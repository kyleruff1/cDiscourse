## QOL-023 — Supabase auth redirect helper (Release 6.5 / deployment hardening)

**Status:** Complete. Auth emails sent by Supabase (signup confirmation, magic link, password reset, admin-triggered reset) now carry an `emailRedirectTo` / `redirectTo` pointing at the deployed origin instead of the dashboard `localhost` Site URL. Issue #109.

- **Deliverables (all in this card):**

Origin-resolution fallback (runtime origin → hosted fallback `https://dev.cdiscourse.com` in non-dev → `http://localhost:8081` only in a true dev build). Scheme/host validation (rejects `file:`/`javascript:`/`data:`/`vbscript:`/`blob:` in every build; rejects `http://` in non-dev; rejects hostless origins). Normalization (trim, strip trailing slashes, lowercase scheme + host, preserve port, discard path). Exports `AuthRedirectKind`, `DEFAULT_AUTH_ROUTES`, `DEV_FALLBACK_ORIGIN`, `HOSTED_FALLBACK_ORIGIN`, `InvalidAuthRedirectOrigin` (error carries only a `reason` enum, never the offending origin string).
  - `src/lib/auth/resolveRuntimeOrigin.ts` — new impure boundary. `resolveRuntimeOrigin()` reads `window.location.origin` → `__CDISCOURSE_RUNTIME_ENV__.EXPO_PUBLIC_APP_ORIGIN` → `process.env.EXPO_PUBLIC_APP_ORIGIN`. `getIsDev()` fail-closes to `false` when `__DEV__` is undefined.

  - `src/features/admin/adminApi.ts` — `adminSendPasswordReset` derives a default `redirectTo` from `buildAuthRedirectUrl` when the caller omits it (signature unchanged; the `admin-users` Edge Function already forwards `redirectTo` — no Edge Function change).
