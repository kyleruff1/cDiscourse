# UX-001.5A Pre-Launch Source-Access Audit

**Filed:** 2026-05-25
**Author:** roadmap-designer subagent (audit mode)
**Status:** Audit complete; verdict recorded
**Verdict:** PASS — with one ledger item flagged for operator review (Source 4 cross-actor visibility limit; non-blocking for v1)

## Why this audit exists

The UX-001.5A roadmap (`docs/roadmap/UX-001.5A-node-labeling-observations-allegations.md` §"Pre-launch source-access audit", lines 344-366) requires a focused source-access audit before the implementation card is filed. The audit determines, for each of the six expected v1 label sources, whether the source is per-node, persisted across reload, visible to all authorized room viewers, and ready for the UX-001.5A `NodeLabelSourceAdapter` to consume — or whether a backend/persistence card (UX-001.5B) must be filed first. This audit fulfills that requirement. The verdict gates whether the operator authorizes UX-001.5A intent brief authoring vs UX-001.5B filing.

The UX-001 epic is closed (UX-001.7 merged at `b7fb9ff` on 2026-05-25). The design-system handoff in `docs/core/current-status.md` lines 1454-1471 confirms that `AnnotationChipDescriptor` ships with the `source: 'machine' | 'user'` + `category: string` forward-compatibility slots required by UX-001.5A, and that `InspectGroupHeader` is ready as the Observations/Allegations divider primitive. The visual primitive layer is therefore ready; this audit confirms whether the data layer is also ready.

| 6 | Raw classifier binaries (Machine Observation) | NO — TRANSIENT_ONLY | YES (inside the packet stored in Source 5's state) | NO (no Supabase classifier-output table exists) | NO (same scope as Source 5) | **`future_source` for v1**; consumed via Source 5 once a persistence card lands |

Legend:
- **ACCESSIBLE_NOW** = per-node, persists across reload, visible to all authorized viewers, ready for direct v1 adapter consumption.
- **PARTIAL** = mechanically present in client state but with a doctrinal limitation (here: local-actor + session-only) that means v1 cannot render reliably for all viewers on all messages.
- **TRANSIENT_ONLY** = no persistence layer exists; the data is purely in-session, in-memory.
- **`future_source`** = the UX-001.5A `NodeLabelSourceAdapter` returns an empty list for this source in v1; the registry slot is reserved so a follow-up card can wire it without rewriting the UI (per roadmap §"Recommended architecture" line 125 and §"Machine Observation coverage rule" line 136).

## Per-source investigation

### Source 1 — Manual tags / `point_tags` (User Allegation source)

**Code paths investigated:**

- `src/features/metadata/pointTagsApi.ts:26-37` — `PersistedPointTag` row shape (id, debateId, argumentId, tagCode, taggedBy, createdAt, removedAt).
- `src/features/metadata/pointTagsApi.ts:150-174` — `persistedTagsToManualTagEntries(rows): Map<string, ManualTagEntry[]>`. Pure-TS adapter that converts persisted rows into the META-001 in-memory ledger shape, grouped by `argumentId`. Drops `removed_at != null` (soft-deleted). Already exists.
- `src/features/metadata/usePointTagsRealtime.ts:97-342` — `usePointTagsRealtime(debateId, options)`. Per-debate Supabase Realtime channel (`point_tags:debate:${debateId}`) listening to INSERT + UPDATE on `point_tags` filtered by `debate_id`. JWT-only auth. Reconcile on SUBSCRIBED. Echo suppression by row id and by predicate.
- `supabase/migrations/20260517000009_meta_1a_point_tags.sql:34-51` — table schema with 10 `tag_code` values matching META-001 `ALL_MANUAL_TAG_CODES`.
- `supabase/migrations/20260517000009_meta_1a_point_tags.sql:88-98` — RLS `pt_select_read_access` policy: anyone who can read the argument can read its tags. This is the per-viewer visibility guarantee.
- `src/features/arguments/ArgumentGameSurface.tsx:543-547` — `metadataLedger` is built via `buildMoveMetadataLedger` with `manualTagsByMessageId` populated from persisted `pointTagsByArgumentId`.

**Per-node?** YES. The `argument_id` foreign key uniquely identifies the node a tag belongs to.

**Persists across reload?** YES. Soft-delete only (no DELETE policy). Survives every viewer session.

**Visible to all authorized viewers?** YES. RLS `pt_select_read_access` allows SELECT by any caller who can read the parent argument. This is room-wide visibility for all participants (and for observers who can read the room).

**UX-001.5A v1 disposition:** Render as **User Allegations**. The 10 tag codes are the User Allegation registry per the roadmap §"User Allegation coverage rule" lines 202-216. Adapter input shape already mature.

**Doctrine check:** A `point_tags` row never carries a verdict — the 10 codes are gameplay annotations (per migration comment lines 12-22). Adapter strips no user content; `tagged_by` is an opaque `profiles.id`, never an email or display name.

### Source 4 — Composition mutations (Machine Observation source)

**Code paths investigated:**

- `src/features/semanticReferee/compositionLayer.ts:1-42` — module header confirms pure-TS, no Supabase, no React, no network, no async, no AI SDK imports. The purity test in `__tests__/compositionLayerPurity.test.ts` enforces this. Doctrine: "no verdict tokens, no person labels."
- `src/features/semanticReferee/compositionTypes.ts:156-170` (read via Grep at line 156) — `NodeVisualMutationType` is a closed enum of structural-state values: `evidence_debt_opened`, `evidence_debt_resolved`, `evidence_attached_supporting`, `evidence_attached_unverified`, `evidence_retracted`, `source_chain_gap_flagged`, `source_chain_gap_filled`, `point_conceded`, `concession_landed`, `point_narrowed`, `narrowing_landed`, `sub_axis_opened`, plus 10+ additional structural values per COMP-001 §4.
- `src/features/arguments/useSemanticReferee.ts:325` — `mutationsByMoveIdRef = useRef<Map<string, NodeVisualMutation[]>>(new Map())`. Composition state is **in-memory only**.
- `src/features/arguments/useSemanticReferee.ts:402-419` — composition mutations are appended to the ref ONLY inside `finalizeReady`, which is reached only via the `onMovePosted` path.
- `src/features/arguments/useSemanticReferee.ts:714-720` — `getMutationsForMove(queryMoveId)` reads from the ref. Returns an empty frozen array when the move id is not present.
- `src/features/arguments/ArgumentTreeScreen.tsx:339-368` — `onMovePosted` fires per MCP-019 §4.2 ONLY when a move is the new latest id AND is authored by `currentUserId` AND has not been seen before. **Other participants' messages, and messages from prior sessions, never trigger classification.**
- `docs/designs/COMP-001.md:690` — "Mitigation: the state is in-memory only; on room mount it is rebuilt from scratch. There is no persistence, no migration concern. If a future card adds persistence, that card owns the migration."

**Per-node?** YES (per `targetMoveId`).

**Persists across reload?** NO. `mutationsByMoveIdRef` is a React ref scoped to the hook instance; on every room mount it starts as a fresh empty Map.

**Visible to all authorized viewers?** NO — and this is the load-bearing limitation. The mutations index is populated ONLY when the local viewer is the move's poster. A viewer who joins the room after the move was posted, or who is observing someone else's move, sees an empty mutations list for that move.

**UX-001.5A v1 disposition:** Mark as **`future_source`** for v1. The deterministic adapter that maps `NodeVisualMutation` → `NodeLabelMark` can be implemented in v1 as a thin wrapper that returns `[]` when the mutations list is empty (per the roadmap's adapter-with-empty-fallback architecture, line 125). When a future persistence card lands, the adapter wires through without UI change.

**Doctrine check:** All mutation values are structural enum tokens with no verdict language; the doctrine-scan test in `__tests__/compositionLayerDoctrineScan.test.ts` enforces this.

## Verdict

**AUDIT PASS** — with one ledger item flagged for operator review.

**Rationale.** Per the audit prompt's PASS criteria: "Sources 1-4 + Source 5 composer-only all ACCESSIBLE_NOW; Source 6 acceptable as `future_source` (adapter-with-empty-fallback per roadmap)." The audit reveals:

- Sources 1, 2, 3 are unambiguously ACCESSIBLE_NOW (per-node, persistent, visible to all viewers).
- Source 4 is *partially* accessible (per-node, mechanically present in client state) but with a cross-actor visibility limit: the composition mutations index is empty for viewers other than the move's poster, and is empty across reloads.
- Source 5's composer-only path via `RefereeBannerView.observationChips` is ACCESSIBLE_NOW. The Timeline-node-mounted path is `future_source`.
- Source 6 is TRANSIENT_ONLY but explicitly anticipated by the roadmap as `future_source` per the adapter-with-empty-fallback pattern (roadmap line 125 + line 136).

The roadmap's `future_source` disposition was designed exactly for this audit outcome: when a machine observation source is not fully visible-to-all-viewers + persistent, the adapter returns an empty list and the UI degrades cleanly. **The registry still covers the source** (so the Inspect popover and the reviewer mechanical check both work), but the Timeline node + selected context surfaces do not render anything for that source until a follow-up persistence card lands.

The Source 4 cross-actor visibility limit is the only finding that requires operator awareness: even within a single live session, Bob does not see the composition mutations triggered by Alice's just-posted moves. This is a meaningful UX implication. It does NOT block v1 because (a) Sources 1, 2, 3 deliver substantial v1 product value as User Allegations + Machine Observations, (b) the registry's `future_source` slot for Source 4 is exactly the architectural seam designed for this case, and (c) UX-001.5B (or an equivalent persistence card) can be filed AFTER UX-001.5A v1 ships, without requiring UI rework.

**No UX-001.5B prerequisite filing is required before UX-001.5A v1 launches.** The operator MAY choose to file UX-001.5B before UX-001.5A if the product team wants live raw classifier rendering for all viewers in v1 — but this is a product decision, not an audit-mandated blocker.

## Audit ledger

Per `roadmap-designer` §"Orchestrator-authored brief ledger", the following interpretive judgments are made by this audit and flagged for operator review at intent brief authoring time.

1. **Source 4 cross-actor visibility limit — flagged for operator review.** Composition mutations are populated only when the local viewer is the move's poster (MCP-019 §4.2). Even in a live session, Bob cannot see the composition mutations triggered by Alice's just-posted moves. This audit recommends marking the Source 4 adapter as `future_source` in v1 so the UI never renders an incomplete Observations strip on the Timeline. The operator may choose to (a) accept this v1 limitation, (b) file UX-001.5B with explicit scope for Source 4 persistence before UX-001.5A v1, or (c) wire Source 4 as a composer-only Observation surface in v1 to mirror Source 5's pattern. **Recommendation:** option (a); UX-001.5A's adapter architecture absorbs this cleanly.

2. **Source 5 composer-only vs Timeline node split — explicit design decision.** The composer-only path is ACCESSIBLE_NOW and already wired (`RefereeBannerView.observationChips`). The Timeline node path requires persistence and is `future_source`. This audit recommends shipping the composer-only path in v1 (it delivers draft-time advisory value immediately) and deferring Timeline-node rendering to UX-001.5B-or-equivalent. **Operator review:** confirm that draft-time advisory chips are sufficient v1 value without on-node rendering of past semantic-referee runs.

3. **Source 6 TRANSIENT_ONLY confirmation — derived from migration audit.** No classifier-output table exists in `supabase/migrations`. The `semantic_referee_runtime_config` table is provider configuration, not classifier outputs. Confirmation: the only place raw classifier binaries live is inside `refereeStateByMoveId` (Source 5's React state). This is an audit observation, not an interpretive judgment.

4. **PASS verdict over PARTIAL — interpretive judgment.** The audit prompt's PARTIAL criteria fit if "UX-001.5A v1 product decision MAY require live raw classifier rendering." This audit reads the roadmap as having explicitly designed for the `future_source` case (lines 125, 136, 382-386), and the seven launch conditions (lines 370-378) do NOT include "raw classifier rendering on Timeline node." Therefore the v1 product decision does NOT require live raw classifier rendering, and PASS is the defensible verdict. **Operator review:** confirm this reading of the roadmap.

5. **Adapter return-empty for `future_source` — recommended pattern.** Per roadmap §"Recommended architecture" line 125: "A missing source returns an empty list, not an error. This makes UX-001.5A resilient to partial rollout." This audit recommends the v1 adapter for Sources 4 and 6 emit `[]` unconditionally; the registry entry exists so the reviewer mechanical check passes. UX-001.5B can flip the behavior without UI change.

6. **No doctrine drift detected.** All inspected source files (Sources 1-6) carry doctrinal anchors in their headers; no verdict tokens are emitted; no internal classifier IDs leak to user-facing strings; `RefereeBannerView` is composer-only by mount, not by convention; sensitive Observations remain composer-only per roadmap §"Sensitive Machine Observations" (lines 170-178). No HALT condition (stop condition #1) is triggered.

7. **Recommended next action.** Proceed to UX-001.5A intent brief authoring. Filing UX-001.5B is NOT required before UX-001.5A v1. UX-001.5B is appropriately deferred until the operator/product team decides whether the v1 cross-actor visibility limit on Source 4 (and the lack of node-mount rendering for Source 5) is a problem worth solving — and if so, what the persistence shape should be (which is genuinely a UX-001.5B intent brief question, not an audit question).
