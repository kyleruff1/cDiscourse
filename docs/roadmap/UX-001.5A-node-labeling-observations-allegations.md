# UX-001.5A — Node Labels: Machine Observations and User Allegations

## Status

Planned adjacent UX-001 card. Launches after UX-001.5 ships and after a source-access pre-launch review confirms which data sources are available for v1 rendering. The card is filed as roadmap direction rather than as a launch-ready implementation card; the implementation issue will be filed only after the launch conditions below are satisfied.

**Decision date:** 2026-05-25
**Decision context:** UX-001 epic in flight; UX-001.1 shipped; UX-001.2 in flight (Timeline-first viewport repair with chrome-nullification authority).

## Decided terminology

Machine-created labels are **Observations**. User-created labels are **Allegations**. This is the schema boundary, the display boundary, the accessibility boundary, and the doctrine boundary for node labels in the UI.

A node can have both:
- Observation: Source gap
- Allegation: Needs source

The UI must never collapse these into a generic "tags" bucket. Generic tags cause the exact confusion this card resolves: users cannot tell whether the app observed something, another participant alleged something, or the system is making a judgment. UX-001.5A solves this permanently.

## Decided card boundary

UX-001.5 ships visual annotation primitives (node chip strips, badges, rings, outlines, edge highlights, overflow chips, Inspect grouping surfaces, accessible chip labels). UX-001.5A ships the source-aware label presentation system that consumes those primitives.

UX-001.5 = visual grammar for node annotations.
UX-001.5A = semantic source model: Observations vs Allegations.

This split keeps each card bounded and gives UX-001.5A a clean visual surface to render into. It avoids enlarging UX-001.5 into an L/XL card that risks delaying the UX sprint, and it avoids deferring UX-001.5A indefinitely (which would leave the semantic classifier layer functionally hidden from users after the UX overhaul).

## Decided data strategy

UX-001.5A v1 consumes currently available client-side sources only. Unavailable raw classifier outputs are marked `future_source` in the registry and handled by follow-up backend/persistence work if needed. The card does not become a backend card by default.

This is the scalable path. The registry and adapter architecture can absorb future machine classifier sources without rewriting the UI. The first release does not depend on making every classifier binary persistent.

## Decided semantic referee treatment

Semantic referee outputs are Machine Observations with `source: "semantic_referee"`. No third top-level label kind in v1.

The top-level taxonomy remains exactly two kinds:
- `machine_observation`
- `user_allegation`

Adding a third top-level class would dilute the clarity of the system and force users to learn a distinction that is more internal than practical. The source distinction appears in Inspect, not as a third default visual category. If future user testing shows that semantic-referee observations need a higher-prominence treatment, that can be added later as a visual tier inside Machine Observations.

## V1 source priority

UX-001.5A v1 consumes these sources in this order:

1. **Manual tags / point_tags** → User Allegations. The most mature user-created label source. META-1A point_tags path provides persistence; hydrated into the room shell.
2. **Auto metadata** → Machine Observations. Deterministic system-derived labels.
3. **Lifecycle state** → Machine Observations. Lifecycle calculations from LIFE-001 and related cards.
4. **Composition mutations** → Machine Observations. Semantic referee / composition layer mutations available via `useSemanticReferee.getMutationsForMove(moveId)` style consumer surface.
5. **Raw classifier binaries** → Machine Observations. Only if the pre-launch source-access audit confirms availability without backend work. Otherwise marked `future_source`.

## V1 UI surfaces

- **Timeline node strip:** compact chips, max one Observation + one Allegation + overflow count
- **Selected context grouped labels:** up to three Observations + three Allegations + overflow
- **Inspect full provenance:** all labels grouped by kind and source, with full detail
- **Composer:** draft-specific advisory observations (e.g., "pause suggested," "person/intent shift")
- **Future Act suggested-action seam:** registry preserves `suggestedAction` field for UX-001.4 or later card to consume

## Recommended architecture

Suggested module structure (names can adjust to repo convention):

- `src/features/nodeLabels/nodeLabelTypes.ts` — source-of-truth types
- `src/features/nodeLabels/machineObservationRegistry.ts` — raw machine keys → Observation labels
- `src/features/nodeLabels/userAllegationRegistry.ts` — raw user tag keys → Allegation labels
- `src/features/nodeLabels/nodeLabelSourceAdapters.ts` — source-adapter functions (missing source returns empty list, never breaks)
- `src/features/nodeLabels/nodeLabelPresentationModel.ts` — combines labels, deduplicates, filters by surface, enforces display caps
- `src/features/nodeLabels/nodeLabelPriorityModel.ts` — priority and overflow rules
- `src/features/nodeLabels/NodeLabelStrip.tsx` — Timeline node chip component
- `src/features/nodeLabels/NodeLabelInspectSection.tsx` — Inspect full-detail component

The adapter architecture is the scalability layer. A missing source returns an empty list, not an error. This makes UX-001.5A resilient to partial rollout.

## Machine Observation coverage rule

Even if v1 cannot render every classifier output, the registry covers every known classifier ID. Each known machine classifier ID has one of these explicit dispositions:

- `rendered_now` — appears on default Timeline node
- `inspect_only` — appears only in Inspect detail
- `composer_only` — appears only in the composer (e.g., sensitive draft observations)
- `hidden_sensitive` — known but not rendered (e.g., person/intent shifts)
- `future_source` — registry entry exists but source not yet available client-side
- `intentionally_silent` — known and deliberately not surfaced

This gives the reviewer a mechanical check: every classifier ID is accounted for, not every classifier ID is visible, no classifier ID leaks raw.

## User Allegation coverage rule

Every existing manual tag becomes a User Allegation:

| Raw manual tag | Allegation label |
|---|---|
| `needs_source` | Needs source |
| `needs_quote` | Needs quote |
| `definition_issue` | Definition fight |
| `scope_issue` | Scope challenge |
| `causal_mechanism` | Mechanism challenge |
| `evidence_debt` | Evidence debt |
| `concession_offered` | Concession offered |
| `narrowed_claim` | Narrowed claim |
| `tangent` | Tangent / side issue |
| `ready_for_synthesis` | Ready for synthesis |

The UI does not render these as neutral tags. They are participant-applied allegations and the provenance must always be present:

- Compact: `Alleged · Needs source`
- Standard: `Allegation: Needs source`
- Inspect: `User Allegation: Needs source — Applied by: Negative participant`

## Copy rules

Machine copy uses one of: "Observed," "Machine observation," "System observed," "Referee note," "Structure note."

User copy uses one of: "Alleged," "User allegation," "Participant allegation," "Applied by user."

Banned across both kinds (in addition to existing doctrine ban-list): "True," "False," "Correct," "Incorrect," "Proven," "Disproven," "Winner," "Loser," "Bad faith," "Dishonest," "Manipulative," "Liar," "Propaganda."

Labels describe the move, edge, branch, or draft. Labels do not describe the person.

Good: `Observation: Person / intent shift` (describes the move)
Bad: `User is attacking intent` (describes the person)

Good: `Observation: Source gap` (describes the source chain)
Bad: `This is unsupported misinformation` (verdict on truth)

Good: `Allegation: Needs source` (describes the allegation)
Bad: `This claim is false` (verdict on truth)

## Confidence and sensitivity gating

Machine Observations from AI classifier sources render on default Timeline nodes only when:

- The classifier fired positively
- The confidence is `medium` or `high`
- The classifier is not intentionally silent for this label
- The label is not duplicative of a higher-priority label
- The label is safe for user-facing display (not on the sensitive list above)
- The label describes the move/edge/branch/draft, not the person

Low-confidence Observations can be hidden or shown only in Inspect under a restrained heading such as "Additional weak signals."

## Deduplication rules

The label system accounts for collisions. Examples:

- `Evidence debt` can be a Machine Observation (`opens_evidence_debt_marker`) or a User Allegation (`evidence_debt`).
- `Ready for synthesis` can appear as lifecycle, runner status, or user tag.
- `Source requested` can appear as lifecycle or auto metadata.
- `Branch suggested` can appear as auto metadata, lifecycle, or classifier output.
- `Mechanism challenge` can be a user-applied Allegation or semantic axis Observation.

The UI does not render labels by text. It renders by kind + source + scope + priority + raw key + display surface. The presentation model deduplicates intelligently:

- Same kind, same raw key → one label
- Same text, different kind → show both, distinguished
- Same text, same kind, different source → show one with the higher-priority source; expose the other source in Inspect

## Non-goals

UX-001.5A does NOT:

- Change classifier logic or semantic referee prompts
- Add new AI calls
- Introduce truth or scoring claims
- Expose raw classifier IDs to normal users
- Let user Allegations appear machine-certified
- Let Machine Observations appear user-applied
- Change backend schema (unless a separate prerequisite card explicitly authorizes it)
- Redesign Timeline layout (UX-001.2 owns that)
- Redesign composer schema (UX-001.3 owns that)
- Redesign Act / Inspect / Go (UX-001.4 owns that)
- Redesign visual annotation primitives (UX-001.5 owns those)
- Block UX-001.2, UX-001.3, UX-001.4, or UX-001.5

## Pre-launch source-access audit

Before UX-001.5A is filed as an implementation card, a 15-30 minute focused audit answers:

1. Where are manual tags hydrated in current client state? What prop or model delivers `pointTagsByArgumentId` or equivalent into the argument surface?
2. Which machine-side semantic outputs are currently available to the argument surface? Does `useSemanticReferee.getMutationsForMove(moveId)` provide enough data to render initial Machine Observations?
3. Are raw classifier binaries accessible per node? Persistent across reloads or only transient? Visible to all room participants or only to the local viewer whose hook ran?
4. Which source should UX-001.5A v1 consume by default?
5. Which source requires a backend follow-up (UX-001.5B)?
6. Which files should remain read-only because they belong to semantic classifier logic rather than UI presentation?

The audit output is a source-access matrix:

| Source | Available? | Scope | UX-001.5A disposition |
|---|---|---|---|
| Manual tags / point_tags | TBD | user labels | render as Allegations |
| Auto metadata | TBD | machine/system labels | render as Observations |
| Lifecycle state | TBD | machine/system labels | render as Observations |
| Composition mutations | TBD | machine/referee labels | render as Observations |
| Raw classifier binaries | TBD | machine/AI labels | render now or future_source |
| Referee banner codes | TBD | machine/referee labels | inspect-only or not used |

The audit does not re-open the product terminology. It only determines the first implementation subset.

## Launch conditions

UX-001.5A can launch when ALL of the following are true:

1. UX-001.5 has shipped visual annotation primitives.
2. Manual tags are confirmed available to the node surface.
3. At least one machine observation source is confirmed available to the node surface.
4. The source-access audit identifies whether raw classifier binaries are available or future-source only.
5. The registry can cover all classifier IDs without requiring all classifier IDs to display.
6. Inspect has enough structure from UX-001.4 to show full label provenance.
7. Timeline layout from UX-001.2 can tolerate compact node chips without reintroducing clutter.

## Escalation rule

If the pre-launch audit proves that no machine observation source is client-accessible, UX-001.5A remains M-effort for User Allegations plus deterministic Observations, and a separate backend/persistence card is filed:

**UX-001.5B — Persist or expose machine observation sources for node labels**

UX-001.5A does NOT automatically expand into L/XL just because some raw classifier outputs are unavailable. The scalable architecture is more important than rendering every machine label on day one.

## Implementation acceptance criteria

UX-001.5A ships when ALL of these hold:

1. All user-applied manual tags render as Allegations, not generic tags.
2. All machine/system/referee-derived labels render as Observations, not generic tags.
3. Timeline nodes visually distinguish Observations from Allegations.
4. Selected context groups Observations and Allegations separately.
5. Inspect shows full provenance for each label.
6. Raw classifier IDs do not appear in normal UI.
7. Raw manual tag codes do not appear in normal UI.
8. Every known manual tag code is covered by the User Allegation registry.
9. Every known machine classifier ID is covered by the Machine Observation registry as rendered, inspect-only, composer-only, hidden, intentionally silent, or future-source.
10. Sensitive machine outputs are not default Timeline chips.
11. Low-confidence machine outputs are not default Timeline chips.
12. A node never renders an unbounded chip list.
13. Default Timeline node display is capped at one Observation + one Allegation + overflow.
14. The label model preserves source and provenance.
15. User Allegations never appear machine-certified.
16. Machine Observations never appear user-applied.
17. No label copy implies truth, proof, correctness, victory, defeat, dishonesty, bad faith, or user intent.
18. No new AI call is introduced.
19. No classifier prompt is changed.
20. No backend schema is changed unless a separate prerequisite card explicitly authorizes it.
