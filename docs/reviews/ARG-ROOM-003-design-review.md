# ARG-ROOM-003 — design review
**Reviewer:** workflow design-reviewer · **Reviewed:** 2026-06-13 · **Design:** docs/designs/ARG-ROOM-003-CREATE-ROOM-UX.md · **Issue:** #614
**Lane:** UI — accessibility matters (StartArgumentPage.tsx)
**Verdict:** changes-requested

## Summary
This is a high-quality, unusually well-grounded design: it correctly treats the card as live-surface exposure over five shipped primitives rather than a rebuild, its file:line citations are essentially exact (I spot-checked 12+ and all but a trivial off-by-one passed), and its doctrine/security posture (no service-role in client, raw token inviter-only, uniform post-create copy, no hard delete) is sound. It does not earn an immediate green light, however, because two cross-card seams are unresolved in a way that determines a large fraction of the implementation: (a) the reused `validateInviteEmailInput` cannot satisfy the "2+ addresses rejected" matrix row, so that logic is net-new in ARG-ROOM-001 and the design under-states it; and (b) the entire post-create failure/retry UX is coupled to a two-call create→invite sequence that the design's own Open Question #1 leaves undecided against ARG-ROOM-002's enforcement model. These are coordination/sequencing issues, not design defects or doctrine violations — hence changes-requested, not block.

## Completeness & edge cases
1. **"2+ rejected" is NOT covered by the reused validator.** `validateInviteEmailInput` (`inviteCopy.ts:189`) is anchored `^[^\s@]+@[^\s@]+\.[^\s@]+$`; given `a@b.com, c@d.com` it returns the **generic** "Enter a valid email address." string, not a multi-invite signal. So the matrix's `2+ → helper_multiple_invites` row (and test 6, which asserts the exact `helper_multiple_invites` string) **requires net-new address-counting in ARG-ROOM-001's `deriveArgumentRoomCreation`** (split on comma/semicolon/whitespace → count>1 → reason `multiple_invites`). The design lists `validateInviteEmailInput` as "reused verbatim" and lists `multiple_invites` only as a reason enum, but never states that the count logic is new in 001. This seam must be explicit or the matrix row silently won't render the intended copy.
2. **Post-create failure UX is contingent on the two-call model (Open Question #1).** The §"Submit sequence" room-created-before-invite flow, the "room exists but invite failed" retry panel, and tests 11/13/14 only make sense if 002 keeps creation as two client calls. If 002 lands as an atomic `create-argument-room` Edge call (its own listed option), there is no "room created but invite failed" intermediate state and the retry panel premise disappears. The design's claim that the sequence "can later collapse to one Edge call without changing any control or copy" holds for the radiogroup/email/capacity portions but **not** for the failure/retry UX. Resolve OQ#1 before building `handleSubmit` + the post-create panel.
3. **Capacity-constant sourcing window.** The design assumes ARG-ROOM-001 owns/exports the reconciled `5`, but roadmap §4.1(2) assigns the `publicSeatModel.ts:62` (6→5) reconciliation to **ARG-ROOM-005**, not 001. So between 001 and 005 the codebase legitimately holds two cap numbers (validator 5, `publicSeatModel` 6). The design is on the right side (reads the validator, never `publicSeatModel`), but review must confirm 001 actually exports `{ total, openSlots, reservedSeat }` so no second UI constant is introduced (the design's OQ#3).
4. **Idempotency of retry is implicit.** Retry calls `onInvite` with the same `debateId`+email; the `argument_room_invites_one_live` partial-unique constraint + `reused` make a double-fire safe, but the design should state this so the implementer doesn't add client-side dedupe.
5. **Private room can persist with zero invites if the user dismisses after an invite-send failure** (design Risk #6) — correctly deferred to 002's durable guarantee, acceptable given the dependency order, but worth a confirmation that 002's grace-window/atomic choice actually closes it.

## Actionable comments
1. **[blocking-for-GATE-A]** Resolve Open Question #1 (ARG-ROOM-002 enforcement shape: deferred-check/trigger vs atomic `create-argument-room` Edge call) **before** implementing §"Submit sequence + failure semantics" and the post-create retry panel. The two-call flow + "room exists, invite failed" retry UX (tests 11/13/14) is not portable to an atomic model. The radiogroup/email/capacity-display portions may be built independently of this answer.
2. **[should]** §"Existing shipped state" / §"Product contract": state explicitly that the **2+ address detection is net-new in ARG-ROOM-001**, since the reused `validateInviteEmailInput` (`inviteCopy.ts:189`) returns the generic invalid-email string for a comma-paste and cannot produce `multiple_invites`. Without this, matrix row 5 / test 6 won't render `helper_multiple_invites`.
3. **[should]** Reconcile the section header `who_can_join_label: 'Who can join'` with the reused radiogroup label `ROOM_VISIBILITY_COPY.group_label = 'Who can see this argument'` (`gameCopy.ts:1610`). As written, a screen-reader announces a **read-access** label ("see") inside a **participation** section ("join") — and per QOL-039's own column comment + roadmap §5, visibility is access, capacity is activity. Either give the radiogroup a join-framed `accessibilityLabel` from the new copy block or confirm the announced label is intentional.
4. **[should]** Confirm ARG-ROOM-001 exports the capacity numbers (`{ total, openSlots, reservedSeat }`) and the reconciled `5`, since the roadmap assigns the `publicSeatModel.ts:62` 6→5 edit to ARG-ROOM-005 (design OQ#3 / Risk #5). Hold 003 from reaching users until 001's exported `total` reads 5.
5. **[should]** Expand the test plan to pin all 8 button-state matrix rows (add Private-malformed and the second-visibility 2+ cases) and to assert the multi-invite vs invalid-email reasons are distinct.
6. **[nit]** Fix the `:122-127` citation to `:123-127` (line 122 is `const trimmed = declaration.trim();`, not part of the `CreateDebateInput` literal).
7. **[nit]** §"Submit sequence" / doctrine note: `CreateRoomInviteResponse` carries `reused`/`status`/`expiresAt` too, not "only `notification` + `inviteLink`." Harmless, but state that post-create copy must not branch on `reused`/`status` to keep the enumeration guard airtight.

## Ready to implement?
Not green-light as-is, but close. The design itself is sound, doctrine-clean, and genuinely build-on-shipped — the changes requested are cross-card coordination, not rework of this document. Two items gate GATE-A: (1) the card cannot be imported until ARG-ROOM-001 lands and must confirm 001 implements address-counting + exports the reconciled capacity numbers (comments 2, 4), and (2) the submit-sequence/post-create-panel portion must wait on Open Question #1 (comment 1) so the page is built against 002's final create shape once. The visibility radiogroup, the email field, the validator-driven button state, the capacity explainer, and the a11y/copy/doctrine surface can proceed as soon as 001's validator exists. Address comments 1-5 (and ideally 3 for screen-reader coherence) and this is a green light.
