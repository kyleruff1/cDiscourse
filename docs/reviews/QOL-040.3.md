## Summary

Surgical, additive-only deep-link node pre-activation extension. One optional
field (`entryHintForArgumentId?: string`) is appended to `GalleryEntryHint`;
a new pure-TS helper `buildDeepLinkEntryHint(link)` constructs the hint from a
`NotificationDeepLink`; a new pure-TS deriver `deriveInitialActiveMessageId`
implements the priority rule (hint id → activate policy → latestId). App.tsx's
`handleOpenNotificationDeepLink` callback is the sole producer; the
ArgumentGameSurface room consumer reads the new field first and falls through
to the existing GAL-002 / Stage 6.4 cascade when the id is absent. No protected
file modified, no schema/Edge Function/migration, no RLS side-channel. The
fallback path is naturally covered by the composition of debate-list RLS +
argument-row RLS + soft-delete filter (`status === 'posted'`) — exactly as the
design's Q3 analysis specified. Test coverage exceeds the design's §9.3
estimate (27 vs the +7 minimum).

## Suggestions (non-blocking)

1. **Consider an integration test for App.tsx's full hand-off.** The routing logic is well-tested via the pure helper, but the actual `setEntryHint(buildDeepLinkEntryHint(link))` → `selectDebate(target, side)` sequence in App.tsx is not directly tested. Operator may wish to add an end-to-end test in a future card (e.g. QOL-040.4 or as part of broader notification integration tests). Not a Block because the helper + deriver tests cover the substantive logic, and the App.tsx integration is mechanically straightforward (one line each).

2. **Case 8 (already-in-room) deferral is correctly out-of-scope.** Design §12 #4 documents this gap explicitly: "the room does not jump to the hinted argument when the user is already inside it." This is acceptable for v1; a follow-up could add a `useEffect` that watches `entryHint` and explicitly calls `setActiveMessageId`. Operator may choose to file a v1.1 card if this surfaces as a real-world complaint.

3. **The `code: 'watch_first'` value on the notification-path hint is semantically a bit odd** (the notification recipient isn't "watching first" — they're being routed to a specific move). Design §5.2 explains the rationale: the field is unused on the notification path because the banner is suppressed, so the value is irrelevant for behavior. Code is fine as-is; the operator may want a more neutral code like `'notification_deep_link'` in a future iteration (would require a one-line addition to `GalleryEntryHintCode` and the related preset/dock tables, plus a banner-suppression rule by code rather than by `verbPhrase` truthiness — non-trivial).
