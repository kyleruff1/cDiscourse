## Completeness & edge cases
The decision record is structurally complete and the reservation→accept atomicity (R4 + the reservation-accounting test) is well-reasoned: a live pending invite consumes a seat, accept must flip status and insert in one transaction so the count never transiently hits 6. Three real gaps an implementer will hit:

- **Counted-quantity definition diverges from the roadmap.** D4 counts `side IN ('affirmative','negative')` + creator-as-seat-1, and R6 *explicitly excludes* the creator's `moderator` row to avoid double-counting. Roadmap §1 instead defines active = `side IN ('affirmative','negative','moderator')` (i.e. `!= observer`). In the default flow (creator is the lone moderator) both yield creator + N debaters, but they are textually different formulas and diverge if a second `moderator` row exists (the data model permits one — `joinDebate` accepts `side='moderator'`). For a keystone the cards "must not re-litigate," the single binding formula must be unambiguous.
- **D8 "atomically attach exactly one invite or reject" vs a non-atomic client create path.** `createDebate` is a client INSERT (verified debatesApi.ts:107-151) and the Data-shape section describes "the create flow then makes … a `manage-room-invite create` call and threads its `rawToken`" — a client-orchestrated two-step that cannot be atomic; a failure after the debate insert leaves exactly the hidden-solo-draft state D8 forbids. The roadmap §5 already resolves this with a `create-argument-room` Edge, but the ADR never names that transactional boundary.
- **Over-cap join failure semantics.** The cap rejection fires as a Postgres RLS/check violation; today's `joinDebate` (debatesApi.ts:160-175) maps only `23505`. Nothing pins how the over-cap denial is distinguished from a generic RLS error so the UX can render the doctrinal "follow as an observer" path rather than a bare failure.

Matrix coverage is exact and internally consistent (private 0 = NO; private 1 = 2 total/0 open; public 0 = 5/4 open; public 1 = 5 total/1 reserved/3 open; any 2+ = NO), and the arithmetic (creator + reserved + open = total) checks out on every row.

1. **[should]** D4 (§ "Chosen approach") vs roadmap §1 — reconcile the *active-participant counted set* to one binding formula. The roadmap's `side != 'observer'` (`affirmative|negative|moderator`) form is simpler and self-handles the creator's +1 without R6's exclusion special-case; whichever you pick, state it as the single truth and flag the other document's line for alignment, since the cap helper SQL derives directly from it.
2. **[should]** Dependent-card table + Divergence #1 + R2 vs roadmap §4.1(2) — the `publicSeatModel.ts:62` 6→5 edit is assigned to the **cap-enforce card (ARG-ROOM-002)** by the ADR but **recommended for the surfacing card (ARG-ROOM-005)** by the roadmap. Pin one owner and one timing in the keystone; if enforcement (002) lands the literal 5 server-side while the derived constant flips only in 005, the seat strip reads "of 6" while the server caps at 5 (an R1/R2 drift window). Pair this resolution with R1.
3. **[should]** D8 + Data/API shape — record that "atomically attach exactly one invite or reject" forces room creation through a transactional boundary (the roadmap's `create-argument-room` Edge / an RPC), or soften the wording to "create-or-compensate" (insert debate → mint invite → on mint failure archive/roll back the debate). As written, "atomic" is asserted against the verified client `createDebate` two-step (debatesApi.ts:107-151), which cannot satisfy it.
4. **[should]** Add a consequence/open-question naming the **over-cap join failure signal**: the WITH-CHECK denial must be mapped to a distinguishable client signal so the UX renders the structural "join as an observer" copy (already drafted in the UI-copy section) instead of `joinDebate`'s generic error path.

6. **[nit]** Matrix section — add a one-line pointer from the "Total capacity" column to D4, clarifying that the ceiling is the *membership* count, not the GAME-005 *posting-derived* seat-map count (the vocabulary-reconciliation table already handles it; this just prevents a first-time reader conflating the two).
