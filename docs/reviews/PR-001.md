## Summary

PR-001 ships a "My preferences" popout: a core React Native `Modal` bottom-sheet
of 9 fields, opened from a header gear. It is a pure-code, no-migration UI card.
Preferences persist device-locally in one versioned `AsyncStorage` JSON blob
(`schemaVersion: 1`, keyed `cdiscourse:preferences:<userId>`); only the display
name is an account write, reusing the existing `updateOwnDisplayName` /
`useAccountProfile` path. The implementation is clean and disciplined: pure-TS
model with defensive parsing, a thin storage layer that never throws, an
optimistic hook, centralized copy, and presentational components built only from
RN primitives (no new dependency). Both coordinator corrections were applied
correctly. Doctrine is respected — the popout exposes no role/admin/secret
surface, no preference reaches a scoring or validation gate, and the
notification/colour-blind/side-label stubs carry honest "not available yet"
copy. Verification battery is fully green (typecheck, lint, 4384/4384 tests
across 149 suites). No blocking concerns. The only follow-ups are non-blocking
suggestions and the standard live browser pass for a visible-UI card.

## Suggestions (non-blocking)

1. **`densityToTimelineMode` is a pure identity function** (`return d;`). It is
   well-justified in the docblock (explicit, test-pinned dependency seam) and
   correct — keep it. No action needed; noted only so a future reader does not
   "simplify" it away.
2. **`useUserPreferences` reads `osReduceMotion` once on mount** and subscribes
   to `reduceMotionChanged`. The one-shot read's effect array is `[]` — fine,
   since the OS value is not user-scoped. No change needed; just confirming this
   is intentional (it is, per the design).
3. **Field 9 (side label) and the 3 colour-blind modes are honest persist-only
   stubs.** The copy is genuinely honest ("Your choice is saved and will apply
   when…", "Colour-blind palettes are coming…"). A future card (PR-002 / a VG
   colour card) should pick these up. The design's "Out of scope" section
   already names them — no action for PR-001.
4. **The contact-email read** (`fetchCurrentAuthUser` in `AppRoot`) fires on
   every `signedIn`/`userId` change. This is a single lightweight auth call and
   cancellation-guarded; acceptable. A future card could move it into
   `useAccountProfile` so the email and profile load together, but that is not
   PR-001's job.

### Browser-check focus (live Expo-web verification)

This is a visible-UI card; the operator should verify these specifically:

1. **One-tap open/close.** Sign in → the header gear (`preferences-trigger`) is
   visible in the header right slot → one tap opens the `Modal` bottom-sheet
   (`preferences-popout`); the × (`preferences-close`) or backdrop dismisses it
   in one tap, and the game screen behind it is untouched (no navigation
   occurred). Note: the backdrop `View` is not itself a `Pressable` — confirm
   the × button is an obvious, reachable close affordance.
3. **Display-name write is the real account write.** Edit the name, Save → a
   "Name saved." confirmation shows → open the Account screen and confirm the
   same name is reflected (same `profiles` row, not a local copy).
3. **Visual density genuinely moves the board.** Set density to Compact, open a
   debate room in Timeline mode → nodes sit visibly closer; set Roomy → they
   spread. Confirm Normal looks identical to the pre-PR-001 default (44px) and
   that the active/selected node is unchanged by the switch.
4. **Reduce-motion override actually wins.** Set Reduce motion to On → reopen the
   popout: it should snap open, not slide. With the OS reduce-motion OFF, the
   user's "On" must still win (this is the `resolveEffectiveReduceMotion`
   override path). The "System" option's helper line should reflect the real OS
   value.
5. **Honest-stub copy is visible and not over-promising.** Toggle the
   notification switch, select a colour-blind mode, and look at the side-label
   note — each must show its "not available yet / coming / saved" helper line so
   the user is never misled into thinking an inert control does something.
6. **No console noise.** DevTools console shows no errors, no warnings, and no
   raw internal codes (`high_contrast`, `for_against`, etc.) printed or rendered
   as visible text. Persistence: change several fields, reload the tab, sign back
   in, reopen — every preference is still selected.
