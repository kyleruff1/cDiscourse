## Suggestions (non-blocking)

1. **Deep-link exact-node activation is a known v1 gap.** The implementer honestly disclosed it in the App.tsx comment block and the operator section. The handler calls `selectDebate(target, side)` but cannot pass an explicit `activeArgumentId` because Stage 6.4's entry-hint mechanism does not accept one. This aligns with design §17 follow-up and the design's "in-app route param" framing. A follow-up card (working name QOL-040.2 or a Stage 6.4 extension) can add an `entryHintForArgumentId` field. Not a Block — the notification still routes correctly to the room, and the room screen's existing logic selects the latest move on entry.
2. **Submit-argument insert path uses raw `serviceClient.from('room_notifications').insert(rows)` without explicit error check.** Supabase JS client returns `{ error }` on the result rather than throwing. A non-thrown `error` would fall through silently without a `console.error`. The doctrine ("notification failure must never block the post") is still honoured — the post still returns 201. Logging an explicit `error` branch would tighten observability without changing behavior. Defer to follow-up.
3. **`source_requested` recipient set in submit-argument is broad.** The submit-argument side-effect delivers `source_requested` to all primaries except the author rather than to a specific source-requester. The implementer left a comment acknowledging this as a §17 enrichment that needs the `source_requester_id` payload from EV-002. For v1 the broader delivery is acceptable (any primary who would care sees it); a follow-up tightens it to the specific requester when EV-003 stores that linkage.
4. **`chime_in_posted` actor-name visibility is caller-controlled.** The Edge Function honours whatever `meta.actorNameVisible` the caller passes (only flipping `false` if no name is present, never overriding `true → false`). The design §9 rule 3 lets the caller pass `true` "only when the recipient's room access would already let them see that observer's participation." That gate is the caller's responsibility (GAME-005's chime-in path will set it appropriately). The Edge Function does not currently re-derive room visibility from the DB to enforce the gate independently. Acceptable for v1; tighten when GAME-005 lands if desired.
