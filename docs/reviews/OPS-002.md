## Notes

### Minor design vs implementation drift in Test 2 (corrected by implementer; not a block)

Design §5.3 Group 1 Test 2 specified expected output
`'feat/QOL-040.3-deep-link-node-pre-activation-via-stage-6.4'` for the
QOL-040.3 input, but the actual PS1 slug logic produces
`'feat/QOL-040.3-deep-link-node-pre-activation-via-stage'` (the
truncation lands inside `stage-6-4` and the trailing-dash trim removes
the trailing dash, leaving `stage`). The implementer landed the test
with the **correct PS1-derived expected value**, not the
design-document expected value. This is the right call: the test pins
*actual* PS1 behaviour, which is the source of truth; the design's
arithmetic was off by ~8 chars. No follow-up needed beyond noting it
here; the design defect did not propagate into the test.

### §7 open-question pinning verified

Design §7 flagged the cosmetic side-issue that the PS1 strip regex
expects a dash separator but modern issue titles use a colon, so the
slug doubles the code prefix
(`feat/OPS-002-ops-002-pipeline-operational-hygiene-spa`). Group 1
Test 3 explicitly asserts this current behaviour with a verbose
docstring explaining why the test pins the cosmetic-but-not-broken
output: a future intentional OPS-003 fix to the strip regex must
update this test as a conscious change, not silently drift the four
shipped cards' branch names.

### Regression-introduction smoke test (passed)

Per the launch-prompt reviewer addendum, I introduced a one-line
temporary regression in the TS-mirror helper (truncation length 40 →
50) and re-ran `npm run test -- spawnCardBranchName`. The suite
correctly failed with 2 of 5 Group 1 tests reporting explicit
expected-vs-received output (the QOL-040.3 truncation test and the
OPS-002 colon-double-prefix test). I reverted the change immediately;
8/8 pass again, `git status` clean. The regression test mirror catches
TS-side drift; it does **not** catch PS1-side drift (the test mirrors
PS1 in TS, it does not execute the PS1 script). Design §5.4 openly
acknowledges this scope choice — the bash twin is also uncovered for
the same reason. Future drift on the PS1 side would surface in the
first card whose branch name comes out wrong, which is a slower but
still terminal feedback loop.

### Branch-name oddity for this card

The branch name for OPS-002 itself is
`feat/OPS-002-pipeline-operational-hygiene-spawn-card-bra`, which is
the legacy slug shape the design discovered in §1.1 (the strip regex
is a no-op only for *colon*-separated titles; this card's branch was
created from a dash-separated title interpretation, so the doubling
did not occur here). The branch shipped under the pre-OPS-002
workflow because the implementer for OPS-002 checked out the
designer's pushed branch directly rather than running through the
auto-branch path; this is acknowledged in the launch prompt and is
fine — the new rename step exercises on the **next** card's
implementer subagent, not this one.
