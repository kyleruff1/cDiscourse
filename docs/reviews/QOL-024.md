## Summary

QOL-024 adds an invite-by-email path to admin user management: a new `invite_user`
action on the `admin-users` Edge Function (`handleInviteUser`, calling Supabase's
`inviteUserByEmail`), a pure-builder module `_shared/adminInvitePayload.ts`, a client
`adminInviteUser` wrapper that derives the redirect URL from QOL-023's
`buildAuthRedirectUrl({ kind: 'invite' })`, and an Invite/Password mode toggle on
`AdminCreateUserForm.tsx` (invite is the new default). The implementation matches the
design's API contracts, data model, and doctrine self-check exactly, and resolves all
four card-vs-reality discrepancies the design called out. The security envelope is
correct — `inviteUserByEmail` (not `generateLink`) keeps the invite link/token out of
function memory and the response; the response carries no link/token/email/userId; the
audit payload stores `emailDomain` + `redirectToProvided` (boolean) only; `role` is
`z.enum(['user','moderator'])` so an invite can never mint an admin. The implementer's
one deviation — extracting form logic into pure helpers + a source scan instead of RTL
`render()` — is well-justified, matches the repo's established `.test.tsx` pattern, and
produces coverage materially equivalent to the design's intent. No migration is needed
(`admin_audit_events.action` is `text NOT NULL`, verified). The Edge Function change is
written-not-deployed, an operator step, consistent with Stage 6.1.8's precedent.

### Note on the implementer's one deviation (form test mechanism)

The design's `AdminCreateUserForm.test.tsx` plan specified React-Testing-Library
`render()`. The implementer instead extracted the form's mode/visibility/dispatch logic
into pure helpers in `adminHelpers.ts`, unit-tested those, and added a static source
scan of the form wiring. This deviation was scrutinized and is **accepted**:

1. **The deviation rationale is accurate.** No existing `.test.tsx` in the repo uses
   RTL `render()` — `argumentReplySidecar.test.tsx` and `appHeaderTagline.test.tsx`
   (the files the design itself cited as the RTL exemplars) both use the
   `fs.readFileSync` static-scan + pure-helper pattern. `react-test-renderer@19.1.0` is
   outside `@testing-library/react-native@13.3.3`'s pinned peer range; installing a new
   dependency to satisfy a test mechanism would be out of scope. Avoiding the install
   was the correct call.
2. **The tests exercise the real code path, not a parallel re-implementation.**
   `AdminCreateUserForm.tsx` imports `isInvitingHuman`, `isModeToggleVisible`,
   `isPasswordFieldVisible`, `resolveCreateUserDispatch` from `adminHelpers.ts` (lines
   4-10) and uses them for its live state derivation and submit dispatch (lines 31-33,
   43). The test imports the same functions. The source scan additionally asserts the
   JSX wires each dispatch branch to the right wrapper (`adminInviteUser` /
   `adminCreateBotUser` / `adminCreateUser`).
3. **Every behavior in the design's 13-case list is genuinely covered** — mode default,
   toggle visibility per branch, password-field visibility per mode (including the
   Bot/password regression guards), dispatch branching invite/password/Bot, the
   invalid-email guard, the "Invite sent." status line + `onCreated()`, the
   `invite_email_not_configured` → plain-copy mapping (exercised via the real
   `adminErrorMessage`), and the doctrine ban-list on rendered text. The
   password→invite "switch" case is covered by the pure function being deterministic on
   both inputs — there is no stateful path to lose. Coverage is materially equivalent
   to the design's intent, not weaker.

## Suggestions (non-blocking)

1. The static source-scan tests in `AdminCreateUserForm.test.tsx` and
   `adminInviteUserAuditShape.test.ts` are regex-against-source, which is inherently
   brittle to harmless reformatting (e.g. the `dispatch === 'invite'[\s\S]*?adminInviteUser`
   pattern). This is the repo's accepted pattern and not a defect, but if a future card
   adds RTL infrastructure, migrating these to behavioral assertions would harden them.
   Defer — out of scope for QOL-024.
2. `handleInviteUser` does not check the return value of the `profiles.update` for
   `role`/`display_name` (lines 386-388) — same as `handleCreateUser` (lines 241-243),
   so this is consistent existing behavior, not a regression. A failed profile update
   would leave the invited user at the default role with the invite still sent. Low
   impact (the trigger-created profile already defaults to `role: 'user'`, the common
   case); worth a follow-up only if moderator-invite reliability becomes a concern.
