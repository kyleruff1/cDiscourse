## Summary

MCP-018 un-stubs MCP-009's reserved `mcp` provider slot in the `semantic-referee`
Edge Function with a real operator-hosted MCP-server adapter — the exact analog
of the MCP-017 `anthropic` provider. It ships two new server-only files:
`mcpAdapterCore.ts` (zod-free, Jest-importable pure core — request-body assembly,
envelope extraction, JSON parse, sanitizer, the `McpUnavailableReason` /
`McpProviderResult` boundary-internal types) and `mcpAdapter.ts` (the Deno-only
orchestrator — the only file that reads `SEMANTIC_REFEREE_MCP_URL` /
`SEMANTIC_REFEREE_MCP_TOKEN` via `Deno.env.get()` and `fetch`es the MCP server).
The `mcp` branch in `providerRoutingCore.ts` is wired through a new injected
async `runMcp` dep with a pure `mcpReasonToOutcomeReason` map; `providerRouting.ts`
wires `runMcp: runMcpAdapter` into `DEFAULT_PROVIDER_DEPS`. The implementation
follows the design step-for-step: `mcp` is opt-in (the `?? 'mock'` fallback is
byte-identical), the adapter never throws (all seven failure paths return a typed
`unavailable` result), every MCP-server response is double-validated through
`SemanticRefereePacketSchema` + `scanPacketContent`, the token only travels over
TLS (a non-`https` URL is rejected before any `fetch`), and no type file is
widened. No production-code concerns remain. The only red test suite is the
documented pre-existing `diagnosticInspectPackage.test.ts` flake — confirmed red
on untouched `main`, unrelated to this card.
