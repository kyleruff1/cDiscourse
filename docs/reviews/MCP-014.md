## Summary

MCP-014 ships the referee feedback banner library designed in MCP-008: a pure-TypeScript 103-entry string library plus a deterministic `selectBanner` selection model that turns already-classified move metadata (a `SemanticRefereePacket` from MCP-011 and/or a `LedgerResult` from MCP-013) into at most one short, encouraging, non-verdict banner. The footprint matches the expected file surface exactly — 6 new files under `src/features/refereeBanners/`, an append-only 103-key extension of `gameCopy.ts` `PLAIN_LANGUAGE_COPY`, 7 new test files, and a `docs/core/current-status.md` update. No `supabase/` file, no `.env`, no Edge Function, no MCP-011 / MCP-013 source file touched. All four verification commands pass; the full suite is 6356 tests / 239 suites (+122 tests / +7 suites over the 6234 / 232 baseline, matching the implementer's reported number). Every safety scan is clean (only documented false positives — secret-name strings appear solely inside the forbidden-import test's *absence* assertions and the design doc). The doctrine is respected throughout: every banner string is ban-list-clean, low confidence softens or suppresses, an unmapped code is suppressed not echoed, and the model is provably pure-TS. The work is ready for the operator to push and PR.

### Documented deviations — verified benign and consistently applied
1. `CLASSIFIER_TO_BANNERS` is keyed off the real 23-id `SemanticClassifierId` catalog (`ALL_SEMANTIC_CLASSIFIER_IDS`), not MCP-008 §5's 90 seed ids. Confirmed against `src/features/semanticReferee/semanticRefereeTypes.ts` (23 ids) — `refereeBannerLibrary.test.ts` asserts the key set equals the catalog. Recorded in the design's first section.
2. `FEEDBACK_CODE_TO_BANNER` is keyed off MCP-013's 22 `RefereeFeedbackCode`s (`ALL_REFEREE_FEEDBACK_CODES`). Confirmed against `src/features/refereeLedger/types.ts` (22 codes including `broad_point_set_down` + `synthesis_named`) — the coverage test asserts the key set.
3. `softenedSiblingCode: string | null` replaces MCP-008's `hasSoftenedSibling: boolean` — a strict superset; recorded as a documented deviation.
4. `BannerSelectionInput` drops the redundant `category` field and `routeSuggestion`, and tightens every field to the merged union types — recorded as a documented deviation.
5. Implementer reword (within the design's own ban-list scope): the design's `pacing_*` copy used the word "beat" (verdict sense). "beat" is on the MCP-014 ban-list, so two banners were reworded to "pause" and `bannerCode` `pacing_take_a_short_beat` → `pacing_take_a_short_pause`. Verified consistent across `refereeBannerLibrary.ts`, `classifierBannerMap.ts`, and the `gameCopy.ts` block. The `refereeBannerBanList.test.ts` `BANNED_TOKENS` array includes `beat`, so any banner string containing it would fail the suite — confirming the reword is enforced. The pre-existing MCP-013 `pacing_is_on` `PLAIN_LANGUAGE_COPY` key still reads "a short beat helps" and is correctly left untouched (append-only discipline; that key is not a banner string and is not scanned by the ban-list test).

- [x] No truth/winner/loser language in user-facing strings — `refereeBannerBanList.test.ts` scans **every** `headline`, **every** `helperLine`, **every** `accessibilityLabel`, and **every** `bannerCode` across the full `REFEREE_BANNER_LIBRARY` (the `userFacingStrings()` helper iterates the entire library, not a sample) against a 35-token whole-word ban-list. - [x] Score never blocks posting — `RefereeBanner` has no `block` field; `selectBanner` has no gate; a banner is a string the UI shows after a move exists. The schema-shape test asserts `block`/`winner`/`loser`/`truthValue`/`score`/`authoritative`/`verdict` keys are all absent.
- [x] No service-role in client code — none; `refereeBannerForbiddenImports.test.ts` asserts no `SERVICE_ROLE` string and no Supabase import in any of the 6 files.
- [x] No direct insert into `public.arguments` — none; the model consumes already-classified metadata and writes nothing.
- [x] No AI calls in production app paths — none; the model is pure-TS, fully synchronous (`refereeBannerForbiddenImports.test.ts` asserts no `async`, no `fetch(`, no `Deno`, no `process.env`). It consumes already-classified data; no provider SDK is imported.
- [x] Plain language only — every `bannerCode` resolves through `gameCopy.toPlainLanguage` to its headline; `refereeBannerPlainLanguage.test.ts` proves an unmapped code returns `null` (suppressed, not echoed) and no resolved string leaks an underscore; `refereeBannerBanList.test.ts` asserts no user-facing field matches `looksLikeInternalCode`.
- [x] Epic-specific doctrine:
  - `cdiscourse-doctrine` §3 (popularity is not evidence): there is no heat / engagement / popularity field anywhere in `BannerSelectionInput`; the `uses_popularity_as_evidence` classifier maps only to `source_chain_gap_popularity_not_proof` ("Popularity isn't proof — what's the source?"); the ban-list test enforces that any banner naming a popularity token must also carry a not-proof phrase.
  - `timeline-grammar` (shape carries meaning): every banner carries a `toneGlyph` (`star`/`arrow`/`branch`) — a non-color shape signal; `refereeBannerAccessibility.test.ts` asserts the `tone → toneGlyph` mapping for every entry, so a grayscale render distinguishes tone.
  - `accessibility-targets` (no color-only meaning; screen-reader labels): every entry carries a pre-authored complete-sentence `accessibilityLabel` with the tone stated in words (`Referee note:` / `Referee suggestion:`); `refereeBannerAccessibility.test.ts` asserts it is non-empty, begins with the tone prefix, and is rebuilt from headline + helper (cannot drift).
  - Describe the move, never the person: every banner is about the posted move; `contains_unplayable_insult_only` is in `INTENTIONALLY_SILENT_CLASSIFIERS` and maps to `[]` so a move is never labelled "an insult"; the ban-list test scans for second-person belittling constructions.
  - `point-standing-economy` / `evidence-doctrine`: concession / debt banners frame as repair, never defeat ("Narrow concession noted — the broad point still stands."); `evidence_debt` banners are `nudge` tone ("Good pressure — an evidence debt opened." / "A source closes it."), never a penalty.

## Suggestions (non-blocking)
1. The `'all_suppressed_low_confidence'` branch in `selectBanner` is currently unreachable via real inputs: every library banner with `minConfidence` above `low` carries a `softenedSiblingCode`, so a low-confidence source always swaps rather than drops. This is by design ("when in doubt, no banner" — and `refereeBannerConfidence.test.ts` already documents this in a code comment) and is a correct defensive backstop. No action needed; flagged only so a future library author who adds a confident banner with `softenedSiblingCode: null` knows the suppression path will then activate (and the test comment in `refereeBannerConfidence.test.ts` already explains this).
2. The schema-shape assertion in `refereeBannerBanList.test.ts` samples `REFEREE_BANNER_LIBRARY[0]` only. This is per the design test plan ("a sampled `RefereeBanner` object") and is sufficient because all entries are built from one `RefereeBannerSeed` shape via a single `.map()`; a future hardening pass could iterate the full library for symmetry with the other scans. Non-blocking.
