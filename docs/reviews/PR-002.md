## Summary

PR-002 ships a Profile tag popout: an optional, identity-light way for a
signed-in user to add up to 5 tags from a closed, curated 28-tag vocabulary,
reached via a new "Profile tags" row inside PR-001's `PreferencesPopout` and
persisted device-locally in `AsyncStorage` (versioned blob per user, no
migration, no Supabase write). The implementation matches the design closely —
every named file is present, the two doctrine-load-bearing tests
(`profileTagVocabularySafety`, `profileTagValidationImmutability`) are real and
exhaustive, and the closed-vocabulary / cap / immutability / defensive-parse
constraints are all genuinely enforced. Typecheck, lint, and the full 4471-test
suite pass. There are no doctrine violations and no concerns that block. The
self-visible-only v1 scope is a correct, honest designer call, not a defect.

## Suggestions (non-blocking)

1. `useProfileTags.ts` defines a local `EMPTY_SELECTION` constant that is
   structurally identical to the exported `DEFAULT_PROFILE_TAG_SELECTION` (which
   is also imported but only used as the initial `useState` value). Reusing a
   single source would be marginally cleaner, though the local one exists to
   hand back a fresh non-frozen object on `clearTags`/error — defensible as-is.
2. `profileTagsStorage.ts` `loadProfileTags` returns
   `{ ...DEFAULT_PROFILE_TAG_SELECTION, selectedTagIds: [] }` in two places; a
   plain `{ schemaVersion: 1, selectedTagIds: [] }` (as the model already does)
   would read more directly. Purely cosmetic.
3. A future v2 "shared profile tags" card will add the Supabase column + RLS +
   an other-user profile surface; the `schemaVersion` field already keeps that
   path open, as the design notes. No action now.

3. **5-cap disables chips:** select 5 tags → count reads "5 of 5", the
   `profile-tag-at-limit` note appears, and every unselected chip is visibly
   dimmed and stops responding; deselecting one re-enables the rest.
4. **Optionality:** with zero tags, `profile-tag-empty` and the optional-helper
   copy plainly say tags are optional; no error, no minimum.
5. **Persistence across reload:** select a few tags, close both popouts, reload
   the Expo-web tab, sign back in, reopen → the same tags are still selected.
6. **Honest copy / no free-text:** confirm there is no text input anywhere in
   the tag popout, no raw internal codes (`topic_climate`) shown as user text,
   and no console errors/404s. Reduce motion = On → the popout snaps open
   instead of sliding.
