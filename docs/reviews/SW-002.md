## Summary

SW-002 ships a pure-TS, total, deterministic activity-model deriver (`src/features/strengthWeakness/heatModel.ts`) plus an `index.ts` barrel and 79 doctrine-anchored tests in `__tests__/heatModel.test.ts`. The three coarse 3-valued unions (`HeatLevel`, `MomentumState`, `TrendDirection`) are locked to the design's exact vocabulary; the 8-kind `ActivityChip` set is locked; helper-line strings either read RULE-003's `LIFECYCLE_UX_MAP` / `MANUAL_TAG_UX_MAP` verbatim or use plain-language NEW strings that pass an extended ban-list (verdict + amplification + person-attribution + block-prevent tokens). The `hot` carve-out is documented in all three required places (HeatLevel JSDoc, `_forbiddenActivityTokens` JSDoc, and an inline-doctrine-pinned test) and explicitly excluded from the ban-list with a doctrine citation. The deriver is room-level only — zero side-aware fields, no `score` / `rank` / `winningSide` / `percentile`. The forbidden-imports test refuses `Date.now`, `Math.random`, `fetch`, `supabase`, `anthropic`, `xai`, `console.log`, secret literals, and direct value-imports of LIFE-001 / META-001 / `argumentGameSurfaceModel` (type-only imports enforced). This is the most doctrine-sensitive card in Wave 2; it is also one of the cleanest implementations of the doctrine layer.

## Suggestions (non-blocking)

1. **`AutoMetadataCode` import unused at runtime.** The `AutoMetadataCode` and `ClusterMetadataSummary` type-imports in `heatModel.ts` (lines 60–65) are referenced only via explicit type annotations on local variables inside `scanMetadata`. Strict `verbatimModuleSyntax` would still keep these — fine — but the imports could be inlined into the scan helper for tightness. Cosmetic only.
2. **Segment-level lifecycle / metadata signals are deliberately room-wide.** `buildSegments` reuses `scanLifecycle(lifecycleMap)` + `scanMetadata(metadataLedger)` for every band rather than restricting cluster signals to nodes in that band. The design doc names this explicitly: "we reuse the room-wide signals to keep the deriver simple and deterministic." Documented and acceptable for v1; flag in a follow-up if segment-marker UI (IX-002) needs depth-banded lifecycle/metadata signals.
3. **The `void comparisonWindowMs;` statement inside `buildSegments`** is a satisfy-noUnusedLocals hint that's unnecessary because the parameter is in fact referenced via `recentLo = clockMs - recentWindowMs` (not `comparisonWindowMs`). Minor — could be removed when threshold tuning lands.
4. **`recentMovesQuietMax === hotMin` path restores BOTH defaults**, not just one. This is the design-documented behaviour and the test passes; a future tuning card could relax it to "restore the conflicting field only." Documented.
