## Summary

QOL-023 ships a pure-TS `buildAuthRedirectUrl` helper plus a thin impure
`resolveRuntimeOrigin` boundary, wires `emailRedirectTo` / `redirectTo` into
signup, the new `sendPasswordResetEmail` wrapper, and `adminSendPasswordReset`,
and extends the HOST-001b runtime-env injector (`mjs` / `ps1` / `sh` +
`server.mjs`) with an optional `EXPO_PUBLIC_APP_ORIGIN` slot. The implementation
matches the design faithfully — including the design's own three documented
discrepancies against the card body (no `publicOrigin` slot existed, no
`expo-constants`, `adminSendPasswordReset` already had the param). The helper is
genuinely pure (zero imports), the fail-soft contract is correct (a bad origin
degrades to `emailRedirectTo` omitted, never blocks signup), and the error class
leaks only a reason enum. Verification is fully green: 4159 tests / 141 suites,
typecheck and lint clean, no secrets, no service-role, no console.*, no
migration, no Edge Function change, no new dependency. The two
implementer-flagged items both check out as correct adaptations. Only one
non-blocking nit: a stale sentence in `docs/current-status.md` (see Suggestions).

**Two implementer-flagged items — both verified correct:**

1. *Explicit `.toLowerCase()` deviation.* The design (line 166) assumed the
   WHATWG `URL` parser lowercases protocol+host "for free"; under the jest-expo
   RN `URL` runtime `.host` is not lowercased. `validateAndNormalizeOrigin`
   (buildAuthRedirectUrl.ts:142) applies `.toLowerCase()` to `parsed.protocol`
   and `parsed.host` only — the route/path is never lowercased. This satisfies
   the design's stated output contract (`https://DEV.CDiscourse.com/` →
   `https://dev.cdiscourse.com/...`, design line 177; test
   buildAuthRedirectUrl.test.ts:131-139). It is a faithful, well-commented
   adaptation, not a deeper problem. No verbatim-route regression — the
   query-string test (line 217) confirms route case is preserved.

2. *`hostOneBuildScripts.test.ts` source-scan update.* The old assertion
   regex-matched `JSON.stringify({...EXPO_PUBLIC_SUPABASE_URL...PUBLISHABLE_KEY`.
   `buildRuntimeEnvFileContents` now assembles an `env` object (to include
   `EXPO_PUBLIC_APP_ORIGIN` only when present) then calls `JSON.stringify(env)`.
   The updated test asserts (a) both required keys still appear in the builder
   source, (b) `JSON.stringify(env)` is present, (c) the
   `window.__CDISCOURSE_RUNTIME_ENV__ = Object.freeze(...)` wrapper is intact.
   The injection-safety property (payload JSON-stringified, both required keys
   bound) is preserved — confirmed against inject-runtime-env.mjs:128-142.

## Suggestions (non-blocking)
1. `docs/current-status.md` (added in commit 4192737) contains a stale,
   inaccurate sentence: "...4140 pass, 19 pre-existing xAI / engagement-intel
   suite failures remain unrelated to this card and operator-gated by missing
   `.env.engagement-intelligence`." Verified false this review: when run in the
   main repo path (where `.env.engagement-intelligence` is present), the full
   suite is **4159/4159 passing across 141/141 suites** — the 19 xAI/bot-fixture
   tests pass on both `main` and this branch. The "+54 tests / +4 suites
   (4086 → 4140)" delta in the same paragraph is also off; the real delta vs the
   `main` baseline is +54 tests / +4 suites landing on **4159 / 141**. This is a
   doc-text accuracy nit only — no code or test is affected, and the verdict is
   not gated on it. Recommend the operator (or a trivial follow-up commit)
   correct the sentence to "+54 tests / +4 suites (4105 → 4159 passing; 141
   suites, all green)". Implementer can address or defer.
2. The injector design suggested exit code `9` for an `EXPO_PUBLIC_APP_ORIGIN`
   forbidden-secret shape; the implementer instead folds the new value into the
   existing `looksForbidden(url) || looksForbidden(key) || looksForbidden(appOrigin)`
   block (exit `5`) and reserves `10` for the malformed-origin case. This is
   *better* than the design's "e.g. 9" suggestion — it keeps all forbidden-shape
   refusals on one exit code consistent with the existing URL/key handling. No
   action needed; noted so a future reader does not flag it as a deviation.
