## Summary

MCP-003 is a design-only card (501 lines, one file) specifying the deterministic "referee ledger" — layer 3 of the MCP semantic-referee architecture. It is a strong, doctrine-disciplined design: it covers all six issue scope items, correctly positions itself as a thin layer ABOVE the Constitution that CONSUMES the point-standing economy rather than replacing it, faithfully references the MCP-001 `SemanticRefereePacket` / `scoreHints` contract without redefining it, and carries a thorough doctrine self-check (§18) and test plan (§15). The design's whole §9/§11/§16 promise is to name REAL exported symbols; this one is wrong and a few others are imprecise. These are cheap to fix and do not require a re-design.

## Suggestions (non-blocking)

These are accuracy fixes to the integration map. The design's stated job (§9 "The names below are the **real exported symbols** verified in `src/features/pointStanding/`") makes naming-accuracy load-bearing — so these should be corrected before the implementation card is filed, even though none is a doctrine issue.

1. **`gameCopy.ALL_INTERNAL_CODES` does not exist.** §8.1, §11 ("extend `toPlainLanguage` + `ALL_INTERNAL_CODES`"), §14 ("`toPlainLanguage` + `ALL_INTERNAL_CODES`"), and §16 ("must extend `ALL_INTERNAL_CODES` and the mapping") all name a symbol that is not exported by `src/features/arguments/gameCopy.ts`. The real surface is `PLAIN_LANGUAGE_COPY` (the record at gameCopy.ts:150), `PlainLanguageKey` (the `keyof typeof PLAIN_LANGUAGE_COPY` type at line 255), and `toPlainLanguage` (line 263). The implementation card would extend `PLAIN_LANGUAGE_COPY` and `PlainLanguageKey`. Update the four references so MCP-003-impl does not chase a non-existent export. (`ALL_INTERNAL_CODES` appears in the `test-discipline` skill's illustrative example — that is a generic skill snippet, not this repo's real API.)

2. **`reasonsFromEligibility` is file-private, not importable.** §7.1 says the ledger emits "an `ineligibilityReasons` array (re-using the economy's `reasonsFromEligibility` strings)" and §9.3 says `LedgerResult.ineligibilityReasons` "Re-uses the economy's eligibility-reason strings." Verified: `reasonsFromEligibility` is an un-exported function inside `scoringEngine.ts` (line 57). The ledger cannot import it. It is, however, trivially reachable: `ChallengeGradingResult.ineligibilityReasons` and `RepairGradingResult.ineligibilityReasons` (both public result fields, types.ts:217/228) already carry exactly those strings. Reword §7.1/§9.3 to source the reasons from the grading-result fields, not from the private function.

3. **`AntiAmplificationResult` and `RepairGradingOptions` are not in `types.ts`.** §9.1's types row claims the ledger's input/output vocabulary (including `AmplificationContext`, etc.) is "Imported as the ledger's input/output vocabulary" from `types.ts`. `AntiAmplificationResult` is exported from `antiAmplification.ts` (line 45) and `RepairGradingOptions` from `scoringEngine.ts` (line 164). Minor — the implementation card imports them from the right files via the package index — but the §9.1 table attributes them to `types.ts`. Tighten the "File" column.

4. **`applyAntiAmplification` / `amplificationContextFromAnnotationFields` are not re-exported by `src/features/pointStanding/index.ts`.** `index.ts` currently re-exports `types`, `mixedClassWeights`, `concessionEffects`, `eligibility`, `scoringEngine` — but not `antiAmplification`. §14 asserts everything is "All verified present (`src/features/pointStanding/index.ts`)"; that is true for the symbols but `antiAmplification.ts` is reachable only by deep import, not via the package index. The implementation card should either deep-import `../pointStanding/antiAmplification` or add `export * from './antiAmplification'` to the economy's `index.ts` (a one-line, doctrine-neutral change). Worth a sentence in §9.1 or §16 so the implementer is not surprised.

5. **`scoreHints` integer range and `HINT_STEP` are assumptions.** §6.2 proposes `0..3` and `HINT_STEP = 0.08`; §16 already flags this as an open item against the seed bank. This is handled honestly — no change required, just confirming the reviewer saw it and agrees it is correctly surfaced as a risk, not buried.
