**(b) `getPromotedEntry` returns `null` for several replay frames because the
hard engine gate removes `reply`.** Accepted ‚Äî documented design behavior,
not a production bug. Verified against the Constitution transition table
(`src/domain/constitution/constitution.v1.ts`): a `claim` parent's allowed
children are `[evidence, rebuttal, clarification_request, concession]` and a
`rebuttal` parent's are `[counter_rebuttal, evidence, clarification_request,
concession]` ‚Äî `claim` is a child of **neither**. The ¬ß3.4 soft-gate table
(`STAGE_PROMOTED_ENTRY`) promotes `reply` for `open` / `answered` /
`rebutted` / `clarified` / `sourced`, and `reply` carries `argumentType:
'claim'`. So `applyEngineGate` correctly removes `reply` off a `claim` or
`rebuttal` parent, and `applyStageGate` (lines 585‚Äì587: "the promoted entry
must have survived the hard gates to be promoted") returns `promoted: null`.
This is design ¬ß3.3 working exactly as written ‚Äî **engine + role *filter*;
stage only *suggests*; a stage can never resurrect an engine-filtered move.**
The affected frames honestly assert the surviving `challenge` box-opening
entry via `mustOfferEntry` (verified: `challenge` carries `rebuttal` and is
kept when `rebuttal` OR `counter_rebuttal` is a valid child ‚Äî true for both
`claim` and `rebuttal` parents) and document the reasoning inline in each
frame's `note`. The one frame that asserts a real promotion ‚Äî Scenario 2 F6,
`source_requested` ‚Üí `add_evidence` off a `rebuttal` parent, where `evidence`
*is* a valid child ‚Äî correctly asserts `isPromoted: true`. The test reproduces
the engine-gate doctrine rather than papering over it. Acceptable.
