## Summary

IX-002 adds a timeline mini-map overview rail to `ArgumentTimelineMap`: a tiny,
fixed-width strip that renders a long conversation at a glance with a
draggable/tappable viewport-window indicator plus Root / Latest / Hot zone /
per-side-branch jump chips. The implementation is a clean two-layer split — a
pure-TS view-model (`timelineMiniMapModel.ts`) plus a thin RN component
(`TimelineMiniMap.tsx`) — wired into `ArgumentTimelineMap.tsx` as purely
additive internal chrome (no new prop crosses the component boundary, the room
shell needs zero changes). The mini-map is a genuine *projection*: it re-labels
the existing `temperatureBand` and reads `branchId` / `branchRootMessageId` /
`lane` verbatim from the already-built `ArgumentTimelineMapModel`; the three
upstream model files are untouched. Doctrine is respected throughout — heat is an
activity signal with no verdict copy, the hot zone is chosen by run length not by
"hottest", no standing band leaks into any mini-map field, and the model never
even reads `standingBand`. All four verification commands pass; the test count
rose by exactly the implementer-reported delta. The two flagged implementer notes
are benign and well-documented. Ready for the operator to push.

## Known implementer notes — both confirmed benign

1. **`.tsx` test discipline (helper + source-scan, not `render()`)** — the repo
   genuinely has no RN render harness; `BranchCollapseStub.test.tsx` and
   `ReceiptChip.test.tsx` follow the identical pattern. Every visual decision in
   `TimelineMiniMap.tsx` is extracted into an exported pure helper that the test
   exercises directly (null-render guard, marker geometry, heat ring, viewport
   rect, 44px, grayscale, reduce-motion); the un-executable wiring is
   source-scanned. This is real coverage, not hollow.
2. **Cluster keying — mainline by `lane===0`, side branches by `branchId`** —
   `buildArgumentTimelineMap` Pass 2 fragments `branchId` per-node along a long
   single chain, so keying the mainline by `branchId` would split a no-branch
   chain into many clusters and break the design's own edge case "no branches →
   only the mainline cluster". The implementer (commit 01209d1) keys the
   mainline on the structural `lane===0` signal and side branches on the stable
   `branchId`. The model test `'a no-branch chain produces a single mainline
   cluster'` proves the edge case is satisfied; `lane` / `branchId` /
   `branchRootMessageId` are still read verbatim — no parallel branch logic is
   invented, the upstream model is untouched. Sound, documented in-code, not a
   silent divergence.

## Suggestions (non-blocking)

1. `buildBranchChipLabel` re-maps `cluster.laneLabel` (a sentence fragment) to a
   compact noun (`Mainline` / `Side branch`) with a local `lane === 0` check
   rather than reusing a shared label source. It is correct and the no-verdict
   test covers it, but if a future card adds a third lane category (e.g.
   "Chime-in", "Detached") this compact-noun map will need a parallel update.
   Consider, in a later card, extending `deriveBranchLabel` (or a sibling) to
   emit the compact noun form so there is one source of branch wording.
2. The `panResponder` `useMemo` depends only on `onScrubViewport`; `railWidth`
   is read via `railWidthRef.current` inside the handler (correct — avoids
   stale closure). This is fine, but a one-line comment noting the deliberate
   ref-read would help the next reader.

Neither is a gate. The implementer may defer both.
