## Summary

REF-004 closes REF-001's ratified disagreement loop (Act changes the issue ·
Inspect explains the issue · Go moves around the issue map) by wiring REF-003's
Referee Card into the three already-shipped one-box popouts. It is a pure `src`
UI change: 7 modified + 1 new src file, 4 new test suites, 1 status-doc line —
exactly the 13 files the design names. The Act half is a behavior-preserving
extraction (`enterBoxForActEntry`) that makes the board-Act path and the card's
zone-3 buttons share **one** composer-open bridge; the Inspect half adds a
read-only sibling overlay that is the single home for the issue's raw codes; the
Go half adds one pure 8-state→lens map that dims-never-hides by construction.
Every verification gate passes, doctrine is clean, and the two posture-critical
claims (gate-neutral; no write path) are **both provable from the diff**. No
blockers. Two cross-device QA items (focus-return on Inspect close; affordance-row
crowding on phone) remain as the design's own flagged follow-ups, not regressions.

## Suggestions (non-blocking)
1. **Inspect focus-return QA not captured in a test.** The design (§A11y) asked the
   implementer to verify focus moves into the Inspect popout on open and returns to
   the "View details" affordance on close (web), and to document it in the a11y test
   or a cross-device QA note. REF-004 correctly adds no new focus logic (it relies on
   the shipped Popout chassis trap), so this is not a defect — but neither the test
   suite nor a QA note records the on-close focus-return observation. Operator may
   want a quick web smoke or a follow-up QA note before/at REF-006.
2. **Affordance-row crowding on phone** (design Risk). The row wraps
   (`flexWrap: 'wrap'`) and "Focus on board" hides when `targetNodeId == null`, which
   mitigates it, but the 390×844 + 1024×1366 cross-device check the design flagged is
   not evidenced in-repo. Fold into the REF-006 usability smoke.
3. **`handleRefereeMove` behavior change is a genuine fix, worth noting.** Dropping the
   v1 `null`-quickAction early-return means a zone-3 entry whose `actEntryToQuickAction`
   is `null` (e.g. `offer_concession`) now opens the composer with no preset instead of
   being a silent no-op. This is intentional and matches the board-Act path; flagging it
   only so the operator is aware the card path's no-op edge is now active behavior.
