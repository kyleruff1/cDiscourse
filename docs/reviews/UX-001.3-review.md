## Findings

### Advisory finding 1 ŌĆö Swipe-down and tap-outside-to-dismiss deferred

- **Severity:** advisory (non-blocking)
- **Location:** `src/features/arguments/ArgumentComposerDock.tsx`, Phase 3 framing in `docs/core/current-status.md` lines 591-604
- **Description:** The UX-001.3 design (┬¦5 + ┬¦8) called for adding a
  `PanResponder` to the drag handle for swipe-down-to-collapse on phone
  and converting the scrim from inert to tap-to-collapse. The
  implementer deferred both, with the documented rationale that the
  existing inert-scrim design protects against accidental draft loss
  (the dock today preserves draft on Escape, hardware-back, and Cancel
  press; turning the scrim into a tap target risks an unintended draft
  collapse from a stray Timeline tap). The implementer chose the
  Esc-to-collapse path as the new dismiss seam, which is also documented
  in the Phase 3 framing as the operator-accepted behavior shift.
- **Why this is advisory not blocker:** The brief leaves "Haptics are
  NOT a hard requirement; if the platform does not support them cleanly,
  the composer functions identically without them" as the precedent for
  pragmatic deferral. The design ┬¦ 8 risk table also pre-acknowledged
  the conflict ("Tap-outside-to-collapse might conflict with the scrim's
  existing inert-touch-shield purpose"). The deferral preserves the
  draft-loss guarantee that is load-bearing for the existing dock; the
  Esc-to-collapse path satisfies the brief's "Esc collapses the
  composer or closes the bottom sheet" contract.
- **Recommended action:** None for UX-001.3 ship-or-don't-ship
  decision. A follow-up card may revisit if the operator decides the
  swipe-down + scrim-tap behaviors are higher priority than the existing
  draft-loss guarantee. The Phase 3 framing already documents the
  deferral for UX-001.4's designer to consume.
