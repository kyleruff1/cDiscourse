### The two operator-mandated checks (explicit)
1. **Genuinely ATOMIC? — YES.** The body is one `LANGUAGE plpgsql` block with a single top-level `BEGIN…END $$` (migration L160–L271). Comment-stripped, it contains zero `COMMIT` / `ROLLBACK` / `SAVEPOINT` / `autonomous` / `dblink` / `pg_background` (read directly + asserted by FIN-13/FIN-14). There is **no `EXCEPTION WHEN` block** (FIN-15) — this is the *correct* posture: any error (e.g. the `confidence` CHECK or an FK violation) propagates and rolls back the entire function-call transaction, so the result-INSERT (L216) and the run-UPDATE (L237) **both commit or both roll back**. A subtransaction (`BEGIN…EXCEPTION`) would have broken that all-or-nothing guarantee for the caught region; its absence is deliberate and right. The success path is INSERT(L216)+UPDATE(L237) in the same body; the terminal-failure path is a single UPDATE(L255); the branches are mutually exclusive, so any call is `{INSERT+UPDATE}` or `{UPDATE}` — atomic either way.
2. **Can a stale / wrong / reclaimed owner finalize? — NO.** The ownership guard (L180–L186) runs FIRST, before any write (FIN-18 proves SELECT-before-INSERT and SELECT-before-both-UPDATEs by string offset): `SELECT … WHERE r.id = p_run_id AND r.lease_owner = p_owner AND r.state = 'leased' FOR UPDATE` → `IF NOT FOUND THEN RETURN false` (L188–L191), a hard no-op (no result rows, no UPDATE). Wrong owner (`lease_owner ≠ p_owner`), stale/reclaimed (`state` moved off `'leased'` to `retry_scheduled`/`dead_letter`, or `lease_owner` cleared), and already-terminal rows all fail the predicate. **The reclaim race is safe:** `FOR UPDATE` holds the row lock for the rest of the transaction, so a concurrent `reclaim_stale_leases` UPDATE (Card-1 `…21` L411, `WHERE r.state='leased' AND r.lease_expires_at < now()`) on the same row blocks until the finalize commits and then re-evaluates to find the row no longer `'leased'`; if reclaim wins the lock first, the finalize's guarded SELECT sees `state != 'leased'` and returns false. No interleaving yields a write by a non-owner or on a non-leased row. There is no code path that writes results or updates the run for a non-owned / non-leased row. `p_terminal_state` validation (L166–L170) precedes even the guard, so an invalid state raises before any lock — also correct.

### Other required verifications
- **`ON CONFLICT (run_id, raw_key) DO NOTHING` is duplicate-safe — YES, via column inference.** L230 uses explicit column inference `(run_id, raw_key)` against the named non-partial constraint `amor_unique_run_rawkey UNIQUE (run_id, raw_key)` (`…18` L117), NOT `ON CONFLICT ON CONSTRAINT` (FIN-26 asserts both the presence of the inference form and the absence of the constraint form in executable SQL). A retried finalize, or a pre-existing `(run_id, raw_key)` row, inserts no duplicate and raises no unique violation — so the finalize never loops toward dead_letter on a genuinely-succeeded cell (the exact failure window the card exists to close). The verify SQL Case 2b proves DO NOTHING does not overwrite the pre-existing row.
- **Result mapping mirrors `persistResults` exactly — YES.** The INSERT column list `(run_id, debate_id, argument_id, schema_version, raw_key, family, confidence, evidence_span)` (migration L217-L218) is byte-identical, in order, to `persistResults`'s `insertPayload` keys (persistenceWriter.ts L141-L148), cross-checked by FIN-33. `debate_id`/`argument_id`/`schema_version` are sourced from the locked run row (L181, L220-L223) — the run's own identity, never passed by the drainer; `raw_key`/`family`/`confidence`/`evidence_span` from each `p_observations` element. `evidence_span` is passed straight through with no `COALESCE` (FIN-34), identical to `persistResults` passing `r.evidenceSpan` through, so NULL stays NULL (verify SQL Case 1 proves it). `COALESCE(p_observations, '[]'::jsonb)` (L228) yields zero rows for null/empty input → a succeeded run with zero positive observations writes no result rows, which is valid.
- **Success vs terminal-failure branches — correct.** Success (`p_terminal_state='succeeded'`, L194): INSERT result rows then UPDATE `state='succeeded', status=p_status, completed_at=now(), lease_owner=NULL, lease_expires_at=NULL` (L237-L243), and the branch writes none of the failure fields (FIN-28). Terminal failure (L255): no result rows (FIN-31 asserts exactly one INSERT in the whole body, in the success branch), UPDATE `state=p_terminal_state, status=p_status, completed_at=now(), failure_reason, failure_sub_reason`, `dead_letter_reason` CASE-gated to fire only when `p_terminal_state='dead_letter'` (L261-L265, FIN-30), lease cleared. Invalid `p_terminal_state` raises (L166-L170). `retry_scheduled` is never written here (FIN-21) — correct, it stays the drainer's run-row-only UPDATE in Card 2.

## Suggestions (non-blocking — operator/Card-2 may address or defer)
1. **Verify Case 4 simulates the reclaimed state rather than exercising the live `FOR UPDATE` vs `reclaim_stale_leases` race.** Case 4 pre-creates a row already in `retry_scheduled` (the post-reclaim state) and asserts the belated finalize is a no-op — which correctly proves the *state-predicate* arm of the guard. It does not run two concurrent sessions to prove the `FOR UPDATE` lock actually serializes a real finalize-vs-reclaim interleaving. The `FOR UPDATE` correctness is sound by reading (and a single-transaction script structurally cannot open a second session), but if the operator wants belt-and-suspenders, the Card-1 verify script (`scripts/arch-001-card1-sql/...`) already documents a two-session concurrency procedure in its Section F; a parallel two-psql-session check (session A holds the finalize mid-transaction via a manual `BEGIN; SELECT finalize_classifier_job(...)` paused, session B runs `reclaim_stale_leases()`) could be added to the post-merge runbook. Strictly optional — does not gate this card.
2. **Card 2 (the drainer) is the consumer that must respect the boolean contract.** This function returns `false` on a no-op; the drainer must treat `false` as "I lost the lease — do not retry-as-if-succeeded, do not double-count." That belongs to Card 2, but flagging it here so the contract is not lost between cards. (The function and its tests are complete and correct as the substrate; this is a forward-looking note, not a defect.)
