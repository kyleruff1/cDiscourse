## Summary

QOL-036 ships exactly what the design specifies: an additive `payment?: PaymentEvidenceMetadata`
sub-object on the EV-001 `EvidenceArtifact`, a seventh `EvidenceArtifactKind` (`payment_screenshot`),
the `EvidenceConfidence` type pinned to `user_asserted`, the `EvidenceAmount` / `PaymentParty` /
`ClaimedApplicability` sub-shapes, and the redaction helpers (`detectRawAccountData`,
`findRawAccountDataFields`, `redactPaymentParty`) plus display helpers (`getPaymentEvidenceLabel`,
`summarizePaymentEvidence`). The change is genuinely additive — every existing `EvidenceArtifact`
field keeps its name/type, every existing consumer (EV-002, EV-003, EV-005, `ReceiptChip`, the
timeline contract) compiles and the full 277-real-suite run stays green. The redaction contract is
load-bearing and correctly placed: `prepareStoredPayment` runs `findRawAccountDataFields` as a hard
gate and *strips* the entire `payment` sub-object (downgrading the kind to `screenshot_redacted`)
when any field carries a card / account / routing / IBAN digit shape — raw account data is rejected
at the model boundary, never masked-and-stored. A payment object is inert: no truth value, no
`PointStandingDelta`, no source-chain auto-promotion, `confidence` pinned regardless of input.
Doctrine is clean across all scans. One concern remains and it is non-blocking: a cosmetic
inconsistency between the detection regex and the masking regex (see Suggestions §1) — it cannot
leak account data because the strict detector, not the masker, is the gate.

## Suggestions (non-blocking — implementer may defer or address)

1. **Cosmetic regex inconsistency between the detector and the masker.**
   `detectRawAccountData` (`evidenceModel.ts:~1330`) collapses *one-or-more* whitespace/hyphens
   between digits (`/(\d)[\s-]+(?=\d)/g`), so it correctly flags a double-spaced card number like
   `"4111  1111  1111  1111"`. But `maskRawAccountRuns` (`evidenceModel.ts:~1400`) matches only a
   *single optional* separator (`/\d(?:[\s-]?\d){11,}/g`), so it does **not** mask that same
   double-spaced run. This is **not a security hole**: in the adapter path `prepareStoredPayment`
   runs `findRawAccountDataFields` (which uses the strict detector) *first* and strips the whole
   `payment` object on any hit, so a double-spaced card number can never reach storage; and
   `summarizePaymentEvidence`'s defence-in-depth re-redaction only ever sees already-gated stored
   objects. The strict detector is the gate; the masker is cosmetic. Still, for consistency and to
   make `redactPaymentParty`'s box-side output match what the gate considers dirty, consider
   widening `maskRawAccountRuns`'s inter-digit class to `[\s-]+`. Low priority.

2. **`detectRawAccountData` IBAN check can be evaded by intra-token spacing.**
   The IBAN branch strips *all* whitespace before testing (`text.replace(/\s+/g, '')`), so
   `"GB29 NWBK 6016 1331 9268 19"` is caught. Good. But the broader point: the heuristic is, by
   design (§16 open question 4), conservative and accepts false negatives at the margins. The
   adapter's strip-on-detect degradation and the `user_asserted` framing mean a missed exotic
   account format degrades a label rather than asserting a falsehood — acceptable per the design's
   stated risk posture. No change required; noting it so a future reviewer does not mistake the
   heuristic for exhaustive PII detection.

3. **Function-name drift between the task brief and the shipped code.**
   The review brief references `assertNoRawAccountData`; the design §7.2 and the implementation
   both name the throw-free guard `findRawAccountDataFields`. The shipped name is the correct one
   (it matches the design and is throw-free, as a render-path guard must be). No action — flagged
   only so the operator is not surprised the brief's name is absent.
