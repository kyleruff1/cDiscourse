### Chassis integrity (explicit judgement point 1)
Verified `boxModel.ts`, `actPopoutModel.ts`, `Popout.tsx`, `PopoutEntry.tsx`,
`PopoutGroup.tsx` are **byte-untouched** — none appear in `git diff main..HEAD`.
The OneBox.tsx +129/-86 diff is genuine component-extraction wiring: it deletes
the inline `buildActPopout` call + inline `Popout`/`PopoutGroup` JSX and the
`popoutGroups` mapping, and replaces them with a single `<ActPopout>` element.
The box state machine is untouched — `switchBoxType`, `setBoxState`,
`quickActionPresets` seeding, `typePatch`/`mergedInitialPatch` all preserved.
The only behavioral change is that `direct`/`role_change` entries now dispatch
to dedicated (no-op-in-this-host) callbacks instead of a shared close handler;
that is a strict refinement of the prior "they just close the popout" comment,
not a state-machine change. Not a finding.

### §4 catalogue gap (explicit judgement point 2)
The implementer note documents that the shipped `buildActPopout` does not
enumerate `Mark moved-on` / `Mark ignored` / `Share` / `Acknowledge / Touché`
or the four-entry `Govern` group from design §4, and that §3.3's
"role-blocked shown disabled" is satisfied via an additive host `disabledEntries`
prop rather than a model change.

Judgement: **this does not block the card.** The card instruction is explicit —
"consume `buildActPopout`, do not re-derive it" and "do not modify the chassis."
`actPopoutModel.ts` is a QOL-030-owned chassis file. Surfacing the missing §4
entries necessarily means editing that file, which QOL-031 is forbidden from
doing. Design §1 itself places "`buildActPopout` (QOL-030)" out of scope, and
§9 is explicit that QOL-031 asserts *rendering*, not the model. Rendering
exactly the model's `ActEntryId` set is therefore the correct, in-scope
behavior — the alternative would be an unauthorized chassis rewrite. The
`disabledEntries` resolution is likewise additive (a new optional prop, no
chassis/model edit) and faithfully delivers the §3.3/§7/§8 "disabled with a
reason, never silently omitted" contract. The implementer correctly logged the
catalogue delta as a **QOL-030 follow-up**, not a QOL-031 defect. The design
note is accurate and the design doc edit is note-only (no spec mutation).

## Suggestions (non-blocking)
1. The §6 narrative-frame test fixtures substitute `clarification_request` /
   `claim` parents to make the storyboard frames reproducible against the
   *shipped* engine gate (e.g. S1 F2/F3 use `clarification_request` so `reply`
   survives). This is honest and well-commented, but several frames assert
   `promoted: null` where design §6 names a specific top entry (S1 F6, S2 F3,
   S2 F5). The tests compensate with separate presence assertions, which is
   fine — but when the QOL-030 follow-up lands the missing §4 catalogue entries
   (`Govern` group, evidence-object promotion), revisit these frames so they
   assert the design's named top entry directly rather than `null`.
2. `handleDirectAction` / `handleRoleChange` in OneBox.tsx are intentional
   no-ops (the composer host owns no seat/moderation surface). This is correct
   and documented, but a future room-shell host wiring these to real behavior
   should be tracked — consider a one-line pointer in the QOL-030 follow-up
   note so the dispatch surface is not forgotten.
