## Suggestions (non-blocking)

These are for the future MCP-010-impl card or an optional design tidy; none block this design-only card.

1. **Line-count drift (cosmetic).** §5.1 says "this design document (~ 520 lines)" and §4.2's file header comment says the same `~ 520 lines`; the file actually landed at 462 lines. Harmless, but the implementation card (or a one-line design tweak) could correct the estimate so it does not read as a stale target.
2. **Own-bubble re-lane permission is a genuine open question.** §11 and §16 decision 3 propose the move author may re-lane their own move, which is *more permissive* than META-001's own-bubble manual-tag rule and than the Stage 6.4 own-bubble action set (CLAUDE.md: own bubble → `Qualifiers · Request deletion` only — no disagree, no flag, no score). The design already flags this and documents the safe fallback (`answers_parent` assertion only on own bubble). The implementation card must resolve it against GAME-004 *before* coding the choice sheet — recommend the implementer treat the fallback as the default and only widen if GAME-004 explicitly permits self-re-laning, to avoid an own-bubble action-set inconsistency.
3. **`MetadataEvent.code` union widening.** §4.1 lets `code` carry `mainline` / `branch` / `tangent` / `answers_parent`. META-001's `MetadataEvent.code` is typed as a plain `string`, so this is type-compatible — but META-001's ban-list / doctrine-anchor tests (`metadataEvents.test.ts`, `metadataDoctrineAnchors.test.ts`) may assert the *set* of codes seen. §13 risk 2 already calls this out; the implementation card should run those two suites in the same change. No design defect — just confirming the risk note is accurate and sufficient.
4. **`SemanticOverrideLane` vs `SemanticRouteSuggestion` mapping.** The trigger model's `suggestedLane: SemanticOverrideLane` (3 values) is derived from MCP-001's `SemanticRouteSuggestion` (7 values incl. `outer_realm`, `synthesis_lane`, `cards_detail`). The design notes the BR-003/BR-004 vocabulary risk (§13) but does not give the implementation card a mapping rule for the four `SemanticRouteSuggestion` values that have no `SemanticOverrideLane` equivalent. Recommend the implementation card specify that collapse explicitly (e.g. `synthesis_lane` / `outer_realm` → `branch` or `tangent`) so it is a deliberate decision, not an implicit default. Non-blocking — the design's contract is sound; this is an implementation-detail to nail down.
