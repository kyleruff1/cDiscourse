## Summary

IX-003 makes the horizontal argument timeline (`ArgumentTimelineMap.tsx`) keyboard-operable on
web and completes its per-node screen-reader contract. A new pure-TS model
(`keyboardNavigationModel.ts`) provides `resolveTimelineNavEffect` (the traversal resolver) and
`buildNodeAccessibilityLabel` (the single label source of truth). The timeline gains a web-only
`onKeyDown` handler on a single focusable group, Prev/Next chips gain a disabled a11y state at the
ends, and the node label now exposes strength + branch. The implementation matches the design
faithfully — including the explicitly-offered deviation of inlining the chronological prev/next
index logic to keep the dependency on `argumentGameSurfaceModel` strictly type-only. Doctrine is
clean: strength is exposed as plain-language gameplay standing, never a verdict, and a ban-list
test locks it. Verification is fully green and no concerns block the merge. One non-blocking
observation about a dropped `actorLabel` fragment is recorded below.

### Acceptance-criteria deep dive

1. **Traversal correctness.** `resolveTimelineNavEffect` is a correct pure function. Arrow L/R
   walk chronological prev/next with no wrap at the ends (`prevNodeId`/`nextNodeId` return `null`
   at `idx <= 0` / `idx >= length-1`). Home/End jump to `rootMessageId`/`latestMessageId` and
   return `none` when already there. No active node → Left/Home go to root, Right/End go to
   latest. Empty map → `none` for every key (no throw, guarded). Single node → arrows + Home + End
   all `none`, Enter → `open_detail`. Detached nodes stay in `map.nodes` so arrows walk them
   chronologically. The keyboard handler routes through the SAME `onActivate` callback tap uses —
   no parallel selection mechanism.
2. **Web capture, no new dependency.** `onKeyDown` + `tabIndex` are spread via
   `{...(Platform.OS === 'web' ? { ... } : {})}` objects — no `as` cast. The handler early-returns
   on `Platform.OS !== 'web'`. `package.json` is unchanged; the wiring test scans the import block
   and asserts no import outside `react` / `react-native` / relative paths. Native path untouched.
3. **Screen-reader contract.** `buildNodeAccessibilityLabel` exposes type, side, ordinal,
   STRENGTH (`STANDING_BAND_SOFT_LABEL[band]`) and BRANCH (`deriveBranchLabel`: mainline / side /
   detached) plus active / latest / junction / detached state — all plain language. The strength
   fragment is gameplay standing, never a verdict; `keyboardNavigationModel.test.ts` includes a
   per-band ban-list assertion over all 9 bands. Node Pressables keep
   `accessibilityRole="button"` + `accessibilityState={{ selected }}`; Prev/Next chips gain
   `accessibilityState={{ disabled }}` + a dimmed `controlChipDisabled` style at the ends.
4. **One label source of truth.** Both `argumentGameSurfaceModel.ts` Pass 4c and `NodeDot` call
   the same `buildNodeAccessibilityLabel` with the same field set — confirmed. The old ad-hoc
   Pass-4 string builder is deleted. No drift possible.
5. **Deviation — inlined prev/next.** Verified against the real `timelineMapPrevId`/
   `timelineMapNextId` (`argumentGameSurfaceModel.ts` 1556–1570): the inlined `prevNodeId`/
   `nextNodeId` use the identical `findIndex` + `idx <= 0` / `idx >= length-1` boundary checks and
   the identical no-wrap semantics for the non-null `currentId` case. The inlined helpers are only
   ever called with a non-null id (the resolver handles `activeMessageId === null` separately), so
   the null-branch difference in the real helpers is never reached and is harmless. The import is
   genuinely `import type { ... }` — `ix003KeyboardA11yWiring.test.ts` asserts there is a `import
   type` from `./argumentGameSurfaceModel` and no plain value import from it.
6. **Up/Down out of scope.** `TIMELINE_NAV_KEYS` binds only `ArrowLeft`, `ArrowRight`, `Home`,
   `End`, `Enter`, ` `, `Spacebar`, `Escape`. `ArrowUp`/`ArrowDown` resolve to `none` and are
   covered by the unhandled-key test. Correctly deferred per the design (page-scroll collision).
7. **One existing test updated.** Only `argumentTimelineMapRootOnboarding.test.ts` changed — the
   `Message 1 of 3` assertion became `position 1 of 3` + an added `opening claim` check, with a
   comment explaining the design-anticipated label-format change. Verified the other
   `Message N of M` references (`argumentGameSurface.test.ts:194` segment label;
   `timelineNodePopoverModel.test.ts` popover `headerLine`) target different label sources and are
   unaffected — the full suite confirms they still pass.

## Suggestions (non-blocking)

1. The old Pass-4 model label included `node.actorLabel` ("Participant" / "Observer" / etc.); the
   new shared `buildNodeAccessibilityLabel` does not carry actor in its field set. This matches the
   design's stated label-order spec (which omits actor), no test depended on it, and the screen
   reader still gets side + type + strength + branch + state — so this is intentional and clean.
   Recorded only so a future card knows the actor fragment was deliberately dropped from the node
   label, not lost by accident. The implementer/operator can ignore this or note it in
   `current-status.md`; no action required for IX-003.
2. The trailing VG-004 fragment "active move" remains redundant with the new "currently active"
   fragment (the design calls this out and chose not to touch VG-004's contract). Fine as-is; a
   future VG-004-adjacent card could de-duplicate.

### Browser-check focus (Expo web — live operator verification)
1. Focus the timeline group (one Tab stop for the whole node set, not one per node), then press
   **Arrow Right / Arrow Left** — the active-node ring moves one node later / earlier and the
   timeline auto-scrolls to keep it in view.
2. **Home** jumps the active node to the root (opening claim); **End** jumps it to the latest move.
3. Arrow Right repeatedly stops at the latest node and Arrow Left repeatedly stops at the root —
   no wrap-around; and arrows do not scroll the whole web page while the timeline is focused.
4. The focused timeline group shows a visible focus outline (browser default ring is acceptable).
5. With a screen reader (NVDA on Windows / VoiceOver on Safari), a focused node announces, in
   plain English, its type, side, ordinal ("position 4 of 12"), strength ("Strongly supported" /
   "Needs work" — never "correct"/"true"), branch ("on the main line" / "on a side branch"), and
   "currently active" / "selected" for the active node.
6. Prev / Next control chips appear dimmed and are announced as "disabled" when the active node is
   at the first / last position; **Enter/Space** opens the per-node popover (Space does not
   page-down) and **Escape** closes an open popover.
