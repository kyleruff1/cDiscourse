## Doctrine & security check
- **§1 (no verdict).** The two new strings ("Add just one email…", "A private argument needs one invite…") are person-neutral and verdict-free; the ban-list test scans them plus `BANNED_INVITE_FRAMING` tokens. Compliant.
- **§5 (purity).** Pure TS, no Supabase/React/network/async/clock/random, frozen output — mirrors `engine.ts:1-5`. Compliant for the module's own code; the one open question is the transitive import graph (see Actionable #1).
- **§8 / RLS / migration-append-only.** No migration, no RLS change, no Edge, no client insert added by this card. No applied migration edited. Compliant.
- **§9 (plain language).** Reason codes never echoed; reuses `plainLanguageForInviteError` and its generic fallback for unknown codes. Compliant.
- **Account enumeration.** Uniform result shape regardless of invitee existence; never queries user existence. Strong — preserves the uniform-response guarantee.
- **Raw token / secrets.** No token, link, or secret handled; masked email is the only echo form. Compliant.
- **Guard-change governance.** The 6→5 flip is not a silent self-approval: it is recorded operator decision 2 (roadmap §4.1), self-gated by a parity test (suite is red until `PUBLIC_ROOM_SEAT_CAP === 5`), and GATE-A-surfaced in the ADR. The flip and its forcing test land in the same PR, so the flip cannot be forgotten — a genuine strength.
- **GATE-C posture.** Correctly N/A (no migration/Edge/Deno/MCP/hosted-config/live send). The design correctly notes the flip is a *client-bundle* behavior change (chime-in capacity 4→3) shipping via the normal bundle, with the client-plane verify lane as belt-and-braces. Right posture.

## Completeness & edge cases
- **Matrix + derivation coverage is complete:** all five contract rows, the open-slot identity (`capacity − 1 creator − reserved`: public/0→4, public/1→3, private/1→0), reserved===count invariant, capacity {2,5}, zeroed seat fields on reject (keeps the literal unions honest). The split-before-validate ordering (so a 2-address paste yields `too_many_invites`, not `invalid_email`) is a thoughtful, correct touch.
- **Null-vs-empty `directInviteEmail`.** The interface is `string | null`, but algorithm step 2 says only "Trim directInviteEmail." The canonical "no email" input is plausibly `null`, and `null.trim()` throws. The reference implementation must null-guard (`(email ?? '').trim()`), and the test plan should treat `null`, `''`, and `'   '` as three distinct "no email" inputs. This is a concrete GATE-B trap the prose glosses (Actionable #2).
- **Seat→side semantics seam.** The arithmetic "creator occupies 1 of 2 private seats" relies on a mapping the pure matrix does not define: `createDebate` auto-joins the creator as `side='moderator'` (debatesApi.ts:148), the roadmap counts `moderator` as an active participant, yet a "private 1v1" is `initiator`/`primary_opponent` (roomContractModel.ts:29). Whether the private creator counts as a debater seat or a moderator seat is unresolved and belongs to ADR-001 / the enforcement card — the pure validator is internally consistent regardless, but the design should flag the seam so capacity counting downstream is unambiguous (Actionable #4).
- **Literal-union typing detail.** `capacity: ArgumentRoomCapacity (2|5)` only compiles because the imported caps retain literal types (`const … = 5/2`). If anyone ever annotates `PUBLIC_ROOM_SEAT_CAP: number`, the assignment breaks. R4 covers the coupling; worth a one-line implementer note (nit).
- **Idempotency:** pure function, no writes — N/A; determinism/non-mutation/frozen are covered.

## Test plan check
Strong and near-exhaustive: every matrix row, capacity 2/5 parity, fitsPublic/Private boundaries (4+1 ok / 4+2 fail; 1+1 ok / 2+1 fail), determinism + non-mutation + frozen, ban-list over the two new strings *and* `plainLanguageForCreationReason` across every reason + unknown, self-invite both visibilities + case/whitespace-insensitive + skip-when-no-creatorEmail, multi-email variants (`,`/`;`/space/trailing-sep/only-separators), invalid email, normalize/mask composition, visibility coercion, and the reuse-proof that `self_invite`/`invalid_email` copy equals the shipped invite strings. Gaps:
1. **No import-purity / Edge-safety assertion.** The headline claim is "same module runs in an Edge Function." That depends on `inviteCopy` + `inviteModel` carrying no React/Supabase imports. Add a test (or lint guard) asserting the transitive import graph is RN/Supabase-free, or at minimum confirm both reused modules are Edge-safe (Actionable #1).
2. **`directInviteEmail: null` not explicitly enumerated** as a "no email" input (Actionable #2).
3. **No JSON round-trip assertion** despite the JSON-serializable claim — a one-line `JSON.parse(JSON.stringify(result))` deep-equal would close it (nit).

## Actionable comments
1. **[should]** (§Chosen approach / Tests) — Back the "client + server shared validator … runs identically in an Edge Function" claim with an import-purity guarantee: add a test asserting no `react`/`react-native`/`@supabase` in the transitive graph of `argumentRoomCreationMatrix.ts`, **or** explicitly confirm `inviteCopy.ts` and `inviteModel.ts` are RN/Supabase-free so the Edge import in the sibling card cannot break. 2. **[should]** (§Chosen approach step 2 + §Tests "Matrix rows") — Specify null-safe trim (`(intent.directInviteEmail ?? '').trim()`) and enumerate `null`, `''`, and `'   '` as three distinct "no email" inputs for both visibilities. As written, a literal `null` (allowed by the interface) would throw on `.trim()`.
