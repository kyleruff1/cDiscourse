## Summary

QOL-032 ships the **Inspect popout** — the strictly read-only node / evidence / branch
detail surface ("understand what is here") that stands on the QOL-030 flash-popout
chassis beside Act (QOL-031) and Go (QOL-033). The card delivers a pure-TS content
model (`inspectPopoutModel.ts`) and a render component (`InspectPopout.tsx`), plus two
new test suites (268 tests). `buildInspectPopout(input)` produces the fixed
seven-section set with LIFE-001 stage-driven emphasis applied as a pure re-order, and
the §5 hand-off names one `ActEntryId` per stage for the host to open the Act popout
with. Inspect itself never writes, posts, edits, or opens the box.

The implementation is faithful to the design with no gaps and no undocumented
file-changes. Every doctrine line checks out: strictly read-only, plain-language only,
no verdict/winner/loser/truth copy, deterministic pure render with no AI/network/clock
dependency. Verification is fully green and the test count lands exactly on the
expected target. This is a clean Approve — no blockers, no required changes.

  independent diff scan.
- [x] Score never blocks posting — N/A; Inspect is read-only and posts nothing.
- [x] No service-role in client code — `InspectPopout.tsx` and `inspectPopoutModel.ts`
  contain no `SERVICE_ROLE` reference (the only hits are negative test assertions).
- [x] No direct insert into public.arguments — no `.insert(` / `.update(` /
  `.delete(`, no `public.arguments` reference; the model's source-scan test enforces
  this.
- [x] No AI calls in production app paths — no `fetch`, no `anthropic`/`openai`/`x.ai`,
  no `XMLHttpRequest`; deterministic pure projection, asserted by both suites.
- [x] Plain language only (no raw internal codes in UI strings) — RULE-001: the
  model's "no internal-code leak" suite scans every rendered string for snake_case and
  for raw multi-token `PointLifecycleState` / `InspectSectionId` codes; all section
  titles and §7 fallbacks are plain English.
- [x] Epic-specific doctrine:
  - **evidence-doctrine** — §E "Evidence detail" renders the EV-001 object as
    host-supplied plain text; a node with no evidence shows "No evidence attached."
    (not hidden), keeping the section set stable. Inspect derives no evidence,
    re-computes no source-chain status, and treats no popularity signal as evidence.
    §3 "What is unresolved" describes the *move*, never the person — the model's
    ban-list test explicitly checks the §3 title/label carry no person-attribution
    token (`liar`/`troll`/`bot`/`astroturfer`/`bad faith`).
  - **timeline-grammar** — "heat is activity, not correctness": Inspect authors no
    heat copy itself (the model has no heat field); §3/§4 content is host-supplied
    already framed as activity. No truth/judgment language enters any label.
  - **accessibility-targets** — every interactive control (7 section headers, the §5
    chip, prev/next) is a `Pressable` with `accessibilityRole="button"`, a verbose
    `accessibilityLabel`, and `accessibilityState` (`expanded` / `disabled`);
    `minHeight: 44` + `PADDED_HIT_SLOP` clear the 44px target; expand state is carried
    by a text caret (`▾`/`▸`) not color; decorative glyphs are
    `accessibilityElementsHidden` + `importantForAccessibility="no"`; reduce-motion is
    threaded into the chassis.
  - **expo-rn-patterns** — RN primitives only (`View`/`Text`/`Pressable`); no new
    dependency, no icon library (`<Text>` glyphs), no Bootstrap; the model file is
    pure TS with no React/Supabase import; the component is presentational and
    consumes the model unchanged.

## Suggestions (non-blocking)

1. The component suite verifies render wiring via static source-scan rather than a
   live render-tree assertion — this is a deliberate, documented repo convention
   (the pinned `react-test-renderer` is held away from `@testing-library`'s peer, see
   the suite header and the sibling `actPopoutComponent.test.tsx` /
   `goPopoutComponent.test.tsx`). No action needed for this card; if the repo later
   unpins the renderer, the popout suites could be upgraded to runtime render
   assertions as a single shared follow-up.
2. The component has no VoiceOver / TalkBack native pass recorded. The accessibility
   contract (roles, labels, states, hit targets, reduce-motion) is fully covered by
   static tests and the chassis is shared with Act/Go which carry the same contract,
   so risk is low — but per `accessibility-targets`, a native assistive-tech smoke
   pass across the three popouts would be a reasonable explicit follow-up. Not a
   blocker for QOL-032.
