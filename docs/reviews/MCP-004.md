## Summary

MCP-004 is a **design-only card** — the issue scope is verbatim "Scope (design doc only)" and the
card's own "Test plan" is explicitly labeled "(future implementation card)". The branch contains
exactly one commit (`b584b66`) adding exactly one file, `docs/designs/MCP-004.md` (391 lines). This
review is therefore a doctrine + design-completeness review of the document, not a code review.

The design formalizes roadmap-expansion §7 into a full trigger-gating, caching, batching,
retry/fallback, and token-budget plan for the semantic-referee call path. It is well-scoped:
it correctly *gates* the MCP-009 boundary's call path without redesigning it, *caches and batches*
the MCP-001 packet without adding a field, and *reads* GAME-003's `semanticClassification` mode as
an input without redesigning GAME-003's mode model. Cost control is treated as a doctrine concern
("an uncapped layer is a failed layer") with four independent caps. Every "no call" path —
forbidden event, mode `'off'`, over-budget, exhausted retry — degrades to the deterministic layer-1
result and never blocks a post. The only concern is a cosmetic internal inconsistency in §4 (the
prose says "four batch groups" then presents five groups A–E); it is non-blocking and does not
affect conformance or doctrine. The design correctly defers all production code, tests, and the
one additive `ClassifyMoveOutcome` reason to a future implementation card.

## Suggestions (non-blocking)

1. **§4 batch-group count wording.** The prose in §4 first says the families "map to **four batch
   groups**", then the table presents **five** groups (A–E), and the line below the table says
   "Five groups (A–E)". The five-group table is the substantive, correct content; the earlier
   "four" is a stale phrasing. A one-word fix ("four" → "five") in the future card's design touch-up,
   or a quick amend, would remove the ambiguity. Not a conformance or doctrine defect — flagged only
   so the implementer reads the table, not the prose, as authoritative.
2. **Pre-send review timeout.** §5 / §"Risks" already flags that the `2 × 8 s` worst case is long on
   the one synchronous trigger and suggests a shorter dedicated timeout for `pre_send_review`
   (e.g. 4 s, single attempt). Consider promoting that from a "Risks" aside to an explicit
   operator-decision item so it is not lost — the design itself already hints at this.
