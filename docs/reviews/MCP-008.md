## Summary

MCP-008 is a **design-only** card; commit 5b6cf35 adds exactly one file —
`docs/designs/MCP-008.md` (569 lines). The design specifies the referee feedback
banner library: a pure-TS string library + classifier-id→banner map +
tone-band model + confidence-handling rule + accessibility contract. It is a
presentation layer over the already-merged MCP-001 packet contract, the MCP-002
seed bank, and the MCP-003 ledger — it defines no new classifier, no new score
category, no migration, no Edge Function, no live model call. All five issue
scope items are addressed concretely. Every exported symbol the design names
from existing source was verified against real code; the `gameCopy.ts` surface
is cited correctly and the design explicitly disavows the non-existent
`ALL_INTERNAL_CODES` symbol (the trap a prior MCP review fell into). Doctrine is
clean: no banner string in the representative sample contains a verdict,
person-label, or popularity-praise token; the doc carries a ban-list, a
12-line doctrine self-check, and a six-file test plan. The only issues found
are cosmetic documentation typos — non-blocking.

## Suggestions (non-blocking — fold into a doc touch-up or carry to the impl card)

1. **Classifier-id range typo (×2).** §5.3 rule 1 writes
   "every id A1…G12, B1…B12, C1…C18, D1…D14, E1…E12, F1…F12, G1…G10" and §13
   dependency 2 repeats "(A1…G12, B1…B12, …)". The leading token "A1…G12" is a
   typo for **"A1…A12"** (group A has 12 ids; the seed bank confirms A1–A12).
   The total (90) and every other range is correct; only the first token is
   garbled. Cosmetic, but the reviewer brief flags symbol/range accuracy
   specifically — worth a one-character fix so the impl card's coverage test
   iterates the right documented list.
2. **Line-count drift.** §9 says this design document is "~520 lines"; the file
   is 569 lines. Minor; update or drop the parenthetical.
3. **§5.1 feedbackCode column is partial.** The "Maps from (MCP-003
   `feedbackCode`)" column lists ~14 of the 20 MCP-003 codes; §5.3 +§12
   correctly require *all 20* to be `FEEDBACK_CODE_TO_BANNER` keys. Not a
   contradiction (the §5.1 column is illustrative, the binding requirement is
   §12), but the impl card author should map all 20, including the six not
   shown in §5.1 (`evidence_connects`, `evidence_needs_connecting`,
   `nice_narrowing`, `concession_noted`, `clarification_in_play`,
   `back_to_the_claim`). Consider a one-line note in §5.1 pointing to §12 as
   the authoritative coverage requirement.
4. **`hasSoftenedSibling` + library-as-array invariant.** §4 / §7.2 make the
   softened sibling its own library entry with its own `bannerCode`. The impl
   card should add a test that for every banner with
   `hasSoftenedSibling: true`, a sibling entry with `minConfidence: 'low'`
   actually exists in `REFEREE_BANNER_LIBRARY` — §12's confidence file implies
   this but does not name the existence check explicitly. Carry to the impl card.
