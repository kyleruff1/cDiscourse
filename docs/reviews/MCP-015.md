## Summary

MCP-015 ships the pure-TypeScript semantic override UX model that implements MCP-010 §5.2: the
trigger model (`evaluateSemanticOverridePrompt`), the append-only record-builder
(`buildSemanticOverrideRecord`), the two `MetadataEvent` projectors, the route-suggestion → lane
mapper, and the in-memory per-room `RepeatedOverrideSignal`. The implementation matches the design
exactly — same file footprint, same type shapes, same algorithm order (actor gate → no-input gate →
L1/L2 conflict → low-confidence → confident path). The only changes outside the new
`src/features/semanticOverride/` directory are the documented additive `'semantic_override'`
`codeFamily` member in META-001's `moveMetadataLedger.ts` and 16 append-only override copy codes in
`gameCopy.ts`. All four verification commands pass; all safety scans are clean (the only grep hits
are documented false positives — ban-list arrays and absence-assertion test code). Every doctrine
invariant for this card holds and is genuinely test-enforced: no-penalty (object-key + export-name +
source-text scans), append-only (`kind: 'add'` always, no remove path), observer-excluded (actor
gate is step 1, tested for both trigger paths), repeated-signal never persisted (exact-key shape
assertion), and pure-TS (forbidden-imports static scan). No concerns remain. The React choice
surface is deferred to a scoped follow-up per the design and the issue — that deferral is
legitimate and the model is independently complete and testable.
