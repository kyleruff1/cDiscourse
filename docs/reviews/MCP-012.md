## Summary

MCP-012 ships the semantic call router as five pure-TypeScript modules under
`src/features/semanticReferee/` ‚Äî `triggerGates.ts`, `classifierBatching.ts`,
`semanticCache.ts`, `tokenBudget.ts`, `retryPolicy.ts` ‚Äî plus an append-only edit
to `index.ts` and eight new `__tests__/` suites. The router is a deterministic
translator from a UI/game event into the decision "should a semantic-referee
classifier call be made, and of what?" ‚Äî it never decides "should this post be
allowed?". The implementation matches the design with no deviation: the file
footprint is exactly the expected set, MCP-011's four source files are untouched,
the four MCP-011 export lines in `index.ts` are byte-identical, and every doctrine
constraint named in the design holds. All four verification commands pass and the
test count moved cleanly from the 5959/218 baseline to 6117/226. No blockers, no
concerns ‚Äî ready for the operator to push.

Harmless and well-commented ‚Äî no change needed, noted only for the operator's awareness that the batcher silently coerces.
2. `semanticCache.ts` keeps `SemanticCacheEntry.lruTick` (and the `tick` counter) for entry-level access ordering, while eviction in practice relies on `Map` insertion order. `lruTick` is currently informational rather than load-bearing for eviction. This matches the design's "for LRU ordering ONLY" comment and is harmless; a future card could drop the field if it stays unused, but it is fine to keep as a debugging aid.
