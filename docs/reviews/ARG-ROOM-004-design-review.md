## Summary
This is a strong, honestly-scoped design that genuinely builds-on-shipped rather than rebuilding: the acceptance spine, the auth-callback consumer, `pendingInviteIntent`, and the routing-priority handoff are all reused, and the citation accuracy is exceptional (23/23 spot-checked citations materially correct). It correctly defers the seat-cap to ARG-ROOM-003, keeps `parseAuthCallbackUrl` un-widened, and respects the no-enumeration / no-token-leak / no-service-role-in-client doctrine. The blocking gap is that the **new-user Auth-invite branch has no in-function gate** and therefore is *not* "off by default" — the cited smoke doc proves the hosted Auth email is live, so 004a as written sends real invite emails on merge, contradicting both the design's own "no behavior change on merge" claim and the roadmap's binding gated-email posture. Fixable with one env gate + one test; the rest is tightening.

## Doctrine & security check
- **No-enumeration:** Uniform response (c) + server-side branch is sound and mirrors the shipped `admin-users` pattern (:397-399, :423). Good. One caveat folded into comment 3.
- **Raw token never surfaced:** `room-notifications` `inviteLink` stays `null` at :416; the new exposure is the raw token inside the server-side `redirectTo` passed to `inviteUserByEmail` and inside the server-side Resend body — both equivalent to the already-shipped `/invite/<token>` capability link, neither logged nor returned. No new client-facing token surface. Compliant.
- **No service-role in client:** Send + Auth-invite run inside `manage-room-invite`; `inviteApi.ts` header (:6-8) confirms the client never imports service-role. Compliant.
- **RLS / migration:** No migration, no new table; reuses `argument_room_invites` + the `(debate_id,user_id)` PK. Compliant.
- **GATE-C posture — VIOLATED for the new-user branch (see comment 1).** The roadmap (§5) binds "Email send stays gated + default OFF (QOL-040 #209 owns the flip)" and the design asserts "both transport gates stay OFF" (§164) and "merge is safe — no behavior change" (§213). But the cited smoke doc (`2026-06-13-auth-live-invite-seed-smoke.md`:50 "rate limits untouched", :56 `notification:"sent"`, :58-68 full success, :70-74 devtest97 hit the live rate limit) proves the **hosted Auth email is armed**. The existing-user Resend path is genuinely gated by `INVITE_EMAIL_ENABLED`; the new-user `inviteUserByEmail` path has **no equivalent in-function flag**, so on 004a merge the create path begins sending live Supabase-branded invite emails to any creator-supplied address (today it sends none — §238 confirms create fires no email). That is a real behavior change + an abuse vector, and there is no gate to withhold for the very path the design says "Never self-approve the gate flip" (§216) protects.

## Completeness & edge cases
- **Idempotency:** Correctly relies on `handleAccept`'s same-redeemer early return (:547-558) and the PK 23505 no-op (:597-607); double-accept consumes exactly one seat; double-handoff (§230) converges on the same accept. Sound — no new accept logic added.
- **Reserved-seat accounting:** Correctly deferred to ARG-ROOM-002/003; `handleAccept` stays count-blind (:561-577). The DAG builds 003 before 004, so the latent "accept enrolls before cap exists" window is 003's responsibility, not a 004 defect. Honest scoping.
- **Email-failure → retryable room:** Row is inserted (:261-295) before any send; send is best-effort after, so a failed send never rolls back the invite. For a new user whose Auth-invite failed, recovery is the copyable `/invite/<token>` link → existing `App.tsx` :170-206 capture → `SignedOutPrompt` (:209-223) → sign-up with the same email → shipped auto-accept. Works, but the design states this only obliquely via the §227 rate-limit risk — make the new-user failed-Auth recovery path explicit.
- **Notification uniformity across the SPLIT:** In 004a (new-user branch only, before 004b lifts the Resend path), an *existing*-user invitee has no transport yet. The implementer must ensure both new and existing invitees still yield a branch-independent `notification` in 004a's interim state (the shipped `handleCreate` already hardcodes `'queued'` on every path — preserve that), or 004a re-introduces the enumeration §221 is trying to prevent.

## Actionable comments
1. **[blocking]** (§70-72, §155-156, §164, §213; evidenced by `2026-06-13-auth-live-invite-seed-smoke.md`:50/:56/:70-74) — Add an explicit in-function env gate on the new-user `inviteUserByEmail` call (e.g. `INVITE_AUTH_BRIDGE_ENABLED`, default OFF), returning the uniform dormant `notification` when unarmed, so 004a lands dormant and the live send is the operator's flip. The "hosted Auth SMTP is the gate" assumption is contradicted by the smoke doc — the hosted Auth email is demonstrably live. This gate must live in **004a** (the first-landing card), not be deferred to 004b.
2. **[should]** (§98, §162 vs §59) — Re-anchor the `App.tsx` insertion point from :125-129 to **:170-206**. :125-129 is a `useState` initializer (render-phase, synchronous) where you cannot `dispatch` or run the async `savePendingInviteIntentToStorage`. The :170-206 cold-start effect already does `parseInviteDeepLink → buildPendingInviteIntent → save → dispatch`; the bridge extractor slots in there in ~3 lines. The timing argument still holds (the higher-priority `AuthCallbackScreen` is gated by `authCallback.active`, set synchronously at :125-129; the intent dispatch landing one tick later does not pre-empt it).
3. **[should]** (§112, §221) — Pin the uniform-`notification` rule to a single gate-posture predicate independent of branch, and state it explicitly. The two branches have *different* gates (`INVITE_EMAIL_ENABLED` vs the new-user Auth gate from comment 1); deriving `notification` from each branch's actual transport return would leak existing-vs-new. Safest: preserve the shipped `notification: 'queued' as const` invariant. Ensure this holds in 004a's interim state too (existing-user has no transport yet).
4. **[nit]** (§49) — Fix the misquoted :416 comment ("the raw token is unrecoverable from the hash" → actual: "The invite link is not exposed via this function — it lives in inviter's create-time response"). The substantive reconciliation-gap fact is correct.
5. **[nit]** (§120, `sendInviteSmoke.js`:45-49) — The dry-smoke "`?invite=` redirect resolves to an allow-listed `/auth/callback` host" assertion should validate against the **hosted** `uri_allow_list` (config.toml :164-170 / hosted readback), not just the script's local `ALLOWED_REDIRECT_HOSTS` (which drifts — it lists `dev.cdiscourse.com`, absent from config.toml), so the dry assertion actually proves the live precondition behind open-question 1.

## Ready to implement?
Needs a bounded design revision before GATE-A, not a full redo. The foundation is sound and the citations are real, so this is **green-light after comment 1 is resolved and comments 2-3 are folded in**. Comment 1 is the gate that keeps 004a from becoming a live email-sender on merge — it is the whole point of the GATE-C "land dormant, flip later, never self-approve" posture and must be settled (gate added, or operator explicitly ratifies that the live hosted Auth email IS the accepted gate and 004a may ship sending). Comments 2-3 prevent a subtle wrong-insertion-point bug and an enumeration regression across the split. With those, the SPLIT (004a first, carrying the gate + uniform-response rule; 004b behind `INVITE_EMAIL_ENABLED`) is sound and ready to build.
