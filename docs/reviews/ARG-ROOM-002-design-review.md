# ARG-ROOM-002 — design review
**Reviewer:** workflow design-reviewer · **Reviewed:** 2026-06-13 · **Design:** docs/designs/ARG-ROOM-002-BACKEND-VISIBILITY-CAPACITY-INVITES.md · **Issue:** #613

**Verdict:** changes-requested

## Summary
This is a strong, unusually well-cited design that genuinely builds on the five shipped primitives rather than rebuilding them: it reuses the QOL-039 visibility column + `SECURITY DEFINER` helpers, the QOL-038 invite table + token-hash discipline + the `manage-room-invite` accept path verbatim, and the `20260516000006` recursion-safe helper pattern. The recursion landmine is honored, the migration is append-only, the reserved-seat accounting is correct against the *actual* accept-path ordering, and GATE-C posture is explicit. It is **not** ready to build as written for one reason: the card's headline promise — "make the room-creation contract true at the database" — is not met for `private ⇒ invite`, because the shipped `"debates: authenticated can create"` INSERT policy is left open, so a client can create a private-no-invite room by inserting into `public.debates` directly via PostgREST, fully bypassing the new RPC. That plus an over-broad helper grant (a minor invite-email oracle) are the two changes I want before GATE-A.

## Build-on-shipped check
The design reuses the right seams and explicitly refuses to re-implement them: capacity is derived from `visibility` (no new column, mirroring `publicSeatModel`'s "NO write path / NO new DB column" doctrine), it reuses the `manage-room-invite` accept path unchanged, and it adds the one-invite-per-room index as *additive* on top of the shipped per-address index. The single deliberate "replace" (client `createDebate` insert → server-authoritative Edge+RPC) is correctly justified by cross-table atomicity.

## Doctrine & security check
- **Migration append-only:** honored. The new file `20260613000001_*` is additive; the participants INSERT policy is replaced via DROP/CREATE in the *new* migration, never by editing `20260516000006`. RLS is never disabled.
- **Recursion landmine:** honored. Every cross-table read in a policy/trigger goes through a `SECURITY DEFINER STABLE` helper; no raw cross-table subquery in any policy.
- **No service-role in client / no direct client write to the protected create path:** *Partially.* The new RPC is `service_role`-only and the client wrapper holds no key — good. **But the protected path is not actually closed** (see [blocking] #1): the shipped `"debates: authenticated can create"` policy (`20260516000002_rls_policies.sql:157-160`, `WITH CHECK (created_by = auth.uid())`) still lets the client insert a `debates` row directly, including `visibility='private'` with no invite. `private ⇒ invite` lives only in the Edge validator + the RPC, not at the DB for all writers.
- **Raw token never surfaced:** honored — hashed in Deno, only `token_hash` persisted, raw token returned once to the creator (mirroring `manage-room-invite:313-324`), never logged/listed/MCP'd.
- **No account enumeration:** the create response is uniform regardless of whether the invited email maps to a user — good. One caveat in the helper grants (see [should] #2).
- **No verdict/heat copy:** the `ROOM_CAPACITY_COPY` strings are structural ("the most people it can hold", "both seats are taken", "open seat"); the ban-list reuse of `_forbiddenChimeInGovernanceTokens()` is sound.
- **GATE-C:** explicit, correct, and well-handled — names the migration+RLS-bearing risk, never-self-approve, the QOL-041 incident class, `config.toml` registration as the deploy trigger, and `npx supabase db reset` before done.

## Completeness & edge cases
- **Reserved-seat accounting: correct.** I traced all six worked rows against the real accept-path ordering. The "exclude the joining user's own pending invite" + the `NOT EXISTS` addressee-already-seated guard together make both the full-private-room accept and the public self-join-while-pending case land on the right counts. This is genuinely well-reasoned.
- **Trigger-fires-for-every-writer:** correct and load-bearing — service-role bypasses RLS but not triggers, so the accept path and the RPC creator-join both pass the cap check. Good.
- **The gap the design omits (the headline):** with the `debates` INSERT policy open, the only thing forcing creation through the RPC is the *client code*, not the *database*. To make `private ⇒ invite` "true at the database," the migration must also revoke/replace the authenticated `debates` INSERT policy so the `service_role` RPC is the sole room creator. The design's own "Alternatives rejected" section argues the cross-table check isn't a simple constraint and routes creation through the RPC — but never closes the other door.
- **Subcard sequencing coupling (002a vs 002b):** 002a tightens the participants INSERT policy to public-only self-join, on the assumption the creator auto-join has moved to the service-role RPC. If 002a merges before 002b, the *legacy* `createDebate` client path's creator auto-join into a **private** room (`debatesApi.ts:146-148`, a client insert) would be refused by the new policy, stranding the creator out of their own room. It's masked today only because the live surface can't create private rooms until ARG-ROOM-003. Make the co-landing constraint explicit.
- **`manage-room-invite` create after the per-room index:** the new `(debate_id) WHERE status='pending'` index means a second pending invite to a *different* address now throws `23505`. If the shipped `create` action relies on the unique index for its "already invited this address" message, a post-creation invite collision will now be mislabeled. Flag which card owns that copy shift (likely ARG-ROOM-006).
- **Idempotent re-accept / observer-uncapped / null-cap-FK branch:** all handled in the trigger sketch. Good.

## Actionable comments
1. **[blocking]** §"Data / API shape" + §"Doctrine compliance" — Close the direct-insert door. The migration must also DROP/replace `"debates: authenticated can create"` (`supabase/migrations/20260516000002_rls_policies.sql:157-160`) so the `service_role` `create_argument_room` RPC is the *only* room creator. As written, `private ⇒ invite` is enforced only in the Edge+RPC path; a direct PostgREST `debates` insert with `visibility='private'` and no invite still succeeds, which contradicts the card's stated goal ("true at the database") and the issue title's "private-requires-invite enforcement." Add the refusal test (Test-plan gap #1). Confirm no other code path legitimately inserts `debates` from the client (the bot-fixture/runner create paths may need to move to the new function — note this for the implementer).
2. **[should]** §"New migration" step 4 — `count_reserved_invites(uuid, text)` is granted to `authenticated`, but it is only ever called from the `SECURITY DEFINER` capacity trigger (which runs as owner and does not need the invoker to hold EXECUTE). Granting it to `authenticated` creates an invite-email-confirmation oracle: `count_reserved_invites(known_debate_id, 'guess@x')` returns 0 when the guess is the room's pending invitee and 1 otherwise — a probing surface the shipped `argument_room_invites` RLS deliberately denies. Apply the design's own `user_email_lower` reasoning ("intentionally NOT granted to authenticated") to `count_reserved_invites` and `count_active_participants`; revoke both from `authenticated`. Only `room_active_seat_cap` is harmless to grant (and the pure-TS twin means even that isn't required).
3. **[should]** §"GATE-C + merge posture" — Make explicit that 002a's tightened participants INSERT policy and 002b's service-role creator-join must co-land (or that the live create surface stays public-only between them). Otherwise a private creation via the legacy `createDebate` path (`debatesApi.ts:146-148`) is refused by the new policy and strands the creator out of their own room.
4. **[should]** §"Open questions" #3 — Note that the new per-room pending index changes what a `23505` means in the shipped `manage-room-invite` `create` action (now "room already has a pending invite to *anyone*", not just "this address"). Name the card that owns the relabeled copy.
5. **[nit]** §"Existing shipped state" — the citation `20260524000013:38-42` for "no invite row is hard-deleted (status flips only)" points at the re-invite partial-index note; the soft-state doctrine actually lives in the table comment at `:166-167`. Re-anchor.
6. **[nit]** Creator is inserted as `side='moderator'`, which the cap counts as active (correct for the matrix), but in a private 1v1 this yields a `moderator` + a `negative` with no `affirmative`. This is pre-existing shipped behavior (`debatesApi.ts:148` + `manage-room-invite:592-595`), faithfully preserved — just confirm it's intended for 1v1 and not a side-assignment smell to fix in this slate.
