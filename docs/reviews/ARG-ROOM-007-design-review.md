## Actionable comments
1. **[should]** `report.js` is CommonJS but "reuse the matrix ban-list (`argumentRoomCreationMatrix:406-439`)" implies importing a `.ts` source — a `require()` of a TS file fails without transpile (and a stray `.ts` import path can trip lint, per the repo's eslint-scan lesson). Follow the established repo idiom: **mirror** the ban-list in the CommonJS harness with a **parity test** pinning it equal to `_forbiddenArgumentRoomCreationTokens()` (the same way `publicSeatModel`/`chimeInGovernance` each author their own `_forbidden*Tokens`). State "mirror + parity," not "reuse."
2. **[should]** "4 accounts exist today (admin + bot A/B/C)" overstates `loadEnv.js`: REQUIRED_KEYS guarantees only admin + A + B (3); bot C is optional (the A/B/C loop `continue`s when absent). Soften to "3 required + C optional," and have the runner report the **resolved N distinct accounts** in the report header (it already reserves "N distinct") so #5/#7 branch on the real runtime count rather than an assumed 4.
3. **[should]** Specify the harness behavior when **004 is not yet deployed** (the current HEAD is 003/8929bde; 004=#615 is a separate GATE-C card). #9/#10/#11 should degrade to an explicit `dependency_not_deployed` skip (like `accounts_insufficient`), never a false FAIL. The dry harness + pure tests + #1-#8/#12/R1-R3 are authorable/mergeable now; only the email-bearing live checks wait on 004 — make that skip path first-class.
4. **[should]** Open Q #1 (which Edge emits the create-time-invite email under deployed 004 — `create-argument-room` vs `manage-room-invite handleCreate`) is a real unknown for #9/#10/#11. The "precondition probe that records which Edge emits the email" lives only in the runner narrative; give it an explicit home in the runner's precondition phase and a line in the report Preconditions table so the operator can't skip it. (Accept is always `manage-room-invite accept` — that half is settled.)
5. **[nit]** Open Q #4 (transiently-visible `open` public rooms during #5/#7) is honestly surfaced; recommend the design pre-commit to the operator decision (low-traffic window + immediate archive, or defer cap-5 to a staging project) rather than leaving it fully open, so the live runbook is unambiguous.
