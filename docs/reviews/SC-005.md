## Summary

SC-005 replaces the dense flex-wrap observer-actions grid plus `App.tsx`'s
separate bottom `actionBar` with a single collapsed contextual dock: a
context-aware pill that expands into a side-anchored panel on wide viewports
or a capped bottom sheet (~28% of the viewport, never full-screen) on narrow
ones. The implementation is a clean, well-scoped LAYOUT card. The Stage 6.4
action-set definitions (`RailActionCode` … `railActionToBubbleControl`) are
byte-for-byte unchanged — confirmed by diffing against the merge-base and by
the SC-001 contract-lock suite (`railActionGrouping.test.ts`) staying green
with zero edits. The new layout logic is correctly split into a pure-TS
module (`ObserverActionDockLayout.ts`) with no React/Supabase/network/AI
imports. Doctrine holds throughout: the dock surfaces moves not verdicts, the
own-bubble matrix stays `[qualifiers, request_deletion]`, no heat/score is
read, and the ban-list suite scans every visible string. typecheck, lint, and
the full 5021-test suite all pass. The only concern is non-blocking: the
branch is one commit behind `origin/main`, which makes the `origin/main..HEAD`
diff show three unrelated MCP-001 docs as "deletions" — they are not deleted
by SC-005, they simply post-date the branch point. Operator should rebase
before opening the PR.

## Suggestions (non-blocking)

1. **Rebase before the PR.** The branch is one commit behind `origin/main`
   (`579e889` — "docs: MCP semantic-referee roadmap expansion …"). Because
   SC-005 branched at `e94c9dd`, the `origin/main..HEAD` diff shows
   `docs/designs/MCP-001.md`,
   `docs/roadmap-expansions/2026-05-20-mcp-semantic-referee-roadmap.md`, and
   `docs/semantic-prompts/mcp-semantic-referee-prompt-bank.md` as deletions.
   They are NOT deleted by SC-005 — they post-date the branch point. A
   `git pull --rebase origin main` before opening the PR makes the diff
   honest and avoids the appearance of removing three unrelated docs. The
   files do not collide with any SC-005 file, so the rebase is clean.

2. **The a11y suite is a source-scan suite, not a render-behavior suite.**
   `sideActionDockA11y.test.ts` greps `ArgumentSideActionRail.tsx` for the
   accessibility attributes rather than mounting the component and asserting
   on a rendered tree. This matches the existing `composerDockA11y.test.ts`
   pattern the design cites, so it is consistent and acceptable — but a
   source-scan can drift if the file is refactored (e.g. an attribute moved
   to a spread prop). Not a blocker; a future render-based a11y assertion
   would be more robust. The design's own § "Operator next steps" already
   recommends a manual VoiceOver/TalkBack + keyboard pass before merge —
   keep that.

3. **The SC-005 design doc is committed at `Status: Design draft`** (line 3).
   Cosmetic — consider bumping to "Implemented" / "Shipped" when the PR
   merges so the doc header reflects reality. Not blocking.
