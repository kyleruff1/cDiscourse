## Summary

AN-002 ships a pure-TS, dev/QA-only library of 8 named deterministic argument-timeline
fixtures (`src/features/analytics/visualQaFixtures.ts`), a 34-test model suite that runs each
fixture through `buildArgumentTimelineMap` and asserts the structural invariant its name
promises, and a hand-authored browser visual checklist (`docs/visual-qa-snapshots.md`) with one
section per fixture. The implementation matches the design closely — every file change in the
design is present, no undocumented file change exists, and the only product code is a new
isolated module the production app never imports. Tests, typecheck, and lint all pass; the
secret and doctrine scans are clean. The card adds product-zero behavior: it is purely a QA
surface plus its automated verification, which is exactly what the card and Epic 13 ("analytics
without AI calls") asked for. No blockers; one minor non-blocking observation on the ban-list
token set.

### Each of the 8 fixtures genuinely builds the promised shape

Verified that the tests assert *shape*, not merely "it builds":

1. `no_rebuttal` — one root `thesis`; test asserts `nodes.length === 1`, `isRoot`,
   `hasRebuttal === false`, `firstRebuttalMessageId === null`, `edges.length === 0`,
   `rootOnboardingHint` non-null.
2. `straight_chain_10` — root + 9 single-child replies; test asserts `nodes.length === 10`,
   deepest `depth === 9`, `edges.length === 9`, **every `lane === 0`**, no junctions. The
   `lane === 0` assertion directly guards the Stage-6.3 first-child-continues-the-rail rule.
3. `source_chain_fight` — test asserts ≥ 5 nodes, ≥ 2 `clarify`-family nodes each carrying a
   `source_request` dropped tag, ≥ 1 `evidence`-family node, and zero `concede`-family nodes
   (the fight is genuinely unresolved).
4. `evidence_heavy_branch` — test asserts ≥ 6 nodes, ≥ 3 `evidence`-family nodes, an
   `Evidence run` band present, and ≥ 1 node on a non-zero lane. Confirmed `ev1`→`ev2`→`ev3`
   are a contiguous chain so the band heuristic (contiguous 2+ evidence-family nodes in node
   order) genuinely fires; `source` correctly maps to the `evidence` color family via
   `KIND_COLOR_FAMILY`.
5. `tangent_kink_branch` — test asserts ≥ 7 nodes, exactly one `isJunction` node with
   `junctionChildCount >= 2`, ≥ 1 node `lane !== 0`, exactly one `isDetached` node, and that
   node's `accessibilityLabel` contains "detached". The detached node (`kink1`) points at a
   genuinely absent `parentId` and is referenced nowhere else.
6. `synthesis_path` — test asserts ≥ 5 nodes, ≥ 1 `concede`-family node, exactly one
   `synthesis` input, and the chronologically-last node is that synthesis node with
   `kindColorFamily === 'concede'`.
7. `stress_board_250` — builder emits root + 50 + 50 + 100 + 50 = 251 inputs; test asserts
   `nodes.length === 251`, exactly 50 detached, unique node + edge ids, strictly monotonic `x`.
   Genuinely 250 nodes plus root, matching the existing inline `buildStressFixture` count.
8. `avatar_profile_display` — test asserts ≥ 4 nodes, ≥ 3 distinct `authorId`,
   `participantTrends.length >= 3`, one node `actorLabel === 'You'` and one `actorLabel === 'Bot'`.
   Builder genuinely exercises self (`AVATAR_PROFILE_DISPLAY_CURRENT_USER_ID`), other, and an
   `isBot: true` author — `classifyActor` maps these to `self`/`other`/`bot` correctly.

### LIFE-001 cross-check tests

The 3 optional cross-check tests were kept and are correct. `buildPointLifecycleMap`'s input
contract is `{ timelineMap, artifactsByMessageId }` (verified in
`src/features/lifecycle/pointLifecycleModel.ts:1076,1092-1095`); the fixtures satisfy it with an
empty `Map` for `artifactsByMessageId`. The tests assert real lifecycle outcomes — a
resolved-family cluster state for `synthesis_path`, at least one cluster for `source_chain_fight`
with `byMessage.size` matching node count, and exactly one `open` cluster for `no_rebuttal`.
The implementer's §R4 judgment ("contract satisfiable, keep them") is sound.

### Two minor implementer choices — both sound

- (a) The "no node resolves to the `default` color family" test asserts
  `node.kindColorFamily !== 'default'` on the built map rather than calling an unexported
  `getKindFamily`. Correct — `getKindFamily` is module-private in `argumentGameSurfaceModel.ts`;
  asserting on the public `TimelineNode.kindColorFamily` is the right seam and avoids exporting
  internals just for a test.
- (b) The extra exported constant `ALL_VISUAL_QA_FIXTURE_IDS` mirrors AN-003's
  `ALL_POINT_LIFECYCLE_STATES` convention and gives the test suite a clean iteration source.
  Consistent and harmless.

## Suggestions (non-blocking)

1. The implementation's `BANNED_TOKENS` array omits `'true'` and `'false'`, which the design's
   §Test plan (line 440) lists. This omission is deliberate and correct — a whole-word
   `\btrue\b` / `\bfalse\b` scan would false-positive on ordinary code/prose and the fixture
   content carries no `true`/`false` as verdict labels. Worth a one-line code comment in the
   test noting why those two design-listed tokens are intentionally excluded, so a future reader
   does not "restore" them and break the suite. Implementer can address or defer.
2. §R3 of the design notes the deliberate duplication between the new `buildStressBoard250Fixture`
   and the inline `buildStressFixture` in `argumentTimelineMap.test.ts`. A one-line follow-up
   card to dedupe (point the existing test at the new builder) would be reasonable — explicitly
   out of AN-002's scope and not required here.
