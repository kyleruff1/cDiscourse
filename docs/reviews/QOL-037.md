## Summary

QOL-037 adds the **evidence applicability axis** — a structured way to dispute
*what* a piece of evidence applies to, as a third independent status axis
alongside EV-001's `SourceChainStatus` (existence) and EV-003's evidence-debt
(obligation). It ships a pure-TS model (`evidenceApplicabilityModel.ts`), locked
copy (`evidenceApplicabilityCopy.ts`), two RN components (`RespondToEvidenceForm`,
`ApplicabilityChip`), an additive `MoveDraftPatch.evidenceResponse` field, and a
deliberately minimal advisory pass-through in `submit-argument`. The
implementation matches the design doc closely and respects every doctrine line:
the applicability axis is a *status* (`undisputed` / `disputed` / `supported`)
with no truth-negative value by design, the required-clarification rule is a
validation gate that never affects score, and `applicability_supported` is
reachable only via corroborating moves, never popularity. The highest-risk item
— the `submit-argument` change — is genuinely minimal, advisory, backward
compatible, and adds no new hard-block, insert-path branch, or RLS change. All
verification passes. The single concern (component-file instrumented coverage)
is a non-blocking note: QOL-037 follows the repo's actual RTL-free test
convention, which the sibling Evidence components also use.

## submit-argument change assessment (the highest-risk item)

The change to `supabase/functions/submit-argument/index.ts` and
`supabase/functions/_shared/validationSchemas.ts` is **safe and exactly what
design §7.4 specifies**. Verified line-by-line:

- **Schema** (`validationSchemas.ts`): adds `EvidenceResponseSchema` (three fields:
  `evidence_artifact_id` / `choice` / `clarification_body`, all `z.string()`) and
  attaches it to `SubmitArgumentSchema` as `evidence_response: …optional()`.
  `choice` is a permissive `z.string()`, **not** a `z.enum` — a malformed/future
  choice can never fail schema validation and block a post. A request omitting
  the field is byte-identical to the pre-QOL-037 shape (backward compatible).
- **Function** (`index.ts` lines 314–327): the only change is `serverValidation`
  is re-typed `Record<string, unknown>` and, **after** the existing blocking
  check (`if (!evalResult.allowPost)` at line 301, unchanged — still the sole
  post gate) and **before** the insert, `if (data.evidence_response)
  serverValidation.evidenceResponse = data.evidence_response;` copies the block
  **verbatim** (no per-field re-mapping) into the `server_validation` JSONB
  snapshot already written.
- **No new hard-block**: there is no `validationFailed` / `forbidden` call keyed
  on `evidence_response`. The function does not validate, branch on, or reject
  the block.
- **No insert-path change**: `argInsert` (lines 322–337) gains no column —
  verified the block appears nowhere in the insert row. No migration is needed
  or present.
- **No RLS change**: no `disable row level` / `alter policy` anywhere.
- **No secret / service-role escalation / Authorization logging**: the diff adds
  no logging at all; the service-role client usage is unchanged from before.

This is a contract addition, not a behaviour change. **The merge does not depend
on the deploy** — an un-redeployed function simply drops the optional field
(the move body still posts; the applicability status just would not persist for
other viewers until the deploy lands). Backward compatible in both directions.

## Suggestions (non-blocking)

1. **Component-file instrumented coverage is ~41–43% on the two `.tsx` files**,
   below the design §11 "≥90% line coverage on `RespondToEvidenceForm.tsx` /
   `ApplicabilityChip.tsx`" target. This is **not a defect** and **not a blocker**:
   `@testing-library/react-native` is not installed in this repo, and every
   sibling Evidence component test (`SourceChainPopover.test.tsx`,
   `ReceiptChip.test.tsx`, `EvidenceDebtChip.test.tsx`) uses the same
   pure-planner-extraction + source-scan pattern QOL-037 uses. QOL-037 correctly
   extracts the load-bearing logic (`planRespondToEvidenceForm`,
   `composeApplicabilityPreviewLine`, `buildApplicabilityChipContainerStyle`)
   into pure helpers that *are* fully covered, and the test files document the
   choice. The design's 90%-`.tsx` target is aspirational and inconsistent with
   the repo's actual RTL-free reality; adding RTL would be a speculative
   dependency the dependency policy forbids. No action required for this card —
   if the team later adopts RTL repo-wide, mounting these two components is a
   natural follow-up.

2. **Constitution argument-type mapping (design open question 1)** — the design
   §9 / §16 flag that the seven choices route to existing argument types
   (`evidence_acceptance` / `evidence_challenge` / `clarification_request`) and
   that `accept` / `accept_with_caveat` may need a `concession` type. QOL-037
   correctly adds **no** new Constitution transition (engine untouched), and the
   `respond_to_evidence` host wiring that actually selects the argument type is
   out of scope for this card (it belongs to QOL-030's `renderSchema`
   integration). When QOL-030 lands, the operator/implementer should confirm the
   Constitution has a clean type for each choice before wiring the host — this
   is already an open question on the design, not a gap in QOL-037's delivery.
