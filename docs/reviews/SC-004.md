## Summary

SC-004 lands a context-aware timeline node action dock that turns the Timeline/Tree into a playable surface. The pure-TS model (`timelineNodeActionDockModel.ts`) consumes LIFE-001 / META-001 / EV-001 outputs read-only and produces a deterministic render contract that surfaces a primary suggestion plus an ordered action list, with full coverage of the 19 LIFE-001 lifecycle states, 10 META-001 manual tag codes, 16 META-001 auto metadata codes, and 6 EV-001 source-chain statuses. The RN component (`TimelineNodeActionDock.tsx`) uses only RN core primitives — no new dependency. The COPY-001 disambiguation rule (cluster header vs move chips with dedup when labels collide) is encoded structurally in `buildMoveChips` and is covered by a dedicated test. The SC-002 popover ↔ SC-004 dock mutual exclusion is enforced at three layers (handleNodeTap short-circuit, handleInfoTap dock-dismiss, JSX `!popoverModel` guard) with 5 dedicated tests. Doctrine is intact: dock RECOMMENDS, never BLOCKS; lifecycle is play state not truth; no verdict tokens; no service-role; no direct insert into `public.arguments`; no route push; no AI inference path; no new dependency; no migration; no Edge Function deploy; no `.env*` change. All 3071 / 113 suites pass. Frozen surfaces (BR-001 / LIFE-001 / META-001 / EV-001 / EV-002 / SC-002 / VG-002) are zero-diff. One known gap: room-shell composer wiring routes `narrow` / `confirm` / `synthesize` through `handleAction('reply', …)` so the composer opens but does not yet auto-seed the preset body. The model returns the correct `MoveDraftPatch` and `actionDockToComposerPreset` is fully tested; the gap is explicitly flagged by the implementer as discovery follow-up #9 and is not required by the design to ship in SC-004 — the design's File-changes wording ("dispatches `onActionDockAction` to the composer ... as appropriate") and Test-plan §12 (composer preset mapping) only require the model-level mapping, which is correct.

## Composer wiring gap — judgment

**Approve with follow-up.**

The implementer's `actionDockToComposerPreset` returns the correct `MoveDraftPatch` for `narrow` / `confirm` / `synthesize` (and the other six post-producing actions). The model-level contract is correct and is fully tested at `timelineNodeActionDockModel.test.ts:899-974`.

The room-shell wiring in `ArgumentGameSurface.tsx:309-374` (`handleActionDockAction`) routes `narrow` / `concede` / `confirm` / `synthesize` / `clarify` / `add_evidence` through `handleAction('reply', targetMessageId)`. This opens the composer but does not auto-seed the preset body. The implementer calls `actionDockToComposerPreset(action, target, null)` and discards the result via `void` at line 322.

**Reasoning for approve-with-follow-up rather than changes-requested:**

1. The design's §"Composer preset contract" specifies the **mapping contract** (model returns the right patch). It does NOT explicitly require the room shell to apply the patch to the composer in SC-004.
2. The design's §"File changes" wording for `ArgumentGameSurface.tsx` is: "wires `selectedTarget` state, threads the dock model build (`buildTimelineNodeActionDockModel`), dispatches `onActionDockAction` to the composer or branch-toggle or Cards-detail surface **as appropriate**." The "as appropriate" wording explicitly allows the room shell to dispatch through the existing composer seam without auto-seeding the body in v1.
3. The design's §"Test plan" item 12 (composer preset mapping) requires the model to return the right patch — which it does. No test category requires the room shell to apply the patch.
4. The design's §"Risks" #3 says the bodies "**will be visible** to every player who taps the corresponding primary action" — this is forward-looking, anticipating eventual room-shell consumption, but does not gate the card.
5. The implementer explicitly flagged the gap as discovery follow-up #9 with a clear name and scope.
6. The cost of fixing it is real plumbing work (the composer seam needs to learn to accept a `MoveDraftPatch` from outside the bubble side-rail), which is appropriately scoped as a separate card.

If the operator disagrees and prefers it shipped in SC-004, the implementer fix is a small follow-up patch: route `narrow` / `confirm` / `synthesize` through a new room-shell helper that threads the `MoveDraftPatch` into the composer state.

## Suggestions (non-blocking)

1. **Composer wiring** — When ST-002 / the composer-preset-aware Cards-detail card lands, wire `actionDockToComposerPreset` results into the composer body for `narrow` / `confirm` / `synthesize`. The model side is already correct; the cost is small. The implementer's discovery follow-up #9 captures this.
2. **`collapsed_stub_must_expand` reason** — The design's `TimelineNodeActionDockDisabledReason` union (§Data model line 170-179) lists 9 reasons; the implementation adds a 10th (`collapsed_stub_must_expand`) to support collapsed-stub gating. This is consistent with design intent (§"Branch / cluster selection" describes the behaviour) but the union is implicitly expanded. The implementation is correct; consider an addendum line in the design doc on the next pass.
3. **`reduceMotion` unused** — `TimelineNodeActionDock.tsx:80` deliberately `void reduceMotion`s the state to silence the unused-var lint. The design's §"Reduce-motion" wording promises a snap-in behaviour, but v1 ships with no slide-up animation anyway. The current "read once, discard" pattern is defensible because there's no animation to disable; consider folding this into discovery follow-up #4.
4. **`isReadModeViewer` unused inside the component** — The component receives `isReadModeViewer` and `void`s it. The model already gates observer actions via `actorRule`, so the prop is currently redundant. Either consume it in the component (e.g. dim the dock root visually for observers) or remove the prop from the component contract.
