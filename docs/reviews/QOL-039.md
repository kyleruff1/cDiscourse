## Summary

QOL-039 introduces room visibility transitions with a tight, doctrine-clean
implementation. The migration adds `debates.visibility`, a one-way trigger,
two SECURITY DEFINER helpers, four RLS policy replacements, and a counts-only
audit table (`room_visibility_changes`). The Edge Function
`record-visibility-transition` atomically performs UPDATE + audit INSERT +
QOL-040 notification dispatch (best-effort, never rolling back the
transition). The pure-TS `roomVisibilityModel.ts` is creator-only per OD-1
with the mod field RESERVED, the flat `RoomVisibilityChangeEvent` shape per
E1.5, and `ROOM_VISIBILITY_COPY` carries every user-facing string. Tests are
exhaustive (+230 tests, +14 suites) with explicit OPS-001 four-class
compliance, doctrine ban-list scans, source-safety scans, migration shape
verification, and confirmation-modal a11y. No QOL-038/QOL-040 files
modified. No reverse transition. No `dropped_observer*` columns.

The room-creator audit policy uses `room_visibility_changes.debate_id` and `d.id = room_visibility_changes.debate_id`; `room_visibility_changes.triggered_by_user_id = auth.uid()`. - **Class 3 (Implicit ordering dependencies):** Statement order is documented in the header (steps 1–10) and verified by `qol039Migration.test.ts` line 261–272: `ALTER TABLE` → trigger function → trigger → 2 helpers → 3 DROP/CREATE POLICY → audit table CREATE → INDEX (×2) → ENABLE RLS → audit policies (×3). `argument_tags` is intentionally NOT replaced (E1.6 — inherited via `EXISTS arguments`).

## Suggestions (non-blocking)

1. **`ROOM_VISIBILITY_COPY` not registered in `ALL_COPY`** — `gameCopy.ts` exports `ALL_COPY` as a convenience grouping for some other test surfaces. The design §4.5 suggested adding `ROOM_VISIBILITY_COPY` to it but the implementer did not. This is harmless: `ROOM_VISIBILITY_COPY` is imported directly by every consumer and is fully ban-list-tested by `roomVisibilityModel.banlist.test.ts`. Worth picking up in a follow-up touch (or in QOL-040.2 if filed) for consistency, but does not block.
2. **Notification dispatch granularity** — the Edge Function returns `'queued'` whenever the cross-function call returns non-2xx, whereas a more granular status (e.g. distinguishing `'not_configured'` from `'queued'`) could be derived by inspecting the response body before draining it. Since the Edge Function deliberately drains-and-discards (per the safety contract — `recordVisibilityTransition.safety.test.ts` line 66–73), the current behavior is correct and intentional. Surfacing the distinction would require widening the API surface; not worth doing pre-product-feedback.
3. **Gallery routing is belt-and-suspenders** — `conversationGalleryModel.ts:1510` routes private cards to `'my_rooms'` even though RLS already withholds them from non-participants. The comment block explains this is intentional belt-and-suspenders. Could be reduced if the participant-derivation seam is tightened later.
