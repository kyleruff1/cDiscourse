## Findings

### Finding 1 — BLOCKING — Lint failure in `__tests__/FistBumpReaction.test.tsx:142`

```
__tests__/FistBumpReaction.test.tsx
  142:16  error  '_m' is assigned a value but never used  @typescript-eslint/no-unused-vars

ESLINT_EXIT=1
```

The repo's `eslint.config.js` line 32 uses `argsIgnorePattern: '^_'` only — it does NOT use `varsIgnorePattern`. The `_` prefix exempts function arguments from the rule but **not** local variable declarations (including `for...of` loop variables). The unused `_m` in the loop at line 142 triggers a hard error; with `--max-warnings 0` set in `package.json`'s `lint` script, this fails the build.

**Note on the test's intent.** The loop at 141–146 iterates `weightMatches` but the assertion inside the loop is constant w.r.t. the iteration (the regex `re` is re-created each iteration, and the `expect(code).not.toMatch(re)` runs against `code` not against `_m`). The whole loop is redundant — the assertion would be equivalent if collapsed to either:

```ts
// (a) Drop the loop — the assertion does not depend on iteration:
if (weightMatches.length > 0) {
  const re = /(?<!font)\bweight\b/gi;
  expect(code).not.toMatch(re);
}

// (b) OR, if the intent is to use `_m`:
for (const m of weightMatches) {
  // assert something about m, e.g. an index probe
  expect(code.indexOf(m)).toBeGreaterThanOrEqual(0);
}
```

The simplest fix to make lint pass without altering test semantics is approach (a). Either is acceptable.

**Impact:** Per CLAUDE.md ("Run `npm run typecheck` before every commit" and the test-discipline skill's "`npm run lint` exits 0"), this blocks the card. The fix is one line in a test file; no production code change is needed.

### Finding 2 — non-blocking (note for follow-up)

The submit-argument soft-rollback marks `arguments.status='deleted'` on failure. This is the documented atomic-approximation per design §5.6, but it does leave a `deleted`-status row in the DB on each failure path. A future PG-function or a single SQL `WITH` chain could give a truer atomic insert (and avoid the soft-deleted detritus). For now the doctrine constraint (rows never hard-deleted) is honored and the row is invisible to readers; this is fine to ship as-is.

### Finding 3 — non-blocking (observation)

The doctrine test `qol041-doctrine.test.ts:124-129` asserts an exact string count (38 = 5+5+4+8+7+9). This is brittle to future copy additions — adding one helper line in `CONCESSION_LIST_SECTION_COPY` would silently fail this test. Acceptable as a tripwire (a copy change should force a deliberate update to the count), but worth noting if the copy churns during operator review.
