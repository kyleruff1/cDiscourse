## Summary

EV-003 ships the **obligation axis** of the evidence model: a pure-TS
render-time derivation (`evidenceDebtModel.ts`) that turns a room's argument
rows into a list of `EvidenceDebt` records, plus an `EvidenceDebtChip` status
component and four wiring patches (timeline popover model, popover component,
`ArgumentGameSurface`, conversation gallery model + hook). The implementation
matches the binding §6.2 contract precisely — the opening rule, the resolution
scan per debt kind, the close-condition table, staleness, the roll-ups, the
locked chip-copy matrix, the worst-status ordering, and graceful degradation
when QOL-037 is absent are all present and tested. Doctrine is clean: the model
is advisory (imports nothing from `src/features/pointStanding/` or the
constitution validator, emits no `PointStandingDelta`, never enters a blocking
path), reads no heat/engagement field, is pure (no React, no Supabase, no
`Date.now()` inside the model), and every produced string is ban-list-asserted.
No migration, no Edge Function, no `supabase/` change, no `.env` touch, no AI
call, no service-role, no direct insert into `public.arguments`. Verification
is fully green. The two concerns below are non-blocking: a coverage shortfall
versus the design's stated targets (explained and repo-consistent) and a couple
of doc-text overstatements. Neither affects shipped behaviour.

## Adjudication — the §5 vs §6.2 `unresolved` inconsistency

**The §6.2 reading is correct. The implementation is right to follow it. Not a
defect.**

The design's §5 type doc-comment says `unresolved` is reached by an explicit
decline/evasion *OR* "the debt has been open and contested long enough." The
precise §6.2 contract table has exactly one `unresolved` row: an explicit
decline/evasion detected only via the `source_declined` / `request_evaded` tag
codes. The implementation produces `unresolved` only via `hasDeclineTag`;
long-dormant open debts go `stale`.

§6.2 governs because (a) the design itself labels §6 "stable contract" while §5
is descriptive prose; (b) the "contested long enough" path is unimplementable
as written — it needs either a second time threshold the design never defines
or the free-text NLP §6.2 explicitly bans; (c) routing a dormant debt to both
`stale` and `unresolved` would be a redundant double-classification — the clean
split is `stale` = silence, `unresolved` = explicit refusal, and §5's own type
comment ("Distinct from `stale` ... silence, not refusal") only coheres under
the §6.2 reading. The implementer correctly trimmed the vestigial clause from
the `EvidenceDebtStatus` source doc-comment. The §5 prose in the *design doc*
still carries the stale clause — a one-line design-doc cleanup, noted as
Suggestion 3; it does not block the card.

## Suggestions (non-blocking — implementer may address or defer)

1. **Coverage versus the design's stated targets.** §11 promised "100% line +
   branch" for `evidenceDebtModel.ts` and "≥ 90% line" for `EvidenceDebtChip.tsx`.
   Measured: model 99.36% line / 85.33% branch; chip 50% line.
   - The model's one uncovered line is `evidenceDebtModel.ts:636` — the
     `requestArgumentId` tie-break arm of the `getNodeEvidenceDebtSummary` sort
     comparator (two debts on one node with identical `requestedAt`). Defensive
     code; the same comparator shape is exercised in the room-level sort. A
     single test deriving two same-timestamp debts on one node would close it.
   - The chip's uncovered lines are 84-90 — the JSX render body. The test file
     uses the repo's pure-helper + source-scan pattern and documents why (the
     pinned `react-test-renderer` is held away from `@testing-library`'s peer).
     `ReceiptChip.tsx`, EV-002's already-merged sibling chip, sits at 57.89%
     line with the identical approach — so 50% here is repo-consistent, and the
     genuinely branchy logic (`buildEvidenceDebtChipContainerStyle`, tone→color)
     *is* fully tested. The "≥ 90%" target was superseded by repo reality.
   - Recommendation: accept as-is, or add the one same-timestamp model test to
     reach 100% branch on the pure model. Not blocking.

2. **`current-status.md` overstates the model coverage.** The new section says
   `evidenceDebtModel.test.ts` has "full line + branch coverage"; measured is
   99.36% / 85.33%. A doc claim, not production code — tighten the wording or
   close the gap (Suggestion 1) so the doc becomes true.

3. **Design-doc §5 cleanup.** `docs/designs/EV-003.md` §5's `unresolved`
   type-doc still carries the unimplementable "open and contested long enough"
   clause that §6.2 supersedes. A one-line edit to the design would remove the
   inconsistency for future readers. (The design doc is out of this card's
   scope to edit; flag for whoever next touches it.)

4. **`receipt` close-condition.** Design §6.2's `receipt` row lists
   `payment_screenshot` (QOL-036) as a discharging artifact kind, but EV-001's
   `EvidenceArtifactKind` has no such value. The implementation correctly omits
   it (checking for a non-existent kind would be dead code). No action needed —
   noted so a future QOL-036 implementer knows to add `payment_screenshot` to
   `artifactDischarges`'s `receipt` arm when that kind lands.
