## Two implementer judgment calls — adjudicated

**1. Reworded lens helper / empty-note strings — ACCEPTED as correct doctrine-conformance,
not a redesign.** The design §4.2's prescribed copy contained five strings with substrings
on the repo's *canonical* ban-list (`_forbiddenLifecycleTokens()`, which `noVerdict`
asserts against): four `emptyNote`s with "right now" (`right` is banned) and
`hot.helper`'s "activity, not correctness" (`correct` is banned). Shipping the design's
literal copy would have failed the card's own ban-list test. The implementer reworded all
five — "right now" → "is waiting on a response" / "at the moment", and "not correctness" →
"not a result". Design intent is fully preserved: the heat disclaimer still names activity
and explicitly denies a verdict; the empty notes still read as plain English. The design
itself is internally inconsistent here — §8's grammar table already prescribes "activity,
not a result" for heat copy — so the implementer chose the wording §8 already mandated.
This is doctrine conformance, correctly handled; no re-design needed.

**2. `TimelineLensNode` augmentation for the missing `lifecycleState` field — ACCEPTED, matches
design §15-Q5.** `ArgumentTimelineMapNode` (verified, lines 532–571) carries no
`lifecycleState` field. The implementer defined `TimelineLensNode = ArgumentTimelineMapNode
& { lifecycleState?: PointLifecycleState | null }` and the timeline lifecycle predicates
read it optionally with a documented fallback (`temperatureBand` / `kindColorFamily` /
`isRoot` / child-edge count) when absent. Design §15-Q5 explicitly directs this: "if not,
the node predicates use the documented fallbacks until LIFE-001 wires it." Exactly
conformant. Related: the implementer added `nodeIdsWithChildren` to `TimelineLensContext`
to supply the "child-edge count" §4.4's `no_rebuttal` table mandates (the §10 interface
block under-specified `TimelineLensContext` to `activePathIds` only) — a well-documented
reconciliation of a design under-specification, not a deviation.

## Suggestions (non-blocking)

1. **§9.2 `ConversationGalleryScreen` wiring is deferred.** The design §9.2 lists
   `ConversationGalleryScreen.tsx` (~+90) and its card renderer (~+40) as "Modified
   files" that adopt the model; the branch ships only the `index.ts` barrel. This is
   *acceptable* — the card brief is explicit that "IX-001 ships NO standalone React shell
   — only the pure `timelineDensityLensModel` module + the small `useDensityLens` hook",
   resolving the design's internal §0/§1/§16-vs-§9.2 tension in favour of the pure core;
   wiring an already-shipped screen is genuinely separable surface work whose deferral
   avoids destabilising the gallery suite for zero model-correctness gain. The implementer
   disclosed it honestly in `docs/core/current-status.md` ("wiring the Density + Lens controls
   into `ConversationGalleryScreen` is a follow-up surface task"). Recommend the operator
   track the gallery-adoption wiring as its own follow-up card (it pairs naturally with
   QOL-033's Go-popout adoption of the same model).
2. **Design §4.4 names a `source_chain` node kind that does not exist.** The §4.4 table
   says `source_chain_pressure` should match "nodes of kind `source_chain`", but
   `TimelineKindColorFamily` (verified, lines 509–516) has no `source_chain` member —
   source-chain nodes fall in the `evidence` color family. The implementer correctly
   matched `kindColorFamily === 'evidence'` and noted it in a code comment. Worth a
   one-line design-doc erratum so a future reader is not confused; no code change needed.
3. **`useDensityLens.test.ts` tests the hook via its pure delegate logic + a source
   scan**, not `react-test-renderer`. The file documents this as the repo's deliberate
   discipline (`botRoomMarker.test.tsx` precedent) and the design §12 only asks for "the
   existing RN-testing setup". Acceptable as-is; flagged only so the operator is aware the
   hook has no runtime-render test.
