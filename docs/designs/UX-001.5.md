## §0 Goal (one paragraph)

UX-001.5 ships a canonical set of twelve visual primitives — chips, chip
strips, overflow chips, badges, badge clusters, focus rings, outlines,
edge highlights, Inspect-section chip strips, group headers, plus pure-TS
accessibility infrastructure — that consolidates the ad-hoc annotation
treatments scattered across the codebase (evidence annotation chips,
metadata ledger output, semantic referee banner, Inspect `flags`
section, future Observations / Allegations) into one coherent visual
language. The card has standalone product value: annotations feel like
one system. It also produces the descriptor contract (`AnnotationChipDescriptor`),
the Inspect `flags` section integration pattern, and the Phase 5 framing
that UX-001.5A's Machine Observations vs User Allegations layer
consumes. Doctrine constraints: no verdict (cdiscourse-doctrine §1),
color is never the only signal (accessibility-targets §2), no AI calls
in the production app (doctrine §7), no schema or backend changes
(brief §"Disallowed"), and UX-001.2's offset caps stay intact (brief
§"Cross-viewport rendering").

### Per-primitive specification

#### 1. `AnnotationChip` (base chip)

- **File:** `src/features/nodeAnnotations/AnnotationChip.tsx`
- **Props:**
  ```ts
  export interface AnnotationChipProps {
    descriptor: AnnotationChipDescriptor;
    onPress?: (descriptor: AnnotationChipDescriptor) => void;
    size?: 'compact' | 'standard'; // default 'compact' on phone, 'standard' on tablet/wide
    testID?: string;
  }
  ```
- **Render output (summary):**
  - `<View>` or `<Pressable>` (Pressable when `onPress` provided).
  - Pill: `borderRadius: RADIUS.pill`, paddingH 10/12, paddingV 4/6 by `size`.
  - Background, border, foreground from `annotationKindTokens.resolveChipColors(kind, iconHint)`.
  - Optional `<Text>` icon glyph (geometric: `★ ⓘ ⚑ ⏱ ▢ ↻` — non-color carrier of meaning).
  - `<Text style={{ fontSize: 11/12 }} numberOfLines={1}>` with the label.
  - Optional trailing dismiss glyph `×` (when `descriptor.dismissible === true`, *deferred to UX-001.5A unless surfaced*; v1 always omits).
- **Tap target / accessibility:**
  - `hitSlop: { top: 12, bottom: 12, left: 12, right: 12 }` to reach 44×44 when visual is ~28px.
  - `accessibilityRole = onPress ? 'button' : 'text'`.
  - `accessibilityLabel` from `buildAnnotationAriaLabel(descriptor)` or `descriptor.ariaLabel` override.
  - `accessibilityState = { disabled: !onPress }` when role is `'button'`.

#### 2. `AnnotationChipStrip` (horizontal scrollable container)

- **File:** `src/features/nodeAnnotations/AnnotationChipStrip.tsx`
- **Props:**
  ```ts
  export interface AnnotationChipStripProps {
    descriptors: ReadonlyArray<AnnotationChipDescriptor>;
    maxVisible?: number; // band default; phone 3 / tablet 4 / wide 6
    onChipPress?: (descriptor: AnnotationChipDescriptor) => void;
    onOverflowPress?: () => void; // expands; v1 toggles internal expanded state
    band?: 'phone' | 'tablet' | 'wide'; // resolved from useHeaderBreakpoint
    testID?: string;
  }
  ```
- **Render output (summary):**
  - When `descriptors.length <= maxVisible`: render all chips in a horizontal `<View style={{ flexDirection: 'row', flexWrap: 'wrap', gap: SPACING.xs }}>`.
  - When `descriptors.length > maxVisible`: render first `maxVisible - 1` chips + one `<AnnotationOverflowChip count={overflowCount} onPress={...} />`.
  - When the overflow chip is pressed AND `onOverflowPress` is omitted, the strip uses internal `expanded` state to render all chips (band cap still applies for wrap layout).
  - Native: no horizontal `ScrollView` (chips wrap with `flexWrap`); web: same wrapping (predictable across viewports).
- **Accessibility:**
  - Container `accessibilityRole="list"` (RN treats unknown roles as `none` on native — safe).
  - Wraps children inside an `AnnotationFocusBoundary` element on web (`Platform.OS === 'web'`) for arrow-key navigation.

#### 11. `AnnotationAriaLabel` (pure-TS label builder)

- **File:** `src/features/nodeAnnotations/annotationAriaLabel.ts`
- **API:**
  ```ts
  export function buildAnnotationAriaLabel(descriptor: AnnotationChipDescriptor): string;
  export function buildAnnotationAriaLabelForCluster(
    badges: ReadonlyArray<{ iconHint?: ...; kind?: ...; ariaLabel: string }>,
  ): string;
  export function buildAnnotationStripAriaLabel(
    descriptors: ReadonlyArray<AnnotationChipDescriptor>,
    sectionId: string,
  ): string;
  ```
- **Behavior (pure TS, deterministic):**
  - `buildAnnotationAriaLabel(d)`: returns `d.ariaLabel` when set; else composes from `d.kind` + `d.label` + optional `d.tooltip`:
    - Example: `"Annotation: Source gap. Needs a primary source to support the claim."`
    - Pattern: `${kindToWord(d.kind)}: ${d.label}.${d.tooltip ? ` ${d.tooltip}.` : ''}`.
  - `kindToWord({state,context,lifecycle,evidence,flag,semantic})`:
    - `state` → `'State'`
    - `context` → `'Context'`
    - `lifecycle` → `'Lifecycle'`
    - `evidence` → `'Evidence'`
    - `flag` → `'Flag'`
    - `semantic` → `'Note'`
    - `undefined` → `'Annotation'`
  - Cluster: `"3 annotations: Source gap, Side branch suggested, Needs source"`.
  - Strip: `"Semantic flags: 4 items"`.

#### 12. `AnnotationFocusBoundary` (focus management for chip strips)

- **File:** `src/features/nodeAnnotations/annotationFocusBoundary.ts` + a thin RN wrapper component.
- **API:**
  ```ts
  // Pure-TS model — keyboard-event interpreter:
  export type FocusBoundaryKeyEffect =
    | { type: 'focus_next' }
    | { type: 'focus_prev' }
    | { type: 'focus_first' }
    | { type: 'focus_last' }
    | { type: 'exit_boundary' }
    | { type: 'noop' };

  export function resolveFocusBoundaryKeyEffect(
    key: string,
    modifiers: { shift?: boolean; alt?: boolean; ctrl?: boolean; meta?: boolean },
  ): FocusBoundaryKeyEffect;
  ```

  - `ArrowLeft` / `ArrowUp` → `focus_prev`

## §3 — `AnnotationChipDescriptor` (Q3)

### Pure-TS type

```ts
// src/features/nodeAnnotations/annotationChipDescriptor.ts

/**
 * UX-001.5 — Source-neutral annotation chip descriptor.
 *
 * The primitive layer's contract for one chip. Source-neutral on purpose:
 * UX-001.5A adds `source` + `category` for the Observations / Allegations
 * presentation. UX-001.5 primitives MUST accept `source` and `category`
 * without rendering changes (forward-compatible).
 *
 * Pure TS. No React. No Supabase. No network.
 */
export interface AnnotationChipDescriptor {
  /** Stable key for React diffing. */
  id: string;

  /** Plain-language label. NO raw internal codes. Routed through
   *  `gameCopy.toPlainLanguage` by emitters when sourced from internal
   *  classifier IDs / lifecycle codes / manual-tag codes. */
  label: string;

  /** Kind family — drives token color via `annotationKindTokens`. */
  kind?: 'state' | 'context' | 'lifecycle' | 'evidence' | 'flag' | 'semantic';

  /** Icon hint glyph + tone — drives the optional leading glyph + secondary
   *  color shade. Non-color carrier of meaning. */
  iconHint?: 'info' | 'warn' | 'check' | 'time' | 'evidence' | 'flag' | 'cluster';

  /** Plain-language explanation; surfaced on tap (composer) or appended to
   *  the screen-reader label. NO raw internal codes. Optional. */
  tooltip?: string;

  /** Explicit screen-reader label override. When absent the builder
   *  `buildAnnotationAriaLabel` composes one from kind + label + tooltip. */
  ariaLabel?: string;

  // ── UX-001.5A forward-compatibility slots ─────────────────────────
  //
  // The UX-001.5A roadmap (docs/roadmap/UX-001.5A-...) declares two
  // top-level taxonomy values and a source enum. UX-001.5 primitives
  // accept these fields without altering rendering — UX-001.5A's renderer
  // reads them to group / decorate chips.

  /** UX-001.5A — top-level taxonomy. `'machine'` = Observation;
   *  `'user'` = Allegation. Accepted but not rendered differently by
   *  UX-001.5 primitives. */
  source?: 'machine' | 'user';

  /** UX-001.5A — source provenance. The full taxonomy
   *  (`MachineObservationSource` | `UserAllegationSource`) is defined in
   *  UX-001.5A; UX-001.5 accepts any string for forward compatibility. */
  category?: string;
}
```

### Forward-compatibility verification

Per `docs/roadmap/UX-001.5A-node-labeling-observations-allegations.md` §"Required label taxonomy":

| UX-001.5A field on `NodeLabelMark` | UX-001.5's `AnnotationChipDescriptor` slot |
|---|---|
| `id` | `id` (same name, same purpose) |
| `kind: NodeLabelKind` (`'machine_observation' \| 'user_allegation'`) | `source: 'machine' \| 'user'` (UX-001.5A normalizes the `_observation` / `_allegation` suffixes — confirmed below) |
| `source: MachineObservationSource \| UserAllegationSource` | `category` (string accepts the full union; UX-001.5A will type-narrow) |
| `rawKey` | (UX-001.5A internal; not rendered) — NOT part of the descriptor |
| `label` | `label` |
| `shortLabel` | (could be carried in a future `compactLabel?: string` field; not needed for UX-001.5) |
| `description` | `tooltip` |
| `defaultSurface: NodeLabelSurface` | (UX-001.5A internal routing decision; descriptor is surface-neutral) |
| `priority` | (UX-001.5A presentation-model concern; descriptor stays neutral) |
| `actionable: boolean` | (UX-001.5A presentation-model concern) |
| `confidence?` | (UX-001.5A presentation-model concern) |

**Source field normalization (load-bearing decision):** UX-001.5A's `NodeLabelKind` uses the suffixed strings `'machine_observation'` and `'user_allegation'`; UX-001.5's `source` field uses the un-suffixed `'machine'` and `'user'`. UX-001.5A's adapter will map `kind → source` at the boundary (a one-liner: `descriptor.source = kind === 'machine_observation' ? 'machine' : 'user'`). This choice keeps the descriptor field short for the v1 source-neutral primitive and lets UX-001.5A's presentation model own the suffixed enum (where it carries semantic meaning).

**Forward compatibility verdict:** ✅ Compatible. UX-001.5A's `NodeLabelMark` is a superset of `AnnotationChipDescriptor`; UX-001.5A will compose primitives by adapting `NodeLabelMark → AnnotationChipDescriptor`.

## §10 — Color, token, verdict-clean (Q10)

### `kind` → token mapping (`annotationKindTokens.ts`)

```ts
import { ARGUMENT, STATUS, SURFACE_TOKENS, CONTROL } from '../../lib/designTokens';
import type { AnnotationChipDescriptor } from './annotationChipDescriptor';

export interface ChipColors {
  bg: string;
  fg: string;
  borderColor?: string;
}

const KIND_TOKEN_TABLE: Record<NonNullable<AnnotationChipDescriptor['kind']>, ChipColors> = {
  state:     { bg: SURFACE_TOKENS.raised,   fg: SURFACE_TOKENS.textPrimary,   borderColor: SURFACE_TOKENS.border },
  context:   { bg: STATUS.neutral.bg,       fg: STATUS.neutral.fg,            borderColor: SURFACE_TOKENS.border },
  lifecycle: { bg: SURFACE_TOKENS.elevated, fg: SURFACE_TOKENS.textSecondary, borderColor: SURFACE_TOKENS.border },
  evidence:  { bg: ARGUMENT.evidence.bg,    fg: ARGUMENT.evidence.fg,         borderColor: SURFACE_TOKENS.border },
  flag:      { bg: SURFACE_TOKENS.elevated, fg: SURFACE_TOKENS.textPrimary,   borderColor: STATUS.warning.bg },
  semantic:  { bg: SURFACE_TOKENS.elevated, fg: SURFACE_TOKENS.textSecondary, borderColor: SURFACE_TOKENS.border },
};

export function resolveChipColors(
  kind?: AnnotationChipDescriptor['kind'],
  iconHint?: AnnotationChipDescriptor['iconHint'],
): ChipColors {
  // Default kind is 'context' (neutral, informative).
  const base = KIND_TOKEN_TABLE[kind ?? 'context'];

  // iconHint may shift the border (the glyph carries meaning regardless;
  // border shade is supplementary).
  if (iconHint === 'warn') return { ...base, borderColor: STATUS.warning.bg };
  if (iconHint === 'check') return { ...base, borderColor: STATUS.success.bg };
  return base;
}
```

The chip background palette is grayscale + neutral indigo/orange/cobalt — informational, not evaluative.
- **Color is informative but never the only meaning.** Every chip carries a `label` (plain text); a label-only render still distinguishes kinds. The `iconHint` glyph (`★ ⓘ ⚑ ⏱ ▢ ↻`) is a secondary non-color carrier. Tested by §11 color-independence.

## §11 — Accessibility (Q11)

### Screen-reader announcements

Every chip carries one of:
- An explicit `descriptor.ariaLabel` (UX-001.5A may supply a category-aware label).
- A composed label from `buildAnnotationAriaLabel(descriptor)` (§2 #11).

Every badge carries a required `ariaLabel` prop (no default — defensive, since badges have no visible text).

Every overflow chip announces `{count} more annotation{plural}`.

### Focus management

`AnnotationFocusBoundary` is the focus-management contract:
- **Web (`Platform.OS === 'web'`):** strip wrapper is `tabIndex={0}`; on focus, arrow keys move focus between chip children. Esc exits the strip and returns focus to the prior tabstop. Tab still advances out of the strip (per accessibility-targets §"Keyboard navigation").
- **Native:** RN's built-in `accessibilityRole="button"` per chip is the rotor target. No custom keydown.

The pure-TS `resolveFocusBoundaryKeyEffect` (§2 #12) is unit-tested independently; the wrapper component is tested for web-only `onKeyDown` attachment.

### Color independence

Each primitive has a non-color signal:
- Chip: text label (always present), optional iconHint glyph, border shade (border presence is a non-color signal — width 1).
- Badge: ariaLabel (always present), optional inner glyph when diameter ≥ 10.
- Overflow chip: `+N` text (always present).
- Focus ring: 2px border (non-color geometric signal).
- Outline: border + opacity changes (geometric + alpha, not pure color).
- Edge highlight: line thickness (geometric).

Test `__tests__/uxOneOneFiveColorIndependence.test.tsx` renders each primitive with token colors replaced by a single neutral gray (via a mock that returns one color for `resolveChipColors`); asserts that:
- `accessibilityLabel` is present and non-empty.
- `<Text>` content is present for chips, badges (when diameter ≥ 10), and overflow.
- For badges with `diameter < 10`, the `accessibilityLabel` carries the meaning.

### 44×44 tap targets

## §16 — Risks

1. **EvidenceAnnotationChip refactor scope creep.** The chip has tone mapping, depth indicator, evidence-specific affordances. The follow-up disposition is the safe call; the risk is the operator decides during review that the refactor should be in-scope. Mitigation: §7 documents the L-effort estimate and the follow-up card name explicitly.

2. **Cross-viewport sizing edge cases.** Three reference viewports (390 / 1024 / 1366 / 1920) per UX-001.2 + UX-001.3. The band sizing table is fixed; risk is a non-canonical viewport (e.g. tablet portrait at 768) lands ambiguously. Mitigation: the band resolver via `useHeaderBreakpoint` is already band-aware — 768dp resolves to `'tablet'` cleanly.

3. **Color-independence verification methodology.** The grayscale mock replaces `resolveChipColors` with a single color. Risk: a test passes because the assertion only checks `accessibilityLabel`, not visual distinguishability. Mitigation: the test also asserts `<Text>` content presence per primitive (label + glyph) — the visual signal carriers.

4. **Forward compatibility regression (UX-001.5A breaks UX-001.5 tests).** UX-001.5A adds source-aware presentation; if UX-001.5A's renderer changes affect `AnnotationChip` rendering (e.g. group dividers), UX-001.5 snapshot tests could fail. Mitigation: UX-001.5 explicitly tests that `source` + `category` populated descriptors render identically to source-neutral ones (the descriptor is source-neutral *at the primitive layer*; UX-001.5A's grouping is wrapper-only).

5. **Semantic referee banner integration affecting composer regression.** The bounded prop addition to `RefereeBannerView` could be misread as a composer change. Mitigation: the banner file is the *banner*, not the composer; the composer (`ArgumentComposer.tsx`, `OneBox.tsx`, etc.) has zero diff (verified by §13 test). The `observationChips` prop is optional; current call sites pass `undefined` → identical behavior.

6. **Inspect `flags` section: section title rename.** UX-001.4's framing referenced `INSPECT_SECTION_TITLE.flags = 'Semantic flags'`. The brief envisions a "richer chip surface, while preserving the read-only semantics" — the title can stay `'Semantic flags'` (compatible) OR UX-001.5A may rename it to `'Notes'` later. UX-001.5 does NOT rename. Mitigation: the title constant is read-only by UX-001.5.

7. **Sidecar chip ↔ descriptor adapter drift.** The `toAnnotationChipDescriptor` adapter sits in `nodeAnnotations/` but reads `SidecarSemanticFlagChip` from `arguments/`. Risk: a sidecar refactor breaks the adapter. Mitigation: the adapter has its own test file; sidecar shape changes will surface.
