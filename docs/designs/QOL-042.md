## 1. Goal & scope

Let a new argument room **reference an earlier settled room** as context. The
reference is a one-directional, immutable **link record** between two rooms:

- The new (referencing) room surfaces a **context chip** carrying the prior
  room's title.
- The link carries an **access check**: a private prior room's *content* opens
  only for users who are authorized on that original room. An unauthorized
  viewer sees the prior room's **title only** — never its body, nodes, or
  evidence.
- Relevant **resolved tangent nodes** from the prior room can surface as
  context.
- The linked prior room is **never re-opened or mutated** by the link. A locked
  room stays immutable; the link is read-only context, not a re-entry.

In scope:

- The `argument_room_links` link record (data model).
- The RLS-enforced access check (not UI-only).
- The context-chip content model (`linkedPriorArgumentModel`) and its two view
  states (authorized vs. unauthorized).
- The resolved-tangent surfacing model.
- The create-link path: who may create a link, from where, through what.
- Integration with the QOL-030 one-box `root_claim` setup box and the QOL-032
  Inspect popout.

Out of scope (see §13).

## 3. Source storyboards / analysis

QOL-042 is driven directly by two storyboard frames and the cross-room taxonomy
entry:

- **Scenario 1 — Roommates / dishes (public), Step 21 "Later reuse."** Weeks
  later the same two users start a *new* argument about the door being unlocked.
  In the new room they reference the prior argument by ID. The new room shows
  *"Linked prior argument: Dishes after the camping week"* and can surface the
  relevant resolved **tangent nodes** (the door-unlocked tangent from Steps
  17–19) as context. The old locked room "remains immutable and is used as
  context, not re-opened."
- **Scenario 2 — Band / rent (private, evidence), Step 14 "Future use."** Next
  month a rent question arises again. Either bandmate references the prior
  *private* room by ID. Because the prior room is private, "only the same
  authorized participants can view its details." The new room shows a **safe
  reference**: *"Linked prior private argument: March practice-room rent"* — the
  title shows, the content does **not** open for anyone not authorized on the
  original room. The link "carries an access check: the linked content is
  viewable only by users authorized on the original room."
- **`interaction-taxonomy.md` § Cross-room — Linked prior argument.** Defines
  the user-facing copy ("Linked prior argument" public / "Linked prior private
  argument" private), the internal field ("a link record between two rooms;
  carries an access check"), the visual ("a context chip on the new room's
  timeline"), and the permission ("Anyone may create a link; viewing the linked
  content respects the prior room's visibility").
- **`missing-capabilities-and-issues.md` § QOL-042.** Classification: *Missing*.
  Kind of work: "Room-to-room link with access check." Acceptance: title shows
  for all; content access-checked; resolved tangent nodes surface as context;
  locked room never mutated.
- **`2026-05-21-roadmap-collision-supersession-analysis.md`.** Confirms
  QOL-042 is the renumbered QOL-029, a new card with no supersession in or out;
  the one-box cards QOL-030…033 are the integration target (§4 conceptual
  layer), and QOL-042's chip is consumed by that surface, not a bespoke screen.

## 4. Problem statement

CDiscourse arguments do not exist in isolation — the same people argue about
related things over time. Today every room is a closed island: a settled room
can be read (if visibility allows) but cannot be **cited** from a new room.
Step 21 / Step 14 both need a new room to say "this builds on / relates to that
earlier settled argument" without:

1. **Re-opening the prior room.** A locked room is immutable doctrine
   (`interaction-taxonomy.md` § Locked room; CLAUDE.md soft-delete + immutable
   bodies). A link must be pure read-context — it adds a row to a *new* table,
   never a row or mutation to the prior room.
2. **Leaking private content.** Scenario 2's prior room is private. A naive
   "store the prior room id and render its title + nodes" leaks the entire
   private argument to anyone who can read the new room. The access check must
   be **RLS-enforced server-side**: even a hand-crafted REST call from an
   unauthorized user must get the title only, never the nodes.
3. **Confusing context with participation.** The viewer of the new room is not
   thereby a participant of the prior room. The link grants *read context*
   scoped to the prior room's own visibility rules — it never enrolls anyone,
   never grants posting rights, never changes a seat.

The design problem is therefore: **a minimal, immutable, one-directional link
record whose content-disclosure is gated by the prior room's existing RLS, with
a UI that degrades to title-only without the implementer having to special-case
it.**

## 5. Data model & file changes

### 5.1 The link record — `public.argument_room_links`

A new table. One row = one reference from a *source* (new) room to a *target*
(prior, settled) room.

```sql
CREATE TABLE public.argument_room_links (
  id                uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  source_debate_id  uuid        NOT NULL REFERENCES public.debates(id) ON DELETE CASCADE,
  target_debate_id  uuid        NOT NULL REFERENCES public.debates(id) ON DELETE RESTRICT,
  created_by        uuid        NOT NULL REFERENCES public.profiles(id) ON DELETE RESTRICT,
  note              text        NOT NULL DEFAULT '' CHECK (char_length(note) <= 280),
  is_removed        boolean     NOT NULL DEFAULT false,
  created_at        timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT no_self_link CHECK (source_debate_id <> target_debate_id),
  CONSTRAINT one_link_per_pair UNIQUE (source_debate_id, target_debate_id)
);
```

Design notes:

- **`source_debate_id`** — the *new* room that carries the chip.
  `ON DELETE CASCADE`: if the new room is ever hard-removed, its links go too.
- **`target_debate_id`** — the *prior* settled room. `ON DELETE RESTRICT`: a
  prior room that is referenced cannot be hard-removed out from under a link.
  (Rooms are not user-deletable today; this is defensive.)
- **`note`** — optional, ≤ 280 chars, the link author's one-line reason
  ("Same door-unlocked issue as before"). Author-supplied free text; held to the
  doctrine ban-list at the composer (client validation) and never rendered as a
  verdict.
- **`is_removed`** — soft-remove only, mirroring the `arguments` soft-delete
  doctrine. A link is **never hard-deleted** by the client; removing a chip sets
  `is_removed = true`. The prior room is untouched either way.
- **`one_link_per_pair`** — a new room links a given prior room at most once;
  re-linking is idempotent (the create path upserts / no-ops on conflict).
- **No `status`, no `updated_at`** — the link is immutable except for the
  `is_removed` flag. There is no edit path for `target_debate_id` or
  `note` after insert (changing the cited room = remove + create a new link).
- **No truth field, no score field, no relationship-type enum.** The link is
  context, not a verdict. It does not say the prior room "won", "proved", or
  "supports" anything. (`relationship_type` was considered and rejected — see
  §14.)

`updated_at` is intentionally omitted; the `debates_set_updated_at` trigger is
**not** attached to this table.

### 5.2 The access check — there is no second table

The access check is **not stored data** — it is **computed by RLS at read
time** against the prior room's existing visibility + participant rules. This is
the core design decision: the link record holds *no* copy of the prior room's
content and *no* materialized authorization list. Every read re-derives access
from `debates` + `debate_participants` live. That means:

- A participant added to the prior room *after* the link was created
  immediately gains content access (no stale snapshot).
- A prior room transitioning public → private (QOL-039) immediately revokes
  content access for non-participants — the link cannot out-live the revocation.

See §7 for the exact policy text.

### 5.3 Indexes

```sql
CREATE INDEX idx_arg_room_links_source ON public.argument_room_links(source_debate_id)
  WHERE is_removed = false;
CREATE INDEX idx_arg_room_links_target ON public.argument_room_links(target_debate_id)
  WHERE is_removed = false;
```

Source index: the timeline loads "links on this room" by `source_debate_id`.
Target index: the back-reference count ("rooms that cite this one") and the
`ON DELETE RESTRICT` check.

## 6. UI states

### 6.1 Where the chip lives

The context chip is **not a node** on the timeline — it is a **room-level
context affordance** rendered in the timeline header / context row of the *new*
(source) room, above the horizontal rail. It is the new room saying "this
argument has prior context", not a move within the argument. A room may carry
**more than one** link chip (a horizontal, wrapping chip row); the common case
is one.

### 6.2 Context chip — three view states

The chip is produced by `buildLinkedPriorArgumentChip`; its `LinkAccessState`
drives which of three states renders.

**State A — `authorized` (viewer may see prior content).**
The viewer is a participant of the prior room, or the prior room is public, or
the viewer is a mod/admin.

```
┌───────────────────────────────────────────────┐
│ ⤴  Linked prior argument                       │
│    "Dishes after the camping week"             │
│    Settled · 6 moves · 1 resolved tangent      │
│    [ Open prior argument ]  [ View context ]   │
└───────────────────────────────────────────────┘
```

- Title is a live read of the prior room's title.
- Sub-line: settled state + a coarse move count + resolved-tangent count.
  Counts are activity facts, never standing/score.
- `Open prior argument` → opens the prior room **read-only** (it is locked).
- `View context` → opens the QOL-032 Inspect popout's "linked prior argument"
  section showing the relevant resolved tangent nodes (§6.4).
- Optional `note` from the link author renders as a single muted line beneath
  the title if present.

**State B — `title_only` (viewer NOT authorized; prior room is private).**

```
┌───────────────────────────────────────────────┐
│ ⤴  Linked prior private argument               │
│    "March practice-room rent"                  │
│    🔒 Private — only its participants can open  │
│    it. You can see the title here as context.  │
│    [ Open prior argument ]  ← disabled         │
└───────────────────────────────────────────────┘
```

- **Only the title** renders. No move count, no tangent count, no body, no
  evidence, no participant names.
- A neutral lock line explains *why* — never shaming, never "access denied" as
  an error. The viewer is given context, not a wall.
- `Open prior argument` is **disabled** with a one-line reason
  (`accessibilityState={{ disabled: true }}`), never a silent omission — this is
  the QOL-031 "disabled entries show a reason" doctrine.
- `View context` is **absent** (not disabled-then-empty) — there is nothing to
  show.

> The title itself is treated as low-sensitivity metadata that the link author
> chose to surface. See §10 and §14 (open question) for the title-disclosure
> rationale and its bound.

**State C — `unavailable` (the prior room or link no longer resolvable).**
The link's `target_debate_id` row cannot be read at all — e.g. the prior room
was archived out of readability, or the link was created against a room that
later became fully unreadable, or the link itself is `is_removed = true` but the
client still has it cached.

```
┌───────────────────────────────────────────────┐
│ ⤴  Linked prior argument is no longer available │
└───────────────────────────────────────────────┘
```

- A single neutral line. No title (the title could not be fetched). No actions.
- The chip is collapsible / dismissible from the viewer's local view; the link
  row is not mutated by viewing.

### 6.3 Creating a link — the create path

A link is created in **one** place: the QOL-030 one-box `root_claim` setup box,
when a new room is being composed. The root box already does room setup (title,
visibility, invite — QOL-030 §5.1 F1). QOL-042 adds an **optional** "Reference a
prior argument" affordance to that setup:

1. The author taps "Reference a prior argument" in the root box.
2. A picker lists the author's **own settled (locked) rooms** — rooms the
   author created or participated in, `status = 'locked'`. (The author can only
   link rooms they can already see; this is RLS-natural and avoids any room
   search surface — see §13 out-of-scope.)
3. The author may add an optional one-line `note`.
4. On room creation, after the new room row exists, the client inserts the
   `argument_room_links` row via `createArgumentRoomLink`. The link insert is a
   **separate write** after `submit-argument` has created the root argument —
   the link is not part of the argument payload and `submit-argument` is
   unchanged.

A link may **also** be added to an *existing* (open, not yet locked) room the
viewer participates in, from the room's context row — same picker, same insert.
Creating a link does not require the new room to be settled; only the *target*
must be locked.

> Whether non-participants of an open public room may add a link to it is an
> open question — see §14. The conservative default for this design: only a
> participant of the source room may add a link to it (enforced by the §7.2
> INSERT policy).

### 6.4 Resolved tangent context

Scenario 1 Step 21 wants the door-unlocked **tangent** from the prior room to
surface in the new room. A "resolved tangent node" is an `arguments` row in the
prior room that (a) sits on a tangent branch (`BranchDirection tangent_diagonal`
/ BR-003/BR-004 routing) and (b) is in a resolved lifecycle stage.

- `buildLinkedTangentContext` takes resolved-tangent rows **already fetched
  through the access-checked path** and produces a read-only context list for
  the Inspect popout. It performs **no fetching** and does **no** access
  decision — it is a pure formatter.
- The fetch (`loadPriorRoomContext`) reads `arguments` for the prior room under
  the caller's JWT. RLS on `arguments` (§7.3) returns tangent rows **only** to
  an authorized viewer. For a `title_only` viewer the fetch returns zero
  argument rows, so the tangent context is simply empty — the implementer never
  special-cases it; the access check already did.
- Tangent context is **read-only**. Surfacing a prior tangent node never copies
  it, never re-posts it, never creates a `submit-argument` call. It is rendered,
  with an attribution that it came from the linked prior room, and a tap-through
  to that node in the (locked, read-only) prior room.

### 6.5 Copy (all held to the terminology rules)

| Surface | Copy |
|---|---|
| Chip header (public prior) | "Linked prior argument" |
| Chip header (private prior, authorized) | "Linked prior private argument" |
| Chip header (private prior, title-only) | "Linked prior private argument" |
| Title-only lock line | "Private — only its participants can open it. You can see the title here as context." |
| Disabled "Open" reason | "This prior argument is private. You are not a participant on it." |
| Unavailable state | "Linked prior argument is no longer available." |
| Create affordance | "Reference a prior argument" |
| Picker empty state | "You don't have any settled arguments to reference yet." |
| Inspect section header | "From the linked prior argument" |

Forbidden everywhere on this surface: "game", "debate" (the room is an

"access denied" as an error verdict. "Settled" is the only end-state word.

## 7. Permission / rules implications — RLS-enforced, not UI-only

The card's non-negotiable: *the access check is enforced by RLS, not only by
UI.* The UI states in §6 are a **rendering** of what RLS already decided — a
hand-crafted REST request from an unauthorized user must get exactly the
title-only result.

### 7.1 `argument_room_links` — SELECT

A viewer may read a link row if they can see its **source** room (so the chip
shows on rooms they can already open). The link row itself contains only the
two room ids, the author, and the note — no prior-room content — so SELECT on
the link is gated on the *source* room, not the target.

```sql
CREATE POLICY "argument_room_links: select if source room visible"
ON public.argument_room_links
FOR SELECT
TO authenticated
USING (
  is_removed = false
  AND (
    public.is_debate_open_or_locked(source_debate_id)
    OR public.is_debate_participant(source_debate_id, auth.uid())
    OR is_moderator_or_admin()
  )
);
```

(If QOL-039's private-source-room rule is in place, `is_debate_open_or_locked`
is replaced by the QOL-039 visibility-aware source-readability helper — §11.)

### 7.2 `argument_room_links` — INSERT

Only a participant of the **source** room may add a link to it, and only they
may be `created_by`. The *target* must be a locked room the author can read —
that part is enforced by the trigger in §7.4 (a `WITH CHECK` subquery into
`debates` would risk the recursion the `…0006` migration fixed, so the
locked-and-readable check is a `BEFORE INSERT` trigger that calls the existing
SECURITY DEFINER helpers).

```sql
CREATE POLICY "argument_room_links: insert by source room participant"
ON public.argument_room_links
FOR INSERT
TO authenticated
WITH CHECK (
  created_by = auth.uid()
  AND public.is_debate_participant(source_debate_id, auth.uid())
);
```

### 7.3 `argument_room_links` — UPDATE (soft-remove only)

The only permitted update is the soft-remove. Only the link author or a
mod/admin may set `is_removed = true`. The policy permits the update; a
`BEFORE UPDATE` trigger (§7.4) rejects any change to a column other than
`is_removed`, making the link immutable in every other respect.

```sql
CREATE POLICY "argument_room_links: soft-remove by author or admin"
ON public.argument_room_links
FOR UPDATE
TO authenticated
USING (created_by = auth.uid() OR is_moderator_or_admin())
WITH CHECK (created_by = auth.uid() OR is_moderator_or_admin());
```

No DELETE policy is created — links are never hard-deleted by the client.

### 7.4 Triggers — locked-target gate + immutability

Two `BEFORE` triggers, each a small `plpgsql` function:

- **`link_target_must_be_locked` (BEFORE INSERT).** Rejects the insert unless
  `target_debate_id` resolves to a room with `status = 'locked'` *and* the
  inserting user can read that room (calls `is_debate_open_or_locked` +
  `is_debate_participant` + visibility, mirroring the SELECT logic for
  `debates`). This enforces "a settled prior room only" and "you can only link a
  room you can see" without putting a recursive subquery in a `WITH CHECK`.
- **`link_columns_immutable` (BEFORE UPDATE).** Raises an exception if any of
  `source_debate_id`, `target_debate_id`, `created_by`, `note`, `created_at`
  differs between `OLD` and `NEW`. Only `is_removed` may change.

This is the structural guarantee that **the link can never re-open or mutate a
locked room**: there is no code path — RLS, trigger, or Edge Function — that
writes to `debates` or `arguments` from QOL-042. The link only ever inserts /
soft-updates its own row.

### 7.5 The access check itself — it rides on existing `debates`/`arguments` RLS

QOL-042 adds **no new policy on `debates` or `arguments`.** The access check is
the *existing* RLS doing its job:

- `loadPriorRoomContext` issues a caller-scoped `SELECT id, title, status FROM
  debates WHERE id = <target>`. With QOL-039's visibility model, the `debates`
  SELECT policy returns this row to a non-participant **only if the room is
  public**. For a *private* prior room, a non-participant gets **zero rows** for
  the full select — so the client cannot even read the title that way.
- To make the **title-only** state work for a private prior room, the title is
  surfaced through the **link row**, which the viewer *can* read (§7.1), via a
  denormalized read: see §7.6. The body/nodes are never denormalized.
- The argument rows are read with a second caller-scoped `SELECT … FROM
  arguments WHERE debate_id = <target> AND status = 'posted'`. The existing
  `arguments` SELECT policy ("posted + open-debate, own, or mod/admin") returns
  rows only to an authorized reader. An unauthorized reader gets zero rows. The
  implementer does not branch on access — empty rows *is* the title-only state.

### 7.6 How the title reaches a title-only viewer — minimal, bounded denormalization

A private prior room's `debates` row is not readable by a non-participant, so
the title cannot come from `debates` for the `title_only` state. Two options
were weighed (§14); the chosen design:

**Store the prior room's title on the link row at creation time** — add
`target_title_snapshot text NOT NULL` to `argument_room_links`, populated by the
INSERT (the link author *is* authorized on the target at creation, so they can
read it). The link row is readable by anyone who can see the source room (§7.1),
so the snapshot title is what renders in the `title_only` state.

Consequences, all acceptable and explicit:

- The snapshot is a **point-in-time title**. If the prior room's title is later
  edited, the chip shows the title as it was when linked. This is fine — the
  link is a historical reference; a stale title is a far smaller problem than a
  content leak, and the `authorized` state can re-read the live title from
  `debates` and prefer it when available.
- The snapshot is **only a title** — never body, never nodes, never evidence,
  never participant identity. The denormalization surface is exactly one short
  text field that the link author deliberately chose to cite.
- The `link_columns_immutable` trigger (§7.4) covers `target_title_snapshot`
  too — it cannot be changed after insert.

So the final table adds one column to §5.1:

```sql
  target_title_snapshot text NOT NULL CHECK (char_length(target_title_snapshot) <= 200),
```

`buildLinkedPriorArgumentChip` prefers a live title from `debates` when the
viewer is authorized; falls back to `target_title_snapshot` otherwise.

## 8. Edge cases

| Case | Handling |
|---|---|
| **Target room not locked** (still open/draft) | INSERT rejected by the `link_target_must_be_locked` trigger. The picker (§6.3) only lists locked rooms, so the UI never offers it; the trigger is the server guard. |
| **Self-link** (`source = target`) | Rejected by the `no_self_link` CHECK constraint. |
| **Duplicate link** (same source→target twice) | `one_link_per_pair` UNIQUE constraint; `createArgumentRoomLink` treats the conflict as idempotent success (no-op). |
| **Private prior room, unauthorized viewer** | RLS returns zero `debates`/`arguments` rows for the target; chip renders `title_only` from `target_title_snapshot`. No body, no count, no names leak. |
| **Public prior room** | All authenticated viewers get `authorized` — public content is already world-readable. |
| **Viewer is mod/admin** | `is_moderator_or_admin()` grants `authorized` on the link and on the prior room's content (consistent with existing `debates`/`arguments` policies). |
| **Prior room becomes private after the link is created** (QOL-039 transition) | No stale snapshot of access — the access check re-derives on every read. A non-participant who could open the prior room yesterday gets `title_only` today, automatically. |
| **A participant is added to the prior room after the link** | They immediately gain `authorized` content access — the check is live against `debate_participants`. |
| **Prior room hard-removed** | `ON DELETE RESTRICT` on `target_debate_id` blocks it while a link exists. (Rooms are not user-deletable today; defensive.) |
| **New (source) room hard-removed** | `ON DELETE CASCADE` removes its links. |
| **Link soft-removed** (`is_removed = true`) | Excluded by every SELECT policy and both indexes; the chip disappears on next load. A stale client cache renders `unavailable` (§6.2 State C). |
| **Resolved tangent context for a `title_only` viewer** | `loadPriorRoomContext` returns zero argument rows under RLS; `buildLinkedTangentContext` produces an empty list; the Inspect "View context" action is absent. No special-casing. |
| **Prior room has no resolved tangents** | `buildLinkedTangentContext` returns an empty list; the chip still shows the title + Open action; "View context" shows "No tangents from the linked argument." |
| **`note` contains a doctrine ban-list word** | Client composer validation flags it before insert (the same ban-list scan used elsewhere); the server stores whatever passes — `note` is never rendered as a verdict regardless. |
| **Multiple links on one room** | The chip row wraps; deterministic order = `created_at ASC`. |
| **Offline / network failure on `loadPriorRoomContext`** | Chip renders from the cached link row (`target_title_snapshot`) in a degraded `title_only`-shaped state with a quiet "Couldn't refresh" note; never blocks the new room. |
| **Concurrent link create + prior-room transition** | The trigger evaluates `status`/visibility at insert time; if the room is locked at that instant the link is valid. A later transition only affects content disclosure, never the link's existence. |
| **QOL-039 not yet shipped** (no `visibility` column) | §11 — the access check degrades to "all `open`/`locked` rooms are content-readable"; the title-only state is unreachable but harmless; the link table + RLS + triggers all still apply. |

## 10. Doctrine & safety self-check

- **No truth labels.** The link is context, not a verdict. It carries no
  relationship type, no score, no "supports/refutes" field; the chip never says
  the prior argument "won", "proved", or "was correct". (§5.1, §14.)
- **Score never blocks; the link never scores.** QOL-042 produces no
  point-standing delta, touches no strength band, and is not an input to any
  scoring path. The move counts on the chip are *activity* facts, not standing.
- **Heat / popularity untouched.** The link is not a signal; a heavily-cited
  prior room is not thereby "important" or "correct". No engagement input.
- **Locked room immutability — structurally enforced.** There is no code path in
  QOL-042 that writes to `debates` or `arguments`. The link only writes its own
  `argument_room_links` row. The `link_columns_immutable` trigger and the
  absence of any `debates`/`arguments` mutation prove the prior room stays
  immutable. (§7.4, §7.5.)
- **Access check is server-enforced, not UI-only.** The title-only result for an
  unauthorized viewer is produced by RLS on `debates`/`arguments` — a
  hand-crafted REST call cannot retrieve the prior room's body or nodes. The UI
  states in §6 only *render* the RLS outcome. (§7.5.)
- **Minimal disclosure.** The single denormalized field is the prior room's
  *title* (≤ 200 chars), which the link author deliberately chose to cite. No
  body, no node text, no evidence, no participant identity is ever copied onto
  the link. (§7.6 — and §14 records the title-disclosure as the one residual
  decision for the operator.)
- **RLS always on; no service-role in client.** `argument_room_links` has RLS
  enabled with four explicit policies. All client reads/writes are caller-scoped
  under the user's JWT. No Edge Function is added; no service-role key is used.
- **No direct insert into `public.arguments`.** The link is not an argument; the
  `submit-argument` write path is untouched. The link insert is a separate write
  to a separate table.
- **No AI.** No Anthropic / xAI / OpenAI / X API call anywhere in QOL-042. The
  models are deterministic pure functions.
- **Soft-delete doctrine.** Links are soft-removed (`is_removed = true`), never
  hard-deleted by the client; no DELETE policy exists.
- **Plain language.** No internal codes in any user-facing string; "argument"
  not "debate", never "game"; "settled", never "case closed". (§6.5.)
- **Migration discipline.** One new, append-only, self-contained migration; no
  existing migration edited; RLS never disabled.

## 11. Dependencies

- **Hard prerequisite — none.** QOL-042 can ship against today's schema. The
  `debates` and `debate_participants` tables, the `status = 'locked'` state, and
  the `is_debate_participant` / `is_debate_open_or_locked` helpers all exist.
- **Soft dependency — QOL-039 (public/private visibility).** The `title_only`
  state is only *reachable* once a room can be private. Until QOL-039 ships,
  every `open`/`locked` room is content-readable by all authenticated users, so
  every link resolves to `authorized` and State B is dead-but-harmless code.
  QOL-042's table, RLS, triggers, and chip model are all written
  visibility-aware from day one:
  - If `debates.visibility` does not yet exist, the §7 policies and the
    `link_target_must_be_locked` trigger use `is_debate_open_or_locked` alone
    (today's behaviour). When QOL-039 lands, a follow-up one-line migration
    swaps in the visibility-aware source-readability helper. The QOL-042
    migration is written so this swap is a clean addition, not an edit.
  - **Recommended sequencing:** ship QOL-039 first so State B is live on
    arrival. If QOL-042 ships first, note in `docs/current-status.md` that the
    title-only path is latent until QOL-039.
- **Soft dependency — QOL-030 (one-box `root_claim`).** The create-link
  affordance is hosted in the root setup box. If QOL-030 has not landed, the
  link can be created from a minimal standalone affordance on the room context
  row instead (the API + model are independent of the box); the root-box
  integration is then a QOL-030 follow-up.
- **Soft dependency — QOL-032 (Inspect popout).** "View context" (resolved
  tangent surfacing) renders in the Inspect popout. Without QOL-032 the chip
  still works (title + Open); the tangent section is a QOL-032 follow-up.
- **Reads existing:** `is_debate_participant`, `is_debate_open_or_locked`,
  `is_moderator_or_admin` (migration `…0006` / RLS doc); BR-003/BR-004 tangent
  routing for "what is a resolved tangent node".
- **Blocks:** nothing downstream depends on QOL-042.

## 12. Supersedes / superseded-by

- **Supersedes:** nothing. QOL-042 is a new capability with no prior card.
- **Superseded by:** nothing. The
  `2026-05-21-roadmap-collision-supersession-analysis.md` records QOL-042
  (renumbered from QOL-029) as a standalone Rules-UX card; the one-box cards
  QOL-030…033 are an *integration target*, not a supersession — the chip is
  consumed by the one-box surface, and no QOL-042 model is re-housed or
  retired.
- **Conceptual neighbour:** QOL-039 (visibility) defines the private/public
  axis QOL-042's access check branches on; QOL-040 (notifications) explicitly
  does *not* notify on link creation (storyboards: "Notifications: None specific
  to the link") — recorded so a future card does not add a link notification by
  reflex.

## 13. Out of scope

- **Argument search / a global room picker.** The create-link picker lists only
  the author's *own* settled rooms (RLS-natural). No search across all rooms,
  no discovery surface — argument search is a v1 scope guard (CLAUDE.md).
- **Editing a link.** A link's target and note are immutable after insert
  (`link_columns_immutable`). Changing a citation = soft-remove + create new.
- **A relationship-type taxonomy** ("builds-on" / "contradicts" / "supersedes").
  Rejected as verdict-adjacent — see §14.
- **Bidirectional / back-reference UI.** The prior room does not show "rooms
  that cite me". The `idx_arg_room_links_target` index supports a future
  back-reference count, but rendering it is a separate card.
- **Notifications on link creation.** Both storyboards say no link notification
  (QOL-040's lifecycle list omits it).
- **Copying prior content into the new room.** A linked tangent is *referenced
  and rendered read-only*, never copied, never re-posted, never re-submitted.
- **Linking individual nodes** (deep-linking a specific argument node across
  rooms). The link is room-to-room; node-level cross-room references are a
  potential future card.
- **An Edge Function for link creation.** RLS + triggers fully enforce the link
  rules; no privileged write is needed, so no Edge Function is added.
- **Operator deploy of QOL-039.** QOL-042 does not deploy or depend on a
  deployed QOL-039; §11 covers the degraded path.

## 14. Open questions

1. **Title disclosure for a private prior room — confirm the bound.** The design
   surfaces the prior room's *title* to anyone who can see the new room, via the
   `target_title_snapshot` on the link row (§7.6). Both storyboards explicitly
   require this ("the title shows, the content does not open" — Scenario 2 Step
   14). The residual question for the operator: is a private room's *title*
   itself ever sensitive enough that even the title should be gated? The
   storyboards say no; this design follows them. If the answer ever changes, the
   fix is local: drop the snapshot, and the `title_only` state collapses to a
   generic "A private prior argument is linked" with no title. **Recommended: keep
   the title (storyboard-faithful); revisit only if a real privacy case
   surfaces.**
2. **Who may add a link to an open *public* room.** §6.3 / §7.2 restrict link
   creation to a *participant* of the source room. Should an observer of an open
   public room be able to attach a prior-argument link? The conservative default
   here is no (participants only). If product wants observer-contributed
   context, that is a one-line policy widening — flagged, not assumed.
3. **`relationship_type` — confirmed omitted.** A typed relationship
   ("builds-on" / "contradicts") was considered and **rejected**: "contradicts"
   is verdict-adjacent and the link is meant to be neutral context. The `note`
   free-text field covers the human-readable "why" without a typed verdict. This
   is recorded as a deliberate decision, not an oversight — re-open only with a
   doctrine review.
4. **Resolved-tangent selection precision.** §6.4 surfaces *resolved* tangent
   nodes. The exact predicate for "resolved tangent" depends on the BR-003/BR-004
   tangent model and the LIFE-001 lifecycle stage vocabulary. If LIFE-001 is not
   yet built at implementation time, `buildLinkedTangentContext` degrades to "all
   tangent-branch nodes from the prior room", coarser but still correct — a
   QOL-042 follow-up tightens it once LIFE-001 lands.
