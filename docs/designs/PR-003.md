## §1 — Goal & scope

Allow a signed-in user to upload, change, and remove their profile avatar
photo, with the storage layer enforcing every safety constraint that
matters: MIME + size validated on **both** client (UX) **and** server (the
trust boundary), EXIF / metadata stripped via server-side decode-and-re-encode,
fixed 256×256 main + 64×64 thumb output, and a default generated avatar (the
existing `GeneratedAvatar` from PR-001) when no upload exists. The card
opens the Profile epic — every subsequent Profile card (PR-004 contact
info, future v2 cover image / bio / etc.) consumes the patterns it
establishes (§17): how a Profile section extends `AccountScreen`, which
upload path the repo uses, how storage RLS is written, the optimistic UI
pattern, and the scaffolding shape for moderation columns the team has
not yet decided how to surface.

Doctrine constraints (`cdiscourse-doctrine`):
- Score / standing / heat are not touched. An avatar is profile cosmetics
  with zero effect on the game.
- No AI call. No moderation decision in v1 — the moderation column ships
  as scaffolding only (Q5).
- No service-role key in the client (acceptance criterion 4). The
  `upload-avatar` Edge Function holds the service-role write.
- No raw `avatar_url` text field anywhere a user can type (acceptance
  criterion 2). The path is derived from `auth.uid()` server-side.
- v1 scope guards: no push notifications (no avatar-moderation alert),
  no public API (the Edge Function is per-user only), no realtime
  collaborative editing.

`expo-rn-patterns`: prefer RN primitives; one new dep (`expo-image-picker`)
is justified because the acceptance criteria require client-side image
selection + MIME / size validation, and there is no RN primitive for that.

`accessibility-targets`: the avatar row exposes role / label / hit-target
≥ 44 px; the picker invocation reports status via `accessibilityLiveRegion`;
removal is a destructive action with confirmation copy and a distinct
visual state.

`supabase-edge-contract`: the migration follows OPS-001 four-class
discipline; the Edge Function follows the standard JWT + caller-scope +
service-role-for-writes contract; the audit insert is best-effort.

### In scope
- New SQL migration creating `profile-avatars` storage bucket, four
  `profiles` columns, storage RLS (server-role-write / public-read), and
  a narrowed `profiles` UPDATE policy that refuses avatar-column writes
  via the client JWT.
- New Edge Function `upload-avatar` (three actions: `upload`, `remove`,
  `read_url_for_user`).
- New client wrapper `src/features/account/avatarApi.ts`.
- New presentational component `src/features/account/AvatarUploadSection.tsx`.
- Extension of `AccountScreen.tsx`, `accountApi.ts` (fetch projection),
  `useAccountProfile.ts` (carry new fields), `types.ts` (extend `UserProfile`).
- `expo-image-picker` added via `npx expo install`.
- Tests: `__tests__/avatarApi.test.ts`, `__tests__/avatarUploadValidation.test.ts`,
  `__tests__/uploadAvatarEdgeFunction.test.ts`, `__tests__/avatarUploadSection.test.tsx`,
  `__tests__/avatarDoctrine.test.ts`.

### Out of scope (explicit — see §15)
- Active moderation review surface (the `avatar_moderation_status` column
  is scaffolding only; no admin UI).
- Signed-URL read path (public-read bucket; a v2 toggle is noted in the
  schema for forward compatibility).
- Cover image, gallery, bio, pronouns — separate v2 cards.
- Push notification on moderation decision (v1 scope guard).
- AI moderation, perceptual hashing, content classification — none of
  these are in scope; the moderation column defaults to `allowed` so
  uploads are immediately visible.

## §2 — Source: issue body + Profile epic role

The issue body (#25, verbatim acceptance criteria):
- User can upload/change/remove avatar.
- Avatar URL not user-editable as raw text.
- Upload validates MIME and size.
- No service-role in client.
- RLS/storage policy prevents arbitrary overwrite.

The recommended policy (also from the issue body):
- Bucket `profile-avatars`
- Max 2 MB
- jpg / png / webp
- Resize 256×256 + 64×64 thumb
- Public-read opt-in else signed
- Strip EXIF / metadata pre-upload
- Default generated avatar

PR-003's role in the Profile epic: this is the first card in the epic
that **touches data** (PR-001 was AsyncStorage-only; PR-002 was
AsyncStorage-only). It establishes the migration / Edge Function /
storage pattern. PR-004 then mirrors the pattern for email change. After
both ship, the Profile epic's data surface is complete.

## §3 — Problem statement

Today a CDiscourse user has no avatar. PR-001 ships a deterministic
`GeneratedAvatar` (initials on a hashed colour) as a placeholder, with an
explicit note that PR-003 will own the upload pipeline. Without PR-003:

1. There is no way for a user to set a face on their profile, weakening
   identity in the gallery, the bubble stack, and the timeline.
2. The placeholder is forever — a deliberate limitation, but one that
   makes long-term use feel impersonal.
3. There is no precedent in the repo for storage-backed file uploads.
   Every future card that needs file uploads (cover image, evidence
   screenshots beyond `screenshot_redacted` metadata, dataset uploads
   for EV-005-class evidence) is blocked on someone designing the
   storage + RLS + upload-path patterns.

PR-003 closes both gaps. The acceptance criteria explicitly forbid the
easy-but-unsafe paths (client raw text URL field, client-side service
role, client-only validation). The design below enforces every
acceptance criterion at the data layer, not the UI layer.

## §4 — Data model + file changes

### Migration

**New file:** `supabase/migrations/20260525000016_pr_003_profile_avatars.sql`

Four objects, OPS-001 four-class compliant. Statement order:
1. `insert into storage.buckets` for `profile-avatars` (public-read, size
   cap 2 MiB, MIME allowlist jpg/png/webp).
2. `alter table public.profiles add column avatar_path text`,
   `avatar_thumb_path text`, `avatar_updated_at timestamptz`,
   `avatar_moderation_status text not null default 'allowed' check (...)`.
3. `drop policy "profiles: users update own"` (the existing UPDATE policy
   from `20260516000002_rls_policies.sql`) **and re-create it narrowed**
   to refuse writes to the four avatar columns via the client JWT (see
   the policy text in §8). The new policy still allows `display_name`
   writes — `AccountScreen.tsx`'s display-name edit path is unaffected.
4. Four `storage.objects` RLS policies for the `profile-avatars` bucket:
   public read (SELECT), service-role write (INSERT / UPDATE / DELETE
   are denied to `authenticated`; only the service-role client used by
   `upload-avatar` can write).

The migration follows the OPS-001 four-class discipline. The header
comment names each class and the line range that addresses it (see §8).
Operator deploys via `npx supabase db push --linked`.

### Edge Function

**New directory:** `supabase/functions/upload-avatar/` containing one file
`index.ts` (~280–340 lines, comparable in size to `request-argument-deletion`
which is 236 lines but PR-003 adds image decode + resize + double-write).
Standard contract per `supabase-edge-contract`:

- `verify_jwt = true` in `config.toml` (so the platform verifies the JWT
  before the function runs).
- Caller-scoped client for the auth check (`auth.getUser()`).
- Service-role client for the bucket writes + the `profiles` column
  update.
- Best-effort audit insert.

Three actions in one function (mirrors `manage-room-invite`'s multi-action
dispatch shape, but the actions here are all caller-owned so there is no
unauthenticated path):

| Action | Body | Verifies | Writes | Returns |
|---|---|---|---|---|
| `upload` | `{ action: 'upload', imageBase64: string, mimeType: 'image/jpeg' \| 'image/png' \| 'image/webp', originalByteLength: number }` | JWT, MIME, decoded byte length ≤ 2 MiB, decode succeeds, resulting dimensions ≥ 64 px on each side | `storage.objects` (2 paths: `<uid>/avatar-256.webp` + `<uid>/avatar-64.webp`); `profiles.avatar_path` + `avatar_thumb_path` + `avatar_updated_at` | `{ avatarPath, avatarThumbPath, avatarUpdatedAt, publicUrl, publicThumbUrl }` |
| `remove` | `{ action: 'remove' }` | JWT, caller is profile owner (trivially true — caller IS the profile owner being edited) | `storage.objects` (delete both paths if present); `profiles.avatar_path = null` + `avatar_thumb_path = null` + `avatar_updated_at = now()` | `{ removed: true }` |
| `read_url_for_user` | `{ action: 'read_url_for_user', userId: string }` | JWT, `userId` UUID shape | (none) | `{ publicUrl: string \| null, publicThumbUrl: string \| null, moderationStatus: 'allowed' \| 'removed' }` |

`read_url_for_user` exists so the gallery and bubble stack can display
**other users'** avatars without raw-URL exposure. v1 returns `null` for
both URLs when `moderation_status = 'removed'` (Q5 scaffolding) —
visually equivalent to no avatar (the `GeneratedAvatar` placeholder shows).

The Edge Function imports `imagescript` from `https://deno.land/x/imagescript@1.2.17/mod.ts`
(the established Deno-compatible image codec). `imagescript` decodes
JPEG / PNG / WebP and encodes to WebP — one library covers both the
input MIME allowlist and the output format. The decode-then-encode cycle
strips EXIF by construction (a fresh encode emits no EXIF block).

### Client wrapper

**New file:** `src/features/account/avatarApi.ts` (~140–180 lines).

Pure-TS validation helpers (testable without RN / Supabase) plus thin
Supabase / Edge Function callers:

```ts
export const AVATAR_MAX_BYTES = 2 * 1024 * 1024; // 2 MiB
export const AVATAR_ALLOWED_MIME_TYPES = ['image/jpeg', 'image/png', 'image/webp'] as const;
export type AvatarMimeType = typeof AVATAR_ALLOWED_MIME_TYPES[number];

export type AvatarValidationError =
  | 'mime_not_allowed'
  | 'too_large'
  | 'empty';

/** Pure — testable without picker / network. */
export function validateAvatarSelection(input: {
  mimeType: string;
  byteLength: number;
}): { ok: true } | { ok: false; error: AvatarValidationError };

/** Builds the request body for the Edge Function. */
export async function buildAvatarUploadPayload(input: {
  uri: string;
  mimeType: AvatarMimeType;
}): Promise<{ imageBase64: string; mimeType: AvatarMimeType; originalByteLength: number }>;

/** Calls upload-avatar Edge Function. */
export async function uploadAvatar(input: { uri: string; mimeType: AvatarMimeType }): Promise<
  | { ok: true; data: AvatarUploadResult }
  | { ok: false; error: string; message: string }
>;

export async function removeAvatar(): Promise<
  | { ok: true }
  | { ok: false; error: string; message: string }
>;

/** Resolves a profile.avatar_path → public URL via Supabase storage API.
 *  Pure — no Edge Function call. Returns null if the path is null. */
export function resolveAvatarPublicUrl(avatarPath: string | null): string | null;
```

`resolveAvatarPublicUrl` uses `supabase.storage.from('profile-avatars').getPublicUrl(path)`
— a pure read API that does not require the Edge Function. The Edge Function
is the **write** path; reading a public path goes through the existing
Supabase client.

## §6 — Q2: Image-selection library

**Decision:** Add `expo-image-picker` via `npx expo install expo-image-picker`.

**Why not RN primitives:** there is no RN primitive for accessing the
photo library / camera. `expo-image-picker` is the standard Expo path
and the only library this repo's `expo-rn-patterns` skill names as
acceptable for Expo-tracked features (the skill explicitly bans
`react-native-image-picker`-class community forks for parity reasons).

**Why not `expo-image-manipulator`:** server-side resize (Q7) eliminates
the need for client-side image manipulation. The picker returns a `uri`
(filesystem URI on native, blob URL on web); we `fetch(uri)` to get a
`Blob`, read its size + MIME, and send to the Edge Function for
processing. Skipping the manipulator dep saves ~6 MB of native binary
and avoids a second platform-API surface to learn.

**Why not `expo-file-system`:** `fetch(uri).then(r => r.blob())` works on
both web and native (RN's `fetch` polyfill handles `file://` and
`asset://` URIs on Expo SDK 54). No filesystem reads needed.

**Version constraint:** let `npx expo install expo-image-picker` pick
the version compatible with the pinned `expo: ~54.0.33`. Do not pin
manually.

**Picker invocation shape** (in `AvatarUploadSection.tsx`):

```ts
const result = await ImagePicker.launchImageLibraryAsync({
  mediaTypes: ImagePicker.MediaTypeOptions.Images,
  allowsEditing: true,                    // OS-native crop UI (free)
  aspect: [1, 1],                         // square crop hint
  quality: 0.85,                          // pre-server-side encode bandwidth save
  exif: false,                            // ask OS not to attach EXIF (best effort)
});
if (result.canceled) return;
const asset = result.assets?.[0];
if (!asset?.uri || !asset?.mimeType) return; // defensive — see §11 edge cases
```

**Permission rationale:** `expo-image-picker` requests photo-library
permission on first use; on web there is no permission prompt. The
component must handle the denied path and surface plain copy ("To upload
a photo, allow CDiscourse to read your photo library in system
settings.").

## §7 — Q3: Upload path (Edge Function-mediated)

**Decision:** Edge Function-mediated upload via the new `upload-avatar`
Edge Function. Client sends base64-encoded image bytes; Edge Function
decodes, resizes, re-encodes (which strips EXIF), and writes both paths
via the service-role client.

**Why not direct upload + signed URL:**
- Direct upload puts the resize + EXIF-strip burden on the client, which
  requires `expo-image-manipulator` (extra dep) AND an explicit EXIF
  stripper. Cross-platform EXIF tooling in RN is fragile.
- "Strip EXIF" via client-side codec is a re-encode anyway — same
  CPU cost, but in the user's browser/device.
- Server-side resize lets us **enforce** the output dimensions. A direct
  upload could produce a 4000×4000 file regardless of the client's
  intent, then we'd need a second function to clean up.
- The Edge Function path is the established repo pattern (every existing
  privileged write is a function call).

**Why not Edge Function-mediated multipart:** `multipart/form-data`
parsing in Deno is awkward (the std library has it but it's a
streaming API). Base64 JSON is `imageBase64: string` — one obvious shape,
parsed by `await req.json()`. At 2 MiB max, base64 inflates to ~2.7 MiB
JSON, still well inside the Edge Function 6 MiB request cap.

**Sequence (`upload` action):**

```
client → buildAvatarUploadPayload
       → fetch(uri).blob().arrayBuffer() → base64
client → POST /functions/v1/upload-avatar { action: 'upload', imageBase64, mimeType, originalByteLength }
       Authorization: Bearer <user JWT>

upload-avatar:
  1. verify_jwt = true → platform rejects 401 unauthenticated
  2. req.json() → parse + zod-validate shape
  3. caller_client.auth.getUser() → derive userId
  4. validate MIME ∈ allowlist (server enforces — client check is UX only)
  5. base64 decode → Uint8Array
  6. validate decoded byteLength ≤ 2 MiB (server enforces)
  7. imagescript.decode(bytes, mimeType) → Image
  8. resize to 256×256 (cover-fit, square crop already enforced by picker)
     → encode as WebP (quality 0.85) → main
  9. resize to 64×64 → encode as WebP (quality 0.80) → thumb
  10. service_client.storage.from('profile-avatars')
        .upload(`${userId}/avatar-256.webp`, main, { upsert: true, contentType: 'image/webp' })
  11. service_client.storage.from('profile-avatars')
        .upload(`${userId}/avatar-64.webp`,  thumb, { upsert: true, contentType: 'image/webp' })
  12. service_client.from('profiles').update({
        avatar_path: `${userId}/avatar-256.webp`,
        avatar_thumb_path: `${userId}/avatar-64.webp`,
        avatar_updated_at: new Date().toISOString(),
      }).eq('id', userId)
  13. Best-effort audit insert into admin_audit_events
        (action: 'avatar_uploaded', actor_user_id, payload: { sizeBytesOriginal, sizeBytesMain, sizeBytesThumb })
  14. publicUrl = service_client.storage.from('profile-avatars').getPublicUrl(avatarPath).data.publicUrl
  15. Return { avatarPath, avatarThumbPath, avatarUpdatedAt, publicUrl, publicThumbUrl }
```

**Sequence (`remove` action):**

```
client → POST /functions/v1/upload-avatar { action: 'remove' }
upload-avatar:
  1. JWT verify, caller_client.auth.getUser() → userId
  2. service_client.storage.from('profile-avatars').remove([
       `${userId}/avatar-256.webp`,
       `${userId}/avatar-64.webp`,
     ])  // does not error if files don't exist
  3. service_client.from('profiles').update({
       avatar_path: null,
       avatar_thumb_path: null,
       avatar_updated_at: new Date().toISOString(),
     }).eq('id', userId)
  4. Audit (best-effort): { action: 'avatar_removed', actor_user_id }
  5. Return { removed: true }
```

**Sequence (`read_url_for_user`):**

```
client → POST /functions/v1/upload-avatar { action: 'read_url_for_user', userId }
upload-avatar:
  1. JWT verify (any signed-in user may resolve any other user's URL —
     same as the existing profile SELECT RLS which lets every
     authenticated caller read display_name + role)
  2. caller_client.from('profiles').select('avatar_path, avatar_thumb_path, avatar_moderation_status').eq('id', userId).maybeSingle()
  3. If moderation_status = 'removed' → return { publicUrl: null, publicThumbUrl: null, moderationStatus: 'removed' }
  4. Otherwise → resolve both paths via service_client.storage.getPublicUrl
  5. Return { publicUrl, publicThumbUrl, moderationStatus: 'allowed' }
```

### Profiles UPDATE policy narrowing

The existing policy from `20260516000002_rls_policies.sql` (full text
varies; the surface shape is "caller can update their own row"). The
narrowed replacement:

```sql
drop policy if exists "profiles: users update own" on public.profiles;

create policy "profiles: users update own — narrow"
on public.profiles
for update
to authenticated
using (id = auth.uid())
with check (
  id = auth.uid()
  -- Avatar columns are server-role write only. The WITH CHECK refuses
  -- any UPDATE that changes a snapshot of any avatar column. The
  -- expression below evaluates to true ONLY when every avatar column
  -- in NEW is byte-equal to the OLD row. PostgreSQL gives us no
  -- direct "compare NEW vs OLD" inside a policy; the workaround is
  -- the established pattern of subselecting OLD via the same table:
  and (
    select
      coalesce(p.avatar_path, '') is not distinct from coalesce(public.profiles.avatar_path, '')
      and coalesce(p.avatar_thumb_path, '') is not distinct from coalesce(public.profiles.avatar_thumb_path, '')
      and p.avatar_updated_at is not distinct from public.profiles.avatar_updated_at
      and p.avatar_moderation_status is not distinct from public.profiles.avatar_moderation_status
    from public.profiles p
    where p.id = public.profiles.id
  )
);
```

**Implementer note on the WITH CHECK pattern:** PostgreSQL RLS does not
expose OLD inside a policy expression. The subselect against
`public.profiles p` where `p.id = public.profiles.id` reads the current
(pre-UPDATE) row — this is the established pattern for "freeze a
column" via RLS without a trigger. The reviewer must verify this
pattern compiles and behaves as designed via `npx supabase db reset
--linked=false` (OPS-001 Docker-available path) OR perform the
heightened textual review against the four classes (Docker-unavailable
path). The pattern works in Postgres ≥ 15 (Supabase is on 17 per
`supabase/config.toml` line 49). If the reviewer's local apply surfaces
unexpected behaviour with this pattern, the fallback is to move the
column-freeze enforcement to a `BEFORE UPDATE` trigger that raises
`exception` when an avatar column is touched and the calling role is
`authenticated`. The trigger fallback is documented in §16.

## §9 — Q5: Moderation scaffolding (minimal column only)

**Decision:** Add a single column on `public.profiles`:

```sql
avatar_moderation_status text not null default 'allowed'
  check (avatar_moderation_status in ('allowed', 'removed'))
```

The two-value enum is the smallest set that lets a future admin card
hide an avatar without deleting the file (`'removed'` flips the
`read_url_for_user` response to `null` URLs; the storage object can
persist for audit). When a future card needs `'pending_review'` or
`'flagged'`, that card adds the value via a new migration and the CHECK
constraint widening.

**Why not a separate `avatar_moderation` table:** at one row per user,
the column is cleaner. The avatar IS a profile attribute; the
moderation status IS attached to the profile, not to a separate
artifact. A separate table is only justified when the entity has its
own lifecycle independent of the profile (which an avatar moderation
status does not).

**Why default `'allowed'`:** zero behaviour change for existing users.
PR-003 ships without an admin UI; the column is dormant scaffolding.

**Edge Function behaviour gating on the column:**
- `upload`: writes regardless of current moderation status (a user
  uploading a new avatar overrides any prior `'removed'` flag —
  intentional: the column is moderation, not a permanent ban; the user
  can replace the offending image with one that is not). A future
  moderation card may want to distinguish "removed-pending-replacement"
  from "removed-permanent"; that is out of scope.
- `remove`: same — works regardless.
- `read_url_for_user`: returns `null` URLs when `moderation_status =
  'removed'`. The client falls back to `GeneratedAvatar`.

**No admin UI in v1.** No admin button to flip the status. No
notification to the affected user. No appeal flow. All of those are
explicit follow-up cards.

## §10 — Q6: Optimistic update pattern

**Decision:** Local-state-first. On successful picker selection (after
validation passes), `AvatarUploadSection` shows the picked `uri`
immediately via `<Image source={{ uri: localPickerUri }} />` with a
translucent `ActivityIndicator` overlay. The Edge Function call runs in
the background. On success, the local state swaps to the returned
`publicUrl` (cache-busted via `?v=${avatarUpdatedAt}`); on failure, the
local state reverts to the previous `avatarPath` (or `GeneratedAvatar`
if there was none) and a plain-copy error appears.

**State machine:**

```
idle (no upload in flight)
  └─ picker selection passes validation
     → uploading (locally-rendered picked image + spinner)
        ├─ on success → idle (with new public URL)
        └─ on error  → idle (reverted, error visible inline)
```

**Cache busting:** the public URL is `<bucket>/<path>` with no version
parameter by default. After an upload, the same path produces the same
URL — but the file content has changed. Browsers / RN's image cache
will show the OLD avatar until the cache is invalidated. The fix:
append `?v=${avatarUpdatedAt}` (the timestamp the Edge Function writes
to `profiles.avatar_updated_at`) to the URL on every render. The
timestamp changes on every upload / remove, so the cache key changes;
when no upload has happened, the timestamp is stable and the cache hit
is fine.

**Pattern PR-004 mirrors:** local state shows the new value
immediately, network call runs in the background, revert on error,
inline plain-copy error message. PR-004's email-change flow is more
delicate (involves a verification email) but the optimistic shell is
the same: show the new email locally with a "verification pending"
badge, revert to the old email if the API call fails before the
verification email is sent.

**Removal confirmation:** two-tap pattern, not a modal. First tap on
"Remove" replaces the button with two buttons: "Confirm remove" and
"Cancel". Confirms via the Edge Function. This avoids the dependency
on a confirmation modal component (the repo has `Modal` patterns but
adding one here would over-engineer a two-tap action).

## §11 — Edge cases

| Case | Handling |
|---|---|
| **Picker cancelled** | `result.canceled === true` → no-op, return to idle state silently. |
| **Picker returns no `uri` or no `mimeType`** | Defensive — the asset object is malformed. Surface "We couldn't read that photo. Try another file." plain copy. No upload attempted. |
| **MIME not in allowlist (e.g. HEIC)** | Client validation rejects with "Use a JPG, PNG, or WebP photo." server-side double-check rejects with `error: 'mime_not_allowed'`. |
| **File > 2 MiB** | Client rejects with "That photo is too large (max 2 MB)." Server rejects with `error: 'too_large'`. The bucket-level cap rejects with a storage 413. Three layers — only the first surfaces to the user under normal conditions. |
| **File is 0 bytes (corrupted)** | Client `validateAvatarSelection` rejects with `error: 'empty'`. Server rejects with `error: 'invalid_image'` after `imagescript.decode` throws. |
| **Decoded dimensions < 64 px on either side** | Server rejects with `error: 'image_too_small'`, plain copy "Use a photo at least 64×64 pixels." (Below 64 the 64-px thumb would be an upscale — visually worse than the placeholder.) |
| **EXIF strip fails (e.g. decode throws on a malformed file)** | The `try { imagescript.decode(...) } catch (e)` returns `error: 'invalid_image'`. The original bytes are NOT written to storage. Strip-via-re-encode means a decode failure means no upload happened. |
| **Storage upload fails (network, quota)** | Edge Function returns `error: 'storage_upload_failed'`. The `profiles` columns are NOT updated. The user's prior avatar (if any) remains in place. The client shows "Upload failed. Try again." |
| **Profiles update fails after a successful storage upload** | The function attempts to delete the just-uploaded paths (best-effort cleanup), then returns `error: 'profile_update_failed'`. If the cleanup also fails, the orphaned objects are logged for the operator. The user retries; the next upload upserts to the same path and the orphan is overwritten. |
| **`read_url_for_user` for a user that does not exist** | Profile SELECT returns null → return `{ publicUrl: null, publicThumbUrl: null, moderationStatus: 'allowed' }`. The caller falls back to `GeneratedAvatar`. Avoids a 404 for what is effectively "this user has no avatar." |
| **`read_url_for_user` for self when caller has no avatar** | Same — both URLs `null`, `GeneratedAvatar` renders. |
| **`moderation_status = 'removed'`** | `read_url_for_user` returns null URLs. `GeneratedAvatar` renders. The user themselves can still see their photo via `AvatarUploadSection` directly reading their own `profiles.avatar_path` (they retain access to their own profile data per existing RLS) — though v1 ships no moderation UI so this edge case is dormant. |
| **Concurrent uploads from two tabs** | The Edge Function is per-call; the last-write-wins via `upsert: true` on storage and a plain UPDATE on profiles. Both tabs eventually display the second upload's avatar after a refresh. No locking needed — the data IS the user's own avatar; no inter-user race. |
| **Permission denied on photo library** | The picker resolves with `result.canceled = true` and a permission status. The component surfaces "To upload a photo, allow CDiscourse to read your photo library in system settings." |
| **Offline (Edge Function unreachable)** | The picker still works; the upload fails with a network error; the client reverts the local optimistic state and shows "Network error. Check your connection." |
| **A user manually crafts a `PATCH /profiles` request to set `avatar_path` to another user's path** | The narrowed UPDATE policy's WITH CHECK refuses (the row's `avatar_path` would change vs the OLD snapshot). The request returns 403. Even if the policy were bypassed, the storage path is namespaced by user id; the user would be claiming credit for an image only the other user can overwrite. |
| **An attacker tries `supabase.storage.from('profile-avatars').upload('victimUserId/avatar-256.webp', maliciousBytes)` from a client** | No INSERT policy on `storage.objects` for `authenticated`. Returns 403. |
| **A user uploads, then removes — but the storage objects are not deleted (e.g. permission glitch)** | The `profiles.avatar_path` is null, so the UI shows `GeneratedAvatar`. The orphan storage objects are invisible to the UI (nothing references them). A future cleanup job is a separate operational concern. |

## §16 — Open questions

**The design has no operator-judgment gaps that require an answer
before implementation.** Every Q1–Q6 framework question is resolved
above. The two minor implementation choices below are flagged for
implementer awareness but do not block — both have a documented
default and a documented fallback.

1. **The narrowed-UPDATE RLS pattern in §8** uses a subselect against
   the same table (`select ... from public.profiles p where p.id =
   public.profiles.id`) to read the OLD row from inside a policy. This
   is an established Postgres pattern and works on Postgres ≥ 15
   (Supabase is on 17). If the reviewer's local `npx supabase db reset
   --linked=false` apply surfaces unexpected behaviour, the documented
   fallback is to replace the WITH CHECK column-freeze with a
   `BEFORE UPDATE` trigger that raises `exception` when any avatar
   column is `NEW.avatar_* IS DISTINCT FROM OLD.avatar_*` AND
   `current_setting('request.jwt.claims', true)::jsonb ->> 'role' =
   'authenticated'`. The trigger fallback is fully replaceable without
   any other migration changes.

2. **The `imagescript` version pin** (`@1.2.17`) is the latest stable
   at design time. The implementer should verify this version still
   exists at install time; if Deno's CDN has moved it, pick the
   newest stable patch within `1.2.x` and update the pin in both the
   Edge Function source AND the corresponding line in the
   `uploadAvatarEdgeFunction.test.ts` regex assertion.
