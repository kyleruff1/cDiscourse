## Goal (one paragraph)

MCP-013 builds the **referee ledger adapter** — layer 3 of the semantic-referee architecture. It is the deterministic, pure-TS model that takes three inputs — (1) deterministic layer-1 move metadata, (2) one advisory `SemanticRefereePacket` from layer 2 (the MCP-011 contract; in this card always a validated **mock** packet), and (3) the existing point-standing economy — and produces **play-quality feedback per move**. The doctrine that shapes every line: the ledger scores the *rules of play* (continuity, evidence hygiene, branch hygiene, constructive movement), never *truth*; it never declares a winner or loser; it never blocks a post; it never assigns a truth value; it keeps engagement credit and factual-standing credit on **separate axes** (per `applyAntiAmplification`); and concession renders as a **repair**, never a defeat. The ledger does not replace the point-standing economy — for the categories the economy owns (`narrowing`, `concession`, `evidence_debt_resolution`) it **calls `gradeChallenge` / `gradeRepair` and adopts the returned delta bit-for-bit**, proven by a `refereeLedgerEconomyParity` test. A layer-1 / layer-2 conflict is never a penalty: it routes to a reversible user choice (`conflict_routed`, `delta: 0`). A low-confidence semantic binary moves no standing on its own.

## Data model

No new database table, no migration, no Edge Function, no persistence. The ledger is a pure-TS in-memory model — the same constraint as `src/lib/constitution/engine.ts` and `src/features/pointStanding/`. All new types live in `src/features/refereeLedger/types.ts`. They are JSON-serializable, side-effect-free, and contain **no `block` / `winner` / `truthValue` / `score`-scalar / per-side-aggregate field** anywhere.

### Core unions (MCP-003 §4, §5)

```ts
// The 14 play-quality categories the ledger may move. None is a truth category.
export type RefereePointCategory =
  | 'continuity'
  | 'direct_response'
  | 'evidence_provided'
  | 'evidence_relevance'
  | 'quote_anchoring'
  | 'narrowing'
  | 'concession'
  | 'clarification'
  | 'synthesis'
  | 'branch_hygiene'
  | 'person_intent_drift'        // negative-only — reads 0 or a small negative; never positive
  | 'evidence_debt_resolution'
  | 'staying_in_mode'
  | 'respecting_pacing';

export type ReconciliationOutcome =
  | 'agreement'                 // L1 and L2 agree → full-confidence delta
  | 'l1_only'                   // no semantic signal for this category → L1 stands
  | 'l2_only'                   // no deterministic signal → L2 stands, magnitude capped at medium tier
  | 'conflict_l1_dominant'      // L1 and L2 disagree AND L1 is authoritative → L1 value, confidence→medium, no penalty
  | 'conflict_routed'           // L1 and L2 disagree, neither dominant → delta 0, reversible user choice
  | 'low_confidence_semantic';  // L2 confidence is 'low' → delta 0, soft feedback code

export type LedgerConfidence = 'high' | 'medium' | 'low';

export type CreditAxis = 'engagement' | 'factual_standing' | 'hygiene';
```

### `LedgerMoveInput` / `LedgerResult` (MCP-003 §9.3)

```ts
export interface LedgerMoveInput {
  pointId: string;
  moveArgumentId: string;
  /** MCP-001 layer 2 — advisory; ALWAYS a validated mock packet in this card.
   *  May be omitted entirely; an absent packet runs the ledger l1_only. */
  semanticPacket?: SemanticRefereePacket;
  /** Layer-1 deterministic facts. */
  deterministicMetadata: DeterministicMoveMetadata;
  /** challenge | repair — decides which economy call fires. */
  moveRole: 'challenge' | 'repair';
  /** The economy grading input the caller already assembled. */
  economyInput: ChallengeGradingInput | RepairGradingInput;

  repairOptions?: RepairGradingOptions;

  debtLedger: IssueDebtLedger;
  /** Per-debt category-credit bookkeeping carried across moves. Caller owns the lifetime. */
  creditStates?: readonly LedgerCreditState[];
}

export interface LedgerResult {
  pointId: string;
  moveArgumentId: string;
  /** One reading per category this move touched. Untouched categories omitted. */
  categoryReadings: readonly CategoryReading[];
  /** Verbatim from the economy — adopted, never re-derived. Null on the ineligible path. */
  economyDelta: PointStandingDelta | null;
  /** Verbatim from applyAntiAmplification. Null when no evidence-axis category was read. */
  antiAmplification: AntiAmplificationResult | null;
  /** Re-uses the ineligibilityReasons strings from the economy's grading results. Empty when eligible. */
  ineligibilityReasons: readonly string[];
  /** True when any reading is conflict_routed — the caller surfaces an MCP-010 choice. */
  needsUserChoice: boolean;
  /** 0..1 advisory audit signal, forwarded UNCHANGED from the economy. The ledger never acts on it. */
  exploitRiskScore: number;
  /** Updated per-debt credit bookkeeping — caller persists this for the next move. */
  creditStates: readonly LedgerCreditState[];
  /** Always the literal true — advisory output. Matches the economy's grading results. */
  userReviewRequired: true;
}
```

> **Schema-shape doctrine (MCP-003 §12, acceptance criterion):** `LedgerResult` has **no** `block`, **no** `winner`, **no** `truthValue`, **no** `score` scalar, **no** per-side aggregate (no `affirmativeTotal` / `negativeTotal` / `scoreToBeat` / `leaderboard`). ## Reconciliation algorithm (layer-1 / layer-2 / economy)

`reconcileMove` runs five stages in order. Each is pure.

### Stage 0 — classify the move and call the economy

`reconcileMove` reads `input.moveRole`:

- `moveRole === 'challenge'` → call `gradeChallenge(input.economyInput as ChallengeGradingInput)`. Adopt `result.delta` (a `PointStandingDelta | null`), `result.newDebt`, `result.eligible`, `result.ineligibilityReasons`. If `newDebt` is non-null, `debtLedger.appendDebt(newDebt)`.
- `moveRole === 'repair'` → call `gradeRepair(input.economyInput as RepairGradingInput, input.repairOptions!)`. Adopt `result.delta`, `result.updatedDebt`, `result.effect`, `result.eligible`, `result.ineligibilityReasons`. If `updatedDebt` is non-null, `debtLedger.updateDebt(updatedDebt.debtId, () => updatedDebt)`.

`LedgerResult.economyDelta` is set to the economy's `delta` **verbatim** (a structural copy is allowed; the *numbers* are never recomputed). `LedgerResult.exploitRiskScore` is `economyDelta?.exploitRiskScore ?? 0` — forwarded unchanged, never acted on (MCP-003 §7.3). `LedgerResult.ineligibilityReasons` is the economy's `ineligibilityReasons` array.

### Stage 1 — anti-amplification on the economy delta

When `economyDelta` is non-null and any factual-standing category is in play (`evidence_provided` / `evidence_relevance` / `evidence_debt_resolution`), build an `AmplificationContext` and run `applyAntiAmplification(economyDelta, ctx)`:

```ts
const ctx = amplificationContextFromAnnotationFields({
  platformSupportWarning: binaryValue('uses_popularity_as_evidence') === 1,
  evidentiaryRisk: binaryValue('creates_source_chain_gap') === 1 ? 'high' : 'low',
  amplificationRisk: binaryValue('uses_popularity_as_evidence') === 1 ? 'high' : 'none_observed',
  amplificationSignals: {
    appeal_to_virality: binaryValue('uses_popularity_as_evidence') === 1,
    appeal_to_crowd_size: binaryValue('uses_popularity_as_evidence') === 1,
    high_engagement_low_evidence:
      binaryValue('uses_popularity_as_evidence') === 1 && !deterministicMetadata.hasAttachedEvidence,
    unknown_source_chain: binaryValue('creates_source_chain_gap') === 1,
  },
  hasEvidence: deterministicMetadata.hasAttachedEvidence,
  hasTargetExcerpt: deterministicMetadata.hasQuoteAnchor,
  hasScopeNarrowing: binaryValue('narrows_claim') === 1,
});
```

`binaryValue(id)` returns the packet binary's `value` for that `classifierId`, or `0` when the packet is absent or the binary is missing. The `AntiAmplificationResult` is stored on `LedgerResult.antiAmplification`. The factual-standing categories are then read against `antiAmplification.adjustedDelta` and `antiAmplification.factualStandingGainSuppressed`: a suppressed move emits the `evidence_debt_open` feedback code with `delta: 0` — **never a penalty** (MCP-003 §6.1, §13). Engagement-credit and hygiene categories are never affected by anti-amplification.

> **Note — anti-amplification only constrains *gains*.** `applyAntiAmplification` preserves negative deltas (the doctrine constrains gains, not consequences — see `antiAmplification.ts` header). The ledger never adds an amplification penalty of its own.

### Stage 2 — anti-exploit gate

Call `evaluateLedgerGate(input)` (see "Anti-exploit gates" below). If `tripped`, short-circuit: build a `LedgerResult` where **every** `CategoryReading.delta === 0`, `ineligibilityReasons` is populated, and one constructive feedback code nudges the move back into eligibility (`you_decide_the_lane` for a tangent route, or the category-appropriate prompt code). No penalty — an ineligible move is a no-op for the ledger, exactly as `gradeChallenge` returns `eligible: false, delta: null`.

### Stage 3 — per-category reconciliation

For each of the 14 categories that this move *touched* (a category is touched when it has a layer-1 derivation, a layer-2 binary, or a relevant `scoreHint` — categories with no signal are omitted from `categoryReadings`), call `reconcileCategory`. The algorithm per `CATEGORY_AUTHORITY` entry (MCP-003 §5.1, §5.3):

| `CATEGORY_AUTHORITY` value | Rule |
|---|---|
| `economy` (`narrowing`, `concession`, `evidence_debt_resolution`) | The reading's `delta` is **adopted bit-for-bit** from the economy result — `narrowing` ← `gradeRepair` `narrowStandingDelta`; `concession` ← `gradeRepair` `broadStandingDelta` (the repair effect); `evidence_debt_resolution` ← the economy's debt-close path. `outcome` is `agreement` if a semantic binary confirms, else `l1_only`. The ledger **never re-derives** these — `refereeLedgerEconomyParity.test.ts` proves it. |
| `l1` (`evidence_provided`, `quote_anchoring` presence, `clarification`, `synthesis`, `branch_hygiene`, `staying_in_mode`, `respecting_pacing`) | L1 fact stands. On agreement with L2 → `agreement`, full magnitude, `confidence: 'high'`. On conflict → `conflict_l1_dominant`: use the L1 value, `confidence` → `'medium'`, **no penalty for the disagreement**. When L2 is absent → `l1_only`, `confidence: 'high'`. |
| `l2` (relevance questions L1 cannot see, e.g. `evidence_relevance`) | L2 stands; the delta magnitude is **capped at the medium tier** (`HINT_STEP * 2 = 0.16`); `confidence` follows the packet's `SemanticConfidence`. When L1 is absent → `l2_only`. |
| `neither` (`continuity`, `direct_response`, `evidence_relevance`, `person_intent_drift`) | On agreement → `agreement`, full magnitude. On disagreement → `conflict_routed`: `delta: 0`, `confidence: 'low'`, `requiresUserChoice: true`, `feedbackCode: 'you_decide_the_lane'`. **Never a penalty.** |

Across all branches: when the L2 binary's `confidence === 'low'`, the outcome is forced to `low_confidence_semantic`, `delta: 0`, and the soft feedback code for that category (`partial_parent_tie` / `question_still_open` / `evidence_needs_connecting` / `almost_a_synthesis` / `belongs_on_a_branch` / `pacing_is_on`). A low-confidence binary never moves standing on its own (MCP-003 §5.1 rule 4).

The hint contribution from `scoreHintMapping` is folded in *before* clamping: a positive category delta = economy-or-zero base + `hintMagnitude`, then `clampCategoryDelta` to `[-0.4, +0.4]`. When the outcome is `low_confidence_semantic` or `conflict_routed`, the hint contributes `0` regardless of its integer value (MCP-003 §6.2).

### Stage 4 — credit-state update + assemble

For each economy-owned or per-debt category (`narrowing`, `concession`, `evidence_debt_resolution`, `quote_anchoring` against a debt), call `applyCreditState(category, debtId, creditStates)` *before* awarding the delta. If `allowed` is false (the category already paid against that `debtId`), the reading's `delta` is forced to `0` (no double-pay — MCP-003 §7.2). Per-move categories (`continuity`, `direct_response`, `staying_in_mode`, `respecting_pacing`, `clarification`, `branch_hygiene`, `person_intent_drift`, `synthesis`) are **not** credit-state-gated — they re-evaluate every move. Assemble `LedgerResult`: `categoryReadings`, `economyDelta`, `antiAmplification`, `ineligibilityReasons`, `needsUserChoice` (true iff any reading is `conflict_routed`), `exploitRiskScore`, the updated `creditStates`, `userReviewRequired: true`.

## The low-confidence gate

A semantic binary whose `confidence === 'low'` (MCP-011 `SemanticConfidence`) **never moves standing on its own** (MCP-003 §5.1 rule 4, acceptance criterion: "Low-confidence semantic binary moves no standing"). Concretely, inside `reconcileCategory`:

1. If `l2Signal !== null && l2Signal.confidence === 'low'` → set `outcome: 'low_confidence_semantic'`, `delta: 0`, `confidence: 'low'`, `requiresUserChoice: false`, and the **soft** feedback code for the category (a gentle prompt, not a credit).
2. This check runs **before** the `CATEGORY_AUTHORITY` branch — a low-confidence binary cannot reach the `agreement` / `conflict` paths at all.
3. The exception: a category that is **economy-owned** ignores the low-confidence semantic entirely — the economy delta is deterministic and authoritative there; the semantic binary is only a confirmation hint. A low-confidence binary on an economy-owned category simply leaves `outcome: 'l1_only'` with the economy's delta intact.
4. A low-confidence binary still cannot *trigger* a `conflict_routed` — a conflict requires a binary the ledger trusts enough to disagree with; a low-confidence binary is treated as "no usable signal", so the category falls to `l1_only` when L1 exists, or `low_confidence_semantic` (delta 0) when it does not.

`refereeLedgerConflict.test.ts` covers this with a `validLowConfidence`-style fixture (MCP-011 `VALID_FIXTURES.validLowConfidence` carries a `confidence: 'low'` binary).

## Anti-exploit gates

`evaluateLedgerGate` reuses the economy's `ScoringEligibility` machinery — it does **not** invent a parallel anti-exploit machine (MCP-003 §7). A move earns **no credit on any category** when any gate trips:

| Gate | Detection (reuses economy + semantic) |
|---|---|
| **tangent** | `buildChallengeEligibility(input.economyInput).isTangent` (true when `mixedAgreementClass === 'tangent_or_joke'`) **or** the semantic binary `suggests_diagonal_tangent` at `confidence !== 'low'` with `value === 1`. |
| **near-duplicate** | `buildChallengeEligibility(input.economyInput).isNearDuplicate` (another open debt on the same axis + intensity). |
| **self-concession-loop** | `buildRepairEligibility(input.economyInput, input.repairOptions).isSelfConcessionLoop` (same author, `previousRepairsByAuthor >= 2`, explicit concession marker). Only checked when `moveRole === 'repair'`. |
| **no-axis-identified** | `!buildChallengeEligibility(input.economyInput).isAxisIdentified` (`disagreementType` is `none` or `framing`). |

When a gate trips, `LedgerResult` carries every `CategoryReading.delta === 0`, `ineligibilityReasons` populated with the economy's reason strings (`is_tangent`, `is_near_duplicate`, `self_concession_loop`, `not_specific_to_parent`, `no_live_issue_debt`, `low_effort_agreement` — these are the `reasonsFromEligibility` strings carried on `ChallengeGradingResult.ineligibilityReasons` / `RepairGradingResult.ineligibilityReasons`; the ledger reads them off the *public* grading result, it does not import the file-private `reasonsFromEligibility` helper), and a **constructive** feedback code that nudges the move back into eligibility — e.g. a tangent → `belongs_on_a_branch` ("This probably belongs on a branch."). No penalty: an ineligible move is a no-op.

**One credit per debt (MCP-003 §7.2).** `applyCreditState` enforces it. Before the ledger awards `evidence_debt_resolution`, `narrowing`, `concession`, or `quote_anchoring` credit *against a specific `debtId`*, it checks `LedgerCreditState.awardedCategories`. If the category is already present for that `debtId` → `delta: 0`. The per-move categories (`continuity`, `direct_response`, `staying_in_mode`, `respecting_pacing`) are not credit-state-gated — they are not farmable because the parent changes each move and a clean tie is expected every move. `LedgerCreditState` is keyed by the same `debtId` the economy mints; `creditStates` flows through `LedgerMoveInput` → `LedgerResult` so the caller threads it across moves.

**Exploit-risk audit signal (MCP-003 §7.3).** `LedgerResult.exploitRiskScore` = the economy's `PointStandingDelta.exploitRiskScore`, forwarded **unchanged**. The ledger never acts on it, never auto-penalizes — it is an advisory signal for a human reviewer.

## scoreHintMapping (`scoreHints` → ledger delta)

`scoreHintMapping.ts` encodes MCP-003 §6. `HINT_STEP = 0.08` (so a max hint of `3` contributes `0.24` — comparable to the economy's `+0.25` broad-lift, deliberately not larger; MCP-011 confirmed `SCORE_HINT_MAX = 3`, closing MCP-003 §16's open item). Every category delta is `clampCategoryDelta`-ed to `[-0.4, +0.4]`.

| `scoreHint` field | Feeds category | Mode | Rule |
|---|---|---|---|
| `continuityCredit` | `continuity`, `direct_response` | `credit` | `magnitude = HINT_STEP * continuityCredit`, then reconciled against L1 parent-id presence. `0` → no credit, never a penalty. |
| `evidencePressure` | `evidence_provided`, `evidence_relevance` | `opens_debt` | `> 0` means the move *needs* evidence — raises an `OpenIssueDebt` via the economy; **never subtracts standing**. A prompt, not a penalty. |
| `branchHygiene` | `branch_hygiene` | `credit` | `magnitude = HINT_STEP * branchHygiene`, reconciled against BR-001 `branchKind` (L1-dominant). |
| `synthesisReadiness` | `synthesis` | `credit` (gated) | Contributes credit **only** when `deterministicMetadata.lifecycleSynthesisReady`. Otherwise produces the `almost_a_synthesis` prompt — never a credit. |
| `sourceChainDebt` | `evidence_debt_resolution` | `opens_debt` | `> 0` sizes the evidence debt the economy tracks; a *later* move that closes it earns `evidence_debt_resolution` credit. |
| `unresolvedRedirectRisk` | `branch_hygiene` (route), `person_intent_drift` (drift) | `routes` | High redirect risk raises a reversible routing prompt (`belongs_on_a_branch`); **never subtracts standing**. |

**The crucial direction rule (MCP-003 §6.1 — encoded as a hard invariant):** every hint function returns a `magnitude >= 0` and a `mode` of `credit | opens_debt | routes`. A hint can never directly subtract standing. The only negative deltas in the whole ledger are (1) the economy's own reviewed `gradeRepair` evasion path (`no_concession` / `performative_concession_no_repair` → narrow-standing drop), adopted verbatim, and (2) the negative-only `person_intent_drift` hygiene reading — and even that only fires when L1 and L2 *agree* the move targets the speaker, or on a clear L1 fact; a guess routes (`conflict_routed`), never penalizes.

## Edge cases

- **Empty / missing semantic packet** — `LedgerMoveInput.semanticPacket` is optional. When absent, every category is treated as `l1_only`: the ledger runs entirely on layer-1 + the economy, emits full-confidence readings, and never errors. The board still keeps points; the semantic layer contributes nothing. (MCP-003 §13 first bullet.)
- **Packet with `authoritative` not `false`** — impossible per the MCP-011 literal type, but a defensive runtime guard in `reconcileMove` treats any non-`false` value as a malformed packet and falls back to `l1_only`. `refereeLedger.test.ts` covers it.
- **Move that is neither a clean challenge nor a clean repair** — `moveRole` is required on `LedgerMoveInput`; the caller classifies it (typically by `deterministicMetadata.authorIsOriginalSpeaker`). When the caller cannot decide, the `continuity` category routes to `conflict_routed` rather than guessing.
- **Layer-1 / layer-2 conflict on a `neither`-authority category** — `conflict_routed`: `delta: 0`, `confidence: 'low'`, `requiresUserChoice: true`, `feedbackCode: 'you_decide_the_lane'`, never a penalty.
- **Layer-1 / layer-2 conflict on an `l1`-authority category** — `conflict_l1_dominant`: use the L1 value, `confidence` → `'medium'`, no penalty.
- **Low-confidence semantic binary** — `low_confidence_semantic`: `delta: 0`, soft feedback code, no standing move (see "The low-confidence gate").
- **Anti-exploit gate trips** (tangent / near-duplicate / self-loop / no-axis) — every `CategoryReading.delta === 0`, `ineligibilityReasons` populated, a constructive nudge, no penalty.
- **Re-stated concession on an already-resolved debt** — `applyCreditState` finds `concession` already in `awardedCategories` for that `debtId` → `delta: 0`. No double-pay.
- **Move that closes a debt the economy already closed** — `debtLedger.getDebt(debtId)?.closed === true` → `evidence_debt_resolution` reading is `delta: 0` (one credit per debt).
- **Amplification-only move** (high engagement, no evidence, no narrowing) — engagement-credit and hygiene categories may still earn; factual-standing categories are suppressed by `applyAntiAmplification` (`factualStandingGainSuppressed: true`); `evidence_provided` renders `evidence_debt_open` ("evidence debt opened"), never a penalty.
- **`person_intent_drift` on a spicy-but-on-topic hot take** — a hot take is *playable*, not drift. `person_intent_drift` reads negative only when L1 and L2 agree the move targets the *speaker*; a coherent spicy claim is `0`. A conflict here routes (`conflict_routed`), never penalizes.
- **Economy returns `eligible: false, delta: null`** — `reconcileMove` short-circuits like the anti-exploit path: all readings `delta: 0`, `economyDelta: null`, `ineligibilityReasons` from the economy result, a constructive nudge.
- **`moveRole === 'repair'` but `repairOptions` is undefined** — a defensive guard treats it as `eligible: false` with a clear `ineligibilityReasons` entry rather than throwing; `repairOptions` is documented required for repairs.
- **Concurrent edits** — argument bodies are immutable after posting (soft-delete only — CLAUDE.md Supabase conventions); `contentHash` is stable, so a move is reconciled at most once per `promptVersion`. No concurrent-edit race exists for a posted move.
- **Offline / network failure** — the ledger is pure-TS with no network; the *packet* is fetched upstream. If a packet fetch fails, the caller omits `semanticPacket` and the ledger runs `l1_only`. The ledger itself never makes a network call.
- **Permission-denied** — not applicable: the ledger model has no auth surface; it consumes already-fetched data.
- **Doctrine edge — "can a hint push standing negative?"** — no. Every hint function returns `magnitude >= 0`. The only negatives come from the economy's reviewed evasion path and the negative-only `person_intent_drift` reading.
- **Doctrine edge — "can heat / popularity earn a category?"** — no. `DeterministicMoveMetadata` carries no engagement / view / heat field; none of the 14 categories takes a crowd count as input. The `uses_popularity_as_evidence` binary *raises evidence pressure on the move's claim* (via `applyAntiAmplification`); it never earns the move points and never penalizes the crowd.
