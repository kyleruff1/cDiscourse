# PM-003 — Research page: playable argument modes, friction, channels, semantic metadata

**Status:** Design draft
**Epic:** Project Mgmt (Epic 14 — UX Project Board)
**Release:** 6.6
**Issue:** https://github.com/kyleruff1/cDiscourse/issues/121

## Goal (one paragraph)

PM-003 is a **design-only / research-only** card. Its deliverable is one long-form research
artifact, `docs/research/playable-argument-modes.md`, that ties together the design vision behind
the wave of sibling cards (RULE-004, RULE-005, RULE-006, BR-003, GAME-002, GAME-003, EV-005,
COMPOSER-002, SC-005, VG-004) and the PvP-roadmap companion PM-004. The research page is the
navigation map for that wave: it decomposes the friction / channels / modes / semantic-metadata
vision into the child cards, builds a live child-issue map and dependency graph, sequences the
first three implementation slices, lists what the agent runner must *not* pick up, and captures
the open product questions for the operator. There is **no Build phase and no production code** —
the card is shaped by `cdiscourse-doctrine` (all 10 rules are reiterated verbatim in the research
doc's §1), specifically: research-section discussion of "winner / right / wrong" is allowed only
inside clearly-marked operator-decision sections, never as shippable copy.

## Edge cases

- **A sibling card's issue is closed/merged.** The research doc's §15 must mark it as closed.
  Handled: VG-004 (#113) is closed (Build complete) and is marked as such; the upstream
  foundations (BR-001, LIFE-001, META-001, SC-004, COMPOSER-001, EV-001, EV-002, RULE-001/002/003)
  are listed as already-done and "do not redo".
- **The PM-004 companion research files do not exist in the repo.** PM-004's issue body references
  `docs/project-research/...` and `docs/roadmap-expansions/...`; neither directory exists. Handled:
  the research doc states plainly that PM-004 *creates* those files; PM-003 does not invent them.
- **The issue body's child-card count drifts from the live issue list.** The issue says "12
  sibling cards"; the enumerated list has 10 implementation siblings + PM-004. Handled: §15 is
  built from the live `gh issue list` retrieval, not the prose count, and notes the relationship.
- **Operator-decision sections must not become answers.** §19 lists the open questions and
  carries the sibling cards' *recommendations as input*, but explicitly does not answer them.
- **Doctrine-sensitive copy.** "Winner / right / wrong" appears only in §1 (scope-guard
  discussion) and §19/§20 (operator decisions, research framing) — never as product copy.

## Dependencies (cards / docs / files)

## Risks

- **Research drift.** The child-issue map (§15) and dependency graph (§16) are snapshots of the
  GitHub issue list at authoring time (2026-05-20). As sibling cards land, close, or get renamed,
  §15 / §16 / §19 go stale. Mitigation: the doc's closing line instructs contributors to keep
  §15 and §19 current; the design-only nature means a refresh is a cheap doc edit, not a rebuild.
- **Scope creep into implementation.** A reader could mistake the research doc's detailed
  descriptions of `preSendReviewModel` etc. for an instruction to build them. Mitigation: §18
  ("Items explicitly NOT for the agent runner") and the explicit "design-only / no Build phase"
  framing in both this doc and the research doc.
- **Doctrine contradiction.** The "do not copy" / false-positive lists could accidentally
  contradict `cdiscourse-doctrine`. Mitigation: the research doc's ban-list (§20) is authored as a
  strict *superset* of doctrine rule 1's list; §1 reiterates all 10 rules verbatim.

## Out of scope

- Any production code, UI, migration, Edge Function, AI call, or `.env` change.
- Implementing any child card (RULE-004 … VG-004) — each has its own trajectory.
- Answering the operator decisions in §19 — the card *captures* them.
- Creating PM-004's PvP market-research companion files — PM-004 owns those.
- The PvP slate (GAME-004…GAME-008, BR-004, RULE-007) — a parallel wave, only cross-referenced.
- Marketing copy or external publication.

## Doctrine self-check

- **cdiscourse-doctrine — no truth labels:** the research doc introduces zero shippable
  user-facing copy. Discussion of "winner / right / wrong" appears only in §1 (scope-guard
  reasoning) and §19/§20 (operator-decision + research framing), each clearly marked. ✅
- **cdiscourse-doctrine — score never blocks posting:** PM-003 ships no gating logic. The research
  doc reiterates (§3, §4, §5) that only structural conditions block; score/advisories never do. ✅
- **cdiscourse-doctrine — no service-role / no AI calls / no secrets:** PM-003 is documentation
  only. No code, no Edge Function, no provider call, no `.env` touch. The research doc's §10/§18
  reiterate that RULE-006 ships zero production AI calls and its implementation is a separate
  operator-gated card. ✅
- **cdiscourse-doctrine — all 10 rules reiterated:** §1 of the research doc lists all 10 rules
  verbatim and walks through where the wave brushes against the v1 scope guards. ✅
- **cdiscourse-doctrine — plain language / supabase conventions / rules engine sacred:** no code
  is written, so no surface can violate these; the research doc nonetheless restates them as the
  constraints the child cards inherit. ✅

None — pure documentation change. No `supabase db push`, no function deploy, no env var.
