## §1 — Goal & scope

Give a signed-in user a small, safe, in-app surface to update their
contact information — display name (which drives the new initials
avatar) and email — without leaving the Account screen. At the same
time, deprecate PR-003's uploaded-avatar pipeline (storage bucket +
RLS + Edge Function + four `profiles` columns + image picker dep)
because the operator chose the simpler initials-based identity glyph as
the v1 avatar story. The result: the Profile epic closes for v1 with
one fewer dep, one fewer Edge Function, one fewer bucket, four fewer
columns, ~195 fewer tests on the picker path, and a clean two-field
contact-info surface that respects every doctrine constraint.

Doctrine constraints (`cdiscourse-doctrine`):
- §1 (score / standing): contact info has zero effect on score, heat,
  truth bands, or any validation gate. A display name change does not
  earn a point. An email change does not move standing.
- §4 (AI moderator limits): no AI call. No moderation of contact info
  in v1.
- §5 (rules engine): not touched.
- §6 (secrets): no new secret. `supabase.auth.updateUser` is the
  client-side auth SDK — it carries the user's own JWT, never a
  service-role key.
- §7 (no AI calls from prod app): not violated.
- §8 (Supabase conventions): migration 17 is **append-only** (a new
  file, not an edit of 16). RLS stays on for every table. The narrowed
  UPDATE policy from migration 16 is dropped and the **original**
  unrestricted-by-avatar-columns policy from `20260516000002_rls_policies.sql`
  is restored — net: profiles UPDATE is back to its pre-PR-003 shape.
- §9 (plain language): no internal code surfaces. `verification_pending`
  is mapped to "Check your inbox to confirm the new email." Validation
  errors map to plain copy.
- §10 (v1 scope guards): no voting, no OAuth, no public API, no push
  notifications, no real-time collaborative editing.

`expo-rn-patterns`: PR-004 REMOVES one dep (`expo-image-picker`),
adds NONE. No new RN library. The initials avatar is pure RN primitives
(`<View>` + `<Text>`) — already shipped as `GeneratedAvatar`.

`accessibility-targets`: every interactive element keeps 44 px hit
target + role + label + state + live region for verification-pending
status. The initials avatar exposes
`accessibilityRole="image"` + `accessibilityLabel="Initials avatar
showing {initials}"`. Color contrast on the initials background is
deterministic and already verified by `userPreferencesModel.test.ts`
line 391 (the 8-color palette is hand-picked for white-text contrast).

`supabase-edge-contract`: migration 17 follows OPS-001 four-class
discipline; the deprecation pattern (drop bucket policies + drop
columns + drop narrowed policy + restore original) is documented
inline. The migration does NOT use any `COMMENT ON POLICY ... ON
storage.*` statement (the 2026-05-24 PR-003 deploy chain lesson). No
new Edge Function. The existing `upload-avatar` function is deleted
both at the filesystem layer (operator unlinks via `npx supabase
functions delete upload-avatar --linked` — see §17) and at the
`config.toml` layer.

`test-discipline`: the deletion removes ~195 tests; PR-004 adds
~80 new tests (`contactApi.test.ts`, `contactInfoSection.test.tsx`,
`contactDoctrine.test.ts`, `deprecateAvatarMigration.test.ts`).
Net test count: roughly **-115**, which is the only roadmap card in
the project that net-removes tests. This is doctrine-compliant because
the deletion explicitly removes a withdrawn feature (`test-discipline`:
"unless a card explicitly removes tests with a documented reason" —
PR-004's reason is the avatar pivot, documented in §0 and
known-blockers).

## §3 — Problem statement

After PR-003, a CDiscourse user can upload a profile photo but cannot
update their contact information (display name edit exists; email is
display-only). The avatar upload pipeline introduced cost (a storage
bucket, an Edge Function, four DB columns, an RN dep, ~195 tests, and
the open question of who moderates uploaded images) that the operator
judged disproportionate for v1.

Two problems to close:

1. **Avatar overhead.** The PR-003 pipeline is unrequired surface area
   that complicates the v1 release. The deterministic initials avatar
   (already shipped as `GeneratedAvatar` in PR-001) is a stable
   identity glyph that needs no storage, no Edge Function, and no
   moderation question. Keeping both paths confuses the design.
2. **Contact-info gap.** The Account screen exposes email read-only
   with no "change" affordance. The card body's acceptance criterion
   ("Email update flow uses Supabase auth update") is unmet. A user
   who needs to change their email today must contact an admin.

PR-004 closes both. The deprecation removes the picker pipeline; the
new `ContactInfoSection` adds the missing contact-info edit surface
with the correct Supabase-auth-driven email flow.

## §4 — Data model + file changes

### Data model (DB)

**No new tables. No new columns. No new RLS policies.** PR-004 is a
**reverse-data-model** card: migration 17 deletes the four PR-003
columns from `public.profiles` and restores the pre-PR-003 UPDATE
policy shape.

`auth.users` (managed by Supabase, untouched by app migrations) holds
the email. `auth.users.email` is the source of truth for the user's
contact email; reads use `supabase.auth.getUser()`; writes use
`supabase.auth.updateUser({ email })`.

After PR-004 applies, `public.profiles` columns are once again:
`id`, `display_name`, `role`, `created_at` — exactly the pre-PR-003
shape from `20260516000001_initial_schema.sql` lines 25–31.

```ts
export type ContactEmailChangeError =
  | 'config_missing'
  | 'no_session'
  | 'invalid_email'
  | 'same_as_current'
  | 'email_already_used'
  | 'rate_limited'
  | 'network_error'
  | 'unknown';

export interface PendingEmailChange {
  /** The email the user submitted (NEW address — not yet verified). */
  newEmail: string;
  /** Local-only timestamp. Display "Verification pending — {newEmail}". */
  submittedAt: string;
}

export type ContactApiResult<T = void> =
  | { ok: true; data: T }
  | { ok: false; error: ContactEmailChangeError; message: string };

/** Pure (testable without Supabase): valid email shape + min length. */
export function validateEmail(input: string): { ok: true } | { ok: false; error: 'invalid_email' };

/** Pure (testable without Supabase): plain-language map for the error code. */
export function messageForContactError(code: ContactEmailChangeError): string;

/** Calls supabase.auth.updateUser({ email }). The session keeps the OLD
 *  email until the user clicks the verification link in the NEW
 *  inbox AND the auth session refreshes. */
export async function requestEmailChange(newEmail: string): Promise<ContactApiResult<PendingEmailChange>>;
```

## §6 — Q2: Validation pattern per field

**Verdict: PASS — inline helpers, no library.**

### Display name validation

- **Min length:** 1 char after trim (existing precedent — `AccountScreen.tsx`
  line 48 `if (!draftName.trim()) return;`).
- **Max length:** 60 chars (existing precedent — `PreferencesPopout.tsx`
  line 84 `DISPLAY_NAME_MAX = 60`; `accountApi.ts` does NOT enforce a
  max but the existing UI `maxLength={60}` does).
- **Trim:** whitespace trimmed before write (existing precedent —
  `accountApi.buildProfileUpdatePayload` line 14 `display_name: input.displayName.trim()`).
- **Initials extraction edge case:** 1-char names produce 1-char
  initials (e.g. `'A'` → `'A'`). 2-char names produce 2-char initials
  (`'AB'` → `'AB'`). This is the existing `deriveAvatarInitials`
  behaviour (`GeneratedAvatar.tsx` lines 34–42). No new lower bound
  needed; the pivot accepts that very-short display names still
  render legibly.
- **Unicode handling:** see §12 edge cases. The existing
  `deriveAvatarInitials` splits on `\s+` which works correctly for
  CJK names without spaces (the whole string is one "word"; first 2
  characters are taken via `.slice(0, 2)`).
- **Error display:** **inline below the field**, plain copy. Existing
  precedent — `AvatarUploadSection.tsx` line 283 `errorText` with
  `accessibilityLiveRegion="polite"`; `AccountScreen.tsx` line 134
  `saveError`. PR-004 follows the same pattern.

### Email validation

- **Shape:** the existing `validateAuthInput` helper in `authApi.ts`
  line 22 uses `email.includes('@') && email.length >= 5`. PR-004
  exports a tighter pure helper `validateEmail` in `contactApi.ts`:

  ```ts
  export function validateEmail(input: string): { ok: true } | { ok: false; error: 'invalid_email' } {
    const e = (typeof input === 'string' ? input : '').trim();
    if (e.length < 5) return { ok: false, error: 'invalid_email' };
    if (e.length > 254) return { ok: false, error: 'invalid_email' };  // RFC 5321 hard limit
    // Defensive minimum-viable check (NOT a full RFC parser):
    if (!/^[^@\s]+@[^@\s]+\.[^@\s]+$/.test(e)) return { ok: false, error: 'invalid_email' };
    return { ok: true };
  }
  ```

  No regex library; one inline regex. Supabase's auth backend
  validates the email shape with its own (more permissive) check on
  the server — client validation is UX, server validation is the trust
  boundary.

- **Same-as-current check:** the wrapper short-circuits with a
  `same_as_current` error before calling `updateUser`. This avoids a
  no-op verification email and a confusing "Verification pending" for
  the user's existing address.

- **Email-already-used:** mapped from the Supabase auth SDK's error
  message via a substring check (`'already registered'` or
  `'already in use'`). Existing precedent — `authApi.ts` line 36 maps
  the same substring for signup.

- **Rate limited:** mapped from `'rate limit'` substring. Existing
  precedent — none in `authApi.ts` but the `updateUser` SDK can
  return rate-limit errors. The plain copy is "Too many email change
  attempts. Try again in a few minutes."

- **Error display:** inline below the email field with
  `accessibilityLiveRegion="polite"`. Same pattern as display-name.

### Why no validation library

The existing repo uses pure-TS helpers + `zod` (`package.json`
line 89), but Zod for two-field validation is overkill. Pure
helpers stay testable without a runtime schema and match the
established `validateAuthInput` precedent.

### Field-level error placement (consistent with `accessibility-targets`)

- Below the input, separate `<Text>` element with `accessibilityLiveRegion="polite"`.
- Test (`contactInfoSection.test.tsx`): the error `<Text>` is mounted
  immediately after the input in the accessibility tree.
- Summary error block at the top of the section: **NO**. Single-field
  errors don't need a summary. If a future card adds multi-field
  forms (e.g. phone + location), a summary may be warranted.

## §7 — Q3: Propagation strategy for identity changes

**Verdict: PASS — none in v1.**

When the user updates their display name, the change is local-first:
the new value writes to `public.profiles.display_name`; the
`ContactInfoSection` re-renders with the new name; the
`useAccountProfile.refresh()` (already in PR-003's plumbing for the
avatar) refetches the row; the initials avatar updates because it
reads `displayName` from the same `UserProfile` slice.

**Other surfaces that display the user's name (admin views, gallery
cards via author lookup, etc.) refresh on their next navigation /
refetch.** PR-004 does NOT wire a realtime propagation channel.

### Why not realtime

META-1B established the realtime pattern (postgres-changes
subscription, channel topic per debate) for **per-room** state. A
per-user "display name changed" channel would be a global subscription
that every authenticated client opens for every other user — way out
of v1 scope. The user updates their own name; the user's own UI
reflects it instantly. Stale display names on other users' screens
(seconds-to-minutes old) are an acceptable v1 trade-off.

### Why not polling

There is no user-pull "what is X's current display name" need in v1.
Display names appear at navigation time (which fetches fresh data) or
when the realtime point-tag pipeline (META-1B) brings in a tag with
the user's name in its label. The point-tag system writes the label
at apply time, so the label is a snapshot; subsequent display-name
changes do not retroactively update past tag labels (this is by
design — a tag is an audit-traced action; the labeller's name at the
moment is the historical fact).

### Why not lazy refresh

Lazy refresh ("refetch on focus") is what the existing
`useAccountProfile` already does indirectly via its `useEffect`-on-userId
load. PR-004 does not add a focus listener. The implicit lazy refresh
on screen navigation is sufficient.

### Email-change propagation

`auth.users.email` is the canonical source. Other surfaces typically
do not display the user's email (it is private — see Q4). The only
exception is the user's own `ContactInfoSection`, which reads it via
`supabase.auth.getUser()` and the `onAuthStateChange` subscription
already wired in `authApi.ts` line 195. When the user clicks the
verification link in the NEW inbox and returns to the app, the
session refresh triggers `onAuthStateChange` → the Section calls
`getUser()` → the new email displays. No additional plumbing.

### What an implementer might be tempted to do that PR-004 does NOT do

- ❌ Subscribe to `public.profiles` realtime to push display-name
  changes to other clients. (Out of scope; future card if requested.)
- ❌ Add an `onFocus` listener to refetch on screen wake. (Existing
  navigation refetch is sufficient.)
- ❌ Refetch every user's display name on every gallery render. (The
  gallery already caches profiles; PR-004 doesn't change that.)
- ❌ Add a "your display name is now X — broadcasting to your active
  rooms" announcement. (Over-engineered; nothing announces it today
  and the operator has not requested it.)

## §8 — Q4: Per-field read authorization

**Verdict: PASS — existing policies cover both fields.**

| Field | Read by | RLS policy enforcing it | Where it lives |
|---|---|---|---|
| `display_name` | Any authenticated caller (own + others) | `"profiles: authenticated can read all"` (SELECT, USING (true)) — `20260516000002_rls_policies.sql` lines 56–59 | `public.profiles.display_name` |
| `email` | Only the owning caller | `auth.users` is managed by Supabase; only the calling user's own row is exposed via `supabase.auth.getUser()` — never via `public.profiles`. No app-level RLS needed; Supabase auth handles it. | `auth.users.email` |

### Why display_name read is public-to-authenticated

The existing policy lets any signed-in user read every profile's
display name (and `role` and `created_at`). This is needed for: showing
author names on argument bubbles, gallery cards, audit logs, admin
tabs, etc. PR-004 changes nothing about this read surface.

### Why email read is private

`auth.users` is intentionally not exposed via `public.profiles` (the
`public.profiles` table has no `email` column). The user's own
`supabase.auth.getUser()` returns their own email; another user's
email is not reachable through any client surface. This is the
existing default — PR-004 changes nothing.

**If a v2 feature ever needs "user X's email"** (e.g. an admin
moderator-contact flow), that feature would be a service-role-bound
Edge Function that returns the email only after an admin-authority
check (mirroring `request-argument-deletion`'s admin-email-redaction
discipline). PR-004 explicitly does NOT introduce such a surface.

### What the OPS-001 four-class checklist demands

PR-004 ships a migration but the migration only **DROPS** policies +
columns. It does not CREATE any new policy or column. The four-class
checklist still applies but is mostly vacuous:

- **Class 1** (ambiguous column references in subqueries): the restored
  policy uses `id = auth.uid()` which is fully qualified by table
  context (`ON public.profiles FOR UPDATE`). No subqueries.
- **Class 2** (column type mismatches): no new columns. The
  `DROP COLUMN` statements act on existing types.
- **Class 3** (implicit ordering dependencies): the order is
  `drop storage SELECT policy → drop 4 columns → drop narrowed UPDATE
  policy → restore original UPDATE policy`. No statement depends on a
  later one. The bucket itself is **not** dropped (the bucket can stay
  empty in `storage.buckets`; deleting it requires
  `supabase_storage_admin` ownership per the same lesson that bit
  PR-003's `COMMENT ON POLICY` — see §10).
- **Class 4** (function / trigger / extension dependencies): the
  restored UPDATE policy uses `is_moderator_or_admin()` (existing
  helper from migration 02 line 17). No new function. No new trigger.
  No new extension.

Full migration spec is in §10.

```sql
--     stays — it is used by other tables).
--
-- Storage ownership lesson (verbatim apply of the 2026-05-24
-- known-blockers entry "PR-003 Storage Schema Comment Ownership"):
--   - `storage.objects` is owned by supabase_storage_admin, not by
--     the standard migration role. The standard migration role can
--     CREATE POLICY and DROP POLICY on storage.objects (this is
--     granted by Supabase's storage RLS setup), but it cannot
--     COMMENT ON POLICY ... ON storage.*. This migration includes
--     NO `COMMENT ON POLICY ... ON storage.*` statement. The reason
--     for each drop is captured in inline `--` SQL comments instead.
--
-- Why we do not drop the storage bucket:
--   - `delete from storage.buckets` requires `supabase_storage_admin`
--     ownership for the same reason `COMMENT ON POLICY ON storage.*`
--     does. Attempting `delete from storage.buckets where id =
--     'profile-avatars'` from a migration may fail with the same
--     SQLSTATE 42501 insufficient_privilege error.
--   - The bucket itself has zero data after this migration applies
--     (PR-003 never reached production with real user uploads
--     because the deploy chain only just stabilized on 2026-05-24;
--     any test uploads are operator-cleanable via the Supabase
--     dashboard).
--   - Keeping the empty bucket is doctrine-safe: the storage SELECT
--     policy is dropped, so no caller can list its contents; the
--     four `profiles` columns are dropped, so no path string can
--     reference it; the `[functions.upload-avatar]` config block
--     is removed, so the Edge Function is unreachable.
--   - **If the operator wants the bucket gone**, they can delete it
--     via the Supabase dashboard (Storage → profile-avatars → ...)
--     which runs under the dashboard's elevated session and bypasses
--     the privilege boundary. Documented in §17 Operator steps.
--
-- Doctrine encoded by this migration:
--   - The four avatar columns on public.profiles are GONE.
--     Anything that referenced them in client code is also gone (the
--     deletion list in §11 enumerates them).
--   - The profiles UPDATE policy is restored to the pre-PR-003 shape.
--     A user can update their own row (display_name in practice;
--     no other client-writable columns exist on profiles); a
--     moderator/admin can update any row.
--   - The storage bucket policies are gone — no caller can list the
--     bucket. Existing files in the bucket (if any) become
--     unreachable from the client and are operator-cleanable.
--
-- Companion design: docs/designs/PR-004.md §4 + §10.
-- ============================================================
```

### Statement 1 — drop storage SELECT policy

```sql
-- The bucket itself is intentionally not dropped (see header). Only
-- the SELECT policy is removed so no caller can list its contents.
DROP POLICY IF EXISTS "profile-avatars: anyone can read" ON storage.objects;
```

### Statement 2 — drop the four avatar columns

```sql
ALTER TABLE public.profiles
  DROP COLUMN IF EXISTS avatar_path,
  DROP COLUMN IF EXISTS avatar_thumb_path,
  DROP COLUMN IF EXISTS avatar_updated_at,
  DROP COLUMN IF EXISTS avatar_moderation_status;
```

The CHECK constraint on `avatar_moderation_status` is dropped
automatically when the column is dropped. The `IF EXISTS` is safety
in case a future fresh-database apply executes this migration before
16 (impossible in practice — `db reset` runs in numeric order — but
the guard is cheap).

### Statement 3 — drop the narrowed UPDATE policy

```sql
DROP POLICY IF EXISTS "profiles: users update own — narrow" ON public.profiles;
```

### Statement 4 — restore the original UPDATE policy

```sql
-- This is a byte-for-byte restoration of the original policy from
-- 20260516000002_rls_policies.sql lines 69–73. The
-- deprecateAvatarMigration.test.ts assertion (§13) verifies the
-- restored body matches the original.
CREATE POLICY "profiles: users update own; mods update any"
ON public.profiles FOR UPDATE
TO authenticated
USING (id = auth.uid() OR is_moderator_or_admin())
WITH CHECK (id = auth.uid() OR is_moderator_or_admin());
```

### What this migration does NOT do

- ❌ `DROP BUCKET profile-avatars` / `delete from storage.buckets` —
  ownership boundary (see header).
- ❌ Any `COMMENT ON POLICY ... ON storage.*` statement — the
  2026-05-24 PR-003 lesson.
- ❌ Drop the `is_moderator_or_admin()` function — used by every other
  RLS policy.
- ❌ Drop the `handle_new_user()` trigger (line 53 of migration 01) —
  unrelated to avatars.
- ❌ Modify `auth.users` — not the migration's surface.

### Rollback considerations

If the operator needs to revert PR-004 (re-enable the avatar
pipeline), the path is a **new** migration that re-creates the four
columns + the bucket SELECT policy + the narrowed UPDATE policy. Not
this migration. Doctrine §8 (append-only) forbids editing this
migration to roll back.

Per OPS-001 the migration is reviewed by either (a) `npx supabase db
reset --linked=false` (when Docker is available) or (b) heightened
textual review against the four classes. The header walks all four
classes inline so option (b) is straightforward.

### Operator deploy command

`npx supabase db push --linked` — applies migration 17.

Followed by `npx supabase functions delete upload-avatar --linked` —
removes the deployed Edge Function (see §17). The function is also
deleted at the filesystem layer by PR-004's source-deletion pass.

## §12 — Edge cases

| Case | Handling |
|---|---|
| **Display name with only emoji** (e.g. `'😀😀😀'`) | `deriveAvatarInitials` returns `'😀😀'.slice(0,2).toUpperCase()` = first emoji surrogate pair. `.toUpperCase()` is identity for emoji. The initials avatar renders the emoji on the hashed background. Test asserts no `'?'` fallback (the string is non-empty after trim). |
| **Single-char display name** | `'A'` → initials `'A'`. `'Z'` → initials `'Z'`. Avatar circle renders one char centered. Test asserts behaviour. |
| **Very long display name** | `DISPLAY_NAME_MAX = 60` (existing). Names > 60 chars are truncated upstream at input time. Initials extraction still gets only first/last word slices, so longer-than-60 names handled gracefully even if upstream limit ever changes. Test defensively asserts behaviour with a 80-char string. |
| **Leading / trailing whitespace** | `deriveAvatarInitials` trims (line 36). The `accountApi.buildProfileUpdatePayload` ALSO trims (line 14). The persisted value is trimmed; the initials always derive from a clean string. |
| **RTL languages** (Arabic, Hebrew) | `'محمد'` → `'محمد'.slice(0,2)` returns `'مح'` correctly. RN renders RTL text within `<Text>` directly. Tested. |
| **CJK names without spaces** | `'李明'` is one "word" per `\s+` split; `.slice(0,2)` returns `'李明'`. Tested. |
| **CJK names with spaces** | `'李 明'` is two words; `first[0] + last[0]` = `'李明'`. Tested. |
| **Mixed-script names** | `'李 Lovelace'` → `'李L'` (one CJK + one Latin). Renders fine. Tested. |
| **Display name = null** | `deriveAvatarInitials(null)` returns `'?'`. Tested. |
| **Display name = empty string** | `deriveAvatarInitials('')` → `trimmed = ''` → returns `'?'`. Tested. |
| **Display name = whitespace only** | `deriveAvatarInitials('   ')` → returns `'?'`. Tested. |
| **Email already in use** | `supabase.auth.updateUser` returns an error with `'already in use'` or `'already registered'` substring; `contactApi.requestEmailChange` maps to `'email_already_used'` → plain copy "Another account is already using that email." UI reverts to the prior email; pending state never enters. |
| **Email is rate-limited** | Supabase returns `'rate limit'` substring → mapped to `'rate_limited'` → plain copy "Too many email change attempts. Try again in a few minutes." UI reverts. |
| **Email same as current** | `contactApi.requestEmailChange` short-circuits before calling Supabase: comparison after `.trim()` and case-insensitive. Returns `'same_as_current'` → plain copy "That's already your email address." UI does not enter pending state. |
| **Email shape invalid** | `validateEmail` rejects client-side before Supabase call. Plain copy "Enter a valid email address." |
| **Network error during email change** | Supabase SDK throws / returns network error → mapped to `'network_error'` → plain copy "Network error. Check your connection." UI reverts; the pending state was never set (we only set it on success). |
| **User cancels pending verification** | UI offers a "Cancel pending change" affordance after submission. Cancel clears local pending state. The verification email has already been sent (Supabase-side); we cannot un-send it. If the user clicks the link in the email AFTER cancelling local state, `onAuthStateChange` still fires and the new email becomes the session email — local UI displays the new email correctly. This is acceptable: the "cancel" only cancels the LOCAL pending UI indicator; the auth state is the truth. Copy: "Cancel pending change" + helper "If you already clicked the verification link in your inbox, the email change still applies." |
| **Verification link click in a different device** | The auth session in the device where the user submitted DOES NOT auto-refresh until the user returns to the app and `getUser()` is called fresh. The verify link itself updates `auth.users.email`; the next `getUser()` returns the new email. Acceptable v1 behaviour. |
| **`supabase.auth.getUser()` returns no email post-verify** | The pending state stays visible until the user returns to the screen and the auth listener fires. If the listener never fires (e.g. background tab), the next navigation refetches. The pending state has no time-out in v1 (a future card can add one). |
| **Display name save while email pending** | Independent paths — display name writes to `profiles`; email writes to `auth.users`. Both can be in-flight concurrently. UI shows both spinners. No interaction. |
| **Two clients on the same account** | If a user changes email on device A and display name on device B simultaneously: both writes succeed (different surfaces); device A's display name refresh will fetch the new value on its next navigation. No conflict. |
| **Email change while signed out** | `getAccessToken()`-equivalent path returns no session → `requestEmailChange` returns `'no_session'` → plain copy "Please sign in again." This should not happen in practice (the section only renders when signed in) but defensive. |
| **PreferencesPopout still imports GeneratedAvatar from old path** | Back-compat shim re-exports from new path. Existing tests + behaviour unchanged. |
| **A future card adds a `profiles.email` column** | Doctrine §10 says no — `auth.users.email` is the source of truth. If a future card needs email caching, it goes through an Edge Function under admin authority. Out of scope for PR-004. |
| **Migration 17 runs against a database without migration 16** | `IF EXISTS` clauses make the column drops no-ops. The narrowed-policy drop is a no-op (`IF EXISTS`). The storage SELECT policy drop is a no-op (`IF EXISTS`). The recreate-original-policy may fail if the original policy ALREADY exists (idempotency hole). **Mitigation:** the restored policy statement should use `DROP POLICY IF EXISTS ... ` immediately before `CREATE POLICY ...` to make the migration idempotent. Spec'd in §10 statement 4. |
| **Pending email change persists across app reloads** | The pending state is component-local (not AsyncStorage-persisted). A reload drops the pending UI; `getUser()` returns the OLD email until verification fires. This is acceptable v1 behaviour — pending state is a UI affordance, not a data invariant. If the operator wants persistence, that is a future card. |
| **Initials avatar background palette collision** | Two users with similar hashes could share the same color. That's fine — the initials differ, and the consumer typically also displays the display name nearby. No new test required (`hashAvatarSeed` collision behaviour is already characterized). |

```sql
select column_name
from information_schema.columns
where table_schema = 'public'
  and table_name = 'profiles'
  and column_name like 'avatar_%';
-- Expect: empty result.

-- Sanity: confirm the restored UPDATE policy exists and the narrowed one does not
select polname
from pg_policy
where polrelid = 'public.profiles'::regclass
  and polcmd = 'w';
-- Expect: 'profiles: users update own; mods update any'
--         'profiles: users can insert own' (existing INSERT — not UPDATE)

-- Sanity: confirm the storage SELECT policy is gone
select polname
from pg_policy
where polrelid = 'storage.objects'::regclass
  and polname like 'profile-avatars:%';
-- Expect: empty result.
```

```
npx supabase functions delete upload-avatar --linked
```

Removes the deployed function from the Supabase project. The

### Phase 3 — Storage bucket cleanup (optional, dashboard-only)

no `[functions.upload-avatar]` block points at it).

### Phase 4 — Smoke verification

1. Sign in as a test user → Account screen → confirm:
   - The initials avatar shows the user's initials (or `?` if empty).
   - The display-name edit affordance is present and works.
   - The email row shows the current email + a "Change email" affordance.
2. Click "Change email" → enter a new email → submit → confirm:
   - The "Verification pending — {newEmail}" badge appears.
   - An email arrives at the new address with a verification link.
3. Click the verification link in the inbox → return to the app →
   confirm the displayed email updates.
4. Cancel a pending change (without clicking the verify link) →
   confirm:
   - The pending badge clears.
   - The OLD email is still displayed (correct — the cancel is UI-only).
5. Negative smoke: submit an invalid email shape ("xyz"), an

If PR-004 introduces a regression and rollback is needed:

No operator should attempt to roll back migration 17 by
hand-editing the SQL or by `truncate pg_migrations` — both violate
