## Goal (one paragraph)

MCP-015 builds the **semantic override UX model** that MCP-010 §5.2 designed: the pure-TypeScript state model and test fixtures that let a participant reverse an *uncertain or conflicting* semantic routing suggestion without ever feeling caged. The semantic referee (MCP-011's `SemanticRefereePacket`) is advisory — `authoritative: false`, always. When a routing-relevant binary's `confidence` is `'low'`, or when MCP-013's reconciliation produced a `conflict_routed` reading and `LedgerResult.needsUserChoice` is `true`, the referee deliberately does **not** act silently — it surfaces a small, reversible choice and lets the **user**, not the model, decide where the move goes. The doctrine that shapes every decision (`cdiscourse-doctrine` §1, §4; MCP-010 §3): a semantic suggestion is never a hard gate — the move is already posted, the user can always proceed; overriding is **never a penalty** — no score cost, no flag, no "you were warned" copy; the override is recorded as **visible, append-only, META-001-style metadata**, never hidden state; no verdict / person-label copy appears on any override surface. MCP-015 delivers a pure-TS model first (the trigger model, the record-builder, the in-memory repeated-override signal); the React choice surface is **deferred to a scoped follow-up** (see §8) — the issue says it "may be a scoped follow-up if needed", and the pure model is independently testable and is what BR-003 / BR-004 and MCP-013 consume. MCP-015 emits a lane *choice* (`mainline` / `branch` / `tangent`) plus an optional "this answers the parent" assertion; that choice is an advisory input — MCP-015 does **not** route branches, score moves, or call any provider.

## Data model

All shapes are pure TypeScript, JSON-serializable, immutable (`Readonly` / `readonly` where structural), no `Date` objects, no functions on the records, no class instances, no React types — the same constraint as `src/lib/constitution/engine.ts`, `src/features/refereeLedger/types.ts`, and the META-001 ledger.

### D.1 The `MetadataEvent.codeFamily` extension (additive, in META-001's file)

META-001's `MetadataEvent` lives in `src/features/metadata/moveMetadataLedger.ts`. Its current `codeFamily` is a closed three-member union:

```ts
codeFamily: 'manual_tag' | 'auto_metadata' | 'lifecycle_causation';
```

MCP-015 adds **exactly one** member — `'semantic_override'`:

```ts
// src/features/metadata/moveMetadataLedger.ts — MetadataEvent.codeFamily (one new member)
codeFamily: 'manual_tag' | 'auto_metadata' | 'lifecycle_causation' | 'semantic_override';
```

This is the **only** change MCP-015 makes outside `src/features/semanticOverride/` and `gameCopy.ts`. It is purely additive — no member removed, no field added, no field removed. The `MetadataEvent` interface keeps **all** its existing fields and gains **none**: it still has `eventId`, `kind`, `codeFamily`, `code`, `messageId`, `clusterId`, `at`, `cause?`. Critically, `MetadataEvent` already has **no** `delta` / `score` / `penalty` / `block` / `flag` field, and MCP-015 adds none — the no-penalty acceptance criterion holds for the event by construction.

For a `semantic_override` event:
- `kind` is **always** `'add'` — an override is an additive, append-only record. There is no `'remove'`; a user who changes their mind appends a *new* event (see Edge cases).
- `codeFamily` is `'semantic_override'`.
- `code` is the chosen lane or assertion: one of `mainline` / `branch` / `tangent` / `answers_parent`. All four already map through `gameCopy` — `mainline`, `branch`, `tangent` need new entries; the lane labels are added in the copy section.
- `messageId` / `clusterId` anchor the override to the move, exactly as a manual-tag event does.
- `at` is the caller-supplied ISO-8601 timestamp of the user's tap.
- `cause` carries an internal debug string (`override_low_confidence` / `override_l1_l2_conflict`) — never rendered (matches META-001's `cause` convention, "internal debug field … never rendered").

**Note on a real codebase difference from MCP-010.** MCP-010 §4.1 quotes a `MetadataEvent` whose `codeFamily` it described as a four-member-after-extension union. The merged META-001 file confirms the current union is the three members above; this design's extension reproduces MCP-010's intent exactly against the *actual* merged file. No deviation — just pinned to the real source.

### D.4 `SemanticOverrideRecord` — the per-move append-only record

The `MetadataEvent` is the *log* entry. The per-move *record* the choice surface produces and the sidecar reads is richer — it carries the original suggestion so the audit trail shows what was reversed.

```ts
// src/features/semanticOverride/types.ts
import type { SemanticRouteSuggestion } from '../semanticReferee/semanticRefereeTypes';

/**
 * One append-only override record attached to one move. Visible metadata —
 * never hidden state. The MetadataEvent (D.1) is the log entry; this is the
 * fuller record the sidecar renders.
 *
 * DOCTRINE — has NO `delta`, NO `score`, NO `penalty`, NO `block`, NO `flag`
 * field. Overriding is never a penalty. Enforced by
 * `__tests__/semanticOverrideNoPenalty.test.ts`.
 */
export interface SemanticOverrideRecord {
  /** Stable id — `semantic_override:${messageId}:${at}`. */
  overrideId: string;
  messageId: string;
  clusterId: string;
  /** The lane the user picked. */
  chosenLane: SemanticOverrideLane;
  /**
   * True when the user explicitly asserted "this answers the parent" to
   * overturn a `responds_to_parent: 0`. Independent of `chosenLane` — a user
   * can assert "answers the parent" AND keep the move on the mainline.
   */
  assertsAnswersParent: boolean;
  /**
   * What the referee suggested before the override — recorded so the audit
   * trail shows what was reversed. Plain-language-mapped before render.
   */
  originalRouteSuggestion: SemanticRouteSuggestion;
  /** Why the choice was offered. */
  triggerReason: SemanticOverrideTriggerReason;
  /**
   * The classifier id family that was contested (e.g. 'responds_to_parent',
   * 'suggests_side_branch'). Internal — mapped through gameCopy before render;
   * never echoed raw to a user surface.
   */
  contestedClassifierId: string;
  /** ISO-8601 — caller-supplied tap time. */
  at: string;
  /** The viewer who made the override (current viewer id, caller-supplied). */
  overriddenByUserId: string;
  /** Actor role of the overrider. */
  overriddenByActorRole: SemanticOverrideActorRole;
}
```

There is intentionally **no** `userReviewRequired`, no `confidence`, no `outcome`, no economy field — MCP-015 produces a *record of a user choice*; it never scores and never re-derives a ledger reading. MCP-013 *consumes* this record (MCP-003/MCP-013 own the re-read); MCP-015 does not.

### D.5 `SemanticOverridePrompt` — the trigger model output

```ts
// src/features/semanticOverride/types.ts

/**
 * Output of the pure trigger model — decides whether the choice surface
 * appears and what it should offer. Computed from a packet + a LedgerResult
 * + the viewer role + the in-memory RepeatedOverrideSignal.
 */
export interface SemanticOverridePrompt {
  /** True when the choice surface should be shown for this move. */
  shouldOffer: boolean;
  /** Why — drives the produced record's triggerReason + the prompt copy. Null when shouldOffer is false. */
  triggerReason: SemanticOverrideTriggerReason | null;
  /** The referee's current routing suggestion the user may reverse. */
  suggestedLane: SemanticOverrideLane;
  /**
   * True when a `responds_to_parent: 0` binary (or its conflict) is in play,
   * so the "this answers the parent" toggle is offered.
   */
  offersAnswersParentToggle: boolean;
  /** The contested classifier id — recorded onto the record, mapped for copy. Null when shouldOffer is false. */
  contestedClassifierId: string | null;
  /**
   * Plain-language code for the prompt headline — a key of PLAIN_LANGUAGE_COPY.
   * One of `semantic_override_prompt_low_conf` / `semantic_override_prompt_conflict`
   * / `semantic_override_prompt_soft`. Empty string when shouldOffer is false.
   */
  promptCopyCode: string;
}
```

### D.6 `RepeatedOverrideSignal` — in-memory, per-room, UX-only

```ts
// src/features/semanticOverride/types.ts

/**
 * Per-room, in-memory, per-render. NOT persisted. NOT a user-level score.
 * NOT sent anywhere. NOT a model-training input. It exists ONLY so the prompt
 * copy can soften after a user has reversed several suggestions in one room.
 *
 * DOCTRINE — this shape MUST NOT gain a persistence field, a profile field,
 * a cross-room aggregate, or any score field. Enforced by
 * `__tests__/semanticOverrideNoPenalty.test.ts`.
 */
export interface RepeatedOverrideSignal {
  roomId: string;
  /** Count of override records in this room this session. */
  overrideCountThisRoom: number;
  /**
   * True once the count crosses SOFTEN_COPY_THRESHOLD. When true the trigger
   * model picks the quieter `semantic_override_prompt_soft` copy code. A UX
   * constant, not a gate — the surface is never suppressed.
   */
  softenCopy: boolean;
}

/** The count at which the prompt copy softens. A UX constant, not a gate. */
export const SOFTEN_COPY_THRESHOLD = 3;
```

### D.7 No new database table, no migration, no Edge Function

MCP-015 is a pure-TS model. The override record lives in the same render-time, `useState`-owned in-memory metadata ledger META-001 established (META-001: "manual tags live in `useState` and disappear at unmount. Persistence is explicitly v2"). Persisting the override record to Supabase is a future card mirroring META-001's in-memory-v1 / persist-v2 split. No `db push`, no `functions deploy`.

## API / interface contracts

```ts
// src/features/semanticOverride/overrideTriggerModel.ts
import type { SemanticRefereePacket } from '../semanticReferee/semanticRefereeTypes';
import type { LedgerResult } from '../refereeLedger/types';

/**
 * Pure. No network, no React, no Supabase, no mutation. Decides whether the
 * override choice surface appears for one move.
 *
 * - `packet` may be undefined (feature flag off — no packet fetched).
 * - `ledgerResult` may be undefined (the ledger ran l1_only or not at all).
 * - `viewerActorRole` gates the actor rule — observers always get shouldOffer:false.
 */
export function evaluateSemanticOverridePrompt(input: {
  packet?: SemanticRefereePacket;
  ledgerResult?: LedgerResult;
  viewerActorRole: SemanticOverrideActorRole;
  repeatedSignal: RepeatedOverrideSignal;
}): SemanticOverridePrompt;
```

```ts
// src/features/semanticOverride/overrideRecordModel.ts
import type { MetadataEvent } from '../metadata/moveMetadataLedger';
import type { SemanticRouteSuggestion } from '../semanticReferee/semanticRefereeTypes';

/**
 * Pure. Builds the append-only override record from a confirmed user choice.
 * `prompt` is the SemanticOverridePrompt that offered the surface — its
 * triggerReason / contestedClassifierId are copied onto the record.
 */
export function buildSemanticOverrideRecord(input: {
  prompt: SemanticOverridePrompt;
  messageId: string;
  clusterId: string;
  chosenLane: SemanticOverrideLane;
  assertsAnswersParent: boolean;
  originalRouteSuggestion: SemanticRouteSuggestion;
  overriddenByUserId: string;
  overriddenByActorRole: SemanticOverrideActorRole;
  at: string; // ISO-8601, caller-supplied
}): SemanticOverrideRecord;

/**
 * Pure. Projects a SemanticOverrideRecord to a META-001 MetadataEvent.
 * ALWAYS returns `kind: 'add'` and `codeFamily: 'semantic_override'`.
 * `code` is the chosen lane; when `assertsAnswersParent` is true the record
 * additionally projects an `answers_parent` event — see note below.
 */
export function toSemanticOverrideMetadataEvent(record: SemanticOverrideRecord): MetadataEvent;

/**
 * Pure. Returns a NEW signal with overrideCountThisRoom + 1 and softenCopy
 * recomputed against SOFTEN_COPY_THRESHOLD. Never mutates the input.
 */
export function bumpRepeatedOverrideSignal(signal: RepeatedOverrideSignal): RepeatedOverrideSignal;

/** Pure factory — a fresh zeroed signal for a room. */
export function emptyRepeatedOverrideSignal(roomId: string): RepeatedOverrideSignal;

/** The ban-list source the doctrine test scans. Not a content filter. */
export function _forbiddenOverrideTokens(): string[];
```

**`toSemanticOverrideMetadataEvent` — the `answers_parent` projection.** A single confirmed override can carry both a lane *and* an `assertsAnswersParent` assertion. To keep `MetadataEvent` one-`code`-per-event (META-001's shape), `toSemanticOverrideMetadataEvent` returns the **lane** event (`code` = `chosenLane`). When `assertsAnswersParent` is true the caller additionally projects the assertion via a second exported helper:

```ts
/**
 * Pure. When `record.assertsAnswersParent` is true, returns the companion
 * `answers_parent` MetadataEvent (kind 'add', codeFamily 'semantic_override').
 * Returns null when the record does not assert it. The two events share the
 * same `at`; their `eventId`s differ by `code`, so both are append-only and
 * distinct in the log.
 */
export function toAnswersParentMetadataEvent(record: SemanticOverrideRecord): MetadataEvent | null;
```

This keeps the projection 1-event-per-`code` and append-only while still recording both facts. The future component card appends both events (when present) to the in-memory ledger.

Contract rules:
- `evaluateSemanticOverridePrompt`, `buildSemanticOverrideRecord`, `toSemanticOverrideMetadataEvent`, `toAnswersParentMetadataEvent`, `bumpRepeatedOverrideSignal`, `emptyRepeatedOverrideSignal` are **all pure** — no network, no React, no Supabase, no mutation, no `Date.now()` inside the model (the `at` is caller-supplied; `overrideId` / `eventId` are derived from `messageId` + `at`).
- The `SemanticRefereePacket` and `LedgerResult` are consumed **read-only**; MCP-015 never constructs or mutates a packet or a ledger result.
- `SemanticOverrideRecord`, `SemanticOverridePrompt`, `RepeatedOverrideSignal`, and the projected `MetadataEvent`s have **no** `delta` / `score` / `penalty` / `block` / `flag` field.
- The dependency arrow points one way: `semanticOverride/` imports types from `semanticReferee/` and `refereeLedger/` and `metadata/`; none of those import from `semanticOverride/`.

## Trigger algorithm — when an override is OFFERED

`evaluateSemanticOverridePrompt` returns `shouldOffer: true` only when the referee declared uncertainty. The algorithm, in order:

1. **Actor gate.** If `viewerActorRole === 'observer'` → return `{ shouldOffer: false, triggerReason: null, suggestedLane, offersAnswersParentToggle: false, contestedClassifierId: null, promptCopyCode: '' }`. Observers never override (see Actor rule). This gate is first — before any packet inspection.

2. **No-input gate.** If `packet` is undefined **and** `ledgerResult` is undefined → `shouldOffer: false`. Nothing was classified; there is nothing to reverse. (Feature flag off / fetch failed → mock-stub or absent packet.)

3. **L1/L2 conflict check (highest priority).** If `ledgerResult` is present, `ledgerResult.needsUserChoice === true`, and at least one `CategoryReading` has `outcome === 'conflict_routed'` on a **routing-relevant category** (`'continuity'` or `'direct_response'`) → `shouldOffer: true`, `triggerReason: 'l1_l2_conflict'`. The `contestedClassifierId` is derived from that reading's category (`continuity → 'responds_to_parent'`, `direct_response → 'responds_to_parent'`). `offersAnswersParentToggle` is `true` when a `direct_response` or `continuity` reading is the conflict (overturning a "this does not answer the parent" suggestion is exactly what the toggle does).

4. **Low-confidence check.** Else, if `packet` is present and **any** routing-relevant binary has `confidence === 'low'` → `shouldOffer: true`, `triggerReason: 'low_confidence'`. The routing-relevant binary set is:
   - `'responds_to_parent'`
   - `'introduces_new_issue'`
   - `'suggests_side_branch'`
   - `'suggests_diagonal_tangent'`

   `contestedClassifierId` is the `classifierId` of the first such low-confidence binary, scanned in the order the binaries appear in `packet.binaries`. `offersAnswersParentToggle` is `true` when that contested binary is `'responds_to_parent'`.

5. **Confident path.** Else → `shouldOffer: false`. The referee was confident and L1/L2 agreed; the normal route chip behaves as usual and the user can still ignore it. The override surface is the *uncertainty* path, not always-on.

**`suggestedLane` derivation** — independent of `shouldOffer`, always populated (the choice sheet pre-selects it). Map `packet.routeSuggestion` (MCP-011's seven-member `SemanticRouteSuggestion`) onto the three-member `SemanticOverrideLane`:

| `SemanticRouteSuggestion` | `SemanticOverrideLane` |
|---|---|
| `mainline` | `mainline` |
| `no_route_change` | `mainline` |
| `synthesis_lane` | `mainline` |
| `cards_detail` | `mainline` |
| `vertical_chime_branch` | `branch` |
| `diagonal_tangent` | `tangent` |
| `outer_realm` | `tangent` |

When `packet` is absent (the L1/L2-conflict-only path), `suggestedLane` defaults to `mainline`. This mapping is a small named function (`routeSuggestionToLane`) exported for the test suite. **Documented note:** MCP-010 §13 flagged that BR-003 / BR-004 may finalize a wider lane vocabulary; this mapping collapses the seven route suggestions into the three lanes BR-003/BR-004 currently expose. The `SemanticOverrideLane` union is deliberately small so a future card can widen it and the mapping together without a contract break.

**`promptCopyCode` selection:**
- `repeatedSignal.softenCopy === true` → `'semantic_override_prompt_soft'` (wins regardless of trigger reason once the room has crossed the threshold).
- else `triggerReason === 'low_confidence'` → `'semantic_override_prompt_low_conf'`.
- else `triggerReason === 'l1_l2_conflict'` → `'semantic_override_prompt_conflict'`.
- `shouldOffer === false` → `''`.

The function reads **only** the packet confidence, the ledger conflict signal, the viewer role, and the in-memory signal. It reads **no** engagement, view count, heat band, or popularity — and it triggers **no** AI call (the packet is already classified; MCP-015 consumes it read-only).

## Override record model — append-only, no penalty

`buildSemanticOverrideRecord` is a pure projection of a confirmed user choice:
- `overrideId` = `` `semantic_override:${messageId}:${at}` ``.
- `chosenLane`, `assertsAnswersParent` come straight from the confirmed choice.
- `originalRouteSuggestion` is the referee's pre-override suggestion (preserved for audit).
- `triggerReason`, `contestedClassifierId` are copied from the `SemanticOverridePrompt` that offered the surface. (`contestedClassifierId` falls back to `''` if the prompt's was `null` — a defensive guard; in practice a confirmed override always came from a `shouldOffer: true` prompt.)
- `at`, `overriddenByUserId`, `overriddenByActorRole` are caller-supplied.

The record carries **no** score, delta, penalty, flag, or block field — the no-penalty doctrine holds by construction. There is no "you ignored the suggestion" field, no negative marker, no moderation linkage.

**Append-only.** Every confirmed override produces a new `SemanticOverrideRecord` and a new `MetadataEvent` (`kind: 'add'`). A user who changes their mind appends a *new* record — earlier records are **never** deleted or mutated. The latest `at` wins for display and for what MCP-013 / BR-004 consume. `toSemanticOverrideMetadataEvent` always emits `kind: 'add'`; there is no remove path.

**`bumpRepeatedOverrideSignal`** returns a new signal: `overrideCountThisRoom + 1`, `softenCopy = (overrideCountThisRoom + 1) >= SOFTEN_COPY_THRESHOLD`. It produces **no** persisted value, **no** profile field, **no** score, **no** cross-room aggregate. It is in-memory, per-room, per-session. It only feeds the prompt copy code.

## Actor rule — who can override

Verified against the MCP-015 issue acceptance criterion ("Observers are not offered the override surface"), MCP-010 §11 ("Observer viewing the move … The override surface is **not offered to observers**"), and META-001's `MANUAL_TAG_ELIGIBILITY` precedent ("Observers may NEVER apply manual tags in v1").

| Viewer role | Offered the override surface? | Rationale |
|---|---|---|
| `participant_affirmative` | **Yes** | A participant decides where their / the room's move is routed. |
| `participant_negative` | **Yes** | Same. |
| `admin` | **Yes** | Admin retains the gameplay surface (consistent with META-001's `allowAdmin`). |
| `observer` | **No** — `shouldOffer: false` | Observers are read-only on the gameplay surface in v1 (Stage 6.4). Re-routing a move is participant territory. Observers still **see** a recorded override read-only (it is visible metadata) — they just cannot create one. |

The actor gate is enforced inside `evaluateSemanticOverridePrompt` (step 1 of the algorithm) — an `observer` never gets `shouldOffer: true`. There is no separate eligibility table; the rule is a single role check because the override has exactly one kind of action.

**Own bubble.** The author of a move **may** override the lane on their own move — deciding "this is a tangent of mine" is legitimate structural self-description, not a pressure tag against another participant. MCP-015 does **not** gate own-bubble override; the trigger model does not even receive an `isOwnBubble` flag. (MCP-010 §16 decision 3 flagged this as an operator decision; this design takes MCP-010 §11's recommended position — own-bubble re-lane is allowed — and records it as Open question 1 below in case the operator wants the stricter fallback.)

## Edge cases

- **Confident packet, L1/L2 agree** — `evaluateSemanticOverridePrompt` returns `shouldOffer: false`. No override surface; the normal route chip behaves as usual. Not every move gets a choice.
- **Empty / missing packet (feature flag off)** — MCP-011's mock-stub packet carries no `low`-confidence routing binary, and an absent packet hits the no-input gate. `shouldOffer: false`. The board still works; no error.
- **`ledgerResult` present but `needsUserChoice: false`** — even if a non-routing category is `conflict_routed`, the routing-relevant filter (`continuity` / `direct_response`) means a non-routing conflict does not trigger the surface. `shouldOffer` falls through to the low-confidence / confident check.
- **`conflict_routed` on a non-routing category only** (e.g. `evidence_relevance`) — not routing-relevant → step 3 does not fire. The override surface is specifically the *routing* override; evidence-axis conflicts are MCP-013's own feedback path, not MCP-015's.
- **Observer viewing the move** — `shouldOffer: false` (actor gate, step 1). The observer still sees any *recorded* override read-only — it is visible metadata.
- **User picks the lane the referee already suggested** — a record is **still** written (an explicit affirmation is auditable). `chosenLane === suggestedLane`; the future confirm button reads `Keep referee suggestion`. MCP-013 may re-read the contested category at `agreement` confidence from that explicit affirmation.
- **User overrides, then changes their mind** — a *new* `SemanticOverrideRecord` and a new `MetadataEvent` (`kind: 'add'`) are appended; the latest `at` wins for display and downstream consumption. Earlier records are never deleted — the audit trail shows the full sequence.
- **Two participants override the same move** — each produces a record keyed by `overriddenByUserId` (same multi-applier pattern as META-001 manual tags). The model just records; the sidecar / MCP-013 / BR-004 decide per-viewer or per-move per their own rules.
- **Own bubble** — the author may override their own move's lane (Actor rule). The model has no `isOwnBubble` gate.
- **`assertsAnswersParent` true but `chosenLane` is `tangent`** — allowed and intentional. The toggle is independent of the lane radio group; a user can mark "this answers the parent" *and* route it as a tangent. `toSemanticOverrideMetadataEvent` emits the `tangent` event; `toAnswersParentMetadataEvent` emits the companion `answers_parent` event.
- **Soft-deleted move** — `arguments` rows are soft-deleted (`is_deleted = true`); the room shell filters deleted messages before the metadata ledger sees them (META-001 precedent). A stale override record for a now-absent move is simply not rendered. The model itself does not need to know — `messageId` is just a string anchor.
- **Concurrent edits** — argument bodies are immutable after posting (soft-delete only). The override anchors to a `messageId`; no concurrent-edit race on the body. Two overrides racing produce two append-only records; latest `at` wins — no lost update.
- **Offline / network failure** — the model is pure-TS with no network. The *packet* is fetched upstream (a future MCP-016 Edge Function); if that fetch failed the packet is absent and `shouldOffer` is `false` — the override surface degrades to absent, the board still works.
- **`repeatedSignal.overrideCountThisRoom` very high** — `softenCopy` only swaps the prompt copy code; it never suppresses the surface. There is no upper bound that disables the feature — a user must always be able to override.
- **Doctrine edge — "does overriding move score?"** — no. MCP-015 writes a record with no score field. MCP-013 *may* re-read the contested category from that record (MCP-003 §5.2's `agreement`-tier `l1_only` at full magnitude, **no penalty for the earlier conflict**) — that re-read is MCP-013's job, not MCP-015's. MCP-015 moves nothing.
- **Doctrine edge — "does overriding flag the user or the move?"** — no. No `flags` row, no moderation entry. The `semantic_override` `MetadataEvent` lives in the in-memory gameplay metadata ledger, which META-001 keeps strictly separate from `public.flags`.
- **Doctrine edge — "does a repeated override penalize the user?"** — no. The count only softens copy. It is not persisted, not summed across rooms, not a profile field, not a score, not a training input.
- **Doctrine edge — "can heat / popularity influence the override trigger?"** — no. The trigger reads only packet confidence + the L1/L2 conflict signal + viewer role + the in-memory count. It never reads engagement, view count, or heat band.
