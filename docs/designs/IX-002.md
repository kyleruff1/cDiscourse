**Release:** 6.6 / 6.7 — Wave 3 (Game constraints)

## Dependency check (resolved first, per the brief)

**IX-002 does NOT hard-depend on IX-001 (issue #20, OPEN).** Confirmed by reading the
code:

- IX-002's only structural input is `ArgumentTimelineMapModel`, the output of
  `buildArgumentTimelineMap` in `src/features/arguments/argumentGameSurfaceModel.ts`.
  That model **already ships** (Stage 6.2+) with `nodes`, `edges`, `bands`,
  `activePathIds`, `branchId` / `branchRootMessageId` per node, `isJunction`,
  `temperatureBand` / `toneBand`, `x` / `scrollWidth`, `rootMessageId`,
  `latestMessageId`, and `width`/`height`. Everything the mini-map needs is on it.
- IX-001 adds *zoom / density / focus lenses* to the **main** timeline. The
  mini-map is a constant-size overview; it does not zoom and does not consume
  density. The one optional touch-point (IX-001's `density` reflows node `x`
  positions) is handled gracefully because the mini-map **normalizes node
  positions to a 0..1 fraction** (see §Data model), so a different `density`
  produces an identical mini-map. There is no ordering, no API, no type that
  IX-002 must wait for.
- The mini-map's "viewport window" needs the host scroll offset + measured
  viewport width. Those are already tracked in `ArgumentTimelineMap.tsx`
  (`scrollX`, `viewportWidth` `useState`) — independent of IX-001.

**Conclusion: IX-002 is designed standalone and can be implemented before IX-001
lands.** When IX-001 ships, no IX-002 change is required. (BR-001 — the tree /
branch contract — is already complete; `branchId` and `isJunction` exist on the
map node today.)

The Timeline-Tree-Game-Board scope additions name LIFE-001 (lifecycle) and
GAME-001 (exhaustion advisory) as dependencies for "unresolved points" and
"exhausted points" markers. **Those two cards are not required for this card and
their markers are explicitly deferred** — see §Out of scope. IX-002 ships with
the markers that are derivable from data that exists today: branch clusters,
heat zone, active path, root, latest, junctions, collapsed-branch count.

## Goal (one paragraph)

For long debates the main `ArgumentTimelineMap` becomes a wide horizontal scroll
surface where the user loses the big picture. IX-002 adds a **mini-map overview
rail** — a tiny, fixed-width strip that renders the whole conversation at a
glance with a draggable/ tappable **viewport window** showing which slice the
main map is currently displaying, plus **jump-to** affordances for Root, Latest,
the hottest zone, and each branch cluster. Tapping a mini-map region pans/focuses
the main map *without a route transition*. Per `cdiscourse-doctrine` §2 and the
`timeline-grammar` skill, the mini-map's heat markers are **activity / friction
signals only — never truth, correctness, winning, or popularity**; the markers
reuse the existing `temperatureBand` / `toneBand` and `branchId` primitives so
no parallel heat or branch logic is invented. Per the acceptance criterion "does
not crowd mobile UI", the mini-map is **collapsed by default** and only auto-
expands (or becomes available to expand) once the debate crosses a length
threshold.

## Data model

No new database table, no migration, no Edge Function. IX-002 is a **pure-TS
view-model + a thin RN component**. The view-model is a deterministic projection
of the already-built `ArgumentTimelineMapModel`.

### New pure-TS module: `src/features/arguments/timelineMiniMapModel.ts`

```ts
// All x-coordinates in the mini-map model are NORMALIZED fractions in [0,1].
// 0 = first chronological node, 1 = last. Normalizing means the mini-map is
// invariant to the main map's xStep / density (IX-001-proof).

/** A single move plotted on the mini-map rail. One per timeline node. */
export interface MiniMapMarker {
  messageId: string;
  ordinal: number;            // 1-based, mirrors ArgumentTimelineMapNode.ordinal
  /** Normalized horizontal position in [0,1]. */
  xFraction: number;
  /** Branch lane the node sits on (from node.lane). Drives vertical offset. */
  lane: number;
  /** Branch cluster id (from node.branchId). */
  branchId: string;
  /** True when on the active path (node.isActivePath). */
  isActivePath: boolean;
  isActive: boolean;          // node.isActive
  isRoot: boolean;            // node.isRoot
  isLatest: boolean;          // node.isLatest
  isJunction: boolean;        // node.isJunction
  /** Heat tier — ACTIVITY, not correctness. Derived from temperatureBand. */
  heatTier: MiniMapHeatTier;
  /** Kind color family — reused token, never re-picked. */
  kindColorFamily: TimelineKindColorFamily;
  /** Marker fill color (kind color, reused from TIMELINE_KIND_COLORS). */
  color: string;
}

/** Heat tier on the mini-map. ACTIVITY level only. Never a verdict. */
export type MiniMapHeatTier = 'quiet' | 'mild' | 'warm' | 'hot';

/** A contiguous run of warm/hot markers — the "hot zone" jump target. */
export interface MiniMapHotZone {
  /** Start/end normalized fractions of the run. */
  xStartFraction: number;
  xEndFraction: number;
  /** The message id to jump to (first move of the run). */
  jumpTargetMessageId: string;
  /** Count of moves in the run. */
  moveCount: number;
}

/** A branch cluster summarized for a jump chip + a region band. */
export interface MiniMapBranchCluster {
  branchId: string;
  /** The branch root message id — the jump target for this cluster. */
  branchRootMessageId: string;
  /** Branch lane (0 = mainline). */
  lane: number;
  /** Move count in this branch cluster. */
  moveCount: number;
  /** Normalized x span of the cluster. */
  xStartFraction: number;
  xEndFraction: number;
  /** True when this branch is collapsed in the host's BranchCollapseState. */
  isCollapsed: boolean;
  /** Hidden-move count when collapsed (0 when expanded). */
  hiddenMoveCount: number;
  /** Plain-language label, e.g. "Mainline" / "Side issue" / "Chime-in". */
  laneLabel: string;
  /** True when the active path runs through this cluster. */
  containsActivePath: boolean;
}

/** The complete mini-map view-model. */
export interface TimelineMiniMapModel {
  /** True when the debate is long enough for the mini-map to be useful. */
  isAvailable: boolean;
  /** Total move count (mirrors timelineMap.nodes.length). */
  moveCount: number;
  markers: MiniMapMarker[];
  branchClusters: MiniMapBranchCluster[];
  /** At most one hot zone — the longest warm/hot run. null when none. */
  hotZone: MiniMapHotZone | null;
  /** Active-path markers' message ids, in chronological order. */
  activePathMessageIds: string[];
  /** Jump targets, always present (root/latest may equal each other). */
  rootMessageId: string | null;
  latestMessageId: string | null;
  /** Min / max lane across all markers — drives the mini-map's lane band height. */
  minLane: number;
  maxLane: number;
  /** Count of collapsed branches (for the "N branches collapsed" summary chip). */
  collapsedBranchCount: number;
  /** Plain-language one-line region summary, e.g. "12 moves · 3 branches · 1 hot zone". */
  summaryLine: string;
}

/** Input. Everything is already-built; this model NEVER re-derives heat or branches. */
export interface BuildTimelineMiniMapInput {
  timelineMap: ArgumentTimelineMapModel;
  /** The host's branch-collapse state (BR-001's BranchCollapseState). Optional. */
  collapseState?: BranchCollapseState;
  /**
   * Length threshold above which the mini-map is `isAvailable`.
   * Defaults to MINI_MAP_MIN_MOVES (see Constants). Caller may override
   * but the default is the product rule.
   */
  minMovesToShow?: number;
}
```

**Viewport-window model** (separate, tiny, also in `timelineMiniMapModel.ts`):

```ts
/** The slice of the conversation currently visible in the MAIN timeline. */
export interface MiniMapViewportWindow {
  /** Normalized left edge of the visible window, clamped to [0,1]. */
  xStartFraction: number;
  /** Normalized right edge of the visible window, clamped to [0,1]. */
  xEndFraction: number;
  /** True when the window covers the entire conversation (no scroll needed). */
  coversAll: boolean;
}

export interface BuildViewportWindowInput {
  /** Host ScrollView contentOffset.x (px). */
  scrollX: number;
  /** Host ScrollView measured viewport width (px). */
  viewportWidth: number;
  /** timelineMap.scrollWidth (px) — total scrollable content width. */
  scrollWidth: number;
}

/** A jump request the mini-map emits back to the host. */
export interface MiniMapJumpRequest {
  kind: 'root' | 'latest' | 'hot_zone' | 'branch' | 'marker' | 'region';
  /** The message id to activate + scroll to. */
  messageId: string;
}
```

**Heat-tier mapping** (deterministic, no AI, reuses `temperatureBand`):

```ts
// temperatureBand is produced by inferTemperatureBand() in
// argumentGameSurfaceModel.ts. The mini-map only RE-LABELS it; it never
// re-infers heat. 'unknown' degrades to 'quiet' (no heat shown).
//   cool    -> quiet
//   mild    -> mild
//   warm    -> warm
//   hot     -> hot
//   unknown -> quiet
```

`branchId` and `branchRootMessageId` are read verbatim from
`ArgumentTimelineMapNode`. `laneLabel` reuses `deriveBranchLabel({ lane,
isDetached })` from `keyboardNavigationModel.ts` (lane 0 → "Mainline",
non-zero → "Side issue", detached → "Detached") so branch wording never drifts.

No new data model beyond these interfaces. Nothing is persisted; the mini-map is
recomputed each render from the already-memoized `timelineMap`.

## API / interface contracts

### `buildTimelineMiniMapModel`

```ts
function buildTimelineMiniMapModel(
  input: BuildTimelineMiniMapInput,
): TimelineMiniMapModel;
```

- `input.timelineMap.nodes.length === 0` → returns a model with
  `isAvailable: false`, empty `markers` / `branchClusters`, `hotZone: null`,
  `summaryLine: ''`.
- `isAvailable === input.timelineMap.nodes.length >= (input.minMovesToShow ?? MINI_MAP_MIN_MOVES)`.
- `markers` is in chronological order (mirrors `timelineMap.nodes` order).
- `xFraction = ordinalIndex / max(1, nodeCount - 1)` (first node → 0, last → 1;
  single node → 0).
- Never throws; every missing/`undefined` field degrades to a safe default.

### `buildViewportWindow`

```ts
function buildViewportWindow(input: BuildViewportWindowInput): MiniMapViewportWindow;
```

- `scrollWidth <= 0` or `viewportWidth >= scrollWidth` →
  `{ xStartFraction: 0, xEndFraction: 1, coversAll: true }`.
- Otherwise `xStartFraction = clamp01(scrollX / scrollWidth)`,
  `xEndFraction = clamp01((scrollX + viewportWidth) / scrollWidth)`.
- Never throws on `NaN` / negative inputs (clamped).

### `TimelineMiniMap` component props

```ts
interface TimelineMiniMapProps {
  model: TimelineMiniMapModel;
  viewportWindow: MiniMapViewportWindow;
  /** Fired for every jump (chip tap, marker tap, region tap, window drag end). */
  onJump: (request: MiniMapJumpRequest) => void;
  /**
   * Fired continuously while the viewport window is dragged so the host
   * can scroll the MAIN map live. Emits a normalized centre fraction.
   */
  onScrubViewport?: (centreFraction: number) => void;
  /** Effective reduce-motion (composed OS + PR-001 preference). */
  reduceMotion?: boolean;
  /** Start expanded. Default false (collapsed) — see Mobile non-crowding. */
  initiallyExpanded?: boolean;
}
```

The component owns its own expanded/collapsed `useState` (seeded from
`initiallyExpanded`). When `model.isAvailable === false` the component renders
**nothing** (`return null`) — see Mobile non-crowding.

### Host wiring (inside `ArgumentTimelineMap.tsx`)

```ts
const handleMiniMapJump = useCallback((req: MiniMapJumpRequest) => {
  // Expand a collapsed branch first (mirrors handleStubPress).
  if (req.kind === 'branch') {
    setCollapseState((prev) => /* expand if collapsed */ ...);
  }
  onActivate(req.messageId);          // existing selection channel
  const node = nodesById.get(req.messageId);
  if (node) {
    try { scrollRef.current?.scrollTo({ x: Math.max(0, node.x - 120), animated: !reduceMotion }); }
    catch { /* swallow — same pattern as the existing auto-scroll effect */ }
  }
}, [onActivate, nodesById, reduceMotion]);
```

No new prop crosses the `ArgumentTimelineMap` boundary; the room shell
(`ArgumentGameSurface.tsx`) needs **zero changes**.

## Viewport-window indicator design

- The mini-map rail is a fixed-width `<View>` (fills the timeline frame width,
  same horizontal padding as `controlsRow`). The rail represents the entire
  conversation, `xFraction 0..1` mapped linearly across its measured width.
- The viewport window is a translucent rounded `<View>` overlay positioned with
  `left: xStartFraction * railWidth` and
  `width: (xEndFraction - xStartFraction) * railWidth`, with a visible 1.5px
  border (`#a5b4fc`, indigo — the same active-ring family) so it is legible in
  grayscale (border, not just fill).
- When `coversAll === true` the window spans the full rail and the drag handle
  is hidden (nothing to scrub).
- The window is a `Pressable` with `hitSlop` to guarantee a ≥44px effective
  drag target even though its visual height is the rail height (~28px). On
  drag (`onResponderMove` via a `PanResponder` built from RN core — no new
  dep) it emits `onScrubViewport(centreFraction)`; the host converts that to
  `scrollRef.current.scrollTo({ x: centreFraction * scrollWidth - viewportWidth/2 })`.
- Tapping anywhere on the rail **outside** the window calls
  `resolveRegionJumpTarget` and emits a `region` jump — the main map scrolls so
  that tapped region is centred and the nearest marker is activated.
- The window is **purely an indicator + scrubber** — it never changes which
  message is *active*; only chip taps and marker taps change the active node.
  (Scrubbing pans; it does not select. This keeps the selection model — shared
  with IX-004's readout — unsurprising.)

## Jump-to behavior

All four jump targets emit a `MiniMapJumpRequest`; the host routes every one
through the existing `onActivate` + imperative-scroll path. **No route push, no
navigation, no `Linking`** — the acceptance criterion "No route transition" is
satisfied structurally because the mini-map only calls in-component callbacks.

| Jump | Target message id | Source | Behavior |
|---|---|---|---|
| **Root** | `model.rootMessageId` | `timelineMap.rootMessageId` | Activate root, scroll its node into view. Chip disabled (dimmed + `accessibilityState.disabled`) when root is already active. |
| **Latest** | `model.latestMessageId` | `timelineMap.latestMessageId` | Activate latest, scroll to end. Chip disabled when latest already active. |
| **Hot zone** | `model.hotZone.jumpTargetMessageId` | first move of the longest warm/hot run | Activate that move, scroll the run into view. Chip **hidden entirely** when `model.hotZone === null` (no warm/hot run exists). Chip copy: `Hot zone` — never "winning"/"important". |
| **Branch** | `cluster.branchRootMessageId` per `MiniMapBranchCluster` | `node.branchRootMessageId` | One chip per non-mainline branch cluster (mainline gets no chip — Root already covers it). Tapping expands the branch if collapsed (via `toggleBranchCollapse`) then activates its root. Chip label: `laneLabel` + move count, e.g. `Side issue · 4`. |

Marker taps: tapping an individual marker emits `{ kind: 'marker', messageId }`.
Region taps (rail background): emit `{ kind: 'region', messageId }` via
`resolveRegionJumpTarget`, which finds the marker whose `xFraction` is nearest
the tapped normalized x.

Active-path indicator: every marker with `isActivePath === true` renders with a
connecting under-line tint (`#a5b4fc` at low alpha) so the user sees the spine
of the current conversation at a glance. The active marker itself gets the ring
treatment (slightly larger + cream border) — geometry, not color-only.

## Mobile non-crowding behavior

This is an explicit acceptance criterion ("Does not crowd mobile UI") and is
handled three ways:

1. **Length threshold — the mini-map does not exist for short debates.**
   `model.isAvailable` is `false` when `moveCount < MINI_MAP_MIN_MOVES`
   (**default 12**). When `isAvailable === false`, `TimelineMiniMap` returns
   `null` — zero pixels, zero layout cost. A 5-message debate (the IX-001
   "isn't sparse" case) shows no mini-map at all; the main map is already
   fully visible so an overview adds nothing.
2. **Collapsed by default.** Even when `isAvailable`, the mini-map renders
   **collapsed**: a single ~28px-tall header chip — `Overview ▾` plus the
   `summaryLine` (e.g. `18 moves · 2 branches · hot zone`). It occupies one
   slim row. The user taps the chip to expand the full rail + jump chips +
   viewport window. Collapsed state persists in component `useState` for the
   session (not across sessions — no storage).
3. **Expanded height is bounded.** The expanded mini-map is a fixed
   `MINI_MAP_RAIL_HEIGHT` (~64px rail + ~36px chip row + ~20px summary ≈ 120px
   total max). Lanes are compressed: `MINI_MAP_LANE_STEP_PX` (~7px) so even a
   ±3-lane spread fits in the rail height. It never grows with message count —
   markers get smaller / closer, never push the rail taller. There is no
   horizontal scroll inside the mini-map (the whole point is "at a glance").

On a narrow phone the collapsed chip is the steady state; the user expands it
only when they want to navigate, then it auto-collapses is **not** done (would
be surprising) — it stays expanded until the user collapses it.

## Heat / branch marker visual grammar (per `timeline-grammar`)

The mini-map reuses the existing token system. It introduces **no new color
families** and **no new shapes that imply truth**.

**Markers (one per move):**
- Shape: a small filled dot/square — same `MINI_MAP_MARKER_SIZE` (~7px) for all
  moves. Marker shape carries *nothing* at mini-map scale (too small); the main
  map's shape grammar is where type lives. The mini-map marker's **fill color**
  is the node's `kindColor` (`TIMELINE_KIND_COLORS[kindColorFamily]`, reused
  verbatim) purely as a familiarity cue, never as a judgement.
- **Heat is shown by a saturation/size overlay, not by an independent color.**
  Per `timeline-grammar`, heat = fill saturation. `hot` markers get a 1px warm
  ring (`rgba(249,115,22,0.55)`); `warm` get a fainter ring; `mild` / `quiet`
  get none. Heat NEVER changes the marker's kind color and NEVER adds a
  "winning"/"correct" glyph. The hot-zone run also gets a faint tinted underlay
  band (`rgba(249,115,22,0.18)`) — the same band-tint approach
  `ConversationMiniTimeline` already uses for `hot_zone`.
- Root marker: a tiny amber tab/notch (reuses the `nodeRoot` amber `#fde68a`).
  Geometry (a notch), color secondary.
- Latest marker: slightly larger dot + cream 1px border (mirrors the main
  map's `nodeRingLatest` cyan/cream cue).
- Junction marker (`isJunction`): a small hollow ring around the dot — geometry,
  visible in grayscale.
- Active marker: cream border + ~1.3× size. Active-path markers: connected by a
  low-alpha indigo under-line.

**Branch lanes:**
- Mini-map markers are offset vertically by `lane * MINI_MAP_LANE_STEP_PX`
  (lane 0 = centre rail, ±1/±2/±3 above/below) — the exact lane semantics from
  `computeLane`. Branch clusters render as a faint tinted band behind their
  marker run (slate `rgba(148,163,184,0.14)` for non-mainline) so a cluster
  reads as a group without any color implying quality.
- A collapsed branch cluster renders a single stub marker with a `+N` count
  badge — mirrors the main map's `BranchCollapseStub` convention.

**Copy:** marker / chip / summary text contains only: ordinals, plain kind
labels (`claim`, `challenge`, …), `Root`, `Latest`, `Hot zone`, `Mainline`,
`Side issue`, `Chime-in`, `Detached`, move counts, and `branch(es) collapsed`.
No verdict tokens. The whole-mini-map `accessibilityLabel` explicitly frames
heat as activity: e.g. *"Conversation overview. 18 moves, 2 side branches, one
hot zone (recent activity, not a result)."*

**Grayscale rule:** every signal — root (notch), latest (size+border), junction
(ring), active (size+border), heat (ring + saturation), branch cluster (band) —
is distinguishable without color. A grayscale snapshot test enforces this.

## Accessibility contract

- **Mini-map root**: `accessibilityRole="summary"` (or `"image"` if `summary`
  is unsupported on a target — match `ConversationMiniTimeline`'s `"image"`
  pattern) with the activity-framed `accessibilityLabel` above.
- **Collapse/expand header**: `Pressable`, `accessibilityRole="button"`,
  `accessibilityState={{ expanded }}`, label `"Conversation overview, tap to
  expand"` / `"… tap to collapse"`. ≥44px hit target (`hitSlop` since the
  visual is ~28px tall).
- **Jump chips** (Root / Latest / Hot zone / each Branch): `Pressable`,
  `accessibilityRole="button"`, descriptive `accessibilityLabel`
  (e.g. `"Jump to the opening claim"`, `"Jump to the hot zone, 4 moves of
  recent activity"`, `"Jump to side issue branch, 4 moves"`),
  `accessibilityState={{ disabled }}` when the target is already active.
  Each chip ≥44×44 (`hitSlop` if visual smaller).
- **Viewport window**: `Pressable` with `accessibilityRole="adjustable"`,
  `accessibilityLabel="Visible portion of the timeline"`,
  `accessibilityValue={{ text: "showing moves X to Y of N" }}`. On web,
  `accessibilityActions` `increment`/`decrement` page the window left/right
  by one viewport so keyboard users can scrub without a pointer drag. ≥44px
  effective target via `hitSlop`.
- **Individual markers**: markers are tiny and numerous; they are **NOT** each
  a separate focus stop. The rail is a single accessible element with the
  summary label; jump chips + the viewport window are the keyboard/SR-operable
  controls. (This mirrors `ConversationMiniTimeline`, where dots carry labels
  but the rail is the `image`.) Marker taps still work for pointer users via
  the rail's region tap.
- **Color independence**: enforced (see grammar §). Verified by grayscale
  snapshot test.
- **Reduce motion**: when `reduceMotion` is true, the expand/collapse height
  transition snaps (no animation) and viewport-window drag does not animate the
  main map's `scrollTo` (`animated: false`). Honors
  `AccessibilityInfo.isReduceMotionEnabled()` composed with the PR-001
  preference, exactly as `ArgumentTimelineMap` already does
  (`reduceMotionOverride`).
- **Focus order (web)**: collapse header → (when expanded) jump chips left-to-
  right → viewport window. The mini-map sits before the main rail in DOM order
  so it is reached first — it is an overview.
- **Keyboard (web)**: chips are standard `Tab`-reachable buttons (the mini-map
  does NOT join the main map's roving-Arrow group; it is its own simple button
  set). The viewport window responds to `Left`/`Right` via `accessibilityActions`.

## Edge cases

- **Empty debate (0 moves)** — `buildTimelineMiniMapModel` returns
  `isAvailable: false`; component renders `null`. Main map already shows its own
  "Timeline appears once any argument is posted" empty state.
- **Single-node debate (1 move)** — `isAvailable: false` (1 < 12). No mini-map.
  `xFraction` for the lone node would be 0 (guarded `max(1, n-1)`); no divide-
  by-zero.
- **Short debate (2–11 moves)** — `isAvailable: false`. No mini-map; the main
  map fits without an overview.
- **Very long debate (250+ moves)** — explicit acceptance criterion. Markers
  are normalized fractions, so 250 markers compress into the fixed rail width;
  marker dots may visually overlap — that is fine (the mini-map is a density
  picture, not a hit-target surface). The model build is O(n) single-pass; no
  per-marker layout cost beyond the dot `<View>`. Test with a 260-node fixture.
- **No branches (all moves on lane 0)** — `branchClusters` contains only the
  mainline cluster, which produces **no branch chip** (mainline excluded);
  jump chips are just Root / Latest (+ Hot zone if present). No empty chip row.
- **No heat (all `temperatureBand` = `unknown`/`cool`)** — every marker is
  `quiet`; `hotZone` is `null`; the Hot-zone chip is hidden. No heat rings, no
  tinted band. The mini-map still shows the rail + Root/Latest.
- **All moves detached (no root)** — `timelineMap.rootMessageId` may be the
  first detached node (the map already assigns `isRoot` to the first node with
  a missing parent). Root chip jumps there. If `rootMessageId` is genuinely
  `null` (only possible for empty map), the Root chip is hidden.
- **Active node inside a collapsed branch** — the host's existing
  `applyActiveAutoExpand` already keeps the active path visible on the main
  map; the mini-map's `containsActivePath` is computed from `node.isActivePath`
  which is set regardless of collapse, so the active-path under-line is still
  drawn. A branch-chip jump into a collapsed branch expands it first.
- **Concurrent edits / new move arrives** — `timelineMap` is rebuilt by the
  host on new data; the mini-map model is a `useMemo` keyed on `map` so it
  recomputes. `latestMessageId` shifts to the new move; the Latest chip and
  marker follow. No stale state — the model holds no internal mutable state.
- **Host viewport not yet measured (`viewportWidth === 0`)** —
  `buildViewportWindow` treats `viewportWidth >= scrollWidth` (0 ≥ anything
  false; but 0 width) → returns `coversAll: false` with a zero-width window.
  Guard: when `viewportWidth <= 0`, return `coversAll: true` (show full-width
  window until layout settles) — documented in the function.
- **`coversAll` (whole conversation fits, no scroll)** — the viewport window
  spans the full rail and the drag handle is hidden; scrubbing is a no-op.
- **Doctrine edge case — heat cannot influence anything but the heat ring.**
  The mini-map reads `temperatureBand` only to pick a `MiniMapHeatTier`. It
  does NOT feed heat into branch clustering, into jump-target selection (hot
  zone is the longest *run*, chosen by length, not by "hottest"), into marker
  ordering, or into any band that resembles a strength/standing band. A
  heavily-replied wrong claim shows a `hot` ring and nothing else — exactly the
  `timeline-grammar` rule.

## Risks

- **Library / platform gotcha — `PanResponder`.** The viewport-window scrubber
  uses RN core `PanResponder` (no new dep). On web, RN-web's `PanResponder`
  works but is less smooth than a pointer-events handler. Mitigation: the
  primary affordance is the **jump chips + region tap**; drag-scrub is an
  enhancement. If web drag proves janky, the implementer may fall back to
  tap-to-region only on web (`Platform.OS === 'web'`) and keep drag on native,
  with the `accessibilityActions` increment/decrement still covering keyboard.
  Document whichever choice is made.
- **Existing test update.** `__tests__/argumentTimelineMap.test.ts` will gain
  assertions and a couple of existing render tests may need their fixtures
  bumped past 12 moves (or explicitly kept short to assert the mini-map is
  absent). This is expected; the test count goes **up**.
- **Marker overlap at 250+ moves.** Acceptable by design (density picture), but
  the grayscale snapshot for a huge fixture will be visually dense — keep that
  snapshot test on a moderate (~30-move) fixture; use a count-only assertion
  for the 260-move fixture.
- **Heat-tier wording drift.** `temperatureBand` already has a `warm`/`hot`
  vocabulary; the mini-map must not introduce a *new* heat adjective that could
  read as a verdict. Mitigation: `MiniMapHeatTier` literals are exactly
  `quiet | mild | warm | hot` and the ban-list test guards the union.
- **No migration, no Edge Function, no operator deploy** — pure client code.
  Low operational risk.

## Doctrine self-check

- **cdiscourse-doctrine §1 — no truth labels, score never blocks posting, no
  service-role.** The mini-map renders no strength/standing band, no
  winner/loser/correct/true/false copy; a ban-list test enforces it. It posts
  nothing, blocks nothing, and is pure client UI — no service-role, no Edge
  Function, no DB write.
- **cdiscourse-doctrine §2 — heat means activity / friction, never
  correctness / popularity.** The mini-map's heat tier is a re-label of the
  existing `temperatureBand` (an activity proxy). Heat drives only a marker
  ring + a faint band; it never feeds branch clustering or jump-target
  selection (the hot zone is the *longest run*, not the "hottest"). The
  whole-mini-map a11y label explicitly says heat is "recent activity, not a
  result". A `pretty_wrong` move can be `hot` — verified by test.
- **cdiscourse-doctrine §3 — popularity is not evidence.** The mini-map shows
  no reply-count / engagement-count markers; nothing on it grants factual
  standing.
- **cdiscourse-doctrine §7 — no AI calls from the production app.** Pure-TS
  model + RN component; no Anthropic / xAI / X API; forbidden-imports test
  enforces no `fetch` / network.
- **cdiscourse-doctrine §9 — plain language.** All mini-map copy is plain
  English (`Overview`, `Root`, `Latest`, `Hot zone`, `Side issue`, move
  counts). No internal codes (`temperature_band`, `branch_id`, etc.) reach a
  user-facing string.
- **cdiscourse-doctrine §10 — v1 scope.** No voting, no search, no push, no
  OAuth, no public API. The mini-map is navigation chrome.
- **timeline-grammar — shape/stroke/texture carry meaning, color is
  supplementary; no truth-laden visuals.** Root = notch (geometry), latest =
  size+border, junction = ring, active = size+border, heat = saturation+ring,
  branch cluster = band. Every signal survives grayscale; a grayscale snapshot
  test enforces it. The mini-map reuses `TIMELINE_KIND_COLORS` and
  `deriveBranchLabel` — no parallel grammar.
- **accessibility-targets — 44px targets, color independence, SR contract,
  reduce-motion.** Every `Pressable` (collapse header, jump chips, viewport
  window) meets 44px via `hitSlop`; all expose role + label + state; the
  viewport window is `adjustable` with increment/decrement actions;
  reduce-motion snaps the expand transition and the scroll. Verified by
  component tests.
- **expo-rn-patterns — RN primitives, no new dependency.** `View` / `Pressable`
  / `Text` / `PanResponder` (RN core) only. No Bootstrap, no SVG, no animation
  lib beyond an optional RN-core `Animated` height. The model file is pure TS
  with no React import — enforced by the forbidden-imports test.
- **test-discipline — tests are part of "done".** Four new test files plus an
  extension of the existing `argumentTimelineMap.test.ts`; pure-model functions
  each have unit tests including failure cases; ban-list + grayscale + a11y +
  forbidden-imports all covered.
