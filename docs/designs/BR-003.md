## Goal

When a participant's draft does not respond to the point it is replying to,
introduces a brand-new issue, or shifts scope, the app should offer to route
the move into a **side branch** instead of letting it poison the mainline or
forcing the participant to abandon the structured room. BR-003 builds the
deterministic assessment for that offer: a pure-TS model
(`tangentRoutingModel.ts`) that, given a draft + its parent + the parent's
lifecycle/metadata + the draft's manual tags, returns a `RedirectRisk`
(`none | possible | strong`), a structural `reason`, and a `suggestedAction`.
The assessment surfaces as **one advisory inside RULE-004's existing
`PreSendReviewSheet`** — the user can always continue; nothing is blocked.

Doctrine constraints that shape the design (`cdiscourse-doctrine` +
`timeline-grammar` "a tangent is a topology label, not a verdict"):

- **§1 — score never blocks posting; advisories are not verdicts.** BR-003 adds
  **zero blocking rules.** A tangent-risk assessment NEVER prevents a post. It
  surfaces a `soft`/`info` advisory inside the RULE-004 sheet, which always keeps
  "Post anyway" live (unless an *existing* Constitution structural block is
  already hit — which is RULE-004's pre-existing engine block, not BR-003's).
  A test asserts `risk: 'possible'` and `risk: 'strong'` never produce a
  `PreSendStructuralBlock`.
- **Structural redirect, no person labels.** Every `reason` value describes the
  MOVE's structural relationship to its parent — `introduces_new_axis`,
  `no_signal_about_parent`, `mode_demands_response`, `repeated_off_path`,
  `user_marked_tangent`. None describes the person. The plain-language copy says
  "this move…", never "you are dodging / evading / off-topic-as-a-person". A
  ban-list test enforces zero verdict / person-attribution tokens.
- **§1 / Stage 6.2 doctrine — no keyword-only gating.** `assessTangentRisk`
  **never inspects raw body text for keyword matching.** It reads typed
  structural fields (argument type, channel, qualifier tag codes), the parent's
  already-derived META-001 `AutoMetadataEntry` codes, and the parent's LIFE-001
  lifecycle state. The one body read is the same deterministic *punctuation
  shape* read RULE-004 already uses (`bodyEndsWithQuestion`) and it only ever
  *raises* an advisory — never blocks. Stage 6.2 converted OFF_TOPIC /
  PARENT_NONRESPONSIVE to advisory precisely so structural routing stays
  keyword-light; BR-003 honors that.
- **§2 / §3 — heat / popularity are not signals.** The model never reads a
  strength band, a heat value, a reply count, or any engagement metric. Two
  drafts identical in structure produce identical assessments regardless of
  (hypothetical) heat. A test asserts this.
- **§4 / §7 — no AI call.** `assessTangentRisk` is deterministic pure TS. No
  Anthropic, no xAI, no client AI, no Edge Function. The card explicitly defers
  the **semantic AI tangent classifier to RULE-006.**
- **§5 — pure models.** `tangentRoutingModel.ts` imports types + frozen copy
  tables only. No React, no Supabase, no network, no mutation, no async.
- **§8 — soft-delete only.** BR-003 never hard-removes content. A move routed to
  a side branch is *posted as a branch*, fully accessible (collapsed, not
  deleted). Mainline demotion is *reversible*. No `is_deleted` write, no row
  removal.
- **§9 — plain language.** Every produced string is read from a frozen
  `gameCopy.ts` block. No new snake_case in any user-facing string; no
  verdict / person token. Enforced by a ban-list test.
- **§10 — no v1 scope item touched.** No voting, search, notifications, OAuth,
  public API.

The model is unit-testable in isolation. The RULE-004 model gains one derivation
step + one union member + one optional input field — all **additive**; every
existing RULE-004 caller that omits `tangentContext` behaves identically. The
sheet component is **not modified at all** — it already iterates
`review.advisories` generically.

**Key architectural decision — BR-003 does NOT own routing execution.** It is an
*assessment + advisory* layer. `assessTangentRisk` returns a `suggestedAction`;
the *carry* of that action (posting as a branch) goes through the **existing**

channel and the existing `branch_this_off` qualifier tag. BR-003 builds **no new
post API, no new branch-creation API, no migration, no Edge Function.**

## §2 — Data model

### 2.1 New pure-TS types (`tangentRoutingModel.ts`)

```ts
/** BR-003 — how strongly the draft reads as a redirect away from its parent. */
export type RedirectRisk = 'none' | 'possible' | 'strong';

/** Frozen list of every risk level. Tests iterate this. */
export const ALL_REDIRECT_RISKS: ReadonlyArray<RedirectRisk> =
  Object.freeze(['none', 'possible', 'strong']);

/**
 * BR-003 — the STRUCTURAL reason a redirect risk was raised. Every value
 * describes the MOVE's relationship to its parent — never the person.
 * The card's §1 verbatim list.
 */
export type RedirectReason =
  | 'introduces_new_axis'      // draft acts on a different disagreement axis
  | 'no_signal_about_parent'  // draft carries no structural link to the parent
  | 'mode_demands_response'   // the channel/lifecycle expects a direct reply, draft is not one
  | 'repeated_off_path'       // this side has gone off-path ≥ threshold times recently
  | 'user_marked_tangent';    // the author themselves tagged the draft a tangent

/** Frozen list of every reason. Tests iterate this. */
export const ALL_REDIRECT_REASONS: ReadonlyArray<RedirectReason> = Object.freeze([
  'introduces_new_axis',
  'no_signal_about_parent',
  'mode_demands_response',
  'repeated_off_path',
  'user_marked_tangent',
]);

/**
 * BR-003 — what BR-003 SUGGESTS the user do. The card's §1 verbatim list.
 * `continue` = no friction; the others surface as a transformation in the
 * RULE-004 advisory card.
 */
export type RedirectSuggestedAction =
  | 'continue'              // post on the mainline as-is — no advisory
  | 'send_to_side_branch'  // post as a branch root off the parent (carry: §5.4)
  | 'ask_clarifying_question' // re-aim the move as a clarification of the parent
  | 'branch_this';         // explicitly branch — the strongest redirect offer

/** The model's single output. */
export interface TangentRoutingAssessment {
  risk: RedirectRisk;
  /** The dominant reason. `null` only when `risk === 'none'`. */
  reason: RedirectReason | null;
  suggestedAction: RedirectSuggestedAction;
}
```

**Doctrine note on the type names.** No `outer_realm` / `outer_orbit` identifier
appears — §0 D3. The type is `RedirectRisk`; the model is `tangentRoutingModel`.
`tangent` is already an established *topology* word in the repo
(`branchTopologyModel`'s doctrine header: "a tangent is a topology label, not a
verdict") so reusing it in code is consistent and safe.

### 2.2 The model input

## §3 — Reconciliation with RULE-004's `PreSendReviewSheet` (load-bearing)

**This is the integration the card brief explicitly requires.** The card scope
§2 says the advisory "surfaces … *inside* `PreSendReviewSheet`" and the brief
says it must "compose with RULE-004's existing advisory list as an additional
advisory kind / input, NOT a new parallel surface and NOT a duplicate."

RULE-004's `preSendReviewModel.ts` exposes a **closed 7-kind `AdvisoryKind`
union**, a frozen `ADVISORY_DEFINITIONS` record, and an 8-step derivation in
`buildPreSendReview`. The sheet (`PreSendReviewSheet.tsx`) iterates
`review.advisories` *generically* — it renders any advisory through one
`advisoryCard` path with `actionableTransformations(kind)`.

BR-003 plugs in exactly here:

### 3.1 One new advisory kind

`AdvisoryKind` gains **one** member: `tangent_redirect`. `ALL_ADVISORY_KINDS`
gains the same. `ADVISORY_DEFINITIONS` gains one frozen entry:

```ts
tangent_redirect: freezeDef('tangent_redirect', 'soft', [
  'branch_tangent',   // primary — "Branch a side issue" (existing RULE-004 label)
  'post_anyway',
]),
```

Severity is `soft` in casual mode. The `suggested` transformations are the
**existing** `AdvisoryTransformation` values — `branch_tangent` already maps to
the `branch` quick action via RULE-004's `transformationToQuickAction`. **BR-003
adds no new `AdvisoryTransformation` and no new `QuickActionLabel`** — the same
discipline RULE-004 §0 D5 applied.

### 3.2 One new derivation step in `buildPreSendReview`

`buildPreSendReview` gains **step 9** (appended after the existing
`permanent_record_warning` step):

```
9. tangent_redirect — call assessTangentRisk(...) on the draft+parent.
   When the assessment's `risk` is 'possible' OR 'strong', append a
   `tangent_redirect` advisory. When `risk === 'none'`, append nothing.
```

The advisory's `plainLanguage` comes from `TANGENT_ROUTING_COPY`:
`advisory_possible` for `risk: 'possible'`, `advisory_strong` for
`risk: 'strong'`. (The per-reason detail line is available for a future
two-line card; v1 shows the headline only — keeps the card compact.)

### 3.3 One new optional input field

`PreSendReviewInput` gains **one optional field**:

```ts
/**
 * BR-003 — context the tangent-risk assessment needs. Optional: when
 * omitted, buildPreSendReview skips step 9 entirely and behaves exactly
 * as the merged RULE-004 model. Additive — no existing caller breaks.
 */
tangentContext?: AssessTangentRiskInput | null;
```

When `tangentContext` is absent or null, step 9 is a no-op — `buildPreSendReview`
produces an identical review to today's merged RULE-004. This is the same
additive-optional discipline RULE-004 used for `onBeforeSubmit` and RULE-005
used for its forward-compatible fields. **A test asserts: with `tangentContext`
omitted, the review is byte-identical to the pre-BR-003 output.**

### 3.4 Why this is composition, not duplication

| Property | How BR-003 satisfies it |
|---|---|
| Inside the sheet, not parallel | `tangent_redirect` is a member of RULE-004's own `AdvisoryKind` union; it renders through the sheet's existing `advisoryCard` loop. Zero new component. |
| Not a duplicate of `asks_new_question` | RULE-004's `asks_new_question` fires on META-001 `branch_suggested` OR a tangent tag OR a `?`-shaped body. ## §4 — The `repeated_off_path` deterministic counter

The card §3 says mainline-demotion fires when `repeated_off_path` is hit "≥ 3
times in a thread". §0 D4 established there is **no META-001 code** for this —
BR-003 owns the counter.

`countRecentTangentMoves(context: TangentThreadContext): number` — pure,
deterministic:

1. Take `context.recentMoves` (already chronologically ordered, already bounded
   by the caller to a recent window — see §7 #8).
2. Count moves where `move.side === context.authorSide` **and**
   `move.inboundBranchKind` is one of `kink_start | tangent | kink_end`.
   (`main` / `detached` do not count.) BR-001's `RailBranchKind` is the
   authoritative topology classification — BR-003 reuses it, never re-derives.
3. Return the count.

`repeated_off_path` is raised by `assessTangentRisk` when
`countRecentTangentMoves(context) >= REPEATED_OFF_PATH_THRESHOLD` (frozen
constant, default **3** — the card's number).

**Doctrine guard:** the counter counts a *side's* moves by their *topology
classification* — it never reads who-is-right, never reads heat, never names a
participant. The card's §3 demotion "on the offending participant's NEXT move"
is honored by the *advisory* (`buildMainlineDemotionAdvisory`) describing the
**thread** ("The main thread has drifted"), not the person — and only the side's
own next composer surfaces it.

`buildMainlineDemotionAdvisory(context)` returns a `MainlineDemotionAdvisory`
with `active: true` when the threshold is met, `reversalActions:
['ask_clarifying_question', 'continue']` (a direct confirm/narrow/synthesize
move clears it — the card's "reversible" requirement), and `plainLanguage:
TANGENT_ROUTING_COPY.demotion_advisory`.

## §5 — API / interface contracts

### 5.1 `assessTangentRisk(input)`

```ts
/**
 * Assesses how strongly a draft reads as a redirect away from its parent.
 * Pure. Deterministic. Idempotent. NO AI, NO network, NO mutation, NO async.
 *
 * Reads ONLY:
 *  - draft.argumentType, draft.selectedTagCodes, draft.body (punctuation
 *    SHAPE only — never keyword matching)
 *  - parent.depth / parent presence
 *  - lifecycle.parentSnapshot.axis, lifecycle.parentClusterSummary.state,
 *    lifecycle.parentLinkage.autoDerivedMetadata (META-001 codes)
 *  - manualTags (the author's own draft tags)
 *  - tangentContext.recentMoves[].inboundBranchKind (BR-001 topology)
 *  - selectedChannel (RULE-005 channel)
 *
 * NEVER reads: strength bands, heat, reply counts, engagement, raw body
 * text for keyword matching.
 */
export function assessTangentRisk(
  input: AssessTangentRiskInput,
): TangentRoutingAssessment;
```

**Derivation order** (the first reason that fires at each tier wins; fully
unit-testable rule list):

1. **`risk: 'none'` fast paths** — root draft (`parent === null`); or the draft
   declares `clarification_request` / a confirm/narrow/synthesis channel
   (those are *direct* moves, never tangents). Returns
   `{ risk: 'none', reason: null, suggestedAction: 'continue' }`.

2. **`user_marked_tangent`** — `manualTags` contains `tangent` /
   `branch_this_off` / `tangent_or_joke`, OR `selectedChannel ===
   'branch_tangent'`. → `risk: 'possible'`, `suggestedAction:
   'send_to_side_branch'`. (When RULE-004's `asks_new_question` already fired
   for the same tag, the RULE-004 model suppresses the `tangent_redirect`
   advisory — §3.4 / §7 #6 — so the user does not see two cards for one tag.)

3. **`introduces_new_axis`** — `lifecycle.parentSnapshot.axis` is non-null AND
   the draft declares a *different* disagreement axis. The draft's axis is read
   from its typed channel + qualifier tags (e.g. an `ask_source` channel on a
   parent whose open axis is a *definition* dispute is not necessarily a
   tangent; but a draft whose tags name a different axis family than the
   parent's `axis` is). **No body keyword read.** When the parent has *no* axis
   yet (`axis === null`), this reason cannot fire. → `risk: 'strong'`,
   `suggestedAction: 'branch_this'`.

4. **`mode_demands_response`** — META-001 `participant_skipped_node` is present
   on `parentLinkage.autoDerivedMetadata` (the parent is a node a same-side
   participant skipped) AND the draft is *not* a direct reply move (channel is
   not `reply` / `challenge` / `clarify` / `concede` / `confirm`). The parent's
   point is still owed a direct response and this draft is not it. →
   `risk: 'strong'`, `suggestedAction: 'ask_clarifying_question'`.

5. **`no_signal_about_parent`** — META-001 `branch_suggested` is present on
   `parentLinkage.autoDerivedMetadata` (META-001 already observed the parent's
   cluster is `branch_recommended` or carries a tangent qualifier) AND the draft
   carries no challenge/evidence/clarify channel that would *engage* the parent.
   → `risk: 'possible'`, `suggestedAction: 'send_to_side_branch'`.

6. **`repeated_off_path`** — `tangentContext` is present AND
   `countRecentTangentMoves(tangentContext) >= REPEATED_OFF_PATH_THRESHOLD`
   (§4). → `risk: 'possible'`, `suggestedAction: 'send_to_side_branch'`.
   (This reason also feeds `buildMainlineDemotionAdvisory` — §4.)

7. **Default** — no reason fired → `{ risk: 'none', reason: null,
   suggestedAction: 'continue' }`.

The order encodes priority: an explicit user tag (2) is honored first; a hard
structural axis mismatch (3) and a skipped-node mode demand (4) are `strong`;
the softer META-001 / counter signals (5, 6) are `possible`. Exactly one
`reason` is returned (the dominant one); the model does not return a list.

**`risk` → `suggestedAction` is NOT 1:1** — it depends on the reason, which is
why the assessment carries both. A test asserts every `(risk, reason)` pair maps
to its specified `suggestedAction`.

### 5.4 How `suggestedAction` is *carried* (the BR-004 boundary)

The card §2 says "'Send to side branch' creates the post as a branch root … using
existing `submit-argument` path." §0 D5 established BR-004 (first-class branch

  quick-action preset. That preset sets the draft's channel to RULE-005's

- The draft is then posted **through the unchanged `submit-argument` path** with
  its `parentId` intact. The resulting `public.arguments` row carries the
  `branch_this_off` qualifier.
- On the next render, **BR-001's `branchTopologyModel` classifies the inbound
  edge** of that move as `kink_start` (its `hasTangentLexicalCode` helper reads
  exactly `branch_this_off` / `tangent_or_joke`). The move now visually lives on
  a side branch — collapsed-capable, fully accessible. **No new branch-root API
  is needed**; BR-001's topology classifier already produces the side-branch
  visual from the qualifier tag.
- **BR-004 is a downstream consumer, not a dependency.** If/when BR-004 ships a
  first-class branch-creation flow, `suggestedActionToQuickAction` can route
  `send_to_side_branch` into BR-004's API instead of the qualifier-tag path —
  a one-line change in BR-003, no redesign. BR-003 does **not** redesign or
  pre-empt BR-004; it just leaves the seam.

This means BR-003 ships a **working "send to side branch"** today (via the
qualifier-tag + BR-001 topology path) without waiting on BR-004.

## §7 — Edge cases

The implementer must handle each:

1. **Root draft (`parent === null`).** `assessTangentRisk` step 1 returns
   `{ risk: 'none', reason: null, suggestedAction: 'continue' }`. A root move
   cannot be a tangent of anything. Test covers it.
2. **Draft is a direct move** — `clarification_request` type or a
   confirm/narrow/synthesize channel. Step 1 fast-paths to `risk: 'none'`. A
   clarification of the parent is the *opposite* of a tangent.
3. **Parent has no axis yet (`parentSnapshot.axis === null`).** The
   `introduces_new_axis` reason cannot fire (nothing to mismatch against). The
   model falls through to the softer reasons. Never throws.
4. **`lifecycle.parentLinkage === null`** (parent posted before META-001 ran,
   or a caller without the ledger). `mode_demands_response` and
   `no_signal_about_parent` (which read `autoDerivedMetadata`) cannot fire; the
   model falls through. Never throws — same null-tolerance RULE-004's
   `parentSuggestsBranch` already has.
5. **`tangentContext` omitted.** `repeated_off_path` never fires;
   `buildPreSendReview` step 9 is skipped entirely; the review is
   byte-identical to merged RULE-004. Test asserts this explicitly.
6. **De-dup with RULE-004's `asks_new_question`.** When `assessTangentRisk`
   would fire *only* because of `user_marked_tangent` (a tangent qualifier tag)
   AND RULE-004's `asks_new_question` already fired for that same tag,
   `buildPreSendReview` step 9 **suppresses the `tangent_redirect` advisory** —
   the user sees one card, not two, for one tag. `tangent_redirect` is appended
   only when its reason is *not* purely `user_marked_tangent`-via-tag, i.e. it
   carries axis-mismatch / no-signal / mode-demands / repeated-off-path
   information `asks_new_question` does not. Documented; tested.
7. **Keyword-laden body, structurally fine.** A draft whose body is full of
   charged words but whose typed channel/tags/axis engage the parent yields
   `risk: 'none'`. `assessTangentRisk` never reads the body for keyword
   matching — only the `bodyEndsWithQuestion` punctuation shape, and even that
   is not a `assessTangentRisk` input in v1 (RULE-004's `asks_new_question`
   owns the `?`-shape read). A test feeds a charged-keyword draft and a benign
   one of identical structure and asserts identical assessments.
8. **`recentMoves` window.** The caller bounds `tangentContext.recentMoves` to a
   recent slice (e.g. the last N moves of the thread). BR-003's counter does not
   walk the whole tree — it counts only what the caller passes. The "recent"
   definition is the caller's; `countRecentTangentMoves` is pure over its input.
   Documented so the implementer does not unbounded-walk the room.
9. **`repeated_off_path` at exactly the threshold.** `>= 3` fires (the card's
   "≥ 3"). At 2 it does not. Boundary test in `mainlineDemotion.test.ts`.
10. **Mainline demotion is reversible.** After `repeated_off_path` fires, a
    direct confirm/narrow/synthesize move by the same side drops the off-path
    count below the threshold on the next assessment — the advisory clears
    automatically because `countRecentTangentMoves` re-counts. No persisted
    "demoted" flag, nothing to manually un-set. Test covers the reversal.
11. **Strict mode (GAME-003, inert in v1).** `tangent_redirect` is `soft`;
    RULE-004's `isPostAnywayEnabled` already gates `soft` advisories in strict
    mode generically. The card's "stronger 'Are you sure?'" upgrade for
    `risk: 'strong'` is the strict-mode `soft`-dismiss path — already handled by
    RULE-004's existing strict logic. **No new strict code in BR-003.** It is
    unreachable in v1 (dock passes `mode: 'casual'`).
12. **Concurrent edits / offline.** `assessTangentRisk` is a pure draft-time
    model with no network. Offline has no effect. There is no write, so no
    concurrency surface. N/A beyond that.
13. **Permission-denied paths.** BR-003 surfaces an advisory; it never posts,
    never writes, never reads RLS-protected rows directly. The eventual post
    goes through the unchanged `submit-argument` path which already enforces
    permissions. N/A for BR-003.
14. **Doctrine edge — can a tangent assessment block a post?** No.
    `assessTangentRisk` returns no block; `tangent_redirect` is `severity:
    'soft'`. The ONLY thing that hides "Post anyway" is a `PreSendStructuralBlock`
    from the Constitution engine. `__tests__/tangentRoutingNoBlock.test.ts`
    asserts `risk: 'possible'` and `risk: 'strong'` both leave
    `hasStructuralBlock === false` and "Post anyway" available.
15. **Doctrine edge — does the assessment ever label the person?** No. Every
    `RedirectReason` describes the move's structural relationship to its parent.
    Every copy string describes "this move" / "the thread". The ban-list test
    scans for `dodge / dodging / evade / evading / troll / bad faith / liar /
    winner / loser` etc. and asserts zero matches.
16. **Doctrine edge — does heat / popularity raise a redirect risk?** No. The
    model reads typed structural fields, META-001 *structural* codes, LIFE-001
    lifecycle state, and BR-001 topology classifications. It never reads a
    strength band, heat, reply count, or engagement. A test feeds two drafts
    identical in structure but different in (mock) heat and expects identical
    assessments.

## §10 — Risks

- **De-dup with `asks_new_question` (§7 #6) is the subtlest part.** RULE-004's
  `asks_new_question` and BR-003's `tangent_redirect` both react to tangent
  qualifier tags. If the suppression logic is wrong the user sees two cards for
  one tag. Mitigation: the rule is explicit (§7 #6) and has a dedicated test in
  the RULE-004 suite extension. The implementer must read RULE-004's
  `asks_new_question` derivation (step 5 of `buildPreSendReview`) before wiring
  step 9.
- **`introduces_new_axis` needs a draft-axis read that is NOT keyword-based.**
  The parent's axis is on `PointLifecycleSnapshot.axis`; the *draft's* axis must
  come from typed fields (channel + qualifier tags), since the draft is not yet
  posted and has no lifecycle snapshot. The implementer must derive the draft's
  axis family from `selectedChannel` + `selectedTagCodes` only — never from body
  text. If a clean typed axis read is not possible for some channel
  combinations, the safe default is **do not fire `introduces_new_axis`** (a
  false negative is acceptable; a keyword-based false positive is not — Stage 6.2
  doctrine). Documented as a Build-phase judgment call.
- **`TangentThreadContext` plumbing.** The caller (the dock / composer) must
  build `recentMoves` from the already-loaded timeline map + BR-001 topology.
  This is read-only assembly, but the implementer must confirm the dock has the
  topology map in scope at Post intent. If it does not, `tangentContext` is
  simply omitted and `repeated_off_path` does not fire (graceful degradation,
  §7 #5) — BR-003 still ships the other four reasons.
- **RULE-004's `_forbiddenPreSendTokens` ban-list test iterates
  `ALL_ADVISORY_KINDS`.** Adding `tangent_redirect` pulls `TANGENT_ROUTING_COPY`
  into that existing test for free — *good*, but the implementer must ensure
  `TANGENT_ROUTING_COPY.reason_user_marked_tangent` ("You marked this as a side
  issue") does not trip a future stricter person-token rule. "You marked" is the
  author describing their own action; it is non-punitive. Documented so a
  reviewer does not flag it.
- **No migration, no Edge Function** — so no deploy-ordering risk. The one
  watch-item is that BR-003 modifies the *merged* `preSendReviewModel.ts`; if
  RULE-004 receives a follow-up patch in parallel there could be a merge
  conflict in `AdvisoryKind` / `ADVISORY_DEFINITIONS` / `buildPreSendReview`.
  Mitigation: BR-003's changes are small, additive, and at the end of each
  structure — low conflict surface.

## §12 — Doctrine self-check

- **cdiscourse-doctrine §1 — no truth labels; score never blocks posting; no
  service-role.** Every `RedirectReason` is structural (`introduces_new_axis`,
  `no_signal_about_parent`, …) — none asserts who is right. `assessTangentRisk`
  returns no block; `tangent_redirect` is `severity: 'soft'`; "Post anyway" is
  always available (§7 #14, `tangentRoutingNoBlock.test.ts`). No service-role,
  no Edge Function, no DB write.
- **cdiscourse-doctrine §1 / Stage 6.2 — no keyword-only gating.**
  `assessTangentRisk` reads typed channels, qualifier tag codes, META-001
  structural codes, LIFE-001 lifecycle state, BR-001 topology — never raw body
  text for keyword matching (§7 #7).
- **cdiscourse-doctrine §2 / §3 — heat / popularity are not signals.** The model
  reads no strength band, heat, reply count, or engagement metric (§7 #16).
- **cdiscourse-doctrine §4 / §7 — no AI call.** `assessTangentRisk` is
  deterministic pure TS. The semantic classifier is explicitly RULE-006.
- **cdiscourse-doctrine §5 — pure models.** `tangentRoutingModel.ts` imports
  types + the frozen copy table only; no React, no Supabase, no network, no
  mutation, no async.
- **cdiscourse-doctrine §8 — soft-delete only.** Routing to a side branch posts
  the move *as a branch*; it is collapsed-capable and fully accessible. Mainline
  demotion is reversible (§7 #10). No `is_deleted`, no row removal.
- **cdiscourse-doctrine §9 — plain language.** Every string is read from
  `TANGENT_ROUTING_COPY`; no snake_case leak; ban-list test enforced.
- **cdiscourse-doctrine §10 — no v1 scope item.** No voting, search,
  notifications, OAuth, public API.
- **timeline-grammar — "a tangent is a topology label, not a verdict."** BR-003
  reuses BR-001's `RailBranchKind` topology classification directly; it adds no
  new node visual and no truth-flavored label. The "Side issue" plain label
  matches `timeline-grammar`'s `branch` node label exactly.
- **BR-003 card doctrine — "structural redirect, no person labels."** The
  assessment describes the move's structural relationship to its parent; the
  copy describes "this move" / "the thread"; the mainline-demotion advisory
  names the *thread*, never the participant (§4). Ban-list test enforces it.
- **BR-003 card doctrine — "advisory only, never blocks."** §7 #14 +
  `tangentRoutingNoBlock.test.ts`.
