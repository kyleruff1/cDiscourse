## Discrepancies between the card body and the real code (read this first)

The card body is mostly accurate, but three of its stated assumptions are wrong against `main` as it stands today. None of them block the card; the design below is written against reality.

1. **`publicOrigin` is NOT already in the runtime-env contract.** The card's context note says `__CDISCOURSE_RUNTIME_ENV__.publicOrigin` and a "`publicOrigin`-style mechanism" are "already referenced in `src/lib/supabase.ts`, `scripts/build/inject-runtime-env.mjs`, and `scripts/runtime/server.mjs`." They are not. HOST-001b shipped exactly two runtime-env slots — `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY` — and nothing else. QOL-023 must **add** the `publicOrigin` slot to all three files, extending the existing HOST-001b pattern (it does not extend an existing slot). This is the largest piece of new work in the card and the card's effort estimate likely under-counted it.

2. **There is no Expo Router `app/` directory and `expo-constants` is NOT installed.** The card says "for native, read from `Expo.Constants.expoConfig?.extra?.publicOrigin`." `expo-constants` is not in `package.json`, and the card simultaneously forbids new dependencies ("No new dependency. Pure TS, RN-safe."). Installing `expo-constants` would also violate the dependency policy (no speculative installs). **Resolution:** the native path does NOT use `expo-constants`. Native builds read `publicOrigin` from `process.env.EXPO_PUBLIC_APP_ORIGIN` (babel-injected `EXPO_PUBLIC_*` is the existing native-config mechanism — see `src/lib/supabase.ts` lines 47-53 and `devEnvironmentModel.ts` which already reads `EXPO_PUBLIC_APP_URL`). This keeps the card dependency-free and consistent with how every other public config value reaches native today. The `extra.publicOrigin` idea is recorded under "Out of scope / future" only.

3. **`adminSendPasswordReset` already accepts an optional `redirectTo`.** `src/features/admin/adminApi.ts` line 84-90 already has `adminSendPasswordReset(input: { userId?, email?, redirectTo? })`, and the `admin-users` Edge Function already forwards `redirectTo` into `generateLink({ type: 'recovery', options: { redirectTo } })` (`supabase/functions/admin-users/index.ts` lines 341-371). So QOL-023 does **not** add a `redirectTo` parameter — it adds a **default**: when the caller passes no `redirectTo`, derive one from `buildAuthRedirectUrl`. No Edge Function change is needed for the admin path.

Also confirmed against reality:
- `signUpWithEmailPassword(email, password, displayName?)` — confirmed signature (`authApi.ts` lines 39-43). The card's name `signUpWithEmailPassword` is correct.
- There is **no** existing password-reset wrapper in `authApi.ts`. The card asks for "a new minimal wrapper around `supabase.auth.resetPasswordForEmail`" — this is genuinely new.
- `src/lib/supabase.ts` already exports `readRuntimeEnv()` and `SUPABASE_CONFIGURED`. The helper will piggy-back on the same `__CDISCOURSE_RUNTIME_ENV__` global, but `readRuntimeEnv()`'s return type is currently narrowed to the two Supabase keys — its interface `CDiscourseRuntimeEnv` must gain a `publicOrigin?` field.
- `src/lib/auth/` does **not** exist yet — it is a new directory.

None of these conflict with doctrine, so the card proceeds.

## Goal (one paragraph)

Auth emails sent by Supabase (signup confirmation, magic link, password reset, email change, invite) currently redirect to whatever **Site URL** is set in the Supabase Dashboard, which today is `localhost`. `src/features/auth/authApi.ts` never passes `emailRedirectTo` to `supabase.auth.signUp`, so a user invited to the deployed dev app receives a link pointing at `http://localhost:8081` and bounces — onboarding is broken in the deployed environment. QOL-023 ships one pure-TS helper, `buildAuthRedirectUrl`, that deterministically computes the correct redirect URL from a runtime origin (the deployed origin in hosted runs, `http://localhost:8081` only in a true local dev build), wires it into the three auth call sites that send emails, and extends the HOST-001b runtime-env injector with a `publicOrigin` slot so the deployed origin reaches the web bundle at container start. Doctrine constraints that shape the design: **no secrets** (the origin is a public URL, never an env-file read at runtime — runtime-env injection only); **no service-role** (the production app and the helper never touch service-role; the admin path already runs server-side); **plain-language only** (the `InvalidAuthRedirectOrigin` error is mapped to user-facing copy with no raw code); **no new dependency** (pure TS, RN-safe); **fail-closed** (a non-https origin in a non-dev build throws rather than silently shipping a broken or unsafe link).

## Data model

No new database tables, no migration, no schema change. QOL-023 is pure client + build-script code.

The only data shapes are TypeScript interfaces in the new helper module:

```ts
// src/lib/auth/buildAuthRedirectUrl.ts

/** The five Supabase auth-email flows that carry a redirect. */
export type AuthRedirectKind =
  | 'confirm_signup'
  | 'magic_link'
  | 'password_reset'
  | 'invite'
  | 'email_change';

export interface BuildAuthRedirectUrlInput {
  /** Which auth flow the email belongs to. Drives the default route. */
  kind: AuthRedirectKind;
  /**
   * The deployed/host origin, e.g. 'https://dev.cdiscourse.com'.
   * `null` when no origin was injected (local dev) — fallback logic applies.
   * Never an env-file read; always sourced via resolveRuntimeOrigin().
   */
  runtimeOrigin: string | null;
  /**
   * True when the build is a local development build (`__DEV__ === true`).
   * Controls the localhost fallback and relaxes the https-only rule.
   */
  isDev: boolean;
  /**
   * Optional explicit route path (must begin with '/'). When omitted,
   * defaults per `kind` (see route-defaulting rules below).
   */
  route?: string;
}
```

The runtime-env contract gains one optional field. `src/lib/supabase.ts` already declares `interface CDiscourseRuntimeEnv`; extend it:

```ts
interface CDiscourseRuntimeEnv {
  EXPO_PUBLIC_SUPABASE_URL?: string;
  EXPO_PUBLIC_SUPABASE_PUBLISHABLE_KEY?: string;
  EXPO_PUBLIC_APP_ORIGIN?: string; // NEW — public deployed origin, e.g. https://dev.cdiscourse.com
}
```

(The runtime-env JSON key is `EXPO_PUBLIC_APP_ORIGIN` for naming-symmetry with the two existing `EXPO_PUBLIC_*` keys. The *concept* the card calls `publicOrigin` is carried by this key. The injector CLI flag is `--app-origin` / `-AppOrigin` / `--app-origin`; see "API / interface contracts".)

## API / interface contracts

### `buildAuthRedirectUrl(input): string`

```ts
export function buildAuthRedirectUrl(input: BuildAuthRedirectUrlInput): string;
```

**Route-defaulting rules.** When `input.route` is omitted, the route is chosen by `kind`:

```ts
export const DEFAULT_AUTH_ROUTES: Record<AuthRedirectKind, string> = {
  confirm_signup: '/auth/callback',
  magic_link:     '/auth/callback',
  invite:         '/auth/callback',
  email_change:   '/auth/callback',
  password_reset: '/auth/reset',
};
```

When `input.route` is provided it overrides the default. The provided route MUST start with `/`; if it does not, the helper prepends one (normalization, not an error). The route is otherwise used verbatim (it may contain a query string or sub-path).

**Origin-resolution rules** (applied in order):

1. If `runtimeOrigin` is a non-empty string after trimming → use it (subject to validation/normalization below).
2. Else if `isDev === false` → use `HOSTED_FALLBACK_ORIGIN = 'https://dev.cdiscourse.com'`.
3. Else (`runtimeOrigin` null/empty AND `isDev === true`) → use `DEV_FALLBACK_ORIGIN = 'http://localhost:8081'`.

This is the only branch that may ever produce a `localhost` URL, and it requires `isDev === true`. A non-dev build with a missing origin lands on rule 2 — the hosted fallback — never on localhost.

**Scheme / host validation.** The resolved origin is validated *after* resolution and *before* the URL is assembled:

- Reject — throw `InvalidAuthRedirectOrigin` — if the origin (case-insensitive, after trim) starts with `file:`, `javascript:`, `data:`, `vbscript:`, or `blob:`. These are rejected in **all** builds, dev and non-dev.
- Reject — throw `InvalidAuthRedirectOrigin` — if the origin does not begin with `https://` **and** the build is non-dev (`isDev === false`). In a dev build, `http://` is permitted (this is how `http://localhost:8081` is allowed). `http://` in a non-dev build throws.
- Reject — throw `InvalidAuthRedirectOrigin` — if the origin has no parseable host (e.g. `https://`, `https:///`, empty after scheme).
- The helper does **not** restrict the host to `dev.cdiscourse.com` — a custom origin injected via `EXPO_PUBLIC_APP_ORIGIN` is accepted as long as it is `https://` (non-dev) and has a host. This mirrors `inject-runtime-env.mjs`'s `looksLikeSupabaseUrl`, which deliberately does not pin the domain.

**Normalization** (applied to a valid origin before assembly):

- Trim leading/trailing whitespace.
- Strip every trailing `/` from the origin (so `https://dev.cdiscourse.com/` and `https://dev.cdiscourse.com` produce identical output).
- Lowercase the **scheme and host** only. The path/route is **not** lowercased (routes can be case-sensitive). Implementation note: parse with the global `URL` (available in RN via `react-native-url-polyfill/auto`, already imported in `src/lib/supabase.ts` line 1, and in Node/JSDOM natively) — `new URL(origin)` then read `.protocol` + `.host` (both already lowercased by the WHATWG URL parser) and discard `.pathname`. Do not hand-roll string casing.
- If a port is present it is preserved (`:8081` survives).

**Assembly.** Result = `<normalized-origin>` + `<route>` where `route` is guaranteed to start with exactly one `/`. The origin contributes no trailing slash, so there is never a `//` between origin and route. Example outputs:

| `kind` | `runtimeOrigin` | `isDev` | `route` | result |
|---|---|---|---|---|
| `confirm_signup` | `https://dev.cdiscourse.com` | `false` | — | `https://dev.cdiscourse.com/auth/callback` |
| `password_reset` | `https://dev.cdiscourse.com` | `false` | — | `https://dev.cdiscourse.com/auth/reset` |
| `magic_link` | `null` | `true` | — | `http://localhost:8081/auth/callback` |
| `magic_link` | `null` | `false` | — | `https://dev.cdiscourse.com/auth/callback` (hosted fallback) |
| `invite` | `https://DEV.CDiscourse.com/` | `false` | — | `https://dev.cdiscourse.com/auth/callback` (host lowercased, slash trimmed) |
| `email_change` | `https://dev.cdiscourse.com` | `false` | `custom/path` | `https://dev.cdiscourse.com/custom/path` (slash prepended) |
| `confirm_signup` | `file:///etc/passwd` | `true` | — | **throws `InvalidAuthRedirectOrigin`** |
| `confirm_signup` | `http://evil.example` | `false` | — | **throws `InvalidAuthRedirectOrigin`** (http in non-dev) |

### `InvalidAuthRedirectOrigin`

```ts
export class InvalidAuthRedirectOrigin extends Error {
  readonly code = 'invalid_auth_redirect_origin';
  /** The reason class — never the raw origin value, to avoid leaking it into logs. */
  readonly reason: 'forbidden_scheme' | 'insecure_scheme' | 'no_host';
  constructor(reason: 'forbidden_scheme' | 'insecure_scheme' | 'no_host') {
    super(`Auth redirect origin rejected: ${reason}`);
    this.name = 'InvalidAuthRedirectOrigin';
    this.reason = reason;
  }
}
```

The error message carries only the `reason` enum, never the offending origin string — so a malformed origin cannot be echoed into a log line or a crash report. Callers map this to plain-language copy; they never surface `error.message` or `error.code` to the user.

### `resolveRuntimeOrigin(): string | null`

```ts
// src/lib/auth/resolveRuntimeOrigin.ts
export function resolveRuntimeOrigin(): string | null;
export function getIsDev(): boolean;
```

`resolveRuntimeOrigin` precedence:

1. **Web with a real location:** if `typeof window !== 'undefined'` and `window.location?.origin` is a non-empty string that is not `'null'` (the string literal a sandboxed iframe yields) → return it. This makes local `npm run web` "just work" — the browser's own origin (`http://localhost:8081`) is used directly, which is the intended local-dev behaviour.
2. **Web Cloud Run shim:** else read `readRuntimeEnv().EXPO_PUBLIC_APP_ORIGIN` (the HOST-001b global). Return it if non-empty.
3. **Native + local dev fallback:** else read `process.env.EXPO_PUBLIC_APP_ORIGIN`. Return it if non-empty.
4. Else return `null` — `buildAuthRedirectUrl`'s fallback rules take over.

`getIsDev()` returns `typeof __DEV__ !== 'undefined' ? __DEV__ : false` — fail-closed to `false` (treat unknown as production, never relax the https rule by accident).

**Why two modules.** `buildAuthRedirectUrl.ts` is 100%-pure and has no `window` / `process` / `__DEV__` references, so its unit suite needs no environment mocking. `resolveRuntimeOrigin.ts` is the thin impure boundary; `authApi.ts` / `adminApi.ts` call `buildAuthRedirectUrl({ runtimeOrigin: resolveRuntimeOrigin(), isDev: getIsDev(), ... })`.

```ts
export async function sendPasswordResetEmail(email: string): Promise<AuthResult> {
  if (!SUPABASE_CONFIGURED) {
    return { ok: false, error: 'config_missing', message: 'Supabase is not configured. …' };
  }
  const redirectTo = safeBuildRedirect('password_reset');
  const { error } = await supabase.auth.resetPasswordForEmail(

  );

```

`safeBuildRedirect` is a small private helper in `authApi.ts`:

```ts
function safeBuildRedirect(kind: AuthRedirectKind): string | null {
  try {
    return buildAuthRedirectUrl({ kind, runtimeOrigin: resolveRuntimeOrigin(), isDev: getIsDev() });
  } catch {
    // InvalidAuthRedirectOrigin — fall back to undefined so Supabase uses its
    // dashboard Site URL rather than crashing the signup/reset call.
    return null;
  }
}
```

**Doctrine note:** when `buildAuthRedirectUrl` throws, the auth call still proceeds — it just omits `emailRedirectTo`, falling back to the dashboard Site URL. A bad origin must never block a user from signing up (parallels "score never blocks posting" — a redirect-config defect is advisory, not a hard gate). The `InvalidAuthRedirectOrigin` → plain-language mapping is for the *callback screen* path (a future card actually renders `/auth/callback`); QOL-023 only needs `mapAuthError` to have a branch and a plain string ready.

`mapAuthError` gains:

```ts

  return 'redirect_invalid';
```

and the plain-language copy `"We couldn't open that link. Try again or contact an admin."` is the message string callers show for `redirect_invalid` (mapped in whatever component renders `AuthError` → copy; QOL-023 adds the union member and the string, does not redesign the auth UI).

## Edge cases

The implementer must handle each of these explicitly; each has a named test (see test plan).

- **Empty / whitespace `runtimeOrigin`** (`''`, `'   '`) → treated as `null` for fallback purposes (trim, then check non-empty). Not an error.
- **`runtimeOrigin` is the string `'null'`** (sandboxed-iframe `window.location.origin`) → `resolveRuntimeOrigin` skips it (rule 1 excludes the literal `'null'`); `buildAuthRedirectUrl` itself, if handed `'null'`, treats it as a non-empty origin and will throw `no_host` (the `URL` parser rejects it) — acceptable, callers catch and fall back.
- **Trailing slashes** — single (`https://x/`) and multiple (`https://x///`) — all stripped.
- **Mixed-case host** (`https://DEV.CDiscourse.COM`) → host lowercased, path preserved.
- **Origin with a port** (`http://localhost:8081`) → port preserved.
- **Origin already including a path** (`https://dev.cdiscourse.com/app`) → the path segment is **discarded**; only scheme+host(+port) form the origin. Documented behaviour — `runtimeOrigin` is an *origin*, not a base URL. (Edge note for the implementer: `EXPO_PUBLIC_APP_ORIGIN` should be set to a bare origin by the operator; if someone sets `https://dev.cdiscourse.com/dev`, the `/dev` is dropped. Call this out in the operator runbook.)
- **Forbidden scheme in a dev build** (`file:`, `javascript:`, `data:`, `vbscript:`, `blob:`) → throws `forbidden_scheme` even when `isDev === true`. Dev does not relax the dangerous-scheme ban.
- **`http://` origin in a non-dev build** → throws `insecure_scheme`.
- **`http://` origin in a dev build** → allowed (this is the `localhost` path).
- **`route` without a leading slash** (`auth/callback`) → `/` prepended; not an error.
- **`route` is an empty string** → treat as omitted; fall back to the per-`kind` default route. (An empty explicit route should not produce `https://host` with no path.)
- **`route` with a query string** (`/auth/callback?foo=bar`) → used verbatim after slash-normalization.
- **Missing `runtimeOrigin` + non-dev build** → hosted fallback `https://dev.cdiscourse.com`, never localhost. This is the core acceptance criterion.
- **Network / offline** — N/A for the helper itself (it is synchronous and pure). The *callers* (`signUp`, `resetPasswordForEmail`) already return `network_error` via `mapAuthError`; QOL-023 does not change that path. A failed `buildAuthRedirectUrl` never throws out of the caller — `safeBuildRedirect` catches and returns `null`.
- **Concurrent edits / permission-denied** — N/A. No DB writes, no rows, no RLS surface in QOL-023.
- **`__DEV__` undefined** (some Jest configs) → `getIsDev()` returns `false` (fail-closed). Tests that need `isDev: true` pass it explicitly to `buildAuthRedirectUrl`; they do not rely on the global.
- **`window` undefined** (native, Node tests) → `resolveRuntimeOrigin` skips rule 1 cleanly via the `typeof` guard, exactly like `readRuntimeEnv` already does.
- **Injector: `EXPO_PUBLIC_APP_ORIGIN` absent** → injector writes the file without the key; the helper's fallback covers it. Not a build failure.
- **Injector: `EXPO_PUBLIC_APP_ORIGIN` carries a forbidden secret shape** → injector refuses (new exit code), same posture as the existing URL/key forbidden-shape refusal.
- **Doctrine edge — could a bad origin block signup?** No. `safeBuildRedirect` catches `InvalidAuthRedirectOrigin` and returns `null`; the auth call proceeds with `emailRedirectTo` omitted. A redirect-config defect degrades to the dashboard Site URL; it never gates a user out of the app.

## Risks

- **`expo-constants` mismatch (already resolved in this design).** The card body asked for `Expo.Constants.expoConfig?.extra?.publicOrigin`, but the package is not installed and the card forbids new deps. The design uses `process.env.EXPO_PUBLIC_APP_ORIGIN` for native instead. **Risk for the implementer:** do not "fix" the design back toward `expo-constants` — that would break the no-new-dependency constraint. The native path is `EXPO_PUBLIC_APP_ORIGIN` via babel-injected env, full stop.
- **`__DEV__` in the production web bundle.** Metro defines `__DEV__` at bundle time; in a production web build it is `false`. The hosted fallback therefore engages correctly. But if a hosted build were ever bundled in dev mode, `getIsDev()` would return `true` and a missing origin would fall to `localhost`. Mitigation: the runtime-env injector *always* supplies `EXPO_PUBLIC_APP_ORIGIN` in the Cloud Run path once the operator wires the secret, so rule 1/2 of `resolveRuntimeOrigin` wins before the fallback is ever consulted. The operator runbook makes setting that secret a required step.
- **`window.location.origin` on web overrides the injected origin.** On web, rule 1 of `resolveRuntimeOrigin` uses the browser's own origin. For the deployed SPA served from `https://dev.cdiscourse.com`, that *is* the correct origin, so this is desirable. The injected `EXPO_PUBLIC_APP_ORIGIN` matters mainly for native and as a documented fallback. This is intentional but worth a comment in the helper so a future reader does not "fix" it.
- **`AuthError` union shape.** The design assumes `AuthError` in `src/features/auth/types.ts` is a string-literal union and adds `'redirect_invalid'`. The implementer must read `types.ts` first — if `AuthError` is structured differently, adapt. (Low risk; `mapAuthError` returns string literals today, strongly implying a literal union.)
- **Static-scan test brittleness.** `noLocalhostInProdAuthPaths.test.ts` must not pass vacuously if the file walker is misconfigured. The negative-control assertion (`> 50` files visited) mitigates this. The implementer must also keep the allowlist tight — adding files to it weakens the guard.
- **Existing auth tests.** Any current test of `signUpWithEmailPassword` that asserts the exact `signUp` argument object will break when `emailRedirectTo` is added. The implementer must `Grep __tests__/ for 'signUpWithEmailPassword'` and update those assertions to tolerate / expect the new `options.emailRedirectTo` field. This is a known, expected test update — not a regression.
- **Operator deploy gap.** The redirect URLs only work end-to-end once the operator updates the Supabase Dashboard **Site URL** and **Additional Redirect URLs** allow-list. Until then, Supabase rejects the redirect and falls back to Site URL. The design degrades safely (no crash), but onboarding is not actually fixed until the operator step runs. This is called out loudly in "Operator steps."
- **No router for `/auth/callback`.** There is currently no `app/` route directory. The constructed URL is correct, but the screen that consumes the token at `/auth/callback` does not exist yet. QOL-023 fixes the *email link target*; a follow-up card builds the callback screen. The acceptance criterion ("no email points to localhost") is still fully met by QOL-023 alone.
