## Â§1 â€” Goal

Replace loose "accuse / flag" affordances with a **procedural allegation object**. A
concern about another participant's move is only expressible as (a) a required target
quote, (b) a bounded concern type, and (c) a bounded requested remedy â€” never as a
free-form label on a person. Every participant pays the same three-step procedural cost,
which is what keeps the playing surface level against social pressure.

The design is shaped by these doctrine constraints (`cdiscourse-doctrine`):

- **Â§10a â€” Observations vs Allegations.** Machine signals are Observations; user-applied
  tags are Allegations. The structured concern is a **third, separate object**: a
  *moderation concern*, distinct from gameplay Allegations and from machine Observations.
  The design erects a wall (Â§7) so the three never cross-contaminate.
- **Â§10a sensitive-surface rule.** Person-directed concerns
  (`about_person_not_claim`, `harassment_concern`) read as accusation if surfaced on the
  target's node. They are gated composer-only / moderator-visible and **never**
  `public_after_review` pre-review.
- **Â§1 / Â§2.2 no person-verdict labels.** A user may never publicly attach a
  person-verdict label to a participant. The procedural allegation is the *only* channel,
  and it is reviewed by a human before any public effect.
- **Acceptance-gate invariant (Â§14).** Nothing in this card blocks, rejects, routes, or
  delays an ordinary user post. The deterministic rules engine is the sole submission
  gate. A concern is composed *after* a move is stored, and never gates posting.
- **Â§8 soft-dismiss.** No `argument_flags` row is ever deleted; v1 introduces no delete
  path of any kind.

## Â§2 â€” Persistence decision (THE central fork) â€” Option (b), with evidence

The issue requires the design to decide, with quoted schema evidence, whether the existing
moderation surface can carry the structured concern (target quote + concern type + remedy
+ visibility) **without any migration or Edge change**. It cannot. Four independent
blockers, any one of which is decisive.

### 2.1 The live `argument_flags` schema

`supabase/migrations/20260516000001_initial_schema.sql:242-261` (read, never edited):

```sql
CREATE TABLE public.argument_flags (
  id          uuid  PRIMARY KEY DEFAULT gen_random_uuid(),
  debate_id   uuid  NOT NULL REFERENCES public.debates(id) ON DELETE CASCADE,
  argument_id uuid  NOT NULL REFERENCES public.arguments(id) ON DELETE CASCADE,
  flag_code   text  NOT NULL REFERENCES public.flag_definitions(code) ON DELETE RESTRICT,
  rule_code   text  REFERENCES public.constitution_rules(code) ON DELETE SET NULL,
  source      text  NOT NULL CHECK (source IN (
                      'client_rules','server_rules','semantic_adapter',
                      'user_report','moderator')),
  confidence  numeric CHECK (confidence BETWEEN 0 AND 1),
  status      text  NOT NULL DEFAULT 'open'
                    CHECK (status IN ('open','needs_review','confirmed','dismissed')),
  payload     jsonb NOT NULL DEFAULT '{}',
  created_by  uuid  REFERENCES public.profiles(id) ON DELETE SET NULL,
  resolved_by uuid  REFERENCES public.profiles(id) ON DELETE SET NULL,
  created_at  timestamptz NOT NULL DEFAULT now(),
  resolved_at timestamptz
);
```

### 2.2 Blocker A (decisive) â€” there is no visibility axis, and the RLS that exists makes flags *public to the target and all participants*

The structured concern requires `visibility: 'composer_only' | 'moderator_visible' |
'public_after_review'`. `argument_flags` has **no visibility column**. Worse, the existing
SELECT policy makes any row routed here readable by exactly the parties the doctrine says
must NOT see a person-directed concern pre-review
(`supabase/migrations/20260516000002_rls_policies.sql:294-308`):

```sql
CREATE POLICY "argument_flags: authors and mods can read"
ON public.argument_flags FOR SELECT TO authenticated
USING (
  is_moderator_or_admin()
  OR EXISTS (SELECT 1 FROM public.arguments a
             WHERE a.id = argument_flags.argument_id AND a.author_id = auth.uid())
  -- Participants can see flags on arguments in debates they joined
  OR EXISTS (SELECT 1 FROM public.debate_participants
             WHERE debate_id = argument_flags.debate_id AND user_id = auth.uid())
);
```

So a `harassment_concern` filed against a move would be readable by **(1) the move's
author â€” i.e. the target of the concern** and **(2) every participant in the room**. That
is the exact "surfacing reads as accusation" failure doctrine Â§10a forbids. Making
`argument_flags` safe for this object requires a **new visibility column + a new SELECT
policy** that gates person-directed rows to moderators only â€” i.e. a migration + RLS
change. That is REF-005B / GATE-C, explicitly out of v1 scope.

### 2.3 Blocker B â€” no home for `targetQuote` or `requestedRemedy`

There is no `target_quote` column and no `requested_remedy` column. The only place they
could go is `payload jsonb`. The issue is explicit: "if the fit requires bending the
schema (overloading columns, stuffing JSON into fields not meant for it), that is NOT a
fit." `payload` is a debugging/summary cache, not a typed remedy/visibility carrier.
Stuffing the remedy + visibility + quote into it is exactly the disqualifying overload.

### 2.4 Blocker C â€” `flag_code` is a NOT NULL FK to a seeded registry

`flag_code text NOT NULL REFERENCES public.flag_definitions(code)`. The 6
`ReviewConcernType` values are not `flag_definitions` codes. Seeding them is a
migration-bearing change (`flag_definitions` is seeded in
`20260516000003_seed_data.sql`; admin-only write per
`20260516000002_rls_policies.sql:128-132`).

### 2.5 Blocker D â€” there is no existing client write path to reuse

The issue permits reusing an existing RLS-gated client write path **only if it already
exists**. It does not. The only client interaction with `argument_flags` is a batched
**SELECT** (`src/features/arguments/argumentsApi.ts:296-298`):

```ts
supabase.from('argument_flags')
  .select('id,debate_id,argument_id,flag_code,rule_code,source,confidence,status,created_at')
  .in('argument_id', argumentIds)
```

A repo-wide search for a client `argument_flags` insert/upsert returns **zero** matches.
The RLS does technically allow an authenticated user to INSERT a `user_report` row
(`...:312-318`, `source IN ('user_report','client_rules')` with `created_by = auth.uid()`),
but no client code wires it, and creating that wire is forbidden by the issue. Even if it
were created, Blockers Aâ€“C still disqualify it.

### 2.6 Soft-dismiss reality (precision finding â€” implementer/reviewer must use these names)

`CLAUDE.md` and the issue body describe the soft-dismiss convention as
`dismissed = true`, `reviewed_by`, `reviewed_at`. **The live schema does not have those
columns.** The actual convention is:

- `argument_flags.status = 'dismissed'` (one of the four CHECK values)
  (`...001_initial_schema.sql:254-255`),
- `argument_flags.resolved_by` (`:258`) + `argument_flags.resolved_at` (`:260`),
- with the decision trail in a separate append-only `public.moderation_reviews` table
  (`reviewer_id`, `decision IN ('confirm','dismiss','escalate')`, `notes`)
  (`...001_initial_schema.sql:301-308`; mods-only RLS `...002_rls_policies.sql:361-370`).

The table comment is binding: *"Flags raised on arguments. Rows are never deleted; use
status = dismissed."* (`...001_initial_schema.sql:263`). The
`requestReviewFlagsConventions.test.ts` plan (Â§11) pins these **real** names and asserts
v1 introduces no delete path. This naming drift is a documentation precision issue, not a
blocker â€” but the implementer MUST encode the real column names.

### 2.7 Decision

**Option (b): draft / composer-local only.** v1 composes and validates the structured
concern entirely client-side. Claim-level remedies launch a real engine-gated move via
the shipped disagreement loop (Â§5). Moderator-queue remedies hand off to the **existing,

Edge function, no migration lands in this card.**

**Named follow-up filed by this design: REF-005B (GATE-C)** â€” adds the
`structured_concerns` table (or `argument_flags` visibility column), the moderator-only
SELECT RLS for person-directed rows, the submission Edge function, and the real moderator

## Â§5 â€” API / interface contracts (`requestReviewModel.ts`)

### 5.1 Validation

```ts
/**
 * True iff the draft is submittable: target node id present, target quote
 * non-empty after trim, concern type in the bounded set, remedy in the
 * bounded set. Pure. Never throws. Returns false for null/undefined/partial.
 */
export function canSubmitConcern(
  draft: Partial<StructuredConcernDraft> | null | undefined,
): boolean;
```

Rule (all four required; matches AC "cannot be submitted without target quote + type +
remedy"):
- `typeof targetNodeId === 'string' && targetNodeId.length > 0`
- `typeof targetQuote === 'string' && targetQuote.trim().length > 0`
- `concernType` âˆˆ `ALL_REVIEW_CONCERN_TYPES`
- `requestedRemedy` âˆˆ `ALL_REVIEW_REQUESTED_REMEDIES`

`visibility` is NOT required as an input to `canSubmitConcern` â€” it is derived. A finalize
helper `buildSubmittableConcern(partial)` returns `StructuredConcernDraft | null`: null
when `canSubmitConcern` is false, otherwise a frozen draft with `visibility` set from
`deriveConcernVisibility`.

unchanged "Send for review" callback (`handleAction('flag', messageId)`) as the interim
notify; REF-005B replaces that with the real persisted queue.

### 5.3 Visibility derivation (the Â§10a pin)

```ts

 * Derive the visibility a concern must carry. Pure, total. v1 is
 * conservative: it NEVER returns 'public_after_review' (no concern has any
 * public effect pre-review in v1; only a human moderator in REF-005B may
```

Binding derivation:

1. If `PERSON_DIRECTED_CONCERN_TYPES.has(concernType)` â†’ **`'moderator_visible'`**.

3. Else (claim-level type + claim-level remedy) â†’ `'composer_only'`
   (the concern is a launcher; the *resulting move* is what becomes public, via the
   engine gate â€” the concern object itself stays composer-local).

**v1 never emits `'public_after_review'` for any of the 36 (type Ã— remedy) combinations.**
The value remains in the union (binding from the issue) but is reserved for REF-005B's
post-review moderator-publish path. Pinned by `requestReviewModel.test.ts`.

## Â§6 â€” Composer UI

### 6.1 Where it lives

- **Entry point.** The `flag` Act entry (`actPopoutModel.ts:324-332`) keeps its id but its
  label becomes **"Request review / Mark concern"** and its handler opens the structured
  composer instead of firing an immediate notify. The bare `ArgumentBubbleActions` `flag`
  chip (`:19`) is relabeled to match and routes to the same composer. Per
  `expo-rn-patterns`, this is a *bounded relabel + handler rebind*, not a new compose
  system â€” it does not rebuild the one-box chassis.
- **Mount.** A new presentational component `RequestReviewComposer.tsx` mounts as a
  composer-side overlay in `ArgumentGameSurface.tsx`, a sibling to the existing Act/Inspect
  mounts â€” analogous to how UX-001.5A mounts `NodeLabelInspectGroups` adjacent to the
  InspectPopout. It is only ever shown on the actor's own composer surface; it never
  renders on a target's node.

### 6.2 The three-step structured flow

Step 1 â€” **Quote.** A required `TextInput` seeded from the target move's
`target_excerpt` / selected quote when available; the user may trim to the exact passage.
Empty quote â†’ step 2/3 disabled, submit disabled. (`accessibilityLabel="Quote the exact
passage you are concerned about"`, `accessibilityLiveRegion="polite"` for the
empty-quote error.)

Step 2 â€” **Concern type.** Six `Pressable` radio chips (`accessibilityRole="radio"`,
`accessibilityState={{ checked }}`), labels from `CONCERN_TYPE_LABELS`. Exactly one
selectable. No free-text field exists anywhere in this step â€” bounded choices only.

Step 3 â€” **Remedy.** Six `Pressable` radio chips, labels from `REMEDY_LABELS`. The
composer calls `deriveConcernVisibility(type, remedy)` live and shows a plain-language
visibility readout ("Only a moderator will see this until it is reviewed." for
`moderator_visible`; "This stays on your screen and opens a move." for `composer_only`).

Submit â€” enabled only when `canSubmitConcern(draft)` is true. On submit:
- routing `act_entry` â†’ `onRouteToActEntry(actEntryId)` (host calls `enterBoxForActEntry`).
- routing `moderator_queue` â†’ `onSendForModeratorReview(draft)` (host calls the existing
  `handleAction('flag', messageId)` interim; draft retained for REF-005B). The composer
  itself **never** hides, deletes, or mutates any content.

### 6.3 Accessibility (per `accessibility-targets`)

- Every chip â‰¥ 44Ã—44 (visual or `hitSlop`); radios expose role + label + `checked` state.
- Color is never the only signal: selected chips carry a check glyph + stroke, not just a

## Â§7 â€” The wall: gameplay Allegations vs moderation concerns vs machine Observations

Three objects, three modules, no shared registry. The design's load-bearing separation:

| Object | Module | Schema | Renders where |
|---|---|---|---|
| **Machine Observation** | `src/features/nodeLabels/machineObservationRegistry.ts` | `NodeLabelMark` (`kind: 'machine_observation'`) | node strips / inspect (composer-only for sensitive) |
| **Gameplay Allegation** (manual tag) | `src/features/nodeLabels/userAllegationRegistry.ts:139`, `src/features/metadata/moveMetadataLedger.ts` | `NodeLabelMark` (`kind: 'user_allegation'`, `source: 'manual_tag'`) â€” 10 `ManualTagCode`s | node strips / inspect |
| **Moderation concern** (this card) | `src/features/requestReview/requestReviewModel.ts` | `StructuredConcernDraft` (`concernType` + `remedy` + `visibility`) | composer-only / (REF-005B) moderator queue |

Wall invariants (enforced by `requestReviewWall.test.ts`, Â§11):

1. `StructuredConcernDraft` is **never** converted to a `NodeLabelMark` and **never**
   passed to a node-label adapter or registry. The `requestReview` module imports nothing
   from `nodeLabels/` registries and exports no `NodeLabelMark`.
2. A `ManualTagCode` (gameplay Allegation) **never** gains a `requestedRemedy` or a
   moderation `visibility`. `moveMetadataLedger` / `userAllegationRegistry` do not import
   from `requestReview`.
3. The name overlap is intentional and harmless: `needs_source` / `needs_quote` exist as
   *both* a `ManualTagCode` (gameplay) and a `ReviewConcernType` (moderation). They are
   distinct objects with distinct lifecycles; the wall test asserts they are produced by
   distinct modules and never merged. (The UX-001.5A `userAllegationRegistry` doctrine
   already separates gameplay tags from machine Observations; this card adds the third
   wall, to moderation concerns.)
4. A moderation concern is **not** the same as a `user_report` `argument_flags` row.
   v1 does not write one; REF-005B introduces the persisted form with its own table/RLS.

## Â§9 â€” Edge cases

- **Empty / whitespace-only quote** â†’ `canSubmitConcern` false; submit disabled; inline
  live-region error. No partial submit.
- **Quote present, no type or no remedy** â†’ submit disabled (all three required).
- **Person-directed type chosen** â†’ `deriveConcernVisibility` forces `moderator_visible`;
  the composer's visibility readout updates; no public/target-node surface is ever
  produced. (UI pin in `requestReviewVisibility.test.tsx`.)
- **`hide_pending_review` chosen** â†’ routing is `moderator_queue`; **nothing hides**; the
  composer shows "A moderator will decide whether to hide this." There is no client code
  path that flips any `arguments.status` to `hidden`.
- **Concern composed against your own move** â†’ the entry is offered only on *other*
  participants' bubbles (mirrors the existing actor-aware `ArgumentBubbleActions` split:
  own bubble exposes only `view_qualifiers` / `request_deletion`,
  `ArgumentBubbleActions.tsx:4-7`). Own-bubble path never reaches the composer.
- **Unknown / foreign concernType or remedy** (defensive) â†’ `canSubmitConcern` false;
  `routeRemedy` is total over the bounded union, so the type system prevents foreign values
  at compile time; a runtime guard drops them.
- **Offline / network failure** â†’ v1 composes locally and (claim-level) launches an
  engine-gated move through the existing submit path, which already owns its own offline
  handling; the concern object adds no new network call, so there is no new failure mode.
- **Concurrent edits / double-submit** â†’ the composer disables submit on first dispatch
  and closes; re-opening starts a fresh draft. No persisted row means no concurrent-write
  race in v1.
- **Permission-denied** â†’ there is no client write in v1, so no RLS denial surfaces; the
  moderator-queue interim uses the existing `handleAction` callback, whose permission
  behavior is unchanged by this card.
- **Doctrine edge â€” "can heat/popularity influence which concerns are allowed?"** â†’ no.
  The concern vocabulary is fixed and bounded; nothing about engagement, view count, or
  heat gates, ranks, or changes a concern. (Â§2/Â§3 doctrine.)
- **Doctrine edge â€” "does a concern delay or block the targeted post?"** â†’ no. The post is
  already stored; a concern is composed after the fact and never feeds the submission gate.
