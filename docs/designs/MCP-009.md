## Goal (one paragraph)

MCP-009 designs the **implementation boundary** for the semantic-referee classifier: the precise seam where a redacted argument move, its parent, and minimal room context cross from the deterministic CDiscourse stack into a server-side AI provider, and where the `SemanticRefereePacket` crosses back. It answers one question — *in which server-side component does the provider call physically live, and how is it shaped, secured, and disabled?* — and it answers it without starting any service or making any call. The design is shaped by four doctrine constraints that are blockers if violated: (1) **no client-side AI call, ever** — the call lives only in an Edge Function or a server-only MCP adapter the Edge Function reaches (cdiscourse-doctrine §4, §7; MCP-001 §3.4); (2) **secrets are server-only** — `ANTHROPIC_API_KEY` and any MCP credential are read solely via `Deno.env.get()` inside the Edge Function, never returned, never logged, never present in `.env` / `app/` / `src/` (cdiscourse-doctrine §6; supabase-edge-contract); (3) **mock-by-default** — the provider registry defaults to a deterministic `mock` provider, a local `fixture` provider serves development, and the live provider is operator-gated behind an explicit env flag (MCP-001 §10; cdiscourse-doctrine §7); (4) **disabled-by-default is silent** — when the feature flag is off the boundary returns `{ enabled: false }` with HTTP 200 and the client falls back to the deterministic layer with no error thrown (MCP-001 §11; matches the existing `process-language-draft` behavior). The boundary is provider-agnostic by construction: it transports MCP-001's packet shape and never widens it — no `block` field, no truth field, `authoritative` always the literal `false`. *When* the call may run, caching, batching, and the token budget are explicitly **out of scope here** and owned by MCP-004; this card documents only that "the MCP-004 trigger plan gates this boundary's call path."

## Data model

**No new database table. No migration. No new persisted entity.** This card adds zero rows to any schema.

The boundary transports two existing/contracted shapes only:

1. **Inbound** — a new Edge Function request payload, `ClassifyMoveRequest` (defined in §"API / interface contracts" below). It is a transient request body, never persisted.
2. **Outbound** — MCP-001's `SemanticRefereePacket` (locked in [`docs/designs/MCP-001.md`](./MCP-001.md) §7), wrapped in a thin `enabled` discriminator: `ClassifyMoveOutcome = { enabled: false; reason: string } | { enabled: true; packet: SemanticRefereePacket }`. The packet is a transient response body in v0; whether a packet is ever persisted to a cache table is explicitly an MCP-004 decision (MCP-001 §12 calls a persisted cache "a future-card option, not v0"). This card does **not** introduce one.

The boundary therefore touches **no `public.arguments` row, no `flags` row, no audit table, no new table**. It reads room/parent context through the caller-scoped client (RLS-enforced) exactly as `process-language-draft` reads the `debates` table today, and writes nothing.

## API / interface contracts

### The three boundary options

The card scope lists three candidate placements. Each is evaluated against: doctrine fit, blast radius, secret surface, testability, MCP-001 invariant compliance, and operator clarity.

#### Option (a) — extend `process-language-draft` with a `classify-move` action

Add an `action: 'process_draft' | 'classify_move'` discriminator to the existing function; the function branches to the language-processing pipeline or the semantic-referee pipeline.

| Dimension | Assessment |
|---|---|
| Reuse | Highest — reuses the existing JWT/CORS/RLS scaffold and one deployed function. |
| Blast radius | **Poor.** Two unrelated features now share one deploy unit. A change to the classifier path forces a redeploy of language processing and vice versa; a bug in one can take down the other. They have different prompts, different schemas, different feature flags, different token budgets. |
| Secret surface | Mixed — `ANTHROPIC_API_KEY` already lives there, but the two features may want different model tiers / different enable flags, and co-locating them blurs which flag gates which call. |
| Flag clarity | **Poor.** `AI_LANGUAGE_PROCESSING_ENABLED` cannot cleanly gate a second, independent feature. Overloading it couples two operator decisions. |
| Testability | Mixed — the action branch adds a conditional that every `process-language-draft` test must now account for. |
| Verdict | **Not recommended.** Conflates two advisory features with different lifecycles, flags, and budgets into one deploy unit. |

#### Option (b) — a new dedicated `semantic-referee` Edge Function — **RECOMMENDED**

A new `supabase/functions/semantic-referee/index.ts`, its own `_shared/semanticReferee/` module tree, its own `SEMANTIC_REFEREE_ENABLED` flag and `SEMANTIC_REFEREE_PROVIDER` selector.

| Dimension | Assessment |
|---|---|
| Reuse | High — *copies the proven pattern* (`http.ts` helpers, `supabaseClients.ts`, the `providers.ts` registry shape, the `enabled`-discriminator response) without *coupling deploy units*. Pattern reuse, not deploy-unit reuse. |
| Blast radius | **Best.** The semantic referee can be deployed, flag-flipped, rolled back, and load-tested independently. A classifier bug cannot affect language processing or `submit-argument`. |
| Secret surface | Clean — one function, one documented set of env var names, one flag. `Deno.env.get()` reads happen in exactly one module tree. |
| Flag clarity | **Best.** `SEMANTIC_REFEREE_ENABLED` is a single dedicated operator switch. Off by default. |
| Testability | **Best.** A focused function with its own schema, its own mock, its own ban-list scan. The future test plan (§"Test plan") maps 1:1 onto it. |
| MCP-001 fit | MCP-001 §11 lists exactly this as boundary option (b); §10 names the provider-registry reuse explicitly. |
| Verdict | **Recommended.** Operator confirmation requested (§"Dependencies / operator decisions"). |

#### Option (c) — a server-only MCP adapter the Edge Function calls

The Edge Function does not call the AI provider's HTTP API directly. Instead it calls an **operator-hosted MCP server** (a separate service that speaks the Model Context Protocol and itself owns the model call). The Edge Function holds an MCP service credential, not necessarily the model key.

| Dimension | Assessment |
|---|---|
| Reuse | Lower in v0 — introduces a *second* server component (the MCP server) that must be hosted, monitored, and secured. |
| Blast radius | The MCP server is a new operational surface and a new failure mode (its own uptime, its own latency). |
| Secret surface | Two credentials instead of one: the MCP service credential (in the Edge Function via `Deno.env.get()`) **and** the model key (now on the MCP server, never in Supabase). This can be a *security improvement* — the model key never touches Supabase secrets — but it doubles the things to rotate. |
| Strength | Genuine MCP capabilities: tool catalogs, multi-model routing, a centralized prompt/version registry, reuse across products. Worth it **if** the operator already runs MCP infrastructure or plans to. |
| Verdict | **Not recommended for v0**, but the design keeps option (c) cheap to adopt later: the provider registry already has an `mcp` slot (MCP-001 §10), so option (c) becomes "add `mcpAdapter.ts`, set `SEMANTIC_REFEREE_PROVIDER=mcp`" — no boundary redesign. The recommendation is **(b) now, (c) as a documented future provider behind the same registry.** |

**Recommendation: option (b)** — a dedicated `semantic-referee` Edge Function — with option (c)'s MCP adapter pre-reserved as a registry provider so the operator can adopt it later without a contract change. This is an operator decision flagged in §"Dependencies / operator decisions".

### Inbound request — `ClassifyMoveRequest`

Sent by the client wrapper `classifyMove()` to the `semantic-referee` function. Everything in it is **redacted** before it is sent (see §"Risks" and MCP-001 §20: strip X handles, URLs, JWT-shaped strings, Bearer/Authorization headers, `sk-ant-*` / `xai-*` / `sb_secret_*` token shapes, emails, 15–20 digit post ids — the redaction itself is a future-card concern shared with MCP-002/MCP-004; the boundary's contract is only that bodies arrive already redacted and the function defends with a second redaction pass before any prompt is built).

```ts
// Future implementation card — shape only.
export interface ClassifyMoveRequest {
  roomId: string;                       // RLS-checked: caller must be able to see this room
  moveId?: string;                      // omitted for a pre-send draft (not yet posted)
  parentId?: string;
  // Redacted bodies. The function runs a second redaction pass defensively.
  moveBodyRedacted: string;             // max 8000 chars, mirrors process-language-draft's cap
  parentBodyRedacted?: string;          // max 8000 chars
  // Deterministic layer-1 context (MCP-001 §6) — drives prompt context and the
  // fallback packet when the layer is disabled or the provider is unavailable.
  roomContext: {
    debateMode?: string;                // GAME-003 room mode code
    selectedAction?: string;            // quick-action preset
    selectedMoveType?: string;          // argumentType
    side?: 'affirmative' | 'negative' | 'observer' | 'moderator';
    actorRole?: 'initiator' | 'primary_opponent' | 'chime_in' | 'observer';
  };
  // Which classifiers to request (<= 5, MCP-001 §9 five-classifier packet prompt).
  // The trigger plan (MCP-004) decides which set; the boundary just transports it.
  requestedClassifiers: string[];       // subset of SemanticClassifierId
  // Provenance the boundary echoes back into the packet.
  promptVersionHint?: string;           // seed-bank prompt revision the caller expects
  contentHash: string;                  // hash of moveBodyRedacted — packet cache-key component
}
```

The request carries **no `block` field, no truth field, no score delta** — there is nothing in the inbound shape that could ask the model for a verdict.

### Outbound response — `ClassifyMoveOutcome`

```ts
// Future implementation card — shape only.
export type ClassifyMoveOutcome =
  | { enabled: false; reason: 'disabled' | 'key_missing' | 'not_configured' | 'not_implemented' }
  | { enabled: true; packet: SemanticRefereePacket };   // SemanticRefereePacket from MCP-001 §7
```

`SemanticRefereePacket` is transported **verbatim from MCP-001**. The boundary does not add, remove, or rename a single field. Key invariants the boundary must hold (all from MCP-001 §7):

- `authoritative` is the literal `false` — enforced by the outbound Zod schema (`z.literal(false)`); a non-`false` value is a validation failure and the boundary returns the deterministic fallback instead.
- `provider` is one of `'mock' | 'mcp' | 'anthropic' | 'operator-selected'` — set by the registry to match the provider that actually answered.
- Every `SemanticBinarySample.value` is `0` or `1`.
- No `block` field exists anywhere in the packet — the boundary cannot, by construction, return something that hard-gates a post.

### Edge Function shape (option b)

The `semantic-referee/index.ts` follows the standard contract from `supabase-edge-contract` and copies `process-language-draft/index.ts` step-for-step:

1. CORS preflight → `corsHeaders`.
2. Method guard → POST only.
3. Auth → require `Authorization` header; `callerClient.auth.getUser()`; 401 on failure.
4. Parse body → 400 on invalid JSON.
5. Validate → `ClassifyMoveRequestSchema.safeParse`; `validation_failed` with issue list on failure.
6. Room-access RLS check → `callerClient.from('debates').select('id').eq('id', roomId).maybeSingle()` — exactly the `process-language-draft` debate-access pattern. The caller-scoped client (anon key + caller JWT) means RLS decides visibility; the function never uses a service-role client because **it performs no privileged write** (supabase-edge-contract: "Build service-role client ONLY if a privileged write is required" — it is not).
7. Defensive redaction pass on `moveBodyRedacted` / `parentBodyRedacted` before any prompt is built.
8. Delegate → `classifyWithConfiguredProvider(request)`.
9. Return `ok(outcome)` — `{ enabled: false, reason }` **or** `{ enabled: true, packet }`, both HTTP 200. Disabled is not an error.

The function **never** calls `submit-argument`, **never** inserts into `public.arguments`, **never** inserts a `flags` row, **never** writes any table. It is a read-context + classify + return function.

### Provider registry — `classifyWithConfiguredProvider`

Reuses `languageProcessing/providers.ts` line-for-line in structure:

```ts
// Future implementation card — shape only. Mirrors languageProcessing/providers.ts.
export async function classifyWithConfiguredProvider(
  request: ClassifyMoveRequest,
): Promise<ClassifyMoveOutcome> {
  const enabled = Deno.env.get('SEMANTIC_REFEREE_ENABLED');
  if (enabled !== 'true') {
    return { enabled: false, reason: 'disabled' };          // ← disabled-by-default
  }
  const providerName = Deno.env.get('SEMANTIC_REFEREE_PROVIDER') ?? 'mock';  // ← mock default

  if (providerName === 'mock')    return { enabled: true, packet: runMockClassifier(request) };
  if (providerName === 'fixture') return { enabled: true, packet: runFixtureClassifier(request) };
  if (providerName === 'anthropic') { /* key check → unavailable → deterministic fallback packet */ }
  if (providerName === 'mcp')     { /* server-only MCP adapter — option (c) */ }
  return { enabled: false, reason: 'not_configured' };
}
```

Routing rules:

- **Default is `mock`** — note this differs from `languageProcessing/providers.ts`, which defaults `AI_LANGUAGE_PROCESSING_PROVIDER` to `'anthropic'`. The semantic referee defaults to `mock` per MCP-001 §10 ("the default in all tests and the fallback when the feature flag is off") and per cdiscourse-doctrine §7 (no live AI from the production app without an explicit operator gate). **Even with the feature flag on, an unset provider variable picks `mock`, never a live provider.** This is a deliberate, doctrine-driven deviation from the language-processing default.
- `mock` — deterministic, no network, schema-valid packet built from the request and the layer-1 context. Test default and the safe fallback.
- `fixture` — local development; reads a checked-in fixture map keyed by `contentHash`, falls through to `mock` on a miss. No key, no network.
- `anthropic` — live provider. If `ANTHROPIC_API_KEY` is missing it returns `{ enabled: false, reason: 'key_missing' }`; on API error or unparseable response it returns a **deterministic fallback packet** (`provider: 'mock'`, low-confidence binaries from layer-1 only) so the caller still gets a valid packet — the call never throws to the client.
- `mcp` — option (c)'s server-only adapter; reaches the operator-hosted MCP server. Same fallback discipline.
- Anything else → `{ enabled: false, reason: 'not_configured' }`.

### Client wrapper — `classifyMove`

Added to `src/lib/edgeFunctions.ts`, identical pattern to `processLanguageDraft` (edgeFunctions.ts lines 197–226):

// Future implementation card — shape only.

  payload: ClassifyMoveRequest,

Hard secret rules (cdiscourse-doctrine §6, supabase-edge-contract logging rules):

- **This design names env vars. It never reads, prints, validates, or uses a value.** No value of any of the above appears anywhere in this doc or in any file this card commits.
- A provider key is read **only** via `Deno.env.get()` inside the Edge Function module tree. It is never imported, never passed to the client, never echoed in a response, never logged.
- **No `SUPABASE_SERVICE_ROLE_KEY`** anywhere in this boundary — the function performs no privileged write, so no service-role client is built (supabase-edge-contract).
- The boundary **never logs**: the `Authorization` header, any provider key, the MCP token, a raw provider response body, or a JWT. It may log: function name, caller user id, `roomId`, decision (`enabled`/`disabled`/provider name), duration, error class — matching the supabase-edge-contract logging contract.
- The raw provider response is run through a sanitizer (`sanitizeRawPayload` — keep only `model`, `stop_reason`, `usage`, as `anthropicProvider.ts` does today) before any field of it could reach a log or a response.
- A source-scan test in the future card asserts `grep -r "ANTHROPIC_API_KEY\|SERVICE_ROLE" app/ src/` returns zero matches (cdiscourse-doctrine §6, MCP-001 §20).
- `.env.example` (if touched at all by the future card) gains only the **names** `SEMANTIC_REFEREE_ENABLED` / `SEMANTIC_REFEREE_PROVIDER` with empty placeholder values, and only as documentation that they are *server-side* secrets — they are not Expo-public and must never be prefixed `EXPO_PUBLIC_`.

### Disabled-by-default behavior

The whole layer is **off unless the operator explicitly turns it on.** The exact contract:

1. `SEMANTIC_REFEREE_ENABLED` unset, empty, or any value other than the string `'true'` → the registry returns `{ enabled: false, reason: 'disabled' }` **before any provider is selected and before any key is read**.
2. The Edge Function returns that object with **HTTP 200** — disabled is a normal outcome, not an error. No 4xx, no 5xx, no stack trace.
3. The client wrapper `classifyMove()` returns `{ ok: true, data: { enabled: false, reason: 'disabled' } }`.
4. The consuming feature (MCP-003 ledger, MCP-008 banners) sees `enabled: false` and **falls back to the deterministic layer-1 result with no error surfaced to the user.** The semantic referee simply does not contribute; the game still works.
5. `SEMANTIC_REFEREE_ENABLED=true` but `SEMANTIC_REFEREE_PROVIDER` unset → provider is `mock` (deterministic, no network) — turning the flag on **never** silently starts a live call.
6. `SEMANTIC_REFEREE_ENABLED=true`, `SEMANTIC_REFEREE_PROVIDER=anthropic`, `ANTHROPIC_API_KEY` missing → `{ enabled: false, reason: 'key_missing' }` — still no throw, still HTTP 200, client still falls back.
7. A room whose GAME-003 `semanticClassification` mode is `'off'` is never sent to the boundary at all (MCP-004's gate). The boundary's own disabled path is the *second* line of defense; the trigger plan is the first.

This is the exact behavior `process-language-draft` already ships (transcript-language-processing.md "Provider Behavior When Disabled" — `{ "enabled": false, "reason": "disabled" }`, HTTP 200, no Anthropic call, no error). The semantic referee copies it.

### MCP-server placement (if the operator picks option c)

If the operator selects option (c), or adopts it later as a registry provider:

- **The MCP server is a separate, operator-hosted, server-side-only service.** It is *not* part of the Expo app, *not* part of `src/` or `app/`, and *not* reachable from the client. The client only ever talks to the `semantic-referee` Edge Function; the Edge Function talks to the MCP server.
- **Where it sits:** alongside the operator's existing backend infrastructure (e.g. the Cloud Run hosting designed by the HOST-*** cards) or a managed MCP host. It exposes one MCP tool — `classify_semantic_move` — that accepts the redacted move/parent/context and returns a `SemanticRefereePacket`-shaped result.
- **How the Edge Function reaches it:** the `mcpAdapter.ts` module connects over HTTPS (MCP streamable-HTTP / SSE transport) to `SEMANTIC_REFEREE_MCP_URL`, authenticating with `SEMANTIC_REFEREE_MCP_TOKEN`. Both are read via `Deno.env.get()` inside the Edge Function; neither is ever in client code.
- **Where the model key lives under option (c):** on the MCP server, not in Supabase. This is a security *advantage* — `ANTHROPIC_API_KEY` need not be a Supabase secret at all — at the cost of a second credential to rotate (`SEMANTIC_REFEREE_MCP_TOKEN`).
- **The doctrine invariants are unchanged:** still server-side only, still `authoritative: false`, still advisory, still no client call, still disabled-by-default (the MCP adapter is only reached when `SEMANTIC_REFEREE_ENABLED=true` *and* `SEMANTIC_REFEREE_PROVIDER=mcp`). The MCP server must itself honor MCP-001's catalog — it classifies move structure, never truth.
- **v0 stance:** the design *reserves* the `mcp` registry slot and documents the adapter shape so adopting option (c) later is a contained, additive change. **No MCP server is started, scaffolded, hosted, or named in this card** — that is explicitly out of scope (card §"Non-scope", roadmap §15).

## Edge cases

The implementer of the future card must handle each of these:

- **Feature flag off** — `{ enabled: false, reason: 'disabled' }`, HTTP 200, no provider selected, no key read. (Covered above; the dominant case.)
- **Flag on, provider unset** — defaults to `mock`; no live call, no key read.
- **Flag on, provider `anthropic`, key missing** — `{ enabled: false, reason: 'key_missing' }`, HTTP 200, no throw.
- **Provider API error / timeout / 5xx** — the provider returns a typed `unavailable`; the registry returns a **deterministic fallback packet** (`provider: 'mock'`, layer-1-only low-confidence binaries, schema-valid). The client never sees an error from a transient provider failure.
- **Provider returns malformed / non-JSON / extra-field output** — outbound Zod validation fails → deterministic fallback packet. A model that tries to return a truth field or `authoritative: true` is rejected here.
- **Empty `moveBodyRedacted`** — request validation rejects an empty body (`validation_failed`), or — operator's choice in the future card — the mock returns a packet with all binaries `0` and `routeSuggestion: 'no_route_change'`. Recommended: reject empty at the schema, since an empty move cannot be classified.
- **Over-length body** — schema caps `moveBodyRedacted` / `parentBodyRedacted` at 8000 chars (matching `process-language-draft`); over-length → `validation_failed`. The function never sends an over-budget payload (the *numeric* token budget is MCP-004's; the *character cap* is the boundary's first guard).
- **`requestedClassifiers` empty or > 5** — schema requires 1–5 entries (MCP-001 §9 caps a packet prompt at 5). Out-of-range → `validation_failed`.
- **`requestedClassifiers` contains an unknown id** — schema rejects ids outside `SemanticClassifierId`. An unmapped id never reaches a prompt.
- **No parent (`parentId`/`parentBodyRedacted` absent)** — valid for a root move; parent-continuity classifiers fall back to their layer-1 derivation and the packet is still produced.
- **Caller cannot see the room (RLS)** — step 6's `debates` lookup returns no row → `validation_failed` (`room_not_found_or_not_accessible`), exactly the `process-language-draft` debate-access behavior. No packet, no provider call.
- **No `Authorization` header** — 401, before any body parse.
- **Permission-denied to call at all** — there is no per-room *write* permission to check; the boundary only *reads* room context (RLS) and *classifies*. It cannot post, so there is no "permission-denied post" path here.
- **Concurrent calls for the same move** — the boundary is stateless and idempotent: two identical `ClassifyMoveRequest`s produce two identical packets (mock/fixture are deterministic; the live provider is called twice, which is a *cost* concern owned by MCP-004's caching, not a *correctness* concern). The boundary does not need a lock.
- **Offline / network failure between client and Edge Function** — `supabase.functions.invoke` surfaces a `FunctionsFetchError`; the wrapper maps it to `{ ok: false, status: 503 }`; the caller falls back to layer 1. No user-facing error.
- **Doctrine edge case — "what if the model tries to return a verdict?"** — it cannot reach the user: the outbound schema has no truth field to populate, `authoritative` is `z.literal(false)`, and any extra/`true` field fails validation → deterministic fallback. The model physically cannot widen the packet.
- **Doctrine edge case — "what if disabling the layer breaks the game?"** — it does not: every consumer of the packet (MCP-003 ledger, MCP-008 banners) is contractually required to have a layer-1 deterministic fallback. The semantic referee is purely additive; `{ enabled: false }` degrades gracefully to the deterministic game.
- **Doctrine edge case — "what if a redacted body still leaks a handle/URL/key?"** — the function runs a **second** defensive redaction pass (step 7) before building any prompt, independent of whatever the client did. Redaction is belt-and-suspenders.

## Risks

Things that might trip up the implementer of the future card:

- **Default-provider divergence.** `languageProcessing/providers.ts` defaults its provider to `'anthropic'`. The semantic referee **must** default to `'mock'`. An implementer copying the language-processing registry verbatim would introduce a doctrine violation (a live call on flag-on with no provider set). The `??  'mock'` fallback is load-bearing and must have its own test (`semanticProviderRegistry.test.ts`).
- **Two type trees drift.** MCP-001's contract is mirrored in both `supabase/functions/_shared/semanticReferee/types.ts` (Deno, `.ts` extensions) and `src/features/semanticReferee/types.ts` (Node, no extensions). The language-processing scaffold already lives with this dual-mirror; a contract change must update both, and a parity test is advisable.
- **Redaction is shared, not owned here.** The boundary assumes bodies arrive redacted and runs a defensive second pass, but the *canonical* redaction implementation is shared with MCP-002/MCP-004. The implementer must not re-invent it — reuse the `scripts/engagement-intelligence/` sanitizer patterns (MCP-001 §20). If redaction lands in a different card first, this card's function imports it.
- **Zod version.** `process-language-draft` uses `npm:zod@4` inside Deno. The new schema module must use the same import specifier; mixing zod versions in `_shared/` is a footgun.
- **`config.toml` `verify_jwt`.** Forgetting the `[functions.semantic-referee]` block with `verify_jwt = true` would deploy an unauthenticated function. The future card must add it and verify it.
- **No migration — but verify.** This card asserts no DB change. The implementer must resist the temptation to add a packet-cache table; a persisted cache is explicitly an MCP-004 decision (MCP-001 §12). If MCP-004 later wants one, that is a *new* migration in *that* card.
- **Operator deploy required for the future card.** The future implementation card produces an Edge Function; Claude does not deploy. The operator must run `npx supabase functions deploy semantic-referee --linked` and, to actually enable the layer, `npx supabase secrets set SEMANTIC_REFEREE_ENABLED=...`. Until then the function simply does not exist and callers fall back to layer 1 — which is the safe default.
- **Existing tests.** Option (b) adds a function and a client wrapper without touching `process-language-draft` or `submit-argument`; no existing test should need updating. If the implementer instead picks option (a), every `process-language-draft` test must be revisited for the new `action` branch — another reason (b) is recommended.

## Doctrine self-check

- **cdiscourse-doctrine §1 — no truth labels:** the boundary transports MCP-001's structural-only packet; the inbound request has no field that can ask for a verdict, and the outbound schema has no truth field to populate. ✓
- **cdiscourse-doctrine §1 — score never blocks posting:** the packet has no `block` field; the boundary returns advisory metadata only and `submit-argument` (the only post path) is untouched. ✓
- **cdiscourse-doctrine §4 / §7 — AI never runs on the client:** the provider call lives only in the `semantic-referee` Edge Function (option b) or a server-only MCP adapter it reaches (option c). The client only calls `supabase.functions.invoke('semantic-referee')`. ✓
- **cdiscourse-doctrine §4 — `authoritative` always `false`:** the outbound Zod schema pins `authoritative` to `z.literal(false)`; a non-`false` value fails validation and the boundary returns the deterministic fallback. ✓
- **cdiscourse-doctrine §6 — secrets policy:** env var **names** are documented; no value is read, printed, validated, or used in this doc. Provider keys are read only via `Deno.env.get()` inside the Edge Function, never returned, never logged, never in `app/` / `src/` / `.env` / git. No `SUPABASE_SERVICE_ROLE_KEY` is used — the function performs no privileged write. ✓
- **cdiscourse-doctrine §7 — no AI from the production app:** the production app makes no provider call; it invokes an Edge Function. The registry defaults to `mock`; the live provider is operator-gated behind `SEMANTIC_REFEREE_ENABLED=true` + an explicit `SEMANTIC_REFEREE_PROVIDER`. ✓
- **cdiscourse-doctrine §9 — plain language:** the boundary's only user-reachable strings are `reason` codes; a ban-list test (`semanticBoundaryBanList.test.ts`) scans them; `classifierId` / `reasonCode` mapping to plain language is MCP-008's job and out of scope here. ✓
- **supabase-edge-contract — no service-role in client, no direct `public.arguments` insert:** the boundary reads room context through the caller-scoped (RLS) client and writes nothing; it never inserts into `arguments`, `flags`, or any audit table, and never calls `submit-argument`. ✓
- **supabase-edge-contract — disabled-by-default returns `{ enabled: false }`:** flag off → `{ enabled: false, reason: 'disabled' }`, HTTP 200, no error thrown, client falls back to the deterministic layer — exactly the `process-language-draft` pattern. ✓
- **evidence-doctrine — engagement vs factual standing stay separate:** the boundary transports a packet whose `scoreHints` are *hints* the MCP-003 ledger reconciles against `antiAmplification.ts`; the boundary itself adds no score, ranks no crowd, and treats no popularity signal as evidence. ✓
- **test-discipline — mock-first, ban-list tests, auth + invalid-input coverage:** the test plan (a future-card responsibility) defaults to the mock provider, includes a ban-list scan, a secrets source-scan, and the auth-refused / invalid-input Edge Function paths. ✓
- **v1 scope guards:** no voting, no argument search, no push notifications, no OAuth, no public API introduced by this boundary. ✓
