## 0. How this design was built

This design is **derived from the storyboards, not asserted.** The two storyboards
record, frame by frame, exactly which notifications must be *generated* and what
each one must say. Both end with a "Notifications generated (summary)" table.
QOL-040's job is to make those tables real: a notification **model**, a **delivery
path** that does not violate doctrine, **per-trigger neutral copy**, and a
**deep-link** that opens the room at the right node.

The work is sequenced in three grounded passes:

1. **The trigger catalogue** (§5) — every storyboard step that says "a
   notification is queued" is enumerated, with its recipients, its copy intent,
   and its deep-link target.
2. **The data model** (§6) — the smallest table + RLS that can carry the
   catalogue without leaking what a room's visibility forbids.
3. **The surface** (§7–§8) — an in-app notification list and badge; **no push.**

Where the storyboards and a generic notification design disagree, **the
storyboard wins** — it is the product canon (QOL-034).

## 1. Goal & scope

CDiscourse currently has **no notification system**. Both storyboards depend on
one at nearly every step: an invite must reach the respondent, a new response
must pull the other primary back, a chime-in must reach both primaries, a
made-private transition must reach the observer. Without QOL-040 a participant
has no way to know an argument moved without re-opening the room manually.

QOL-040 builds the **in-app notification lifecycle** for an argument room:

- A `room_notifications` table — one row per delivered notification, per
  recipient.
- An **append-only** write path: notifications are *produced as a side effect of
  the actions that already write to the room*, inside the existing
  `submit-argument` Edge Function and a small new `room-notifications` Edge
  Function for the non-argument triggers (invite, made-private, chime-in
  rejected, settled).
- Per-trigger **neutral copy** — a deterministic, pure copy builder.
- A **deep link** payload that opens the room with the relevant node active.
- An in-app **notification list** screen + an unread **badge** on the Arguments
  tab.

### In scope

- The notification data model + RLS.
- The 9 storyboard triggers (§5) and their copy + deep links.
- The notification list UI, unread badge, mark-read-on-open.
- A pure copy/deep-link model (`notificationModel.ts`).
- The visibility-safe copy rule (§9) — copy never names data the recipient's
  access to the room would forbid.

### Out of scope (explicit — see §16)

- **Push notifications** — excluded by v1 scope (`CLAUDE.md` "What Not to Build").
  In-app only.
- The invite *backend* itself (record, email send, auth deep link) — **QOL-038**.
  QOL-040 consumes the invite record QOL-038 creates and produces the *in-app*
  invite notification; the *email* invite is QOL-038's.
- The visibility *transition* mechanics + RLS — **QOL-039**. QOL-040 only
  produces the neutral notifications QOL-039's transition fires.
- The concession gradient, evidence applicability flow, chime-in governance
  internals — QOL-041 / QOL-037 / GAME-005. QOL-040 *observes* their state
  changes and notifies; it does not own them.
- Notification *preferences* (mute a room, digest cadence) — a possible v2
  follow-up; §17 records it.

## 4. Problem statement

Today, a participant only learns an argument changed by re-opening the room.
Both storyboards are **real-time conflict de-escalation** — a roommate dispute, a
band rent dispute — where speed matters; a silent room is a stalled argument.
There is also a **safety hazard** in the naive fix: a notification is a tiny
piece of text delivered *outside* the room's access boundary. If the chime-in
notification said *"Observer C wrote: 'you may need to specify…'"* it would leak
room content to a recipient (or, after a made-private transition, to a
now-removed observer) the room's visibility no longer authorises. A notification
must therefore carry **a pointer and a neutral summary, never room content**.

QOL-040 must:

1. Generate a notification for each of the 9 triggers, addressed to the right
   recipients.
2. Phrase every one in **neutral, plain language** — no verdict, no "game", no
   "debate", no shaming.
3. Carry a **deep link** that opens the room at the relevant node — but only for
   recipients who still have access.
4. **Never** put room body text, evidence detail, or any visibility-restricted
   data into the notification copy.
5. Stay **in-app** — no push (v1 scope).

## 5. The notification trigger catalogue

The 9 triggers. Each row: **what fires it · recipients · neutral copy intent ·
deep-link target · the storyboard step it comes from**.

The *copy intent* column states the **message shape**, not the literal string —
the literal string is produced by the deterministic builder in §6.3 and uses
only the room title + a neutral noun. **No copy below contains room body text.**

| # | Trigger (`type`) | Fires when | Recipients | Neutral copy intent | Deep-link target |
|---|---|---|---|---|---|
| 1 | `invite` | An `argument_room_invites` row is created (QOL-038), **or** an invited user accepts and joins | The invitee (in-app notification appears once they have an account; the *email* is QOL-038's) | "You were invited to respond to an argument: `<room title>`." Private rooms: "…to a private argument: `<room title>`." Never "debate challenge" / "game invite". | The room root (`argument_id` = root claim). |
| 2 | `new_response` | A `posted` argument is inserted whose `argument_type` is a plain response (`rebuttal`, `counter_rebuttal`, `clarification`, `synthesis`) and whose author is a primary participant | The **other** primary participant(s) — every primary except the author | "New response in: `<room title>`." | The newly posted `argument_id`. |
| 3 | `concession_challenged` | A `posted` argument records a concession-acceptance whose gradient is a *disagree* level (QOL-041) | The author of the **challenged concession** | "Your concession was challenged in: `<room title>`." The active classification (`framing` / `context` / `fact`) **may** be named as a neutral noun — it is a move-category label, not a verdict. | The challenging `argument_id`; the room opens with the challenged concession highlighted. |
| 4 | `source_requested` | A `posted` argument is an "Ask for source" / "Ask quote" move (EV-002), opening an evidence-debt marker (EV-003) | The participant the source was asked of | "A source was requested in: `<room title>`." | The `argument_id` carrying the evidence-debt marker. |
| 5 | `evidence_supplied` | An evidence object is attached that resolves an open evidence-debt marker | The participant who requested the source | "Evidence was supplied in: `<room title>`." | The `argument_id` the evidence is attached to. |
| 6 | `chime_in_posted` | A `posted` argument is written with a derived `chime_in` seat (GAME-005), attached to a node | **Both** primary participants | "Someone chimed in on: `<room title>`." See §9 — the observer's display name is shown **only if** the recipient's access to the room would already let them see it; otherwise "Someone". | The node the chime-in attaches to. |
| 7 | `room_made_private` | A room's visibility transitions public → private (QOL-039) | Every **non-participant who had read access before the transition** — i.e. observers and chime-in authors who are now removed | "This argument was made private: `<room title>`." Neutral; states a fact, assigns no cause. | **No deep link** — the recipient no longer has access. The notification opens *itself* (a read-only confirmation), not the room. See §9. |
| 8 | `chime_in_rejected` | Two distinct primaries both mark a chime-in "Not useful" (GAME-005), muting it | The chime-in's author (the observer) | "Your chime-in was marked unwanted by both primary participants in: `<room title>`." **Neutral — describes the move's fit to the room, never the person.** No "rejected by", no "bad", no count framed as a verdict. | If the recipient still has read access: the muted chime-in node (it is retained, never deleted — Scenario 1 Step 14). If access was revoked by a concurrent made-private: no deep link. |
| 9 | `argument_settled` | A room's `status` transitions to `locked` (settled) | **Both** primary participants | "This argument is settled: `<room title>`." Never "case closed", "winner", "you won/lost". | The room (root or final summary node). The room is read-only when opened. |

### 5.1 Trigger discrimination

Triggers 2, 3, 4, 5 all fire on an `arguments` insert. They are disambiguated
*inside* `submit-argument` by a **pure classifier** (`classifyArgumentTrigger`,
§6.3) reading the just-inserted argument's `argument_type` and payload — the same
deterministic input the rules engine already sees. The classifier returns
**at most one** argument-derived trigger per insert (an insert that both supplies
evidence *and* challenges a concession is rare; if it occurs, the classifier
returns the **more specific** trigger and the design accepts one notification per
insert — see §10 *Edge case: multi-meaning move*).

Triggers 1, 6, 7, 8, 9 are **not** argument inserts — they are produced by the
new `room-notifications` Edge Function, called by the card that owns the action
(QOL-038 for invite, QOL-039 for made-private, GAME-005 for chime-in
posted/rejected, the room-settle path for settled).

## 6. Data model & file changes

### 6.1 New migration — `room_notifications`

New file: `supabase/migrations/20260522000010_qol040_room_notifications.sql`
(number is the next free sequential slot after `20260517000009`; the implementer
must re-check `supabase/migrations/` for the true next number at build time and
**never edit an applied migration**).

```sql
create table if not exists public.room_notifications (
  id            uuid        primary key default gen_random_uuid(),
  recipient_id  uuid        not null references auth.users(id) on delete cascade,
  debate_id     uuid        not null references public.debates(id) on delete cascade,
  -- The node to open. NULL for room-level notifications (settled, made-private).
  argument_id   uuid        references public.arguments(id) on delete set null,
  type          text        not null check (type in (
                              'invite', 'new_response', 'concession_challenged',
                              'source_requested', 'evidence_supplied',
                              'chime_in_posted', 'room_made_private',
                              'chime_in_rejected', 'argument_settled'
                            )),
  -- Denormalised room title SNAPSHOT at delivery time. See §9: this is the
  -- ONLY room content a notification ever carries, and only because the
  -- recipient was authorised to see it at the moment of delivery.
  room_title    text        not null default '',
  -- Optional small JSON for neutral metadata only (e.g. classification noun
  -- 'fact'|'context'|'framing'). NEVER carries body text. Length-checked.
  meta          jsonb       not null default '{}',
  -- TRUE once the recipient opens the notification.
  read_at       timestamptz,
  created_at    timestamptz not null default now(),
  constraint room_notifications_room_title_length
    check (char_length(room_title) <= 200)
);

create index if not exists room_notifications_recipient_idx
  on public.room_notifications (recipient_id, created_at desc);
create index if not exists room_notifications_unread_idx
  on public.room_notifications (recipient_id)
  where read_at is null;

alter table public.room_notifications enable row level security;
```

**RLS (the load-bearing safety boundary):**

```sql
-- SELECT: a user sees ONLY their own notifications.
create policy room_notifications_select_own
  on public.room_notifications for select
  using (auth.uid() = recipient_id);

-- UPDATE: a user may mark ONLY their own notification read.
-- The WITH CHECK forbids changing anything except read_at by pinning every
-- other column to its existing value.
create policy room_notifications_update_own_read
  on public.room_notifications for update
  using (auth.uid() = recipient_id)
  with check (auth.uid() = recipient_id);

-- INSERT: NO client insert policy. Notifications are inserted ONLY by the
-- service-role inside the submit-argument / room-notifications Edge Functions.
-- With RLS enabled and no INSERT policy, an anon/auth client cannot insert.

-- DELETE: NO delete policy. Notifications are not deleted by clients.
-- (Cascade on debate/user removal handles cleanup.)
```

**Why no client INSERT policy:** a notification's recipient set is a *trust
decision* (who is a primary, who had access before a transition). If clients
could insert notification rows, a user could fabricate a notification addressed
to anyone. Insert is therefore service-role-only, from inside the Edge
Functions, exactly as `submit-argument` already owns the `arguments` insert. The
client never inserts and never holds the service-role key (doctrine §6, §8).

**Why `room_title` is denormalised:** the deep link may be dead for a
made-private notification (trigger 7) — the recipient lost room access. The
notification must still render its own copy. The title is **snapshotted at
delivery time** while the recipient was still authorised; it is the *only* room
content a notification ever carries, and §9 covers the one case where even the
title must be withheld.

> Restricting the UPDATE to `read_at` only: PostgREST `with check` cannot, by
> itself, forbid editing other columns. The implementer enforces "only `read_at`
> changes" by **scoping the client write to a single column** in the API wrapper
> (`.update({ read_at })`) **and** by an Edge-Function-side check is *not* needed
> because no column other than `read_at` is ever meaningful to a recipient — the
> worst a malicious recipient could do is overwrite *their own* `room_title`,
> which only degrades their own view. This is acceptable; the row is already
> theirs. If stricter immutability is wanted later, move the mark-read through a
> tiny RPC. Recorded as a §17 follow-up, not a v1 blocker.

### 6.2 No change to existing tables

QOL-040 adds **one new table**. It does not alter `arguments`, `debates`, or
`debate_participants`. The invite record is QOL-038's table; the visibility
column is QOL-039's.

### 6.3 New pure model — `notificationModel.ts`

New file: `src/features/notifications/notificationModel.ts` (~220 lines). Pure
TypeScript — **no React, no Supabase, no network, no `Date.now()`** in the copy
functions. This mirrors the doctrine that copy/decision models are deterministic
and testable.

```ts
export type RoomNotificationType =
  | 'invite' | 'new_response' | 'concession_challenged'
  | 'source_requested' | 'evidence_supplied' | 'chime_in_posted'
  | 'room_made_private' | 'chime_in_rejected' | 'argument_settled';

export interface RoomNotification {
  id: string;
  recipientId: string;
  debateId: string;
  argumentId: string | null;
  type: RoomNotificationType;
  roomTitle: string;
  meta: NotificationMeta;        // neutral nouns only — see below
  readAt: string | null;
  createdAt: string;
}

export interface NotificationMeta {
  // Optional neutral classification noun for concession_challenged.
  classification?: 'framing' | 'context' | 'fact';
  // TRUE when the room is private — selects the private copy variant.
  roomIsPrivate?: boolean;
  // TRUE when the actor's display name may be shown (recipient is authorised).
  // When false, copy uses "Someone". See §9.
  actorNameVisible?: boolean;
  actorDisplayName?: string;     // present only when actorNameVisible === true
}

// The deterministic copy builder. Pure: (type, title, meta) -> string.
// NEVER receives or emits room body text.
export function buildNotificationCopy(
  type: RoomNotificationType,
  roomTitle: string,
  meta: NotificationMeta,
): { title: string; body: string };

// Deep-link resolution. Returns null when the recipient has no access
// (room_made_private, or a revoked chime_in_rejected) — the caller then
// renders the notification as a non-navigable confirmation.
export function resolveDeepLink(
  n: RoomNotification,
): { debateId: string; activeArgumentId: string | null } | null;

// The argument-insert trigger classifier (used inside submit-argument).
// Returns at most one argument-derived trigger, or null (no notification).
export function classifyArgumentTrigger(input: {
  argumentType: string;
  parentArgumentType: string | null;
  concessionAcceptanceGradient: string | null; // QOL-041 payload, if present
  resolvesEvidenceDebt: boolean;               // EV-003 payload, if present
  opensEvidenceDebt: boolean;                  // EV-002/EV-003 payload
}): RoomNotificationType | null;

// The recipient resolver (pure): given the trigger + participant list +
// author, return the recipient user-id set. No author self-notification.
export function resolveRecipients(
  type: RoomNotificationType,
  ctx: {
    authorId: string;
    primaryIds: string[];
    observerIds: string[];
    challengedConcessionAuthorId?: string;
    sourceRequestTargetId?: string;
    sourceRequesterId?: string;
    priorReadAccessIds?: string[];   // for room_made_private
    chimeInAuthorId?: string;        // for chime_in_rejected
  },
): string[];
```

`buildNotificationCopy` is the single home of every user-facing notification
string. Its strings are scrubbed against the doctrine ban-list by a test (§10).
Any internal code that ever needs to surface is routed through
`gameCopy.toPlainLanguage` — `notificationModel` never embeds raw `snake_case`.

### 6.4 Edge Function changes

**Modified — `supabase/functions/submit-argument/index.ts`** (~40 added lines):
after a successful, RLS-checked argument insert, the function:

1. Calls `classifyArgumentTrigger(...)` on the just-inserted row.
2. If it returns a trigger, calls `resolveRecipients(...)` against the
   participant list it already loaded.
3. Inserts one `room_notifications` row per recipient **using the service-role
   client the Edge Function already holds** — never the caller's JWT, never a
   client.
4. A notification-insert failure is **caught and swallowed** — it must **never**
   roll back or block the argument post (doctrine: validation can block,
   nothing else can; a notification is strictly less important than the post).
   The failure is logged via the structured logger, not `console.log`.

**New — `supabase/functions/room-notifications/index.ts`** (~150 lines): a small
JWT-verified Edge Function for the **non-argument** triggers (invite,
made-private, chime-in posted, chime-in rejected, settled). Its callers are the
sibling cards' write paths:

- It is **JWT-verified** — the caller must be authenticated.
- It **authorises the caller** for the action: e.g. `room_made_private` may be
  requested only by the room creator; `argument_settled` only by a primary;
  `chime_in_rejected` only when GAME-005's governance state already shows two
  rejections (the function *re-derives* this from the DB, it does not trust the
  caller's claim).
- It inserts the notification rows with the service-role client.
- It **never** returns other users' data to the caller — the response is a bare
  `{ delivered: <count> }`.
- It **never** logs `Authorization`, JWTs, or recipient emails.

Standard Edge Function shape (per `supabase-edge-contract`): CORS, JWT verify,
input validation, service-role client constructed from `Deno.env`, structured
error responses, no secret in any log line.

## 7. UI states

The notification list and badge.

### 7.1 Notification list

| State | Render |
|---|---|
| **Loading** | A skeleton list; the badge shows the last-known count. |
| **Empty** | "No notifications yet. You'll see invites and replies to your arguments here." — calm, not an error. |
| **Populated** | Newest-first rows. Each row: a type icon, `buildNotificationCopy` title + body, a relative timestamp, an unread dot for `read_at === null`. |
| **Error** | "Couldn't load notifications. Pull to retry." — actionable, never a raw error code. |
| **Unread present** | A "Mark all read" affordance in the header. |

### 7.2 Notification row interaction

- **Tap a navigable notification** → `resolveDeepLink` returns a target → open
  the room with `activeArgumentId` set; mark the row read on open.
- **Tap a non-navigable notification** (`room_made_private`, or a
  `chime_in_rejected` whose access was revoked) → expand a small read-only
  confirmation card with the notification body; mark read. **No navigation** —
  the recipient has no room access; the app must not attempt to open a room it
  will then deny.
- **Unread → read** transition is visible (the dot clears) and persisted via
  `markNotificationRead`.

### 7.3 Badge

- `NotificationBadge` shows the unread count on the Arguments tab.
- Zero unread → no badge (not a "0").
- Count is capped at a display ceiling ("9+") to avoid layout drift.

### 7.4 Accessibility

Per `accessibility-targets`: every row is a ≥44px tap target with an
`accessibilityRole="button"`; the unread dot is **not** the only unread signal —
the row also exposes an `accessibilityState`/label ("unread"); the badge has an
`accessibilityLabel` ("3 unread notifications"); reduce-motion disables any
list-entry animation.

## 8. Deep-link behaviour

A notification's job after delivery is to **return the recipient to the room at
the right node** — Scenario 1 Step 7: *"Opening it returns directly to the
argument room with the latest response (Node 2) active."*

- `resolveDeepLink(n)` returns `{ debateId, activeArgumentId }` or `null`.
- The room screen accepts `activeArgumentId`; it reuses the **existing**
  open-room-at-node path (Stage 6.4 seamless entry already routes invited users
  straight into a room — QOL-040 adds the *node* selection on top, it does not
  invent a new router).
- When the deep link resolves but the recipient has **lost access** between
  delivery and tap (a room went private after a `new_response` notification was
  delivered to a then-observer), the room screen's existing RLS denies the read
  and shows the standard "this argument is private" state. QOL-040 does not need
  a special case — RLS is the backstop; the notification is just stale.
- `null` deep links never attempt navigation (§7.2).

QOL-040 deliberately does **not** build OS-level / URL deep links — those are
for *push* and external links, which are out of scope. The "deep link" here is
purely an **in-app route param**.

## 9. Permission / rules implications — visibility-safe copy

This is the safety core of the card. A notification is text delivered *outside*
the room. The rules:

1. **A notification never carries room body text.** Not the argument body, not
   the concession text, not the evidence detail, not the chime-in text. The copy
   builder (`buildNotificationCopy`) physically cannot emit body text — it is
   never passed any. The only room content a notification carries is the
   **room title**, and only as a delivery-time snapshot (§6.1).

2. **Recipient set is computed from access, not convenience.** `resolveRecipients`
   only ever returns users who are authorised to know the room exists:
   - room-scoped triggers → current `debate_participants`;
   - `room_made_private` → the set of users who had read access **before** the
     transition (computed by QOL-039's transition path and handed to the
     `room-notifications` function);
   - `chime_in_rejected` → the chime-in's author only.

3. **Actor names are gated.** Trigger 6 (`chime_in_posted`) names the observer
   *only if* `meta.actorNameVisible === true`. The Edge Function sets that flag
   true only when the recipient's room access would already let them see that
   observer's participation. In a public room the primaries see the observer
   anyway → name shown. After a made-private transition the name is **not**
   re-derived — a previously delivered notification is not edited; future ones
   use "Someone". The storyboard's *"[Observer C] chimed in"* copy is therefore
   honoured **when the room is public**, and degrades to *"Someone chimed in"*
   when it would leak.

4. **The made-private notification is the hardest case.** Its recipient is, by
   definition, a user who **just lost access**. The notification:
   - carries the room **title** — they were authorised the moment before, and
     the storyboard explicitly delivers *"This argument was made private:
     `<room title>`"*. The title is delivery-time snapshotted.
   - carries **no deep link** — `resolveDeepLink` returns `null` for
     `room_made_private`.
   - if a room was *created* private and an observer never had access, no
     `room_made_private` notification is ever generated for them (there was
     nothing to lose) — only the *public→private transition* generates it.

5. **The chime-in-rejected notification never shames.** Per Scenario 1 Step 14's
   doctrine note — *"'Not useful' describes the move's fit to this room, never
   the person."* The copy says the **chime-in** was marked unwanted; it never
   says the **observer** was rejected, never names the rejecting primaries,
   never frames the two rejections as a verdict or a score. A §10 ban-list test
   asserts the string contains none of: "rejected you", "bad", "wrong",
   "unwelcome person", "loser", a name of a rejecter.

6. **No verdict copy anywhere.** No notification says winner / loser / correct /
   true / false / proven. The `argument_settled` copy says *"settled"* — never
   *"case closed"*, never an outcome. This mirrors the evidence-doctrine
   forbidden-word list and the storyboard's explicit *"even at settlement, the
   word is 'settled.'"*

7. **Normal-user language.** No notification says "game" or "debate"; it says
   "argument". `new_response` is "New response", not "New move in the game".

## 10. Edge cases

- **Author of a move never notifies themselves.** `resolveRecipients` excludes
  `authorId` from every recipient set. Scenario 1 Step 3: *"None to A (it is A's
  own move)."*
- **Multi-meaning move.** A single insert that *both* supplies evidence and
  challenges a concession: `classifyArgumentTrigger` returns the **more
  specific** trigger (`concession_challenged` over `new_response`,
  `evidence_supplied` over `new_response`) and the design accepts **one
  notification per insert**. Rationale: one insert is one room event from the
  recipient's view; a second notification for the same insert is noise.
- **Concurrent transition.** A `new_response` notification is delivered to an
  observer, then the room goes private. The notification row already exists; its
  deep link still resolves to `{ debateId, argumentId }`, but the room screen's
  RLS denies the read on tap → the standard private-room state shows. No data
  leaks; the notification is merely stale. Acceptable.
- **Recipient has no account yet (invite).** The QOL-038 *email* invite reaches
  the person before they have an account. The **in-app** `invite` notification
  row is created keyed to `recipient_id = auth.users(id)` — it can only exist
  once the user has an account. The Edge Function therefore creates the in-app
  `invite` notification at the moment the invited user **accepts and an account
  exists** (or is skipped entirely if the email path is the only delivery for a
  brand-new user — QOL-038 owns that handoff). QOL-040's in-app `invite`
  notification is primarily for an invitee who **already has an account**.
- **Notification insert fails inside `submit-argument`.** Swallowed and logged;
  the argument post still succeeds (§6.4). A notification is never allowed to
  block a post.
- **Room deleted / argument soft-deleted.** `argument_id` is `ON DELETE SET
  NULL`; `debate_id` is `ON DELETE CASCADE`. A notification whose `argument_id`
  became null falls back to a room-level deep link; a notification whose room
  was deleted is cascade-removed.
- **Duplicate delivery.** If a trigger path is retried (Edge Function retry),
  two identical rows could appear. Mitigation: the implementer adds an idempotency
  guard — a partial unique index on `(recipient_id, type, argument_id)` for the
  argument-derived triggers (`new_response`, `concession_challenged`,
  `source_requested`, `evidence_supplied`) so a retry is a no-op insert. Triggers
  without an `argument_id` (settled, made-private) are naturally low-frequency
  and one-shot; no index needed.
- **Read of a notification whose room the user can no longer see.** The
  notification row is still the user's own (RLS allows the SELECT) — the *row*
  belongs to the recipient. The deep link is what is gated, not the row. The list
  always renders; navigation is what `resolveDeepLink` may refuse.
- **Empty room title.** `room_title` defaults to `''`; `buildNotificationCopy`
  falls back to a neutral phrase ("an argument") when the title snapshot is
  empty — mirroring the storyboard's "blank title → root-claim excerpt" rule,
  except a notification uses a generic noun, never the root body (no body text).
- **Self-chime-in-rejected.** Not possible — the chime-in author is an observer;
  the rejecters are primaries; the author cannot be a rejecter. No self-notify
  guard needed beyond the standard `authorId` exclusion.
- **A primary leaves the room before a notification fires.** `resolveRecipients`
  reads the *current* participant list at trigger time; a departed primary is
  simply not in the set.

## 16. Out of scope — push notifications

**Push notifications are explicitly out of scope.** `CLAUDE.md` "What Not to
Build (v1 Scope)" lists "No push notifications"; the `cdiscourse-doctrine` skill
§10 repeats it. QOL-040 delivers **in-app notifications only**:

- No OS push registration, no APNs / FCM, no `expo-notifications` push token.
- No background delivery — notifications are read when the user opens the app and
  visits the list (or sees the badge).
- No email *digest* — the only email in the storyboards is the QOL-038 *invite*
  email, which QOL-038 owns; QOL-040 sends **no email**.

If a future release wants push, it is a **new card** that builds on QOL-040's
`room_notifications` table as its source of truth. QOL-040's table is
push-ready (it is the canonical record) but QOL-040 itself ships no push path.

Also out of scope: notification **preferences** (mute a room, per-type opt-out,
digest cadence), notification **realtime** (Supabase realtime subscription
instead of poll-on-focus), and the QOL-033 popout re-housing. All recorded in
§17.

## 17. Open questions

1. **Invite in-app vs email split (for the operator / QOL-038).** QOL-038 owns
   the invite *email*. QOL-040's in-app `invite` notification is only meaningful
   for an invitee who **already has an account**. Confirm: should an invitee who
   signs up *because of* the email also get an in-app `invite` notification on
   first login (a "you were invited here" landing cue), or is the email + direct
   room landing enough? This design assumes the in-app `invite` notification is
   created on invite *acceptance* so an existing-account user has a record; a
   brand-new user lands in the room directly via QOL-038's deep link. Operator
   confirmation would let the implementer finalise the QOL-038 ↔ QOL-040 handoff.

2. **Mark-read immutability (for the implementer).** §6.1 accepts that a
   recipient could, in principle, overwrite their *own* notification's
   `room_title` via the UPDATE policy (it only degrades their own view). Is that
   acceptable for v1, or should mark-read go through a tiny RPC that updates only
   `read_at`? This design recommends accepting it for v1 (the row is the
   recipient's own; the blast radius is their own list) and recording the RPC as
   a hardening follow-up.

3. **Realtime vs poll.** v1 uses poll-on-focus (no Supabase realtime
   subscription) to keep the card L-sized. Confirm that "notifications refresh
   when you open the app / the list" is acceptable UX for the de-escalation use
   case, or whether realtime should be pulled into this card. This design keeps
   realtime out (a §16 follow-up) — the storyboards never require *instant*
   delivery, only *reliable* delivery.

4. **Notification retention.** Should old read notifications be pruned (e.g. a
   scheduled cleanup after N days)? v1 keeps everything (cascade-delete on room
   removal is the only cleanup). A retention policy is a possible follow-up;
   not a v1 blocker.

**Context.** The original design above is dated 2026-05-21, predating
QOL-038's shipped invite backend (PR #264, merged 2026-05-24). The
post-QOL-038 verification pass identified three gaps that this
enrichment fills without invalidating the rest of the design. Each
sub-section names which section of the original design it amends or

[`docs/designs/QOL-040-verification.md`](./QOL-040-verification.md)

## E1 — Email scaffold for the `invite` trigger (amends §1 / §5 row 1 / §16)

**Why.** QOL-038's shipped `CreateRoomInviteResponse.notification` type
already lists `'queued' | 'sent' | 'not_configured'`, with QOL-038's
design §5.1.1 and review verdict naming *"QOL-040 owns the email flip."*
Today the `'sent'` outcome is unreachable: no code path in the codebase
calls Resend for invites. The original design's §16 ("QOL-040 sends no
email") would leave the QOL-038 contract surface permanently
vestigial. This enrichment adds a narrow email scaffold for the
`invite` trigger **only** — push notifications stay strictly out of
scope per `CLAUDE.md` / `cdiscourse-doctrine` §10. The other 8 triggers
remain in-app only.

### E1.1 — Channel matrix

| Trigger | In-app | Email |
|---|---|---|
| `invite` | yes | **yes, gated** (off by default) |
| `new_response` | yes | no |
| `concession_challenged` | yes | no |
| `source_requested` | yes | no |
| `evidence_supplied` | yes | no |
| `chime_in_posted` | yes | no |
| `room_made_private` | yes | no |
| `chime_in_rejected` | yes | no |
| `argument_settled` | yes | no |
| `invite_accepted_by_invitee` (new — see E2) | yes | no |

### E1.2 — Edge Function behaviour

The `room-notifications` Edge Function (per §6.4) gains one helper that
mirrors `request-argument-deletion`'s `maybeSendAdminNotification`:

```ts
type InviteEmailStatus = 'sent' | 'not_configured' | 'failed_sanitized';

async function maybeSendInviteEmail(input: {
  inviteId: string;
  recipientEmail: string;
  inviterDisplayName: string | null;
  roomTitle: string;
  inviteLink: string;        // built from APP_BASE_URL + token
  roomIsPrivate: boolean;
}): Promise<InviteEmailStatus>;
```

Gating rules (verbatim mirror of `request-argument-deletion`):

1. Read `INVITE_EMAIL_ENABLED` from `Deno.env`. If unset or `!== 'true'`,
   return `'not_configured'`. No network call is made.
2. Read `RESEND_API_KEY` and `INVITE_EMAIL_FROM` from `Deno.env`. Either
   missing → return `'not_configured'`. No network call.
3. Build the email body in plain text. Subject: *"You were invited to
   respond to an argument."* Body uses the same neutral framing as the
   in-app notification copy from §5 row 1.
4. POST to `https://api.resend.com/emails` with the `Authorization`
   header built **in-place** — never logged, never assigned to a
   variable that lands in a log line.
5. On non-2xx response: drain the body without echoing it, return
   `'failed_sanitized'`. On exception: same.
6. On 2xx: return `'sent'`.

The function call site (the caller of `room-notifications` from
QOL-038's `manage-room-invite` create action — see E3) records the
returned `InviteEmailStatus` in the existing
`CreateRoomInviteResponse.notification` field on a 1:1 mapping
(`'sent' | 'not_configured' | 'failed_sanitized'` →
`'sent' | 'not_configured' | 'failed_sanitized'`, replacing the v1
hard-coded `'queued'`).

**Backwards compatibility.** The QOL-038 client wrapper already accepts
all three values (`inviteApi.ts:46`). Replacing `'queued'` with one of
the three real statuses is a contract-preserving change.

### E1.3 — Doctrine

- **No service-role in client.** `RESEND_API_KEY` and
  `INVITE_EMAIL_FROM` are Edge-Function-only env vars (per
  `cdiscourse-doctrine` §6). Never named in `src/`.
- **No Authorization header logging.** Source-scan test
  (`__tests__/roomNotifications.email.safety.test.ts`) asserts the new
  helper file does not contain a `console.log` or string-concat of any
  Authorization header value.
- **Neutral copy.** Email subject + body pass the same ban-list scan
  the in-app copy passes (`__tests__/notificationModel.banlist.test.ts`
  extends to cover the email-text builder).

## E2 — Trigger catalogue correction (amends §5)

Two changes to the trigger catalogue, one correction and one addition.

### E2.1 — Correct the `invite` trigger moment (amends §5 row 1, §10 row 7)

The original §5 row 1 fires the `invite` notification on the invited
user's *accept*. This is correct only for the brand-new-user case (the
invitee signs up because of the email, lands in the room directly via
QOL-038's `pendingInviteIntent` deep link, and an in-app notification
at that moment is redundant).

For the **existing-account-invitee** case (Bob already has an account
when Alice invites him), the right moment is **invite create**, not
accept. Bob should see the invite in his notification list immediately,
not wait to tap a copy-pasted link.

The corrected behaviour:

| Invitee state at invite-create time | Where the invite trigger fires |
|---|---|
| Email matches an existing `auth.users` row | **At create** — `room-notifications` invite action runs immediately after `manage-room-invite` create succeeds. Recipient is the existing user's id. |
| Email does not match any `auth.users` row | **Not at create** — the invite reaches the user via email (E1) or copy-pasted link. The in-app `invite` notification is **not created** for this case; the user lands in the room directly via `pendingInviteIntent`. |

The lookup ("does this email match an existing user?") is performed by
the `room-notifications` Edge Function using the service-role client's
`svc.auth.admin.listUsers({ perPage: 200 })` pattern (the same pattern
`request-argument-deletion` uses to resolve admin emails). The email
match is **lower-cased + trimmed** to align with QOL-038's
`invitee_email_lower` discipline. If the lookup itself fails, the
function logs and returns success (notification creation is best-effort
per §6.4).

### E2.2 — New trigger 10: `invite_accepted_by_invitee` (extends §5)

| # | Trigger (`type`) | Fires when | Recipients | Neutral copy intent | Deep-link target |
|---|---|---|---|---|---|
| 10 | `invite_accepted_by_invitee` | An `argument_room_invites.status` row transitions `pending → accepted` (i.e. the invitee redeems and is enrolled as a participant) | The **inviter** (`invited_by`) | "`<invitee display name>` joined: `<room title>`." The trigger fires from `manage-room-invite` accept action calling
`room-notifications` with `type: 'invite_accepted_by_invitee'` and
`recipientUserId: invite.invited_by`. The Edge Function authorisation
check: the caller must have a JWT and that JWT must belong to a user
whose id matches the invite's `invitee_profile_id` (the user who just
accepted). The function re-derives this from the DB; it does not trust
the caller's claim of "I just accepted invite X".

### E2.3 — Optional trigger 11: `invite_expired_notice` — DEFERRED

The de-escalation use case suggests an inviter should know when their
invitee did not redeem (the invite expires silently at TTL). Adding
this requires either a `pg_cron` job (out of scope for v1; CDiscourse
does not use `pg_cron` yet) or a lazy-derivation pass on the inviter's
notification list (acceptable but adds complexity). Recommend
**deferring** to a follow-up card and listing as a §17 open question.

### E2.4 — `classifyArgumentTrigger` is unchanged (§6.3)

The new trigger 10 is invite-derived, not argument-derived. The
existing `classifyArgumentTrigger` returns at most one
argument-derived trigger; trigger 10 is added to
`resolveRecipients` + the `RoomNotificationType` union but does not
change the classifier.

## E3 — Preferences integration (amends §1 / §16 / §17)

The pre-survey found an existing
`src/features/preferences/userPreferencesModel.ts` with:

```ts
/** Honest stub — persisted, drives nothing in v1 (no push notifications). */
notificationsOptInStub: boolean;
```

…and matching copy in `preferencesCopy.ts`:

```ts
helper: "Notifications aren't available yet. This remembers your choice for when they are.",
```

QOL-040 is the card that makes notifications available. The stub's
copy explicitly promises that the user's choice will be honoured when
the feature arrives. The original design's §1 ("notification
*preferences* — a possible v2 follow-up") predates the awareness that
the stub already exists. This enrichment adopts **Option A** from the
verification doc's §5:

### E3.1 — Preferences blob change

- Rename `notificationsOptInStub: boolean` →
  `notificationsEnabled: boolean`.
- Change the default from `false` → `true` (the storyboard
  de-escalation use case is opt-out, not opt-in).
- Preserve the *user's* prior choice across the rename: if a user
  previously set `notificationsOptInStub: true`, the rename script in
  `mergeWithDefaults()` reads either old or new key, with the new key
  winning. The user's intent (they flipped the stub) is honoured.

### E3.2 — Preferences copy change

`preferencesCopy.ts` `NOTIFICATIONS_COPY`:

```ts
export const NOTIFICATIONS_COPY = {
  label: 'Notifications',
  helper:
    'Show in-app notifications for replies, invites, and settlements in your arguments.',
  switchAccessibilityLabel: 'Show in-app notifications',
} as const;
```

The doctrine ban-list scanner
(`__tests__/preferencesDoctrine.test.ts`) continues to enforce that
none of `winner`, `loser`, `correct`, `true`, `false`, `proven`,
`game`, etc. appear. The new copy passes.

### E3.3 — Client-side behaviour

`useNotifications` (§6.6) reads `notificationsEnabled` from the
preferences hook:

- If `true` (default): poll-on-focus + render the list + show the
  badge.
- If `false`: do not poll. The list screen renders an empty-state
  message: *"You've turned notifications off. Turn them back on in
  My preferences."* The badge is hidden.

**The Edge Function ignores the preference.** Notifications are still
written to `room_notifications` regardless. This is intentional: if the
user turns notifications back on, they catch up on history (their list
is not silently truncated). This also keeps the Edge Function
stateless about per-user preferences — no preference table lookup per
notification write.

### E3.4 — Test changes

- `__tests__/userPreferencesModel.test.ts` — update fixture and

  one test asserting the rename migration works (a v1 blob with
  `notificationsOptInStub: true` is read as
  `notificationsEnabled: true`).
- `__tests__/preferencesDoctrine.test.ts` — likely no change (the
  doctrine of the new copy is the same as the doctrine of the old
  copy).
- New: `__tests__/useNotifications.preference.test.ts` — asserts the

## E5 — Display grouping for high-volume rooms (extends §17)

The original §17 lists four open questions. Add a fifth:

5. **List-side display grouping.** Ten replies in a hot room produce
   ten notification rows. The model handles this correctly (no
   collapsing at insert time, so each tap can navigate to a distinct
   node), but the UI may want to visually group them:
   *"5 new responses in `<room title>`. Tap to see the latest."* This
   is purely a `NotificationListScreen` render concern — no schema or
   model change is required. **Recommend deferring** unless the
   operator flags a specific room where the un-grouped list becomes
   noisy. If the operator wants v1 grouping, add a `groupingRule:
   'by_room_and_type'` helper in `notificationModel.ts` that takes
   `RoomNotification[]` and returns a grouped render plan; this
   keeps the model pure and the screen thin.
