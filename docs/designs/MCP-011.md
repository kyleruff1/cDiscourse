## 1. Goal

MCP-011 implements the **first safe, no-live-call layer of the MCP semantic-referee stack**: the pure-TypeScript foundation that every later MCP implementation card (MCP-012 … MCP-016) builds on. It establishes the directory `src/features/semanticReferee/` with four things and nothing else:

1. **The canonical Node-side type contract** — `SemanticRefereePacket` / `SemanticBinarySample` and their enums, re-stated verbatim from MCP-001 §7, so downstream cards import one shared definition instead of minting local mirrors.
2. **A strict packet parser / validator** — `parseSemanticPacket(raw: unknown)` — that accepts a known-good packet and **rejects every malformed, off-contract, or unsafe provider output**: non-JSON, top-level arrays, `authoritative: true`, unknown classifier ids, non-binary values, verdict / person tokens, secret-shaped strings, chain-of-thought fields, raw prompt echoes, and any field that could ask the game to block a post or declare a winner.
3. **A deterministic mock fixture provider** plus an **adversarial malformed-output fixture set** — 18 fixture groups covering valid packets and every rejection path.
4. **A stable, order-normalized cache-key helper** — `buildSemanticCacheKey` / `serializeSemanticCacheKey`.

MCP-011 calls **no provider**, adds **no Edge Function**, makes **no network call**, reads **no env var**, renders **no user-facing UI**, and adds **no dependency**. It is pure TypeScript and fixtures. It is the structural guarantee that no later MCP card can let an untrusted model output reach the game unvalidated — and the proven contract MCP-012 / MCP-013 / MCP-016 consume rather than re-derive.

## 2. Product thesis

The MCP semantic referee is an **advisory** layer: a low-cost model answers tightly bounded yes/no structural questions about a move and returns a compact `SemanticRefereePacket`. The whole layer's safety rests on one fact — **the packet is `authoritative: false` and the deterministic game never trusts it blindly.** That fact is only true if something *enforces* it.

A model can be asked to return strict JSON; it can also return something else — an extra `block: true` field, an `authoritative: true`, a `winner` verdict, a chain-of-thought paragraph, a leaked URL inside an `evidenceSpan`, a classifier id that does not exist. If any of that reaches the referee ledger (MCP-013) or a banner (MCP-014), the advisory layer has quietly become an authority. The validator is the wall that makes "advisory" structurally true: an off-contract packet is **rejected**, the consumer falls back to the deterministic layer 1, and the game is unaffected.

MCP-011 builds that wall first, in pure TypeScript, with an adversarial fixture set that proves every rejection path — **before** any card builds a provider call. It is deliberately the cheapest, most boring, most foundational card in the slate: no network, no UI, no Edge Function, no cleverness. That is the point. The exciting cards (a live classifier) are only safe to build *after* the boring wall exists and is tested.

## 3. Non-goals

MCP-011 explicitly does **not**:

- call any provider (Anthropic, OpenAI, xAI, MCP, X) — live or otherwise;
- add or modify an Edge Function;
- add a provider adapter, a provider registry, or any runtime env lookup;
- read or modify `.env*`;
- add a new dependency (`zod@4` is already a dependency — §7);
- render any user-facing UI, banner, chip, or string;
- wire anything into a screen, the composer, the submit path, or the timeline;
- compute a score, a credit, a delta, a winner, or a verdict (that is MCP-013);
- redact move/parent bodies (redaction is upstream, at the MCP-009 boundary — §12);
- implement trigger gating, batching, retry, or the in-memory cache *store* (that is MCP-012);
- build the referee ledger, the banner library, or the override UX (MCP-013/014/015);
- persist anything — no table, no migration, no Supabase write;
- compute `contentHash` or `inputHash` (those are supplied by the upstream boundary — §11).

MCP-011's `mockFixtureProvider` is a **test/dev fixture source**, not a runtime provider — it is never wired into a running screen or a post path.

## 4. Doctrine constraints

From `cdiscourse-doctrine`, `test-discipline`, `evidence-doctrine`, `point-standing-economy`, `supabase-edge-contract`, and the MCP roadmap expansion §3. **A violation is a blocker, not a comment.** Each line states the constraint and how MCP-011 respects it.

1. **AI never decides who is right.** The validator **rejects** any packet field that asserts a truth value (`truthValue`, `isCorrect`, `verdict`, …) — they are unknown fields and fail the strict schema. No validated packet can carry a truth field.
2. **AI never declares a winner / loser.** The content scanner rejects any string field containing a verdict token (`winner`, `loser`, `won`, `lost`, `right`, `wrong`, `true`, `false`, `correct`, `proven`, `defeated`, …) — including inside a `reasonCode`.
3. **AI never blocks an ordinary post by itself.** The packet contract has no `block` field; the validator rejects any `block` / `blocked` / `shouldBlock` field with a specific `block_field` rejection. A rejected packet → the consumer falls back to layer 1; **rejection never blocks a post.**
4. **AI never runs on the client.** MCP-011 contains no provider call and no network code; `semanticRefereeNoLiveAi.test.ts` statically proves it (§15).
5. **`authoritative` is the literal `false`.** The schema pins `authoritative` to `false`; `true`, absent, or any other value → `authoritative_not_false` rejection. The TS type uses the literal `false`.
6. **Popularity / heat / virality is never evidence.** MCP-011 reads no engagement signal; the cache key and the validator have no heat/like/view input. (The seed-bank classifier `uses_popularity_as_evidence` classifies *text* — MCP-011 only validates that the binary is `0/1`.)
7. **No verdict / person labels.** Constraint 2 plus a person-label scan (`liar`, `troll`, `dishonest`, `bad faith`, `manipulative`, `propagandist`, `extremist`, `stupid`, `idiot`, …) over every string field → `person_label` rejection.
8. **Mock-first.** The `mockFixtureProvider` is deterministic and synchronous; it is the default and the only provider MCP-011 ships. Live providers are a separate operator-gated card.
9. **No `snake_case` internal codes in user-facing strings.** The `SemanticRefereePacket` has **no user-facing copy field** (its `classifierId` / `reasonCode` are internal and mapped to plain language by MCP-013/MCP-014 via `gameCopy`). The validator rejects any *unknown* field — so a smuggled `bannerText` / `displayMessage` copy field fails as `copy_field_smuggled`. MCP-011 exports a `looksLikeInternalCode` helper for the downstream rendering cards; it adds no `gameCopy` code itself.
10. **Pure TypeScript.** Every MCP-011 file is pure TS — no Supabase, no React, no network, no `Deno`, no `process.env` — same constraint as `src/lib/constitution/engine.ts`. Enforced by `semanticRefereeNoLiveAi.test.ts`.
11. **No service-role, no direct `public.arguments` insert, no `.env*` access.** MCP-011 has no Supabase surface at all.
12. **v1 scope guards.** No voting, no winner-producing scoring, no search, no push, no OAuth, no public API. MCP-011 is a validator + fixtures; it introduces none.

## 7. Semantic packet schema — import / definition strategy

**MCP-011 defines the canonical Node-side type contract; it does not import it from anywhere.**

- `semanticRefereeTypes.ts` re-states MCP-001 §7's `SemanticRefereePacket` and all its sub-types **verbatim** as TypeScript. This is the single source of truth for the Node/Expo side (`src/`).
- It does **not** import from `supabase/functions/_shared/semanticReferee/` — that Deno tree (a) does not exist yet (MCP-016 creates it) and (b) uses Deno `.ts`-extension import specifiers that do not resolve under the Node/Jest toolchain. The Deno side is a **mirror**: MCP-016 copies MCP-011's contract into the Deno tree and adds a parity test. MCP-011's `semanticRefereeTypes.ts` is the upstream of that mirror.
- The contract is **frozen at `packetVersion: 'mcp-semantic-referee-v0'`**. MCP-011 does not widen it. A future contract change bumps `packetVersion` and is a new design card (MCP-001's rule, §18).

**Validation library:** the validator uses **`zod@4`** for the structural layer. `zod@^4.4.3` is **already a dependency** (`package.json`) and is the exact library MCP-009's Deno `schema.ts` uses — so the Node and Deno validators stay structurally parallel, which is what makes the MCP-016 parity test meaningful. Reusing an existing dependency does not violate the "no new dependency" rule. The **content-safety scanner** (ban-list, secret/PII shapes, unknown-field classification) is a **pure-TS layer on top** of the zod schema — zod handles "is this the right shape?", the scanner handles "is this string safe to let through?". `semanticRefereeTypes.ts` itself stays dependency-free (pure type declarations + `const` arrays); only `semanticRefereeValidator.ts` imports `zod`. (If the operator prefers a fully hand-rolled validator, see §18 decision 1 — the design works either way; zod is the recommendation.)

## 8. Strict JSON validation plan

`parseSemanticPacket(raw: unknown): SemanticPacketValidationResult` runs five stages. The first failing stage short-circuits where continuing is meaningless (e.g. non-JSON); otherwise the validator **collects all rejections** so a fixture test sees the complete failure set.

**Stage 1 — JSON acceptance.** `raw` is `unknown`. If it is a `string`, attempt `JSON.parse` inside `try/catch`; a throw → `{ ok: false, rejections: [{ code: 'non_json', ... }] }`. If it is already an object, use it. A `number` / `boolean` / `null` / `undefined` → `not_an_object`.

**Stage 2 — top-level shape.** Must be a non-null, non-array plain object. `Array.isArray(value)` → `top_level_array`. Anything else non-object → `not_an_object`.

**Stage 3 — structural schema (zod `.strict()`).** A `zod@4` object schema, every nested object also `.strict()`:
- `packetVersion` — `z.literal('mcp-semantic-referee-v0')`.
- `authoritative` — `z.literal(false)`. `true` / absent / other → `authoritative_not_false`.
- `provider` — `z.enum(ALL_SEMANTIC_PROVIDERS)`.
- `promptVersion`, `modelVersion`, `inputHash`, `contentHash`, `roomId` — non-empty bounded strings.
- `moveId`, `parentId`, `selectedAction`, `selectedMoveType`, `debateMode` — optional bounded strings.
- `binaries` — array; each element a `.strict()` object: `classifierId` ∈ `ALL_SEMANTIC_CLASSIFIER_IDS` (else `unknown_classifier_id`); `value` = `z.union([z.literal(0), z.literal(1)])` (a float, `'1'`, `2`, `true` → `non_binary_value`); `confidence` ∈ `ALL_CONFIDENCE_VALUES` (else `unknown_confidence`); `reasonCode` a bounded string (≤ `MAX_COPY_FIELD_LEN`); optional `evidenceSpan` / `parentSpan` bounded strings.
- `routeSuggestion` ∈ `ALL_ROUTE_SUGGESTIONS` (else `unknown_route_suggestion`).
- `frictionSuggestion` ∈ `ALL_FRICTION_SUGGESTIONS` (else `unknown_friction_suggestion`).
- `scoreHints` — a `.strict()` object of exactly the six MCP-001 §7 integer fields, each `z.int().min(SCORE_HINT_MIN).max(SCORE_HINT_MAX)` (out of range → `score_hint_out_of_range`).
- **`.strict()` everywhere** means any **unknown top-level or nested key** is a rejection. Stage 4 then classifies *which* kind.

**Stage 4 — unknown-field classification.** When `.strict()` reports an unrecognized key, the validator maps the key name to the most specific rejection code:
- `block` / `blocked` / `shouldBlock` / `gate` → `block_field`.
- `reasoning` / `thinking` / `chain_of_thought` / `cot` / `rationale` / `analysis` / `scratchpad` / `deliberation` → `chain_of_thought_field`.
- `prompt` / `systemPrompt` / `system_prompt` / `rawPrompt` / `raw_prompt` / `promptEcho` / `inputPrompt` → `raw_prompt_field`.
- `bannerText` / `displayMessage` / `userMessage` / `copy` / `message` / any field that reads as user-facing copy → `copy_field_smuggled`.
- anything else → `unknown_field`.
A missing required field → `missing_field`.

**Stage 5 — content-safety scan (pure-TS, after structure passes).** For **every string-valued field** in the packet (`reasonCode`, `evidenceSpan`, `parentSpan`, `modelVersion`, `debateMode`, `selectedAction`, …):
- **Verdict-token scan** — reject (`verdict_token`) if the string contains, as a whole word or a `snake_case` segment, any of: `winner · loser · won · lost · right · wrong · true · false · correct · incorrect · proven · defeated`. (This catches a `reasonCode` like `winner_detected`.)
- **Person-label scan** — reject (`person_label`) for: `liar · lying · dishonest · bad faith · manipulative · troll · propagandist · extremist · stupid · idiot · dumb · smart`.
- **Secret-shape scan** — reject (`secret_shape`) for the `sk-ant-` / `xai-` / `sb_secret_` token prefixes, JWT-shaped `eyJ…` triplets, `Bearer ` headers, `Authorization:` headers, or service-role-shaped strings. Reuse the existing `scripts/engagement-intelligence/` sanitizer patterns — do not re-invent them.
- **PII-shape scan** — reject (`pii_shape`) for an `@handle` (1–15 word chars), a bare `x.com` / `t.co` / `twitter.com` host or status URL, any `http(s)://` URL, an email address, or a 15–20-digit run (a post id).
- **Reason-code family check** — `reasonCode` must be lowercase `snake_case`, ≤ `MAX_COPY_FIELD_LEN`, and begin with a known `REASON_CODE_FAMILIES` prefix; otherwise `unknown_reason_code`. (Family + shape, not an exact 90-entry allowlist — §18 decision 3.)

**Stage 6 — coherence check.** Two `binaries` entries with the same `classifierId` → `duplicate_classifier_id` (a packet classifies each structural property at most once; a downstream ledger cannot deterministically reconcile a self-conflicting packet).

**Result.** `{ ok: true, packet }` only if **zero** rejections — and `packet` is the typed, frozen `SemanticRefereePacket`. Otherwise `{ ok: false, rejections }`.

### 8.1 Low confidence produces no point credit

A low-confidence packet is **valid** — it passes `parseSemanticPacket`. But MCP-001 §14 and the readiness doctrine require that a low-confidence binary never moves score on its own. MCP-011 exports the predicate — it does **not** compute credit (that is MCP-013):

```ts
export function isCreditEligible(sample: SemanticBinarySample): boolean; // false iff confidence === 'low'
export function creditEligibleBinaries(packet: SemanticRefereePacket): readonly SemanticBinarySample[];
```

MCP-013's ledger calls these; a low-confidence binary may feed only an advisory / manual-choice signal, never a credit. Fixture group 15 proves a low-confidence packet validates **and** yields zero credit-eligible binaries.

## 9. Fixture provider contract

`semanticRefereeFixtures.ts` exports three things — and a hard rule binds all of them: **every fixture is synthetic, minimal, and obviously fake. No fixture contains a real secret, a real API key, a real `@handle`, a real post id, a real URL, or copied hostile text.** (§12.)

```ts
// 1. Valid packets — pass parseSemanticPacket, typed SemanticRefereePacket.
export const VALID_FIXTURES: Readonly<Record<string, SemanticRefereePacket>>;

// 2. Malformed payloads — invalid by construction, so typed `unknown`.
export interface MalformedFixture {
  id: string;
  group: number;                              // 7..16 per §10
  raw: unknown;                               // a string for non-JSON; an object for the rest
  expectedRejectionCodes: readonly SemanticRejectionCode[];
  note: string;                               // plain-language description of the defect
}
export const MALFORMED_FIXTURES: readonly MalformedFixture[];

// 3. Cache-key fixtures — input pairs with an expected key relationship.
export interface CacheKeyFixture {
  id: string;
  group: number;                              // 17 | 18
  inputA: SemanticCacheKeyInput;
  inputB: SemanticCacheKeyInput;
  expect: 'same_key' | 'different_key';
  note: string;
}
export const CACHE_KEY_FIXTURES: readonly CacheKeyFixture[];

// 4. The deterministic mock provider — a TEST/DEV fixture source, NOT a runtime provider.
export interface MockClassifyRequest {
  fixtureId?: string;                         // pick a named valid fixture
  roomId: string;
  classifierIds: readonly SemanticClassifierId[];
  contentHash: string;
  promptVersion?: string;
}
export function mockFixtureProvider(request: MockClassifyRequest): SemanticRefereePacket;
```

`mockFixtureProvider` is:
- **pure and synchronous** — no `async`, no `Promise`, no network, no I/O;
- **deterministic** — the same request returns a byte-identical packet (a fixture test asserts this);
- **always valid** — every packet it returns passes `parseSemanticPacket`. It **never** returns a malformed fixture. The malformed fixtures are a *separate* export consumed only by the validator's rejection tests.
- **provenance-honest** — every packet it returns carries `provider: 'mock'` and `authoritative: false`.

It is the stand-in MCP-012 / MCP-013 / MCP-016 tests use wherever a real provider would sit — so those cards can be built and fully tested with zero network.

## 10. Malformed-output rejection rules — the 18 fixture groups

| # | Fixture group | Kind | Expected rejection |
|---|---|---|---|
| 1 | Valid minimal packet (one binary, `no_route_change`, all `scoreHints` 0) | valid | accepted |
| 2 | Valid continuity packet (`responds_to_parent: 1`, `mainline`) | valid | accepted |
| 3 | Valid evidence-debt packet (`creates_source_chain_gap: 1`, `sourceChainDebt` hint) | valid | accepted |
| 4 | Valid branch-recommendation packet (`suggests_side_branch: 1`, `routeSuggestion: vertical_chime_branch`) | valid | accepted |
| 5 | Valid synthesis-readiness packet (`ready_for_synthesis: 1`, `synthesisReadiness` hint) | valid | accepted |
| 6 | Valid conflict-routed packet (`responds_to_parent: 0`, `confidence: low`, `frictionSuggestion: suggest_branch`) | valid | accepted |
| 7 | Malformed JSON (`raw` is a non-parseable string) | malformed | `non_json` |
| 8 | `authoritative: true` | malformed | `authoritative_not_false` |
| 9 | Winner / truth language attempted (a `reasonCode` containing a verdict token) | malformed | `verdict_token` |
| 10 | Person-label attempted (a string field containing a person token) | malformed | `person_label` |
| 11 | Service-role / secret-shaped string attempted (a synthetic key-shaped string in a field) | malformed | `secret_shape` |
| 12 | Unknown classifier id | malformed | `unknown_classifier_id` |
| 13 | Unknown route suggestion | malformed | `unknown_route_suggestion` |
| 14 | Non-binary classifier value (`value: 2` / `0.5` / `'1'`) | malformed | `non_binary_value` |
| 15 | Low-confidence no-credit case (valid packet, `confidence: low`) | **valid** | accepted; `isCreditEligible` → false |
| 16 | Duplicate classifier conflict (two binaries, same `classifierId`, conflicting `value`) | malformed | `duplicate_classifier_id` |
| 17 | Cache-key stable case (two inputs identical up to classifier-id order) | cache-key | `same_key` |
| 18 | Cache-key invalidation case (two inputs differing only in `promptVersion`) | cache-key | `different_key` |

Additional malformed coverage the implementation card adds beyond the named 18 (each a one-line fixture, same `MalformedFixture` shape): top-level array → `top_level_array`; a `block: true` field → `block_field`; a `reasoning` field → `chain_of_thought_field`; a `prompt` field → `raw_prompt_field`; a `bannerText` field → `copy_field_smuggled`; a `scoreHints.continuityCredit: 9` → `score_hint_out_of_range`; an `@handle` / URL / email / post-id in `evidenceSpan` → `pii_shape`; an over-length `reasonCode` → `field_too_long`; a `reasonCode` with no known family prefix → `unknown_reason_code`.

**Rejection rule summary** — the validator rejects (never silently sanitizes — §18 decision recorded in §8) a packet that:
- is not parseable JSON, is a top-level array, or is not an object;
- carries `authoritative: true` or omits `authoritative: false`;
- carries any unknown field — and classifies a `block` / chain-of-thought / raw-prompt / smuggled-copy field with its specific code;
- carries an unknown `classifierId` / `routeSuggestion` / `frictionSuggestion` / `confidence`;
- carries a non-`0/1` binary value;
- carries a `scoreHints` integer outside `0..3`;
- carries a `reasonCode` outside the known family + shape;
- carries a verdict token or a person label in any string field;
- carries a secret-shaped, JWT-shaped, Bearer / Authorization, `@handle`, URL, email, or post-id-shaped string in any field;
- carries two binaries for the same `classifierId`.

A rejected packet is never partially used. The consumer (MCP-013/MCP-016) treats `{ ok: false }` exactly as it treats `{ enabled: false }` — fall back to deterministic layer 1, no error to the user, **the post is unaffected**.

## 11. Cache-key helper contract

```ts
export interface SemanticCacheKeyInput {
  roomId: string;
  parentId?: string;                          // omitted/'' for a root move
  contentHash: string;                        // supplied — hash of the redacted move body (MCP-009)
  promptVersion: string;
  classifierIds: readonly string[];           // the requested classifier set
  roomMode?: string;                          // GAME-003 debate mode — a prompt input
  selectedAction?: string;                    // quick-action preset — a prompt input
}
export interface SemanticCacheKey {
  roomId: string;
  parentId: string;                           // '' sentinel for a root move
  contentHash: string;
  promptVersion: string;
  classifierSetHash: string;                  // order-normalized hash of classifierIds
  roomMode: string;                           // '' sentinel when absent
  selectedAction: string;                     // '' sentinel when absent
}
export function buildSemanticCacheKey(input: SemanticCacheKeyInput): SemanticCacheKey;
export function serializeSemanticCacheKey(key: SemanticCacheKey): string;
export function hashClassifierSet(ids: readonly string[]): string;
```

**Rules:**
- `buildSemanticCacheKey` order-normalizes `classifierIds` (sort lexically, **dedupe**), feeds them to `hashClassifierSet`, and applies empty-string sentinels for absent `parentId` / `roomMode` / `selectedAction` so the key is **total** (no `undefined`-vs-`''` ambiguity).
- `hashClassifierSet` is a small **pure-TS deterministic string hash** (FNV-1a / djb2 — no `crypto` import, so the helper is isomorphic and matches the engine-purity rule). `['a','b']` and `['b','a']` and `['a','a','b']` all hash identically.
- `serializeSemanticCacheKey` joins the seven components with the unit-separator character `` (which cannot appear in a roomId / hash / version / mode / action) → a stable, total, collision-safe string.
- MCP-011 **does not compute `contentHash`** — it is supplied (the upstream MCP-009 boundary hashes the redacted body). MCP-011 only hashes the classifier-id *list*.

**Reconciliation with MCP-004.** MCP-004 §"Data model" fixed the cache tuple `{roomId, parentId, contentHash, promptVersion, classifierSet}`. MCP-011's `SemanticCacheKey` is a **strict superset**:
- it keeps all five MCP-004 components (`classifierSet` becomes `classifierSetHash`, an order-normalized hash — MCP-004 itself requires order-normalization, §"Data model");
- it **adds `roomMode` and `selectedAction`** because both are prompt inputs (MCP-009 `ClassifyMoveRequest.roomContext` carries `debateMode` and `selectedAction`) that MCP-004's tuple omitted — two moves with an identical body but a different room mode would otherwise collide on one key yet must classify differently.

Because it is a superset, no MCP-004 behavior breaks. MCP-012's future `semanticCache.ts` (the in-memory *store*) imports `serializeSemanticCacheKey` from MCP-011 rather than re-deriving a divergent key. If the operator prefers MCP-004's verbatim five-tuple, dropping `roomMode` / `selectedAction` is a two-field edit — flagged in §18 decision 2, not assumed.

The operator's MCP-011 acceptance test — "cache key includes promptVersion, classifierSetHash, contentHash, parentId, roomMode, selectedAction" and "changes when promptVersion changes" — is satisfied: all six named components are present (plus `roomId`), and changing any one changes the serialized string.

## 12. Redaction / safety expectations

**MCP-011 does not redact.** Redaction of move and parent bodies happens **upstream** — at the MCP-009 Edge Function boundary, before any text enters a prompt (MCP-001 §20, MCP-009 §"Risks"). By the time a packet reaches MCP-011's validator, the bodies are long gone; only short derived strings remain (`reasonCode`, optional `evidenceSpan` / `parentSpan`, provenance ids).

MCP-011's role is the **last line of defense**: the content-safety scan (§8 stage 5) treats every string field as untrusted and **rejects** — never sanitizes, never trims, never echoes — a packet whose `evidenceSpan` or any field still carries a secret shape, a JWT, a Bearer token, an `@handle`, a URL, an email, or a post id. If redaction upstream missed something, the packet does not validate and the consumer falls back to layer 1.

**Rejection messages are sanitized.** A `SemanticPacketRejection.detail` describes the *kind* of defect and the *field path* — e.g. `"binaries[1].evidenceSpan contained a key-shaped string"` — and **never includes the offending raw value**. The validator must not log, echo, or return the secret it caught. `field` is a dotted path, not a value.

**Fixtures are synthetic by construction.** The malformed fixtures that must *trip* the secret / PII / verdict scanners use **assembled, obviously-fake** strings so no real-secret-shaped literal sits in the committed source:
- the secret-shape fixture builds its string at module load by concatenation (e.g. `'sk-' + 'ant-' + 'X'.repeat(24)`) — the contiguous key-shaped literal never appears in the file;
- the verdict / person fixtures use minimal synthetic tokens (a single banned word inside a short `snake_case` `reasonCode`, or a 3–4-word synthetic phrase) — never copied hostile text;
- no fixture contains a real `@handle`, a real URL, a real email, or a real post id — only synthetic placeholders that match the *shape* the scanner detects.
`semanticRefereeFixtures.test.ts` self-scans the fixture module to prove no real secret slipped in.

## 13. No-live-call boundary

MCP-011 makes **zero network calls** and contains no path that could become one:

- no `fetch`, `XMLHttpRequest`, `node-fetch`, `undici`, `axios`, `http`, `https`, or `net` import;
- no `@anthropic-ai/sdk`, `openai`, `xai`, or any provider SDK import;
- no `Deno` reference (MCP-011 is Node/Expo-side only);
- the `mockFixtureProvider` is synchronous and returns canned fixtures — it has no I/O and cannot be made to call out;
- no `async` function in the card at all (the validator, the fixtures, and the cache-key helper are all synchronous pure functions).

`semanticRefereeNoLiveAi.test.ts` enforces this with a **source-text scan** of every MCP-011 file: it fails the build if any of the forbidden import specifiers or the token `Deno` appears. Because there is no provider seam in MCP-011, there is nothing to spy on — the absence is structural, not behavioral.

## 14. No-client-provider boundary

MCP-011 lives under `src/` (the client / Expo side). It is therefore bound by the doctrine "no AI call from the production app" (`cdiscourse-doctrine` §7). MCP-011 honors it structurally:

- MCP-011 contains **no provider adapter** — the real provider call lives only in MCP-016's Edge Function;
- the `mockFixtureProvider` is a **fixture source for tests and dev**, explicitly not a runtime provider — it is **not** imported by any screen, the composer, the submit path, the timeline, or any `App.tsx` route. The implementation card must not wire it into a running surface; the `roadmap-reviewer` blocks the card if it is;
- MCP-011 reads **no env var** — no `process.env`, no `.env`, no feature flag. There is no "is the layer on?" check in MCP-011; that gating is MCP-012 (the trigger gate) and MCP-016 (the Edge Function flag);
- `index.ts` exports the validator, the types, the fixtures, the cache-key helper, and the small predicates **for consumption by later cards and tests** — exporting a pure function is not wiring a provider.

The net effect: MCP-011 adds reusable pure-TS infrastructure to `src/` and **zero** new client behavior. Nothing runs until a later card wires it, and the only card that can wire a *provider* is MCP-016's Edge Function.

## 20. Open questions

1. **Catalog v0 (23 ids) vs the 90-seed bank.** MCP-001 §7's `SemanticClassifierId` union is the 23-id catalog v0; MCP-002's seed bank holds 90 ids (e.g. `responds_to_parent_directly` vs catalog `responds_to_parent`). MCP-011's validator validates against the **23-id catalog v0** — the locked contract. If a later card ships more of the 90, the `SemanticClassifierId` union widens, which is an MCP-001 contract touch (a new design card), and `ALL_SEMANTIC_CLASSIFIER_IDS` follows. Flagged so the implementer does not silently widen the union.
2. **`scoreHints` ceiling.** `0..3` is MCP-003's assumption, not an MCP-001-fixed value. If MCP-002's seed prompts can emit a larger integer, `SCORE_HINT_MAX` must follow — confirm with the seed-bank owner. (Operator decision 4.)
3. **`reasonCode` exactness.** Family-prefix validation accepts any `<family>_<suffix>` token. A model could emit a well-formed-but-meaningless `reasonCode` within a known family; MCP-011 accepts it (it is shape-valid and safe) and a downstream `gameCopy` mapping suppresses an unmapped code at render time (MCP-013/MCP-014). Is shape-validity enough, or should MCP-011 ship the exact 90-entry allowlist? Recommendation: shape-validity now (operator decision 3).
4. **Duplicate-classifier policy.** MCP-011 rejects any packet with two binaries for the same `classifierId`, even if their values agree. Confirm a hard reject is preferred over accept-and-dedupe — the design assumes reject (a self-referencing packet is incoherent and a model that duplicates a classifier is off-contract).
5. **Deno parity ownership.** MCP-011 owns the Node canonical `semanticRefereeTypes.ts`; MCP-016 owns the Deno mirror and the parity test. If the operator wants a single shared source (e.g. a codegen step), that is a future tooling card — MCP-011 assumes the documented-mirror + parity-test approach MCP-009 already described.
