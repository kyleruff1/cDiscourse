## Goal

Bring the Argument Surface Pivot surfaces up to the project's accessibility contract (`accessibility-targets`) and pin that conformance against regression with tests that follow the house `uxOneOneSix*` conventions. The card is an **audit + regression-guard card**, not a redesign: most cells are already green from the cards that shipped these surfaces, so the design's job is to (a) distinguish **verified-green** from **needs-fix** cell-by-cell with `file:line` evidence, (b) fix the small real gaps, (c) ship one documented reduce-motion primitive + a guard so future motion consults it, and (d) add the amendment's board-axis color-independence guard. The reduce-motion audit found the ASP set is overwhelmingly static-by-construction and every surface that genuinely animates already gates on reduce-motion — so the reduce-motion work is a **shared primitive + guard + docs**, not net-new gating.

Doctrine constraints that shape the design (`cdiscourse-doctrine`, `accessibility-targets`, `timeline-grammar`): color is never the only signal (shape/glyph/label/geometry carry meaning); no verdict / red-green traffic-light pairing; every interactive element carries role + label (+ state where a boolean applies) and a ≥44×44 target; reduce-motion degrades non-essential motion to static; screen-reader labels describe the **move / state**, never a person, and "unknown" reads as unknown. No score, no truth label, no AI call, no Supabase write, no service-role, no migration.

## Edge cases

- **Reduce-motion API absent / rejects** (jest, RN Web shim): `useReduceMotion` keeps `false`, subscription setup is wrapped in try/catch, `remove()` in cleanup is swallowed — identical to the shipped inline pattern. Test the reject path explicitly.
- **`reduceMotionOverride` supplied:** wins over the OS value both directions (`true` and `false`), even mid-session. Test both.
- **Mid-session OS toggle:** `reduceMotionChanged` listener updates the value; DisagreementPointsRail must still snap/animate correctly after adoption (render test with `reduceMotionOverride` on a narrow sheet viewport, mirroring `openIssuesRailA11y.test.tsx`).
- **RingsideCard `accessible={false}`:** verify a11y tree now exposes the nested controls; verify tap-to-activate on the card still fires `onPress` (the activate handler stays on the root). Confirm no existing test asserts the root card as a *single* a11y node (reconciliation — see Risks).
- **ProofDrawer selected affix:** the `✓ ` prefix must not leak into the `accessibilityLabel` (SR already says "selected" via state) — keep the affix visual-only on the `<Text>`, not in the label string. Empty prior-move list → no rows → guard skips.
- **Amendment — empty/terminal state:** a point whose display projection is `resolved_or_settled` renders **no** badge (`v4RowBadgeLabel` returns `''`, DisagreementPointsRail.tsx:84-88). The guard must treat "no badge for a terminal/suppressed state" as correct, and only require a non-empty label for **live** display states.
- **Amendment — `looksLikeInternalCode` suppression:** if a label would read as a raw internal code it is suppressed to `''` (MediatorNodeMarker.tsx:32; `displaySafe` in the rail). The guard asserts the *mapped* label is non-empty for every live state — i.e., `MEDIATOR_STATE_COPY[state]` is a real word, so suppression never fires on a legitimate state.
- **Doctrine edge — does heat/score leak into any a11y label?** No. The node label builder omits the standing band on the default path (`includeStandingBand:false`); the mediator labels are structural states only. Assert the ban-list holds on every new/rendered string.
