## 0. How this design was built

QOL-030 left a deliberate hole: it named the `respond` composite box type (an
optional **concession forced-list** section + a refutation body) and the
`respond_to_concession` box type (a **forced list mirroring** an incoming
concession set, each row a 5-level gradient) ‚Äî and explicitly deferred the
*internals* of both to "QOL-041" (QOL-030 ¬ß1 out-of-scope, ¬ß6.2, ¬ß12).

This design fills that hole. It is derived in three passes:

1. **Storyboard replay** (¬ß3) ‚Äî Steps 6‚Äì8 of the roommates/dishes storyboard,
   frame by frame, recording exactly what the concession list and the gradient
   must store and show.
2. **Doctrine binding** (¬ß4) ‚Äî the point-standing economy is the existing,
   shipped scoring layer. This card **consumes** `ConcessionEffect` /
   `gradeRepair`; it does not invent a parallel scoring path.
3. **Schema + UI fall out** (¬ß6‚Äì¬ß7) ‚Äî the data model and the box internals are
   the consequence of ¬ß3‚Äì¬ß4, not asserted independently.

Where the abstract model and the storyboard disagree, the storyboard wins
(QOL-030's own rule).

## 1. Goal & scope

Build the two structured surfaces a concession exchange needs, plus a
no-score acknowledgment reaction:

1. **The itemized concession list** ‚Äî the concession section of the QOL-030
   `respond` box is a **forced list** of separately-addressable point items.
   Each item is one conceded point, stored as its own row, individually
   targetable by the receiver.
2. **The per-concession acceptance gradient** ‚Äî the QOL-030
   `respond_to_concession` box renders a forced list **mirroring the incoming
   concession set row-for-row**. On each row the receiver picks one of five
   levels ‚Äî *Agree ¬∑ Agree with caveat ¬∑ Disagree based on framing ¬∑ Disagree
   based on context ¬∑ Disagree based on fact*. Any level other than *Agree*
   makes a **clarification body required** on that row. The set of chosen
   levels sets the room's **active-disagreement classification**
   (`framing` / `context` / `fact`).
3. **The fist-bump / acknowledge reaction** ‚Äî a low-friction
   partial-resolution affordance that carries **no score, no verdict, no
   standing change** ‚Äî a human "I hear you" beat.

**In scope.** The concession-item data model; the forced-list renderer for the
concession section; the gradient control + its conditional clarification; the
per-concession acceptance-decision data model; the active-disagreement
classification derivation; the fist-bump reaction data model + affordance; the
plain-language copy for all of it; how each plugs into QOL-030's
`renderSchema`.

**Out of scope (own cards / already shipped).** The one-box chassis itself
(QOL-030 ‚Äî this card *plugs into* it); the point-standing scoring math
(`src/features/pointStanding/` ‚Äî shipped Stage 6.1.4, this card *consumes* it);
the evidence form / evidence-response schemas (QOL-036 / QOL-037); the flash
menu (QOL-031); auto-wiring the point-standing engine into production (an
explicit future stage ‚Äî see `docs/point-standing-economy.md` ¬ß"How a future
stage would wire this in"); notifications (QOL-040). See ¬ß13.

## 2. Problem statement

The roommates/dishes storyboard (Steps 6‚Äì8) is the canonical concession
exchange, and today it cannot happen:

- **Step 6** ‚Äî Roommate B's single response is *two concessions (a list)* **and**
  *one refutation (a body)*. The `arguments` table has one `body text` column.
  Two concessions today would be one blob of prose. The storyboard is explicit:
  "stored as **separate, individually-addressable sub-items**, not as one blob
  of text ‚Äî Step 8 needs to act on each one." There is **no concession-item
  model**.
- **Step 8** ‚Äî Roommate A reviews B's two concessions and applies a **per-item
  5-level acceptance gradient** (`Agree` on item 1, `Disagree based on fact` on
  item 2). Any non-`Agree` choice **requires a clarification body**. The chosen
  levels set the room's **active-disagreement classification**. None of this
  exists: no gradient control, no per-concession acceptance row, no
  active-disagreement field.
- **Step 19** ‚Äî Roommate A taps a **fist-bump / acknowledgment reaction** to
  mark a partial resolution. There is no reaction model and ‚Äî by v1 doctrine ‚Äî
  there must never be a *voting* reaction. The fist-bump is the one allowed
  reaction precisely because it carries **no score**.

The `missing-capabilities-and-issues.md` audit classifies all three as
**Missing** (rows: "Concession list + 5-level acceptance gradient",
"fist-bump reaction").

The point-standing economy (`src/features/pointStanding/`) already models the
*scoring consequence* of a concession (`ConcessionEffect`, `gradeRepair`). What
is missing is the **structured input surface** that produces a clean,
machine-addressable concession set, and the **structured response surface** the
receiver uses to react to each item. QOL-041 builds those two surfaces.

**Doctrine the problem imposes.** A concession is a scoring **repair, not a
defeat** (`docs/point-standing-economy.md` doctrine; point-standing-economy
skill rule 1). The UI must make conceding feel like a *constructive move*, never
a *loss*. No "winner"/"loser"/"truth"/"verdict" copy anywhere. The gradient is
**gameplay analysis**, never a truth ruling.

## 3. Storyboard replay ‚Äî Steps 6‚Äì8 (and 19)

Columns: storyboard frame ‚Üí the data it must produce ‚Üí the UI surface QOL-041
must build.

| Frame | Data produced | QOL-041 surface |
|---|---|---|
| **Step 6** ‚Äî B writes 2 concessions + 1 refutation | 1 response `arguments` row (the refutation body) + **2 `concession_items` rows**, each individually addressable, each with `ordinal` 1/2 | The `respond` box's **concession section = forced list** (QOL-030 W4); `[+ Add a point]`; no single body for that section |
| **Step 7** ‚Äî A is notified, opens room, Node 2 active | (notification ‚Äî QOL-040; no QOL-041 data) | ‚Äî |
| **Step 8** ‚Äî A sets gradient per concession: item 1 ‚Üí `Agree`, item 2 ‚Üí `Disagree based on fact`; writes a required clarification for item 2 | 1 response `arguments` row + **2 `concession_acceptances` rows** (one per concession item), the item-2 row carrying a `clarification_body`; the room's `active_disagreement` derives to `fact` | The `respond_to_concession` box = **forced list mirroring** B's 2 items, each row a **5-level gradient** + conditional-required clarification |
| **Step 19** ‚Äî A taps a fist-bump on B's tangent-branch reply | 1 **`move_reactions` row** (`kind = fist_bump`), no standing delta, no score | A **fist-bump / acknowledge** affordance on a node (a QOL-030/031 *direct entry* ‚Äî no box opens) |

**Findings the replay forces:**

- **F1.** A concession item must be its **own row**, not a sub-string of a body.
  Step 8 must `select` exactly one concession item and attach an acceptance to
  it ‚Äî that requires a stable per-item id and a foreign key.
- **F2.** The acceptance is **per-concession**, not per-response. One response
  to a concession set carries N acceptance rows (one per item). A is allowed to
  `Agree` item 1 and `Disagree based on fact` item 2 in a *single* posted move.
- **F3.** The clarification is **conditional and per-row**: required iff the
  row's level ‚âÝ `agree`. The `respond_to_concession` box cannot be posted while
  any non-`Agree` row has an empty clarification ‚Äî this is a **validation
  block**, not a score penalty (doctrine: validation may block, score never).
- **F4.** The room's `active_disagreement` is **derived** from the chosen
  gradient levels, not chosen directly. The storyboard sets it to `fact` because
  A chose `Disagree based on fact` on item 2. ¬ß5.3 specifies the derivation.
- **F5.** The fist-bump is a **reaction**, structurally separate from a posted
  argument move. It is **not** a concession, **not** an acceptance, **not** a
  standing-affecting move. It is the one v1-allowed reaction *because* it carries
  no score (v1 scope bans voting/scoring; the fist-bump is acknowledgment, not a
  vote).

## 4. Doctrine binding ‚Äî consume, do not re-derive, the point-standing economy

The card's non-negotiable: **a concession is a scoring repair, never a defeat;
consume, do not re-derive, the point-standing economy.**

The point-standing economy is **shipped** (Stage 6.1.4,
`src/features/pointStanding/`). It exposes pure functions:

- `gradeChallenge(input)` ‚Äî grades a challenge, opens an `OpenIssueDebt`.
- `gradeRepair(input, options)` ‚Äî grades a repair (concession / narrowing /
  synthesis / evasion), returns a `PointStandingDelta`, may close a debt.
- `classifyConcessionEffect(...)` ‚Äî detects which `ConcessionEffect` a move is.

QOL-041 **does not call these functions in production code in this card.** The
point-standing engine is "not auto-wired into production" by design
(`docs/point-standing-economy.md` ¬ß"What we do NOT do"; the skill's anti-pattern
list). QOL-041 instead does two things that respect that boundary:

1. **It produces the structured inputs** the point-standing engine *will*
   consume when a future stage wires it in. A clean, itemized concession set +
   an explicit per-item acceptance gradient is exactly the
   `RepairGradingInput` shape that engine needs. QOL-041 builds the surface;
   the future wiring stage (out of scope here) routes it through `gradeRepair`.
2. **It maps the gradient ‚Üí `ConcessionEffect` *vocabulary* deterministically**,
   in a pure model, **without computing any delta**. The mapping is a label
   correspondence the receiver and future grader share ‚Äî not a score.

The mapping (pure, no numbers, no delta):

| Acceptance gradient level | Maps to `ConcessionEffect` family | Meaning |
|---|---|---|
| `agree` | the concession **lands** ‚Äî repair accepted | the conceding party's narrowing/concession is accepted; the debt it was paying *can* close |
| `agree_with_caveat` | repair **lands with a rider** | the concession is accepted but the receiver adds a scoped caveat ‚Äî a new, narrower item |
| `disagree_framing` | repair **contested on framing** | the concession's *frame* is rejected ‚Äî opens a `framing` axis |
| `disagree_context` | repair **contested on context** | the concession ignores context the receiver supplies ‚Äî opens a `context` axis |
| `disagree_fact` | repair **contested on fact** | the concession rests on a factual claim the receiver disputes ‚Äî opens a `fact` axis |

> **Doctrine guard.** This table is a **vocabulary bridge**, not a scoreboard.
> QOL-041 stores the *level* and the *clarification*; it never stores or
> displays a `broadStandingDelta` / `narrowStandingDelta` / pressure number.
> The mapping above is so the future wiring stage knows which `gradeRepair`
> path each level corresponds to. No `winner`, no `loser`, no `truth`.

**What QOL-041 must NOT do (anti-patterns from the point-standing skill):**

- Must not declare a concession a "loss" or the receiver a "winner".
- Must not infer a concession from silence ‚Äî a concession is an **active**
  forced-list item the conceding party types.
- Must not auto-derive an acceptance ‚Äî the receiver must explicitly pick a
  level on every mirrored row before the box can post.
- Must not aggregate the gradient into a single percentage or score.
- Must not let the fist-bump touch standing ‚Äî it is reaction-only.

## 5. Data model

Three new tables, one new migration. All `arguments` writes still go only
through the `submit-argument` Edge Function ‚Äî QOL-041 adds **child rows** that
hang off an argument; it never inserts into `public.arguments` directly and
never uses the service role from the client.

Migration file: `supabase/migrations/20260521000010_qol_041_concession_acceptance.sql`
(next sequential number after `20260517000009_meta_1a_point_tags.sql`).

### 5.1 `concession_items` ‚Äî the itemized concession list (F1)

```sql
create table public.concession_items (
  id            uuid        primary key default gen_random_uuid(),
  debate_id     uuid        not null references public.debates(id)   on delete cascade,
  -- the response argument whose concession SECTION this item belongs to
  argument_id   uuid        not null references public.arguments(id) on delete cascade,
  -- the node this concession concedes a point TO (the parent being responded to)
  conceded_to_argument_id uuid not null references public.arguments(id) on delete cascade,
  author_id     uuid        not null references public.profiles(id)  on delete restrict,
  ordinal       int         not null check (ordinal >= 0),
  item_text     text        not null check (char_length(item_text) between 1 and 600),
  -- soft-delete only; rows are never hard-deleted (repo doctrine)
  removed_at    timestamptz,
  removed_by    uuid        references public.profiles(id) on delete set null,
  created_at    timestamptz not null default now(),
  unique (argument_id, ordinal)
);
```

- One row per conceded point. The forced list in the `respond` box produces N
  rows for one response argument.
- `ordinal` preserves list order; `unique (argument_id, ordinal)` keeps the
  mirrored list stable for the receiver.
- `conceded_to_argument_id` is the node the concession concedes *to* ‚Äî the
  receiver of the concession is that node's author; this is who may set the
  gradient (¬ß6, permissions).
- No score column, no count column, no `is_accepted` ‚Äî acceptance lives in its
  own table (¬ß5.2) so multiple receivers/turns never overwrite each other.

### 5.2 `concession_acceptances` ‚Äî the per-concession gradient (F2, F3)

```sql
create table public.concession_acceptances (
  id                  uuid  primary key default gen_random_uuid(),
  debate_id           uuid  not null references public.debates(id)            on delete cascade,
  concession_item_id  uuid  not null references public.concession_items(id)   on delete cascade,
  -- the response argument that carries this acceptance (the receiver's move)
  argument_id         uuid  not null references public.arguments(id)          on delete cascade,
  receiver_id         uuid  not null references public.profiles(id)           on delete restrict,
  acceptance_level    text  not null check (acceptance_level in (
                              'agree',
                              'agree_with_caveat',
                              'disagree_framing',
                              'disagree_context',
                              'disagree_fact'
                            )),
  -- required iff acceptance_level <> 'agree'; enforced in the Edge Function
  -- AND by this CHECK as defense-in-depth.
  clarification_body  text  not null default '',
  created_at          timestamptz not null default now(),
  constraint clarification_required_unless_agree check (
    acceptance_level = 'agree'
    or char_length(trim(clarification_body)) >= 1
  ),
  -- one acceptance per item per receiving argument (a move grades each item once)
  unique (concession_item_id, argument_id)
);
```

- One row per `(concession_item, receiving argument)`. A single
  `respond_to_concession` move that grades B's 2 items writes **2 rows**.
- The `clarification_required_unless_agree` CHECK is doctrine in the schema: a
  disagreement that does not explain itself cannot be stored. (F3.)
- `acceptance_level` is the 5-value enum ‚Äî verbatim the
  interaction-taxonomy vocabulary.
- The point-standing **delta is not stored here.** A future wiring stage reads
  these rows + the `ConcessionEffect` mapping (¬ß4) and computes standing in its
  own table ‚Äî QOL-041 stores only the receiver's stated level + reasoning.

### 5.3 Active-disagreement classification (F4)

The room's "active disagreement is fact-based" metadata (storyboard Step 8,
Step 10 node metadata) is **derived, not a new column.** Derivation is a pure
function over the most recent `concession_acceptances` rows in a room:

```ts
// src/features/concessions/activeDisagreement.ts
type ActiveDisagreementKind = 'framing' | 'context' | 'fact' | 'none';

function deriveActiveDisagreement(
  acceptancesNewestFirst: ConcessionAcceptanceRow[],
): ActiveDisagreementKind;
```

Rule (deterministic, no AI):

1. Look at the acceptances attached to the **most recent** `respond_to_concession`
   argument in the room.
2. If every level is `agree` (or `agree_with_caveat`) ‚Üí `none` (no active
   factual/context/framing disagreement ‚Äî a caveat is a rider, not a dispute).
3. Otherwise pick the **most material** disagreement among the non-`agree`
   levels using the precedence `fact > context > framing` ‚Äî a factual dispute
   is the most concrete open axis. (Precedence is *materiality ordering* for UI
   focus, **not** a truth ranking ‚Äî documented as such in the source comment.)
4. That kind is shown on the node ("Unresolved: fact-based") and in the room
   status strip ("Active disagreement: fact").

Because it is derived, no migration column is needed and historical rooms
re-derive correctly on read. The room status surface (existing) consumes the
derived value; QOL-041 ships the pure model + a read-only hook.

### 5.4 `move_reactions` ‚Äî the fist-bump / acknowledge reaction (F5)

```sql
create table public.move_reactions (
  id            uuid        primary key default gen_random_uuid(),
  debate_id     uuid        not null references public.debates(id)   on delete cascade,
  argument_id   uuid        not null references public.arguments(id) on delete cascade,
  reactor_id    uuid        not null references public.profiles(id)  on delete restrict,
  -- v1 vocabulary is a SINGLE value. No up/down, no score, no vote.
  kind          text        not null default 'fist_bump'
                            check (kind in ('fist_bump')),
  removed_at    timestamptz,
  removed_by    uuid        references public.profiles(id) on delete set null,
  created_at    timestamptz not null default now(),
  -- a participant may fist-bump a given move at most once (toggle = soft-delete + re-insert)
  unique (argument_id, reactor_id, kind)
);
```

- The `kind` CHECK contains exactly **one** value. This is deliberate: it makes
  the table structurally incapable of becoming a voting system (v1 scope bans
  voting). A future "agree"/"disagree" reaction would require a new migration
  *and* a doctrine review ‚Äî the schema does not pre-authorize it.
- No `weight`, no `score`, no count column. A fist-bump count may be *rendered*
  (a small "¬∑3" affordance) but is computed at read time from row count, never
  stored, and never feeds standing.
- Soft-delete (`removed_at`) lets a reaction be toggled off without a hard
  delete ‚Äî consistent with repo doctrine (rows are never hard-deleted).

### 5.5 RLS (all three tables; modeled on `point_tags`)

`concession_items`:

- **INSERT** ‚Äî `auth.uid() = author_id` AND the `argument_id` belongs to
  `debate_id` AND the author is a primary participant on that debate. The
  itemized list is created *with* the response argument; the Edge Function is
  the real gate, RLS is defense-in-depth.
- **SELECT** ‚Äî anyone who can read the parent argument can read its concession
  items.
- **UPDATE** ‚Äî only the original author OR an admin, and only to set
  `removed_at` / `removed_by` (soft-delete). No `body` mutation ‚Äî argument-like
  immutability.
- **DELETE** ‚Äî **no policy.** Soft-delete only.

`concession_acceptances`:

- **INSERT** ‚Äî `auth.uid() = receiver_id` AND `receiver_id` is the author of the
  `conceded_to_argument_id` node behind the `concession_item_id` (only the
  participant the concession was *made to* may grade it ‚Äî interaction-taxonomy
  "Permissions: only the participant who received the concession") AND the CHECK
  constraint enforces the conditional clarification.
- **SELECT** ‚Äî anyone who can read the room.
- **UPDATE / DELETE** ‚Äî **no policy.** An acceptance is an immutable record of
  a posted move. To change a stance, the receiver posts a *new*
  `respond_to_concession` move (a new argument, new acceptance rows) ‚Äî this is
  why ¬ß5.3 derives from the *newest* acceptances.

`move_reactions`:

- **INSERT** ‚Äî `auth.uid() = reactor_id` AND the room is readable by the
  reactor. Observers **may** fist-bump (it is a no-score acknowledgment, not a
  governing move) ‚Äî but this is decided by the Edge Function eligibility table,
  not RLS; see Open Question Q3.
- **SELECT** ‚Äî anyone who can read the room.
- **UPDATE** ‚Äî only `reactor_id` or admin, only to set `removed_at` (toggle
  off).
- **DELETE** ‚Äî **no policy.**

### 5.6 Write path ‚Äî the Edge Functions

The card adds **two** Edge Functions and **extends none of `submit-argument`'s
core math** (it only carries new payload through):

| Function | Purpose |
|---|---|
| **`submit-argument` (extended)** | Already the sole `arguments` write path. Extended to accept an optional `concession_items[]` array on a `respond` move and an optional `concession_acceptances[]` array on a `respond_to_concession` move, and to insert the child rows **in the same transaction as the argument**, after the existing validation passes. If clarification is missing on a non-`agree` acceptance, the function returns a `validation_failed` blocking error ‚Äî a **validation block, never a score**. |
| **`react-to-move`** (new) | The fist-bump write path. JWT-verified, caller-scoped to `reactor_id`, idempotent (re-fist-bump on an already-bumped move is a no-op `ok`; toggling off soft-deletes). Never touches `arguments`, never touches standing, never returns any score. |

Both follow the shipped Edge-Function shape (`submit-argument`,
`request-argument-deletion`, `apply-manual-tag`): `verify_jwt = true`,
`createCallerClient` for authz, `createServiceClient` only for the
authoritative insert after the caller check, no AI / model calls, never logs
`Authorization`.

**Why concession rows go through `submit-argument` and not a third function:**
a concession item has no meaning without its parent response argument ‚Äî they
are born together (storyboard Step 6: one move = list + refutation). Splitting
them into two round-trips would create a window where an `arguments` row exists
with no concession items, which the receiver would see as an empty list.
Inserting them in the same Edge-Function transaction keeps the move atomic.

## 8. Permission / rules implications

| Action | Who may | Enforced where |
|---|---|---|
| Add concession items to a `respond` move | The move's author (a primary participant responding to another's node) | `submit-argument` (the existing argument-author check) + `concession_items` INSERT RLS |
| Set the acceptance gradient on a concession | **Only the participant the concession was made to** ‚Äî the author of `conceded_to_argument_id` (interaction-taxonomy "Permissions") | `submit-argument` (the `respond_to_concession` path checks `receiver_id` = the conceded-to node's author) + `concession_acceptances` INSERT RLS |
| Post a non-`Agree` gradient row without a clarification | **Nobody** ‚Äî it is a validation block | `respondToConcessionModel.isPostable()` (client) + `submit-argument` (authoritative) + the CHECK constraint |
| Fist-bump a move | Any room reader (participant or observer) ‚Äî see Q3 | `react-to-move` eligibility + `move_reactions` INSERT RLS |
| Fist-bump one's **own** move | **No** ‚Äî consistent with QOL-030 own-bubble doctrine (own bubble exposes no reaction/score affordances) | `react-to-move` rejects `reactor_id` = the argument's `author_id` |
| Edit a posted concession item / acceptance | **No** ‚Äî argument-like immutability; soft-delete only on items, no update path on acceptances | RLS (no UPDATE for body; no UPDATE/DELETE on acceptances) |

**Rules-engine implications:** none. The Constitution transition table is not
touched. `concession` is already a valid `argument_type`. The gradient is a
**post-hoc structured response**, layered above the engine ‚Äî it never changes
which transitions are valid. The conditional clarification is a
**submit-argument validation**, in the same family as the existing length /
topic checks ‚Äî it can *block a post*, exactly as doctrine permits validation to
do, and exactly as doctrine forbids *score* from doing.

**RULE-004 (advisory pre-send review) interplay:** RULE-004 fires advisory
warnings on Post. The QOL-041 clarification requirement is **stronger than
advisory ‚Äî it is blocking** ‚Äî but only for the narrow, deterministic,
structural case "you picked Disagree and left the reason empty." That is a
*structural* completeness check (the form is incomplete), not a *semantic*
judgement (RULE-004's territory). The two coexist: RULE-004 may still surface
an advisory note on the clarification's *content*; QOL-041 only blocks on its
*absence*.

## 9. Edge cases

- **Zero concession items.** A `respond` move with an empty concession section
  is valid ‚Äî it is a pure refutation. The forced list is *optional*; only its
  *itemization* is forced when used.
- **Concession item with no later response.** A concession can sit ungraded
  indefinitely. `deriveActiveDisagreement` returns `none` until a
  `respond_to_concession` move exists. An ungraded item is **not** auto-agreed
  and **not** auto-anything ‚Äî silence is never a concession or an acceptance
  (point-standing anti-pattern).
- **Receiver grades only some items.** The `respond_to_concession` box mirrors
  **all** incoming items and requires a level on **every** row before Post ‚Äî
  partial grading cannot be posted. (If a future need for partial grading
  appears, it is a new card; this card requires completeness.)
- **The conceding party edits/removes an item after it was graded.** Items are
  soft-delete-only and never body-editable. A removed item's `concession_items`
  row keeps `removed_at`; any existing `concession_acceptances` row pointing at
  it survives (FK `on delete cascade` only fires on hard delete, which has no
  policy). The receiver's UI shows a removed item as "(point withdrawn)" with
  its prior acceptance intact for the record.
- **Two concurrent `respond_to_concession` moves on the same set.** Each posts
  its own argument + its own acceptance rows; `unique (concession_item_id,
  argument_id)` is per-argument so they do not collide.
  `deriveActiveDisagreement` uses the **newest** argument's acceptances, so the
  later post wins the room-status derivation ‚Äî deterministically, by
  `created_at`.
- **Re-stating the same concession.** A second, identical concession item in a
  later move is just another row. QOL-041 stores it; the **point-standing
  engine's** near-duplicate gate (`isNearDuplicate`) is what prevents *score
  farming* ‚Äî and that engine is not wired in this card, so there is no farmable
  score here at all.
- **Offline / submit failure.** The `respond` and `respond_to_concession`
  drafts live in QOL-030's per-type draft buffers ‚Äî a failed Post parks the
  draft, including every concession item and gradient choice, losslessly. A
  failed `react-to-move` shows a retry affordance; the fist-bump is
  idempotent so a retry is safe.
- **Permission-denied.** A user who is not the conceded-to participant never
  sees the `respond_to_concession` box for that set (the QOL-030 flash menu
  role-gate hides it); if the request is forged, RLS + the Edge Function
  reject it. A user fist-bumping their own move is rejected by `react-to-move`.
- **Empty clarification that is only whitespace.** Trimmed before the length
  check ‚Äî both client and the CHECK constraint use `trim()`.
- **Observer fist-bumps then becomes a participant** (or vice-versa). The
  reaction row is keyed to the user, not the role; it survives a role change. A
  reaction is a no-score acknowledgment, so a role change has no scoring
  consequence to reconcile.
- **`agree_with_caveat` and the active-disagreement derivation.** A caveat is a
  *rider*, not a dispute ‚Äî ¬ß5.3 rule 2 treats an all-`agree`/`agree_with_caveat`
  set as `none`. A caveat does **not** raise a `framing`/`context`/`fact`
  active disagreement; it requires a clarification only so the rider is on the
  record.
- **Doctrine edge ‚Äî could a fist-bump count influence standing?** No. The count
  is read-time-derived and feeds nothing. The anti-amplification doctrine
  (engagement ‚âÝ evidence) is respected structurally: there is no path from a
  reaction row to a `PointStandingDelta`.

## 11. Doctrine & safety self-check

- **A concession is a repair, not a defeat.** No QOL-041 surface labels a
  concession a loss or the receiver a winner. The gradient is the *receiver's
  stance*, mapped to `ConcessionEffect` *vocabulary* only ‚Äî no delta, no score
  is stored or shown by this card (¬ß4). ‚úî
- **Score never blocks posting; validation may.** The only thing QOL-041 blocks
  is a structurally incomplete form (a non-`Agree` row with no clarification) ‚Äî
  a validation block, in the same family as the existing length check. No score
  gates any post. ‚úî
- **No truth labels.** "Disagree based on fact" is the *receiver's stated
  stance toward a point*; the source comment and the ban-list test make
  explicit it is not a ruling that the point is false. `fact > context >
  framing` is materiality ordering for UI focus, documented as not a truth
  rank. ‚úî
- **Popularity / engagement is not evidence.** The fist-bump carries no score
  and feeds nothing. The `move_reactions.kind` CHECK has exactly one value ‚Äî
  the table is structurally incapable of becoming a vote tally. There is no
  path from a reaction to a `PointStandingDelta`. ‚úî (Anti-amplification
  respected by construction.)
- **No AI.** All QOL-041 models are pure deterministic TypeScript. No
  production file calls Anthropic / xAI / OpenAI / X. `deriveActiveDisagreement`
  is a pure function over rows. ‚úî
- **The rules engine is sacred.** `engine.ts` is not modified. No new
  `arguments.argument_type` value. The gradient layers above the Constitution. ‚úî
- **Security.** No service-role in client code. The only `arguments` write path
  is `submit-argument`; the new `react-to-move` function never touches
  `arguments`. RLS enabled on all three new tables; no `DELETE` policy
  anywhere ‚Äî soft-delete only. Edge Functions never log `Authorization`. ‚úî
- **Plain language.** No internal codes in user strings; the 5 levels +
  active-disagreement kinds + fist-bump all have plain-language copy. No
  "game"; "argument" preferred over "debate". ‚úî
- **Accessibility.** The gradient is not color-only (icon + text per segment,
  non-color selection marker); 44px targets; reduce-motion instant; screen
  reader announces every segment and the fist-bump toggle. ‚úî (Confirm full
  contract with `accessibility-targets` at implementation.)
- **v1 scope.** No voting system: the fist-bump is a single-value, no-score
  acknowledgment; the schema cannot host an up/down vote without a new
  migration + doctrine review. No search, no push, no OAuth, no public API. ‚úî

## 15. Open questions

- **Q1 ‚Äî concession item cap.** This design proposes **‚â§8 concession items per
  response** (a forced-list usability ceiling ‚Äî beyond that the mirrored
  gradient list becomes unwieldy, and a response conceding 9+ separate points
  is likely a structure problem better served by branching). Confirm the
  number, or confirm "no cap."
- **Q2 ‚Äî `agree_with_caveat` clarification.** This design makes the
  clarification **required** for `agree_with_caveat` (the caveat must be stated
  to be on the record) and treats it as a *rider*, not a `framing`/`context`/
  `fact` active disagreement (¬ß5.3 rule 2). Confirm: should `agree_with_caveat`
  also be able to *open* an active-disagreement axis, or is "rider only"
  correct?
- **Q3 ‚Äî may observers fist-bump?** This design lets any room *reader*
  (participant or observer) fist-bump, since the reaction carries no score and
  no governing weight ‚Äî it is a human acknowledgment beat. The alternative
  (participants only) is defensible if a fist-bump should feel like a
  *participant* gesture. Confirm the eligibility.
- **Q4 ‚Äî operator deploy ordering.** The migration + two Edge Functions need an
  operator deploy. Should `react-to-move` ship in the same deploy as the
  `submit-argument` extension, or can the fist-bump land in a later deploy
  (the concession list + gradient are the storyboard-critical path; the
  fist-bump is the lower-priority Step 19 affordance)?
- **Q5 ‚Äî count visibility.** Should the fist-bump count be shown at all
  (a tiny "¬∑3"), or should the affordance only show *whether the viewer*
  fist-bumped (binary, no count)? A visible count is the mildest possible
  "popularity" signal ‚Äî showing it is defensible (it is acknowledgment, not a
  vote) but hiding it is the more conservative reading of the
  anti-amplification doctrine.

## 16. Operator steps

After the implementer commits, the operator runs (in order):

1. `npx supabase db push --linked` ‚Äî applies
   `20260521000010_qol_041_concession_acceptance.sql` (the three tables + RLS).
2. `npx supabase functions deploy submit-argument --linked` ‚Äî picks up the
   `concession_items[]` / `concession_acceptances[]` payload handling.
3. `npx supabase functions deploy react-to-move --linked` ‚Äî deploys the new
   fist-bump function (may be deferred ‚Äî see Q4).

No `.env` changes. No secrets. No service-role key handling beyond the existing
Edge-Function pattern. Until the deploy lands, the new box types render but
posts that carry concession arrays would be rejected by the un-extended
deployed `submit-argument` ‚Äî so the deploy must precede enabling the QOL-041
box types in the client.
