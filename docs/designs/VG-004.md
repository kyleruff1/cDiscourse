## 0. Card-vs-reality discrepancies (read this first)

Prior VG/QOL designers found multiple card-vs-reality mismatches. VG-004 has **five**. None of them is a blocker, but each one changes how the implementer must scope the work. The implementer must follow the reconciliations below, not the card body literally.

### D1 ‚Äî There is no separate "node view" component file

The card says: *"Modified: ... node view (or equivalent)"*.

**Reality:** the timeline node is an **internal function component `NodeDot`** declared inside `src/features/arguments/ArgumentTimelineMap.tsx` (lines 139‚Äì213). It is not exported and has no own file. A separate file `src/features/arguments/ArgumentTimelineNode.tsx` exists but is a **different, older Stage 6.1.0 surface** ‚Äî a vertical "compact node card" with a `borderLeftColor`, consumed by `ArgumentTrack.tsx`, *not* by the Timeline/Cards board this card targets.

**Reconciliation:** "node view" = the internal `NodeDot` inside `ArgumentTimelineMap.tsx`. Do **not** touch `ArgumentTimelineNode.tsx` ‚Äî it is out of scope. All node-level polish (glow, halo, receipt mark) lands inside `ArgumentTimelineMap.tsx` plus new pure helpers in a new model file. Do not extract `NodeDot` into its own file in this card ‚Äî that is a refactor with its own risk surface; keep it internal and just enrich it.

### D2 ‚Äî Timeline nodes are circles, not hexagons

The card says item 3 is *"small inner triangle ... on hexagon nodes when `evidenceModel.kind` is non-empty"*.

**Reality:** `NodeDot` renders **every** node as a circle (`styles.node` ‚Üí `borderRadius: TIMELINE_NODE_SIZE / 2`). The board has no hexagon-shaped nodes at all. The `timeline-grammar` skill *does* describe a per-kind shape table (evidence = hexagon), but that table is aspirational ‚Äî the current board encodes kind by **fill color + ordinal text + legend chips + a `kindColorFamily`**, not by polygon shape. Rendering true hexagons in pure RN (no `react-native-svg`) means CSS-border triangle hacks or rotated `<View>` stacks ‚Äî that is a board-wide shape refactor far beyond a "polish" card.

**Reconciliation:** VG-004 does **not** introduce hexagon node shapes. The "evidence receipt chip" (item 3) is reconciled as a **shape-agnostic inner mark on the circular node** ‚Äî a small corner badge composed from `<View>`s ‚Äî shown when the node's message has at least one `EvidenceArtifact` whose `kind` is a real evidence kind. Converting nodes to per-kind polygons is explicitly **Out of scope** (see ¬ß"Out of scope") and should be filed as a separate VG card if wanted.

### D3 ‚Äî There is no "density mode" today

The card says item 6 is *"Density mode tightening: Compact reduces inter-node spacing to 28px, Normal to 44px, Expanded to 64px."*

**Reality:** there is **no density mode** anywhere in the board. Inter-node horizontal spacing is a single frozen constant `TIMELINE_NODE_GAP = 28` in `argumentGameSurfaceModel.ts` (line 668), consumed once as `xStep = TIMELINE_NODE_SIZE + TIMELINE_NODE_GAP` inside `buildArgumentTimelineMap` (line 1206). `BuildTimelineMapInput` has no density field. `ArgumentGameSurface.tsx` has no density state, no density toggle, and never passes density to the map builder. (The `compact` prop on `ArgumentBubbleCard` is unrelated ‚Äî it is a stack-mode body-truncation flag.)

Also note the card's three numbers are internally **contradictory with the current default**: today's gap is already `28`, and the card calls `28` the *Compact* value. The card's "Normal = 44" would actually *loosen* the board vs. today.

**Reconciliation:** VG-004 introduces density mode as a **pure-model layout parameter**, not a new UI control in this card. Specifically:
- Add an optional `density?: TimelineDensityMode` to `BuildTimelineMapInput` (default `'normal'`).
- Add a `TIMELINE_NODE_GAP_BY_DENSITY` lookup. To keep the *current* board pixel-identical by default, `'normal'` maps to **28** (today's value), `'compact'` to **20**, `'expanded'` to **44**. This deliberately diverges from the card's literal numbers, which would silently re-space every existing board.
  - **Open question for the operator (Q1):** the card's literal numbers (Compact 28 / Normal 44 / Expanded 64) would make today's board the "Compact" preset and loosen the default. Confirm whether you want (a) the design's "default stays 28" mapping, or (b) the card's literal mapping with a deliberate one-time loosening of the default board. The design below assumes (a).
- VG-004 ships the *plumbing and the spacing values*. It does **not** ship a density-picker control ‚Äî wiring a visible toggle is a small follow-on (`ArgumentGameSurface` state + a chip) and is called out in ¬ß"Out of scope" so a later card or the implementer-with-extra-time can add it without re-deriving the model.
- "Density preserves active node" (card text): because x-position is purely `LEFT_PAD + ordinal * xStep`, changing `xStep` already preserves *which* node is active (active is an id, not a coordinate). The auto-scroll effect already re-centers on `map.activeNode.x` whenever the map rebuilds. No extra work ‚Äî but the design records it as an explicit invariant + a test.

### D4 ‚Äî Active-path glow and tone overlay already exist on the RAIL

The card lists item 1 (active-path glow) and item 5 (tone overlay) as if new. **Reality:** VG-002 already ships both **on the rail segments**:
- `railSegmentModel.ts` `deriveGlow(isActivePath)` returns a `RailGlowLayer` (shadow + elevation) for active-path edges; `GradientWaveRail.tsx` renders it as "Layer 4 ‚Äî active-path glow".
- `deriveToneWash(toneBand, temperatureBand)` returns the tone overlay; `GradientWaveRail.tsx` renders it as "Layer 1 ‚Äî tone wash". The tone wash is already keyed off `toNode.toneBand` / `temperatureBand`, i.e. **activity, not correctness**.

**Reconciliation:** VG-004's item 1 and item 5 are the **node-level** counterparts the rail does not have:
- Item 1 = an active-path glow on the **node** (`NodeDot`), distinct from the rail-segment glow. Today `NodeDot` only has `nodeRingActive` (a flat ring on the *single active* node). There is no glow on the broader *active path* of nodes.
- Item 5 = applying a tone overlay tint to nodes **on the active path only** ‚Äî the card is explicit: "apply gradient rail tone overlay to active path only, never to claim/standing surfaces." Today nodes carry no tone tint at all; only the rail does.

Both reuse VG-002's existing tokens (`RAIL_ACTIVE_PATH_GLOW`, the tone-band hues) ‚Äî no new color token, consistent with VG-002.

### D5 ‚Äî `ArgumentTimelineNode.tsx` is a false friend

Stated under D1 but repeated for emphasis because the filename collision is exactly the kind of trap prior designers warned about: a glob for "node view" surfaces `ArgumentTimelineNode.tsx`. **Do not modify it.** It is not on the Timeline/Cards board. Modifying it would be wasted work and could destabilize `ArgumentTrack.tsx`.

## Goal (one paragraph)

Make the horizontal argument board (`ArgumentTimelineMap`) read as a *playable map* rather than a flat form, by adding six visual-only refinements: a node-level active-path glow, a selected-node halo, an evidence "receipt" inner mark, branch-stub icon refinement, a tone tint on active-path nodes, and density-aware inter-node spacing. Every refinement is a render-only change driven by pure helpers; none of it touches Supabase, the rules engine, scoring, or any AI path. The doctrine that shapes the design: per `cdiscourse-doctrine` and `timeline-grammar`, **the glow/halo/tint communicate selection and activity ‚Äî never truth or correctness**. The active-path glow says "this is the line you are reading", not "this line is winning". The tone tint says "this branch is active/heated", not "this branch is right". The evidence receipt mark says "an inspectable artifact is attached", not "this claim is proven". Every new state has a non-color signal (geometry, stroke width, or text) and a screen-reader phrase, and every animation is gated on `AccessibilityInfo.isReduceMotionEnabled`.

## Data model

**No new persisted data model. No SQL. No migration. No Edge Function.**

VG-004 adds pure-TypeScript view-model types only. They live in a **new pure model file** `src/features/arguments/timelineNodeVisualModel.ts` (no React, no Supabase, no network ‚Äî testable in isolation per `expo-rn-patterns`).

```ts
// src/features/arguments/timelineNodeVisualModel.ts

/** Density preset for inter-node horizontal spacing. */
export type TimelineDensityMode = 'compact' | 'normal' | 'expanded';

export const ALL_TIMELINE_DENSITY_MODES: ReadonlyArray<TimelineDensityMode> =
  Object.freeze(['compact', 'normal', 'expanded']);

/**
 * Inter-node gap in px per density mode. `normal` === today's
 * TIMELINE_NODE_GAP (28) so the default board is pixel-identical.
 * See design D3 for why these diverge from the card's literal numbers.
 */
export const TIMELINE_NODE_GAP_BY_DENSITY: Readonly<Record<TimelineDensityMode, number>> =
  Object.freeze({ compact: 20, normal: 28, expanded: 44 });

/** Glow tier for a node. `active_node` is the single active node;
 *  `active_path` is any other node on the active path; `none` otherwise. */
export type NodeGlowTier = 'none' | 'active_path' | 'active_node';

/** The visual treatment block for one timeline node. Pure-derived. */
export interface TimelineNodeVisualStyle {
  /** Outer glow tier. */
  glowTier: NodeGlowTier;
  /** Outer stroke width in px for the glow ring (reduce-motion still draws this). */
  glowStrokeWidthPx: number;     // 0 | 2  (2px outer stroke per card item 1)
  /** Soft shadow radius in px. 0 when reduce-motion is on (stroke-only fallback). */
  glowShadowRadiusPx: number;    // 0 | 12
  /** Halo ring width in px for the *selected* node. 0 when not selected. */
  haloRingWidthPx: number;       // 0 | 3   (3px ring per card item 2)
  /** True when this node carries the evidence receipt inner mark. */
  showsReceiptMark: boolean;
  /** Tone tint applied to active-path nodes only. null = no tint. */
  toneTint: { color: string; alpha: number } | null;
  /** Plain-English a11y fragment describing the added states.
   *  Appended to the node's existing accessibilityLabel. Never verdict copy. */
  accessibilityFragment: string;
}

export interface DeriveTimelineNodeVisualInput {
  isActive: boolean;            // node.isActive
  isActivePath: boolean;        // node.isActivePath
  isSelected: boolean;          // node.messageId === selectedTarget?.messageId (SC-004 dock target)
  toneBand: TimelineToneBand;   // node.toneBand
  temperatureBand: TimelineTemperatureBand;
  hasEvidenceArtifact: boolean; // artifactsByMessageId[id] has >= 1 real evidence-kind artifact
  prefersReducedMotion: boolean;
}
```

Two public functions in the same file:

```ts
export function deriveTimelineNodeVisualStyle(
  input: DeriveTimelineNodeVisualInput,
): TimelineNodeVisualStyle;

export function resolveNodeGapPx(density: TimelineDensityMode | undefined): number;
```

No new field is added to `ArgumentTimelineMapNode` or `ArgumentTimelineMapModel` ‚Äî the visual style is derived at render time from fields that already exist (`isActive`, `isActivePath`, `toneBand`, `temperatureBand`) plus the `artifactsByMessageId` map the component already receives and the `selectedTarget` prop it already receives. This mirrors VG-002's "surface lock" discipline: no model-shape churn.

`BuildTimelineMapInput` gains exactly one optional field: `density?: TimelineDensityMode`.

## API / interface contracts

### `timelineNodeVisualModel.ts`

```ts
/** Inter-node gap for a density mode. Undefined ‚Üí 'normal' (28px, today's value). */
export function resolveNodeGapPx(density: TimelineDensityMode | undefined): number;
// resolveNodeGapPx(undefined)  === 28
// resolveNodeGapPx('normal')   === 28
// resolveNodeGapPx('compact')  === 20
// resolveNodeGapPx('expanded') === 44

/**
 * Derive the full visual treatment for one node. Pure. Deterministic.
 * Same input ‚Üí deep-equal output (so the renderer may memoize by node id).
 *
 *  - glowTier: 'active_node' if input.isActive; else 'active_path' if
 *    input.isActivePath; else 'none'.
 *  - glowStrokeWidthPx: 2 when glowTier !== 'none', else 0.
 *  - glowShadowRadiusPx: 0 if prefersReducedMotion (stroke-only fallback,
 *    card item 1), else 12 when glowTier !== 'none', else 0.

 *    VG-002 tone-band hue at a capped low alpha (<= 0.18) ‚Äî activity, never

 *    "active path only, never to claim/standing surfaces").

 *    "selected", "has an attached source". Empty string when nothing applies.
 *    Never contains verdict / amplification / snake_case tokens.

  input: DeriveTimelineNodeVisualInput,
```

## Edge cases

The implementer must handle each of these:

1. **Empty timeline** ‚Äî `map.nodes.length === 0` already short-circuits to the empty state before any `NodeDot` renders. No new code path needed, but the test must assert `deriveTimelineNodeVisualStyle` is never called with a malformed node (it is a pure helper ‚Äî feed it a node-less fixture is N/A; just assert the empty board test still passes).
2. **Single-node timeline (root only)** ‚Äî root is active, on the active path, possibly selected. `glowTier` must be `active_node`, not double-counted. Receipt mark only if the root has an artifact. Verify the root marker pill + first-clash pill + glow + halo do not visually collide (they stack at different `marginTop` / absolute offsets ‚Äî the glow is `position:absolute` behind the node; the halo wraps it; the pills are below).
3. **Node is BOTH `isActive` AND `isSelected`** ‚Äî the active node can also be the SC-004 dock target. Glow tier = `active_node` AND halo width = 3. The two are different `<View>` layers (glow behind, halo around) and must both render. The `accessibilityFragment` must list both ("active move, selected") without contradiction.
4. **Detached node** ‚Äî `node.isDetached` nodes are never on the active path (the model guarantees `isActivePath === false` for detached). So `toneTint` is `null` and `glowTier` is `none` for detached nodes. The dashed detached border (`styles.nodeDetached`) is preserved ‚Äî the receipt mark can still show if a detached node has an artifact (evidence can be attached to a detached move). Assert this.
5. **Reduce-motion ON** ‚Äî `glowShadowRadiusPx` is `0`; the **2px outer stroke is still drawn** (card item 1: "Reduced-motion: stroke only"). The halo (`haloRingWidthPx`) is a static border, not motion ‚Äî it is kept on under reduce-motion. The tone tint is a static color, not motion ‚Äî kept. Receipt mark is static ‚Äî kept. Only the soft drop shadow is removed.
6. **`AccessibilityInfo.isReduceMotionEnabled()` rejects / is unavailable** (jest, web shim) ‚Äî wrap in `try/catch`, default `prefersReducedMotion = false` BUT also default `glowShadowRadiusPx` resolution so a throw never leaves the board un-rendered. Mirror the existing `try/catch` around `announceForAccessibility` (lines 317‚Äì323).
7. **`artifactsByMessageId` is `undefined`** ‚Äî the prop is already optional on `ArgumentTimelineMap`. `hasEvidenceArtifact` must default to `false`. Receipt mark simply does not render.
8. **Artifact list present but every artifact is a non-evidence kind** ‚Äî `EvidenceArtifactKind` is a closed union of six real evidence kinds (`url`, `quote`, `source_text`, `dataset`, `screenshot_redacted`, `manual_citation`); popularity/identity are deliberately NOT kinds. The receipt mark shows when the list is non-empty (all six kinds are legitimate evidence). The design does **not** invent a kind filter ‚Äî every `EvidenceArtifact` already passed EV-001's classifier. Test asserts an empty list ‚Üí no mark, a one-artifact list ‚Üí mark.
9. **High node count (250+)** ‚Äî the rail is already virtualized (`visibleSegmentSlice`); **nodes are not virtualized** today (`map.nodes.map(...)` renders all). VG-004 must not regress this ‚Äî the per-node visual is a small fixed set of extra `<View>`s. Receipt mark is one `<View>` + one inner `<View>`; glow is one `<View>`; halo is one `<View>`; tone tint is one `<View>` (and only on active-path nodes, a small subset). Worst case ~4 extra `<View>`s per node. This is acceptable for v1; if profiling later shows a problem, node virtualization is a separate card. Note this in ¬ß"Risks".
10. **Selected target is a non-node target** (`selectedTarget.kind === 'cluster' | 'collapsed_stub'`) ‚Äî `isSelected` for a node is only true when `selectedTarget?.kind === 'node' && selectedTarget.messageId === node.messageId`. For cluster/stub targets, no node shows the halo. Assert this in the test.
11. **Density change mid-session** ‚Äî if a future toggle changes `density`, `buildArgumentTimelineMap` re-runs, every node `x` shifts, the auto-scroll effect re-centers on `map.activeNode.x`. The active *node* (an id) is unchanged. Test `resolveNodeGapPx` determinism and assert that two maps built from the same messages with different density produce the same node **ids** in the same **ordinal order** (only `x` differs).
12. **Doctrine edge case ‚Äî "what if heat tries to influence the halo / glow?"** ‚Äî it does not. The halo is driven *only* by `isSelected` (a UI selection state). The glow is driven *only* by `isActive` / `isActivePath` (navigation state). Neither reads `temperatureBand`, `standingBand`, `heat`, or any score. The **tone tint** is the only state that reads `toneBand`/`temperatureBand`, and it is capped at alpha ‚â§ 0.18, applied to active-path nodes only, and explicitly framed as activity. A strong-band node and a needs-work node on the active path get the *same* glow ‚Äî the glow never encodes strength. Test asserts: two nodes identical except `standingBand` produce identical `glowTier` / `glowStrokeWidthPx` / `haloRingWidthPx`.
13. **Doctrine edge case ‚Äî "can the tone tint reach a claim/standing surface?"** ‚Äî no. `toneTint` is `null` for any node not on the active path, and the tint `<View>` only renders inside `NodeDot` (the timeline node), never on `ArgumentBubbleCard`, the sidecar, or `ArgumentScoreTracker`. Card item 5 is explicit and the test enforces `toneTint === null` when `isActivePath === false`.

## Test plan

New file: `__tests__/argumentNodeVisualVG004.test.ts` ‚Äî a **pure / style-rule test suite**, consistent with `BranchCollapseStub.test.tsx` and `railSegmentModel.test.ts` (this repo's `.test.tsx`/`.test.ts` files do **not** call React-Testing-Library `render()` for these surfaces ‚Äî they import pure helpers and assert returned objects, plus they source-scan component files for invariants). All new visual logic is extracted into pure helpers precisely so it is unit-testable without a renderer.

Estimated **~34 new tests** across these groups:

1. **`resolveNodeGapPx` (4 tests)** ‚Äî `undefined ‚Üí 28`; `normal ‚Üí 28`; `compact ‚Üí 20`; `expanded ‚Üí 44`. Plus an exhaustive loop over `ALL_TIMELINE_DENSITY_MODES` asserting every mode resolves to a positive integer.
2. **`deriveTimelineNodeVisualStyle` ‚Äî glow (6 tests)** ‚Äî active node ‚Üí `glowTier === 'active_node'`, `glowStrokeWidthPx === 2`; active-path-not-active node ‚Üí `'active_path'`, stroke 2; off-path node ‚Üí `'none'`, stroke 0; reduce-motion ON ‚Üí `glowShadowRadiusPx === 0` while `glowStrokeWidthPx` stays 2 (stroke-only fallback); reduce-motion OFF ‚Üí `glowShadowRadiusPx === 12`; off-path node has shadow 0 regardless of reduce-motion.
3. **`deriveTimelineNodeVisualStyle` ‚Äî halo (4 tests)** ‚Äî selected ‚Üí `haloRingWidthPx === 3`; not selected ‚Üí `0`; active AND selected ‚Üí glow `active_node` AND halo `3` co-exist; reduce-motion does not change halo width (halo is a static stroke, not motion).

## Risks

1. **`react-native-svg` is NOT in `package.json`.** Confirmed: `dependencies` has no `react-native-svg` (it appears only in `jest.transformIgnorePatterns` as a defensive entry, not as an installed package). The card says STOP-and-convert-to-discovery-card if a dep is needed. **No dep is needed** ‚Äî every VG-004 visual is achievable with `<View>` borders, `shadow*`/`elevation` props, and absolute-positioned `<View>`s. The receipt inner mark is a small rotated/clipped `<View>` composition (a square `<View>` with `borderRadius`, or a CSS-border triangle via `borderWidth` tricks on a zero-size `<View>`). The implementer must NOT install anything.
2. **Halo vs. existing `nodeRingActive` / `nodeRingLatest`.** `NodeDot` already draws a ring on the active and latest nodes (`styles.nodeRing*`). The VG-004 *selected* halo is a different concept (selection ‚âÝ active ‚âÝ latest). Risk: stacking three rings on one node looks noisy. Mitigation in the design: the halo is the *outermost* layer with a distinct cream color (`BRAND.accent.cream`) vs. the indigo active ring; it only appears on the SC-004 dock target. The implementer must visually verify the active+latest+selected triple-stack on one node (edge case #2/#3) and, if it reads as clutter, suppress the legacy `nodeRingActive` when the new glow covers the same node (the glow already signals "active"). Flag for the implementer to make a judgment call and document it.
3. **`RailStubViewModel` has no `branchKind` field today.** Item 4 wants a "branchKind icon". `RailStubViewModel` (in `branchTopologyModel.ts`) carries `borderColor`, `containsActive`, `hiddenMessageCount`, `label`, `accessibilityLabel` ‚Äî no branch-kind. The design deliberately does **not** add a field to `branchTopologyModel.ts` (that pure model is a BR-001 surface and adding a field risks its tests). Instead `buildBranchCollapseStubLabelParts` derives a single generic side-branch glyph. If a *kind-specific* glyph (tangent vs. evidence-thread vs. kink) is wanted, that needs a `RailStubViewModel` field and becomes a small BR-001-adjacent change ‚Äî out of scope here; noted as a possible follow-on.
4. **Nodes are not virtualized.** `map.nodes.map(...)` renders every node (only the *rail* is virtualized). VG-004 adds ~4 `<View>`s per node. On a 250-node room this is ~1000 extra `<View>`s. This is within RN's tolerance for a horizontally-scrolled board and matches the existing per-node `<View>` count (each node already renders 5‚Äì9 `<View>`s for ring/pills/chips). No regression vs. today's order of magnitude, but if a future profiling card finds the board slow, node virtualization is the fix ‚Äî not a VG-004 concern.
5. **Existing tests that may need a touch.** `__tests__/argumentTimelineMap.test.ts` and `__tests__/railSegmentModel.test.ts` exercise the map builder. Adding an optional `density` field to `BuildTimelineMapInput` is non-breaking (default `'normal'` === 28 === today). The implementer must run the full suite; expectation is **zero existing-test edits**. If any snapshot or layout assertion in `argumentTimelineMap.test.ts` hard-codes an `x` value, it stays correct because the default gap is unchanged ‚Äî but the implementer must confirm.
6. **Reduce-motion is read async.** `AccessibilityInfo.isReduceMotionEnabled()` returns a Promise. The first render will have `prefersReducedMotion = false` until the effect resolves; this is acceptable (a one-frame soft-shadow flash before settling). To be safe, also subscribe to the `reduceMotionChanged` event so a mid-session OS change is honored, and clean the listener up on unmount.
7. **No migration / operator step.** Pure UI. If the implementer accidentally reaches for a Supabase call or an Edge Function, that is a doctrine violation ‚Äî there is none required.
