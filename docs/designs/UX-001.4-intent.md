**Effort:** L

## Why this brief exists

UX-001.1 made the brand shell professional, UX-001.2 made the Timeline the primary board, and UX-001.3 made the composer feel like one persistent contextual command center. The three menus that govern in-room interaction — Act, Inspect, Go — already exist as separate popouts (`src/features/arguments/oneBox/ActPopout.tsx`, `InspectPopout.tsx`, `GoPopout.tsx`), but their relationship to each other and to the broader interaction surface is not yet coherent.

The user looking at the room should see one consistent three-affordance model: Act executes a move on the selected node, Inspect explains what the selected node means, Go moves around the board. Today these affordances are split across overlapping surfaces — the Act popout, the `ArgumentSideActionRail`, the `TimelineNodeActionDock`, and the observer dock layout all expose action entries; the Inspect popout overlaps with the `TimelineSelectedReadoutPanel` and the `ArgumentReplySidecar`; the Go popout overlaps with what the dissolved roomToolbar used to surface (per UX-001.2's chrome consolidation). The user does not yet feel that these are one three-menu model.

UX-001.4 consolidates the three menus into one coherent interaction model. The card does NOT replace Act/Inspect/Go (per the epic's non-goals); it removes the duplication, establishes consistent key badges for browser, consistent bottom sheets for touch, and consistent disabled-with-reason patterns across all three.

## What UX-001.4 consumes from prior cards (fixed, not relitigated)

UX-001.1's Phase 1 framing (`docs/core/current-status.md` line 25+) establishes the breakpoints (phone 0-599dp, tablet 600-1279dp, wide 1280dp+), header heights (64/96/120), logo heights (44/80/96). UX-001.4 does not modify these.

UX-001.2's Phase 2 framing (line 159+) establishes the compact room/context strip, hidden tab bar in room-active mode, dissolved roomToolbar, repositioned readout below or beside the Timeline, repositioned score tracker, Timeline first-row offset hard caps (128/168/200), and the QOL-040.3 microMoment dismiss contract. UX-001.4 does not modify these and must not push the Timeline back above its offset cap.

UX-001.3's Phase 3 framing (line 346+) establishes the composer's 14 canonical modes (root claim, standard response, challenge/rebuttal, branch/tangent, synthesize, evidence submission, respond to evidence, ask source, ask quote, clarify, narrow, confirm, concession list/offer_concession, respond to concession), the persistent collapsed strip below the score tracker, the target display contract (heights 48-64/56-72/64-80 per breakpoint), the validation surface pattern (inline chips, header chips, disabled-with-reason; no modals/banners/toasts), the keyboard shortcut allocations (Cmd/Ctrl+Enter submit, Cmd/Ctrl+K mode switcher, Esc collapse-then-dismiss), the touch ergonomics (44x44, bottom sheet, swipe-to-dismiss), and the "Acting on" coordination model (composer reads `activeMessageId` one-way via `composerActingOnModel`).

UX-001.4 consumes these contracts. Act opens the composer in the relevant mode per UX-001.3's mode list. Inspect's expanded detail shows what the composer's compact target display references in shortened form. Go provides navigation that integrates with UX-001.2's hidden tab bar (Go provides in-room navigation; the tab bar provides out-of-room navigation when room is exited).

## The central product rule

Act, Inspect, and Go are the three menus of in-room interaction. Each menu answers exactly one user question:

- **Act answers: "What can I do here?"** — and executing an entry opens the composer in the relevant mode (per UX-001.3) or fires a direct action (governance reactions, role changes, request deletion).
- **Inspect answers: "What am I looking at?"** — expanded context about the selected node: full body, evidence chain, metadata, lifecycle state, classifier outputs (Observations), user annotations (Allegations once UX-001.5A ships).
- **Go answers: "Where do I want to be?"** — navigation within the board (jump-to-root, jump-to-latest, branch navigation, room exit) plus the room-exit affordance that the hidden tab bar requires.

The three menus must NOT bleed into each other. Act does not show inspection content. Inspect does not surface composer-opening entries. Go does not execute moves. The user's question is answered by exactly one menu.

The three menus must feel like one system. They share chrome (same trigger style, same popout/bottom-sheet pattern, same dismiss behavior), share keyboard shortcut conventions (A/I/G plus Esc), and share validation patterns (disabled-with-reason copy where an entry is unavailable).

## The duplicate rail removal contract

The pre-launch codebase survey identified overlapping action surfaces:
- `src/features/arguments/oneBox/ActPopout.tsx` (the canonical Act surface)
- `src/features/arguments/ArgumentSideActionRail.tsx` (the side action rail; observer/participant variants)
- `src/features/arguments/TimelineNodeActionDock.tsx` (the timeline node action dock)
- `src/features/arguments/ObserverActionDockLayout.ts` (observer dock layout model)

UX-001.4 designates `ActPopout` as the canonical Act surface. Entries that currently render in `ArgumentSideActionRail` or `TimelineNodeActionDock` and overlap with Act are either:
- **Migrated into Act** if the entry is a move (open composer, fire direct action, change role) — the rail/dock loses the entry
- **Preserved as a quick affordance** if the entry is a frequently-used shortcut (e.g., reply to the active message) — the rail/dock keeps the entry but the entry's behavior reduces to "open Act with this preset" rather than executing inline

The pre-launch source audit (the designer's first deliverable) maps each entry currently in `ArgumentSideActionRail` and `TimelineNodeActionDock` against the Act popout's entries to determine which are migrated and which are preserved as shortcuts. The audit produces a table with disposition per entry.

Inspect and Go do not currently have duplicate rails. Their consolidation is structural (consistent chrome, consistent dismiss, consistent keyboard) rather than de-duplication.

## The Act contract

Act surfaces three entry kinds per the existing `actPopoutModel`:
- `box_opening` — selecting opens the composer in the relevant `BoxType`
- `direct` — direct action (governance reactions, make-private, Touché, fist-bump, request deletion)
- `role_change` — Watch / Join / Chime in

Act respects the existing three-gate model (engine gate hard, role gate hard, stage gate soft — promotes but never removes). UX-001.4 does not change the gates.

Act's presentation:
- **Phone**: bottom sheet (slides up from below; swipe-to-dismiss; tap-outside-to-close; Esc closes)
- **Tablet**: bottom sheet OR anchored popout (designer chooses with rationale)
- **Wide**: anchored popout near the trigger (a small panel; not full-width)

Act's trigger:
- Default: a single `Act` button on the selected node (or on the strip when a node is selected)
- Keyboard: `A` when the board has focus (board-vs-composer focus context per UX-001.3 — A does NOT fire when composer is focused)
- Touch: tap the trigger or long-press the node

Act's entries display:
- Engine + role gating already removes invalid entries
- Stage-promoted entry is visually emphasized (existing `actPopoutModel` behavior preserved)
- Disabled-with-reason: if an entry would be removed by a future role change but is currently allowed, do NOT disable it; if an entry is unavailable for a reason the user should know (e.g., "Synthesis ready after 2 more challenges"), render the entry disabled with the reason as a tooltip / inline helper

Act's "Acting on" line:
- When Act is open, the popout header names the target (per UX-001.3's `composerActingOnModel` — a compact target excerpt with parent/author hint)
- When the user executes a `box_opening` entry, the composer opens with the same target — no re-selection required

## The Inspect contract

Inspect surfaces expanded context about the currently selected node. Inspect is read-only — it shows; it does not execute.

Inspect's content (per `inspectPopoutModel`):
- Full body of the selected node
- Parent / branch context (lineage)
- Metadata (lifecycle state, classifier outputs as Observations, user annotations as Allegations once UX-001.5A ships)
- Evidence chain (linked evidence items with their applicability)
- Role / authorship hints

Inspect's presentation:
- **Phone**: bottom sheet (max 50% viewport height; scrolls if content exceeds)
- **Tablet**: bottom sheet OR side sheet (designer chooses with rationale)
- **Wide**: anchored panel near the trigger (max 35% viewport width)

Inspect's trigger:
- Default: a single `Inspect` button on the selected node (or on the strip)
- Keyboard: `I` when the board has focus
- Touch: tap the trigger

Inspect's relationship to the composer's compact target display:
- The composer's `ComposerContextStrip` (UX-001.3) shows a compact excerpt (kind label, body excerpt, parent hint, "Acting on:")
- Inspect shows the expanded version of the same target (full body, evidence chain, metadata, lifecycle, classifier outputs)
- The user can transition Inspect → Act on the same target via a single trigger in Inspect's footer (an "Act on this node" button that closes Inspect and opens Act)

Inspect's relationship to the repositioned `TimelineSelectedReadoutPanel` (UX-001.2):
- The readout shows a compact 5-line summary (UX-001.2's contract; read-only)
- Inspect shows the expanded version
- The readout's `Show full details ▾` expander already shows the existing sidecar; UX-001.4's Inspect popout is the new canonical expanded view — the brief specifies whether to coordinate with the readout's expander (open Inspect from it) or keep them separate; the designer chooses with rationale, bounded by zero modification of the readout's internals

## The Go contract

Go surfaces navigation within the board:
- Jump to root (board origin)
- Jump to latest message
- Jump to a specific branch (branch picker)
- Jump to a specific lane (if applicable)
- Leave room (returns to the room list; restores the hidden tab bar from UX-001.2)
- Search within the room (if the implementation already supports it via the existing per-room search infrastructure — the v1 scope guard "No argument search" still applies for cross-room search)

Go's presentation:
- **Phone**: bottom sheet
- **Tablet**: bottom sheet OR side sheet
- **Wide**: anchored panel

Go's trigger:
- Default: a `Go` button on the strip (the room's persistent chrome)
- Keyboard: `G` when the board has focus
- Touch: tap the trigger

Go's integration with the hidden tab bar (UX-001.2):
- The hidden tab bar restores on room exit (UX-001.2's contract; read-only)
- Go's "Leave room" entry triggers the same `handleLeaveRoom` path that UX-001.2 already wired
- The brief does NOT introduce a new room-exit path

## Position and height contracts per breakpoint

Each menu's presentation respects these height bounds; the menu's trigger does NOT add chrome above the Timeline beyond what UX-001.2 already accounts for.

**Phone (390x844 baseline)**:
- Act bottom sheet: ≤ 50% viewport height (~422 px)
- Inspect bottom sheet: ≤ 50% viewport height
- Go bottom sheet: ≤ 40% viewport height (~338 px — Go is typically shorter)
- Trigger row (if on the strip): ≤ 8 px additional chrome (the strip is already 48 px per UX-001.2; triggers fit within or extend by ≤ 8 px)

**Tablet (1024x1366 baseline)**:
- Each menu's expanded surface: ≤ 40% viewport height (or ≤ 40% width for side sheet)
- Trigger row: ≤ 8 px additional chrome

**Wide (1366x768 + 1920x1080 baselines)**:
- Each menu's anchored panel: ≤ 35% viewport height/width
- Trigger row: ≤ 8 px additional chrome

The expanded menus overlay the Timeline (or push downward); they do NOT displace the Timeline upward. The brief MANDATES that UX-001.2's Timeline first-row offset caps (128/168/200) remain satisfied after UX-001.4's changes.

## Validation surface

None expected. Act executes actions which may surface validation through the composer per UX-001.3's pattern. Inspect does not validate (it shows). Go does not validate (it navigates).

If an Act entry is disabled (e.g., "Synthesis ready after 2 more challenges"), the disabled-with-reason pattern from UX-001.3 applies: render the entry dimmed with a tooltip / inline helper naming the reason. The reason copy must NOT contain verdict tokens, raw internal codes, or accusatory language. Use `gameCopy.toPlainLanguage` for any internal-code translation.

## Action path

Read-only with respect to existing submit paths, action permissions, and role gating. Act opens the composer; the composer submits via the existing `submit-argument` Edge Function per UX-001.3's wiring. Direct actions (governance reactions, role changes, request deletion) route through their existing API surfaces. UX-001.4 does NOT introduce a new write path, a new permission model, or new role gating.

The existing `actPopoutModel`'s engine + role + stage gates are preserved verbatim. The existing role permission model (observer / participant / own-bubble) is preserved verbatim.

## Keyboard and shortcut contract

The board-focus keyboard shortcuts:
- `A` opens Act with the currently selected node as the target
- `I` opens Inspect with the currently selected node as the target
- `G` opens Go (board-level, no specific target)
- `Esc` closes the open menu (or, if no menu is open, the user is in board focus and Esc has no effect)

These shortcuts fire only when the board has focus. When the composer is focused (per UX-001.3's `useComposerFocusContext`), A/I/G do NOT fire — the keystroke is consumed by the composer's text input.

Existing board shortcuts (arrows for node navigation, etc.) are preserved verbatim.

Each menu's expanded surface supports its own keyboard:
- Arrow keys move focus among entries
- Enter/Space activates the focused entry
- Esc closes the menu and returns focus to the trigger

**Key badges for browser**: each Act/Inspect/Go trigger displays a small key badge (e.g., `A`, `I`, `G`) when the browser viewport is active. Key badges do NOT appear on phone or tablet (no keyboard expected).

## Touch ergonomics

Touch targets meet 44x44 minimum on all interactive Act/Inspect/Go elements (triggers, entries, bottom-sheet dismiss affordances).

Bottom sheets on mobile:
- Slide up from below
- Swipe-down to dismiss (or operator-accepted fallback if the existing inert-scrim pattern is preserved — see UX-001.3's deferred PanResponder note in `docs/reviews/UX-001.3-review.md`; UX-001.4 may inherit the same pragmatic deferral)
- Tap-outside-to-close (subject to the same scrim-preservation note)
- Esc closes

Haptic feedback (if available without dependency burden):
- Light haptic on menu open
- Light haptic on entry select
- Use the existing `composerHaptics.ts` no-op shim from UX-001.3 (the same API; do NOT introduce `expo-haptics` as a dependency)

## Cross-device parity

Mobile and web are equally required. The three menus must work cleanly on phone, tablet, and browser at all four required viewports (390x844, 1024x1366, 1366x768, 1920x1080).

Failure conditions:
- Do not ship UX-001.4 if the Act bottom sheet feels janky on phone but the popout works on wide
- Do not ship UX-001.4 if the key badges render at viewports that don't have keyboards
- Do not ship UX-001.4 if the duplicate rail removal regresses any existing action (every removed rail entry must have an Act entry that does the same thing)
- Do not ship UX-001.4 if any menu's expanded state pushes the Timeline above UX-001.2's offset cap

## Acceptance criteria

UX-001.4 ships when all of these hold:

1. Act, Inspect, and Go each answer exactly one user question and do not bleed into each other.
2. The three menus share chrome, dismiss behavior, and keyboard shortcut conventions.
3. Act's content is governed by the existing 3-gate `actPopoutModel` (engine hard, role hard, stage soft) without modification to the gates.
4. Act's entries open the composer (UX-001.3) in the correct mode for `box_opening` entries; `direct` and `role_change` entries fire via existing API surfaces.
5. The Act "Acting on" line names the same target the composer's `composerActingOnModel` would name.
6. Inspect surfaces expanded context (full body, evidence chain, metadata, lifecycle, classifier outputs) for the selected node.
7. Inspect does not modify state.
8. Inspect has a footer transition to Act on the same node ("Act on this node" trigger).
9. Inspect coordinates with the UX-001.2 readout's `Show full details ▾` expander OR keeps them separate, with the designer's rationale recorded.
10. Go surfaces in-room navigation (jump-to-root, jump-to-latest, branch picker, leave room) without introducing a new room-exit path.
11. The duplicate rail removal audit is complete — every entry currently in `ArgumentSideActionRail` and `TimelineNodeActionDock` is dispositioned (migrated to Act, preserved as a quick affordance with Act-redirect behavior, or explicitly retained with rationale).
12. The board-focus keyboard shortcuts A/I/G fire when the board has focus and do NOT fire when the composer is focused.
13. Each menu's expanded state respects the per-breakpoint height bounds.
14. Each menu's expanded state does NOT push the Timeline above UX-001.2's offset caps (128/168/200).
15. Mobile bottom sheets meet 44x44 touch targets, support swipe-to-dismiss (or the inert-scrim fallback per UX-001.3), and support Esc-to-close.
16. Key badges (`A`, `I`, `G`) render on browser viewports and do NOT render on phone/tablet.
17. Disabled-with-reason patterns use plain-language copy (no verdict tokens, no raw internal codes).
18. The submit path is unchanged.
19. No migration.
20. No Edge Function.
21. No new write path.
22. No new permission model.
23. No raw internal codes in user-facing copy.
24. No verdict, truth, winner, loser, proof, or correctness language.
25. The Phase 4 framing section is added to `docs/core/current-status.md` with the Inspect surface contract as the most critical input for UX-001.5.

## The designer's framework

Translation questions, not invention questions. The brief specifies product intent; the designer translates intent into implementation specifications.

Q1 — Source audit + duplicate rail audit (the brief's first deliverable).
Q2 — Three-menu consolidation (chrome shared, dismiss shared, keyboard shared).
Q3 — Act implementation (presentation per breakpoint, trigger, entries, "Acting on" line, transitions to composer).
Q4 — Inspect implementation (content, presentation per breakpoint, footer transition to Act, coordination with UX-001.2 readout).
Q5 — Go implementation (entries, presentation per breakpoint, integration with hidden tab bar).
Q6 — Duplicate rail removal implementation (the audit sub-table's dispositions translated into code).
Q7 — Keyboard implementation (A/I/G board-focus routing; focus context coexistence with composer's Cmd/Ctrl shortcuts; Esc routing per menu).
Q8 — Key badges implementation (browser-only rendering; absent on phone/tablet).
Q9 — Bottom sheet implementation (44x44, swipe-to-dismiss with the UX-001.3 fallback note, Esc).
Q10 — Disabled-with-reason implementation (plain-language copy; no verdicts; no raw internal codes).
Q11 — Cross-viewport verification (Act/Inspect/Go expanded states do not push Timeline above caps at any of 390x844, 1024x1366, 1366x768, 1920x1080).
Q12 — Regression preservation (submit path, engine/role/stage gates, action permissions, role gating, semantic doctrine, no-route-transition).
Q13 — Reviewer measurement strategy.

If any question surfaces a gap that requires operator judgment, the designer HALTS.
