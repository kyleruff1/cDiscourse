# SC-005 — Collapsed observer action dock and contextual pop-outs

**Status:** Design draft
**Epic:** Epic 4 — Sidecars, Popovers, Quick Tools
**Release:** Stage 6.6 board-interaction wave (follows SC-001 / SC-002 / SC-004)
**Issue:** https://github.com/kyleruff1/cDiscourse/issues/112

## Goal (one paragraph)

The argument room currently renders the observer action set as a flex-wrap
grid of full-width buttons (`ArgumentSideActionRail.tsx`, `actionButton`
style: `minWidth: 120`, `flexBasis: '30%'`, `maxWidth: '49%'`, helper text
on every button), and *also* renders a separate bottom `actionBar` in
`App.tsx` (lines 439–457) for "Start an argument". Together they consume
~30% of vertical space on a phone and push the active timeline node
off-screen. SC-005 keeps **the exact Stage 6.4 action-set definitions**
(observer set, participant-other set, own-bubble set) and **the SC-001
category taxonomy** unchanged, and only changes *layout*: a collapsed
contextual dock (pill chip on narrow, bottom-right anchor on wide) that
expands into a compact dock — side-anchored on wide viewports, a capped
bottom sheet (~28% of viewport, never full-screen) on narrow ones — with
short labels only, helper text demoted to `accessibilityHint` plus
long-press, category headers shown only when ≥ 2 categories are non-empty,
a context-aware collapsed primary label, mutual exclusion with the SC-002
node popover, and the full accessibility contract (44×44 targets, Tab/Esc/
Enter keyboard nav on web, reduce-motion fallback = no slide). This is a
doctrine-neutral layout card: no verdict copy, no internal codes in labels,
no new dependency, no migration, no AI call.

## Data model

**No new persistent data model. No migration. No new types in `src/lib/types.ts`.**

One **new pure-TS module** holds layout/derivation logic so it is unit
testable without a render harness. All types below live in
`src/features/arguments/ObserverActionDockLayout.ts`:

```ts
/** Which physical layout the dock takes for the current viewport. */
export type DockLayoutVariant = 'sheet' | 'side';
//  'sheet' — narrow: bottom sheet, capped height.
//  'side'  — wide:   anchored panel at bottom-right next to the board.
//  (Same string union name as ArgumentComposerDock.DockLayoutVariant by
//   intent — see "Risks" for why we do NOT share the symbol.)

/** Collapsed/expanded state of the dock. Local UI state, never persisted. */
export type DockExpansionState = 'collapsed' | 'expanded';

/**
 * The four-way "what is the user looking at" context that drives the
 * collapsed-state primary label. Derived from existing inputs only:
 * viewerRole (RailViewerRole) + bubbleActor (RailBubbleActor) +
 * whether a node is selected.
 */
export type DockContext =
  | 'observer_no_node'      // observer, no node selected
  | 'observer_node'         // observer, a node is selected
  | 'participant_own'       // participant viewing own bubble
  | 'participant_other';    // participant viewing another's bubble

/** Render-ready strings for the collapsed pill / anchor chip. */
export interface CollapsedDockLabel {
  /** Short visible text on the chip, e.g. "Watch". The caret glyph is
   *  appended by the component, not stored here. */
  primary: string;
  /** accessibilityLabel — fuller phrasing for screen readers. */
  accessibilityLabel: string;
  /** accessibilityHint — what expanding does. */
  accessibilityHint: string;
}

/** A category section as rendered in the expanded dock. */
export interface DockCategorySection {
  category: RailActionCategory;       // re-used from ArgumentSideActionRail
  /** RAIL_ACTION_CATEGORY_LABEL[category]; only consumed when showHeader. */
  headerLabel: string;
  actions: RailActionWithCategory[];  // re-used from ArgumentSideActionRail
}

/** The fully-derived expanded-dock view model. */
export interface ExpandedDockViewModel {
  sections: DockCategorySection[];
  /** True only when ≥ 2 sections are non-empty. When false, the single
   *  section renders header-less (flat list). */
  showCategoryHeaders: boolean;
  /** Title shown in the dock header — context-derived, no verdict copy. */
  title: string;
}
```

Inputs are all values that already exist at the `ArgumentSideActionRail`
call site: `viewerRole`, `bubbleActor`, `participantSide`,
`activeMessageId`, plus `windowWidth` (from `useWindowDimensions()`) and an
effective reduce-motion boolean. **The dock derives, never stores,
gameplay state.** Heat, strength bands, and lifecycle are not read by this
card.

`railActionGrouping.test.ts` (the existing SC-001 contract-lock suite)
must continue to pass untouched.

### Mutual exclusion contract (SC-002 / SC-004)

The dock and the timeline node popover are **one-open-at-a-time**:

- When `isAnyPanelOpen === true`, the dock renders in collapsed form only
  and ignores expand presses (or, simpler: it still renders the chip but
  expanding it first calls `onExpandedChange(true)` which the parent uses
  to close the popover). The chosen mechanism: **parent-owned exclusion.**
  `ArgumentGameSurface` holds both `selectedDockTarget` (popover/dock) and
  a new `railExpanded` boolean. The rail calls `onExpandedChange(true)`;
  the parent's handler sets `setSelectedDockTarget(null)` then
  `setRailExpanded(true)`. Conversely, when the parent opens the popover
  (`onSelectTarget`), it sets `setRailExpanded(false)`. The rail receives
  `isAnyPanelOpen` so it can also collapse itself reactively if the popover
  opens while the rail is expanded.
- This mirrors the existing SC-004 ↔ SC-002 exclusion already implemented
  for `selectedDockTarget` (see `timelineNodeActionDockSelectionExclusion.test.ts`).
  SC-005 extends the same single-owner pattern; it does not introduce a
  competing exclusion mechanism.

## Edge cases

The implementer must handle:

- **No node selected (observer):** collapsed primary = "Watch ▾". Expanded
  dock still shows the full observer set (Watch is itself one of the six
  actions). This is the default room-entry state.
- **Node selected but observer:** collapsed primary = "Actions on this
  point ▾". Action set is still the observer set (Stage 6.4 does not give
  observers per-bubble tactical actions — `getRailActions` returns
  `OBSERVER_ACTIONS` whenever `viewerRole === 'observer'` regardless of
  `bubbleActor`). The label changes; the action set does not. Tests must
  assert this explicitly so nobody "fixes" it into a bug.
- **Participant on own bubble:** action set = `SELF_ACTIONS` (Qualifiers +
  Request deletion **only**). With only two actions both in the
  `review_flag` category, `groupRailActionsByCategory` yields ONE section
  → `showCategoryHeaders` is `false` → the dock renders a flat,
  header-less two-chip list. No "Reply"/"Disagree"/"Flag"/"Edit" ever
  appears. This is doctrine-critical.
- **Participant on other bubble:** action set = `PARTICIPANT_OTHER_ACTIONS`
  (7 actions across `reply` / `evidence` / `branch` / `review_flag`) →
  ≥ 2 non-empty categories → headers shown.
- **Participant already on the side a Join chip targets:** the existing
  redundant-join filter (`join_aff` hidden when `participantSide ===
  'affirmative'`, etc.) stays. Note: observers never carry a
  `participantSide`, so the filter is inert for the observer set. Keep the
  filter exactly as-is.
- **Empty action list:** `getRailActions` can never return empty for the
  three real roles, but a defensive guard renders the collapsed chip and
  an empty (header-less) expanded body rather than crashing.
- **Narrow sheet height vs tiny viewport:** `resolveSheetMaxHeightPx`
  clamps to `SHEET_MIN_HEIGHT_PX` (168) so a 400 px-tall device still
  gets a usable sheet, and never returns `>= windowHeight` so the sheet is
  never full-screen. The expanded body inside is itself a `ScrollView` so
  the 7-action participant set scrolls within the 28% cap rather than
  overflowing.
- **`windowWidth <= 0` / non-finite (web static export first paint):**
  `resolveObserverDockVariant` returns `'side'` (polished layout first
  paint), matching `resolveHeaderBreakpoint` / `resolveDockLayoutVariant`.
  Hydration corrects to `'sheet'` on the next paint for real narrow
  viewports.
- **Reduce-motion on:** the narrow sheet snaps open/closed (no slide-up
  `translateY`); only opacity / static positioning is used. The wide side
  dock has no slide animation in either mode (it is anchored, not a
  sheet). Verified by `sideActionDockA11y.test.ts`.
- **SC-002 popover open while the dock is expanded:** `isAnyPanelOpen`
  flips true → the dock collapses itself (`expanded` forced to false in a
  derived render guard, not a destructive state write that fights the
  parent). Re-expanding the dock closes the popover via
  `onExpandedChange`.
- **Permission-denied paths:** none — the dock surfaces actions; the
  *actions themselves* (flag, request_deletion, etc.) carry their own
  permission/Edge-Function handling downstream and are unchanged by this
  card.
- **Doctrine edge case — heat / strength must not enter the dock:** the
  dock reads only `viewerRole` / `bubbleActor` / `participantSide` /
  `hasSelectedNode` / viewport size. It does NOT read `lifecycleMap`,
  `metadataLedger`, score bands, or heat. The collapsed label "Actions on
  this point" describes *selection*, never *standing*. A hot node and a
  quiet node produce the identical dock.
- **Doctrine edge case — observer label must not imply a verdict:**
  "Watch" / "Actions on this point" are activity words, not judgments.
  The ban-list test covers this.

## Risks

- **`DockLayoutVariant` name collision.** `ArgumentComposerDock.tsx`
  already exports a `DockLayoutVariant` type and a `resolveDockLayoutVariant`
  function with the *same shape*. SC-005 intentionally defines its **own**
  copy in `ObserverActionDockLayout.ts` rather than importing the
  composer's — importing would couple the observer dock to the composer
  module and risk a circular-ish dependency through the arguments feature.
  The duplication is two tiny pure functions; the alternative (a shared
  `dockLayout.ts`) is a separate refactor card. The implementer must NOT
  re-export the composer's symbol; keep them independent. Note this in the
  PR.
- **Prop chain through `ArgumentTreeScreen`.** Folding the `App.tsx`
  `actionBar` into the rail needs `startArgumentAction` threaded through
