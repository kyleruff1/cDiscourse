## Goal (one paragraph)

Give a signed-in user a lightweight, **optional** way to add a small handful of
**identity-light profile tags** that describe their interests and how they like
to debate — opened as a `Modal` bottom sheet, the same affordance PR-001
established. The user picks tags from a **closed, curated, safe vocabulary**
across four categories (topic interests, debate style, availability,
accessibility self-description); they never type a free-text tag, so a hostile
label or a protected-class target can never enter the system. At most **5 tags**
may be selected; zero is fine (tags are optional). Doctrine constraints that
shape the design: tags are **purely cosmetic social context** — they have
**zero effect on truth, score, heat, point standing, or any validation gate**
(`cdiscourse-doctrine` §1 — a tag never makes a point stronger, weaker, or a
post blocked); the vocabulary contains **no protected-class targeting, no
party-affiliation requirement, no unverified "expert" claim, no hostile labels,
and no ideology/personality scoring** (the card's DISALLOWED list, which is
doctrine-load-bearing). Tags persist **device-locally via `AsyncStorage`** to
keep PR-002 migration-free, exactly like PR-001's preference blob; cross-device
and other-user-visible tags are an explicit v2 follow-up. The two named tests —
a **validation-gate immutability test** and a **tag-vocabulary safety test** —
prove the Constitution engine / scoring / validation is byte-identical with and
without tags, and that the vocabulary carries no forbidden token.

## Reuse-vs-new decision

The parent prompt asks for an explicit, justified decision on whether PR-002's
tags live in PR-001's *same* preferences store/popout, or a *separate* surface.

**Decision: a SEPARATE `ProfileTagPopout` `Modal` and a SEPARATE
`profileTagsStorage` `AsyncStorage` blob — but reachable via a NEW row added
inside PR-001's existing `PreferencesPopout`, and built from PR-001's reusable
primitives.**

Justification:

1. **Keep the stores separate.** PR-001's `UserPreferences` blob is documented
   in `userPreferencesModel.ts` as holding "only enums + one boolean", with a
   pinned `schemaVersion: 1` and a `mergeWithDefaults` parser tuned for that
   exact shape. Tags are a *list of strings from a vocabulary* — a structurally
   different value with a different defensive-parse need (vocabulary membership
   filtering, length cap). Folding a `tags: string[]` field into
   `UserPreferences` would (a) force a `schemaVersion` bump and a re-test of the
   whole PR-001 model, (b) mix two concerns in one blob, and (c) make the
   PR-001 doctrine test (which asserts the blob shape carries no role field and
   only enums/boolean) brittle. A separate `cdiscourse:profile-tags:<userId>`
   blob is cleaner and isolates risk.

2. **Keep the popout separate.** PR-001's `PreferencesPopout` already renders
   nine fields in a `ScrollView` and is ~320 lines. Tag selection is a
   multi-select chip grid across four categories — a substantial sub-surface of
   its own. Cramming it into the existing popout would make a long screen
   unwieldy. A dedicated `ProfileTagPopout` keeps each popout focused (the
   `Modal` self-manages its overlay, so stacking a second `Modal` over the
   first is supported by RN core — `DeletionRequestSheet` proves this pattern).

3. **But reuse everything reusable.** PR-002 imports/mirrors:
   - the **`Modal` bottom-sheet shell shape** from `PreferencesPopout.tsx`
     (`transparent`, `animationType` conditioned on reduce motion, the
     `backdrop` / `sheet` / `header` / close-button style structure);
   - the **`AsyncStorage` versioned-blob pattern** from `preferencesStorage.ts`
     (`load` never throws → defaults; `save` swallows; one JSON blob per user;
     key built in `sessionKeys.ts`);
   - the **pure-helpers + source-scan test style** from
     `userPreferencesModel.test.ts` + `preferencesDoctrine.test.ts` (no
     React-Testing-Library `render()`);
   - the **copy-module pattern** (`preferencesCopy.ts`) — all user strings in
     one file so the ban-list test scans one place.
   - It **may** reuse `PreferenceSegmentedControl` for a category filter if the
     implementer finds it natural, but the core control is a new
     `ProfileTagChip` (multi-select, not radio) — see §File changes.

4. **Discoverability.** PR-001's popout is the user's one self-service surface.
   Adding a single "Profile tags" row there — a labelled `Pressable` that opens
   `ProfileTagPopout` — means users find tags without a second header button.
   This is a ~12-line change to `PreferencesPopout.tsx` (one new field block)
   plus one new prop. PR-001's design doc itself anticipated this:
   *"PR-002 (profile-tag popout) will likely reuse `PreferencesPopout`/
   `PreferenceRow` primitives."*

This satisfies the parent prompt's "strongly prefer reusing PR-001's
infrastructure over building a parallel one" — PR-002 reuses the *patterns and
primitives* and the *entry surface*, while keeping the *store* and the *popout
component* appropriately separate for the structurally different data.

## Data model

**No new SQL. No migration. No Supabase column. No Edge Function. No RLS work.**

One new pure-TypeScript tag model, persisted as a single JSON blob in
`AsyncStorage`, mirroring PR-001's `userPreferencesModel.ts` shape and discipline.

```ts
// src/features/profileTags/profileTagModel.ts  (pure TS — no React, no Supabase, no network)

/** The four allowed tag categories (the card's ALLOWED list). */
export type ProfileTagCategory =
  | 'topic_interest'
  | 'debate_style'
  | 'availability'
  | 'accessibility_note';

/**
 * A single tag in the closed vocabulary. `id` is the stable internal code
 * (never user-visible); `label` is plain-language UI prose; `category` groups
 * it. The vocabulary is CLOSED — users select from `PROFILE_TAG_VOCABULARY`,
 * they never construct a ProfileTagDefinition.
 */
export interface ProfileTagDefinition {
  /** Stable internal code, e.g. 'topic_climate'. Never rendered to the user. */
  id: string;
  category: ProfileTagCategory;
  /** Plain-language label, e.g. 'Climate & environment'. The only thing shown. */
  label: string;
}

/** The full per-user profile-tag blob. Versioned for forward-safe migration. */
export interface ProfileTagSelection {
  /** Schema version — bump on any breaking shape change. */
  schemaVersion: 1;
  /**
   * The selected tag ids, in selection order. Max length MAX_PROFILE_TAGS (5).
   * Every id MUST be a member of PROFILE_TAG_VOCABULARY — the parser drops any
   * unknown id defensively.
   */
  selectedTagIds: string[];
}

/** The selection cap. The card's "3-5 visible" ceiling. Single source of truth. */
export const MAX_PROFILE_TAGS = 5;

/** Canonical empty selection — tags are optional, so the default is zero tags. */
export const DEFAULT_PROFILE_TAG_SELECTION: ProfileTagSelection = Object.freeze({
  schemaVersion: 1,
  selectedTagIds: [],
});
```

The vocabulary itself is a frozen array (defined in full in §Tag vocabulary):

```ts
export const PROFILE_TAG_VOCABULARY: ReadonlyArray<ProfileTagDefinition> = Object.freeze([
  /* ...the ~28 curated tags below... */
]);
```

**Why a separate blob, not a field on `UserPreferences`:** see §Reuse-vs-new
decision. The two blobs are siblings under different `AsyncStorage` keys
(`cdiscourse:preferences:<userId>` vs `cdiscourse:profile-tags:<userId>`).

## Tag vocabulary (the closed, curated allow-list)

This is the **complete v1 vocabulary**. Users pick from exactly these tags;
there is **no free-text entry anywhere in PR-002**. Every label below is
plain-language, identity-light, and carries **no protected-class targeting, no
party affiliation, no unverified "expert" claim, no hostile label, and no
ideology/personality score**. The implementer ships this list verbatim in
`profileTagVocabulary.ts`; the tag-vocabulary safety test asserts it.

### Category 1 — Topic interests (`topic_interest`) — 10 tags

Broad subject areas, deliberately neutral. No partisan framing.

| id | label |
|---|---|
| `topic_climate` | Climate & environment |
| `topic_technology` | Technology & internet |
| `topic_science` | Science & research |
| `topic_economy` | Economy & work |
| `topic_education` | Education & learning |
| `topic_health` | Health & wellbeing |
| `topic_culture` | Arts & culture |
| `topic_history` | History |
| `topic_cities` | Cities & transport |
| `topic_ethics` | Ethics & philosophy |

*Excluded on purpose:* no "abortion", "immigration", "gun control" style
hot-button single-issue tags — those invite party-affiliation signalling and
protected-class proximity. Subject areas are broad enough to be identity-light.

### Category 2 — Debate style (`debate_style`) — 8 tags

How the person likes to *engage* — process descriptors, never personality
verdicts. No "good"/"bad", no "aggressive", no "smart", no scoring words.

| id | label |
|---|---|
| `style_evidence_first` | Likes evidence-first arguments |
| `style_steelman` | Enjoys steelmanning |
| `style_concise` | Prefers concise points |
| `style_deep_dive` | Enjoys deep dives |
| `style_questions` | Asks lots of questions |
| `style_definitions` | Likes pinning down definitions |
| `style_friendly` | Keeps it friendly |
| `style_learning` | Here mostly to learn |

*Excluded on purpose:* no "wins debates", "rarely concedes", "expert debater",
"aggressive", "ruthless", "always right" — those are verdict/personality-score
tokens or unverified expertise claims, all on the DISALLOWED list.

### Category 3 — Availability (`availability`) — 5 tags

When the person tends to be around. Purely scheduling context.

| id | label |
|---|---|
| `avail_weekdays` | Usually around on weekdays |
| `avail_weekends` | Usually around on weekends |
| `avail_evenings` | Usually around in the evenings |
| `avail_async` | Replies when I can (no rush) |
| `avail_quick` | Happy to reply quickly |

*Excluded on purpose:* no time-zone-as-location tags (a precise timezone can
proxy a protected-class/geographic target); coarse "weekdays/weekends/evenings"
stays identity-light.

### Category 4 — Accessibility self-description (`accessibility_note`) — 5 tags

Optional, **self-described** notes so other people can be considerate. These are
NOT functional settings (PR-001 owns `reduceMotion` / `colorMode`); they are
social context only and drive nothing in the renderer.

| id | label |
|---|---|
| `a11y_screen_reader` | I use a screen reader |
| `a11y_plain_language` | I prefer plain language |
| `a11y_short_messages` | Shorter messages help me |
| `a11y_more_time` | I may need more time to reply |
| `a11y_captions` | I prefer text over audio/video |

*Excluded on purpose:* nothing here names a diagnosis, a disability category, or
any protected-class medical status — each tag describes a **preference or a
practice**, chosen by the user, never a clinical/identity label. This keeps the
category identity-light and out of protected-class territory.

**Total v1 vocabulary: 28 tags across 4 categories.** All ids are stable
internal codes; all labels are plain-language UI prose. The vocabulary is frozen
(`Object.freeze`) and is the single source of truth for both the UI and the
safety test.

### Doctrine rationale for the closed list

A **closed vocabulary is the doctrine mechanism** that makes the DISALLOWED list
enforceable. If tags were free-text, no test could guarantee the absence of a
hostile label or a protected-class target — a user could type anything. Because
the vocabulary is a fixed, code-checked array, the tag-vocabulary safety test
can exhaustively assert that **every** shipped tag is safe, and the UI can
**only** offer safe tags. There is no "other / custom" escape hatch. This is the
same principle PR-001 used for its enum preferences — the user chooses from a
fixed set, never constructs a value.

export function canAddTag(
  selection: ProfileTagSelection,

export function toggleTag(
  selection: ProfileTagSelection,
  tagId: string,

  tagId: string,

### Storage (`profileTagsStorage.ts`)

```ts
export function loadProfileTags(userId: string | null): Promise<ProfileTagSelection>;
export function saveProfileTags(userId: string | null, sel: ProfileTagSelection): Promise<void>;
```

Both never throw — `load` returns `DEFAULT_PROFILE_TAG_SELECTION` on any failure
(missing key, corrupt JSON, storage error, over-cap/garbage blob — the last
truncated/filtered by `mergeTagSelectionWithDefaults`). `save` swallows errors.
`userId === null` → keyed under the anon key, identical to `preferencesStorage`.

### Hook (`useProfileTags.ts`)

```ts
interface UseProfileTagsResult {
  selection: ProfileTagSelection;   // DEFAULT_PROFILE_TAG_SELECTION until load resolves
  loading: boolean;
  count: number;                    // selection.selectedTagIds.length
  atLimit: boolean;                 // count >= MAX_PROFILE_TAGS
  toggleTag: (tagId: string) => void;   // optimistic; fires save
  clearTags: () => void;                // resets to default; fires save
}
export function useProfileTags(userId: string | null): UseProfileTagsResult;
```

## Edge cases

- **Signed out / no `userId`:** the preferences popout (and therefore the
  "Profile tags" row that opens this popout) is only rendered inside the
  signed-in shell. The hook still tolerates `userId === null` (anon key)
  defensively, identical to `useUserPreferences`.
- **`AsyncStorage` read failure / corrupt JSON:** `loadProfileTags` catches and
  returns `DEFAULT_PROFILE_TAG_SELECTION`. `mergeTagSelectionWithDefaults`
  handles a partial/garbage object (non-object → default; non-array
  `selectedTagIds` → `[]`).
- **Blob contains MORE than 5 tag ids** (e.g. a hand-edited or future-version
  blob): `mergeTagSelectionWithDefaults` **truncates to `MAX_PROFILE_TAGS`** —
  the cap is enforced on read, not only on write. The validation never depends
  on the UI having enforced it.
- **Blob contains an unknown / removed tag id** (vocabulary shrank between
  versions): the parser **drops** any id not in `PROFILE_TAG_VOCABULARY`. The
  user silently loses a tag that no longer exists; the surviving tags are
  unaffected. (A future vocabulary change should prefer adding over removing.)
- **Blob contains duplicate tag ids:** the parser de-duplicates, keeping first
  occurrence.
- **`AsyncStorage` write failure:** `saveProfileTags` swallows the error; the
  in-memory selection still reflects the change for the session. Non-fatal,
  mirrors `preferencesStorage`.
- **User reaches the 5-tag cap:** unselected chips become `disabled` (the model
  `toggleTag` is a no-op for an add past the cap; the UI also visually disables
  and the chip stops responding). A helper line explains "You can add up to 5
  tags — remove one to add another." Already-selected chips stay tappable (so
  the user can deselect to make room).
- **User selects zero tags:** fully valid — tags are optional. The popout shows
  the `profile-tag-empty` line ("No tags yet — tags are optional."). The
  preferences-popout row shows "No tags yet".
- **Concurrent edits across two devices:** tags are device-local
  `AsyncStorage`, so there is **no cross-device conflict by construction** —
  each device has its own selection. (This is also why shared visibility is a
  v2 card — see §Out of scope.)
- **Offline:** tag toggles write to `AsyncStorage` — they work fully offline.
  There is no network path in PR-002 at all.
- **Reduce motion:** the `ProfileTagPopout` `Modal` passes `animationType="none"`
  when `reduceMotion` is true (read from PR-001's `useUserPreferences`
  `effectiveReduceMotion`), so it snaps open instead of sliding —
  `accessibility-targets`.
- **Doctrine edge — "can a tag change my score or unblock a post?"** No. Tags
  are read **only** by `ProfileTagPopout` (display). No tag value is ever an
  input to `argumentScoreModel`, the Constitution engine, `antiAmplification`,
  the point-standing economy, or any validation gate. Enforced by the
  validation-gate immutability test and a source-scan import ban.
- **Doctrine edge — "can a tag carry a hostile label or a protected-class
  target?"** No. There is **no free-text entry** anywhere in PR-002. The user
  selects only from `PROFILE_TAG_VOCABULARY`, every member of which is asserted
  safe by the tag-vocabulary safety test. There is no "custom / other" option.
- **Doctrine edge — "does the 'accessibility self-description' tag duplicate or
  override PR-001's `reduceMotion` / `colorMode`?"** No. The
  `accessibility_note` tags are **inert social context** — they drive nothing in
  the renderer. PR-001's functional accessibility preferences are unchanged and
  untouched. The copy makes the distinction explicit.
- **Doctrine edge — "can a tag escalate my role / permissions?"** No. The tag
  blob holds only `schemaVersion` + a `string[]` of vocabulary ids. It carries
  no role, no auth material. The doctrine test scans for `role`/`admin`/
  `moderator`/`service_role` tokens.

## Risks

- **Stacking two `Modal`s.** Opening `ProfileTagPopout` from inside
  `PreferencesPopout` puts a second core `Modal` over the first. RN core
  `Modal` supports this (`DeletionRequestSheet` opens from inside the room while
  other overlays exist), and on Expo web it renders fine. The implementer
  should verify the close affordances behave (closing the tag popout returns to
  the preferences popout; closing the preferences popout while the tag popout
  is open is an edge — the simplest correct behaviour is: the "Profile tags"
  row does not need to *close* the preferences popout, the tag popout simply
  layers above it). The browser-verification checklist exercises exactly this.
- **The preferences-popout field-list test.** `preferencesDoctrine.test.ts` has
  a `FIELD_TEST_IDS` array asserting the nine PR-001 field testIDs are present;
  it uses `toContain`, so a tenth field does not break it. But the same file
  also asserts `the only TextInput in the popout is the display-name input`
  (`inputs.length).toBe(1)`). PR-002's new "Profile tags" row is a `Pressable`,
  **not** a `TextInput`, so this assertion still holds — but the implementer
  **must not** add any text input to the preferences popout, and must re-run
  `preferencesDoctrine.test.ts` after the edit to confirm it stays green.
- **Vocabulary churn.** If the vocabulary ever shrinks, users silently lose the
  removed tag (the parser drops unknown ids). Prefer **adding** tags over
  removing in future cards; if a tag must be removed for safety reasons, that is
  acceptable (safety wins) and the drop-on-read behaviour handles it gracefully.
- **"Accessibility self-description" category sensitivity.** This category is
  the one closest to protected-class territory (disability-adjacent). The
  design deliberately frames every tag as a **practice or preference the user
  chooses to share** ("I use a screen reader", "Shorter messages help me") —
  never a diagnosis or a clinical/identity label. The tag-vocabulary safety
  test explicitly bans diagnosis/disability-category tokens. The implementer
  must not add tags like "blind", "deaf", "autistic", "disabled" — those would
  fail the safety test and are out of bounds.
- **Scope creep toward shared visibility.** It is tempting to "just add a
  column" so other users see tags. That is a migration + RLS + an other-user
  profile screen that does not exist — out of scope for PR-002 (see §Out of
  scope). The implementer must not add a Supabase write.
- **No new tests removed / no `.skip`.** Standard discipline, called out
  because PR-002 adds four test files.
