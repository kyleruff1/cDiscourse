## Goal (one paragraph)

When a Timeline node/message is selected, the board must prominently and unmistakably display the selected message and its full active context — a persistent **instrument panel** for the selected point, not a transient popover. Today the selected-message detail lives in `ArgumentReplySidecar` (SC-003), rendered *below* `ArgumentTimelineMap` inside `ArgumentGameSurface`'s timeline branch (`ArgumentGameSurface.tsx` L623–637). It is visually subordinate, frequently off-screen, and gives the user no unmistakable confirmation of *which* node is selected or *which* message the action dock will act on. IX-004 elevates this readout into a prominent, persistent **selected-message readout panel** that sits **above** the timeline map (and the action dock), refreshes on every selection change with unmistakable click feedback (the existing VG-004 selection glow + a panel re-render announced to screen readers), shares one selection id (`activeMessageId`) across Timeline and Cards/Stack, follows Prev/Next/Latest/Back-to-root, handles collapsed-branch/stub/cluster clicks, and degrades gracefully when the selected message is gone ("That message is no longer here — showing the latest move."). Per the issue's decisive implementation note, **this card reuses `buildSidecarViewModel`; the change is layout prominence + click-feedback, NOT a new view model.** Doctrine constraints that shape the design: standing/heat/tone copy is non-verdict (cdiscourse-doctrine §1, §2; the readout reuses SC-003's already-doctrine-clean band lines); heat = activity not correctness; no internal `snake_case` codes leak (routed through RULE-001/003 plain-language maps already used by `buildSidecarViewModel`); selection cue is color-independent (border weight, not color alone — timeline-grammar); strictly in-surface (no route push).

## Data model

**No new persisted data model. No migration. No new domain type.**

The card adds one small **pure-TS view-model wrapper** (a presentation projection over `SidecarViewModel`), not a new data model. `buildSidecarViewModel` already carries every required field — body excerpt, parent hint + parent body preview, standing/tone/heat lines, lifecycle ("Why it matters"), unresolved items, where-it-sits (branch / ordinal / total / path), semantic flags. The one field the issue asks for that SC-003 does not currently surface as a discrete value is **direct reply/child count** (the issue's "direct reply/child count"). It is derivable from data the room already has and must NOT require a new model: `ArgumentTimelineMapNode` already exposes `isJunction` + `junctionChildCount`, and the full timeline map can count direct children of any `messageId`. The wrapper computes it from existing inputs.

### New pure-TS module: `timelineSelectedReadoutModel.ts`

A thin **projection** layer. It does not re-derive standing, lifecycle, or metadata — it consumes the already-built `SidecarViewModel` plus three small extras and produces a flat, render-ready `TimelineSelectedReadoutViewModel` for the new panel component. This keeps the panel a thin presentation layer and keeps all the new logic unit-testable without React.

```ts
// src/features/arguments/timelineSelectedReadoutModel.ts
import type { SidecarViewModel } from './argumentReplySidecarModel';
import type { ArgumentTimelineMapModel, ArgumentTimelineMapNode } from './argumentGameSurfaceModel';

/** Why the readout is showing what it shows — drives the fallback banner. */
export type ReadoutSelectionStatus =
  | 'explicit'        // user clicked / navigated to this node
  | 'entry_hint'      // pre-activated from the Stage 6.4 gallery entry hint
  | 'default_latest'  // no explicit pick yet — showing latest (or root)
  | 'stale_fallback'; // previously-selected id is gone — snapped to latest

export interface TimelineSelectedReadoutViewModel {
  /** True when the room has zero nodes — panel shows the empty hint. */
  isEmpty: boolean;
  /** The message id this readout describes. null only when isEmpty. */
  selectedMessageId: string | null;
  /** Why this message is selected — drives the optional banner. */
  status: ReadoutSelectionStatus;
  /**
   * One-line banner shown above the body when status === 'stale_fallback'.
   * Verbatim: "That message is no longer here — showing the latest move."
   * null for every other status.
   */
  staleNotice: string | null;
  /** The SC-003 view-model the panel renders verbatim (reused, not rebuilt). */
  sidecar: SidecarViewModel;
  /**
   * Direct reply count for the selected node (children whose parentId ===
   * selectedMessageId in the built map). 0 when none. Plain-language label
   * is `replyCountLabel`.
   */
  directReplyCount: number;
  /** "No direct replies yet" | "1 direct reply" | "N direct replies". */
  replyCountLabel: string;
  /**
   * Short label for the action dock's "Acting on:" line. ≤ 48 chars.
   * Form: "<kindLabel> · #<ordinal>" (e.g. "Challenge · #4").
   * Mirrors the node ordinal so the dock target is never ambiguous.
   */
  actingOnShortLabel: string;
  /**
   * Full a11y label for the panel root + the announcement string fed to
   * accessibilityLiveRegion on a selection change. Summarises kind +
   * ordinal + standing band. Reuses sidecar.accessibilityRootLabel and
   * appends the reply count.
   */
  accessibilityPanelLabel: string;
  /**
   * The string announced via accessibilityLiveRegion when the selection
   * changes. Includes the staleNotice when status === 'stale_fallback'.
   */
  accessibilitySelectionAnnouncement: string;
}

export interface BuildTimelineSelectedReadoutInput {
  /** The SC-003 view-model — already built by the room shell. */
  sidecar: SidecarViewModel;
  /** The built timeline map — source of node order + child counts. */
  timelineMap: ArgumentTimelineMapModel;
  /** The currently-selected message id (room shell's activeMessageId). */
  selectedMessageId: string | null;
  /** Why this id is selected. See ReadoutSelectionStatus. */
  status: ReadoutSelectionStatus;
}

export function buildTimelineSelectedReadoutViewModel(
  input: BuildTimelineSelectedReadoutInput,
): TimelineSelectedReadoutViewModel;
```

**Banded values** (standing / heat / tone) are NOT re-derived — they come through `sidecar.sections[0]` (`SidecarSection_WhatThisMoveSays.standingLine / toneLine / heatLine`), which `argumentReplySidecarModel.ts` already builds from `node.standingBand / toneBand / temperatureBand` in non-verdict language. IX-004 renders them; it does not author them.

`directReplyCount` is computed by counting `timelineMap.nodes` whose `parentId === selectedMessageId`. (`ArgumentTimelineMapNode` exposes `parentId`; see `argumentGameSurfaceModel.ts` L1429.) This is O(nodes) per render, acceptable for v1 room sizes; the room shell already iterates `timelineMap.nodes` several times.

### Selection-status state in the room shell

`ArgumentGameSurface` already owns `activeMessageId`. IX-004 adds **one** companion state value — `selectionStatus: ReadoutSelectionStatus` — so the readout can tell an explicit pick from a stale fallback. No new id is introduced; `activeMessageId` stays the single selection source of truth shared across Timeline and Cards.

## API / interface contracts

### `buildTimelineSelectedReadoutViewModel(input): TimelineSelectedReadoutViewModel`

See the Data model section for the full signature. Contract notes:

- **Pure.** No side effects, no async, no React, no Supabase. Deterministic given inputs.
- When `input.timelineMap.nodes.length === 0` → returns `{ isEmpty: true, selectedMessageId: null, status, staleNotice: null, sidecar: input.sidecar, directReplyCount: 0, replyCountLabel: 'No direct replies yet', actingOnShortLabel: '', accessibilityPanelLabel: <empty-state copy>, accessibilitySelectionAnnouncement: <empty-state copy> }`.
- `staleNotice` is the frozen string `"That message is no longer here — showing the latest move."` **only** when `status === 'stale_fallback'`; `null` otherwise.
- `replyCountLabel`: `0 → "No direct replies yet"`, `1 → "1 direct reply"`, `n → "${n} direct replies"`.
- `actingOnShortLabel`: `"${kindLabel} · #${ordinal}"` from the selected node; empty string when no node resolved.
- `accessibilitySelectionAnnouncement`: when `stale_fallback`, prefix the announcement with `staleNotice`; otherwise it equals `accessibilityPanelLabel`.
- Never throws on malformed input — missing node / null `selectedMessageId` degrades to the `isEmpty`-style safe shape.

### `TimelineSelectedReadoutPanel` props

```ts
interface Props {
  viewModel: TimelineSelectedReadoutViewModel;
}
```

The panel is **read-only** — no callbacks, no dispatch, no `onAction`. It mirrors SC-003's "view-model is pure data, component is a thin layer" boundary. Render contract:

- Root `<View testID="timeline-selected-readout-panel" accessibilityLabel={viewModel.accessibilityPanelLabel}>`.
- When `viewModel.staleNotice` is non-null: a `<View testID="timeline-readout-stale-banner">` containing a `<Text>` of the notice, styled as an informational (not error) banner — amber/slate, no red, no verdict tone.
- A `<Text testID="timeline-readout-reply-count">` showing `replyCountLabel`.
- The wrapped `<ArgumentReplySidecar viewModel={viewModel.sidecar} />`.
- An `accessibilityLiveRegion="polite"` host `<View>` (web) whose text content is `viewModel.accessibilitySelectionAnnouncement`; the text is keyed on `selectedMessageId` so it re-announces on selection change. On native, `AccessibilityInfo.announceForAccessibility(viewModel.accessibilitySelectionAnnouncement)` is fired from a `useEffect` keyed on `selectedMessageId` (mirrors the existing pattern in `ArgumentTimelineMap.tsx` L514–522 — wrapped in try/catch).

### `ArgumentTimelineMap` new prop


Rendered only when `selectedTarget?.kind === 'node' && actionDockModel && !popoverModel && actingOnLabel`. Single `<Text testID="timeline-acting-on">Acting on: {actingOnLabel}</Text>`.

### Selection-glow / click feedback

**No new visual token.** Click feedback reuses VG-004's existing node treatment already wired in `ArgumentTimelineMap.tsx`:
- The active node already gets `nodeRingActive` + `deriveTimelineNodeVisualStyle` glow (indigo 2px stroke via `GLOW.activePath`, soft shadow dropped under reduce-motion).
- The SC-004 dock target node already gets the cream `nodeHalo` (`GLOW.selectedHalo`, static stroke kept under reduce-motion).
- IX-004's contribution to "unmistakable feedback" is the **panel refresh** — the prominent panel re-rendering with the new subject is itself the loud, persistent confirmation the issue asks for. The selection cue is therefore **color-independent**: border weight (the 2px glow stroke + 3px halo ring) plus geometry (scale/zIndex) plus the panel content change. No reliance on color alone.

## Edge cases

The implementer must handle each of these:

- **Empty room (zero nodes).** Panel renders the SC-003 empty state ("Pick a message on the timeline to see details.") inside the panel frame. No stale banner, no reply-count line, no crash. `buildTimelineSelectedReadoutViewModel` returns the `isEmpty` shape.
- **No explicit selection yet (first paint).** `selectionStatus` starts as `default_latest` (or `entry_hint` when a gallery entry hint exists). Panel shows the entry-hint target or latest/root per the existing `initialActiveId` logic (`ArgumentGameSurface.tsx` L180–192) — IX-004 does not change which message is initially active, only labels *why*.
- **Selected message soft-deleted / removed upstream.** The existing stale-snap `useEffect` snaps `activeMessageId` to `latestId` and sets `selectionStatus = 'stale_fallback'`. Panel shows the stale banner verbatim. If the room is now empty after removal, fall through to the empty-room case.
- **Selected message inside a collapsed branch.** `ArgumentTimelineMap` already auto-expands ancestors of the active node (`applyActiveAutoExpand`, L511–523). The readout follows `activeMessageId`, so selecting a node inside a collapsed branch (via Prev/Next/keyboard) auto-expands and the panel updates. No extra IX-004 work — document the dependency.
- **Branch stub clicked.** Per BR-001 contract: tapping a `collapsed_stub` should select the branch root and expand. `ArgumentTimelineMap`'s `handleStubPress` currently only toggles collapse. IX-004's readout requirement is "branch stub click selects the branch root and expands if collapsed." **Resolution:** the design wires `handleStubPress` to *also* call `onActivate(stub.branchRootMessageId)` so the readout shows the branch root. This is a 1-line addition inside the existing `handleStubPress` callback — documented here so the implementer does not treat it as out of scope. It does not change the action dock's `collapsed_stub` handling.
- **Cluster target selected (action dock `kind: 'cluster'`).** The action dock can target a cluster; the readout panel is message-centric. When `selectedDockTarget.kind === 'cluster'`, the readout continues to show the *node* identified by `activeMessageId` (the branch root, which is what the dock cluster target also points at). `actingOnShortLabel` for a cluster target falls back to the branch-root node's label. No separate "cluster readout" is built — out of scope, documented below.
- **Junction node (multiple direct children).** `directReplyCount` correctly reflects `> 1`; `replyCountLabel` pluralizes. The panel does not need a special junction layout.
- **Detached node selected.** `buildSidecarViewModel` already handles detached nodes (parentHint is null). The readout panel renders normally; reply count is whatever children exist.
- **Concurrent edits / new message arrives while a node is selected.** The room hook re-renders with new `messages`; `activeMessageId` is preserved (it is not auto-snapped unless the *selected* message vanished). The panel re-renders with refreshed reply count + bands. New messages do NOT steal the selection.
- **Offline / network failure.** IX-004 is pure UI over already-loaded data — no fetch. Offline simply means no new messages arrive; the panel keeps showing the last-known selection. No special handling.
- **Permission-denied / observer.** Observer sees the full readout (issue: "Observer: readout fully visible"). The readout has no action affordances of its own, so there is no permission gate on the panel. The action dock below it already gates observer actions (SC-004 actor matrix) and shows observer affordances only.
- **Own bubble selected.** Readout fully visible; the action dock below it already restricts own-bubble actions to Qualifiers + Request deletion (SC-004 actor matrix). IX-004 adds nothing here — it must just NOT introduce any action affordance on the panel that would bypass the matrix.
- **Doctrine edge case — "what if heat tries to influence the standing band?"** It does not. `heatLine` and `standingLine` are independent fields on `SidecarSection_WhatThisMoveSays`, sourced from `node.temperatureBand` and `node.standingBand` respectively. The panel renders them as separate chips (SC-003 already does this — `sidecar-standing-band` / `sidecar-temperature-band` testIDs). IX-004 must not concatenate or cross-derive them.
- **Doctrine edge case — "what if a high reply count looks like the message is 'winning'?"** `replyCountLabel` is strictly a count ("3 direct replies"). It carries no verdict word. The ban-list test asserts this. Heat (activity) and standing (game support) stay separate; reply count is an activity signal, never a standing signal.

## Risks

- **`ArgumentGameSurface.tsx` is the only non-trivial production edit and it is a hot file.** The timeline-branch JSX is rebuilt: the sidecar moves from after the map to a panel before it. Existing tests that assert the sidecar renders *below* the map, or that assert a specific child order in `argument-game-surface`, may need updating. The implementer should grep `__tests__/` for `argument-reply-sidecar` ordering assertions and for any test that renders the timeline branch and checks layout order — `argumentReplySidecar.test.tsx` tests the component in isolation (safe), but any `ArgumentGameSurface` integration test that depends on sidecar position must be updated, not deleted.
- **`directReplyCount` definition drift.** "Direct reply" must mean *immediate children* (`parentId === selectedMessageId`), not descendants. If the implementer counts the whole subtree, the number will be wrong for deep threads. The test must pin immediate-child semantics with a 3-level fixture.
- **`accessibilityLiveRegion` double-announcement.** RN-web's `accessibilityLiveRegion="polite"` and a native `announceForAccessibility` call can both fire. The implementer must gate native announcement behind `Platform.OS !== 'web'` so a web build does not announce twice. Mirror the `try/catch` + cancelled-flag pattern from `ArgumentTimelineMap.tsx` L408–443.
- **Stale-status latching.** Once `selectionStatus` becomes `stale_fallback`, the *next* explicit user action (tap / Prev / Next / Latest / root / keyboard) must reset it to `explicit`, or the banner will linger after the user has moved on. Every selection-mutating callback must set the status — the test suite covers this.
- **Panel height on small viewports.** SC-003's `ArgumentReplySidecar` has `maxHeight: 360` and its own `ScrollView`. Wrapping it in the panel plus the stale banner + reply-count row adds vertical chrome. On a short phone the timeline map could be squeezed. - **Reduce-motion is not newly at risk** — IX-004 adds *no* animation. The only motion-bearing element (the node glow shadow) is VG-004's, already reduce-motion-gated. The implementer must not add a panel fade/slide; the panel snaps.
