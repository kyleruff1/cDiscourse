## Why this design doc exists

UX-001.6 is the visible quality baseline for the entire UX-001 epic. The five prior cards (UX-001.1 through UX-001.5) shipped product contracts in code: a brand-forward shell with band-aware breakpoints, a Timeline-first room viewport with hard offset caps, a persistent contextual composer with per-target draft persistence, a three-menu interaction model with browser-only key badges and dismissible scrim, and 12 source-neutral visual annotation primitives plus the additive `semanticFlagsChips` Inspect integration. UX-001.6 verifies those contracts hold across the device matrix in actual viewport behavior — not only through component-level prop tests.

The card is mostly QA, not feature work. Bounded visual polish (acute fixes surfaced by QA) is permitted within an operator-set budget of 5-10 fixes. Any pre-existing regression or larger scope-defect surfaced during QA is documented as a separate-card candidate, not folded into UX-001.6.

UX-001.6 is the last card before UX-001.7's visual design system consolidation. Its QA findings inform UX-001.7's scope: which surfaces passed cleanly, which needed acute fixes, which carry pre-existing regressions UX-001.7 must address.

The four mandatory skills (`cdiscourse-doctrine`, `test-discipline`, `expo-rn-patterns`, `accessibility-targets`) were invoked at the start of design. The four hard-blocker viewports (390×844, 1024×1366, 1366×768, 1920×1080) are binding from `expo-rn-patterns` §Cross-device QA viewport conventions. The Observations/Allegations boundary doctrine (`cdiscourse-doctrine` §10a) applies to every annotation chip the matrix verifies. The platform-conditional keyboard-hint pattern (`accessibility-targets` §Keyboard hints platform-conditional pattern) is the load-bearing accessibility contract.

## §2 — Viewport matrix per-surface verification plan (Q2)

The matrix is **6 viewports × 18 surfaces = 108 cells**. Each cell produces a verdict: pass / fail / finding.

### The 6 viewports

| # | Viewport | Label | Interaction context | Per-band |
|---|---|---|---|---|
| 1 | 390 × 844 | Phone (iPhone-class) | Touch-first; no key badges | phone band |
| 2 | 412 × 892 | Phone (large Android, **extension**) | Touch-first; no key badges | phone band |
| 3 | 768 × 1024 | Tablet portrait (iPad-class, **extension**) | Touch-first; no key badges | tablet band |
| 4 | 1024 × 1366 | Tablet portrait (iPad Pro 11-class) | Touch-first; native iPad; no key badges | tablet band |
| 5 | 1366 × 768 | Narrow browser (small laptop) | Keyboard-first; key badges render | wide band |
| 6 | 1920 × 1080 | Wide browser (desktop) | Keyboard-first; key badges render | wide band |

Cell encoding: `{ platformOs, windowWidth, windowHeight, band }` per the platform interpretations in audit #13. // For each viewport, the test asserts surfaces 1-18 via subtests.
```ts
for (const vp of VIEWPORTS) {
  describe(`UX-001.6 matrix — ${vp.label}`, () => {
    // Sub-suite per surface, ~3 assertions each (presence, sizing, doctrine).
  });
}
```

### Per-surface verification strategy

The matrix uses three verification techniques, chosen per surface:

**Technique A — pure-TS token / model assertion** (no render). Surfaces 1, 2, 4, 7, 8, 9, 12, 13 read token / model output directly and assert against the per-band contract. Faster, no JSDOM cost.

**Technique B — RN test renderer rendering** (with mocked `Dimensions` + `Platform`). Surfaces 5, 6, 14, 15, 16, 17 mount the actual component in `react-test-renderer` with a per-viewport mock and assert on the rendered tree.

**Technique C — file-scan boundary verification** (no render). Surface 3 (App.tsx structure), surface 17 (RefereeBannerView mount-site scope), surface 18 (EvidenceAnnotationChip presence) read the source files and assert on structure.

The matrix file may split into 2-3 sibling files if test count growth warrants it (e.g., `uxOneOneSixViewportMatrix.test.ts`, `uxOneOneSixViewportMatrixComposer.test.ts`, `uxOneOneSixViewportMatrixAnnotations.test.ts`). Designer-default: start with one file; implementer splits if a single file exceeds 600 lines.

### Pattern coexistence with existing matrix tests

`composerViewportMatrix.test.ts` (UX-001.3) and `uxOneOneTwoOffsetAcceptance.test.ts` (UX-001.2) remain unchanged. The new matrix complements them — it does not subsume. The new matrix's offset assertions cross-check the prior offset test's contract; the prior tests serve as the canonical contract source.

## §3 — Touch target compliance test plan (Q3)

Per `accessibility-targets` §Keyboard hints platform-conditional pattern: compact controls must pass **focus visibility + touch target size + screen-reader name** at every viewport.

### Test file

`__tests__/uxOneOneSixTouchTargets.test.ts` — a single suite that walks every interactive primitive across the UX-001 source surface and verifies 44×44.

### Verification approach

For each interactive primitive, the test:

1. Mounts the primitive in `react-test-renderer` with each viewport's mocked `Dimensions`.
2. Reads the rendered `Pressable` (or `TouchableOpacity`, though the repo uses `Pressable` per `expo-rn-patterns`).
3. Asserts EITHER: visual width ≥ 44 AND visual height ≥ 44; OR: `hitSlop` extends the effective tap region to ≥ 44 × 44 (computed as `width + hitSlop.left + hitSlop.right` ≥ 44 AND `height + hitSlop.top + hitSlop.bottom` ≥ 44).

### Interactive primitives in scope

From the UX-001 source surface (enumerated via `Grep "<Pressable"` over `src/features/nodeAnnotations/`, `src/features/arguments/composer/`, `src/features/arguments/oneBox/`, `src/components/AppHeader*`, `src/features/refereeBanners/`):

- **UX-001.1:** AppHeader's right-slot triggers
- **UX-001.2:** TimelineSelectedReadoutPanel expand button, ArgumentTimelineMap overlay controls (Prev / Next / Latest / Back-to-root / Cards-toggle)
- **UX-001.3:** ComposerContextStrip controls, CollapsedComposerStrip expand action, OneBox type chip, ArgumentComposer validation chip dismiss
- **UX-001.4:** Popout entries (`PopoutEntry`), Popout group headers (`PopoutGroup`), key-badge wrapped triggers, ActPopout / InspectPopout / GoPopout primary action rows
- **UX-001.5:** AnnotationChip, AnnotationOverflowChip (both have `hitSlop={{ top: 12, bottom: 12, left: 12, right: 12 }}` per UX-001.5 review §O — verified intact)

### Test count estimate

Roughly 6 viewports × ~25 primitives × 3 assertions (focus / size / SR-name) = ~450 generated tests. The Jest counter sees each `it()` as one test. The test file uses parameterized loops to keep the source file under 300 lines.

## §4 — Color independence test plan (Q4)

Per `accessibility-targets` §Color independence: shape, glyph, label distinguish state in a desaturated render.

### Test file

`__tests__/uxOneOneSixColorIndependence.test.tsx`

### Verification approach

A grayscale snapshot is functionally equivalent to: "render the surface with every color token replaced by a neutral gray, and verify the resulting tree still encodes its meaning through shape, glyph, label, and border."

In practice (per the existing `uxOneOneFiveColorIndependence.test.tsx` pattern at UX-001.5 review §N): assert that the rendered surface's accessibility labels + shape tokens (borderWidth, borderRadius, glyphs) carry sufficient distinguishing information without referencing color values.

### Surfaces in scope (at least 3 representative)

Per the brief's acceptance criterion 11:

1. **A chip strip** — `AnnotationChipStrip` rendered with mixed `kind` values (state / context / lifecycle / evidence / flag / semantic). Verify: each chip's `aria-label` contains its plain-language label + kind word; chip distinguishability via `borderWidth` + `glyph` is asserted; no color-only distinguishing.
2. **A timeline node selection state** — `ArgumentTimelineMap` node rendered in three states (default, active, focused). Verify: each state has a distinct `borderWidth` / `borderRadius` / `transform` value, not relying on color alone.
3. **A focus ring** — `AnnotationFocusRing` rendered with focus on / off. Verify: `borderWidth` change is the distinguishing signal, not color (the existing `uxOneOneFiveColorIndependence.test.tsx` already confirms; this matrix re-asserts at each viewport).

The test runs at each of the 6 viewports to confirm color independence is preserved cross-viewport (e.g., a chip strip that uses different padding at phone vs wide must remain color-independent at both).

### Test count estimate

3 surfaces × 6 viewports × ~3 assertions = ~54 generated tests.

## §11 — Risks and edge cases

### Common cross-device QA failure modes

**Viewport-specific layout regressions.** A surface that renders correctly at 390 × 844 may overflow at 412 × 892 due to a 22-px width difference that hits a wrap boundary. Mitigation: the matrix explicitly tests both phone widths; surface 2 (DebateDetailHeader) is the highest-risk surface for this class.

**React Native Web vs native parity.** A surface that renders correctly via `react-test-renderer` may behave differently on actual web due to flexbox quirks. The matrix's test-renderer assertions are necessary but not sufficient — operator-led real-browser smoke verification at UX-001.6 review time supplements the automated matrix.

**Color-mode rendering differences.** A surface that distinguishes states via color in light mode may fail in dark mode if the token swap doesn't preserve the discriminating signal. The color independence test (§4) mitigates by asserting shape/glyph/label distinguishability.

### The "may be active" stale branch question (informational)

Per the launch prompt, the operator's earlier cleanup session left open a question about whether stale branches (e.g., older UX-001.x worktrees) might inadvertently affect the QA verification. **Informational only — no action required of the designer or implementer.** The branch `feat/UX-001.6-cross-device-qa-and-visual-polish` is fresh from `main` at `f79da7c`; all QA assertions run against the current HEAD, not against branch state. If a prior-card worktree exists outside this branch, its state does not affect UX-001.6's diff or test runs.

### Test count delta expectation calibration

The designer-default estimate (§10) of ~1,250 new tests is conservative — the actual count depends on how aggressively the implementer parameterizes. If the implementer uses tighter parameterization (e.g., one `it()` per viewport-surface pair instead of per-assertion), the count could be ~108 + ~150 + ~18 + ~600 + ~50 ≈ 926 tests. If the implementer is more granular, the count could exceed 1,500. **The reviewer's gate is exit code = 0 + positive delta, not a specific count.** The implementer's completion report records the actual count; the reviewer verifies the count matches their re-run.

### Audit clarification risk (surface #13)

The audit found that the A/I/G key badge model's threshold (1024 dp web) means a tablet WEB build at exactly 1024 width WOULD render the badge. The brief's intent is that the 1024 × 1366 cell is native iPad Pro 11 portrait (`Platform.OS === 'ios'`), where the badge does NOT render. The QA matrix MUST encode `{ platformOs, windowWidth }` per cell, not just viewport dimensions. **Risk:** if the implementer encodes only `windowWidth`, the matrix would report a false-positive failure at the 1024 cell (claiming badges should be absent when the model would render them on the web build at that width). The §2 verification plan explicitly enumerates `platformOs` per cell to prevent this.

### EvidenceAnnotationChip conditional risk

The audit's surface #18 finding is conditional: if the matrix surfaces a regression, UX-001.5C is filed separately. The implementer's risk is choosing to fold a regression into UX-001.6's acute-fix budget when the brief's §13 explicitly forbids `EvidenceAnnotationChip` refactor. **Mitigation:** the implementer treats any EvidenceAnnotationChip regression finding as a separate-card candidate, never as an acute fix slot. The reviewer's verdict matrix item O catches budget violations.

### Operator-deferred review surface

The §13 brief ledger names items where the operator's review at completion may surface judgment changes (e.g., the 5-10 budget ceiling, the 2 extension viewport selection). The implementer's completion report MUST surface these explicitly so the operator can evaluate at review time. **Mitigation:** the completion report cross-references the brief ledger and notes any divergence between designer-default resolution and observed implementer behavior.
