## 1. Goal

MCP-001 designs the **semantic-referee architecture** for CDiscourse and locks the **binary classifier contract** — the `SemanticRefereePacket` / `SemanticBinarySample` shapes — that the rest of the MCP-** family depends on.

The semantic referee is a provider-agnostic layer that, at a small number of strategic moments, asks a low-cost model a handful of tightly bounded yes/no questions about the **structure** of an argument move — "did this answer the parent?", "is this a branch or a reply?", "does this evidence attach to the claim?" — and returns a compact packet of `0/1` booleans, bounded enums, short reason codes, confidence, and prompt/model versions. That packet is **advisory metadata only**. It feeds the deterministic game engine, route suggestions, evidence-pressure prompts, "you are here" map context, and playful referee banners. It is `authoritative: false`, always.

The layer exists to make the argument game feel like it has an **objective referee keeping points** — objective about the **rules of play**, never about the **truth**. It is the concrete, MCP-framed answer to RULE-006 (#116), which asked whether and how CDiscourse should use a semantic AI metadata layer. The doctrine that shapes every decision below: the referee scores quality of play (continuity, evidence hygiene, branch hygiene, constructive movement) and never tells anyone they are right or wrong.

## 2. Product thesis

CDiscourse is a PvP argument game. The first-generation room used deterministic keyword matching to gate posts and it produced false positives that made the room feel like a cage — users had to write with "magic keywords" or be rejected. Stage 6.2 correctly made that layer advisory. But that left a real gap: deterministic code reliably answers *button-and-artifact* questions ("did the user tap Ask source?", "is there an evidence row?") and cannot reliably answer *structural-meaning* questions ("did this paragraph engage the parent's mechanism, or just restate a slogan?").

The semantic referee fills exactly that gap and only that gap. It makes argument **structure** harder to manipulate — volume, topic drift, insult, and crowd appeal naturally route to lower-priority lanes — without making the user feel caged, punished, or forced to write with keywords. A funny, contrarian, devil's-advocate, or spicy-but-on-topic move is recognised as **playable**, not suppressed. A move that changes the subject is offered a **branch**, not rejected. A claim that needs evidence raises **source-chain pressure**, not a truth verdict.

The objective judge is objective about the **rules of play**, not about **truth**.

## 3. Doctrine constraints

From `cdiscourse-doctrine` and the MCP roadmap expansion §3. A violation is a blocker.

1. **AI never decides who is right.** No classifier in the catalog asks a truth question.
2. **AI never declares a winner / loser.** The referee scores rules of play, never truth.
3. **AI never blocks an ordinary post by itself.** Only deterministic structural validation (empty body, invalid transition, explicit-evidence-without-source, over-length, auth failure, opted-into cooldown) blocks. The semantic packet can at most surface advisory friction.
4. **AI never runs on the client.** Layer 2 executes only in an Edge Function or a server-side MCP adapter.
5. **`authoritative` is always `false`** on every `SemanticRefereePacket` and every `SemanticBinarySample` — enforced by the TypeScript literal type and a schema test.
6. **Popularity / heat / virality / crowd reaction are never evidence.** `uses_popularity_as_evidence` classifies the *text move*; it never ranks the crowd. Engagement credit and factual-standing credit stay separate (per `antiAmplification.ts`).
7. **Evidence / source-chain state affects gameplay pressure, never truth verdicts.**
8. **No verdict / person labels** — never *liar, dishonest, bad faith, manipulative, troll, propagandist, wrong, right, true, false, winner, loser, proven, defeated, extremist, stupid, idiot*. Classify the move and the text behavior, never the person.
9. **Mocked by default in tests, fixture-driven in development.** Live calls require a separate operator-approved card.
10. **No `snake_case` internal codes in user-facing strings.** Every `classifierId` and `reasonCode` maps through a plain-language catalog (extends `gameCopy.toPlainLanguage`).

## 4. Trigger points (when the classifier MAY run)

The semantic call is allowed only at these moments (full gating plan is MCP-004):

- **Post-submit classification** — after a move is posted, classify it (feature-flagged per room mode). This is the default and the cheapest-to-reason-about trigger: the move already exists, nothing is blocked.
- **Pre-send review** — only in strict room modes the room creator explicitly opted into (GAME-003 `semanticClassification: 'metadata_and_chip'`). Even here the packet only feeds advisory friction (RULE-004's `PreSendReviewSheet`); it never hard-gates.
- **Evidence-inspection classification** — when a user attaches or challenges an evidence artifact, classify whether the evidence attaches to the claim.
- **Branch-routing classification** — when the deterministic layer cannot tell whether a move replies to the parent or redirects, classify for a `routeSuggestion`.
- **Synthesis-readiness classification** — once enough lifecycle events exist on a cluster, classify whether synthesis is genuinely ready.
- **Referee-feedback generation** — banners are generated **from already-classified metadata only**, never from a fresh raw-conversation scrape.

## 5. Non-trigger points (when the classifier MUST NOT run)

- Never on **keystroke** / draft edit.
- Never on **hover**.
- Never on **timeline node selection**.
- Never during **observer-only browsing**.
- Never **twice** for the same `{roomId, parentId, contentHash, promptVersion, classifierSet}` — the cached result is returned.
- Never from the **client**.
- Never on a move in a room mode whose `semanticClassification` is `'off'`.

## 6. Deterministic metadata inputs (layer 1)

Layer 1 runs first, costs nothing, and is authoritative for what it covers. It reads existing models only — no new persistence. Inputs:

- button / quick action selected (`quickActionPresets.ts`);
- reply type / `argumentType`;
- side (`affirmative` / `negative` / `observer` / `moderator`);
- actor role (initiator / primary opponent / chime-in / observer — GAME-004 `roomContractModel.ts`);
- selected parent id + target excerpt;
- manual tags + auto metadata (META-001);
- source-chain status + evidence artifact state (EV-001 `evidenceModel.ts`);
- lifecycle state (LIFE-001);
- branch state (BR-001 `branchTopologyModel.ts`);
- room mode (GAME-003);
- turn count + pacing / rate-limit state (GAME-002 `pacingModel.ts`).

Layer 1 outputs obvious flags with **zero cost and full confidence** — e.g. `selectedAskSource`, `selectedClarify`, `selectedConcede`, `hasAttachedEvidence`, `hasQuote`, `hasParent`, `sameUserRepeated`, `publicRoom`, `observer`, `primaryOpponent`, `jumpBranchUsed`. The semantic layer (layer 2) is only invoked for questions layer 1 **cannot** answer reliably. Where both layers answer the same question, layer 1 wins on its own turf and layer 2's value is reconciled (§14).

## 7. Semantic classifier output schema (layer 2)

The semantic call returns a small strict JSON object — never a naked `1`/`0`. The contract:

```ts
export type Binary01 = 0 | 1;

export type SemanticConfidence = 'low' | 'medium' | 'high';

export type SemanticClassifierId =
  | 'responds_to_parent'
  | 'introduces_new_issue'
  | 'asks_for_evidence'
  | 'provides_evidence'
  | 'evidence_supports_claim'
  | 'quote_anchors_parent'
  | 'narrows_claim'
  | 'concedes_narrow_point'
  | 'requests_clarification'
  | 'answers_clarification'
  | 'shifts_to_person_or_intent'
  | 'uses_popularity_as_evidence'
  | 'contains_playable_hot_take'
  | 'contains_unplayable_insult_only'
  | 'is_satire_or_parody'
  | 'uses_satire_as_evidence'
  | 'cites_retraction'
  | 'creates_source_chain_gap'
  | 'suggests_side_branch'
  | 'suggests_diagonal_tangent'
  | 'fits_selected_debate_mode'
  | 'needs_pre_send_pause'
  | 'ready_for_synthesis';

export type SemanticBinarySample = Readonly<{
  classifierId: SemanticClassifierId;
  value: Binary01;
  confidence: SemanticConfidence;
  reasonCode: string;        // bounded, snake_case, mapped to plain language before UI
  evidenceSpan?: string;     // optional short span from the move
  parentSpan?: string;       // optional short span from the parent
}>;

export type SemanticRouteSuggestion =
  | 'mainline'
  | 'vertical_chime_branch'
  | 'diagonal_tangent'
  | 'outer_realm'
  | 'cards_detail'
  | 'synthesis_lane'
  | 'no_route_change';

export type SemanticFrictionSuggestion =
  | 'none'
  | 'soft_chip'
  | 'pre_send_pause'
  | 'ask_for_quote'
  | 'ask_for_source'
  | 'suggest_branch'
  | 'suggest_narrow'
  | 'cooldown_notice';

export type SemanticRefereePacket = Readonly<{
  packetVersion: 'mcp-semantic-referee-v0';
  promptVersion: string;
  modelVersion: string;
  provider: 'mock' | 'mcp' | 'anthropic' | 'operator-selected';
  authoritative: false;                    // literal — never anything else
  inputHash: string;                       // hash of the full classifier input
  contentHash: string;                     // hash of the move body (cache key component)
  roomId: string;
  moveId?: string;
  parentId?: string;
  selectedAction?: string;
  selectedMoveType?: string;
  debateMode?: string;
  binaries: readonly SemanticBinarySample[];
  routeSuggestion: SemanticRouteSuggestion;
  frictionSuggestion: SemanticFrictionSuggestion;
  scoreHints: Readonly<{
    continuityCredit: number;
    evidencePressure: number;
    branchHygiene: number;
    synthesisReadiness: number;
    sourceChainDebt: number;
    unresolvedRedirectRisk: number;
  }>;
}>;
```

**Contract rules:**
- Every binary is `0` or `1` — never a float, never a string verdict.
- `confidence` is the model's **own uncertainty**, not a verdict and not a truth probability.
- `authoritative` is the literal `false`. The type makes any other value a compile error.
- `reasonCode` is a bounded `snake_case` token from the classifier's documented reason-code family; it is mapped to plain language before it can reach a user surface.
- `scoreHints` are integers in a small bounded range; they are *hints* for MCP-003's ledger, not score deltas.
- The MCP/AI layer **cannot alone block posting** — there is no `block` field and no truth field anywhere in the packet.
- On a layer-1 / layer-2 conflict, the deterministic ledger downgrades confidence and surfaces a reversible user choice (§14, §15) — it never silently overrides the user.

Why a tiny object rather than a naked `1`/`0`: the `0/1` lets the deterministic ledger (MCP-003) score cheaply, while `reasonCode` + `confidence` let layer 3 handle ambiguity, conflict, and false positives **without making the user feel trapped**.

## 8. Binary classifier catalog v0

The catalog v0 is the **strategic minimal set** — the smallest set that gives the game leverage. It maps 1:1 onto the `SemanticClassifierId` union. The full 90-seed prompt library is MCP-002 / the seed bank doc; the catalog here is the curated subset that ships in v0. Every classifier defines: **trigger moment · inputs required · deterministic fallback · user-facing copy family · no-go cases**.

### Family 1 — Parent continuity
| classifierId | `value: 1` when | Deterministic fallback | No-go |
|---|---|---|---|
| `responds_to_parent` | the move engages the parent's claim / mechanism / question / evidence | parent-id present + same-axis tag | never infer dodging-as-character |
| `quote_anchors_parent` | the move quotes/paraphrases a parent span and addresses it | `hasQuote` + excerpt overlap | a decorative quote → `0`, not a verdict |
| `answers_clarification` | the parent asked a question and the move gives a usable answer | lifecycle `clarified` | — |
| `introduces_new_issue` | the move raises a separately-debatable issue | new-axis auto tag | a new issue is not "off-topic-as-fault" |

### Family 2 — Branch routing
| classifierId | `value: 1` when | Deterministic fallback | No-go |
|---|---|---|---|
| `suggests_side_branch` | a chime-in adds a same-topic perspective best on a vertical branch | BR-001 branch kind | routing, never demotion-as-punishment |
| `suggests_diagonal_tangent` | two active participants drift to a related distinct issue | BR-003 `assessTangentRisk` | — |
| `fits_selected_debate_mode` | the move's register matches the room mode | GAME-003 mode profile | mode-fit, never tone-policing the person |
| `contains_unplayable_insult_only` | the move is only an insult with no answerable claim | — | flags the *move*; the user still posts |

### Family 3 — Evidence pressure
| classifierId | `value: 1` when | Deterministic fallback | No-go |
|---|---|---|---|
| `asks_for_evidence` | the move requests a source / receipt / origin | `selectedAskSource` | — |
| `provides_evidence` | the move includes / references an evidence artifact | `hasAttachedEvidence` | — |
| `evidence_supports_claim` | the attached evidence supports the exact claim made | — | "supports" ≠ "true"; gameplay pressure only |
| `creates_source_chain_gap` | the move asserts an evidence-bearing claim with no source path | `sourceChainStatus = no_source` | gap ≠ falsehood |
| `uses_popularity_as_evidence` | likes / virality / "everyone says" is doing evidentiary work | anti-amplification lexicon | classifies text, never the crowd or person |
| `cites_retraction` | the move cites a retraction / correction / changed record | — | — |
| `uses_satire_as_evidence` | satire / parody is used as factual support | — | — |
| `is_satire_or_parody` | the cited item appears to be satire / parody / fiction | — | a label on the *cited item*, not the user |

### Family 4 — Constructive movement
| classifierId | `value: 1` when | Deterministic fallback | No-go |
|---|---|---|---|
| `narrows_claim` | the move limits a broader claim to a defensible scope | `narrowed_claim` tag | — |
| `concedes_narrow_point` | the move accepts a specific limited point | `concession_offered` tag | render as "concession offered", never "lost" |
| `requests_clarification` | the move asks what the other person means | `selectedClarify` | — |
| `ready_for_synthesis` | clear shared ground + limited unresolved debt | lifecycle `synthesis_ready` | premature synthesis → `0` |

### Family 5 — Friction & pacing
| classifierId | `value: 1` when | Deterministic fallback | No-go |
|---|---|---|---|
| `needs_pre_send_pause` | the move could be repaired by the author before sending | strict-mode + structural risk | only surfaces in opted-in strict modes |
| `shifts_to_person_or_intent` | the move targets the speaker / their motive, not the claim | — | flags the *move*; never names a person label |
| `contains_playable_hot_take` | the move is spicy / contrarian / funny but still a coherent claim | — | hot takes are playable, never suppressed |

Every classifier's full seed (binary condition, reason-code family, false-positive risk, copy mapping) is in the seed bank doc. Catalog v0 is **23 classifiers**; the seed bank holds 90 so the catalog can grow without a contract change.

## 9. Prompt packet structure

Two prompt shapes; both demand strict JSON, both state the doctrine in the system prompt. Full templates in the seed bank doc §"Prompt shapes".

**Single-classifier prompt** — system prompt: *"You are a CDiscourse semantic classifier. Return strict JSON only. You do not decide who is right. You classify whether a move has one structural property. The output is advisory metadata. Ordinary posts are not blocked by this result."* User payload carries `classifierId`, `debateMode`, redacted `parent`, redacted `move`, `allowedValues: [0, 1]`. Output: one `SemanticBinarySample`.

**Five-classifier packet prompt** — system prompt: *"You are a CDiscourse semantic referee packet generator. Return strict JSON only. You may fill only the requested classifiers. Values must be 0 or 1. You do not evaluate truth. You classify game structure."* User payload carries `requestedClassifiers` (≤ 5), `roomContext`, `parentContext`, `moveContext`. Output: a partial `SemanticRefereePacket` (`binaries`, `routeSuggestion`, `frictionSuggestion`, `scoreHints`, `authoritative: false`).

Move and parent bodies are **redacted** before they enter a prompt (§20 safety scans).

## 10. Provider strategy

Provider-agnostic by construction. The `provider` field on the packet is one of `'mock' | 'mcp' | 'anthropic' | 'operator-selected'`.

- **`mock`** — deterministic stub; the **default in all tests** and the fallback when the feature flag is off. Produces a schema-valid packet from the deterministic layer alone.
- **`anthropic`** — a low-cost model (Haiku-class) reached through the existing `process-language-draft` Edge Function provider-registry pattern (`providers.ts`).
- **`mcp`** — an operator-hosted MCP classifier service, reached server-side.
- **`operator-selected`** — a placeholder for a future provider the operator wires.

**No default live provider is chosen in this design.** The recommendation (operator decision §23): provider-agnostic abstraction with the mock provider as the test/dev default, and the live provider decided only at the live-pilot card. This mirrors RULE-006's flagged operator decision.

## 11. MCP / Edge Function boundary

Boundary design is MCP-009; MCP-001 states the invariants the boundary must hold:

- The provider call lives in an **Edge Function or a server-only MCP adapter** — never the client.
- Three boundary options for MCP-009 to evaluate: (a) extend `process-language-draft` with a `classify-move` action; (b) a new dedicated `semantic-referee` Edge Function; (c) a server-only MCP adapter the Edge Function calls.
- The provider key is read only via `Deno.env.get()` inside the Edge Function; never returned to the caller, never logged, never in `.env` / `app/` / `src/`. Env var **names** may be documented; values are never read or printed.
- When the feature flag is off, the function returns `{ enabled: false }` and the client falls back to the deterministic layer with no error.
- No service-role key in client code; no direct insert into `public.arguments`.

## 12. Caching and batching

- **Cache key:** `{roomId, parentId, contentHash, promptVersion, classifierSet}`. A packet for an identical key is returned from cache; the provider is not called again.
- **Cache invalidation:** only on a `promptVersion` bump. A move body never changes after posting (soft-delete only), so `contentHash` is stable; pre-send drafts are not cached (they are not posted).
- **Batching:** up to **5 classifiers per call** in one packet prompt. The batching policy groups classifiers that share input context (e.g. the four parent-continuity classifiers travel together).
- **Cache location:** server-side (Edge Function scope); the design does not add a client cache. A persisted cache table is a future-card option, not v0.

## 13. Cost-control plan

- **Trigger gating** (MCP-004) is the primary cost control — post-submit only by default, never on keystroke / hover / selection / observer browsing.
- **Batching** — 5 classifiers per call collapses what would be 5 calls into 1.
- **Caching** — the `{roomId, parentId, contentHash, promptVersion, classifierSet}` key guarantees each unique move is classified at most once per prompt version.
- **Token budget** — a documented max input + output token ceiling per packet; the boundary refuses to send an over-budget payload (operator confirms the number — §23).
- **Model tier** — a low-cost (Haiku-class) model for the binary catalog; the expensive tier is reserved for on-demand synthesis summaries, which are out of scope here.
- **Feature flag** — the whole layer is off by default; a room mode whose `semanticClassification` is `'off'` makes zero calls.
- **Retry/fallback** — bounded retries; on failure the deterministic layer answers and no call is retried indefinitely.

## 14. False-positive mitigation

The original keyword cage was a false-positive machine; the semantic layer must not repeat it.

- **Every semantic binary is paired with a deterministic derivation where one exists.** The two are reconciled.
- **On agreement** — confidence stands; the ledger uses the value.
- **On conflict** — confidence is **downgraded** and a **reversible user choice** is surfaced (MCP-010), never a penalty and never a silent override.
- **Low confidence never hard-acts** — a `low`-confidence binary produces a softer banner or no banner (MCP-008) and never moves score on its own.
- **`friction_would_feel_punitive` is itself a classifier** — if the layer judges that a proposed pause would feel arbitrary given the available metadata, the friction is downgraded.
- **`manual_override_should_be_offered`** — when the layer is uncertain, it explicitly recommends letting the user pick the lane.
- **No classifier hard-gates** — there is no path from a semantic binary to a blocked ordinary post.

## 15. Manual override flow

Owned by MCP-010; MCP-001 states the contract:

- When packet confidence is `low`, or a layer-1 / layer-2 conflict exists, the UI surfaces a reversible choice instead of acting silently.
- The user can pick the lane (mainline / branch / tangent) or explicitly assert "this answers the parent" to overturn a `responds_to_parent: 0`.
- The override is recorded as **visible metadata** (a META-001-style `semantic_override` entry), not hidden state.
- Overriding carries **no score cost, no flag, no "you were warned" copy**.
- Repeated overrides may inform future UX tuning but **do not train any live model in v1**.

## 16. Score / referee ledger integration

Owned by MCP-003; MCP-001 states the seam:

- The packet's `scoreHints` block (`continuityCredit`, `evidencePressure`, `branchHygiene`, `synthesisReadiness`, `sourceChainDebt`, `unresolvedRedirectRisk`) are **hints**, not deltas. MCP-003's deterministic ledger decides the actual score movement.
- The ledger consumes layer 1 + layer 2 + the existing `src/features/pointStanding/` economy. The semantic packet is **one input among many**, never the master.
- The ledger may award play-quality points for continuity, evidence relevance, narrowing, concession, clarification, synthesis, branch hygiene, resolving evidence debt, staying in mode, respecting pacing. It **must not** award for popularity, heat, volume, humour alone, rhetorical force alone, perceived truth, perceived political alignment, crowd vote, user identity, or user reputation.
- Engagement credit and factual-standing credit stay separate (per `antiAmplification.ts`).

## 17. UI contracts

The packet drives, via plain-language mapping only:

- **Route chips** — `routeSuggestion` becomes a chip ("This looks like a branch, not a reply" → tap to branch). Advisory; the user can ignore it.
- **Friction surfaces** — `frictionSuggestion` feeds RULE-004's `PreSendReviewSheet` in opted-in strict modes; in casual modes it is at most a soft post-submit chip.
- **"You are here" context** — `routeSuggestion` + branch state feed IX-004's readout panel.
- **Referee banners** — classifier ids + `scoreHints` map to MCP-008's banner library.
- **Sidecar semantic flags** — SC-003's "Semantic flags" section renders plain-language classifier results, deeper in Cards mode.

**Hard rule:** a raw `classifierId` or `reasonCode` **never** reaches a user surface. Every code is mapped through a plain-language catalog (extends `gameCopy.toPlainLanguage`); an unmapped code is suppressed, not echoed.

## 18. Prompt versioning

- Every packet carries `promptVersion` (the seed-bank prompt revision) and `modelVersion` (the provider's model id string).
- `packetVersion` (`'mcp-semantic-referee-v0'`) versions the *contract shape*; a contract change bumps it and is a new design card.
- A `promptVersion` bump invalidates the cache (§12) — a re-worded prompt may classify differently, so old cached packets must not be reused.
- The seed bank doc is the source of truth for the current `promptVersion`; each seed records the version it was last revised in.

## 19. Test plan

All tests are mock-provider / fixture-driven; **no test makes a live call.** For the future implementation card:

- `__tests__/semanticRefereePacket.test.ts` — schema validation; `authoritative` is the literal `false`; `Binary01` domain; `routeSuggestion` / `frictionSuggestion` enum domains.
- `__tests__/semanticClassifierCatalog.test.ts` — every catalog-v0 classifier has trigger / inputs / fallback / copy family / no-go documented; ids unique; ids ⊆ `SemanticClassifierId`.
- `__tests__/semanticPacketBanList.test.ts` — ban-list across every `reasonCode` and every produced user-facing string (verdict + person tokens).
- `__tests__/semanticMockProvider.test.ts` — mock provider is deterministic and schema-valid.
- `__tests__/semanticNoCallOnDraftEdit.test.ts` — keystroke / hover / selection / observer browsing emit zero calls (cost-control invariant).
- `__tests__/semanticConflictDowngrade.test.ts` — a layer-1 / layer-2 conflict downgrades confidence and surfaces a choice, never a penalty.
- `__tests__/semanticForbiddenImports.test.ts` — the pure model imports nothing from Supabase / React / network.

## 20. Safety scans

- **Body redaction** — move and parent bodies are redacted before entering a prompt: strip X handles, URLs (`x.com` / `t.co` / `twitter.com` and bare hosts), JWT-shaped strings, Bearer / Authorization headers, `sk-ant-*` / `xai-*` / `sb_secret_*` token shapes, emails, 15–20 digit post ids. Reuse the existing sanitiser patterns from `scripts/engagement-intelligence/`.
- **Secret scan** — `grep -r "ANTHROPIC_API_KEY\|SERVICE_ROLE" app/ src/` must return zero matches; a source-scan test enforces it.
- **Ban-list scan** — verdict + person tokens are scanned out of every seed field, every reason-code family, every banner string, and every produced user-facing string.
- **No-secret-logging** — the boundary never logs the Authorization header, the provider key, or a raw provider response body.
- **No `.env*` access** — the design and its future implementation read env var **names** only; values are read solely via `Deno.env.get()` inside the Edge Function.
