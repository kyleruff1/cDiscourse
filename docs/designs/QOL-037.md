## 0. How this design was built

This design is **derived from the storyboard, not asserted.** It is the behaviour
spec for one box type — the QOL-030 `respond_to_evidence` box. It is built in
three grounded passes, and the model falls out as the consequence:

1. **The storyboard frames** (§3) — Steps 6–13 of the band-rent scenario replayed
   frame-by-frame, recording exactly which structured choice was made, what
   clarification was required, and what status the evidence object moved to.
2. **The axis separation** (§4) — the doctrine that *existence*, *source-chain*,
   and *applicability* are three independent axes on one evidence object, and
   QOL-037 owns only the third.
3. **The choice set** (§5) — the seven structured responses, the required-
   clarification rule, and the applicability status ladder, derived as the
   minimal vocabulary the storyboard frames demand.

Where the storyboard and a prior design disagree, **the storyboard wins** — it
is the binding source per `docs/ux-storyboards/`. QOL-037 produces no new bespoke
surface; it is content plugged into QOL-030's `renderSchema` and read by QOL-032.

## 1. Goal & scope

Add the **evidence applicability axis** — a structured way to dispute *what* a
piece of evidence applies to (e.g. which month a payment covers), separate from
disputing that the evidence exists or that its source chain is anchored.

QOL-037 ships the **behaviour spec** for the QOL-030 `respond_to_evidence` box
type. It defines:

- A structured **evidence-response choice set** of seven choices: `accept`,
  `accept_with_caveat`, `dispute_date`, `dispute_amount`, `dispute_applicability`,
  `request_source`, `request_clarification`.
- The rule that **any non-`accept` choice requires a clarification body** — a
  structured form, not a free-text afterthought.
- An **applicability status axis** on the evidence object: `applicability_undisputed`
  → `applicability_disputed` → `applicability_supported`, distinct from the
  EV-001 `SourceChainStatus` (existence / source-chain) and the EV-003 evidence
  debt (obligation) axes.
- The doctrine that disputing applicability **changes a status, never declares
  the evidence false.** Copy is "Applicability disputed" — never "proof",
  "true", "false", "winner", "loser", "wrong".

**In scope:**

- A pure-TS model (`src/features/evidence/evidenceApplicabilityModel.ts`): the
  choice-set type, the applicability status type, the validation rule (non-accept
  → clarification required), the status-transition function, and the
  view-model the `respond_to_evidence` box renders.
- The locked plain-language copy for every choice label, helper, and status chip.
- The schema-renderer entry that QOL-030's `renderSchema('respond_to_evidence',
  evidenceObject)` returns.
- The Inspect-popout (QOL-032 §E) read-view fields for the applicability axis.
- The `submit-argument` payload extension (an optional `evidenceResponse` block
  on the draft) — designed here, contract-only, no Edge Function rewrite beyond
  passing the new advisory block through into the validation snapshot.

**Out of scope (own cards):**

- The QOL-030 box chassis / flash-menu / popout primitives themselves.
- The QOL-036 payment-evidence metadata fields (amount, date, redacted
  payer/payee, claimed applicability) — QOL-037 *reads* them but does not define
  them. If QOL-036 is not yet built, see §11 fallback.
- The EV-003 evidence-debt obligation layer — `request_source` from this choice
  set *opens* an EV-003 debt; QOL-037 does not re-implement the debt tracker.
- The EV-002 source-chain popover — `request_source` reuses EV-002's preset.
- The QOL-041 concession acceptance gradient — a separate forced-list axis.
- A persisted `evidence_applicability` table — v1 derives applicability status
  at render time from `arguments` rows + the new advisory `evidenceResponse`
  block (mirrors EV-001 / EV-003 "no migration in v1").
- Any AI assessment of whether a dispute is correct.

## 3. Storyboard frames — applicability dispute in action

Steps 6–13 of the band-rent scenario, replayed as the box state + the model
output. Columns: actor → selected evidence object → `respond_to_evidence` choice
→ clarification → resulting applicability status.

| Step | Actor | Target evidence | Choice | Clarification required? | Applicability status after |
|---|---|---|---|---|---|
| 6 | A | B's $120 payment object (Node 2) | `dispute_applicability` | **yes** — A writes the "practice space grouping / March 10 due date" body | `applicability_undisputed` → `applicability_disputed` |
| 8 | B | A's Node 3 challenge | `accept_with_caveat` | **yes** — B writes "right that rent was due Mar 10, but I paid early" | unchanged on the payment object (the caveat targets A's claim, not B's evidence) — see §6 edge case 4 |
| 9 | A | B's caveat (Node 4) | `request_source` | (the request *is* the body — a short prompt, EV-002 preset) | unchanged; **opens an EV-003 debt** |
| 10 | B | — supplies the group-chat message as a *new* evidence object | (an `add_evidence` move, QOL-036 — not a `respond_to_evidence`) | n/a | the first payment object: `applicability_disputed` → `applicability_supported` |
| 11 | A | B's caveat | `accept` (two concessions) + a refutation introducing the **amount** sub-axis | **no** for the `accept`; the refutation is a separate `respond` move | the payment object's applicability dispute is effectively resolved |
| 12 | B | A's amount refutation | `dispute_amount` | **yes** — B writes "the increase started in April; March was still $120" | n/a (the amount axis — a different evidence object, the landlord message) |
| 13 | A | B's landlord-message evidence | `accept` + a settlement summary | **no** | the room settles; status is read-only, labelled "Evidence accepted by both" |

### 3.1 Findings forced by the frames

- **F-Step6 → applicability is a status, not a verdict.** A's `dispute_applicability`
  move flips the payment object to `applicability_disputed`. The model never
  records "A is right" / "the evidence is false" — it records that a *dispute on
  the applicability axis is open*. The settlement at Step 13 is the only thing
  that closes it, and even then the label is "accepted by both", not "true".
- **F-Step6 → the clarification is structurally required.** The box cannot post
  a non-`accept` choice with an empty clarification body. This is a *validation*
  rule (it can block the post) — distinct from a *score* rule (which never
  blocks). It is the same kind of structural requirement as a forced-list having
  item fields (QOL-030 D6).
- **F-Step8 → `accept_with_caveat` is a first-class choice.** A caveat is "I
  accept this, but —". It is NOT `dispute_*` (it does not contest the evidence's
  date/amount/applicability); it is NOT `accept` (it carries a reservation). It
  needs its own choice with its own required clarification.
- **F-Step9 → `request_source` is a bridge, not a duplication.** Choosing
  `request_source` from the `respond_to_evidence` choice set opens an EV-003
  evidence debt and reuses EV-002's `ASK_SOURCE_PRESET_BODY`. QOL-037 does not
  re-implement either — it routes to them.
- **F-Step10 → applicability improves via a *later move*, never a retroactive
  edit.** When B supplies the group-chat message, the first payment object's
  applicability status moves to `applicability_supported`. This is computed at
  render time from the move history; the original evidence row is never mutated
  (consistent with EV-003 "debt is resolved by later moves, not edited into the
  original" and the immutable-`arguments`-body rule).
- **F-Step13 → `accept` is the only zero-clarification choice.** Every other
  choice carries a reservation or a dispute that the other party needs spelled
  out. `accept` is unconditional; a clarification would be noise.

## 4. The three axes — what QOL-037 owns and does not own

An evidence object carries **three independent status axes**. The storyboard's
"Evidence object model" section is explicit that these must be separable so a
dispute is precise.

| Axis | Type | Owner | Question it answers |
|---|---|---|---|
| **Existence / source-chain** | `SourceChainStatus` (EV-001) — `no_source` / `unverified` / `source_no_quote` / `source_and_quote` / `broken` / `primary_present` | EV-001, EV-002 | Does an inspectable record exist, and how anchored is its trail? |
| **Obligation (evidence debt)** | EV-003 debt status — `requested` / `supplied` / `challenged` / `accepted_by_both` / … | EV-003 | Has a source/quote/receipt/context been *asked for* and is it owed? |
| **Applicability** | `ApplicabilityStatus` (QOL-037 — **new**) — `applicability_undisputed` / `applicability_disputed` / `applicability_supported` | **QOL-037** | Does the evidence apply to *what the submitter says it applies to* (which month / which obligation)? |

These never collapse into one another:

- A payment object can be `source_and_quote` (the screenshot is inspectable) **and**
  `applicability_disputed` (we disagree about which month it covers). A clear
  trail does not settle applicability.
- A payment object can be `applicability_supported` **without** ever becoming
  `primary_present` — applicability is supported by *argument and corroborating
  moves*, not by the source chain reaching a primary record.
- Heat never feeds any of the three. None of the three is a truth label.

> **Doctrine edge case the model must encode:** "what if a strong source chain
> auto-resolves the applicability dispute? — it does not." `source_and_quote`
> tells you the screenshot is real; it says nothing about which month it was
> for. The two axes are computed independently and never read each other.

## 5. The structured evidence-response choice set

The `respond_to_evidence` box renders exactly **seven** choices. The set is
locked at design time. Each choice has: an internal id, a plain-language label,
a plain-language helper, whether it requires a clarification body, what
applicability transition it drives (if any), and where it routes.

| Choice id | Label | Helper | Clarification required | Applicability transition | Routes to |
|---|---|---|---|---|---|
| `accept` | "Accept evidence" | "This evidence holds up for me as presented." | **no** | none (or, if a dispute was open, contributes to `applicability_supported` — see §6.1) | a `respond_to_evidence` move, type `evidence_acceptance` |
| `accept_with_caveat` | "Accept with caveat" | "Mostly fine — note one reservation alongside it." | **yes** | none | a `respond_to_evidence` move, type `evidence_acceptance` |
| `dispute_date` | "Dispute the date" | "The date on this record is not what it's being used for." | **yes** | none (date is a separable axis; QOL-036 owns the date field) | a `respond_to_evidence` move, type `evidence_challenge` |
| `dispute_amount` | "Dispute the amount" | "The amount is not what it's being used for." | **yes** | none (amount is a separable axis; QOL-036 owns the amount field) | a `respond_to_evidence` move, type `evidence_challenge` |
| `dispute_applicability` | "Dispute what it applies to" | "The record is real, but it applies to something other than the point being made." | **yes** | `applicability_undisputed` → `applicability_disputed` | a `respond_to_evidence` move, type `evidence_challenge` |
| `request_source` | "Ask for the source" | "Ask where this record comes from." | (the request body is seeded — EV-002 preset) | none | reuses EV-002 `ASK_SOURCE_PRESET_BODY`; **opens an EV-003 `source` debt** |
| `request_clarification` | "Ask for clarification" | "Ask the author to explain how this record supports the point." | **yes** | none | a `respond_to_evidence` move, type `clarification_request` |

Notes the implementer MUST observe:

- **`dispute_date` and `dispute_amount` do NOT drive an applicability transition.**
  They are separate axes (QOL-036's `date` / `amount` fields). They share the
  `respond_to_evidence` box because the storyboard groups them under one choice
  set, but only `dispute_applicability` touches `ApplicabilityStatus`. Disputing
  the date/amount of a record is "this field is wrong for the use", not "the
  applicability axis is contested". The model keeps these strictly separate.
- **`request_source` clarification.** Unlike the other non-`accept` choices,
  `request_source` does not demand a free clarification — the *request itself*
  is the body, and EV-002 already seeds it (`ASK_SOURCE_PRESET_BODY`). The user
  may still edit that seeded body. The validation rule (§5.1) treats a non-empty
  seeded body as satisfying the requirement.
- **The seven are exhaustive and closed.** No UI surface may add an eighth
  choice. A new evidence-response kind is a new card that edits this table.

### 5.1 The required-clarification rule

```
A respond_to_evidence draft is structurally valid iff:
  choice === 'accept'
  OR (choice !== 'accept' AND clarificationBody.trim().length >= MIN_CLARIFICATION_CHARS)
```

- `MIN_CLARIFICATION_CHARS = 12` — long enough to be a sentence fragment, short
  enough not to be a busywork gate. (Chosen to match the spirit of the existing
  composer's minimum-body checks; the implementer should confirm against
  `evaluateArgumentDraft` and align if that file has a different floor.)
- This is a **validation** rule. It MAY block the post — the box's Post button
  is disabled (with a visible reason, never silent) until the clarification is
  non-empty. This is doctrine-consistent: *validation can block, score cannot*
  (cdiscourse-doctrine §1). The clarification requirement is structural, like a
  forced-list item field — not a score gate.
- `request_source` satisfies the rule via its seeded body (the body is non-empty
  by construction). If the user clears the seeded body below the floor, the rule
  re-engages.
- The rule is enforced **client-side in the box** (immediate feedback) AND
  re-asserted in the `submit-argument` advisory snapshot (the server records
  whether the clarification was present — see §7). The server does not need a
  new hard block: the choice + clarification travel as an advisory
  `evidenceResponse` block; the *body* of the move still flows through the
  existing `evaluateArgumentDraft` path unchanged.

### 5.2 The applicability status ladder

```
applicability_undisputed  ──dispute_applicability──▶  applicability_disputed
        ▲                                                     │
        │                                       a later corroborating move
        │                                       (add_evidence / accept that
        │                                        resolves the open dispute)
        │                                                     ▼
        └──────────  (only via settlement)  ──────  applicability_supported
```

- **`applicability_undisputed`** — the default. No one has contested what the
  evidence applies to. (Note: "undisputed" ≠ "true" — it only means no
  applicability dispute is open.)
- **`applicability_disputed`** — at least one `dispute_applicability` move is
  open against this evidence object. The evidence pill shows the "disputed"
  state; the Inspect §E read-view shows `claimedApplicability` vs
  `disputedApplicability`.
- **`applicability_supported`** — a dispute was opened and later a corroborating
  move (a new evidence object that addresses it, or an `accept` of the contested
  point by the disputing party) resolved it. The storyboard's word is
  **"stronger"** — the model carries `applicability_supported`; the chip copy is
  "Applicability supported (stronger)". This is **not** "true" / "proven" —
  it is "better supported than when the dispute opened".
- The ladder is **monotonic in v1**: once `applicability_supported`, it does not
  fall back unless a *new* `dispute_applicability` move opens (which returns it
  to `applicability_disputed`). There is no "applicability disproven" state —
  the model has no truth-negative value by design.
- Status is **derived at render time** from the move history (the set of
  `respond_to_evidence` moves targeting the evidence object + later
  corroborating moves). It is not persisted in v1. See §6.1 for the derivation.

## 6. How it integrates with the one-box interface

QOL-037 produces **no standalone surface**. It is content for two existing
QOL-030/032 surfaces.

### 6.1 The QOL-030 `respond_to_evidence` box

Per QOL-030 §6.2, `respond_to_evidence` is a box type with a **structured-form**
schema kind, targeting an **evidence object**. QOL-030 §8 names it: "the choice
set (Accept / Dispute applicability / …) + required clarification on any
non-Accept". QOL-037 supplies that schema.

```
│╭ ⬡ Respond to evidence ▾ ── to: ⬡ $120 payment ──╮│
││ This evidence: payment · Mar 3 · $120             ││  read summary of the target
││  note "practice space" · claims: March rent       ││  (QOL-036 fields, read-only)
││ ┌───────────────────────────────────────────────┐ ││
││ │ ○ Accept evidence                              │ ││  the 7-choice radio set
││ │ ○ Accept with caveat                           │ ││
││ │ ○ Dispute the date                             │ ││
││ │ ○ Dispute the amount                           │ ││
││ │ ● Dispute what it applies to                   │ ││  ← selected
││ │ ○ Ask for the source                           │ ││
││ │ ○ Ask for clarification                        │ ││
││ └───────────────────────────────────────────────┘ ││
││ ┌───────────────────────────────────────────────┐ ││
││ │ Clarification — required for this choice        │ ││  required clarification field
││ │ The March 3 payment note says "practice         │ ││  appears for any non-accept
││ │ space," but we'd already agreed February's…     │ ││
││ └───────────────────────────────────────────────┘ ││
││ This will mark the evidence: Applicability disputed││  preview of the status change
││               [⌫ park]  [Post ⏎ — needs a note]   ││  Post disabled w/ reason if empty
│╰───────────────────────────────────────────────────╯│
```

Wiring:

- `renderSchema('respond_to_evidence', evidenceObject)` → returns the QOL-037
  structured-form descriptor: the 7-choice radio group + a conditional
  clarification field. QOL-030's schema renderer already supports the
  structured-form kind; QOL-037 is the descriptor.
- The choice set, labels, helpers, and the required-clarification rule come from
  QOL-037's pure model — the box renders them, never invents them.
- Selecting a choice updates a **preview line** ("This will mark the evidence:
  Applicability disputed") so the user sees the status consequence before
  posting. The preview is computed by `previewApplicabilityTransition(choice,
  currentStatus)`.
- `accept` hides the clarification field entirely (it is not required and not
  shown — no empty box). Every other choice shows it, marked "required".
- The Post button follows QOL-030's lifecycle: `empty → typed → drafting →
  review → posted`. The `review` step (RULE-004 pre-send review) still fires.
  QOL-037 adds one gate to `drafting → review`: the §5.1 clarification rule. The
  disabled Post button shows a visible one-line reason ("Add a note explaining
  the dispute") — never a silent no-op (consistent with QOL-025's
  no-silent-no-op rule).
- `request_source` is special: selecting it seeds the body with EV-002's
  `ASK_SOURCE_PRESET_BODY` (the box already supports body seeding via
  `MoveDraftPatch.body`, added in EV-002) and, on post, the resulting move opens
  an EV-003 `source` debt. The box does not need a separate code path — it routes
  through EV-002's existing `quickActionToPreset('source', …)`.
- The box posts through the existing `submit-argument` Edge Function. No new
  write path.

### 6.2 The QOL-032 Inspect popout (§E Evidence detail)

QOL-032 §3.2 already lists section **E — Evidence detail** ("date, amount, note,
status (QOL-036/024)") and §5 frame S2-F3 already expects "status: applicability
disputed". QOL-037 supplies the applicability fields that §E reads:

| §E field | Source |
|---|---|
| Applicability status | `ApplicabilityStatus` — rendered via the locked chip copy (§8) |
| Claimed applicability | QOL-036 `claimedApplicability` (e.g. "March rent") |
| Disputed applicability | the most recent open `dispute_applicability` move's clarification, summarised (e.g. "February reimbursement") |
| Who challenged it | the author of the open `dispute_applicability` move |
| Dispute clarification | the clarification body of the open dispute (read-only) |

Inspect is **read-only** (QOL-032 §3.1). It renders the applicability axis; it
never mutates it. The hand-off chip (§5 "Suggested next move") may route to the
`respond_to_evidence` box pre-set to a choice — but that is QOL-032's hand-off
mechanism, not a new QOL-037 surface.

### 6.3 The evidence pill / receipt chip

EV-002's `ReceiptChip` already renders the source-chain axis. QOL-037 adds a
**second, separate** chip token for the applicability axis — `ApplicabilityChip`
— so the two axes are visually distinct (a node can show "Source and quote" AND
"Applicability disputed" at once). The applicability chip:

- Renders only when `ApplicabilityStatus !== 'applicability_undisputed'` (the
  undisputed default is the absence of the chip — no clutter on uncontested
  evidence).
- Carries a text label (§8) — never color-only (accessibility-targets:
  color-independence).
- Is non-pressable in v1 (it is a status indicator; tapping the node opens the
  Inspect popout which shows the detail). If a later card wants the chip to be a
  shortcut into Inspect §E, that is a follow-up.

## 7. Data model & file changes

**No schema migration in v1.** Consistent with EV-001 (adapter-only) and EV-003
(render-time-derived). The applicability status is derived from the move history;
the choice + clarification travel as an **advisory block** inside the existing
`client_validation` / `server_validation` JSONB snapshot — exactly where
`attachedEvidence` already lives. No new table, no new RLS policy, no new Edge
Function.

### 7.1 New pure-TS model

`src/features/evidence/evidenceApplicabilityModel.ts` — pure TypeScript, no
React, no Supabase, no network. Sits beside `evidenceModel.ts`.

```ts
/** The seven structured responses to an evidence object. Closed set. */
export type EvidenceResponseChoice =
  | 'accept'
  | 'accept_with_caveat'
  | 'dispute_date'
  | 'dispute_amount'
  | 'dispute_applicability'
  | 'request_source'
  | 'request_clarification';

/** The applicability axis — distinct from SourceChainStatus and EV-003 debt.
 *  'undisputed' is NOT a truth claim — it only means no dispute is open. */
export type ApplicabilityStatus =
  | 'applicability_undisputed'
  | 'applicability_disputed'
  | 'applicability_supported';

/** A single recorded evidence response — the advisory block persisted into the
 *  argument's validation snapshot. Render-time-derivable; no own table in v1. */
export interface EvidenceResponseRecord {
  /** The evidence object this response targets. */
  evidenceArtifactId: string;
  /** The argument (move) that carried this response. */
  argumentId: string;
  choice: EvidenceResponseChoice;
  /** Required for every choice except 'accept'. May be '' only when choice==='accept'. */
  clarificationBody: string;
  /** The author of the response move. */
  respondedByUserId: string;
  /** ISO-8601 — copied from the argument's created_at. */
  respondedAt: string;
}

/** The per-choice descriptor the box renders. Frozen. */
export interface EvidenceResponseChoiceDescriptor {
  choice: EvidenceResponseChoice;
  /** Plain-language label. Never a snake_case code. */
  label: string;
  /** Plain-language helper, one line. */
  helper: string;
  /** True when a clarification body is structurally required. */
  requiresClarification: boolean;
  /** The applicability transition this choice drives, or null. */
  appliesApplicabilityTransition: 'open_dispute' | null;
  /** True when this choice opens an EV-003 evidence debt. */
  opensEvidenceDebt: boolean;
}

export const EVIDENCE_RESPONSE_CHOICES:
  ReadonlyArray<EvidenceResponseChoiceDescriptor>;          // exactly 7, frozen
export const ALL_EVIDENCE_RESPONSE_CHOICES:
  ReadonlyArray<EvidenceResponseChoice>;                    // the 7 ids, frozen
export const ALL_APPLICABILITY_STATUSES:
  ReadonlyArray<ApplicabilityStatus>;                       // the 3, frozen

export const MIN_CLARIFICATION_CHARS = 12;

/** Structural validity of a respond_to_evidence draft. Pure. */
export interface EvidenceResponseValidation {
  isValid: boolean;
  /** Plain-language reason when !isValid — shown on the disabled Post button. */
  blockingReason: string | null;
}
export function validateEvidenceResponseDraft(
  choice: EvidenceResponseChoice,
  clarificationBody: string,
): EvidenceResponseValidation;

/** Derive the applicability status of one evidence object from the full set of
 *  responses targeting it, in chronological order. Pure. */
```

### 7.2 The `deriveApplicabilityStatus` algorithm

Pure, deterministic, no I/O. Given the chronological list of
`EvidenceResponseRecord`s targeting one evidence object:

1. Start at `applicability_undisputed`.
2. Walk the responses oldest-first.
3. On a `dispute_applicability` → set `applicability_disputed`, mark a dispute
   open.
4. On an `accept` *while a dispute is open and the accept is by the disputing
   party* → the dispute is resolved → set `applicability_supported`, clear the
   open flag. (An `accept` by anyone else does not resolve it.)
5. A *later corroborating move* — modelled as the presence of a newer
   `add_evidence` move whose target is this evidence object's parent and whose
   `respond` relationship addresses the open dispute — also resolves it to
   `applicability_supported`. **In v1, since `add_evidence` corroboration is not
   yet wired (QOL-036 dependency), the model accepts an explicit
   `corroboratedByArgumentIds` option; absent that, only rule 4 advances to
   `applicability_supported`.** See §11 fallback.
6. A `dispute_applicability` *after* `applicability_supported` re-opens →
   `applicability_disputed`.
7. The final state is the derived status.

`dispute_date`, `dispute_amount`, `accept_with_caveat`, `request_source`,
`request_clarification` **never** change the applicability status (they touch
other axes). The walk ignores them for this derivation.

### 7.4 The `submit-argument` change — minimal and advisory

The one Edge Function touch is deliberately the smallest possible:

- `submit-argument` already receives an `attached_evidence` array and writes a
  `client_validation` / `server_validation` JSONB snapshot. QOL-037 adds **one
  optional field** — `evidenceResponse` — to the request body, and the function
  copies it into that same snapshot. It does not validate it, does not hard-block
  on it, does not branch on it.
- This keeps the applicability axis **render-time-derivable** (the consuming
  client reads `evidenceResponse` blocks off the argument rows it already fetches
  and runs `deriveApplicabilityStatus`).
- It is **advisory**: a malformed `evidenceResponse` block can never block a post
  — the existing body validation is the only gate, and it is unchanged.
- The clarification-required rule is enforced **in the box** (the Post button is
  disabled). The server stores whatever the box sent; it does not re-block. This
  matches the Stage 6.2 advisory-validation doctrine.
- **Operator step:** the function must be re-deployed once (`npx supabase
  functions deploy submit-argument --linked`) so it knows to pass the new field
  through. This is the only operator action. See §15.

> **Alternative considered:** keep `evidenceResponse` purely client-side and
> never send it to the server. Rejected — then the applicability status would
> not survive a page reload for *other* viewers (only the move body would). The
> storyboard (Step 7) requires the evidence side panel to show "claimed month
> vs disputed month" to *both* parties, which means the choice + clarification
> must persist. The JSONB snapshot is the existing persistence surface; using it
> avoids a migration. A real `evidence_applicability` table is a deferred
> follow-up (see §13).

## 8. UI states — the applicability status axis

The status axis renders through one chip token (`ApplicabilityChip`) and the
Inspect §E read-view. Locked plain-language copy:

| `ApplicabilityStatus` | Chip label | Chip helper | Tone | Chip visible? |
|---|---|---|---|---|
| `applicability_undisputed` | (none) | (none) | — | **no** — uncontested evidence shows no applicability chip |
| `applicability_disputed` | "Applicability disputed" | "A participant disagrees about what this evidence applies to." | `attention` | yes |
| `applicability_supported` | "Applicability supported" | "After the dispute, this evidence is better supported for what it's used for." | `info` | yes |

Locked choice-set copy is in §5. Locked disabled-Post reasons:

- non-accept choice, empty clarification → **"Add a note explaining your
  response before posting."**
- non-accept choice, clarification below the floor → **"Your note is too short —
  add a sentence explaining your response."**

**Forbidden tokens in every QOL-037 string** (choice labels, helpers, chip
labels, chip helpers, disabled reasons, preview line, accessibility labels):
`proof`, `proven`, `disproven`, `true`, `false`, `correct`, `incorrect`,
`winner`, `loser`, `verdict`, `case closed`, `wrong`, `right` (as a verdict —
"the right month" style usage is avoided entirely; copy says "what it applies
to"), `liar`, `dishonest`, `bad faith`, `manipulative`, plus the
amplification tokens (`likes`, `retweets`, `views`, `followers`, `viral`,
`trending`, `engagement`). A ban-list test enforces every string.

State narrative:

- **Undisputed → disputed**: A posts `dispute_applicability` with a clarification.
  The evidence pill on B's node gains the "Applicability disputed" chip. The room
  status (if EV-003 is wired) may also surface — but the applicability chip is
  the QOL-037-owned signal.
- **Disputed → supported**: the disputing party later posts `accept` of the
  contested point (Step 11), or a corroborating evidence move resolves it (Step
  10). The chip flips to "Applicability supported". Storyboard word: "stronger".
- **Supported is not a finish line**: a fresh `dispute_applicability` re-opens
  the axis. There is no terminal "applicability proven" — by design.
- **Settlement**: when the room settles (Step 13), the chips freeze read-only.
  The room-level copy is "Evidence accepted by both" / "settled". QOL-037's chip
  still reads "Applicability supported" if that was the last derived state.

## 9. Permission / rules implications

- **Who may post a `respond_to_evidence` move:** any **primary participant who is
  not the evidence's author** (interaction-taxonomy "Evidence applicability
  challenge"). The author of the evidence cannot `dispute_*` their own evidence
  — that is incoherent. The author *may* still `accept` another party's later
  evidence in the same room (a different evidence object).
- **Own-evidence case:** when the active viewer is the evidence's author, the
  `respond_to_evidence` box is not offered for that object. This follows the
  QOL-030 role gate — own-bubble keeps only `qualifiers` + `request_deletion`.
  The author fixes their own evidence by attaching a *new* evidence move, not by
  responding to themselves.
- **Observer case:** observers (read-mode, Stage 6.4) cannot post. The
  `respond_to_evidence` box is not offered. If the Inspect popout surfaces a
  hand-off chip for an observer, it is disabled with "Join a side to respond"
  (consistent with EV-002's observer contract).
- **The clarification-required rule is a validation gate, not a score gate.** It
  can disable Post; it never affects standing. Doctrine-consistent
  (cdiscourse-doctrine §1: validation can block, score cannot).
- **Disputing applicability is advisory toward standing.** Opening an
  applicability dispute does not by itself lower the evidence author's
  factual-standing score; it records that an axis is contested. The point-standing
  economy may *read* `applicability_disputed` as an open issue, but QOL-037 does
  not compute a score delta — that is the point-standing engine's job, and it
  treats a dispute as an *open issue*, never a *defeat* (point-standing-economy:
  concession/dispute is a repair surface, not a loss).
- **Engagement vs factual standing stay separate.** An evidence object that is
  amplified (many replies, high heat) earns no applicability support from that
  amplification. `applicability_supported` is only reachable via the §7.2
  derivation — corroborating *moves*, never popularity (anti-amplification
  doctrine).
- **Engine gate.** The `respond_to_evidence` move's resulting argument type
  (`evidence_acceptance` / `evidence_challenge` / `clarification_request`) must
  be a Constitution-valid child of the parent. QOL-037 does **not** add new
  argument types to the Constitution — it maps each choice to an *existing*
  argument type (see §5 "Routes to"). If the Constitution lacks a suitable type,
  that is a `RULE`/Constitution card, flagged as an open question (§16), not
  silently invented here.

## 10. Edge cases

The implementer MUST handle each. Tests in §11.

1. **Empty clarification on a non-accept choice.** Post is disabled with the
   visible reason "Add a note explaining your response before posting." Never a
   silent no-op (QOL-025 rule). The box stays open with the draft intact.
2. **Clarification below `MIN_CLARIFICATION_CHARS`.** Same — disabled Post,
   reason "Your note is too short…". Whitespace-only is treated as empty.
3. **`accept` chosen, then user types in a clarification, then switches back to
   a dispute choice.** The clarification text is preserved (QOL-030 per-type
   draft buffers — the field is parked, not destroyed). When `accept` is
   re-selected the field hides but its content is retained for the next
   non-accept choice.
4. **`accept_with_caveat` vs the applicability axis.** `accept_with_caveat` does
   NOT flip the evidence to `applicability_disputed` — a caveat is a reservation,
   not an applicability dispute. The model's derivation (§7.2) ignores
   `accept_with_caveat` for the applicability walk. (Storyboard Step 8: B's
   caveat targets A's *claim*, not the applicability of B's own payment object.)
5. **Multiple `dispute_applicability` moves on one evidence object.** The status
   is `applicability_disputed` while any dispute is open. The Inspect §E
   "disputed applicability" field shows the **most recent** open dispute's
   clarification; a "+N earlier" affordance is a follow-up, not v1.
6. **A dispute opened, then the disputing party never returns.** The status
   stays `applicability_disputed` indefinitely. There is no auto-resolution and
   no "stale" auto-flip in QOL-037 — staleness is EV-003's concern (an evidence
   debt can go `stale`; an applicability dispute just stays open). The Inspect
   §E read-view shows it as still disputed.
7. **`request_source` from this choice set.** Routes to EV-002's seeded preset
   and opens an EV-003 `source` debt. It does NOT change `ApplicabilityStatus`.
   If EV-003 is not yet built, the move still posts (it is a normal
   `clarification_request` body) — the debt simply is not tracked. See §11.
8. **Disputing the date or amount.** `dispute_date` / `dispute_amount` require a
   clarification (they are non-accept) but drive **no** applicability transition.
   They are separate axes (QOL-036 fields). The model must not let them touch
   `ApplicabilityStatus`. A test asserts this.
9. **QOL-036 not yet built — no `claimedApplicability` field.** The box's read
   summary and the Inspect §E "claimed applicability" field render "Not
   specified" rather than crash. `dispute_applicability` still works — the
   disputed applicability comes from the clarification body; the *claimed* side
   just shows "Not specified". See §11 fallback.
10. **Concurrent disputes.** Two participants each post a `dispute_applicability`
    near-simultaneously. Both are recorded; the status is `applicability_disputed`
    (idempotent — one or many open disputes is the same status). No lock needed —
    the derivation is order-independent for "is any dispute open".
11. **Settled / locked room.** The `respond_to_evidence` box is not offered;
    Inspect §E renders the last derived status read-only with a settled banner
    (QOL-032 §7).
12. **Evidence object soft-deleted (parent argument `is_deleted = true`).** The
    evidence object is not shown; no `respond_to_evidence` box for it. Existing
    `respond_to_evidence` moves targeting it remain in the record (arguments are
    never hard-deleted) but the derivation skips a deleted target.
13. **Malformed `evidenceResponse` block in a JSONB snapshot** (e.g. an unknown
    `choice` value from a future client). `deriveApplicabilityStatus` ignores any
    record whose `choice` is not in `ALL_EVIDENCE_RESPONSE_CHOICES` — it never
    crashes, it degrades to "as if that record were absent".
14. **A user pastes a verdict word ("this proves you're a liar") into the
    clarification body.** QOL-037 does **not** sanitise user free-text — that is
    a moderation concern, not the model's. The ban-list only governs
    QOL-037's *system-generated* strings. The clarification renders as the user
    typed it. (Same stance as EV-001 edge case 13.)
15. **Reduce-motion.** The `respond_to_evidence` box's clarification field
    show/hide uses `LayoutAnimation` only when reduce-motion is off; otherwise it
    snaps. The `ApplicabilityChip` has no animation.
16. **Color-blind / desaturated render.** The `ApplicabilityChip` carries a text
    label in every visible state; the tone color is redundant, never the sole
    signal (accessibility-targets: color-independence).
17. **Offline / submit failure.** The box behaves exactly as any other move on a
    failed `submit-argument` — the draft is parked, the user retries. QOL-037
    adds no new network path; it rides the existing submit flow.

## 12. Doctrine & safety self-check

Each line confirms a constraint from cdiscourse-doctrine / evidence-doctrine /
the spawn prompt is respected:

- **Score is gameplay analysis, never truth (cdiscourse-doctrine §1).**
  `ApplicabilityStatus` describes the *applicability axis* (`undisputed` /
  `disputed` / `supported`) — none of these is a verdict. There is deliberately
  **no** "applicability proven / disproven / true / false" value. The model has
  no truth-negative state. Ban-list test enforces.
- **Disputing applicability changes a status, never declares the evidence
  false.** Encoded structurally: `dispute_applicability` flips
  `applicability_undisputed → applicability_disputed` — a *status*, not a truth
  flag. Copy is "Applicability disputed", never "false"/"proof" (acceptance
  criterion of the card; §8 forbidden list enforces).
- **Validation can block, score cannot.** The required-clarification rule is a
  *validation* gate — it disables Post structurally, like a forced-list item
  field. It does not touch any score. The disabled Post always shows a visible
  reason (QOL-025 no-silent-no-op).
- **Heat / popularity is not evidence (cdiscourse-doctrine §2, §3;
  anti-amplification).** `applicability_supported` is reachable only via the §7.2
  derivation — corroborating *moves*, never replies/views/heat. The model
  consumes only `EvidenceResponseRecord`s; it has no engagement input. Ban-list
  test forbids amplification tokens.
- **Evidence ≠ truth (evidence-doctrine).** A payment screenshot proves at most
  that a payment object exists; QOL-037's whole point is the *applicability* axis
  is separate from existence and from source-chain. §4 encodes the three-axis
  separation; a test asserts `dispute_date`/`dispute_amount` never touch
  `ApplicabilityStatus`.
- **AI moderator hard limits (cdiscourse-doctrine §4, §7).** QOL-037 introduces
  no AI call — not in the client, not in an Edge Function. `deriveApplicabilityStatus`
  and the validation rule are pure deterministic TypeScript. No AI decides
  whether a dispute is valid.
- **The rules engine stays pure (cdiscourse-doctrine §5).** QOL-037 does not edit
  `src/lib/constitution/engine.ts`. The new model lives in
  `src/features/evidence/`, pure TS. QOL-037 adds **no new Constitution argument
  types** — each choice maps to an existing type (§9; §16 open question if a gap
  is found).
- **Secrets policy (cdiscourse-doctrine §6).** No env var, no `.env*` change, no
  key reference.
- **No service-role; no direct insert into `public.arguments`.** The
  `respond_to_evidence` box posts through the existing `submit-argument` Edge
  Function. The one `submit-argument` change copies an advisory field into a
  JSONB snapshot — no service-role, no new insert path, RLS untouched.
- **Soft-delete only (cdiscourse-doctrine §8).** QOL-037 deletes nothing. Edge

  helper, chip label, and disabled reason is plain English. No `snake_case`
  codes in any user-facing string; the internal `EvidenceResponseChoice` ids
  never render. Ban-list test enforces.
- **v1 scope guards (cdiscourse-doctrine §10).** No voting, no search, no push,

  vote that produces a winner — they record a status, never tally a score.
- **No standalone bespoke surface.** QOL-037 is content for the QOL-030

  the integration target, as required.

## 14. Supersedes / superseded-by

- **Supersedes:** nothing. QOL-037 is a new axis the storyboards drive; it does
  not replace a shipped surface.
- **Superseded-by:** nothing.
- **Relationship to QOL-030:** QOL-037 is *content for* the QOL-030
  `respond_to_evidence` box type — it is the named schema follow-up
  (QOL-030 §12: "the evidence + concession schema internals — QOL-036 / QOL-037
  / QOL-041"). It does not supersede QOL-030; it completes one of its box types.
- **Relationship to EV-002:** EV-002 owns the *source-chain* axis (existence /
  trail). QOL-037 owns the *applicability* axis. They are siblings on one
  evidence object, never overlapping — §4 is explicit. QOL-037 does not change
  EV-002's `SourceChainStatus` or its popover.
- **Relationship to EV-003:** EV-003 owns the *obligation* axis (debt). QOL-037's
  `request_source` *opens* an EV-003 debt but does not implement debt tracking.
- **Relationship to QOL-041:** QOL-041 owns the concession **acceptance
  gradient** (a forced-list axis on a concession set). QOL-037's choice set is a
  *radio* over an *evidence object*. Different target, different schema kind —
  no overlap. (`accept_with_caveat` here is an evidence response; the QOL-041
  gradient is a per-concession-item decision.)

## 16. Open questions

1. **Does the Constitution already have argument types for `evidence_acceptance`
   and `evidence_challenge`?** §9 maps the seven choices to existing argument
   types. If `engine.ts`'s transition table lacks a clean type for an
   evidence-acceptance or evidence-challenge move, QOL-037 must NOT invent one
   — that is a `RULE`/Constitution card. **Recommendation for the implementer:**
   before building, grep `src/lib/constitution/` for the available argument
   types; if `clarification_request` / `rebuttal` / a counter-type covers all
   seven choices, use those; if not, stop and flag a Constitution card. (Lean:
   `clarification_request` covers `request_source` / `request_clarification`;
   an existing rebuttal/counter type covers `dispute_*`; `accept` /
   `accept_with_caveat` may need either an existing "concession" type or a new
   one — this is the most likely gap.)
2. **Persist `evidenceResponse` in a table now, or keep render-time-derived?**
   v1 keeps it render-time-derived from the JSONB snapshot (no migration,
   matching EV-001 / EV-003). A real `evidence_applicability` table becomes
   worthwhile once a usage needs to query "all rooms with an open applicability
   dispute" without loading every argument row. **Lean:** defer the table to a
   follow-up card, exactly as EV-001's path-(c) does.
3. **Should `accept` of a contested point auto-flip to `applicability_supported`,
   or only mark the dispute "withdrawn"?** §7.2 rule 4 advances to `supported`.
   An alternative reading: an `accept` by the disputing party *withdraws* the
   dispute (back toward `undisputed`) rather than *supports* it. **Lean:**
   `supported` — the storyboard Step 11 narrative ("I concede the $120 payment
   was intended for March rent") plus Step 10's "stronger" language both point to
   "the evidence ended up better supported than when the dispute opened", not
   "as if the dispute never happened". Operator confirm at Review.
4. **Should the `ApplicabilityChip` be pressable (a shortcut into Inspect §E)?**
   v1 makes it a non-pressable status indicator (tap the node → Inspect). A
   pressable chip is a minor follow-up. **Lean:** non-pressable in v1.
5. **`MIN_CLARIFICATION_CHARS = 12` — is the floor right?** Chosen to be a
   sentence fragment. The implementer should align it with any existing
   minimum-body constant in `evaluateArgumentDraft`; if that file has no floor,
   12 stands. Operator confirm at Review.

## 17. Out of scope

Explicit list to prevent scope creep:

- **The QOL-030 box chassis, flash menu, popout primitives** — QOL-037 supplies
  one schema descriptor; it does not build or modify the chassis.
- **The QOL-036 payment-evidence metadata fields** (amount, date, redacted
  payer/payee, `claimedApplicability`) — QOL-037 reads them; QOL-036 defines
  them.
- **The EV-003 evidence-debt tracker** — `request_source` opens a debt; QOL-037
  does not implement the debt model, debt chips, or the gallery debt lane.
- **The EV-002 source-chain popover** — `request_source` reuses EV-002's preset;
  QOL-037 does not change the popover.
- **The QOL-041 concession acceptance gradient** — a separate forced-list axis.
- **A persisted `evidence_applicability` table / migration** — v1 is
  render-time-derived from the JSONB snapshot (open question 2).
- **New Constitution argument types** — if a type gap exists, that is a separate
  `RULE` card (open question 1).
- **A "+N earlier disputes" history affordance** in Inspect §E — v1 shows the
  most recent open dispute only.
- **Auto-resolution / staleness of an open applicability dispute** — a dispute
  stays open until a move resolves it; staleness is EV-003's concern.
- **Any AI assessment** of whether a dispute is correct, or AI-suggested
  clarification text — forbidden by cdiscourse-doctrine §7.
- **A score delta for opening a dispute** — the point-standing engine may *read*
  `applicability_disputed` as an open issue, but computing the standing change
  is the point-standing engine's job, not QOL-037's.
- **Voting on whether the evidence applies** — explicitly forbidden
  (cdiscourse-doctrine §10; popularity is not evidence).
- **Notifications** for an applicability dispute — the QOL-040 notification
  lifecycle card owns notification copy/routing.
- **Settlement / room-lock mechanics** — already shipped (room status model);
  QOL-037 only renders the read-only state inside a settled room.
