## Phase 0 — locked findings (non-spend RCA, complete)

- **0a NON-TERMINAL.** Idempotency guard is success-only (`autoTriggerDispatcher.ts:254` — `existing.status === 'success'` skips; a failed/null run proceeds). No idempotency change is forced. BUT auto-trigger fires only at submit time → no automatic heal; a failed family is a coverage hole until an in-dispatch fix lands.
- **0b HOLE RATE.** Run-completeness census (`status=success` per (argument,family) cell, era-safe): **~0.8% historical (1/123), ~11% burst (4/35), 3.16% combined.** A burst-concurrency phenomenon, not pervasive. Holes: critical_question 2, evidence_source_chain 2 (1 pre-existing), argument_scheme 1.
- **0c LOGS NOT CLI-QUERYABLE.** `supabase functions logs` is not a CLI subcommand here → do not depend on logs; the sub-reason must travel on the RETURN.
- **0d TYPING IS EDGE-ONLY.** `mcpBooleanObservationSchema.ts` `parseMcpBooleanObservationResponse` already returns granular `{ok:false, reason, details}` (`not_json` / `wrong_schema_version` / `wrong_shape` / `missing_required_field` / `flag_count_too_high`, each with a `details` string). `booleanObservationMcpAdapter.ts:162–169` COLLAPSES all of it to `{ kind:'unavailable', reason:'validation_failed' }`, discarding `parsed.reason` + `parsed.details`. No MCP-server change needed.
- **0e COMPUTE-AT-FAILURE is trivial** — the granular reason is already computed at the failure point; the fix is to stop discarding it.
- **Retry status:** `RETRYABLE_FAILURE_REASONS = {mcp_network_error, mcp_api_error, mcp_rate_limited}` (`autoTriggerDispatcher.ts:119`). `mcp_validation_failed` is NOT retryable → the 4 burst failures got one attempt → hole. (Retry is a Phase 3 decision, NOT Phase 1.)

**Discriminator (make permanent in the vocab):** two failures share `mcp_validation_failed` with opposite fixes — fast-reject (<~1s) = request-side (Card 1A subset class); slow-fail (~full classifier duration) = response-side (this burst). Duration is the tell.

## Phase 1 — TYPE at the failure point (THIS merge cycle; non-spend)

Exact anchors:
- `booleanObservationMcpAdapterCore.ts:74` `BooleanObservationUnavailableReason` union; `:100–102` `BooleanObservationAdapterResult` (`unavailable` variant currently `{ kind:'unavailable'; reason }`).
- `booleanObservationMcpAdapter.ts:156` (parse_failure), `:163` + `:168` (validation_failed collapse sites), plus the HTTP/network sites (`network_error`/`rate_limited`/`api_error`) earlier in the file.
- `classifyArgumentCore.ts:65` `PerArgumentSummary` (`failureReason: string|null`); `:145` `unavailableReasonToFailureReason`; `:227–247` the `kind==='unavailable'` consumption that builds the summary.

Scope:
1. **Stop the discard.** Extend the adapter `unavailable` result with an optional typed `subReason` + a bounded, sanitized `detail`. Thread the validator's `{reason, details}` through at the collapse sites. **PRESERVE `reason:'validation_failed'`** so `unavailableReasonToFailureReason` still yields `failure_reason='mcp_validation_failed'` (audit-lint + prior smokes + existing readers key on that literal).
2. **Typed sub-reason vocabulary** (controlled enum in code; the request_/response_/provider_ split is the durable value): `request_unsupported_family | request_unsupported_raw_key | request_invalid_source_subset | response_not_json | response_wrong_schema_version | response_wrong_shape | response_missing_required_field | response_flag_count_too_high | response_evidence_span_invalid | response_ban_list_violation | provider_timeout | provider_rate_limited | provider_api_error | provider_network_error | unknown`. Map ONLY what the code actually emits today (validator `not_json→response_not_json`, `wrong_schema_version→response_wrong_schema_version`, `wrong_shape→response_wrong_shape`, `missing_required_field→response_missing_required_field`, `flag_count_too_high→response_flag_count_too_high`; adapter `parse_failure→response_not_json`, `network_error→provider_network_error`, `rate_limited→provider_rate_limited`, `api_error→provider_api_error`; `url_missing`/`token_missing` are operator-config, already distinct failure_reasons — leave `subReason` unset or `unknown`). Entries with no current emitter (`request_*` beyond what the subset path produces, `response_evidence_span_invalid`, `response_ban_list_violation`, `provider_timeout`) are RESERVED in the enum; the designer decides whether to add finer detection now or reserve — do not invent emitters speculatively.
3. **Surface in TWO places, neither a DB column:** (a) the classify function's structured RETURN — extend `PerArgumentSummary` with an optional `failureSubReason` (+ bounded `failureDetail`) read synchronously by the Phase 2 admin_validation harness; (b) the structured log (`emitAutoTriggerLog` / classify log) — supplementary. The run row's `failure_reason` is unchanged.
4. **SANITIZATION (the new detail field is a secret-surface — guard it).** Allowlist ONLY `{validatorReason, path, expected, receivedType, receivedKeys, checkedRawKey, schemaVersion, family}`; cap serialized detail (~2000 chars). HARD BAN: raw prompt, argument body, raw/whole model response, full provider payload, JWT, Bearer, service-role, Authorization header, API key. The existing adapter "never logs the token/URL/Authorization/Bearer/raw body" posture extends to this field.
5. **Tests (non-spend):** a fast unsupported-key/request-side failure types as `request_*`; a slow response-shape failure types as `response_*`; the compat `reason:'validation_failed'` / `failure_reason='mcp_validation_failed'` still emit; a HOSTILE fixture that smuggles banned strings (a prompt, a `Bearer x`, a service-role key shape) is stripped/rejected; existing readers tolerate the new optional fields.

OUT of Phase 1: any retry/fix (that is Phase 3, branch-selected by the Phase 2 diagnosis); any migration/column; any concurrency change; any prompt/taxonomy/schema/flag change.
