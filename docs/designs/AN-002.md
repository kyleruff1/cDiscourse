## 0. Card-vs-reality discrepancies (read first)

Every prior card on this roadmap shipped with several card-vs-code drifts. I verified each
noun in the AN-002 card body against the real code. Findings:

### D1 — "Fixture timelines" has no defined SHAPE in the card. This design defines it.

The card says "Fixture timelines for visual review" and lists 8 names. It does **not** say
whether these are inline test fixtures, JSON files, a shared TS module, or screens. Reality
check across the repo found three distinct existing "fixture" patterns, none of which is the
right fit as-is:

- **Inline test fixtures** — `__tests__/argumentTimelineMap.test.ts` builds a 250-node stress
  fixture *inside the test file* (`buildStressFixture()`, lines 371–411). `treePlayabilityDiagnostics.test.ts`
  builds deep-tree fixtures inline via `fakeNode()` / `buildTimelineMap()`. These are private to
  one test file — they cannot be *referenced by a checklist doc* or reused by a second consumer.
- **JSON scenario files** — `fixtures/argument-scenarios/*.json` (4 files) are author-facing
  debate scripts consumed by the `argument-fixture-author` skill and `validateScenario`. They are
  *move-script* shaped (`challenge_parent` / `ask_clarification` / etc.), **not**
  `ArgumentTimelineMapMessageInput`-shaped. They are not built into a timeline map. Wrong shape.
- **Bot-fixture scaffolding** — `scripts/bot-fixtures/` produces *runtime* corpora by posting
  to a live `submit-argument` Edge Function. That is operator-gated network tooling, not a
  deterministic in-memory fixture for visual QA.

**Decision (see §"Data model"):** AN-002 ships a NEW pure-TS fixtures module,
`src/features/analytics/visualQaFixtures.ts`, exporting **8 named builder functions** that each
return `ArgumentTimelineMapMessageInput[]` plus a small registry. A builder returns the *input
array*, not a built map — so a test, the checklist, and any future dev harness can each call
`buildArgumentTimelineMap()` (and `buildPointLifecycleMap()`) on the same deterministic input.
This is the only form that satisfies *both* acceptance criteria: tests import the builders;
the checklist doc names the same builders.

### D2 — The card's "browser visual checklist" is a DOC, not a tool.

The card's second acceptance criterion is "Browser visual checklist references fixtures." A
checklist that *references fixtures* is a Markdown document a human QA reviewer walks through.
AN-002 does **not** ask the agent to run a browser, take screenshots, or diff images. There is
already a `docs/browser-visual-test.md` (391 lines) — a section-by-section manual checklist with
the exact `- [ ]` tone this card wants. AN-002 adds a **new dedicated doc**,
`docs/visual-qa-snapshots.md`, rather than bloating `browser-visual-test.md`, because the AN-002
checklist is fixture-indexed (one section per named fixture) and `browser-visual-test.md` is
app-flow-indexed (one section per screen). See §"File changes" for the rationale.

### D3 — `src/features/analytics/` has NO `index.ts` barrel.

AN-001 and AN-003 are both imported by full path (`../src/features/analytics/boardDiagnostics`).
AN-002 follows the same convention — a new file at the same path level, imported by full path.
This design does **not** create an `index.ts` (a gratuitous refactor of two existing call sites).

### D4 — AN-003's §5 dev-script blocker does NOT affect AN-002.

AN-003's design (`docs/designs/AN-003.md` §5) records that the originally-specified `.js`
snapshot script could not `require()` TS builders from `src/` (no TS runtime loader in the repo;
the card forbids adding one) and resolved it with a `tsc`-precompile step. **AN-002 sidesteps
this entirely.** AN-002 ships *no dev script* and *no Markdown generator*. The 8 fixtures are
plain TS builders consumed by the Jest test suite (which already transpiles TS). The "browser
visual checklist" is a *static, hand-authored* Markdown file — it is not machine-generated, so
there is no script to run and no compile mechanism to choose. This is a deliberate scope choice
and is the simplest reading of the card's two acceptance criteria.

### D5 — `ArgumentTimelineMapMessageInput` is the correct fixture element type.

Verified: `buildArgumentTimelineMap({ messages, currentUserId })` accepts
`ArgumentMessageInput[] | ArgumentTimelineMapMessageInput[]`
(`argumentGameSurfaceModel.ts` line 937). The superset type
`ArgumentTimelineMapMessageInput` (line 923) adds `flagCodes`, `tagCodes`, `topicScore`,
`hasEvidence`, `standingBandOverride` — every field a visual-QA fixture needs to drive node
shape, strength band, and tone. AN-002 fixtures emit `ArgumentTimelineMapMessageInput[]`.

### D6 — The 8th fixture, "avatar/profile display", is NOT a timeline-shape variant.

Seven of the eight named fixtures are timeline *structures* (chain depth, branch shape, node
mix). The eighth — "avatar/profile display" — is about *participant rendering*: distinct
`authorId`s, a `currentUserId` that matches one of them (self vs other), a bot author, and a
mix the timeline map turns into `participantTrends` + actor labels. It is still expressed as an
`ArgumentTimelineMapMessageInput[]` builder (so it stays uniform with the other 7), but its
*assertions* target `participantTrends`, `actorLabel`, and `isBot` rather than depth/branch
shape. No card discrepancy — just a precision note so the implementer does not try to force a
branch-shape assertion onto it.

None of D1–D6 is a blocker. AN-002 proceeds with the adaptations above.

## Goal

AN-002 ships a **pure-TS, dev/QA-only** library of 8 named, deterministic argument-timeline
fixtures, plus a hand-authored "browser visual checklist" doc that tells a human reviewer what
to look at when each fixture is rendered. The purpose is a *stable visual review surface*: when
someone changes the timeline renderer (Epic 2 visual grammar, Epic 3 branches, Epic 7 strength
bands), there is a fixed set of named scenes — "the no-rebuttal one", "the 250-node stress one",
"the tangent/kink one" — that QA can eyeball for regressions, and a model-test suite that
guarantees each fixture still *builds* into the timeline shape its name promises.

This card is the analytics epic's QA-fixtures card: it does not add product behavior. It adds
(a) reusable fixtures and (b) a checklist. The "fixtures" half is verified by automated model
tests (acceptance criterion 1); the "checklist" half is the doc (acceptance criterion 2).

Doctrine constraints that shape the design (from `cdiscourse-doctrine`, `timeline-grammar`,
`test-discipline`):

- **No verdict, no truth labels.** Fixture bodies, fixture qualifier labels, fixture flag codes,
  and every line of the checklist doc must be free of winner / loser / correct / true / false /
  liar / dishonest / bad faith / etc. Fixture content uses neutral, low-stakes topics (bike
  lanes, remote work — the same register as `fixtures/argument-scenarios/`). A ban-list test
  enforces this.
- **No popularity / engagement signal.** Fixtures carry no like counts, view counts, retweet
  counts, follower counts. "Heat" in a fixture is expressed only as the structural inputs the
  timeline map already reads (`flagCodes` like `ad_hominem` / `civility_risk`, contiguous
  challenge nodes). Heat ≠ truth — the strength-band fixture deliberately includes a *heated*
  node that still sits in a low strength band, to keep that distinction visible.
- **Dev / QA-only.** The fixtures module is never imported by `app/`. A structural test asserts
  no `app/**` file imports `visualQaFixtures`. The checklist doc is operator/QA tooling.
- **Pure TS.** No React, no Supabase, no network, no AI, no `Date.now()` / `Math.random()` in
  any builder body. Timestamps are derived from a fixed epoch anchor (the
  `argumentTimelineMap.test.ts` `isoAt(offset)` pattern) so every builder is deterministic.
- **No re-derivation, no new product model.** AN-002 produces *input arrays*. It consumes
  `buildArgumentTimelineMap` (and optionally `buildPointLifecycleMap`) only inside tests to
  *verify* the fixtures. The builders themselves import nothing but a type and a tiny epoch
  helper.

AN-002 ships **no migration, no Edge Function, no schema change, no Supabase write, no `.env*`
change, no new dependency, no UI screen, no AI path, no dev script.**

## Data model

### No new domain model. One new fixtures module + one element type re-use.

All new code lives in **`src/features/analytics/visualQaFixtures.ts`** (NEW). It imports the
existing `ArgumentTimelineMapMessageInput` type and emits arrays of it. It introduces *no* new
product type; the only new exported types are descriptors for the fixture registry.

```ts
import type { ArgumentTimelineMapMessageInput }
  from '../arguments/argumentGameSurfaceModel';

/** Stable identifier for one of the 8 visual-QA fixtures. */
export type VisualQaFixtureId =
  | 'no_rebuttal'
  | 'straight_chain_10'
  | 'source_chain_fight'
  | 'evidence_heavy_branch'
  | 'tangent_kink_branch'
  | 'synthesis_path'
  | 'stress_board_250'
  | 'avatar_profile_display';

/** What a reviewer is meant to verify visually — the bridge to the checklist doc. */
export interface VisualQaFixtureDescriptor {
  /** Stable id; matches the builder export name's snake_case form. */
  id: VisualQaFixtureId;
  /** Human title used as the checklist section header. No verdict tokens. */
  title: string;
  /** One-paragraph plain-language description of the scene. No verdict tokens. */
  summary: string;
  /**
   * The structural invariant the fixture guarantees, stated so a model test
   * and a human reviewer agree. e.g. "Exactly one root node, zero rebuttals."
   */
  structuralInvariant: string;
  /** The deterministic builder. Pure; returns the same array every call. */
  build: () => ArgumentTimelineMapMessageInput[];
}

/**
 * The canonical registry. Order is stable and is the order the checklist
 * doc walks. `VISUAL_QA_FIXTURES.length === 8` is asserted by a test.
 */
export const VISUAL_QA_FIXTURES: ReadonlyArray<VisualQaFixtureDescriptor>;

/** Lookup by id; returns null for an unknown id (never throws). */
export function getVisualQaFixture(id: string): VisualQaFixtureDescriptor | null;
```

Plus 8 named builder functions (also exported individually so a test can import one without
the registry):

```ts
export function buildNoRebuttalFixture(): ArgumentTimelineMapMessageInput[];
export function buildStraightChain10Fixture(): ArgumentTimelineMapMessageInput[];
export function buildSourceChainFightFixture(): ArgumentTimelineMapMessageInput[];
export function buildEvidenceHeavyBranchFixture(): ArgumentTimelineMapMessageInput[];
export function buildTangentKinkBranchFixture(): ArgumentTimelineMapMessageInput[];
export function buildSynthesisPathFixture(): ArgumentTimelineMapMessageInput[];
export function buildStressBoard250Fixture(): ArgumentTimelineMapMessageInput[];
export function buildAvatarProfileDisplayFixture(): ArgumentTimelineMapMessageInput[];
```

### The 8 fixtures — exact shape each one must produce

Each builder returns a deterministic `ArgumentTimelineMapMessageInput[]` with stable ids and
timestamps derived from a fixed epoch (`isoAt(offset)` — anchor `1715000000000`, the same
anchor `argumentTimelineMap.test.ts` already uses, so fixture timestamps are consistent across
the repo). `argumentType` values are drawn only from the recognized vocabulary in
`KIND_COLOR_FAMILY` (`thesis` / `claim` → claim family, `rebuttal` / `counter_rebuttal` /
`disagree` → challenge, `evidence` / `source` → evidence, `clarification_request` / `question`
→ clarify, `concession` / `synthesis` / `narrow` → concede, `flag` / `delete` → flag) so each
fixture exercises real node-shape mappings.

| # | Builder / id | Scene | Structural invariant the model test asserts |
|---|---|---|---|
| 1 | `buildNoRebuttalFixture` / `no_rebuttal` | A single root `thesis` and nothing else. The "empty room, opening move only" case. | `nodes.length === 1`; the one node `isRoot === true`; `map.hasRebuttal === false`; `map.firstRebuttalMessageId === null`; `edges.length === 0`; `rootOnboardingHint` is non-null (the "be the first rebuttal" copy path). |
| 2 | `buildStraightChain10Fixture` / `straight_chain_10` | A 10-move linear chain: root + 9 single-child replies, no branching. | `nodes.length === 10`; deepest node `depth === 9`; every non-root node has exactly one parent and (except the leaf) exactly one child; **every node `lane === 0`** (first-child-continues-the-rail rule); `edges.length === 9`; no junctions. |
| 3 | `buildSourceChainFightFixture` / `source_chain_fight` | A claim challenged by a `clarification_request` tagged `source_request`, answered by `evidence`, re-challenged with another source ask — the "where's your receipt" exchange. | `nodes.length` ≥ 5; contains ≥ 2 nodes whose `kindColorFamily === 'clarify'` carrying a `source_request` tag in `droppedTags`; contains ≥ 1 `evidence`-family node; no concession/synthesis node (the fight is unresolved). |
| 4 | `buildEvidenceHeavyBranchFixture` / `evidence_heavy_branch` | A branch lane where 3+ contiguous `evidence` nodes stack — drives the "Evidence run" band heuristic. | `nodes.length` ≥ 6; ≥ 3 nodes with `kindColorFamily === 'evidence'`, of which ≥ 2 are contiguous in chronological order so `map.bands` contains an `Evidence run` band; at least one node sits on a non-zero `lane` (a real branch). |
| 5 | `buildTangentKinkBranchFixture` / `tangent_kink_branch` | A mainline chain with one *side issue*: a parent with 2+ replies, producing a junction and an off-rail branch (lane ≠ 0), plus one *detached* node (missing parent — the "kink"). | `nodes.length` ≥ 7; exactly one node `isJunction === true` with `junctionChildCount >= 2`; ≥ 1 node with `lane !== 0`; exactly 1 node `isDetached === true`; the detached node's `accessibilityLabel` contains "detached". |
| 6 | `buildSynthesisPathFixture` / `synthesis_path` | A clash that resolves: claim → challenge → `concession` (narrow) → `synthesis`. The "this point got resolved" arc. | `nodes.length` ≥ 5; contains ≥ 1 `concession`-family and exactly 1 `synthesis` node; the `synthesis` node is chronologically last; `kindColorFamily` of the synthesis node is `concede`. |
| 7 | `buildStressBoard250Fixture` / `stress_board_250` | The performance / layout stress case: a 250-node board mixing a wide first level, a deep sub-chain, and a block of detached nodes. Mirrors the existing `buildStressFixture()` in `argumentTimelineMap.test.ts` but as a *named, reusable* builder. | `nodes.length === 251` (250 + root, matching the existing test's `251` assertion — see §Risks R3); `nodes.filter(isDetached).length === 50`; no duplicate node ids; no duplicate edge ids; `x` positions strictly monotonic. |
| 8 | `buildAvatarProfileDisplayFixture` / `avatar_profile_display` | A short multi-author thread: ≥ 3 distinct `authorId`s, one of which equals the `currentUserId` the test passes (self), one a non-self human (other), one with `isBot === true`. Drives participant-trend + actor-label rendering. | `nodes.length` ≥ 4; ≥ 3 distinct `authorId` values; the built map's `participantTrends.length >= 3`; the fixture exports its intended `currentUserId` (see below) so the test can assert one node renders as self and one as bot. |

For fixture 8 the builder also exports a companion constant so the test and checklist agree on
which author is "you":

```ts
/** The currentUserId callers should pass to buildArgumentTimelineMap for fixture 8. */
export const AVATAR_PROFILE_DISPLAY_CURRENT_USER_ID = 'qa-author-self';
```

The other 7 builders are `currentUserId`-agnostic (their invariants are structural); tests pass
a fixed `'qa-reviewer'` for them.

## API / interface contracts

The implementer (and any future consumer — a dev harness, an AN-003 snapshot script) needs
these exact contracts:

1. **Each `buildXxxFixture()`** — zero arguments, returns a fresh
   `ArgumentTimelineMapMessageInput[]`. Pure, total, deterministic: calling it twice returns two
   arrays that are deep-equal. It must construct a *new* array each call (no shared mutable
   module-level array a caller could mutate). Timestamps come from the private `isoAt` epoch
   helper, never `Date.now()`. Ids are stable string literals (`'root'`, `'m1'`, `'L1-0'`, …).

2. **`VISUAL_QA_FIXTURES`** — a frozen, ordered `ReadonlyArray<VisualQaFixtureDescriptor>` of
   length exactly 8. `VISUAL_QA_FIXTURES[i].id` values are unique and cover every member of the
   `VisualQaFixtureId` union. `VISUAL_QA_FIXTURES[i].build` is the corresponding `buildXxxFixture`
   function reference.

3. **`getVisualQaFixture(id)`** — returns the descriptor for a known id, `null` for any unknown
   string. Never throws.

4. **`AVATAR_PROFILE_DISPLAY_CURRENT_USER_ID`** — the `string` a caller passes as
   `currentUserId` to `buildArgumentTimelineMap` when rendering fixture 8, so "you" resolves to
   a real author in that fixture.

5. **Consumption contract (for tests / harness / docs).** A consumer renders a fixture by:
   ```ts
   const messages = buildTangentKinkBranchFixture();
   const map = buildArgumentTimelineMap({ messages, currentUserId: 'qa-reviewer' });
   // map.nodes / map.edges / map.bands / map.participantTrends are now the scene.
   ```
   AN-002 does not wrap this call — it deliberately exposes the *input array* so the consumer
   chooses density, `activeMessageId`, `currentUserId` etc. The checklist doc states this
   one-liner per fixture.

6. **No `argumentType` outside the recognized vocabulary.** Builders use only the keys present
   in `KIND_COLOR_FAMILY` so that node shape / color family assertions are meaningful. An
   unknown `argumentType` falls back to `'default'` family — a test asserts no fixture node
   resolves to `'default'` *unless that is the fixture's explicit intent* (none is).

## Edge cases

The implementer must handle each:

- **Empty-array misuse.** No builder returns `[]` — the `no_rebuttal` fixture returns a
  one-element array (a root). A test asserts every builder returns `length >= 1`. (The
  *empty-timeline* path is already covered by `argumentTimelineMap.test.ts`; AN-002 does not
  re-test it.)
- **Detached node in fixture 5.** The `tangent_kink_branch` fixture intentionally includes one
  node whose `parentId` points at a non-existent id. `buildArgumentTimelineMap` marks it
  `isDetached` and produces *no edge* for it — the test asserts both. The builder must NOT
  accidentally also reference that missing id elsewhere.
- **Stress fixture count drift.** Fixture 7 must produce exactly the node count its invariant
  states. The existing inline `buildStressFixture` yields `251` total (root + 250). AN-002's
  `buildStressBoard250Fixture` must match that exact total and detached count (`50`) so the
  number is unambiguous and the checklist can state "≈250 nodes" honestly. See §Risks R3.
- **Determinism under repeated calls.** Each builder is called twice in the test; the arrays
  must be deep-equal. No `Math.random`, no `Date.now`, no closure over a mutated counter that
  persists between calls.
- **Junction vs first-child rule (fixtures 2, 4, 5).** Per `timeline-grammar`, the *first* child
  of a parent stays on the parent's lane; only the *second+* sibling branches. Fixture 2 (straight
  chain) must therefore have **every node on lane 0** — the test asserts this, and it is the
  easiest way to catch a regression in `computeLane`. Fixture 5's junction must give a parent
  *two* children so a real off-rail lane appears.
- **Doctrine edge case — heat must not look like truth.** Fixture 4 (evidence-heavy) and any
  fixture carrying a `civility_risk` / `ad_hominem` flag must keep verdict-free bodies. A *heated*
  node is still just a node; the strength band is driven by `inferStandingBand`, not by heat. The
  checklist doc for any heated fixture explicitly says "a heated node is not a wrong node — check
  the strength stroke, not the warmth."
- **Doctrine edge case — popularity is not in the fixture.** No fixture field encodes likes /
  views / shares. There is no such input on `ArgumentTimelineMapMessageInput` — confirmed — so
  this is structurally impossible to violate, but the ban-list test still scans for the words.
- **Concurrent edits / offline / network / permission-denied.** Not applicable — AN-002 is pure
  in-memory data and a static doc. No network, no I/O, no RLS surface, no Supabase.

## Test plan

All tests in **`__tests__/visualQaFixtures.test.ts`** (pure-model — no React, no Supabase, no
network). Estimated **~32–40 tests**, raising the suite total from 1805 toward ~1840.

### Registry + builder-shape tests (~8)

- `__tests__/visualQaFixtures.test.ts` — `VISUAL_QA_FIXTURES.length === 8`.
- — every `VisualQaFixtureId` union member appears exactly once in the registry.
- — every descriptor's `build` is a function and its `id` matches the builder.
- — `getVisualQaFixture` returns a descriptor for each known id and `null` for `'bogus'`.
- — every builder returns `length >= 1`.
- — every builder is deterministic: `build()` deep-equals `build()`.
- — every fixture's node ids are unique within the fixture.
- — no fixture node uses an `argumentType` that resolves to the `'default'` color family.

### Per-fixture structural-invariant tests (~16, two per fixture)

For each of the 8 fixtures, build the timeline map (`buildArgumentTimelineMap`) and assert the
exact invariant from the §"Data model" table — happy-path + the one distinguishing edge:

- `no_rebuttal` — one node, `isRoot`, `hasRebuttal === false`, `edges.length === 0`,
  `rootOnboardingHint` non-null.
- `straight_chain_10` — `nodes.length === 10`, deepest `depth === 9`, **every `lane === 0`**,
  no `isJunction`.
- `source_chain_fight` — ≥ 2 `clarify`-family nodes with a `source_request` dropped tag, ≥ 1
  `evidence`-family node, zero `synthesis`/`concession` nodes.
- `evidence_heavy_branch` — ≥ 3 `evidence`-family nodes, an `Evidence run` band present in
  `map.bands`, ≥ 1 node with `lane !== 0`.
- `tangent_kink_branch` — exactly one `isJunction` node (`junctionChildCount >= 2`), ≥ 1 node
  `lane !== 0`, exactly one `isDetached` node, that node's `accessibilityLabel` contains
  `'detached'`.
- `synthesis_path` — exactly one `synthesis` node, it is chronologically last, ≥ 1
  `concession`-family node.
- `stress_board_250` — `nodes.length === 251`, 50 detached, unique node + edge ids, `x`
  strictly monotonic.
- `avatar_profile_display` — ≥ 3 distinct `authorId`, `participantTrends.length >= 3`; building
  with `currentUserId = AVATAR_PROFILE_DISPLAY_CURRENT_USER_ID` yields one node whose
  `actorLabel === 'You'` and one with `isBot`-driven actor (`'Bot'`).

### Lifecycle cross-check tests (~3)

A subset of fixtures is also fed through `buildPointLifecycleMap` (the LIFE-001 builder) to
prove the fixtures are usable by the *other* analytics consumers (AN-003), not just the timeline
map:

- `synthesis_path` produces a `PointLifecycleMap` with a cluster that reaches a resolved-family
  state.
- `source_chain_fight` produces a cluster carrying an open source/quote request.
- `no_rebuttal` produces exactly one cluster in an `open` state.

(If `buildPointLifecycleMap`'s input contract requires a projection AN-002 fixtures do not
directly satisfy, the implementer drops these 3 tests and notes it — they are a *nice-to-have*
cross-check, not the card's acceptance criterion. The 8 timeline-map invariant tests are the
acceptance criterion.)

### Doctrine ban-list tests (~3)

Following the `test-discipline` ban-list pattern:

## Risks

- **R1 — Builders drift from the renderer they verify.** If `computeLane` or the band heuristic
  changes, a fixture invariant test breaks. *This is the point* — the test suite is the
  regression net. The risk is only that an implementer "fixes" a failing fixture test by
  loosening the assertion. Mitigation: each invariant is stated tightly in the §"Data model"
  table and cross-referenced from the checklist doc, so loosening it is visibly wrong.
- **R2 — The checklist doc and the registry diverge.** A reviewer trusts the doc; if the doc
  names a fixture the code dropped, QA wastes time. Mitigation: the doc/structural test in
  §"Test plan" asserts every `VisualQaFixtureId` appears in `docs/visual-qa-snapshots.md`. The
  doc cannot silently drift.
- **R3 — Duplicate stress fixtures.** After AN-002, there are two 250-node stress fixtures: the
  existing inline `buildStressFixture` in `argumentTimelineMap.test.ts` and the new
  `buildStressBoard250Fixture`. They are deliberately kept *identical in count* (`251` total,
  `50` detached) so no number is ambiguous, but they are separate code. This is accepted to keep
  AN-002's blast radius to new files only. A one-line follow-up card ("dedupe: point
  `argumentTimelineMap.test.ts` at `buildStressBoard250Fixture`") may be filed; AN-002 does not
  do it, because refactoring an existing green test is out of this card's scope and would risk
  its 5 stress assertions.
- **R4 — `buildPointLifecycleMap` input mismatch.** The 3 lifecycle cross-check tests assume
  `buildPointLifecycleMap` accepts something derivable from the fixtures. If its input contract
  needs fields AN-002 fixtures don't carry, those 3 tests are dropped (documented inline) — they
  are a bonus, not the acceptance criterion. The implementer must verify the LIFE-001 input
  shape before writing them.
- **R5 — Test count.** This card adds ~32–40 tests. The `test-discipline` rule is "test count
  goes up." The implementer captures the exact new count and updates `docs/core/current-status.md`
  only after `npm run test` confirms it.
- **R6 — Library/platform gotchas.** None. Pure TS, no new dependency, no platform API. The
  only `fs` use is in two tests reading a committed doc — a standard pattern already used by
  `treePlayabilityDiagnostics.test.ts` and `syncUxProjectBoard.test.ts`.

## Out of scope

AN-002 explicitly does **NOT** include:

- **Any browser automation.** No Playwright, Puppeteer, Cypress, headless Chrome. The "browser
  visual checklist" is a *human-walked Markdown doc*. The agent does not open a browser.
- **Any visual-regression image-diffing.** No screenshot capture, no pixel-diff tooling, no
  `jest-image-snapshot`, no new snapshot dependency. If image-diffing were ever wanted it would
  need a new dependency — that is a separate future card, explicitly not AN-002.
- **A dev harness screen.** AN-002 ships fixtures + a doc, not an in-app "render any fixture"
  screen. A harness screen is a plausible follow-up card; it is not in this one. (When such a
  screen exists, it can `import { VISUAL_QA_FIXTURES }` — the registry is designed for it — but
  AN-002 builds no UI.)
- **A Markdown snapshot generator / dev script.** Unlike AN-003 (which has a `tsc`-precompiled
  snapshot script), AN-002 has no machine-generated output. The checklist is hand-authored
  static Markdown. No `scripts/` file, no `tsconfig.diagnostics.json`, no `package.json` script.
- **Refactoring AN-001 / AN-003** or the inline stress fixture in `argumentTimelineMap.test.ts`
  (see §Risks R3).
- **Any product behavior, migration, Edge Function, schema change, RLS policy, or `.env`
  change.**
- **Wiring fixtures into AN-003's diagnostics script.** AN-003's §5 script may later use these
  fixtures; making that connection is AN-003's (or a follow-up's) job.
