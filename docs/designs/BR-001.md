## Goal

BR-001 replaces VG-002's three-rule placeholder (`derivePlaceholderBranchKind`) with the real branch-topology classifier the rail was designed to consume. The card body asks for four observable behaviors:

1. **First child may stay on parent lane for mainline continuity.**
2. **Additional children branch up or down.**
3. **Source-chain / evidence branches get matching visual style.**
4. **Collapse hides children but leaves a stub.** Active node inside a collapsed branch auto-expands.

The locked VG-002 seam (`RailBranchKind`, `RailSegmentInput`, dispatch table, `GradientWaveRail.tsx`) does NOT change. BR-001 ships:

- a new pure-TS module `src/features/arguments/branchTopologyModel.ts` exporting `deriveBranchKindFromConstitutionModel` (the BR-001 helper VG-002 anticipated by name) plus a thin adapter `derivePlaceholderBranchKind` with the **same exact signature** the existing module already exports, so the renderer's call site changes from "consume the placeholder" to "consume the BR-001 helper" without touching `GradientWaveRail.tsx`;
- the collapse-state shape (`BranchCollapseState`) and a pure helper that produces a *pre-collapsed* `RailSegmentInput[]` to feed the existing rail renderer — collapse never reaches into the renderer;
- an auto-expand rule that resolves "active node inside a collapsed branch" by **silently uncollapsing** any ancestors of `map.activeNode.messageId` so the active path is always visible;
- a stub view-model that the room shell renders alongside the rail (a tiny RN `Pressable` chip whose visual style consumes existing VG-001 tokens — no new color or stroke tokens introduced).

The card is doctrine-anchored on one sentence: **a tangent is a topology label, not a verdict**. "Tangent" describes that a child argument leaves the main thread; it does NOT describe whether the child is wrong, off-topic-as-a-content-judgment, low-value, popular, hot, viral, supported, or true. Tests assert this via the same ban-list pattern VG-002 introduced.

BR-001 ships **no new dependency**, **no migration**, **no Edge Function**, **no schema change**, **no AI inference path**, **no Supabase write**, and **no `.env*` change**. The renderer, the rail's enum, the rail's dispatch table, and EV-001/EV-002 surfaces stay frozen.

## Topology model — decision table

The five-value enum (`'main' | 'tangent' | 'kink_start' | 'kink_end' | 'detached'`) is locked at design time by VG-002's surface lock. BR-001 owns the deterministic *population* of those values from the existing argument-game-surface model. The classifier consumes one edge plus its endpoint nodes plus an "evidence-thread membership" precomputation (one O(n) pass over the tree, see §"Evidence-thread detection").

Inputs to the classifier per edge:

- `fromNode` and `toNode` (`ArgumentTimelineMapNode` — already carries `kindColorFamily`, `droppedTags`, `siblingIndex`, `branchId`, `branchRootMessageId`, `replyCount`, `descendantCount`, `isJunction`, `parentId`).
- `siblingIndex` of `toNode` (from `argumentGameSurfaceModel`'s pass 1; 0-indexed chronological among `toNode.parentId`'s direct children).
- `isDetached` (parent missing from the loaded set — already on the edge today).
- `isEvidenceThread` (BR-001-derived per branch — see §"Evidence-thread detection"; one boolean per `branchRootMessageId`).
- `tangentLexicalCodes` on either endpoint's `droppedTags` (`'branch_this_off'`, `'tangent_or_joke'` — these are existing qualifier codes already populated by `mapDroppedTags`).

Decision table (16 rows — exhaustive over the orthogonal axes the enum cares about). The implementer codes the table verbatim; each row maps `(isDetached, siblingIndex, isEvidenceThread, hasTangentLexicalCode) → branchKind`. The classifier never reads color tokens, never reads heat / popularity / engagement, never reads strength bands.

| # | isDetached | siblingIndex | isEvidenceThread | hasTangentLexicalCode | → branchKind | Why |
|--:|:---:|:---:|:---:|:---:|---|---|
| 1 | true  | —   | —     | —     | `detached` | parent missing — VG-002 row 10; strip not rendered |
| 2 | false | 0   | true  | —     | `main`     | evidence-thread first child stays on parent lane (AC #1 + AC #3) |
| 3 | false | 0   | false | false | `main`     | non-evidence first child stays on parent lane (AC #1) |
| 4 | false | 0   | false | true  | `kink_start` | first child but explicitly tagged tangent → leaves mainline immediately (rare; only fires when the qualifier deriver tagged the move) |
| 5 | false | ≥1  | true  | —     | `main`     | additional evidence-thread sibling — evidence branches keep saturated trail (AC #3) — lane assignment alternates, but the rail style stays `main` |
| 6 | false | ≥1  | false | false | `kink_start` | additional non-evidence sibling — this edge IS the branch leaving mainline (AC #2) |
| 7 | false | ≥1  | false | true  | `kink_start` | additional sibling explicitly tagged tangent — same as #6, more explicit |
| 8 | false | —   | —     | —     | `kink_end` | this edge is the LAST edge of a tangent subtree (its `toNode` has `replyCount === 0` AND lives on a `kind_start`-rooted branch). Computed in a second pass — see "Two-pass algorithm" below. |
| 9 | false | —   | —     | —     | `tangent` | interior edge of an already-kinked branch (parent of this edge has `branchKind === 'kink_start'` OR `'tangent'`). Computed in a second pass. |
| — | — | — | — | — | — | (rows 10–16 reserved for future enum extensions; the enum is locked, so no further values can be produced today — design slot kept for symmetry with VG-002's matrix) |

Rows 1–9 partition the input space exhaustively. The classifier is a single switch over the four boolean axes; the second pass is a single walk of the edge list keyed by parent edge's `branchKind`.

### Why this row table and not the three alternatives

- **(a) "First child = main, additional children = tangent (chosen — rows 2/3 vs 6/7).** This is the simplest deterministic rule and matches the existing Stage 6.3 lane-continuity fix. Reading the lane model: when `siblingIndex === 0`, `computeLane` already keeps the child on the parent's lane; when `siblingIndex ≥ 1`, lanes alternate above / below. The kink visual is the rail-surface mirror of that lane behavior. The two surfaces agree.
- **(b) Argument-type-aware (rejected as primary).** Letting argument-type drive `main` vs `tangent` would force the rail to know that, e.g., a `rebuttal` stays on lane regardless of sibling order. That is *already* how the lane model behaves when the rebuttal is the first sibling, and it *isn't* when the rebuttal is the third sibling — because a third rebuttal on the same parent IS a topological branch. Argument-type can't override the topology truth. **However**, argument-type still matters for evidence-thread detection (row 5), so option (b) is partially folded in via §"Evidence-thread detection".
- **(c) Hybrid (rejected as primary).** A "first child + on-topic = main; first child + topic-shift = tangent" rule would require BR-001 to take a position on what "on-topic" means at the topology layer — that decision lives in the moderator pipeline / `messageQualifiers.deriveMessageCategory`, not here. We surface the explicit `branch_this_off` / `tangent_or_joke` qualifier codes (row 4) only because they are user/moderator-tagged signals, not topology inferences.

### Two-pass algorithm

The classifier needs to know whether an edge is **interior** to a tangent subtree (`tangent`, row 9) or the **last** edge of one (`kink_end`, row 8). Both require knowing the classification of the **parent edge**. Solution:

1. **Pass 1** — walk edges in chronological order, applying rows 1–7 (the four-axis switch). Output: a `Map<edgeId, RailBranchKind>` where every edge is one of `'main'`, `'kink_start'`, `'detached'`.
2. **Pass 2** — walk edges again; for each non-`main`/non-`detached` parent in the map, mark its child edges as `'tangent'` (row 9). For each `'tangent'` or `'kink_start'` edge whose `toNode.replyCount === 0` OR whose only descendant edges are all also tangents AND its `toNode` is the latest message in that subtree, mark it `'kink_end'` (row 8). The "last edge of the tangent" is uniquely identified by `descendantCount === 0` within the tangent subtree.

Both passes are O(n) over edge count. No nested walks; each edge is touched at most twice.

## Lane assignment

BR-001 does NOT replace `computeLane` in `argumentGameSurfaceModel.ts`. The Stage 6.3 fix already implements the policy the card body asks for (first child on parent's lane, additional siblings alternating). The relevant rule, verbatim from `computeLane`:

```ts
if (siblingIndex === 0) {
  // First child stays on the parent's lane (parent-lane continuity).
  return clamp(parentLane);
}
const direction = siblingIndex % 2 === 1 ? -1 : 1;
const offset = Math.ceil(siblingIndex / 2);
return clamp(parentLane + direction * Math.min(offset, maxLane));
```

That gives a deterministic, sibling-index-stable lane:

- `siblingIndex 0` → parent lane (continuation; `main` in BR-001's table).
- `siblingIndex 1` → parentLane − 1 (above).
- `siblingIndex 2` → parentLane + 1 (below).
- `siblingIndex 3` → parentLane − 2 (further above).
- `siblingIndex 4` → parentLane + 2 (further below).
- … clamped to ±3 (`TIMELINE_MAX_LANES / 2`).

BR-001 introduces **no change** to `computeLane`. The kink classifier reads `toNode.siblingIndex` (already populated by `argumentGameSurfaceModel`'s pass 1) to decide between rows 3 vs 6 in the decision table. The rail's visual then renders the kink at the edge where the lane jumps — which the renderer already does today, because the wrapper geometry uses `(x1,y1)→(x2,y2)` and a non-parent lane produces a visibly angled strip.

For evidence-threads (row 5), lanes still alternate as `computeLane` dictates — BR-001 does NOT force evidence-thread siblings back onto the parent lane. The branch *visual style* stays `main` (saturated teal evidence track, no opacity dimming) but the *geometry* still angles up or down. This is intentional: the card body says "matching visual style", not "same lane". Coherent evidence threads remain visually grouped via the teal evidence-track overlay VG-002 already renders.

### Deterministic tie-breaks

- `siblingIndex` is already deterministic — populated chronologically by `argumentGameSurfaceModel`'s pass 1 (oldest `createdAt` wins, then id-sort).
- When two siblings share a `createdAt` (rare; bot test fixtures), the existing id-sort in `sortMessagesChronologically` is the tie-break.
- Lane clamping at ±3 means siblings 7+ all collide at the maxLane; the existing `occupied` collision-nudge in pass 1 handles vertical separation. BR-001 does not need to introduce a new clamp.

## Evidence-thread detection

The card body's AC #3 — "Source-chain / evidence branches get matching visual style" — means the BR-001 classifier must NOT tag a coherent evidence sub-conversation as `tangent`. Evidence threads are visually distinguished by the EV-001 → EV-002 → VG-002 stack already in place (teal saturated track on the rail when `sourceChainStatus ∈ {source_and_quote, primary_present}`). BR-001's job is the negative: do not strip that styling by mis-classifying.

### Rule (one sentence)

A subtree is an **evidence-thread** when at least 50 % of its direct messages (excluding the root of the subtree) carry an evidence-related argument type or qualifier, AND the subtree contains at least 2 messages.

### Inputs the rule consumes

Per `toNode` whose `parentId !== null`:

- `argumentType` from `ArgumentTimelineMapNode.kindColorFamily` — evidence family is `'evidence'` (already mapped from the existing `KIND_COLOR_FAMILY` table: `evidence`, `source`).
- `droppedTags` — evidence qualifier codes: `'evidence'`, `'evidence_challenge'`, `'source_request'`, `'quote_request'`. Already populated by `mapDroppedTags`.
- Future-compat: `MessageQualifier` codes `'ask_receipts'`, `'quote_exact_bit'`, `'evidence_challenge'` from `messageQualifiers.ts`. BR-001 reads these from the qualifier code already attached to the node's `droppedTags` (the existing surface) — it does NOT call `deriveMessageCategory` directly. (This avoids re-deriving per render.)

### Algorithm (O(n))

One pass over `nodes`, walking parent → child via the existing `parentId` map:

```text
1. Build childIdsByParent: Map<parentId, string[]> from nodes.
2. For each node in chronological order:
     if node has parentId AND node is the branch root of its branch
         (i.e. node.branchRootMessageId === node.messageId — which the existing
         pass 2 in argumentGameSurfaceModel computes for every junction's
         non-first-child):
       count = 0
       evidenceCount = 0
       stack = [node.messageId]
       while stack not empty:
         id = stack.pop()
         if id !== node.messageId:
           count += 1
           if isEvidenceLike(nodeById[id]) evidenceCount += 1
         for childId in childIdsByParent[id] || []:
           stack.push(childId)
       isEvidenceThread = count >= 2 AND evidenceCount / count >= 0.5
       evidenceThreadByBranchRoot[node.messageId] = isEvidenceThread
3. For non-branch-root nodes, inherit the parent's evidenceThreadByBranchRoot
   value (look up via node.branchRootMessageId).
```

Result: a `Map<branchRootMessageId, boolean>` consumed by the classifier in pass 1. One pass, visited-set guards against cycles (the existing `argumentGameSurfaceModel` already guarantees acyclic; BR-001 adds a defensive `visited` set anyway).

### Why this rule and not the alternatives

- **A "≥ 50 % evidence-typed nodes" rule** is robust to one Q&A clarification interleaved into an evidence thread, and rejects threads where evidence is incidental rather than dominant.
- **A "≥ 1 evidence node" rule** (rejected) would tag any subtree with a single `ask_for_source` reply as an evidence-thread, which is too aggressive and would mis-classify ordinary side-conversations.
- **A "root has evidence type" rule** (rejected) would fail for threads where the evidence move is the *response*, not the trigger — exactly the case AC #3 is asking us to handle.

### What this rule does NOT do

- It does NOT read `sourceChainStatus` (that comes from EV-001's `summarizeArtifactsForReceiptChip` and lives on the rail's evidence-track layer, not BR-001's topology classifier).
- It does NOT decide whether the evidence is *good*. "Evidence-thread" describes that the conversation is *about* evidence; it never claims the evidence is sufficient, primary, or correct.
- It does NOT change the rail's teal evidence-track overlay — that is EV-001/EV-002/VG-002 territory.

## Collapse / stub model

### State shape (pure-TS, no React)

```ts
/** In-memory only. Not persisted in v1. Keyed by the branch's root message id
 *  (matches `ArgumentTimelineMapNode.branchRootMessageId`). */
export type BranchCollapseState = Readonly<Record<string, 'expanded' | 'collapsed'>>;

/** Empty default — nothing is collapsed at first render. */
export const EMPTY_COLLAPSE_STATE: BranchCollapseState;

/** Toggle one branch. Returns a new state object (immutable update). */
export function toggleBranchCollapse(
  state: BranchCollapseState,
  branchRootMessageId: string,
): BranchCollapseState;

/** Auto-expand: given the active message id + the node map, return a state
 *  where every ancestor branch root of the active path is `'expanded'`,
 *  preserving the existing collapse state for unrelated branches. */
export function applyActiveAutoExpand(
  state: BranchCollapseState,
  activeMessageId: string | null,
  nodeById: ReadonlyMap<string, ArgumentTimelineMapNode>,
): BranchCollapseState;
```

A branch is **collapsed** when its `branchRootMessageId` maps to `'collapsed'`. Anything not in the map is treated as `'expanded'` (the default). Storing only collapsed entries keeps the object small.

### Pre-collapsed rail-input feeder

```ts
/** The function the room shell calls. Walks the edge list, filters out
 *  segments inside collapsed branches, and replaces each collapsed branch
 *  with a single `RailStubViewModel` so the renderer continues to receive
 *  the same `RailSegmentInput[]` it already understands.
 *
 *  Per VG-002's surface lock: this function does NOT alter `RailSegmentInput`
 *  in any way and never emits a new `RailBranchKind` value. Stubs render
 *  outside the rail (as a separate `<Pressable>` chip from the room shell). */
export function buildCollapsedRailInputs(args: {
  segments: ReadonlyArray<RailSegmentInput>;
  nodeById: ReadonlyMap<string, ArgumentTimelineMapNode>;
  collapseState: BranchCollapseState;
}): {
  visibleSegments: RailSegmentInput[];
  stubs: RailStubViewModel[];
};
```

Each `RailStubViewModel` is the input the stub `<Pressable>` consumes:

```ts
export interface RailStubViewModel {
  /** Stable id derived from the branch root. */
  stubId: string;
  /** The branch root message id. Tap re-expands this branch. */
  branchRootMessageId: string;
  /** Anchor pixel position (where the branch leaves the mainline). */
  anchorX: number;
  anchorY: number;
  /** Number of messages hidden inside the collapsed branch. */
  hiddenMessageCount: number;
  /** Plain-English label, e.g. "3 hidden replies on the source-trail branch." */
  label: string;
  /** Plain-English accessibility label (per `accessibility-targets`). */
  accessibilityLabel: string;
  /** When true, the stub renders a subtle "active inside" indicator
   *  because the room's `activeMessageId` is in this collapsed subtree
   *  BUT auto-expand was suppressed by the caller (rare; defensive). */
  containsActive: boolean;
}
```

### Stub rendering contract

The stub is a small RN `<Pressable>` rendered **adjacent to the rail**, anchored to the parent edge's child endpoint `(x, y)`. It is **not** a rail segment — the rail's `pointerEvents: 'none'` invariant from VG-002 is preserved.

- Visual: a 24-wide × 24-tall pill `<View>` containing a `<Text>` count badge (e.g. `+3`). Background uses VG-001's `BRAND.surface.appElevated`. Border uses the parent's `node.kindColor` so the stub visually inherits the branch's family color (no new color token).
- Tap target: `hitSlop` of 14px each side so the effective tap target is ≥44px. Per `accessibility-targets` minimum.
- `accessibilityRole: 'button'`, `accessibilityLabel: '3 hidden replies on the source-trail branch. Tap to expand.'`, `accessibilityState: { expanded: false }`. Plain language. No verdict tokens.
- On tap: room shell calls `toggleBranchCollapse(state, branchRootMessageId)` and re-renders.

### Auto-expand rule

**When `map.activeNode.messageId` lives inside a collapsed branch, silently expand every collapsed ancestor branch of the active node so the active path is always visible.**

`applyActiveAutoExpand(state, activeMessageId, nodeById)` walks from the active node up the `parentId` chain, removing any `branchRootMessageId` it encounters from the collapsed set. It is called by the room shell after every active-node change, *before* `buildCollapsedRailInputs`. If the user manually collapses a branch that contains the active node, the room shell respects the user's collapse for one render, then a follow-up active-node-change re-expands. The simpler rule — *active path is never hidden* — wins.

When auto-expand changes the visible structure, the room shell fires `AccessibilityInfo.announceForAccessibility('Branch expanded to show the active move.')` per the `accessibility-targets` skill's guidance on dynamic-change announcements.

### Persistence

**Collapse state is in-memory only. Not persisted across sessions.** The room shell keeps it in a `useState<BranchCollapseState>` and it resets when the room mounts. A future card (none currently scheduled — flag as v2 if asked) can add Supabase-backed persistence via `profiles.preferences` per PR-001. BR-001 does NOT introduce persistence, does NOT introduce a migration, and does NOT touch any Supabase surface.

### Reduce-motion

The expand/collapse transition is a **static swap** in v1 — no animation. `AccessibilityInfo.isReduceMotionEnabled()` does not change behavior. A future card may add a brief opacity-fade; that card MUST gate on the preference.

## Performance plan

### Complexity

The classifier runs in two well-bounded passes plus one evidence-thread pre-walk:

| Step | Cost | Notes |
|---|---|---|
| Build `childIdsByParent` map | O(n) | one pass over `nodes` |
| Evidence-thread per-branch-root pre-walk | O(n) amortized | each node visited once across all branch-root walks (the visited-set guarantees no node is walked twice); the inner stack is bounded by subtree size, summed = `n` |
| Pass 1 — four-axis switch per edge | O(e) ≈ O(n) | `e ≤ n − 1` for a tree |
| Pass 2 — interior `tangent` + `kink_end` | O(e) ≈ O(n) | one walk over edges |
| `buildCollapsedRailInputs` filter | O(e) | one pass; skips segments inside collapsed branches |

**Total: O(n)** per render, where `n` = message count.

### Memoization

The classifier's output is a `Map<edgeId, RailBranchKind>` derived from immutable inputs. The room shell caches this map keyed by:

```ts
(treeHash, collapseSetHash)
```

where `treeHash` is a stable hash of `(nodes.length, lastNode.messageId, lastNode.updatedAt ?? lastNode.createdAt)` and `collapseSetHash` is a sorted-key concatenation of `BranchCollapseState`. Identical inputs → reference-equal outputs. The classifier itself never mutates inputs.

The `activeMessageId` is NOT in the memoization key for the classifier — `branchKind` is independent of which node is active. `isActivePath` lives on the existing `ArgumentTimelineMapEdge` and is already memoized in `argumentGameSurfaceModel`. Auto-expand (which DOES depend on `activeMessageId`) runs against the cached classifier output without recomputing it.

### Back-of-envelope at 250 messages

- 250 edges (one per non-root node).
- Pass 1 + Pass 2 + evidence-thread pre-walk ≈ 3 × 250 = **750 simple comparisons** per uncached render. At ~10 ns each on V8, ≈ **7.5 µs**. Imperceptible.
- Memoized re-renders return the cached map immediately — O(1).
- Memory: ~250 entries × ~50 bytes = ~12 KB for the classifier map.

This is well under VG-002's already-virtualized rendering budget (peak ~330 `<View>` instances regardless of message count). BR-001 introduces zero rendering cost — it only changes what `branchKind` value the existing renderer reads.

## API / interface contracts

### `src/features/arguments/branchTopologyModel.ts`

```ts
import type {
  ArgumentTimelineMapEdge,
  ArgumentTimelineMapNode,
} from './argumentGameSurfaceModel';
import type { RailBranchKind, RailSegmentInput } from './railSegmentModel';

// ── Evidence-thread detection ─────────────────────────────────

/** Argument-types / qualifier codes that count as "evidence-like" for the
 *  purposes of evidence-thread detection. Reads existing node fields only —
 *  does NOT re-derive `MessageCategory`. */
export function isEvidenceLikeNode(node: ArgumentTimelineMapNode): boolean;

/** Build a `Map<branchRootMessageId, boolean>` over the tree. One O(n) pass
 *  with a visited-set guard. Returns a frozen map. */
export function buildEvidenceThreadMap(
  nodes: ReadonlyArray<ArgumentTimelineMapNode>,
): ReadonlyMap<string, boolean>;

// ── Topology classifier ────────────────────────────────────────

export interface DeriveBranchKindInput {
  fromNode: ArgumentTimelineMapNode;
  toNode: ArgumentTimelineMapNode;
  isDetached: boolean;
  /** Chronological sibling index of `toNode` among `toNode.parentId`'s direct
   *  children. Already populated on `toNode.siblingIndex`. */
  siblingIndex: number;
  /** Whether the subtree rooted at `toNode.branchRootMessageId` is an
   *  evidence-thread per `buildEvidenceThreadMap`. */
  isEvidenceThread: boolean;
  /** Whether either endpoint carries an explicit tangent qualifier code
   *  (`'branch_this_off'` or `'tangent_or_joke'`) in `droppedTags`. */
  hasTangentLexicalCode: boolean;
  /** Parent edge's branch kind, when known (pass 2 only). Pass 1 leaves
   *  this `undefined`. */
  parentEdgeBranchKind?: RailBranchKind;
  /** Whether `toNode` is a leaf in the tangent subtree (computed in pass 2). */
  isTangentSubtreeLeaf?: boolean;
}

/** Pure. Deterministic. Same input → identical output. Never reads color
 *  tokens, heat, popularity, engagement, strength bands, or any AI signal. */
export function deriveBranchKindFromConstitutionModel(
  input: DeriveBranchKindInput,
): RailBranchKind;
```

## Edge cases

The implementer MUST handle each. Tests are listed in §"Test plan".

1. **Root-only tree (`nodes.length === 1`, `edges.length === 0`).** `buildBranchKindMap` returns an empty map. `buildEvidenceThreadMap` returns an empty map. `buildCollapsedRailInputs` returns `{ visibleSegments: [], stubs: [] }`. No stubs render.
2. **Single-message tree (alias of #1).** Same.
3. **All-tangent tree.** Every edge from row 6 of the table. Every edge → `kink_start` (pass 1) → `tangent` (pass 2) → last leaf → `kink_end`. Renderer applies VG-002's tangent + kink visuals on each. No stubs by default (nothing collapsed).
4. **All-evidence-thread tree.** Every branch is an evidence-thread. Every edge stays `main` regardless of `siblingIndex`. Rail renders saturated teal evidence track via VG-002. The evidence-thread map flips to `true` for every `branchRootMessageId`.
5. **Deeply nested tangents (e.g. tangent off tangent off tangent).** Pass 2 walks parent-edge `branchKind` correctly — each interior edge becomes `tangent` regardless of depth. Auto-expand walks the full `parentId` chain.
6. **Circular references.** Defensive `visited` set in `buildEvidenceThreadMap` guards. The existing `argumentGameSurfaceModel` already guarantees acyclic via `idIndex`; the defensive guard is belt-and-suspenders.
7. **Deleted nodes (`is_deleted: true` upstream).** The room shell already filters deleted nodes before passing them to the timeline model (consistent with Stage 6.1.8 soft-delete contract). BR-001 sees them as if they don't exist. No special handling needed.
8. **`activeMessageId === rootMessageId`.** Auto-expand walks `parentId` chain, hits `null`, returns unchanged state. No-op.
9. **`activeMessageId === null` (no active node).** `applyActiveAutoExpand` returns state unchanged. No-op.
10. **Collapsed branch with no children (degenerate).** `hiddenMessageCount === 0` would produce an empty stub. The stub-builder skips any branch whose `hiddenMessageCount === 0`. The collapse state entry stays, but no stub renders.
11. **Active node inside double-collapsed branch.** Auto-expand walks from the active node up `parentId`, removing every `branchRootMessageId` in the collapsed set along the way. Both ancestors expand atomically.
12. **Detached node** (parent missing from loaded set). Row 1 → `detached`. Strip not rendered. Same as VG-002 today.
13. **Sibling at clamp boundary** (`siblingIndex` produces a lane beyond ±3). `computeLane` clamps to ±3. BR-001's row choice does not depend on the lane number, only on `siblingIndex` — so the classification is correct even when the geometry is clamped.
14. **Concurrent edits / realtime new message arrives.** The room shell re-renders; `buildBranchKindMap` runs again with the new `nodes`. Memoization invalidates because `treeHash` changes. Cost: one O(n) pass.
15. **Branch root that is itself a junction with many evidence children.** The evidence-thread rule counts ≥ 50 % evidence-like among the **subtree**, so the junction qualifies regardless of how many non-evidence first-children's children exist. Aligns with AC #3.
16. **First child carries explicit `branch_this_off` qualifier (row 4).** This is the one place where the classifier overrides the "first child stays on lane" rule — but only at the rail-style level. `computeLane` still puts the first child on the parent's lane geometrically. The visual then renders a tiny kink-start stub at the parent end of the edge (VG-002 row 8). This is a narrow case (user / moderator explicitly tagged the move) and the rail's "explicit tangent" visual is the doctrine-safe way to surface that without re-running lane assignment.
17. **A user collapses every branch in a tree.** `buildCollapsedRailInputs` returns `visibleSegments` = the mainline only (segments NOT inside any collapsed branch), plus one stub per collapsed branch. The rail still has a baseline; navigation is preserved via stub-tap.

## Risks

1. **Evidence-thread detection over-tags ordinary side conversations as evidence.** The 50 % threshold is a heuristic; a 2-message branch with one evidence reply would qualify. Mitigation: the rule requires `≥ 2 messages` AND `≥ 50 %` — a single non-evidence reply pulls the threshold below 50 %. Test fixture exercises the boundary. If false-positives become a problem in production, the threshold can be tuned without changing any public API.
2. **The 4-axis switch (rows 1–9) misclassifies an edge case I have not anticipated.** Mitigation: decision-table coverage is exhaustive over the orthogonal axes; rows 1–9 partition the input space; row 4 (first-child + explicit-tangent-qualifier) is the narrow override and is fully tested.
3. **Auto-expand-on-active-change is chatty.** Every active-node tap re-runs `applyActiveAutoExpand`, which walks `parentId` up to the root. At depth 250 worst case = 250 lookups. Mitigation: O(depth) per active-node change; depth ≤ 50 in any realistic conversation. Tests assert < 5 ms at depth 250.
4. **Collapse state in-memory only.** Users may expect collapse to persist across reload. Mitigation: documented as v2 follow-up; no card currently scheduled. The user's typical reload behavior re-collapses nothing, which is the safer default (no branches hidden by surprise).
5. **The thin adapter in `railSegmentModel.ts` could drift from the new helper's semantics.** Mitigation: a regression test asserts the adapter's three-rule output exactly matches the new helper when given the no-BR-001-context input (`isEvidenceThread: false`, `hasTangentLexicalCode` derived from `droppedTags`).
6. **The implementer might be tempted to extend `RailSegmentInput` with `isEvidenceThread` to "do it right".** Mitigation: VG-002 surface-lock test asserts the field set is unchanged. The evidence-thread map is threaded as a side-channel via `buildRailSegmentInput`'s arg, not via `RailSegmentInput` itself.
7. **`buildBranchKindMap`'s two-pass algorithm could produce inconsistent results if pass 2 reads stale data.** Mitigation: pass 2 reads the `Map<edgeId, RailBranchKind>` pass 1 produces (no shared mutable state across passes); the map is built up monotonically; once an edge is set, it is not re-read by pass 1.
8. **`branchTopologyModel.ts` could be confused with `argumentGameSurfaceModel.ts` and the implementer may try to merge them.** Mitigation: explicit comment block at the top of the new file calling out the separation: `argumentGameSurfaceModel` owns layout (lanes, positions, bands, banner geometry); `branchTopologyModel` owns *topology classification* and *collapse state*. Do not merge.
9. **Stub anchor position (`anchorX`, `anchorY`) might drift relative to the rail when the rail is virtualized.** Mitigation: the anchor is computed from `toNode.x` / `toNode.y` of the branch root's first edge — the same coordinates the rail uses. Both surfaces see the same `(x, y)`. Tests assert with a fixture.
10. **The auto-expand announcement fires every time the active node changes, creating screen-reader noise.** Mitigation: announce only when auto-expand actually changed the collapse state (i.e. at least one branch was uncollapsed by the call). The room shell wraps the announce in `if (oldState !== newState)`.
