# QOL-030 — One-Box Composer + Flash-Popout Chassis — Design

**Card:** QOL-030 (Interaction · P0 · XL · Release 6.6 · foundation).
**Status:** Design.
**Companion:** [`docs/ux-storyboards/one-box-interface-model.md`](../ux-storyboards/one-box-interface-model.md) (overview + wireframes), [`docs/ux-storyboards/keyboard-map.md`](../ux-storyboards/keyboard-map.md).

## 0. How this design was built

This design is **derived, not asserted.** It is built in two grounded passes,
then the component model falls out as the consequence:

1. **Node stages in timeline mode** (§3) — what a node *is* at each point in its
   life, and what the timeline must offer when it is selected.
2. **Narrative walk-throughs** (§4–§5) — the two storyboard scenarios replayed
   frame by frame, recording exactly what the box and flash menu must do.

The component model (§6) is the *output* of §3–§5. Every box type, every flash-
menu rule, every schema exists because a narrative frame or a node stage
demanded it. Where the abstract model doc and the narrative disagree, **the
narrative wins** — see the `respond` finding in §4.

## 1. Goal & scope

Build the single switchable composer **box** and the shared **flash-popout
chassis** the whole MVP interaction surface stands on. In scope:

- The box as one component identified by `(type, target, view, stageContext)`.
- The flash menu (Act popout content model) — engine ∩ role ∩ stage.
- The popout chassis (`Popout` / `PopoutEntry` / `PopoutGroup`) reused by
  QOL-031/032/033.
- The per-type schema renderer (free-body / forced-list / structured-form /
  composite).
- Per-type draft buffers.

Out of scope (own cards): the Act/Inspect/Go popout *contents* beyond the chassis
(QOL-031/032/033), the keyboard layer (reopened IX-003), the evidence schema
internals (QOL-036/024), the concession-gradient internals (QOL-041).

## 3. Node stages in timeline mode

A node on the timeline is never just "a message" — it is a message **at a
lifecycle stage**. The stage drives both how the node renders and what the flash
menu promotes when that node is the target.

### 3.1 Stage vocabulary (LIFE-001)

`open · answered · rebutted · clarified · source_requested · quote_requested ·
sourced · narrowed · conceded · confirmed · synthesis_ready · moved_on ·
ignored · exhausted · branch_recommended · archived_or_resolved`.

### 3.2 Timeline rendering per stage

The node keeps its VG-001 *shape* (claim circle, challenge diamond, evidence
hexagon…) and gains a **stage indicator** — a non-color signal, reusing VG-004:

| Stage group | Stage indicator (timeline) |
|---|---|
| `open` | plain node |
| `answered` · `rebutted` | child-edge present on the rail |
| `source_requested` · `quote_requested` | dotted ring (evidence-debt) |
| `sourced` | receipt inner mark |
| `narrowed` · `conceded` | concession-pill tint |
| `confirmed` | solid stroke |
| `synthesis_ready` | merge glyph / double border |
| `moved_on` · `ignored` · `exhausted` | muted fill |
| `archived_or_resolved` | grey, locked |

### 3.3 The three gates

The flash menu, for a selected node, is computed by three gates in order:

1. **Engine gate (hard)** — `engine.ts`: argument types that are invalid
   children of this parent are **removed**.
2. **Role gate (hard)** — observer / participant / own-bubble: disallowed
   actions are **removed** (own-bubble keeps only qualifiers + request-deletion).
3. **Stage gate (soft)** — LIFE-001 stage: the fitting action is **promoted**
   to the top and visually emphasized; less-fitting valid actions are kept,
   ordered lower. **The stage gate never removes a valid move** — it orders.

This split is doctrine-critical: engine + role *filter*; stage only *suggests*.
The system never blocks a move because of a stage.

### 3.4 Stage → promoted action (the soft gate table)

| Node stage | Promoted (top of flash menu) | Kept, lower |
|---|---|---|
| `open` | Respond | Clarify · Ask source · Add evidence · Branch |
| `answered` | Respond (to the reply) | Challenge · Branch |
| `rebutted` | Respond (concede + refute) | Clarify · Add evidence |
| `clarified` | Respond | Challenge the clarification |
| `source_requested` | Add evidence *(author)* | Ask again · Branch |
| `quote_requested` | Add quote *(author)* | — |
| `sourced` | Respond to evidence | Add evidence · Challenge |
| `narrowed` | Confirm | Challenge the narrowed claim |
| `conceded` | Set acceptance gradient · Synthesize | Respond |
| `confirmed` | Synthesize | Branch a tangent |
| `synthesis_ready` | Synthesize | Respond |
| `moved_on` · `ignored` · `exhausted` | Branch | Respond *(de-emphasized, never blocked)* |
| `archived_or_resolved` | Share · Reference | *(read-only — no box)* |

Replaying [`roommates-dishes-public-argument.md`](../ux-storyboards/roommates-dishes-public-argument.md).

| Frame | Actor | Selected · stage | Box `(type · target)` | Flash menu | Result |

| F2 | B | Node 1 · `open` | **`respond` · Node 1** | promotes Respond | Node 2 · Node 1 → `rebutted` |
| F3 | A | Node 2 · `rebutted` | **`respond_to_concession` · Node 2's concession set** + refutation | gradient per item | Node 3 · active dispute = `fact` |
| F4 | B | Node 3 · `rebutted` | `respond · Node 3` | promotes Respond | Node 4 |
| F5 | Observer C | Node 4 · `open` | **`chime_in` · branch off Node 4** | observer menu: Watch · Join · Chime in | chime-in vertical branch node |
| F6 | A & B | chime-in node | governance reactions (**direct entries**) | Useful · Not useful · Hide · Convert · Let stand | chime-in → muted |
| F7 | A | **room** | make private (**direct entry, room target**) | room-level menu | room visibility = `private` |
| F8 | A | Node 4 | `respond · Node 4` | promotes Respond | Node 5 |
| F9 | B | Node 5 | `respond · Node 5` (+ raises "door unlocked") | — | Node 6 |
| F10 | A | Node 6 | Touché (**direct**) + flag-tangent (**direct**) | — | dishes line partial-resolved; "door" → diagonal tangent branch |
| F11 | B | tangent node | `respond · tangent node` | — | tangent node |
| F12 | A | tangent | fist-bump acknowledge (**direct**) | — | tangent → partial-resolved |
| F13–F14 | B, A | mainline | summary + settle (**direct**) | — | room `settled` → all nodes `archived_or_resolved`, board locked |

### 4.1 Design findings forced by Scenario 1

- **F2 → the `respond` composite type.** B's single move is *concessions
  (a list)* **and** *a refutation (a body)*. The abstract model treated
  `concede` and `challenge` as separate types — the narrative shows the common
  real move is **both at once**. **Decision:** the headline type for acting on a
  node is **`respond`** — a composite schema: an optional forced-list concession
  section + a refutation body. "Reply / Challenge / Concede / Clarify" in the
  flash menu are **entry points into the `respond` box**, each focusing one
  section. This collapses four types into one adaptive box.
- **F3 → `respond_to_concession` targets a *set*.** The target axis must accept
  a **concession set**, not only a node; the box renders a forced list mirroring
  it row-for-row (QOL-041 owns the gradient internals).
- **F5 → observer flash menu + branch target.** A non-participant's flash menu
  is participation actions (Watch / Join / Chime in), not compose types; a
  chime-in's target is **a branch off a node**.
- **F6/F7/F10/F12 → direct entries.** Governance reactions, make-private,
  Touché, fist-bump are **direct flash-menu entries — no box opens**.
- **F7 → `room` is a target.** Make-private targets the *room*. The target
  taxonomy must include `room`.
- **Every frame → the flash menu re-derives.** Each post advances the parent's
  stage; the flash menu is recomputed on every selection and every post.

| F10 | B | room | confirm-resolution (**direct**) | room locked; all nodes `archived_or_resolved` |

### 5.1 Design findings forced by Scenario 2

- **F2 → `add_evidence` is a structured form** with the QOL-036 payment fields;
  it is a *different schema kind* from `respond`. The box must render
  structured forms, not only bodies and lists.
- **F3 → `respond_to_evidence` targets an *evidence object*.** Another target
  kind. Its schema is the QOL-037 structured choice set + required clarification.
- **F5/F6 → stage drives the flash menu directly.** "Ask source" moves Node 4 to
  `source_requested`; from §3.4 that stage **promotes "Add evidence" for the
  author** — so on B's next visit the box practically pre-selects itself. This
  is the node-stage-driven design working end to end.
- **F1 → `root_claim` carries room setup** (title, visibility, invite) — the
  root box is the *only* box that also configures the room.
- **Evidence status is on the evidence object, not the node** — the node stage
  (`sourced`) and the evidence status (`applicability_disputed`) are separate
  axes; the box reads both.

## 6. Derived component design

Everything below is the consequence of §3–§5.

### 6.1 The box state machine

The box's identity at any moment:

```
Box = {
  type:        BoxType            // §6.2 — set by the flash menu
  target:      BoxTarget          // none | room | node | concessionSet
                                  //        | evidenceObject | branch
  view:        'timeline' | 'cards'
  stageContext: LifecycleStage | null   // the target node's LIFE-001 stage
  lifecycle:   'empty'|'typed'|'drafting'|'review'|'posted'|'parked'
  draftBuffers: Record<BoxType, Draft>  // per-type, never destroyed on switch
}
```

`What's in the box = type × target`. The type picks the schema *kind*; the
target populates the *instance* (e.g. `respond_to_concession` + a concession set
→ a list mirroring that set).

### 6.2 Box types (final set — narrative-derived)

| Type | Schema kind | Origin |
|---|---|---|
| `root_claim` | free-body + room setup | F1 both scenarios |
| **`respond`** | **composite** — optional concession forced-list + refutation body | **F2 Scenario 1** |
| `respond_to_concession` | forced list mirroring a concession set + gradient | F3 Scenario 1 |
| `respond_to_evidence` | structured choice set + required clarification | F3 Scenario 2 |
| `add_evidence` | structured form | F2 Scenario 2 |
| `ask_source` / `ask_quote` | short targeted prompt | F5 Scenario 2 |
| `clarify` | free-body, question-shaped | §3.4 |
| `narrow` / `confirm` / `synthesize` | preset free-body | §3.4 / `quickActionPresets` |
| `branch_tangent` | free-body + side-branch affordance | F10 Scenario 1 |

`Reply` / `Challenge` / `Concede` are **not** separate types — they are flash-
menu **entry points into the `respond` box**, each focusing one section.

### 6.3 The flash-menu (Act popout content) model

`buildActPopout(target, role, stage, engine) → PopoutGroup[]` — a pure function:

1. Enumerate candidate entries for the target kind.
2. **Engine gate** — drop entries whose argument type is an invalid child.
3. **Role gate** — drop entries the role forbids; own-bubble keeps only
   qualifiers + request-deletion.
4. **Stage gate** — order: the §3.4 promoted action first and emphasized; the
   rest kept, lower. Never removes.
5. Group: Respond · Evidence · Resolve · Structure · Direct · Role.

Direct and role-change entries (§4 findings) are first-class entry kinds beside
box-opening entries.

### 6.4 The popout chassis

`Popout` (anchored container · board-non-blocking scrim · 120–160 ms flash ·
focus trap · `Esc`), `PopoutEntry` (icon · label · key badge · enabled/disabled
+ reason · kind), `PopoutGroup`. Reused verbatim by QOL-031/032/033.

### 6.5 Schema renderer

`renderSchema(type, target)` → one of: **free-body** · **forced-list** ·
**structured-form** · **composite** (`respond` = forced-list section +
free-body section). Forced lists have **no single body field** — itemization is
structural; no AI parsing.

### 6.6 Draft buffers

One buffer per `BoxType`. Switching type or target **parks** the current buffer
and restores the destination's; never destructive. Seeds: the shipped
`ComposerDraftRecoveryNotice`.

## 8. Edge cases

- **LIFE-001 not yet built** — the stage gate degrades to a 3-stage derivation
  (`open` / `answered` / `archived_or_resolved`) from `arguments` rows alone;
  the flash menu still works, just with coarser promotion. No hard dependency.
- **Type-switch mid-draft** — parks; never destroys.
- **Target change mid-draft** — parks against the *original* target; explicit.
- **Engine says zero valid types** (e.g. on a locked node) — the flash menu
  shows only direct entries (Share, Inspect); the box does not open.
- **Observer with no compose rights** — flash menu = Watch / Join / Chime-in
  only; box stays minimized.
- **Stage and engine disagree** — engine always wins (a stage never enables an
  invalid transition).
- **Concurrent stage change** — the flash menu re-derives on focus; a stale
  popout is dismissed on the next render.

## 9. Test plan

- `boxModel` — type/target/view state transitions; per-type draft park/restore
  is lossless; `renderSchema` returns the right kind for every type.
- `actPopoutModel` — the 3-gate pipeline: engine drops invalid types · role
  drops forbidden · **stage only re-orders, never removes** (the doctrine test) ·
  the §3.4 promotion table · observer/own-bubble matrices.
- Narrative replay tests — Scenario 1 frames F1–F14 and Scenario 2 F1–F10
  reproduced: given the selected node + stage, assert the expected box type and
  flash-menu top entry.
- Chassis — focus trap, `Esc`, reduce-motion instant, 44px targets.
- Doctrine ban-list scan over every produced label.
- Source scan — no service-role, no direct insert, no AI/network import.

## 10. Doctrine & safety self-check

- The stage gate **only orders** — it can never block a move (§3.3). Verified by
  a named test.
- The flash menu is engine-gated + role-gated — it can only ever offer valid,
  permitted moves. No AI.
- All popout/box content models are deterministic pure functions — no network,
  no AI, no `Date.now()` in the model.
- The box posts only through `submit-argument`; no service-role, no direct
  insert into `public.arguments`.
- No verdict / winner / loser / truth copy; the box previews a node's *type and
  stage*, never its correctness. Heat is never an input.
- Terminology rules hold for every label (`npm run ux:terminology:audit`).
