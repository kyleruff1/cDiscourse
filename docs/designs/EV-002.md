## Goal

EV-002 is the first visible-UI surface for the evidence object model. It promotes the per-move artifact list (already produced by EV-001's `summarizeArtifactsForReceiptChip`) into a click-through **source-chain popover** anchored to the active timeline node, plus a chip in the existing `TimelineNodePopover`. The popover answers one inspection question — *where did this claim come from?* — and dispatches a non-accusatory action per `SourceChainStatus`.

The card encodes three doctrine anchors:

- **Status describes the trail, not the truth.** The popover never tells the user a claim is "wrong" or "proven"; it tells the user the trail's *shape* and offers an action that asks for more trail (cdiscourse-doctrine §1; evidence-doctrine "Source-chain status").
- **Popularity is not evidence.** No affordance references engagement, virality, follower count, "verified" badges, or trending state. The composer preset never offers "many people agree" as a substitute for a primary source (cdiscourse-doctrine §3; evidence-doctrine "What does NOT count as evidence").
- **Score never blocks posting.** The popover is purely an inspection / prompt surface. It never gates the composer, never removes existing controls, and never silences a move (cdiscourse-doctrine §1).

EV-002 ships:

1. A pure-TS dispatch model (`src/features/evidence/sourceChainPopoverModel.ts`) that maps a `SourceChainStatus` → `SourceChainPopoverAction` (label + preset key + tone + read-only flag + accessibility hint).
2. A composer preset extension — exactly three new entries in `quickActionPresets.ts` (`source` already exists; add the body-seed contract and add `weak_source` and `inspect_receipt` keys).
3. A `body` field added to `MoveDraftPatch` (and consumed in `ArgumentComposer.handleMovePatch`) so an "ask" preset can seed non-accusatory trail-language text the user can edit before posting.
4. A `SourceChainPopover.tsx` React Native component anchored under the active node when the `TimelineNodePopover` hosts an evidence-bearing message (or when the receipt-chip is tapped).
5. A `ReceiptChip.tsx` React Native chip that consumes `summarizeArtifactsForReceiptChip` and renders the locked EV-001 strings. The chip carries the dotted-teal source-chain ring per VG-001 when `showsSourceChainPressure === true`.
6. A wiring patch to `TimelineNodePopover.tsx`: receipt-chip + popover-open trigger live inside the existing popover surface; no new top-level overlay primitive.

No persistence change. No Edge Function. No migration. No new `submit-argument` mutation. No production AI call.

## State-to-action dispatch table

Locked at design time. The implementer MUST keep this table aligned with the strings already locked in EV-001's `RECEIPT_CHIP_COPY`.

| `SourceChainStatus`   | Popover headline              | Primary action label  | Preset key       | Read-only? | Composer body seed |
|-----------------------|-------------------------------|-----------------------|------------------|:----------:|--------------------|
| `no_source`           | "No source yet"               | "Ask for source"      | `source`         |    no      | `ASK_SOURCE_PRESET_BODY` |
| `unverified`          | "Receipt attached"            | "Ask for source"      | `source`         |    no      | `ASK_SOURCE_PRESET_BODY` |
| `source_no_quote`     | "Source attached"             | "Ask for quote"       | `quote`          |    no      | `ASK_QUOTE_PRESET_BODY` |
| `source_and_quote`    | "Source and quote"            | "Inspect receipt"     | (none)           |   **yes**  | (none — read-only) |
| `broken`              | "Source trail is weak"        | "Ask for stronger source" | `weak_source` |    no      | `ASK_STRONGER_SOURCE_PRESET_BODY` |
| `primary_present`     | "Source trail anchored"       | "Inspect receipt"     | (none)           |   **yes**  | (none — read-only) |

Notes the implementer MUST observe:

- **`unverified` resolution (per hard-constraint #3).** The card body says the popover should choose the most appropriate prompt — "likely 'Ask for source' if no URL, else 'Ask for quote'." Resolution: in v1 we always pick **"Ask for source"** for `unverified`. Rationale: per EV-001's decision table, the only path to `unverified` is `quote present, no URL, no sourceText` — i.e. there is *no source pointer at all*, only a free-floating quote. "Ask for quote" would be redundant (a quote is already attached); "Ask for source" matches the structural gap. The design test asserts this mapping.
- **Read-only states (`source_and_quote`, `primary_present`)** render the popover with a single neutral "Inspect receipt" affordance: tapping it expands the popover to show the artifact's `label` + `url` (if any) + `quote` (if any) in a non-interactive read-only block. No composer trigger, no preset, no body seed.
- **`broken` keeps trail language.** The headline reuses EV-001's locked chip label "Source trail is weak"; the action label is "Ask for stronger source" (no "broken", no "false", no "wrong", no person attribution).
- **No "request_deletion" / "flag" wiring.** The popover does not expose flagging or deletion paths. Users already have those affordances in the existing sidecar; the popover is inspection-only.

`invitesFollowup` from the existing `ReceiptChipContract` is the source of truth for "should we render the action?" — the popover MUST read `contract.invitesFollowup` directly and NOT re-derive it. The mapping above is consistent with EV-001's table: read-only states have `invitesFollowup === false`; the four "ask" states have `invitesFollowup === true`.

## Composer preset text — exact strings

Locked at design time. Every string is plain English, max ~140 chars, uses *trail / inspection* language, and is free of every verdict/person-label token in the cdiscourse-doctrine ban list. The implementer copies these verbatim into a new pure-TS constants module `src/features/evidence/sourceChainPresetCopy.ts` (so tests can import them).

```ts
/** Seeded composer body for `ask_for_source` from the source-chain popover. */
export const ASK_SOURCE_PRESET_BODY =
  "Could you point to the source you're working from here? A link, citation, or quoted excerpt would help me follow the trail.";

/** Seeded composer body for `ask_for_quote` when a source pointer exists but
 *  no excerpt is attached. */
export const ASK_QUOTE_PRESET_BODY =
  "Could you quote the specific passage in that source this is from? Pulling the exact line would tighten the trail.";

/** Seeded composer body for the `broken` state — asks for a stronger source. */
export const ASK_STRONGER_SOURCE_PRESET_BODY =
  "This trail looks weak — it dead-ends, cycles, or contradicts itself for me. Is there a more primary record we could anchor on?";
```

Implementer note: use **double-quoted** string literals (as above) so the apostrophe in "you're" does not require escaping. Single-quoted versions would need `\'`; double-quoted versions are cleaner. Either form is accepted by ESLint's default config in this repo — pick whichever is consistent with neighboring code.

Style notes the implementer MUST respect:

- First-person plural / second-person voice ("Could you…", "Is there…"). Never imperative ("Tell me…") and never accusatory ("You haven't…").
- Always asks a question. Never asserts a state of the author.
- The `broken` preset names the trail as weak (lifted from EV-001's chip helper "The chain dead-ends, cycles, or contradicts itself."). The preset MUST NOT call the author dishonest / manipulative / etc.
- Never references engagement / popularity / "many people".
- The user can edit the seeded body before posting. The composer's existing `evaluateArgumentDraft` validation runs over the final body, exactly as it does today.

A ban-list test enforces every string against the cdiscourse-doctrine forbidden tokens (see Test plan).

## API / interface contracts

Every export below is a stable contract. EV-003 will read the same artifact lists; EV-004 will read the same chip strings.

### `src/features/evidence/sourceChainPopoverModel.ts`

```ts
import type {
  ReceiptChipContract,
  SourceChainStatus,
  TimelineEvidenceContract,
} from './evidenceModel';

/**
 * The action a popover renders for a given SourceChainStatus. Pure data —
 * never references React, Supabase, or network.
 */
export interface SourceChainPopoverAction {
  /** Plain-English primary CTA label. Never a snake_case code. */
  label: string;
  /**
   * The QuickActionLabel passed to `quickActionToPreset(...)`. `null` for
   * read-only states (`source_and_quote`, `primary_present`).
   */
  presetKey: 'source' | 'quote' | 'weak_source' | null;
  /**
   * The ArgumentBubbleControl dispatched to the existing onAction callback
   * when the user confirms the action. `null` for read-only states.
   */
  bubbleControl: 'ask_for_source' | 'ask_for_quote' | null;
  /** Display tone for the CTA. Mirrors EV-001's chip tone for consistency. */
  tone: 'neutral' | 'info' | 'attention';
  /** True when this popover state opens a composer flow. */
  invitesFollowup: boolean;
  /** Screen-reader hint appended to the popover root accessibilityLabel. */
  accessibilityHint: string;
}

/**
 * The full popover view-model. Built once per artifact list. Pure.
 */
export interface SourceChainPopoverModel {
  /** Plain-English headline. Reuses EV-001's locked chip label verbatim. */
  headline: string;
  /** Plain-English helper. Reuses EV-001's locked chip helper verbatim. */
  helper: string;
  /** Underlying status. EV-003 may read this to drive its own branching. */
  status: SourceChainStatus;
  /** Total artifact count. 0 in the `no_source` aggregate state. */
  artifactCount: number;
  /** True when the dotted-teal source-chain ring should render on the node /
   *  chip. Mirrors `ReceiptChipContract.showsSourceChainPressure` so the
   *  popover and ring never drift. */
  showsSourceChainPressure: boolean;
  /** True for `source_and_quote` and `primary_present` — popover is inspection-only. */
  isReadOnly: boolean;
  /** The one primary CTA for this state. `null` only when `isReadOnly` is true. */
  primaryAction: SourceChainPopoverAction | null;
  /** Accessibility label for the popover root. Built deterministically. */
  accessibilityLabel: string;
}

/** Build the popover model from EV-001's `TimelineEvidenceContract`. Pure. */
export function buildSourceChainPopoverModel(
  contract: TimelineEvidenceContract,
): SourceChainPopoverModel;

/** Build the popover model from a `ReceiptChipContract` directly (used when
 *  the chip lives on a non-evidence node and the timeline contract isn't
 *  needed). Pure. */
export function buildSourceChainPopoverModelFromChip(
  chip: ReceiptChipContract,
): SourceChainPopoverModel;

/** Frozen — covers every status, used by tests for exhaustiveness. */
export const ALL_SOURCE_CHAIN_POPOVER_ACTIONS:
  ReadonlyArray<{ status: SourceChainStatus; action: SourceChainPopoverAction | null }>;
```

## Edge cases

The implementer MUST handle each of these. Tests live in §"Test plan".

1. **Empty artifacts (`no_source`).** Chip renders "No source yet". Popover headline + helper come straight from EV-001's `RECEIPT_CHIP_COPY.no_source`. Primary CTA is "Ask for source"; tapping it dispatches `ask_for_source`. Body seed: `ASK_SOURCE_PRESET_BODY`.
2. **Popover on the viewer's own move (`isOwn === true`).** Author cannot "ask themselves for a source." When `model.actor === 'self'` (resolved from `ArgumentBubbleViewModel.actor`), `SourceChainPopover` hides the "ask" CTA entirely. It still renders the headline + helper + artifact summary as a read-only inspection block. This applies in EVERY status — even `broken` — because authors fix their own trail by attaching a new evidence move, not by asking themselves. (`request_deletion` and the existing self-bubble controls remain available via the regular sidecar.)
3. **Observer / read-mode viewer (`isReadModeViewer === true`).** Observers cannot post. The "ask" CTA renders in a disabled state with `accessibilityState={{ disabled: true }}` and a single-line helper "Join a side to ask". Tapping is a no-op. The read-only "Inspect receipt" affordance remains available in every state. This mirrors the existing Stage 6.4 observer-first action-rail contract.
4. **Missing `argumentId` / activeMessageId.** `ArgumentTimelineMap` already guards on `map.activeNode` and `activeViewModel`. The popover model is built only when both are non-null. If `messageId` is somehow empty string, `SourceChainPopover` short-circuits and renders nothing — defensive guard.
5. **Node off-screen.** The popover is anchored inside the existing `TimelineNodePopover`, which already lives in a fixed `popoverDock` region above the scrollable timeline. The new chip + section are children of that dock. No new offscreen logic — if the timeline-node popover is visible, our section is visible.
6. **Multiple artifacts on one move.** EV-001's `summarizeArtifactsForReceiptChip` already collapses the worst status across artifacts. `SourceChainPopover` renders a stacked list of the first 3 artifacts (each: kind label + hostname / source-text prefix / quote excerpt) with a "+N more" counter when `count > 3`. No new artifact list state — order is preserved exactly as EV-001's adapter emitted them.
7. **`broken` status overlap with `source_and_quote` artifact.** Per EV-001 doctrine, an admin can override an otherwise-`source_and_quote` artifact to `broken`. The aggregate worst-status rule means the chip + popover render the `broken` form. The read-only "Inspect receipt" affordance still shows the URL / quote — users can see what the admin marked as weak.
8. **`primary_present` with zero `quote`.** Possible if an admin marks a URL-only artifact as primary. Popover renders read-only with `Inspect receipt` showing the URL but no quote. Accessibility label suffix already handles this via EV-001's `getTimelineEvidenceContract` ("Evidence node, primary source attached.").
9. **`attachedEvidence` JSONB is malformed.** EV-001's `buildEvidenceArtifacts` drops empty / all-null entries; the popover never sees malformed data. If the JSONB key is missing entirely, the artifact map for that message is `[]` → `no_source` path → "Ask for source" CTA. This is the intended fallback (we never crash on missing data).
10. **Reduce-motion (`AccessibilityInfo.isReduceMotionEnabled() === true`).** The popover's inline expand/collapse uses `LayoutAnimation` only when `reduceMotion === false`. When `true`, the section snaps open/closed with no animation. Implementer reads the preference once per mount via `useEffect` + `AccessibilityInfo`.
11. **Keyboard navigation (web).** `Esc` closes the popover (already handled by `TimelineNodePopover.onClose`). When the `SourceChainPopover` section is expanded, `Tab` order is: chip → expand toggle → primary CTA → existing popover action chips. The primary CTA is a regular `Pressable` with `accessibilityRole="button"` so focus order is automatic on web.
12. **Color-blind / desaturation safety.** The dotted-teal ring carries the same information as the chip's text label ("Source trail is weak", "No source yet", etc.). A grayscale render still distinguishes the chip via text + the dotted border style. The popover never relies on color alone — every visual signal has a text equivalent.
13. **Screen-reader contract.** Popover root `accessibilityLabel`: `"Source-chain inspection for message {ordinal}. {headline}. {helper}. {primaryActionLabel-or-Inspect-only}."` Computed by `buildSourceChainPopoverModel`. Tests assert no banned token in the constructed label across all six statuses.
14. **Doctrine: heat / popularity must not influence chip or popover.** Tests assert: (a) `buildSourceChainPopoverModel` ignores any extra fields on the artifact (it consumes only `status` + `count` from the chip contract); (b) the rendered popover has no React state that reads engagement metrics, view counts, follower counts, or any other amplification-flavored signal.
15. **Concurrent edits.** Not applicable — the popover is read-only over already-fetched artifact data. The user's "ask" CTA opens the composer with a seeded body; the existing composer flow handles concurrency exactly as it does for any other reply.
16. **Author posts a reply via the seeded preset.** The seeded body flows through `evaluateArgumentDraft` like any other reply. The "ask" preset is a `clarification_request` with `suggestedTagCodes`; the user can edit the body, change the type, attach evidence, etc. EV-002 doesn't bypass validation.

### `__tests__/sourceChainPopoverModel.test.ts` (~220 lines)

Pure dispatch coverage. No React.

- **Shape / enum coverage**
  - `buildSourceChainPopoverModel` returns a model with all required fields for every `SourceChainStatus`.
  - `ALL_SOURCE_CHAIN_POPOVER_ACTIONS` has exactly six entries (one per status), and the entries match the dispatch table verbatim.
- **Dispatch table per row**
  - `no_source` → headline "No source yet", `primaryAction.bubbleControl === 'ask_for_source'`, `presetKey === 'source'`, `isReadOnly === false`.

  - `source_and_quote` → headline "Source and quote", `primaryAction === null`, `isReadOnly === true`.
  - `broken` → headline "Source trail is weak", `primaryAction.bubbleControl === 'ask_for_source'`, `presetKey === 'weak_source'`, `tone === 'attention'`.
  - `primary_present` → headline "Primary source", `primaryAction === null`, `isReadOnly === true`.

## Risks

1. **`MoveDraftPatch.body` is contract surface.** Adding `body?: string` to `MoveDraftPatch` extends a shared interface that 30+ callers reach into. The field is optional, so existing callers are not affected, but a future refactor (e.g. EV-003 adding `evidenceDebtSummary`) must keep the field optional. The implementer must NOT promote `body` to required.
2. **Composer body overwrite UX.** When a user has a draft in progress and then opens the popover and taps "Ask for source", the seeded body OVERWRITES their in-progress text. This is the existing behavior of every other preset (challenge / quote / etc.) — `handleMovePatch` calls `updateField` which replaces fields wholesale. The design accepts this as consistent with current UX. Implementer adds NO new "merge" logic. If a future card wants a confirm-on-overwrite prompt, that's a separate card.
3. **No `RING.sourceChain` token in VG-001 yet.** Using `borderStyle: 'dotted'` directly in `ReceiptChip.tsx` is a pragmatic shortcut. If a designer later adjusts the source-chain ring (e.g. to dashed at small sizes, or to a different teal hue at high-contrast mode), the change has to touch the chip directly. Mitigation: a `// TODO(VG-001-ext): migrate to RING.sourceChain token if/when introduced` comment in the chip file. Not a blocker for this card; called out in the "Out of scope" section.
4. **`unverified` mapping is opinionated.** The card body left the resolution open. We picked "Ask for source" (not "Ask for quote") with rationale documented above. If the operator wants a different mapping at Review time, the change is a one-line dispatch-table edit. Tests will need to update.
5. **`isOwnMessage` hides "ask" CTAs.** Some users might expect a "Add a source" affordance on their own message. That's a distinct affordance (modify-own-evidence) which belongs in the composer, not the popover. Documented in "Out of scope".
6. **Observer disable contract.** `isReadModeViewer === true` keeps the CTA visible but disabled. An alternative would be to hide it. We keep it visible so observers learn what the action would be once they join a side. This matches the Stage 6.4 observer-first action-rail copy ("Join a side to ask"). If reviewers prefer hide-over-disable, change is also a one-line patch.
7. **Test suite size.** ~50-60 new tests will push the total to ~1855-1865 across ~72-73 suites. Suite count grows by ~3. No timing impact expected.
8. **The chip lives inside the existing `TimelineNodePopover`.** This couples EV-002 to that file. If a future card (e.g. EV-003 debt chips on nodes) wants the chip on the timeline node itself (outside the popover), the chip component is reusable — but `TimelineNodePopover` will need to accept the extra prop. Mitigation: keep `ReceiptChip` and `SourceChainPopover` as separate exports, not nested components.
