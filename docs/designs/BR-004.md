## Goal (one paragraph)

CDiscourse's Timeline Tree currently appends every move "somewhere" — a new participant's contribution and a topic drift land structurally indistinguishable from the 1v1 spine. BR-004 defines an explicit, **teachable direction grammar** so the board reads at a glance: the **mainline** is the horizontal OP ↔ Primary Opponent 1v1 spine and is always the most prominent path; a **public chime-in branch** is a **vertical** offshoot meaning "a new participant's perspective on the same topic"; a **tangent / redirect branch** is a **diagonal** offshoot meaning "a side issue routed off the mainline so it cannot derail it." BR-004 is the **grammar + rendering-contract layer** — it does not detect tangents (BR-003 does) and does not classify topology edges (BR-001 does). It ships a pure-TS `branchGrammarModel.ts` that maps a branch's structural origin/cause to a `BranchDirection`, a collapsed-branch summary builder (count · recency · unresolved · primary-party-engaged), a rendering contract that conveys direction by **position + shape + stroke** (never color alone, per `timeline-grammar`), and a clean selection hand-off to IX-004's existing "You are here" readout. The doctrine anchor is one sentence, inherited verbatim from BR-001: **a branch direction describes the structural position of a move, never its truth, its heat, its popularity, or its author.** Vertical is not "better" than diagonal; diagonal is not "wrong"; the mainline is not "the correct path" — it is the OP↔Primary spine by definition. A tangent branch is "a side issue", never "a bad move" or "a dodge". BR-004 ships **no migration, no Edge Function, no Supabase write, no AI call, no new dependency, no `.env*` change.**

The two new modules are pure TS, unit-testable in isolation. `branchGrammarModel.ts` owns the **direction grammar** (origin/cause → direction) and the **collapsed-summary builder**. `branchGrammarRenderContract.ts` owns the **shape/position/stroke token mapping** — it is split out so the grammar model never imports a render concern and a future card can swap render tokens without touching the grammar logic. The `ArgumentTimelineMap.tsx` change is a **minimal additive wiring step** (§7) — it passes the new view-models to the existing rail and stub; it does not rewrite rendering.

**Key architectural decision — BR-004 owns grammar, not topology and not routing.** BR-001 classifies *topology* (`RailBranchKind`: is this edge `main` / `tangent` / a kink?). BR-003 decides *routing* (`assessTangentRisk`: should this draft go to a side branch?). BR-004 is the *grammar* layer between them and the renderer: it answers "given a branch's structural origin, which of the three teachable **directions** does it draw in?" and "what does the collapsed stub summarize?" BR-004 reads BR-001's `RailBranchKind` and BR-003's `suggestedAction` as inputs — it never re-derives either.

## §2 — Data model

**No new persisted data model. No migration. No new domain type. No new storage. All BR-004 state is derived per-render from data the timeline map already carries.**

### 2.1 `BranchDirection` — the new presentational enum (`branchGrammarModel.ts`)

```ts
/**
 * BR-004 — the teachable structural DIRECTION a branch draws in. Distinct
 * from BR-001's RailBranchKind topology enum. A direction describes the
 * STRUCTURAL POSITION of a branch on the board — never its truth, heat,
 * popularity, or author. Vertical is not "better" than diagonal.
 */
export type BranchDirection =
  | 'mainline'             // horizontal — the OP <-> Primary Opponent 1v1 spine
  | 'chime_in_vertical'    // vertical offshoot — a new participant's perspective
                           //   on the SAME topic
  | 'tangent_diagonal'     // diagonal offshoot — a side issue routed off the
                           //   mainline so it cannot derail it
  | 'evidence_passthrough';// PASS-THROUGH — this branch is already styled by
                           //   BR-001 as an evidence thread (teal track,
                           //   `main` rail style). BR-004 does NOT restyle it.
                           //   §0 D2: there is no separate evidence/source/
                           //   definition branch kind to collide with.

/** Frozen list of every direction. Tests iterate this. */
export const ALL_BRANCH_DIRECTIONS: ReadonlyArray<BranchDirection> =
  Object.freeze([
    'mainline',
    'chime_in_vertical',
    'tangent_diagonal',
    'evidence_passthrough',
  ]);
```

**Why a fourth `evidence_passthrough` value and not "three plus a boolean":** the card explicitly requires BR-004 to **not collide with or restyle** evidence/source/definition branches. Making evidence-thread membership a *value of the same enum* means the rendering contract has a single exhaustive `switch` and the implementer cannot accidentally apply a chime-in or tangent visual to an evidence branch — the type system forbids it. `evidence_passthrough` means "BR-004 yields; render exactly as BR-001/VG-002 already do." It is never the direction of a non-evidence branch.

### 2.2 `BranchOrigin` — the deterministic input to direction derivation

```ts
/**
 * BR-004 — the structural CAUSE a branch exists, fed to deriveBranchDirection.
 * Every field is read off data the timeline map / BR-001 / BR-003 / MCP-010
 * already produced. BR-004 re-derives nothing.
 */
export interface BranchOrigin {
  /** The branch's id (ArgumentTimelineMapNode.branchId). */
  branchId: string;
  /** The branch root message id (ArgumentTimelineMapNode.branchRootMessageId). */
  branchRootMessageId: string;
  /**

   * Primary Opponent both sit on. Determined structurally: the branch
   * whose root has no parent (the opening claim) is the mainline branch.
   * (The OP<->Primary 1v1 spine is, by construction, the chain of `main`
   * RailBranchKind edges descending from the root.) §2.4.
   */
  isMainlineBranch: boolean;
  /**
   * BR-001's topology classification of the edge that ENTERS this branch
   * root (the inbound `RailBranchKind`). `main` => stays on mainline;
   * `kink_start` => a real branch leaves the spine here; `tangent` =>
   * interior of a kinked subtree; `detached` => parent missing.

  inboundBranchKind: RailBranchKind;

   */
  isEvidenceThread: boolean;
  /**
   * Whether the branch root carries an explicit tangent qualifier code
   * (`branch_this_off` / `tangent_or_joke`) in its `droppedTags`. Read
   * off the node — NOT re-derived. (BR-001's `hasTangentLexicalCode`
   * helper is the same read; BR-004 reuses it.)
   */
  hasTangentLexicalCode: boolean;
  /**
   * Optional advisory routing inputs. When present they REINFORCE the
   * topology read; they never override `isEvidenceThread` or the
   * mainline. Both are advisory per their own cards.
   *   - `routedSuggestedAction` — BR-003's `assessTangentRisk` result
   *     for the branch root's draft, if it was carried.
   *   - `userOverrideLane` — MCP-010's recorded user lane choice, if any.

  userOverrideLane?: SemanticOverrideLane | null;

```

### 2.3 `BranchGrammarNode` — the per-branch grammar record (the card's named shape)

The card body names `BranchGrammarNode { branchId, direction, originNodeId, participantCount, lastActivityAt, unresolvedAxisCount, primaryPartyEngaged }`. BR-004 keeps that shape and derives every field from existing timeline-map data (§0 D3).

```ts

 * BR-004 — one record per branch. Pure derivation — no persisted shape.
 * Built by `buildBranchGrammarMap` from the timeline map + BR-001 maps.
 */
export interface BranchGrammarNode {
  /** ArgumentTimelineMapNode.branchId. */
  branchId: string;
  /** The teachable direction this branch draws in. */
  direction: BranchDirection;
  /** The branch root message id (the card's `originNodeId`). */
  originNodeId: string;
  /**
   * Distinct author count across the branch's messages. Derived by
   * counting unique `actorLabel` values among the branch's nodes.
   * (`actorLabel` is the only stable per-author handle on the node.)
   */
  participantCount: number;
  /**
   * ISO timestamp of the most-recent message in the branch — the max
   * `createdAt` across the branch's nodes. §0 D3: BR-004 computes this;

   */
  lastActivityAt: string | null;
  /**
   * Count of branch messages that carry an UNRESOLVED-challenge qualifier
   * code in `droppedTags` (e.g. an open `evidence_challenge` / a
   * `source_request` with no answering evidence in the branch). A
   * deterministic count over existing fields — NOT a LIFE-001 re-derive,
   * NOT an axis classifier. §0 D3 / §2.5.
   */
  unresolvedAxisCount: number;
  /**
   * True when the OP OR the Primary Opponent has posted at least one
   * message inside this branch. For the mainline this is trivially true.
   * For a chime-in / tangent it answers "did a principal engage here?"
   * Derived by checking branch node `actorLabel`s against the room's
   * OP + Primary Opponent labels (passed in — §3.1).
   */
  primaryPartyEngaged: boolean;
  /**
   * BR-004 — true when this branch is a tangent whose offshoot depth has
   * reached `TANGENT_DEPTH_CAP`. Drives the "summarize this offshoot"
   * advisory prompt. Deterministic; NO AI. §0 D7 / §8.
   */
  offshootDepthCapReached: boolean;

```

### 2.4 How "the mainline" is determined (deterministic, structural)

The card defines the mainline as the **OP ↔ Primary Opponent 1v1 spine** and is emphatic that a chime-in **never auto-promotes to the mainline even if it becomes the most active path** — "the mainline is the OP↔Primary spine by definition."

BR-004 honors this with a purely structural rule, **never an activity / heat / popularity read**:

- The **mainline branch** is the branch whose root is the conversation's opening claim — `ArgumentTimelineMapNode.isRoot === true` (already a field on the node). Every node sharing that node's `branchId` is on the mainline.
- The spine of `main`-`RailBranchKind` edges descending from the root *is* the OP↔Primary 1v1 thread by construction (BR-001 keeps the first child on the parent lane; additional siblings kink off).
- A `chime_in_vertical` or `tangent_diagonal` branch can have any participant count, any recency, any unresolved count — `deriveBranchDirection` **never reads those fields**. A test feeds a chime-in branch with 10× the mainline's message count and asserts its direction stays `chime_in_vertical`.

This is the doctrine guard the card's first edge case demands: *"A chime-in branch that later becomes the most active path — it still does not auto-promote to the mainline."* The grammar is structural; activity cannot move it.

### 2.5 `unresolvedAxisCount` — the deterministic derivation

`unresolvedAxisCount` is **not** a LIFE-001 axis classifier and **not** an AI read. It is a count of branch nodes carrying an open-challenge qualifier code that has no answering node in the same branch:

- A node "raises an unresolved item" when its `droppedTags` includes one of the open-challenge codes (`evidence_challenge`, `source_request`, `quote_request`, or an open `challenge`/`disagree` qualifier — the exact frozen list lives in the model and is reviewed against `messageQualifiers.ts`).
- A raised item is "resolved" when a later node in the same branch carries an answering qualifier (`evidence`, `concede`, `synthesize`, `confirm`). The count is `raised − resolved`, clamped at `0`.
- This is the same family of qualifier codes BR-001's `EVIDENCE_QUALIFIER_CODES` already reads off `droppedTags` — BR-004 reuses the existing surface, never re-derives `MessageCategory`.

This keeps `unresolvedAxisCount` deterministic, doctrine-clean (it counts *structural* open items, never "who is losing"), and unit-testable.

  // Depth-cap advisory — describes the offshoot, never the person.
  depth_cap_prompt: 'This side thread has gone a few layers deep. A short summary keeps it readable.',

## §3 — API / interface contracts (`branchGrammarModel.ts`)

### 3.1 `deriveBranchDirection(origin) → BranchDirection`

```ts
/**
 * Map a branch's structural origin/cause to its teachable direction.
 * Pure. Deterministic. Idempotent. Same input -> identical output.
 *
 * Reads ONLY structural fields (topology classification, evidence-thread
 * boolean, explicit qualifier codes, advisory routing inputs). NEVER
 * reads heat, popularity, reply count, participant count, recency, or
 * any strength / standing band.
 */
export function deriveBranchDirection(origin: BranchOrigin): BranchDirection;
```

**Derivation order** (first match wins; fully unit-testable rule list):

1. **`evidence_passthrough`** — `origin.isEvidenceThread === true`. The card forbids restyling evidence/source/definition branches; the evidence-thread flag wins over everything else, including an explicit tangent tag. (Doctrine note: an evidence conversation that *also* drifts is still rendered as an evidence thread — BR-004 yields; BR-001/VG-002 own the visual.)
2. **`mainline`** — `origin.isMainlineBranch === true`. The OP↔Primary spine. This branch is the mainline by construction (§2.4); activity never changes it.
3. **`tangent_diagonal`** — any of:
   - `origin.inboundBranchKind` is `tangent` or `kink_end` (BR-001 already classified this edge as interior/leaf of a kinked subtree), OR
   - `origin.hasTangentLexicalCode === true` (an explicit `branch_this_off` / `tangent_or_joke` tag), OR
   - `origin.routedSuggestedAction` is `send_to_side_branch` or `branch_this` (BR-003 routed it to a side branch), OR
   - `origin.userOverrideLane === 'tangent'` (MCP-010: the user explicitly chose the tangent lane).
4. **`chime_in_vertical`** — `origin.inboundBranchKind === 'kink_start'` AND none of the tangent signals in rule 3 fired. A real branch leaves the spine here, it is not a tagged/routed tangent, and it is not the mainline — so it is a new participant's perspective on the same topic.
5. **Default → `chime_in_vertical`** — a non-mainline, non-evidence, non-tangent branch with an unclassified inbound kind (defensive; e.g. `inboundBranchKind === 'detached'` on a non-root branch). A detached branch with a missing origin attaches to the nearest ancestor (§8 edge case) and renders as a chime-in rather than guessing a tangent — the conservative, non-accusatory default.

The order encodes priority: **evidence pass-through is sacred (rule 1)**; the mainline is structural (rule 2); an explicit tangent signal — whether from BR-001 topology, a user tag, a BR-003 route, or an MCP-010 override — produces a diagonal (rule 3); everything else that genuinely branches is a vertical chime-in (rules 4/5).

`deriveBranchDirection` returns exactly one `BranchDirection`. It is the only place the mapping lives — the rendering contract and the summary builder both read its output, never re-derive it.

### 3.2 `buildBranchGrammarMap(args) → ReadonlyMap<branchId, BranchGrammarNode>`

```ts
/**
 * Build one BranchGrammarNode per branch in the timeline map. Pure. O(n)
 * over node count. Frozen output.
 */
export function buildBranchGrammarMap(args: {
  /** The built timeline map — source of nodes, branch ids, createdAt. */
  timelineMap: ArgumentTimelineMapModel;
  /** BR-001's per-edge topology map (buildBranchKindMap output). */
  branchKindByEdgeId: ReadonlyMap<string, RailBranchKind>;
  /** BR-001's evidence-thread map (buildEvidenceThreadMap output). */
  evidenceThreadByBranchRoot: ReadonlyMap<string, boolean>;
  /** The room's principal actor labels — OP + Primary Opponent. Used to
   *  derive `primaryPartyEngaged`. The room shell already knows these. */
  principalActorLabels: ReadonlyArray<string>;
  /** Optional advisory inputs keyed by branchRootMessageId. */
  routedSuggestedActionByBranchRoot?: ReadonlyMap<string, RedirectSuggestedAction>;
  userOverrideLaneByBranchRoot?: ReadonlyMap<string, SemanticOverrideLane>;
}): ReadonlyMap<string, BranchGrammarNode>;
```

Algorithm (one pass to group nodes by `branchId`, one pass per branch to aggregate):

1. Group `timelineMap.nodes` by `branchId`.
2. For each branch: find the branch root (`branchRootMessageId`), build a `BranchOrigin` (the inbound `RailBranchKind` comes from `branchKindByEdgeId` for the edge ending at the branch root; `isMainlineBranch` from the root node's `isRoot`; `isEvidenceThread` from the evidence map; `hasTangentLexicalCode` from the root node's `droppedTags`), call `deriveBranchDirection`.
3. Aggregate `participantCount` (distinct `actorLabel`), `lastActivityAt` (max `createdAt`), `unresolvedAxisCount` (§2.5), `primaryPartyEngaged` (any branch `actorLabel` ∈ `principalActorLabels`), `offshootDepthCapReached` (§8).
4. Freeze and return the map.

## §4 — Rendering contract (`branchGrammarRenderContract.ts`)

The card requires direction to be conveyed by **position + shape + stroke, never color alone** (`timeline-grammar` doctrine + `accessibility-targets`). This module is the pure-TS token layer.

```ts
/**
 * BR-004 — the render tokens for one BranchDirection. Consumed by the
 * rail / stub components. Color is SUPPLEMENTARY — shape + position +
 * stroke carry the meaning (grayscale-snapshot test enforces).
 */
export interface BranchDirectionVisual {
  direction: BranchDirection;
  /**
   * Geometry token — how the branch leaves the mainline.
   *   'horizontal'      — mainline: continues straight along the spine
   *   'vertical_offset' — chime-in: leaves at a near-vertical angle
   *   'diagonal_kink'   — tangent: leaves at a visible diagonal kink
   *   'inherit'         — evidence_passthrough: BR-004 yields, no override
   */
  positionToken: 'horizontal' | 'vertical_offset' | 'diagonal_kink' | 'inherit';
  /**
   * Shape token for the branch's connector node — reuses VG-001's
   * existing shape vocabulary. NO new shape primitive is invented.
   *   mainline   -> 'spine'           (the rail's straight strip)
   *   chime_in   -> 'bracket'         (a square-shouldered vertical join)
   *   tangent    -> 'kink'            (VG-001's existing bent-connector)
   *   evidence   -> 'inherit'
   */
  shapeToken: 'spine' | 'bracket' | 'kink' | 'inherit';
  /**
   * Stroke token — reuses VG-001's existing stroke weights.
   *   mainline   -> 'solid_prominent' (thickest — always most prominent)
   *   chime_in   -> 'solid_subordinate' (thinner; visually subordinate
   *                  to the mainline until clicked into — card requirement)
   *   tangent    -> 'dashed'          (VG-001's existing dashed branch edge)
   *   evidence   -> 'inherit'
   */
  strokeToken: 'solid_prominent' | 'solid_subordinate' | 'dashed' | 'inherit';
  /**
   * Whether this direction's branch renders visually subordinate until
   * the user clicks into it. True only for `chime_in_vertical` — the card
   * requires chime-ins to be "visually subordinate to the mainline until
   * clicked" and to require "intentional click-in".
   */
  subordinateUntilSelected: boolean;
  /** Plain-language a11y fragment, e.g. "chime-in branch, vertical". */
  accessibilityFragment: string;
}

/** Pure. Total over BranchDirection. The single source of the token map. */
export function buildBranchDirectionVisual(
  direction: BranchDirection,
): BranchDirectionVisual;
```

**Coordination with `timeline-grammar` / VG-001 / VG-002 (load-bearing):**

- BR-004 introduces **no new color token, no new shape primitive, no new stroke weight.** Every token value above maps onto VG-001's existing vocabulary and VG-002's existing rail `RailSegmentStyleWrapper` fields (`transformRotateZDeg`, `showKinkStartStub`, `showKinkEndStub`, `opacity`). The `diagonal_kink` position reuses VG-002's existing `TANGENT_KINK_DEG` rotation and `kink_start`/`kink_end` stub flags. The `vertical_offset` for a chime-in reuses the lane geometry `computeLane` already produces — a chime-in lands on a non-zero lane (above/below the spine), which `argumentGameSurfaceModel` already turns into a vertical pixel offset; BR-004's contribution is the *bracket shape token* + the *subordinate stroke*, not new geometry.
- The three directions are distinguishable **without color**: `horizontal` straight + thick (mainline), `vertical_offset` bracket + thin (chime-in), `diagonal_kink` + dashed (tangent). A grayscale snapshot test asserts all three remain distinct.
- `evidence_passthrough` returns `'inherit'` for every token — the renderer is contractually told "do not apply a BR-004 visual; render exactly as BR-001/VG-002 already do." This is the explicit no-collision guarantee the card demands.

**This module is presentation tokens only.** It does not render `<View>`s. The actual `react-native` rendering stays in `GradientWaveRail.tsx` / `ArgumentTimelineMap.tsx` — BR-004's wiring step (§7) hands these token values to the existing rail; the rail's existing dispatch already understands kink rotation and stub flags. A `timeline-grammar` review of the final token-to-pixel mapping is **OD-2**.

## §5 — Consuming BR-003's `suggestedAction` and MCP-010's lane choice

The coordinator brief is explicit: BR-004 **consumes** BR-003's tangent-routing output and MCP-010's lane choice — it does **not** redesign either.

- **BR-003** (`tangentRoutingModel.ts`): `assessTangentRisk` returns a `RedirectSuggestedAction`. When the room carries a routed move (a draft posted via the `branch_tangent` channel after the user accepted BR-003's advisory), the resulting branch root's `routedSuggestedAction` is threaded into `BranchOrigin`. `suggestedActionToBranchDirection` (§3.5) maps it: `send_to_side_branch` / `branch_this` → `tangent_diagonal`; `continue` / `ask_clarifying_question` → `mainline`. `deriveBranchDirection` rule 3 then produces the diagonal. **BR-004 never calls `assessTangentRisk` and never makes a routing decision** — it reads the already-made decision and draws it.
- **MCP-010** (semantic override): MCP-010 records a user `SemanticOverrideLane` (`mainline | branch | tangent`) as an advisory routing input "for BR-003 / BR-004 to consume" (MCP-010 design §4.2 / §6 verbatim). When a branch root has a recorded override, its `userOverrideLane` is threaded into `BranchOrigin`. `overrideLaneToBranchDirection` (§3.5) maps `branch` → `chime_in_vertical` and `tangent` → `tangent_diagonal`. `deriveBranchDirection` rule 3 honors a `tangent` override; rule 4 honors a `branch` override. **BR-004 never modifies the MCP-010 record and never re-runs the semantic referee.**
- **RULE-005**'s `branch_tangent` `MoveChannel` is the *channel* a routed move is posted on; BR-001's `hasTangentLexicalCode` already reads the resulting `branch_this_off` qualifier off `droppedTags`. BR-004 reads BR-001's classification — it does not re-read the channel directly. RULE-005 is an upstream input via BR-001, not a BR-004 dependency.

When **both** a BR-003 route and an MCP-010 override are present and disagree, the **MCP-010 user override wins** — the user's explicit lane choice is the authoritative advisory input (consistent with MCP-010's design: the latest override record wins for what BR-004 consumes). `deriveBranchDirection` checks `userOverrideLane` before `routedSuggestedAction` inside rule 3. A test pins this precedence.

## §8 — Edge cases

The implementer must handle each. Tests are listed in §10.

1. **Empty room (`timelineMap.nodes.length === 0`).** `buildBranchGrammarMap` returns an empty map. `buildCollapsedBranchSummary` is never called. No stub, no crash.
2. **Root-only tree (one node, no branches).** The single branch is the mainline (`isRoot === true`). `deriveBranchDirection` → `mainline`. No chime-in, no tangent.
3. **A chime-in branch that becomes the most active path.** `deriveBranchDirection` never reads `participantCount` / `lastActivityAt` / message count — its direction stays `chime_in_vertical` regardless of activity. The mainline is structural (§2.4). A test feeds a chime-in with 10× the mainline's messages and asserts the direction is unchanged. **This is the card's headline edge case.**
4. **Tangent off a tangent (diagonal off diagonal).** BR-001 already classifies each interior edge of a kinked subtree as `tangent`; `deriveBranchDirection` rule 3 produces `tangent_diagonal` at every depth. When the offshoot depth (the count of nested tangent branches from the mainline) reaches `TANGENT_DEPTH_CAP`, `BranchGrammarNode.offshootDepthCapReached` is `true` and the room shows the `depth_cap_prompt` plain-language advisory. The grammar **does not auto-collapse, does not block, does not summarize** — the prompt is advisory only (doctrine §1 — nothing blocks).
5. **A branch with a missing / deleted origin node.** Per the card: graceful fallback — attach to the nearest ancestor. `argumentGameSurfaceModel` already marks such an edge `isDetached` and BR-001 classifies the inbound kind `detached`. `deriveBranchDirection` rule 5 default → `chime_in_vertical` (the conservative, non-accusatory default — a detached branch is treated as a chime-in, never guessed as a tangent). The `BranchGrammarNode` still builds; `lastActivityAt` / counts come from whatever nodes remain.
6. **250+ node stress.** `buildBranchGrammarMap` is O(n) (two passes). The grammar is color-independent and shape-driven, so it remains legible. A 250-node fixture asserts the map builds in well under a frame budget and every branch resolves to exactly one direction. Density-mode rendering tuning coordinates with IX-001 but is not BR-004's component.
7. **Evidence branch that also drifts.** `isEvidenceThread === true` → `deriveBranchDirection` rule 1 → `evidence_passthrough`, even if the branch also carries a tangent tag. BR-004 yields; BR-001/VG-002 own the evidence visual. The card forbids restyling evidence branches — rule-1 priority enforces it. A test pins this.
8. **A branch with a BR-003 route AND an MCP-010 override that disagree.** The MCP-010 user override wins (§5). A test feeds `routedSuggestedAction: 'send_to_side_branch'` + `userOverrideLane: 'mainline'` and asserts the direction is `mainline`.
9. **A branch root whose inbound edge has no entry in `branchKindByEdgeId`.** Defensive: treat the inbound kind as `detached`; rule 5 default → `chime_in_vertical`. Never throws.
10. **`primaryPartyEngaged` when `principalActorLabels` is empty** (room shell did not supply principals). `primaryPartyEngaged` is `false` for every non-mainline branch and `true` for the mainline (by construction). Never throws.
11. **Concurrent edits / a new message arrives in a branch.** The room shell re-renders; `buildBranchGrammarMap` re-runs (memo invalidates on the timeline-map hash change). `lastActivityAt` and counts refresh. Direction is recomputed but, being structural, only changes if the branch's *topology* changed — not its activity.
12. **Offline / network failure.** BR-004 is pure UI over already-loaded data — no fetch, no write. Offline simply means no new messages; the grammar map keeps describing the last-known tree. No special handling.
13. **Permission-denied / observer.** Branch grammar is presentational/structural — actor rules are unchanged. An observer sees the same direction grammar as a participant; BR-004 introduces no action affordance, so there is no permission gate on the grammar. The card states this explicitly under "Actor rules".
14. **Doctrine edge — does heat / popularity move a branch direction?** No. `deriveBranchDirection` reads only topology classification, the evidence-thread boolean, explicit qualifier codes, and advisory routing inputs. It never reads a heat band, a strength band, a reply count, or a participant count. A test feeds two structurally-identical branches with different (mock) heat and asserts identical directions.
15. **Doctrine edge — does a tangent direction imply the move is wrong?** No. `tangent_diagonal` is a *structural position*. The copy says "Side issue", never "wrong" / "off-topic" / "dodge". The ban-list test enforces it.
16. **Doctrine edge — can a branch direction block a post?** No. BR-004 is a rendering/grammar layer with no post path, no validation, no `PreSendStructuralBlock`. The depth-cap prompt is advisory copy only.

## §9 — `TANGENT_DEPTH_CAP` constant

```ts
/** BR-004 — the deterministic offshoot depth at which a tangent branch
 *  surfaces the "summarize this offshoot" advisory. Frozen constant; NO
 *  AI involved in detecting it. OD-1 confirms the value. Proposed: 3. */
export const TANGENT_DEPTH_CAP = 3;
```

"Offshoot depth" is the count of nested tangent branches between the mainline and the branch (mainline = 0; first tangent off the mainline = 1; tangent off that = 2; …). It is derived deterministically by walking the `branchRootMessageId` → parent chain and counting `tangent_diagonal` branches. When depth `>= TANGENT_DEPTH_CAP`, `BranchGrammarNode.offshootDepthCapReached` is `true`. ## §12 — Risks

1. **Grammar collision with the existing evidence/source/branch visuals.** Mitigation: the `evidence_passthrough` enum value + rule-1 priority in `deriveBranchDirection` make "do not restyle evidence branches" a *type-level* guarantee; the render contract returns `'inherit'` for every token. A test pins it. A `timeline-grammar` review of the final token mapping is **OD-2**.
2. **`ArgumentTimelineMap.tsx` is a hot file.** The wiring step must stay minimal — supply token *values* to the rail the rail already understands; do not rewrite the rail's dispatch. If the implementer finds the rail cannot consume the tokens without a structural rewrite, **stop and surface it** — that is a VG-002 refactor card, not BR-004.
3. **Map illegibility at 250+ nodes.** Mitigation: `buildBranchGrammarMap` is O(n); the grammar is shape/position-driven (color-independent). The stress fixture (§10) proves legibility. Density-mode tuning coordinates with IX-001.
4. **`BranchDirection` vs `RailBranchKind` confusion.** Two enums describing branches in the same folder invites a merge attempt. Mitigation: the file header comment explicitly states `RailBranchKind` = BR-001 *topology* (`main`/`tangent`/kink), `BranchDirection` = BR-004 *grammar* (`mainline`/`chime_in_vertical`/`tangent_diagonal`/`evidence_passthrough`). Do not merge.
5. **`primaryPartyEngaged` depends on the room shell supplying `principalActorLabels`.** If the shell does not yet know the OP + Primary Opponent (a 1v1 room model concern), the field degrades to `false` for non-mainline branches (§8 #10) — it never throws. If a future card needs a harder principal identity, that is a GAME-004 (1v1 PvP room contract) follow-up.
6. **MCP-010 `branch` → `chime_in_vertical` mapping is a judgment call.** MCP-010's `branch` lane is "a deliberate branch that is not a tangent" — which is exactly a chime-in. If a later card distinguishes "branch" from "chime-in" semantically, `overrideLaneToBranchDirection` is a one-line change. Documented; not a blocker.
7. **Depth-cap value.** `TANGENT_DEPTH_CAP = 3` is a proposal. Too low → noisy prompt; too high → unreadable offshoots. It is a frozen constant tunable without an API change — **OD-1**.
8. **The wiring step touches the collapse stub.** BR-001's `BranchCollapseStub` currently shows `+N`. BR-004 enriches it to the four-field summary.
