## §1 — Goal & scope

Add a single optional field — `entryHintForArgumentId?: string` — to the
`GalleryEntryHint` interface. Wire `App.tsx`'s
`handleOpenNotificationDeepLink` to set the entry hint with that field
populated from the notification's `link.activeArgumentId`. Extend
`ArgumentGameSurface.tsx`'s `initialActiveId` deriver to honour the new
field on mount when present and accessible, and fall back to the
existing "select latest move" path when the field is absent, references
an inaccessible argument, or references an argument that lives in a
different debate. No schema change, no Edge Function change, no
migration, no operator step.

### §1.1 — Scope

In scope:
- Additive optional field on `GalleryEntryHint`.
- `App.tsx`'s `handleOpenNotificationDeepLink` populates the hint and
  calls `setEntryHint(hint) → selectDebate(target, side)`.
- `ArgumentGameSurface.tsx`'s `initialActiveId` deriver honours
  `entryHintForArgumentId` when present and the id exists in the loaded
  `sorted` slice.
- Existing fallback (`latestId` selection) carries every other case.
- Selection status remains the existing `'entry_hint' | 'default_latest'
  | 'explicit' | 'stale_fallback'` union (no new status needed; the
  fallback maps to `'entry_hint'` because the hint *was* honoured — the
  message just isn't visible to this caller).
- Tests under `__tests__/`.

Out of scope (see §12 for the full list):
- Realtime delivery, notification preferences, native deep-link path.
- Any change to existing `GalleryEntryHint` callers (the new field is
  optional; all callers pass `undefined` until they need it).

## §3 — Problem statement

Today, opening a `room_notifications` row via the notification list
(QOL-040) routes correctly to the room (`debateId`) but the room shell
lands on the *latest move* regardless of which move the notification
referenced. For most triggers this is harmless — `new_response` /
`chime_in_posted` / `argument_settled` all naturally interest the
recipient in the most recent move. But three triggers point at a
specific, older move that the recipient should see *first* on entry:

| Trigger | Why landing on the specific move matters |
|---|---|
| `concession_challenged` | The recipient's earlier concession is the move that was challenged. Landing on the latest move (the challenger's reply) hides the concession context. |
| `source_requested` | A primary was asked to source a specific earlier move. Landing on the latest move (the requester's question) is fine *unless* a third party has since posted on top, in which case the source request becomes invisible. |
| `evidence_supplied` | A specific older "needs evidence" move was just closed by an attached document. Landing on the latest move (the evidence-supplying move) is correct only for the immediate sequence; if any move has landed on top, the resolution context disappears. |

The data is already present: `room_notifications.argument_id` is
populated by `submit-argument` (QOL-040 design §5.1.3) and the deep-link
resolver (`notificationModel.resolveDeepLink`) already returns it as
`NotificationDeepLink.activeArgumentId`. The hand-off layer between
`App.tsx` and `ArgumentGameSurface.tsx` drops it.

This card closes that drop. No new data is needed; one field threads
through three call sites.

## §5 — Hint extension shape

### §5.1 — TypeScript signature

```ts
// src/features/debates/conversationGalleryModel.ts
export interface GalleryEntryHint {
  activate: 'root' | 'latest' | 'first_open_challenge';
  code: GalleryEntryHintCode;
  verbPhrase: string;
  helperLine: string;
  presetKey: QuickActionLabel | null;
  dockAction: TimelineNodeActionDockActionCode | null;
  /**
   * QOL-040.3 — Optional override. When set, the room shell uses this
   * id as the initial active message instead of the deriver's
   * `activate` policy, IF the id is present in the loaded message
   * slice. Falls through to the `activate` policy when absent or
   * inaccessible. Set only by notification deep-link hand-offs; the
   * gallery deriver never sets this field.
   */
  entryHintForArgumentId?: string;
}
```

The field is **purely additive** (answer to Q1). The existing four
`activate` / `code` / `verbPhrase` / `helperLine` consumers (the room
shell, the gallery card renderer, the micro-moment banner) require no
change; their access of these fields is independent.
`deriveGalleryEntryHint` never sets `entryHintForArgumentId`. All
existing callers continue to work unchanged when the new field is
absent (`undefined`).

### §5.2 — App.tsx hand-off

```ts
// App.tsx (replacing the lines 368–384 region)
const handleOpenNotificationDeepLink = React.useCallback(
  (link: NotificationDeepLink, _n: RoomNotification): void => {
    // Close the list, then select the debate.
    // QOL-040.3 — Forward the notification's argumentId as a hint so
    // the room shell can pre-activate the specific move. The room
    // consumer (ArgumentGameSurface) falls back to the latest move if
    // the id is no longer in the loaded slice (soft-deleted, wrong
    // room, or RLS-hidden), so this is always safe.
    setNotificationsOpen(false);
    const target = debates.find((d) => d.id === link.debateId);
    if (!target) return;
    const side = target.myParticipantSide ?? 'observer';
    if (link.activeArgumentId) {
      // Build a minimal hint object. The room consumer reads only
      // `activate` and `entryHintForArgumentId` (the other fields are
      // for the gallery's micro-moment banner; not needed for the
      // notification path).
      const hint: GalleryEntryHint = {
        activate: 'latest',  // safe fallback; only used if the id is
                             // not in the loaded slice
        code: 'watch_first', // neutral; the banner is suppressed below
        verbPhrase: '',      // empty → no micro-moment banner renders
        helperLine: '',
        presetKey: null,
        dockAction: null,
        entryHintForArgumentId: link.activeArgumentId,
      };
      setEntryHint(hint);
    } else {
      setEntryHint(null);  // notification has no specific move; let
                           // the room shell pick latest as today
    }
    selectDebate(target, side);
  },
  [debates, selectDebate],
);
```

**Why empty `verbPhrase` is safe:** `ArgumentGameSurface.tsx:716` reads
`entryHint?.verbPhrase` and only renders the micro-moment banner when
the value is truthy. Empty string is falsy → no banner. The gallery
path remains the only consumer that surfaces the banner; the
notification path is silent (we don't want a "Be the first rebuttal"
banner appearing after a `concession_challenged` deep-link).

**Why `activate: 'latest'` is safe:** the new `entryHintForArgumentId`
branch is checked FIRST in `initialActiveId`. `activate: 'latest'` is
only consulted when the new field's id is absent from `sorted`, at
which point "show the latest move" is exactly the requested fallback.

### §5.3 — Frozen behavior summary

| Field set? | Argument in `sorted`? | `initialActiveId` resolves to |
|---|---|---|
| Yes | Yes | `entryHintForArgumentId` |
| Yes | No | Existing fallback (`latestId` per `activate: 'latest'`) |
| No | n/a | Existing fallback per `activate` (`root` / `first_open_challenge` / `latest`) |

The "field absent" path is byte-identical to today's behavior — every
existing test continues to pass.

## §6 — UI behavior

### §6.1 — Mount-time activation

The new field is honoured **on mount only**. Once the user explicitly
selects a different message (existing `setActiveMessageId` path), the
hint no longer influences selection — exactly as Stage 6.4's `activate`
field works today. No selection-restoration on re-render.

### §6.2 — Fallback to latest

When the hint id is absent from the loaded slice, the deriver returns
`latestId` (the `else` branch in the `initialActiveId` useMemo). The
`selectionStatus` state initialiser at line 247–249
(`entryHint ? 'entry_hint' : 'default_latest'`) keeps the
`'entry_hint'` value — because the hint *was* present, just unmet —
which means the IX-004 readout treats this as a hint-driven selection
rather than a stale fallback. This is correct: the user came here from
a notification, so the readout should reflect that.

A *separate* path exists at lines 265–272 that detects an active
message disappearing from the loaded slice mid-session
(admin-removed); that path sets `selectionStatus` to `'stale_fallback'`.
The deep-link mount-time path does NOT route through this stale-fallback
banner because the message was never set as active in the first place
— the deriver just returned `latestId` directly. Implementer should
ensure no spurious `'stale_fallback'` event fires on mount when the
deep-link id is absent.

### §6.3 — Neutral logging

Per acceptance criteria: "hint references soft-deleted argument →
fallback to latest-move with neutral log line." Add a single
`console.warn` (the codebase convention for noticeable-but-non-blocking
events) when the implementer detects the fallback path:

```ts
// Inside the new initialActiveId branch, when the hint id is absent:
if (entryHint?.entryHintForArgumentId &&
    !sorted.find(m => m.id === entryHint.entryHintForArgumentId)) {
  // Neutral diagnostic. No PII, no doctrine-loaded language.
  // The message body is never logged.
  if (process.env.NODE_ENV !== 'production') {
    console.warn(
      '[QOL-040.3] Notification hint argument not in loaded slice; ' +
      'falling back to latest move.',
      { debateId: messages?.[0]?.debateId, hintId: entryHint.entryHintForArgumentId },
    );
  }
}
```

The log is dev-only (`NODE_ENV !== 'production'`) to honour CLAUDE.md
*"No `console.log` in committed code — use a structured logger utility
or remove before commit"* in production builds. The log carries no PII
beyond two opaque ids (debate, argument) — both already in URL fragments
elsewhere.

### §6.4 — No copy change

Per the issue's "Plain language: no user-facing copy changes — pure
routing improvement." The notification list, the room banner, the
selection-status readout, the rail — every user-facing surface remains
byte-identical. The only visible difference is *which message is active
when the room mounts*.

## §7 — Permission / rules implications (Q3)

**Q3 verdict: confirmed — the hint extension does not introduce a
visibility side-channel.**

Three independent layers enforce that a notification deep-link cannot
pre-fetch an argument from a room the caller cannot read:

1. **Notification list RLS** (QOL-040 design §6.1). The
   `room_notifications` table's SELECT policy restricts to
   `recipient_id = auth.uid()` — the user only sees notifications
   addressed to them. If the room was made private after the
   notification was sent, the row remains visible (the recipient owns
   it), but `resolveDeepLink` returns `null` for
   `room_made_private` and for revoked `chime_in_rejected` — these
   never reach `handleOpenNotificationDeepLink` as a navigable target.

2. **Debate-list RLS** (QOL-039 visibility rules). `App.tsx`'s
   `useDebates()` hook loads only debates RLS-visible to the caller.
   `handleOpenNotificationDeepLink`'s `debates.find((d) => d.id ===
   link.debateId)` returns `undefined` for any debate the user cannot
   see (private room without participation, archived rooms RLS-hidden,
   etc.). The `if (!target) return;` early-return then no-ops the
   entire callback — no `setEntryHint`, no `selectDebate`. The hint
   never gets to the room shell because the room never gets selected.

3. **Argument-row RLS + soft-delete filter** (CLAUDE.md doctrine §8 +
   `useArgumentRoomMessages.ts` lines 9–12). Even after a room IS
   selected, the argument loader pulls only `status === 'posted'` rows
   for the debate. Soft-deleted rows (`is_deleted = true` → status
   transitions) never reach `sorted`. An argument that lives in a
   *different* debate also never reaches `sorted` (the loader filters
   on `debate_id`). When the new branch in `initialActiveId` does
   `sorted.find(m => m.id === entryHintForArgumentId)`, the lookup
   fails naturally for any inaccessible argument, and the fallback
   fires.

The result: **the hint extension is applied AFTER room-level access is
granted**, exactly as the prompt required. No new check is needed —
the natural composition of the three existing layers is sufficient.
Implementer assertion: do NOT add a separate RLS check or a separate
visibility re-derive in the room shell; the layers above are
load-bearing and a duplicate check would be both redundant and a
maintenance liability.

## §8 — Edge cases (Q2)

Q2 verdict: soft-deleted detection is **natural and layered**, not a
new mechanism. Eight edge cases, with disposition for each:

| # | Case | Detection mechanism | Behavior |
|---|---|---|---|
| 1 | Argument was soft-deleted (`is_deleted=true`) | `useArgumentRoomMessages` excludes (status !== 'posted'). Loader filters at the SQL boundary. | Hint id absent from `sorted` → fallback to `latestId`. Dev-only `console.warn`. |
| 2 | Argument was hard-deleted (NOT permitted per CLAUDE.md §8 but defensive) | Same as #1 — id absent from `sorted`. | Same fallback. |
| 3 | Argument exists but is in a *different* debate (data corruption or stale notification across re-imported rooms) | Loader filters on `debate_id`. Wrong-room arguments never appear in `sorted`. | Fallback to `latestId`. |
| 4 | Argument exists in this debate but RLS hides it from this user (e.g., admin-only flag, soft-archive of one branch) | RLS at the row level; the loader receives only visible rows. | Fallback to `latestId`. |
| 5 | The room itself is private and the user lost access between notification creation and deep-link click | `debates.find((d) => d.id === link.debateId)` returns `undefined`. Callback early-returns. | No-op. Notification list will dim the row on next refresh (existing QOL-040 behavior). |
| 6 | The room exists and the argument is loaded, but the user's `myParticipantSide` is `null` (they were never a participant) | `target.myParticipantSide ?? 'observer'` defaults to `'observer'`. Stage 6.4 observer-first entry path applies. | Argument is activated; the user sees the move; the rail is collapsed (observer-default). |
| 7 | `link.activeArgumentId` is `null` (root-level notifications: `invite`, `chime_in_rejected` without specific move) | `if (link.activeArgumentId)` branch is skipped → `setEntryHint(null)`. | Existing behavior: room mounts with latest move active. Identical to today. |
| 8 | The notification fires while the user is already in the *same* room | `selectDebate` dispatches `DEBATE_SELECTED` to the session reducer — idempotent on the same id. `entryHint` state is reset via `setEntryHint`. The mount-time `initialActiveId` only runs on initial mount; if the room is already mounted, no re-activation happens. | Documented gap: the room does not jump to the hinted argument when the user is already inside it. This is **acceptable for v1** (the deep-link is a "take me there" gesture, not a "re-focus" gesture). A follow-up could add an effect that watches `entryHint` and explicitly `setActiveMessageId` — but that introduces a "the screen moved under me" race that has its own UX cost. Defer with a comment. |

Implementer note: cases 1–4 share the same code path (id absent from
`sorted` → fallback). Cases 5 and 7 are upstream of the hint branch
entirely. Case 6 is the QOL-039-compatible happy path. Case 8 is
explicitly out-of-scope for this card.
