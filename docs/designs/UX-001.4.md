## Goal

UX-001.4 consolidates the three in-room interaction menus — Act, Inspect, Go —
into one coherent three-menu model. Each menu answers exactly one user question
("what can I do here?" / "what am I looking at?" / "where do I want to be?"),
all three share chrome and dismiss behavior, and the keyboard shortcut
allocation (`A` / `I` / `G` / `Esc`) is uniform across the three.

The card removes the duplicate action surfaces that have accumulated alongside
`ActPopout`: `ArgumentSideActionRail` and `TimelineNodeActionDock` overlap with
Act's entries today; UX-001.4 either migrates a rail/dock entry into Act,
preserves it as a single-tap shortcut that opens Act with a preset, or retains
it as a high-frequency inline affordance with explicit rationale.

UX-001.4 is a presentation + wiring card. It does not modify the engine /
role / stage gate logic in `actPopoutModel`, does not modify the submit path,
does not modify any role permission, and does not introduce a new menu kind.

Doctrine constraints that shape the design (`cdiscourse-doctrine`):

- Disabled-with-reason copy is plain language only; no verdict tokens; no raw
  internal codes; every internal-code mapping goes through
  `gameCopy.toPlainLanguage`.
- Act preserves the 3-gate model verbatim (engine hard, role hard, stage
  soft); UX-001.4 does not touch `buildActPopout`.
- No AI calls. No service-role. No client-side write path beyond the existing
  composer + the existing direct-action callbacks already in
  `ArgumentSideActionRail`.

## §2 — Three-menu consolidation (Q2)

The Q1 audit shows that the three menus already share the same chassis
(`Popout.tsx` provides Modal + scrim + flash + Esc + hardware back) but have
no breakpoint variance, no uniform trigger contract, and no uniform keyboard
binding. §2 specifies the shared contract.

### 2.1 — Shared chrome contract

All three menus consume the same `Popout` chassis with identical prop wiring:

```ts
<Popout
  visible={isOpen}
  title="Act" | "Inspect" | "Go"  // already set by each component
  onClose={onClose}
  anchor="bottom"  // bottom for all three on phone/tablet sheet variants
  reduceMotionOverride={reduceMotionOverride}  // PR-001 effective reduce-motion
  testID={`one-box-act-popout` | `inspect-popout` | `go-popout`}
>
  {/* component body */}
</Popout>
```

The chassis already enforces:
- Modal mount with `accessibilityViewIsModal`
- Flash animation 140 ms (Reduce-motion → instant per `AccessibilityInfo`)
- Esc closes on web (`Popout.tsx:147-161`)
- Hardware back closes on native (`Modal onRequestClose`)
- Scrim tap dismisses (`Popout.tsx:199-205`)

UX-001.4 adds breakpoint-aware presentation via a new wrapper. The wrapper is
a thin pure-TS module:

```ts
// New: src/features/arguments/oneBox/menuPresentationModel.ts
// Pure TS. No React. No Supabase.

export type MenuVariant = 'sheet_bottom' | 'panel_side' | 'panel_anchored';

export interface MenuPresentationInput {
  band: 'phone' | 'tablet' | 'wide';
  menu: 'act' | 'inspect' | 'go';
  /**
   * Tablet portrait (~600-1023 wide) prefers bottom sheet; tablet landscape
   * (~1024+) prefers side panel. The host derives orientation from
   * useWindowDimensions; the model accepts the resolved width/height.
   */
  windowWidth: number;
  windowHeight: number;
}

export interface MenuPresentationOutput {
  variant: MenuVariant;
  /** Max height for sheet variants (logical px). */
  maxHeight: number;
  /** Width for side/anchored panel variants (logical px). */
  width: number | null;
}

export function resolveMenuPresentation(input: MenuPresentationInput): MenuPresentationOutput;
```

The resolver is uniform across the three menus, with the only per-menu
difference being the phone-band sheet `maxHeight` cap (Act 50%, Inspect 50%,
Go 40% — Go is typically shorter).

### 2.2 — Shared dismiss behavior

Per A19 (Q1) and §9, the dismiss paths uniformly are:

1. **Esc** (web) — closes the menu, returns focus to the trigger.
2. **Hardware back** (native) — via `Modal onRequestClose`.
3. **Scrim tap** — dismissible (NOT inert; design choice §9).
4. **Trigger re-tap** — closes when the menu is open. Implemented by wiring
   the trigger's onPress to toggle visibility, not unconditionally open it.
5. **Selection** — Act / Go entries close on press; Inspect's hand-off chip
   closes when the chip routes to Act.
6. **Swipe-down on phone** — deferred (see §9.3). Documented for UX-001.5.

The chassis already gives us 1, 2, and 3. UX-001.4 adds 4 (trigger toggle)
and preserves 5 (per-popout). Swipe-down is explicitly deferred.

### 2.3 — Shared keyboard contract

| Key | Effect | Where |
|---|---|---|
| `A` | Open Act (no other modifier) | Board focus only |
| `I` | Open Inspect | Board focus only |
| `G` | Open Go | Board focus only |
| `Esc` | Close the open menu | Open menu |
| `Arrow` keys | Move focus within open menu's entry list | Open menu |
| `Enter` / `Space` | Activate the focused entry | Open menu |

Important constraints:
- A/I/G fire **only** when the board has focus. Composer-focused keystrokes
  consume the letter normally (text input). `useComposerFocusContext` provides
  the boolean.
- A/I/G are exclusive — only one menu can be open at a time. Opening Inspect
  while Act is open closes Act first (the chassis's Modal stack handles this
  cleanly with `visible={false}` first).
- The existing Timeline board nav (arrow / Home / End / Enter / Space / Esc
  for node traversal — `keyboardNavigationModel.ts`) continues to fire when
  no menu is open. When a menu IS open, arrow keys are routed to the menu
  body's focus traversal; the Timeline's `handleKeyDown` short-circuits via
  the new `hasOpenMenu` boolean check.

### 2.4 — Bleed prevention

Each menu answers exactly one question. The "no bleeding" rule:

- **Act NEVER shows inspection content.** Act's body is the
  `buildActPopout`-produced entry groups. No description, no metadata, no
  classifier outputs. The optional "Acting on" header (§3.4) names the
  target but does not describe it.
- **Inspect NEVER surfaces composer-opening entries.** Inspect's §5 hand-off
  chip closes Inspect and OPENS Act on the same selection (Inspect → Act
  bridge). Inspect itself never opens the composer.
- **Go NEVER executes moves.** Go's entries only re-configure viewport, view
  toggle, density, lens, and now `leave_room` (which calls back to
  `handleLeaveRoom` — not a move, an app navigation).

Tests in §15.2 enforce these invariants by source-scanning the three
component files for forbidden cross-imports.

Keyboard `A` on board focus:

```ts
// New: src/features/arguments/boardMenuKeyboardModel.ts
import type { Platform } from 'react-native';

export type BoardMenuKey = 'a' | 'A' | 'i' | 'I' | 'g' | 'G' | 'Escape';

export type BoardMenuKeyEffect =
  | { type: 'none' }
  | { type: 'open_act' }
  | { type: 'open_inspect' }
  | { type: 'open_go' }
  | { type: 'close_open_menu' };

export interface BoardMenuKeyInput {
  key: string;
  metaKey: boolean;
  ctrlKey: boolean;
  shiftKey: boolean;
  altKey: boolean;
  composerFocused: boolean;
  hasOpenMenu: boolean;
}

export function resolveBoardMenuKeyEffect(input: BoardMenuKeyInput): BoardMenuKeyEffect {
  // 1. Composer focused → none (composer text input consumes the letter).
  if (input.composerFocused) return { type: 'none' };
  // 2. Esc when a menu is open → close it. (When no menu is open, Esc is
  //    handled by the existing Timeline nav handler.)
  if (input.key === 'Escape' && input.hasOpenMenu) {
    return { type: 'close_open_menu' };
  }
  // 3. A/I/G — ONLY when no modifier keys held (avoid Cmd+A select-all, etc).
  if (input.metaKey || input.ctrlKey || input.altKey) return { type: 'none' };
  if (input.key === 'a' || input.key === 'A') return { type: 'open_act' };
  if (input.key === 'i' || input.key === 'I') return { type: 'open_inspect' };
  if (input.key === 'g' || input.key === 'G') return { type: 'open_go' };
  return { type: 'none' };
}
```

The handler is bound to the Timeline root `<View>` (the same `onKeyDown` site
where `resolveTimelineNavEffect` runs at `ArgumentTimelineMap.tsx:883`). The
order is: A/I/G handler runs FIRST; if it returns `none`, the existing
Timeline nav handler runs.

### 4.5 — Coordination with UX-001.2 readout's `Show full details ▾` expander

**Designer choice: KEEP THEM SEPARATE (option b in the brief).**

Rationale:

1. **The readout's expander shows the existing sidecar** — the sidecar
   (`ArgumentReplySidecar`) is itself a 6-section detail view that has
   shipped for many releases. Users have built muscle memory around it.
   Replacing it with Inspect would force re-learning.
2. **Inspect is a separate surface with separate dismiss / trigger / keyboard
   semantics.** The readout's expander is "expand in place"; Inspect is "open
   an overlay panel". Conflating them would force one set of semantics on
   the other.
3. **Inspect can be opened independently of the readout's expand state.**
   When the user wants Inspect detail without losing the timeline view
   below, they can open Inspect via `I` / the strip trigger without first
   navigating to the readout. The readout's expander only works if the user
   has the readout visible AND taps the chevron.
4. **The two views have different content evolution paths.** Inspect's
   evolution under UX-001.5 / UX-001.5A adds Observations and Allegations
   labels. The readout's expander stays focused on the existing sidecar's
   META-001 / EV-001 / LIFE-001 fields. Decoupling them keeps each surface's
   evolution clean.

**Trade-off acknowledged:** there is some redundancy — Inspect's `says`
section and the sidecar's body excerpt show overlapping content. The brief
explicitly accepts either choice; this design accepts the redundancy in
exchange for stable existing behavior + independent evolution paths.

**No modification to UX-001.2 readout internals.** The readout panel keeps
its existing `Show full details ▾` expander and continues to render
`ArgumentReplySidecar`. UX-001.4 does not touch
`TimelineSelectedReadoutPanel.tsx` or `ArgumentReplySidecar.tsx`.

### 4.6 — Inspect must NOT modify state

The `InspectPopout` component itself does not call Supabase, does not fire
direct actions, and does not open the composer. The only "mutation" it
triggers is `onHandoffToAct(actEntryId)` which closes Inspect and opens
Act. Inspect itself is read-only.

A source-scan test asserts that `InspectPopout.tsx` does not import
`supabase`, `fetch`, `axios`, any router, or any `Linking` module
(`__tests__/inspectPopoutReadOnlyImports.test.ts` — new file in §15).

## §7 — Keyboard implementation (Q7)

### 7.1 — A/I/G handler placement

New pure-TS file: `src/features/arguments/boardMenuKeyboardModel.ts`
(per §3.3). Exports `resolveBoardMenuKeyEffect` + the union types.

The handler is wired into `ArgumentGameSurface.tsx` (NOT into
`ArgumentTimelineMap.tsx` — placing it one level higher makes it work for
both Timeline and Cards views). New `useEffect` block:

```tsx
useEffect(() => {
  if (Platform.OS !== 'web') return;
  if (typeof document === 'undefined' || !document.addEventListener) return;
  const handleKeyDown = (event: KeyboardEvent) => {
    const effect = resolveBoardMenuKeyEffect({
      key: event.key,
      metaKey: event.metaKey,
      ctrlKey: event.ctrlKey,
      shiftKey: event.shiftKey,
      altKey: event.altKey,
      composerFocused,  // from existing useComposerFocusContext consumer
      hasOpenMenu: boardActVisible || inspectVisible || goVisible,
    });
    switch (effect.type) {
      case 'open_act':
        event.preventDefault();
        setBoardActPopoutVisible(true);
        return;
      case 'open_inspect':
        event.preventDefault();
        setInspectVisible(true);
        return;
      case 'open_go':
        event.preventDefault();
        setGoVisible(true);
        return;
      case 'close_open_menu':
        event.preventDefault();
        setBoardActPopoutVisible(false);
        setInspectVisible(false);
        setGoVisible(false);
        return;
      case 'none':
      default:
        return;
    }
  };
  document.addEventListener('keydown', handleKeyDown);
  return () => document.removeEventListener('keydown', handleKeyDown);
}, [composerFocused, boardActVisible, inspectVisible, goVisible]);
```

### 7.2 — Coexistence with UX-001.3 composer shortcuts

The composer's `Cmd/Ctrl+Enter` / `Cmd/Ctrl+K` / `Esc` shortcuts in
`composerKeyboardModel.resolveComposerKeyEffect` already check
`input.composerFocused`. The board-menu handler ALSO checks
`composerFocused` (and returns `'none'` when true). They are mutually
exclusive:

| State | Cmd+Enter | Cmd+K | Esc | A | I | G |
|---|---|---|---|---|---|---|
| Composer focused, no menu open | submit | mode switcher | close composer | letter typed | letter typed | letter typed |
| Board focused, no menu open | (no effect) | (no effect) | (no effect; Timeline nav handles) | open Act | open Inspect | open Go |
| Board focused, menu open | (no effect) | (no effect) | close menu | (consumed by chassis: focus menu entries via arrows; A would just be a letter the menu ignores) | (same) | (same) |

When a menu is open, the menu owns the keyboard. Inside the menu, A/I/G are
not special — they're just letters that don't navigate.

### 7.3 — Coexistence with existing board shortcuts

`resolveTimelineNavEffect` handles `ArrowLeft` / `ArrowRight` / `Home` / `End`
/ `Enter` / `Space` / `Escape`. None of these overlap with A/I/G. The new
board-menu handler runs FIRST (it's bound at the `ArgumentGameSurface` level,
which is HIGHER in the DOM tree than the `ArgumentTimelineMap` level where
`handleKeyDown` is bound). When the new handler returns `none` for arrow /
Home / End / Enter / Space, the existing Timeline handler still fires.

There is a subtle Esc collision: when a menu is open, the new handler
returns `close_open_menu` and preventDefaults; when no menu is open and the
Timeline has an overlay (popover / dock target selected), the existing
Timeline handler's Esc closes the overlay. The order is:

1. Board-menu handler fires first; if `hasOpenMenu === false`, returns
   `none` (no preventDefault).
2. Timeline handler fires next; if `hasOpenOverlay === true`, returns
   `close_overlay` and preventDefaults.

This ordering means Esc always closes the most-recently-opened thing. The
test in §15.1.G asserts this ordering by simulating the keydown with both
states.

### 7.4 — Focus context model + rationale for new file

The new file `boardMenuKeyboardModel.ts` is chosen instead of extending
`keyboardNavigationModel.ts`:

**Rationale:** `keyboardNavigationModel.ts` (IX-003) is scoped to TIMELINE
node traversal — it's a pure model whose inputs are an `ArgumentTimelineMapModel`
and an `activeMessageId`. The board-menu handler has different inputs
(composerFocused, hasOpenMenu) and produces different effect types. Extending
the IX-003 model would muddle two concerns (node-traversal vs menu-open).
A separate, small, focused model is cleaner per the `expo-rn-patterns`
"Model files (`*Model.ts`) MUST be pure TypeScript" pattern.

The new file imports nothing except React Native's `Platform` (used only
for the unused `Platform`-aware web check — actually, the model is pure TS
and platform-agnostic; the host code does the `Platform.OS` check). Final
imports for the model file: NONE (zero imports — purely string-key
classification).

## §8 — Key badges (Q8)

### 8.1 — Browser-only rendering

The chassis `PopoutEntry` already supports an optional `keyBadge?: string`
prop (`PopoutEntry.tsx:113-118`) that renders a small monospace chip. UX-001.4
extends the per-popout content models to populate `keyBadge` for the
top-level menu triggers AND for first-line entry shortcuts (e.g., Act's
"Reply" gets `R` if applicable — deferred decision; v1 only badges the
top-level menu triggers).

For the menu trigger badges (the `A` / `I` / `G` on the strip-level
triggers), UX-001.4 introduces a new pure-TS helper:

```ts
// In: src/features/arguments/oneBox/menuKeyBadgeModel.ts
// Pure TS.

export type MenuKeyBadgeContext = 'browser_keyboard' | 'touch' | 'unknown';

export interface KeyBadgeInput {
  context: MenuKeyBadgeContext;
  reduceMotion: boolean;  // not used for badges in v1; kept for future
}

export function resolveKeyBadgeVisibility(input: KeyBadgeInput): boolean {
  return input.context === 'browser_keyboard';
}

export function deriveMenuKeyBadgeContext(input: {
  platformOs: 'web' | 'ios' | 'android' | 'windows' | 'macos';
  windowWidth: number;
  hasPointerFine?: boolean;  // optional CSS pointer media query reading
}): MenuKeyBadgeContext {
  // Web + wide viewport + (optional fine pointer signal) → browser keyboard.
  if (input.platformOs !== 'web') return 'touch';
  // The brief: badges on browser, not on phone/tablet. We treat wide and
  // tablet-landscape viewports as browser. Tablet portrait stays 'touch'
  // because keyboard is unlikely.
  if (input.windowWidth >= 1024) return 'browser_keyboard';
  return 'touch';
}
```

The host (`ArgumentGameSurface`) calls `deriveMenuKeyBadgeContext` to derive
the context, then passes `keyBadge="A"` / `"I"` / `"G"` to the strip-level
triggers when `resolveKeyBadgeVisibility(...)` returns true.

### 8.2 — Accessibility for badges

The badge is rendered next to the trigger label. Its accessibility:

- The badge `<Text>` has `accessibilityElementsHidden` and
  `importantForAccessibility="no"` (existing pattern in
  `PopoutEntry.tsx:209-215`).
- The trigger Pressable's `accessibilityLabel` includes the keyboard
  shortcut info: `accessibilityLabel="Open Act menu. Keyboard shortcut: A."`

The badge is purely visual; screen readers learn the shortcut via the
trigger's accessibilityLabel.

### 8.3 — Touch laptop / hybrid devices

Hybrid devices (Surface, Chromebook with touch) are edge cases. The badges
render based on viewport width (`>= 1024`), not on touch capability. A user
on a 1024+ tablet with no keyboard still sees the badge; this is acceptable
because (a) the badge is small and unobtrusive, (b) the trigger is still
tappable, (c) testing every input combination is out of scope. The
accessibility label informs screen-reader users regardless of the badge's
visual presence.

### 9.1 — 44x44 hit targets

Every interactive element in the three popouts already meets 44x44 via the
chassis:

- `InspectPopout` traversal: `minHeight: 44, minWidth: 72`
  (`InspectPopout.tsx:504-505`)

UX-001.4 adds no new interactive elements that violate the 44x44 rule. The
new "Acting on" header in Act (§3.4) is a `<Text>` (not Pressable), so no
hit-target requirement. The new `leave_room` Go entry consumes the existing

strip uses `hitSlop` to lift sub-44 visuals to 44+ — see
`DebateDetailHeader.tsx`).

### 9.2 — Dismiss behavior: dismissible scrim (NOT inert)

**Designer choice: ADOPT the chassis's existing dismissible-scrim pattern.**

Rationale:

1. **Act/Inspect/Go are low-commitment surfaces.** They show information /
   surface actions; closing them costs nothing. The user can re-open by
   pressing A/I/G or the trigger.
2. **The composer's inert scrim guards an in-progress draft.** A draft has
   value (the user typed it). An open menu has no analogous user investment.
3. **The chassis already implements dismissible scrim** (`Popout.tsx:199-205`).
   Deviating to inert would require modifying the chassis, which is broader
   than UX-001.4's scope.
4. **Brief allows either choice.** The brief explicitly says "Swipe-down to
   dismiss (or operator-accepted fallback if the existing inert-scrim
   pattern is preserved — see UX-001.3's deferred PanResponder note in
   docs/reviews/UX-001.3-review.md; UX-001.4 may inherit the same pragmatic
   deferral)". The "may inherit" is permissive; the design adopts the
   chassis's existing behavior.

This document records the choice for UX-001.5 (per brief §"Phase 4 framing
handoff"): popout dismissible scrim, composer inert scrim.

### 9.3 — Swipe-down to dismiss

**Deferred.** Rationale:

1. The chassis (`Popout.tsx`) does not currently support swipe gestures
   (no PanResponder). Adding it would require either:
   - Adding `react-native-gesture-handler` as a dep (banned per
     `expo-rn-patterns` policy; speculative dep)
   - Or implementing PanResponder ad-hoc (UX-001.3 deferred this exact path
     for the composer with the same rationale)
2. The chassis has 4 working dismiss paths (Esc, hardware back, scrim tap,
   trigger re-tap). User research can determine if swipe is missed; if so,
   a future card can add it once the dep policy / PanResponder ergonomics
   are settled.

### 9.4 — Tap-outside to close

Same as 9.2: the chassis's scrim tap closes the popout. Tap-outside is the
same as scrim tap (the scrim covers all non-panel area).

### 9.5 — Esc closes

The chassis already binds Esc (`Popout.tsx:147-161`). UX-001.4 does not
override; it adds the §7.1 Esc handler at the `ArgumentGameSurface` level
that also closes the open menu (via `setBoardActPopoutVisible(false)`
etc.). The two handlers are not redundant: the chassis Esc closes the
chassis itself (via the chassis's own state); the `ArgumentGameSurface`
Esc closes the menu via parent-side state. Both fire on the same Esc
keypress; the chassis preventDefaults; the `ArgumentGameSurface` handler
preventDefaults; the chassis's Esc fires first because the chassis Modal
captures focus. In practice the result is identical (the menu closes).

The §7.3 ordering note already covers the Esc ambiguity.

## §16 — Risks and edge cases

### 16.1 — A/I/G collision with existing board single-letter shortcuts

Risk: an existing IX-003 binding might use A, I, or G.

Mitigation: Verified empirically. IX-003 binds:
`ArrowLeft / ArrowRight / Home / End / Enter / Space / Escape`
(`keyboardNavigationModel.ts:66-75`). No single-letter bindings exist.
Search for other single-letter handlers in `src/features/`:

```bash
grep -rn "event.key === 'a'" src/features
```

Returns no matches. **A/I/G are free.**

### 16.2 — Bottom sheet on tablet portrait vs landscape

Risk: an iPad rotation mid-session changes the variant. The chassis Modal
remounts on prop changes, which could cause a visible re-flash.

Mitigation: `useWindowDimensions` updates on orientation change; the
host's `menuPresentationModel.resolveMenuPresentation` re-runs and the
prop changes. The chassis's flash animation is 140 ms; the perceived
re-flash is acceptable. Alternative would be to memoize the variant
against a coarse "is_landscape" boolean; deferred unless reviewer reports
a UX regression.

### 16.3 — Inspect content rendering when classifier outputs are heavy

Risk: a node with many semantic flags + a long evidence chain could
produce a very tall Inspect body.

Mitigation: The chassis ScrollView handles overflow. The per-band
maxHeight caps the panel. The §6 `flags` section concatenates
semanticFlags with ` · ` separator (no per-chip rendering in v1);
UX-001.5 changes this to chips.

Lazy-load deferred: not necessary in v1 because the content is built
synchronously from already-resident client state. If UX-001.5 adds
per-chip metadata fetching, lazy loading becomes relevant then.

### 16.4 — Duplicate rail removal regressing a high-frequency shortcut

Risk: removing an entry the user relies on muscle memory for.

Mitigation: The `preserve-as-shortcut` disposition keeps every
high-frequency entry. The B.2 `reply` / `disagree` retain direct
behavior (no Act roundtrip). The B.1 observer entries (`watch`,
`join_aff`, `join_neg`) preserve as shortcuts (one-tap into a state
change).

Reviewer-measurement test (§6.2 / §15.3 B-2) verifies behavioral
parity between rail-shortcut and Act-entry for the preserved entries.

### 16.5 — Inspect → Act footer transition affecting selected-node state

Risk: when Inspect's hand-off chip closes Inspect and opens Act, the
selected node id must remain consistent.

Mitigation: Both Inspect and the board-level Act mount share the same
`selectedActTarget` state in `ArgumentGameSurface.tsx`. The hand-off
flow:
1. Inspect's `handleHandoff` calls `onHandoffToAct(actEntryId)`.
2. Host's `handleInspectHandoffToAct` sets `setBoardActPopoutVisible(true)`.
3. The board-level Act mount opens with the SAME `selectedActTarget`.
4. Inspect's `onClose()` runs (state: `inspectVisible = false`).

The `selectedActTarget` is never mutated by the transition. Asserted by
`__tests__/inspectToActHandoffPreservesTarget.test.ts` (additional NEW;
add to §15.1).

### 16.6 — Key badge accessibility on browser+touch hybrids

Risk: hybrid devices (Surface, Chromebook touch) show the key badge but
the user uses touch.

Mitigation: The badge is visual-only; touch users ignore it. The trigger
still works via tap. Accessibility label includes the shortcut info for
screen-reader users regardless of input modality.

### 16.7 — Disabled-with-reason copy crossing into verdict territory

Risk: a host accidentally passes a verdict-token string to
`disabledEntries`.

Mitigation: `__tests__/disabledReasonCopyBanList.test.ts` scans every
authored disabled-reason string in the codebase. The chassis's
`PopoutEntry.tsx` could also defensively scan via the existing
`looksLikeInternalCode` helper from `gameCopy.ts`, but this is deferred
to keep the chassis untouched.

### 16.8 — Menu mutual exclusion implementation race

Risk: two menus open momentarily if the user triggers A and I in quick
succession (within a few ms).

Mitigation: React state updates batch synchronously within an event
handler. The handler in §7.1 fires for a single keydown; the keydown can
only carry one key. Cross-keydown races: when the second keydown fires,
the first menu's state is already updated. The board-menu handler reads
the latest state via the dependency array. Worst case: a single frame of
double-mounting; acceptable.

Asserted by `__tests__/uxOneOneFourMenuExclusivity.test.ts` (§15.1).

### 16.9 — Strip-level trigger placement requires UX-001.2 strip modification

Risk: Adding Act / Inspect / Go triggers to `DebateDetailHeader.tsx`
(the UX-001.2 strip) requires modifying a UX-001.2-owned file. UX-001.2
framing says "UX-001.3 may add new strip prop wiring only with operator
approval".

Mitigation: UX-001.4's brief §16 explicitly lists
`src/features/arguments/oneBox/Popout.tsx`, `PopoutEntry.tsx`,
`PopoutGroup.tsx` as allowed modifications. The strip
(`DebateDetailHeader.tsx`) is not in the brief's allowed list. **Design
mitigation:** add the triggers to the strip's overflow `⋯` panel via a
new prop on `DebateDetailHeader`. This is the same prop-wiring extension
that the operator pre-authorized for UX-001.3 ("UX-001.3 may add new
strip prop wiring only with operator approval"). UX-001.4 extends the
same allowance.

**Operator confirmation surfacing:** §17 §17.B notes this; the operator
reviews at the completion report.

### 16.10 — Boundary risk: brief allows board-menu handler in ArgumentGameSurface

The brief §16 lists `src/features/arguments/oneBox/` as a primary
allowed modification scope but does not explicitly list
`ArgumentGameSurface.tsx`. The brief's allowed modifications include
"New pure-TS modules for menu keyboard coordination (e.g.,
`boardMenuKeyboardModel.ts` if needed; coexists with
`keyboardNavigationModel.ts`)" and "New pure-TS modules for key-badge
presentation". The HOST wiring in `ArgumentGameSurface.tsx` is implied:
the brief explicitly lists the mounts as "duplicate rail removal" scope
on `ArgumentSideActionRail.tsx` / `TimelineNodeActionDock.tsx` /
`ObserverActionDockLayout.ts` / `timelineNodeActionDockModel.ts`, but
not the parent surface.

**Design mitigation:** The mount additions to `ArgumentGameSurface.tsx`
are BOUNDED — only the 3 new mounts (board-level Act, Inspect, Go),
the new `useEffect` for A/I/G keyboard, the new prop pass-through for
`onLeaveRoom`, and the new prop wiring for `actingOnLabel`. No
modification to the existing rail mount semantics, no modification to
the existing TimelineSelectedReadoutPanel rendering, no modification
to the existing tab bar conditional.

**Operator confirmation surfacing:** §17 §17.B notes this; the operator
reviews at the completion report.
