## 1. Goal (one paragraph)

MCP-003 designs the **referee ledger** — layer 3 of the semantic-referee architecture locked by MCP-001. It is the deterministic model that takes three inputs — (1) the deterministic metadata layer (layer 1), (2) the advisory `SemanticRefereePacket` from layer 2 (MCP-001's contract), and (3) the existing deterministic point-standing economy in `src/features/pointStanding/` — and produces **play-quality feedback** that makes the board feel like a referee is keeping points about the **rules of play**. The ledger is **deterministic**; the semantic packet is one advisory input among many, never its master. The doctrine that shapes every decision: the referee scores continuity, evidence hygiene, branch hygiene, and constructive movement — it **never** decides who is right, never declares a winner or loser, never assigns a truth value, never blocks a post, and keeps engagement credit and factual-standing credit on **separate** axes (per `antiAmplification.ts`). Concession renders as a **repair**, never a defeat. This card does not replace the point-standing economy — it is a thin reconciliation-and-presentation layer **above** the Constitution that **consumes** that economy's existing `gradeChallenge` / `gradeRepair` / `applyAntiAmplification` / `IssueDebtLedger` exports.

## 2. Why this is a layer above, not a rewrite

`src/features/pointStanding/` (Stage 6.1.4, doctrine in `docs/point-standing-economy.md`) already answers "what happened to a point after pressure?" with `PointStandingDelta`, `OpenIssueDebt`, `ConcessionEffect`, `gradeChallenge`, `gradeRepair`, `IssueDebtLedger`, and `applyAntiAmplification`. That economy is **the substrate**. What it does not have is:

1. A way to **fold in the semantic packet** — its inputs are `GradingFlags` + `AgreementDisagreementVector`, not a `SemanticRefereePacket`.
2. A way to **reconcile** a layer-1 deterministic signal with a layer-2 semantic signal when they disagree.
3. A **broader category set** — it moves `broadStanding` / `narrowStanding` plus four engagement-credit fields; the referee feedback wants finer-grained categories (continuity, branch hygiene, staying in mode, respecting pacing) that the economy does not model.
4. A **playful, non-verdict presentation** mapping from internal deltas to user-facing copy.

MCP-003 adds exactly those four things and nothing else. It does **not** rewrite `gradeChallenge` / `gradeRepair`, does **not** touch `CONCESSION_EFFECT_WEIGHTS` or `MIXED_CLASS_WEIGHTS`, and does **not** bypass `applyAntiAmplification`. Where the referee category overlaps the economy (concession, narrowing, evidence debt), the ledger **calls the economy and adopts its delta** rather than re-deriving it. The ledger's own categories are the ones the economy does not cover (continuity tie, branch hygiene, staying in mode, pacing).

This document is the design only. The exported function/type names below are the **proposed** surface for the future implementation card; they are not yet code.

## 3. Doctrine constraints

From `cdiscourse-doctrine`, `point-standing-economy`, `evidence-doctrine`, and the MCP roadmap expansion. A violation is a blocker.

1. **The ledger scores rules of play, never truth.** No point category is "correct", "true", or "won". Categories describe *how the move played*, not whether the claim is right.
2. **The ledger never declares a winner or a loser.** There is no `winner` field, no aggregate "score to beat", no leaderboard, no per-side total that crowns anyone. A user-facing winner/resolution outcome is explicitly out of scope (it is RULE-007 / GAME-007, doctrine-gated and separate).
3. **The ledger never blocks a post.** It produces feedback *after* a move exists; only deterministic structural validation (empty body, invalid transition, explicit-evidence-without-source, over-length, auth failure, opted-in cooldown) blocks. The ledger has no `block` field.
4. **The ledger never assigns a truth value.** It records standing movement in the game; `broadStanding` / `narrowStanding` are gameplay weights, not truth probabilities.
5. **AI is advisory only.** The `SemanticRefereePacket` carries `authoritative: false` (MCP-001 enforces the literal type). The ledger treats every semantic input as a *hint* and downgrades its own output confidence when the hint is uncertain or conflicts with deterministic metadata.
6. **Popularity / heat / virality / volume / rhetorical force / perceived political alignment / crowd vote / user identity / user reputation never earn points.** None of these is an input to any positive delta. Heat is activity, not correctness.
7. **Engagement credit and factual-standing credit stay separate.** The ledger preserves the `antiAmplification.ts` separation: a move's engagement-credit categories and its factual-standing categories are scored on independent axes and an amplification-only move can earn engagement credit while factual-standing gain is suppressed until evidence arrives.
8. **Concession is a scoring repair, not a scoring defeat.** A narrow concession that preserves the broad point *lifts* broad standing. The ledger never renders a concession as "lost" / "defeated" / "gave up".
9. **No verdict or person labels** anywhere in the ledger's output — never *liar, dishonest, bad faith, manipulative, troll, propagandist, wrong, right, true, false, winner, loser, won, lost, proven, defeated, extremist, stupid, idiot*. The ledger classifies the *move*, never the *person*.
10. **No `snake_case` internal codes in user-facing strings.** Every category id and feedback code maps through a plain-language catalog (extends `gameCopy.toPlainLanguage`); an unmapped code is suppressed, not echoed.
11. **The ledger model is pure TypeScript.** No imports from Supabase, React, or any network library — same rule as the rules engine and the point-standing economy. It runs identically on client and in an Edge Function.

## 4. Scope item 1 — Point categories the ledger may move

The ledger moves **fourteen** point categories. Each category is a named, bounded axis of *play quality*. None is a truth category. The proposed enum:

```ts
export type RefereePointCategory =
  | 'continuity'                  // the move tied cleanly to the parent
  | 'direct_response'             // the move answered the question / pressure on the table
  | 'evidence_provided'           // the move attached or referenced an evidence artifact
  | 'evidence_relevance'          // the attached evidence connects to the exact claim
  | 'quote_anchoring'             // the move anchored to a parent span / exact quote
  | 'narrowing'                   // the move limited a broad claim to a defensible scope
  | 'concession'                  // the move accepted a specific limited point (a repair)
  | 'clarification'               // the move asked for / supplied a needed clarification
  | 'synthesis'                   // the move named genuine shared ground
  | 'branch_hygiene'              // a new issue was routed to a branch, not jammed on the mainline
  | 'person_intent_drift'         // (negative-only) the move targeted the speaker, not the claim
  | 'evidence_debt_resolution'    // the move closed an open evidence / issue debt
  | 'staying_in_mode'             // the move's register matched the room's selected mode
  | 'respecting_pacing';          // the move respected the room's pacing / turn rules
```

### 4.1 Category rules

- **Each category carries its own delta**, signed. Positive = the move played that axis well; negative = the move worked against that axis. They are **never summed into one number**. A "score" the user sees is a *set of category readings*, not a scalar — this mirrors the point-standing economy's refusal to collapse all debts into one score.
- **`person_intent_drift` is negative-only.** It can read `0` (no drift) or a small negative value (drift present). It can never be a positive credit — there is no reward for *not* attacking a person; that is the baseline.
- **Two credit axes.** Categories split into:
  - **Engagement-credit categories** — `continuity`, `direct_response`, `quote_anchoring`, `narrowing`, `concession`, `clarification`, `branch_hygiene`, `staying_in_mode`, `respecting_pacing`. These reward *constructive movement and play hygiene*. They are earned by activity-shaped behavior and are **never gated by evidence**.
  - **Factual-standing-credit categories** — `evidence_provided`, `evidence_relevance`, `evidence_debt_resolution`. These are truth-adjacent (still gameplay, not Truth™) and are **subject to the anti-amplification gate**: an amplification-only move earns nothing on these even if it earns engagement-credit categories.
  - `synthesis` and `person_intent_drift` straddle: `synthesis` earns engagement credit for naming shared ground but only earns the factual-standing portion if the synthesized subtree's debts were genuinely resolved; `person_intent_drift` is a negative-only hygiene reading on neither axis's *credit* side.
- **No category is named after a verdict.** "continuity" is not "correctness"; "evidence_relevance" is "the evidence connects to the claim", never "the claim is true".
- **Categories are advisory.** Like every output in this family, the ledger result carries `userReviewRequired: true` (consistent with the point-standing economy's existing contract).

### 4.2 Category → economy mapping (so we do not double-count)

| Category | Owned by the economy? | How the ledger sources its delta |
|---|---|---|
| `continuity` | No | Ledger-derived (§6, from `continuityCredit` hint + layer-1 parent-id check) |
| `direct_response` | No | Ledger-derived (semantic `responds_to_parent` + layer-1 answer-state) |
| `evidence_provided` | Partly (`OpenIssueDebt` close) | Ledger-derived; routed through `applyAntiAmplification` |
| `evidence_relevance` | No | Ledger-derived; routed through `applyAntiAmplification` |
| `quote_anchoring` | No | Ledger-derived |
| `narrowing` | **Yes** | Adopt `gradeRepair`'s `broadStandingDelta` / `narrowStandingDelta` for the narrow-concession effects |
| `concession` | **Yes** | Adopt `gradeRepair`'s `ConcessionEffect` + delta directly; render as repair |
| `clarification` | No | Ledger-derived |
| `synthesis` | Partly | Ledger-derived; closes economy debts via `IssueDebtLedger.updateDebt` |
| `branch_hygiene` | No | Ledger-derived (from `branchHygiene` hint + BR-001 layer-1 branch kind) |
| `person_intent_drift` | No | Ledger-derived (negative-only) |
| `evidence_debt_resolution` | **Yes** | Adopt the economy's debt-close path; the ledger does not invent a parallel debt store |
| `staying_in_mode` | No | Ledger-derived (GAME-003 mode profile, layer-1 dominant) |
| `respecting_pacing` | No | Ledger-derived (GAME-002 pacing state, layer-1 dominant) |

**Hard rule:** where a category is "owned by the economy", the ledger **calls the economy function and adopts its number** — it never re-computes a parallel standing delta. This guarantees the worked examples in `docs/point-standing-economy.md` stay true after MCP-003 ships.

## 5. Scope item 2 — Reconciliation rules (deterministic vs semantic packet)

The ledger has, for most categories, **two signals**: a layer-1 deterministic derivation and a layer-2 semantic binary. MCP-001 §14 sets the doctrine; MCP-003 makes it concrete.

### 5.1 The reconciliation algorithm

For each category, the ledger computes a `CategoryReading`:

```ts
export type ReconciliationOutcome =
  | 'agreement'              // L1 and L2 agree → full-confidence delta
  | 'l1_only'                // no semantic signal exists for this category → L1 stands
  | 'l2_only'                // no deterministic signal exists → L2 stands, capped confidence
  | 'conflict_l1_dominant'   // L1 and L2 disagree AND L1 is authoritative for this category
  | 'conflict_routed'        // L1 and L2 disagree, neither is dominant → route to user choice
  | 'low_confidence_semantic'; // L2 confidence is 'low' → softened, no standing move on its own

export type LedgerConfidence = 'high' | 'medium' | 'low';

export interface CategoryReading {
  category: RefereePointCategory;
  delta: number;                 // signed; clamped to the category's bounded range
  outcome: ReconciliationOutcome;
  confidence: LedgerConfidence;
  /** Set when the outcome is `conflict_routed`. The caller surfaces a reversible MCP-010 choice. */
  requiresUserChoice: boolean;
  /** Plain-language code → mapped by gameCopy before any UI surface. Never raw. */
  feedbackCode: string;
  /** True only for evidence_* categories that passed/were suppressed by anti-amplification. */
  factualStandingAxis: boolean;
}
```

Per category, the rule:

1. **`l1_only`** — some categories have no semantic counterpart (`respecting_pacing`, `staying_in_mode` are layer-1-dominant; `jump_branch` state is deterministic-dominant). L1 stands, `confidence: 'high'`.
2. **`agreement`** — L1 and L2 produce the same sign (both say "responds to parent" / both say "does not"). The delta fires at full magnitude; `confidence: 'high'`.
3. **`l2_only`** — L1 has no derivation for this category (e.g. "is this evidence *relevant* to the claim" — layer 1 can see an artifact exists but not whether it connects). L2 stands but its delta magnitude is **capped at the medium tier** and `confidence` follows the packet's own `SemanticConfidence`.
4. **`low_confidence_semantic`** — the semantic binary's `confidence` is `'low'`. Per MCP-001 §14, a low-confidence binary **never moves standing on its own**. The category reading is emitted with `delta: 0`, `confidence: 'low'`, and a *softer* feedback code (a gentle prompt, not a credit). MCP-008 renders it as a soft banner or no banner.
5. **`conflict_l1_dominant`** — L1 and L2 disagree, and L1 is the authoritative layer for this category (the deterministic fact is unambiguous — e.g. `hasAttachedEvidence` is a button-and-artifact fact, layer 1 wins; `respecting_pacing` is a clock fact, layer 1 wins). The ledger uses the L1 value, downgrades `confidence` to `'medium'` (because the layers disagreed), and **emits no penalty for the disagreement itself**.
6. **`conflict_routed`** — L1 and L2 disagree, and **neither is authoritative** (the question is genuinely structural-meaning — e.g. "did this paragraph engage the parent's *mechanism*?"). The ledger:
   - sets `delta: 0` for that category (it does **not** guess),
   - sets `confidence: 'low'`,
   - sets `requiresUserChoice: true`,
   - emits a `feedbackCode` that maps to a *reversible MCP-010 choice prompt* ("Reply here, branch, or tangent? — you decide"), **never a penalty and never a silent override**.

### 5.2 The non-negotiable conflict doctrine

> **On a layer-1 / layer-2 conflict the ledger downgrades its own confidence and routes to a reversible user choice. It never converts the disagreement into a penalty, never silently overrides the user, never moves standing on the contested category.**

This is the single most important rule in MCP-003. The original keyword cage produced false positives that punished users; the referee must never repeat that. Concretely:

- A conflict **never** produces a negative delta on the contested category. The worst a conflict can do is `delta: 0` + a reversible prompt.
- A conflict **never** flags the user, never adds a `flags` row, never emits "you were warned" copy. (Consistent with MCP-001 §15.)
- The user's manual override (MCP-010) carries **no score cost**. If the user asserts "this answers the parent" to overturn a `responds_to_parent: 0`, the `continuity` and `direct_response` categories are re-read from the override as `agreement`-tier `l1_only` (the user's explicit assertion becomes the authoritative layer-1 fact), at full magnitude, with no penalty for the earlier conflict.
- The override is recorded as **visible metadata** (a META-001-style `semantic_override` entry), not hidden state — but MCP-003 only *consumes* that entry; the entry's creation is MCP-010's job.

### 5.3 Which layer is authoritative for which category

| Category | Authoritative layer on conflict | Why |
|---|---|---|
| `continuity` | neither → `conflict_routed` | "engaged the parent's mechanism" is structural-meaning |
| `direct_response` | neither → `conflict_routed` | "answered the question on the table" is structural-meaning |
| `evidence_provided` | **layer 1** | "an artifact row exists" is a button-and-artifact fact |
| `evidence_relevance` | neither → `conflict_routed` | "the evidence connects to the *exact* claim" is meaning |
| `quote_anchoring` | **layer 1** (presence) / neither (relevance) | quote presence is deterministic; "addresses the quote" is meaning |
| `narrowing` | economy (`gradeRepair`) | the economy's `ConcessionEffect` classifier is the authority |
| `concession` | economy (`gradeRepair`) | same — the economy owns concession classification |
| `clarification` | **layer 1** (was a clarify action selected) | `selectedClarify` is a button fact |
| `synthesis` | **layer 1** (lifecycle `synthesis_ready`) | lifecycle state gates; semantic only confirms readiness |
| `branch_hygiene` | **layer 1** (BR-001 branch kind) | branch kind is a deterministic topology fact |
| `person_intent_drift` | neither → `conflict_routed` | "targets the speaker" is structural-meaning; route, never penalize on a guess |
| `evidence_debt_resolution` | economy (`IssueDebtLedger`) | the economy owns the debt store |
| `staying_in_mode` | **layer 1** (GAME-003 mode profile) | register-fit is mode-config-driven |
| `respecting_pacing` | **layer 1** (GAME-002 pacing state) | pacing is a clock fact |

## 6. Scope item 3 — `scoreHints` → ledger delta mapping

MCP-001's `SemanticRefereePacket.scoreHints` carries six integers. Per MCP-001 §16 they are **hints, not deltas** — the ledger decides the actual movement. The proposed mapping:

```ts
// MCP-001 contract:
// scoreHints: { continuityCredit, evidencePressure, branchHygiene,
//               synthesisReadiness, sourceChainDebt, unresolvedRedirectRisk }
```

Each hint is a small bounded integer (MCP-001 §7 says "integers in a small bounded range"; MCP-003 assumes the range `0..3` and the implementation card confirms the exact ceiling with the seed bank). The mapping table:

| `scoreHint` field | Feeds category | Direction | Mapping rule |
|---|---|---|---|
| `continuityCredit` | `continuity`, `direct_response` | positive | `delta = HINT_STEP * continuityCredit`, then reconciled (§5) against layer-1 parent-id presence. `0` → no credit, never a penalty. |
| `evidencePressure` | `evidence_provided`, `evidence_relevance` | **opens debt, does not score** | `evidencePressure > 0` means the move *needs* evidence — it raises an `OpenIssueDebt` via the economy, it does **not** subtract standing. Evidence pressure is a prompt, never a penalty. |
| `branchHygiene` | `branch_hygiene` | positive | `delta = HINT_STEP * branchHygiene`, reconciled against BR-001 branch kind (layer-1 dominant). |
| `synthesisReadiness` | `synthesis` | positive (gated) | Only contributes credit when layer-1 lifecycle is `synthesis_ready`. Otherwise it produces a *prompt* ("Almost a synthesis — name the shared point"), never a credit. |
| `sourceChainDebt` | `evidence_debt_resolution` (negative side: an *open* debt) | **opens / sizes debt** | `sourceChainDebt > 0` sizes the evidence debt that the economy tracks; a *later* move that closes it earns `evidence_debt_resolution` credit. Aligns with EV-003's evidence-debt tracker. |
| `unresolvedRedirectRisk` | `branch_hygiene` (routing), `person_intent_drift` (drift) | **routes, does not penalize** | High redirect risk means "this looks like it changes the subject" → it raises a `routeSuggestion`-backed reversible choice (MCP-010), it does **not** subtract standing. |

### 6.1 The crucial direction rule

> **`scoreHints` may only ever *add* credit on a constructive category, *open or size a debt*, or *raise a reversible routing prompt*. A `scoreHint` may never directly subtract standing.**

The only negative deltas in the whole ledger come from:
1. the **economy's own** `gradeRepair` evasion path (`no_concession` / `performative_concession_no_repair` → `unresolvedDebtPenalty`, narrow-standing drop) — which is the existing, doctrine-reviewed Stage 6.1.4 behavior; and
2. the `person_intent_drift` negative-only hygiene reading — and even that is `conflict_routed` (so a guess never penalizes; only an *agreement* between L1 and L2, or a clear layer-1 fact, produces the small negative).

`scoreHints` are advisory; an advisory AI signal can encourage but cannot punish. This keeps the ledger inside doctrine constraint 5 (AI is advisory only) and 3 (never blocks / never punishes a post).

### 6.2 `HINT_STEP` and clamping

- `HINT_STEP` is a small constant (proposed `0.08`) so the maximum a hint-driven category can move is bounded (`0.08 * 3 = 0.24`) — comparable to the economy's `+0.25` broad-lift, deliberately *not larger*. The semantic layer must never out-weigh the deterministic economy.
- Every category delta is clamped to `[-0.4, +0.4]` (the economy's own deltas already sit inside this; `narrow_shrink` is `-0.15`, evasion narrow-drop is `-0.30`).
- When a category is `low_confidence_semantic`, the hint contributes `0` regardless of its integer value (a low-confidence hint cannot move standing — §5.1 rule 4).

## 7. Scope item 4 — Anti-exploit gates

Gamification can be drilled, but not farmed. MCP-003 **reuses the point-standing economy's existing `IssueDebtLedger` + `ScoringEligibility` patterns** rather than inventing a parallel anti-exploit machine.

### 7.1 The four gate conditions (issue scope item 4, verbatim)

A move earns **no credit on any category** when any of these is true:

| Gate | Reuses | Detection |
|---|---|---|
| **tangent** | `ScoringEligibility.isTangent` | `mixedAgreementClass === 'tangent_or_joke'` (already computed by `buildChallengeEligibility`) **or** the semantic `suggests_diagonal_tangent` / `tangent_is_escape_hatch` binary at `agreement` confidence |
| **near-duplicate** | `ScoringEligibility.isNearDuplicate` | another open debt exists on the same axis + intensity (`buildChallengeEligibility` already does this); the ledger additionally compares the move's `contentHash` (from the packet) against recent moves on the same point |
| **self-concession-loop** | `ScoringEligibility.isSelfConcessionLoop` | `buildRepairEligibility`'s existing check: same author, `previousRepairsByAuthor >= 2`, explicit concession marker present |
| **no-axis-identified** | `ScoringEligibility.isAxisIdentified` (false) | `buildChallengeEligibility`'s existing check: `disagreementType` is `none` or `framing`. A move that just expresses disagreement without naming an axis opens no debt and earns no pressure credit. |

When the gate trips, the ledger emits a `LedgerResult` with **every** `CategoryReading.delta = 0`, an `ineligibilityReasons` array (re-using the economy's `reasonsFromEligibility` strings), and a *constructive* feedback code that nudges the move **back into eligibility** — e.g. `tangent_or_joke` → "This may be a tangent — branch it or tie it to the parent point." The gate produces **no penalty** — an ineligible move is a no-op for the ledger, exactly as `gradeChallenge` already returns `eligible: false, delta: null`.

### 7.2 One credit per debt

> **Each credit component is awarded at most once per debt** (issue scope item 4, verbatim).

The economy already enforces this with `OpenIssueDebt`'s `explicitConcessionAwarded` / `impliedConcessionRecorded` / `recoveryCreditAwarded` flags and the `ConcessionCreditState` snapshot. MCP-003 **extends the same pattern** to the ledger's own categories with a per-debt category-credit set:

```ts
/** Mirrors ConcessionCreditState — one entry per (debtId), tracks which
 *  ledger categories have already paid out against that debt. */
export interface LedgerCreditState {
  debtId: string;
  awardedCategories: ReadonlySet<RefereePointCategory>;
}
```

Rule: before the ledger awards `evidence_debt_resolution`, `narrowing`, `concession`, or `quote_anchoring` credit *against a specific debt*, it checks `LedgerCreditState.awardedCategories`. If the category is already present for that `debtId`, the delta is `0` — a re-statement of the same concession or a second source attached to an already-resolved debt pays nothing. The `continuity` / `direct_response` / `staying_in_mode` / `respecting_pacing` categories are **per-move**, not per-debt — they re-evaluate every move (a user *should* tie cleanly to the parent on every move; that is not farmable because the parent changes).

This is a thin wrapper over the economy's existing bookkeeping, not a new store: `LedgerCreditState` is keyed by the *same* `debtId` the economy's `IssueDebtLedger` already mints in `gradeChallenge` (`shortHash(pointId::argumentId::axis)`).

### 7.3 Exploit-risk audit signal

The ledger forwards the economy's `PointStandingDelta.exploitRiskScore` (already a `0..1` audit signal raised by `applyAntiAmplification` and the `performative_concession_no_repair` path) into its `LedgerResult` unchanged. The ledger does **not** act on it — it is an advisory signal for a human reviewer, never an auto-penalty. (Doctrine: AI / heuristics never auto-punish.)

## 8. Scope item 5 — Doctrine-safe presentation

The ledger's output is rendered as **fun, non-verdict feedback**. Banner rendering itself is MCP-008; MCP-003 owns the **copy mapping** — the deterministic map from a category reading to a plain-language string.

### 8.1 The copy-mapping contract

Every `CategoryReading.feedbackCode` is a bounded `snake_case` token. It is **never** shown to a user raw — it is mapped through a plain-language catalog that **extends `gameCopy.toPlainLanguage`** (`src/features/arguments/gameCopy.ts`). An unmapped code is **suppressed, not echoed** (doctrine constraint 10; this is already how `gameCopy.toPlainLanguage` behaves and is test-enforced).

Proposed mapping (the implementation card moves this into `gameCopy` + a `refereeLedgerCopy.ts` table; copy strings are drawn from the seed bank's user-facing copy column so the voice is consistent):

| `clean_parent_tie` | `continuity` (positive) | "Clean parent tie." |
| `partial_parent_tie` | `continuity` (low) | "Answers part of it — the main point is still open." |
| `answered_the_question` | `direct_response` (positive) | "That answers the question on the table." |
| `question_still_open` | `direct_response` (low/0) | "The question on the table is still waiting for an answer." |
| `source_attached` | `evidence_provided` (positive) | "Source attached." |
| `evidence_debt_open` | `evidence_provided` (debt) | "Good pressure — evidence debt opened." |
| `evidence_connects` | `evidence_relevance` (positive) | "The evidence lines up with the claim." |

| `debt_resolved` | `evidence_debt_resolution` (positive) | "Evidence debt resolved — nice." |
| `fits_the_room` | `staying_in_mode` (positive) | "Fits the room." |
| `pacing_is_on` | `respecting_pacing` (notice) | "Pacing is on in this room — a short beat helps." |
| `you_decide_the_lane` | any `conflict_routed` | "Reply here, branch, or tangent? — you decide." |

### 8.2 The presentation ban-list

The ledger's feedback strings are scanned against the doctrine ban-list (verdict + person tokens): `winner · loser · won · lost · right · wrong · true · false · correct · proven · defeated · liar · lying · dishonest · bad faith · manipulative · troll · propagandist · extremist · stupid · idiot · dumb · smart`. A future `__tests__/refereeLedgerBanList.test.ts` enforces it across every produced string (and every `feedbackCode` reason family), matching MCP-001 §19's `semanticPacketBanList` test.

### 8.3 Concession renders as a repair, never a defeat

The `concession` category copy is deliberately *positive-framed*: "Narrow concession noted — the broad point still stands." There is no "you conceded", no "you gave ground", no "you lost the point". A broad concession (`explicit_broad_concession_abandons_point`) renders as "Broad point set down — honest move" (the economy already rewards integrity here; the copy matches). This is doctrine constraint 8 and the point-standing economy's central doctrine made visible.

### 8.4 No aggregate winner surface

The ledger emits a `LedgerResult` *per move*. It deliberately does **not** emit a per-side total, a "score to beat", a leaderboard, or any aggregate that could read as "X is winning". The user-facing surface is a *set of category readings* and *playful banners* — the same way the point-standing economy refuses to collapse debts into one number. A resolution / winner outcome is RULE-007 / GAME-007 and explicitly out of scope (§13).

## 13. Edge cases

- **Empty / missing semantic packet** — when the feature flag is off the packet is a `provider: 'mock'` deterministic stub (MCP-001 §10). The ledger treats every category as `l1_only`: it runs entirely on layer-1 + the economy, emits full-confidence readings, and never errors. The board still keeps points; the semantic layer simply contributes nothing.
- **Semantic packet with `authoritative` not `false`** — impossible per MCP-001's literal type, but a defensive runtime guard treats any non-`false` value as a malformed packet and falls back to `l1_only` (and the implementation card adds a schema test).
- **Move that is neither a clean challenge nor a clean repair** — `moveRole` is required on `LedgerMoveInput`; the caller (the future room wiring) classifies it. If the caller cannot decide, it passes `moveRole` based on whether the author is the original speaker; an ambiguous case routes the `continuity` category to `conflict_routed` rather than guessing.
- **Layer-1 / layer-2 conflict on a category** — §5.1 rule 6: `delta: 0`, `confidence: 'low'`, `requiresUserChoice: true`, a reversible prompt, never a penalty.
- **Low-confidence semantic binary** — §5.1 rule 4: `delta: 0`, soft feedback code, no standing move.
- **Anti-exploit gate trips** (tangent / near-duplicate / self-loop / no-axis) — every category `delta: 0`, `ineligibilityReasons` populated, a constructive nudge, no penalty (§7.1).
- **Re-stated concession on an already-resolved debt** — `LedgerCreditState.awardedCategories` already contains `concession` for that `debtId` → `delta: 0`. No double-pay (§7.2).
- **Amplification-only move** (high engagement, no evidence, no narrowing) — engagement-credit categories may still earn (continuity, etc.); factual-standing categories are suppressed by `applyAntiAmplification` (`factualStandingGainSuppressed: true`); the `evidence_provided` reading renders the "evidence debt opened" prompt, never a penalty (§4.1, §6.1).
- **Move that closes a debt the economy already closed** — `IssueDebtLedger.getDebt(debtId).closed === true` → `evidence_debt_resolution` reading is `0` (debt already resolved; one credit per debt).
- **`person_intent_drift` on a spicy-but-on-topic hot take** — a hot take is *playable*, not drift (seed bank §G, §E11). `person_intent_drift` only reads negative when L1 and L2 agree the move targets the *speaker*; a coherent spicy claim is `0`. A conflict here routes (`conflict_routed`), never penalizes.
- **Concurrent edits** — argument bodies are immutable after posting (soft-delete only — CLAUDE.md Supabase conventions); `contentHash` is stable, so a move is reconciled at most once per `promptVersion`. No concurrent-edit race exists for a posted move.
- **Offline / network failure** — the ledger is pure-TS with no network; the *packet* is fetched upstream (Edge Function, MCP-009). If the packet fetch fails the caller passes the mock stub and the ledger runs `l1_only` (first bullet). The ledger itself never makes a network call.
- **Permission-denied** — not applicable: the ledger model has no auth surface; it consumes already-fetched data. RLS lives on whatever future table persists results, which is out of scope.
- **Doctrine edge case — "can a hint push standing negative?"** — no. §6.1: a `scoreHint` may only add credit, open/size a debt, or raise a routing prompt. The only negatives come from the economy's reviewed evasion path and the negative-only `person_intent_drift` hygiene reading.
- **Doctrine edge case — "can heat / popularity earn a category?"** — no. None of the 14 categories takes engagement count, view count, or crowd reaction as an input. `uses_popularity_as_evidence` (a semantic binary) *raises evidence pressure on the move's claim*; it never earns the move points and never penalizes the crowd.

## 16. Risks

- **`scoreHints` integer range is assumed `0..3`.** MCP-001 §7 says "integers in a small bounded range" without fixing the ceiling. The implementation card must confirm the exact range against the seed bank before fixing `HINT_STEP`; if the range is wider, `HINT_STEP` shrinks so the per-hint cap stays ≤ the economy's `+0.25`. Flagged as an open item for the seed-bank owner.
- **`DeterministicMoveMetadata` shape depends on six not-yet-merged models.** LIFE-001 / META-001 / EV-001 / BR-001 / GAME-002 / GAME-003 / GAME-004 each own a slice of layer-1. If any of those models ship a different field shape, the implementation card adjusts `DeterministicMoveMetadata`. This is a coordination risk, not a doctrine risk.
- **EV-003 vs `OpenIssueDebt` vocabulary.** EV-003's evidence-debt tracker may model debt slightly differently from the economy's `OpenIssueDebt`. This design assumes one debt vocabulary (`OpenIssueDebt`). If EV-003 diverges, the implementation card needs a small adapter — surfaced here so it is not a surprise.
- **`gameCopy.toPlainLanguage` is test-enforced.** Adding new `feedbackCode`s without adding mappings is an existing test failure (`test-discipline`, doctrine constraint 10). The implementation card must extend `ALL_INTERNAL_CODES` and the mapping in the same change.
- **Scope creep toward a winner surface.** The ledger produces per-move category readings; it is tempting for a later card to sum them into a "who is winning" number. That is a doctrine violation (constraint 2) and is RULE-007 / GAME-007 territory. The `LedgerResult` shape deliberately has no aggregate field and the ban-list test asserts it — but reviewers of future cards must hold this line.
- **No migration / no operator deploy** — pure-TS model, in-memory like the economy. Low operational risk.

## 18. Doctrine self-check
