## Goal (one paragraph)

Give a signed-in user a lightweight way to inspect and adjust their own
profile and UI preferences **without leaving the game** — opened as a `Modal`
bottom sheet from a single header affordance, closeable with one tap. The
popout surfaces nine fields (display name, avatar preview, contact email,
notification stub, default room entry, visual density, colour accessibility
mode, reduce motion, default side label). Doctrine constraints that shape the
design: it exposes **no role / permission control** (no escalation surface — a
non-admin must not see anything that could change their role; `cdiscourse-doctrine`
§ secrets / scope guards), **no hidden auth fields** (no token, no service-role,
no email-change credential capture — that is PR-004), and it has **zero effect
on truth, score, heat, or validation gates** (`cdiscourse-doctrine` §1 — a
preference is cosmetic; it never makes a point stronger or a post blocked).
Preferences persist **device-locally via `AsyncStorage`** to keep PR-001
migration-free; only the display name — which is real account data — writes to
Supabase, reusing the existing `updateOwnDisplayName` path. The visual-density
preference drives VG-004's already-shipped `TimelineDensityMode`; the
reduce-motion preference is a user **override** that composes on top of the OS
`AccessibilityInfo` read.

## Data model

**No new SQL. No migration. No Supabase column. No Edge Function.**

One new pure-TypeScript preference shape, persisted as a single JSON blob in
`AsyncStorage`.

```ts
// src/features/preferences/userPreferencesModel.ts  (pure TS — no React, no Supabase)

/** Visual density — drives VG-004's TimelineDensityMode 1:1. */
export type DensityPreference = 'compact' | 'normal' | 'expanded';

/** Reduce-motion override that composes on top of the OS setting. */
export type ReduceMotionPreference = 'system' | 'on' | 'off';

/**
 * Colour accessibility mode. v1: only 'default' and 'high_contrast' have a
 * real effect (see §"Color accessibility mode — honest scope"). The three
 * colour-blind simulation values are PERSISTED but inert in v1 — no
 * palette-swap layer exists to drive them yet (a separate VG card).
 */
export type ColorAccessibilityMode =
  | 'default'
  | 'high_contrast'
  | 'protanopia'      // persisted, v1 no-op
  | 'deuteranopia'    // persisted, v1 no-op
  | 'tritanopia';     // persisted, v1 no-op

/** Which mode a room opens in. Mirrors the existing Observer-first default. */
export type DefaultRoomEntryPreference = 'observe' | 'last_used';

/** Default side label shown in the composer / side rail. Cosmetic only. */
export type DefaultSideLabelPreference = 'for_against' | 'side_a_b';

/** The full per-user preference blob. Versioned for forward-safe migration. */
export interface UserPreferences {
  /** Schema version — bump on any breaking shape change. */
  schemaVersion: 1;
  density: DensityPreference;
  reduceMotion: ReduceMotionPreference;
  colorMode: ColorAccessibilityMode;
  defaultRoomEntry: DefaultRoomEntryPreference;
  defaultSideLabel: DefaultSideLabelPreference;
  /** Honest stub — persisted, drives nothing in v1 (no push notifications). */
  notificationsOptInStub: boolean;
}

/** The canonical defaults. resolveNodeGapPx(undefined) already returns the
 *  VG-004 'normal' value (44px), so 'normal' is the density default. */
export const DEFAULT_USER_PREFERENCES: UserPreferences = {
  schemaVersion: 1,
  density: 'normal',
  reduceMotion: 'system',
  colorMode: 'default',
  defaultRoomEntry: 'observe',
  defaultSideLabel: 'for_against',
  notificationsOptInStub: false,
};
```

**Display name and contact email are NOT in this blob.** Display name is
account data (`profiles.display_name`, written via `updateOwnDisplayName`).
Contact email is read-only here, from `fetchCurrentAuthUser()`. Putting either
in `AsyncStorage` would let the device copy drift from Supabase — they are
deliberately excluded.

### Storage (`preferencesStorage.ts`)

```ts
export function loadUserPreferences(userId: string | null): Promise<UserPreferences>;
export function saveUserPreferences(userId: string | null, prefs: UserPreferences): Promise<void>;
```
Both never throw — `load` returns `DEFAULT_USER_PREFERENCES` on any failure
(missing key, corrupt JSON, storage error), `save` swallows errors. `userId ===
null` → keyed under an anon key, identical to `sessionStorage`'s anon fallback.

### Hook (`useUserPreferences.ts`)

```ts
interface UseUserPreferencesResult {
  preferences: UserPreferences;          // DEFAULT_USER_PREFERENCES until load resolves
  loading: boolean;
  osReduceMotion: boolean;               // live OS value
  effectiveReduceMotion: boolean;        // resolveEffectiveReduceMotion(pref, os)
  updatePreference: <K extends keyof UserPreferences>(
    key: K, value: UserPreferences[K],
  ) => void;                             // optimistic; fires save
}
export function useUserPreferences(userId: string | null): UseUserPreferencesResult;
```

## Color accessibility mode — honest scope

`accessibility-targets` names a 5-value enum and says the token system should
"swap the colour family but preserve shape/stroke." **That token-swap layer
does not exist** (D4). PR-001 will not pretend it does.

- **`default`** — today's palette. No change.
- **`high_contrast`** — has a **real, small, deliverable effect**: it raises
  border/stroke weight and forces the highest-contrast text token pair on the
  preferences popout itself (and any surface PR-001 newly touches). It is
  *persisted* so a future VG card can widen its reach across the timeline.
  PR-001 does **not** repaint the whole app — that would be scope creep — but
  the preference is real and the popout demonstrably honours it.
- **`protanopia` / `deuteranopia` / `tritanopia`** — **persisted but inert in
  v1.** Selecting one stores the value and shows an honest helper line:
  *"Colour-blind palettes are coming. Your choice is saved and will apply when
  the timeline palette update ships."* No fake daltonization, no broken promise.
  The timeline already satisfies colour-independence via shape/stroke
  (`timeline-grammar`), so a colour-blind user is **not** blocked today — the
  helper says exactly that.

This keeps the design honest: a real toggle, a real (bounded) effect, and
explicit "coming later" copy for the part that genuinely isn't built.

## Persistence

- **Mechanism:** `AsyncStorage`, one JSON blob per user, keyed
  `cdiscourse:preferences:<userId>` (anon key when signed out — though the
  popout is only reachable while signed in).
- **Why local, not Supabase:** preferences are device-scoped UI state. Putting
  them in `profiles` would require a **migration** (new columns or a new
  `user_preferences` table + RLS) and an operator deploy. The card explicitly
  prefers a no-migration v1, and nothing about density/motion/colour needs to
  follow a user across devices in v1. `AsyncStorage` is the established
  local-persistence primitive (`sessionStorage.ts`).
- **Forward-safe:** the blob carries `schemaVersion: 1`. `mergeWithDefaults`
  defensively reconstructs the blob from whatever is on disk — a future schema
  bump or a v2 "sync preferences to Supabase" card can read the local blob and
  migrate it without data loss.
- **No secrets:** the blob contains only enums and one boolean. No token, no
  email, no auth material ever touches `AsyncStorage` here.

**Migration required? NO.** Pure code + `AsyncStorage`. No `supabase db push`,
no Edge Function, no new table.

## Edge cases

- **Signed out / no `userId`:** the preferences trigger is only rendered inside
  `MainAppShell` (signed-in shell), so the popout is unreachable when signed
  out. The hook still tolerates `userId === null` (anon key) defensively.
- **`AsyncStorage` read failure / corrupt JSON:** `loadUserPreferences`
  catches and returns `DEFAULT_USER_PREFERENCES`. `mergeWithDefaults` handles a
  partial/garbage object (missing keys filled, wrong-typed values replaced,
  unknown keys dropped). The popout always has a valid `UserPreferences`.
- **`AsyncStorage` write failure:** `saveUserPreferences` swallows the error;
  the in-memory state still reflects the change for the session. Non-fatal,
  mirrors `sessionStorage` behaviour.
- **Display-name save failure (network / RLS):** the display-name field shows
  the `saveError` from `useAccountProfile` inline; the rest of the popout is
  unaffected. The user can retry or close.
- **Display-name empty / whitespace-only:** Save is disabled when
  `draftName.trim()` is empty (same rule as `AccountScreen`). Max length 60
  (mirrors `AccountScreen`'s `maxLength={60}`).
- **OS reduce-motion changes while the popout is open:** the hook subscribes to
  `AccessibilityInfo`'s `reduceMotionChanged` event; `effectiveReduceMotion`
  recomputes. If the user's override is `on`/`off`, the OS change is ignored
  (override wins) — the "System" option's helper line shows the current OS
  value so the user understands what `System` would do.
- **Density change mid-session:** changing density re-runs
  `buildArgumentTimelineMap` next render; node `x` positions shift but node
  **ids** and ordinal order are unchanged (the active node stays active). No
  jump-to-wrong-message.
- **Concurrent edits (display name) across two devices:** last-write-wins at
  the `profiles` row — acceptable; display name is low-stakes and the existing
  `AccountScreen` already has this property. Preferences blob is device-local
  so there is no cross-device conflict by construction.
- **Offline:** preference writes go to `AsyncStorage` — they work fully
  offline. Only the display-name write needs the network; its failure path is
  the existing `useAccountProfile` error surface.
- **Doctrine edge — "can a preference change my score / unblock a post?"** No.
  Preferences are read only by render-layer code (timeline gap, motion, colour,
  labels). No preference is an input to `argumentScoreModel`, the
  Constitution engine, `antiAmplification`, or any validation gate. This is
  asserted by the doctrine test (source scan: the preferences module must not
  import the score/engine modules).
- **Doctrine edge — "can the popout escalate my role?"** No. The popout never
  renders, reads-for-write, or references `profiles.role`. The doctrine test
  scans `PreferencesPopout.tsx` + `preferencesCopy.ts` for the tokens `role`,

  `{compact:28, normal:44, expanded:64}` — `normal` is `44`, not the historical
  `28`. `resolveNodeGapPx(undefined)` already returns `44`, so if VG-004's wire
  is *not* yet done, soldering it (even with `density` defaulted) changes the
  default board from 28→44px. This is **intended** per VG-004's operator
  decision, but the implementer should be aware that the visible board spacing
  changes the moment the wire lands. The `preferencesModelVG004Wire` test pins
  that the *preference default* and the *un-wired default* agree (both 44).
- **Default side-label resolver may not exist.** If `src/features/arguments/`
  has no central side-label string source, field 9 ships persist-only with
  honest copy (see §Field spec caveat). The implementer must verify and not
  fake a wire.

  new dep). `building-native-ui` notes `Switch` has built-in haptics on iOS —

  This is intentional scope-limiting, not a bug — but it means "Reduce motion:

## Browser verification checklist

PR-001 is a visible-UI card and gets a real Expo-web verification before
merge. Concrete things a human clicks/observes:

1. **Open:** sign in → a gear / "Preferences" control is visible in the app
   header's right slot (`preferences-trigger`). Click it → a bottom-sheet
   `Modal` slides up (`preferences-popout`).
2. **Close:** click the × (`preferences-close`) → the sheet dismisses and the
   game screen behind it is untouched (no navigation occurred).
3. **Display name:** edit `pref-display-name-input`, click `pref-display-name-save`
   → a "saved" confirmation shows; reopen the Account tab → the new name is
   reflected there too (same `profiles` row).
4. **Avatar preview:** `pref-field-avatar` shows a coloured initials circle; it
   changes initials when the display name changes. No upload button is present.
5. **Contact email:** `pref-contact-email-value` shows the signed-in email;
   `pref-contact-email-update` is a clearly-labelled link, not an editable
   field — clicking it routes to the Account screen (or shows the interim
   note), it does NOT capture a new email inline.
6. **Notification stub:** toggle `pref-notifications-switch` → the visible
   helper copy makes clear notifications aren't available yet; the toggle state
   persists.
7. **Visual density:** set `pref-field-density` to `Compact` → reopen a debate
   room in Timeline mode → nodes sit visibly closer together; set `Expanded` →
   nodes spread out. Node selection/active state is unchanged by the switch.
8. **Colour accessibility:** set `pref-field-color-mode` to `High contrast` →
   the popout's own borders/text visibly sharpen; set a colour-blind option →
   an honest "coming later" helper line appears.
9. **Reduce motion:** set `pref-field-reduce-motion` to `On` → close and
   reopen the popout → it snaps open instead of sliding. Set to `System` → the
   helper line reflects the current OS reduce-motion value.
10. **Default room entry / default side label:** change each segmented control;
    the selection is highlighted with a visible active state.
11. **Persistence across reopen:** change several fields, close the popout,
    reload the Expo-web tab, sign back in, reopen the popout → every changed
    preference is still selected (display name from Supabase, the rest from
    `AsyncStorage`).
12. **No console errors:** the DevTools console shows no 404s, no warnings, no
    raw validation/internal codes printed.
13. **Keyboard (web):** `Tab` reaches the trigger, the close button, every
    field control, and the display-name input in reading order; `Enter`/`Space`
    activates them; `Esc` closes the `Modal`.
