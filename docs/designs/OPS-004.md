### §1.2 — Class 3 sub-check (DROP COLUMN / DROP TABLE ordering)

The existing Class 3 row's "Syntactic markers" cell ends with
`(d) A drop policy if exists followed by a create policy with the same
name — confirm the create block names the same target table as the
drop. **Resolution:** reorder the statements so every referenced object
exists before it is referenced.`

OPS-004 inserts a new marker `(e)` **before** the `**Resolution:**`
sentence. The new marker reads exactly:

```
(e) Any `DROP COLUMN` or `DROP TABLE` statement that is not preceded by `DROP` of every policy, index, view, trigger, or function that references the column or table. For every `DROP COLUMN` or `DROP TABLE` statement, grep the entire migration set for the column or table name and verify no `CREATE POLICY`, `CREATE INDEX`, `CREATE VIEW`, `CREATE TRIGGER`, or `CREATE FUNCTION` body references it earlier in the same migration or in any prior migration whose object would be affected. If a reference exists, the dependent object must be `DROPPED` before the referenced object. The PR-004 motivating example was `ALTER TABLE public.profiles DROP COLUMN avatar_path, …` failing with `SQLSTATE 2BP01 cannot drop column avatar_path … because other objects depend on it (policy "profiles: users update own — narrow" depends on column avatar_path)` — the fix was to swap the migration's steps 2 and 3 so the narrowed policy was dropped before the columns it referenced.
```

The existing `**Resolution:**` sentence remains the cell's terminal
text and is **not edited**; the OPS-004 sub-check is a marker addition,
not a resolution rewrite.

### §1.3 — Class 4 sub-check (storage schema COMMENT ownership)

The existing Class 4 row's "Syntactic markers" cell ends with
`(e) Trigger functions calling public.is_admin(auth.uid()) or similar
helpers — confirm the helper was created by an earlier applied
migration. **Resolution:** name the prior migration that provides the
dependency in a SQL comment above the dependent statement, or add the
CREATE EXTENSION IF NOT EXISTS / CREATE OR REPLACE FUNCTION block to
this migration.`

OPS-004 inserts a new marker `(f)` **before** the `**Resolution:**`
sentence. The new marker reads exactly:

```
(f) Any `COMMENT ON {POLICY,TABLE,COLUMN,FUNCTION,TYPE,DOMAIN,VIEW,INDEX,SEQUENCE,TRIGGER,RULE} ON storage.*` statement. The Supabase `storage` schema's `objects` table is owned by `supabase_storage_admin`, not by the standard migration runner role. `CREATE POLICY ... ON storage.objects` is permitted to non-owners via Supabase's storage RLS setup, but `COMMENT ON {…} ON storage.*` requires ownership the standard migration role lacks. Surfaces as `SQLSTATE 42501 must be owner of relation objects` at apply time. The PR-003 motivating example was `COMMENT ON POLICY "profile-avatars: anyone can read" ON storage.objects IS '...'` failing on statement 12. Resolution options, ranked: (i) **preferred** — omit the COMMENT statement entirely (the policy body and inline SQL `--` comments serve as the policy's documentation); (ii) wrap in a privilege-tolerant `DO $$ BEGIN EXECUTE $sql$ COMMENT ON … $sql$; EXCEPTION WHEN insufficient_privilege THEN NULL; END $$;` block; (iii) run the migration as `supabase_storage_admin` — **not available** in standard Supabase migrations.
```

The existing `**Resolution:**` sentence remains the cell's terminal
text and is **not edited**; the OPS-004 sub-check is a marker addition
inside the existing cell.

### §1.4 — Why "inside the existing cell" and not "as a new class"

A fifth class would require renumbering the existing class 1–4 cells
across the table, the table header, the design references in
`docs/designs/OPS-001.md`, the OPS-001 review at `docs/reviews/OPS-001.md`,
the cross-references at `CLAUDE.md:165–171` / `docs/core/agent-charters.md:62–68`
/ `docs/core/agent-workflow.md:40`, and the four-class assertion at
`docs/reviews/OPS-001.md:39` ("The 4 classes are exactly: …"). The
sub-check shape inside the existing cells respects the OPS-001
acceptance-criterion 3 ("the four-class table … with all four classes
present in their numbered order") without modifying the row count, and
nests both new sub-checks inside the class they semantically belong to:
PR-004 DROP COLUMN ordering IS a statement-ordering dependency (Class 3);
PR-003 storage COMMENT ownership IS a function/role/extension dependency
(Class 4, marker subset (c) "role-or-privilege gap").

### §2.3 — Verbatim replacement text for lines 49–53

The implementer pastes the block below verbatim, replacing the
existing 5-line prose paragraph at lines 49–53:

```markdown
   The rename is a local-only operation (no remote refs, no force
   push) and is safe to run on a fresh worktree before any commit.

   If `git branch -m` reports `fatal: a branch named
   'feat/<code>-<slug>' already exists`, the canonical branch is held
   somewhere else in the local repo. **Do not force.** Recover with the
   sequence below; this is the post-PR-004 documented path:

   ```
   # 1. Identify which worktree holds the canonical branch.
   git worktree list | grep "feat/<code>-<slug>"
   #   → expect 1 result; the path is a sibling worktree under
   #     .claude/worktrees/agent-<other-hash>

   # 2. Inspect the holder. If it is a designer's post-design worktree
   #    (no uncommitted work; HEAD is the design commit), the canonical
   #    branch is safe to claim. If it is an active session, STOP and
   #    surface to the operator.
   git -C ".claude/worktrees/agent-<other-hash>" status -sb
   git -C ".claude/worktrees/agent-<other-hash>" log -1 --format="%h %s"

   # 3. When the holder is a clean designer worktree, switch this
   #    worktree onto the canonical branch and clean up the auto-branch.
   #    --ignore-other-worktrees is the documented git override for the
   #    "already used by worktree at" guard rail.
   git switch --ignore-other-worktrees feat/<code>-<slug>
   git branch -d worktree-agent-<hash>     # the original auto-branch
   git branch --show-current               # must now be feat/<code>-<slug>
   ```

   When the holder is NOT a clean designer worktree (uncommitted work,
   active session, or HEAD diverged from the design commit), STOP and
   surface to the operator. Do not force the override; loss-of-work
   risk is real when the holder has uncommitted changes the
   `--ignore-other-worktrees` override would silently abandon.

   The PR-004 incident (`docs/core/known-blockers.md` § "OPS-002
   Stale-Worktree-Branch-Claim (2026-05-25)") is the motivating
   example; OPS-004 codifies the recovery sequence so future
   implementers do not re-discover it.
```

### §2.4 — Why the recovery path is documented but not automated

The `--ignore-other-worktrees` override is safe only when the holder
worktree has no uncommitted work, and "no uncommitted work" is an
operator-judgement check (the implementer subagent cannot reliably
distinguish a clean post-design state from a paused mid-session state
without operator confirmation). OPS-004 documents the *sequence* so
the implementer knows what to run; it does NOT script an automatic
recovery (which would risk silent loss-of-work). The STOP-and-surface
escape hatch is preserved for every case the sequence does not cover.

## §3 — spawn-card script regex correction

### §3.1 — The bug

`.claude/scripts/spawn-card.ps1` line 74:

```
$slug = ($issue.title -replace "^$Code\s*-\s*", '' -replace "[^a-zA-Z0-9]+", '-' -replace '^-+|-+$', '').ToLower()
```

The leading-character-class `^$Code\s*-\s*` assumes a dash separator
between the issue code and the title. Modern GitHub issue titles in
this repo use a colon (`OPS-002:`), so the strip is a no-op for current
titles and the resulting slug duplicates the code prefix
(`feat/OPS-002-ops-002-pipeline-…` instead of
`feat/OPS-002-pipeline-…`). The bash twin at
`.claude/scripts/spawn-card.sh` line 112 has the identical bug in
sed-shape (`s/^${CODE}[[:space:]]*-[[:space:]]*//`).

OPS-002 §7 flagged this and pinned the current behaviour in
`__tests__/spawnCardBranchName.test.ts` Group 1 Test 3 so a silent
"fix" later would not break the four shipped cards' branch names.
OPS-004 makes the fix conscious: the regex is corrected, the test is
updated to assert the corrected behaviour, and the shipped cards'
branch names remain in PR history unchanged (history is immutable
post-merge; the regex change affects only *future* spawn-card invocations).

### §3.2 — The fix

Single-character-class change in each script. Each script gains a `:`
inside an existing character bracket. Net delta per script: **+1
character**.

**`.claude/scripts/spawn-card.ps1` line 74.** Current:

```powershell
$slug = ($issue.title -replace "^$Code\s*-\s*", '' -replace "[^a-zA-Z0-9]+", '-' -replace '^-+|-+$', '').ToLower()
```

Corrected (the change is `-` → `[-:]` inside the first `-replace`'s
character expression):

```powershell
$slug = ($issue.title -replace "^$Code\s*[-:]\s*", '' -replace "[^a-zA-Z0-9]+", '-' -replace '^-+|-+$', '').ToLower()
```

The PowerShell `[-:]` character class matches either a literal dash
or a literal colon. Dash and colon are the only two separators
observed in this repo's issue titles (legacy `OPS-002 - title` and
modern `OPS-002: title`). Future separators (semicolon, em-dash, etc.)
are not anticipated; if they arise, a follow-up OPS card extends the
class.

**`.claude/scripts/spawn-card.sh` line 112.** Current:

```bash
  | sed -E "s/^${CODE}[[:space:]]*-[[:space:]]*//" \
```

Corrected (the change is `-` → `[-:]` inside the sed expression's
character expression):

```bash
  | sed -E "s/^${CODE}[[:space:]]*[-:][[:space:]]*//" \
```

The POSIX bracketed character expression `[-:]` matches either a
literal dash or a literal colon. The expression remains within a
`-E` (extended regex) sed call, matching the existing shape exactly.

### §3.3 — No other edit

The two regex changes above are the **only** edits to the spawn-card
scripts. The slug-collapse `[^a-zA-Z0-9]+ → -` regex is untouched (it
already collapses colons inside the title body to dashes). The
40-character truncation, the lowercasing, and the
leading/trailing-dash trim are untouched. The Phase-switch block, the
Project-item-lookup block, the context-file write block, and the
NEXT STEP printout are untouched.

The reviewer of the OPS-004 implementer diff confirms by `git diff
main..HEAD --shortstat -- .claude/scripts/spawn-card.ps1
.claude/scripts/spawn-card.sh` that the per-file delta is **+1
character / -1 character** for each script (the dash character
becomes the two-character class `[-:]`, net +1 char per script;
git's line-diff sees this as one modified line per script).

### §3.4 — Symmetry requirement

The two scripts are functional twins; their behaviour must remain
symmetric so the orchestrator can use either platform interchangeably.
OPS-004 keeps the symmetry by applying the same single-character-class
change to both. A test asserting twin-symmetry is **out of scope** for
OPS-004 (the existing test in `__tests__/spawnCardBranchName.test.ts`
mirrors only the PS1 regex per the OPS-002 §5.4 documented triple-parity
exclusion); the bash twin is a manual operator fallback and a future
regression there would surface on the operator's first attempted use.

### §5.4 — Why exactly 3 new tests, not more

The OPS-004 contract surface is three distinct charter / template

future regression where one extension is deleted; adding more tests

existing Group 2 / Group 3 contract pattern is the precedent: one
test per extension, asserting a unique substring or signal phrase
