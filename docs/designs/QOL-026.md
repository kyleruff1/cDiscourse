# QOL-026 — Fix Admin All Arguments loader (selected_tag_codes + attached_evidence)

**Status:** Design draft
**Epic:** UI/UX Behavior Repair Wave (QOL — admin data layer)
**Release:** Behavior Repair Wave (see `docs/project-audits/2026-05-19-ui-ux-behavior-repair-design-map.md`)
**Issue:** https://github.com/kyleruff1/cDiscourse/issues/137

## Goal (one paragraph)

The Admin → All Arguments tab is fully broken: it shows "Could not load
argument activity. Check admin access and try again." with detail
`loadAdminArguments failed: column arguments.selected_tag_codes does not exist`.
The loader at `src/features/admin/adminArgumentsApi.ts` requests **two columns
that have never existed** on `public.arguments`: `selected_tag_codes` and
`attached_evidence`. Both are listed in the same `.select()` call (lines 83–84).
PostgREST reports unknown columns one at a time, so the page would still fail on
`arguments.attached_evidence does not exist` even after the first is fixed. The
issue's acceptance criterion is "Admin All Arguments loads and renders rows";
fixing only one column leaves the page broken. This card therefore fixes **both
columns in the same `.select()`** — same root cause (loader selects columns that
do not exist), same file, no sibling card. The two columns get **different
treatments**: `selected_tag_codes` is replaced with a correct PostgREST nested
embed of `argument_tags(tag_code)` (the join table exists and admins have RLS
SELECT), while `attached_evidence` is **dropped** because there is no evidence
table or column anywhere in the schema to join to. No schema migration, no
service-role, no Edge Function change. The fix respects doctrine trivially: tags
are advisory category/qualifier signals, never truth verdicts, and the read
stays inside RLS using the shared anon client.

## Cannot proceed?

No conflict. The card is well-specified once widened (per coordinator decision
below). Both fixes are provably safe — see "Risks" and "Doctrine self-check".

## Scope decision (COORDINATOR-CONFIRMED)

The coordinator widened QOL-026 to fix **both** non-existent columns in the same
`.select()`. This is **required, not scope creep**: the acceptance criterion is
"Admin All Arguments loads and renders rows", and fixing only `selected_tag_codes`
leaves the page broken on `attached_evidence`. Same root cause, same file, no
sibling card.

Independently verified during design:

- `attached_evidence` is **NOT** in any migration. `grep -rn attached_evidence
  supabase/migrations/` returns zero matches.
- There is **NO evidence table anywhere** in `supabase/migrations/`. A grep for
  `create table .*evidence` (case-insensitive) returns nothing.
- `AdminArgumentsTab.tsx:325` renders an `evidence` badge gated on `r.hasEvidence`,
  which is derived in the loader (`adminArgumentsApi.ts:112`) from
  `attached_evidence`. Since that column never existed, the badge has **never
  shown real data** — `attached_evidence` resolved to `undefined`, so
  `hasEvidence` was always `false` (when the query did not outright fail).

The two columns therefore get **different treatments**:

| Column                | Treatment | Why |
|-----------------------|-----------|-----|
| `selected_tag_codes`  | **Option B — correct join** | The `public.argument_tags` join table exists, has the right FK, and admins have RLS SELECT. Real tag data should be restored. |
| `attached_evidence`   | **Option A — drop**         | There is no evidence table or column to join to. Nothing to embed. Remove it cleanly. |

### `selected_tag_codes` → Option B (correct nested join)

Replace the bad scalar column with a PostgREST nested embed
`argument_tags(tag_code)`, mapped into `AdminArgumentRow.selectedTagCodes`.
Two concrete facts make this the right call:

1. **`AdminArgumentsTab.tsx` already consumes `selectedTagCodes`.** Lines 272
   and 279 feed `r.selectedTagCodes` into `deriveMessageCategory` and
   `derivePrimaryQualifier` (`src/features/arguments/messageQualifiers.ts`,
   which declares `selectedTagCodes?: string[] | null` on its input at L70).
   Today that input is always the broken `null` from a failed query — the
   category/qualifier badges silently lose a real signal. Dropping the field
   would permanently delete that signal; Option B restores it with real data.
2. **The join is RLS-safe and cheap.** `public.argument_tags` has a SELECT
   policy that mirrors argument visibility and grants admins access via
   `is_moderator_or_admin()` (`20260516000002_rls_policies.sql:244`). It also
   has an index on `argument_id` (`20260516000005_...:92`), so the embed for a
   capped 50–500 row page is inexpensive.

### `attached_evidence` → Option A (drop)

There is no evidence table or column to join to, so there is **nothing to embed
and nothing to correct**. The fix is to remove `'attached_evidence'` from the
`.select()` list, drop `attached_evidence` from `RawArgumentRow`, and **remove
`hasEvidence` and the evidence badge entirely**.

Rationale for dropping `hasEvidence` rather than hardcoding `hasEvidence: false`:

- The badge never showed real data — keeping a permanently-`false` field and a
  badge that can never render is dead code that misleads future readers.
- A future real Evidence card (the `EV-*` series — e.g. `EV-001` Evidence
  object model) will reintroduce evidence as a first-class concept with a real
  table. When it does, it should add `hasEvidence` (or a richer evidence
  summary) back **deliberately**, wired to the real source. Leaving a fake
  `hasEvidence: false` here would invite that card to "just flip it on" against
  a column that still does not exist.
- The change stays minimal and self-contained: one field removed from a type,
  one badge line removed from the tab, one style entry optionally removed.

The implementer **must not** hardcode `hasEvidence: false`. Drop it.

## Data model

**No new data model. No migration.** Existing schema only.

`public.argument_tags` (from `20260516000001_initial_schema.sql:223`):

```sql
CREATE TABLE public.argument_tags (
  argument_id uuid NOT NULL REFERENCES public.arguments(id) ON DELETE CASCADE,
  tag_code    text NOT NULL REFERENCES public.tag_definitions(code) ON DELETE RESTRICT,
  created_by  uuid NOT NULL REFERENCES public.profiles(id) ON DELETE RESTRICT,
  created_at  timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (argument_id, tag_code)
);
```

The loader needs only `tag_code`. The FK `argument_tags.argument_id ->
arguments.id` is what lets PostgREST resolve the embed `argument_tags(tag_code)`.

There is **no evidence table**. `attached_evidence` corresponds to nothing in the
schema. (Note: `composerSubmit.ts` builds an `attached_evidence` array as part of
the **`submit-argument` Edge Function request payload** — that is a transient
request field consumed by the Edge Function, NOT a persisted column on
`public.arguments`. The admin loader cannot and should not select it.)

Existing TypeScript shapes:

```ts

```

### New mapper helper

```ts
/** Flattens the embedded argument_tags relation into a plain tag-code array.
 *  Returns null when no tag rows came back (keeps AdminArgumentRow's
 *  string[] | null contract — null = "no tag data"). */
export function asTagCodes(j: RawArgumentRow['argument_tags']): string[] | null {
  if (!j || !Array.isArray(j) || j.length === 0) return null;
  return j
    .map((t) => (t && typeof t.tag_code === 'string' ? t.tag_code : null))
    .filter((c): c is string => c !== null);
}
```

Decision: return `null` (not `[]`) for the empty case, so the public
`AdminArgumentRow.selectedTagCodes: string[] | null` contract is unchanged and
the qualifier derivers (which accept `string[] | null`) behave exactly as they
did when the column was assumed-null. `[]` would be equally valid for the
derivers; `null` is chosen purely to keep the externally-visible shape
byte-identical to the pre-fix declared type. Export the helper so it has a
direct unit test (test-discipline: every pure helper gets a direct test).

### RLS / policy text (UNCHANGED — referenced for proof, not modified)

`argument_tags` SELECT policy already grants admins read access
(`20260516000002_rls_policies.sql:244`):

```sql
CREATE POLICY "argument_tags: select with argument"
ON public.argument_tags FOR SELECT TO authenticated
USING (EXISTS (
  SELECT 1 FROM public.arguments a
  WHERE a.id = argument_id
    AND (a.author_id = auth.uid()
         OR is_moderator_or_admin()
         OR (a.status = 'posted' AND EXISTS (
              SELECT 1 FROM public.debates d
              WHERE d.id = a.debate_id AND d.status IN ('open','locked'))))));
```

No new policy. No migration.

## Edge cases

- **Argument with zero tags** — embed returns `[]` (or `null`); `asTagCodes`
  returns `null`; `deriveMessageCategory` / `derivePrimaryQualifier` already
  handle `null` (input type is `string[] | null`). No badge regression.
- **Argument with multiple tags** — `asTagCodes` returns all codes; the derivers
  already iterate the array (`messageQualifiers.ts` uses `.some(...)` /
  `.includes(...)`). The category badge reflects the first matching signal as
  it does today; no ordering guarantee is added or needed.
- **Malformed embed row** (a `tag_code` that is not a string) — `asTagCodes`
  filters it out rather than crashing the whole page. Defensive, cheap.
- **Evidence badge removal** — no argument row will lose visible data: the
  badge has been showing nothing real since `attached_evidence` never resolved.
  Removing it is a no-op for the user, a cleanup for the code.
- **Non-admin caller** — RLS on both `arguments` and `argument_tags` returns
  zero rows; `loadAdminArguments` returns `[]`. The tab is also gated by
  `AdminScreen` so a non-admin never mounts it. No code path leaks tags.
- **`SUPABASE_CONFIGURED === false`** — loader returns `[]` early (existing
  guard, unchanged). The embed never executes.
- **Query genuinely fails** (network, RLS misconfig) — existing
  `throw new Error('loadAdminArguments failed: ...')` path is preserved; the
  tab still renders its accessible error state. The fix removes the *specific*
  `column ... does not exist` failures, not the generic error handling.
- **Soft-deleted arguments** — `includeDeleted` filter is unchanged; tags of a
  hidden argument are simply never fetched because the parent row is filtered.
- **Doctrine edge case — "do tags become a verdict?"** — No. `tag_code` values
  are advisory category/qualifier signals consumed by `messageQualifiers.ts`,
  which (by an existing ban-list test) contains zero verdict tokens. Surfacing
  real tags cannot introduce a truth label; it only makes an existing advisory
  badge accurate.
- **PostgREST relation ambiguity** — `arguments` has exactly one FK from
  `argument_tags` (`argument_tags.argument_id`), so `argument_tags(tag_code)`
  resolves unambiguously. No `!hint` disambiguator needed. (If a future
  migration adds a second FK between these tables, the embed would need
  `argument_tags!argument_tags_argument_id_fkey(tag_code)` — out of scope now.)

## Risks

- **PostgREST embed resolution.** If the project's PostgREST cannot resolve
  `argument_tags(tag_code)` (e.g. the FK metadata is stale in the schema cache),
  the query errors. Mitigation: the FK `argument_tags_argument_id_fkey` exists
  in the initial migration, so a correctly-migrated database resolves it. If the
  operator sees a `could not find a relationship` error after deploy, the
  fallback is to drop the embed (same treatment applied to `attached_evidence`)
  and set `selectedTagCodes: null` — see Fallback below. This is a deploy-time
  observable, not a code risk.
- **Tests that read source text.** `adminArguments.test.ts` asserts on literal
  substrings. The new `.select()` must keep `argument_tags(tag_code)` as a
  contiguous literal (do not split it across template concatenation) or the
  contract test will not match. Likewise the `not.toContain` assertions for
  `selected_tag_codes` / `attached_evidence` / `hasEvidence` will fail if any of
  those strings linger anywhere in the file — the implementer must remove every
  occurrence in `adminArgumentsApi.ts` (select list, `RawArgumentRow`, `.map`).
- **`hasEvidence` removal ripple.** `hasEvidence` is consumed **only** by
  `AdminArgumentsTab.tsx:325`. (The many other `hasEvidence` matches in the repo
  belong to unrelated models — `argumentScoreModel`, `argumentVisualGrammar`,
  `claimStanding`, `gameStatus`, `timelineNodeVisualModel`, etc. — and take
  their own inputs; they are NOT fed by `AdminArgumentRow`.) Removing
  `AdminArgumentRow.hasEvidence` therefore touches exactly one render line plus
  one type field. Verified by grep during design.
- **No live Supabase available.** The implementer cannot run `npx supabase
  start` to integration-test the embed. The contract tests cover query shape;
  the operator confirms the live load. This is acceptable per the card's "no
  live Supabase mutation" constraint.

### Fallback — drop the tag embed too (only if Option B's embed cannot resolve)

If, and only if, the operator confirms PostgREST cannot resolve the embed:
remove `'argument_tags(tag_code)'` from `.select()` with no replacement, drop
`argument_tags` from `RawArgumentRow`, and set `selectedTagCodes: null` in the
`.map(...)` result (keeping `AdminArgumentRow.selectedTagCodes: string[] | null`).
The badge derivers then behave exactly as they have while the page was broken —
category/qualifier still derive from `argument_type` / `disagreement_axis` /
`body`, just without the tag signal. This restores the page; it does not restore
the tag badge. Do not delete `selectedTagCodes` from `AdminArgumentRow`
(downstream derivers expect the property to exist as nullable). This is the
documented fallback, not the plan. The `attached_evidence` drop is unconditional
either way.
