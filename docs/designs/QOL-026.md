# QOL-026 — Fix Admin All Arguments selected_tag_codes load error

**Status:** Design draft
**Epic:** UI/UX Behavior Repair Wave (QOL — admin data layer)
**Release:** Behavior Repair Wave (see `docs/project-audits/2026-05-19-ui-ux-behavior-repair-design-map.md`)
**Issue:** https://github.com/kyleruff1/cDiscourse/issues/137

## Goal (one paragraph)

The Admin → All Arguments tab is fully broken: it shows "Could not load argument
activity. Check admin access and try again." with detail
`loadAdminArguments failed: column arguments.selected_tag_codes does not exist`.
The loader at `src/features/admin/adminArgumentsApi.ts` requests a column
(`selected_tag_codes`) that has never existed on `public.arguments` — tags live
in the separate join table `public.argument_tags (argument_id, tag_code)`. The
name collides with the composer's in-memory draft field `selectedTagCodes`,
which is draft state, not a persisted column. This card replaces the bad column
reference with a correct PostgREST nested embed of `argument_tags(tag_code)`,
maps the returned `{ tag_code }[]` into the existing `string[]` shape, and
leaves types and the tab's badge rendering honest. No schema migration, no
service-role, no Edge Function change — `public.argument_tags` already exists
and admin SELECT is already permitted by RLS via `is_moderator_or_admin()`. The
fix respects doctrine trivially: tags are advisory category/qualifier signals,
never truth verdicts, and the read stays inside RLS using the shared anon
client.

## Cannot proceed?

No conflict. The card is well-specified. Option B (correct join) is chosen and
is provably safe — see "Risks" and "Doctrine self-check" for why.

## Decision: Option B (correct nested join)

The coordinator recommended Option B unless a concrete reason makes it unsafe.
There is none. Two concrete facts make Option B the right call:

1. **`AdminArgumentsTab.tsx` already consumes `selectedTagCodes`.** Lines 272
   and 279 feed `r.selectedTagCodes` into `deriveMessageCategory` and
   `derivePrimaryQualifier` (`src/features/arguments/messageQualifiers.ts`,
   which declares `selectedTagCodes?: string[] | null` on its input at L70).
   Today that input is always the broken `null` from a failed query — the
   category/qualifier badges silently lose a real signal. Option A would
   permanently delete that signal; Option B restores it with real data.
2. **The join is RLS-safe and cheap.** `public.argument_tags` has a SELECT
   policy that mirrors argument visibility and grants admins access via
   `is_moderator_or_admin()` (`20260516000002_rls_policies.sql:244-260`). It
   also has an index on `argument_id` (`20260516000005_...:92`), so the embed
   for a capped 50–500 row page is inexpensive.

Option A (drop the field entirely) is documented below only as the fallback if
the implementer hits a PostgREST embed-resolution problem (see Risks). It is
**not** the chosen path.

## Data model

**No new data model. No migration.** Existing schema only.

`public.argument_tags` (from `20260516000001_initial_schema.sql:223`):

```sql
CREATE TABLE public.argument_tags (
  argument_id uuid NOT NULL REFERENCES public.arguments(id) ON DELETE CASCADE,
  tag_code    text NOT NULL REFERENCES public.tag_definitions(code) ON DELETE RESTRICT,
  created_by  uuid NOT NULL REFERENCES public.profiles(id) ON DELETE RESTRICT,
  created_at  timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (argument_id, tag_code)
);
```

The loader needs only `tag_code`. The FK `argument_tags.argument_id ->
arguments.id` is what lets PostgREST resolve the embed `argument_tags(tag_code)`.

Existing TypeScript shape stays the same to the rest of the app:

```ts
// src/features/admin/types.ts — AdminArgumentRow (UNCHANGED field)
selectedTagCodes: string[] | null;
```

The only internal change is the **raw row** shape inside the loader: instead of
a non-existent scalar column, it carries the embedded relation array.

```ts
// adminArgumentsApi.ts — RawArgumentRow (CHANGED field)
// before: selected_tag_codes: string[] | null;
// after:
argument_tags: { tag_code: string }[] | null;
```

## API / interface contracts

### Loader — `loadAdminArguments` (signature UNCHANGED)

### PostgREST select string (the fix)

The `.select(...)` argument changes from:

```
'id','debate_id','author_id','argument_type','side','body','status',
'created_at','updated_at','disagreement_axis','selected_tag_codes',
'target_excerpt','attached_evidence','server_validation','is_deleted',
'debates(title)','profiles(display_name)'
```

to (only the one token swapped):

```
'id','debate_id','author_id','argument_type','side','body','status',
'created_at','updated_at','disagreement_axis','argument_tags(tag_code)',
'target_excerpt','attached_evidence','server_validation','is_deleted',
'debates(title)','profiles(display_name)'
```

PostgREST returns the embed as an array of objects keyed by the selected
columns: `argument_tags: [{ tag_code: 'fact_disagreement' }, ...]` (or `[]`
when an argument has no tags; `null` is also tolerated by the mapper).

### New raw-row shape

```ts
interface RawArgumentRow {
  // ...all other fields unchanged...
  argument_tags: { tag_code: string }[] | null;   // replaces selected_tag_codes
  // ...
}
```

### New mapper helper

```ts
/** Flattens the embedded argument_tags relation into a plain tag-code array.
 *  Returns null when no tags rows came back (keeps AdminArgumentRow's
 *  string[] | null contract — null = "no tag data", [] would mean the same
 *  but null preserves the pre-fix nullable shape used by callers). */
function asTagCodes(j: RawArgumentRow['argument_tags']): string[] | null {
  if (!j || !Array.isArray(j) || j.length === 0) return null;
  return j
    .map((t) => (t && typeof t.tag_code === 'string' ? t.tag_code : null))
    .filter((c): c is string => c !== null);
}
```

Decision: return `null` (not `[]`) for the empty case, so the public
`AdminArgumentRow.selectedTagCodes: string[] | null` contract is unchanged and
the qualifier derivers (which accept `string[] | null`) behave exactly as they
did when the column was assumed-null. `[]` would be equally valid for the
derivers; `null` is chosen purely to keep the externally-visible shape
byte-identical to the pre-fix declared type and avoid any downstream surprise.

### Result mapping (the one line in `.map`)

```ts
// before: selectedTagCodes: r.selected_tag_codes,
selectedTagCodes: asTagCodes(r.argument_tags),
```

### RLS / policy text (UNCHANGED — referenced for proof, not modified)

`argument_tags` SELECT policy already grants admins read access
(`20260516000002_rls_policies.sql:244`):

```sql
CREATE POLICY "argument_tags: select with argument"
ON public.argument_tags FOR SELECT TO authenticated
USING (EXISTS (
  SELECT 1 FROM public.arguments a
  WHERE a.id = argument_id
    AND (a.author_id = auth.uid()
         OR is_moderator_or_admin()
         OR (a.status = 'posted' AND EXISTS (
              SELECT 1 FROM public.debates d
              WHERE d.id = a.debate_id AND d.status IN ('open','locked'))))));
```

No new policy. No migration.

## Edge cases

- **Argument with zero tags** — embed returns `[]` (or `null`); `asTagCodes`
  returns `null`; `deriveMessageCategory` / `derivePrimaryQualifier` already
  handle `null` (input type is `string[] | null`). No badge regression.
- **Argument with multiple tags** — `asTagCodes` returns all codes; the derivers
  already iterate the array (`messageQualifiers.ts` uses `.some(...)` /
  `.includes(...)`). The category badge reflects the first matching signal as
  it does today; no ordering guarantee is added or needed.
- **Malformed embed row** (a `tag_code` that is not a string) — `asTagCodes`
  filters it out rather than crashing the whole page. Defensive, cheap.
- **Non-admin caller** — RLS on both `arguments` and `argument_tags` returns
  zero rows; `loadAdminArguments` returns `[]`. The tab is also gated by
  `AdminScreen` so a non-admin never mounts it. No code path leaks tags.
- **`SUPABASE_CONFIGURED === false`** — loader returns `[]` early (existing
  guard, unchanged). The embed never executes.
- **Query genuinely fails** (network, RLS misconfig) — existing
  `throw new Error('loadAdminArguments failed: ...')` path is preserved; the
  tab still renders its accessible error state. The fix removes the *specific*
  `column ... does not exist` failure, not the generic error handling.
- **Soft-deleted arguments** — `includeDeleted` filter is unchanged; tags of a
  hidden argument are simply never fetched because the parent row is filtered.
- **Doctrine edge case — "do tags become a verdict?"** — No. `tag_code` values
  are advisory category/qualifier signals consumed by `messageQualifiers.ts`,
  which (by an existing ban-list test) contains zero verdict tokens. Surfacing
  real tags cannot introduce a truth label; it only makes an existing advisory
  badge accurate.
- **PostgREST relation ambiguity** — `arguments` has exactly one FK from
  `argument_tags` (`argument_tags.argument_id`), so `argument_tags(tag_code)`
  resolves unambiguously. No `!hint` disambiguator needed. (If a future
  migration adds a second FK between these tables, the embed would need
  `argument_tags!argument_tags_argument_id_fkey(tag_code)` — out of scope now.)

## Risks

- **PostgREST embed resolution.** If the project's PostgREST cannot resolve
  `argument_tags(tag_code)` (e.g. the FK metadata is stale in the schema cache),
  the query errors. Mitigation: the FK `argument_tags_argument_id_fkey` exists
  in the initial migration, so a correctly-migrated database resolves it. If the
  operator sees a `could not find a relationship` error after deploy, the
  fallback is Option A (drop the field) — documented below. This is a deploy-
  time observable, not a code risk.
- **`attached_evidence` is a SECOND non-existent column — CONFIRMED.** The
  `.select()` list at `adminArgumentsApi.ts:84` also requests
  `'attached_evidence'`. `grep -rn attached_evidence supabase/migrations/`
  returns **zero matches** — it is not a column on `public.arguments`. PostgREST
  reports unknown columns one at a time; once `selected_tag_codes` is fixed, the
  query will fail next on `arguments.attached_evidence does not exist` and the
  Admin All Arguments page **stays broken**. QOL-026 as scoped by the card
  (`selected_tag_codes` only) is therefore **insufficient on its own** to make
  the page load. See "Open question" — the coordinator must decide whether to
  widen QOL-026 by one line or spin a sibling card. The implementer must NOT
  silently expand scope without that decision.
- **Tests that read source text.** `adminArguments.test.ts` asserts on literal
  substrings. The new `.select()` must keep `argument_tags(tag_code)` as a
  contiguous literal (do not split it across template concatenation) or the
  contract test will not match.
- **No live Supabase available.** The implementer cannot run `npx supabase
  start` to integration-test the embed. The contract tests cover query shape;
  the operator confirms the live load. This is acceptable per the card's "no
  live Supabase mutation" constraint.

### Fallback — Option A (only if Option B's embed cannot resolve)

If, and only if, the operator confirms PostgREST cannot resolve the embed:
remove `'selected_tag_codes'` from `.select()` with no replacement, drop
`selected_tag_codes` from `RawArgumentRow`, and set
`selectedTagCodes: null` in the `.map(...)` result (keeping
`AdminArgumentRow.selectedTagCodes: string[] | null`). The badge derivers then
behave exactly as they have while the page was broken — category/qualifier
still derive from `argument_type` / `disagreement_axis` / `body`, just without
the tag signal. This restores the page; it does not restore the tag badge. Do
not delete `selectedTagCodes` from `AdminArgumentRow` (downstream derivers
expect the property to exist as nullable). Option A is the documented fallback,
not the plan.

## Open question for the operator / coordinator

**CONFIRMED during design: `attached_evidence` is a second non-existent column.**
`adminArgumentsApi.ts:84` selects both `'selected_tag_codes'` AND
`'attached_evidence'`. A migration grep proves neither exists on
`public.arguments`. PostgREST surfaces unknown columns one at a time, so fixing
only `selected_tag_codes` (the card's literal scope) will just move the error to
`column arguments.attached_evidence does not exist` — the Admin All Arguments
page will **still not load**.

The card title and acceptance criteria name only `selected_tag_codes`. To make
the page actually load, the implementer also needs to handle `attached_evidence`
(it is mapped at L112 into `hasEvidence`). The smallest correct fix is the same
pattern: there is an `argument_evidence` / evidence join table in the schema —
the implementer should grep for the real evidence relation and either embed it
(`<evidence_table>(id)` → `hasEvidence = array length > 0`) or, if no evidence
table exists yet, drop `attached_evidence` and set `hasEvidence: false`.

**Decision needed before Build:** either (a) widen QOL-026's acceptance criteria
to "Admin All Arguments loads" and fix both bad columns in one PR (recommended —
they are the same root cause, the same file, ~5 extra lines), or (b) spin a
sibling card QOL-027 for `attached_evidence` and accept that QOL-026 alone does
not fully unbreak the page. The design above is written for `selected_tag_codes`
per the card's literal scope; the implementer must get this decision before
claiming the acceptance criterion "Admin All Arguments loads without error".
