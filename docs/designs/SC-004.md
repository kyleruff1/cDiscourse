## Goal

SC-004 turns the Timeline / Tree into a **playable surface** by replacing SC-002's flat actor-driven action menu with a **context-aware action dock** that surfaces one suggested **primary action** plus a short ordered list of secondary actions, derived from LIFE-001 lifecycle state + META-001 manual tags + META-001 auto metadata + EV-001 source-chain status + BR-001 branch / cluster topology + the existing Stage 6.4 actor matrix. The dock is the *action surface*; SC-003 / Cards-detail stays the *detail inspector*; SC-002's popover keeps the actor-anchored hover affordance.

The model is doctrine-anchored on six sentences (mirrors research plan ¬ß15 verbatim):

1. **The dock RECOMMENDS, never BLOCKS.** Even `exhausted` / `archived_or_resolved` only re-order suggestions; ordinary `reply` is always available where the actor matrix allows.
2. **Lifecycle is play state, not truth.** No code path infers truth / winner / loser / correctness from a lifecycle state, metadata code, or evidence status. Wrong-but-loud and right-but-quiet produce identical dock models when the move structure matches.
3. **No verdict tokens.** No `winner` / `loser` / `liar` / `dishonest` / `bad faith` / `manipulative` / `extremist` / `propagandist` / `troll` / `astroturfer` / `true` / `false` / `correct` / `incorrect` / `right` / `wrong` / `defeated` / `lost` / `won` / `proof` / `proven` / `validated` / `verdict` in any dock-produced string.
4. **No person-attribution drift.** Labels describe a move or a cluster. `ignored_by_negative` is a cluster property; the dock surfaces it as "Negative did not respond" ‚ÄĒ never as an accusation against a user.
5. **No re-derivation of upstream signals.** `timelineNodeActionDockModel.ts` MUST NOT value-import `deriveMessageCategory`, `derivePrimaryQualifier`, `deriveMessageQualifiers`, `applyAntiAmplification`, `gradeChallenge`, `gradeRepair`, or `buildPointLifecycleMap`. It consumes outputs (`PointLifecycleMap`, `MoveMetadataLedger`, `ClusterMetadataSummary`, `TimelineEvidenceContract`) only. A forbidden-imports test enforces this.
6. **The dock is the dispatch surface, never the post path.** It produces an `ArgumentBubbleControl` / `TimelineNodeActionDockActionCode` event + an optional `MoveDraftPatch` preset. The room shell threads those into the existing composer + `submit-argument` Edge Function. The model file MUST NOT reference `SERVICE_ROLE`, `from('arguments').insert`, `supabase.from`, `fetch(`, or any network call.

SC-004 ships **no new dependency**, **no migration**, **no Edge Function deploy**, **no schema change**, **no AI inference path**, **no Supabase write**, **no `.env*` change**, and **no route push**. The model is pure-TS. The component is `Pressable` + `View` + StyleSheet + safe-area inset only.

## Module ownership ‚ÄĒ where does this code live?

**Decision:** new sibling files inside the existing `src/features/arguments/` module, *not* a new `src/features/dock/` directory.

Rationale:

- The dock layers on top of `argumentGameSurfaceModel.ts` (`ArgumentBubbleActor`, `ArgumentBubbleControl`, `getBubbleControlsForActor`, `BubbleControlsContext`), `timelineNodePopoverModel.ts` (selection / tap-effect helpers), and `quickActionPresets.ts` (composer-preset seam). All three live in `src/features/arguments/`. Splitting the dock into a parallel module would force re-export gymnastics and would invite the "two action universes" failure mode the research plan ¬ß4 explicitly warns against.
- LIFE-001 lives in `src/features/lifecycle/` because it owns a tightly-scoped 18-state vocabulary. SC-004 reads that vocabulary but produces a UI-shaped artefact; per the META-001 precedent (which lives in `src/features/metadata/` and reads lifecycle), SC-004's UI-adjacent model belongs next to the rest of the argument-room UI model.
- `src/features/dock/` is reserved for a *future* shared dock primitive if other surfaces (composer, Cards detail) grow a similar contextual action palette. v1 does not need it.

File layout (additions only):

```
src/features/arguments/
  timelineNodeActionDockModel.ts   # NEW ‚ÄĒ pure-TS dock model
  TimelineNodeActionDock.tsx       # NEW ‚ÄĒ RN component
  # (existing files untouched except ArgumentTimelineMap.tsx integration)
```

`branchTopologyModel.ts`, `pointLifecycleModel.ts`, `moveMetadataLedger.ts`, `evidenceModel.ts`, `sourceChainPresetCopy.ts`, and `quickActionPresets.ts` are NOT modified by SC-004. `gameCopy.ts` MAY receive a small `OBSERVER_COPY` extension for dock-specific helper strings (see ¬ß10 ‚ÄĒ Plain-language copy).

## Branch / cluster selection behavior

Mirrors research plan ¬ß9 verbatim.

Selection targets:

- **Node target** (`{ kind: 'node', messageId }`) ‚ÄĒ user tapped a single message node. Dock renders for the SELECTED node, using its move-level `MoveLinkageRecord` from META-001 + the cluster's `PointLifecycleClusterSummary` + `ClusterMetadataSummary`. - **Cluster target** (`{ kind: 'cluster', branchRootMessageId }`) ‚ÄĒ user tapped a branch area or cluster header chip. - **Collapsed stub target** (`{ kind: 'collapsed_stub', branchRootMessageId }`) ‚ÄĒ user tapped a `BranchCollapseStub`. The dock offers:

  - `open_cards_detail` (secondary) ‚ÄĒ surfaces the cluster summary in Cards-detail without expanding the branch.
  - All other actions are DISABLED with a helper "Expand the branch first to act on a specific move."

**Auto-expand consistency with BR-001:** when the dock dispatches `expand_branch`, the room shell calls `toggleBranchCollapse(state, branchRootMessageId)` and re-renders. ## Edge cases

Mirrors research plan ¬ß11 / category 11 and operator's run-prompt ¬ß15.

1. **Root selected.** Node target where `node.isRoot === true`. The cluster id IS the message id. Primary action is `lifecycle:open` ‚Üí `reply`. Cluster header reads `getPointLifecyclePlainLabel(cluster.state)`. Move chips are empty (root has no parent metadata signals like `has_reply`).
2. **Own message selected.** Actor `'self'`. Most participant actions are DISABLED with `own_bubble` reason. The dock surfaces `open_cards_detail` as primary, `narrow` / `concede` / `synthesize` as enabled (META-001 own-bubble eligibility), `request_deletion` is NOT a dock action (it lives on the bubble side rail per Stage 6.1.8 and stays there). `mark_moved_on` / `mark_ignored` are disabled with `manual_tag_not_in_vocabulary`.
3. **Observer selects a node.** Actor `'observer'`. Participant actions show with `observer_must_join` helper copy that invites a side join. `ask_source` is the only participant-shaped action enabled (per Stage 6.4). `flag` is also enabled (CLAUDE.md: observers may flag). `open_cards_detail` is enabled.
4. **Selected node inside a collapsed branch.** The active node's ancestors silently uncollapse (per BR-001 `applyActiveAutoExpand`); the dock target switches to the activated node. The dock NEVER renders `expand_branch` for a node target.
5. **Selected cluster header with multiple nodes.** Cluster target. The cluster summary's lifecycle is used; primary action speaks to the cluster (`synthesize` if `synthesis_ready`, `branch` if `branch_recommended`, etc.). Node-scoped actions (`ask_source` on a specific message) render disabled with `node_action_on_cluster` helper "Pick a specific message to ask source on."
6. **Archived / resolved cluster.** Lifecycle `archived_or_resolved`. Primary is `open_cards_detail`. Secondary actions render with `archived_resolved` disabled reason and helper "This point is resolved. Open details to read." `reply` is the fallback and remains enabled (the dock RECOMMENDS, never BLOCKS).
7. **`source_requested` with existing source attached.** Move-level `source_attached` auto code present. Primary stays `lifecycle:source_requested` (cluster-level) ‚Üí `add_evidence` / `ask_source`. The move chip surfaces `"Source attached"`. `ask_source` for the selected move is DISABLED with `evidence_already_attached`. The dock helper copy: "A source is already attached to this move; try asking for a quote or challenging the evidence."
8. **`quote_requested` with existing quote attached.** Symmetric to #7. `ask_quote` disabled with `quote_already_attached`. Helper: "A quote is already attached; try challenging the evidence."
9. **`exhausted` cluster with user trying to reply.** `reply` is NOT disabled. It is just not the primary. The dock primary is `narrow`; the secondary list contains `branch` and `reply`. User can still post a reply. Doctrine: lifecycle never blocks posting.
10. **`branch_recommended` cluster.** Primary `branch`. Branch dispatch flow opens BR-001's existing branch creation UI (not yet wired in this card ‚ÄĒ the dispatch event `{ action: 'branch', target }` reaches the room shell, which may forward to a future `BranchComposer` or fall back to opening the composer with no preset). Acceptance: the dock's responsibility ends at dispatch.
11. **`synthesis_ready` cluster.** Primary `synthesize`. Composer opens with `SYNTHESIZE_PRESET_BODY`. Cluster header reads `"Ready for synthesis"`.
12. **250+ messages.** Performance gate: dock model build must complete in < 50 ms. Achieved by reading already-built `lifecycleMap.byCluster` + `metadataLedger.byCluster` (both `ReadonlyMap`, O(1) lookups). No walk across all messages ‚ÄĒ only the cluster the selected target belongs to.
13. **Missing lifecycle summary.** `lifecycleMap.byCluster.get(clusterId)` returns `undefined`. The dock falls back to `lifecycle:open` primary and renders the cluster header label as `getPointLifecyclePlainLabel('open')` = `"Open for response"`. No crash.
14. **Missing metadata summary.** `metadataLedger.byCluster.get(clusterId)` returns `undefined`. The dock skips all manual-tag / auto-metadata refinements; it renders the lifecycle-default primary and empty `moveChips`. No crash.
15. **Missing evidence contract.** `evidenceContractFor(messageId)` returns `null`. The dock skips evidence-status refinements. The cluster header `evidenceLabel` is empty string.
16. **No parent id (detached node).** Node target where `parentId === null` AND `isRoot === false`. The action list renders with primary = `open_cards_detail` and all post-producing actions disabled with `detached_node` reason and helper "This message isn't attached to the conversation tree yet."
17. **Selection cleared (`selectedTarget === null`).** The dock is not rendered. The component returns `null`.

## Risks (top 3 for the implementer)

1. **SC-002 popover vs SC-004 dock interaction.** Both surfaces listen to node taps and both render anchored overlays. The implementer MUST verify that tapping a node does NOT cause both to open. SC-002's existing `decideNodeTapEffect` returns `'activate'` / `'open_popover'` / `'close_popover'`; the room shell must extend that decision to also drive `selectedTarget`. Recommended pattern: the activation (`'activate'`) sets `selectedTarget = { kind: 'node', messageId }` AND the popover stays closed; the user must explicitly tap the info icon for the popover. The dock and popover never coexist on the screen at the same time; tapping the info icon dismisses the dock (sets `selectedTarget = null`) and opens the popover, OR vice-versa. A test asserts they're mutually exclusive.
2. **Snapshot-diff for the input hash (memoization).** The room shell builds `lifecycleMap` + `metadataLedger` each render. The dock model build is O(cluster members) so this is cheap, but if the room shell rebuilds the lifecycle / metadata maps on every keystroke in the composer, the dock model rebuilds too. Recommended pattern: the room shell memoizes `lifecycleMap` + `metadataLedger` by `inputHash` (META-001 already exposes one) and rebuilds the dock model only when target / lifecycle / metadata / evidence change. The dock model itself does not need internal memoization.
3. **`narrow` / `confirm` / `synthesize` preset bodies are user-facing strings.** They ship in `quickActionPresets.ts` and will be visible to every player who taps the corresponding primary action. The implementer MUST run the ban-list test on `ALL_SC004_PRESET_BODIES` and MUST run a manual read-aloud check (does the body sound like the user said it, not like an AI said it?). The bodies in this design are deliberately first-person scaffolding with bracketed placeholders ‚ÄĒ the user is expected to edit them. If a copy review during Review surfaces a tone issue, file SC-1B as a copy-only follow-up; do NOT block SC-004's merge.

## Resolution of research plan ¬ß14 open questions

1. **Inline vs bottom dock decision.** **Bottom dock only in v1.** Inline-on-wide is deferred to discovery follow-up SC-1A. Rationale: SC-002's popover already covers the per-node hover affordance on wide; layering a third surface adds risk without clear value pre-merge.
2. **Composer presets for `narrow` / `concede` / `synthesize`.** **Ship in SC-004.** Three new `QuickActionLabel` values + three new preset body constants added to `quickActionPresets.ts`. EV-002 set the seam; SC-004 extends it. Doctrine-safe; ban-list tested. The alternative (defer to SC-1B) would leave the primary action on `narrowed` / `conceded` / `synthesis_ready` clusters surfacing a blank composer ‚ÄĒ that defeats the contextual recommendation entirely.
3. **Cluster target vs node target for `synthesize`.** **Require cluster target.** On a node target, `synthesize` renders disabled with `cluster_action_on_node` helper "Select the cluster to synthesize." Tests assert this.
4. **`mark_moved_on` / `mark_ignored` vs META-001 manual tag vocabulary.** **Dock action is a no-op UI advisory in v1, DISABLED with `manual_tag_not_in_vocabulary` reason and helper "This is coming in a later update."** Filed as discovery follow-up #2 ‚ÄĒ META-001 patch to add `moved_on_by_<side>` / `ignored_by_<side>` manual tags. SC-004 does NOT block on this; the dock surfaces the disabled affordance so the player knows the feature is coming.
5. **`expand_branch` UI primitive routing.** **Dispatched to `toggleBranchCollapse(state, branchRootMessageId)` from BR-001.** The dock model emits an event with `action: 'expand_branch'` and `target.branchRootMessageId`; the room shell calls `setCollapseState(toggleBranchCollapse(state, target.branchRootMessageId))`. The dock model NEVER calls `toggleBranchCollapse` itself (purity). No new `argument_type`. No composer preset.
