## Documented deviation from MCP-008 (read this first)

MCP-008 was written before MCP-011 merged. **MCP-008 §5 keys the banner map off "all 90 MCP-002 seed-bank classifier ids (A1…G10)".** That assumption no longer matches the merged code, and the implementer must follow this section, not MCP-008 §5.3 verbatim.

**What actually merged (verified against `src/features/semanticReferee/semanticRefereeTypes.ts`):** MCP-011 shipped the **23-id catalog v0** as the `SemanticClassifierId` union — `responds_to_parent`, `introduces_new_issue`, `asks_for_evidence`, `provides_evidence`, `evidence_supports_claim`, `quote_anchors_parent`, `narrows_claim`, `concedes_narrow_point`, `requests_clarification`, `answers_clarification`, `shifts_to_person_or_intent`, `uses_popularity_as_evidence`, `contains_playable_hot_take`, `contains_unplayable_insult_only`, `is_satire_or_parody`, `uses_satire_as_evidence`, `cites_retraction`, `creates_source_chain_gap`, `suggests_side_branch`, `suggests_diagonal_tangent`, `fits_selected_debate_mode`, `needs_pre_send_pause`, `ready_for_synthesis`. A `SemanticBinarySample.classifierId` is **typed** as this 23-id union — a real `SemanticRefereePacket` can never carry an `A1`/`B4`/`G7`-style seed id. The 90 seed ids in `docs/semantic-prompts/mcp-semantic-referee-prompt-bank.md` §22 are explicitly "the bank holds 90 so the catalog can grow without a contract change" — they are a **design reference**, not a runtime vocabulary.

**Resolution (binding on the implementer):**
1. `CLASSIFIER_TO_BANNERS` is keyed off the **23 `SemanticClassifierId` values**, imported from `ALL_SEMANTIC_CLASSIFIER_IDS` (MCP-011 exports it). The coverage test asserts the key set equals `ALL_SEMANTIC_CLASSIFIER_IDS` — not "90 ids". This makes the test track the real contract automatically; if MCP-011's catalog later widens, the test fails until the new id gets a mapping, which is exactly the safety MCP-008 §5.3 rule 1 intended.
2. `INTENTIONALLY_SILENT_CLASSIFIERS` is a subset of the 23. The catalog v0 has **one** structurally-silent id worth flagging: `contains_unplayable_insult_only` — its presence is a *routing* signal (the move goes to a low-priority lane), not user feedback; surfacing a banner for "this is only an insult" would label the move punitively. It maps to `[]`. The four seed-bank ids MCP-008 §5.3 rule 3 named (`F2`, `F11`, `B5`, `B6`) are not in catalog v0 at all, so they need no mapping. The test asserts `INTENTIONALLY_SILENT_CLASSIFIERS` is exactly `['contains_unplayable_insult_only']` and that its mapping is `[]`.
3. The **banner library itself still holds ≥100 strings across the 12 categories** — MCP-008 §5.1's category floors are unchanged. The library is sized by user-feedback coverage, not by classifier count; 23 ids fanning out to ≥100 banners is the normal one-to-many shape (an id like `creates_source_chain_gap` offers several candidates).
4. The **23-id → banner mapping table in §6 below** is the authoritative map. It is derived from MCP-008 §5.1's per-category "Maps from" columns, collapsed onto the 23 real ids (e.g. MCP-008's `A1 responds_to_parent_directly` collapses onto the catalog's `responds_to_parent`).
5. The `FEEDBACK_CODE_TO_BANNER` map is keyed off MCP-013's **22** `RefereeFeedbackCode` values (`ALL_REFEREE_FEEDBACK_CODES`), not "the 20 in MCP-003 §8". MCP-013 added `broad_point_set_down` and `synthesis_named`; MCP-008 §13 says "20" — superseded by the merged 22-code union.

This is a naming/contract-version reconciliation, not a redesign. Every doctrine rule, the tone-band model, the confidence rule, the accessibility contract, the `selectBanner` algorithm shape, and the ≥100-string floor are followed from MCP-008 exactly. The reconciliation is recorded here so the implementer applies it instead of re-deriving a divergent 90-key map that would not type-check against a real packet.

## Goal (one paragraph)

MCP-014 builds the **referee feedback banner library** designed in MCP-008 — a pure-TypeScript string library plus a deterministic selection model that turns already-classified move metadata into one short, encouraging, non-verdict banner ("Clean parent tie.", "Good pressure. Evidence debt opened.", "This probably belongs on a branch."). A banner is generated **only** from a `SemanticRefereePacket` (MCP-011) and/or a `LedgerResult.categoryReadings` (MCP-013) that already exist — never from a fresh conversation scrape, never from a live model call. The doctrine that shapes every line (`cdiscourse-doctrine`, `timeline-grammar`, `accessibility-targets`): a banner describes the **move**, never the **person**; it carries no verdict / winner / loser / truth / right / wrong token; it is encouragement or a suggested next move, never a punishment; it inherits `authoritative: false` and never escalates a `0/1` binary into a hard claim; popularity / heat is never an input and never praised as evidence; a low-confidence input produces a *softer* banner or *no* banner, never a confident-sounding line; a raw `classifierId` / `reasonCode` / `feedbackCode` / `bannerCode` never reaches a user surface, and an unmapped code is **suppressed, not echoed**; the model is pure-TS with no Supabase / React / network import; and the banner never blocks posting and never auto-acts. MCP-014 owns `src/features/refereeBanners/*` exactly and extends `gameCopy.ts` `PLAIN_LANGUAGE_COPY` append-only with the `bannerCode` family. The render component is deliberately **deferred** (MCP-008 §15): MCP-014 ships the string library, the selection model, the tone-band model, the confidence rule, and the ready-to-render `accessibilityLabel` / `toneGlyph` — everything a thin follow-up `<View>`/`<Text>` card consumes — and no `<View>`.

### `RefereeBanner` — one library entry

```ts
export interface RefereeBanner {
  /** Stable internal id — snake_case. Added to PLAIN_LANGUAGE_COPY. NEVER user-visible raw. */
  bannerCode: string;
  category: RefereeBannerCategory;
  tone: RefereeBannerTone;
  /** The visible banner line. <= 64 chars. Ban-list clean. A complete clause. */
  headline: string;
  /** Optional second line — a concrete suggested move. <= 80 chars. Ban-list clean. */
  helperLine?: string;
  /** Non-color tone marker the render layer shows. */
  toneGlyph: RefereeBannerToneGlyph;
  /** The complete screen-reader sentence (MCP-008 §8.1). Always a full sentence
   *  with the tone stated in words. Built by buildBannerAccessibilityLabel and
   *  frozen into the library entry at module load. */
  accessibilityLabel: string;
  /** Minimum confidence at which this banner may show. A banner whose effective
   *  confidence falls below this is softened or suppressed (confidence rule). */
  minConfidence: RefereeBannerConfidence;
  /** The bannerCode of this banner's softened sibling, or null. When the source
   *  confidence is below minConfidence, selectBanner swaps to the sibling. A
   *  softened banner itself has softenedSiblingCode: null and minConfidence: 'low'. */
  softenedSiblingCode: string | null;
}
```

> **Deviation note vs MCP-008 §4 field name:** MCP-008's interface used `hasSoftenedSibling: boolean`. MCP-014 uses `softenedSiblingCode: string | null` instead — a boolean is not enough to *find* the sibling at selection time; `selectBanner` step 2 needs the actual code to swap to. This is a strict superset (`softenedSiblingCode !== null` answers the boolean question) and removes a lookup ambiguity. Recorded as a documented deviation, not a silent change.

### Selection input / result

```ts
/** The deterministic selection input — built UPSTREAM from a SemanticRefereePacket
 *  and/or a LedgerResult. The banner library NEVER reads a raw conversation. */
export interface BannerSelectionInput {
  /** The packet's binaries that read value === 1, in packet order. Read-only.
   *  classifierId is a SemanticClassifierId (the 23-id catalog). */
  positiveBinaries: ReadonlyArray<{
    classifierId: SemanticClassifierId;
    confidence: SemanticConfidence;        // 'low' | 'medium' | 'high'
  }>;
  /** The ledger's category readings, if a LedgerResult is available. Read-only. */
  categoryReadings: ReadonlyArray<{
    feedbackCode: RefereeFeedbackCode;     // a MCP-013 RefereeFeedbackCode
    confidence: LedgerConfidence;          // 'low' | 'medium' | 'high'
    outcome: ReconciliationOutcome;        // a MCP-013 ReconciliationOutcome
    requiresUserChoice: boolean;
  }>;
}

/** The selection result — at most ONE banner per move (MCP-008 §5.4). */
export interface BannerSelectionResult {
  /** The chosen banner, or null when nothing is worth surfacing / all suppressed. */
  banner: RefereeBanner | null;
  /** Why this banner won (or why none did) — dev/debug + tests, NEVER user-visible. */
  selectionTrace: string;
}
```

> **Deviation note vs MCP-008 §4 `BannerSelectionInput`:** MCP-008's `categoryReadings` entry carried a separate `category: string` field and the example used loose `string` types; MCP-014 drops the redundant `category` (the `feedbackCode` already determines the banner — `FEEDBACK_CODE_TO_BANNER` is keyed off `feedbackCode`) and tightens every field to the merged MCP-011/MCP-013 union types so the model type-checks against a real `CategoryReading`. MCP-014 also adds `requiresUserChoice` to the input because the confidence rule (MCP-008 §7.1) needs it to force the `conflict_routed` routing banner; MCP-008 §7.1 derives it from `outcome === 'conflict_routed'`, which the implementer may still do — both `outcome` and `requiresUserChoice` are present so either check works. MCP-008's optional `routeSuggestion?: string` is **dropped**: §6's 23-id map already covers `suggests_side_branch` / `suggests_diagonal_tangent`, so the route suggestion would double-count; recorded as a deliberate simplification.

All `RefereeBanner` fields are JSON-serializable strings/enums; the model is side-effect-free and pure — same constraint as `src/lib/constitution/engine.ts`, `src/features/pointStanding/`, and `src/features/refereeLedger/`.

### `buildBannerAccessibilityLabel` contract

Format: `"<framingWord> <tone-prefix>: <headline> <helperLine, if present>"`.
Examples (framingWord = `Referee`):
- celebratory `continuity_clean_tie` → `"Referee note: Clean parent tie. The referee likes how this picks up the thread."`
- nudge `quote_needed_exact_excerpt` → `"Referee suggestion: Source is here — the exact quote is still needed."`
- routing_hint `branch_belongs_on_branch` → `"Referee suggestion: This probably belongs on a branch."`

The function is pure: same input → same output, no `Date.now()`, no randomness. Changing `framingWord` changes only the prefix; every `accessibilityLabel` in the library is regenerated from `headline` + `helperLine`, so the operator's scoreboard-framing-word decision is a one-argument change.

### `selectBanner` — the deterministic selection algorithm

Given a `BannerSelectionInput`, `selectBanner` returns the same `BannerSelectionResult` every call. No randomness, no clock, no model call.

1. **Build the candidate pool.** For each `positiveBinaries[i]`, look up `CLASSIFIER_TO_BANNERS[classifierId]`; append each `bannerCode` paired with that binary's `confidence`. Skip ids in `INTENTIONALLY_SILENT_CLASSIFIERS` (their value is `[]` so this is automatic — the skip is also asserted defensively). For each `categoryReadings[j]`, look up `FEEDBACK_CODE_TO_BANNER[feedbackCode]`; append the `bannerCode` paired with that reading's `confidence`. A `bannerCode` not present in `BANNER_BY_CODE` is skipped defensively and recorded in `selectionTrace` (the coverage test makes this impossible in practice).
2. **Force the conflict-routed banner.** If any `categoryReadings[j]` has `outcome === 'conflict_routed'` (equivalently `requiresUserChoice === true`), the pool is **replaced** with exactly one candidate: the `pacing_you_decide_the_lane` routing banner at the reading's confidence. A routed conflict never produces a confident celebratory/nudge line (MCP-008 §7.3) — this short-circuit is the enforcement.
3. **Apply confidence handling** (see next section) to every pooled candidate: keep, swap to softened sibling, or drop.
4. **Rank by `BANNER_CATEGORY_PRIORITY`.** Resolve each surviving `bannerCode` to its `RefereeBanner` via `BANNER_BY_CODE`, group by `category`, pick the highest-priority category present. Within one category, ties break by the candidate's **first appearance in pool order** (which is itself deterministic: `positiveBinaries` order, then `categoryReadings` order, then the `CLASSIFIER_TO_BANNERS` array order). No randomness.
5. **Return at most one banner.** `{ banner, selectionTrace }`. If the pool is empty or every candidate was suppressed, return `{ banner: null, selectionTrace: '<reason>' }`. `null` is a valid, common, correct result for an ordinary move.

### Confidence handling — softening / suppression rule

A banner never invents confidence. It reads it from `SemanticBinarySample.confidence` (packet path) or `CategoryReading.confidence` (ledger path). For each pooled candidate, compare the source confidence against the banner's `minConfidence`:

| Effective source confidence | Banner `minConfidence` | Action |
|---|---|---|
| `high` or `medium` | `low` or `medium` | Banner stays as authored. |
| `high` | `high` | Banner stays as authored. |
| `medium` | `high` | Downgrade: if `softenedSiblingCode !== null`, swap to the sibling; else **drop** the candidate. |
| `low` | any | If `softenedSiblingCode !== null`, swap to the sibling (the sibling has `minConfidence: 'low'`); else **drop** the candidate. |

After a swap, the candidate's `bannerCode` becomes the sibling's; the sibling is itself a library entry with its own `bannerCode` (e.g. `continuity_clean_tie_soft`), its own `PLAIN_LANGUAGE_COPY` mapping, and `minConfidence: 'low'`, so it is never suppressed by step 3. A softened sibling has `softenedSiblingCode: null` (no infinite chain) and its `headline` contains a hedge token (`looks like` / `might` / `reads as` / `may`) — a test asserts this for every sibling.

**The hard rule low confidence enforces:** a `low`-confidence binary never produces a celebratory line that sounds like a verdict ("Clean parent tie." → "This looks like it ties to the parent." or dropped); a `low`-confidence binary never produces a `nudge` that asserts a defect as fact ("The evidence doesn't cover the whole claim" → "The evidence might not cover the whole claim yet"); a `conflict_routed` reading produces exactly the `pacing_you_decide_the_lane` routing banner (step 2). When in doubt, no banner — `{ banner: null }` is always safe.

### Unmapped-code suppression rule

- A `classifierId` not in `CLASSIFIER_TO_BANNERS` — impossible at the type level (the key set is `SemanticClassifierId`), but `selectBanner` skips an unknown key defensively and records it in `selectionTrace` rather than throwing. The coverage test catches a real gap.
- A `feedbackCode` not in `FEEDBACK_CODE_TO_BANNER` — same: skipped, recorded, caught by the coverage test.
- A `bannerCode` referenced by a map but absent from `REFEREE_BANNER_LIBRARY` — skipped defensively; the coverage test asserts every mapped `bannerCode` exists in `BANNER_BY_CODE`.
- A `bannerCode` selected but missing from `PLAIN_LANGUAGE_COPY` — the render layer (a follow-up card) passes the `headline` through `toPlainLanguageOrSuppress`; an unmapped code returns `null` and the render layer shows nothing. **The user never sees a raw `bannerCode`.** The plain-language coverage test asserts every `bannerCode` resolves through `toPlainLanguage` to a non-null string, so this never happens in practice — but the suppression behavior is the doctrine backstop (`cdiscourse-doctrine` §9).

### `gameCopy.ts` `bannerCode` extension

Append a new block to the end of the `PLAIN_LANGUAGE_COPY` object literal (after the MCP-013 ledger block, before the closing `} as const;`):

```ts
  // MCP-014 — Referee banner library bannerCode family (>=103 codes). Every
  // code is a gameplay signal about the MOVE, never a verdict, never a person
  // label. Plain English, no snake_case leak, ban-list-clean. The VALUE is the
  // banner's `headline` verbatim. Read by toPlainLanguage so an unmapped
  // bannerCode is suppressed, not echoed. Append-only — no existing key changed.
  continuity_clean_tie: 'Clean parent tie.',
  continuity_clean_tie_soft: 'This looks like it ties to the parent.',
  continuity_engages_mechanism: 'Good — this engages the actual mechanism.',
  // ... one entry per bannerCode in REFEREE_BANNER_LIBRARY, including every
  //     softened sibling (e.g. *_soft) ...
  pacing_you_decide_the_lane: 'Reply here, branch, or tangent? — you decide.',
```

Rules for the extension:
- One `PLAIN_LANGUAGE_COPY` entry per `bannerCode` in the library. The value is the banner's `headline` verbatim (so `toPlainLanguage(bannerCode)` returns the visible line — this is the contract the render layer relies on).
- Every softened sibling code (`*_soft`) gets its own entry.
- The block is purely additive. No existing key is renamed, revalued, or moved. (Same precedent as META-001 / LIFE-001 / MCP-013, which each appended to this record.)
- A `bannerCode` must not collide with an existing `PLAIN_LANGUAGE_COPY` key. The library author picks `bannerCode`s namespaced by category prefix (`continuity_*`, `evidence_debt_*`, `source_chain_*`, …); a pre-commit check (and the `refereeBannerLibrary.test.ts` uniqueness test) catches any collision with an existing key. If a desired `bannerCode` would collide (e.g. `source_attached` already exists from META-001), the library author **renames the bannerCode** (e.g. `source_chain_attached`) — the banner library never reuses another feature's key, because the two would then share a single string and drift.
- `helperLine` strings are **not** added to `PLAIN_LANGUAGE_COPY` — only `bannerCode → headline`. The `helperLine` is carried on the `RefereeBanner` object and rendered directly; it is ban-list scanned but it is not a code-keyed lookup.

## Edge cases

The implementer must handle each:

- **Empty `BannerSelectionInput`** (no positive binaries, no readings — a quiet, unremarkable move). `selectBanner` → `{ banner: null, selectionTrace: 'empty_pool' }`. The render layer shows nothing. This is the **common** case — most moves get no banner, and that is correct.
- **All candidates suppressed by low confidence** (non-empty pool, every candidate fell below `minConfidence` with no softened sibling). → `{ banner: null, selectionTrace: 'all_suppressed_low_confidence' }`. Never a confident fallback.
- **A `conflict_routed` reading co-occurring with a high-confidence celebratory binary.** Step 2 short-circuits: the pool is replaced with only the `pacing_you_decide_the_lane` routing banner. An unresolved routing decision outranks a compliment — correct.
- **Multiple `conflict_routed` readings.** Step 2 still produces exactly one `pacing_you_decide_the_lane` banner (idempotent — the pool is replaced, not appended to).
- **Conflicting candidates from packet path and ledger path** (both fed the pool, different categories). `BANNER_CATEGORY_PRIORITY` resolves it deterministically; there is no random tie-break.
- **A binary with `value: 0`.** `positiveBinaries` carries only `value === 1` binaries by construction (the upstream builder filters). A `value: 0` binary contributes no banner — the *absence* of a property is never surfaced as a failure ("does not respond to parent" is not a banner; the *presence* of `responds_to_parent: 1` surfaces the constructive `continuity_*` banner instead).
- **`contains_unplayable_insult_only: 1` in the packet.** It is in `INTENTIONALLY_SILENT_CLASSIFIERS`; its mapping is `[]`; it contributes no banner. The move is never labelled "an insult" — that would be a punishment and a near-person-label. Routing of an insult-only move is owned by the route layer, not the banner library.
- **`uses_popularity_as_evidence: 1` in the packet.** It maps to a `source_chain_gap` banner that asks for a source ("Popularity isn't proof — what's the source?"). The banner *names* popularity only to say it is **not** proof and route to a source request; it never praises it. There is no engagement/heat input to `BannerSelectionInput`, so no banner can ever be selected *because* a move is popular.
- **A classifier id added to MCP-011's catalog later without a banner mapping.** Caught by `refereeBannerLibrary.test.ts` (key set must equal `ALL_SEMANTIC_CLASSIFIER_IDS`) — a test failure, not a runtime crash.
- **A `feedbackCode` added to MCP-013's union later without a banner mapping.** Caught by the same coverage test (key set must equal `ALL_REFEREE_FEEDBACK_CODES`).
- **A packet whose `authoritative` is somehow not `false`.** Impossible per MCP-011's literal `false` type; and the banner model never reads `authoritative` at all — it only reads `positiveBinaries` / `categoryReadings`. A malformed packet cannot make a banner assert authority.
- **Concurrent edits.** Argument bodies are immutable after posting (soft-delete only — `cdiscourse-doctrine` §8). The upstream packet is computed once per move; a banner is stable for a posted move. No concurrent-edit race exists in the banner model.
- **Offline / network failure.** The banner model is pure-TS with no network. The *packet* is fetched upstream; if that fetch fails, the upstream passes an empty/mock input and `selectBanner` returns `null`. The banner model itself never makes a network call and never errors on a missing packet.
- **Permission-denied.** Not applicable — the model has no auth surface; it consumes already-fetched, already-classified data. The same banner string is doctrine-safe regardless of who reads it.
- **Doctrine edge case — can heat boost a banner to celebratory?** No. There is no heat/engagement field in `BannerSelectionInput`. A banner is `celebratory` only for a *structural* property of the move (it tied to the parent, named a mechanism, is a coherent hot take).
- **Doctrine edge case — can a banner say the other side lost a point?** No. A banner is about *the move that was just posted*, never about the opponent and never about a point's resolution. "Evidence debt resolved — nice." celebrates the *repairing move*; it does not say the challenger "lost". Concession banners ("Narrow concession noted — the broad point still stands.") follow MCP-013's `concession_noted` copy exactly — a repair, never a defeat.
- **Doctrine edge case — a low-confidence binary is the only signal.** It shows in softened form if it has a softened sibling, otherwise no banner. It never shows as a confident line.
- **`helperLine` absent.** Many banners are a single line. `accessibilityLabel` is built from `headline` alone in that case; the format string handles the missing `helperLine` (no trailing space, no empty clause).

## Test plan

All tests are pure-model; **no test makes a live call.** Seven files. Coverage bar (`test-discipline`): every public function (`selectBanner`, `buildBannerAccessibilityLabel`) has unit tests including failure / empty / suppressed cases; the data files are covered by the library + ban-list + plain-language tests.

### `__tests__/refereeBannerLibrary.test.ts` — coverage

- `REFEREE_BANNER_LIBRARY.length >= 100` (MCP-008 §5.1; the design target is ≥103).
- Each of the 12 `RefereeBannerCategory` values has at least its MCP-008 §5.1 floor count of banners (`continuity` ≥10, `evidence_debt` ≥9, `hot_take` ≥9, `clever_rebuttal` ≥9, `source_chain_gap` ≥9, `branch_suggestion` ≥8, `tangent_suggestion` ≥8, `synthesis_readiness` ≥9, `quote_needed` ≥8, `mechanism_needed` ≥8, `mode_mismatch` ≥8, `pacing_cooldown` ≥8).
- Every `bannerCode` is unique across the library.
- Every `RefereeBanner.headline` is ≤64 chars; every `helperLine` (when present) is ≤80 chars.
- Every `softenedSiblingCode` (when non-null) resolves to a real library entry whose `minConfidence === 'low'` and whose own `softenedSiblingCode === null`.
- `Object.keys(CLASSIFIER_TO_BANNERS)` equals `ALL_SEMANTIC_CLASSIFIER_IDS` (the 23-id catalog) as a set — every catalog id is a key.
- Every `CLASSIFIER_TO_BANNERS` value is a non-empty array **except** for ids in `INTENTIONALLY_SILENT_CLASSIFIERS`, whose value is exactly `[]`.
- `INTENTIONALLY_SILENT_CLASSIFIERS` equals `new Set(['contains_unplayable_insult_only'])` exactly — so a deliberate silence is distinguishable from a missing-mapping bug.
- `Object.keys(FEEDBACK_CODE_TO_BANNER)` equals `ALL_REFEREE_FEEDBACK_CODES` (all 22) as a set.
- Every `bannerCode` referenced by `CLASSIFIER_TO_BANNERS` or `FEEDBACK_CODE_TO_BANNER` exists in `BANNER_BY_CODE`.

### `__tests__/refereeBannerBanList.test.ts` — doctrine ban-list (the acceptance-criterion test)

- Scan **every** `headline`, **every** `helperLine`, **every** `accessibilityLabel`, and **every** `bannerCode` in the full library against the doctrine ban-list (whole-word, case-insensitive): `winner · loser · won · lost · win · lose · right · wrong · true · false · correct · incorrect · proven · disproven · defeated · beat · crush · liar · lying · dishonest · bad faith · manipulative · troll · bot · astroturfer · propagandist · extremist · stupid · idiot · dumb · smart · clueless · pathetic · ignorant`. (Whole-word matching so legitimate substrings like "narrowing" are not false hits.)
- No user-facing field (`headline`, `helperLine`, `accessibilityLabel`) matches `looksLikeInternalCode` (imported from `gameCopy.ts`).
- No `bannerCode` string appears verbatim inside any `headline` / `helperLine` / `accessibilityLabel`.
- No popularity-as-praise: a banner whose string contains `popular` / `trending` / `viral` / `everyone agrees` / `most people` MUST also contain a not-proof phrase (`isn't proof` / `not proof` / `not evidence` / `what's the source`). The test scans for praise-context, not bare token presence.
- No second-person insult: no string matches `/you('re| are) (wrong|losing|behind|bad|clueless)/i` or similar belittling constructions.
- Schema-shape assertion: a sampled `RefereeBanner` object has no `block`, no `winner`, no `truthValue`, no `score`, no `authoritative` key — `RefereeBanner` carries none of those (a structural assertion that the type did not drift).
- Every softened sibling's `headline` contains a hedge token (`looks like` / `might` / `reads as` / `may`).

### `__tests__/refereeBannerConfidence.test.ts` — confidence handling

- A `low`-confidence source for a banner that has a softened sibling yields the **softened** banner; the softened banner's `headline` contains a hedge token.
- A `low`-confidence source for a banner with no softened sibling yields **no banner** (suppressed; `selectionTrace` records it).
- A `medium`/`high` source yields the banner as authored (no swap, no drop) when `minConfidence` allows.
- A `medium` source against a `minConfidence: 'high'` banner downgrades (swap to sibling if present, else drop).
- A `conflict_routed` `CategoryReading` yields exactly the `pacing_you_decide_the_lane` routing banner — never a confident celebratory/nudge line — even when a high-confidence celebratory binary co-occurs.
- A `requiresUserChoice: true` reading produces the same routing-banner result as `outcome === 'conflict_routed'`.
- Low confidence never produces a banner whose `tone` is `celebratory` with a verdict-shaped headline (assert: when the source is `low`, the result banner, if any, is the softened sibling).

### `__tests__/refereeBannerSelection.test.ts` — `selectBanner` determinism + priority

- `selectBanner` is deterministic: the same `BannerSelectionInput` yields a deeply-equal `BannerSelectionResult` across repeated calls.
- An empty input yields `{ banner: null, selectionTrace: 'empty_pool' }`.
- The category-priority order is respected: an input that fields a `source_chain_gap` candidate **and** a co-occurring `hot_take` candidate selects the `source_chain_gap` banner.
- A `pacing_cooldown` candidate beats every other category.
- At most one banner is ever returned (the result `banner` is a single object or `null`, never an array).
- A within-category tie breaks by pool order (a stable, documented order) — assert two same-category candidates always pick the first-in-pool one.
- A `null`/malformed input does not throw (defensive guard returns `{ banner: null }`).

### `__tests__/refereeBannerForbiddenImports.test.ts` — purity guard

- Source-scan every file under `src/features/refereeBanners/`: no import from `@supabase/*`, `react`, `react-native`, `expo*`, a network/fetch library, or any provider SDK. (The MCP-013 `refereeLedgerForbiddenImports.test.ts` pattern.)
- The only cross-feature imports allowed: `import type` from `src/features/semanticReferee/semanticRefereeTypes.ts`, `import type` from `src/features/refereeLedger/types.ts`, and the runtime import of `gameCopy.ts` (the plain-language map). The test asserts no other cross-feature import.

### `__tests__/refereeBannerAccessibility.test.ts` — accessibility guard

- Every `RefereeBanner.accessibilityLabel` is a non-empty complete sentence that begins with the tone prefix (`<framingWord> note:` for `celebratory`, `<framingWord> suggestion:` for `nudge`/`routing_hint`).
- `buildBannerAccessibilityLabel` is pure: same input → same output across repeated calls.
- Changing `framingWord` changes only the prefix; the headline/helper portion is byte-identical.
- Every banner has a `toneGlyph` of `star` | `arrow` | `branch` (a non-color tone signal always exists — `accessibility-targets` "color is never the only signal").
- `toneGlyph` is consistent with `tone`: `celebratory → star`, `nudge → arrow`, `routing_hint → branch` (so a grayscale render distinguishes the tone by shape — `timeline-grammar` "shape carries the meaning"). A test asserts this mapping for every library entry.
- `accessibilityLabel` is rebuilt from `headline` + `helperLine` and equals the value frozen on the entry (the label cannot drift from the headline).

### `__tests__/refereeBannerPlainLanguage.test.ts` — plain-language coverage (`cdiscourse-doctrine` §9)

- For every `bannerCode` in `REFEREE_BANNER_LIBRARY`, `toPlainLanguage(bannerCode)` returns a non-null string equal to the banner's `headline`.
- `toPlainLanguage(bannerCode)` for every banner contains no `_` (no snake_case leak).
- An unknown made-up code (`'not_a_real_banner_zzz'`) returns `null` from `toPlainLanguage` — the suppression backstop works.
- No new `bannerCode` collides with a pre-existing `PLAIN_LANGUAGE_COPY` key that belongs to another feature (assert the MCP-014 block's keys are disjoint from the pre-MCP-014 key set — the test reads a frozen snapshot list of the pre-existing keys).

## Risks

- **Voice drift across ≥103 hand-authored strings.** A library this size can drift into scolding or verdict-shaped copy on the long tail. *Mitigation:* MCP-008 §5.2 locks the voice with a representative sample across all 12 categories — the implementer extends each category to its floor using that voice; `refereeBannerBanList.test.ts` scans *every* string including the long tail; the no-second-person-insult and no-popularity-praise assertions catch the two most likely drifts. A reviewer should still read the full library.
- **`bannerCode` collision with `gameCopy.ts`.** `PLAIN_LANGUAGE_COPY` already has ~150 keys (validation rail, semantic axes, LIFE-001 lifecycle, META-001 tags, MCP-013 ledger). A `bannerCode` that duplicates an existing key would silently share a string. *Mitigation:* category-prefixed `bannerCode`s; the `refereeBannerPlainLanguage.test.ts` disjoint-keys assertion; and a note in the §"gameCopy.ts extension" rule that the library author renames on collision (e.g. `source_chain_attached`, not `source_attached`). The implementer should diff the new keys against the existing record before committing.
- **The 23-vs-90 reconciliation.** If a future implementer reads MCP-008 §5.3 without reading this doc's deviation section, they may mint a 90-key map that does not type-check against a real `SemanticBinarySample`. *Mitigation:* the deviation section is the first section of this doc; the `refereeBannerLibrary.test.ts` coverage test keys off `ALL_SEMANTIC_CLASSIFIER_IDS` directly, so an over-wide map fails the test immediately.
- **The category-priority order is a judgement call.** `BANNER_CATEGORY_PRIORITY` ranks `pacing_cooldown` / `source_chain_gap` highest and `hot_take` lowest. If product feedback later wants celebratory banners to win more often, it is a single documented constant — a low-risk, well-localized change, not a contract change. Flagged so it is a conscious decision.
- **The render component is a separate card.** MCP-014 delivers the string library + selection model + accessibility strings but not the `<View>`. The color-independence, reduce-motion, announce-once, and 44×44-hit-target requirements (`accessibility-targets`) are *designed* here (the `toneGlyph` + `accessibilityLabel` make them possible) but *enforced* by the render card's tests. If the render card is not filed, the accessibility contract is designed but not shipped — flagged so the render card is not forgotten.
- **No migration / no operator deploy.** Pure-TS model + an additive `gameCopy.ts` extension. Lowest operational risk class — the same precedent META-001 / LIFE-001 / MCP-013 set when they extended `PLAIN_LANGUAGE_COPY`. No runner-side regression because the pipeline reads codes, not labels.

## Doctrine self-check

- **cdiscourse-doctrine §1 — score is gameplay analysis, never truth:** every banner describes a *structural play property* of the move (continuity, branch fit, mechanism present, source attached); no banner asserts a claim is true/false/proven; the ban-list test scans `true`/`false`/`proven`/`correct`/`incorrect`. ✓

- **cdiscourse-doctrine §2 — heat means activity, not truth:** there is no heat field; a banner is never warmer because a move is active. ✓

- **cdiscourse-doctrine — describe the move, never the person; no insults:** every banner is about the posted move; the ban-list excludes person labels (`liar`, `troll`, `manipulative`, `propagandist`, `extremist`, …); `contains_unplayable_insult_only` is intentionally silent so a move is never labelled "an insult"; the no-second-person-insult assertion scans for belittling constructions. ✓

- **timeline-grammar — heat is not truth; shape carries meaning:** the `toneGlyph` (`star`/`arrow`/`branch`) is a *shape* signal independent of color; `refereeBannerAccessibility.test.ts` asserts the `tone → toneGlyph` mapping so a grayscale render distinguishes the tone; no banner copy uses standalone judgment words (`right`/`wrong`/`correct`). ✓
