## Phase 2 finding (the diagnosis this fix addresses)

Reproduced live via concurrent admin_validation (maxObservedConcurrency≈30): **3/30 = 10%** failures (matches the ~11% production burst), ALL typed `response_wrong_schema_version` / `mcp_validation_failed`, across 3 distinct families (argument_scheme, critical_question, evidence_source_chain), durations 5.9–9.1s (response-side slow-fail). The decisive evidence: `failureDetail.receivedKeys = ['isError','reason','path','detail']` — the MCP server returns its OWN error envelope `{ isError, reason, path, detail }` under concurrent load, and the Edge adapter mis-types it as `wrong_schema_version` (it lacks a `schemaVersion`). Root cause = **server-side under load** (the operator-hosted MCP server's error signal); the Edge mis-classification is the secondary bug. Cross-family uniformity → a shared server-under-load behavior, not a per-family prompt issue.

## Stage 2B decision (operator-approved)

Detect the `{isError}` envelope BEFORE schema validation, type it correctly as a provider/server error, and **retry it narrowly**. The retry/dispatch change is approved BY THIS DECISION.
