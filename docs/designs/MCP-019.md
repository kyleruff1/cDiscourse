## 0. Issue-vs-code reconciliation (read this first)

The issue body assumes a tidy `classifyMove тЖТ refereeLedger тЖТ refereeBanners тЖТ semanticOverride` chain. The merged code is mostly there, but **three of its assumptions do not match the actual module shapes**. The implementer must build to this section, not the issue's prose.

### Defect 1 тАФ `reconcileMove` cannot be called from the room with the data the room has today

The issue says MCP-019 "feeds the returned packet into MCP-013's `refereeLedger` adapter." But MCP-013's `reconcileMove(input: LedgerMoveInput)` is **not a thin packet adapter** тАФ `LedgerMoveInput` (verified in `src/features/refereeLedger/types.ts`) requires:

```
pointId, moveArgumentId, semanticPacket?, deterministicMetadata: DeterministicMoveMetadata,
moveRole: 'challenge' | 'repair', economyInput: ChallengeGradingInput | RepairGradingInput,
repairOptions?, debtLedger: IssueDebtLedger, creditStates?
```

The argument room **does not assemble** a `ChallengeGradingInput` / `RepairGradingInput`, an `IssueDebtLedger`, a `DeterministicMoveMetadata`, or a `moveRole` today тАФ MCP-013's own design (┬з"Data model", "Risks") states the point-standing economy and the ledger are **"not auto-wired into the live argument room"** and that wiring is "a separate future stage." Wiring the *full* economy-driven ledger is a large, separate card (it pulls in `src/features/pointStanding/` grading, debt ledgers, credit-state threading) and is **explicitly out of scope** for a P2 mock-mode UI card.

**Resolution.** MCP-019 introduces a thin, packet-only **`buildBannerSelectionInputFromPacket`** adapter and does **not** call `reconcileMove`. The issue's wording ("feeds the packet into MCP-013's `refereeLedger` adapter") is satisfied in spirit: MCP-013's *consumable contract surface* тАФ `CategoryReading` / `RefereeFeedbackCode` / `ReconciliationOutcome` тАФ is what `BannerSelectionInput.categoryReadings` is typed against, and MCP-014's `selectBanner` is fed directly from the packet's positive binaries. Concretely:

- When **no** `LedgerResult` is available (the v1 reality тАФ the economy/ledger is not wired into the room), `BannerSelectionInput.categoryReadings` is `[]` and `BannerSelectionInput.positiveBinaries` is built from `packet.binaries.filter(b => b.value === 1)`. This is exactly the input shape MCP-014 ┬з"Out of scope" names: *"The upstream builder that assembles `BannerSelectionInput` from a `SemanticRefereePacket` + `LedgerResult` тАж is a room-wiring concern."* MCP-019 **is** that room-wiring card for the banner path.
- The override path (MCP-015's `evaluateSemanticOverridePrompt`) accepts `ledgerResult?: LedgerResult` as **optional** and triggers purely off `packet` confidence when the ledger is absent тАФ verified in `overrideTriggerModel.ts`. So MCP-019 calls `evaluateSemanticOverridePrompt({ packet, viewerActorRole, repeatedSignal })` with `ledgerResult` omitted.

The design records this as the honest scope line: **MCP-019 wires the packet тЖТ banner тЖТ override path. It does NOT wire the economy-driven `reconcileMove` ledger.** A future card ("referee-ledger room wiring") does that, alongside the point-standing economy. MCP-019's `buildBannerSelectionInputFromPacket` is forward-compatible: when that card lands it passes a real `LedgerResult.categoryReadings` into the same builder.

### Defect 2 тАФ there is no client-side redaction helper; `classifyMove` requires `moveBodyRedacted`

`classifyMove`'s `ClassifyMoveRequest` (verified in `src/lib/edgeFunctions.ts`) requires `moveBodyRedacted` and `parentBodyRedacted?`, and the wrapper's own doc says *"Bodies MUST be redacted before calling this wrapper; the function runs a defensive second redaction pass."* The Edge Function boundary has `supabase/functions/_shared/semanticReferee/redaction.ts` (Deno-side), but **there is no `src/`-side redaction module** тАФ `grep` confirms `classifyMove` has zero callers today and no client redaction helper exists.

**Resolution.** MCP-019 adds a small, pure-TS **`src/features/semanticReferee/clientRedaction.ts`** тАФ a conservative client-side first-pass redactor (strips `@handle`-shape tokens, URLs, email-shape, long digit runs that look like IDs/phone numbers). It does **not** duplicate the Deno boundary redactor's algorithm byte-for-byte; it is a defensive client pass so a raw handle/URL never leaves the device, and the boundary's `redaction.ts` remains the authoritative second pass. This is the same belt-and-suspenders posture `submit-argument` already uses (client validation + Edge re-validation). The helper is pure (no network, no React) and tested with a ban-list / leak scan.

### Defect 3 тАФ the cache is owned by the Edge Function, not by the client; the "in-memory LRU cache" the issue cites is server-side

The issue's acceptance criterion says "the in-memory LRU cache is consulted first." MCP-012's `SemanticPacketCache` is a pure-TS class, but MCP-012's design (┬з"Integration seam", ┬з"Operator steps") states the cache is instantiated **inside MCP-016's Edge Function** тАФ *"the cache store the Edge Function will instantiate"* тАФ keyed by `contentHash` + `promptVersion`. A posted argument body is immutable, so the server-side cache already de-duplicates repeat classifications of the same move.

**Resolution.** MCP-019 adds a **second, client-side `SemanticPacketCache` instance** scoped to the room session (in `useState` / a `useRef`, room-scoped, capacity 256, the MCP-012 default). Its purpose is **call-rate reduction across the room's render lifecycle** тАФ if the user navigates away from a node and back, or the room re-renders, the client cache returns the already-fetched packet for that `(roomId, moveId, contentHash, classifier set)` key instead of re-invoking the Edge Function. The client cache and the server cache are independent and complementary: the client one cuts redundant *network round-trips*, the server one cuts redundant *provider calls*. MCP-019 uses MCP-012's `SemanticPacketCache` + `buildSemanticCacheKey` verbatim тАФ it adds no new cache type. The cache is consulted **before** `evaluateTrigger` is allowed to result in a `classifyMove` call (cache hit тЗТ no trigger-driven call at all).

These three resolutions keep MCP-019 a genuine, shippable, P2-sized mock-mode card. The implementer must not try to wire the full economy ledger or re-implement boundary redaction.

## 1. Goal (one paragraph) and scope

MCP-019 makes the dormant semantic-referee capability **observable in a real room** in mock mode. It threads one `SemanticRefereePacket` from the argument-room surface: a room hook calls `classifyMove` **only** when MCP-012's `evaluateTrigger` returns `allowed: true` for a `post_submit` moment **and** the client-side `SemanticPacketCache` has no hit; the requested classifier set is planned through MCP-012's `planClassifierBatches` (тЙд 5 per batch, тЙд 2 batches per `post_submit`); the returned packet is turned into a `BannerSelectionInput` by a new packet-only adapter and rendered through MCP-014's `selectBanner` as one short, non-verdict banner; and MCP-015's `evaluateSemanticOverridePrompt` is run on the same packet so a low-confidence routing binary surfaces a small, reversible lane-choice sheet. The doctrine that shapes every line: the semantic referee is **OFF by default** and `{ enabled: false }` for *any* reason (disabled, not deployed, key missing, rate limited, validation failed) falls back to the deterministic layer-1 result with **no error shown to the user**; an ordinary post is **never blocked** by a semantic result and **score never blocks posting**; no banner ever shows a verdict / winner / truth / person-label string; the override choice is **never a penalty** тАФ it moves no score punitively; and **no AI call is ever made from the client** тАФ `classifyMove` calls the Edge Function, which (only when the operator has flipped the provider flag) calls Anthropic. MCP-019 ships two new pure-TS adapter/state modules, two new thin RN render components (the deferred MCP-014 banner component and the deferred MCP-015 choice sheet, both named as follow-ups in their parent designs), and the room-hook wiring.

### In scope

- A new room hook `useSemanticReferee` (`src/features/semanticReferee/useSemanticReferee.ts`) тАФ owns the client-side `SemanticPacketCache`, the `RepeatedOverrideSignal`, the trigger evaluation, the `classifyMove` call, and the resulting packet/banner/override state.
- A new pure-TS adapter `buildBannerSelectionInputFromPacket` (`src/features/refereeBanners/bannerSelectionInputFromPacket.ts`) тАФ turns a `SemanticRefereePacket` (and, forward-compatibly, an optional `LedgerResult`) into a `BannerSelectionInput`.
- A new pure-TS client redactor `src/features/semanticReferee/clientRedaction.ts` тАФ a conservative first-pass body redactor (Defect 2).
- A new pure-TS module `src/features/semanticReferee/semanticTriggerInput.ts` тАФ builds a `TriggerEvaluationInput` and the per-`post_submit` classifier set from room state (so the room component does not hand-assemble MCP-012 inputs).
- The deferred **MCP-014 render component** `RefereeBannerView.tsx` тАФ a thin `<View>`/`<Text>` that renders a `BannerSelectionResult.banner` (MCP-014 ┬з"Out of scope" explicitly defers it; MCP-019 is its home).
- The deferred **MCP-015 choice sheet** `SemanticOverrideChoiceSheet.tsx` тАФ the thin RN-primitive sheet MCP-015 ┬з8 fully specified and deferred (MCP-019 is its home).
- Wiring all of the above into `ArgumentGameSurface.tsx` + `ArgumentTreeScreen.tsx` so a posted move can show a referee banner and (on low-confidence) an override sheet, without a route transition.
- Tests тАФ see ┬з9.

### Explicitly out of scope

- **The economy-driven `reconcileMove` ledger wiring** тАФ Defect 1. `reconcileMove` needs the point-standing economy assembled into the room (`ChallengeGradingInput`, `IssueDebtLedger`, `moveRole`, `DeterministicMoveMetadata`). That is a separate, larger card. MCP-019 wires the packetтЖТbannerтЖТoverride path only and passes `categoryReadings: []`.
- **Enabling the live provider.** `SEMANTIC_REFEREE_ENABLED` / `SEMANTIC_REFEREE_PROVIDER` are operator server secrets. MCP-019 never sets them, never deploys the function, makes no live Anthropic call. Mock mode is the v1 default for this surface.
- **Any change to the `semantic-referee` Edge Function, the provider registry, the boundary redactor, or the packet contract.** MCP-019 is pure client code.
- **`pre_send_review`, `branch_routing`, `synthesis_readiness`, `evidence_inspection` triggers.** MCP-019 wires exactly one moment тАФ `post_submit` тАФ the trigger of record. The other moments are valid future wiring; MCP-019 keeps the surface small. (`evaluateTrigger` still refuses them correctly if a future caller passes them.)
- **Persisting the `SemanticOverrideRecord` to Supabase.** MCP-015's record is in-memory (the META-001 in-memory-v1 / persist-v2 split). MCP-019 keeps it in-memory; persistence is a later card.
- **A migration, a new table, a new RLS policy, a new Edge Function, a new dependency.** None.
- **v1-scope-excluded features** тАФ no voting, no winner-producing scoring, no argument search, no push notifications, no OAuth, no public API. MCP-019 introduces none.

## 2. Doctrine constraints (the lines a reviewer enforces)

Any violation is a blocker.

1. **Mock is the default; the live provider is an operator flag flip.** MCP-019 reads no `SEMANTIC_REFEREE_*` env var, sets none, deploys nothing. When the operator has not enabled the layer (the v1 default), `classifyMove` returns `{ enabled: false, reason: 'disabled' | 'not_configured' }` and MCP-019 renders no banner and offers no override sheet. The live `anthropic` provider is reached only after an operator flips `SEMANTIC_REFEREE_PROVIDER=anthropic` тАФ never by this card.
2. **`{ enabled: false }` for ANY reason falls back to deterministic layer 1 with no user-visible error.** The nine `ClassifyMoveDisabledReason` values (`disabled`, `not_configured`, `not_implemented`, `key_missing`, `api_error`, `rate_limited`, `network_error`, `parse_failure`, `validation_failed`) and a thrown/`ok:false` wrapper result are **all** treated identically: the room shows whatever it already shows (the deterministic layer-1 timeline / surface), and **no toast, no error banner, no red text, no console.error** appears. The `useSemanticReferee` hook never surfaces an error string to the UI.
3. **An ordinary post is never blocked by a semantic result; score never blocks posting.** `classifyMove` runs **after** `submit-argument` has already posted the move (the `post_submit` trigger). The room hook never gates the composer, never gates `submit-argument`, never delays a post on a pending classification. A pending or failed classification has zero effect on posting.
4. **No banner shows a verdict / winner / truth / person-label string.** The banner copy comes entirely from MCP-014's `REFEREE_BANNER_LIBRARY` (already ban-list-tested in MCP-014). MCP-019's `RefereeBannerView.tsx` renders only `banner.headline` / `banner.helperLine` / `banner.accessibilityLabel` тАФ it authors **zero new user-facing strings**, so it cannot introduce a banned token. A ban-list test still scans the component's rendered output as a backstop.
5. **The override choice never moves score punitively.** MCP-015's `SemanticOverrideRecord` / `SemanticOverridePrompt` / `RepeatedOverrideSignal` have **no** `delta` / `score` / `penalty` / `block` / `flag` field (MCP-015-enforced). MCP-019's choice sheet writes only an in-memory `SemanticOverrideRecord` via `buildSemanticOverrideRecord`; it never calls a scoring path, never writes a `flags` row, never delays or marks the move.
6. **No AI runs on the client.** The only outbound call MCP-019 makes is `classifyMove тЖТ supabase.functions.invoke('semantic-referee')`. The Edge Function тАФ and only the Edge Function, only when the operator has flipped the provider flag тАФ calls Anthropic. `useSemanticReferee.ts` and every new module import no provider SDK, no `fetch` to a model host. A source-scan test enforces it.
7. **Cost is bounded by MCP-012's gates.** A `classifyMove` call fires only when (a) the client `SemanticPacketCache` misses, **and** (b) `evaluateTrigger` returns `allowed: true`. The classifier set is `planClassifierBatches`-planned (тЙд 5 per batch); the per-`post_submit` set spans тЙд 2 groups (тЙд 10 classifiers). MCP-019 adds no new caller and no new trigger beyond `post_submit`.
8. **An ordinary draft keystroke / edit is NOT a trigger.** `evaluateTrigger` is only ever called by MCP-019 with `event.kind: 'trigger', moment: 'post_submit'` тАФ after a successful post. MCP-019 never calls `classifyMove` on a composer keystroke, a draft edit, a node tap, a scroll, an observer browse, or a mode toggle. (Those are MCP-012's `SemanticForbiddenEvent`s; MCP-019 simply never originates them.)
9. **Plain language.** The banner library and override copy are already routed through `gameCopy.toPlainLanguage` (MCP-013/014/015). MCP-019 surfaces no raw `classifierId` / `reasonCode` / `feedbackCode` / `bannerCode`. The override sheet's strings come from MCP-015's `PLAIN_LANGUAGE_COPY` entries.
10. **Observer safety.** An observer is never offered the override sheet (`evaluateSemanticOverridePrompt` returns `shouldOffer: false` for `observer` тАФ MCP-015 actor gate). An observer's room never *originates* a `post_submit` classification because an observer cannot post. A banner *may* be shown read-only to an observer viewing a participant's classified move тАФ a banner is doctrine-safe regardless of who reads it (MCP-014 ┬з"Edge cases").

## 4. MCP-012 trigger-gate integration

The exact integration point, what is *not* a trigger, the batching, and the cache-first order.

### 4.1 Where the gate is consulted

`evaluateTrigger` is called in **exactly one place**: inside `useSemanticReferee.onMovePosted`, which `ArgumentTreeScreen` invokes from its **existing** `handleSubmitSuccess` path (the callback that already fires after `submit-argument` posts a move and the room refreshes). The `TriggerEvaluationInput` is built by the new pure-TS `semanticTriggerInput.ts`:

```ts
// src/features/semanticReferee/semanticTriggerInput.ts
export function buildPostSubmitTriggerInput(args: {
  roomId: string;
  moveId: string;
  parentId?: string;
  featureLayerEnabled: boolean;            // see ┬з6 тАФ derived, not an env read
  semanticClassificationMode: SemanticClassificationMode;  // see ┬з6
  actorRole: SemanticActorRole;            // mapped from the room's participant side
}): TriggerEvaluationInput;
// returns { event: { kind: 'trigger', moment: 'post_submit' }, preSendReviewOptIn: false, ... }
```

`onMovePosted` then does, in order: redact тЖТ cache key тЖТ **cache lookup** тЖТ `evaluateTrigger` тЖТ `planClassifierBatches` тЖТ `isWithinBudget` тЖТ `classifyMove`. If `evaluateTrigger` returns `allowed: false` (layer disabled, room mode off, observer/moderator role), `onMovePosted` returns immediately тАФ **no `classifyMove` call**, the room keeps its deterministic layer-1 surface.

### 4.2 What is NOT a trigger

MCP-019 calls `evaluateTrigger` **only** with `{ kind: 'trigger', moment: 'post_submit' }`, and **only** after a confirmed successful post. It never originates a classification on:

- a composer keystroke or any draft edit (the issue's named "ordinary draft edit" case),
- a node tap / timeline selection,
- a scroll, a focus/blur, a mode toggle (Stack тЖФ Timeline),
- an observer browsing the gallery or the room,
- a refresh that brings in *other* participants' messages (only the *local user's own just-posted move* triggers тАФ `onMovePosted` is called with the specific `moveId` returned by `submit-argument`, never for every message in a refresh).

These are MCP-012's `SemanticForbiddenEvent`s. MCP-019 does not even construct a `{ kind: 'forbidden' }` input тАФ it simply never calls the hook's `onMovePosted` for them. A test (`semanticRoomNoCallOnDraftEdit.test.ts`) drives a composer-edit / node-tap / mode-toggle sequence through the hook's public surface and asserts the `classifyMove` spy records **zero calls**.

### 4.3 The тЙд 5-classifier batching

`semanticTriggerInput.ts` exports a frozen `POST_SUBMIT_CLASSIFIER_SET` тАФ the curated classifier ids a `post_submit` moment requests. Per MCP-012's design, a single trigger's set spans **at most two batch groups** (тЙд 2 batches, тЙд 10 classifiers). The recommended `post_submit` set, drawn from MCP-012's `SEMANTIC_BATCH_GROUPS` group A (parent continuity) and group C (evidence pressure) тАФ the two groups most relevant to "did this just-posted move tie to its parent and is its evidence hygiene OK":

```
POST_SUBMIT_CLASSIFIER_SET (тЙд 10, spans groups A + C):
  responds_to_parent, quote_anchors_parent, answers_clarification, introduces_new_issue,   // group A
  asks_for_evidence, provides_evidence, evidence_supports_claim, creates_source_chain_gap,  // group C
  uses_popularity_as_evidence
```

`planClassifierBatches(POST_SUBMIT_CLASSIFIER_SET)` returns **two** `ClassifierBatch`es (one for group A's intersection, one for group C's), each тЙд `BATCH_CAP` (5). MCP-019 sends the **flattened** set as `ClassifyMoveRequest.requestedClassifiers` тАФ but `requestedClassifiers` is capped at `1..5` entries by `ClassifyMoveRequest` (verified in `edgeFunctions.ts`). **This is a real constraint:** MCP-019 must issue **one `classifyMove` call per batch** (тЙд 2 calls per `post_submit`), each call carrying that batch's тЙд 5 ids. The hook loops over the planned batches, calls `classifyMove` per batch, and merges the returned packets' `binaries` arrays into one combined positive-binary list before `selectBanner`. A test asserts no `classifyMove` call ever carries > 5 `requestedClassifiers` and that a `post_submit` issues тЙд 2 calls. (If the implementer prefers a single classifier set тЙд 5, that is also acceptable and simpler тАФ see ┬з10 operator decision OD-3 тАФ but the design's default honors the full A+C set via 2 calls because that is what the issue's "batching тЙд 5 classifiers" criterion implies.)

### 4.4 LRU-cache-first lookup order

The client `SemanticPacketCache` (Defect 3) is consulted **before** the trigger gate produces a call тАФ concretely, the cache lookup is step 2 and `classifyMove` is step 6 in ┬з3's flow. The key is MCP-012's `buildSemanticCacheKey` (imported verbatim), built from `roomId`, `moveId`, `contentHash`, the planned classifier set (via MCP-011's `hashClassifierSet`, which order-normalizes), and the `promptVersion` hint. Order:

1. **Cache hit** тЗТ use the cached packet, **do not call `evaluateTrigger`-into-a-call**, do not call `classifyMove`. (A posted body is immutable, so a hit is always the correct packet for that move + classifier set.)
2. **Cache miss** тЗТ run `evaluateTrigger`; if `allowed`, plan batches, budget-check, call `classifyMove` per batch.
3. **On a successful `{ enabled: true, packet }`** тЗТ `clientCache.set(key, packet)`. A `{ enabled: false }` or errored result is **never cached** (caching a degraded result would pin it past a provider/operator recovery тАФ MCP-012's documented cache contract).

Because the cache is per-room-session (created once per `ArgumentTreeScreen` mount, lives in a `useRef`), re-rendering the room or navigating between nodes never re-fires a classification for an already-classified move. A test asserts a second `onMovePosted` for the same `(moveId, contentHash, classifier set)` records zero additional `classifyMove` calls (cache hit).

## 6. Mock-mode default тАФ how MCP-019 stays OFF by default

### 6.1 The env-var boundary

`SEMANTIC_REFEREE_ENABLED` and `SEMANTIC_REFEREE_PROVIDER` are **Supabase function secrets**, read **only** inside the Edge Function via `Deno.env.get()` (MCP-016/MCP-017). They are **not** `EXPO_PUBLIC_`-prefixed and **never** reach the client. MCP-019's client code therefore **cannot and must not** read them. The client has no way to know whether the layer is on тАФ and it does not need to.

The client's `featureLayerEnabled` value passed to `evaluateTrigger` is **not** a mirror of the server secret тАФ it is a **client-side intent flag**, defaulting to a single named constant:

```ts
// src/features/semanticReferee/semanticTriggerInput.ts
/**
 * Client-side intent flag for the semantic-referee layer. This is NOT the
 * server's SEMANTIC_REFEREE_ENABLED secret (that lives only in the Edge
 * Function). It defaults to `true` so the client WILL attempt a classifyMove
 * тАФ and the SERVER is the real gate: when the operator has not enabled the
 * layer, the Edge Function returns `{ enabled: false }` and the client falls
 * back silently. Set to `false` only to disable the client attempt entirely
 * (a kill-switch for the room-wiring itself, independent of the server).
 */
export const SEMANTIC_REFEREE_CLIENT_ATTEMPT_DEFAULT = true;
```

This is the honest model: **the server is the authority.** The client always *attempts* (cheaply тАФ one gated call), and `{ enabled: false }` from the boundary is the universal "the operator has not turned this on" signal. Mock mode is the default because the *operator has not deployed/flipped anything* тАФ not because the client suppresses the call.

`semanticClassificationMode` (MCP-012's `'off' | 'metadata_only' | 'metadata_and_chip'`) is GAME-003's room-mode field. GAME-003 is **not merged** (it is open тАФ confirmed by MCP-012's design and the board). MCP-019 therefore passes `semanticClassificationMode` as a **client constant defaulting to `'metadata_and_chip'`** (so the trigger gate's mode check passes and the banner can render) тАФ and records GAME-003's absence as **operator decision OD-1** below. When GAME-003 lands, a one-line change reads the real per-room mode. Until then, the *server* `{ enabled: false }` gate is still the real off-switch, so defaulting the client mode to `metadata_and_chip` does not turn anything on by itself.

### 6.2 Graceful `{ enabled: false }` тЖТ layer-1 fallback, no user-visible error

`useSemanticReferee` collapses **every** non-success outcome into a single inert state:

| `classifyMove` result | `RefereeMoveState.status` | What the user sees |
|---|---|---|
| `{ ok: true, data: { enabled: true, packet } }` | `'ready'` | the banner (if `selectBanner` returns one) + override sheet (if low-confidence) |
| `{ ok: true, data: { enabled: false, reason } }` тАФ **any** of the 9 reasons | `'fallback'` | **nothing** тАФ the deterministic layer-1 surface, unchanged |
| `{ ok: false, error, status }` тАФ network / wrapper error | `'fallback'` | **nothing** |
| `classifyMove` promise rejects (defensive тАФ the wrapper "never throws", but belt-and-suspenders) | `'fallback'` | **nothing** |

`'fallback'` and `'idle'` render **identically** тАФ no banner, no sheet, no toast, no error text, no `console.error`, no retry button. The hook logs nothing user-visible. The `useSemanticReferee` public surface has **no error field** тАФ there is intentionally no way for a consumer to render an error, so a reviewer can prove the no-error rule structurally. A test (`semanticRoomFallback.test.ts`) drives each of the nine `reason` values plus an `ok:false` result through the hook and asserts `getMoveState(moveId)` is an inert `'fallback'` state and no error string is exposed.

This is the issue's hardest criterion and the ┬з2.2 doctrine line: **`{ enabled: false }` for any reason тАФ including `semantic-referee` not deployed at all тАФ is a normal, expected outcome, not an error.**

## 7. Edge cases

The implementer must handle each:

- **Low-confidence binary.** `evaluateSemanticOverridePrompt` returns `shouldOffer: true, triggerReason: 'low_confidence'` when a routing-relevant binary (`responds_to_parent`, `introduces_new_issue`, `suggests_side_branch`, `suggests_diagonal_tangent`) has `confidence: 'low'` (MCP-015 trigger algorithm step 4). MCP-019 renders `SemanticOverrideChoiceSheet`. The banner path *also* runs тАФ MCP-014's `selectBanner` softens or suppresses a low-confidence banner itself (MCP-014 confidence rule); MCP-019 does not need to special-case it.
- **Packet-vs-layer-1 conflict.** MCP-019 does **not** run the economy ledger (Defect 1), so a `LedgerResult.needsUserChoice` / `conflict_routed` reading is **not produced by this card**. The override sheet therefore triggers in MCP-019 **only** via the `low_confidence` path. The `l1_l2_conflict` path of `evaluateSemanticOverridePrompt` is correctly handled by MCP-015's model and will activate automatically when the future economy-ledger card passes a real `LedgerResult` into the hook тАФ MCP-019's `useSemanticReferee` already passes `ledgerResult` through (currently `undefined`), so no rework is needed then. **This is the honest scope line and the implementer must state it:** in MCP-019 the override sheet is a *low-confidence* surface; the *conflict* surface lights up with the ledger card. (The issue lists "packet-vs-layer-1 conflict" as a trigger тАФ that requires the layer-1 ledger comparison, which is the out-of-scope economy wiring.)
- **`semantic-referee` function not deployed.** `supabase.functions.invoke('semantic-referee')` for a function that does not exist returns a `FunctionsFetchError` тЖТ `classifyMove` returns `{ ok: false, status: 503 }` тЖТ `'fallback'` state тЖТ **nothing shown, no error.** Identical handling to `{ enabled: false }`.
- **Batching overflow.** `POST_SUBMIT_CLASSIFIER_SET` has 9 ids spanning 2 groups; `planClassifierBatches` returns 2 batches of тЙд 5. MCP-019 issues one `classifyMove` per batch (┬з4.3) so no single request exceeds the `requestedClassifiers: 1..5` cap. If a future change grows the set past 2 groups, `planClassifierBatches` still partitions it; the hook still loops per batch. A test asserts no request carries > 5 ids.
- **Cache hit vs miss.** Miss тЖТ gated `classifyMove`. Hit тЖТ the cached packet, no call, no trigger-into-call. A re-render, a node re-selection, or a second `onMovePosted` for the same `(moveId, contentHash, classifier set)` is a hit (a posted body is immutable). A *different* move is a different key тЖТ a miss тЖТ a fresh gated call.
- **A packet arriving after the user moved on / left the room.** `classifyMove` is async; the user may select a different node or unmount the room before it resolves. The hook guards the `setState` with a mounted-ref check (standard RN pattern) тАФ a resolved packet for an unmounted room is dropped. A resolved packet for a *still-mounted* room but a *non-active* move is still stored in `refereeStateByMoveId[moveId]`; it simply does not render until/unless that move becomes active again (and then it is a cache hit, instant). No stale banner flashes on the wrong node тАФ the banner is keyed strictly to the active move id.
- **Observer posts? тАФ impossible.** An observer cannot post a move, so `onMovePosted` is never called for an observer's own action. An observer *viewing* a participant's already-classified move: the banner renders read-only (doctrine-safe тАФ MCP-014); the override sheet does **not** (`evaluateSemanticOverridePrompt` actor gate тЖТ `shouldOffer: false` for `observer`).
- **Empty / whitespace move body.** `submit-argument` already rejects empty bodies deterministically; `onMovePosted` is only called on a *successful* post, so the body is non-empty. `clientRedaction.redactBody` on a short body is a no-op-ish pass; `ClassifyMoveRequest.moveBodyRedacted` requires `1..8000` chars тАФ a successful post guarantees тЙе 1.
- **Over-budget body.** `isWithinBudget` (MCP-012) over an 8000-char body that with classifier overhead exceeds `SEMANTIC_PACKET_TOKEN_BUDGET` тЖТ MCP-019 skips the `classifyMove` call for that batch тЖТ `'fallback'` тЖТ nothing shown. The post is unaffected (it already happened). MCP-019 does **not** truncate the body to fit (truncation classifies a different text тАФ MCP-012's documented rule).
- **Two posts in quick succession.** Each successful post calls `onMovePosted` with its own `moveId`; each is an independent cache key and an independent gated call. `refereeStateByMoveId` holds both. No race тАФ the keys differ.
- **Concurrent edits.** Argument bodies are immutable after posting (soft-delete only тАФ CLAUDE.md). `contentHash` is stable; a move is classified at most once per `(classifier set, promptVersion)`. No concurrent-edit race.
- **Reduce-motion on.** `RefereeBannerView` and `SemanticOverrideChoiceSheet` both honor `reduceMotionOverride` тАФ no slide/fade; the banner appears statically, the sheet snaps. The non-color glyph + text remain.
- **Doctrine edge тАФ "could a slow classification delay the post?"** No. `classifyMove` runs strictly *after* `submit-argument` succeeds; `onMovePosted` is fire-and-forget from `handleSubmitSuccess`'s perspective тАФ the composer is already cleared, the move is already on the timeline. The classification populates a banner *later*; it never gates anything.
- **Doctrine edge тАФ "could heat/popularity reach the classifier?"** No. `ClassifyMoveRequest` carries the redacted move body, the parent body, room context (mode / action / type / side), and the classifier ids тАФ no engagement/view/like/heat field exists in the request shape. The `uses_popularity_as_evidence` classifier *flags* popularity-as-evidence inside the move's text; it never grants standing.

## 11. Does MCP-019 need an Edge Function deploy or a DB migration?

**No to both тАФ and this is load-bearing, so it is stated plainly.**

- **No Edge Function deploy.** MCP-019 writes no Edge Function code. The `semantic-referee` function already exists (MCP-016) and already has the `anthropic` live provider (MCP-017). MCP-019 only *calls* it via the existing `classifyMove` wrapper. If the operator has never deployed `semantic-referee`, MCP-019 still ships and behaves correctly тАФ `invoke('semantic-referee')` returns a fetch error, which MCP-019 treats as `{ enabled: false }` тЖТ silent layer-1 fallback. The Edge Function deploy + the provider-flag flip remain **MCP-017's operator steps**, unchanged and independent of MCP-019.
- **No DB migration.** MCP-019 adds no table, no column, no RLS policy, no index. The `SemanticOverrideRecord` is in-memory (`useState`, the META-001 in-memory-v1 model); persisting it is an explicitly future card. The client `SemanticPacketCache` is in-memory (`useRef`). Nothing is written to Postgres by this card.

If the implementer finds they need a migration or an Edge Function change to make MCP-019 work, that is a signal the scope has drifted тАФ most likely into the out-of-scope economy-ledger wiring (Defect 1) or into persisting the override record. Stop and re-scope.
