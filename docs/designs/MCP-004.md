## Goal (one paragraph)

MCP-004 designs **exactly when a semantic-referee classifier call is allowed and forbidden**, and the **caching, batching, retry/fallback, and token-budget plan** that keeps the layer cheap. MCP-001 locked *what* the layer returns (the `SemanticRefereePacket`); MCP-009 locked *where* the call physically lives (the `semantic-referee` Edge Function, default provider `mock`, disabled-by-default). This card locks *when* — and *how often, how batched, how bounded* — that boundary's call path may fire. It exists because the roadmap expansion §7 enumerates the allowed and forbidden trigger moments and ends with "This is owned by MCP-004"; this doc formalizes §7 into a full design. The design is shaped by doctrine constraints that are blockers if violated: (1) **the classifier is advisory — gating it never blocks an ordinary post**: a trigger that does not fire, or a call that fails, must degrade silently to the deterministic layer-1 result, never to a rejected post (cdiscourse-doctrine §1, §4; MCP-001 §3); (2) **no call ever runs on the client** — every trigger evaluated here results, when it fires, in a call to the `semantic-referee` Edge Function, never a direct provider call (cdiscourse-doctrine §4, §7; MCP-009); (3) **cost control is a doctrine concern** — "an uncapped layer is a failed layer"; trigger gating, caching, batching, and a token budget together cap the spend, and any uncapped path is a design defect (roadmap §13; MCP-001 §13); (4) **never on keystroke / hover / timeline-selection / observer-browsing** — these are the false-positive, cost-blowout, and "feels invasive" failure modes the card must structurally prevent (card §2; MCP-001 §5). This card does **not** redesign the packet contract (MCP-001), the Edge Function boundary (MCP-009), or the seed bank (MCP-002) — it references them and gates the call path between them.

## Data model

**No new database table. No migration. No new persisted entity.**

The card's caching plan is **process-scoped, not persisted** — consistent with MCP-001 §12 ("a persisted cache table is a future-card option, not v0") and MCP-009's data-model section ("whether a packet is ever persisted to a cache table is explicitly an MCP-004 decision … This card does not introduce one"). **This card explicitly decides: v0 uses an in-Edge-Function in-memory cache only; no cache table, no migration.** A persisted cross-instance cache is a documented future option (§"Risks").

This card defines two new pure-TS data shapes — a **trigger descriptor** and a **cache key** — that the future implementation card builds. Both are JSON-serializable, free of network/React/Supabase imports (they live in a pure model file, see "File changes"), and are *transient* — never written to any table.

```ts
// Future implementation card — shape only. Pure TS, no I/O.

/** The six moments at which a semantic call MAY originate. */
export type SemanticTriggerMoment =
  | 'post_submit'              // after a move is successfully posted (default)
  | 'pre_send_review'          // strict-mode only, room-creator opt-in
  | 'evidence_inspection'      // on evidence attach / challenge
  | 'branch_routing'           // when reply-to-parent is deterministically ambiguous
  | 'synthesis_readiness'      // once enough lifecycle events exist on a cluster
  | 'referee_feedback';        // banner generation from already-classified metadata only

/** Events the gate sees but MUST refuse — present so the gate is exhaustive. */
export type SemanticForbiddenEvent =
  | 'keystroke'                // draft edit / every character
  | 'hover'                    // pointer hover on any surface
  | 'timeline_selection'       // selecting a node on the horizontal timeline
  | 'observer_browsing'        // observer / read-mode browsing a room
  | 'scroll'
  | 'focus'
  | 'blur'
  | 'gallery_browsing';        // browsing the Conversation Gallery

/** The advisory mode a room is in, sourced from GAME-003's room mode field.
 *  Reproduced here as the input the gate reads; GAME-003 owns the canonical type. */
export type SemanticClassificationMode =
  | 'off'                      // never call for this room
  | 'metadata_only'            // post-submit + structural triggers allowed; no pre-send
  | 'metadata_and_chip';       // additionally allows pre-send review (strict)

/** What the gate is asked about. Pure input — no body text, no network. */
export interface TriggerEvaluationInput {
  event:
    | { kind: 'trigger'; moment: SemanticTriggerMoment }
    | { kind: 'forbidden'; event: SemanticForbiddenEvent };
  roomId: string;
  parentId?: string;
  moveId?: string;
  /** GAME-003 room mode — when absent, treated as 'off' (fail-closed). */
  semanticClassificationMode: SemanticClassificationMode;
  /** True only if the room creator explicitly opted into pre-send review. */
  preSendReviewOptIn: boolean;
  /** Layer-feature flag mirror (MCP-009 SEMANTIC_REFEREE_ENABLED). The gate
   *  is the FIRST line of defense; the Edge Function's disabled path is the second. */
  featureLayerEnabled: boolean;
  /** Actor role — observers never trigger a call. */
  actorRole: 'initiator' | 'primary_opponent' | 'chime_in' | 'observer' | 'moderator';
  /** Deterministic ambiguity flag (layer 1) — branch_routing only fires when true. */
  branchRouteAmbiguous?: boolean;
  /** Lifecycle-event count on the cluster — synthesis_readiness threshold gate. */
  clusterLifecycleEventCount?: number;
}

export type TriggerDecision =
  | { allowed: true; moment: SemanticTriggerMoment; reason: string }
  | { allowed: false; reason: string };   // reason is a bounded code, plain-mapped before UI

/** The cache key — the exact tuple from the card scope, MCP-001 §12, MCP-009 §data-model. */
export interface SemanticCacheKey {
  roomId: string;
  parentId: string;            // '' (empty sentinel) for a root move with no parent
  contentHash: string;         // hash of the redacted move body (MCP-001 SemanticRefereePacket.contentHash)
  promptVersion: string;       // seed-bank prompt revision (MCP-001 §18)
  classifierSet: string;       // canonical, ORDER-NORMALIZED join of the requested classifier ids
}

/** A cache entry. value is MCP-001's packet, unchanged. */
export interface SemanticCacheEntry {
  key: SemanticCacheKey;
  packet: import('./types').SemanticRefereePacket;  // MCP-001 contract
  storedAtMs: number;          // for in-memory LRU eviction, NOT for time-based invalidation
}
```

`classifierSet` is the **order-normalized** serialization of the requested classifier ids (e.g. the ids sorted lexically and joined with `,`). Two requests for the same classifiers in a different order must produce the **same** key — otherwise the cache misses a guaranteed-identical packet. This is a load-bearing detail the future card must test.

`parentId` uses an explicit empty-string sentinel (not `undefined`) inside the key so the serialized key string is stable and total — a root move and a missing parent both key the same way, and the key never depends on `undefined`-vs-`''` JSON quirks.

## API / interface contracts

### 1. Allowed trigger moments

A semantic call MAY originate **only** at these six moments. Each is `allowed: true` from `evaluateTrigger` **only when** the layer feature flag is on, the room's `semanticClassificationMode` permits it, and the per-moment precondition holds.

| `SemanticTriggerMoment` | Fires when | Per-moment precondition | Default | Mode requirement |
|---|---|---|---|---|
| `post_submit` | A move was **successfully posted** (the `submit-argument` Edge Function returned success). The move row already exists; nothing is blocked. | `actorRole` is a participant (not `observer`); the post succeeded. | **On by default** — *if* post-submit classification is operator-defaulted on (operator decision §"Operator decisions"). | `metadata_only` or `metadata_and_chip`. |
| `pre_send_review` | The composer is about to send and the room is a **strict mode** the **room creator explicitly opted into**. | `semanticClassificationMode === 'metadata_and_chip'` **AND** `preSendReviewOptIn === true`. The packet only feeds RULE-004's advisory `PreSendReviewSheet`; it **never hard-gates** (MCP-001 §4, §17). | **Off by default.** Opt-in only. | `metadata_and_chip` only. |
| `evidence_inspection` | A user **attaches or challenges** an evidence artifact (EV-001 / EV-002 surfaces). Classifies whether the evidence attaches to the claim. | An evidence attach/challenge action just completed. | On when the layer is on. | `metadata_only` or `metadata_and_chip`. |
| `branch_routing` | The deterministic layer-1 **cannot tell** whether a move replies to the parent or redirects. | `branchRouteAmbiguous === true` — layer 1 already decided it is ambiguous. If layer 1 is confident, **no call** (layer 1 wins on its own turf, MCP-001 §6). | On when the layer is on. | `metadata_only` or `metadata_and_chip`. |
| `synthesis_readiness` | **Enough lifecycle events** exist on a cluster to ask whether synthesis is genuinely ready. | `clusterLifecycleEventCount >= SYNTHESIS_TRIGGER_MIN_EVENTS` (a small constant, recommended `6` — operator-tunable, §"Operator decisions"). Below threshold → no call. | On when the layer is on. | `metadata_only` or `metadata_and_chip`. |
| `referee_feedback` | A referee banner (MCP-008) needs generating. | Generated **from already-classified metadata only** — it reads packets that already exist in the cache / ledger. **It never originates a fresh provider call.** If no classified metadata exists, the banner is simply not shown. | On when the layer is on. | `metadata_only` or `metadata_and_chip`. |

Three structural rules bind all six:

- **Each allowed moment is event-driven and discrete** — it corresponds to a *completed user action* (a post, an attach, a send-intent, a lifecycle accumulation), never to a continuous input stream.
- **`referee_feedback` is special: it is a derived trigger, not a provider trigger.** It consumes already-classified metadata; it can never be the *cause* of a provider call. The gate returns `allowed: true` for it but the consuming code path reads the cache/ledger, not the Edge Function. This is the structural guarantee that banners cannot cause a cost.
- **A room whose `semanticClassificationMode` is `'off'` produces `allowed: false` for every moment** — fail-closed. An absent mode is treated as `'off'`.

### 2. Forbidden trigger moments

`evaluateTrigger` returns `allowed: false` — unconditionally, before any mode/flag check — for every `SemanticForbiddenEvent`. These are not "off by default"; they are **structurally unreachable** as a provider trigger.

| Forbidden event | Why it is forbidden | What happens instead |
|---|---|---|
| `keystroke` (every draft-edit character) | Per-keystroke calls are the cost-blowout failure mode (roadmap §13) and would re-create the magic-keyword cage (roadmap §2). | Nothing. The draft is local until the user posts (→ `post_submit`) or explicitly invokes pre-send review in an opted-in strict room. |
| `hover` | Hover is ambient, high-frequency, and not a user decision. | Nothing. |
| `timeline_selection` | Selecting a node on the horizontal timeline is *navigation*, not a move. Stage 6.3's `ConversationMiniTimeline` / Stage 6.1.8 timeline scrubber must never trigger a call. | The IX-004 "you are here" readout uses **already-classified** metadata only (MCP-001 §17). |
| `observer_browsing` | An observer in read mode (Stage 6.4 default entry) is consuming, not playing. Classifying their browsing is invasive and pays for nothing. | Nothing — `actorRole === 'observer'` also independently blocks every *allowed* moment. |
| `scroll`, `focus`, `blur`, `gallery_browsing` | Same family as hover/selection — ambient UI events, not move decisions. Listed so the gate's forbidden enum is exhaustive and the future card's test table is complete. | Nothing. |

**Belt-and-suspenders:** even if a caller wires a forbidden event by mistake, the gate refuses it; even if the gate were bypassed, MCP-009's Edge Function still requires a valid `ClassifyMoveRequest` (a posted-or-send-intent move with redacted bodies) and the disabled-by-default flag. There is no single point whose failure produces a keystroke-driven call.

### 3. Caching

**Cache key:** `{ roomId, parentId, contentHash, promptVersion, classifierSet }` — the exact tuple from the card scope, MCP-001 §12, and MCP-009's data-model section.

```ts
// Future implementation card — shape only.
export function serializeSemanticCacheKey(key: SemanticCacheKey): string {
  // ORDER-NORMALIZED, total, stable. parentId '' sentinel for root moves.
  return [
    key.roomId,
    key.parentId || '',
    key.contentHash,
    key.promptVersion,
    key.classifierSet,           // already order-normalized by planClassifierBatches
  ].join('');              // unit-separator — cannot appear in any component
}
```

Cache rules:

- **Never call twice for the same key.** Before `classifyWithConfiguredProvider` selects a provider, it computes the key, calls `cache.get(key)`, and on a hit returns the cached `SemanticRefereePacket` with **no provider call**. This is the primary per-move cost guarantee: each unique `{move body, prompt version, classifier set}` is classified **at most once**.
- **A move body is immutable after posting.** `arguments` rows are soft-deleted, never edited (cdiscourse-doctrine §8). So `contentHash` for a posted move is stable forever — a `post_submit` packet for a given move is a permanent cache target within the prompt version.
- **Pre-send drafts are not cached.** A draft body changes until it is sent; there is no stable `contentHash` and no `moveId`. A `pre_send_review` call computes its key from the *current draft snapshot at send-intent*; if the user edits and re-invokes, that is a different `contentHash` and a legitimately different (cheap, opt-in, strict-mode-only) call. The future card MUST NOT cache pre-send drafts under a moveId they do not have.
- **Invalidation is on `promptVersion` bump ONLY.** There is **no time-based TTL** and **no manual invalidation API**. A re-worded prompt may classify the same body differently (MCP-001 §18), so a `promptVersion` bump changes the key and old entries become unreachable — they are *naturally* invalidated by never being looked up again, not deleted. The in-memory store's LRU eviction (capacity cap) handles memory; it is **not** a correctness mechanism and **must not** be confused with invalidation. `storedAtMs` exists only for LRU ordering.
- **Cache scope (v0):** an **in-memory `Map`-backed store at Edge Function module scope** (MCP-009 §providers.ts). It survives across invocations of one *warm* instance and is naturally empty on a cold start. This is a deliberate v0 choice: it is correct (a miss just re-classifies — same packet, only a cost), needs no migration, and no RLS surface. A **persisted cross-instance cache table** is a documented future option (§"Risks") — explicitly NOT introduced here, consistent with MCP-001 §12 and MCP-009.
- **Capacity:** the in-memory store is bounded (recommended **256 entries per warm instance**, operator-tunable but not an operator *decision* — a sane default). On overflow, LRU eviction. An evicted entry just causes a future re-classification (a cost, not a bug).
- **`classifierSet` normalization** (repeated for emphasis because it is the most likely cache-correctness bug): the requested classifier ids are **sorted** before being joined into `classifierSet`. `['responds_to_parent','quote_anchors_parent']` and `['quote_anchors_parent','responds_to_parent']` produce one key.

### 4. Batching

- **Batch cap = 5 classifiers per call (one packet).** This is the MCP-001 §9 / §12 five-classifier packet prompt limit and the card's acceptance criterion. `BATCH_CAP = 5` is a hard constant; `planClassifierBatches` never emits an inner array longer than 5.
- **Batching policy — classifiers travel with the family that shares their input context.** The five MCP-001 §8 catalog families map to **four batch groups** (the catalog is 23 classifiers; four ≤5-sized groups plus a small remainder — see below). Grouping by shared input context means one prompt's `parentContext` / `moveContext` / `roomContext` serves every classifier in the batch, so the *input tokens are paid once per batch, not once per classifier* — this is where batching saves money.

  | Batch group | Classifiers (from MCP-001 §8) | Shared input context | Count |
  |---|---|---|---|
  | **A — Parent continuity** | `responds_to_parent`, `quote_anchors_parent`, `answers_clarification`, `introduces_new_issue` | parent body + move body + parent argumentType | 4 |
  | **B — Branch routing & mode fit** | `suggests_side_branch`, `suggests_diagonal_tangent`, `fits_selected_debate_mode`, `contains_unplayable_insult_only`, `contains_playable_hot_take` | move body + room mode + branch state | 5 |
  | **C — Evidence pressure** | `asks_for_evidence`, `provides_evidence`, `evidence_supports_claim`, `creates_source_chain_gap`, `uses_popularity_as_evidence` | move body + claim text + evidence artifact state | 5 |
  | **D — Evidence provenance & constructive movement** | `cites_retraction`, `uses_satire_as_evidence`, `is_satire_or_parody`, `narrows_claim`, `concedes_narrow_point` | move body + parent body + cited-item text | 5 |
  | **E — Constructive movement & friction (remainder)** | `requests_clarification`, `ready_for_synthesis`, `needs_pre_send_pause`, `shifts_to_person_or_intent` | move body + parent body + lifecycle state | 4 |

  Five groups (A–E) of ≤5 cover the 23-classifier catalog. The future card MAY merge two small remainder groups if and only if their combined size stays ≤ 5 and they share enough input context that input tokens are not wasted — but the **safe, documented default is the five groups above**.
- **`planClassifierBatches(requested)` does not always emit five calls.** A trigger moment requests **only the classifiers relevant to that moment**, then `planClassifierBatches` packs them into the *fewest* ≤5-sized batches respecting the group boundaries. Examples:
  - `branch_routing` requests group B's routing subset → **1 call**.
  - `evidence_inspection` requests group C → **1 call**.
  - `post_submit` in `metadata_only` mode requests a curated cross-family subset (e.g. continuity + the single most relevant evidence/branch classifier) → typically **1 call**, at most 2.
  - `synthesis_readiness` requests `ready_for_synthesis` (+ optionally `narrows_claim`, `concedes_narrow_point`) → **1 call**.
- **Per-trigger classifier budget:** a single trigger moment requests **at most 5 classifiers → at most 1 provider call** by default. A moment that genuinely needs more than one family is the rare case and is capped at **2 calls** (10 classifiers). A moment never fans out to all five groups — that would be an `evaluateTrigger` design defect. The recommended per-moment classifier sets are an MCP-002/MCP-003 detail; this card fixes only the *cap* (≤2 calls per trigger, ≤5 classifiers per call).
- **Batching is per-trigger, not cross-trigger.** The card does not batch two different moves' classifications into one call — each move's `contentHash` is a distinct cache key and each post is a distinct, well-separated event. Cross-move batching would add latency and complexity for no doctrine benefit.

### 5. Retry / fallback policy

- **Fallback target:** the **deterministic layer-1 derivation** (MCP-001 §6) — the free, authoritative-for-what-it-covers metadata. On any failure of layer 2, the consumer uses the layer-1 result and shows **no error to the user**. The game still works; the semantic referee simply does not contribute this turn. This is the doctrine guarantee: **gating the classifier never blocks a post; failing the classifier never blocks a post** (cdiscourse-doctrine §1; MCP-009 §"disabled-by-default", §"edge cases").
- **Bounded retries:** `SEMANTIC_RETRY_POLICY = { maxAttempts: 2, backoffMs: [250] }` — **one initial attempt plus at most one retry**, a single short backoff. Retries happen **inside the Edge Function** (MCP-009), never on the client. The retry loop is bounded and total — it can never spin.
- **Retryable vs terminal:** `shouldRetry(errorClass)` returns `true` only for **transient** classes — `network_timeout`, `provider_5xx`, `provider_rate_limited`. It returns `false` for **terminal** classes — `key_missing`, `validation_failed` (the request is malformed), `parse_failure` (the model returned non-JSON or an off-contract shape, e.g. tried to set `authoritative: true`), `disabled`. A terminal error is **not** retried — it goes straight to fallback.
- **Failure is exhausted, then fallback:** after `maxAttempts` retryable failures, or immediately on a terminal failure, the Edge Function returns either `{ enabled: false, reason }` or a **deterministic fallback packet** (`provider: 'mock'`, low-confidence layer-1-only binaries — MCP-009's registry behavior). Either way the consuming code path falls back to layer 1; **the call never throws to the client and never blocks the post.**
- **No indefinite retry, no retry storm:** because `maxAttempts` is 2 and the result is cached on success, a repeatedly-failing provider produces at most 2 attempts per *unique cache key*, then stops. A cold cache after a provider outage means re-attempts, but still capped at 2 per key. There is no global retry queue and no background retry — a failed classification is simply absent until the next *new* trigger.
- **Timeout:** each provider attempt has a wall-clock timeout (recommended **8 s**, operator-tunable). A timeout is a `network_timeout` (retryable once). The total worst case for a trigger is `2 × 8s + 250ms backoff` ≈ 16.25 s before fallback — well inside an Edge Function budget, and the user never waits on it because classification is **off the post's critical path** (post-submit classification runs *after* the post already succeeded; pre-send review is the only synchronous case and it is opt-in strict-mode only, where the user has chosen to wait for review).
- **Retry/fallback interaction with the cache:** a *successful* classification (including a successful retry) is cached. A *fallback packet* is **not** cached — it is layer-1-derived and cheap to recompute, and caching it would pin a degraded result past a provider recovery. The next trigger for the same key gets a fresh attempt.

### 6. Token budget

- **A single documented ceiling: `SEMANTIC_PACKET_TOKEN_BUDGET`** — the **maximum combined input + output tokens per packet call**. The number itself is an **operator decision** (§"Operator decisions"); this design fixes the *mechanism* and *recommends* a starting value.
- **Recommended starting ceiling: 1,500 tokens per packet** (≈ 1,100 input + ≈ 400 output). Rationale: MCP-009 caps `moveBodyRedacted` and `parentBodyRedacted` at 8,000 *characters* each; the system prompt (MCP-001 §9) plus a redacted parent + redacted move + ≤5 classifier ids realistically fits well under 1,100 input tokens for a normal move, and a five-classifier `SemanticRefereePacket` (≤5 binaries, two enums, a small `scoreHints` block, provenance) is a few hundred output tokens. 1,500 is a **conservative cap with headroom**; the operator confirms or adjusts.
- **Estimation without a tokenizer dependency:** `estimatePacketTokens` uses a **conservative char-based heuristic** — `ceil(totalChars / 3.5)` for input (3.5 chars/token is deliberately pessimistic; real English is ~4) plus a fixed `OUTPUT_TOKEN_RESERVE` (recommended 450) for the bounded JSON response. No `tiktoken` / no tokenizer package is added (dependency policy + "no speculative deps"). The heuristic over-estimates, so it fails *safe* — it refuses slightly before the true ceiling, never after.
- **Refuse over-budget — the boundary, not the post:** `isWithinBudget(payload)` is checked **twice**: (1) client-side in the future call-site wiring before `classifyMove` is invoked, and (2) **defensively inside the `semantic-referee` Edge Function** before any provider call (MCP-009 §"file changes" notes this re-check). An over-budget payload → the call is **not sent**; the Edge Function returns `{ enabled: false, reason: 'over_budget' }` (a new `ClassifyMoveOutcome` reason this card adds to MCP-009's union — see §"Dependencies"). The consumer falls back to layer 1. **Refusing an over-budget packet never blocks the post** — the post already happened (`post_submit`) or proceeds normally (`pre_send_review` is advisory). An over-budget move is simply not semantically classified; it still posts, still scores on layer 1.
- **Why refuse rather than truncate:** truncating a move body to fit the budget would classify a *different* text than the user wrote — a false-positive risk and a doctrine smell. MCP-009's 8,000-char cap already bounds body size at the schema; a move that still exceeds the *token* budget after that is an outlier, and the honest behavior is "this move was too large to semantically classify" (degraded gracefully to layer 1), not "we classified a truncated version."
- **The token budget is one of four cost controls, not the only one.** Trigger gating (§1–2) caps *how often*; caching (§3) caps *re-calls*; batching (§4) caps *input-token duplication*; the token budget (§6) caps *per-call size*. Together they make "an uncapped layer" structurally impossible — the doctrine requirement (roadmap §13).

### 7. Integration seam (where the gate sits)

The future implementation card wires `evaluateTrigger` as the **single chokepoint** every potential semantic call passes through:

```
user action / UI event
        │
        ▼
  evaluateTrigger(input)  ──► allowed:false ──► do nothing (or read cached metadata
        │                                       for referee_feedback). No call.
        │ allowed:true
        ▼
  planClassifierBatches(requestedClassifiers)  ──► ≤5 per batch, ≤2 batches
        │
        ▼  per batch:
  build SemanticCacheKey ──► cache.get(key) ──► HIT ──► return cached packet. No call.
        │ MISS
        ▼
  isWithinBudget(payload) ──► over budget ──► no call; fall back to layer 1.
        │ within budget
        ▼
  classifyMove()  ──►  semantic-referee Edge Function (MCP-009)
                       └─ disabled-by-default 2nd line; retry policy; cache.set on success
```

The gate, the batch planner, the cache-key builder, and the budget check are all **pure TS** and run wherever the call site lives (client wiring before `classifyMove`, and re-checked in the Edge Function). The provider call itself is **only** in the Edge Function (MCP-009). This card adds **no client-side AI call** and **no new network surface** — it only decides whether MCP-009's existing boundary is invoked.

## Edge cases

The implementer of the future card must handle each of these:

- **Layer feature flag off (`featureLayerEnabled === false`)** — `evaluateTrigger` returns `allowed: false` for every moment; no call, no error. The Edge Function's own disabled path (MCP-009) is the redundant second line.
- **Room mode `'off'` or absent** — `allowed: false` for every moment (fail-closed on an absent mode).
- **`pre_send_review` requested but `preSendReviewOptIn === false`** — `allowed: false`. Pre-send review is opt-in even in a `metadata_and_chip` room; the mode *permits* it, the creator's opt-in *enables* it. Both required.
- **`pre_send_review` requested in a non-strict mode** — `allowed: false` (mode is not `metadata_and_chip`).
- **`actorRole === 'observer'`** — `allowed: false` for every moment. An observer never originates a semantic call (also independently blocked by `observer_browsing` being a forbidden event).
- **`actorRole === 'moderator'`** — moderators are not players; treated like observers for trigger purposes — no semantic call on a moderator action. (A moderator post, if one exists, is out of the PvP loop the referee scores.)
- **`branch_routing` requested but `branchRouteAmbiguous` is false/absent** — `allowed: false`. Layer 1 is confident; layer 2 adds nothing and costs money. Layer 1 wins on its own turf (MCP-001 §6).
- **`synthesis_readiness` requested below the lifecycle-event threshold** — `allowed: false`. Premature synthesis classification wastes a call and would surface a premature banner.
- **`referee_feedback` requested but no classified metadata exists** — the gate may return `allowed: true`, but the consuming path finds nothing in the cache/ledger and **shows no banner**. It **must not** originate a provider call to fill the gap — `referee_feedback` is a derived trigger by definition.
- **Empty / whitespace-only move body** — there is nothing to classify; the call is not made (MCP-009's schema also rejects an empty `moveBodyRedacted`). `post_submit` for an empty post cannot happen — `submit-argument` rejects empty bodies deterministically (cdiscourse-doctrine: deterministic validation blocks; that block is unrelated to and upstream of the classifier).
- **Concurrent triggers for the same move (e.g. two rapid `post_submit` retries from a flaky UI)** — both compute the **same cache key**; the first populates the cache, the second is a hit. The cache is the de-dup. In a true cold-cache race two identical calls may briefly run — a small bounded cost, not a correctness bug (MCP-009 §"concurrent calls" notes the boundary is idempotent). A future optimization is an in-flight-promise map keyed by the cache key; not required for v0.
- **Same move re-triggered after a `promptVersion` bump** — different `classifierSet`/`promptVersion` → different key → a new, legitimate call. The old packet is unreachable and LRU-evicted eventually. This is the *intended* invalidation path.
- **Cache eviction under load** — an LRU-evicted entry causes one re-classification (a cost) on the next trigger for that key. Never a wrong answer.
- **Cold Edge Function start** — the in-memory cache is empty; the first trigger after a cold start is a miss and re-classifies. Acceptable — correctness is preserved, only a cost.
- **Over-budget payload after the 8,000-char schema cap** — `isWithinBudget` refuses; `{ enabled: false, reason: 'over_budget' }`; fall back to layer 1; **the post is unaffected**.
- **`classifierSet` ordering** — `['a','b']` vs `['b','a']` must key identically; the order-normalization in `planClassifierBatches` / `serializeSemanticCacheKey` guarantees it. A future-card test asserts it (§"Test plan").
- **Provider transient failure** — bounded retry (≤2 attempts), then deterministic fallback packet; never thrown, never blocking.
- **Provider returns off-contract output (e.g. `authoritative: true`, a `block` field, a truth field)** — MCP-009's outbound schema rejects it (`parse_failure`, terminal — no retry) → deterministic fallback. The classifier physically cannot produce a verdict that reaches the user.
- **Network failure between client and Edge Function** — `classifyMove` maps it to a non-`ok` result; the consumer falls back to layer 1; no user-facing error (MCP-009 §"edge cases").
- **Doctrine edge case — "what if a gate misfire blocks a post?"** — it cannot: `evaluateTrigger` returns a *decision about whether to classify*, never a decision about whether to *post*. The post path (`submit-argument`) does not consult the gate. A gate that returns `allowed: false` for everything simply means "no semantic metadata this session" — the game runs on layer 1.
- **Doctrine edge case — "what if caching serves a stale verdict?"** — a cached packet is keyed by `contentHash` + `promptVersion`; a posted body is immutable and a prompt change bumps the version, so a cache hit is always the *correct* packet for *that body under that prompt*. There is no staleness window. (And the packet is advisory regardless — even a hypothetically stale packet could not block or mislabel; it is one input to the deterministic ledger.)
- **Doctrine edge case — "what if heat / popularity sneaks into a trigger condition?"** — no trigger fires on a heat, like-count, view-count, or virality signal. Triggers fire on *structural completed actions* (a post, an attach, a send-intent, a lifecycle accumulation) only. `evaluateTrigger`'s input has no engagement field.

## Test plan

**No tests are written in this card.** The card scope is design-only and the card's own "Test plan" section is explicitly labeled "(future implementation card)." The tests below are the **responsibility of the future implementation card** and are listed so that card's runner executes without re-deriving them. All tests are mock-provider / fixture-driven; **no test makes a live call** (cdiscourse-doctrine §7; test-discipline; MCP-001 §19).

- `__tests__/semanticTriggerGates.test.ts` — the **allowed-vs-forbidden moment table.** For every `SemanticTriggerMoment`, `evaluateTrigger` returns `allowed: true` only when flag on + mode permits + precondition holds, and `allowed: false` for every disqualifying combination (flag off, mode `'off'`/absent, observer/moderator role, missing opt-in, `branchRouteAmbiguous` false, below synthesis threshold). For every `SemanticForbiddenEvent`, `allowed: false` unconditionally. A table-driven test with one row per (event × mode × flag × role) combination.
- `__tests__/semanticCache.test.ts` — `serializeSemanticCacheKey` is total and order-normalized (`['a','b']` and `['b','a']` → same string; root move `parentId` `''` sentinel stable). The same key never re-calls: a second `get` after a `set` returns the cached packet with **zero provider invocations** (assert via an injected provider spy). A `promptVersion` bump produces a different key → a fresh call. LRU eviction at capacity drops the least-recently-used entry. **No time-based expiry exists** — assert that advancing a fake clock does not invalidate an entry.
- `__tests__/semanticNoCallOnDraftEdit.test.ts` — **the acceptance-criterion proof.** Feed `keystroke`, `hover`, `timeline_selection`, `observer_browsing`, `scroll`, `focus`, `blur`, `gallery_browsing` events through `evaluateTrigger` and through the integration seam; assert the provider spy records **zero calls**. Also: an observer role through a *post_submit*-shaped event → zero calls.
- `__tests__/semanticBatching.test.ts` — `planClassifierBatches` never emits an inner array longer than `BATCH_CAP` (5); `BATCH_CAP === 5`; the four/five batch groups partition the 23-classifier catalog; classifiers are order-normalized within a batch; a trigger requesting one family produces exactly one batch; a trigger never fans out to more than 2 batches.
- `__tests__/semanticTokenBudget.test.ts` — `estimatePacketTokens` is conservative (over-estimates vs a known fixture); `isWithinBudget` returns `ok: false` for an over-ceiling payload and `ok: true` at/under it; the Edge-Function defensive re-check refuses an over-budget payload with `{ enabled: false, reason: 'over_budget' }`; **refusing over-budget never blocks a post** (the `submit-argument` path is untouched in the test).
- `__tests__/semanticRetryPolicy.test.ts` — `shouldRetry` returns `true` only for transient classes (`network_timeout`, `provider_5xx`, `provider_rate_limited`) and `false` for terminal ones (`key_missing`, `validation_failed`, `parse_failure`, `disabled`); `maxAttempts === 2`; an exhausted retry yields a deterministic fallback packet, **never a thrown error and never a blocked post**; a fallback packet is **not** cached.
- `__tests__/semanticTriggerBanList.test.ts` — **doctrine ban-list:** scan every `reason` code `evaluateTrigger` and the budget/retry layers can produce for verdict/person tokens (`winner`, `loser`, `liar`, `true`, `false`, `correct`, `dishonest`, `bad faith`, `manipulative`, `troll`, `propagandist`, `extremist`, `wrong`, `right`, `proven`, `defeated`). Zero matches. The card touches user-reachable strings via `TriggerDecision.reason` codes (mapped through `gameCopy.toPlainLanguage` before UI).
- `__tests__/semanticTriggerForbiddenImports.test.ts` — `triggerGates.ts`, `semanticCache.ts`, `classifierBatching.ts`, `tokenBudget.ts`, `retryPolicy.ts` import **nothing** from Supabase, React, or a network library — they are pure decision/data modules (mirrors MCP-001 §19's engine-purity test).

The card's own "Test plan" section names four files (`semanticTriggerGates`, `semanticCache`, `semanticNoCallOnDraftEdit`, `semanticBatching`); this design **keeps all four** and adds `semanticTokenBudget`, `semanticRetryPolicy`, `semanticTriggerBanList`, and `semanticTriggerForbiddenImports` so the future card's coverage matches test-discipline's bar (every public function tested, ban-list scan, forbidden-import scan).

### Operator decisions

1. **Per-packet token-budget ceiling (`SEMANTIC_PACKET_TOKEN_BUDGET`).** A single number the operator confirms. **Recommendation: 1,500 tokens** combined input + output per packet call (≈ 1,100 input + ≈ 400 output), with a conservative char-based estimator (`ceil(chars / 3.5)` + a 450-token output reserve) that fails safe. The operator may raise it (if real moves are routinely larger) or lower it (for tighter cost control). The mechanism — refuse over-budget, fall back to layer 1, never block the post — does not change with the number.
2. **Whether post-submit classification is on by default or opt-in per room.** The card scope and acceptance criteria say "post-submit is the **default** trigger" — i.e. post-submit is the *trigger of record*. The operator decision is whether, **once the layer feature flag is on**, post-submit classification runs for **every room in a non-`off` mode by default**, or whether each room **additionally opts in**. **Recommendation: on by default for any room whose `semanticClassificationMode` is `metadata_only` or `metadata_and_chip`** — the GAME-003 mode field already *is* the per-room switch (a room set to `'off'` has opted out; a room set to a non-`off` mode has opted in by choosing that mode). A second per-room opt-in on top of the mode field would be redundant. The whole layer remains off until the operator flips `SEMANTIC_REFEREE_ENABLED` (MCP-009), so "on by default" still means "off until the operator enables the layer, then on for non-`off` rooms." If the operator wants a more conservative rollout, the fallback is: post-submit is `metadata_only`-gated *and* requires an explicit room-creator toggle — a one-field addition to GAME-003's setup flow.
3. **Synthesis-readiness lifecycle-event threshold (`SYNTHESIS_TRIGGER_MIN_EVENTS`).** Recommendation: `6`. A sane default, operator-tunable; not a hard operator *decision* but flagged so the operator can adjust if synthesis banners fire too early or too late.
4. **Provider attempt timeout.** Recommendation: `8 s` per attempt. Tunable; flagged for visibility because it bounds the pre-send-review worst case (`2 × 8s + 250ms` ≈ 16.25 s before fallback).

## Risks

- **`classifierSet` ordering bug.** If the requested classifier ids are not order-normalized before the cache key is built, two requests for the same classifiers in different orders miss the cache and double the cost. The order-normalization in `planClassifierBatches` / `serializeSemanticCacheKey` is **load-bearing** and must have its own test (`semanticCache.test.ts`).
- **In-memory cache is not cross-instance.** Supabase Edge Functions can run multiple instances; an in-memory cache is per-instance. Two instances classifying the same move both miss and both call — a bounded cost duplication, not a correctness bug. If real-world cost shows this matters, the documented future option is a **persisted cache table** (`semantic_referee_cache`, keyed by `serializeSemanticCacheKey`, RLS read-scoped to room participants, written by the Edge Function). That is a *new migration in a future card*, explicitly NOT introduced here (consistent with MCP-001 §12 and MCP-009's no-DB-change stance). The implementer must resist adding it speculatively.
- **Cold-start cache miss.** Every cold Edge Function start empties the cache; the first batch of triggers re-classifies. Acceptable for v0; the persisted-cache option above also fixes it if it becomes a cost problem.
- **GAME-003 not yet landed.** `semanticClassificationMode` is GAME-003's field and GAME-003 (#119) is open. The future implementation card of MCP-004 must either land after GAME-003 or stub the mode input behind a feature flag defaulting to `'off'` (fail-closed). The gate *logic* is stable; only the input type is GAME-003-shaped.
- **Token estimator drift.** The char-based heuristic is deliberately conservative, but if a future model's tokenization differs sharply the estimate could be off. Mitigation: the estimator over-counts (fails safe — refuses early), the ceiling has headroom, and MCP-009's 8,000-char schema cap is the hard upstream bound. If precision is ever needed, that is a future change — **not** a speculative tokenizer dependency now (dependency policy).
- **Pre-send review latency on the user's critical path.** `pre_send_review` is the one synchronous trigger — the user waits for it before sending. The worst-case `2 × 8s` retry path is long. Mitigation: pre-send review is **strict-mode + opt-in only** (the user chose a mode that reviews before send), the retry result still falls back to layer 1 (the user is never *stuck*), and the future card should consider a shorter timeout for the `pre_send_review` moment specifically (e.g. 4 s, single attempt) — flagged for the implementer.
- **Existing tests.** This card's future implementation adds pure-TS modules and one additive `ClassifyMoveOutcome` reason; it touches no existing screen and no existing Edge Function logic beyond MCP-009's (not-yet-built) function. No existing CDiscourse test should need updating. If MCP-009's implementation lands first, its `ClassifyMoveOutcome` schema test must be extended for `'over_budget'` — a one-line addition.
- **Migration that requires operator deploy.** **None for v0** — the in-memory cache needs no migration. *If* the operator later adopts the persisted-cache option, that future card carries a migration and an `npx supabase db push --linked` step. This card produces no migration.
