## Discrepancies between the card body and the real code (read this first)

The card body is directionally correct but four of its stated shapes are wrong against `main` as it stands today. None of them block the card; the design below is written against reality.

1. **The audit-log entry shape in the card is not the real `admin_audit_events` shape.** The card says the audit entry is `{ kind: 'admin_invite_user', actor_user_id, target_email, target_role, requested_redirect, sent }`. There is no `kind` column and no `admin_invite_user` value anywhere. Every existing handler calls `writeAdminAudit({ actorUserId, targetUserId?, targetAuthUserId?, action, reason?, payload })` (`supabase/functions/_shared/adminAudit.ts` lines 63-94). `action` is a constrained enum — `AdminAuditAction = typeof WHITELISTED_ACTIONS[number]` — and `writeAdminAudit` inserts columns `actor_user_id`, `target_user_id`, `target_auth_user_id`, `action`, `reason`, `source`, `payload`. **Resolution:** the new action is `'invite_user'` (snake_case, matching `create_user` / `send_password_reset`). The card's per-invite fields (`target_email`, `target_role`, `requested_redirect`, `sent`) live **inside `payload`**, not as top-level columns. Email is recorded as `emailDomain` only (the domain after `@`) — `handleSendPasswordReset` already establishes this exact precedent (`payload: { emailDomain: email.split('@')[1] ?? null, ... }`, `index.ts` line 367) precisely so a raw email is never stored in an audit row.

2. **`'invite_user'` must be added to TWO whitelists, not one — and the request union too.** The Edge Function rejects an unknown action twice before any handler runs: `AdminUsersRequestSchema.safeParse` (the discriminated union in `_shared/adminSchemas.ts`) fails first with a `validation_failed` 422, and then `isWhitelistedAction(body.action)` (`_shared/adminAudit.ts` `WHITELISTED_ACTIONS`) returns `bad_request` `unknown_action` 400. Adding `invite_user` requires: a new `InviteUser` zod object added to the `AdminUsersRequestSchema` discriminated union, the string `'invite_user'` appended to `WHITELISTED_ACTIONS`, and a `case 'invite_user':` in the `switch` in `index.ts`. The client-side `AdminUsersAction` union in `src/lib/edgeFunctions.ts` (lines 236-250) must also gain `'invite_user'` or the `adminInviteUser` wrapper won't type-check.

3. **`adminSchemas.test.ts` is a hand-maintained mirror, not an import.** `__tests__/adminSchemas.test.ts` re-declares the zod schemas locally because the Edge Function file uses Deno-style `npm:zod@4` imports Jest cannot load (see the file's header comment, lines 1-8). It currently mirrors only six of the fourteen actions. If QOL-024 wants schema coverage for `invite_user`, the implementer must add the `InviteUser` mirror schema to that test file too. This is expected drift-by-design, not a bug — the test file header explicitly says "If you change the Edge Function schemas, mirror the change here."

4. **`buildAuthRedirectUrl` cannot be reused inside the Edge Function — and should not be.** The card says "redirectTo built via QOL-023's `buildAuthRedirectUrl`." That helper lives in `src/lib/auth/buildAuthRedirectUrl.ts`. Edge Functions are Deno and import via URL or relative path — they cannot import from `src/`. More importantly, the redirect origin is a **runtime/browser concept** (`window.location.origin`, the runtime-env-injected `EXPO_PUBLIC_APP_ORIGIN`) that the Edge Function has no clean access to. The QOL-023 admin path already solved this exact problem: `adminSendPasswordReset` (`src/features/admin/adminApi.ts` lines 86-109) builds `redirectTo` **on the client** via `buildAuthRedirectUrl` and passes it into the Edge Function as a parameter; the Edge Function forwards it verbatim into `generateLink({ ..., options: { redirectTo } })`. **Resolution:** QOL-024 follows that pattern exactly — `adminInviteUser` builds `redirectTo` client-side with `buildAuthRedirectUrl({ kind: 'invite', ... })`; the Edge Function receives it as an optional, validated string and forwards it. This is the cleaner client/Deno split and is verified-feasible because QOL-023 already does it for `password_reset`. No `buildAuthRedirectUrl` import is added inside `supabase/functions/`.

Also confirmed against reality:
- `admin-users` is JWT-verified at two layers: `verify_jwt = true` in `supabase/config.toml` (line 376) **and** `requireAdmin(req)` in `index.ts` line 56 — which calls `callerClient.auth.getUser()` then loads `profiles.role` via the service client and requires `role === 'admin'`. The card says "Caller must satisfy `is_moderator_or_admin()`" — the *real* gate is stricter: it requires `role === 'admin'` exactly (`_shared/adminAuth.ts` lines 69-71), not moderator-or-admin. QOL-024 inherits this gate unchanged; `invite_user` is admin-only like every other action. (The SQL helper `is_moderator_or_admin()` exists for RLS policies on other tables; the `admin-users` function does not use it — it does an explicit `role === 'admin'` check. Do not loosen this.)
- The existing email-sending precedent inside `admin-users` is `generateLink`, **not** a Resend call. `handleSendPasswordReset` calls `sc.auth.admin.generateLink({ type: 'recovery', email, options: { redirectTo } })` (`index.ts` lines 355-359). The function has **no** `RESEND_API_KEY` path and no `inviteUserByEmail` call today. (`request-argument-deletion` is the only function that does a Resend call; `admin-users` does not.) This drives the `inviteUserByEmail` vs `generateLink('invite')` decision below.
- `AdminCreateUserForm.tsx` (87 lines) today has a Human/Bot toggle, not an Invite/Password toggle. `isBot` defaults to `true`. The human branch always sends `emailConfirm: true` and an optional password. There is **no existing test file** for `AdminCreateUserForm` — `__tests__/` has `adminApi.test.ts`, `adminArguments*.test.ts`, `adminSchemas.test.ts`, `adminSecurity.test.ts`, but no `AdminCreateUserForm.test.tsx`. The card says "Existing `AdminCreateUserForm` tests extended" — there are none to extend; QOL-024 **creates** the first one.
- `adminCreateUser` and `adminCreateBotUser` wrappers exist in `adminApi.ts` (lines 53-75) and are thin pass-throughs to `adminUsers({ action, ...input })`. `adminInviteUser` will follow the same shape.

None of these conflict with doctrine, so the card proceeds.

## Goal (one paragraph)

Today an admin can only mint an **already-confirmed** user with a password via `AdminCreateUserForm` → `adminCreateUser` → the `admin-users` Edge Function `create_user` action. For real human testers this forces the operator to hand-issue credentials out-of-band, because there is no path that emails a new user a self-serve first-sign-in link. QOL-024 adds an **invite-by-email** path: a new `invite_user` action on the `admin-users` Edge Function that calls Supabase's admin invite API, a `adminInviteUser(input)` client wrapper that derives the invite-email callback URL from QOL-023's `buildAuthRedirectUrl({ kind: 'invite', ... })`, and a top-level toggle on `AdminCreateUserForm` so an admin chooses **Invite by email** (new default) vs **Create with password** (the existing flow) — the Bot/Test branch is untouched. Doctrine constraints that shape the design: **no service-role in the client** (the invite call runs only inside the Edge Function, which already builds its own service-role client server-side after the admin check); **no secrets / no token leak** (the function never logs `Authorization`, never returns the generated invite link or token, never returns another user's email); **plain-language status copy** (the UI shows "Invite sent" / "Invite could not be sent" — never an internal code, never the link); **the invite-email body is not constructed in client or function code** — the Supabase "Invite user" email template is the single source of truth; **fail-soft** (a bad redirect origin degrades to the Supabase dashboard Site URL, it never blocks the invite).

## Data model

**No new database table. No migration.** This is the most important scoping decision in the card, and it is deliberate:

- The Supabase **`auth.users`** table is the user store. `inviteUserByEmail` / `generateLink('invite')` create the `auth.users` row (and the trigger-created `public.profiles` row) — QOL-024 writes nothing to a new table.
- The **`admin_audit_events`** table already exists and already receives a row for every admin action via `writeAdminAudit`. The `invite_user` action reuses it. Its `action` column is a free-text column constrained only by the application-side `WHITELISTED_ACTIONS` array — adding a new string to that array requires **no schema change** (the column is `text`, not a Postgres enum; this is confirmed by the fact that all fourteen current actions are added by editing the JS array, never a migration). If the implementer discovers `admin_audit_events.action` is in fact a Postgres `enum` type, that would require a migration — **see the Risks section; verify this before claiming no-migration.**

The only new *types* are TypeScript interfaces:

```ts
// supabase/functions/_shared/adminSchemas.ts — new zod object in the union
const InviteUser = z.object({
  action: z.literal('invite_user'),
  email: z.string().email().max(320),
  displayName: z.string().min(1).max(120).optional(),
  role: z.enum(['user', 'moderator']).default('user'), // NOT 'admin' — see below
  redirectTo: z.string().url().max(500).optional(),
});
```

```ts
// src/features/admin/adminApi.ts — wrapper input/output
export interface AdminInviteUserInput {
  email: string;
  displayName?: string;
  role?: 'user' | 'moderator';
  redirectTo?: string; // omitted by the form; adminInviteUser derives it
}

export interface AdminInviteUserResult {
  ok: true;
  invited: true;
  notification: 'sent' | 'not_configured' | 'send_failed';
}
```

**Role restriction (deliberate, stricter than the card).** The card's body says `role?: 'user' | 'moderator'` — note it already excludes `'admin'`, and QOL-024 keeps that exclusion. An admin must **never** be created via an unauthenticated invite link: the existing `create_user` action requires `confirmAdminCreate === true` to mint an admin (`adminSchemas.ts` lines 40-43), a deliberate friction gate. An invite email cannot carry that confirmation, and an invited admin would gain admin role on first sign-in with a self-chosen password. The `InviteUser` schema therefore types `role` as `z.enum(['user', 'moderator'])` — `'admin'` is rejected at validation time with a 422. Promoting an invited user to admin remains the `update_role` action's job (with its `confirmAdminGrant` gate). This is recorded again under "Doctrine self-check."

## API / interface contracts

### Edge Function: `invite_user` action

**Request body** (validated by the `InviteUser` zod object):

```jsonc
{
  "action": "invite_user",
  "email": "tester@example.com",        // required, valid email, <= 320 chars
  "displayName": "Test User",           // optional, 1-120 chars
  "role": "user",                       // optional, 'user' | 'moderator' — default 'user', NEVER 'admin'
  "redirectTo": "https://dev.cdiscourse.com/auth/callback" // optional, valid URL <= 500 chars
}
```

**Gating** — identical to every other `admin-users` action, no new code:

1. `supabase/config.toml` `[functions.admin-users] verify_jwt = true` — Supabase rejects a missing/invalid JWT before the function body runs.
2. `req.method !== 'POST'` → `methodNotAllowed()` 405.
3. `AdminUsersRequestSchema.safeParse` — a malformed `invite_user` body (bad email, `role: 'admin'`, etc.) → `validationFailed` 422 with `{ error: 'validation_failed', issues: [...] }`.
4. `isWhitelistedAction(body.action)` — once `'invite_user'` is appended to `WHITELISTED_ACTIONS`, this passes.
5. `requireAdmin(req)` — verifies the JWT via `callerClient.auth.getUser()`, loads `profiles.role` via the service client, requires `role === 'admin'`. Non-admin → `forbidden('admin_required')` 403; missing/invalid token → `unauthorized()` 401.

The handler only runs after all five gates pass. **No new gating logic is written** — `invite_user` inherits the standard `admin-users` security envelope.

**Handler — `inviteUserByEmail` vs `generateLink('invite')`: pick `inviteUserByEmail`.**

| Option | What it does | Sends an email? |
|---|---|---|
| `sc.auth.admin.inviteUserByEmail(email, { data, redirectTo })` | Creates the `auth.users` row AND triggers the Supabase **"Invite user"** email template. | **Yes** — Supabase sends it via the configured SMTP. |
| `sc.auth.admin.generateLink({ type: 'invite', email, options: { redirectTo } })` | Creates the row and returns the invite link/token in the response. Does **not** send an email unless custom SMTP delivers it; the caller is expected to send the email itself. | No — caller must email it. |

**Decision: use `inviteUserByEmail`.** Rationale: (a) The card requires "an invite path … + email" and the `notification: 'sent'` response — `inviteUserByEmail` is the API whose entire purpose is *send the invite email*. (b) `generateLink('invite')` returns the raw invite link in the response body — the card explicitly forbids returning the link to the client and forbids the function constructing the email body; `generateLink` would force the function to either (wrongly) return the link or (wrongly) build+send the email itself via Resend. `inviteUserByEmail` keeps the email body entirely in the Supabase template, which is the doctrine-required source of truth. (c) `handleSendPasswordReset` uses `generateLink('recovery')` only because recovery there relies on Supabase's auto-send for that type; for invites, `inviteUserByEmail` is the documented, purpose-built call and avoids handling a token in function code at all. (d) The `request-argument-deletion` Resend path is a *separate* concern (admin notification, not an auth email) — `admin-users` deliberately has no Resend dependency and QOL-024 does not add one.

**`notification` field semantics with `inviteUserByEmail`:** `inviteUserByEmail` does not return a granular "did SMTP deliver" flag — it returns the created user (or an error). The handler maps:

- Call succeeds, no error → `notification: 'sent'` (Supabase accepted the invite and queued the email).
- Call fails with an error whose message indicates SMTP / email is not configured (e.g. matches `/smtp|email.*not.*config|email provider/i`) → `notification: 'not_configured'` and **still** return `ok: true, invited` is `false` in that case — see below.
- Call fails for any other reason (duplicate email, rate limit, etc.) → return a `422 validationFailed` with `{ error: 'invite_user_failed', detail: <sanitized> }` — the `detail` is the Supabase error message, which the existing `create_user` handler already surfaces the same way (`index.ts` line 233). The detail must be scanned to ensure it carries no token (Supabase invite errors do not, but the implementer asserts it in a test).

To keep the `{ ok, invited, notification }` contract clean, the handler returns:

- **Happy path:** `ok({ ok: true, invited: true, notification: 'sent' })`.
- **Email-not-configured but user created:** in practice `inviteUserByEmail` fails atomically when SMTP is unset (no user row is created), so this collapses into the failure path. To honour the card's `notification: 'not_configured'` value, the handler detects the SMTP-not-configured error class and returns `validationFailed({ error: 'invite_email_not_configured', invited: false, notification: 'not_configured' })` — a clear, plain-mappable signal that the *invite mechanism* needs operator setup, distinct from a bad email address. The client maps this to "Invite could not be sent — email is not configured. Ask the operator."
- **`send_failed`** is reserved for the case where Supabase reports the user was created but the email dispatch failed — if the installed `@supabase/supabase-js@2` version surfaces that distinctly. If it does not (current versions treat it atomically), `send_failed` is still a valid member of the response union for forward-compat; the implementer wires it only if the SDK exposes the distinction, otherwise it is simply never returned. Document this in a code comment.

**Pseudo-handler:**

```ts
async function handleInviteUser(
  body: Req<'invite_user'>,
  caller: Caller,
  sc: SC,
): Promise<Response> {
  const { data: inviteRes, error: inviteErr } = await sc.auth.admin.inviteUserByEmail(
    body.email,
    {
      data: body.displayName ? { display_name: body.displayName } : {},
      redirectTo: body.redirectTo, // undefined => Supabase uses dashboard Site URL
    },
  );

  if (inviteErr || !inviteRes?.user) {
    const msg = inviteErr?.message ?? 'unknown';
    if (/smtp|email.*not.*config|email provider|sending.*disabled/i.test(msg)) {
      // Plain-mappable: the invite mechanism itself needs operator setup.
      return validationFailed({
        error: 'invite_email_not_configured',
        invited: false,
        notification: 'not_configured',
      });
    }
    return validationFailed({ error: 'invite_user_failed', detail: msg });
  }

  const newId = inviteRes.user.id;

  // Profile is auto-created by trigger; set role + displayName if provided.
  // role is 'user' | 'moderator' only (schema-enforced) — never 'admin'.
  await sc.from('profiles')
    .update({ display_name: body.displayName ?? null, role: body.role })
    .eq('id', newId);

  await writeAdminAudit({
    actorUserId: caller.userId,
    targetUserId: newId,
    targetAuthUserId: newId,
    action: 'invite_user',
    payload: {
      emailDomain: body.email.split('@')[1] ?? null,  // domain only — never the raw address
      role: body.role,
      redirectToProvided: Boolean(body.redirectTo),    // boolean — never the URL string
      invited: true,
      notification: 'sent',
    },
  });

  return ok({ ok: true, invited: true, notification: 'sent' });
}
```

Notes on the handler:
- It mirrors `handleCreateUser` (`index.ts` lines 221-266) almost exactly — same `profiles.update` for role/displayName, same audit shape, same `Req<>` / `Caller` / `SC` type aliases already declared in the file.
- The audit `payload` stores `emailDomain` not `email`, and `redirectToProvided: boolean` not the URL. This is the established `handleSendPasswordReset` pattern (`index.ts` line 367) and keeps a raw address / a URL with a possible token out of the audit row. `writeAdminAudit`'s `sanitizePayload` additionally strips any `token`/`password`/`api_key` key as a backstop.
- The response object `{ ok, invited, notification }` contains **no** user id, **no** email, **no** link, **no** token. (Contrast `handleCreateUser`, which returns `userId` + `email` — that is acceptable for a password-create where the admin needs the id immediately, but the invite flow has no such need and the card explicitly says "never returns admin email addresses or tokens." We omit even the invited user's email to be maximally conservative; the admin already typed it.)
- `inviteUserByEmail` is called on `sc` — the **service client** built by `requireAdmin` *after* the admin check. No service-role key touches the client. `Authorization` is never logged: the function's only `console.error` is the generic `admin_users_error` catch in `index.ts` line 96, which logs `body.action` + the error object — `body.action` is `'invite_user'` (a literal), not a secret, and the error from `inviteUserByEmail` carries no token.

**Response shapes:**

| Outcome | HTTP | Body |
|---|---|---|
| Invite sent | 200 | `{ ok: true, invited: true, notification: 'sent' }` |
| Email/SMTP not configured | 422 | `{ error: 'invite_email_not_configured', invited: false, notification: 'not_configured' }` |
| Bad email / duplicate / other Supabase failure | 422 | `{ error: 'invite_user_failed', detail: '<sanitized message>' }` |
| Schema validation failed (e.g. `role: 'admin'`) | 422 | `{ error: 'validation_failed', issues: [...] }` |
| Not an admin | 403 | `{ error: 'forbidden', reason: 'admin_required' }` |
| Missing/invalid JWT | 401 | `{ error: 'unauthorized' }` |

### Client wrapper: `adminInviteUser`

```ts
// src/features/admin/adminApi.ts
export async function adminInviteUser(
  input: AdminInviteUserInput,
): Promise<AdminUsersResult<{ ok: true; invited: true; notification: 'sent' }>> {
  // QOL-023: derive the invite-email callback URL from buildAuthRedirectUrl so
  // the email points at the deployed origin, never a hard-coded localhost.
  // A bad origin degrades to `undefined` — the Edge Function then lets Supabase
  // fall back to the dashboard Site URL. The invite is never blocked by a
  // redirect-config defect (fail-soft, mirrors adminSendPasswordReset).
  let redirectTo = input.redirectTo;
  if (!redirectTo) {
    try {
      redirectTo = buildAuthRedirectUrl({
        kind: 'invite',
        runtimeOrigin: resolveRuntimeOrigin(),
        isDev: getIsDev(),
      });
    } catch {
      redirectTo = undefined;
    }
  }
  return adminUsers({
    action: 'invite_user',
    email: input.email,
    displayName: input.displayName,
    role: input.role ?? 'user',
    redirectTo,
  });
}
```

- `buildAuthRedirectUrl`, `resolveRuntimeOrigin`, `getIsDev` are **already imported** at the top of `adminApi.ts` (lines 10-11) — no new imports needed.
- `kind: 'invite'` resolves to `DEFAULT_AUTH_ROUTES.invite === '/auth/callback'` (`buildAuthRedirectUrl.ts` line 49) — the invite link lands on `/auth/callback`, the same route as signup confirmation. Correct: an invited user clicking the link is completing a confirmation-style flow.
- The wrapper passes only `action`, `email`, `displayName`, `role`, `redirectTo` — it does **not** spread `...input`, so no stray field can leak into the payload (contrast `adminCreateUser`, which spreads; the explicit field list here is a tighter, intentional choice and is asserted by `adminInviteUserClient.test.ts`).
- Return type is `AdminUsersResult<...>` (the existing discriminated `{ ok: true; data } | { ok: false; error; status }` envelope from `edgeFunctions.ts`). The form reads `r.ok` and `r.data` / `r.error` the same way every other wrapper's caller does.

### `AdminCreateUserForm` UI contract

New state: `const [mode, setMode] = useState<'invite' | 'password'>('invite');` — **invite is the default** (per the card).

The form has two independent toggle rows for the **human** path:
1. Existing Human/Bot row (`create-human-toggle` / `create-bot-toggle`) — unchanged.
2. **New** Invite/Password row, rendered **only when `!isBot`** — `accessibilityLabel="invite-mode-toggle"` and `"password-mode-toggle"`, label text "Invite by email" / "Create with password". ## Edge cases

The implementer must handle each explicitly; most have a named test.

- **Empty / whitespace email** — the form's existing guard (`!email.trim() || !email.includes('@')`) blocks submit before any call. The zod `z.string().email()` is the server backstop. Both layers stay.
- **`role: 'admin'` somehow reaches the function** — rejected by `z.enum(['user','moderator'])` with a 422 `validation_failed`. The client wrapper never sends `'admin'` (the form hard-codes `role: 'user'`), but the schema is the real gate.
- **Duplicate email (user already exists)** — `inviteUserByEmail` returns an error; the handler returns `422 { error: 'invite_user_failed', detail: <message> }`. The form shows the sanitized detail. The card lists "duplicate-email path" as a required test case. Note: Supabase's behaviour for inviting an already-registered email may be either an error or a re-invite depending on the user's confirmation state — the handler treats any non-success as the failure path; the test asserts the wrapper surfaces it as `ok: false` and shows a non-empty message. Do **not** try to special-case "already exists" into a success — that would leak account-existence information; a generic "could not send" is the safer copy. (See Risks.)
- **SMTP / email provider not configured** — `inviteUserByEmail` fails; handler returns `422 { error: 'invite_email_not_configured', invited: false, notification: 'not_configured' }`. The form shows the operator-directed copy. This is the card's `notification: 'not_configured'` value.
- **Bad / non-https redirect origin (non-dev build)** — `buildAuthRedirectUrl` throws `InvalidAuthRedirectOrigin`; `adminInviteUser`'s `try/catch` sets `redirectTo = undefined`; the function receives no `redirectTo` and Supabase uses the dashboard Site URL. The invite still sends. (Fail-soft — a redirect-config defect must never block an invite.)
- **Offline / network failure** — `supabase.functions.invoke` errors; `adminUsers` maps it to `{ ok: false, error: { error: 'network_error' }, status: 503|500 }`; `adminErrorMessage` surfaces it. The form shows the error and stays open so the admin can retry. No partial state on the client.
- **`admin-users` function not yet deployed** (operator has not run the deploy step) — `invoke` returns a 404; `adminUsers` → `{ ok: false, status: 404 }`; `adminErrorMessage` returns "admin-users function is not deployed yet." This is expected until the operator deploys; the card flags the deploy as a post-merge operator step.
- **Permission-denied (non-admin caller)** — cannot happen from the UI (the Admin tab is role-gated by `getVisibleTabs`, `adminSecurity.test.ts`), but a non-admin calling the Edge Function directly hits `requireAdmin` → 403. `adminInviteUserClient.test.ts` covers the 403 path by mocking the transport's 403 response.
- **Concurrent invites of the same email** — two admins inviting the same address near-simultaneously: the second `inviteUserByEmail` fails on the now-existing user; handled as the duplicate path above. No DB corruption — `auth.users` enforces email uniqueness.
- **Display name with only whitespace** — the form sends `displayName.trim() || undefined`; the schema's `z.string().min(1)` rejects an empty string, so `undefined` is correct.
- **Doctrine edge — could an invite mint an admin?** No. `role` is `z.enum(['user','moderator'])`; `'admin'` is a 422. An admin is only ever created via `create_user` + `confirmAdminCreate` or `update_role` + `confirmAdminGrant`. Verified in `adminInviteUserAuditShape.test.ts` / the schema mirror test.
- **Doctrine edge — does the invite link or token ever reach the client?** No. `inviteUserByEmail` (not `generateLink`) is used precisely so the function never holds the link. The response is `{ ok, invited, notification }` — no `link`, no `token`, no `email`, no `userId`. Asserted in `adminInviteUserAuditShape.test.ts`.
- **Doctrine edge — does a raw email land in an audit row or a log?** No. The audit `payload` stores `emailDomain` only; `writeAdminAudit`'s `sanitizePayload` strips token-shaped keys; the function logs only `body.action`. Asserted in the audit-shape test.

### `__tests__/adminInviteUserClient.test.ts` — client wrapper, mocked transport (~11 cases)

## Risks

- **`admin_audit_events.action` column type.** This design asserts no migration is needed because the column is `text` (all fourteen current actions are added by editing a JS array, never SQL). **The implementer must verify this** by reading the migration that creates `admin_audit_events` (grep `supabase/migrations/` for `admin_audit_events`). If the column is a Postgres `enum`, adding `'invite_user'` requires an `ALTER TYPE ... ADD VALUE` migration — a new, append-only migration file, and a **loud operator-gated `npx supabase db push --linked` step**. The design's "no migration" claim is contingent on this check. The strong prior (the existing 14-action array pattern) says `text`, but verify before claiming done.
- **`inviteUserByEmail` availability + signature in the pinned SDK.** `supabaseClients.ts` imports `npm:@supabase/supabase-js@2`. `auth.admin.inviteUserByEmail` has existed since early v2, so availability is near-certain — but the implementer must confirm the exact signature (`inviteUserByEmail(email, options?)` where `options` carries `data` and `redirectTo`) against the installed minor version, because older v2 minors took `redirectTo` differently. If the signature differs, adapt the handler — do not silently drop `redirectTo`.
- **Account-existence enumeration on duplicate email.** Inviting an email that already has an account produces a different error than inviting a fresh one. Surfacing a precise "this user already exists" message would let an admin (or anyone who reached the function) enumerate accounts. The design deliberately shows a **generic** "could not send the invite" for any non-success. The implementer must not "improve" this into a specific existence message. (Low impact — the caller is already an authenticated admin — but it is the correct conservative default and the test asserts the generic copy.)
- **The `notification: 'send_failed'` member may be unreachable.** Current `@supabase/supabase-js@2` treats `inviteUserByEmail` failures atomically (no user row on SMTP failure), so the handler in practice returns `'sent'` or `'not_configured'` or a hard failure. `'send_failed'` stays in the response union for forward-compat but may never be emitted. Document this with a code comment so a future reader does not think the path is dead by mistake.
- **`adminSchemas.test.ts` drift.** The schema mirror is hand-maintained. If the implementer changes the `InviteUser` schema in the Edge Function but forgets the mirror, the mirror test passes against a stale shape. The card-level mitigation: the implementer adds the `InviteUser` mirror in the **same commit** as the Edge Function schema, and the audit-shape test cross-checks the field list.
- **No existing `AdminCreateUserForm` test = no regression net for the existing form.** Because there is no current test file, the implementer is adding the *first* coverage while also changing the form. The Bot-branch and password-branch regression cases in the new test file are therefore load-bearing — they must be written and passing before the invite branch is considered done.
- **Operator deploy gap.** The `invite_user` action does nothing until the operator runs `npx supabase functions deploy admin-users --linked`. Until then, the form's invite button hits a function whose deployed version has no `invite_user` case — the request fails the `isWhitelistedAction` / union check on the *old* deployed code and returns a 400/422. The UI degrades safely (shows an error, stays open). This is the same pattern as Stage 6.1.8's `request-argument-deletion` — written by the implementer, deployed by the operator — and is called out loudly in "Operator steps."
- **Invite-email template + redirect allow-list.** Even after the function deploys, the invite email only works end-to-end if the Supabase **"Invite user"** email template is enabled and `https://dev.cdiscourse.com/auth/callback` is in the Auth redirect allow-list (the same allow-list QOL-023's runbook covers). If it is not, `inviteUserByEmail` may still succeed (queue the email) but the link bounces. Operator-gated; in the runbook.
