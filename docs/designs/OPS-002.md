## §1 — Current state

### §1.1 — `spawn-card.ps1` generates a `feat/<code>-<slug>` branch name

`.claude/scripts/spawn-card.ps1` lines 73–76 compute a branch name from the
GitHub issue title and embed it in the next-step Agent-tool prompt:

```
# 4. Compute slug for the branch name (used by the agent prompt)
$slug = ($issue.title -replace "^$Code\s*-\s*", '' -replace "[^a-zA-Z0-9]+", '-' -replace '^-+|-+$', '').ToLower()
if ($slug.Length -gt 40) { $slug = $slug.Substring(0, 40).TrimEnd('-') }
$branchName = "feat/$Code-$slug"
```

The bash twin at `.claude/scripts/spawn-card.sh` lines 110–118 produces the
same result via `sed | tr | cut -c1-40`. The branch name flows into the
designer prompt at PS1 line 100 (`Branch (create with this name):
$branchName`) and into the build/review prompts at PS1 lines 116 and 132
(`Branch: $branchName (already exists)`).

For OPS-002 itself, the title `OPS-002: Pipeline operational hygiene
(spawn-card branch naming + worktree cleanup)` yields the slug
`pipeline-operational-hygiene-spawn-card-bra` (40 chars, truncated; the
final word `branch` becomes `bra`), producing the branch name
`feat/OPS-002-pipeline-operational-hygiene-spawn-card-bra` — exactly the
branch the orchestrator gave to this designer subagent. (Side note: the
strip regex `^$Code\s*-\s*` expects a dash separator between the code and
the title; modern issue titles use a colon `OPS-002:` so the strip is a
no-op and the slug starts with `ops-002-` doubled. This is cosmetic and
out of scope for OPS-002 — flagged in §7.)

### §1.2 — The implementer subagent's `isolation="worktree"` runtime creates an auto-branch

When the orchestrator paste the spawn-card prompt and the subagent starts
with `isolation="worktree"`, the Claude Code runtime creates a fresh git
worktree under `.claude/worktrees/agent-<hash>/` and checks out a new
local branch named `worktree-agent-<hash>` (not the
`feat/<code>-<slug>` name the spawn-card prompt asks for). The subagent
does its work, runs `git status` / `git log` / `git diff`, and commits to
`worktree-agent-<hash>`. The implementer charter at
`.claude/agents/roadmap-implementer.md:14` describes the working
directory as "the existing worktree on branch `feat/<code>-<slug>`," but
the runtime does not honour that name; the charter sentence is
aspirational, not enforced.

Direct empirical evidence is visible from this OPS-002 designer
subagent's own session:

- `git rev-parse --show-toplevel` →
  `C:/Users/kyler/cdiscourse/debate-constitution-app/.claude/worktrees/agent-a0312bb12c8ece35e`
- `git branch --show-current` → `worktree-agent-a0312bb12c8ece35e`

The orchestrator's pre-implementer survey reported the same pattern
across QOL-039, RECON-001, QOL-040.3, and QOL-036.1.

### §1.3 — Push-time reconciliation workaround

The four cited sessions (QOL-039, RECON-001, QOL-040.3, QOL-036.1) all
shipped successfully via the operator's manual push reconciliation:

```
git push origin worktree-agent-<hash>:feat/<code>-<slug>
```

The colon-separated refspec pushes the auto-generated local branch to a
remote branch named per the charter convention. The PR then opens
against `feat/<code>-<slug>` and squash-merges to `main` (verified via
recent merge log: PRs #272, #273, #274). The mechanic works; it is not a
correctness bug. It is a friction bug: the orchestrator must remember
the refspec, and a typo breaks the next step.

### §1.4 — Orphan worktree accumulation

`git worktree list` at OPS-002 designer time returns 96 entries (the
issue-time snapshot was 94; the count grew by two during the subsequent
cards). Every entry under `.claude/worktrees/agent-<hash>/` is marked
`locked`. The two branch-name forms — `worktree-agent-<hash>` and
`feat/<code>-<slug>` — both appear in `git branch -a` for cards whose
implementer used the auto-branch path and whose operator pushed via the
reconciliation refspec.

Concrete friction signals:

- `git worktree list` output is hundreds of lines, drowning out the
  current-card worktree.
- `gh pr merge --delete-branch` fails when any worktree still
  references the merged branch — Git refuses to delete a branch a
  worktree is on. The operator currently works around this with
  `git worktree remove --force <path>` before the delete, which is
  manual and undocumented.
- Disk usage: each worktree carries a full project tree (~1.3 GB at
  Stage 6.4 with `node_modules/` per worktree depending on whether the
  subagent installed). At 96 worktrees the disk hit is non-trivial on
  a developer laptop.

The orchestrator captured the four-session pattern in the issue body's
"Trigger" section; the survey is accurate.

## §2 — Proposed alignment

### §2.1 — Three options considered

The issue body explicitly enumerates three options:

- **Option A** — Update spawn-card.ps1's output to *surface* the
  auto-branch reality in the prompt. Add a sentence to the designer /
  build / review prompt blocks: "The worktree's auto-generated branch
  `worktree-agent-<hash>` is what your commits land on; the
  `feat/<code>-<slug>` name is the *intended* remote name and is used
  only at push time." No behavioural change to the subagent; the
  orchestrator and operator both see the truth.
- **Option B** — Add an instruction to the implementer charter so the
  subagent's first checkout step renames the auto-branch:
  `git branch -m worktree-agent-<hash> feat/<code>-<slug>`. The local
  branch then matches the prompt, and the operator pushes a normal
  `git push -u origin feat/<code>-<slug>` with no refspec surgery.
- **Option C** — Skip branch naming in spawn-card.ps1 entirely. The
  auto-branch becomes the source of truth; the spawn-card prompt
  states the auto-branch shape; the operator's push command is
  `git push origin <auto-branch>:feat/<code>-<slug>` and the
  charter documents the refspec as the standard path.

### §2.2 — Recommendation: Option B

Option B is the smallest change, eliminates push-time refspec friction,
and matches the existing charter language (the implementer charter
already *says* the branch is `feat/<code>-<slug>`; the implementer just
needs one explicit early step to make it so). Justification matrix:

| Criterion | A | B | C |
|---|---|---|---|
| Lines of edit | ~12 (3 prompt blocks × 4 lines) | ~6 (one new step in implementer charter) | ~15 (script edit + charter rewrite) |
| Push-time friction | persists | gone | normalised but still refspec-shaped |
| Matches existing charter language | no — charter remains aspirational | yes — charter becomes enforceable | requires charter rewrite |
| Risk of subagent skipping the step | n/a (no behaviour change) | low — first-checkout step is mechanical and grepable | n/a (charter explicitly endorses auto-branch) |
| Operator mental model | "two branch names; push translates" | "one branch name throughout" | "two branch names; push uses refspec" |
| Compatibility with previously-shipped 4 sessions | no change needed | no change needed; future cards only | no change needed |

Option B wins on every row except the no-op "previously-shipped"
compatibility tie. The cited cards shipped under the workaround; OPS-002
prevents the workaround from being needed on the *next* card.

### §2.3 — Option B mechanical specification

The implementer adds a new **first** step to
`.claude/agents/roadmap-implementer.md`'s "Phase order" (currently the
first step is "Verify clean baseline" at line 33). The new step is
inserted as step 1; the existing steps renumber to 2–7.

Exact text to insert (verbatim, including code block):

```markdown
1. **Rename the worktree auto-branch to the named feat branch.**
   The `isolation="worktree"` runtime checks out a new local branch
   named `worktree-agent-<hash>`, not the `feat/<code>-<slug>` branch
   the spawn-card prompt names. Rename it as your first action so the
   rest of the pipeline (commits, charter language, operator push,
   PR title, squash-merge) uses one consistent name.

   ```
   # Confirm the current auto-branch (sanity check).
   git rev-parse --show-toplevel        # must be .claude/worktrees/agent-<hash>
   git branch --show-current            # must start with "worktree-agent-"

   # Rename to the named feat branch from the spawn-card prompt.
   git branch -m feat/<code>-<slug>
   git branch --show-current            # must now be feat/<code>-<slug>
   ```

   The rename is a local-only operation (no remote refs, no force
   push) and is safe to run on a fresh worktree before any commit.
   If `git branch -m` reports "a branch named feat/<code>-<slug>
   already exists," the worktree was reused from a prior card with

```

The existing "Verify clean baseline" step becomes step 2; all
subsequent steps renumber. No other charter text changes.

The charter's "Inputs you'll receive" block at line 14 already reads

`feat/<code>-<slug>`." After OPS-002 the rename step makes that
sentence factually true (it is currently aspirational; the rename step
operationalises it). No edit to that line is needed.

- The implementer subagent's `isolation="worktree"` runtime behaviour
  is NOT modified. The runtime is owned by the Claude Code harness;

## §3 — Worktree cleanup procedure

### §3.1 — When the cleanup runs

The cleanup runs **after the PR has merged on GitHub** (squash-merge
into `main`), not before. Two reasons: (1) `gh pr merge --delete-branch`
removes the *remote* branch; the cleanup removes the *local* worktree
plus its auto-branch; (2) running cleanup before merge risks deleting
work-in-progress if the reviewer requested changes.

### §3.2 — The procedure (verbatim text to add to a charter)

The procedure lives as a new top-level section in
`.claude/agents/roadmap-reviewer.md` after the existing "What you must
NOT do" block at line 188 and before the "Verdict rules" block at line
195. The section is reviewer-owned because the reviewer's "Operator
next steps" block already lives there and the cleanup is the operator's
immediate follow-up to a successful merge.

The section is titled exactly **"Post-merge worktree cleanup
(operator step)"** and its body matches the block below verbatim.

```markdown
## Post-merge worktree cleanup (operator step)

After the PR merges to `main`, the orchestrator runs the cleanup
procedure below to remove the implementer's worktree and the
auto-generated local branch. This step is the operator's, not the
reviewer subagent's — the reviewer cannot delete its own worktree
mid-session. The reviewer's "Operator next steps" block names the
deploy command; this section adds the cleanup command beneath it.

Run from the main repo root (not from inside a worktree):

```
# 1. Identify the worktree path for this card.
git worktree list | grep "feat/<code>-<slug>"
#   → C:/Users/kyler/cdiscourse/debate-constitution-app/.claude/worktrees/agent-<hash>

# 2. Remove the worktree. --force is required because the worktree
#    was created with isolation="worktree" and is marked locked.
git worktree remove --force ".claude/worktrees/agent-<hash>"

# 3. Delete the local auto-branch (the rename step in the implementer
#    charter renames worktree-agent-<hash> to feat/<code>-<slug>, so
#    after merge the local feat branch is the only one left; if a
#    pre-OPS-002 card was used, both branches may exist).
git branch -D feat/<code>-<slug>          # post-OPS-002 cards
git branch -D worktree-agent-<hash>       # pre-OPS-002 cards (if present)

# 4. Verify cleanup.
git worktree list | grep -c "agent-<hash>"   # must print 0
git branch -a | grep -E "feat/<code>-<slug>|worktree-agent-<hash>"
#   → only the remote-tracking branch remains, and after `gh pr merge
#     --delete-branch` even that is gone.
```

**Cross-platform notes.**

- The commands above are POSIX-shape but run identically in PowerShell
  (`git` accepts the same arguments on Windows). The path separator
  `/` works in PowerShell for `git worktree remove`; backslashes are
  not required.
- On Windows, `grep` is not native PowerShell; substitute
  `Select-String` if running PowerShell-only: `git worktree list |
  Select-String "feat/<code>-<slug>"`. The orchestrator typically
  runs the cleanup from the Bash tool (Git-for-Windows MSYS bash) where
  `grep` works directly.
- `git worktree remove --force` will fail if the worktree path is
  the operator's current working directory. The operator must `cd` to
  the main repo root first (the procedure's first sentence
  enforces this).
- If `gh pr merge --delete-branch` ran *before* `git worktree
  remove`, the remote-branch delete may have failed silently because
  a worktree was still on it. Re-run `gh api -X DELETE
  repos/{owner}/{repo}/git/refs/heads/feat/<code>-<slug>` after
  cleanup to finish the remote delete, or simply let
  `git remote prune origin` clear the stale tracking ref on the next
  fetch.

**Bulk cleanup of historical orphans.**

For the 96 existing orphan worktrees (one-time historical debt, not
created by post-OPS-002 cards), the operator may run a single
PowerShell pass to remove every locked worktree whose branch has
already merged:

```
# Dry run first — list every worktree under .claude/worktrees/ except
# the currently-active one.
git worktree list --porcelain |
  Select-String -Pattern '^worktree '

# Then for each non-active path the orchestrator confirms is safe,
# run:
git worktree remove --force "<path>"
git branch -D "<branch>"
```

The bulk pass is **operator-judgement-gated** — the orchestrator does
not automate it because mis-identifying a non-merged worktree as
orphan loses work. OPS-002 only mandates the per-card cleanup in steps
1–4 above; the bulk pass is informational.
```

### §3.3 — Why the procedure lives in the reviewer charter (not a new file)

The issue body offers three placement options:
`.claude/agents/roadmap-implementer.md` post-build, `.claude/agents/
roadmap-reviewer.md` post-merge, or a new
`.claude/agents/operational-hygiene.md`. The reviewer charter wins:

| Location | Why it is NOT chosen |
|---|---|
| `roadmap-implementer.md` post-build | The implementer cannot run cleanup on its own worktree mid-session (it is sitting in it). The cleanup is post-merge, which is post-implementer. |
| New `.claude/agents/operational-hygiene.md` | Adds a fourth agent file; `docs/core/agent-charters.md` lines 17–26 explicitly states "new specialized `.claude/agents/*.md` files are NOT created" — the three pipeline agents are the executable layer. A new file violates that doctrine. |
| `roadmap-reviewer.md` post-merge ✅ | The reviewer charter already owns the "Operator next steps" block (lines 182–185). The cleanup is the next operator step after push, PR open, and merge. Same charter, same operator-step flow, one new section. |

The reviewer subagent does not *execute* the cleanup; it documents it
in the review doc's "Operator next steps" block (which the design
extends — see §6 criterion 4). The operator (the human running the
orchestrator) executes the commands after the merge.

---

### ✅ Pipeline Branch-Naming Misalignment + Worktree Orphan Accumulation (OPS-002)

**Incident.** Four consecutive autonomous pipeline sessions — QOL-039
(#268, PR shipped), RECON-001 (#272, PR shipped), QOL-040.3 (#274, PR
shipped), and QOL-036.1 (#273, PR shipped) — required push-time
refspec reconciliation between the `spawn-card.ps1` prompt's named
branch (`feat/<code>-<slug>`) and the implementer subagent's actual
working branch (`worktree-agent-<hash>`). All four shipped successfully
via `git push origin worktree-agent-<hash>:feat/<code>-<slug>`, but the
refspec is fragile (operator-typed each time, no automation) and the
mismatch leaks into review docs and operator-next-step lists where the
branch name is referenced. Separately, 96 orphan worktrees accumulated
under `.claude/worktrees/` (all marked `locked`), each tying up disk
space and blocking `gh pr merge --delete-branch` for the merged
card. The gap is operational hygiene, not correctness — the pipeline
shipped — but the friction compounds with each card.

**Root cause.** The `isolation="worktree"` runtime in the Claude Code
harness creates a worktree on a new local branch named
`worktree-agent-<hash>` (the runtime's convention, not configurable
from the subagent prompt). The `spawn-card.ps1` and `spawn-card.sh`
scripts compute and embed `feat/<code>-<slug>` into the agent
prompt — a branch name the runtime never adopts. The implementer
charter at `.claude/agents/roadmap-implementer.md:14` describes the
working directory as the `feat/<code>-<slug>` branch, but the
description was aspirational: the charter did not previously include a
rename step. The mismatch propagated through every subsequent
charter sentence that referenced the branch name.

**Resolution (OPS-002, #275).** Two coordinated changes:

1. **Rename step added to the implementer charter as a new Phase
   order step 1.** Before any baseline check or implementation, the
   implementer subagent runs `git branch -m feat/<code>-<slug>` to
   rename the auto-branch to the named feat branch. The rest of the
   pipeline (commits, charter sentences, operator push, PR title,
   squash-merge) then uses one consistent name. The `spawn-card.ps1`
   and `spawn-card.sh` scripts are unchanged; the implementer charter
   absorbs the alignment.

2. **Worktree cleanup procedure added to the reviewer charter as a
   new section "Post-merge worktree cleanup (operator step)."** The
   operator runs the procedure after each PR merges: `git worktree
   remove --force <path>` plus `git branch -D <branch>` for both
   branch-name forms (the auto-branch for pre-OPS-002 cards is
   preserved as a fallback). Cross-platform notes cover the
   PowerShell-vs-Bash invocation; a bulk pass for the 96 historical
   orphans is documented as operator-judgement-gated and is NOT
   automated by OPS-002.

**Lesson for future sessions.** Two rules:

- **Rule 1 (alignment):** The implementer charter is the canonical
  place to operationalise a branch-name expectation. When a script
  prints a branch name into a subagent prompt, the subagent's charter
  must include the step that makes the local branch match the
  printed name. Aspirational charter sentences ("your branch is X")
  do not configure runtime behaviour; the subagent must run the
  command. The spawn-card scripts remain the source of the *name*,
  but the implementer is the source of the *binding*.

- **Rule 2 (cleanup):** Worktree creation is automated; worktree
  removal is not. Every pipeline that uses `isolation="worktree"` for
  per-card work must document a per-card cleanup procedure adjacent
  to the success-path documentation. The procedure runs after merge,
  not before. Bulk cleanup of historical orphans is
  operator-judgement-gated because mis-identifying a non-merged
  worktree as orphan loses work.

The OPS-001 reviewer-template lesson and the OPS-002 alignment lesson
together form the foundation of the OPS-* series: OPS cards capture
process bugs that ship product correctly but accumulate operational
friction; they fix the process, not the product. Future OPS cards
follow the same shape (a single design doc, a verbatim lesson block,
a minimal-footprint implementer diff, and a verification table in the
reviewer doc that confirms the lesson is recorded).

### §5.1 — Lightweight options considered

- **Option β** — Contract assertion in

Justification:

| Criterion | α | β | γ | δ |
|---|---|---|---|---|
| Catches a future spawn-card slug regression | yes | yes (via parity test) | yes | no |
| Catches a future charter-step deletion | no | yes (via charter file scan) | no | no |
| Requires PS1 refactor | yes — extract function | no | no | no |
| Requires CI infrastructure change | no | no | yes — new CI job | no |
| Runs in existing `npm run test` | no (PS only) | yes | no | n/a |

| Surfaces a failure in a code-review diff | only if PS test exists | yes — in the test run | only on CI | only on human read |

Options α and γ require new infrastructure (PowerShell test harness
or a new CI job). Option δ is the no-cost option but provides no
mechanical signal. Option β fits the existing
`npm run test` battery, requires no script refactor, and can encode

### §5.3 — Test file specification

The implementer creates one new test file:
`__tests__/spawnCardBranchName.test.ts`. Approximate length: 120
lines. The file has two test groups:

**Group 1 — Slug parity (mirrors spawn-card.ps1 lines 74–76 in TS).**

A pure-TS helper `computeSpawnCardBranchName(code, issueTitle)` lives
in the test file (not in `src/`; this is a test-only mirror, not
production code). The helper implements the exact regex sequence the
PS1 script uses:

```ts
function computeSpawnCardBranchName(code: string, issueTitle: string): string {
  // 1. Strip "<Code><whitespace>-<whitespace>" prefix.
  const stripped = issueTitle.replace(new RegExp('^' + code + '\\s*-\\s*'), '');
  // 2. Replace non-alphanumeric runs with single dash.
  const dashed = stripped.replace(/[^a-zA-Z0-9]+/g, '-');
  // 3. Trim leading/trailing dashes, lowercase.
  let slug = dashed.replace(/^-+|-+$/g, '').toLowerCase();
  // 4. Truncate to 40 chars and trim any trailing dash.
  if (slug.length > 40) slug = slug.slice(0, 40).replace(/-+$/, '');
  return `feat/${code}-${slug}`;
}
```

Tests (5):

1. Known input `('TL-001', 'TL-001 - Make Timeline...')` → known
   output `'feat/TL-001-make-timeline'`. (Verifies the dash-separator
   strip works for legacy-formatted titles.)
2. Known input `('QOL-040.3', 'QOL-040.3 - Deep-link node
   pre-activation via Stage 6.4 entry-hint extension')` → known output
   `'feat/QOL-040.3-deep-link-node-pre-activation-via-stage-6.4'`.
   (Verifies the truncation lands at a non-dash and the dot in the
   code survives.)
3. Known input `('OPS-002', 'OPS-002: Pipeline operational hygiene
   (spawn-card branch naming + worktree cleanup)')` → known output
   `'feat/OPS-002-ops-002-pipeline-operational-hygiene-spa'`. (Verifies
   the colon-separator behaviour: the strip is a no-op so `OPS-002`
   appears twice in the slug. This is the cosmetic side-issue flagged
   in §7; the test pins the *current* behaviour so a future
   spawn-card edit that "fixes" the strip does not silently break
   the four shipped cards' branch names.)
4. Empty input `('X', '')` → `'feat/X-'` (or the equivalent trailing-
   dash result the regex produces). Pins the empty-title behaviour
   so a future "trim empty" change is intentional.
5. Special-character input `('X', 'X - Foo bar! @baz $qux %quux')` →
   `'feat/X-foo-bar-baz-qux-quux'`. Pins the multi-special-character
   collapsing.

**Group 2 — Charter rename-step contract (file scan).**

A `fs.readFileSync` scan of `.claude/agents/roadmap-implementer.md`
asserts the rename step is present and well-formed:

1. The file contains the exact heading text
   `**Rename the worktree auto-branch to the named feat branch.**`
   (case-sensitive substring match).
2. The file contains the exact command `git branch -m feat/<code>-<slug>`
   (literal substring match).
3. The rename step appears *before* the "Verify clean baseline"
   step (string-index comparison) — guards against a future edit that
   re-orders the steps and demotes the rename.

Total test count: 5 (Group 1) + 3 (Group 2) = **8 new tests**. The
baseline at OPS-002 design time is 10479 tests / 417 suites (per
`docs/core/current-status.md` line 2). After OPS-002 the count is
expected to be 10487 tests / 418 suites.

### §5.4 — Why this is "lightweight"

- Zero production source under `src/` or `app/` is added.
- Zero new dependency.
- The test-only helper mirrors 4 lines of PS1 logic; no PS1 refactor
  needed.
- Runs in the existing `npm run test` battery; no new CI job.
- Failure mode is informative: the test name names the gap, the
  expected branch name is in the assertion message.

The bash twin (`spawn-card.sh`) is *not* covered by this test;
maintaining a triple parity (PS1 + bash + TS) is out of scope. The bash
script is a manual operator fallback and a future regression there
would surface during an operator's first attempt to use it. The
implementer charter rename step is the binding contract; the
PS1/bash *names* are inputs.

## §7 — Open question for operator (none blocking)

One cosmetic side-issue surfaced during §1.1 analysis: the
`spawn-card.ps1` strip regex `^$Code\s*-\s*` assumes a dash
separator between the issue code and the title (`OPS-002 - title`),
but modern GitHub issue titles in this repo use a colon
(`OPS-002: title`). The strip is therefore a no-op for current
titles, and the resulting slug doubles the code prefix
(`feat/OPS-002-ops-002-pipeline-operational-hygiene-spa` instead of
`feat/OPS-002-pipeline-operational-hygiene-spawn-card-bra`).

This is **out of scope for OPS-002**:

- The four cited shipped cards all shipped with the current
  behaviour. Their branch names are recorded in PR titles and merge
  commits; "fixing" the regex would silently change branch names for
  the next card and break the §5.3 Group 1 Test 3 parity assertion.
- The slug doubling is cosmetic — the branch still encodes the code
  uniquely; the duplicate prefix is at worst ugly, not broken.
- Fixing it requires a coordinated PS1 + bash + test update, which
  is its own OPS-003-shape card.

OPS-002 pins the *current* slug behaviour in the regression test
(§5.3 Group 1 Test 3) so a future intentional change is a conscious
test update, not a silent behavioural drift. The operator may file a
follow-up OPS-003 to fix the strip regex if desired; this design does
NOT propose it.
