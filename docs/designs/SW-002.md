## Goal

The roadmap card SW-002 ships the **activity model layer** that downstream UI cards (Gallery cards GAL-001 / GAL-002, Timeline mini-map IX-002, future segment-marker UI) will read to color, sort, and signal "what is going on" in a room or a subtree. It is the doctrine-sensitive replacement for the engagement / popularity / heat-as-truth instincts that the rest of the consumer web treats as default.

The card is built around a single non-negotiable rule from `cdiscourse-doctrine` §2: **heat means activity / friction, never correctness**. The model produces three coarse 3-valued unions — `HeatLevel`, `MomentumState`, `TrendDirection` — plus a small set of `ActivityChip` reason codes, and a deterministic `RoomActivityProfile`. Every literal string the module emits passes the COPY-001 ban-list (verdict + amplification + person-attribution + block-prevent tokens), with the single doctrine-approved carve-out that `hot` is a permitted `HeatLevel` value because it describes activity density.

Doctrine constraints that shape the design (all from `cdiscourse-doctrine` + the issue body + the coordinator brief):

1. **Heat / momentum / trend describe activity patterns, never correctness or who is right.**
2. **Popularity is not evidence; engagement velocity is not standing.** The deriver never reads view counts, like counts, reply velocity in absolute terms, or any participant-identity field.
3. **A "hot" cluster** is an activity descriptor (lots of recent posts, unresolved debt, repeated axis pressure) — never a claim that a side is right.
4. **A "quiet" room** is an easy-entry opportunity, never a verdict of "the conversation is decided."
5. **No `winning / losing / dominant / losing ground / gaining ground / proven / disproven / true / false / right / wrong / verdict / validated / correct / incorrect`** in any literal the module emits.
6. **Side-stratified heat is out of scope for v1** — the model is room-level only. Stratifying by side would invite a leaderboard reading, which is exactly what the doctrine refuses.
7. **No new presets, no new dependencies, no `Date.now()`, no `Math.random()`** inside the deriver. The deriver is pure, total, deterministic. `clockMs` is injected by the caller.
8. **The model produces ACTIVITY-DESCRIPTIVE data; UIs decide how to render it.** SW-002 ships no React, no rendering, no animation, no styling. Consumers (GAL-001 / GAL-002 / IX-002) read the profile read-only and are responsible for their own ban-list scans of any consumer-side derived strings.

## What SW-002 IS and IS NOT

### SW-002 IS

- A pure-TS deriver: `deriveRoomActivityProfile(input): RoomActivityProfile`. Total. Deterministic. Same input → same output (deep-equal).
- A locked set of three 3-valued unions that consumer UIs MUST use as-is (`HeatLevel`, `MomentumState`, `TrendDirection`).
- A locked set of 8 `ActivityChip` kinds — each describes a structural / temporal / lifecycle pattern, never a verdict.
- A small `ACTIVITY_HELPER_LINES` lookup that maps each chip kind to one ≤ 80-char plain-language phrase. RULE-003's helper lines are reused verbatim where an entry exists; new phrases pass the COPY-001 ban-list.
- A doctrine-table ban-list helper `_forbiddenActivityTokens()` consumed by tests.
- A set of reusable signal helpers consumed later by GAL-001 / GAL-002 / IX-002.

### SW-002 IS NOT

- A heat-map renderer. SW-002 ships the model + tests + doctrine table only. No React component, no animation, no visual layer.
- A scoring engine. Point standing / standing bands belong to `src/features/pointStanding/` and are NOT touched. SW-002 never reads `standingBand`, never writes to standing, never routes through anti-amplification.
- A side leaderboard. Sides do not get heat scores. The model is room-level only.
- A lifecycle / metadata / branch-topology deriver. SW-002 READS `PointLifecycleMap` + `MoveMetadataLedger` + `ArgumentTimelineMapModel`. It never imports `buildPointLifecycleMap`, `buildMoveMetadataLedger`, `deriveAutoMetadataForMessage`, `gradeChallenge`, `gradeRepair`, or `applyAntiAmplification`. Enforced by a forbidden-imports test.
- A copy authoring layer for downstream UIs. Consumers compose their own segment / card labels; SW-002 supplies typed signal codes + a small helper-line table.
- A cross-room aggregator. Gallery-level signals (hot rooms, easy-entry rooms across many rooms) are GAL-001's concern; SW-002 is **per-room**.
- A heat-decay tuning surface. v1 uses a simple recent-N-moves window; tuning is a follow-up.

```ts
// src/features/strengthWeakness/heatModel.ts

import type {
  PointLifecycleClusterSummary,
  PointLifecycleMap,
  PointLifecycleState,
} from '../lifecycle';
import type {
  AutoMetadataCode,
  ClusterMetadataSummary,
  MoveMetadataLedger,
} from '../metadata';
import type {
  ArgumentTimelineMapModel,
  ArgumentTimelineMapNode,
} from '../arguments/argumentGameSurfaceModel';

/**
 * SW-002 — Heat level. A 3-valued coarse band describing activity density
 * in the room or subtree. **NEVER** a number ranking. **NEVER** a
 * percentile. **NEVER** a claim that a side is right.
 *
 * Doctrine: `hot` is an activity descriptor (lots of recent posts,
 * unresolved debt, repeated axis pressure). It is the COPY-001-approved
 * carve-out — the only token in the SW-002 vocabulary that survives the
 * verdict-adjacent scan because doctrine §2 explicitly permits "hot =
 * activity". The ban-list test EXCLUDES `hot` for this exact reason.
 */
export type HeatLevel = 'quiet' | 'active' | 'hot';

export const ALL_HEAT_LEVELS: ReadonlyArray<HeatLevel> = Object.freeze([
  'quiet',
  'active',
  'hot',
]);

/**
 * SW-002 — Momentum state. A 3-valued coarse band describing activity
 * CHANGE between an earlier window and the most recent window. Never
 * "who is winning." `cooling` simply means recent activity is lower than
 * earlier activity; `building` means the inverse. `steady` means no
 * meaningful change.
 */
export type MomentumState = 'cooling' | 'steady' | 'building';

export const ALL_MOMENTUM_STATES: ReadonlyArray<MomentumState> = Object.freeze([
  'cooling',
  'steady',
  'building',
]);

/**
 * SW-002 — Trend direction. A 3-valued coarse band describing the
 * STRUCTURAL pattern the room is taking on. Never a verdict.
 *
 * - `unresolved_open` — at least one cluster has an open source/quote
 *    request or rebut without a closing concession / synthesis / sourced
 *    move. The default for any active room.
 * - `narrowing_toward_synthesis` — at least one cluster sits in
 *    `synthesis_ready` and no later rebut reopened it.
 * - `branch_proliferating` — three or more branches are actively engaged
 *    in the recent window. Reads as "side issues are multiplying", never
 *    as "the conversation is decided."
 */
export type TrendDirection =
  | 'unresolved_open'
  | 'narrowing_toward_synthesis'
  | 'branch_proliferating';

export const ALL_TREND_DIRECTIONS: ReadonlyArray<TrendDirection> = Object.freeze([
  'unresolved_open',
  'narrowing_toward_synthesis',
  'branch_proliferating',
]);

/**
 * SW-002 — Activity chip kinds. Discriminated union of the 8 reason codes
 * a segment / room profile can surface. Each kind is an observation about
 * structural / temporal / lifecycle activity. **No truth chips.**
 *
 * Naming rule: snake_case internally; the chip kind itself is never
 * rendered as plain text. Consumers look up the chip's helper-line via
 * `ACTIVITY_HELPER_LINES[chip.kind]`.
 */
export type ActivityChip =
  | { kind: 'recent_post' }
  | { kind: 'open_source_request' }
  | { kind: 'repeated_axis_pressure' }
  | { kind: 'no_rebuttal' }
  | { kind: 'synthesis_ready' }
  | { kind: 'branch_tangent' }
  | { kind: 'evidence_debt_open' }
  | { kind: 'cool_room_easy_entry' };

export type ActivityChipKind = ActivityChip['kind'];

export const ALL_ACTIVITY_CHIP_KINDS: ReadonlyArray<ActivityChipKind> = Object.freeze([
  'recent_post',
  'open_source_request',
  'repeated_axis_pressure',
  'no_rebuttal',
  'synthesis_ready',
  'branch_tangent',
  'evidence_debt_open',
  'cool_room_easy_entry',
]);

/**
 * SW-002 — One segment of the timeline. A segment is a DEPTH-BAND of the
 * argument tree (e.g. depth 0–2, depth 3–5), NOT a side-band, NOT a
 * winning-band.
 *
 * The deriver builds segments by chunking nodes by `depth` into a small
 * fixed number of bands (≤ 4 in v1). Empty bands are not emitted.
 *
 * `heatLevel` is computed per-band from the same coarse mix the room
 * profile uses, restricted to nodes in that band.
 *
 * `helperLine` is composed from the segment's `reasonChips` (longest
 * helper-line wins on tie; never authored fresh).
 */
export interface ActivitySegment {
  depthRange: { from: number; to: number };
  heatLevel: HeatLevel;
  reasonChips: ReadonlyArray<ActivityChip>;
  helperLine: string;
}

/**
 * SW-002 — Where in the room is a new participant most likely to find a
 * useful first move? `null` when no clear opportunity exists (e.g.
 * archived / fully-resolved room).
 *
 * - `easy_first_move` — quiet room, root open, no rebuttal yet. The first
 *    reply will land cleanly.
 * - `mid_thread_join` — active room, multiple open clusters, no single
 *    obvious anchor. A new participant can pick any open cluster.
 * - `deep_existing_clash` — hot room with sustained same-axis pressure
 *    deep in the tree. A new participant can add evidence or narrow
 *    scope on the deepest active node.
 */
export type EntryOpportunity =
  | 'easy_first_move'
  | 'mid_thread_join'
  | 'deep_existing_clash';

export const ALL_ENTRY_OPPORTUNITIES: ReadonlyArray<EntryOpportunity> = Object.freeze([
  'easy_first_move',
  'mid_thread_join',
  'deep_existing_clash',
]);

/**
 * SW-002 — Per-room activity profile. The single thing the deriver
 * returns. Pure data, JSON-serializable. Memoization key is upstream
 * (`PointLifecycleMap.inputHash` + `MoveMetadataLedger.inputHash` +
 * `clockMs`).
 */
export interface RoomActivityProfile {
  heatLevel: HeatLevel;
  momentumState: MomentumState;
  trendDirection: TrendDirection;
  segments: ReadonlyArray<ActivitySegment>;
  entryOpportunity: EntryOpportunity | null;
}

/**
 * SW-002 — Input bundle. Optional fields are gracefully handled (null /
 * missing → cool-room default; never thrown). The deriver is total.
 *
 * Every named field from the issue body is mapped here:
 *
 *  - `move_count`             → derived from `timelineMap.nodes.length`.
 *  - `recent_activity`        → derived from a deterministic count of
 *                               nodes whose `createdAt` falls within
 *                               `recentWindowMs` relative to `clockMs`.
 *  - `branch_depth`           → derived from `max(node.depth)` over
 *                               `timelineMap.nodes`.
 *  - `unresolved_source_evidence_debt`
 *                             → derived from
 *                               `lifecycleMap.byCluster[*].hasOpenSourceOrQuoteRequest`
 *                               + `metadataLedger.byCluster[*].autoMetadataCodes`
 *                               containing `'point_stalled'` /
 *                               `'no_response_after_n_turns'` on a cluster
 *                               with `source_requested` lifecycle.
 *  - `repeated_axis_pressure` → derived from `metadataLedger.byCluster[*]
 *                               .autoMetadataCodes.includes('repeated_axis_pressure')`.
 *  - `no_rebuttal_state`      → derived from `lifecycleMap.byCluster[*]
 *                               .state === 'open'` AND the cluster's root
 *                               node has `replyCount === 0`.
 *  - `synthesis_ready_state`  → derived from `lifecycleMap.byCluster[*]
 *                               .state === 'synthesis_ready'`.
 */
export interface HeatInputBundle {
  /** Timeline map for the room. May be empty. */
  timelineMap: ArgumentTimelineMapModel | null;
  /** Lifecycle map for the room. May be empty. */
  lifecycleMap: PointLifecycleMap | null;
  /** Metadata ledger for the room. May be empty. */
  metadataLedger: MoveMetadataLedger | null;
  /** Caller-injected "now". REQUIRED. The deriver NEVER reads `Date.now()`.
   *  Callers pass `Date.now()` (or a test clock) explicitly. */
  clockMs: number;
  /** Recent-activity window in milliseconds. Default: 24 hours
   *  (86_400_000 ms). Values ≤ 0 → default. */
  recentWindowMs?: number;
  /** Pre-recent comparison window in milliseconds. Used to compute
   *  momentum (cooling vs building). Default: 72 hours (259_200_000 ms)
   *  starting at `clockMs - recentWindowMs`. Values ≤ 0 → default. */
  comparisonWindowMs?: number;
  /** Thresholds. Bounded; ignored if non-finite. Each defaults to a safe
   *  v1 value documented next to the property. */
  thresholds?: ActivityThresholds;
}

/**
 * SW-002 — Thresholds. Coarse on purpose. The four bands are
 * `quiet < active < hot` for two counts (moves and recent moves). Tuning
 * is a follow-up; v1 ships safe defaults that match the doctrine.
 */
export interface ActivityThresholds {
  /** Recent-window moves at-or-below this count → `quiet`. Default 1. */
  recentMovesQuietMax: number;
  /** Recent-window moves at-or-above this count → `hot`. Default 6. */
  recentMovesHotMin: number;
  /** Branches actively engaged in the recent window at-or-above this
   *  count → `branch_proliferating`. Default 3. */
  activeBranchesProliferatingMin: number;
  /** Recent ÷ earlier ratio at-or-above this → `building`. Default 1.5. */
  momentumBuildingRatioMin: number;
  /** Recent ÷ earlier ratio at-or-below this → `cooling`. Default 0.5. */
  momentumCoolingRatioMax: number;
  /** Max number of depth bands the segmenter emits. Default 4. */
  maxSegments: number;
}

export const DEFAULT_ACTIVITY_THRESHOLDS: Readonly<ActivityThresholds> = Object.freeze({
  recentMovesQuietMax: 1,
  recentMovesHotMin: 6,
  activeBranchesProliferatingMin: 3,
  momentumBuildingRatioMin: 1.5,
  momentumCoolingRatioMax: 0.5,
  maxSegments: 4,
});

/**
 * SW-002 — Activity chip helper-line lookup. Each entry is a single
 * plain-language phrase ≤ 80 chars. Where an entry exists in RULE-003's
 * `LIFECYCLE_UX_MAP[state].helperLine`, the SW-002 entry reads from
 * RULE-003 verbatim (no parallel copy authored).
 *
 * Frozen at module-load time. Consumers MUST NOT mutate.
 */
export const ACTIVITY_HELPER_LINES: Readonly<Record<ActivityChipKind, string>>;

/**
 * SW-002 — The deriver. Pure, total, deterministic. Never throws. Never
 * returns `undefined`. Same input twice → deep-equal output.
 *
 * Order of resolution:
 *   1. Validate `clockMs` (finite number; non-finite → 0 fallback).
 *   2. Compute counts: `totalMoves`, `recentMoves`, `comparisonMoves`,
 *      `maxDepth`, `activeBranchesInRecentWindow`.
 *   3. Heat = compare(`recentMoves`, `recentMovesQuietMax`,
 *      `recentMovesHotMin`).
 *   4. Momentum = ratio(`recentMoves`, `comparisonMoves`); apply
 *      `momentumBuildingRatioMin` / `momentumCoolingRatioMax` thresholds.
 *   5. Trend = priority resolution:
 *      (a) `narrowing_toward_synthesis` if ≥ 1 cluster in `synthesis_ready`
 *          AND no later rebut.
 *      (b) `branch_proliferating` if `activeBranchesInRecentWindow ≥
 *          activeBranchesProliferatingMin`.
 *      (c) `unresolved_open` otherwise.
 *   6. Segments — chunk nodes by `depth` into at most `maxSegments` bands;
 *      compute heatLevel + chips per band.
 *   7. EntryOpportunity = if heat === 'quiet' AND root unanswered →
 *      `easy_first_move`; else if heat === 'hot' AND maxDepth ≥ 4 →
 *      `deep_existing_clash`; else if total > 0 → `mid_thread_join`;
 *      else `easy_first_move`.
 *
 * Empty / null inputs → cool-room default:
 *   { heatLevel: 'quiet', momentumState: 'steady', trendDirection:
 *     'unresolved_open', segments: [], entryOpportunity: 'easy_first_move' }
 */
export function deriveRoomActivityProfile(input: HeatInputBundle): RoomActivityProfile;

/**
 * SW-002 — Pure helper exposed for AN-003 diagnostics + tests. Returns
 * the activity chips that apply to the entire room (the same chips the
 * room-level helper-line composer would draw on).
 */
export function deriveRoomActivityChips(input: HeatInputBundle): ReadonlyArray<ActivityChip>;

/**
 * SW-002 — Ban-list scan helper for tests. Returns the concatenated
 * dedup'd token list from:
 *   - `_forbiddenLifecycleTokens()` (LIFE-001),
 *   - `_forbiddenMetadataTokens()` (META-001),
 *   - The COPY-001 person-attribution tokens (`you`, `your`, `they`,
 *     `their`, `the user`, `the author`, `the poster`, `the participant`).
 *   - The SW-002-specific verdict tokens: `winning`, `losing`, `dominant`,
 *     `losing ground`, `gaining ground`, `proven`, `disproven`, `verdict`,
 *     `validated`, `correct`, `incorrect`.
 *
 * `hot` is DELIBERATELY NOT in the returned list. SW-002 uses `hot` as
 * an activity descriptor per doctrine §2 (`cdiscourse-doctrine` §2 +
 * COPY-001 §5.1 + §8 carve-out). The ban-list test in
 * `__tests__/heatModel.test.ts` documents this carve-out inline.
 */
export function _forbiddenActivityTokens(): string[];
```

### Mapping: input bundle → derived counts

The deriver computes the following intermediate counts deterministically. None of them are exported; all are exercised by tests via outputs.

| Internal count | Source field(s) |
|---|---|
| `totalMoves` | `timelineMap.nodes.length` |
| `recentMoves` | `count(node : node.createdAt parsed → ms, ms ≥ clockMs − recentWindowMs, ms ≤ clockMs)` |
| `comparisonMoves` | `count(node : ms ≥ clockMs − recentWindowMs − comparisonWindowMs, ms < clockMs − recentWindowMs)` |
| `maxDepth` | `max(timelineMap.nodes[*].depth, default 0)` |
| `activeBranchesInRecentWindow` | `distinct(node.branchId for node in recent window where node.kindLabel ≠ 'root')` |
| `clustersInSynthesisReady` | `lifecycleMap.byCluster.values()` filtered by `state === 'synthesis_ready'` |
| `clustersWithOpenSourceRequest` | `lifecycleMap.byCluster.values()` filtered by `hasOpenSourceOrQuoteRequest === true` |
| `clustersWithRepeatedAxisPressure` | `metadataLedger.byCluster.values()` filtered by `autoMetadataCodes.includes('repeated_axis_pressure')` |
| `clustersWithEvidenceDebt` | `metadataLedger.byCluster.values()` filtered by `autoMetadataCodes.includes('point_stalled')` OR `manualTagCodes.includes('evidence_debt')` |

`createdAt` parsing uses `new Date(node.createdAt).getTime()` once per node, guarded by `Number.isFinite` — bad timestamps are silently skipped (not thrown).

## API / interface contracts

Consumers (GAL-001 / GAL-002 / IX-002, future segment-marker UI) MUST follow these rules:

1. **Read-only.** Consumers receive a `RoomActivityProfile` and read it without mutation. The profile is REGENERATED every render — there is no persistence, no Supabase column, no cache outside the caller's memoization.
2. **No content ordering or filtering.** The profile is NEVER used for ordering, scoring, or filtering CONTENT. It MAY be used for ordering UI HINTS (e.g. "show easy-entry rooms higher in the gallery") but never for hiding content.
3. **No truth labels.** Consumers MUST NOT compose a label from a profile field that implies any side is right. Acceptable: "Recent activity here." Unacceptable: "Affirmative is winning."
4. **Memoization key.** Callers memoize on the upstream hashes plus `clockMs`:
   ```ts
   const key = `${lifecycleMap?.inputHash ?? '-'}|${metadataLedger?.inputHash ?? '-'}|${clockMs}|${thresholdsHash}`;
   ```
   `thresholdsHash` is a simple stable string `recentMovesQuietMax|recentMovesHotMin|...`. SW-002 ships a `_buildThresholdsHash(thresholds): string` internal helper for tests (not exported by `index.ts`).
5. **clockMs ownership.** Callers (typically a React hook that takes `Date.now()` once per render) pass `clockMs` explicitly. SW-002 NEVER reads the clock itself. This makes the deriver testable with frozen timestamps.

The doctrine boundary: SW-002 produces ACTIVITY-DESCRIPTIVE data; UIs decide how to render. If a UI later misuses the data to imply truth (e.g. composes a string `"the dominant side"`), that is a UI doctrine violation tracked in that consumer card, not in SW-002.

## Edge cases

Each is exercised by an explicit test.

1. **Empty room** — `timelineMap.nodes.length === 0`. Returns the cool-room default: `{ heatLevel: 'quiet', momentumState: 'steady', trendDirection: 'unresolved_open', segments: [], entryOpportunity: 'easy_first_move' }`.
2. **Null inputs** — any of `timelineMap`, `lifecycleMap`, `metadataLedger` is `null`. Returns the cool-room default. The deriver never throws.
3. **Malformed `clockMs`** — `Number.isFinite(clockMs) === false` (NaN, Infinity, undefined coerced). Treated as `0`. The deriver still returns a valid profile (typically cool-room because no node's `createdAt` parses as ≥ 0 within any non-positive window).
4. **Single root, no replies** — `move_count = 1`, root open. Returns `quiet / steady / unresolved_open` with chip `no_rebuttal`. `entryOpportunity = 'easy_first_move'`.
5. **Many moves but all > 7 days old** — `totalMoves > 0`, `recentMoves === 0`, `comparisonMoves > 0`. Returns `active / cooling / unresolved_open`. (The room reads as "active" because total > recent-quiet threshold, but momentum cools because recent ÷ comparison < `momentumCoolingRatioMax`.)
6. **Many recent moves, no source / synthesis** — high `recentMoves`. Returns `hot / building / unresolved_open` (never "winning"). Chips include `recent_post`. `entryOpportunity = 'deep_existing_clash'` if `maxDepth ≥ 4`; else `mid_thread_join`.
7. **All clusters in `synthesis_ready`** — at least one cluster has `state === 'synthesis_ready'` with no later rebut. Returns `active / steady / narrowing_toward_synthesis`. Chip `synthesis_ready` is present.
8. **3+ active branches in recent window** — Returns `… / … / branch_proliferating`. Chip `branch_tangent` is present.
9. **Heat AND no_rebuttal in same room (rare)** — root is open AND `recentMoves` is high (the latest recent moves landed on a different cluster). Heat wins for `heatLevel`; `reasonChips` includes both `recent_post` and `no_rebuttal`. This is the case the issue body singles out.
10. **All inputs zero / cool room** — `totalMoves === 0`, every cluster map empty. Returns the cool-room default plus chip `cool_room_easy_entry` on the (empty) segment summary.
11. **Deeply nested branch but only one chain active** — `maxDepth = 6`, `activeBranchesInRecentWindow = 1`. Returns `active / steady / unresolved_open`. Segments emit ≥ 2 depth bands.
12. **Concurrent edits** — N/A. SW-002 is a pure deriver; concurrency is the upstream model layer's concern. Calling twice with the same input is idempotent (same deep-equal output). Calling once during render and once during memoization yields the same profile.
13. **Offline / network failure** — N/A. SW-002 issues zero network calls.
14. **Permission-denied paths** — N/A. SW-002 reads in-memory model objects only; auth gating lives upstream at the row-level (RLS) before the lifecycle map is built.
15. **Bad threshold input** — non-finite or negative `recentMovesQuietMax` / `recentMovesHotMin`. The deriver substitutes `DEFAULT_ACTIVITY_THRESHOLDS[field]` per-field. A test fixture asserts each substitution path.
16. **`recentMovesQuietMax >= recentMovesHotMin`** (operator-misconfigured) — the deriver applies a defensive swap so `quiet ≤ active ≤ hot` always holds. A test asserts the swap.

## Risks

1. **Date parsing portability.** `new Date(node.createdAt).getTime()` returns NaN on malformed ISO strings on some legacy Node + RN versions. The deriver guards every parse with `Number.isFinite(...)` and falls through to "outside any window" (the node contributes to `totalMoves` but not to `recentMoves`). The risk is silent under-counting in `recentMoves` if a debug fixture passes bad strings — the test fixture for "malformed timestamps" explicitly exercises this path.
2. **`PLAIN_LANGUAGE_COPY` changes in RULE-003.** SW-002 reads RULE-003's helper-line strings at module-load time. If RULE-003 changes a helper text (intentionally or otherwise), SW-002's `ACTIVITY_HELPER_LINES` reflects the change. The RULE-003 ban-list test catches verdict tokens at RULE-003's layer; SW-002 ALSO scans its own composed lines in `__tests__/heatModel.test.ts` to catch any drift that slips past RULE-003. **No coordination required** — both tests must pass for the suite to go green.
3. **Threshold tuning is a follow-up.** v1 ships safe defaults (1 / 6 / 3 / 1.5 / 0.5 / 4) that match the doctrine but have not been validated against a real corpus. **Out of scope for SW-002**; flagged in `## Out of scope`.
4. **GAL-001 / GAL-002 misuse risk.** A downstream consumer could compose a label that violates doctrine (`"Hot room — Affirmative is winning"`). This is a consumer-side doctrine violation, tracked in that card. SW-002 mitigates by (a) shipping coarse 3-valued unions only (no numeric score) and (b) explicitly documenting the consumer rules in `## API / interface contracts`.
5. **`hot` confusion with truth in copy review.** A future copy-review pass could mistakenly flag `hot` as a verdict-adjacent token. The doctrine carve-out is documented in three places: the `HeatLevel` JSDoc, the `_forbiddenActivityTokens` JSDoc, and the explicit "hot carve-out" test. A fourth backstop is the COPY-001 doc reference in this design.
6. **Segmenter complexity drift.** The depth-band segmenter is the most complex helper (~ 60 LOC). The simplest implementation (4 equal-width bands by max depth) is sufficient for v1. The risk is that a future "smarter" segmenter (semantic bands, clustering by cluster boundary) might be inserted without doctrine review. Mitigation: the `maxSegments` threshold and the `depthRange: { from, to }` shape are the only public surface — the implementation is hidden behind them.

## Out of scope

Explicit list of related work that this card does **NOT** include:

- **Rendering of any kind.** GAL-001, GAL-002, IX-002, and any future segment-marker UI render the data. SW-002 ships zero React, zero animation, zero styling, zero rail.
- **Side-aware activity.** A future card may stratify by side (`HeatLevel` per affirmative / negative). SW-002 v1 is room-level only — stratifying by side invites leaderboard reading, which the doctrine refuses.
- **Score weighting in `pointStanding`.** Standing math is a separate domain (`src/features/pointStanding/`). SW-002 never touches it.
- **Cross-room aggregation.** Gallery-level signals (hot rooms across many rooms, cool rooms across many rooms) are GAL-001's concern. SW-002 is per-room.
- **Time-zone display / locale formatting.** SW-002 emits no formatted timestamps. Consumer concern.
- **Heat decay curves.** v1 uses a simple recent-N-moves window. A future card may add an exponential decay or a per-axis decay. Tuning is a follow-up.
- **Persistence / Supabase column.** The profile is regenerated every render. No DB column, no migration, no RLS policy.
- **A "heatmap" timeline overlay.** Visual layer; future card.
- **AI-driven heat detection.** No external AI call. The deriver is pure deterministic TS.
- **Push notifications about hot rooms.** Out of v1 scope (CLAUDE.md "What Not to Build").
- **New `ActivityChip` kinds beyond the 8 listed.** Adding a new kind requires a follow-up card + a new helper-line entry + a new test path. SW-002 ships the 8 and freezes the union.

## Doctrine self-check

A walk through each relevant doctrine point and how the design respects it.

### `cdiscourse-doctrine` §1 — Score is gameplay analysis, never truth
SW-002 emits no truth labels. The three unions are coarse activity bands. `HeatLevel` carries no implication of correctness. The ban-list test (`__tests__/heatModel.test.ts`) explicitly refuses `winner / loser / true / false / correct / incorrect / verdict / proven / disproven / validated / right / wrong` and the SW-002-specific `winning / losing / dominant / losing ground / gaining ground`.

### `cdiscourse-doctrine` §2 — Heat means activity / friction
This is the core doctrine of the card. `HeatLevel === 'hot'` describes activity density: recent move count above threshold, plus open debt, plus repeated axis pressure. The `hot` token is the COPY-001-approved carve-out (see `_forbiddenLifecycleTokens` source comment + COPY-001 §5.1 + §8). The test `hot carve-out` documents this inline. No copy in SW-002 says "this side is winning" or "the conversation is decided."

### `cdiscourse-doctrine` §3 — Popularity is not evidence
SW-002 reads zero engagement / popularity / virality / view-count fields. The model has no input for "likes" or "retweets" because the platform does not surface them, and the deriver is structurally incapable of accepting one. The deriver routes nothing through `applyAntiAmplification` because it touches no standing.

### `cdiscourse-doctrine` §4 — AI moderator limits
SW-002 issues zero AI calls. It is a pure deterministic deriver. It does not call Anthropic, xAI, X API, OpenAI, or any external service.

### `cdiscourse-doctrine` §5 — Rules engine is sacred
SW-002 does not touch `src/lib/constitution/engine.ts`. It does not import from it. It does not extend the transition matrix.

### `cdiscourse-doctrine` §6 — Secrets policy
SW-002 reads no secrets. The deriver issues no network call. There is no `.env*` touch. The forbidden-imports test asserts zero `supabase`, `fetch`, `anthropic`, `xai` references.

### `cdiscourse-doctrine` §7 — No AI calls from production app
SW-002 lives under `src/features/strengthWeakness/`. It contains no AI client, no external HTTP call, no Edge Function hop.

### `cdiscourse-doctrine` §8 — Supabase conventions
SW-002 is pure-TS. No migration, no RLS policy, no Edge Function. No DB column. The profile is regenerated every render.

### `cdiscourse-doctrine` §9 — Plain language for users
The helper lines are plain English. Snake_case appears only as internal `ActivityChipKind` strings (typed enum values); helper-line text contains no underscores (test enforces). Internal codes never leak to UI.

### `cdiscourse-doctrine` §10 — v1 scope guards
SW-002 is per-room. No voting, no scoring winner, no search, no OAuth, no push, no public API. The activity model is a doctrine-clean alternative to engagement-style heat that other apps ship.

### `point-standing-economy` doctrine
SW-002 does not touch standing. The card explicitly refuses to read `standingBand` from the timeline map. The deriver never routes through anti-amplification because no scoring decision is made. The boundary is enforced by the forbidden-imports test.

### `timeline-grammar` doctrine
The timeline visual grammar reserves heat for activity / friction (`timeline-grammar` §"Heat is not truth"). SW-002 produces the data layer the rail will consume; the rendering rules (`heat ≠ truth`, `quiet rooms are easy entry not unimportant`) are baked into the unions and helper-line phrasings.

### COPY-001 doctrine
The ban-list helper aggregates `_forbiddenLifecycleTokens()` + `_forbiddenMetadataTokens()` + the COPY-001 person-attribution tokens + the SW-002-specific verdict tokens. Every emitted literal is scanned. The `hot` carve-out is documented in three places (JSDoc on `HeatLevel`, JSDoc on `_forbiddenActivityTokens`, explicit test).

### `accessibility-targets` doctrine
SW-002 ships no UI, so there are no accessibility-level requirements on this card directly. The downstream consumers (GAL-001 / GAL-002 / IX-002) inherit accessibility-targets in their cards. SW-002's data shape (typed unions; helper-line strings instead of icon-only signals) is structured so a screen-reader-friendly consumer can render it with `accessibilityLabel` strings composed from `helperLine` directly.
