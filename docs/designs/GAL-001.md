## Goal (one paragraph)

The Conversation Gallery's entry screen groups cards into six fixed sections today ("Jump into a live dispute · Needs first rebuttal · Source trail fights · Hot but unresolved · Easy first move · My rooms"). The issue (#30) asks for a richer set of "play lanes" matching the **kind of move** a user wants to make — so the gallery reads like a scrum board where each lane invites a specific play. GAL-001 expands `ConversationGallerySection` from 6 to 10 codes (adding `evidence_needed`, `definition_fights`, `logic_traps`, `tangents_branches`, `almost_synthesis`, `quiet_beginner_rooms`) while preserving the four Stage 6.4 codes whose semantics already match a lane shape (`my_rooms`, `needs_rebuttal`, `source_trail`, `jump_in`). The grouping stays deterministic (one card → exactly one lane), the doctrine constraints from `cdiscourse-doctrine` shape every label (no truth/popularity tokens, heat framed as activity/friction), and the change is pure-model + chip-row UI. No new dependency, no Supabase touch, no AI call, no migration. A new lane-filter chip row above the gallery lets the user narrow to a single lane.

> **Rename note.** Stage 6.4's `easy_first_move` becomes `quiet_beginner_rooms`. The Stage 6.4 `hot_unresolved` code is **removed**; its cards now route to `jump_in` (active back-and-forth) or `logic_traps` (deep unresolved chain) depending on lifecycle. See "Migration: removed codes" below.

### Frozen lane order

```ts
const SECTION_ORDER: ConversationGallerySection[] = [

  'jump_in',               // 3. active live dispute
  'source_trail',          // 4. source-chain fights
  'evidence_needed',       // 5. evidence asks open
  'definition_fights',     // 6. definition / scope disputes
  'logic_traps',           // 7. exhausted axis / deep unresolved chain

  'almost_synthesis',      // 9. closing-in rooms

```

Order matches scan-flow priority: a returning user sees **their** rooms first, then the lowest-friction first-move opportunities, then progressively heavier engagement.

## API / interface contracts

### `classifyCardToSection(card: ConversationGalleryCard): ConversationGallerySection`

Internal function (not exported). Pure, side-effect-free, deterministic.

```ts
function classifyCardToSection(card: ConversationGalleryCard): ConversationGallerySection {
  // 1. Joined-room override — user's own rooms always come first.
  if (card.hasUserJoined) return 'my_rooms';

  // 2. No-rebuttal short-circuit — lowest-friction first move.
  if (card.hasNoRebuttal) return 'needs_rebuttal';

  // 3. Lifecycle-primary routing (when available).
  const lc = card.rootClusterLifecycleState ?? null;
  if (lc === 'synthesis_ready' || lc === 'narrowed' || lc === 'conceded' || lc === 'confirmed') {
    return 'almost_synthesis';
  }
  if (lc === 'exhausted') return 'logic_traps';
  if (lc === 'branch_recommended') return 'tangents_branches';

  // 4. Bucket-derived routing (Stage 6.4 parity, expanded).
  if (card.bucket === 'source_chain_fight') return 'source_trail';
  if (card.bucket === 'evidence_fight') return 'evidence_needed';
  if (card.bucket === 'definition_scope_fight') return 'definition_fights';
  if (card.bucket === 'unresolved_deep_chain') return 'logic_traps';
  if (card.bucket === 'hot_now') return 'jump_in';
  if (card.bucket === 'gaining_heat') return 'jump_in';

  // 5. Heat-overheated fallback — overheated is still an active dispute lane.
  if (card.heatLevel === 'overheated') return 'jump_in';

  // 6. SW-002 entry-opportunity tie-breaker for mid-thread cards.
  if (card.entryOpportunity === 'deep_existing_clash') return 'logic_traps';
  if (card.entryOpportunity === 'mid_thread_join') return 'jump_in';

  // 7. Default fallback — quiet rooms, easy first move.
  return 'quiet_beginner_rooms';
}
```

The function reads **only** fields already on `ConversationGalleryCard` (no new field added). All inputs are deterministic descriptors of move structure / activity / lifecycle — none are popularity / engagement / virality signals (doctrine §3).

## Filter UX

### Decision: single-select chip row, value `'all' | ConversationGallerySection`.

The screen renders a horizontal `ScrollView` of chip `Pressable`s above the gallery list. The leftmost chip is `"All lanes"`; the remaining 10 chips are the lane labels from `GALLERY_SECTION_DEFINITIONS`. Tapping a chip sets `activeLane`. Tapping the active chip again or tapping `"All lanes"` clears the filter.

### Single-select vs multi-select — why single

The card is a P1/M-effort polish on the gallery; multi-select adds two costs that don't pay back in v1:

1. **Render ambiguity.** When two lanes are selected, the gallery must either (a) render both lanes as separate sections (effectively the same as `'all'` mode pre-filtered to two lanes), or (b) render a merged unsegmented list. Option (a) is identical to the `'all'` mode minus the unselected lanes — a degenerate UX. Option (b) breaks the scrum-board metaphor (the whole point of the card).
2. **Discoverability cost.** Multi-select requires a "clear" affordance and visible-state for the selection set. Single-select reads from one chip's `accessibilityState.selected` and is screen-reader-clean by default.

Multi-select is a viable v2 if data shows users routinely want a hybrid (e.g. `needs_rebuttal + quiet_beginner_rooms`). Flagged in §"Out of scope".

### Render contract when a lane filter is active

When `activeLane !== 'all'`:
- The gallery renders **only** the selected lane's cards.
- The lane header is still rendered above the card list (label + helperLine) so the user can see what they are filtered to.
- The lane's `emptyCopy` is shown via `<EmptyState>` when the filter yields zero matches.
- The bucket-filter chip row (Stage 6.3 — still present as `BucketChip`) and search remain functional. A bucket filter + lane filter compose with AND semantics (rare combo but well-defined: lane wins for ordering, bucket narrows further).

### Render contract when `activeLane === 'all'`

- Each non-empty lane renders as a section with `<Text>{label}</Text>` (heading) + `<Text>{helperLine}</Text>` (sub-heading muted), then the card list.
- Section ordering matches `SECTION_ORDER`.
- Lanes with zero cards are elided (existing Stage 6.4 behaviour).

### Observer-mode compatibility

The lane filter is **observer-safe**. The user is not yet a participant when they touch the chip row, so:
- Selecting `my_rooms` while not having joined any room produces the `emptyCopy` `"You haven't joined a room yet — try another lane to find one."` This message both explains the state and points the user back to other lanes.
- No lane requires a side selection to be readable.
- Tapping a card from any lane still routes through Stage 6.4's observer-default entry (no auto-join, no side modal).

### Accessibility contract for the lane chip row

Each chip `<Pressable>`:
- `accessibilityRole="button"`
- `accessibilityLabel={\`Filter by ${definition.label}\`}` for lane chips, `"Show all lanes"` for the `'all'` chip
- `accessibilityState={{ selected: activeLane === id }}`
- `accessibilityHint={definition.helperLine}` so a screen-reader user knows what the lane means before activating
- `hitSlop={{ top: 8, bottom: 8, left: 6, right: 6 }}` — the visual chip is ~32px tall; `hitSlop` lifts the effective hit target to 48px (≥ 44 requirement).

The chip row scrolls horizontally; on web a keyboard user can `Tab` to each chip and `Enter` / `Space` to activate. On native, VoiceOver / TalkBack swipe traversal hits every chip in order.

## Deterministic grouping

A card maps to **exactly one** lane. The priority order below is the canonical reference. The implementation is the if-else chain in `classifyCardToSection` (see §"API / interface contracts").

### Priority order (first match wins)

| Priority | Condition | Lane | Rationale |
| --- | --- | --- | --- |
| 1 | `card.hasUserJoined === true` | `my_rooms` | A returning user always sees their rooms before any other lane, regardless of bucket/lifecycle. |
| 2 | `card.hasNoRebuttal === true` | `needs_rebuttal` | Lowest-friction first move. Wins over `hot_now` / `source_chain_fight` even if both apply because "be the first rebuttal" is the most actionable hint. |
| 3 | `rootClusterLifecycleState ∈ { synthesis_ready, narrowed, conceded, confirmed }` | `almost_synthesis` | Lifecycle says the cluster is converging. Beats bucket because lifecycle is a stronger signal of "what move lands cleanly". |
| 4 | `rootClusterLifecycleState === 'exhausted'` | `logic_traps` | Same-axis pressure repeated — the cluster is stuck. Beats `evidence_fight` / `definition_scope_fight` because exhaustion is the dominant move-structure signal. |
| 5 | `rootClusterLifecycleState === 'branch_recommended'` | `tangents_branches` | Off-axis pressure built up — a branch reads cleaner. |
| 6 | `bucket === 'source_chain_fight'` | `source_trail` | Stable Stage 6.4 mapping. |
| 7 | `bucket === 'evidence_fight'` | `evidence_needed` | New lane for the evidence bucket. |
| 8 | `bucket === 'definition_scope_fight'` | `definition_fights` | New lane for the definition/scope bucket. |
| 9 | `bucket === 'unresolved_deep_chain'` | `logic_traps` | Stage 6.4 routed this to `hot_unresolved`; GAL-001 routes it to `logic_traps` (more specific). |
| 10 | `bucket === 'hot_now'` OR `bucket === 'gaining_heat'` | `jump_in` | Active back-and-forth — friction is the framing, not correctness. |
| 11 | `heatLevel === 'overheated'` | `jump_in` | Stage 6.4 routed overheated to `hot_unresolved`; GAL-001 routes it to `jump_in` because overheated rooms still have active moves. (`logic_traps` already catches the stuck variant via priority 4 / 9.) |
| 12 | `entryOpportunity === 'deep_existing_clash'` | `logic_traps` | SW-002 tie-breaker for deep clashes that didn't trip the lifecycle path. |
| 13 | `entryOpportunity === 'mid_thread_join'` | `jump_in` | SW-002 tie-breaker for mid-thread join opportunities. |
| 14 | _fallback_ | `quiet_beginner_rooms` | Cold rooms / plain temperament / no other signal. |

### Doctrine guards encoded in the priority order

- **No popularity / engagement signal.** Every input (`hasUserJoined`, `hasNoRebuttal`, `rootClusterLifecycleState`, `bucket`, `heatLevel`, `entryOpportunity`) is structural / activity-derived. None reads participant count, vote count, view count, follower count, virality. The heat band is "activity / friction" per doctrine §2.
- **Heat does not block or promote a lane based on correctness.** `heatLevel === 'overheated'` routes to `jump_in` (an action lane), not to a "best" or "winning" lane. There is no lane called "winning" / "losing" / "best rooms".
- **`logic_traps` does not imply the user is correct or incorrect.** It describes the move structure (same-axis pressure repeated). Helper line: "Same-axis pressure has repeated without new information." No verdict.
- **`almost_synthesis` does not imply a winner.** It describes structural convergence. Helper line: "The two sides have converged enough to summarise."
- **`my_rooms` is determined by joinedness, not by score / standing.** A user in a room with low standing still sees the room in `my_rooms` — no filter strips it out.

### Migration: removed codes

- **`easy_first_move` → renamed to `quiet_beginner_rooms`.** Card classification unchanged. The `seamlessConversationEntry.test.ts` test that asserts `'routes cold/plain cards to easy_first_move'` is updated in lockstep.
- **`hot_unresolved` → removed.** Existing routes split:
  - `bucket === 'hot_now'` (was → `hot_unresolved`) now → `jump_in`.
  - `heatLevel === 'overheated'` (was → `hot_unresolved`) now → `jump_in`.
  - `bucket === 'unresolved_deep_chain'` (was → `hot_unresolved`) now → `logic_traps`.
  - This is the only behavioural change to existing cards. The seamlessConversationEntry test that asserts `routes hot_now to hot_unresolved` is rewritten to assert `routes hot_now to jump_in`.

## Edge cases

The implementer MUST handle every case below. Tests in `__tests__/galleryLaneGrouping.test.ts` cover each one explicitly.

| # | Case | Expected behaviour |
| --- | --- | --- |
| 1 | Card with `moveCount === 0` (no root posted yet — draft/empty debate) | `hasNoRebuttal` is `false` per existing model (it requires `moveCount >= 1`). The fallback `quiet_beginner_rooms` catches it. |
| 2 | Card with `hasUserJoined === true` AND `hasNoRebuttal === true` | Routes to `my_rooms`, not `needs_rebuttal`. Joinedness wins. |
| 3 | Card with `rootClusterLifecycleState === 'archived_or_resolved'` | No lifecycle priority hits (archived isn't in the 4 priority tables). Falls through to bucket; typically `resolved_or_synthesized` bucket → quiet_beginner_rooms (no synthesis lane covers archived). The hint deriver (GAL-002) shows `watch_first` for archived. The lane shows them as quiet rooms — observer-readable but flagged as low-action. |
| 4 | Card with `bucket === 'resolved_or_synthesized'` (Stage 6.3 stop reason synthesis/concession/resolved) | Falls through priority 6-13; lands in `quiet_beginner_rooms`. Resolved rooms are still readable; they are not promoted to `almost_synthesis` because synthesis already happened — the lifecycle path (priority 3) catches the in-progress case. |
| 5 | Card with `rootClusterLifecycleState === null` AND `bucket === 'all_open'` (catch-all bucket from Stage 6.3) | Lands in `quiet_beginner_rooms` via the fallback. No lane is left empty in the union. |
| 6 | Generated / test duplicate rooms (Stage 6.3 dedupe — `duplicateCount > 1`) | Lane derivation runs on the deduplicated card. The collapsed duplicates land in **one** lane (the primary card's lane). The `duplicateCount > 1 ? ` ${card.duplicateCount} duplicate runs collapsed` : ''` UI text is preserved verbatim from Stage 6.3. |
| 7 | Filter `activeLane === 'my_rooms'` when the user has not joined any room | The lane has zero cards. `<EmptyState>` renders the `emptyCopy` from the table. No crash. |
| 8 | Filter `activeLane === '<lane>'` while `search` term is non-empty | AND semantics. The search input filters first; the lane filter narrows further. Tested in `gallerySectionCopy.test.ts` integration tests. |
| 9 | Search yields zero results with a lane filter active | Render the lane's `emptyCopy` (preferred over a generic "no results" string because the filter is the proximate cause). |
| 10 | Card with both `hasNoRebuttal === true` AND `rootClusterLifecycleState === 'synthesis_ready'` (impossible in practice, but defensible) | Priority 2 wins; lands in `needs_rebuttal`. The lifecycle priority (3) is below the no-rebuttal priority because "be the first rebuttal" is the most actionable hint a fresh user can act on. |
| 11 | Permission-denied / RLS-stripped card (debate not visible) | Gallery API never returns such rows; the gallery model never sees them. No defensive code needed in `classifyCardToSection`. |
| 12 | Offline / cached state with stale `rootClusterLifecycleState` | The lane derivation is pure; it works on whatever fields are present. If the loader has not populated `rootClusterLifecycleState` (GAL-002 graceful fallback path), the bucket path (priority 6-10) still classifies the card. |
| 13 | A new bucket is added to `ConversationBucket` (future work) | The fallback `quiet_beginner_rooms` catches it. The implementer should add an explicit lane mapping when a new bucket lands. |
| 14 | A new lane is added to `ConversationGallerySection` | The `SECTION_ORDER` array and `GALLERY_SECTION_DEFINITIONS` must be updated in the same commit. A typecheck failure surfaces the gap. |
| 15 | The user filters to a lane and the screen re-renders mid-scroll | `pageIndex` is **reset to 0** on `setActiveLane` (existing `setPageIndex(0)` pattern matches the bucket-chip and sort-chip handlers). |

## Risks

### R1 — Stage 6.4 `hot_unresolved` removal breaks an external assumption

**Likelihood:** Low. **Impact:** Medium.

The Stage 6.4 `hot_unresolved` section is referenced by:
- `seamlessConversationEntry.test.ts:118, 270` — updated in this card.
- `docs/seamless-conversation-entry.md` — doc reference, updated in the card.
- `CLAUDE.md` "Current stage" line — doc reference. The implementer must update this line when landing the card.

No production runtime code reads `hot_unresolved` by string. The grep above confirms 6 files mention it, all docs / tests / CLAUDE.md. The risk is a future test that hard-codes the string; nothing currently does.

**Mitigation:** the cdiscourse-doctrine §9 plain-language test would catch any code that emits `'hot_unresolved'` as a user-facing string (none does today). The seamlessConversationEntry test update covers the test surface.

### R2 — `easy_first_move` rename breaks fixtures

**Likelihood:** Low. **Impact:** Low.

Same shape as R1. Single test file references the old name. Rename in lockstep.

### R3 — Lane priority drift between the priority table here and the implementation

**Likelihood:** Medium during initial review. **Impact:** Medium.

The priority order is encoded as an if-else chain in `classifyCardToSection`. A reviewer or future implementer could reorder branches and break the documented contract.

**Mitigation:** `__tests__/galleryLaneGrouping.test.ts` asserts every row of the priority table explicitly. The exhaustive-switch pattern (with a `: never` fallback) catches missing lane members at compile time.

### R4 — Bucket chips vs lane chips coexist (UX confusion risk)

**Likelihood:** Medium. **Impact:** Low.

The Stage 6.3 `BucketChip` row still exists in `ConversationGalleryScreen.tsx`. Adding a `LaneChip` row creates two filter rows. The two compose with AND semantics, which can produce empty results for surprising combinations (e.g. bucket=`evidence_fight` + lane=`source_trail` is rare).

**Mitigation options the implementer should choose between (and document the choice in the PR description, not in this design doc):**
1. Keep both rows visible. The bucket row is the secondary "advanced filter"; the lane row is primary. Rendering order: search → lane chips (primary, with `helperLine` heading) → bucket chips (secondary, smaller).
2. Move the bucket row behind a "More filters" disclosure. Default-collapsed.

The design doc does **not** prescribe one. Both are doctrine-clean and accessibility-clean.

### R5 — Lane label authoring drift from RULE-003

**Likelihood:** Low. **Impact:** Low.

Some lane `helperLine` strings (`logic_traps`, `tangents_branches`, `almost_synthesis`) echo RULE-003 lifecycle helperLine wording. If RULE-003 changes its `exhausted` / `branch_recommended` / `synthesis_ready` helpers, the gallery lane helpers may go out of sync.

**Mitigation:** the audit doc for COPY-001 already notes this kind of cross-surface authoring as acceptable when bounded; the lane table is small enough (10 entries) to manually audit per copy review cycle. The ban-list test catches the worst-case drift (a verdict token slipping in).

### R6 — `rootClusterLifecycleState` not yet plumbed in the live loader

**Likelihood:** Medium. **Impact:** Low.

GAL-002 added the field as optional and graceful. If the live gallery query has not yet been wired to populate it (Stage 6.5 work), every card lands with `rootClusterLifecycleState === null`, the lifecycle priorities (3-5) never fire, and the gallery degrades cleanly to bucket-only routing.

**Mitigation:** the lane derivation is null-safe by construction. The implementer should confirm the loader populates the field if Stage 6.5 has landed; if not, the lanes still work via bucket routing.

### R7 — Implementation file size grows past comfort

**Likelihood:** Low. **Impact:** Low.

`conversationGalleryModel.ts` is currently ~1350 lines. Adding +145 lines lands near 1500. The file is pure-TS with no React, so unit-testability is unaffected, but readability suffers.

**Mitigation:** if the implementer feels the file is too dense, the lane derivation can be split into `conversationGalleryLaneModel.ts` (~250 lines) alongside the main model. The change is internal and does not affect the public API. The design doc does not require the split.

## Out of scope

Explicitly **not** included in GAL-001. Each is a candidate for a future card.

1. **AI lane suggestion.** No AI call decides which lane a card belongs to. The card is structurally classified. Future cards may add an AI-advisory "suggested lane" chip; this card does not.
2. **Persisted user-selected lane preference.** The lane filter resets to `'all'` on screen reload / app restart. No AsyncStorage write, no Supabase column, no server-side preference. A future card could add this — see "Will block" above.
3. **Lane-specific notifications.** No push, no email, no in-app notification fires when a new card enters a lane. (Push notifications are out of scope for v1 per CLAUDE.md.)
4. **Lane analytics / event firing.** No telemetry event is added for "user filtered to lane X". A future analytics card can add it.
5. **New compose presets per lane.** Lane filtering does not pre-fill the composer. The existing GAL-002 hint deriver still drives per-card hints. A future GAL-003 could route "user filtered to evidence_needed and then opened a card" → pre-select the "ask source" preset. Not in GAL-001.
6. **Multi-select lane chips.** Single-select only. Multi-select is a v2 candidate.
7. **Drag-and-drop scrum-board layout.** The "scrum-like" framing in the card name refers to the **lane vocabulary**, not a drag-and-drop UI. The gallery remains a vertically-scrolling stack of horizontal lane sections. A literal Kanban-style drag UI is out of scope.
8. **New bucket / lifecycle / heat signals.** GAL-001 reads existing signals only. Adding new structural signals to drive lane routing is future work.
9. **Persisted lane reorder.** The 10 lanes render in `SECTION_ORDER` and the user cannot reorder them.
10. **Reverting the Stage 6.4 `hot_unresolved` lane.** The lane is removed permanently. If the implementer or operator decides the change is wrong, the revert is a one-line change to add the lane back — but it is not designed-in as an option.

## Doctrine self-check

### `cdiscourse-doctrine` (the 10 universal rules)

1. **Score is gameplay analysis, never truth.** No lane label or helper line contains `winner`, `loser`, `correct`, `incorrect`, `true`, `false`, `verdict`, `proven`. Verified by the ban-list test. The `logic_traps` lane describes move structure, not correctness.
2. **Heat means activity / friction.** The `jump_in` and `logic_traps` helpers use "active back-and-forth" and "same-axis pressure" — activity language, not popularity. No lane mentions `popular`, `trending`, `viral`.
3. **Popularity is not evidence.** No lane derivation reads view count, vote count, follower count, share count, or any engagement metric. All inputs are structural (`hasUserJoined`, `hasNoRebuttal`, `bucket`, `heatLevel`, `rootClusterLifecycleState`, `entryOpportunity`). Heat itself is the existing Stage 6.3 score, which is activity-derived per its own doctrine self-check.
4. **AI moderator hard limits.** No AI call from production code. The lane derivation is pure TypeScript, sync, no network.
5. **Rules engine is sacred.** No changes to `src/lib/constitution/engine.ts`.
6. **Secrets policy.** No new env var, no new key. `grep -r "ANTHROPIC_API_KEY\|SERVICE_ROLE" app/ src/` remains zero matches.
7. **No AI calls from the production app.** Confirmed — no new fetch, no Anthropic, no xAI, no X API.
8. **Supabase conventions.** No DB migration. No table touched. No RLS change.
9. **Plain language for users.** Every lane label is plain language (≤ 8 words, ≤ 64 chars). No internal code (`source_chain_fight`, `evidence_fight`, etc.) reaches the rendered UI. The `helperLine` strings are also code-free.
10. **v1 scope guards.** No voting, no real-time collaboration, no OAuth, no public API, no push, no search. The lane vocabulary does not introduce any of these.

### `expo-rn-patterns`

- **No new dependency.** The chip row uses `<Pressable>` + `<ScrollView>` (existing primitives). No Bootstrap CSS, no animation lib, no icon lib.
- **All text in `<Text>`.** Lane headings, sub-headings, chip labels, empty-state copy all wrap in `<Text>`.
- **Color is supplementary.** The active-lane chip uses a border color and a font-weight delta; even with grayscale, `accessibilityState.selected` carries the meaning.
- **Existing component reuse.** The lane chip pattern reuses the existing `BucketChip` style (same paddings, same chip shape). No new visual primitive.
- **File structure.** The model change stays in `conversationGalleryModel.ts`; the UI change stays in `ConversationGalleryScreen.tsx`. Tests live in `__tests__/`. No new directory.

### `accessibility-targets`

- **44×44 hit target.** Lane chips are visually ~32px tall; `hitSlop={{ top: 8, bottom: 8, left: 6, right: 6 }}` lifts the effective hit target to 48px.
- **All Pressables have role + label + state.** Every lane chip has `accessibilityRole="button"`, `accessibilityLabel`, `accessibilityState={{ selected }}`, `accessibilityHint`.
- **Tab order.** Chip row sits after the search input and before the gallery list. Reading order is preserved on web.
- **Reduce motion.** No new animation. The chip selection swap is instant (no transition). Already passes the reduce-motion check.
- **Screen-reader announcement.** When the user activates a lane chip, the gallery re-renders; the new section heading is the first focusable element. No `announceForAccessibility` call needed — the lane heading + chip's `accessibilityState.selected` carry the change.
- **Color contrast.** The active chip uses the existing `#a5b4fc` border + `#1e1b4b` background, which already passes WCAG AA per Stage 6.3 visual tests.

### `test-discipline`

- **Tests are part of done.** +65 new tests, +20 updated tests. Test count goes UP.
- **Pure-model coverage.** `classifyCardToSection` has full priority-table coverage in `galleryLaneGrouping.test.ts`.
- **Ban-list test.** `gallerySectionCopy.test.ts` scans every label / helperLine / emptyCopy for the doctrine ban-list.
- **No .skip / .only.** None added.
- **No `console.log` in committed code.** None added.
