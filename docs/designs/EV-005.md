# EV-005 — Evidence-to-evidence interaction (annotations on evidence)

**Status:** Design draft — full scope
**Epic:** Epic 6 — Evidence
**Release:** v1
**Issue:** https://github.com/kyleruff1/cDiscourse/issues/120

## Goal (one paragraph)

EV-005 makes evidence a first-class object that can be *responded to with more
evidence or context* — never with a truth verdict on the original author. Today
an `EvidenceArtifact` (EV-001) carries a `sourceChainStatus` describing the
trail, and EV-002 surfaces that status in a popover. EV-005 adds a thin
**annotation layer** on top of the artifact: a participant or admin attaches a
small, descriptive note to a specific artifact — `primary_source`,
`retraction_attached`, `paywalled_source`, `satire_parody_context`,
`source_missing_quote`, etc. The result is a derived **status chip**
(`anchored | conflict_open | context_open | paywalled | broken | unknown`) that
the EV-002 popover and the timeline receipt chip surface. EV-005 ships the model
**and** the interactive surface: the read-only annotation stream, the
**"Add an annotation" picker** (18 plain-language tags), the **write path** that
persists an added annotation, and the Timeline/Cards count integration. The
doctrine that shapes every line of this design: annotations describe the
**source / record**, never accuse a **person**; an annotation never converts
popularity into factual standing (anti-amplification); an annotation has no path
to a `PointStandingDelta`; an annotation never blocks an ordinary post; AI never
authors an annotation. Evidence-inception is capped — annotations on annotations
are limited to one level; beyond that the UI replaces the picker with a
"Summarise this evidence thread" prompt that feeds a synthesis move, so the
surface cannot spiral into an unbounded nested argument.

## Scope — ONE FULL CARD

Per the binding operator decision, EV-005 ships its full written scope in a
single card. There is **no EV-005a**. Issue #120 closes fully satisfied. The
card delivers, all in this pass:

1. **The pure-TS annotation model** — `evidenceModel.ts` extension: the
   `EvidenceAnnotation` type, the 18-kind enum, `summariseAnnotations`, the
   depth-cap helper, the constructor + adapter, the actor-eligibility table, and
   the plain-language label/copy map.
2. **The read-only annotation stream + derived status chip** — rendered inside
   the existing EV-002 `SourceChainPopover` body, below "View receipt".
3. **The interactive "Add an annotation" picker** — a bottom-sheet built from RN
   primitives that presents the 18 tags as plain-language options, gated by the
   actor-eligibility table, with an optional ≤140-char note field.
4. **The evidence-meta write path** — a thin Edge Function `annotate-evidence`
   that persists an added annotation into the target argument's existing
   `client_validation` jsonb under a new `evidenceAnnotations` key. **No DB
   migration. No service-role in the client. No new client dependency.**
5. **Timeline / Cards integration** — the receipt chip count on evidence-bearing
   nodes reflects annotation status.
6. **The depth cap** — annotations on annotations limited to one level; the
   picker is replaced by a "Summarise this evidence thread" prompt at the cap.
7. **The full test suite** — model, summarise, depth-cap, ban-list over every
   label/copy string, forbidden-imports, anti-amplification separation, the
   picker, the write path (success + every failure), and a no-service-role
   structural assertion.

The `evidence_annotations` **DB table / migration is explicitly V2** and does
NOT ship in this card (see "Out of scope").

## Write-path decision (EXPLICIT — read this before designing the write)

The issue's Scope #5 says annotations persist as `meta.attached_evidence`
payload extensions, "same pattern as EV-001's pure-TS adapter", V1 with **no DB
migration**. Investigating the codebase resolves the loose wording into a precise
design, and surfaces one genuine constraint the implementer must not paper over:

**Findings:**

- **There is no `meta` column on `public.arguments`.** The table
  (`20260516000001_initial_schema.sql`) has `client_validation jsonb` and
  `server_validation jsonb`. EV-001's "`meta.attached_evidence`" is shorthand for
  the `attached_evidence` array that lives *inside* those jsonb snapshots
  (`docs/designs/EV-001.md` § "Persistence decision": EV-001 reads
  `arguments.client_validation.attachedEvidence` /
  `arguments.server_validation.attachedEvidence`).
- **`attached_evidence` is written only at insert time by `submit-argument`,
  which uses the service-role client.** There is no client *write* path to those
  jsonb columns today — EV-001 explicitly shipped a read adapter only.
- **The `arguments` UPDATE RLS policy is author-only**
  (`20260516000002_rls_policies.sql`: `author_id = auth.uid() OR
  is_moderator_or_admin()`). The *normal* EV-005 case is a participant
  annotating **someone else's** evidence-bearing argument — a **non-author
  write**. A caller-scoped client `supabase.from('arguments').update(...)`
  **cannot do that** — RLS blocks it. Only an own-bubble annotation (author
  annotating their own source — 3 self-descriptive kinds, see actor rules) could
  pass author-only RLS, and even then a raw client UPDATE of `client_validation`
  is fragile (last-write-wins on the whole blob, no eligibility enforcement).
- The issue forbids a DB migration in V1, so the natural insert-by-eligible-actor
  pattern that META-1A used (`point_tags` table) is **not available** to EV-005.

**Conclusion: EV-005 requires a new Edge Function. A client-only RLS write
cannot satisfy the cross-author case.** This is not optional and is reported to
the operator. The function is thin and follows the existing `apply-manual-tag`
shape exactly:

- New Edge Function `annotate-evidence` — JWT-verified, caller-scoped
  eligibility check, then a **service-role read-modify-write of the *existing*
  `client_validation` jsonb** on the target argument: append one annotation
  under a new `evidenceAnnotations` array key. **No schema change, no migration**
  — it writes only to a column that already exists. `submit-argument` already
  proves the column is service-role-writable.
- The client never holds the service-role key; it calls
  `supabase.functions.invoke('annotate-evidence', ...)`, identical to how
  `pointTagsApi.ts` calls `apply-manual-tag`.

This honors every constraint: no migration in V1, no service-role in client,
RLS-respecting (the function does a caller-scoped read to enforce visibility +
eligibility *before* the privileged write — same as `apply-manual-tag`), no new
dependency, RN primitives only. The `evidence_annotations` table remains the V2
upgrade path; when it lands, `annotate-evidence` swaps its persistence target
and the pure-TS model is unchanged.

## Data model

All new types live in `src/features/evidence/evidenceModel.ts` (extends EV-001;
does **not** re-create or modify it). No DB migration — v1 persists the

### Annotation kind enum (18 kinds — verbatim from the issue)

```ts
/**
 * EV-005 — Evidence-annotation kind. Every value describes the SOURCE or the
 * RECORD, never a person. No verdict, no person-attribution. Aligned with
 * META-001's manual-tag philosophy: a participant annotation, never a flag,
 * never a moderation action.
 */
export type EvidenceAnnotationKind =
  | 'primary_source'             // the artifact is (or reaches) a primary record
  | 'secondary_analysis'         // the artifact is secondary commentary on a source
  | 'quote_attached'             // a verbatim quote is present alongside the source
  | 'source_missing_quote'       // a source pointer with no quote excerpt
  | 'quote_disputed'             // the quoted text may not match the linked source
  | 'context_requested'          // more context is being asked for on this artifact
  | 'retraction_attached'        // a retraction / correction notice is linked
  | 'source_later_updated'       // the source was revised after it was first cited
  | 'satire_parody_context'      // the source is satire / parody, not a literal report
  | 'screenshot_only_chain_weak' // only a screenshot exists; no inspectable chain
  | 'misreporting_alleged'       // an alternate account of the same facts exists
  | 'translation_context_issue'  // translation may affect the meaning of the source
  | 'outdated_source'            // the source predates more current information
  | 'methodology_dispute'        // the source's method/sampling is contested
  | 'broken_link'                // the linked source no longer resolves
  | 'paywalled_source'           // the source is behind a paywall / not freely open
  | 'conflicting_source'         // another source disagrees with this one
  | 'source_chain_anchored';     // the chain reaches an inspectable anchor

export const ALL_EVIDENCE_ANNOTATION_KINDS: ReadonlyArray<EvidenceAnnotationKind> =
  Object.freeze([
    'primary_source', 'secondary_analysis', 'quote_attached', 'source_missing_quote',
    'quote_disputed', 'context_requested', 'retraction_attached', 'source_later_updated',
    'satire_parody_context', 'screenshot_only_chain_weak', 'misreporting_alleged',
    'translation_context_issue', 'outdated_source', 'methodology_dispute', 'broken_link',
    'paywalled_source', 'conflicting_source', 'source_chain_anchored',
  ]);
```

`misreporting_alleged` is the one kind whose internal code reads close to an
accusation; its plain-language label is deliberately **"An alternate account
exists"** (record-descriptive). The ban-list test asserts every label/helper is
free of person-attribution tokens. See "The 18 plain-language labels" and
"Non-accusatory copy contract".

### Non-accusatory copy contract

Three kinds (`misreporting_alleged`, `quote_disputed`, `methodology_dispute`)
have internal codes that read close to an accusation. Their plain-language
labels/helpers are deliberately written about the **record**, not the person:
"An alternate account exists", "The quoted text may not match what the linked
source says", "The source's method or sampling is being questioned". The
ban-list test gives these three special focus and asserts they contain no
`lied / deceived / misleading reader / liar / dishonest / bad faith` token. The
implementer must not "improve" the copy toward a verb about a person.

### The annotation object

```ts
/**
 * EV-005 — One annotation attached to one EV-001 EvidenceArtifact. Frozen at
 * construction. JSON-serializable. v1 persists inside the existing
 * client_validation jsonb on the argument (no DB migration).
 */
export type EvidenceAnnotation = Readonly<{
  /** Stable id. Adapter mints: `<evidenceArtifactId>:annotation:<index>`.
   *  The write path (Edge Function) mints the same shape on append. */
  id: string;
  /** FK back to the EV-001 EvidenceArtifact.id this annotation describes. */
  evidenceArtifactId: string;
  kind: EvidenceAnnotationKind;
  /** Optional ≤ 140-char free-text note. Describes the SOURCE; never a
   *  person. Rendered read-only beneath the chip. */
  note?: string;
  /** The participant/admin who added the annotation. Never AI. */
  addedByUserId: string;
  /** ISO-8601. */
  createdAt: string;
  /**
   * Annotation-nesting depth. 0 = annotation directly on an artifact.
   * 1 = annotation on an annotation (the one permitted nested level).
   * The model rejects/suppresses anything > 1 — see depth cap.
   */
  depth: 0 | 1;
  /** When depth === 1, the id of the depth-0 annotation it responds to.
   *  null/absent for depth 0. */
  parentAnnotationId?: string | null;
}>;
```

### The summary

```ts
/** EV-005 — derived status chip. Describes the annotation set's shape,
 *  never a truth verdict. */
export type EvidenceAnnotationStatusChip =
  | 'anchored'       // a primary_source / source_chain_anchored annotation present
  | 'conflict_open'  // a conflicting/disputed/retraction/misreporting annotation present
  | 'context_open'   // a context_requested annotation present, no conflict
  | 'paywalled'      // a paywalled_source annotation present, no conflict/context
  | 'broken'         // a broken_link / screenshot_only_chain_weak annotation present
  | 'unknown';       // no annotations, or none that map to a higher-priority chip

export interface EvidenceAnnotationSummary {
  /** primary_source + source_chain_anchored + quote_attached annotations. */
  primary: ReadonlyArray<EvidenceAnnotation>;
  /** conflicting_source + quote_disputed + retraction_attached +
   *  misreporting_alleged + methodology_dispute + outdated_source +
   *  source_later_updated + translation_context_issue annotations. */
  conflicts: ReadonlyArray<EvidenceAnnotation>;
  /** context_requested annotations. */
  contextRequests: ReadonlyArray<EvidenceAnnotation>;
  /** The single derived status chip (priority resolution — see below). */
  statusChip: EvidenceAnnotationStatusChip;
  /** Total annotation count (depth 0 + accepted depth 1). Drives the "+N"
   *  affordance and the timeline receipt-count reflection. */
  count: number;
  /** Plain-English label for statusChip. Max 32 chars. Never a code. */
  statusLabel: string;
  /** Plain-English helper for statusChip. Max 90 chars. */
  statusHelper: string;
  /** Logical tone for the chip renderer. NOT a truth label. */
  tone: 'neutral' | 'info' | 'attention' | 'muted';
}
```

**Status-chip priority resolution** (deterministic, asserted by tests). When
multiple kinds are present the chip reflects the *most action-worthy* state, in
this order: `conflict_open` > `context_open` > `broken` > `paywalled` >
`anchored` > `unknown`. Rationale: a conflict/retraction is the state a reader
most needs to see; an anchored chip must never *hide* an open conflict.
`broken` outranks `paywalled` because a dead link is a harder failure than a
paywall. Empty annotation list → `unknown` with label "No annotations yet".

Status-chip plain copy (exact strings, all ban-list-clean):

| statusChip | statusLabel | statusHelper | tone |
| --- | --- | --- | --- |
| `anchored` | `Source trail is anchored` | `An annotation marks this source as reaching an inspectable record.` | `neutral` |
| `conflict_open` | `Open conflict` | `An annotation flags a conflicting, disputed, or retracted source.` | `attention` |
| `context_open` | `Context asked for` | `Someone has asked for more context on this source.` | `info` |
| `paywalled` | `Behind a paywall` | `An annotation notes this source is not freely open.` | `info` |
| `broken` | `Trail is weak` | `An annotation notes the trail is broken or screenshot-only.` | `attention` |
| `unknown` | `No annotations yet` | `No one has added context to this source.` | `muted` |

### Doctrine note on the summary

`summariseAnnotations` reads **only** the annotation set. It never reads
engagement, likes, view counts, follower counts, or the artifact author's
identity. It never *upgrades* `EvidenceArtifact.sourceChainStatus` and never
touches `PointStandingDelta`. Adding a `retraction_attached` annotation produces
`conflict_open` — it adds context; it does not "defeat" anyone and it does not
subtract factual standing. Standing changes remain owned by EV-004 /
`antiAmplification.ts` and only respond to *new evidence moves*, not annotations.

export function buildEvidenceAnnotation(input: {
  evidenceArtifactId: string;
  kind: EvidenceAnnotationKind;
  addedByUserId: string;
  createdAt: string;
  index: number;                 // for the deterministic id
  note?: string | null;
  depth?: 0 | 1;                 // defaults to 0
  parentAnnotationId?: string | null;
}): EvidenceAnnotation;
// Trims note to 140 chars; whitespace-only note → undefined; freezes the
// result; mints id = `${evidenceArtifactId}:annotation:${index}`.

// ── Adapter: raw payload array → typed, depth-capped annotations ──
export function buildEvidenceAnnotations(input: {
  evidenceArtifactId: string;
  raw: ReadonlyArray<{
    kind: EvidenceAnnotationKind;
    note?: string | null;
    addedByUserId: string;
    createdAt: string;
    depth?: number;
    parentAnnotationId?: string | null;
  }>;
}): ReadonlyArray<EvidenceAnnotation>;
// Drops entries with an unknown kind. Preserves order. Runs every entry
// through enforceAnnotationDepthCap before returning.

// ── Summary ──
export function summariseAnnotations(
  annotations: ReadonlyArray<EvidenceAnnotation>,
): EvidenceAnnotationSummary;
// Pure. Empty input → { statusChip: 'unknown', count: 0, ... }.

// ── Depth cap ──
export interface AnnotationDepthCapResult {
  accepted: ReadonlyArray<EvidenceAnnotation>;    // depth-0 + valid depth-1
  suppressed: ReadonlyArray<EvidenceAnnotation>;  // depth ≥ 2 or orphan depth-1
  showsSynthesisPrompt: boolean;                  // true when any depth-1 accepted
  synthesisPromptLabel: string;                   // '' when showsSynthesisPrompt false
}
export function enforceAnnotationDepthCap(
  annotations: ReadonlyArray<EvidenceAnnotation>,
): AnnotationDepthCapResult;
// depth 0 always accepted. depth 1 accepted IF parentAnnotationId resolves to
// an accepted depth-0 annotation. depth >= 2 (or depth-1 with missing/depth-1
// parent) is suppressed. showsSynthesisPrompt is true whenever any depth-1
// annotation is accepted (the next level is closed → synthesis).
// synthesisPromptLabel === 'Summarise this evidence thread' when shown.

// ── Plain-language copy lookup ──
export function getEvidenceAnnotationLabel(kind: EvidenceAnnotationKind): string;
export function getEvidenceAnnotationHelper(kind: EvidenceAnnotationKind): string;
// Frozen plain-English strings; never a snake_case code.

// ── Actor eligibility ──
export type EvidenceAnnotationActorRole =
  | 'participant_other_bubble'
  | 'participant_own_bubble'
  | 'observer'
  | 'admin';

export interface EvidenceAnnotationEligibilityContext {
  actorRole: EvidenceAnnotationActorRole;
  /** depth of the annotation being added: 0 (on artifact) or 1 (on annotation). */
  targetDepth: 0 | 1;
}

/** Frozen table — actorRole → the kinds that actor may add. */

  Readonly<Record<EvidenceAnnotationActorRole, ReadonlyArray<EvidenceAnnotationKind>>>;

  kind: EvidenceAnnotationKind,
  ctx: EvidenceAnnotationEligibilityContext,

// false for observers (always). false when targetDepth > 1. own-bubble: only
// the 3 self-descriptive kinds. other-bubble + admin: all 18.

export function eligibleAnnotationKinds(

  debateId: string;
  /** The argument that carries the EvidenceArtifact being annotated. */
  argumentId: string;
  /** EvidenceArtifact.id — `<argumentId>:evidence:<n>`. */
  evidenceArtifactId: string;

  argumentId: string;
  evidenceArtifactId: string;

### Edge Function contract (annotate-evidence/index.ts)

Standard mock-safe shape, modelled on `apply-manual-tag`:

```
POST /functions/v1/annotate-evidence
Auth: Bearer <user JWT>   (required — 401 without)

Request body:
{
  debateId: string (uuid),
  argumentId: string (uuid),
  evidenceArtifactId: string,        // `<argumentId>:evidence:<n>`
  kind: EvidenceAnnotationKind,      // must be one of the 18
  note?: string | null,             // server trims to 140
  depth: 0 | 1,
  parentAnnotationId?: string | null // required when depth === 1
}

Response 200:
{ argumentId, evidenceArtifactId, annotations: EvidenceAnnotation[] }

Errors (stable shape { error, message }):
  400 invalid_json | invalid_kind | depthId_required | debate_argument_mismatch
      | argument_deleted | depth_cap_exceeded | evidence_artifact_not_found
  401 unauthorized
  403 not_eligible | argument_not_visible
  405 method_not_allowed
  500 argument_lookup_failed | persist_failed
```

**Function workflow** (each step mirrors `apply-manual-tag`):

1. CORS preflight / `POST`-only gate.
2. Require `Authorization` header → else `401`.
3. Parse + validate body: `kind` ∈ the 18; `depth` ∈ {0,1}; when `depth === 1`,
   `parentAnnotationId` present; `debateId`/`argumentId` are UUIDs.
4. Build the **caller-scoped client** (anon key + caller JWT). `getUser()` →
   `callerId`; else `401`.
5. Caller-scoped `SELECT` the target argument (`id, author_id, debate_id,
   status, client_validation`). Missing/invisible row → `403
   argument_not_visible` (no existence leak). `debate_id` mismatch → `400`.
   `status === 'deleted'` → `400 argument_deleted`.
6. Derive the caller's `EvidenceAnnotationActorRole` from
   `debate_participants.side` + `profiles.role` + `author_id === callerId` —
   identical derivation to `apply-manual-tag`'s eligibility block.
7. Enforce `isAnnotationAllowed(kind, { actorRole, targetDepth: depth })`
   (the same pure rule the client uses, mirrored into
   `_shared/evidenceAnnotationEligibility.ts`). Ineligible → `403 not_eligible`.
8. Read the existing `client_validation.evidenceAnnotations` array; verify the
   `evidenceArtifactId` resolves to an artifact on this argument (re-run
   `buildEvidenceArtifacts` against the stored `attached_evidence`); else `400
   evidence_artifact_not_found`.
9. Run `enforceAnnotationDepthCap` on `[...existing, candidate]`. If the
   candidate would be suppressed (depth ≥ 2, or depth-1 with a non-resolving
   parent) → `400 depth_cap_exceeded`.
10. Mint the new annotation with `buildEvidenceAnnotation` (deterministic id =
    `<evidenceArtifactId>:annotation:<nextIndex>`), append it to the array.
11. **Service-role client** read-modify-write: `UPDATE public.arguments SET
    client_validation = <merged jsonb> WHERE id = argumentId`. This writes only
    to an existing column — **no migration**. The merge is a *spread* so the
    existing `attachedEvidence` / `flags` / `valid` keys are untouched. The
    service-role client is used *only* for this single privileged write, never
    logged. On error → `500 persist_failed`.
12. Best-effort audit row in `admin_audit_events` (service-role; `action:
    'evidence_annotation_added'`, payload carries shortened ids + kind only —
    never the note text, never an email).
13. Return `{ argumentId, evidenceArtifactId, annotations }`.

**Logging rules:** never log `Authorization`, the service-role key, the user's
note text, or any email. Log function name, `callerId`, `argumentIdShort`,
`kind`, decision, duration, error class — same as `apply-manual-tag`.

**Concurrency:** the function does a read-modify-write of the whole
`client_validation` blob. Two simultaneous annotations on the same argument
race; the loser's append can be lost. V1 mitigation: the function re-selects
the argument inside the same request immediately before the write and the
window is tiny; the deterministic-index id-mint means a lost write is *safe*
(no corruption, just a missing annotation the user can re-add). The race-free
fix is the V2 `evidence_annotations` table (one row per annotation, no blob
rewrite). This limitation is documented in the function header and in
"Risks". This is acceptable for V1 per the issue's no-migration constraint.

### Actor rules — who may annotate

Aligned with META-001's `MANUAL_TAG_ELIGIBILITY` philosophy. Encoded as the
frozen `EVIDENCE_ANNOTATION_ELIGIBILITY` table + `isAnnotationAllowed` helper,
mirrored into `supabase/functions/_shared/evidenceAnnotationEligibility.ts` so
the client picker and the Edge Function enforce the *same* rule.

| Actor | May add annotations? | Which kinds |
| --- | --- | --- |
| Participant on **another** bubble | Yes — the normal case | All 18 |
| Participant on **own** bubble | Yes, but only self-descriptive | `source_later_updated`, `retraction_attached`, `context_requested` |
| **Observer** | No | none — stream is read-only for observers |
| **Admin** | Yes | All 18 |
| **AI** | Never | none — no code path constructs an annotation from an AI call; the constructor + Edge Function require an explicit human `addedByUserId` |

Depth rule layered on top: **any** actor adding at `targetDepth > 1` is refused
by `isAnnotationAllowed` (the depth cap). At `targetDepth === 1` the eligible-kind
set is unchanged per actor.

Eligibility is a *gate on adding*, not on *reading*: everyone (including
observers) reads the annotation stream and the status chip.

## The add-annotation flow — end to end

1. A participant opens the EV-002 `SourceChainPopover` on an evidence-bearing
   node (via `TimelineNodePopover`) and expands the source-chain section.
2. Below the read-only artifact rows / "Inspect receipt", the
   `EvidenceAnnotationStream` renders: the status-chip header, the existing
   depth-0 chips, and — for an eligible viewer (`canAddAnnotation` true) — an
   **"Add an annotation"** `Pressable` (`accessibilityRole="button"`, ≥44×44).
   For observers the trigger is hidden; for own-bubble authors it is shown
   (their eligible set is the 3 self-descriptive kinds).
3. Tapping the trigger sets `TimelineNodePopover`'s `addSheetVisible` state and
   opens `AddAnnotationSheet`. The parent computes `eligibleKinds` via
   `eligibleAnnotationKinds({ actorRole, targetDepth: 0 })` for the viewer.
4. The sheet renders a scrollable list of option rows — each row is a
   `Pressable` showing the **plain-language label** + helper subtitle, with
   `accessibilityRole="radio"` and `accessibilityState.selected`. Below the list
   is an optional `TextInput` note field (`maxLength={140}`, `accessibilityLabel`
   "Optional note about this source", a live char-count, and the hint
   *"Describe the source, not the person."*). A `Confirm` button (disabled until
   a kind is selected) and a `Cancel` button.
5. On `Confirm` the sheet calls `onSubmit(kind, note)`. `TimelineNodePopover`
   sets `isSubmitting`, then calls
   `addEvidenceAnnotation({ debateId, argumentId, evidenceArtifactId, kind,
   note, depth: 0 })`.
6. `addEvidenceAnnotation` invokes the `annotate-evidence` Edge Function. The
   function enforces auth + visibility + eligibility + depth cap, then
   service-role-appends the annotation into the argument's `client_validation`
   jsonb (spread-merge, never replace) and returns the full annotation set.
7. On `ok: true` the parent closes the sheet and updates the annotation list
   from `data.annotations`; the popover re-renders — `summariseAnnotations` +
   `enforceAnnotationDepthCap` recompute the status chip, the chip list, and the
   timeline receipt-count. On `ok: false` the sheet stays open with
   `submitError` set (inline, polite live region); the post is never blocked —
   the failure only means the annotation was not saved.
8. **Depth cap:** if the viewer tries to annotate a *depth-1* annotation (the
   stream is already at one nested level), `EvidenceAnnotationStream` does **not**
   render an "Add an annotation" trigger on that nested chip — instead it shows
   the **"Summarise this evidence thread"** prompt row. Pressing it calls
   `onPressSynthesisPrompt`, which `TimelineNodePopover` wires to the existing
   synthesis composer preset (`argumentType: 'synthesis'`, as `SC-004` already
   does in `quickActionPresets`). No third annotation level is ever created;
   continued disagreement converts into a first-class synthesis move.

The model machinery (steps 6–8 pure logic), the picker (steps 3–5), the write
path (step 6 network), and the re-read (step 7) all ship in EV-005.

## Depth-cap design (full)

The cap is **one nested level**: an annotation may sit on an artifact (depth 0)
or on a depth-0 annotation (depth 1). Nothing deeper exists.

- **Model enforcement.** `enforceAnnotationDepthCap` partitions an annotation
  list into `accepted` (every depth-0; every depth-1 whose `parentAnnotationId`
  resolves to an accepted depth-0) and `suppressed` (depth ≥ 2; orphan depth-1;
  depth-1 whose parent is itself depth-1). `buildEvidenceAnnotations` runs it on
  every read; the Edge Function runs it on `[...existing, candidate]` before
  persisting and rejects a candidate that would be suppressed with
  `400 depth_cap_exceeded`. So depth ≥ 2 can never be persisted *and* is
  defensively dropped if it somehow appears in stored data.
- **UI enforcement.** `EvidenceAnnotationStream` renders the "Add an annotation"
  trigger only against artifact-level (depth-0 add) and depth-0 chips (depth-1
  add). For a depth-1 chip the trigger is replaced by the synthesis-prompt row.
- **Synthesis prompt.** When `depthCap.showsSynthesisPrompt` is true the stream
  renders a `Pressable` row labelled **"Summarise this evidence thread"**
  (`accessibilityRole="button"`, ≥44×44). Pressing it calls
  `onPressSynthesisPrompt` → `TimelineNodePopover` opens the composer with the
  existing `argumentType: 'synthesis'` preset. The disagreement that would have
  become a depth-2 annotation becomes a real synthesis move instead.
- **Observer.** When `isReadModeViewer` the synthesis-prompt row renders
  disabled with the EV-002 "Join a side to ..." helper.
- **Test:** `evidenceAnnotationDepthCap.test.ts` asserts depth-0 always
  accepted; one valid depth-1 → accepted + `showsSynthesisPrompt: true` +
  `synthesisPromptLabel === 'Summarise this evidence thread'`; depth-2 →
  suppressed; orphan depth-1 → suppressed; depth-1-on-depth-1 → suppressed; no
  depth-1 → `showsSynthesisPrompt: false`, `synthesisPromptLabel: ''`. The chip
  test asserts the synthesis-prompt row a11y + the disabled observer variant.

## Accessibility contract

The picker is keyboard- and screen-reader-accessible and meets WCAG AA.

- **`AddAnnotationSheet`** uses RN `Modal` with `accessibilityViewIsModal` on the
  content `View` and `accessibilityRole="dialog"` on the root. The scrim is a
  full-bleed `Pressable` (`accessibilityLabel="Close"`) — pressing it calls
  `onClose`.
- **Escape / outside-press / hardware back close the sheet.** `Modal`'s
  `onRequestClose` is wired to `onClose` (covers Android back). On web
  (`Platform.OS === 'web'`) a `keydown` listener closes on `Escape` and returns
  focus to the "Add an annotation" trigger. Scrim press closes via the
  `Pressable`.
- **Option rows** are `Pressable` with `accessibilityRole="radio"`,
  `accessibilityState={{ selected }}`, `accessibilityLabel = label`,
  `accessibilityHint = helper`, and a ≥44×44 hit target (`hitSlop` if the
  visual row is shorter). Exactly one row is selected at a time.
- **Confirm/Cancel** are `Pressable` with `accessibilityRole="button"`,
  `accessibilityState={{ disabled }}`, ≥44×44. Confirm is `disabled` until a
  kind is selected and while `isSubmitting` (`accessibilityState.busy`).
- **Note `TextInput`** has an `accessibilityLabel`, a visible char-count, and
  `maxLength={140}`. Submit failures render in a `Text` with
  `accessibilityLiveRegion="polite"`.
- **Focus order** follows reading order: header → option list → note field →
  Cancel → Confirm. No `tabIndex` jumps.
- **Color is never the only signal.** Each option row and each chip carries its
  full meaning in the text label; tone color is supplementary. A grayscale test
  (tone swapped to neutral) asserts the rendered text still distinguishes every
  kind and every status chip.
- **Reduce motion:** the sheet uses `Modal`'s default present/dismiss; when
  `reduceMotion` is true the sheet sets `animationType="none"`. The annotation
  stream itself has no animation.
- **Chips** are `accessibilityRole="text"` (not pressable in v1) with a label
  carrying the kind label + note. The status-chip header label carries the count
  + status label.
- **Screen-reader announcement:** on a successful add, the parent calls
  `AccessibilityInfo.announceForAccessibility('Annotation added: <label>.')`
  once — not on every recompute.

## Edge cases

The implementer must handle each:

- **Empty annotation list** — `summariseAnnotations([])` → `statusChip:
  'unknown'`, `count: 0`, label "No annotations yet"; `SourceChainPopover`
  renders exactly as today (no stream regression to EV-002 behaviour). The
  "Add an annotation" trigger still shows for eligible viewers (you can add the
  first annotation).
- **Single annotation** — chip renders, `+N` affordance absent, status chip
  reflects that one kind's priority bucket.
- **Conflicting kinds present together** — e.g. `primary_source` AND
  `retraction_attached`: priority resolution picks `conflict_open`; the
  `primary` array still contains the `primary_source` annotation. Test asserts
  both.
- **Unknown / malformed kind in the persisted payload** — `buildEvidenceAnnotations`
  drops the entry silently (matches EV-001's "drop empty attachment"); never
  throws. `evidenceAnnotationsFromMeta` returns `[]` for a malformed
  `client_validation`.
- **depth-1 annotation whose `parentAnnotationId` does not resolve** — suppressed
  by `enforceAnnotationDepthCap` (orphan); `showsSynthesisPrompt` unaffected by
  an orphan.
- **depth ≥ 2 in the payload** — suppressed; the synthesis prompt is shown
  because the level is already at the cap. The Edge Function rejects a
  depth ≥ 2 *add* with `400 depth_cap_exceeded` before it ever persists.
- **note exceeds 140 chars** — `buildEvidenceAnnotation` truncates to 140 (both
  client and Edge Function); empty/whitespace-only note stored as `undefined`.
- **note containing a person-attribution token** — not runtime-filtered (notes
  are free text); the ban-list test covers only app-authored labels/helpers/
  prompts. The picker carries the visible hint *"Describe the source, not the
  person."* (in-scope copy, asserted ban-list-clean).
- **Observer opening the popover** — stream + status chip fully visible; the
  "Add an annotation" trigger is hidden; if an observer somehow reaches the
  sheet, `eligibleKinds` is `[]` and the Edge Function refuses with `403
  not_eligible` (defense in depth).
- **Own-bubble author** — sees the full stream; the picker offers only the 3
  self-descriptive kinds; the Edge Function re-enforces.
- **Write succeeds but the popover was closed** — the `addEvidenceAnnotation`
  result is still applied to `TimelineNodePopover` state; on next open the
  re-read shows the new annotation. Guard the `setState` with a mounted ref so
  an unmounted component does not warn.
- **Write fails (network / 403 / 500)** — the sheet stays open with
  `submitError`; the user can retry or cancel. **The failure never blocks an
  ordinary post** — annotating is orthogonal to posting.
- **Concurrent annotations from two participants** — see "Concurrency" in the
  Edge Function contract: deterministic-index ids mean a lost race is safe (a
  missing annotation, not corruption); re-add works. V2 table removes the race.
- **Artifact with no annotations but a `broken` `sourceChainStatus`** — the
  EV-002 receipt chip still shows "Source trail is weak" from EV-001; EV-005's
  annotation status chip independently shows `unknown`. The two chips are
  distinct signals and may legitimately disagree — documented so the implementer
  does not "reconcile" them.
- **Anti-amplification edge — "does a popular retraction defeat the point?"** —
  No. An annotation never feeds `PointStandingDelta`. Adding `retraction_attached`
  produces `conflict_open` (context), full stop. The forbidden-imports test
  asserts `evidenceModel.ts` and `evidenceAnnotationApi.ts` import nothing from
  `pointStanding`/`antiAmplification`.
- **`evidenceArtifactId` that does not resolve to an artifact on the argument**
  — Edge Function returns `400 evidence_artifact_not_found`.
- **Reduce motion** — the stream has no animation; the sheet sets
  `animationType="none"` under reduce-motion.

  EV-002 "ask" CTA, observer-disable, and own-message paths must keep their
  existing tests green.
- **A new Edge Function ships in this card.** `annotate-evidence` must be
  written and tested but **not deployed by Claude** — the operator runs
  `npx supabase functions deploy annotate-evidence --linked`. Until then the
  client wrapper's calls fail gracefully (the sheet shows `submitError`); the
  rest of EV-005 (model, read rendering of pre-existing annotations, chips)
  works without the deploy.
- **Read-modify-write concurrency on `client_validation`.** Two simultaneous
  adds on the same argument race; the loser's append can be lost. Mitigated by
  the immediate re-select before write + deterministic-index ids (a lost write
  is a missing annotation, never corruption). The race-free fix is the V2
  `evidence_annotations` table. Documented in the function header.
- **`client_validation` is an EV-001 read source.** EV-005 *appends* a sibling
  key `evidenceAnnotations` to that jsonb; it must **not** disturb the existing
  `attachedEvidence` / `flags` / `valid` keys. The Edge Function does a
  spread-merge, never a replace. Tested.
- **Two chips on one node.** The EV-002 receipt chip and the EV-005 annotation
  status chip both summarise "evidence health." They are distinct and may
  disagree. Mitigation: distinct labels ("Source trail is weak" vs "Open
  conflict"), distinct testIDs, the annotation stream sits inside the expanded
  popover while the receipt chip stays in the band row.
- **`misreporting_alleged` copy drift.** Its internal code is the closest to an
  accusation. The plain-language label/helper must stay record-descriptive
  ("An alternate account exists"). The ban-list test is the guard; do not
  "improve" the copy toward "misreporting" as a *verb about a person*.
- **Plain-language code map.** EV-005 introduces an annotation copy map *local to
  `evidenceModel.ts`* (parallel to EV-001's `RECEIPT_CHIP_COPY`), **not** an
  addition to `gameCopy.PLAIN_LANGUAGE_COPY`. Consistent with EV-001/EV-002. The
  18 kind codes are never rendered raw — only via `getEvidenceAnnotationLabel` —
  so the `gameCopy` plain-language coverage test needs no new entry. Document
  this in the model file header.
- **No migration this card.** Annotations live in the existing `client_validation`
  jsonb. A reviewer expecting a table is pointed at the issue's explicit "V1: no
  DB migration … V2 candidate: separate `evidence_annotations` table".
- **`client_validation` shape coupling.** EV-005 introduces a new key into a
  jsonb column EV-001 reads and `submit-argument` writes at insert. The key name
  `evidenceAnnotations` is namespaced so it cannot collide with EV-001's
  `attachedEvidence`/`flags`/`valid`. If a future card formalises this column's
  shape, the `evidenceAnnotations` key must be carried forward.

## Out of scope

- The `evidence_annotations` **DB table / migration** — explicit V2 per the
  issue. EV-005 V1 persists into the existing `client_validation` jsonb.
- **Cross-room evidence linking** — explicit non-scope per the issue.
- A **public evidence library** — explicit non-scope per the issue.
- Any change to `PointStandingDelta`, `antiAmplification.ts`, or the strength
  bands. Annotations never feed scoring.
- Any change to EV-001's `EvidenceArtifact` shape, `sourceChainStatus` values,
  or `buildEvidenceArtifacts`.
- Any change to EV-002's six-row source-chain dispatch table or the "ask" CTAs.
- AI-generated annotations or AI annotation suggestions — no AI in EV-005, ever.
- Runtime profanity/accusation filtering of user-typed `note` text (the model
  ban-list covers app-authored copy; the visible picker hint is the UX
  mitigation).
- Annotations *nested deeper than one level* — the depth cap converts further
  disagreement into a synthesis move; no third annotation level exists.
- Editing or deleting an annotation after it is added — V1 is append-only
  (consistent with the read-modify-write constraint). An edit/remove path is a
  V2 concern that pairs with the `evidence_annotations` table.
- Cards-screen *new component* work. The issue says "Cards + Timeline"; EV-005
  wires the **Timeline** path fully (`TimelineNodePopover` → `SourceChainPopover`)
  and surfaces the annotation count in the shared `ReceiptChip` contract, which
  the Cards screen already consumes via the same model — so the count reflection
  reaches Cards without a new component. Any Cards-specific chip *placement*
  redesign, if one is wanted, is a separate card; the count integration itself
  is in scope and delivered through the shared contract.

## Doctrine self-check

**cdiscourse-doctrine**
- *No truth labels / verdicts.* Every annotation kind, label, helper, status
  chip, picker string, and the synthesis prompt describes a *source record* —
  "Behind a paywall", "Retraction attached", "An alternate account exists". The
  ban-list test scans all of them for verdict tokens. ✔
- *Score never blocks posting.* An annotation produces only a descriptive status
  chip. It never gates `submit-argument`, never blocks a reply. A failed
  annotation write leaves posting entirely unaffected. ✔
- *Heat ≠ truth, popularity ≠ evidence.* `summariseAnnotations` reads only the
  annotation set — no likes, views, follower counts, or author identity. The
  forbidden-imports test forbids `pointStanding`/`antiAmplification` imports so
  an annotation structurally cannot convert engagement into standing. ✔
- *AI moderator limits.* No AI authors, suggests, or scores an annotation. The
  constructor and Edge Function require an explicit human `addedByUserId`. No AI
  call anywhere in EV-005's files. ✔
- *Rules engine sacred.* EV-005 does not touch `src/lib/constitution/engine.ts`. ✔
- *Secrets / service-role.* The service-role key lives only in the
  `annotate-evidence` Edge Function (used solely for the one privileged
  `client_validation` write + the best-effort audit row), never in the client,
  never logged. The client wrapper calls `supabase.functions.invoke` only — the
  no-service-role structural test asserts this. ✔
- *Plain language.* The 18 kind codes are never rendered raw — only via
  `getEvidenceAnnotationLabel`/`getEvidenceAnnotationHelper`. ✔
- *v1 scope guards.* No voting, no winner, no search, no push, no public API,
  no OAuth. ✔

**evidence-doctrine**
- *Evidence is a first-class object responded to with more evidence/context.*
  EV-005 makes the `EvidenceArtifact` addressable by `EvidenceAnnotation`
  records — a participant responds to a source with descriptive context, not a
  verdict on the author. ✔
- *Engagement credit vs factual-standing credit stay separate.* An annotation
  has no path to `PointStandingDelta`. `retraction_attached` → `conflict_open`
  (context), it does not subtract standing or "defeat" anyone. Standing changes
  remain owned by EV-004 / `antiAmplification.ts` and respond only to new
  evidence moves. The forbidden-imports test enforces the structural gap. ✔
- *Anti-amplification.* No annotation kind, label, helper, or picker string
  references popularity; the amplification ban-list test enforces this. ✔
- *Source-chain status stays an inspection signal.* EV-005 never *upgrades*
  `EvidenceArtifact.sourceChainStatus` — `primary_source` as an *annotation* is
  a participant's descriptive claim, kept distinct from EV-001's
  `sourceChainStatus = 'primary_present'` value (admin/automated-only). ✔
- *Non-accusatory framing.* `misreporting_alleged` renders as "An alternate
  account exists"; `source_missing_quote` describes the *source*; banned person
  labels (`troll`, `bot`, `liar`, `bad faith`, …) never appear in any field. ✔
- *No hard refusals triggered.* EV-005 does not count likes toward standing,
  does not label users, does not auto-delete unsourced claims, does not treat
  missing evidence as falsehood, does not surface another user's risk score. ✔

**supabase-edge-contract**

  Edge Function only; the service-role key is referenced only inside
