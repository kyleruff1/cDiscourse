# EV-005 — Evidence-to-evidence interaction (annotations on evidence)

**Status:** Design draft
**Epic:** Epic 6 — Evidence
**Release:** v1
**Issue:** https://github.com/kyleruff1/cDiscourse/issues/120

## Goal (one paragraph)

EV-005 makes evidence a first-class object that can be *responded to with more
evidence or context* — never with a truth verdict on the original author. Today
an `EvidenceArtifact` (EV-001) has a `sourceChainStatus` describing the trail,
and EV-002 surfaces that status in a popover. EV-005 adds a thin **annotation
layer** on top of the artifact: a participant or admin can attach a small,
descriptive note to a specific artifact — `primary_source`, `retraction_attached`,
`paywalled_source`, `satire_parody_context`, `source_missing_quote`, etc. The
result is a derived **status chip** (`anchored | conflict_open | context_open |
paywalled | broken | unknown`) that the EV-002 popover and the timeline receipt
chip can surface. The doctrine that shapes every line of this design:
annotations describe the **source / record**, never accuse a **person**; an
annotation never converts popularity into factual standing (anti-amplification);
an annotation never blocks an ordinary post; and AI never authors an annotation.
Evidence-inception is capped — annotations on annotations are suppressed at one
level and replaced with a "Summarise this evidence thread" synthesis prompt, so
the surface cannot spiral into an unbounded nested argument.

## Scope decision — model is the core deliverable, one thin chip component ships with it

The card body lists three UI touch-points (popover chips, an "Add an annotation"
flow, a timeline receipt-count reflection) plus a depth cap. To keep this
buildable in one implementer pass without a redesign, EV-005 ships:

- **In scope (this card):**
  1. The pure-TS annotation model (`evidenceModel.ts` extension) — the full
     `EvidenceAnnotation` type, the 18-kind enum, `summariseAnnotations`, the
     depth-cap helper, and the plain-language label/copy map. This is the core.
  2. `EvidenceAnnotationChip.tsx` — a thin presentational chip that renders one
     annotation kind (consistent with `ReceiptChip`'s visual grammar) **plus** a
     small list/stream renderer for the popover. No new screen.
  3. A read-only annotation stream + the derived status chip wired into the
     **existing** `SourceChainPopover` body. Annotations are *displayed* here.
- **Scoped follow-up — NOT this card (EV-005a):** the interactive
  **"Add an annotation" composer flow** (the picker UI that lets a user choose
  one of the 18 kinds and persist it). This card defines the data model, the
  actor-eligibility rules, the plain-language picker vocabulary, and the
  `buildEvidenceAnnotation` constructor that the follow-up will call — but the
  picker modal itself is left to EV-005a so this card stays a single, reviewable
  pass. The model + read path are fully testable without the picker.

Rationale: the card body itself flags "if the UI surface is large, scope the
model as the core deliverable." The annotation *picker* is a genuine composer
surface (18 options, eligibility gating, note text input, persistence wiring)
and pairs naturally with EV-001's `attached_evidence` payload write path, which
does not yet exist on the client. Splitting it keeps both cards honest.

The acceptance criterion "Evidence can carry annotations" is met by the model +
read path: annotations supplied to the model (in tests, and in EV-005a from the
picker) flow through `summariseAnnotations` to a status chip rendered in the
popover and reflected on the timeline node.

### Annotation kind enum (18 kinds — verbatim from the issue)

```ts
/**
 * EV-005 — Evidence-annotation kind. Every value describes the SOURCE or the
 * RECORD, never a person. No verdict, no person-attribution. Aligned with
 * META-001's manual-tag philosophy: a participant annotation, never a flag,
 * never a moderation action.
 */
export type EvidenceAnnotationKind =
  | 'primary_source'           // the artifact is (or reaches) a primary record
  | 'secondary_analysis'       // the artifact is secondary commentary on a source
  | 'quote_attached'           // a verbatim quote is present alongside the source
  | 'source_missing_quote'    // a source pointer with no quote excerpt
  | 'quote_disputed'           // the quoted text may not match the linked source
  | 'context_requested'        // more context is being asked for on this artifact
  | 'retraction_attached'      // a retraction / correction notice is linked
  | 'source_later_updated'     // the source was revised after it was first cited
  | 'satire_parody_context'    // the source is satire / parody, not a literal report
  | 'screenshot_only_chain_weak' // only a screenshot exists; no inspectable chain
  | 'misreporting_alleged'     // an alternate account of the same facts exists
  | 'translation_context_issue' // translation may affect the meaning of the source
  | 'outdated_source'          // the source predates more current information
  | 'methodology_dispute'      // the source's method/sampling is contested
  | 'broken_link'              // the linked source no longer resolves
  | 'paywalled_source'         // the source is behind a paywall / not freely open
  | 'conflicting_source'       // another source disagrees with this one
  | 'source_chain_anchored';   // the chain reaches an inspectable anchor
```

`misreporting_alleged` is the one kind whose internal code reads close to an
accusation; its plain-language label is deliberately **"An alternate account
exists"** (record-descriptive), and the ban-list test asserts the label/helper
do not contain person-attribution tokens. See "Non-accusatory copy contract".

### The annotation object

```ts
/**
 * EV-005 — One annotation attached to one EV-001 EvidenceArtifact. Frozen at
 * construction. JSON-serializable. v1 persists inside the existing
 * meta.attached_evidence payload (no DB migration).
 */
export type EvidenceAnnotation = Readonly<{
  /** Stable id. Adapter mints: `<evidenceArtifactId>:annotation:<index>`. */
  id: string;
  /** FK back to the EV-001 EvidenceArtifact.id this annotation describes. */
  evidenceArtifactId: string;
  kind: EvidenceAnnotationKind;
  /** Optional ≤ 140-char free-text note. Describes the SOURCE; never a
   *  person. v1 renders it read-only beneath the chip. */
  note?: string;
  /** The participant/admin who added the annotation. Never AI. */
  addedByUserId: string;
  /** ISO-8601. */
  createdAt: string;
  /**
   * Annotation-nesting depth. 0 = annotation directly on an artifact.
   * 1 = annotation on an annotation (the one permitted nested level).
   * The model rejects/suppresses anything > 1 — see depth cap.
   */
  depth: 0 | 1;
  /** When depth === 1, the id of the depth-0 annotation it responds to.
   *  null/absent for depth 0. */
  parentAnnotationId?: string | null;
}>;
```

### The summary

```ts
/** EV-005 — derived status chip. Describes the annotation set's shape,
 *  never a truth verdict. */
export type EvidenceAnnotationStatusChip =
  | 'anchored'       // a primary_source / source_chain_anchored annotation present
  | 'conflict_open'  // a conflicting/disputed/retraction/misreporting annotation present
  | 'context_open'   // a context_requested annotation present, no conflict
  | 'paywalled'      // a paywalled_source annotation present, no conflict/context
  | 'broken'         // a broken_link / screenshot_only_chain_weak annotation present
  | 'unknown';       // no annotations, or none that map to a higher-priority chip

export interface EvidenceAnnotationSummary {
  /** primary_source + source_chain_anchored + quote_attached annotations. */
  primary: ReadonlyArray<EvidenceAnnotation>;
  /** conflicting_source + quote_disputed + retraction_attached +
   *  misreporting_alleged + conflicting/methodology annotations. */
  conflicts: ReadonlyArray<EvidenceAnnotation>;
  /** context_requested annotations. */
  contextRequests: ReadonlyArray<EvidenceAnnotation>;
  /** The single derived status chip (priority resolution — see below). */
  statusChip: EvidenceAnnotationStatusChip;
  /** Total annotation count (depth 0 + depth 1). Drives the "+N" affordance
   *  and the timeline receipt-count reflection. */
  count: number;
  /** Plain-English label for statusChip. Max 32 chars. Never a code. */
  statusLabel: string;
  /** Plain-English helper for statusChip. Max 90 chars. */
  statusHelper: string;
  /** Logical tone for the chip renderer. NOT a truth label. */
  tone: 'neutral' | 'info' | 'attention' | 'muted';
}
```

**Status-chip priority resolution** (deterministic, asserted by tests). When
multiple kinds are present the chip reflects the *most action-worthy* state, in
this order: `conflict_open` > `context_open` > `broken` > `paywalled` >
`anchored` > `unknown`. Rationale: a conflict/retraction is the state a reader
most needs to see; an anchored chip must never *hide* an open conflict.
`broken` outranks `paywalled` because a dead link is a harder failure than a
paywall. Empty annotation list → `unknown` with label "No annotations yet".

### Doctrine note on the summary

`summariseAnnotations` reads **only** the annotation set. It never reads
engagement, likes, view counts, follower counts, or the artifact author's
identity. It never *upgrades* `EvidenceArtifact.sourceChainStatus` and never
touches `PointStandingDelta`. Adding a `retraction_attached` annotation produces
`conflict_open` — it adds context; it does not "defeat" anyone and it does not
subtract factual standing. Standing changes remain owned by EV-004 /
`antiAmplification.ts` and only respond to *new evidence moves*, not annotations.

## API / interface contracts

```ts
// ── Constructor (used by EV-005a's picker; tested directly here) ──
export function buildEvidenceAnnotation(input: {
  evidenceArtifactId: string;
  kind: EvidenceAnnotationKind;
  addedByUserId: string;
  createdAt: string;
  index: number;                 // for the deterministic id
  note?: string | null;
  depth?: 0 | 1;                  // defaults to 0
  parentAnnotationId?: string | null;
}): EvidenceAnnotation;
// Trims note to 140 chars; freezes the result; mints
// id = `${evidenceArtifactId}:annotation:${index}`.

// ── Adapter: raw payload array → typed, depth-capped annotations ──
export function buildEvidenceAnnotations(input: {
  evidenceArtifactId: string;
  addedByUserId: string;
  createdAt: string;
  raw: ReadonlyArray<{
    kind: EvidenceAnnotationKind;
    note?: string | null;
    depth?: number;
    parentAnnotationId?: string | null;
  }>;
}): ReadonlyArray<EvidenceAnnotation>;
// Drops entries with an unknown kind. Runs every entry through
// enforceAnnotationDepthCap before returning.

// ── Summary ──
export function summariseAnnotations(
  annotations: ReadonlyArray<EvidenceAnnotation>,
): EvidenceAnnotationSummary;
// Pure. Empty input → { statusChip: 'unknown', count: 0, ... }.

// ── Depth cap ──
export interface AnnotationDepthCapResult {
  /** Annotations that are kept (depth 0, plus depth-1 whose parent exists). */
  accepted: ReadonlyArray<EvidenceAnnotation>;
  /** Annotations suppressed for exceeding the one permitted nested level. */
  suppressed: ReadonlyArray<EvidenceAnnotation>;
  /** True when at least one depth-1 annotation exists — the UI then shows the
   *  "Summarise this evidence thread" prompt instead of allowing a 3rd level. */
  showsSynthesisPrompt: boolean;
  /** Plain-English prompt copy. Empty string when showsSynthesisPrompt false. */
  synthesisPromptLabel: string;
}
export function enforceAnnotationDepthCap(
  annotations: ReadonlyArray<EvidenceAnnotation>,
): AnnotationDepthCapResult;
// Rule: depth 0 always accepted. depth 1 accepted IF parentAnnotationId
// resolves to an accepted depth-0 annotation. depth >= 2 (or a depth-1 with a
// missing/depth-1 parent) is suppressed. showsSynthesisPrompt is true whenever
// any depth-1 annotation is accepted (the next level is closed → synthesis).

// ── Plain-language copy lookup ──
export function getEvidenceAnnotationLabel(kind: EvidenceAnnotationKind): string;
export function getEvidenceAnnotationHelper(kind: EvidenceAnnotationKind): string;
// Both return frozen plain-English strings; never a snake_case code.
```

### Actor rules — who may annotate (defines the EV-005a contract)

Aligned with META-001's `MANUAL_TAG_ELIGIBILITY` philosophy. EV-005 ships the
**read** path; these rules are encoded as a pure exported table
(`EVIDENCE_ANNOTATION_ELIGIBILITY`) and an `isAnnotationAllowed(kind, context)`
helper so EV-005a's picker has no decisions left to make.

| Actor | May add annotations? | Notes |
| --- | --- | --- |
| Participant on **another** bubble | Yes | The normal case — respond to evidence with context. |
| Participant on **own** bubble | Yes, but only the three *self-descriptive* kinds: `source_later_updated`, `retraction_attached`, `context_requested` | Mirrors META-001's own-bubble allowance for `concession_offered` / `narrowed_claim` / `ready_for_synthesis`. An author may annotate their own source to *add context*, never to grade another's. |
| **Observer** | No | Mirrors META-001 exactly: observers never apply tags in v1. The annotation stream is read-only for observers; the synthesis prompt renders disabled with "Join a side to add context". |
| **Admin** | Yes — all 18 kinds | Moderation-review parity with META-001's admin row. |
| **AI** | Never | No code path constructs an annotation from an AI call. The constructor takes an explicit `addedByUserId`; no AI identity exists. |

Eligibility is a *gate on adding*, not on *reading*: everyone (including
observers) can read the annotation stream and the status chip.

## Annotation-add flow (EV-005a — documented here so the model is built right)

1. A participant opens the EV-002 `SourceChainPopover` on an evidence-bearing
   node and expands the source-chain section.
2. Below the read-only `EvidenceAnnotationStream`, an "Add context to this
   source" affordance appears (EV-005a renders it; it is hidden for observers
   and gated by `isAnnotationAllowed`).
3. Tapping it opens a picker listing the *eligible* kinds (own-bubble → 3 kinds;
   other-bubble → all 18; admin → all 18) by their **plain-language labels**.
4. The user picks a kind, optionally types a ≤140-char note describing the
   source, and confirms.
5. EV-005a calls `buildEvidenceAnnotation(...)`, appends it to the
   `meta.attached_evidence` payload for that artifact (same write path EV-001
   defined for artifacts), and the existing argument-update flow persists it.
   **No new Edge Function, no service-role, no AI call.**
6. The stream re-renders via `summariseAnnotations` + `enforceAnnotationDepthCap`.
7. If the user tries to annotate a depth-1 annotation, the picker is replaced by
   the **"Summarise this evidence thread"** prompt — which routes to the
   existing synthesis composer preset (`ready_for_synthesis` move). No third
   level is ever created.

EV-005 (this card) delivers steps 6's pure machinery and the read rendering of
steps 1, 6, 7. Steps 2–5 (the picker + persistence write) are EV-005a.

## Edge cases

The implementer must handle each:

- **Empty annotation list** — `summariseAnnotations([])` → `statusChip: 'unknown'`,
  `count: 0`, label "No annotations yet"; `SourceChainPopover` renders exactly as
  today (no stream, no regression to EV-002 behaviour).
- **Single annotation** — chip renders, `+N` affordance absent, status chip
  reflects that one kind's priority bucket.
- **Conflicting kinds present together** — e.g. `primary_source` AND
  `retraction_attached`: priority resolution picks `conflict_open`; the
  `primary` array still contains the `primary_source` annotation (the anchor is
  not erased, just not the headline). Test asserts both.
- **Unknown / malformed kind in the raw payload** — `buildEvidenceAnnotations`
  drops the entry silently (matches EV-001's "drop empty attachment" behaviour);
  it never throws.
- **depth-1 annotation whose `parentAnnotationId` does not resolve** — suppressed
  by `enforceAnnotationDepthCap` (counts as orphaned); `showsSynthesisPrompt`
  unaffected by an orphan.
- **depth ≥ 2 in the payload** (should not happen, but defend) — suppressed; the
  synthesis prompt is shown because the level is already at the cap.
- **note exceeds 140 chars** — `buildEvidenceAnnotation` truncates to 140;
  empty/whitespace-only note is stored as `undefined`, not `''`.
- **note containing a person-attribution token** — out of scope for runtime
  filtering (notes are free text); the *ban-list test* covers only the
  app-authored labels/helpers/prompts. The picker copy (EV-005a) will carry a
  one-line "describe the source, not the person" hint; documented here so
  EV-005a inherits it.
- **Observer viewing the stream** — stream + status chip fully visible; the
  synthesis-prompt row and (in EV-005a) the add affordance render disabled with
  the EV-002-style helper.
- **Own-bubble author** — sees the full stream; may add only the 3
  self-descriptive kinds (enforced by `isAnnotationAllowed`).
- **Artifact with no annotations but a `broken` sourceChainStatus** — the EV-002
  receipt chip still shows "Source trail is weak" from EV-001; EV-005's
  annotation status chip independently shows `unknown` ("No annotations yet").
  The two chips are distinct signals and may legitimately disagree — documented
  so the implementer does not try to "reconcile" them.
- **Anti-amplification edge case — "does a popular retraction defeat the
  point?"** — No. An annotation never feeds `PointStandingDelta`. Adding
  `retraction_attached` produces `conflict_open` (context), full stop. The model
  has no path to standing; the doctrine self-check asserts this by the *absence*
  of any `pointStanding` import in `evidenceModel.ts`.
- **Concurrent annotations from two participants** — each is a separate payload
  entry with its own deterministic id (`...:annotation:<index>`); the adapter
  preserves order. v1 has no merge conflict because annotations are append-only
  within the artifact payload. (True last-write-wins on the whole
  `attached_evidence` blob is an EV-005a/persistence concern, noted there.)
- **Reduce motion** — the stream has no animation; the depth-cap prompt appears
  without a transition. Nothing to gate. (`SourceChainPopover` already handles
  its own expand animation.)

- **`SourceChainPopover` is a shipped EV-002 surface.** The change must be purely
  additive — a new optional prop with a `[]` default, a new conditional block
  below the existing artifact rows. If the implementer finds they need to
  restructure the popover body, that is a redesign — stop and flag it. The
  EV-002 "ask" CTA, observer-disable, and own-message paths must keep their
  existing tests green.
- **Two chips on one node.** The EV-002 receipt chip (source-chain status) and
  the EV-005 annotation status chip both summarise "evidence health." They are
  distinct and may disagree. The risk is a reader conflating them. Mitigation:
  distinct labels ("Source trail is weak" vs "Open conflict"), distinct testIDs,
  and the annotation stream sits *inside* the expanded popover (secondary
  surface) while the receipt chip stays in the band row (primary surface).
- **`misreporting_alleged` copy drift.** Its internal code is the closest to an
  accusation. The plain-language label/helper must stay record-descriptive
  ("An alternate account exists"). The ban-list test is the guard; the
  implementer must not "improve" the copy toward "misreporting" as a *verb about
  a person*.
- **Plain-language code map.** EV-005 introduces an annotation copy map *local
  to `evidenceModel.ts`* (parallel to EV-001's `RECEIPT_CHIP_COPY`), **not** an
  addition to `gameCopy.PLAIN_LANGUAGE_COPY`. This is intentional and consistent
  with how EV-001/EV-002 keep their own frozen copy tables. The 18 annotation

  — so the `gameCopy` plain-language coverage test does not need a new entry.
  If a future card surfaces annotation kinds through `gameCopy`, that is its

  does not flag a "missing `gameCopy` mapping."
- **No migration this card.** Annotations live in the `meta.attached_evidence`
  payload (EV-001 path b). If a reviewer expects a table, point them at the

  `evidence_annotations` table — DOES NOT ship in this card."
- **Test count.** Expect roughly +60–80 tests across four files. The implementer

## Doctrine self-check

**cdiscourse-doctrine**
- *No truth labels / verdicts.* Every annotation kind, label, helper, status
  chip, and the synthesis prompt describes a *source record* — `paywalled_source`,
  `retraction_attached`, `outdated_source`, `An alternate account exists`. The
  ban-list test scans all of them for `winner/loser/true/false/liar/dishonest/
  fake/fraud/hoax/bad faith/manipulative/correct/wrong/proof/verdict`. ✔
- *Score never blocks posting.* An annotation produces only a descriptive status
  chip. It never gates `submit-argument`, never blocks a reply, never deletes or
  hides content. ✔
- *Heat ≠ truth, popularity ≠ evidence.* `summariseAnnotations` reads only the
  annotation set — no likes, views, follower counts, or author identity. The
  forbidden-imports test forbids `pointStanding`/`antiAmplification` imports so
  an annotation structurally *cannot* convert engagement into standing. ✔
- *AI moderator limits.* No AI authors, suggests, or scores an annotation. The
  constructor requires an explicit human `addedByUserId`. No AI call anywhere in
  EV-005's files. ✔
- *Rules engine sacred.* EV-005 does not touch `src/lib/constitution/engine.ts`. ✔
- *Secrets / service-role.* No service-role, no Edge Function, no secrets. v1 is
  pure-TS + RN + the existing `meta.attached_evidence` payload. ✔
- *Plain language.* The 18 kind codes are never rendered raw — only via
  `getEvidenceAnnotationLabel`/`getEvidenceAnnotationHelper`. ✔
- *v1 scope guards.* No voting, no winner, no search, no push, no public API,
  no OAuth. ✔

**evidence-doctrine**
- *Evidence is a first-class object responded to with more evidence/context.*
  EV-005 makes the `EvidenceArtifact` addressable by `EvidenceAnnotation`
  records — a participant responds to a source with descriptive context, not a
  verdict on the author. ✔
- *Engagement credit vs factual-standing credit stay separate.* An annotation
  has no path to `PointStandingDelta`. `retraction_attached` → `conflict_open`
  (context), it does not subtract standing or "defeat" anyone. Standing changes
  remain owned by EV-004 / `antiAmplification.ts` and respond only to new
  evidence moves. ✔
- *Anti-amplification.* No annotation kind, label, or helper references
  popularity; the amplification ban-list test enforces this; the
  forbidden-imports test enforces the structural separation. ✔
- *Source-chain status stays a gameplay/inspection signal.* EV-005 never
  *upgrades* `EvidenceArtifact.sourceChainStatus` — `primary_source` as an
  *annotation* is a participant's descriptive claim about the record, kept
  distinct from the EV-001 `sourceChainStatus = 'primary_present'` value, which
  remains admin/automated-only. ✔
- *Non-accusatory framing.* "evidence of dishonesty" → never used; the issue's
  prescribed replacements (`contradictory record` → `conflicting_source`,
  `retraction_attached`, `context_requested`) are the vocabulary;
  `misreporting_alleged` renders as "An alternate account exists";
  `source_missing_quote` describes the *source*, not the person. ✔
- *No hard refusals triggered.* EV-005 does not count likes toward standing,
  does not label users, does not auto-delete unsourced claims, does not treat
  missing evidence as falsehood, does not surface another user's risk score. ✔
