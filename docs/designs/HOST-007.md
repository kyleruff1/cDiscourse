## Goal

HOST-007 is the **IAM + IAP access control** card. It is the last card on the critical path to a usable, gated, Google-sign-in-protected dev URL at `https://dev.cdiscourse.com`. After this card lands and the operator runs the runbook, the dev URL behaves as follows:

| Caller | Status | Result |
|---|---|---|
| Anonymous browser, no Google session | unauth | 302 to Google IAP sign-in page |
| Google-signed-in user, no `roles/iap.httpsResourceAccessor` grant on `cdiscourse-dev` | denied | IAP "You don't have access" 403 page |
| Google-signed-in user with `roles/iap.httpsResourceAccessor` grant | allowed | App loads; Supabase auth then handles app-level sign-in |
| `curl` with `gcloud auth print-identity-token --audiences=<IAP_CLIENT_ID>` (operator) | allowed | App responds 200 (or whatever the actual route returns) |

The card delivers the **contract, IAM templates, helper scripts, runbook, and tests** for:

- OAuth consent screen configuration (manual GCP Console flow — captured in the runbook, not in code).
- IAP enablement on the Cloud Run service `cdiscourse-dev` (via the **Cloud Run IAP integration**, NOT an HTTPS Load Balancer).
- Tester IAM grant procedure (`gcloud iap web add-iam-policy-binding ... --role=roles/iap.httpsResourceAccessor`).
- Onboarding helper that takes a list of tester emails from a YAML template and prints the operator-runnable `gcloud` grant commands.
- Offboarding helper that prints the matching `remove-iam-policy-binding` commands.
- Post-IAP verification helper that confirms anonymous → 302 + token-authenticated → 200.
- New operator runbook `docs/deployment/host-007-iap-runbook.md` with 9 numbered steps and explicit "agent does not run this" markers.
- Doc edits: extend HOST-001 Phase 6 hand-off + append to `docs/current-status.md`.

The card explicitly does **NOT** own:

- **Cloud Armor IP allowlist.** D4 + D6 retired it; not designed here.
- **HTTPS Load Balancer + serverless NEG.** Not built unless IAP-on-Cloud-Run is currently unsupported (see §"IAP enablement path decision" below — if the surface check forces an LB, surface as a deviation and STOP).
- **DNS records.** HOST-006 owns DNS.
- **Cloud Run service spec changes.** `infra/cloud-run/cdiscourse-dev.template.yaml` is HOST-001 territory.
- **Runtime / deployer SA changes.** `infra/iam/cdiscourse-dev-runner.iam.yaml` and `infra/iam/cdiscourse-deployer.iam.yaml` are fixed by HOST-001. IAP tester bindings live in a NEW IAM template (`infra/iam/iap-testers.iam.yaml.template`).
- **Production parallels.** `cdiscourse-prod` IAP + prod OAuth consent screen are HOST-008, stub-only.
- **Modifications** to `src/lib/supabase.ts`, `scripts/runtime/server.mjs`, `scripts/build/inject-runtime-env.*`, `infra/secrets/*`, or any HOST-005 helper.
- **Modifications** to HOST-006 helpers (parallel card).

Doctrine constraints that shape this design (per `cdiscourse-doctrine` skill + master plan §16):

- **Agent never executes any `gcloud` command** that mutates IAM, IAP, OAuth consent, or service state. Helpers print operator-runnable command shapes only.
- **Agent never sees real tester emails.** The tester YAML committed to the repo is a `.template` file with placeholder addresses (`tester-1@example.com`). The operator copies it to a gitignored working file and adds real emails locally.
- **Agent never sees / commits OAuth client IDs or secrets.** Those are generated by GCP at consent-screen creation. The operator stores them locally in their working environment; they are never pasted into Claude or committed.
- **No service-role / Anthropic / xAI / Resend keys** anywhere in this card's surface.
- **No new dependency.** Node built-ins only (`node:fs`, `node:path`, `node:child_process`, `node:https`).
- **No `console.log` in committed code** outside test files.
- **TypeScript strict mode** preserved (test files only — the helpers are `.mjs`).
- **HOST-001 / HOST-005 / HOST-006 files unchanged.** Tests assert this.
- **No service-account JSON files.** Operator ADC + impersonation only.
- **No verdict / truth copy.** This is plumbing — no user-facing scoring text is introduced.
- **No AI calls** from the production app — IAP only authenticates Google identities; it does not call any AI provider.

## Why IAP, not Cloud Armor — short justification

The operator-locked decision is D6: IAP. The rationale (already established in master plan §9.3 and confirmed by D4) is:

- **Tester onboarding cost.** One IAM grant per tester (≤ 5 seconds) vs capturing each tester's CIDR (home ISPs rotate; mobile carriers are too broad). For invite-only dev, IAM wins.
- **Cost.** IAP IAM is free-tier. Cloud Armor + HTTPS LB + serverless NEG adds ~$18/month base plus per-request costs.
- **Surface.** Cloud Armor on Cloud Run forces an HTTPS LB build-out (master plan §9.2). IAP-on-Cloud-Run does not. Less infra, less drift.
- **Reversibility.** Revoking a tester is one IAM revoke. Removing a tester's IP from a Cloud Armor allow rule is the same shape but the IP may be shared with non-testers.
- **Sign-in friction is acceptable.** The operator explicitly accepts the Google sign-in step in front of Supabase sign-in for an invite-only sandbox. Wider-release revisit is tracked at master plan R14.

This section exists in the design so the implementer does not relitigate the choice. If a constraint surfaces during implementation that makes IAP unworkable, the implementer must STOP and surface the deviation (see §"IAP enablement path decision" below).

## OAuth consent screen design

The OAuth consent screen is a **manual GCP Console flow**. There is no clean `gcloud` equivalent for the full consent-screen lifecycle as of last documented surface. The runbook captures the UI steps; this design locks the values.

### Decision matrix

| Field | Value | Rationale |
|---|---|---|
| User type | **External / Testing status** (default) | Operator does not have a Google Workspace org for `cdiscourse.com` (verified at runbook step 2). External + Testing keeps the screen unpublished; Google's OAuth verification flow is not needed for an invite-only sandbox. **If the operator does have a Workspace org bound to `cdiscourse.com`, prefer Internal** — runbook step 2a captures the branch. |
| Product name | **`CDiscourse Dev`** | Distinguishes from a future `CDiscourse` production consent screen (HOST-008). |
| User support email | **Operator's personal Google account** | The only viable choice for v0. |
| App logo | **None (placeholder OK)** | Branding deferred. Operator may upload `/dev-logo.png` later — out of scope for HOST-007. |
| Application home page | `https://dev.cdiscourse.com` | Matches D3. |
| Application privacy policy | **Blank in Testing status** | Privacy policy required only when publishing to production verification. |
| Application terms of service | **Blank in Testing status** | Same. |
| Authorized domains | `cdiscourse.com` | Minimum. `*.cdiscourse.com` is implicitly covered. |
| Developer contact | Operator's personal Google account | Required field. |
| Scopes | `openid`, `email`, `profile` | IAP standard. No additional scopes — IAP only needs identity confirmation. |
| Test users | Operator email + tester emails from `infra/iam/iap-testers.iam.yaml` (operator-edited working copy, NOT the committed template) | External / Testing status caps at **100 test users**. Sufficient for invite-only dev. |
| Publishing status | **Testing** (do NOT publish) | Publishing triggers Google OAuth verification (logo / domain ownership / privacy policy review). Out of scope. |

### Consent screen creation step contract

The runbook (§"Operator runbook" below, step 2) walks the operator through:

1. GCP Console → APIs & Services → OAuth consent screen.
2. Pick User type (Internal vs External — runbook decision tree).
3. Fill the fields above.
4. Add test users (paste operator email + tester emails — agent never sees these).
5. Save. Note: the operator does NOT promote to Production.

**Outputs the operator captures locally** (never pasted into Claude, never committed):

- OAuth client ID for IAP (auto-generated when IAP is enabled in step 3 — different from the consent screen client ID; this is for the IAP IAM `audiences` parameter in `gcloud auth print-identity-token`).

Per the doctrine: the agent never reads, prints, or echoes any of these values. They live only in the operator's working notes.

## IAP enablement path decision

### The two paths

| Path | Description | LB needed? | Status |
|---|---|---|---|
| **A. Cloud Run IAP integration** (chosen) | `gcloud iap web enable --resource-type=cloud-run --service=cdiscourse-dev --region=us-central1`. IAP attaches directly to the Cloud Run service. Requests still hit Cloud Run's `*.run.app` URL (or the mapped custom domain) — IAP intercepts unauthenticated requests before the Cloud Run service is reached. | **No** | Supported via `gcloud iap` post-2024 (with Cloud Run direct integration). Surface flux possible — see "Known unknowns". |
| **B. HTTPS LB + serverless NEG + IAP on backend service** | Old path. Operator builds an HTTPS LB, points a serverless NEG at Cloud Run, enables IAP on the LB's backend service. | **Yes** | Strictly more infra. Forces D4-violating LB cost. |

### Decision: Path A — Cloud Run IAP integration

Justification:
- D6 chose IAM + IAP without explicitly requiring an HTTPS LB. Direct Cloud Run IAP integration is the minimum-surface implementation.
- HOST-001's locked decision D6 explicitly notes "direct Cloud Run domain mapping (no HTTPS LB)" as the default.
- HOST-006 (parallel) maps `dev.cdiscourse.com` directly to the Cloud Run service. Adding an LB now would invalidate HOST-006's design.
- Cost: zero added infra beyond the IAP attach. Path B adds ~$18/mo base for the LB.
- Operator action surface: 1 `gcloud iap web enable` + 1 grant batch vs Path B's LB-create + backend-service-create + NEG-create + IAP-enable + 4× redirect.

### Surface-flux risk + the STOP gate

The `gcloud iap web enable --resource-type=cloud-run` surface is the **post-2024 supported** Cloud Run direct IAP integration. Two known risks:

1. **Flag rename / `gcloud beta` migration.** If the implementer or operator finds `--resource-type=cloud-run` is not accepted by the installed `gcloud` version, fall back to `gcloud beta iap web enable --resource-type=cloud-run --service=cdiscourse-dev --region=us-central1`. The runbook lists both shapes.
2. **Region-availability constraint.** If IAP-on-Cloud-Run is not available in `us-central1` at runbook time (unlikely; it has been GA in us-central1), **STOP** and surface — do NOT silently fall back to building an HTTPS LB. Path B is a HOST-007 deviation that requires operator re-approval.

The verification helper (§"Helper scripts" below) does NOT enable IAP and does NOT itself run `gcloud iap web enable`. It only verifies the resulting HTTP behaviour.

### The IAP enablement command shape (verbatim, for the runbook)

```bash
gcloud iap web enable \
  --resource-type=cloud-run \
  --service=cdiscourse-dev \
  --region=us-central1 \
  --project=cdiscourse-host
```

Fallback (only if the operator's gcloud version does not accept the above):

```bash
gcloud beta iap web enable \
  --resource-type=cloud-run \
  --service=cdiscourse-dev \
  --region=us-central1 \
  --project=cdiscourse-host
```

Disable / rollback shape:

```bash
gcloud iap web disable \
  --resource-type=cloud-run \
  --service=cdiscourse-dev \
  --region=us-central1 \
  --project=cdiscourse-host
```

## Tester IAM grant contract

### Role + scope

| Field | Value |
|---|---|
| Role | `roles/iap.httpsResourceAccessor` |
| Scope | **Per-resource** (the Cloud Run service `cdiscourse-dev`) — NOT project-wide |
| Member shape | `user:<email>` (real human Google accounts only; no `serviceAccount:` for testers) |

**Why per-resource, not project-wide:** project-wide `roles/iap.httpsResourceAccessor` would also grant access to any future IAP-protected resource in the project (prod, Cloud Functions, etc.). The per-resource grant scopes the access to exactly one Cloud Run service.

### Grant command shape (operator copies + runs)

```bash
gcloud iap web add-iam-policy-binding \
  --resource-type=cloud-run \
  --service=cdiscourse-dev \
  --region=us-central1 \
  --member="user:<email>" \
  --role="roles/iap.httpsResourceAccessor" \
  --project=cdiscourse-host
```

### Revoke command shape

```bash
gcloud iap web remove-iam-policy-binding \
  --resource-type=cloud-run \
  --service=cdiscourse-dev \
  --region=us-central1 \
  --member="user:<email>" \
  --role="roles/iap.httpsResourceAccessor" \
  --project=cdiscourse-host
```

### Surface-flux fallback

If `gcloud iap web add-iam-policy-binding --resource-type=cloud-run` is not accepted in the installed gcloud version, the equivalent shape via `gcloud beta`:

```bash
gcloud beta iap web add-iam-policy-binding \
  --resource-type=cloud-run \
  --service=cdiscourse-dev \
  --region=us-central1 \
  --member="user:<email>" \
  --role="roles/iap.httpsResourceAccessor" \
  --project=cdiscourse-host
```

The helper script outputs both the GA shape (default) and the beta fallback (commented).

## IAM YAML template

### File path

`infra/iam/iap-testers.iam.yaml.template`

The `.template` suffix is intentional. The committed file contains placeholder addresses only. The operator copies it to `infra/iam/iap-testers.iam.yaml` (gitignored — added to `.gitignore` in this card) and edits the working copy with real tester emails.

### File contents (placeholders only — NO real emails)

```yaml
# HOST-007 — IAP tester membership template for cdiscourse-dev.
#
# THIS IS A TEMPLATE. The committed copy contains placeholder addresses.
# The operator copies this to `infra/iam/iap-testers.iam.yaml` (gitignored)
# and replaces the placeholders with real tester emails.
#
# The agent NEVER sees real tester emails. The committed file is intentionally
# inert — the helpers refuse to print grant commands if the file still
# contains placeholder addresses.
#
# Doctrine encoded:
#   - All members are `user:<email>` — no `serviceAccount:` allowed.
#   - Role is fixed to `roles/iap.httpsResourceAccessor`.
#   - Resource is fixed to the Cloud Run service `cdiscourse-dev` in
#     `us-central1` / `cdiscourse-host`.
#   - environment must be `dev`. `prod` is reserved for HOST-008.
#   - At most 100 entries (OAuth Testing-status cap; helpers refuse if exceeded).

version: 1
environment: dev
project: cdiscourse-host
region: us-central1
service: cdiscourse-dev
role: roles/iap.httpsResourceAccessor

testers:
  # Replace each placeholder with a real tester email. Helpers refuse if any
  # entry still matches `*@example.com` or `<...>` patterns.
  - email: tester-1@example.com
    note: placeholder — operator replaces
  - email: tester-2@example.com
    note: placeholder — operator replaces
```

### JSON Schema (no comments — separate file)

A draft-07 JSON Schema for the YAML lives at `infra/iam/iap-testers.schema.json`. Even though the file is YAML, the schema is JSON-shaped because:

- Node built-ins parse JSON natively; YAML parsing requires a dep (forbidden).
- The helpers use a **minimal hand-rolled YAML parser** for this single file's two-level shape (top-level scalars + a single `testers:` list of `{ email, note }`). No nested anchors, no flow style. The parser refuses anything the schema doesn't allow.

Schema enforces:

- `version == 1`, `environment == "dev"` (until HOST-008), `project == "cdiscourse-host"`, `region == "us-central1"`, `service == "cdiscourse-dev"`, `role == "roles/iap.httpsResourceAccessor"`.
- `testers` is an array; `1 ≤ length ≤ 100`.
- Each tester `email` matches `^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$`.
- Each tester `email` does NOT match the forbidden-placeholder patterns (`example.com$`, `^<.*>$`).

### Helpers read the template, emit operator-runnable commands

The helpers (next section) parse the YAML, validate against schema, and emit the `add-iam-policy-binding` / `remove-iam-policy-binding` commands on stdout. They never call `gcloud`.

## Helper scripts

All helpers live under `scripts/deploy/`. Each is a `.mjs` Node entrypoint with `.ps1` + `.sh` wrappers (matching HOST-005 convention; the wrappers do not re-implement logic).

### Common doctrine (asserted by tests)

- Refuse to execute if any IAM/IAP mutation would be required.
- Refuse if `gcloud` is not on PATH (when the helper needs to invoke it — only `verify-iap.mjs` does).
- Refuse if project mismatch (where applicable).
- Never log a token, a Bearer header, an Authorization value, or an OAuth client secret.
- Never write to a file outside the test fixture dir; stdout only for command-printing helpers.
- Exit codes documented at file head.

### `scripts/deploy/print-iap-tester-commands.mjs` (+ `.ps1`, `.sh`)

**Purpose:** Read the operator's working `iap-testers.iam.yaml`, validate against schema, and print operator-runnable `gcloud iap web add-iam-policy-binding` commands.

**Flags:**
- `--testers=<path>` (required) — path to the YAML.
- `--check-only` — validate the YAML and exit; print no commands.
- `--include-beta-fallback` (default: on) — also print the `gcloud beta` shape as a commented fallback.
- `--no-include-beta-fallback` — suppress the beta line.
- `--help` / `-h`.

**Behaviour:**
1. Parse YAML via the minimal hand-rolled parser. Refuse on parse error (exit 2).
2. Schema-check (exit 2 on violation).
3. Forbidden-placeholder check (exit 4 if any email matches `@example.com$` or `^<.*>$`).
4. Forbidden-token check on every string field (`anthropic`, `xai`, `bearer`, `resend`, `service-role`, `service_role`, JWT-shape, `sk-ant-`, etc. — same list as HOST-005). Exit 3 on hit.
5. Empty-testers refusal (exit 5).
6. Over-100-testers refusal (exit 6).
7. If `--check-only`, exit 0 here.
8. Print header (agent-non-execution marker, project / region / service identifiers, ISO timestamp).
9. For each tester, print the `gcloud iap web add-iam-policy-binding` command (and the beta fallback as a comment if enabled).
10. Print footer linking to `verify-iap.mjs`.

**Exit codes:**
- `0` printed successfully
- `2` YAML parse / schema error
- `3` forbidden token in any field
- `4` placeholder email still present
- `5` empty testers list
- `6` over 100 testers
- `7` `--testers` flag missing

**Doctrine:**
- Never reads `.env*`.
- Never invokes `gcloud`.
- Never accepts a value via `--member=` or `--token=`.
- Never writes a file.
- Output is stdout only.

### `scripts/deploy/print-iap-tester-removals.mjs` (+ `.ps1`, `.sh`)

**Purpose:** Print operator-runnable `gcloud iap web remove-iam-policy-binding` commands.

**Two modes:**
1. `--removed-email=<email>` — print a single revoke command for that email. Does NOT require the email to appear in the YAML (it might have already been removed).
2. No `--removed-email` — read the YAML and print revoke commands for every tester in it (used for full bulk-offboard or transcript-style audits).

**Flags:**
- `--testers=<path>` (required) — same shape as the grant helper.
- `--removed-email=<email>` (optional) — single-email mode.
- `--include-beta-fallback` / `--no-include-beta-fallback` (default: on).
- `--help` / `-h`.

**Behaviour mirrors the grant helper** except the action is `remove-iam-policy-binding`. Same email-shape validation. Same forbidden-token scan. Same agent-non-execution header.

**Exit codes:** same as grant helper (0/2/3/4/5/6/7).

### `scripts/deploy/verify-iap.mjs` (+ `.ps1`, `.sh`)

**Purpose:** Diagnostic. Confirm anonymous HEAD → 302 and token-authenticated HEAD → 200. Operator runs this after the consent + enable + grant steps.

**Flags:**
- `--url=<https-url>` (required) — defaults rejected; explicit URL.
- `--gcloud-bin=<path>` (default: `gcloud`) — for testability; tests inject a stub.
- `--no-auth-check` — skip the token-authenticated HEAD (anonymous-only mode).
- `--no-anon-check` — skip the anonymous HEAD (token-only mode).
- `--help` / `-h`.

**Behaviour:**
1. Validate `--url`: must start with `https://`. Refuse if `http://` or non-URL.
2. **Anonymous HEAD**: `node:https.request({ method: 'HEAD' })` (built-in only) against the URL. Expected: status in `{302, 303}` AND `Location` header starting with `https://accounts.google.com/` (or the IAP login URL prefix `https://iap.googleapis.com/`).
3. **Token-authenticated HEAD**: only run if `--no-auth-check` is absent. Invoke `${gcloudBin} auth print-identity-token --audiences=<IAP_CLIENT_ID>` via `child_process.spawnSync`. The `IAP_CLIENT_ID` is read from `GOOGLE_IAP_CLIENT_ID` env var (operator sets locally; helper never echoes the value). Helper exits 9 if the env var is unset. Use the resulting token in an `Authorization: Bearer ${TOKEN}` HEAD request. Expected: status in `{200, 204, 304}` (the app's actual healthy responses).
4. Emit a compact diagnostic block to stdout:
   ```
   verify-iap — diagnostic
   url:                <url>
   anonymous HEAD:     302  (Location: https://accounts.google.com/…  -- redacted)
   authenticated HEAD: 200
   verdict:            PASS
   ```
   The `Location` header is **redacted past the path component** so query-string state tokens aren't logged.
5. Exit 0 on PASS, 1 on FAIL with a single-line summary on stderr.

**Doctrine asserted by tests:**
- Never logs the identity token. The token string never appears in stdout or stderr.
- Never logs `Authorization` header values.
- Never logs the `GOOGLE_IAP_CLIENT_ID` value (the env var name is fine; the value is not echoed).
- Never reads `.env*` files.
- Never invokes any `gcloud` command other than `gcloud auth print-identity-token --audiences=...` (and `gcloud --version` to check PATH). No mutation surface.
- Refuses if `--url` is missing or not `https://`.
- Refuses if `GOOGLE_IAP_CLIENT_ID` is unset and `--no-auth-check` is absent.

**Exit codes:**
- `0` PASS
- `1` FAIL (clear message on stderr)
- `2` flag error
- `3` `gcloud` not on PATH
- `4` `--url` not `https://`
- `9` `GOOGLE_IAP_CLIENT_ID` env var unset (when auth check requested)

## Rollback

| Failure mode | Operator action |
|---|---|
| IAP attach succeeded but no testers granted yet → URL returns 302 to Google but all sign-ins are rejected | Continue to step 5; grant the operator's own email first. |
| Wrong tester granted (typo) | Run `print-iap-tester-removals.mjs --removed-email=<typo>` and execute the revoke command. |

| Operator must restore access without IAP (emergency only) | Temporarily flip Cloud Run service to `--allow-unauthenticated` with `gcloud run services update cdiscourse-dev --allow-unauthenticated --region=us-central1 --project=cdiscourse-host`. The runbook prints a loud red warning. Restore via `--no-allow-unauthenticated` within minutes — the URL becomes globally readable in the interim. **Strongly discouraged for v0.** |
| Full IAP detach | `gcloud iap web disable --resource-type=cloud-run --service=cdiscourse-dev --region=us-central1 --project=cdiscourse-host`. Service remains unreachable (`--no-allow-unauthenticated` still gates). |

## Edge cases

- **Empty tester YAML.** Helper exits 5 with stderr `testers list is empty`. Operator must add at least their own email before granting.
- **Operator email matches `<...>` placeholder shape.** Helper exits 4 — refuses to print commands while any tester still looks like a template placeholder.
- **Duplicate emails in YAML.** Helper deduplicates (case-insensitive). The output count differs from the YAML count; helper warns to stderr `note: N duplicate(s) removed`.
- **Mixed-case email (`Alice@Example.com` vs `alice@example.com`).** Helper lowercases for the IAM `--member=user:` flag — Google IAM is case-insensitive on the local-part for routing but IAM stores normalized lowercase.
- **Tester removed from YAML but still has IAM binding.** The bulk-removal mode of `print-iap-tester-removals.mjs` reads the YAML and emits revokes for every email there — operator must use `--removed-email=<email>` for the one-off offboarding case where the email was already deleted from the YAML.
- **`verify-iap.mjs` against a URL that returns 200 anonymously.** This means IAP is NOT attached. Helper exits 1 with `verdict: FAIL (anonymous HEAD returned 200; IAP not enforcing)`.
- **`verify-iap.mjs` against a URL where the cert chain isn't Google Trust Services.** The cert chain is HOST-006's responsibility; the helper does NOT check the cert chain (would duplicate H1 from `deployment-smoke-checklist.md`). Helper only checks status code + Location header.
- **Operator runs `gcloud iap web enable` while IAP is already enabled.** Idempotent. The runbook notes this.
- **Operator runs `gcloud iap web disable` while testers still have IAM bindings.** Bindings remain; they just become inert until IAP is re-enabled. Runbook notes this — bindings are not auto-revoked.
- **YAML contains `service: cdiscourse-prod` accidentally.** Schema rejects (environment must be `dev`). Helper exits 2.
- **gcloud not on PATH for `verify-iap.mjs`.** Helper exits 3 with `gcloud not found on PATH; cannot fetch identity token`. The print-* helpers do NOT depend on gcloud being installed (they don't invoke it).
- **`GOOGLE_IAP_CLIENT_ID` env var set to an empty string.** Helper exits 9 (same as unset).
- **`--no-auth-check` AND `--no-anon-check` both set.** Helper exits 2 — at least one check must run.
