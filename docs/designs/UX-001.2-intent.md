# UX-001.2 Design Intent Brief — Timeline-First Viewport Repair and Room-Active Chrome Consolidation

**Card:** UX-001.2 — Timeline-first viewport repair and room-active chrome consolidation
**Epic:** UX-001 (Brand-Forward Simplified UX/UI Consolidation)
**Phase:** 2 of 7
**Priority:** P0 / Urgent
**Effort:** L (revised upward from M after pre-launch review surfaced the chrome floor)
**Filed:** 2026-05-25 (revised)
**Author:** Operator (product intent + chrome-nullification directive)
**Status:** Binding for UX-001.2 designer phase

## Governing sentence

UX-001.2 supersedes prior aesthetic and layout decisions that conflict with the Timeline-first viewport contract. Prior cards remain historically informative, but their visual placement decisions are not protected APIs. Functional behavior, state, permissions, route safety, composer submission, keyboard navigation, branch topology, doctrine constraints, and backend boundaries remain protected. Layout chrome, panel placement, visual hierarchy, duplicate controls, persistent top-of-board surfaces, and aesthetic spacing are explicitly mutable when they block the Timeline from rendering as the primary in-room surface.

## Why this brief was revised

The original UX-001.2 brief specified a four-file scope (ArgumentGameSurface, ArgumentTimelineMap, TimelineSelectedReadoutPanel, ArgumentReplySidecar) and a 200px hard cap for the Timeline first-row offset at the 1366x768 wide-band viewport. A pre-launch review against the current codebase proved the offset cap is mathematically unachievable within the declared file scope because approximately 170-250px of immutable chrome lives outside the four files: the MainAppShell tabBar in App.tsx (~48px), the DebateDetailHeader (~70-90px), the MainAppShell roomToolbar in App.tsx (~45px), and the ArgumentScoreTracker (~114px). The arithmetic floor across the required viewports exceeded every brief hard cap before any in-scope element rendered.

The pre-launch review surfaced three resolution options: expand the brief's file scope (Option 1), re-scope as a bounded card with relaxed targets and a follow-up card (Option 2), or relax the offset caps to acknowledge the floor (Option 3). The operator chose Option 1 with an explicit override directive that nullifies prior aesthetic and layout decisions where they conflict with the Timeline-first contract. The original 200px hard cap is preserved; the file scope expands to include App.tsx, DebateDetailHeader.tsx, and ArgumentScoreTracker.tsx as authorized modification targets.

This brief is the revised product specification. The designer reads this brief in full as the binding input.

## The hierarchy of authority

UX-001.2 documents this hierarchy explicitly and the designer enforces it:

1. Product doctrine and safety constraints remain highest. No verdict copy, no truth labels, no backend scope expansion.
2. UX-001 executive intent governs visible layout hierarchy.
3. Timeline-first viewport contract governs room-active layout.
4. Existing functional behavior is preserved.
5. Prior visual and aesthetic placements are historical references, not binding constraints.
6. Legacy spacing, panel placement, duplicate chrome, and top-of-board controls may be nullified.

This hierarchy prevents the implementer or reviewer from treating old aesthetic placements as protected APIs.

## What is superseded (the prior aesthetic blockers being nullified)

UX-001.2 explicitly supersedes the following prior aesthetic and layout decisions where they create vertical chrome above the Timeline in room-active view:

- Always-visible global tab bar (Arguments / Account / Admin) inside active room view
- Large DebateDetailHeader above the argument board
- Separate roomToolbar row above the board
- Duplicate Timeline/Cards toggles in multiple vertical layers
- Large ArgumentScoreTracker above the Timeline
- Full TimelineSelectedReadoutPanel above the Timeline
- Full ArgumentReplySidecar content above the Timeline (when used as the readout body)
- Large ArgumentGameSurface.header above the Timeline
- Large ArgumentTimelineMap.controlsRow above the rail
- Internal rail offset of approximately 120px that pushes the first node down
- Historical padding, hairlines, and spacers that exist only because of older visual decisions

These are aesthetic and layout decisions. UX-001.2 may collapse, remove, relocate, merge, overlay, hide, or move them below or aside so the Timeline becomes the first substantive in-room object beneath the UX-001.1 header and one compact room/context strip.

The supersession does not authorize functionality removal. Any displaced feature must remain available through a compact control, lower panel, side panel, overflow affordance, existing navigation control, or later Act/Inspect/Go integration. Behavior is preserved; placement is mutable.

- Authentication flow

UX-001.2 is aggressive about layout hierarchy and conservative about functional behavior.

## The viewport matrix (hard blockers and stress checks)

UX-001.2's acceptance is verified across a specific viewport matrix. Four viewports are hard blockers; failure on any of them fails the card. Four additional viewports are stress checks.

Hard blockers:
- 390 x 844 (iPhone-class phone)
- 1024 x 1366 (iPad portrait)
- 1366 x 768 (laptop browser — the critical web case)
- 1920 x 1080 (desktop browser)

Stress checks:
- 360 x 800 (small Android phone)
- 412 x 892 (large Android phone)
- 768 x 1024 (narrow tablet / narrow browser)
- 2560 x 1440 (wide desktop)

## The Timeline first-substantive-row offset contract

The offset is measured from the top edge of the viewport to the first substantive Timeline artifact: the rail, the first node row, or the equivalent board element. The cap is NOT satisfied by an empty Timeline wrapper appearing within the cap while the first node renders below.

Per breakpoint:

| Breakpoint | Header | Compact strip | Internal rail top | Preferred first row | Hard cap |
|---|---|---|---|---|---|
| Phone (0-599dp) | 64 | 48 | 0-12 | 112 | 128 |
| Tablet (600-1279dp) | 96 | 56 | 0-12 | 152 | 168 |
| Wide (1280dp+) | 120 | 64 | 0-16 | 184 | 200 |

The 200px wide cap is binding at 1366 x 768. The Timeline must be usable at 100% browser zoom at this viewport. No relaxation to 220px or any other value.

The hard caps apply to the first substantive Timeline artifact. If the rail container appears at 184px but the first node appears at 304px because of a 120px internal aesthetic offset, the card fails.

## The compact room/context strip (replaces DebateDetailHeader + roomToolbar)

The room/context strip is the only in-room object allowed to sit above the Timeline by default.

Maximum heights:
- Phone: 48px
- Tablet: 56px
- Wide: 64px

The strip is one row unless a true accessibility need requires otherwise. Multi-row strip is disallowed by default.

Allowed content in the strip:
- Room or debate title excerpt (truncated if necessary; the Timeline gets priority)
- Timeline/Cards state indicator or compact toggle (one toggle, not duplicated)
- A compact room-exit / back / "Arguments" control providing escape path
- A single tiny status chip if required
- One Go, Act, or Inspect trigger if the design requires it

Disallowed content in the strip:
- Full selected-message body
- Full sidecar detail
- Resolution preview (moves to expanded details or Inspect)
- Participant badge cluster (reduces to compact chips or moves to details)
- Full score tracker
- Full action dock
- Composer fields
- Long explanatory text
- Multiple rows of controls
- Duplicate Timeline/Cards toggles

If the room title is long, truncate. The Timeline gets priority.

The strip subsumes:
1. DebateDetailHeader's content (collapsed)
2. roomToolbar's Timeline/Cards toggle (consolidated)
3. Room-exit affordance (so the user can leave the room with the global tab bar hidden)

## The MainAppShell tabBar room-active behavior

The global tab bar (Arguments / Account / Admin) is hidden when a room is active.

Required behavior:
- When no room is active, the tab bar renders normally.
- When a room is active, the tab bar is hidden.
- When the user exits the room or returns to the Arguments list, the tab bar restores.
- Account remains reachable outside active room mode.
- Admin remains reachable outside active room mode for eligible users.
- Direct room or deep-link entry provides an escape path back to Arguments via the compact strip's room-exit control.

A compact tab bar (32-36px) is NOT acceptable. The full Timeline-first contract requires the tab bar to be absent in room-active view; a compact version preserves the old aesthetic hierarchy.

## The MainAppShell roomToolbar dissolution

The roomToolbar's separate ~45px row is dissolved entirely. Its contents are decomposed:

- Timeline/Cards toggle: moves into the compact room/context strip
- Invite button: moves into compact overflow, into Go (since "invite users" is a navigation action), or into a secondary detail action
- Dev chips: remain dev-only; must not affect production layout
- Duplicate Timeline/Cards toggle (currently present in both roomToolbar and ArgumentGameSurface.header): consolidated to one toggle in the compact strip
- Other nonessential controls: move out of default top chrome

The final room-active top stack is:
1. AppHeader (UX-001.1 height: 64 / 96 / 120)
2. Compact room/context strip (48 / 56 / 64)
3. Timeline board

Not three rows. Not four rows. Two rows of chrome plus the Timeline.

## The ArgumentGameSurface.header treatment

ArgumentGameSurface.header currently contributes ~83-105px of pre-Timeline chrome. UX-001.2 treats this as mutable.

Disposition options:
- Remove if duplicative with the compact room strip
- Merge essential controls into the compact strip
- Collapse to a thin band (e.g., 32px) with only board-specific controls
- Relocate controls to a compact overlay or lower context area

The designer chooses with stated rationale. The header must NOT remain as a separate large pre-Timeline block.

## The ArgumentScoreTracker repositioning

ArgumentScoreTracker (~114px) is preserved as functionality but cannot consume top-of-board space before the Timeline appears.

Preferred default disposition: move below the Timeline, with a compact summary state if feasible.

Alternative acceptable disposition: place side-adjacent on wide screens if it does not delay the Timeline's first row.

Future disposition (UX-001.4 or UX-001.5 scope): move score and semantic detail into Inspect or attach as object-bound metadata chips.

UX-001.2 does NOT remove the tracker's functionality. UX-001.2 does NOT allow the tracker to remain as a large pre-Timeline panel.

## The selected-message readout repositioning (IX-004 supersession)

IX-004 promoted the selected-message readout above the Timeline. That decision was correct at the time because the immediate problem was selected-node ambiguity. UX-001 now establishes a stronger executive priority: the Timeline is the primary board and must not be buried. UX-001.2 supersedes IX-004's default above-map placement.

Preserved from IX-004:
- The readout model
- The selected state
- The activeMessageId single source of truth
- The accessibility work
- The context content
- The semantics

Superseded by UX-001.2:
- The default placement of the full readout panel above the Timeline

The reviewer's read-only API verification for UX-001.2 does NOT treat IX-004's above-map placement as a protected API. The protected API is the selected context model and its accessibility behavior; the vertical placement is mutable.

Default placements per breakpoint:

| Breakpoint | Placement | Default height |
|---|---|---|
| Phone | Below Timeline or attached to composer | 64-72px |
| Tablet | Below Timeline or side-adjacent if cleanly supported | 72-80px |
| Wide | Right-side or lower-adjacent context panel | 80-96px |

Default compact readout shows only:
- Selected node type or move type
- A short body excerpt
- A parent or reply-target hint
- Direct reply count or branch hint where applicable
- A small "Acting on" label if the action dock or composer is bound to that node

The default compact readout does NOT show:
- The full six-section detail panel
- Long-form node body content
- Full evidence chains or source detail
- Comprehensive metadata or classifier output

Detailed content moves to:
- The Inspect popout (UX-001.4 scope, not delivered in UX-001.2)
- An expanded readout state triggered by explicit user action
- A side panel on wide screens that the user can expand on demand

The expanded readout may consume up to 30% of viewport height but appears only after explicit user action.

## The internal rail offset nullification

ArgumentTimelineMap currently has an internal top offset of approximately 120px before the first node renders. This is legacy aesthetic spacing.

UX-001.2 reduces this to a tokenized minimal value:
- Phone internal top padding: 0-12px
- Tablet internal top padding: 0-12px
- Wide internal top padding: 0-16px

This is necessary for the cap to mean something. The acceptance criterion measures the first substantive Timeline row, not the empty container top.

## The scroll containment contract

On web (laptop and desktop browser), the room surface behaves like a viewport-managed application area. The Timeline board has its own scroll or pan behavior. The body and page do not become a long document where the Timeline is buried below panels.

On mobile, vertical space is limited and the board may share space with lower docks. The first visible in-room surface after the header and compact strip must still be the Timeline.

The failure mode the design must prevent:
1. Header
2. Tab bar
3. Room header (full)
4. Room toolbar
5. Argument surface header
6. Score tracker
7. Selected readout (full)
8. Sidecar
9. Composer
10. Timeline (buried below all of the above)

The target order:
1. AppHeader (UX-001.1 height)
2. Compact room/context strip (within the strip's height cap)
3. Timeline board (the first substantive board object, with internal rail offset minimized)
4. Compact selected-context or readout or action context (below or beside the Timeline)
5. Composer or lower dock
6. Expanded Inspect, Act, or Go only when requested by explicit user action

On wide screens, the target order may use horizontal layout:
1. AppHeader
2. Compact room/context strip
3. Timeline board plus side context panel in the same horizontal row
4. Composer or lower dock

The Timeline still starts at the top of the in-room area in the wide layout.

## The reference aesthetic

The intended Timeline aesthetic draws from specific aspects of three reference products, used only as directional anchors.

From Linear: the level of restraint and component discipline. Every visible element has earned its place. Dark surfaces, professional density, no decorative chrome.

From Figma's inspector and canvas patterns: the principle that the canvas stays primary and details live in side or lower panels. The inspector never displaces the canvas by default.

From GitHub's commit graph and branch graph: branch legibility, parent-child structure, visual continuity through branching history. The Timeline communicates structure, not chronology.

Explicitly NOT references for UX-001.2: Notion's and Are.na's whitespace-heavy block layouts.

The intended feel: dark, minimal, precise, board-first, keyboard-capable, touch-friendly, contextual without clutter, professional enough for executive review, readable without browser zoom. The Timeline should feel closer to a game board, DAW track map, argument graph, or branchable operating surface than a comment section.

## The Timeline topology (read-only constraint)

The Timeline is an adaptive branching argument map. UX-001.2 preserves the existing model:

- A branching Timeline board with a clear root
- A visible active path
- Parent-child connectors
- Collapsed branch stubs where necessary
- Branch and tangent support without a separate route
- Selected-node state that survives Timeline-Cards switching
- Dense but legible node rendering
- Board-level navigation that scales to long rooms

UX-001.2 preserves the existing branch and junction-ready model. UX-001.2 does not change the underlying topology. UX-001.2 is about viewport hierarchy and board placement, not rewriting the argument graph.

## The credibility framing

UX-001.2 is the visible credibility repair for the whole UX-001 epic. UX-001.1 made the brand shell professional, but UX-001.2 determines whether the product's primary interaction surface actually feels usable. The Timeline is not secondary documentation, not an afterthought under a readout, and not a panel the user should have to zoom out to find. It is the argument board. This card must make the Timeline the first substantive in-room object beneath the UX-001.1 header and one compact room/context strip, while preserving selected-node context in a compact, relocated, or on-demand form. If the Timeline remains visually buried after UX-001.2, the card has failed even if the code is technically correct. The expanded file scope and the explicit chrome nullification are the minimum required to deliver this; this is not scope creep, it is the minimum scope required to make the original product requirement achievable.
