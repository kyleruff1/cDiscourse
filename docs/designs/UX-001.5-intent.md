## Why this brief exists

UX-001.1 made the brand shell professional. UX-001.2 made the Timeline the primary board. UX-001.3 made the composer a persistent contextual command center. UX-001.4 consolidated Act/Inspect/Go into one three-menu interaction model and established the Inspect surface contract — including the `flags` section that will house annotation chips.

Today, metadata, lifecycle state, classifier outputs, semantic referee flags, evidence annotations, and user tags render through ad-hoc visual treatments scattered across the codebase. `EvidenceAnnotationChip` has one pattern; `moveMetadataLedger` produces another shape; the semantic referee's banner uses a third visual treatment; the existing inspect popup model has a flat `flags` section that UX-001.4 leaves stable for UX-001.5 to evolve.

UX-001.5 creates the visual annotation primitives — rings, badges, chips, outlines, chip strips, overflow chips, edge highlights, accessible chip labels — that consolidate all annotation rendering into one coherent visual language. The card has standalone product value: the current labels look professional and consistent. UX-001.5A then layers the Observations vs Allegations distinction on top using these primitives.

The user sees the board with annotations that feel like one system: same chip shape, same badge density, same focus ring style, same overflow pattern. Nothing dominates the board; everything is compact and attached to its relevant object.

## The central product rule

The visual annotation primitives are one design language with these properties:

- **Compact**: every primitive fits the brief's compact-annotation requirement. No primitive renders larger than necessary to be readable and tappable.
- **Object-attached**: every primitive renders attached to its relevant object (the node, the evidence item, the section, the composer field). Nothing floats freely or detaches from the thing it annotates.
- **Source-agnostic at the primitive layer**: a chip is a chip regardless of whether it renders a machine observation or a user allegation. UX-001.5A adds the source-aware presentation; UX-001.5's primitives are source-neutral.
- **Verdict-clean**: no primitive's default rendering implies truth, correctness, winning, losing, or proof. The primitives' visual language reads as "this is information about this object" not "this is a judgment of this object."
- **Accessible**: every interactive primitive meets 44x44 tap targets. Every primitive supports screen-reader announcement with plain-language labels. No primitive relies on color alone to communicate state.

The primitives feel like one system across the timeline body (badges + outlines on nodes), the composer (existing validation chips), Inspect's `flags` section (chip strips with overflow), the side action rail (where annotations may surface adjacent to actions), and any other surface that today renders annotations ad-hoc.

## The canonical primitive set

UX-001.5 ships these primitives. The designer may rename for clarity but must NOT add or remove primitives without operator decision.

**Chips** — compact horizontal pill containers for labels.
- `AnnotationChip` (base) — single label + optional icon + optional dismiss
- `AnnotationChipStrip` — horizontal scrollable container for multiple chips
- `AnnotationOverflowChip` — `+N` indicator when more chips exist than the strip displays

**Badges** — small status indicators attached to a parent object.
- `AnnotationBadge` — small dot or pill that signals state without consuming label space
- `AnnotationBadgeCluster` — multiple badges grouped on one anchor

**Rings and outlines** — visual differentiation for state.
- `AnnotationFocusRing` — focus state ring (keyboard or selection)
- `AnnotationOutline` — selected / active / dimmed outline (state, not focus)
- `AnnotationEdgeHighlight` — parent-child relationship emphasis (used on branch edges)

**Inspect grouping surfaces** — sections within Inspect that group chips by kind / source / category.
- `InspectSectionChipStrip` — chip strip inside an Inspect section
- `InspectGroupHeader` — small group divider with optional count

**Accessibility infrastructure** — screen reader + focus management.
- `AnnotationAriaLabel` — pure-TS label builder for screen-reader announcements
- `AnnotationFocusBoundary` — focus management for chip strips (arrow-key navigation between chips)

## The chip descriptor contract

The primitives consume a pure-TS descriptor shape. The descriptor is source-neutral; UX-001.5A adds source-aware fields.

```typescript
type AnnotationChipDescriptor = {
  id: string;                    // stable key
  label: string;                 // plain-language; no raw internal codes
  kind?: 'state' | 'context' | 'lifecycle' | 'evidence' | 'flag' | 'semantic';
  iconHint?: 'info' | 'warn' | 'check' | 'time' | 'evidence' | 'flag' | 'cluster';
  // Color: chip uses a token-derived color from kind/iconHint, NOT direct hex.
  // No verdict colors. No green=correct / red=wrong patterns.
  tooltip?: string;              // plain-language explanation; no internal codes
  ariaLabel?: string;            // explicit screen-reader override
};
```

UX-001.5A extends with `source: 'machine' | 'user'` and `category` fields; UX-001.5's primitives MUST accept these fields without rendering changes (forward-compatible).

## The Inspect `flags` section integration

The current `inspectPopoutModel.ts` `flags` section renders a flat list of strings. UX-001.5 changes the shape:

- `flags: string[]` becomes `flags: AnnotationChipDescriptor[]`
- `InspectPopout` renders the `flags` section using `InspectSectionChipStrip`
- The strip displays up to N chips inline (designer chooses N per breakpoint); excess chips become `AnnotationOverflowChip` ("+N")
- Tapping the overflow chip expands to show all chips
- Each chip is keyboard-focusable with `AnnotationFocusBoundary` for arrow-key navigation
- Plain-language labels routed through `gameCopy.toPlainLanguage` for any internal code

The section interface remains stable for UX-001.5A (add `source` and `category` fields to the descriptor; UX-001.5A's renderer reads these to categorize chips into Observations vs Allegations groups).

## Color and visual treatment

The primitives use surface tokens from `src/lib/designTokens.ts` (read-only). The chip's background, border, and text color derive from `kind` and `iconHint` mapped through a token table. No direct hex literals in primitive code.

Verdict-clean color rules:
- No green = correct / red = wrong
- No traffic-light coloring
- Neutral information colors (slate / cobalt / amber for distinct categories) instead of evaluative colors
- Color is informative but never carries the only meaning; shape/icon/label also distinguish kinds

Color independence (per accessibility): every chip's meaning is communicable without color. Screen readers receive plain-language labels; keyboard users see focus rings; color-blind users see distinct shapes/icons.

## Cross-viewport rendering

The primitives respect breakpoint sizing:
- **Phone (390x844)**: chip strips show 2-3 chips before overflow; chip height ≤ 28px; badge diameter ≤ 8px
- **Tablet (1024x1366)**: chip strips show 3-4 chips before overflow; chip height ≤ 32px; badge diameter ≤ 10px
- **Wide (1366x768, 1920x1080)**: chip strips show 4-6 chips before overflow; chip height ≤ 32px; badge diameter ≤ 10px

The primitives MUST NOT push the Timeline above UX-001.2's offset caps. If a badge mount on a timeline node would shift the rail's vertical position, the badge sits as an overlay (`pointerEvents: 'none'`) rather than displacing layout.

## Keyboard and shortcut contract

UX-001.5 does NOT introduce new keyboard shortcuts. The primitives respect the existing shortcut allocations:
- Cmd/Ctrl+Enter, Cmd/Ctrl+K — composer (UX-001.3)
- A/I/G — board menus (UX-001.4)
- Arrows — node navigation (existing)

Within a chip strip, arrow keys move focus between chips (per `AnnotationFocusBoundary`). Enter on a focused chip activates its tooltip / handler. Esc removes focus from the chip strip.

## Touch ergonomics

Every interactive chip meets 44x44 (via `hitSlop` if visual size is smaller). Tap a chip to surface its tooltip; long-press for fuller context (if the primitive supports it). Bottom-sheet expanded chip lists (when overflow is large) follow UX-001.4's dismissible-scrim pattern.

The audit produces a table with `Audit item | Current state (cited) | Brief contract | Gap or alignment | Disposition`.

Disposition values: `consolidate-into-primitive` (the existing surface uses the new primitive), `mark-for-follow-up` (the existing surface is left alone with a follow-up card filed), `retain-with-rationale` (the existing surface is intentionally different — designer documents why).
