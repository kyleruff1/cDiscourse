## §1 — Goal & scope

QOL-039 defines and enforces **room visibility** — whether an argument room appears on public lists and who may read it — and the **one-way public → private transition** the room creator may perform. Two visibility states:

- **`public`** — the room appears on the Conversation Gallery and is readable by any authenticated user. This is today's behavior for every `open`/`locked` debate; QOL-039 makes that behavior an *explicit, stored* property rather than an implicit one.
- **`private`** — the room never appears on any public list and is readable **only** by its participants (and moderators/admins). A room may be **created** private (band-rent Scenario 2, Step 1) or **transitioned** from public to private by its creator (roommates Scenario 1, Step 15).

In scope:

1. A new `visibility` column on `public.debates` with a safe default and a `CHECK` constraint.
2. RLS policy replacements on `debates`, `debate_participants`, `arguments`, and `argument_tags` so a private room is invisible to non-participants — without disabling RLS and without breaking the existing `debates ↔ debate_participants` non-recursion contract.
3. The **transition flow**: a creator-only, one-way `public → private` action, its preconditions, and its deterministic consequences.
4. A pure-TS model that decides whether a given caller may transition a room, and computes the **transition consequences summary** (what changes, in neutral language).
5. The **client API wrapper** for the transition (a single `UPDATE` under RLS — no Edge Function needed; see §4.4).
6. The **transition-event payload** QOL-040 consumes for notifications, plus the redaction contract that payload must honor.
7. UI states: the visibility control at room creation, the `make private` action on a live public room, the neutral confirmation, and the private-room read-time chrome.

Out of scope — see §11.

The doctrine anchor, inherited from GAME-005: **visibility is a structural access property of the room, never a verdict on the room or the people in it. Making a room private is a structural transition, never a punishment, never shaming. No content is hard-deleted, soft-deleted, or mutated by the transition.**

## §3 — Problem statement

Today every `open` or `locked` debate is **globally readable** by any authenticated user — the `arguments` SELECT policy grants read on any posted argument in an `open`/`locked` debate, and `listDebates` (`src/features/debates/debatesApi.ts`) returns *every* debate row with no visibility filter. There is no way to:

1. **Create a room that is not publicly listed.** Band-rent Scenario 2 requires a private-from-creation room; the schema cannot express it.
2. **Take a public room private.** Roommates Scenario 1 Step 15 requires a creator to revoke non-participant read access after the fact; nothing supports it.
3. **Do either without leaking content.** A naive "hide from the list" that left the `arguments` SELECT policy untouched would still let any user read every argument by `debate_id` — RLS, not the list query, is the real boundary.

Three doctrine traps a careless design would fall into, and how QOL-039 avoids each:

- **The shaming trap.** Making a room private *after* a chime-in was rejected could read as "we kicked you out and locked the door." QOL-039 forbids that: the transition copy is structural and neutral, the rejected chime-in branch is **retained in full** (never deleted), and the only person-relative notification (chime-in-marked-unwanted) is QOL-040's, phrased per QOL-040's neutral-copy contract — QOL-039 never authors shaming copy and never deletes the branch.
- **The recursion trap.** Migration `20260516000006` documents a real production incident: a `debates` ⇄ `debate_participants` RLS recursion took down every `GET /rest/v1/debates`. Any new visibility predicate that joins back across those tables re-introduces it. QOL-039 reuses the existing `SECURITY DEFINER` helpers and adds **at most one** new same-pattern helper (§4.3) — never a raw cross-table subquery in a policy.
- **The hard-delete trap.** "Rejected chime-in branches are retained" is non-negotiable: the transition mutates **zero `arguments` rows**. It changes only one column on one `debates` row.

## §4 — Data model (the visibility field + RLS) & file changes

### 4.1 The `visibility` column

One new migration, `supabase/migrations/20260521000001_qol039_room_visibility.sql` (timestamp must sort after the latest existing migration `20260517000009_meta_1a_point_tags.sql` — confirm at write time). Append-only; never edits an applied file.

```sql
-- Add the visibility column. Default 'public' so every existing row keeps
-- today's behavior (open/locked debates stay globally readable). NOT NULL
-- with a default backfills every existing row in one statement.
ALTER TABLE public.debates
  ADD COLUMN visibility text NOT NULL DEFAULT 'public'
    CHECK (visibility IN ('public', 'private'));

COMMENT ON COLUMN public.debates.visibility IS
  'public = listed and globally readable; private = readable only by '
  'participants and mods/admins. One-way: public->private only, by the '
  'creator. Never affects argument content; only read access + listing.';
```

Why a `text` + `CHECK` and not a Postgres `enum`: the codebase already uses `text + CHECK` for `debates.status` and `debate_participants.side`. Consistency beats a marginally tighter type, and a `CHECK` is editable by a later append-only migration without an `ALTER TYPE` dance.

Why default `public`: backward compatibility. Every existing `open`/`locked` debate must keep being readable and listed after the migration; flipping the default to `private` would silently hide live rooms. New private-from-creation rooms set `visibility = 'private'` explicitly at `INSERT`.

**No second column for "transitioned vs created private."** The card needs only the current state. If audit history is later wanted, that is a separate append-only `room_visibility_changes` audit table — explicitly out of scope (§11), and §5.4 already defines the event shape it would use.

### 4.2 The one-way invariant — enforced in the DB, not just the app

`public → private` is the only legal transition; `private → public` is forbidden (re-exposing a room people argued in believing it was private is a privacy violation). The app's UPDATE policy and the API wrapper both forbid it, but a `BEFORE UPDATE` trigger makes it impossible even for a hand-crafted REST call by the creator:

```sql
CREATE OR REPLACE FUNCTION public.enforce_room_visibility_one_way()
RETURNS trigger
LANGUAGE plpgsql
AS $$
BEGIN
  IF OLD.visibility = 'private' AND NEW.visibility = 'public' THEN
    RAISE EXCEPTION 'room_visibility_is_one_way: a private room cannot be made public again';
  END IF;
  RETURN NEW;
END;
$$;

CREATE TRIGGER debates_enforce_visibility_one_way
  BEFORE UPDATE OF visibility ON public.debates
  FOR EACH ROW EXECUTE FUNCTION public.enforce_room_visibility_one_way();
```

This is structural, not stylistic: the one-way rule is a privacy guarantee, so it belongs at the lowest enforceable layer. Mods/admins are **not** exempted — re-publicizing a private room is never appropriate, even for an admin. (An admin who must do so for a legal/abuse reason performs it as a service-role migration with an audit trail — outside this card.)

### 4.3 RLS policy replacements

RLS is **never disabled**. Four SELECT policies are dropped and recreated so a private room is invisible to non-participants. Each replacement reuses the existing `SECURITY DEFINER` helpers (`is_debate_participant`, `is_moderator_or_admin`) to stay non-recursive — see the §3 recursion trap.

**4.3.1 — one new helper.** The `arguments` and `argument_tags` policies need to know "is this argument's debate private?" without a raw subquery into `debates` (which would recurse for an `arguments` policy that some `debates` policy transitively reads). Add a narrow `SECURITY DEFINER` helper, modeled exactly on `is_debate_open_or_locked` from migration `20260516000006`:

```sql
CREATE OR REPLACE FUNCTION public.is_debate_private(p_debate_id uuid)
RETURNS boolean
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.debates d
    WHERE d.id = p_debate_id AND d.visibility = 'private'
  );
$$;

REVOKE ALL ON FUNCTION public.is_debate_private(uuid) FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.is_debate_private(uuid) TO authenticated;
```

**4.3.2 — `debates` SELECT.** Today: `status IN ('open','locked') OR created_by = uid OR mod/admin OR is_debate_participant(id, uid)`. New: a private room must drop the unconditional `open/locked` arm.

```sql
DROP POLICY IF EXISTS "debates: select open, own, or participant" ON public.debates;

CREATE POLICY "debates: select public-open, own, or participant"
ON public.debates
FOR SELECT
TO authenticated
USING (
  created_by = auth.uid()
  OR is_moderator_or_admin()
  OR public.is_debate_participant(id, auth.uid())
  OR (visibility = 'public' AND status IN ('open', 'locked'))
);
```

`visibility` is a column **on the row being filtered**, so this adds no cross-table join — no recursion risk.

**4.3.3 — `debate_participants` SELECT.** Today the third arm is `is_debate_open_or_locked(debate_id)` — meaning anyone can enumerate the participant list of any open/locked debate. For a private room, that arm must not apply. Replace it with a private-aware helper:

```sql
DROP POLICY IF EXISTS "debate_participants: select own or open debate" ON public.debate_participants;

CREATE POLICY "debate_participants: select own, participant, or public-open debate"
ON public.debate_participants
FOR SELECT
TO authenticated
USING (
  user_id = auth.uid()
  OR is_moderator_or_admin()
  OR public.is_debate_participant(debate_id, auth.uid())
  OR public.is_debate_open_or_locked_public(debate_id)
);
```

`is_debate_open_or_locked_public` is `is_debate_open_or_locked` plus an `AND d.visibility = 'public'` term — add it as a second new `SECURITY DEFINER` helper alongside §4.3.1, same pattern. (Alternatively, modify the existing helper — but helpers are referenced by other policies, so adding a sibling is the append-only-safe choice.) The added `is_debate_participant(debate_id, auth.uid())` arm ensures a participant can still see *co-participants* of a private room — without it, a private-room participant could not load the seat strip.

**4.3.4 — `arguments` SELECT.** Today: `author_id = uid OR mod/admin OR (status='posted' AND debate is open/locked)`. The third arm leaks every argument of a private room. New:

```sql
DROP POLICY IF EXISTS "arguments: select posted in readable debates or own" ON public.arguments;

CREATE POLICY "arguments: select own, participant-private, or posted-public"
ON public.arguments
FOR SELECT
TO authenticated
USING (
  author_id = auth.uid()
  OR is_moderator_or_admin()
  OR (
    status = 'posted'
    AND (
      -- public room: any authenticated user
      (public.is_debate_open_or_locked_public(debate_id))
      -- private room: participants only
      OR public.is_debate_participant(debate_id, auth.uid())
    )
  )
);
```

This is the policy that makes "retained (muted), never deleted" and "non-participant read access revoked" simultaneously true: the rejected chime-in author's own row is still readable to **them** (`author_id = auth.uid()` arm), still readable to **participants** (the participant arm), and no longer readable to **other non-participants** once the room is private. Nothing is deleted; only the audience shrinks.

**4.3.5 — `argument_tags` SELECT.** `docs/rls.md` states `argument_tags` SELECT "mirrors argument visibility." If its policy contains a duplicated `debates`-status subquery, replace it to call `is_debate_open_or_locked_public` / `is_debate_participant` the same way — so a private room's tags are not a side-channel leak. The implementer must read the live `argument_tags` SELECT policy and apply the identical pattern; if it already delegates through an `arguments` `EXISTS`, it inherits §4.3.4 automatically and needs no change. **The design instruction: tags must not be more visible than the arguments they tag.**

**4.3.6 — what does NOT change.** `topic_satisfaction_checks`, `argument_flags`, `audio_submissions` SELECT policies already gate on `is_debate_participant` / own-row / mod-admin and do **not** have an unconditional public-open arm — confirm against the live policies, but per `docs/rls.md` they are participant-scoped already. INSERT/UPDATE/DELETE policies on all tables are **unchanged** — visibility affects read access and listing only, never the write path.

### 4.4 Why a client `UPDATE`, not an Edge Function

The `debates` UPDATE policy already permits `creator OR mod/admin`. The transition is a single-column update (`visibility = 'private'`) on a row the caller owns, with the one-way invariant enforced by the §4.2 trigger. There is **no privileged write, no service-role need, no cross-user mutation** — so the supabase-edge-contract rule "move privileged writes to an Edge Function" does not apply. The transition is a plain RLS-gated `UPDATE` from the client wrapper (§4.5). The notification fan-out (QOL-040) may later be an Edge Function or a DB trigger; that is QOL-040's call, not QOL-039's.

`submit-argument` remains the only write path into `public.arguments` — QOL-039 never inserts or updates an `arguments` row.

## §5 — The transition flow & its consequences

### 5.1 Preconditions (`canTransitionToPrivate`)

A pure function in `roomVisibilityModel.ts`:

```ts
export interface VisibilityTransitionContext {
  roomId: string;
  currentVisibility: RoomVisibility;   // 'public' | 'private'
  roomStatus: 'draft' | 'open' | 'locked' | 'archived';
  callerUserId: string;
  createdByUserId: string;
  callerIsModeratorOrAdmin: boolean;
}

export interface TransitionEligibility {
  allowed: boolean;
  /** Stable internal reason code; routed through gameCopy.toPlainLanguage
   *  for any user-facing surface. Never shown raw. */
  reason:
    | 'eligible'
    | 'already_private'
    | 'not_room_creator'
    | 'room_archived';
}

export function canTransitionToPrivate(
  ctx: VisibilityTransitionContext,
): TransitionEligibility;
```

Rules:

- **`already_private`** — `currentVisibility === 'private'` → not allowed (no-op; the action is hidden, not errored — consistent with QOL-025 "no silent no-op": a private room simply does not show the `make private` entry).
- **`not_room_creator`** — only `createdByUserId === callerUserId` may transition. **A moderator/admin may also transition** (the `debates` UPDATE policy already permits it, and a mod taking an abusive public room private is legitimate). So: `allowed` requires `callerUserId === createdByUserId || callerIsModeratorOrAdmin`. The card says "the room creator" — mods are a documented superset, not a contradiction (same pattern as every other mod-or-owner gate in the repo).
- **`room_archived`** — an `archived` room is read-only history; transitioning its visibility is allowed in principle (it still changes who can read the history) but the v1 decision is to **allow** it for `open` / `locked` / `archived` and only block nothing on status grounds. The `room_archived` code is reserved in case the operator wants to forbid it — see Open Question OQ-3. Default v1: status never blocks the transition.
- One-way: there is **no** `canTransitionToPublic`. The function only ever evaluates `→ private`.

### 5.2 The transition action

1. The creator opens the `make private` entry (§7) on a `public` room.
2. The client calls `buildTransitionConsequences(ctx, seatMap)` (§5.3) and renders the **neutral confirmation** — a plain-language list of exactly what changes. The confirmation is **required**; the transition is irreversible, so it is never a one-tap action.
3. On confirm, `debatesApi.transitionRoomToPrivate(debateId, userId)` issues `UPDATE public.debates SET visibility='private' WHERE id=$1` under the caller's JWT. RLS (`debates` UPDATE: `creator OR mod/admin`) authorizes it; the §4.2 trigger guarantees one-way; the §4.1 `CHECK` guarantees the value is valid.
4. On success, the client emits a `RoomVisibilityChangeEvent` (§5.4) for QOL-040 to turn into notifications, refreshes the room (so the private badge appears and the gallery list drops it for non-participants), and shows a neutral inline confirmation toast.
5. On failure (RLS denial, network), surface a neutral, actionable error via `gameCopy` — never a raw Postgres string.

### 5.3 Consequences (`buildTransitionConsequences`)

The confirmation must tell the creator the truth, neutrally. A pure function returns the structured consequence list; the UI renders it as bullets — no prose concatenation, no AI.

```ts
export interface TransitionConsequences {
  /** Stable codes; each maps to a neutral ROOM_VISIBILITY_COPY bullet. */
  effects: ReadonlyArray<
    | 'leaves_public_list'        // room no longer on the Conversation Gallery
    | 'non_participants_lose_read'// people who aren't in the room can't read it
    | 'participants_keep_access'  // everyone currently in the room keeps access
    | 'content_unchanged'         // no argument is deleted, hidden, or edited
    | 'chime_in_branches_retained'// any muted chime-in branches stay, fully
    | 'one_way'                   // this cannot be undone
  >;
  /** Count of chime-in branches that are currently observer_only (muted)
   *  per GAME-005. Drives the 'chime_in_branches_retained' bullet wording
   *  ("the N side branch(es) stay in the record"). 0 omits that nuance but
   *  the 'content_unchanged' bullet still shows. */
  retainedChimeInBranchCount: number;
}

export function buildTransitionConsequences(
  ctx: VisibilityTransitionContext,
  seatMap: PublicRoomSeatMap | null,   // GAME-005; null when not a public-seat room
): TransitionConsequences;
```

The `effects` list is **fixed and always the same six codes** when the transition is eligible — the confirmation never varies its *meaning* by room; only `retainedChimeInBranchCount` varies the wording of one bullet. This keeps the copy auditable and prevents per-room shaming language from creeping in.

### 5.4 The transition event (`RoomVisibilityChangeEvent`) — QOL-040's input

QOL-039 does not send notifications; it **emits a structured event** QOL-040 consumes. Defining it here means QOL-040 has a stable contract and QOL-039 owns the redaction rules.

```ts
export interface RoomVisibilityChangeEvent {
  roomId: string;
  /** Always 'public' -> 'private' in v1 (one-way). */
  from: RoomVisibility;
  to: RoomVisibility;
  /** The user who performed the transition (creator or mod). Used by
   *  QOL-040 only to suppress self-notification — never rendered as
   *  "X made this private" person-attribution copy. */
  actorUserId: string;
  /** ISO timestamp the client recorded the successful UPDATE. */
  occurredAt: string;
  /**
   * The audiences QOL-040 must notify, already classified so QOL-040
   * never re-derives a leak-prone audience itself.
   */
  audiences: {
    /** Participants who keep access — get a neutral "room is now private". */
    retainedParticipantUserIds: ReadonlyArray<string>;
    /** Users who were observer_only via GAME-005 governance (rejected
     *  chime-in) — get the QOL-040 "made private" + the separate
     *  QOL-040 "chime-in marked unwanted" notice. QOL-039 only LISTS
     *  them; QOL-040 owns the neutral wording. */
    rejectedChimeInUserIds: ReadonlyArray<string>;
    /** Observers/overflow who keep read access only while public — get
     *  the neutral "this argument was made private" notice. */
    nonParticipantObserverUserIds: ReadonlyArray<string>;
  };
}
```

### 5.5 Redaction contract for QOL-040 (QOL-039 owns this rule)

QOL-040's copy must obey two QOL-039-defined invariants:

1. **No post-private content leak.** A notification delivered *because of* the transition must not embed any argument body, evidence detail, or participant identity that the recipient can no longer read after the room is private. The "this argument was made private" notice to a non-participant observer may carry the room **title** (a non-participant could already see the title when it was public; the title was already public) but **must not** carry any new argument text. QOL-039 surfaces only IDs in the event — never bodies — so QOL-040 cannot accidentally leak.
2. **No shaming.** The `rejectedChimeInUserIds` notice is the storyboard's "Your chime-in was marked unwanted by both primary participants." QOL-040 phrases it; QOL-039's contract: it describes the **move's fit to the room**, never the person; it never uses "rejected/booted/removed/unwanted person" framing about the human; it is consistent with GAME-005's `CHIME_IN_GOVERNANCE_COPY` non-punitive vocabulary. QOL-039's `roomVisibilityModel.test.ts` includes a ban-list scan over `ROOM_VISIBILITY_COPY`; QOL-040's tests extend it to its own copy.

## §6 — UI states

All copy lives in `ROOM_VISIBILITY_COPY` (`gameCopy.ts`); the strings below are intent, finalized in the copy block. Normal-user copy: "argument" / "room", never "game", never "debate" in user-facing strings.

### 6.1 Create-time visibility control

In the room-setup section (`CreateDebateForm.tsx` today, the `root_claim` box's room-setup section once QOL-030 lands — §7): a two-option segmented control.

- **Public** (default, selected) — helper: "Anyone can find and read this argument."
- **Private** — helper: "Only people you invite can find and read this argument."
- The control is a single `accessibilityRole="radiogroup"` of two 44px targets; the selected state is conveyed by more than color (label weight + a check mark) per `accessibility-targets`.
- Choosing Private at creation writes `visibility: 'private'` straight into `createDebate` — no transition, no confirmation needed (nothing is being revoked).

### 6.2 The `make private` action on a live public room

- Appears **only** when `canTransitionToPrivate` returns `allowed` — i.e. the room is currently `public` and the caller is creator or mod/admin. A private room shows no such control (no silent no-op; the action is simply absent — per the QOL-025 doctrine).
- Rendered as a `make private` **direct flash-menu entry** on the `room`-target Act popout (QOL-030 §6.3 names exactly this entry). Until QOL-030 lands, it is a labeled button in the room header for the creator (§9).
- Activating it opens the confirmation (§6.3); it never transitions on a single tap.

### 6.3 The neutral confirmation

A modal/sheet listing the `TransitionConsequences.effects` as plain bullets:

- "This argument leaves the public list — it won't show up for people browsing."
- "People who aren't already in this argument will no longer be able to read it."
- "Everyone currently in this argument keeps full access."
- "Nothing is deleted or hidden — every message stays exactly as it is."
- When `retainedChimeInBranchCount > 0`: "The {N} side branch{es} stay in the record, visible to the people in this argument." — **never** "the rejected/unwanted branch."
- "This can't be undone — a private argument stays private."

Two buttons: **Make private** (primary) and **Cancel** (secondary). No scare color, no warning iconography that implies wrongdoing — this is a routine setting change. The confirmation is the *only* place the irreversibility is stated before the action; it must be unmissable but calm.

### 6.4 Private-room read-time chrome

- **For participants of a private room:** a small, non-alarming **"Private"** badge near the room title (`accessibilityLabel`: "This argument is private — only invited people can read it"). The badge is a state indicator, never a verdict.
- **The gallery / list:** a private room the caller can read (because they're a participant) may still show in **"My rooms"** sections of the Conversation Gallery — RLS returns it to them. It must **never** appear in the public-discovery sections ("Jump into a live dispute", "Needs first rebuttal", etc.). Because RLS already withholds private rooms from non-participants, the gallery model needs no new filter for *non-participants*; for the *participant's own* view, the gallery's section grouping (`groupGalleryCardsBySection`, `conversationGalleryModel.ts`) should route a `visibility === 'private'` card into "My rooms" only and exclude it from the public-discovery buckets. This is a small, additive grouping rule — see §7.
- **The muted chime-in branch** (roommates Step 15): it was *already* visually muted by GAME-005 governance before the transition. After the transition it is simply visible to fewer people. QOL-039 renders no new mute state — it relies on GAME-005's existing `observer_only` rendering.

### 6.5 Error & edge states

- **RLS-denied transition** (e.g. a stale client where a non-creator's button rendered): a neutral "You can't change this argument's visibility." — routed through `gameCopy`, never a raw error.
- **Network failure mid-transition:** the room is unchanged (the `UPDATE` is atomic); a neutral retry prompt. The event is emitted **only** after a confirmed success.
- **Concurrent transition** (creator + mod both tap): the first `UPDATE` wins; the second is an idempotent no-op (`visibility` is already `private`); the §4.2 trigger does not fire (no `public → public` and no `private → public`). The second client refreshes and finds the room already private — no error shown.

## §7 — Permission / rules implications

| Concern | Rule |
|---|---|
| Who may create a private room | Any authenticated user, at room creation (`CreateDebateInput.visibility`). |
| Who may transition `public → private` | The room **creator**, or a **moderator/admin**. Enforced three ways: `canTransitionToPrivate` (UI gate), the `debates` UPDATE RLS policy (`creator OR mod/admin`), and — for the one-way rule — the §4.2 trigger. |
| Who may transition `private → public` | **No one.** No API, no UI, and the §4.2 trigger rejects it at the DB. |
| Who may read a private room | Participants (`debate_participants` row), moderators/admins. Enforced by the §4.3 RLS policies. |
| Constitution engine (`engine.ts`) | **Untouched.** Visibility is an access property; it is not an argument type, a transition, or a rule. The engine never reads it. |
| `submit-argument` Edge Function | **Untouched.** Posting into a private room works exactly as a public room for a participant; visibility never gates the write path, only the read path. Non-participants already cannot post (they have no `debate_participants` row and the room is not joinable to them once private — see below). |
| Joining a private room | A private room is **not** publicly joinable. The existing `debate_participants` INSERT policy gates on `is_debate_joinable` (status `draft`/`open`) — it does **not** currently check visibility. v1 decision: QOL-039 does **not** change the INSERT policy, because the *only* way a non-participant could attempt to join is if they could already see the room — and §4.3.2/4.3.3 make a private room invisible to them. A non-participant with a stale `debate_id` who POSTs a join would succeed at the RLS layer (the room is `open`) but immediately see nothing (the SELECT policies hide everything). This is a benign no-op, not a leak. **Tightening the join INSERT policy to also require visibility-or-invite is a clean follow-up** (Open Question OQ-2) — QOL-038 (invite/auth) is the natural home for invite-gated joins. QOL-039 flags it; it is not in scope here. |
| One-box integration (QOL-030..033) | The create-time control (§6.1) is a `root_claim` box room-setup field; the `make private` action (§6.2) is a `room`-target direct flash-menu entry. QOL-030 §6.3 already enumerates `make private` as a direct entry — QOL-039 supplies its handler + confirmation. The gallery grouping nuance (§6.4) is a one-line rule in `conversationGalleryModel.ts`'s `groupGalleryCardsBySection`. |
| GAME-005 seats | Visibility does not change seat assignment. A private room still has an Initiator, Primary Opponent, and chime-in seats; chime-in *acquisition* simply can't happen from a non-participant because they can't see the room. GAME-005's `MovedToObserverRecord` set is read **as-is** to populate `audiences.rejectedChimeInUserIds` (§5.4). |
| Heat / popularity / standing | Never inputs. Visibility is chosen by the creator; it is never auto-set by activity, reply count, or standing. A "hot" room is not auto-made-public; a "quiet" room is not auto-made-private. |

## §8 — Edge cases

- **Existing rooms after migration.** Default `'public'` → every current `open`/`locked` debate stays listed and globally readable. No live room is hidden by the deploy. (Verified by the `db reset` test against the seed data — §10.)
- **`draft` room made private.** A draft is creator-only-visible already. Setting it private at creation is fine; transitioning a draft is allowed and harmless (it was never public-listed). The consequences confirmation still shows; the `leaves_public_list` bullet is technically a no-op for a draft but is not misleading.
- **`archived` room.** v1 allows transitioning an archived room's visibility (it still changes who can read the history). See OQ-3 if the operator wants to forbid it.
- **Room with zero non-participants / zero chime-ins.** `buildTransitionConsequences` returns `retainedChimeInBranchCount: 0`; the chime-in bullet is omitted; the other five bullets still show. `audiences.rejectedChimeInUserIds` and `nonParticipantObserverUserIds` are empty arrays — QOL-040 simply sends fewer notifications.
- **Caller is a mod, not the creator.** Allowed (§5.1). The event's `actorUserId` is the mod; QOL-040 must still phrase the notification structurally ("this argument was made private"), never "a moderator made this private."
- **Creator leaves the room after going private.** Out of QOL-039 scope (leaving is GAME-005 / participant management); the room stays private regardless — visibility is a room property, not a creator-presence property.
- **Concurrent argument post during the transition.** A participant posting at the same instant the room goes private is unaffected — the post is a participant write, still permitted; the new argument is private along with the room.
- **A non-participant has the room open when it goes private.** Their already-loaded data stays on their screen until they refresh; the next `listDebates` / room fetch returns nothing for them (RLS). No client-side scrub is required or attempted — RLS is the boundary; stale in-memory data is not a leak vector worth a forced logout. A refresh / re-navigation naturally drops it.
- **Private room shared by deep link.** A non-participant opening a private room's deep link gets the same empty result as any other non-participant read — RLS denies it; the UI shows a neutral "This argument isn't available to you" (routed through `gameCopy`). QOL-042 (linked prior argument) already needs this exact "title-only / no-access" state — QOL-039's neutral copy is the seed.
- **Doctrine edge — "does making it private change any score?"** No. Visibility is not an input to point standing, heat, topic-fit, or any band. A private room's arguments score identically to a public room's. The transition is purely an access change.
- **Doctrine edge — "is a private room a 'losing' room?"** No. Going private is never a verdict. The confirmation copy and the badge are strictly neutral; the test suite scans for verdict tokens.

## §11 — Doctrine & safety self-check

- **cdiscourse-doctrine — no truth labels.** Visibility is an access property, not a verdict. No `winner/loser/true/false/correct` copy; the confirmation, badge, and event are strictly structural. Verified by the §10 ban-list scan.
- **cdiscourse-doctrine — score never blocks posting.** Visibility does not gate the write path — `submit-argument` is untouched; a participant posts into a private room exactly as a public one. Visibility only changes *reads* and *listing*.
- **cdiscourse-doctrine — heat / popularity not inputs.** Visibility is creator-chosen; it is never auto-set by reply count, heat, or standing. §7.
- **cdiscourse-doctrine — no service-role in client.** The transition is a plain RLS-gated client `UPDATE` (creator/mod own the row); no service-role, no Edge Function needed (§4.4).
- **supabase-edge-contract — RLS always on.** No policy disables RLS; four SELECT policies are *replaced* with stricter ones; INSERT/UPDATE/DELETE policies are unchanged. The one-way invariant is a `BEFORE UPDATE` trigger, not an RLS bypass.
- **supabase-edge-contract — no direct `arguments` insert; no hard delete.** QOL-039 mutates **one column on one `debates` row** and **zero `arguments` rows**. The "retained (muted), never deleted" requirement is met by definition — the rejected chime-in branch's rows are never touched; only the read audience shrinks (§4.3.4).
- **supabase-edge-contract — migration discipline.** One new append-only migration; no applied file edited; reuses the existing `SECURITY DEFINER` helper pattern from `20260516000006` so the documented `debates ⇄ debate_participants` recursion incident does not recur (§3 recursion trap).
- **GAME-005 doctrine inheritance.** "A seat / a governance reaction describes structure, never the person" extends to visibility: making a room private is a structural transition, never a penalty or shaming. The rejected chime-in branch is retained in full and the only person-relative notification is QOL-040's, governed by QOL-039's redaction contract (§5.5).
- **Plain language.** Every internal code routes through `gameCopy.toPlainLanguage`; no raw code reaches the UI. User-facing copy says "argument" / "room", never "game", never "debate". The `debates` table name stays internal.
- **v1 scope guards.** No voting, no search, no push notifications (QOL-040 is in-app only), no OAuth, no public API. QOL-039 adds none of these.
- **No AI calls.** `roomVisibilityModel.ts` is pure deterministic TS — no Anthropic / xAI / OpenAI / X API, no network, no `Date.now()` in the model (timestamps are passed in / set by the API wrapper).
