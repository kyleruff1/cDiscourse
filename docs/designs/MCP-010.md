## 1. Goal (one paragraph)

MCP-010 designs the **semantic override and appeal UX** — the surface that lets a user reverse an uncertain or wrong semantic routing / friction suggestion without ever feeling caged. The semantic referee (MCP-001) is advisory: `authoritative: false`, always. When its packet confidence is `low`, or when its layer-2 reading conflicts with the deterministic layer-1 metadata (MCP-003's `conflict_routed` / `low_confidence_semantic` outcomes), the referee deliberately **does not act silently** — it surfaces a small, reversible choice and lets the **user**, not the model, decide where the move goes. The doctrine that shapes every decision: a semantic suggestion is never a hard gate (the user can always proceed and the move is already posted); overriding is never a penalty — no score cost, no flag, no "you were warned" copy; the override is recorded as **visible, auditable metadata**, not hidden state; and no verdict / person-label copy appears on any override surface. The override produces a **lane choice** (`mainline` / `branch` / `tangent`) plus an optional "this answers the parent" assertion; that lane choice is an **advisory input** that BR-003 / BR-004's branch grammar will later route — MCP-010 does **not** redesign branch grammar, it only emits the choice. The override is written as a META-001-style metadata event of a new `semantic_override` kind attached to the move, visible in the sidecar's semantic section, and consumed read-only by MCP-003 to re-read the contested categories at full magnitude with **no penalty for the earlier conflict**.

## 2. Why this is a layer on top of MCP-001 + MCP-003, not a new classifier

The MCP family already produced everything the override UX needs as **input**, and the override doctrine is already stated as a contract MCP-010 must implement:

- **MCP-001 §7** locked the `SemanticRefereePacket` — each `SemanticBinarySample` carries `classifierId`, `value: 0|1`, `confidence: 'low' | 'medium' | 'high'`, `reasonCode`; the packet carries `routeSuggestion` and `frictionSuggestion`. **MCP-001 §14** ("False-positive mitigation") states the rule: "On conflict — confidence is downgraded and a reversible user choice is surfaced (MCP-010), never a penalty and never a silent override." **MCP-001 §15** ("Manual override flow") names MCP-010 as the owner and fixes the contract: low confidence or L1/L2 conflict surfaces a reversible choice; the user picks the lane or asserts "this answers the parent"; the override is recorded as visible META-001-style metadata; overriding carries no score cost, no flag, no "you were warned" copy; repeated overrides may inform future UX tuning but do not train any live model in v1.
- **MCP-003 §5** authored the reconciliation algorithm — the `CategoryReading` shape (`category`, `delta`, `outcome`, `confidence`, `requiresUserChoice`, `feedbackCode`, `factualStandingAxis`), the `ReconciliationOutcome` union (including `conflict_routed` and `low_confidence_semantic`), and `LedgerResult.needsUserChoice`. **MCP-003 §5.2** states the non-negotiable conflict doctrine and §5.2 explicitly says: "The user's manual override (MCP-010) carries no score cost… The override is recorded as visible metadata (a META-001-style `semantic_override` entry), not hidden state — but MCP-003 only consumes that entry; the entry's creation is MCP-010's job."
- **META-001** (#62, closed) shipped the `MetadataEvent` shape and the `MoveMetadataLedger`. The override record reuses that shape directly.

So MCP-010 does **not** define a new classifier, does **not** define a new score category, does **not** scrape conversation, and does **not** redesign branch grammar. It is a **UX + record layer**. It does exactly five things and nothing else (the five scope items in the card body):

1. **Conflict / low-confidence trigger** — a pure-TS model that decides, from a packet + a `LedgerResult`, whether the override choice surface should appear (scope item 1).
2. **Choice surface** — a React Native component that presents the reversible lane choice and the "this answers the parent" toggle (scope item 2).
3. **Override record** — a META-001-style `semantic_override` metadata event, written to the in-memory metadata ledger, visible in the sidecar (scope item 3).
4. **Repeated-override signal** — an in-memory, per-room count that informs UX copy only; no model training, no user-level score (scope item 4).
5. **Copy** — the plain-language, non-punitive strings, added to `gameCopy.PLAIN_LANGUAGE_COPY` (scope item 5).

It is a consumer of MCP-001 and MCP-003 and a producer of one new metadata-event family. It never modifies a packet, never modifies the MCP-003 ledger model, never re-derives a score, and never routes a branch.

## 3. Doctrine constraints

From `cdiscourse-doctrine`, `timeline-grammar`, `accessibility-targets`, `expo-rn-patterns`, `test-discipline`, the MCP roadmap expansion §3, and MCP-001 §3 / MCP-003 §3. **A violation is a blocker, not a comment.**

1. **A semantic suggestion is never a hard gate.** The override surface appears *after* a move is posted (post-submit classification — MCP-001 §4). The move already exists; the user can always proceed; there is nothing to "unblock". The override only re-routes a lane *suggestion* and re-reads advisory categories. Only deterministic structural validation (empty body, invalid transition, explicit-evidence-without-source, over-length, auth, opted-in cooldown) ever blocks — and the override surface touches none of that.
2. **Overriding is not a penalty.** No score cost, no `flags` row, no negative delta, no "you were warned" / "you were wrong" / "you ignored the suggestion" copy. MCP-003 §5.2 already guarantees the contested category re-reads at full magnitude with no penalty for the earlier conflict; MCP-010 must not introduce any penalty of its own.
3. **The override is visible, auditable metadata — never hidden state.** It is a `semantic_override` `MetadataEvent` written to the metadata ledger and rendered in the sidecar's semantic section. There is no silent field, no shadow flag, no quiet model nudge.
4. **No verdict / person labels** on any override surface — never *winner, loser, won, lost, right, wrong, true, false, correct, incorrect, proven, defeated, liar, lying, dishonest, bad faith, manipulative, troll, propagandist, extremist, stupid, idiot, dumb, smart*. Copy classifies the *move's structure / lane*, never the person. Test-enforced ban-list (§15).
5. **AI is advisory only; the user decides on uncertainty.** The override surface appears *because* the model declared uncertainty (`low` confidence) or because layer 1 and layer 2 disagreed. The model never overrides the user; the user overrides the model. The override is the doctrine "when it is unsure, the user — not the model — decides" made into UI.
6. **No AI call from the override surface.** The override UX consumes an already-classified packet / ledger result; it never triggers a fresh semantic call (MCP-001 §5: never on selection, never on a non-trigger moment). The override surface runs entirely on the client off already-fetched data. No client-side AI, ever.
7. **Repeated overrides never train a live model in v1 and never accumulate into a user-level score.** The repeated-override count is an in-memory, per-room, per-render signal that may only soften copy. It is not persisted, not sent anywhere, not summed across rooms, and not attached to a profile. (Card body acceptance criterion + operator decision §16.)
8. **Popularity / heat / virality is never an input.** The override trigger reads only packet confidence + L1/L2 conflict; it never reads engagement, view count, or heat band.
9. **Pure-TS model + thin React component split.** The trigger / record-builder logic is a pure-TS model (`src/features/semanticOverride/`) — no Supabase, no React, no network — identical on client and Edge Function. The choice surface is the only React file and it only consumes the model.
10. **Plain language only.** No raw `classifierId` / `reasonCode` / `ReconciliationOutcome` reaches the user. Every override string is keyed off a code mapped through `gameCopy.toPlainLanguage`; an unmapped code is suppressed, not echoed (already how `toPlainLanguage` behaves, test-enforced).
11. **Accessibility is part of done.** The choice surface meets `accessibility-targets`: 44×44 hit targets, `accessibilityRole` / `accessibilityLabel` / `accessibilityState` on every control, color is never the only signal, reduce-motion respected.
12. **v1 scope guards.** No voting, no winner-producing scoring, no search, no push, no OAuth, no public API is introduced. The override surface is a reversible choice + a metadata record; it adds none of those.

## 4. Data model

**No new database table, no migration, no Edge Function.** MCP-010 is design-only, and even the future implementation card delivers a **pure-TS model + a React component**. The override record is an in-memory `MetadataEvent` of a new family — it lives in the same render-time, `useState`-owned metadata ledger META-001 established (META-001 §"Goal": "manual tags live in `useState` and disappear at unmount. Persistence is explicitly v2"). The override record follows the same v1-in-memory / v2-persist split.

### 4.1 The `semantic_override` metadata event — reusing META-001's `MetadataEvent`

META-001's `MetadataEvent` (in `src/features/metadata/moveMetadataLedger.ts`) is:

```ts
export interface MetadataEvent {
  eventId: string;
  kind: 'add' | 'remove' | 'transition';
  codeFamily: 'manual_tag' | 'auto_metadata' | 'lifecycle_causation';
  code: string;
  messageId: string;
  clusterId: string;
  at: string;            // ISO-8601
  cause?: string | null; // internal debug only — never rendered
}
```

MCP-010 reuses this shape **verbatim** and adds **one** value to the `codeFamily` union — `'semantic_override'`:

```ts
// Proposed extension to META-001's MetadataEvent.codeFamily (one new member).
codeFamily: 'manual_tag' | 'auto_metadata' | 'lifecycle_causation' | 'semantic_override';
```

This is the minimal, doctrine-neutral change to META-001 the implementation card makes — the same pattern META-001 itself anticipated ("Later cards… consume the ledger"). For a `semantic_override` event:

- `kind` is always `'add'` (an override is an additive, append-only record — there is no `remove`; a user changing their mind appends a *new* override event, see §6.4).
- `codeFamily` is `'semantic_override'`.
- `code` is the chosen lane: one of `mainline` / `branch` / `tangent` / `answers_parent` (the last is the explicit "this answers the parent" assertion — see §4.2). These four codes map through `gameCopy` (§9).
- `messageId` / `clusterId` anchor the override to the move, exactly as a manual tag does.
- `at` is the caller-supplied ISO-8601 timestamp of the user's tap.
- `cause` carries the internal debug string (`override_low_confidence` / `override_l1_l2_conflict`) — never rendered (matches META-001's `cause` convention).

The metadata-event log already drives the sidecar "what changed on this move" affordance (META-001 §"Product context"); a `semantic_override` event joins that stream and is rendered by the sidecar's semantic section (§7).

### 4.2 The `SemanticOverrideRecord` — the richer per-move shape

The `MetadataEvent` is the *log* entry. The per-move *record* the choice surface produces and the sidecar reads is richer (it carries the original suggestion so the audit trail is complete). Proposed:

```ts
// src/features/semanticOverride/  (proposed — not yet code)

/** The lane the user chose. The 'tangent' / 'branch' / 'mainline' values feed
 *  BR-003 / BR-004's branch grammar as an ADVISORY input — MCP-010 does not
 *  route; it records the choice and BR-003/BR-004 consume it later. */
export type SemanticOverrideLane = 'mainline' | 'branch' | 'tangent';

/** Why the override surface was offered in the first place. */
export type SemanticOverrideTriggerReason =
  | 'low_confidence'        // packet binary confidence === 'low'
  | 'l1_l2_conflict';       // MCP-003 reconciliation produced a conflict_routed reading

/** A single, append-only override record attached to one move. Visible
 *  metadata — never hidden state. Reuses the META-001 MetadataEvent shape for
 *  the log entry; this is the fuller record the sidecar renders. */
export interface SemanticOverrideRecord {
  /** Stable id — `semantic_override:${messageId}:${at}`. */
  overrideId: string;
  messageId: string;
  clusterId: string;
  /** The lane the user picked. */
  chosenLane: SemanticOverrideLane;
  /** True when the user explicitly asserted "this answers the parent" to
   *  overturn a responds_to_parent: 0. Independent of `chosenLane` — a user
   *  can both assert "answers the parent" AND keep it on the mainline. */
  assertsAnswersParent: boolean;
  /** What the referee had suggested before the override — recorded so the
   *  audit trail shows what was reversed. Plain-language-mapped before render. */
  originalRouteSuggestion: SemanticRouteSuggestion; // from MCP-001 §7
  /** Why the choice was offered. */
  triggerReason: SemanticOverrideTriggerReason;
  /** The classifier id family that was contested (e.g. 'responds_to_parent',
   *  'suggests_side_branch'). Internal — mapped through gameCopy before render. */
  contestedClassifierId: string;
  /** ISO-8601 — caller-supplied tap time. */
  at: string;
  /** The viewer who made the override. In v1 in-memory this is the current
   *  viewer id passed by the caller (matches META-001's ManualTagEntry pattern). */
  overriddenByUserId: string;
  /** Actor role of the overrider — same enum META-001 ManualTagEntry uses. */
  overriddenByActorRole:
    | 'participant_affirmative'
    | 'participant_negative'
    | 'observer'
    | 'admin';
}
```

### 4.3 The trigger model output

```ts
/** Output of the pure trigger model — decides whether the choice surface
 *  appears and what it should offer. Computed from a packet + a LedgerResult. */
export interface SemanticOverridePrompt {
  /** True when the choice surface should be shown for this move. */
  shouldOffer: boolean;
  /** Why — drives the trigger reason on the produced record + the copy. */
  triggerReason: SemanticOverrideTriggerReason | null;
  /** The referee's current suggestion the user may reverse. */
  suggestedLane: SemanticOverrideLane;
  /** True when a responds_to_parent: 0 binary (or its conflict) is in play,
   *  so the "this answers the parent" toggle is shown. */
  offersAnswersParentToggle: boolean;
  /** The contested classifier id — recorded onto the record, mapped for copy. */
  contestedClassifierId: string | null;
  /** Plain-language code for the prompt headline — mapped via gameCopy. */
  promptCopyCode: string;
}
```

### 4.4 The repeated-override signal (in-memory, UX-only)

```ts
/** Per-room, in-memory, per-render. NOT persisted. NOT a user-level score.
 *  NOT sent anywhere. NOT a model-training input. It exists only to let the
 *  copy soften after a user has reversed several suggestions in one room
 *  (e.g. switch the prompt from a chip to a quieter inline line). */
export interface RepeatedOverrideSignal {
  roomId: string;
  /** Count of override records in this room this session. */
  overrideCountThisRoom: number;
  /** When the count crosses a small threshold (proposed 3), the trigger model
   *  sets a softer promptCopyCode. The threshold is a UX constant, not a gate. */
  softenCopy: boolean;
}
```

All of these shapes are pure-TS, JSON-serializable, immutable (`readonly` where structural), no `Date` objects, no functions, no class instances, no React types — same constraint as `src/lib/constitution/engine.ts`, the point-standing economy, and the META-001 ledger.

## 6. UI / UX behavior

### 6.1 Scope item 1 — the conflict / low-confidence trigger

The choice surface is **not** shown on every move. It is shown only when the referee declared uncertainty. The pure trigger model `evaluateSemanticOverridePrompt(packet, ledgerResult, repeatedSignal)` returns `shouldOffer: true` when **either**:

- **Low confidence** — the `SemanticRefereePacket` carries a routing-relevant binary (`responds_to_parent`, `introduces_new_issue`, `suggests_side_branch`, `suggests_diagonal_tangent`) whose `confidence` is `'low'` (MCP-001 §7). The trigger reason is `low_confidence`.
- **L1/L2 conflict** — the MCP-003 `LedgerResult.needsUserChoice` is `true` *and* at least one `CategoryReading` has `outcome: 'conflict_routed'` on a routing-relevant category (`continuity`, `direct_response`). The trigger reason is `l1_l2_conflict`. (MCP-003 §5.1 rule 6: `conflict_routed` sets `requiresUserChoice: true` and a `feedbackCode` that "maps to a reversible MCP-010 choice prompt".)

When `shouldOffer` is `false` (the referee was confident and L1/L2 agreed), **no override surface appears** — the route chip / banner behaves normally (MCP-001 §17) and the user can still ignore it. The override surface is the *uncertainty* path, not the always-on path. This keeps the surface rare and meaningful instead of a permanent piece of chrome.

The trigger model is pure-TS and reads only the packet + the ledger result + the in-memory `RepeatedOverrideSignal`. It triggers **no AI call** — the packet is already classified (MCP-001 §5: no semantic call on selection / non-trigger moments).

### 6.2 Scope item 2 — the choice surface

When `shouldOffer` is `true`, the `SemanticOverrideChoiceSheet` renders inside the existing selected-message sidecar (`ArgumentReplySidecar`) — it is **not** a blocking modal. The move is already posted and visible; the surface is a calm, optional panel below the move body. It contains:

1. **A one-line, non-punitive prompt** — copy code `semantic_override_prompt` → "Not sure this is a reply? You decide where it goes." (low-confidence) or "The referee and the room rules disagree here — you choose the lane." (conflict). The prompt never says the move was wrong or off-topic.
2. **Three lane choices**, each a 44×44+ `Pressable` styled as a selectable chip, mutually exclusive (radio-group semantics):
   - **Reply here (mainline)** — copy "Reply here — this stays on the mainline."
   - **Branch** — copy "Branch — this opens a related thread."
   - **Tangent** — copy "Tangent — this is a separate side issue."
   The currently-suggested lane is pre-selected (so "keep the suggestion" is one tap of confirm, not a hunt). The lane labels and the `branch` / `tangent` distinction follow the `timeline-grammar` skill's branch vocabulary; MCP-010 emits the choice, BR-003 / BR-004 own how the lane is drawn and routed.
3. **A "This answers the parent" toggle** — shown only when `offersAnswersParentToggle` is `true` (a `responds_to_parent: 0` or its conflict is in play). It is an independent checkbox, not part of the radio group: the user can assert "this answers the parent" *and* keep the move on the mainline, or branch it, etc. Copy: "This answers the parent — count it as a direct reply." Toggling it on overturns the `responds_to_parent: 0` suggestion.
4. **A confirm affordance** — "Set the lane" (when the user changed something) / "Keep referee suggestion" (when nothing changed). Confirm writes the override record (§6.3). There is no separate "cancel" that punishes — dismissing the sheet without confirming simply leaves the referee's advisory suggestion in place, unchanged, no record written.

The surface is **fully reversible**: a user who set `tangent` and then reconsiders can reopen the sheet and pick `mainline` — that appends a *new* override record (§6.4); the latest record wins for display and for what MCP-003 / BR-004 consume.

The surface is **advisory in both directions** — it neither blocks the user from leaving the move as-is, nor blocks the user from re-routing it. Per MCP-001 §3.3 the move is never gated.

### 6.3 Scope item 3 — the override record

On confirm, the record-builder `buildSemanticOverrideRecord(input)` produces a `SemanticOverrideRecord` (§4.2) and `toSemanticOverrideMetadataEvent(record)` produces the META-001 `MetadataEvent` (§4.1, `codeFamily: 'semantic_override'`, `kind: 'add'`). The caller (room shell) appends the event to the in-memory metadata ledger — exactly the path META-001's `applyManualTag` already uses for manual tags. The record is then:

- **Visible in the sidecar** — the `ArgumentReplySidecar` semantic section renders a read-only line: "You set the lane: Tangent." plus, if asserted, "You marked this as a direct reply." The original referee suggestion is shown alongside for audit ("Referee had suggested: branch"). No raw codes — all mapped via `gameCopy`.
- **Consumed read-only by MCP-003** — MCP-003 §5.2: "If the user asserts 'this answers the parent'… the `continuity` and `direct_response` categories are re-read from the override as `agreement`-tier `l1_only`… at full magnitude, with no penalty for the earlier conflict." MCP-010 *produces* the record; MCP-003 *consumes* it. MCP-010 does not move score.
- **Consumed read-only by BR-003 / BR-004** — the `chosenLane` is an advisory routing input to the branch grammar. BR-003 / BR-004 decide how `branch` / `tangent` are drawn and where the move sits; MCP-010 only records the user's lane intent. (Card body: "the lane choice routes into their branch grammar"; this design references BR-003 / BR-004 as downstream consumers and does **not** redesign branch grammar.)

The record is **append-only and never deleted** — consistent with the doctrine that metadata is auditable. `MetadataEvent` for `semantic_override` is always `kind: 'add'`.

### 6.4 Scope item 4 — the repeated-override signal

Every confirmed override increments the in-memory `RepeatedOverrideSignal.overrideCountThisRoom`. When the count crosses a small UX threshold (proposed 3), the trigger model sets `softenCopy: true`, which only swaps the prompt copy code for a quieter inline variant ("You decide the lane here." instead of the fuller two-line chip). That is the **entire** effect.

The signal is explicitly **not**:

- persisted to Supabase (in-memory, per render, gone at unmount — matches META-001's manual-tag store);
- summed across rooms or attached to a profile (no user-level score);
- sent to any provider or model (no live-model training in v1 — card body acceptance criterion);
- a gate, a penalty, or an input to score.

It is purely a politeness affordance so a user who clearly knows what they want stops getting a loud prompt. Whether a future UX-tuning surface ever reads an *aggregate* of overrides is an operator decision (§16); the v1 answer is "metadata only, no training".

### 6.5 Scope item 5 — copy

All override copy is plain-language, non-punitive, and ban-list-clean. The full table is §9. The voice rule: every string is about *where the move goes* or *what the user decided*, never about the move being wrong and never about the person. There is no "you were warned", no "the referee flagged this", no "off-topic", no "rejected". The two anchor lines the card body quotes are honored verbatim: "Not a reply? Choose where this goes." and "You decide — this stays on the mainline."

## 7. Sidecar integration

The override surface lives in the existing selected-message sidecar (`ArgumentReplySidecar.tsx`), not a new screen. Two distinct sidecar states:

- **Choice state** — when `shouldOffer` is `true` and no override record exists yet for the move, the sidecar shows the `SemanticOverrideChoiceSheet` (§6.2).
- **Recorded state** — once an override exists, the sidecar shows a compact, read-only "You set the lane" summary (§6.3) with a small "Change" affordance that reopens the choice sheet (appending a new record on confirm).

The semantic section of the sidecar is also where the referee's *normal* advisory route chip / banner appears (MCP-001 §17, MCP-008 banners). The override surface sits directly below it so the relationship — "the referee suggested X; here is your choice" — is spatially obvious. The sidecar already renders metadata-event-driven content (META-001 §"Product context": the metadata-event log drives the sidecar "what changed" affordance), so the `semantic_override` event slots into the existing rendering path.

## 8. Accessibility

Per `accessibility-targets`, the `SemanticOverrideChoiceSheet` must meet:

- **Hit targets ≥ 44×44** on every lane `Pressable` and the toggle and the confirm button; `hitSlop` where the visual chip is smaller.
- **Radio-group semantics** for the three lanes: each lane `Pressable` has `accessibilityRole="radio"`, `accessibilityState={{ selected: isChosen }}`, and an `accessibilityLabel` that includes the lane name + a short consequence ("Reply here, stays on the mainline"). The "answers the parent" control has `accessibilityRole="checkbox"` with `accessibilityState={{ checked }}`.
- **Confirm button** has `accessibilityRole="button"`, a label that reflects its current meaning ("Set the lane to tangent" / "Keep referee suggestion"), and `accessibilityState={{ disabled }}` if nothing is selectable.
- **Color is never the only signal** — the selected lane is shown by a check glyph + a text-weight change + a border, not color alone (grayscale-snapshot test, `timeline-grammar` rule). The branch / tangent / mainline chips additionally carry the `timeline-grammar` shape vocabulary cue where space allows (mainline = straight, branch = bent connector, tangent = kink), but the **label text** is the primary signal.
- **Screen-reader announcement on confirm** — `AccessibilityInfo.announceForAccessibility` fires once: "Lane set to tangent." (sparingly, per the skill — one announcement per confirmed override, never on selection or hover).
- **Reduce motion** — the sheet open/close snaps instead of animating when `AccessibilityInfo.isReduceMotionEnabled()` is true; the recorded-state transition is a fade, not a slide, under reduce-motion.
- **Focus order** — the prompt line → three lanes → toggle → confirm, matching reading order; on web, `Tab` walks them in that order and `Esc` closes the sheet returning focus to the node (consistent with the IX-003 keyboard contract in `accessibility-targets`).

## 10. API / interface contracts

The load-bearing signatures the future implementation card must honor:

```ts
// src/features/semanticOverride/overrideTriggerModel.ts
export function evaluateSemanticOverridePrompt(input: {
  packet: SemanticRefereePacket;            // MCP-001 §7 — read-only
  ledgerResult: LedgerResult;               // MCP-003 §9.3 — read-only
  repeatedSignal: RepeatedOverrideSignal;
}): SemanticOverridePrompt;

// src/features/semanticOverride/overrideRecordModel.ts
export function buildSemanticOverrideRecord(input: {
  prompt: SemanticOverridePrompt;
  messageId: string;
  clusterId: string;
  chosenLane: SemanticOverrideLane;
  assertsAnswersParent: boolean;
  originalRouteSuggestion: SemanticRouteSuggestion;
  overriddenByUserId: string;
  overriddenByActorRole: SemanticOverrideRecord['overriddenByActorRole'];
  at: string;                               // ISO-8601, caller-supplied
}): SemanticOverrideRecord;

export function toSemanticOverrideMetadataEvent(
  record: SemanticOverrideRecord,
): MetadataEvent;                           // codeFamily: 'semantic_override', kind: 'add'

export function bumpRepeatedOverrideSignal(
  signal: RepeatedOverrideSignal,
): RepeatedOverrideSignal;                  // pure — returns a new signal, count + 1
```

Contract rules:
- `evaluateSemanticOverridePrompt` is **pure** — no network, no React, no Supabase, no mutation. Same constraint as the rules engine.
- The `SemanticRefereePacket` and `LedgerResult` are consumed **read-only**; MCP-010 never constructs or mutates a packet or a ledger result (those are MCP-001 / MCP-003 / MCP-009).
- `SemanticOverrideRecord` has **no** `delta`, **no** `score`, **no** `penalty`, **no** `block`, **no** `flag` field — enforced by a schema-shape assertion in `semanticOverrideNoPenalty.test.ts` (§15).
- `toSemanticOverrideMetadataEvent` always returns `kind: 'add'` — overrides are append-only.
- The override model imports from `src/features/metadata` (the `MetadataEvent` type) and the MCP-001 / MCP-003 type modules only — never the reverse (MCP-001 / MCP-003 must not depend on `semanticOverride`).

## 11. Edge cases

- **Confident packet, L1/L2 agree** — `evaluateSemanticOverridePrompt` returns `shouldOffer: false`; no override surface, the normal route chip behaves as in MCP-001 §17. Not every move gets a choice surface.
- **Empty / missing semantic packet** (feature flag off — the packet is a `provider: 'mock'` stub, MCP-001 §10) — the mock packet carries no `low`-confidence routing binary and the ledger reads `l1_only`, so `shouldOffer` is `false`. The override surface simply never appears; the board still works. No error.
- **User dismisses the sheet without confirming** — no record is written; the referee's advisory suggestion stays in place unchanged; no penalty, no "you ignored this" copy. Dismissing is a no-op, not a decision.
- **User picks the lane the referee already suggested** — a record is *still* written (the user explicitly affirmed it). The confirm button reads "Keep referee suggestion"; the record's `chosenLane === originalRouteSuggestion`'s lane. This is intentional — an explicit affirmation is auditable and lets MCP-003 re-read at `agreement` confidence.
- **User overrides, then changes their mind** — a *new* `semantic_override` `MetadataEvent` (`kind: 'add'`) is appended; the latest `at` wins for sidecar display and for MCP-003 / BR-004 consumption. Earlier records are never deleted — the audit trail shows the full sequence. (§6.4.)
- **Two participants on the same move** — each can override independently; each produces a record keyed by `overriddenByUserId` (same multi-applier pattern as META-001's manual tags, §"Edge cases" #5: "Tag application is additive… There is no 'winning' tag."). The sidecar shows the viewer's own latest override prominently and notes others exist. MCP-003 / BR-004 consume per-viewer or per-move per their own rules — MCP-010 just records.
- **Observer viewing the move** — observers are read-only on the gameplay surface in v1 (Stage 6.4). The override surface is **not offered to observers** — the choice changes how a move is routed/scored, which is participant territory. The trigger model returns `shouldOffer: false` when the viewer's actor role is `observer`. (Matches META-001's "Observers may NEVER apply manual tags in v1.") Observers still *see* a recorded override (it is visible metadata) read-only.
- **Own bubble** — the author of the move may override the lane on their own move (deciding "this is a tangent of mine" is a legitimate self-routing call, and routing is not a pressure tag). This is the one place MCP-010 is more permissive than META-001's manual-tag own-bubble rule, because a lane choice is structural self-description, not a challenge tag. The implementation card confirms this against the GAME-004 own-bubble action set; if a conflict is found, the safe fallback is "own-bubble may only assert `answers_parent`, not re-lane" — flagged as a small open item (§16).
- **Soft-deleted move** — if the move id no longer exists in the timeline map, the override surface is not shown and any stale record is not rendered (the room shell filters deleted messages before the metadata ledger sees them — META-001 §"Edge cases" #3).
- **Repeated-override count high** — `softenCopy` only swaps the prompt copy; it never suppresses the surface entirely (a user must always be able to override). There is no upper bound that disables the feature.
- **Doctrine edge case — "can an override move score?"** — no. MCP-010 writes a record; MCP-003 re-reads the contested category from that record. The re-read is MCP-003 §5.2's `agreement`-tier `l1_only` at full magnitude *with no penalty for the earlier conflict*. MCP-010 itself has no score field and moves nothing.
- **Doctrine edge case — "does overriding flag the user or the move?"** — no. There is no `flags` row, no moderation entry. The `semantic_override` `MetadataEvent` lives in the in-memory gameplay metadata ledger, which META-001 §"doctrine" #3 keeps strictly separate from `public.flags`.
- **Doctrine edge case — "does a repeated override penalize the user?"** — no. The count only softens copy. It is not persisted, not summed across rooms, not a profile field, not a score, not a training input.
- **Concurrent edits** — argument bodies are immutable after posting (soft-delete only — CLAUDE.md Supabase conventions). The override anchors to a `messageId`; no concurrent-edit race exists for the move body. Two overrides racing produce two append-only records; latest `at` wins. No lost update.
- **Offline / network failure** — the override model is pure-TS with no network; the choice surface writes to the in-memory ledger. The *packet* is fetched upstream (MCP-009 Edge Function); if that fetch failed the packet is the mock stub and `shouldOffer` is `false` — the override surface degrades to absent, the board still works.

## 13. Risks

- **BR-003 / BR-004 are open — the lane vocabulary could drift.** MCP-010 emits `mainline` / `branch` / `tangent`. If BR-003 / BR-004 finalize a different lane vocabulary (e.g. an additional `outer_realm` lane, which MCP-001's `SemanticRouteSuggestion` already hints at), the implementation card must reconcile `SemanticOverrideLane` against the final branch grammar. Mitigation: keep `SemanticOverrideLane` a small, named union the implementation card can widen without a contract break; treat the lane choice as *advisory input*, never a routing command, so BR-004 stays the routing authority. Flagged as a coordination risk, not a doctrine risk.
- **`MetadataEvent.codeFamily` is a closed union — extending it touches META-001's file.** The implementation card adds `'semantic_override'` to the union in `src/features/metadata/moveMetadataLedger.ts`. META-001's existing tests (`metadataEvents.test.ts`, `metadataDoctrineAnchors.test.ts`) may assert the exact `codeFamily` set; the implementation card updates them in the same change. This is the only file outside `semanticOverride/` + `gameCopy.ts` + the sidecar that the implementation card edits. Low risk, but called out so it is not a surprise.
- **Modal-vs-inline pull.** It is tempting to make the choice surface a blocking modal so the user "must decide". That would violate doctrine constraint 1 (a semantic suggestion is never a hard gate). The design fixes it as a non-blocking sidecar panel; reviewers of the implementation card must hold this line.
- **Scope creep toward training.** A future card might want to feed the repeated-override count into model tuning. That is explicitly out of v1 scope (card body acceptance criterion + operator decision §16). The `RepeatedOverrideSignal` shape is deliberately in-memory and per-room so it *cannot* accidentally become a training input; reviewers must keep it that way.
- **Own-bubble re-lane permission.** §11 proposes the author may re-lane their own move. If the GAME-004 own-bubble action set turns out to forbid this, the fallback (`answers_parent` assertion only on own bubble) is documented. The implementation card confirms against GAME-004 before finalizing.
- **`gameCopy` is test-enforced.** Adding override codes without adding mappings is an existing test failure (`test-discipline`, doctrine constraint 10). The implementation card must extend `PLAIN_LANGUAGE_COPY` in the same change. Low risk — well-trodden path (LIFE-001, META-001, MCP-003, MCP-008 all did it).
- **No migration / no operator deploy** — pure-TS model + a React component + an in-memory record. The override store is in-memory like META-001's manual-tag store; persistence is a future card. Low operational risk.
