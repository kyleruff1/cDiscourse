## 0. How this design was built

This design is **derived, not asserted.** It is built in three grounded passes,
and the model falls out as the consequence:

1. **The storyboard frames** (§2) — Steps 9–10 of the band-rent scenario
   replayed: "Ask for source" → an evidence-debt marker appears on the node →
   the room status line reads "Evidence requested" → B supplies the message →
   the marker flips to resolved. Every field of the debt record exists because a
   frame demanded it.
2. **The three-axis separation** (§4) — the doctrine, already locked by QOL-037
   §4, that an evidence object carries three *independent* axes: existence /
   source-chain (EV-001), **obligation / evidence debt (EV-003 — this card)**,
   and applicability (QOL-037). EV-003 owns the middle axis and must never
   collapse it into the other two.
3. **The render-time derivation** (§5–§6) — the decision, locked by EV-001
   (adapter-only, path b) and QOL-037 (render-time applicability status, no
   migration in v1), that EV-003 derives debts from existing `arguments` rows at
   read time. The persistence question is deferred to a named follow-up (§14).

Where the storyboard and a prior design disagree, **the storyboard wins** — it
is the binding source per `docs/ux-storyboards/`. EV-003 produces one new
visible chip token (`EvidenceDebtChip`) and one model; everything else is
content plugged into surfaces EV-002 / QOL-032 / the gallery already establish.

This design supersedes the **stub** at this path; it preserves every decision
the stub locked (the status set, the `debtKind` set, the forbidden labels, the
render-time-derived persistence choice) and deepens each to implementation depth.

## 1. Goal & scope

When a participant asks another for a source, quote, receipt, or context, the
room takes on an **unresolved source obligation**. Today nothing tracks that
obligation: there is no per-node marker, no room-status signal, no gallery-level
indicator. A reader cannot tell that a claim is leaning on unbacked assertion,
and the asker cannot tell at a glance whether their request was ever answered.

EV-003 — the **Evidence Debt Tracker** — closes that gap. It introduces a pure-TS
model that, given a room's argument rows (and the EV-001 artifact lists already
derived from them), derives a list of **evidence debts**: structured records of
the shape *"a source was asked for on node X and is owed / supplied / resolved."*
Each debt renders with a plain-language status in four surfaces: the timeline
node (a debt chip), the QOL-032 Inspect popout ("What is unresolved" section),
the conversation gallery card (an "Evidence requested" indicator), and a future
diagnostic package.

**Doctrine framing — the load-bearing sentence of this card.** An evidence debt
is a **structural** fact: *a source was asked for and is owed*. It is **never a
verdict** on who is right. Resolving a debt — even "accepted by both" — proves
at most that a record exists and that both parties stopped asking; it never
declares the original claim true. A debt never declares the original claim
false either: an open debt means "this point lacks X", never "this point is
wrong" (evidence-doctrine "Evidence debt"). The status set, the chip copy, and
the gallery indicator are all phrased as *obligation* observations, never truth
labels.

### In scope

- A pure-TS model `src/features/evidence/evidenceDebtModel.ts` (beside
  `evidenceModel.ts`): the `EvidenceDebt` record interface, the `EvidenceDebtKind`
  enum, the `EvidenceDebtStatus` enum, the render-time derivation
  `deriveEvidenceDebts(input)`, the per-node roll-up `getNodeEvidenceDebtSummary`,
  the room roll-up `getRoomEvidenceDebtSummary`, the debt-chip display contract
  `summarizeEvidenceDebtChip`, the staleness derivation, and the frozen
  enumeration constants.
- An `EvidenceDebtChip.tsx` React Native chip component (a status indicator,
  modeled on EV-002's `ReceiptChip` but for the *obligation* axis — a node can
  show "Source and quote" AND "Source requested" at once).
- A wiring patch to `TimelineNodePopover.tsx` / `timelineNodePopoverModel.ts` so
  the debt chip renders on a node carrying an open debt, beside the EV-002
  receipt chip.
- A wiring patch to QOL-032's Inspect popout §3 ("What is unresolved") so the
  per-node debt list is one of its inputs. (QOL-032 is a *design*, not yet
  built; EV-003 supplies the contract §3 reads and notes the §11 fallback if
  QOL-032 has not landed.)
- A wiring patch to `src/features/debates/conversationGalleryModel.ts` so the
  gallery card carries an `evidenceDebtSummary` and the "Evidence requested"
  signal / `source_trail` lane have a **real data source** (today the lane is
  derived only from `source`-substring *tag counts*, which cannot tell open
  from resolved — EV-003 adds the resolution dimension).
- The locked plain-language copy for every debt-status chip label, helper, and
  accessibility suffix.
- The doctrine reinforcement that an evidence debt is **advisory** — it never
  blocks an ordinary reply (consistent with the Stage 6.2 advisory-validation
  doctrine), never emits a `PointStandingDelta` on its own, and never feeds heat.

### Out of scope (own cards — see §15)

- **Evidence file upload + storage** — the binary screenshot, a storage bucket,
  EXIF stripping. EV-001 and QOL-036 already name this as a separate deferred
  card; EV-003 inherits that boundary.
- **Voting on evidence** — forbidden by `cdiscourse-doctrine` §3 (popularity is
  not evidence) and the v1 scope ban on voting.
- **Any AI assessment of evidence quality** — no Anthropic / xAI / OpenAI / X
  API call, in the app or anywhere. Debt derivation is deterministic pure TS.
- **A persisted `evidence_debt` table / migration** — v1 derives debts at render
  time from existing `arguments` rows + the `ask_source` move type + EV-001
  artifacts. The persistence follow-up is named in §14.
- **The applicability dispute flow** — `applicability_disputed` status, the
  seven structured evidence responses, the `dispute_applicability` choice. All
  **QOL-037**. EV-003's `challenged` status is *fed by* QOL-037's
  `dispute_applicability` / `request_source` choices but EV-003 does not
  implement the choice set.
- **The `ask_source` / `ask_quote` box UI** — the one-box composer's box types,
  the short-prompt schema. **QOL-030**. EV-003 *reads* the moves those boxes
  produce; it does not build the box.
- **Auto-wiring evidence debt into the point-standing engine** — EV-003 emits no
  `PointStandingDelta`. The `OpenIssueDebt` ledger in `src/features/pointStanding/`
  is a *separate* obligation model (axis-pressure debt, not evidence-source
  debt); §4.1 explains why EV-003 does not reuse or mutate it.
- **An admin path to mark a debt `stale` or force-resolve it** — EV-003 derives
  `stale` deterministically from a dormancy threshold; no admin override UI.
- **A risk classifier** that promotes `EvidenceRisk` off `unknown` — EV-001
  reserved that field for "an EV-003-era classifier"; on review, EV-003 keeps
  `EvidenceRisk` derivation out of scope (a debt is an obligation, not a risk
  band) and leaves the field for a later card. §16 records this as an open
  question resolved toward "not now".

## 3. Problem statement

EV-001 modelled the evidence **artifact** (the receipt). EV-002 gave it an
inspection surface (the source-chain popover) and a way to *ask* — the `source` /
`quote` composer presets that seed a non-accusatory request body. But asking is
only half a loop. Today:

- **The request leaves no trace.** When A posts an "Ask for source" move
  (a `clarification_request` carrying the `source_request` tag, per EV-002's
  `quickActionToPreset('source', …)`), the move is just another node in the
  tree. Nothing records that the *parent* node now carries an unmet obligation.
  A reader scrolling the timeline sees a question, not a standing debt.
- **There is no resolution signal.** When B later attaches the evidence, nothing
  connects B's new artifact to A's earlier request. The asker cannot tell at a
  glance whether their request was ever answered; the loop never visibly closes.
- **The room has no obligation status.** The storyboard's Step 9 requires the
  room status line to read "Evidence requested." There is no per-room roll-up of
  open debts to drive that line.
- **The gallery's "Evidence requested" lane has no honest data source.** The
  conversation gallery already has a `source_trail` lane and a `source_chain_fight`
  bucket, but they are derived from *tag counts* (`tagCode.includes('source')`).
  A tag count cannot distinguish "a source was asked for and is still owed" from
  "a source was asked for and supplied two moves later." The lane therefore
  surfaces rooms that have *no live obligation at all*.
- **QOL-032's "What is unresolved" section has a named-but-missing input.**
  QOL-032 §3.2 lists "open axes / evidence debt (LIFE-001 + EV-003)" as the
  source for its §3 section. EV-003 is that source; without it, §3 has nothing
  to render for the evidence-debt half.

The cost of not fixing this: the storyboard's central Step 9–10 loop ("Ask for
source" → marker → resolved) is impossible to render, the gallery mis-sorts
rooms into the source-trail lane, and a reader cannot tell an unbacked
assertion from a backed one.

**What EV-003 is *not* solving.** EV-003 does not decide whether a supplied
source is *good*. It records that a source was *asked for* and that *something
was later attached in response*. Whether that something anchors the trail is
EV-001's `sourceChainStatus` axis; whether it applies to the right month is
QOL-037's applicability axis. EV-003 owns one question only: **is the obligation
still open?**

## 4. The three axes — what EV-003 owns and does not own

An evidence object carries **three independent status axes**. QOL-037 §4 locked
this separation; EV-003 restates it because EV-003 owns the middle axis and the
implementer must never collapse it into the other two.

| Axis | Type | Owner | Question it answers |
|---|---|---|---|
| **Existence / source-chain** | `SourceChainStatus` (EV-001) — `no_source` / `unverified` / `source_no_quote` / `source_and_quote` / `broken` / `primary_present` | EV-001, EV-002 | Does an inspectable record exist, and how anchored is its trail? |
| **Obligation (evidence debt)** | `EvidenceDebtStatus` (EV-003 — **this card**) — `requested` / `supplied` / `challenged` / `accepted_by_participant` / `accepted_by_both` / `unresolved` / `stale` / `branched` | **EV-003** | Has a source / quote / receipt / context been *asked for*, and is it still owed? |
| **Applicability** | `ApplicabilityStatus` (QOL-037) — `applicability_undisputed` / `applicability_disputed` / `applicability_supported` | QOL-037 | Does the evidence apply to *what the submitter says it applies to* (which month / which obligation)? |

These never collapse into one another:

- A node can be `source_and_quote` on the existence axis (a screenshot is
  inspectable) **and** still carry a `requested` evidence debt — because the
  *quote* attached does not answer a request for a *different* source. A clear
  trail on one artifact does not discharge an obligation opened against another
  claim.
- A debt can reach `accepted_by_both` **without** the artifact ever becoming
  `primary_present` — the parties can simply stop asking; "both stopped asking"
  is not "the chain reached a primary record."
- **Heat never feeds the debt axis.** None of the eight `EvidenceDebtStatus`
  values is a truth label, and none is influenced by reply velocity, view count,
  or participant count. A debt's status is a pure function of the *moves* that
  open and answer it.

> **Doctrine edge case the model must encode (stub Open Question 1 + QOL-037 §4
> pattern):** "what if a strong source chain auto-resolves an open debt? — it
> does not." `source_and_quote` on artifact A says artifact A is inspectable; it
> says nothing about whether the *request* B made was answered. The two axes are
> derived independently and never read each other's *status*. EV-003's
> derivation reads EV-001's artifact *list* (to find the resolving artifact) but
> never reads EV-001's `sourceChainStatus` *as a resolution signal* — except in
> the one explicit, documented place (§6.2) where a debt's *kind* names a
> source-chain threshold as its close-condition, exactly as EV-001's
> "How EV-003 will consume EV-001" section specified.

### 4.1 Why EV-003 does NOT reuse `OpenIssueDebt` from `src/features/pointStanding/`

The point-standing module already has an `OpenIssueDebt` interface
(`src/features/pointStanding/types.ts`). It is tempting to reuse it. EV-003 does
**not**, for three reasons:

1. **Different obligation.** `OpenIssueDebt` is *axis-pressure* debt — it is
   opened when a reply applies pressure along a `DisagreementType` axis (scope,
   definition, causal…) and is closed by a *repair move* (concession, narrowing,
   synthesis). EV-003's `EvidenceDebt` is *source* debt — opened by an explicit
   "ask for a source/quote/receipt/context" move and closed by *attaching the
   asked-for artifact*. A concession does not resolve an evidence debt; a
   supplied screenshot does not resolve an axis-pressure debt.
2. **Different scoring posture.** `OpenIssueDebt` carries credit bookkeeping
   (`recoveryCreditAwarded`, `explicitConcessionAwarded`) and drives
   `PointStandingDelta`s. EV-003 emits **no** `PointStandingDelta` — an evidence
   debt is a *visible obligation marker*, not a scoring event. (The
   anti-amplification interaction is one-directional: see §9.)
3. **Different lifecycle.** `OpenIssueDebt` is a stateful ledger entry the
   point-standing engine mutates. EV-003's `EvidenceDebt` is **render-time
   derived** — it is recomputed from `arguments` rows on every read and never
   stored in v1 (consistent with EV-001 path b and QOL-037).

The two models are siblings, not parent/child. The doctrine skill
(`evidence-doctrine` "Evidence debt" vs `point-standing-economy` "How challenges
create debt") describes them as two distinct debt concepts; EV-003 keeps them
distinct in code. `evidenceDebtModel.ts` imports **nothing** from
`src/features/pointStanding/`.

> **One naming-collision note for the implementer:** the `evidence-doctrine`
> skill's "Evidence debt — the gameplay model" table lists debt *types* as
> `source_needed`, `quote_needed`, `scope_example_needed`, `definition_needed`,
> `mechanism_needed`, `counterexample_needed`, `primary_record_needed`. Those
> seven are the **point-standing** axis-debt types (they are what
> `OpenIssueDebt.axis`-driven debts map to). EV-003's `EvidenceDebtKind` is the
> narrower **source-obligation** set — `source` / `quote` / `receipt` /
> `context` / `primary_record` — the five kinds the *storyboard's evidence-debt
> marker* and the `ask_source` / `ask_quote` box types actually produce. The
> overlap (`source` ⇄ `source_needed`, `primary_record` ⇄ `primary_record_needed`)
> is conceptual, not a shared type. EV-003 deliberately does not adopt the
> `_needed` suffix so the two models never alias.

## 5. Data model

All public types live in a new pure-TS module
`src/features/evidence/evidenceDebtModel.ts`. **No React. No Supabase. No
network. No `Date.now()` in the model** — the derivation takes an injected `nowMs`
for the staleness calculation, exactly as `conversationGalleryModel.ts` does, so
tests stay deterministic.

```ts
/**
 * The kind of source obligation a request opens. A debt names WHAT was asked
 * for. Closed, exhaustive set — no UI surface may invent a sixth kind.
 *
 * NOTE: deliberately NOT suffixed `_needed`. The point-standing module's
 * axis-debt types (`source_needed`, `primary_record_needed`, …) are a DIFFERENT
 * model (see §4.1). EV-003 never aliases them.
 */
export type EvidenceDebtKind =
  | 'source'          // "Where did this come from?" — a link / citation / pointer.
  | 'quote'           // "Quote the exact passage." — a verbatim excerpt of a known source.
  | 'receipt'         // "Show the record itself." — a screenshot / document / transaction note.
  | 'context'         // "Show the surrounding context." — the message/thread a claim leans on.
  | 'primary_record'; // "Reach a primary record." — the strongest ask; an inspectable original.

/**
 * The lifecycle status of one evidence debt. Strictly an OBLIGATION
 * observation — never a verdict. None of these values says the claim is
 * true or false; they say whether a source request is still owed.
 *
 *   requested              a request move exists; nothing has been attached in answer yet.
 *   supplied               a later move attached an artifact that answers the request,
 *                          but neither the asker nor both parties have acknowledged it.
 *   challenged             the asker (or a primary) contested the supplied artifact —
 *                          fed by QOL-037's `dispute_applicability` / a follow-up ask.
 *                          The obligation is live again.
 *   accepted_by_participant  ONE primary participant explicitly accepted the supplied artifact.
 *   accepted_by_both       BOTH primary participants explicitly accepted it. The obligation
 *                          is discharged. (User label "settled" — never "case closed".)
 *   unresolved             a request move exists, the asked party explicitly declined /
 *                          evaded, OR the debt has been open and contested long enough that
 *                          the model summarises it as "still unresolved". Distinct from
 *                          `stale` (which is silence, not refusal).
 *   stale                  the debt has been open with NO answering or accepting move past
 *                          the dormancy threshold (§6.3). Advisory only — never auto-acts.
 *   branched               the obligation moved onto a side-branch (a `split_branch` move
 *                          off the requested node). The debt is not closed; it is relocated.
 */
export type EvidenceDebtStatus =
  | 'requested'
  | 'supplied'
  | 'challenged'
  | 'accepted_by_participant'
  | 'accepted_by_both'
  | 'unresolved'
  | 'stale'
  | 'branched';

/**
 * One evidence debt. Render-time-derived in v1 (no own table); a later
 * persistence card (§14) would store rows of this exact shape.
 *
 * DOCTRINE: this record is an OBLIGATION marker. It carries no truth value,
 * emits no PointStandingDelta, and is never read by the heat model.
 */
export interface EvidenceDebt {
  /**
   * Stable deterministic id. The derivation mints:
   *   `<requestArgumentId>:debt`
   * One request move opens at most one debt (§6.1 rule 2), so the request
   * move's id uniquely identifies the debt across reads.
   */
  id: string;
  /** The room this debt lives in. FK to public.debates.id (internal table name). */
  debateId: string;
  /**
   * The node the debt is ATTACHED to — the move whose claim now carries an
   * unmet source obligation. This is the PARENT of the request move (the asker
   * asked *about* the parent), falling back to the request move itself when the
   * request has no parent. See §6.1 rule 3.
   */
  nodeId: string;
  /** The request move that OPENED the debt. */
  requestArgumentId: string;
  debtKind: EvidenceDebtKind;
  /** The user who opened the debt (author of the request move). May be null if the row's author is unknown. */
  requestedByUserId: string | null;
  /** ISO-8601 — copied from the request move's created_at. */
  requestedAt: string;
  status: EvidenceDebtStatus;
  /**
   * The later move that supplied / resolved the debt, when one exists.
   * `undefined` while the debt is still `requested` / `stale`.
   * For `supplied` / `accepted_*` this is the move that attached the answering
   * artifact; for `branched` it is the `split_branch` move.
   */
  resolvedByNodeId?: string;
  /** ISO-8601 of the resolving move, when resolved. `undefined` otherwise. */
  resolvedAt?: string;
  /**
   * Whole-day age of the debt at derivation time (from `requestedAt` to the
   * injected `nowMs`). Exposed so the chip / Inspect can show "open 6 days"
   * without re-deriving. 0 for a debt opened today.
   */
  ageDays: number;
  /**
   * True when `status === 'stale'`. Duplicated as a boolean for cheap
   * filtering in the gallery roll-up. (Redundant with `status` — kept for
   * the same reason `ConversationGalleryCard.hasNoRebuttal` duplicates a
   * derivable fact: it is a hot filter key.)
   */
  isStale: boolean;
}

/** Per-node roll-up — what the timeline node + Inspect §3 render. */
export interface NodeEvidenceDebtSummary {
  nodeId: string;
  /** Every debt attached to this node, in `requestedAt` order. */
  debts: ReadonlyArray<EvidenceDebt>;
  /** Count of debts whose status is "still owed" (`requested` / `challenged` / `unresolved` / `stale`). */
  openCount: number;
  /** Count of debts that are discharged (`accepted_by_both`). */
  settledCount: number;
  /** True when openCount > 0 — drives whether the node renders a debt chip. */
  hasOpenDebt: boolean;
  /**
   * The single status the node's chip displays when multiple debts are
   * attached — the worst-status-wins pick (§6.4 ordering). `null` when the
   * node has no debts at all (no chip rendered).
   */
  chipStatus: EvidenceDebtStatus | null;
}

/** Per-room roll-up — what the gallery card + the room status line render. */
export interface RoomEvidenceDebtSummary {
  debateId: string;
  /** Total debts in the room across all nodes and all statuses. */
  totalCount: number;
  /** Debts that are still owed (`requested` / `challenged` / `unresolved` / `stale`). */
  openCount: number;
  /** Debts that are `stale` specifically (subset of openCount). */
  staleCount: number;
  /** Debts that are `accepted_by_both`. */
  settledCount: number;
  /**
   * True when openCount > 0 — drives the gallery "Evidence requested"
   * indicator and the storyboard's "Evidence requested" room status line.
   */
  hasOpenEvidenceDebt: boolean;
  /**
   * Plain-language one-liner for the room status line / gallery card.
   * e.g. "Evidence requested" (openCount > 0) or "" (no open debt).
   * Locked copy — see §8.
   */
  statusLine: string;
}

/** The debt-chip display contract — the EvidenceDebtChip component binds to this. */
export interface EvidenceDebtChipContract {
  /** Plain-language label, ≤ 28 chars. Never a snake_case code, never a verdict token. */
  label: string;
  /** Plain-language helper, ≤ 90 chars. Empty string when not applicable. */
  helper: string;
  /** Logical tone — the renderer maps this to a color token. NOT a truth label. */
  tone: 'neutral' | 'info' | 'attention' | 'muted';
  /**
   * False for a node with no open debt — the chip is not rendered then
   * (no clutter on a node that carries no obligation).
   */
  isVisible: boolean;
  /** The underlying status — kept so consumers branch without re-deriving. Never rendered raw. */
  status: EvidenceDebtStatus;
  /** The debt kind, for the chip's kind word ("source", "quote", …) rendered in plain language. */
  debtKind: EvidenceDebtKind;
  /** Plain-language accessibility label, the full sentence a screen reader announces. */
  accessibilityLabel: string;
}
```

## 6. API / interface contracts

Every export below is a stable contract. QOL-032's Inspect popout, the gallery
model, and EV-004 will all consume it; locking it here keeps them aligned.

### 6.1 `deriveEvidenceDebts` — the core derivation

```ts
/**
 * Derive every evidence debt in a room from its argument rows. Pure,
 * deterministic, side-effect-free. The single entry point for EV-003.
 *
 * Returns debts in `requestedAt` ascending order (ties broken by
 * `requestArgumentId` string order) so the output is stable across reads.
 */
export function deriveEvidenceDebts(
  input: DeriveEvidenceDebtsInput,
): ReadonlyArray<EvidenceDebt>;
```

**Derivation rules** (the implementer follows these exactly):

1. **A request move opens a debt.** A move opens a debt iff it carries a
   recognised request tag code. The recognised codes and the kind they map to:

   | Tag code on the move | `debtKind` |
   |---|---|
   | `source_request` | `source` |
   | `quote_request` | `quote` |
   | `receipt_request` | `receipt` |
   | `context_request` | `context` |
   | `primary_record_request` | `primary_record` |

   These are exactly the tag codes EV-002's `quickActionToPreset` already emits
   for `source` / `quote` (`source_request`, `quote_request`) plus three new
   codes EV-003 reserves for the `receipt` / `context` / `primary_record` kinds.
   The three new codes are **derived-from, not created-by** EV-003 — they are
   tag *strings* the QOL-030 `ask_source` box (its kind selector) and a future
   EV-004 affordance will attach; EV-003 only *reads* them. If none of the five
   codes is present, the move opens **no debt** (a plain reply, a clarification
   that does not ask for a source, etc.). This is the doctrine guard: a move
   that just expresses disagreement without naming an obligation opens nothing.
2. **One debt per request move.** A request move opens at most one debt, even
   if it somehow carries two request tags — the kind is the *first* match in the
   table order above (`source` > `quote` > `receipt` > `context` >
   `primary_record`). The debt id is `<requestArgumentId>:debt`. This keeps ids
   stable and the count honest (no double-counting a single ask).
3. **The debt attaches to the asked-about node.** `nodeId = requestMove.parentId
   ?? requestMove.id`. The asker asked *about* the parent; the obligation
   belongs on the parent. A request move with no parent (a root-level ask —
   unusual but possible) attaches the debt to itself.
4. **Resolution is by a later move** (§6.2). After every debt is opened, the
   derivation scans moves *created after* `requestedAt`, in chronological order,
   for the resolving / accepting / challenging / branching move and sets
   `status` + `resolvedByNodeId` + `resolvedAt` accordingly.
5. **Staleness is derived last** (§6.3) from `nowMs` and `requestedAt`, and only
   for debts still in `requested` (never for `supplied` / `accepted_*` / etc.).
6. **`ageDays`** is `floor((nowMs - Date.parse(requestedAt)) / 86_400_000)`,
   clamped to `>= 0` (a request "in the future" relative to a stale injected
   clock yields 0, never negative).
7. **Determinism.** Same input → deeply-equal output. No `Date.now()`, no
   `Math.random()`, no iteration over a `Set` whose order is insertion-defined
   in a way the input does not control.

### 6.2 The resolution scan — `requested` → `supplied`/`challenged`/`accepted_*`/`branched`

For each open debt, the derivation examines the moves created after it, in
chronological order, and applies the **first** matching rule. The scan is scoped
to moves *in the same room*; a move resolves a debt when it is plausibly an
answer to that debt — defined precisely below.

A later move **M** is considered an *answer* to debt **D** when **M** is a
descendant of, or a direct reply to, `D.nodeId` **or** `D.requestArgumentId`
(the asker asked on / under the requested node; the answer lands on / under the
same subtree). This descendant check uses the `parentId` chain already present
in the input — no new traversal primitive; EV-003 reuses the same parent-walk
shape `buildArgumentTree` uses.

| Condition on a later answering move **M** | Resulting `EvidenceDebt.status` | `resolvedByNodeId` |
|---|---|---|
| **M** carries ≥ 1 `EvidenceArtifact` whose `kind` matches the debt kind's close-condition (table below) | `supplied` | `M.id` |
| **M** carries a QOL-037 `evidenceResponse` with `choice === 'accept'` and the responder is a primary participant — **and** this is the second distinct primary to accept | `accepted_by_both` | `M.id` |
| …same, but only one distinct primary has accepted so far | `accepted_by_participant` | `M.id` |
| **M** carries a QOL-037 `evidenceResponse` with `choice === 'dispute_applicability'` or `choice === 'request_source'`, targeting the supplied artifact | `challenged` | `M.id` |
| **M** is a `split_branch` move (`moveKindHint === 'split_branch'` or argument type indicates a branch) off `D.nodeId` | `branched` | `M.id` |
| **M** is an explicit decline / evasion of the request — detected ONLY via a recognised tag code (`source_declined` / `request_evaded`), never via free-text NLP | `unresolved` | `M.id` |

**Close-condition by debt kind** — what `kind` of artifact discharges a debt
(this is the EV-001 contract "EV-003 will use `kind` + `sourceChainStatus` as the
close-condition", made concrete):

| `debtKind` | An answering move's artifact discharges it when the artifact's `kind` is… | …and its `sourceChainStatus` is at least… |
|---|---|---|
| `source` | `url`, `source_text`, `dataset`, or `manual_citation` | `source_no_quote` (a source pointer exists) |
| `quote` | any kind **with a non-empty `quote` field** | `source_and_quote` (a quote tied to a source) |
| `receipt` | `screenshot_redacted` or `payment_screenshot` (QOL-036) or `source_text` | `unverified` (an inspectable record exists at all) |
| `context` | any kind with a non-empty `sourceText` or `quote` | `unverified` |
| `primary_record` | any kind | `primary_present` (admin-set; the strongest bar) |

The `sourceChainStatus` threshold is the **one** place EV-003 reads EV-001's
existence axis — and it reads it strictly as a *close-condition input named by
the debt's own kind*, never as a generic "the chain is good so the debt is
done" shortcut (the §4 doctrine edge case). A `quote` debt is not discharged by
a bare URL with no quote; a `primary_record` debt is not discharged until an
admin has marked an artifact `primary_present`.

**Status precedence when a debt has multiple later events.** The scan applies
the **last** event chronologically that changes the status, with one exception:
once `accepted_by_both` is reached it is **terminal** unless a *later*
`dispute_applicability` / `request_source` reopens it (→ `challenged`) — mirror
of QOL-037's monotonic-with-reopen ladder. So the realistic sequences are:

- `requested → supplied → accepted_by_participant → accepted_by_both` (the happy
  storyboard loop, Steps 9–10–13).
- `requested → supplied → challenged → supplied → accepted_by_both` (a contested
  supply that is re-answered).
- `requested → unresolved` (the asked party declines/evades).
- `requested → branched` (the obligation is split onto a side-branch).
- `requested → stale` (silence past the threshold — §6.3).

### 6.3 Staleness derivation

```ts
/** Dormancy threshold: a `requested` debt with no answering move for this many
 *  whole days is summarised as `stale`. Advisory only — never auto-acts. */
export const STALE_DEBT_THRESHOLD_DAYS = 7;
```

A debt is `stale` iff **all** of:

- its status (before staleness is applied) is still `requested` — never
  `supplied` / `accepted_*` / `challenged` / `branched` / `unresolved`;
- `ageDays >= staleThresholdDays` (default `STALE_DEBT_THRESHOLD_DAYS`).

`stale` is a *re-labelling* of a still-open `requested` debt, not a new
lifecycle branch — it tells the reader "this has been owed a while" so the
gallery can surface it and the Inspect popout can mention it. Per stub Open
Question 1, **a `stale` debt never auto-advises a branch and never auto-acts** —
it only changes the chip copy and the gallery `staleCount`. The decision to
branch, re-ask, or drop it stays entirely with the user.

**Worst-status ordering** for the node chip when multiple debts are attached
(most-attention-demanding first):

```
challenged > unresolved > stale > requested > branched > supplied
           > accepted_by_participant > accepted_by_both
```

`challenged` wins because a contested supply is the most live obligation;
`accepted_by_both` ranks last because it demands no attention. The node chip
shows the worst; the Inspect popout (§7) lists *all* debts so nothing is hidden.

### 7.3 Conversation gallery — the "Evidence requested" indicator

The conversation gallery (`conversationGalleryModel.ts`) already has a
`source_chain_fight` bucket and a `source_trail` lane — but they are derived from
**tag counts** (`tagCode.includes('source')` → `sourceChainHits`), which cannot
tell an *open* debt from a *resolved* one. EV-003 adds the resolution dimension.

EV-003 extends `ConversationGalleryCard` with one new field:

```ts
/** EV-003 — per-room evidence-debt roll-up. Drives the "Evidence requested"
 *  indicator and gives the source-trail lane an honest data source. */
evidenceDebtSummary: RoomEvidenceDebtSummary;
```

and `buildGallery` computes it by:

1. running `deriveEvidenceDebts` over each debate's `GalleryArgumentInput[]`
   (the gallery already fetches every room's argument rows in one batched
   `in()` query — `useGalleryArguments`; EV-003 reuses that data, no new fetch);
2. `getRoomEvidenceDebtSummary` to roll the debts up;
3. attaching the summary to the card.

The card's existing `signals` array gains one entry when
`evidenceDebtSummary.hasOpenEvidenceDebt` is true:

```
{ code: 'evidence_debt_open', label: 'Evidence requested', tone: 'warning' }
```

— and a second when `staleCount > 0`:

```
{ code: 'evidence_debt_stale', label: 'Source still owed', tone: 'neutral' }
```

The `source_chain_fight` **bucket** derivation (`deriveBucket`) is upgraded:
today it fires on `sourceChainHits >= 1 && (sourceChainRisk high/medium || …)`.
EV-003 makes the *open-debt* count the primary signal — a room with an
`evidenceDebtSummary.openCount > 0` is a genuine source-trail fight; a room
whose `source` tags all resolved is **not** routed into the lane. This is a
precision fix: the lane stops surfacing rooms with no live obligation.

The `GalleryArgumentInput` shape gains the two fields EV-003's derivation needs
that are not already present: `tagCodes` is **already** available to the gallery
(it builds `tagsByArgumentId`); `artifacts` is derived from
`clientValidation.attachedEvidence` via EV-001's `buildEvidenceArtifacts` — the
gallery loader adds that one call (the same call `ArgumentGameSurface` already
makes). No new query, no new prop drilling beyond the loader.

> The gallery indicator is the card's Acceptance criterion: *"a gallery-level
> indicator marks rooms that carry an unresolved evidence debt — the 'Evidence
> requested' lane has a real data source."* EV-003 satisfies it by making the
> open-debt count the lane's driving signal.

- **Who can open an evidence debt.** Any participant who can post an "Ask for
  source / quote / receipt / context" move — i.e. any primary participant (and,
  per EV-002's observer contract, *not* an observer: observers see the "ask" CTA
  disabled with "Join a side to ask"). EV-003 adds **no new permission**; it
  rides the existing composer + role gates. The debt is simply *derived* from
  the request move that the existing gates already governed.
- **Who can resolve a `source` / `quote` debt.** Per the stub: a `source` /
  `quote` debt is resolved by *attaching the asked-for artifact*. In practice the
  resolving move is most often by the move's **author** (the person whose claim
  was asked about) — but EV-003's derivation does **not** restrict the resolving
  author. Anyone who can post an evidence-bearing move under the requested
  subtree can supply; the model records *who* via `resolvedByNodeId`'s author,
  and the UI shows it. (Restricting "only the author may supply" is a
  composer-permission decision, not a derivation decision; EV-003 keeps the
  derivation permissive and leaves any tightening to the composer.)
- **"Accepted by both" requires both primaries.** `accepted_by_both` is reached
  only when two *distinct* primary participant user ids have each posted an
  `accept` evidence-response targeting the supplied artifact. When
  `primaryParticipantUserIds` has fewer than two entries the model can still
  reach `accepted_by_participant` but never `accepted_by_both` (edge case §10.9).
- **A debt is advisory — it never blocks a reply.** This is doctrine-critical
  (`cdiscourse-doctrine` §1; the Stage 6.2 advisory-validation rule). An open
  evidence debt **never** appears in a blocking-error path: it is not a flag, it
  is not consumed by `evaluateArgumentDraft`, it cannot disable the Post button.
  A node with five open debts is just as postable-onto as a node with none. A
  §11 test (`'an evidence debt never appears in a blocking-error path'`) asserts
  the model exports nothing that `evaluateArgumentDraft` imports and that no debt
  status maps to a validator error code.
- **No point-standing delta from EV-003.** Opening or resolving a debt emits
  **no** `PointStandingDelta`. The point-standing engine is untouched. Standing
  moves only when a *move* does something the engine already grades (concession,
  narrowing, sourcing, synthesis, evasion).
- **Anti-amplification stays intact — one-directional.** The `evidence-doctrine`
  anti-amplification rule says an amplified-but-unsourced claim earns *engagement
  credit* but not *factual-standing credit* until evidence arrives. EV-003's open
  evidence debt is the **visible marker** of exactly that state ("a source is
  owed"). The interaction is one-directional: the anti-amplification gate in
  `src/features/pointStanding/antiAmplification.ts` is what *suppresses standing*;
  EV-003 only *displays* the obligation. EV-003 does not feed, strengthen, or
  weaken the anti-amplification gate — it is a read-side mirror of the same
  doctrine, not a second enforcer. (If a future card wants the gate to *read*
  EV-003's open-debt count as a suppression input, that is that card's decision;
  EV-003 ships them decoupled.)
- **RLS.** EV-003 adds no table and no policy. A debt is derived from
  `arguments` rows; it is visible exactly when those rows are. In the band-rent
  storyboard the room is *private* — the existing private-room RLS (QOL-039's
  territory) is what keeps the debt private. EV-003 relies on it; it does not
  re-implement it. The derivation performs no auth check of its own (consistent
  with EV-001 / EV-002).

## 10. Edge cases

The implementer must handle each. Tests in §11.

1. **A room with no request moves.** `deriveEvidenceDebts` returns `[]`.
   `getRoomEvidenceDebtSummary` → `totalCount 0`, `hasOpenEvidenceDebt false`,
   `statusLine ''`. The gallery card shows no evidence-debt signal; the
   `source_chain_fight` bucket does not fire on debt grounds. No crash.
2. **A request move with no parent (a root-level "Ask for source").** Unusual
   but valid. `nodeId = requestMove.id` (the debt attaches to itself, rule 3).
   Everything else derives normally.
3. **A request move carrying two request tags** (`source_request` *and*
   `quote_request`). One debt only (rule 2); kind is the first in precedence
   order (`source`). The count is not inflated.
4. **A request move whose parent was soft-deleted** (`is_deleted = true`). The
   parent row may be absent from the input or present with a deleted flag. The
   debt still attaches to `parentId` (the id is stable even if the row is
   hidden); the chip renders on whatever the timeline shows for that node. EV-003
   does not resurrect or special-case deleted nodes — it derives from whatever
   rows the caller passed (the caller already applies the room's soft-delete
   filter).
5. **An answering move attached *before* the request** (clock skew, or an
   artifact attached then asked-about). The resolution scan only considers moves
   with `createdAt > requestedAt`. An earlier artifact does **not** resolve a
   later request — a request is answered by a *later* move, never retroactively
   (the storyboard + `evidence-doctrine` "resolved by later moves" rule).
6. **An answering move with the wrong artifact kind.** A `quote` debt is *not*
   discharged by a move attaching only a bare `url` (no quote) — the
   close-condition table requires `source_and_quote`. The debt stays `requested`.
   The reader correctly still sees an open quote obligation.
7. **Multiple debts on one node.** The node's `NodeEvidenceDebtSummary.debts`
   lists all of them; `chipStatus` is the worst-status pick (§6.4). Inspect §3
   lists every debt; the timeline chip shows the worst. Nothing is hidden.
8. **The same artifact answers two debts.** Possible — one screenshot answers
   both a `source` and a `receipt` ask on the same subtree. Each debt is
   evaluated independently; both can reach `supplied`. The artifact is not
   "consumed" by the first match.
9. **`accepted_by_both` with fewer than two known primaries.** When
   `primaryParticipantUserIds` is undefined / has one entry, the model caps the
   status at `accepted_by_participant` even if two `accept` responses exist from
   ids it cannot confirm are *primaries*. Documented degradation — never a false
   `accepted_by_both`.
10. **A `dispute_applicability` response after `accepted_by_both`.** The debt
    reopens to `challenged` (the §6.2 monotonic-with-reopen exception). The
    gallery `openCount` rises again. This is correct — a settlement can be
    reopened by a fresh dispute.
11. **A `stale` debt that then receives an answer.** Once a `supplied` /
    `accepted_*` / `branched` move lands, the debt is no longer `requested`, so
    the staleness re-label does not apply — it takes the resolved status. `stale`
    is only ever a label on a *still-`requested`* debt.
12. **Clock skew making `nowMs` earlier than `requestedAt`.** `ageDays` clamps
    to `0` (rule 6); the debt is not `stale` (age `0 < threshold`). No negative
    age, no crash.
13. **A `split_branch` move off the requested node.** The debt → `branched`,
    `resolvedByNodeId` = the branch move. The obligation is *relocated*, not
    *closed* — `branched` is **not** in `OPEN_EVIDENCE_DEBT_STATUSES`, so it does
    not inflate `openCount`, but it is also not `settledCount`. It is a quiet
    "see the branch" state. (Open question §16.3 records whether `branched`
    should count as open — leaning *no*.)
13b. **QOL-037 not yet built — `evidenceResponses` absent.** The derivation
    degrades gracefully: with no `evidenceResponse` records it can still reach
    `requested` / `supplied` / `stale` / `branched` (those depend only on tags +
    artifacts + branch moves), but it cannot reach `challenged` /
    `accepted_by_participant` / `accepted_by_both` (those need the response
    records). A `supplied` debt simply stays `supplied`. No crash; the model is
    consumable before QOL-037 ships (§11 fallback). When QOL-037 lands, the
    acceptance / challenge statuses light up with no EV-003 code change.
14. **Concurrent edits.** Not applicable — the model is pure and read-only over
    already-fetched rows; the derivation has no I/O. Two readers deriving the
    same room get the same debts.
15. **Offline / network failure.** Not applicable — the model does not fetch.
    The caller's existing offline handling for the room load applies unchanged.
16. **Permission-denied.** RLS on `public.arguments` (and the private-room RLS)
    already gates which rows the caller fetched. The model derives only from
    rows it was given; it performs no auth check.
17. **A verdict token in a user's request body** ("show me the proof you're not
    a liar"). EV-003 stores no user free text in the debt record — the debt
    carries `requestArgumentId`, not the request *body*. The ban-list governs
    only EV-003's **system-generated** strings (chip labels, helpers, status
    line, accessibility labels). The user's body is rendered by the existing
    timeline, exactly as today; EV-003 does not sanitise it (consistent with
    EV-001 edge case 13).
18. **A very old room with dozens of debts.** `deriveEvidenceDebts` returns them
    all (no cap). The roll-ups sum honestly. The gallery shows one indicator
    regardless of count; the Inspect popout lists per-node, so per-node counts
    stay small. No pagination needed in v1.
19. **A request move that is itself later soft-deleted.** If the request row is
    absent from the input, its debt is simply not derived (no orphan debt). If
    present-but-deleted, the debt still derives (the asker did ask); the caller's
    timeline decides whether to show the deleted request node. EV-003 does not
    special-case this — it derives from the rows passed.

### `evidenceDebtModel.test.ts`

**Shape + enum coverage**
- `EvidenceDebtKind` union has exactly the five documented values;
  `ALL_EVIDENCE_DEBT_KINDS` is a frozen array covering them.
- `EvidenceDebtStatus` union has exactly the eight documented values;
  `ALL_EVIDENCE_DEBT_STATUSES` covers them.
- `OPEN_EVIDENCE_DEBT_STATUSES` is exactly `['requested', 'challenged',
  'unresolved', 'stale']` and is a strict subset of the status union.
- An `EvidenceDebt` literal type-checks with every field at example values; the
  optional `resolvedByNodeId` / `resolvedAt` omitted on a `requested` debt.

**`deriveEvidenceDebts` — opening rule (§6.1)**
- A room with one `source_request`-tagged move → one debt, kind `source`, id
  `<requestId>:debt`, `status 'requested'`, `nodeId` = the request's `parentId`.
- Each of the five request tag codes → the matching `debtKind`.
- A plain reply with no request tag → no debt.
- A move with two request tags → one debt, kind = first in precedence.
- A root-level request (no `parentId`) → debt `nodeId` = the request's own id.
- Determinism: same input twice → deeply-equal output (incl. id + order).

**The resolution scan (§6.2) — one test per row**
- `requested → supplied`: a later move under the subtree carrying a matching
  artifact → `supplied`, `resolvedByNodeId` = that move.
- The close-condition table per kind: a `quote` debt is **not** discharged by a
  bare-`url` artifact (status stays `requested`); it **is** discharged by an
  artifact with a non-empty `quote` and `source_and_quote` status.
- A `primary_record` debt is discharged only by an artifact with
  `sourceChainStatus === 'primary_present'`.
- `supplied → accepted_by_participant`: one primary's `accept` response.
- `accepted_by_participant → accepted_by_both`: the second distinct primary's
  `accept`.
- `supplied → challenged`: a `dispute_applicability` response.
- `accepted_by_both → challenged`: a `dispute_applicability` *after* settlement
  reopens it (the monotonic-with-reopen exception).
- `requested → branched`: a `split_branch` move off the node.
- `requested → unresolved`: a move tagged `source_declined` / `request_evaded`.
- An answering move *before* the request does **not** resolve it (edge case 5).
- The full storyboard happy loop: `requested → supplied → accepted_by_participant
  → accepted_by_both` reproduced from a Steps 9–10–13 fixture.

**Staleness (§6.3)**
- A `requested` debt with `ageDays >= STALE_DEBT_THRESHOLD_DAYS` → `stale`,
  `isStale true`.
- A `requested` debt below the threshold → stays `requested`, `isStale false`.
- A `supplied` debt past the threshold → stays `supplied` (staleness applies
  only to `requested`).
- A custom `staleThresholdDays` is honoured.
- `nowMs` earlier than `requestedAt` → `ageDays` clamped to 0, not stale, no
  negative.

**Roll-ups (§6.4)**
- `getNodeEvidenceDebtSummary` on a node with no debts → `openCount 0`,
  `hasOpenDebt false`, `chipStatus null`, `debts []`.

  `settledCount` sum correctly; `hasOpenEvidenceDebt` true iff `openCount > 0`.

  resolved-by-acceptance; `""` for an empty room.

## 12. Doctrine & safety self-check

Each line confirms a hard constraint is respected by this design:

- **Evidence is never truth; a debt is a structural obligation, not a verdict.**
  Every `EvidenceDebtStatus` is an *obligation* observation (`requested` /
  `supplied` / `settled by both` / …). No status, label, helper, or status line
  is a truth claim. An open debt is rendered as "a source is owed", never "this
  point is false"; a resolved debt is "settled", never "proven" or "case
  closed". The ban-list test enforces it across every produced string.
- **Score / status is gameplay analysis, never truth** (`cdiscourse-doctrine`
  §1). EV-003 produces an obligation marker, not a score. It emits no
  `PointStandingDelta`; it imports nothing from `src/features/pointStanding/`.
- **Score never blocks posting; the debt is advisory** (Stage 6.2 doctrine).
  An evidence debt never enters `evaluateArgumentDraft`, is never a flag, never
  disables the Post button. A named test asserts the model exports nothing a
  blocking path consumes.
- **Popularity is not evidence** (`cdiscourse-doctrine` §3; `evidence-doctrine`).
  The derivation input carries no engagement / view / follower / "verified"
  field. A debt's status is a pure function of *moves*, never of reach. The
  amplification ban-list test enforces it over every string. EV-003 mirrors the
  anti-amplification doctrine (an open debt *is* the visible "unsourced" state)
  but does not let popularity discharge a debt.
- **Heat = activity, not correctness.** EV-003 does not touch the heat model and
  the debt axis never reads heat (§4).
- **AI moderator hard limits — no AI.** EV-003 introduces no Anthropic / xAI /
  OpenAI / X API call, in the client or anywhere. The derivation is pure
  deterministic TypeScript. No AI classifies, values, or resolves a debt.
- **No service-role, no direct insert into `public.arguments`.** EV-003 adds no
  Supabase write path. The write path stays `submit-argument`. No client code
  touches a service-role key. Debts are derived from rows already stored by the
  existing flow.
- **No migration, no RLS change, no Edge Function.** v1 derives debts at render
  time from existing `arguments` rows + tag codes + EV-001 artifacts (adapter-
  only, per EV-001 path b and QOL-037). A debt is visible exactly when its rows
  are — gated by the existing `public.arguments` RLS (and the private-room RLS
  for private rooms).
- **The rules engine stays pure TS and untouched.** `src/lib/constitution/engine.ts`
  is not edited. EV-003's model lives in `src/features/evidence/`, pure TS.
- **Soft-delete only.** EV-003 deletes nothing; it derives over whatever rows the
  caller passes (the caller already applies the room's soft-delete filter).
- **Secrets policy.** No env var, no `.env*` change, no key reference.
- **Plain language; the `debates` table name stays internal.** Every system
  string is plain English; no `snake_case` code surfaces (ban-list +
  `looksLikeInternalCode` test). User-facing copy says "argument" / "room", not
  "debate" and never "game"; the internal `debates` table / `debateId` field
  names stay in code only.
- **v1 scope guards.** No voting (a debt is not a vote — §1 out-of-scope), no
  search, no push (the storyboard's "source requested" notification is QOL-040,
  out of scope — EV-003 only ships the roll-up a future QOL-040 would read), no
  OAuth, no public API.
- **Three-axis separation honoured** (`evidence-doctrine`; QOL-037 §4). EV-003
  owns the obligation axis only; it never reads the existence axis or the
  applicability axis as a resolution shortcut, except the one documented
  kind-named close-condition threshold (§6.2). A doctrine test enforces it.
- **Terminology audit stays green.** `npm run ux:terminology:audit` — the chip
  copy and status line use approved vocabulary.

## 13. Risks

1. **Render-time derivation cost.** `deriveEvidenceDebts` runs per room render
   and, in the gallery, per room across the whole list. The cost is O(N·M) for
   N moves and M debts (each debt scans later moves). For a typical room
   (< 40 moves, < 5 debts) this is negligible; for the gallery (50 rooms) it is
   50 small derivations on data already in memory. Mitigation: the derivation is
   pure, so `ArgumentGameSurface` / `buildGallery` can `useMemo` it on the row
   array; the design already says "build once per render." If a future room
   grows past a few hundred moves, the persistence follow-up (§14) removes the
   per-render cost. Not a blocker for v1.
2. **The three new request tag codes (`receipt_request`, `context_request`,
   `primary_record_request`) are read-but-not-yet-emitted.** EV-002 emits only
   `source_request` / `quote_request`. The `receipt` / `context` /
   `primary_record` debt kinds will not appear until the QOL-030 `ask_source`
   box (its kind selector) or an EV-004 affordance attaches the new codes.
   EV-003 ships the *reader* for all five so those kinds light up with no EV-003
   change. The implementer should add a `// reserved — emitted by QOL-030 ask_source
   kind selector` comment beside the three new codes and a test that the reader
   handles them. Risk: a reviewer flags "you read a tag nothing sends" — the
   defence is identical to EV-001's `screenshot_redacted` / `manual_citation`
   ("reserved so the consumer can already build for them").
3. **`challenged` / `accepted_*` depend on QOL-037.** Those three statuses need
   QOL-037's `evidenceResponse` records. If QOL-037 ships after EV-003, those
   statuses are simply never reached until it does (§10.13b graceful
   degradation). The design is explicit that this is acceptable — EV-003 is
   consumable as a `requested` / `supplied` / `stale` / `branched` tracker on day
   one. The implementer must **not** stub QOL-037's choice set inside EV-003 to
   force the statuses; the `EvidenceResponseLite` structural-minimum shape keeps
   the cards decoupled.
4. **The gallery `deriveBucket` upgrade changes existing routing.** Making the
   open-debt count the `source_chain_fight` primary signal will move some rooms
   *out* of that bucket (rooms whose `source` tags all resolved) and possibly
   some *in*. Existing gallery snapshot tests will need updating. This is an
   intended precision fix, not a regression — but the implementer must update
   the affected snapshots deliberately and the reviewer must confirm the new
   routing matches the §7.3 intent. Documented so it is not a surprise at Review.
5. **`STALE_DEBT_THRESHOLD_DAYS = 7` is opinionated.** Seven days is a guess at
   "long enough to be dormant, short enough to be useful." It is exported as a
   named constant and injectable per call (`staleThresholdDays`) so it is easy to
   tune. The implementer should add a `// TODO(EV-004): consider moving to
   constitution config` comment, exactly as EV-001 did for its dataset
   allowlist. Tuning the number is a follow-up, not a blocker.
6. **`branched` semantics — "open" or not?** EV-003 decides `branched` is **not**
   in `OPEN_EVIDENCE_DEBT_STATUSES` (the obligation is relocated, the asker can
   follow the branch — it is not "owed and ignored"). A reviewer might argue a
   branched debt is still unmet. The design records this as Open Question §16.3;
   the lean is "not open" so a branched debt does not keep a room flagged
   "Evidence requested" forever. Tunable in one line if the operator disagrees.
7. **Two debt models in the codebase.** `OpenIssueDebt` (point-standing) and
   `EvidenceDebt` (EV-003) both contain the word "debt". §4.1 documents why they
   are separate; the risk is a future contributor conflating them. Mitigation:
   the §4.1 note, the deliberate non-`_needed` naming, and a comment at the top
   of `evidenceDebtModel.ts` pointing to §4.1 of this doc.
8. **Test suite size.** ~70–90 new tests push the total from 1805 toward
   ~1880–1895 across ~71–72 suites (+2 suites). Group the per-status / per-kind
   assertions with `describe.each(ALL_EVIDENCE_DEBT_STATUSES)` /
   `describe.each(ALL_EVIDENCE_DEBT_KINDS)` to keep the file readable. No timing
   impact expected.

### Persistence — explicit decision

**v1: render-time derived. No migration, no `evidence_debt` table.** Consistent
with EV-001 (adapter-only, path b) and QOL-037 (render-time applicability
status). A debt is recomputed from `arguments` rows on every read. The request
move, the answering move, the artifacts, and the QOL-037 response records all
already persist (in `arguments` rows + the `client_validation` JSONB snapshot);
EV-003 needs **no new stored field**.

**Why not persist now** — the same three reasons EV-001 gave: (a) committing a
migration before EV-004 settles risks a second migration; (b) it blocks the
operator on a deploy step for a card that does not need one; (c) it violates the
"small, reversible, testable" cadence. The derivation is the authoritative
classifier; a future table would store rows of the `EvidenceDebt` shape and
backfill via the same derivation — read-time code would not change.

**The named follow-up (path c).** After EV-004, a follow-up card may add
`public.argument_evidence_debts` (`id`, `debate_id` FK, `node_argument_id` FK,
`request_argument_id` FK, `debt_kind`, `status`, `resolved_by_node_id`,
`requested_by_user_id`, `requested_at`, `resolved_at`) with RLS mirroring
`arguments` visibility, backfilled by `deriveEvidenceDebts`, and a
`submit-argument` extension that upserts a debt row when a request / answering
move posts. EV-003's derivation signature is the stable contract that follow-up
consumes — operator-gated, scheduled when the gallery / Inspect usage justifies
removing the per-render cost (Risk 1).

- **Evidence file upload + storage** — the binary screenshot, a storage bucket,

  binaries.

- **A persisted `evidence_debt` table / migration** — render-time-derived in v1
  (§14). The persistence card is the named follow-up.

  QOL-037's choice set.
- **The `ask_source` / `ask_quote` box UI** — the one-box composer's box types
  and short-prompt schema. **QOL-030**. EV-003 reads the moves they produce.
- **Auto-wiring evidence debt into point-standing** — EV-003 emits no
  `PointStandingDelta`; the `OpenIssueDebt` ledger is a separate model (§4.1).
- **The notification path** — the storyboard's "source requested" / "evidence
  supplied" notifications. **QOL-040**. EV-003 ships the room roll-up; QOL-040
  ships the delivery.
- **An admin override** to force a debt `stale` / resolved / reopened. EV-003
  derives every status deterministically from moves; no admin write path.
- **A risk classifier** promoting `EvidenceRisk` off `unknown`. A debt is an

- **Editing a request after it is posted** — `arguments` bodies are immutable

  resolves by later moves, never by editing the original (the storyboard +
  `evidence-doctrine` rule).
- **Cross-room debt aggregation** — a debt is room-scoped. No "all my open
  evidence debts across every room" view in v1 (a profile-epic candidate).
- **A keyboard chord to jump to the next open debt** — IX-003 territory; EV-003
  adds no chord.

## 16. Open questions

1. **Does a `stale` debt auto-advise a branch?** *Resolved — no.* Per the stub
   lean and §6.3: a `stale` debt only re-labels the chip and feeds the gallery
   `staleCount`. It never auto-acts, never auto-recommends a branch. The Inspect
   popout's §5 "Suggested next move" *may* surface "ask again or branch" as a
   user-chosen hand-off, but the model never decides for the user. Locked.
2. **Should EV-003 populate `EvidenceRisk`?** EV-001 reserved `EvidenceRisk` for
   "an EV-003-era risk classifier." *Lean — not in EV-003.* A debt is an
   *obligation* axis; a risk band is a separate *quality* judgement that edges
   toward truth-scoring. EV-003 keeps `EvidenceRisk` unpopulated and leaves any
   risk classifier to a later, separately-scoped card. Confirm at design review;
   if the operator wants it, it is an additive follow-up, not an EV-003 change.
3. **Does `branched` count as an "open" debt?** *Lean — no* (§6.4, Risk 6):
   `branched` is excluded from `OPEN_EVIDENCE_DEBT_STATUSES`, so a relocated
   obligation does not keep a room flagged "Evidence requested" indefinitely.
   The counter-argument: a branched debt is still unmet. One-line change if the
   operator disagrees. Flagged for design review.
4. **Is "accepted by both" reversible if a party retracts?** *Resolved — yes,
   via a fresh dispute.* Per §6.2's monotonic-with-reopen exception: a
   `dispute_applicability` / `request_source` after `accepted_by_both` reopens
   the debt to `challenged`. There is no separate "retract acceptance" move —
   reopening is done by a new dispute, mirroring QOL-037's ladder and GAME-005's
   retractable governance reactions. Locked.
5. **Persist debts now or stay render-time-derived?** *Resolved — render-time in
   v1* (§14). The persistence follow-up is named and operator-gated. Revisit
   when gallery / Inspect usage makes the per-render cost (Risk 1) material.
6. **Should the three reserved request tag codes be locked to a constitution
   tag list?** EV-002's `source_request` / `quote_request` are plain tag
   strings. The three new codes (`receipt_request`, `context_request`,
   `primary_record_request`) are likewise plain strings EV-003 *reads*. Whether
   they should become first-class constitution-registered tags is an EV-004 /
   tag-taxonomy question, not an EV-003 blocker. Noted for EV-004.
