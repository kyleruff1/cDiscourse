## Problem statement

When a participant asks another for a source, receipt, quote, or context, the
room takes on an **unresolved source obligation**. Today nothing tracks that
obligation: there is no per-node marker, no room-status signal, and no
gallery-level indicator. A reader cannot tell that a claim is leaning on
unbacked assertion, and the asker cannot tell at a glance whether their request
was ever answered.

**Evidence is not truth.** An evidence debt is a *structural* fact ("a source
was asked for and is owed"), never a verdict on who is right. A supplied source
proves at most that a record exists — never automatically what it means.

## Why this card matters now

- The one-box `ask_source` / `ask_quote` box types (QOL-030) produce these
  obligations; without EV-003 there is nowhere for the obligation to live.
- The QOL-032 Inspect popout's "What is unresolved" section needs evidence debt
  as an input.
- The conversation gallery (Epic 11) needs an "Evidence requested" lane
  (IX-001 focus lens) — that lane has no data source without EV-003.

## User-facing behavior

- Asking for a source/quote/receipt/context opens an **evidence debt** on the
  targeted node.
- The debt is **per move**, not global — resolvable by a later move.
- The debt is visible, with plain-language status, in: the room timeline node,
  the active-node detail (QOL-032 Inspect popout), the conversation gallery
  card, and a future diagnostic package.
- Supplying evidence, or both parties accepting it, advances the debt status.
- A debt never declares the original claim false; it influences strength
  *visuals*, never a truth label.

## Data / semantic fields needed

An evidence debt record (render-time-derivable in v1, persisted in a later card):

| Field | Meaning |
|---|---|
| `nodeId` | the move the debt is attached to |
| `debtKind` | `source · quote · receipt · context · primary_record` |
| `requestedByUserId` / `requestedAt` | who opened it |
| `status` | see status set below |
| `resolvedByNodeId?` | the later move that supplied/resolved it |

Evidence is a **structured object** — these axes stay separable (shared with
EV-001 / QOL-036 payment-evidence metadata):

- evidence **existence** · **amount** · **date** · **note/label** ·
  **applicability** · **interpretation** · **source chain**.

Payment/transfer examples must be **redacted and generic** — no real account
data, no raw payer/payee, no raw transaction IDs.

## UI states (evidence debt status set)

`requested · supplied · challenged · accepted_by_participant · accepted_by_both
· unresolved · stale · branched`

Plain-language user-facing labels:

- "Source requested"
- "Evidence attached"
- "Applicability disputed"
- "Evidence accepted by both"
- "Still unresolved"

**Forbidden labels:** "proof", "true", "winner", "loser". "Case closed" is
forbidden unless **both** parties explicitly settle — and even then prefer
"settled".

## Permission / rules implications

- Any participant may open an evidence debt by asking for a source.
- Only the move's author can resolve a `source`/`quote` debt by supplying.
- "Accepted by both" requires both primary participants.
- An evidence debt is **advisory** — it never blocks an ordinary reply
  (consistent with the Stage 6.2 advisory-validation doctrine).
- Engagement credit and factual-standing credit stay on **separate axes** — an
  amplified-but-unsourced claim earns no factual standing until the debt
  resolves (anti-amplification doctrine).

## Dependencies

- EV-001 (`EvidenceArtifact`) — the evidence object EV-003 reads.
- QOL-036 — payment/screenshot evidence metadata (the applicability axis).
- QOL-037 — evidence applicability dispute flow (drives `challenged`).
- QOL-030 (`ask_source` box type) / QOL-032 (Inspect "What is unresolved").
- IX-001 — the "Evidence requested" / "Source-chain pressure" focus lenses.

## Supersedes / superseded-by

EV-003 supersedes nothing and is superseded by nothing. It is a blank card the
storyboards depend on. It is *consumed by* QOL-032 and IX-001.

## Out of scope

- Evidence file upload + storage (a separate, explicitly-deferred card).
- Voting on evidence.
- Any AI assessment of evidence quality.
- A persisted `evidence_debt` table — v1 derives debts at render time from
  existing `arguments` rows + the `ask_source` move type.

## Acceptance criteria

- An evidence debt is derivable per node from existing data; no migration in v1.
- Debt status renders with a plain-language label in the timeline node, the
  Inspect popout, and the gallery card.
- **A gallery-level indicator marks rooms that carry an unresolved evidence
  debt** (the "Evidence requested" lane has a real data source).
- A debt never blocks an ordinary reply.
- No produced string contains "proof" / "true" / "winner" / "loser"; "case
  closed" appears only behind a both-parties-settled gate, and "settled" is
  preferred.
- Evidence existence / amount / date / note / applicability / interpretation /
  source-chain stay separable.

## Test targets

- Debt derivation per `debtKind`; status transitions `requested → supplied →
  accepted_by_both` and `requested → challenged → unresolved`.
- Stale derivation (debt open past a dormancy threshold).
- Ban-list scan over every produced label (no verdict tokens).
- The gallery indicator fires for a room with an open debt and not for one
  without.
- Advisory guarantee — a debt never appears in a blocking-error path.

## Implementation notes

- Pure-TS model first (`src/features/evidence/` beside `evidenceModel.ts`),
  render-time-derived. No migration, no Edge Function, no service-role.
- Reuse EV-001's `sourceChainStatus`; EV-003 adds the *obligation* layer above
  the *artifact* layer.

## Open questions

- Does a `stale` debt auto-advise a branch, or only surface in the Go popout's
  lens? (Lean: surface only — never auto-act.)
- Is "accepted by both" reversible if a party retracts? (Lean: yes — mirror
  GAME-005's retractable governance reactions.)
- Persist debts now or keep render-time-derived until a usage forces a table?
