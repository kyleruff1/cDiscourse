## 1. Goal & scope

Build the **invite-by-email backend** and the **auth deep link** so an invited
respondent lands **directly in the specific argument room** — fast, because
these rooms exist to de-escalate live in-person conflicts.

The card delivers four things:

1. **The invite data model** — a real `argument_room_invites` table (the shape
   `docs/invite-flow.md` reserved), with RLS that lets the inviter manage their
   own invites and the invitee see only theirs.
2. **The invite write path** — a `manage-room-invite` Edge Function. The client
   never inserts invite rows directly (the token + the email-side enrolment are
   privileged), and never sends email itself.
3. **The auth deep link** — a `/invite/<token>` route that survives signup (new
   user) and sign-in (existing user) and resolves to "open argument room
   `<debate_id>`".
4. **Seamless return entry** — the invited respondent enters the room as a
   **primary participant** with **no observer/side modal**, consistent with
   Stage 6.4 seamless conversation entry.

### In scope

- `argument_room_invites` table + RLS migration (written, not deployed).
- `manage-room-invite` Edge Function — actions `create`, `revoke`, `accept`,
  `list_for_debate`, `lookup_by_token`.
- Pure-TS invite model: token-shape validation, status machine, deep-link
  parsing/serialisation, the redeem-after-auth resolver.
- A `pendingInviteIntent` slice in the app session so the destination room
  survives the signup/auth round-trip.
- Wiring the existing `InvitePanel` placeholder to the real `create` path.
- Replacing the `InvitePanel` placeholder copy with real, doctrine-safe copy.

### Out of scope (own cards / explicit non-goals)

- **Live email delivery** (QOL-040 — invite & response notification lifecycle).
  This card defines the invite-row lifecycle and the *queued* state; the actual
  email send is operator-gated and owned by QOL-040. See §13.
- **Public ↔ private room visibility transition** (QOL-039). QOL-038 reads
  `debates` visibility; it does not own the toggle.
- **The concession composer / acceptance gradient** (QOL-041).
- **Deep links to anything other than an invite** (no share links, no node
  links, no public-room deep links — those are later cards).
- **OAuth / social login** — v1 is email+password only (doctrine §10).
- **A broad user-search API.** Invites are email-keyed only; this card adds no
  user lookup surface.

## 3. Problem statement

`docs/invite-flow.md` (Stage 6.1.0) shipped `InvitePanel` as a **pure
placeholder**: it accepts an email, lets the user mark a "planned invite" in
local state, and shows *"Invite sending coming later."* There is:

- **no invite table** — nothing is persisted;
- **no write path** — no Edge Function;
- **no token** — no `/invite/<token>` route, and the app has **no router at
  all** (routing is state-driven through the session reducer; see §6);
- **no auth-return memory** — if an invited person has to sign up, the
  destination room is lost the moment they leave the app for their email
  client and come back through a cold start.

The result: a person invited to a live conflict cannot actually be brought into
the right room. Scenario 1 calls this *"the single largest gap in Scenario 1."*

The hard parts:

1. **Surviving the auth round-trip.** The invitee leaves the app (or never had
   it), opens an email, taps a link, and re-enters through a Supabase auth
   redirect. The target `debate_id` must survive a cold start and a possible
   account-creation step.
2. **Enrolment timing.** Seamless entry (Stage 6.4) shows a non-participant as
   an *observer*. The invitee must be enrolled as a **primary participant
   before** the room renders, or they get the wrong (observer) view.
3. **Doing it without a router.** The app is a single `App.tsx` with
   state-driven screens. A deep link must be translated into session state, not
   a navigation push.
4. **Privilege.** Minting a token, writing the invite row, and email-side
   enrolment touch data the invitee does not yet own (the invitee may not even
   have an account when the row is created). RLS alone cannot express
   "enrol the holder of this token" — so the accept step needs the Edge
   Function.

## 4. Data model & file changes

### 4.1 The `argument_room_invites` table

A new migration creates the table `docs/invite-flow.md` reserved, with several
hardening changes over that draft (noted inline).

```sql
-- supabase/migrations/20260521000010_qol038_argument_room_invites.sql

create table public.argument_room_invites (
  id                  uuid primary key default gen_random_uuid(),
  debate_id           uuid not null references public.debates(id) on delete cascade,
  invited_by          uuid not null references auth.users(id),
  -- Stored lower-cased + trimmed. Always present: invites are email-keyed.
  invitee_email_lower text not null,
  -- Bound at accept time, never at create time. The invitee may have no
  -- account when the invite is minted.
  invitee_profile_id  uuid references public.profiles(id),
  -- The seat the invitee takes on accept. 'respondent' = the opposing primary
  -- seat (the storyboard's "named respondent"). NOT a public side word.
  intended_seat       text not null default 'respondent'
                        check (intended_seat in ('respondent','co_primary')),
  status              text not null default 'pending'
                        check (status in ('pending','accepted','revoked','expired')),
  -- SHA-256 hex of the raw token. The raw token is NEVER stored — only e-mailed.
  token_hash          text not null,
  created_at          timestamptz not null default now(),
  -- Default 14 days; set by the Edge Function, never null in practice.
  expires_at          timestamptz not null default (now() + interval '14 days'),
  accepted_at         timestamptz,
  revoked_at          timestamptz
);

-- One live (pending) invite per (debate, email). A revoked/expired invite does
-- not block re-inviting the same address.
create unique index argument_room_invites_one_live
  on public.argument_room_invites (debate_id, invitee_email_lower)
  where status = 'pending';

create index argument_room_invites_token_hash
  on public.argument_room_invites (token_hash);
create index argument_room_invites_invitee_email
  on public.argument_room_invites (invitee_email_lower);
create index argument_room_invites_debate
  on public.argument_room_invites (debate_id);

alter table public.argument_room_invites enable row level security;
```

**Deviations from the `docs/invite-flow.md` draft, and why:**

| Draft field | Change | Reason |
|---|---|---|
| `role_or_side text default 'any'` | → `intended_seat` enum `respondent` \| `co_primary` | A public-side word ("affirmative"/"negative") on an invite leaks game framing. The invitee enters as a *primary*; which sub-side they argue is theirs to choose in the room. `respondent` is doctrine-neutral. |
| `invitee_email_lower text` (nullable) | → `not null` | The card scope is **email invites only**. No display-name / user-id invite path — that would require a broad user search this card explicitly forbids. |
| `expires_at timestamptz` (nullable) | → `not null default now()+14d` | A never-expiring token is a standing credential. A bounded TTL is doctrine-safe. |
| — | + `revoked_at` | The inviter can revoke; we keep the audit timestamp. Rows are never deleted. |
| — | + partial unique index | Prevents duplicate live invites to the same address on the same room. |

**Token doctrine.** The raw token is a 32-byte random value, base64url-encoded
(~43 chars), generated **inside the Edge Function** via the Deno Web Crypto API.
The DB stores only `token_hash = sha256(raw)`. The raw token appears in exactly
two places: the invite email link, and the `lookup_by_token` request body. It is
never logged, never stored, never returned by `list_for_debate`.

### 4.2 RLS policies

```sql
-- The inviter manages their own invites.
create policy "inviter selects own invites"
  on public.argument_room_invites for select
  to authenticated
  using (invited_by = auth.uid());

create policy "inviter sees invites on rooms they created"
  on public.argument_room_invites for select
  to authenticated
  using (
    exists (
      select 1 from public.debates d
      where d.id = argument_room_invites.debate_id
        and d.created_by = auth.uid()
    )
  );

-- The invitee sees invites addressed to their own confirmed email.
-- auth.jwt() ->> 'email' is the caller's verified email claim.
create policy "invitee selects own invites"
  on public.argument_room_invites for select
  to authenticated
  using (
    invitee_email_lower = lower(auth.jwt() ->> 'email')
  );

-- Moderators/admins can read all invites (support + abuse review).
create policy "mods select all invites"
  on public.argument_room_invites for select
  to authenticated
  using (public.is_moderator_or_admin());
```

**No `INSERT`, `UPDATE`, or `DELETE` policy exists for `authenticated`.** Every
write goes through the `manage-room-invite` Edge Function's service-role client.
This is deliberate and matches the `annotate-evidence` precedent: the create
path mints a token (privileged), and the accept path must enrol the invitee in
`debate_participants` and bind `invitee_profile_id` — a write the invitee cannot
express through their own RLS at create time (the invite row pre-dates their
account). `anon` has no policies, so an unauthenticated client sees nothing —
the only unauthenticated touch-point is `lookup_by_token`, which is itself an
Edge Function call (see §5.4).

**Why the inviter does not get an `INSERT` policy even though they "own" the
row:** the token must be generated server-side and never seen by the inviter's
client; a client-side insert would mean the client picks the token. Centralising
in the function keeps token generation, hashing, and TTL in one audited place.

### 4.3 No change to `debates` / `debate_participants` schema

Enrolment reuses the existing `debate_participants` table. The accept step
inserts the standard participant row. No new column on `debates`. Room
visibility (`public`/`private`) is read-only here and owned by QOL-039 — if a
`visibility` column does not yet exist, QOL-038 treats every room as visible to
its participants (the existing `debates` SELECT RLS already gates that) and
QOL-039 layers the public-listing rule on later.

## 5. The `manage-room-invite` Edge Function

Standard `_shared/http.ts` shape: CORS preflight, non-POST → 405, JSON parse →
400, Zod validation → 422. A discriminated-union request body keyed on `action`,
exactly like `admin-users`.

```
POST /functions/v1/manage-room-invite
Content-Type: application/json
Authorization: Bearer <jwt>     (required for every action EXCEPT lookup_by_token)

{ "action": "create" | "revoke" | "list_for_debate" | "lookup_by_token" | "accept", ... }
```

The function builds:

- a **caller-scoped client** (anon key + the caller's `Authorization`) for every
  visibility/ownership read — RLS does the enforcing;
- a **service-role client** used **only** after the authorization check passes,
  for the privileged writes (invite row insert/update, participant enrolment).

### 5.1 `create`

```json
{ "action": "create", "debateId": "uuid", "inviteeEmail": "string",
  "intendedSeat": "respondent" }
```

1. Require JWT → `callerId` (else 401).
2. Validate `inviteeEmail` is a syntactically valid address; normalise to
   `lower(trim(...))`.
3. Caller-scoped `SELECT` of the debate (`id, created_by, status`). Not visible
   / not found → `403 room_not_visible` (no existence leak).
4. **Authorization:** caller must be the room creator **or** an existing primary
   participant **or** a mod/admin. A pure observer cannot invite. (Reuses the
   `debate_participants` read.) Else `403 not_allowed_to_invite`.
5. Reject if the debate `status` is `locked`/`settled` → `409 room_closed`
   (you cannot invite into a finished argument).
6. Reject self-invite: `inviteeEmail` equals the caller's own JWT email →
   `400 cannot_invite_self`.
7. Service-role check for an existing **pending** invite on `(debateId,
   inviteeEmail)`. If one exists → return it idempotently (`reused: true`) —
   re-inviting is not an error and does not mint a second token.
8. `generateInviteToken()` → raw token; `hashInviteToken(raw)` → hash.
9. Service-role `INSERT` the invite row (`status='pending'`,
   `expires_at = now()+14d`).
10. Build the link via `buildInviteLink(origin, raw)` where `origin` is the
    deployed origin (HOST-001b runtime-env; same source `buildAuthRedirectUrl`
    uses). The function does **not** send email in this card (QOL-040 owns
    delivery). It returns `notification: 'queued'`.
11. Best-effort `admin_audit_events` row: `action: 'room_invite_created'`,
    payload carries `debateId`, `emailDomain` (substring after `@` only — never
    the raw address, mirroring `adminInvitePayload.ts`), and
    `tokenIssued: true`. **Never** the raw token, never the full email.
12. Return `{ inviteId, status: 'pending', expiresAt, notification: 'queued',
    reused: <bool> }`. **The raw token is NOT in this response** — see §5.1.1.

#### 5.1.1 How the inviter gets the link (operator gating)

The card forbids live email until operator-gated. Two delivery paths, selected
by an Edge-Function env flag, with the **same** `manage-room-invite` contract:

- **`INVITE_EMAIL_ENABLED` unset (default, this card's shipped state):** the
  function does not send email and returns `notification: 'queued'`. For the
  invite to be testable, the function returns the **link** (not the bare token)
  in the `create` response **only to the inviter who is the room creator/primary
  and only over the authenticated call** — this is the inviter's own invite, the
  inviter is trusted with it, and `InvitePanel` shows a "Copy invite link"
  affordance. This keeps the manual-share path (the Stage 6.1.0 placeholder's
  intent) working without any email infrastructure.
- **`INVITE_EMAIL_ENABLED=true` (operator-gated, delivered by QOL-040):** the
  function calls the email provider (Resend, the same dependency
  `request-argument-deletion` already uses, with the same graceful
  `not_configured` fallback). When email is on, the `create` response **omits**
  the link — the link travels only in the email.

This is doctrine-safe: the link is a capability the *inviter already holds* (they
created the room and the invite); returning it to *them* is not a leak. It is
never returned to anyone else, never logged, and never in `list_for_debate`.

### 5.2 `revoke`

```json
{ "action": "revoke", "inviteId": "uuid" }
```

1. Require JWT.
2. Caller-scoped `SELECT` of the invite. RLS already restricts this to the
   inviter / room creator / mod. Not visible → `403`.
3. If `status != 'pending'` → `409 not_pending` (cannot revoke an accepted or
   already-revoked invite).
4. Service-role `UPDATE status='revoked', revoked_at=now()`.
5. Audit row `room_invite_revoked`.
6. Return `{ inviteId, status: 'revoked' }`.

### 5.3 `list_for_debate`

```json
{ "action": "list_for_debate", "debateId": "uuid" }
```

1. Require JWT.
2. Caller-scoped `SELECT … where debate_id = ?`. RLS returns only invites the
   caller may see (inviter / room creator / mod / their own invitee row).
3. Return `{ invites: [{ inviteId, inviteeEmailMasked, intendedSeat, status,
   createdAt, expiresAt, acceptedAt }] }`. `inviteeEmailMasked` masks the local
   part (`j•••@example.com`) — the inviter typed it so a partial echo is fine,
   but full re-emission is avoided. **No `token`, no `token_hash`.**

### 5.4 `lookup_by_token`

```json
{ "action": "lookup_by_token", "token": "string" }
```

This is the **only action callable without a JWT** — a brand-new invitee who has
not yet signed up taps the link and needs to see *what room they were invited
to* before they create an account.

1. Validate the token shape (length, base64url charset). Bad shape →
   `400 invalid_token`.
2. `hashInviteToken(token)`; service-role `SELECT` the invite by `token_hash`.
   Not found → `404 invite_not_found`.
3. Compute the live status: if `status='pending'` but `expires_at < now()` →
   treat as `expired`. If `revoked` / `accepted` / `expired` → return that
   status with **no room details** beyond a generic message.
4. If live-`pending`: service-role `SELECT` the debate's **display-safe**
   fields only — `title` (or a root-claim excerpt if blank), the inviter's
   `display_name`, `created_at`. **Never** the room body, never the argument
   tree, never participant emails. A private room exposes only its title here;
   the full content is gated behind accept + RLS.
5. Return `{ status, room: { title, invitedByDisplayName } | null,
   tokenEcho: <token> }`. (`tokenEcho` lets the gate carry the token into the
   `accept` call without re-parsing.)

**Why service-role for a public-ish read:** the caller has no JWT and no RLS
identity, so a caller-scoped client would see nothing. The token *is* the
authorization — possession of a valid, unexpired, unrevoked token authorises
seeing the minimal room title. The function deliberately returns the narrowest
possible projection.

### 5.5 `accept`

```json
{ "action": "accept", "token": "string" }
```

Called **after** the invitee is authenticated (signed up or signed in). This is
the privileged step that turns a token into a seat.

1. Require JWT → `callerId`, `callerEmail` (from the verified `email` claim).
   No JWT → `401`.
2. `hashInviteToken(token)`; service-role `SELECT` the invite. Not found →
   `404`.
3. Live-status check: revoked → `409 invite_revoked`; expired (or
   `pending`+past TTL) → `409 invite_expired`; already `accepted` →
   **idempotent success** if `invitee_profile_id = callerId`, else
   `409 invite_already_accepted` (someone else already redeemed it).
4. **Email-binding check.** `lower(callerEmail)` must equal
   `invitee_email_lower`. Mismatch → `403 invite_email_mismatch`. This is the
   core safety property: a token alone is not enough — the redeemer must be
   signed in as the addressed email. (A new user signs up *with* the invited
   email; an existing user must already own it. See §6.5 for the mismatch UX.)
5. Load the debate. If `status` is `locked`/`settled` → `409 room_closed`.
6. **Enrol the participant.** Service-role `INSERT` into `debate_participants`
   `(debate_id, user_id=callerId, side)`. The `side` is the doctrine-safe
   neutral primary value the room model uses for the second primary seat
   (`neutral` in today's schema — the invitee picks affirmative/negative inside
   the room; the GAME-004 room contract treats them as the claimable opponent
   seat). If a participant row already exists (the invitee was already in the
   room) the insert is a no-op via the existing `(debate_id, user_id)`
   uniqueness — accept is still idempotent.
7. Service-role `UPDATE` the invite: `status='accepted'`, `accepted_at=now()`,
   `invitee_profile_id=callerId`.
8. Audit row `room_invite_accepted`.
9. Return `{ debateId, status: 'accepted', enteredAsParticipant: true }`.

The client then calls `selectDebate(debateId)` — the room opens, and because
the invitee is now a `debate_participants` row, Stage 6.4 seamless entry shows
them as a **primary participant**, not an observer. No side modal fires.

### 5.6 Error shape table

```
400  invalid_json | invalid_token | cannot_invite_self | invalid_email
401  unauthorized
403  room_not_visible | not_allowed_to_invite | invite_email_mismatch
404  invite_not_found
405  method_not_allowed
409  room_closed | not_pending | invite_revoked | invite_expired
     | invite_already_accepted
422  validation_failed
500  invite_lookup_failed | enrolment_failed
```

All errors return `{ error: 'code', message: 'human readable' }`. No stack
traces, no token echo on an error path.

### 5.7 Logging rules (per `supabase-edge-contract`)

- **Never log:** the `Authorization` header, the raw token, `token_hash`, the
  service-role key, the `RESEND_API_KEY`, the full invitee email.
- **Do log:** function name, `action`, caller user id, `debateId`, the decision,
  duration, error class, and `emailDomain` only.

## 6. The auth deep-link flow

### 6.1 Constraint: there is no router

`App.tsx` is a single component tree; "screens" are chosen by a `switch` on
`state.status` and on whether a debate is selected. Routing into a room is
`selectDebate(debateId)` (a `debate_participants`-aware dispatch), and "go home"
is re-dispatching `SIGNED_IN`. **QOL-038 must not add React Navigation or any
router** — that would break the TL-003 no-route invariant the codebase relies
on. The deep link is therefore translated into **session state**, not a
navigation event.

### 6.2 The link itself

- **Web (the primary target for v1 — the deployed SPA):** the invite link is an
  app-origin URL: `https://dev.cdiscourse.com/invite/<token>`. Opening it loads
  the SPA; `App.tsx` reads `window.location.pathname` once at cold start.
- **Native (Expo):** the same path is also registered as an Expo Linking deep
  link (`cdiscourse://invite/<token>` / the universal-link form). The impure
  read uses `expo-linking`'s `getInitialURL()` + `addEventListener`. `expo-linking`
  is already an Expo SDK package (no new third-party dep; install via
  `npx expo install expo-linking` if not already present — verify against
  `package.json` first).
- The Supabase **auth** redirect (`buildAuthRedirectUrl`, kind `confirm_signup`)
  continues to land on `/auth/callback`; QOL-038 does **not** change that route.
  The invite token survives the auth round-trip in **session state** (§6.3),
  not in the auth redirect URL — so a token is never appended to a Supabase
  email link and never appears in auth provider logs.

`parseInviteDeepLink(url)` (pure) extracts `{ token }` from either form and
validates the token shape. Anything else returns `null` and the app cold-starts
normally.

### 6.3 The `pendingInviteIntent` session slice

The destination must survive: (a) a cold start, (b) leaving the app for the
email client, (c) a full signup. So it lives in the **persisted** session, not
React state.

```ts
// added to AppSessionSnapshot
pendingInviteIntent: {
  token: string;          // the raw invite token
  capturedAt: string;     // ISO — used to expire a stale intent (24h)
} | null;
```

- New reducer actions: `SET_PENDING_INVITE_INTENT` and
  `CLEAR_PENDING_INVITE_INTENT`.
- `sessionStorage.ts` persists `pendingInviteIntent` alongside the existing
  snapshot fields, so a cold start mid-signup restores it.
- **Critical:** the `SIGNED_OUT → SIGNED_IN` reducer transition **preserves**
  `pendingInviteIntent`. Today `SIGNED_IN` rebuilds the snapshot; QOL-038's
  reducer change carries the intent through so it is still present the moment
  the new account exists.
- A `pendingInviteIntent` older than 24h is dropped on read (a stale intent
  must not silently redirect a returning user weeks later).

### 6.4 New user (no account)

```
Email → tap /invite/<token>
  → cold start; App.tsx reads the URL; parseInviteDeepLink → { token }
  → dispatch SET_PENDING_INVITE_INTENT({ token })
  → InviteRedeemGate mounts:
      → inviteApi.lookupInviteByToken(token)   (no JWT — §5.4)
      → status 'pending' → show "You were invited to respond to an
        argument: <room title> — by <inviter>." + a Continue button
  → user is signed_out → AuthScreen, in SIGN-UP mode, with the
    email field PRE-FILLED from the invite and (softly) locked to the
    invited address (an editable-but-warned field — see §6.5)
  → user completes signup (Supabase confirm-email flow may apply)
  → on the FIRST signed_in state with a non-null pendingInviteIntent:
      → App.tsx calls inviteApi.acceptRoomInvite(token)
      → §5.5 enrols the participant, marks the invite accepted
      → dispatch CLEAR_PENDING_INVITE_INTENT
      → selectDebate(debateId) → the room opens
  → seamless entry (Stage 6.4) sees a participant row → primary view,
    no observer/side modal.
```

**Email-confirmation wrinkle.** If the Supabase project requires email
confirmation, signup does not immediately yield a session — the user confirms
via a second email and re-enters through `/auth/callback`. Because
`pendingInviteIntent` is **persisted**, it is still there when the confirmed
session finally arrives, and the accept-on-first-signed-in step fires then. If
the project has confirmation **off**, the session is immediate and accept fires
right away. Both paths work without special-casing because the trigger is "first
signed_in with a pending intent", not "the signup call returned".

### 6.5 Existing user (has an account)

```
Email → tap /invite/<token>
  → InviteRedeemGate → lookupInviteByToken → 'pending' → Continue
  → user is signed_out → AuthScreen
      → if the gate's email matches a known pattern, default to SIGN-IN
        mode (still switchable to sign-up); email pre-filled
  → user signs in
  → first signed_in with pendingInviteIntent → acceptRoomInvite(token)
      → §5.5 step 4 email-binding check:
          - signed-in email == invited email → enrol → room opens
          - signed-in email != invited email → 403 invite_email_mismatch
  → on mismatch: InviteRedeemGate shows
      "This invite was sent to a different email address. Sign in with
       the address the invite was sent to, or ask <inviter> to re-send
       it to <your address>."
    + a "Sign in as someone else" action (sign out, keep the intent).
```

The email-binding check (§5.5 step 4) is the security spine of the existing-user
path: a forwarded link cannot be redeemed by whoever receives the forward — only
by the addressee, proven by their authenticated email claim.

### 6.6 Already-signed-in user

If the invitee is **already signed in** when they tap the link (e.g. they have
the app open), `App.tsx` still captures the intent and `InviteRedeemGate` runs;
accept fires immediately (no auth step). If the signed-in email matches → straight
into the room. If it does not → the §6.5 mismatch screen.

### 6.7 Why accept is a separate explicit step (not auto on lookup)

`lookup_by_token` is unauthenticated and must stay read-only — it cannot enrol
anyone. `accept` is the authenticated, side-effecting step. Splitting them means
the unauthenticated surface can never mutate state, and the enrolment write
always happens against a verified identity.

## 8. Permission / rules implications

- **Who can invite:** room creator, an existing primary participant, or a
  mod/admin. A pure observer cannot (§5.1 step 4). Enforced server-side.
- **Who can revoke:** the inviter, the room creator, or a mod/admin (RLS +
  §5.2).
- **Who can accept:** only a signed-in user whose verified email equals the
  invited address (§5.5 step 4).
- **Constitution engine:** untouched. Invites do not create arguments; the
  argument write path remains `submit-argument`. No engine import in any new
  file.
- **`debate_participants`:** the accept step is the only new writer, and it
  writes the same row shape `joinDebate` already writes — through the Edge
  Function's service-role client, because the invitee is being enrolled by the
  *system on the strength of a token*, not by their own self-service join.
- **Room visibility:** QOL-038 reads it, never sets it. A private room's invite
  works identically; the room simply never appears in the public list (QOL-039).
  The invite is the *only* way into a private room for a non-creator — which is
  exactly the band/rent storyboard.
- **Seamless entry doctrine (Stage 6.4):** by enrolling the invitee *before*
  `selectDebate`, the invitee is an "existing participant" by the time the room
  renders, so seamless entry shows them the primary view — no observer default,
  no "choose side" modal. This is the card's headline acceptance criterion and
  it is satisfied by **ordering**, not by a new UI branch.

## 9. Doctrine & safety self-check

- **No "game" / "debate" framing in user copy.** Invite email subject:
  *"You were invited to respond to an argument."* Body names the room title and
  carries one link. `InvitePanel` helper copy says *"invite someone to this
  argument"*. Forbidden in every invite string: *debate challenge, game invite,
  game, debate, challenger, opponent, winner, loser, beat, defeat*. The existing
  `INVITE_ROLE_LABELS` ("Take the other side", "challenger") are **removed** —
  `intended_seat` is `respondent`, surfaced to users as *"respond to the
  argument"*. A doctrine ban-list test asserts this over every exported invite
  string.
- **Plain language.** Internal codes (`room_closed`, `invite_email_mismatch`,
  `not_allowed_to_invite`, …) never reach the UI raw — each maps through a
  plain-language string in `inviteCopy.ts` (and, where the existing
  `gameCopy.toPlainLanguage` map is the surfacing path, a new entry is added
  there, since adding a code without the mapping is a test failure).
- **No truth / verdict language.** An invite is content-neutral; it carries a
  room *title* only. Nothing in this card labels a claim, a person, or an
  argument.
- **No service-role in the client.** The service-role key lives only in the
  `manage-room-invite` function. The client uses
  `supabase.functions.invoke(...)`. A `grep -r "SERVICE_ROLE" src/` stays
  clean.
- **No direct write to `public.arguments`.** Invites never touch `arguments`.
  The argument write path is still `submit-argument`.
- **No broad user search.** Invites are email-keyed. The card adds no endpoint
  that enumerates or searches users. `lookup_by_token` returns one room's title
  and one display name, gated by token possession.
- **Token safety.** 32-byte random, base64url; only the SHA-256 hash is stored;
  the raw token is never logged and is returned only to the inviter (their own
  capability) or carried in the email. TTL ≤ 14 days. A revoked/expired/used
  token is inert.
- **RLS always on.** `argument_room_invites` has RLS enabled with no
  `authenticated` write policy; `anon` has no policy at all.
- **No AI calls.** No Anthropic / xAI / X API anywhere in this card. The
  function makes at most one outbound call — to the email provider — and only
  when `INVITE_EMAIL_ENABLED=true`, which is operator-gated (QOL-040).
- **No PII leak.** `list_for_debate` masks the email local part; audit rows
  store `emailDomain` only; `lookup_by_token` returns a display name, never an
  email; no function logs a full email.
- **Fail-open on routing defects.** A malformed/unknown invite URL cold-starts
  the app normally; a redirect-origin defect degrades exactly as
  `buildAuthRedirectUrl` already does. An invite problem never blocks signup or
  traps a user.
- **Terminology audit.** `npm run ux:terminology:audit` must stay clean over the
  rewritten `inviteCopy.ts` and `InvitePanel.tsx`.

## 10. Edge cases

| # | Case | Handling |
|---|---|---|
| 1 | Token tapped twice (accepted, then re-tapped by the same user) | `accept` is idempotent for the same `invitee_profile_id` — re-opens the room. |
| 2 | Forwarded link redeemed by a third party | §5.5 step 4 email-binding check → `403 invite_email_mismatch`. |
| 3 | Invite expired before redemption | `lookup_by_token` and `accept` both report `expired`; the gate shows the expired screen with "ask for a new one". |
| 4 | Inviter revokes after the email was sent but before redemption | `accept` → `409 invite_revoked`; gate shows "no longer active". |
| 5 | Same address invited twice to the same room | Partial unique index + §5.1 step 7 → the second `create` returns the existing pending invite (`reused: true`), no duplicate token. |
| 6 | Invitee already a participant in the room | `accept` step 6 insert is a no-op via `(debate_id,user_id)` uniqueness; invite still flips to `accepted`; room opens. |
| 7 | Room locked/settled between invite and accept | `accept` → `409 room_closed`; gate explains the argument is settled. |
| 8 | Room deleted between invite and accept | `lookup_by_token`/`accept` → `404` (FK `on delete cascade` removes the invite too). |
| 9 | Cold start mid-signup (app killed between tap and account creation) | `pendingInviteIntent` is persisted → restored on next launch → accept fires on the first signed_in state. |
| 10 | Stale intent (user signed up, abandoned, returns days later) | Intent older than 24h is dropped on read — no surprise redirect. |
| 11 | New user signs up with a *different* email than the invited one | Signup succeeds (we don't hard-lock the field); `accept` then returns `invite_email_mismatch`; gate guides them to use the invited address or request a re-invite. |
| 12 | Network failure during `lookup`/`accept` | Gate shows a Retry; the intent is kept so a retry works. Never a dead end. |
| 13 | Concurrent accept by the addressee on two devices | First write wins; the second sees `accepted` with the same `invitee_profile_id` → idempotent success on both. |
| 14 | Concurrent `create` race (two inviters, same address, same instant) | The partial unique index makes one insert fail; the function catches the unique violation and returns the now-existing row (`reused: true`). |
| 15 | Token in a server log / crash report | Prevented by the logging rules (§5.7); a ban-list test scans the function source for `console.*` touching `token`. |
| 16 | Observer opens `InvitePanel` | `create` → `403 not_allowed_to_invite`; panel renders the "only participants can invite" state, no email field. |
| 17 | Supabase email-confirmation enabled | Accept fires on the first *confirmed* signed_in state, not on the signup call return — works because the trigger is session-based and the intent is persisted (§6.4). |
| 18 | Self-invite (creator invites their own address) | §5.1 step 6 → `400 cannot_invite_self`. |
| 19 | `INVITE_EMAIL_ENABLED` unset | `create` returns `notification:'queued'` + the link (to the inviter only); `InvitePanel` shows "Copy invite link". No email sent — satisfies the "no emails until operator-gated" non-negotiable. |
| 20 | Invite link opened on web vs native | `parseInviteDeepLink` handles both the `https://…/invite/<token>` and the `cdiscourse://invite/<token>` forms. |

contradict or invalidate any other section of this design.

### 17.1 The gap

§10 row 8 ("Room deleted between invite and accept") assumes a hard
`on delete cascade` removes the invite row when the debate is deleted. That
path is correct for *hard* deletes — the `argument_room_invites.debate_id`
FK in §4.1 does carry `on delete cascade`. The DB-level reality, however, is
that the realistic "the room is no longer available" state is the **status
transition to `archived`**:

```
debates.status enum (per supabase/migrations/20260516000001_initial_schema.sql
line 145):
   'draft' → 'open' → 'locked' → 'archived'
```

An `archived` room's invite row still exists. `lookup_by_token` would find
the invite and return room details; `accept` would attempt to enrol the
invitee. Neither §5.4 nor §5.5 names what to return for an archived room. The
existing `room_closed` error (§5.6) is reserved for `locked`/`settled`
states, which are doctrinally distinct from `archived` (the storyboards use
`settled` as a UX-level concept; `locked` is the closest DB state for a
finished argument that participants choose not to amend; `archived` is the
post-life state where the room is no longer active for interaction).

### 17.2 The fix — three small changes

**(a) `lookup_by_token` and `accept` reject archived rooms explicitly.**

Add to §5.4 step 3 (live-status compute for lookup): if the room's
`debates.status = 'archived'`, return `status: 'room_archived'` with no room
details beyond a generic message — identical projection to the
revoked/expired/already-accepted paths.

Add to §5.5 step 5 (load the debate inside accept): the existing rule
"If `status` is `locked`/`settled` → `409 room_closed`" gains a sibling rule
"If `status` is `archived` → `409 room_archived`". The archived check
happens before the participant enrolment.

**(b) New error code in §5.6.**

```
409  room_closed | not_pending | invite_revoked | invite_expired
     | invite_already_accepted | room_archived          ← NEW
```

The 409 status class is correct: the room exists and the invite is
syntactically valid, but the target's state forbids the operation. Mirrors
`room_closed` semantics exactly.

**(c) New `InviteRedeemGate` state in §7.2.**

| State | UI |
|---|---|
| **Room archived** | *"This argument was archived and is no longer active. Ask <inviter> if there is a newer argument to join."* + "Go to my arguments". |

Placement: between the existing **Revoked** and **Already accepted** rows
in the §7.2 table. Mirrors the Expired state's shape (clear copy, escape
hatch to the gallery, never traps the user).

### 17.3 Updated §10 edge-case table

Row 8 of the existing §10 table is split into two rows so the two paths
(soft-delete via status vs. hard-delete via row removal) are both explicit:

| # | Case | Handling |
|---|---|---|
| 8a | **Room archived** between invite and accept (status transition to `archived`) | `lookup_by_token` returns `status: 'room_archived'`; the gate shows the §7.2 "Room archived" state with a clear escape. `accept` returns `409 room_archived` if the user attempts to proceed past the gate. The invite row is **not** mutated by the archive — re-activating the room (if an admin un-archives it) lets the invite continue to work until its normal TTL expires. |
| 8b | **Room hard-deleted** between invite and accept (row removal — administrative cleanup) | `on delete cascade` removes the invite row; `lookup_by_token` and `accept` both return `404 invite_not_found`. The gate's existing "Not found / malformed" state handles this. |

### 17.4 Test plan addition

One new test in the `manage-room-invite` contract test suite:

- `__tests__/manageRoomInviteArchivedRoom.test.ts` — `lookup_by_token`
  against a debate whose `status = 'archived'` returns the
  `room_archived` status; `accept` against the same returns
  `409 room_archived`; the invite row is not mutated by either call;
  the participant enrolment never fires.

One new test in the `InviteRedeemGate` UI test:

- The §7.2 "Room archived" state renders correctly given the
  `room_archived` lookup response and exits cleanly on "Go to my
  arguments" (clears the intent, drops to the gallery, identical to
  the Expired-state shape).

### 17.5 What does NOT change

- §4 (data model): no schema change. `debates.status` already carries
  `archived`; the function reads it.
- §4.1: the invite table's `on delete cascade` on `debate_id` stays — it
  is the correct semantics for the hard-delete case (row 8b above).
- §4.2 (RLS): unchanged.
- §5.1 (create) step 5: stays as "Reject if the debate `status` is
  `locked`/`settled` → `409 room_closed`". A creator inviting into an
  already-archived room is a separate case the §5.1 rule should also
  cover — append `archived` to that step's status check, returning
  `409 room_archived` for create-time consistency.
- §5.2 (revoke): unchanged. Revoke is a metadata operation; the
  room's status does not block it.
- §5.3 (list_for_debate): unchanged. Listing invites on an archived
  room is read-only and useful for audit.
- §6 (the auth deep-link flow): unchanged. The new state lives in the
  gate, not in the session reducer.
- §9 (doctrine self-check): unchanged. The new copy *"This argument
  was archived and is no longer active. Ask <inviter> if there is a
  newer argument to join."* contains no banned tokens and uses the
  post-QOL-035 terminology (`argument`, not `debate`).
- §13 (supersedes / hands-off to QOL-040): unchanged.
- §16 (deploy steps): unchanged.

### 17.6 Operator-judgment items raised by the enrichment

None. The fix is fully specified at the function and UI layers; no new
env var, no new migration, no operator decision required.
