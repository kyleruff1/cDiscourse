# UX-001.3 Design Intent Brief — Composer and Context Consolidation

**Card:** UX-001.3 — Composer and context consolidation
**Epic:** UX-001 (Brand-Forward Simplified UX/UI Consolidation)
**Phase:** 3 of 7
**Priority:** P0 / Urgent
**Effort:** L
**Filed:** 2026-05-25
**Author:** Operator (product intent)
**Status:** Binding for UX-001.3 designer phase

## Why this brief exists

UX-001.1 corrected the brand shell with an eight-question framework because the shell is structurally bounded. UX-001.2 succeeded with a twelve-question framework plus chrome-nullification authority because the viewport repair required overriding prior aesthetic placements. UX-001.3 needs a different shape: the composer is not about repair (it is mostly functional today) but about consolidation. The work is to make the composer feel like one persistent contextual command center rather than a collection of mode-specific forms.

The composer's current state has accumulated working surfaces across multiple shipped cards (QOL-030 one-box composer chassis, COMPOSER-002 in-room dock, SC-004 timeline node action-to-composer preset mapping, the per-action mode switches that have shipped piecemeal). Each surface works individually, but the user-facing experience does not yet feel like one system. The composer can feel like an unrelated form when its target context is ambiguous, when mode switching loses draft state, when validation appears as scattered messages rather than compact contextual signals, or when the composer's position drifts from the persistent dock the one-box concept implies.

This brief converts the operator's intent for the composer into measurable thresholds and explicit dispositions. The designer reads the brief as binding product specification; the designer's job is to translate the brief's contracts into an implementation specification.

## What UX-001.3 consumes from prior cards (fixed, not relitigated)

UX-001.1's Phase 1 framing in docs/core/current-status.md establishes the breakpoints (phone 0-599dp, tablet 600-1279dp, wide 1280dp+), header heights (64/96/120), logo heights (44/80/96), and surface hierarchy. UX-001.3 does not modify these.

UX-001.2's Phase 2 framing in docs/core/current-status.md establishes the compact room/context strip (max 48/56/64 heights), the hidden tab bar in room-active mode, the dissolved roomToolbar, the selected-message readout's repositioning to below or beside the Timeline (compact heights 64-72 / 72-80 / 80-96), the score tracker's repositioning, the internal rail offset reduction, the Timeline first-row offset hard caps (128/168/200), and the QOL-040.3 microMoment dismiss-on-first-Timeline-interaction contract. UX-001.3 does not modify these and must not push the Timeline back above its offset cap.

The composer in Timeline mode currently sits below the Timeline (the lower dock position UX-001.2 confirmed). UX-001.3 preserves the lower dock placement. The composer does NOT return to an above-Timeline position.

The "Acting on" label that UX-001.2's repositioned selected-message readout displays is the same context UX-001.3's composer surfaces. UX-001.3 coordinates with the readout so the "Acting on" semantic is consistent across both surfaces: the readout names what is selected; the composer names what the user is acting on; these are typically the same node but may diverge during type switches.

## The central product rule

The composer must always show its target. The user looking at the composer should never have to wonder what node, evidence item, concession set, or branch their next action applies to. The target context is visible in compact form by default; the target can be expanded for fuller context on explicit user action.

The composer must remain contextual and persistent. The composer is not a separate screen or a detached form. It is one persistent box that adapts its schema based on the action type the user has selected. Mode switching does not lose drafts. Mode switching does not require navigation. Mode switching feels like changing what kind of move the user is making, not changing what app they are using.

The composer's modes are presented as one system with shared chrome and shared persistence, not as separate forms with separate states.

## The composer's mode list (canonical)

The composer supports the following modes in UX-001.3's scope. Each mode has its own schema (which fields appear, which are required, which are validated how) but shares the composer's chrome, target display, draft persistence, and submit path.

Authoring modes (create new content):
- Root claim (the room's opening argument)
- Standard response (a reply to a parent node)
- Challenge / rebuttal (a structured opposition)
- Branch / tangent (a side path from a parent)
- Synthesize (a cluster summary)

Evidence modes:
- Evidence submission (attach evidence to a claim)
- Respond to evidence (acknowledge or contest existing evidence)

Source modes:
- Ask source (request a source for a claim)
- Ask quote (request a verbatim quotation)

Clarification modes:
- Clarify (ask for or provide clarification)
- Narrow (refine the scope of a claim)
- Confirm (acknowledge agreement on a narrow point)

Concession modes:
- Concession list (offer one or more concessions)
- Respond to concession (accept, modify, or reject offered concessions)

The mode list is canonical for UX-001.3. New modes (if any emerge during design) require operator decision before inclusion. Modes not listed here are out of scope for UX-001.3 and may be addressed in UX-001.4 (Act popout) or later cards.

## The target display contract

The composer displays its target in compact form by default. The target display answers: what node, evidence item, concession set, or branch is this action applying to?

Default target display per mode:
- Authoring modes targeting a parent: parent node type + short excerpt + author/role hint
- Authoring modes with no parent (root claim): room/resolution title + room context
- Evidence modes: target claim's short excerpt + evidence applicability context (which claim this evidence supports)
- Source modes: target claim's short excerpt + the specific axis being questioned
- Clarification modes: target node's short excerpt + clarification request context
- Concession modes: target node's short excerpt + concession items list (compact)
- Synthesis mode: cluster identifier + nodes-being-synthesized count + cluster summary excerpt

Maximum target display heights:
- Phone: 48-64px
- Tablet: 56-72px
- Wide: 64-80px

The target display can expand to show fuller context on explicit user action (tap, hover-and-hold, expand affordance), but the expanded state never becomes default. Expanded state may consume up to 25% of viewport height.

## The mode-switching contract

The user can switch action type without losing drafts. Each mode has its own draft buffer. Switching from Reply to Add Evidence preserves the Reply draft; switching back restores it.

Mode-switching persistence rules:
- Drafts are per-mode and per-target. The Reply draft for node A is separate from the Reply draft for node B.
- Drafts persist across mode switches within the same composer session.
- Drafts do NOT need to persist across app sessions or page navigations (that is a separate product decision, deferred).
- Mode switching does not require navigation. The composer's mode changes; the composer's position does not.
- Mode switching takes effect within one user interaction (tap a mode toggle, the composer schema changes immediately).

The mode switcher's placement is a designer decision (header of the composer, dropdown, segmented control, etc.), but the switcher must be reachable in compact form without consuming significant composer real estate.

## The composer position contract (read-only)

The composer position is fixed by UX-001.2's Phase 2 framing. UX-001.3 does not relitigate this.

On Timeline view:
- The composer sits below the Timeline (bottom dock).
- The composer does NOT sit above the Timeline.
- The composer's height has an upper bound (specified below) so the Timeline retains substantive board space.

On Cards/Stack view:
- The composer's position follows the existing COMPOSER-002 in-room dock pattern.
- UX-001.3 may refine the position if necessary but does not redesign it.

The composer's default height:
- Phone collapsed: 56-72px (just the target display + a "tap to compose" affordance)
- Phone expanded: up to 50% viewport height (the user has tapped into compose mode)
- Tablet collapsed: 64-80px
- Tablet expanded: up to 40% viewport height
- Wide collapsed: 72-88px
- Wide expanded: up to 35% viewport height

The collapsed state preserves Timeline visibility. The expanded state lets the user actually compose. The transition is triggered by explicit user action (tap the composer, tap a mode, tap the Act affordance for an action that opens the composer).

The composer's mobile expanded state behaves as a bottom sheet: the composer slides up from the bottom dock; the Timeline remains visible above (potentially obscured by the sheet but not displaced); the sheet can be dismissed via swipe, tap-outside, or explicit close.

The composer's tablet and wide expanded state may use the bottom sheet pattern or expand-in-place; the designer chooses with stated rationale.

## The validation surface contract

Composer validation appears as compact contextual signals attached to the relevant field or the composer's header. Validation does NOT appear as scattered banners, modals, or full-page error states.

Validation surfaces:
- Inline field validation (e.g., "Source URL required" appearing below the source field)
- Composer header chip (e.g., "Draft incomplete" as a small chip when required fields are empty)
- Submit affordance disabled with reason (e.g., the submit button is dimmed with a tooltip "Source URL required to submit")
- Semantic validation chips (e.g., "Pause suggested" if the QOL-040.3 microMoment-equivalent classifier fires on the draft; these are advisory, not blocking)

Validation does NOT appear as:
- Full-screen modals
- Banner notifications above the composer
- Toast notifications outside the composer's surface
- Verdict copy (no "this argument is wrong," "this evidence is invalid")

Semantic validation (per the Observations vs Allegations roadmap UX-001.5A): if classifier-derived advisory signals fire on a draft, they appear as composer-only Observations (e.g., "Pause suggested," "Person/intent shift may need review"). These are advisory copy directed at the author; they are never accusatory and never block submission. UX-001.5A's full registry is not yet shipped; UX-001.3 uses the existing semantic referee surface (refereeBanner / overridePrompt props) for composer-only advisory signals and the registry consolidates them in UX-001.5A.

## The submit path contract (read-only)

UX-001.3 does NOT modify the submit path. The composer continues to use the existing submit-argument Edge Function with the existing argument insert path. No new direct insert, no new service-role usage, no new backend write path.

If the composer's UI changes require backend coordination (e.g., a new field needs persistence), the requirement is documented and the operator decides whether to file a separate backend card or scope the UI feature to use existing backend fields. UX-001.3 does NOT extend backend behavior inline.

## The keyboard and shortcut contract

The composer respects IX-003's keyboard navigation model. Keyboard shortcuts on the composer:
- Cmd/Ctrl + Enter submits the composer (preserves existing IX-003 behavior)
- Cmd/Ctrl + K opens the mode switcher (new in UX-001.3; designer specifies the mode switcher's keyboard affordance)
- Tab / Shift+Tab moves between composer fields (standard form navigation)
- Esc collapses the composer or closes the bottom sheet (preserves existing patterns)

The composer's keyboard shortcuts do NOT conflict with the board's keyboard shortcuts (arrow keys for node navigation, A/I/G for Act/Inspect/Go). The shortcut system distinguishes board-focus from composer-focus contexts.

When the composer is focused, board shortcuts do NOT fire. When the composer is unfocused (Esc or click outside), board shortcuts resume.

## The touch ergonomics contract

The composer is touch-first on phone and tablet. Touch targets meet the 44x44 minimum.

Touch behaviors:
- Tap the collapsed composer to expand (single tap)
- Tap a mode in the mode switcher to switch (single tap)
- Tap a field to focus and bring up the keyboard (single tap; standard input behavior)
- Swipe down on the expanded composer (bottom sheet) to collapse
- Long-press on a node's Act affordance to open Act, which can then open the composer in the relevant mode (existing pattern; UX-001.3 does not redesign Act)

Haptic feedback (if available without dependency burden):
- Light haptic on mode switch
- Light haptic on submit success
- Medium haptic on validation error

Haptics are NOT a hard requirement; if the platform does not support them cleanly, the composer functions identically without them.

## The cross-device parity rule

Mobile and web are equally required. The composer is touch-first but keyboard-capable.

Failure conditions:
- Do not ship UX-001.3 if the composer only works cleanly on phone but feels broken on desktop browser
- Do not ship UX-001.3 if the composer only works cleanly on desktop but the mobile bottom sheet feels janky or obstructs the Timeline
- Do not ship UX-001.3 if the composer's expanded state pushes the Timeline below the UX-001.2 offset cap on any required viewport
- Do not ship UX-001.3 if mode switching loses drafts on any platform
- Do not ship UX-001.3 if the keyboard and touch behaviors conflict (e.g., keyboard shortcut interferes with normal typing in a field)

## The acceptance criteria

UX-001.3 ships when all of these hold:

1. The composer displays its target in compact form by default at every required viewport, with the target display heights within the brief's bounds.
2. The target display answers "what is this action applying to?" without ambiguity for every supported mode.
3. The composer's expanded target display is explicit-user-action only.
4. Mode switching preserves drafts per-mode and per-target within the composer session.
5. Mode switching does not require navigation or full-page transitions.
6. Mode switching takes effect within one user interaction.
7. The composer sits below the Timeline in Timeline view (lower dock position from UX-001.2).
8. The composer's collapsed height does not push the Timeline above the UX-001.2 offset caps (128/168/200).
9. The composer's expanded state does not displace the Timeline upward; it overlays or pushes downward.
10. The mobile expanded state behaves as a clean bottom sheet with swipe-to-dismiss.
11. Validation appears as compact contextual signals attached to the relevant field or the composer header.
12. Validation does NOT appear as full-screen modals, banner notifications, or verdict copy.
13. Semantic validation (advisory classifier signals) appears as composer-only Observations directed at the author.
14. The submit path is unchanged (existing submit-argument Edge Function; no new direct insert; no new service-role usage).
15. Keyboard shortcuts (Cmd/Ctrl+Enter, Cmd/Ctrl+K, Tab, Esc) work reliably without conflicting with board shortcuts.
16. Touch targets meet the 44x44 minimum on all interactive composer elements.
17. The "Acting on" label coordinates with UX-001.2's repositioned selected-message readout (the readout's "Acting on" and the composer's "Acting on" reference the same target).
18. Each canonical mode in the brief's mode list is supported.
19. No new backend write path is introduced.
20. No migration is introduced.
21. No Edge Function is introduced.
22. No raw internal codes appear in user-facing copy.
23. No verdict, truth, winner, loser, proof, or correctness language appears.
24. The Phase 3 framing section is added to docs/core/current-status.md and is detailed enough for UX-001.4 to consume.

## The pre-launch source audit (designer's first deliverable)

Before producing the design document, the designer audits the current composer implementation to identify:

1. The composer's actual component file paths (the brief uses "composer component files" generically; the audit names them)
2. The current mode list (which of the brief's canonical modes are already implemented, which are new for UX-001.3)
3. The current target display behavior (does the composer currently show its target? If yes, how? If no, what needs to be added?)
4. The current mode-switching behavior (does the composer currently support mode switching with draft persistence? Where do drafts live?)
5. The current validation surface (where does validation currently appear? What patterns are in use?)
6. The current keyboard shortcut allocations (which shortcuts does the composer currently consume?)
7. The current touch behaviors (does the composer currently behave as a bottom sheet on mobile? If not, what needs to change?)
8. The integration points with UX-001.2's repositioned readout (where does the composer currently coordinate with the readout? Where do the integration points live?)

The audit produces a table at the top of docs/designs/UX-001.3.md with one row per audit item. The columns are:

| Audit item | Current state | Brief contract | Gap or alignment |

The audit's findings determine the implementation scope. If significant gaps surface (e.g., the composer currently has no mode-switching infrastructure and all modes are separate components), the designer surfaces the gap and the operator decides whether the brief's scope holds or needs revision.

## The designer's framework

The designer's questions for UX-001.3 are translation questions, not invention questions. The brief specifies the product intent; the designer translates the intent into implementation specifications.

Q1 — Composer architecture audit. Map the current composer implementation per the eight audit items above. Produce the audit table.

Q2 — Mode list reconciliation. For each canonical mode in the brief, identify whether it is currently implemented or new. For each currently-implemented mode, identify whether its schema needs changes to fit the consolidated composer. For each new mode, specify the schema.

Q3 — Target display implementation. Specify per-mode target display content and the rendering pattern. Match the brief's height bounds.

Q4 — Mode-switching implementation. Specify the mode switcher's placement, interaction pattern, and draft persistence mechanism.

Q5 — Composer position and height implementation. Specify the collapsed and expanded heights per breakpoint within the brief's bounds. Verify that the expanded heights do not push the Timeline above UX-001.2's offset caps.

Q6 — Validation surface implementation. Specify the validation patterns (inline field validation, header chips, disabled-with-reason, semantic advisory chips) with examples per mode.

Q7 — Keyboard shortcut implementation. Specify the shortcut allocations and the focus context handling (board vs composer focus).

Q8 — Touch ergonomics implementation. Specify the bottom sheet behavior, swipe-to-dismiss, haptic feedback (if applicable), and 44x44 touch targets.

Q9 — Selected-message readout coordination. Specify the integration point with the readout's "Acting on" label.

Q10 — Cross-viewport verification. Specify how the design holds across the four required viewports (390x844, 1024x1366, 1366x768, 1920x1080).

Q11 — Regression preservation. Specify how the no-regression list (submit path, selected-node persistence, IX-003 keyboard navigation, branch topology, action dock permissions, role gating, semantic doctrine, no-verdict copy, no-route-transition principle) is preserved.

Q12 — Reviewer measurement strategy. Specify how the reviewer mechanically verifies the target display, mode switching, position contracts, validation surfaces, keyboard shortcuts, touch behaviors, and the no-regression requirements.

## The credibility framing

UX-001.3 is the card that makes the composer feel like one system. UX-001.1 made the shell professional, UX-001.2 made the Timeline the primary board, and UX-001.3 makes the action surface coherent. If the composer continues to feel like a collection of separate forms after UX-001.3 ships, the consolidation premise is compromised. The card must deliver a composer where the user always knows what they are acting on, can switch action types without losing work, sees validation as contextual signals rather than scattered messages, and experiences the composer as a persistent contextual command center on every device.

The composer is also where the no-verdict doctrine receives its most visible test. Composer-only advisory signals (semantic referee outputs directed at the author) must read as helpful editorial assistance, not as judgment. The user composing a draft should feel guided, not accused. UX-001.5A's full registry will consolidate this; UX-001.3 establishes the pattern using the existing semantic referee surface.
