## Goal (one paragraph)

Produce the first **human** usability evidence for the disagreement loop. The synthetic corpus runs validate the
**instrument** — they prove skill gates, determinism, and zero provider spend
(`docs/testing-runs/2026-06-12-xai-adversarial-bot-corpus-dry.md`, `docs/testing-runs/2026-06-12-ai-driven-bot-corpus-dry.md`;
slate priority-order item 7). They cannot tell us whether a **first-time human** understands the surface — whether they can
find the point under dispute, ask for a source on the right target, critique a claim without attacking a person, branch a
tangent without losing the mainline, and read the Referee Card back in their own words. This card produces that evidence
path as two committed Markdown artifacts: a facilitator-run **protocol** and a **results template**. The doctrine that
shapes every line: the protocol is **procedural only** — the facilitator never hints, never asks the participant to judge
who is right or wrong, and the captured fields describe the participant's *navigation and comprehension*, never a verdict
on the debate content (cdiscourse-doctrine §1, §2, §9). The acceptance-gate invariant is honored by construction: the
protocol exercises only the shipped surfaces, and the shipped surfaces never let a classifier block a post (the
deterministic engine is the sole gate). With n ≤ 3 this is a **directional first baseline only** — explicitly **no
quantitative claims** — and every stuck point becomes a **new card**, not an inline fix.

## Capture-field hardening decision (required; recorded for traceability)

The issue lists 10 per-task capture fields. **Three of them embed prohibited verdict tokens at camelCase joints**
(`…Correct…`, `…Truth`). They pass today's word-boundary ban scanners (the token is glued inside an identifier), but the
slate's validation pass recommended token-free renames at REF-006 design time. **This design ADOPTS the token-free names**
and carries a mapping table back to the issue's original names so the measurement intent stays traceable. All field names
appear in backticks everywhere (they are code-style identifiers, never participant-facing prose).

| Issue field name (original, superseded) | Adopted token-free name | Why renamed | Measurement intent (unchanged) |
|---|---|---|---|
| `askedSourceOnCorrectTarget` | `askedSourceOnIntendedTarget` | the `…Correct…` joint glues a verdict token (`correct`) into the identifier | Did the participant aim the source/quote ask at the node currently under dispute, rather than a different node? |
| `refereeCardParaphraseCorrect` | `refereeCardParaphraseMatched` | the `…Correct` suffix glues a verdict token (`correct`) into the identifier | Did the participant's paraphrase of the Referee Card match its three zones (relation · open task · next move)? |
| `confusedByScoreHeatOrTruth` | `mixedUpScoreHeatWithStanding` | the `…Truth` joint glues a verdict token (`truth`) into the identifier | Did the participant read heat or a number as truth/winner rather than as activity / point-standing-in-the-game? |

The remaining seven keep their issue names verbatim:
`timeToFindDisputedPointSec` · `pickedBackendTypeManually` · `rawCodeSeen` · `branchPreservedMainline` ·
`usedAccusationInsteadOfProcedure` · `stuckPointText` · `suggestedCopyFix`.

Notes that keep the rename honest:
- `usedAccusationInsteadOfProcedure` is **kept** — `accusation` is not a prohibited token, and the field describes the
  *move's relation to the claim* (procedural framing, exactly like REF-005's bounded "About the person rather than the
  claim" concern type), not a person-verdict.
- `mixedUpScoreHeatWithStanding` uses `standing` (point standing in the game), which is doctrine-clean (cdiscourse-doctrine
  §1 — strength/standing describes the *point's* gameplay state, never truth).
- `rawCodeSeen` is a **failure capture**, not a neutral observation: any `snake_case` token, family ID, rawKey, classifier
  ID, or internal code reaching the participant is a **task failure** (the whole loop's plain-language doctrine, §9, is the
  thing under test). The results template frames it as such.

## Scope-reality audit (pre-launch; required for a layout/data-dependent card)

A card whose success depends on **what the participant actually touches on the current build** must be audited against the
live surfaces before the protocol is written, not against the brief's assumptions. Audited at `b515da6`:

### Finding 1 (load-bearing) — the side action rail is minimal; **Act is the canonical home** for Ask-source / Split-branch / Flag

The brief's mental model (and CLAUDE.md's Stage 6.4 narrative) describes an expanded observer rail with
`Ask source · Open timeline · Share` and a participant set with `Ask source · Split branch · Flag · Qualifiers`. **The
shipped rail has since consolidated** (`src/features/arguments/ArgumentSideActionRail.tsx:88-125`,
UX-001.4 migration):

- **Observer rail** renders only `Watch · Join For · Join Against · Share` (`ArgumentSideActionRail.tsx:92-101`).
- **Participant-on-another-bubble rail** renders only `Reply · Disagree` (`:103-111`).
- **Own-bubble rail** is empty — it collapses to an `Open Act ▾` label (`:113-119`).
- The file's own comment is explicit: *"Act is the canonical home — open Act on a node to access Ask source / Ask quote /
  Split branch / Flag / Qualifiers / Request deletion."* (`:90-91`).

**Consequence for the protocol:** Task 2 ("ask for a source on the right target") and Task 4 ("branch a tangent") do **not**
start from a top-level rail button. Their real entry points are **(a)** a **Referee Card zone-3 next-move button** when one
is present (`Ask for a source` / `Open a side issue`, rendered on the active node's detail by REF-003/REF-004), or **(b)**
**open Act on the node** and pick the entry there. The protocol's task steps below name **both** valid paths and instruct
the facilitator to observe **which** path the participant discovers first (this is exactly the `whereTheyTappedFirst`
signal the brief wants). This is the founding reason a pre-launch reality audit is mandatory here: a protocol written
against the brief's stale rail model would have sent the participant hunting for a button that no longer exists, producing
a false "stuck point."

### Finding 2 (worth flagging to the facilitator) — a cosmetic label split on the Request-review affordance

REF-005's implementer note records a deferred relabel: the **bubble chip** reads `Request review`
(`ArgumentBubbleActions.tsx:23`) and the **composer title** reads `Request review`
(`RequestReviewComposer.tsx:62`), but the **board-Act popout `flag` entry still reads `Send for review`**
(`actPopoutModel.ts:328`) because that file is pinned by a zero-diff boundary test. None of the 5 tasks targets
request-review directly, so this does not change a task step — but the facilitator is told (in the protocol's ground-rules)
not to be surprised by the two labels for the same structured flow, and an incidental participant remark about it is a
valid `suggestedCopyFix`.

### Finding 3 — the gallery action label a first-timer actually sees

`ConversationGalleryScreen.tsx:636` renders the card action as `Continue →` only when `card.hasUserJoined`, else `Observe →`
when the room is open, else `Open →`. A **first-time participant has joined nothing**, so they will see `Observe →` or
`Open →` — **never** `Continue →`. The protocol's Task 1 names `Observe →` / `Open →` as the labels to expect and does not
mention `Continue →` (which would mislead the facilitator with a first-timer).

### Finding 4 — the micro-moment hint is conditional

The one-line "micro-moment" prompt above the body (`ArgumentGameSurface.tsx:1796-1808`, `entryHint.verbPhrase`) surfaces on
**gallery** entry per bucket (e.g. `needs_rebuttal` pre-activates the root), but is **suppressed** (empty `verbPhrase`) on
notification/deep-link entry (`deepLinkEntryHint.ts:51-53`). The protocol routes every participant through the **gallery**
(not a deep link), so the banner is in play — but the protocol tells the facilitator the banner "may or may not appear"
and to capture whether the participant notices/uses it, never to depend on it.

### Allowed scope vs required scope

- **Allowed by the brief:** the two `docs/testing-runs/` files + design doc + status entry. **Required to deliver the
  contract:** identical. **No scope correction needed** — the deliverables are docs-only and the reality audit changed the
  *content* of the task steps (which entry points to name), not the *file scope*. Effort estimate **S** stands.

## Deliverable 1 — `docs/testing-runs/REF-006-usability-smoke-protocol.md` (full section structure)

The implementer builds the file with exactly these sections. Verbatim copy is given where it is load-bearing (doctrine
hygiene), so the implementer does not re-author neutral wording and risk a verdict/person token or a `snake_case` leak.

### § Title + status block
`# REF-006 — Usability smoke protocol (5-task first-user pass)` · run-lane note (human-run; zero provider spend) ·
verified-at-HEAD `b515da6` · a one-line pointer to the results template.

### § Why this run exists (instrument vs usability — verbatim framing)
A short paragraph, copy fixed here so it cannot drift toward a quantitative claim:

> *The synthetic corpus runs prove the instrument: skill gates fire, the engine is deterministic, and no provider is
> billed. They do not prove that a first-time person understands the surface. This run is that evidence. With one to three
> participants it is a directional first baseline — it is not a metric, not a pass/fail gate, and produces no quantitative
> claim. Its output is a list of verbatim stuck points, each of which becomes a new card.*

### § Recruitment note
- 1–3 first-time participants (people who have not used CDiscourse and have not read its docs).
- **No documentation is provided** to the participant — the point is to see the surface cold.
- Participants are identified only as **P1 / P2 / P3** (see § Privacy).
- One facilitator runs the session and records; the participant drives the device.

### § Facilitator ground rules (neutral; the doctrine spine of the protocol)
Copy fixed here:

- The facilitator reads each task's script **verbatim** and then **stays quiet**. No hints, no leading questions, no
  pointing at the screen.
- The facilitator **never asks the participant to judge who is right or wrong**, never asks which side is winning, and
  never uses an evaluative word about the debate content. Tasks are **procedural only** — they ask the participant to
  *do* a thing or *say back* what they see, never to *rule* on the argument.
- If the participant asks "what should I do?", the facilitator replies only: *"Whatever makes sense to you — there's no
  wrong move here."* and records the question as a stuck point.
- The facilitator records, in the participant's **own words**, what they call each thing (the vocabulary table feeds this).
- The facilitator notes **where the participant taps first** for each task and the **time to first move**.
- Reminder (Finding 2): the structured concern flow reads `Request review` on the bubble chip and the composer but
  `Send for review` in the Act popout — this is a known label split, not a participant error.

### § The 5 tasks (verbatim from the issue, each with start screen + neutral script + capture rules)

Each task block has four parts: **(a) Start screen** (the real shipped surface + labels, from the reality audit);
**(b) Task — verbatim from the issue**; **(c) Facilitator script — neutral, read aloud**; **(d) Capture** (the fields to
record). The per-task start-screen mapping is consolidated in "Per-task start-screen mapping" below; the protocol embeds
the same mapping inline.

1. **Open a room and identify the exact point currently under dispute.**
   - *Start:* the Conversation Gallery (`ConversationGalleryScreen`), grouped into lanes — e.g. **Needs first rebuttal**,
     **Jump in now**, **Source trail fights**, **Evidence needed**, **Definition fights** (live labels,
     `gameCopy.ts:1022-1083`). The card action button reads **`Observe →`** or **`Open →`** for a first-timer. Tapping it
     opens the room **in read / observer mode** (no "choose side" modal). The active node's detail shows the **Referee
     Card** with a **`Point under dispute: "…"`** quoted anchor and a **`Referee note: …`** line.
   - *Script (verbatim):* *"Open one of these rooms, then tell me in your own words the exact point the two sides are
     arguing about right now."*
   - *Capture:* `timeToFindDisputedPointSec` · `whereTheyTappedFirst` (free note) · `stuckPointText` (verbatim) ·
     `rawCodeSeen` (failure if any internal code appears) · their word for "the point under dispute".

2. **Ask for a source or quote on the right target.**
   - *Start:* the active node's detail. **Two valid paths** (Finding 1): the **Referee Card zone-3 button**
     `Ask for a source` (when present in next-best-moves), **or** **open Act on the node** and pick **`Ask source`**
     (`timelineNodeActionDockModel.ts:470`). The composer then shows what it is acting on
     (`Ask source for {type} · "{excerpt}"`, `composerActingOnModel.ts:166`) so the participant can confirm the target.
   - *Script (verbatim):* *"Ask the other side to back up that specific point with a source or a quote."*
   - *Capture:* `askedSourceOnIntendedTarget` · `whereTheyTappedFirst` (Referee Card button vs Act vs hunted elsewhere) ·
     `pickedBackendTypeManually` (expected false — the one-box infers the move) · `stuckPointText` · `rawCodeSeen`.

3. **Reply with a claim-level critique without referring to the person.**
   - *Start:* the active node's detail → **`Reply`** or **`Disagree`** on the rail (`ArgumentSideActionRail.tsx:109-110`),
     **or** Act → **Respond** (the one-box `respond` composite, QOL-030). The one-box infers the type; the participant is
     not asked to pick a backend type.
   - *Script (verbatim):* *"Write a reply that pushes back on the point itself — not on the person who made it."*
   - *Capture:* `usedAccusationInsteadOfProcedure` (did the wording target the person vs the claim) · `pickedBackendTypeManually`
     (expected false) · `stuckPointText` · `rawCodeSeen`. (Note for the facilitator: the app **never blocks** the post — the
     engine is the sole gate; advisory surfaces only suggest. Record what they *wrote*, not whether the system "allowed" it.)

4. **Branch a tangent without losing the main point.**
   - *Start:* the active node's detail. **Two valid paths** (Finding 1): the **Referee Card zone-3 button**
     `Open a side issue` (the `branch_tangent` entry / recovery route), **or** **open Act** and pick the branch entry.
     After branching, the horizontal timeline shows the branch on its own lane while the mainline node stays selectable.
   - *Script (verbatim):* *"Start a side point that's related but separate — without losing the main argument."*
   - *Capture:* `branchPreservedMainline` (did the mainline stay anchored/findable) · `whereTheyTappedFirst` ·
     `stuckPointText` · `rawCodeSeen`.

5. **Read the Referee Card and explain: what the move is doing, what remains open, and which next move you would take.**
   - *Start:* the **Referee Card** on the active node (`RefereeCardView`). The three zones map one-to-one to the three
     things asked: **zone 1** `Referee note: {relation}.` = what the move is doing; **zone 2** `The open task is {burden}.`
     (or a terminal-state line) = what remains open; **zone 3** = 2–3 candidate next-move buttons. When the move has no
     observation yet, zone 1 reads **`No referee notes yet on this move.`** (`RefereeCardView.tsx:60`).
   - *Script (verbatim):* *"Read this card out loud, then tell me three things in your own words: what this move is doing,
     what's still open, and what you'd do next."*
   - *Capture:* `refereeCardParaphraseMatched` (did the paraphrase match the three zones) · `mixedUpScoreHeatWithStanding`
     (did they read heat or a number as truth/winner) · `stuckPointText` · `rawCodeSeen` · `suggestedCopyFix` (any wording
     they tripped on).

### § Capture rules (how to record)
- **Verbatim stuck points** — write the participant's exact words in quotes; do not paraphrase.
- **Time-to-first-move** — start a stopwatch when the script ends; stop at the participant's first deliberate action.
- **Where they tapped first** — note the first interactive element they reached for, even if it was wrong.
- **Their words** — record the participant's own name for each thing (feeds the vocabulary table).
- **`rawCodeSeen`** — if any `snake_case` token, family ID, rawKey, or internal code ever reaches the screen the
  participant is reading, that task is a **failure**; record exactly what appeared and where.

### § Privacy
- No PII beyond what the participant volunteers. Do not record names, emails, faces, or device identifiers.
- Quotes in the committed results are **anonymized**; participant identifiers are **P1 / P2 / P3** only.
- If a participant volunteers identifying detail in a quote, the facilitator redacts it before committing.

### § Operator dry walk-through (acceptance criterion — the card does not close until this passes)
A short checklist the **operator** runs **once**, solo, before recruiting any participant, to confirm every task is
executable on the current build. This is an acceptance criterion (issue AC: "One dry walk-through of the protocol by the
operator confirms the tasks are executable on current builds"). The checklist names the build/screen each task starts from
(see "Operator dry walk-through checklist" below) and has a checkbox + a "confirmed on build ___" slot per task.

### § What happens to findings
- Every stuck point and every `suggestedCopyFix` becomes a **new GitHub issue** (a new card), filed against the right epic.
- **Findings are not fixed inline in this card.** REF-006 produces the evidence; the fixes are separate cards.
- The "their words vs our labels" table is offered as **input to META-1D (#79)** — its current scope is the 26 META-001
  codes, so any extension of its scope to the REF rail-state vocabulary is **proposed via a comment on #79**, never assumed
  here.

card **does not close** until every line is confirmed. (This checklist is reproduced inside the protocol file; named here

1. [ ] **Gallery loads** with grouped lanes; at least one room shows `Observe →` or `Open →`. *(build ___)*
2. [ ] **Open a room as a first-timer** → it enters in **read / observer** mode with **no "choose side" modal**. *(build ___)*

## Edge cases (the facilitator / operator must handle)

- **Empty gallery / no open rooms** → the operator seeds a room (any existing dev room works) before the session; the
  protocol notes that a first-time participant needs at least one `Observe →` / `Open →` card on screen.
- **A room with no Referee observation yet** → Task 5's card reads `No referee notes yet on this move.` — that is a valid,
  non-error state; the participant reads zones 2–3 and the empty zone-1 teaching line. Not a stuck point by itself.
- **Participant joins a side** (taps `Join For` / `Join Against`) mid-Task-1 → allowed; the rail switches to the
  participant set (`Reply · Disagree`); the facilitator records it and continues. The tasks still resolve.
- **Participant lands on their own bubble** → the rail collapses to `Open Act ▾` (own-bubble safety: no edit/disagree/flag
  on your own move). The facilitator records any confusion as a stuck point; it is not a protocol error.
- **Micro-moment banner absent** (Finding 4) → expected on some buckets / on deep-link entry; the protocol never depends
  on it.
- **The two Request-review labels** (Finding 2) → if a participant wanders into the concern flow and remarks on
  `Request review` vs `Send for review`, that is a valid `suggestedCopyFix`, not a task failure.
- **Participant asks "who's winning?"** → the facilitator does **not** answer and does **not** evaluate; replies only
  *"This isn't about who's winning — just do what makes sense to you,"* and records the question (it is itself a usability
  signal that the surface read as a scoreboard, captured under `mixedUpScoreHeatWithStanding`).
- **A raw code appears on screen** → record under `rawCodeSeen`, mark the task failed, and file it as a new card (a
  plain-language §9 regression).
- **Offline / network failure during a post** → the existing submit path owns its own offline handling; the protocol notes
  that a failed post is a system event to record, never a participant error, and never a verdict.
- **n = 1** (only one participant available) → still valid as a directional baseline; the template's quantitative-claims
  reminder applies even more strongly.

## Doctrine self-check

**Acceptance-gate invariant (verbatim, slate §2.1):**
*AI/MCP classifiers MUST NEVER be the submission acceptance gate. The deterministic rules engine
(`src/domain/constitution/engine.ts`, mirrored for the server at `supabase/functions/_shared/constitution/`) is the sole
gate. Classifiers run after an argument is stored. No path may block, reject, route, or delay an ordinary user post.*
→ **Respected.** The protocol adds no code and no gate; it exercises only the shipped surfaces, where the Referee Card /
Open Issue layer is advisory and post-storage. Task 3's facilitator note explicitly tells the facilitator the post is
**never blocked** (the engine is the sole gate) and to record what the participant *wrote*, not whether the system
"allowed" it — so the run can never be mis-read as a classifier acting as a gate.

- **cdiscourse-doctrine §1 (no truth labels; score never blocks).** Tasks are **procedural only**; the facilitator never
  asks who is right/wrong and never uses an evaluative word about the content. The renamed fields drop the three
  camelCase-glued verdict tokens (`…Correct…` → `…Intended…` / `…Matched`; `…Truth` → `…Standing`). The prohibited tokens
  (winner, loser, correct, incorrect, truth, untrue, dishonest, liar, manipulative, extremist, propagandist, stupid, idiot,
  verdict, "bad faith", "proof of") appear in this doc and the deliverables **only** as prohibitions / inside the
  ban-scan framing.
- **§2 (heat = activity/friction).** `mixedUpScoreHeatWithStanding` is a capture for the *failure* mode where a participant
  reads heat as truth/winner — the protocol treats that as a usability signal, never endorses it; the gallery lane labels
  it names (`Jump in now`, `Source trail fights`) are the shipped activity-framed copy, not popularity claims.
- **§3 (popularity is not evidence).** No task ranks, gates, or rewards by engagement/views/virality; the protocol reads
  the shipped surfaces as-is.
- **§4 (AI moderator limits) / §7 (no app-side AI).** Zero provider spend, zero AI call of any kind — the deliverables are
  hand-run Markdown.
- **§9 (plain language).** `rawCodeSeen` is a **failure** capture: any `snake_case` / family ID / rawKey reaching the
  participant fails the task. The participant-facing facilitator scripts and template seed rows carry **zero** internal
  codes in prose; every code identifier in this design is in backticks.
- **§10a (Observations vs Allegations).** Not directly exercised, but Finding 2 surfaces the Request-review label split
  honestly to the facilitator; the protocol never asks a participant to attach a person-label.
- **§10 (v1 scope guards).** No voting/winner scoring, OAuth, push, public API, or search introduced.
- **Doctrine-constraint edge case — "what if the facilitator's wording nudges the participant toward a verdict? — it
  doesn't":** the scripts are fixed verbatim, procedural-only, and the "whatever makes sense to you" non-answer is the only
  permitted response to a request for guidance.
