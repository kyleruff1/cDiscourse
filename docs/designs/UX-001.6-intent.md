## §1 — Why this brief exists

UX-001.6 is the visible quality baseline for the entire UX-001 epic. The five prior cards (UX-001.1 through UX-001.5) shipped product contracts in code: a brand-forward shell with band-aware breakpoints (UX-001.1), a Timeline-first room viewport with hard offset caps (UX-001.2), a persistent contextual composer with per-target draft persistence (UX-001.3), a three-menu interaction model with browser-only key badges and dismissible scrim (UX-001.4), and 12 source-neutral visual annotation primitives plus the additive `semanticFlagsChips` Inspect integration (UX-001.5). UX-001.6 verifies those contracts hold across the device matrix in actual viewport behavior — not only through component-level prop tests.

The card is mostly QA, not feature work. Bounded visual polish (acute fixes surfaced by QA) is permitted within an operator-set budget. Any pre-existing regression or larger scope-defect surfaced during QA is documented as a separate-card candidate, not folded into UX-001.6.

UX-001.6 is the last card before UX-001.7's visual design system consolidation. Its QA findings inform UX-001.7's scope: which surfaces passed cleanly, which needed acute fixes, which carry pre-existing regressions UX-001.7 must address.

## §3 — The central product rule

The cross-device QA matrix verifies that every product contract established by UX-001.1 through UX-001.5 holds in actual viewport behavior across the device matrix — phone, tablet, narrow browser, wide browser, and at least two extension viewports. Bounded visual polish is permitted within the operator-set acute-fix budget; any defect that cannot be fixed within budget is documented as a candidate for a separate card, not folded into UX-001.6's scope. The matrix produces a pass / fail / finding verdict per surface per viewport, and the resulting findings inform UX-001.7's visual design system consolidation scope.

## §4 — The viewport matrix

The four hard-blocker viewports are binding from `expo-rn-patterns` §Cross-device QA viewport conventions (committed at 13dccf3 with operator approval):

| Viewport | Label | Interaction context |
|---|---|---|
| 390 × 844 | Phone (iPhone-class) | Touch-first; no keyboard hints |
| 1024 × 1366 | Tablet portrait (iPad Pro 11-class) | Touch-first; optional keyboard |
| 1366 × 768 | Narrow browser (small laptop / desktop window) | Keyboard-first; key badges render |
| 1920 × 1080 | Wide browser (desktop) | Keyboard-first; key badges render |

The two extension viewports proposed (from the skill's enumerated list, two of four):

| Viewport | Label | Rationale for inclusion |
|---|---|---|
| 412 × 892 | Phone (large Android) | The 390 × 844 hard-blocker captures the iPhone class; 412 × 892 captures the large-Android edge case (Pixel 8 Pro / Samsung Galaxy S24) where the layout budget is similar but glyph rendering and touch-target spacing can drift independently of iOS. Including it ensures the compact strip's 45px content min-height holds on the busiest large-Android density. |
| 768 × 1024 | Tablet portrait (iPad-class) | The 1024 × 1366 hard-blocker captures iPad Pro 11. 768 × 1024 captures the iPad-class tablet (iPad mini, iPad 10.2") that the hard-blocker doesn't represent — the narrower portrait dimension stresses the compact strip's tablet-band rendering (51px / 56px cap) and verifies the Timeline offset cap (168px) holds without the wider iPad Pro layout. |

The total QA matrix is therefore 4 hard-blocker + 2 extension = 6 viewports. The matrix stays tractable; the per-surface verification table is 6 columns wide.

Operator may add a third or fourth extension during Stage 0 review (360 × 800 small-Android phone, 2560 × 1440 4K-class desktop) if QA coverage gaps warrant it; the brief caps the total at 4 extensions.

## §5 — The QA matrix contract

The QA matrix tests every surface listed in the Phase 5 alternate handoff's "outputs UX-001.6 must NOT modify" list, plus every surface in the Phase 1-4 framing handoffs' read-only file reference tables. For each surface at each viewport, the matrix verifies three classes of contract:

**Structural verification:**
- The surface renders without overflow or clipping
- The surface respects its committed offset cap (e.g., UX-001.2 Timeline offset cap, UX-001.3 collapsed strip cap, UX-001.4 popout sizing)
- The surface's API contract is intact (e.g., `semanticFlagsChips` additive sibling unchanged, `RefereeBannerView` composer-only rendering)

**Accessibility verification (per `accessibility-targets` §Keyboard hints platform-conditional pattern):**
- Focus visibility — keyboard-focused control has a visible focus ring distinct from hover and active state
- Touch target size — every interactive control meets 44×44 logical pixels via visual size or `hitSlop`
- Screen-reader name — the `accessibilityLabel` reads naturally without including the keyboard shortcut (the key badge is visual-only; screen readers get the action name)

**Visual-polish observation:**
- Spacing, alignment, color-token consistency, copy consistency
- Findings categorized as **acute** (fixable within the §6 budget) or **pre-existing-regression** (documented for separate card, not fixed in this card)

The matrix produces a pass / fail / finding verdict per cell (surface × viewport). Pass cells require no action. Fail cells block the card. Finding cells route to either the acute-fix budget or the separate-card register.

## §6 — The bounded acute-fix budget

UX-001.6 may apply visual polish fixes to defects surfaced during QA. The budget is **5-10 acute fixes maximum** — operator-set ceiling. The Stage 1 designer refines the exact number based on the scope-reality audit's surface-level findings (e.g., if the audit surfaces 7 likely-acute surfaces, the designer proposes a budget of 8 to leave headroom).

Acute fixes MAY include:
- Small spacing issues (4-8 px adjustments within a single component)
- Token-based color inconsistencies (a component using a near-token hex literal where a token exists)
- Text wrapping / truncation defects (an `ellipsizeMode` missing on a one-line label)
- Touch target sizing defects (a primitive sized < 44×44 without `hitSlop` — add the missing `hitSlop`)
- Focus-ring visibility defects (a control whose focus state is invisible against its background)
- Overflow or scroll containment bugs (a chip strip whose overflow chip count is wrong)
- Viewport-specific layout bugs (a strip that wraps to two rows at one specific extension viewport)
- Minor copy consistency issues (`Share` vs `share` between two surfaces; the canonical case wins)

Acute fixes MAY NOT include:
- New product features (UX-001.7 territory)
- Architectural changes (e.g., refactoring a state-lifting pattern)
- Backend modifications (no migrations, no Edge Functions, no schema changes)
- New write paths
- Anything that warrants its own card (per the §13 disallowed list)
- Changes to UX-001.{1-5} pure-TS model files (e.g., `actPopoutModel.ts`, `composerKeyboardModel.ts`, `boardMenuKeyboardModel.ts`, `inspectContentBuilder.ts`, `annotationKindTokens.ts`) — these are the load-bearing models the prior cards locked

If QA surfaces more than 10 acute findings, the card halts via the designer's stop condition. UX-001.6 ships as QA-only with a UX-001.6.1 follow-up card filed for the remaining acute fixes. The operator decides the cutoff during Stage 0 review of the audit.

## §7 — The doctrine surface verification contract

UX-001.6's QA matrix MUST verify the following via file-scan tests (not only visual inspection):

**Verdict ban-list (per `cdiscourse-doctrine` §1):**
- No `winner`, `loser`, `liar`, `dishonest`, `bad faith`, `manipulative`, `extremist`, `propagandist`, `stupid`, `idiot`
- No `true`, `false`, `correct`, `proof`, `proven`, `verified` as truth assertions in user-facing copy
- Scan covers all UX-001.{1-5} source files plus any new test file UX-001.6 adds

**Internal-code ban-list (per `cdiscourse-doctrine` §9):**
- No raw classifier IDs in user-facing UI (e.g., `topic_satisfaction_lexical`, `source_chain_lexical`, `anti_amplification`, `evidence_debt`, `platform_support_warning`, `validation_failed_after_retries`, `max_depth_reached`, `synthesis_ready`, `submit_failed`)
- Every code routes through `gameCopy.toPlainLanguage` before reaching a `<Text>` node
- Unknown codes are suppressed, not echoed

**Service-role + AI ban-list (per `cdiscourse-doctrine` §6, §7):**
- No service-role usage in any client code (file-scan ban on `SUPABASE_SERVICE_ROLE_KEY`, `service_role`, `serviceRoleClient` patterns in `app/` and `src/`)
- No new AI calls from the production app (file-scan ban on `Anthropic`, `xAI`, `@anthropic-ai/sdk`, `openai`, `gemini` imports in `app/` and `src/`)

**Observations/Allegations doctrine (per `cdiscourse-doctrine` §10a):**
- If any annotation chip renders with a `source` field, the field's value is `'machine'` or `'user'` (no other values)
- Composer-only Observations (e.g., `shifts_to_person_or_intent`, `contains_unplayable_insult_only`, `needs_pre_send_pause`) do NOT render on the target's node — only in the composer (verified via the bounded `RefereeBannerView` integration UX-001.5 shipped)

These verifications are blocking. Failure on any one is a fail cell in the matrix; multiple fails block the card.

## §9 — Cross-device parity

The matrix gives equal weight to each viewport. A failure on phone is as blocking as a failure on wide-browser; the brief does not privilege one device class over another.

Touch-first and keyboard-first divergence is acceptable when documented (per `expo-rn-patterns` §Cross-device QA viewport conventions). The same screen can render different chrome — most notably:

- Browser-only key badges (A/I/G next to menu triggers) verified ABSENT at phone (390×844) and tablet (1024×1366) widths
- Browser-only key badges verified PRESENT at narrow browser (1366×768) and wide browser (1920×1080) widths
- Composer keyboard shortcuts (Cmd/Ctrl+Enter, Cmd/Ctrl+K) verified active on web at all widths; native path does not install the keydown handler (touch only) — verified by `Platform.OS` branching, not by viewport width

The no-color-only rule (per `accessibility-targets`) and the 44×44 touch target rule apply to BOTH touch and keyboard paths.

## §15 — The designer's framework

Translation questions for Stage 1, not invention questions. The designer answers each with a concrete pattern grounded in the existing codebase, the prior framing handoffs, and this brief.

- **Q9 — Reviewer measurement strategy.** What commands the reviewer runs (e.g., `npm run test -- uxOneOneSix`, the diff check, the doctrine ban-list scan).
