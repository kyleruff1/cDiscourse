## Goal

The sidecar surfaces a "Suggested next move" slot today (SC-003) that is **always null** — typed `SuggestedMoveStub = never`. ST-002 fills that slot with a small ordered list of suggested moves derived deterministically from upstream gameplay state. Each suggestion carries a one-line plain-language rationale built from the same RULE-003 helper copy the rest of the sidecar already uses, plus a preset key that lets the dock and the composer round-trip the suggestion through `quickActionToPreset(...)` if the user picks it.

The doctrine constraints that shape the design (all from `cdiscourse-doctrine` + the issue body):

1. **Suggestions describe the move shape, never the author.** No `you / your / they / their / the user / the author / the poster` in any rationale line. The rationale describes the board state: "A primary source has been requested.", "The cluster is ready to summarise.", etc.
2. **Score never blocks posting.** ST-002 is advisory. An empty suggestion list is a valid output, and the composer must keep its ordinary `reply` path open regardless.
3. **No popularity / engagement / heat input.** The deriver reads lifecycle / manual tag / auto metadata / axis / source-chain risk / evidentiary risk / depth / branch state / latest move type / no-rebuttal / stopReason / standing. It never reads engagement counts, view counts, or anything from the temperature band.
4. **No new presets in this card.** Every suggestion's `presetKey` is an existing entry in `quickActionPresets.ts` `QuickActionLabel`. Where the issue's 9-code suggestion vocabulary lacks a 1:1 preset (`challenge_mechanism`, `challenge_scope`, `branch_tangent`), the deriver maps to the closest existing preset (`challenge` / `narrow` / `branch`) and documents the gap. New presets are a follow-up card.
5. **No live AI.** Pure deterministic TS. Same inputs → same outputs, stable ordering, no async, no fetch, no Anthropic / xAI / Edge Function hop.
6. **The dock owns execution.** ST-002 produces data; the dock (SC-004) and composer (COMPOSER-001) execute. The view model exposes no `onPress`, no callback, no dispatch field.

## What ST-002 IS and IS NOT

### ST-002 IS

- A pure-TS deriver: `deriveSuggestedMoves(input): SuggestedMove[]`. Total. Deterministic. O(constant) on input vocabulary size + O(suggestions) on output.
- The producer that widens SC-003's reserved slot: `SuggestedMoveStub = never` → `SuggestedMoveStub = SuggestedMove`. Type-level widening only. No SC-003 behaviour changes — the SC-003 builder still always returns `suggestion: null` because the slot is populated by a downstream caller passing a built `SuggestedNextMove` model in.
- A small ordered list of suggestions (0–3 in v1), each carrying:
  - A typed `SuggestedMoveCode` (the 9 issue-body codes).
  - A `presetKey: QuickActionPresetKey` that already exists in `quickActionPresets.ts`.
  - A `dockAction: TimelineNodeActionDockActionCode | null` that lets the dock highlight the matching primary if it lines up.
  - A `rationale: string` ≤ 80 chars assembled from RULE-003 helper lines + a small phrase table — never authored fresh per call.
  - A list of `SuggestionSignal` records explaining what triggered the suggestion (debug + AN-003 diagnostics).

### ST-002 IS NOT

- An action executor. The dock executes. ST-002 only suggests.
- A verdict assigner. No code path turns "Should concede" into an accusation against the author. The rationale describes the cluster state ("The broad point was conceded; the narrowed claim is now in play.") never the player.
- A forced gate. The user can ignore every suggestion and post an ordinary reply. The composer keeps `reply` available.
- A lifecycle / metadata deriver. ST-002 reads `PointLifecycleMap` + `MoveMetadataLedger` only. It never imports `buildPointLifecycleMap`, `deriveAutoMetadataForMessage`, or any deriver entry point. Enforced by a forbidden-imports test.
- A NAV-005 queue. Cross-room "next move" surfacing is a separate follow-up card.

### ST-002 ↔ SC-003 ↔ SC-004 boundary

| Concern | Owner | Forbidden in others |
|---|---|---|
| Suggestion vocabulary (`SuggestedMoveCode`) | ST-002 | SC-003, SC-004 may not re-author. |
| Action vocabulary (`TimelineNodeActionDockActionCode`) | SC-004 | ST-002 type-imports only; never values. |
| Preset vocabulary (`QuickActionLabel`) | COMPOSER-001 | ST-002 reads, never invents. |
| Rationale phrasing | ST-002 (composing RULE-003 helper lines) | No new helper lines authored in ST-002. |
| `SuggestedNextMove` typed slot in sidecar | SC-003 (reserved) → ST-002 (widened) | ST-002 widens via a one-line type re-export only. |
| Composer state | COMPOSER-001 | ST-002 must not import any composer file. |

The boundary is enforced two ways:
- A source-scan test asserts `suggestedMovesModel.ts` contains zero references to `ArgumentComposer`, `argumentGameSurfaceModel` value imports (type-only is allowed), `buildPointLifecycleMap`, `deriveAutoMetadataForMessage`, `gradeChallenge`, `gradeRepair`, `applyAntiAmplification`, `supabase`, `fetch`, `anthropic`, `xai`.
- A field-name scan asserts the public types contain no fields whose name matches `onPress|onAction|dispatch|callback|handler`.

### Public types

```ts
// src/features/arguments/suggestedMovesModel.ts

import type { PointLifecycleState, PointLifecycleClusterSummary } from '../lifecycle';
import type { ManualTagCode, AutoMetadataCode, ClusterMetadataSummary, MoveLinkageRecord } from '../metadata';
import type { TimelineNodeActionDockActionCode } from './timelineNodeActionDockModel';
import type { QuickActionLabel } from './quickActionPresets';
import type { SourceChainStatus } from '../evidence/evidenceModel';

/**
 * ST-002 — the 9-code suggestion vocabulary from the issue body's Wave 2
 * scope additions. Each code maps to (a) a RULE-003-style rationale phrase
 * built from the lifecycle / tag / auto-metadata helper lines, (b) a
 * QuickActionLabel preset, and (c) optionally a SC-004 dock action code so
 * the dock can highlight the matching primary.
 */
export type SuggestedMoveCode =
  | 'ask_source'
  | 'ask_quote'
  | 'narrow'
  | 'concede'
  | 'confirm'
  | 'challenge_mechanism'   // maps to preset 'challenge', dock 'challenge'
  | 'challenge_scope'       // maps to preset 'narrow', dock 'narrow'
  | 'branch_tangent'        // maps to preset 'branch', dock 'branch'
  | 'synthesize';

export const ALL_SUGGESTED_MOVE_CODES: ReadonlyArray<SuggestedMoveCode> = Object.freeze([
  'ask_source',
  'ask_quote',
  'narrow',
  'concede',
  'confirm',
  'challenge_mechanism',
  'challenge_scope',
  'branch_tangent',
  'synthesize',
]);

/**
 * Why a suggestion was emitted. Used by AN-003 diagnostics + tests; never
 * rendered as raw text. The rationale STRING is what surfaces to the user;
 * signals are the structural evidence.
 *
 * Discriminated union — adding a new signal requires extending the
 * rationale builder's case table (compile-time enforced).
 */
export type SuggestionSignal =
  | { kind: 'unresolved_source_request' }
  | { kind: 'unresolved_quote_request' }
  | { kind: 'manual_tag_present'; tag: ManualTagCode }
  | { kind: 'auto_metadata_present'; code: AutoMetadataCode }
  | { kind: 'lifecycle_state_match'; state: PointLifecycleState }
  | { kind: 'narrow_concession_opportunity' }
  | { kind: 'synthesis_ready' }
  | { kind: 'point_exhausted_same_axis' }
  | { kind: 'branch_recommended_off_axis' }
  | { kind: 'broken_source_chain'; status: SourceChainStatus }
  | { kind: 'no_response_after_n_turns' };

/**
 * One suggestion record. Returned in deterministic priority order.
 *
 * - `presetKey` MUST be a member of `QuickActionLabel`. No new presets in
 *   ST-002.
 * - `dockAction` is the closest matching SC-004 action when one exists;
 *   `null` when no dock action lines up (branch / synthesise have one;
 *   challenge_scope reuses `narrow`; challenge_mechanism reuses `challenge`).
 * - `rationale` is built by `buildRationale(signals)`. Cap 80 chars. The
 *   builder concatenates RULE-003 helper lines + a small fixed-phrase
 *   table; it never authors new sentences per call.
 */
export interface SuggestedMove {
  code: SuggestedMoveCode;
  label: string;                                       // plain-language, ≤ 24 chars
  rationale: string;                                   // plain-language, ≤ 80 chars
  presetKey: QuickActionLabel;
  dockAction: TimelineNodeActionDockActionCode | null;
  sourceSignals: ReadonlyArray<SuggestionSignal>;
}

/**
 * Input bundle. All optional fields are gracefully handled (missing →
 * suggestion is suppressed; not thrown). The deriver is total.
 */
export interface SuggestionDerivationInput {
  /** Cluster lifecycle summary. Null → only minimal `reply` suggestion. */
  clusterSummary: PointLifecycleClusterSummary | null;
  /** Cluster metadata aggregate. Null → only lifecycle-driven suggestions. */
  clusterMetadata: ClusterMetadataSummary | null;
  /** Per-move linkage for the *selected* move. Null when no move is selected. */
  moveLinkage: MoveLinkageRecord | null;
  /** Source-chain status of the selected move. Null when no contract. */
  sourceChainStatus: SourceChainStatus | null;
  /** EV-001 evidentiary-risk label, plain-language. Reserved; null in v1. */
  evidentiaryRisk: 'low' | 'medium' | 'high' | 'unknown' | null;
  /** Latest move's argument type. Null when undetectable. Drives preset
   *  selection via `quickActionToPreset(label, parentType)`. */
  latestMoveType: 'rebuttal' | 'counter_rebuttal' | 'clarification_request' |
                  'evidence' | 'concession' | 'synthesis' | 'claim' | null;
  /** Active path depth (0 = root). Reserved for future heuristics. */
  activePathDepth: number;
  /** True when the selected move has been outstanding without rebuttal for
   *  longer than threshold. Mirrors LIFE-001 `no_response_after_n_turns`. */
  isNoRebuttal: boolean;
  /** Last validation stopReason from `submit-argument` (gameCopy code).
   *  Reserved; null in v1. */
  stopReason: string | null;
  /** True when the selected move is on a side branch. */
  isOnSideBranch: boolean;
  /** True when the selected move is flagged as a tangent (manual tag /
   *  auto metadata). */
  isTangent: boolean;
  /** Standing band, plain-language. Reserved; never used to add or remove
   *  a suggestion (doctrine: standing is play state, not truth). */
  standingBand: string | null;
  /** Caller-side cap on suggestions returned. Default 3. Values ≤ 0 →
   *  default. */
  maxSuggestions?: number;
}

/**
 * The public type that widens SC-003's stub. ST-002's deriver returns
 * arrays of `SuggestedMove`; the SC-003 sidecar slot becomes
 * `SuggestedMove | null` (still one-or-none; if a caller wants to surface
 * "the top suggestion only", it passes `[0]ish` into the slot).
 */
export type SuggestedNextMove = SuggestedMove;

/**
 * Public deriver. Pure, deterministic, total. Empty array is a valid
 * result (the user just sees no suggestion + their ordinary reply path).
 *
 * Order of resolution (mirrors SC-004 §"Order of resolution"):
 *   1. Validate inputs (null safety; empty array fallback).
 *   2. Source-chain status override (EV-001) — emits `ask_source` /
 *      `ask_quote` when the contract demands.
 *   3. Lifecycle-state primary — read from `LIFECYCLE_UX_MAP[state].allowedDockActions`.
 *      The deriver picks the first 1–2 allowedDockActions whose preset
 *      mapping is defined.
 *   4. Manual-tag override — promote tag-driven suggestions to the front
 *      when present.
 *   5. Auto-metadata refinement — push `synthesise` / `branch` /
 *      `narrow` forward when the cluster aggregate carries
 *      `synthesis_candidate` / `branch_suggested` / `repeated_axis_pressure`.
 *   6. Truncate to `maxSuggestions` (default 3).
 *   7. Build rationale per suggestion from RULE-003 helper lines.
 *
 * Never throws. Never mutates input. Stable order for identical inputs.
 */
export function deriveSuggestedMoves(input: SuggestionDerivationInput): SuggestedMove[];

/**
 * Pure function that composes a rationale STRING from the signal set. ≤
 * 80 chars. Uses RULE-003 helper lines verbatim where possible (no parallel
 * copy author). Falls back to a tiny fixed-phrase table for the four
 * non-1:1 signals (`narrow_concession_opportunity`,
 * `point_exhausted_same_axis`, `branch_recommended_off_axis`,
 * `no_response_after_n_turns`).
 *
 * Doctrine: the function MUST NOT use `you / your / they / their / the
 * user / the author / the poster`. The output passes
 * `_forbiddenLifecycleTokens()` + `_forbiddenMetadataTokens()` + the
 * COPY-001 verdict ban-list.
 */
export function buildRationale(signals: ReadonlyArray<SuggestionSignal>): string;

/**
 * Ban-list scan helper for tests. Concatenates the lifecycle + metadata
 * + COPY-001 person-attribution + verdict + amplification + block-prevent
 * token lists into a single deduped array.
 */
export function _forbiddenSuggestionTokens(): string[];
```

## Rationale text generation

```ts
export function buildRationale(signals: ReadonlyArray<SuggestionSignal>): string;
```

**Construction rules.**

1. The rationale describes the cluster / move state. **No `you / your / they / their / the user / the author / the poster / the speaker / the participant`** in any output.
2. RULE-003 helper lines are reused verbatim where possible. Example: `unresolved_source_request` → reuses `LIFECYCLE_UX_MAP['source_requested'].helperLine` (`'Someone asked for a primary source.'`).
3. For signals that don't map to a single RULE-003 line, a fixed phrase table supplies the prose:
   - `narrow_concession_opportunity` → `'The claim was narrowed to a tighter scope.'` (reuses `LIFECYCLE_UX_MAP['narrowed'].helperLine`).
   - `synthesis_ready` → `'The two sides have converged enough to summarise.'` (reuses `LIFECYCLE_UX_MAP['synthesis_ready'].helperLine`).
   - `point_exhausted_same_axis` → `'Same-axis pressure has repeated without new information.'` (reuses `LIFECYCLE_UX_MAP['exhausted'].helperLine`).
   - `branch_recommended_off_axis` → `'Off-axis pressure has built up; a branch reads cleaner.'` (reuses `LIFECYCLE_UX_MAP['branch_recommended'].helperLine`).
   - `no_response_after_n_turns` → `'No reply landed after the configured turn threshold.'` (reuses `AUTO_METADATA_UX_MAP['no_response_after_n_turns'].helperLine`).
   - `broken_source_chain` (status) → reads from `evidenceModel`'s plain-language label for the status; ST-002 imports `evidenceModel`'s read-only helper without re-authoring copy.
4. When multiple signals are present, the highest-priority signal owns the rationale (signal priority mirrors lifecycle table order). The remaining signals are kept in `sourceSignals[]` for diagnostics but do not concatenate into the rationale (keeps the cap at 80 chars).
5. Cap: 80 chars after assembly. If the assembled string exceeds 80 chars, the builder truncates at the last word boundary and appends `…` (mirrors the truncation helper SC-003 uses).
6. The output passes:
   - `_forbiddenLifecycleTokens()` (LIFE-001 ban-list — includes `winner / loser / right / wrong / validated / hot / popular / amplification / block / prevent / reject / forbid`).
   - `_forbiddenMetadataTokens()` (META-001 ban-list — same shape).
   - The COPY-001 person-attribution token list, asserted in ST-002's own test: `['you', 'your', "you're", 'yours', 'they', 'their', "they're", 'theirs', 'the user', 'the author', 'the poster', 'the speaker', 'the participant', 'this person', 'this user']` (case-insensitive, whole-word).

A test exhaustively builds a rationale for every `SuggestedMoveCode` × every supported signal kind and scans the output against the combined ban-list.

## API / interface contracts

### `deriveSuggestedMoves`

```ts
deriveSuggestedMoves(input: SuggestionDerivationInput): SuggestedMove[]
```

- Total. Never throws.
- Returns `[]` when:
  - `input.clusterSummary === null` AND `input.clusterMetadata === null` AND `input.sourceChainStatus === null` AND no manual / auto signals are derivable.
  - The matched lifecycle state's allowedDockActions is empty (e.g. `archived_or_resolved`) AND no override fires.
- Order is deterministic for identical inputs. Comparison is structural (same length, same `code` per index, same `rationale`, same `presetKey`, same `dockAction`, same `sourceSignals` length + per-index kind).
- Caps:
  - `maxSuggestions` (default 3, min 0).
  - Each `rationale` ≤ 80 chars.
  - Each `label` ≤ 24 chars.

### `buildRationale`

```ts
buildRationale(signals: ReadonlyArray<SuggestionSignal>): string
```

- Total. Empty `signals` → `''` (caller must not pass empty; deriver guarantees ≥ 1).
- Output is a single-line plain-language sentence assembled from RULE-003 helper lines + the fixed-phrase table. ≤ 80 chars (truncated at word boundary + `…` if longer).

### `_forbiddenSuggestionTokens`

```ts
_forbiddenSuggestionTokens(): string[]
```

- Returns a deduped concatenation of `_forbiddenLifecycleTokens()` + `_forbiddenMetadataTokens()` + the COPY-001 person-attribution list.
- Test-only consumer; not part of any production code path.

## Edge cases

- **`clusterSummary === null`** → No lifecycle signal. Deriver returns either `[]` or, when source-chain status is present, the source-chain override only. No throw.
- **`clusterMetadata === null`** → No manual/auto signals. Deriver runs lifecycle-only and source-chain-only paths. No throw.
- **`moveLinkage === null`** → No per-move tags. Deriver still reads cluster aggregate from `clusterMetadata`. No throw.
- **`sourceChainStatus === null`** → No EV-001 override fires; lifecycle path runs as primary.
- **Unknown / future `PointLifecycleState`** → Compile-time impossible (the type is closed). The deriver's switch is exhaustive (a default `assertNever`-style branch returns `[]`). Test asserts `ALL_POINT_LIFECYCLE_STATES.every(state => derive(...).every(s => ALL_SUGGESTED_MOVE_CODES.includes(s.code)))`.
- **Unknown / future `ManualTagCode` / `AutoMetadataCode`** → Same compile-time guarantee. Future codes fall through to no-promotion.
- **`archived_or_resolved` state** → `LIFECYCLE_UX_MAP[state].allowedDockActions === []`. Deriver returns `[]`. The user can still post a reply via the ordinary composer path; the dock surfaces `open_cards_detail` + `reply` (SC-004's responsibility), unaffected by ST-002.
- **`exhausted` state without a manual tag** → emits `[narrow, branch_tangent]`. Even though "concede" is in the lifecycle's allowedDockActions (it isn't, in fact — `narrow / branch / synthesize / flag`), ST-002 only emits suggestions whose preset exists; `flag` is not a suggested move.
- **Conflicting overrides** (e.g. source-chain says `ask_source` but lifecycle says `synthesize`) → source-chain wins. Suggestion order becomes `[ask_source, synthesize]`. Capped to `maxSuggestions`.
- **Concurrent state** — pure deriver; no concurrency to worry about. The caller (SC-003 / room shell) is responsible for not re-running on every keystroke.
- **`maxSuggestions === 0`** → returns `[]` (still total; doctrine: never throw on a configuration value).
- **Doctrine edge** — what if `standingBand === 'Strongly supported'`? It does NOT alter the suggestion list. Standing is play state, not truth; a "strongly supported" move can still attract a `challenge_mechanism` suggestion if the manual tag is present. A test pins this.
- **Doctrine edge** — what if `clusterMetadata.autoMetadataCodes` contains a heat-flavored code like `point_stalled`? It only promotes `synthesize` (a structural suggestion); it does NOT label the cluster "stale", "boring", "cold". The label comes from `AUTO_METADATA_UX_MAP['point_stalled'].label`.

## Risks

- **Person-attribution drift in rationale.** The rationale builder is small but it would be easy to slip a `'someone'`/`'they'` token in. *Mitigation:* the ban-list test exhaustively exercises every signal kind × every code, and the COPY-001 person-attribution list is included in `_forbiddenSuggestionTokens()`.
- **Preset gap codes confuse the dock.** Three `SuggestedMoveCode` values (`challenge_mechanism`, `challenge_scope`, `branch_tangent`) reuse another preset's mapping. If a future card adds a dedicated preset (PRESET-001), ST-002's mapping table must update in lockstep. *Mitigation:* the gap is documented in this design + asserted by a test that pins the three mappings, so a future preset addition will visibly need both a preset change and a mapping change.
- **SC-003 widening could trip the type-pinning test.** The existing test `SuggestedMoveStub type is 'never' so ST-002 can widen` (line 495 of `argumentReplySidecarModel.test.ts`) is a runtime tautology, so it should keep passing. But if a future SC-003 test ever asserts `SuggestedMoveStub` is exactly `never` at the type level (e.g. via `Expect<Equal<SuggestedMoveStub, never>>`), ST-002 will break it. *Mitigation:* before commit, the implementer greps for any test that pins the literal type `never` to `SuggestedMoveStub`; today there is none.
- **Rationale truncation at 80 chars feels tight for some lifecycle states.** Helper lines from RULE-003 already cap at 80 chars in their own author guidance, so the assembled rationale should usually fit. *Mitigation:* the truncation helper uses a word-boundary cut with `…`, mirroring SC-003's existing `truncateAtWordBoundary` so the failure mode is graceful, not jagged.
- **`branch_tangent` preset returns null `MoveDraftPatch`.** COMPOSER-001's `quickActionToPreset('branch', parentType)` deliberately returns `null` because branch opens a separate UI. The deriver still surfaces the suggestion; the dock routes the click into the branch flow. *Mitigation:* documented in the mapping table; no special handling required in ST-002.
- **Snapshot tests on lifecycle-state fixtures.** Locking the rationale string per state pins a small piece of copy. If COPY-001 later edits a RULE-003 helper line, the rationale snapshot must update. *Mitigation:* the test asserts the output array's structure (codes, signal kinds) without snapshotting the full rationale string verbatim — only its length, ban-list compliance, and presence of the expected RULE-003 helper line by `includes()`.
