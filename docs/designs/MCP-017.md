## 1. Goal (one paragraph)

MCP-017 builds the **Anthropic live provider** for the `semantic-referee` Edge Function: the implementation of MCP-009's `anthropic` provider slot — a Haiku-class model reached via the same `Deno.env.get` + `fetch` pattern the `process-language-draft` provider already uses — so the semantic referee can produce real (non-mock) routing / friction / score-hint suggestions when the operator opts in. The card is shaped by doctrine constraints that are blockers if violated: (1) **the live provider is opt-in by environment variable, never the default** — `SEMANTIC_REFEREE_PROVIDER` defaults to `'mock'` via MCP-016's `?? 'mock'` fallback; flipping it to `'anthropic'` is an explicit operator action; an unset value picks `mock`. (2) **The provider call lives only in the Edge Function** — the live `fetch` is in `anthropicProvider.ts` inside the `_shared/semanticReferee/` tree and `ANTHROPIC_API_KEY` is read there via `Deno.env.get()`; no client code, no `src/`, no `app/`, no git, no `.env*` file ever names or contains the key. A source-scan test enforces this. (3) **The provider can fail without failing the user** — key-missing, API error, rate limit, network failure, parse failure, or a provider output that fails the schema + content-safety validator all return a typed `unavailable` outcome that `providers.ts` converts to MCP-016's deterministic minimal fallback packet (`buildFallbackPacket`); the caller still gets a valid `ClassifyMoveOutcome` and the user never sees an error. (4) **Every output is re-validated through the boundary contract** — even a successful Anthropic response runs through the Deno `SemanticRefereePacketSchema` and a Deno-side content-safety scanner (mirroring MCP-011's `parseSemanticPacket` ban-lists) before it leaves the boundary; any smuggled `block` field, `authoritative: true`, verdict token, or off-contract value forces the deterministic fallback. (5) **The packet is structural only** — the seed prompt asks for `0/1` answers to bounded yes/no questions about the **structure** of an argument move (parent continuity, evidence hygiene, branch hygiene); no prompt field asks the model for a verdict, a winner, a truth value, or a popularity reading.

## 2. Scope

### In scope

- A new `anthropicProvider.ts` under `supabase/functions/_shared/semanticReferee/` — the Deno-only live-call orchestrator (key read, `fetch`, schema + content validation dispatch).
- A new `anthropicClassifierCore.ts` under the same directory — the **zod-free, Jest-importable** pure logic: prompt assembly, Anthropic-response-text extraction, JSON-from-content parse, `sanitizeRawPayload`. (See §"Test reality" — this split is what makes the provider unit-testable; it is a refinement of the card draft, which assumed `runAnthropicClassifier` was directly Jest-importable. It is not — `anthropicProvider.ts` transitively imports `npm:zod@4`.)
- A new `seedPrompt.ts` — the canonical structured prompt template + the per-classifier structural-question text keyed by the 23 catalog-v0 ids.
- A new `contentSafetyScan.ts` under the same directory — a **zod-free** Deno-side content-safety scanner mirroring `semanticRefereeValidator.ts`'s ban-lists (verdict / person / secret / PII / smuggled-field token detection). MCP-011's `parseSemanticPacket` lives in `src/` and imports Node `zod`; the Deno boundary cannot import it, so a parity-tested mirror is required (the same dual-mirror arrangement `types.ts` already uses).
- Un-stubbing the `anthropic` slot in `providerRouting.ts` (the pure routing core) and `providers.ts` (the Deno registry wrapper).
- Two new operator-controlled environment variables: `ANTHROPIC_API_KEY` (the existing key — reused if `process-language-draft` is already live, see §7) and `SEMANTIC_REFEREE_MODEL` (optional — defaults to a Haiku-class model-id constant in `anthropicClassifierCore.ts`).
- Tests: see §8.
- Documentation: a new operator runbook `docs/deployment/mcp-017-live-provider-runbook.md`.

### Out of scope (explicitly)

- **An MCP-server-hosted provider.** The `mcp` slot stays stubbed in this card. A future MCP-018 (reserved for exactly this) is the operator-hosted MCP adapter.
- **A persisted packet cache table.** MCP-012's in-memory LRU cache remains the cache of record; a persisted cache is a separate future migration card (MCP-001 §12).
- **Wiring `classifyMove` into a live room surface.** MCP-014's render component, MCP-015's choice sheet, and the in-room "assemble `BannerSelectionInput` from a live packet" builder remain a follow-up card (named in §10). MCP-017 unlocks the *capability*; the capability is not auto-mounted into any UI.
- **Synthesis summaries / expensive-tier models.** Out of scope per MCP-001 §10.
- **A second Anthropic credential.** This card reuses `ANTHROPIC_API_KEY` if it is already set for `process-language-draft`; see §7 and §12.
- **Any change to the inbound `ClassifyMoveRequest` contract, the catalog-v0 23-id union, the outbound packet shape, or the Node ↔ Deno type files.** MCP-017 changes no type. It un-stubs a provider slot and adds provider-only files.
- **Re-introducing `'key_missing'` to a *type* file the parity test scans.** `ClassifyMoveDisabledReason` in `types.ts` lists `'disabled' | 'not_configured' | 'not_implemented'`. The new live-provider failure reasons (`key_missing`, `api_error`, `rate_limited`, `network_error`, `parse_failure`, `validation_failed`) are **boundary-internal `ProviderUnavailable` reasons**, not `ClassifyMoveDisabledReason` values — see §"Defect resolution 3" in §4.

## 3. Doctrine constraints (the lines a reviewer enforces)

Any violation is a blocker.

1. **Mock is still the default after MCP-017.** `providerRouting.ts` keeps the `?? 'mock'` fallback unchanged. `SEMANTIC_REFEREE_PROVIDER=anthropic` is opt-in; unset means `mock`. The existing `semanticProviderRegistry.test.ts` "default is mock" test must keep passing untouched.
2. **The provider call never throws to the client.** Every error path — `key_missing`, `api_error`, `rate_limited`, `network_error`, `parse_failure`, `validation_failed` — returns a typed `ProviderUnavailable` outcome that `providers.ts` converts to a deterministic minimal fallback packet (`buildFallbackPacket`). `classifyMove` callers always receive a structurally valid `ClassifyMoveOutcome`. The Edge Function returns HTTP 200 on every one of these paths.
3. **The key is read only via `Deno.env.get()` inside `anthropicProvider.ts`.** `ANTHROPIC_API_KEY` and `SEMANTIC_REFEREE_MODEL` are never: imported from another file; passed to the client; echoed in a response body; logged (the function may log: function name, caller user id, `roomId`, decision name, provider name, duration, error class — nothing else); mentioned in `app/`, `src/`, `.env`, `.env.local`, `.env.example`, or any git-tracked file other than as a documented name in this design doc and the runbook.
4. **Every Anthropic response is re-validated through the boundary contract.** The provider's parsed JSON runs through (a) the Deno `SemanticRefereePacketSchema` in `schema.ts` — strict objects, `authoritative: z.literal(false)`, `value` in `{0,1}`, `scoreHints` in `0..3`; then (b) the Deno-side `contentSafetyScan.ts` — verdict / person / secret / PII / smuggled-`block` / chain-of-thought / raw-prompt detection. A failure on **either** step returns `validation_failed` and the caller falls back to the deterministic packet. `providers.ts`'s existing `validateOrFallback` is the second, registry-level wall — even a provider that bypassed step (a)/(b) would still hit it.
5. **The prompt asks for structure, never for truth.** The seed prompt's yes/no questions are about the move's *structure*: "Does this response engage the parent's claim, mechanism, question, or evidence?", "Does this evidence reference attach to the claim being made?", "Does this move raise a new issue that could be debated separately?". No prompt field asks the model whether anything is true, correct, right, wrong, factual, proven, popular, or who is winning. A ban-list test scans the seed prompt for forbidden vocabulary.
6. **The model can't smuggle a verdict.** The Deno `SemanticRefereePacketSchema` is `.strict()` everywhere and has no `block`, no `verdict`, no `truthValue`, no `winner` field to populate. Any such field fails the schema before the packet leaves the boundary; the content scanner additionally rejects a verdict token hiding inside a `reasonCode`.
7. **Sanitized payloads only in logs.** `sanitizeRawPayload` keeps only `model`, `stop_reason`, `usage` — never the raw body, never the response text, never request headers, never the `content` array.
8. **Cost is bounded by upstream gating.** MCP-012's trigger gates, batching (≤5 classifiers per call), in-memory LRU cache (capacity 256), token budget (≤8000 char body cap at the schema), and 2-attempt retry policy together cap the per-room call rate. MCP-017 changes none of these and adds no new caller.

### New files

  - `SEMANTIC_REFEREE_SYSTEM_PROMPT` — the system-prompt boilerplate (mirrors MCP-002 seed bank §"System-prompt boilerplate").

  - `extractAnthropicContentText(responseJson): string | undefined` — pulls the first `content[].type === 'text'` text (mirrors the sibling's `contentText` block).
  - `parseJsonFromContent(text): unknown | null` — extracts and `JSON.parse`s the first `{...}` object; returns `null` on failure (never throws).
  - `sanitizeRawPayload(raw): { model, stop_reason, usage }` — copied verbatim from the sibling provider; retains exactly those three keys.
  This file imports only `seedPrompt.ts` (zero `npm:` imports) so the `_helpers/semanticRefereeDeno.ts` Jest bridge can `require()` it.
- **`supabase/functions/_shared/semanticReferee/seedPrompt.ts`** (~140 lines) — the canonical structured prompt template. Exports:
  - `CLASSIFIER_QUESTION_TEXT: Readonly<Record<SemanticClassifierId, string>>` — one bounded structural yes/no question for each of the **23 catalog-v0 classifier ids** (the union in `types.ts`). The question text is the catalog-v0 condensation of MCP-002's seed-bank `value: 1 when` column — see §"Open item 2" resolution for the sync mechanism.
  - `buildClassifierPrompt(request: ClassifyMoveRequest): string` — assembles the user-message string: the per-id question list for `request.requestedClassifiers`, the strict-JSON output instruction (respond with a packet conforming to the contract — no prose, no markdown, no chain-of-thought), and the redacted move + parent bodies + room context as the input to classify.
  - `SEED_PROMPT_VERSION = 'mcp-semantic-referee-prompt-v0'` — the prompt-version string the packet is stamped with (matches MCP-002's `promptVersion` and MCP-016's `MOCK_PROMPT_VERSION`).
  Zod-free; imports only `types.ts`. Carries no verdict / truth / popularity vocabulary (ban-list scanned).
- **`supabase/functions/_shared/semanticReferee/contentSafetyScan.ts`** (~130 lines) — a **zod-free** Deno-side content-safety scanner. Exports `scanPacketContent(packet: unknown): { ok: true } | { ok: false; reason: 'validation_failed'; detail: string }`. Mirrors the ban-lists in `src/features/semanticReferee/semanticRefereeValidator.ts` (`VERDICT_TOKENS`, `PERSON_LABEL_TOKENS`, `PERSON_LABEL_PHRASES`, `SECRET_SHAPE_PATTERNS`, `PII_SHAPE_PATTERNS`, `BLOCK_FIELD_NAMES`, `CHAIN_OF_THOUGHT_FIELD_NAMES`, `RAW_PROMPT_FIELD_NAMES`) so a verdict token in a `reasonCode`/span, or a smuggled `block`/`reasoning` key, is rejected. It is the Deno twin of `parseSemanticPacket`'s Stage-5 content scan; a parity test asserts the ban-list constants match the Node upstream. Like `redaction.ts`, secret-prefix literals are assembled from `RegExp` fragments so the repo secret-literal scan stays green. Zod-free; imports only `types.ts` (for `ALL_SEMANTIC_CLASSIFIER_IDS` / field-name constants if reused). **Note:** this is additive insurance; the `.strict()` Deno `SemanticRefereePacketSchema` in `schema.ts` already rejects an unknown key — `contentSafetyScan.ts` adds the *value-content* scan (verdict token *inside* a contract-valid string field) that a schema cannot express.
- **`supabase/functions/_shared/semanticReferee/anthropicProvider.ts`** (~140 lines) — the Deno-only live-call orchestrator. Exports `runAnthropicClassifier(request: ClassifyMoveRequest): Promise<ProviderResult>` where `ProviderResult = { kind: 'success'; packet: SemanticRefereePacket } | { kind: 'unavailable'; reason: ProviderUnavailableReason }`. It:
  1. Reads `ANTHROPIC_API_KEY` via `Deno.env.get()` → `{ kind: 'unavailable', reason: 'key_missing' }` if absent.
  2. Reads `SEMANTIC_REFEREE_MODEL` via `Deno.env.get()` ?? `DEFAULT_SEMANTIC_REFEREE_MODEL`.
  3. Builds the body with `buildAnthropicRequestBody` and `POST`s `https://api.anthropic.com/v1/messages` (headers `x-api-key`, `anthropic-version: '2023-06-01'`, `Content-Type`) — identical call shape to the sibling.
  4. Maps a thrown `fetch` → `'network_error'`; a non-OK response → `'rate_limited'` for HTTP 429, `'api_error'` otherwise.
  5. `sanitizeRawPayload`s the response; `extractAnthropicContentText` + `parseJsonFromContent` → `'parse_failure'` on any miss.
  6. Stamps `provider: 'anthropic'`, `authoritative: false`, `packetVersion`, `promptVersion: SEED_PROMPT_VERSION`, `roomId`/`contentHash`/optional ids onto the parsed object.
  7. Validates the result through the Deno `SemanticRefereePacketSchema` (`schema.ts`) **and** `scanPacketContent` (`contentSafetyScan.ts`) → `'validation_failed'` on either failure.
  8. Returns `{ kind: 'success', packet }` only on a clean pass.
  This file imports `schema.ts` (→ `npm:zod@4`) so it is **not Jest-importable** — covered by source-scan + `supabase functions serve` integration tests, exactly as the sibling `anthropicProvider.ts` is. It mirrors the sibling step-for-step.
- **`docs/deployment/mcp-017-live-provider-runbook.md`** (~230 lines) — operator runbook (outline in §9).
- Test files — see §8 for the full list and rationale.

#### Defect resolution 3 — where the new failure reasons live in the type system

MCP-016's `types.ts` (Deno) and the Node twin in `src/lib/edgeFunctions.ts` define `ClassifyMoveDisabledReason = 'disabled' | 'not_configured' | 'not_implemented'`. MCP-016's `types.ts` header comment explicitly says `'key_missing'` "re-enters the union only when the live-provider pilot card adds the `anthropic` provider" — i.e. **this card**. The design resolves it cleanly:

- `ProviderUnavailableReason` (a **new boundary-internal type**, declared in `anthropicProvider.ts` or `anthropicClassifierCore.ts`) = `'key_missing' | 'api_error' | 'rate_limited' | 'network_error' | 'parse_failure' | 'validation_failed'`. This is the provider's own failure vocabulary.
- `ClassifyMoveDisabledReason` is **widened once** to the union of the original three plus the six provider reasons: `'disabled' | 'not_configured' | 'not_implemented' | 'key_missing' | 'api_error' | 'rate_limited' | 'network_error' | 'parse_failure' | 'validation_failed'`. The widening is applied to **both** the Deno `types.ts` and the Node twin in `src/lib/edgeFunctions.ts` (the parity test `semanticDenoNodeParity.test.ts` compares the *contract* surface — classifier ids, `ALL_*` arrays, packet field set, `PACKET_VERSION`, score-hint bounds — and per its own header comment does **not** scan `ClassifyMoveDisabledReason` because it is a boundary-only transport type; widening it touches no parity-scanned symbol). This is a deliberate, documented contract touch — small, additive, and exactly what MCP-016's comment anticipated. A reviewer should expect it.

```
                  providers.ts: validateOrFallback(request, packet)   ← 2nd wall
                  (schema fail → buildFallbackPacket; enabled:false passes through)
                                           ▼
                  semantic-referee/index.ts → ok(outcome)  (HTTP 200 always)
                                           ▼
                  classifyMove → caller; enabled:false → deterministic layer 1
```

## 7. Secret reuse

`ANTHROPIC_API_KEY` is **already a Supabase secret** for the `process-language-draft` Edge Function. MCP-009 §"Secret policy" recommends reusing it for the semantic referee rather than provisioning a second key.

- **Pro:** one fewer key to rotate; no duplicate billing surface; the same Anthropic workspace bills both features.
- **Con:** a compromised key affects both features at once; rotation is a coordinated action.
- **Mitigation:** the runbook (§9) includes a rotation procedure that swaps the secret in one `npx supabase secrets set` call — both functions pick up the new key on their next cold start.

If the operator prefers separate keys, name the second one `SEMANTIC_REFEREE_ANTHROPIC_API_KEY` and have `anthropicProvider.ts` read it with `ANTHROPIC_API_KEY` as a fallback (`Deno.env.get('SEMANTIC_REFEREE_ANTHROPIC_API_KEY') ?? Deno.env.get('ANTHROPIC_API_KEY')`). This is a one-line change and the design accommodates either choice. **Operator decision — flagged in §12.**

## 8. Test plan

### Test reality (the constraint that shapes this plan)

The `_shared/semanticReferee/` Deno tree splits into Jest-importable and not-Jest-importable files. The established repo convention (documented in `__tests__/_helpers/semanticRefereeDeno.ts` and used by `semanticBoundaryFallback.test.ts` / `semanticProviderRegistry.test.ts`):

- **Zod-free Deno files** (`mockProvider.ts`, `fixtureProvider.ts`, `redaction.ts`, `providerRouting.ts`* , `types.ts`, `fixtures.ts`) are loaded by the `_helpers/semanticRefereeDeno.ts` bridge via `require()` and tested directly in Jest.
- **Zod-coupled Deno files** (`schema.ts`, `providers.ts`) import `npm:zod@4`, which Jest cannot resolve. They are covered by **re-declared-local-zod-schema mirror tests** + **source-scan tests** (the `adminSchemas.test.ts` convention).

MCP-017's new files therefore split: `anthropicClassifierCore.ts`, `seedPrompt.ts`, `contentSafetyScan.ts` are **zod-free → Jest-importable via the bridge**. `anthropicProvider.ts` imports `schema.ts` → **not Jest-importable** → source-scan + integration coverage. *`providerRouting.ts` was zod-free in MCP-016; this card makes `DEFAULT_PROVIDER_DEPS` import `anthropicProvider.ts`, so `providerRouting.ts` becomes non-Jest-importable for a *default-deps* import — but the bridge already imports it and exercises `classifyWithProvider` with **injected spy deps**, never `DEFAULT_PROVIDER_DEPS`, so the routing-logic tests still run. The bridge's `require('providerRouting')` will now transitively pull `anthropicProvider.ts` → `schema.ts` → `npm:zod@4`; **the bridge must therefore stop re-exporting `classifyWithProvider` from the live `providerRouting.ts`** and instead the routing-logic test re-declares the tiny routing core locally (the `adminSchemas.test.ts` mirror convention) OR the implementer extracts the zod-free routing switch into a `providerRoutingCore.ts` the bridge can still load and `providerRouting.ts` re-exports. **The implementer must pick one; the design recommends the `providerRoutingCore.ts` extraction** — it keeps a real (not re-declared) routing core under test. This is called out as an implementation decision, not left implicit.

This whole subsection is a **refinement of the card draft's §4/§8**, which listed `__tests__/semanticAnthropicProvider.test.ts` directly unit-testing `runAnthropicClassifier` with a mocked `fetch`. That is not directly possible — `runAnthropicClassifier` lives in a zod-coupled file. The plan below preserves every test *intent* from the draft and routes each to a harness that actually runs.

## 11. Risks

- **Cost runaway from a misconfigured trigger.** Mitigation: MCP-012's trigger gates, batching, in-memory cache, token budget, and retry policy cap the rate; the operator must verify the cost alert is configured before flipping the flag. A per-Edge-Function call-rate limiter is a possible future card; out of scope here.
- **Provider returns plausible but doctrine-violating output.** Mitigation: the Deno `SemanticRefereePacketSchema` is `.strict()` and rejects every smuggleable field; `contentSafetyScan.ts` catches a verdict / person / secret token inside a contract-valid string; either failure → `validation_failed` → deterministic fallback; and `providers.ts`'s `validateOrFallback` is a third, registry-level wall. A misbehaving model cannot reach the user.
- **The Deno content scanner drifts from the Node validator.** Mitigation: `semanticAnthropicContentScanParity.test.ts` reads both files as source and fails the build on a ban-list divergence. New verdict/person tokens must be added to both.
- **The seed prompt drifts from MCP-002's seed bank / the catalog-v0 union.** Mitigation: `semanticAnthropicSeedPromptBanList.test.ts` asserts `CLASSIFIER_QUESTION_TEXT` has exactly the 23 `ALL_SEMANTIC_CLASSIFIER_IDS` keys. A catalog change without a `CLASSIFIER_QUESTION_TEXT` change fails CI. (Note: a *wording* change to a question is intentionally **not** asserted byte-identical against the seed bank — see §"Open item 2"; the test guards id-coverage and the ban-list, which is the safety-relevant invariant.)
- **The Anthropic API surface changes.** Mitigation: the sibling `_shared/languageProcessing/anthropicProvider.ts` is the working reference; a breaking change there needs the same fix here. Both are mock/intercept-tested, so the change is mechanical.
- **`async` ripple breaks the existing `semanticNoLiveCall.test.ts`.** Known and resolved — the test is retargeted (§4); the design calls it out explicitly so the reviewer does not flag the retarget as a removed assertion.
- **`providerRouting.ts` becoming non-Jest-importable silently breaks the routing tests.** Known and resolved — §8 "Test reality" specifies the `providerRoutingCore.ts` extraction (recommended) or a re-declared local routing core, so the routing logic stays under real test.
- **`ANTHROPIC_API_KEY` deployed to the wrong Supabase project.** Mitigation: `secrets set` / `functions deploy` operate on the `--linked` project; the runbook calls this out. A test cannot enforce it — operator discipline.
- **Reusing the existing key couples failure modes.** Mitigation: rotation affects both features at once; the runbook documents the coordinated rotation; §12 flags the decision.

### Open item 2 — the `seedPrompt.ts` source and how it stays in sync with the seed bank

`seedPrompt.ts` lives at `supabase/functions/_shared/semanticReferee/seedPrompt.ts`. Its `CLASSIFIER_QUESTION_TEXT` map is keyed by the **23 catalog-v0 ids** frozen in `types.ts` (`ALL_SEMANTIC_CLASSIFIER_IDS`) — `responds_to_parent`, `introduces_new_issue`, `asks_for_evidence`, … `ready_for_synthesis`. Each value is a single bounded structural yes/no question, authored as the catalog-v0 condensation of the corresponding family in `docs/semantic-prompts/mcp-semantic-referee-prompt-bank.md` (the MCP-002 seed bank). Examples of the mapping (the implementer authors all 23):

| catalog-v0 id (`types.ts`) | seed-bank family / row (MCP-002) | structural question in `CLASSIFIER_QUESTION_TEXT` |
|---|---|---|
| `responds_to_parent` | §A A1 `responds_to_parent_directly` | "Does this move directly engage the parent's claim, mechanism, question, evidence, or requested clarification?" |
| `introduces_new_issue` | §A A9 `introduces_new_issue` | "Does this move raise a new issue that could be debated separately?" |
| `asks_for_evidence` | §C C1/C2 `asks_for_source` / `asks_for_quote` | "Does this move request a source, citation, primary source, receipt, or exact quote?" |
| `evidence_supports_claim` | §C C6 `evidence_directly_supports_claim` | "Does the attached evidence appear to support the exact claim made in this move?" |
| `uses_popularity_as_evidence` | §C C17 `popularity_used_as_evidence` | "Does this move use likes, shares, virality, or 'everyone says' as evidentiary support?" |
| `needs_pre_send_pause` | §F F1 `pre_send_pause_recommended` | "Could this move be tightened by its author before sending?" |
| `ready_for_synthesis` | §D D10 `synthesis_candidate` | "Is there clear shared ground plus only limited unresolved debt?" |

**The sync mechanism (the mirror test):** `semanticAnthropicSeedPromptBanList.test.ts` is the structural mirror. It does **not** assert byte-identity against the seed bank — that is impossible because the id vocabularies differ (90 verbose seed ids vs 23 catalog ids; OQ-2). Instead it asserts the safety-relevant invariants:
1. `CLASSIFIER_QUESTION_TEXT` has an entry for **every** id in `ALL_SEMANTIC_CLASSIFIER_IDS` and **no extra keys** — so a future catalog change without a `seedPrompt.ts` change fails CI (id-coverage parity).
2. Every question string and the system prompt are free of the doctrine ban-list vocabulary (verdict / person / popularity / handle / URL / secret shapes) — so the prompt can never ask the model for a truth verdict.
3. `buildClassifierPrompt` emits the question for exactly the `requestedClassifiers` in the request and no others.

This is the strongest sync test that is actually implementable given the id-vocabulary mismatch. The design records the mismatch as OQ-2 so the operator can, if desired, file a separate MCP-002 documentation card to reconcile the seed bank to the 23-id catalog and make a future byte-identity test possible — but that reconciliation is **not** in MCP-017's scope.
