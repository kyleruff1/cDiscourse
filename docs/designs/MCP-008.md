## 1. Goal (one paragraph)

MCP-008 designs the **referee feedback banner library** — the playful, high-tech, non-verdict feedback strings that make the "objective referee keeping points" feeling reach the user. A banner is a short, fun, encouraging line ("Sharp — and it stays on point.", "Good pressure. Evidence debt opened.", "This probably belongs on a branch.") that the board surfaces **after a move is classified**. Banners are generated **from already-classified metadata only** — the `SemanticRefereePacket` binaries / `scoreHints` (MCP-001 §4 "Referee-feedback generation") and the `CategoryReading[]` the MCP-003 ledger produces — never from a fresh raw-conversation scrape. The banner library is a **pure-TS lookup-and-selection model**: it maps a classified packet/reading to a deterministic banner choice. The doctrine that shapes every line: a banner describes the **move**, never the **person**; it is encouragement or a suggested move, never a punishment; it carries no verdict / winner / loser / right / wrong / true / false token; it reflects `authoritative: false` metadata; popularity / heat is never praised as evidence; and a raw `classifierId` / `reasonCode` never reaches the banner surface — every banner is keyed off a code that maps through plain language. This card does not render UI chrome (color, animation, dismiss button) — it owns the **string library, the classifier-id → banner mapping, the tone-band model, the confidence-handling rule, and the accessibility contract**; the actual `<View>`/`<Text>` banner component is a thin follow-up that consumes this model.

## 2. Why this is a layer on top of MCP-002 + MCP-003, not a new classifier

The MCP family already produced everything a banner needs as **input**:

- **MCP-001** locked the `SemanticRefereePacket` — the `binaries: SemanticBinarySample[]` (each carries `classifierId`, `value: 0|1`, `confidence`, `reasonCode`), the `routeSuggestion`, the `frictionSuggestion`, and the six `scoreHints` integers. §4 of MCP-001 explicitly names "Referee-feedback generation — banners are generated from already-classified metadata only".
- **MCP-002** authored the 90-seed bank. §G already holds **10 "Fun referee banner classifiers"** (`clever_rebuttal_detected`, `funny_but_relevant_detected`, `pop_culture_reference_detected`, `unpopular_opinion_detected`, `probably_immediate_pushback_detected`, `hard_hitting_but_needs_receipt`, `logic_gap_opened`, `story_inconsistency_opened`, `evidence_hole_opened`, `moderator_needs_less_heat_more_anchor`) plus a "More banner copy" list and the explicit note "Full ≥ 100-banner library is MCP-008." Every other seed family (A–F) also carries a `User-facing copy` column whose voice the banner library extends.
- **MCP-003** authored the referee ledger. §8 already defines a `feedbackCode` → plain-language **copy-mapping table** (20 rows, e.g. `clean_parent_tie` → "Clean parent tie."), the `CategoryReading` shape (`category`, `delta`, `outcome`, `confidence`, `requiresUserChoice`, `feedbackCode`, `factualStandingAxis`), the `RefereePointCategory` 14-value union, and the `LedgerConfidence` (`'high' | 'medium' | 'low'`) type.

So MCP-008 does **not** define a new classifier, does **not** define a new score category, and does **not** scrape conversation. It is a **presentation library**. It does exactly four things and nothing else:

1. Holds a curated library of **≥ 100 banner strings** across the 12 named categories (scope item 1).
2. Maps **every MCP-002 classifier id** (A1–G10, all 90) to one or more banner candidates, with **deterministic selection** given a packet (scope item 2).
3. Defines **three tone bands** (celebratory / nudge / routing hint) and the rule for which band a banner sits in (scope item 3).
4. Defines **confidence handling** — low-confidence packets soften or suppress — and the **accessibility contract** — every banner works as a screen-reader announcement, no meaning by color alone (scope items 4 + 5).

It is a pure consumer: it imports the seed-bank classifier-id vocabulary (as a documented list) and MCP-003's `feedbackCode`/`CategoryReading` types; it never modifies a packet, never modifies the ledger, never re-derives a score.

## 3. Doctrine constraints

From `cdiscourse-doctrine`, `accessibility-targets`, `point-standing-economy`, `evidence-doctrine`, `test-discipline`, and the MCP roadmap expansion. **A violation is a blocker, not a comment.**

1. **A banner describes the MOVE, never the person.** Every string is about *the text move that was posted*. No banner names, characterises, or judges the author. "Sharp — and it stays on point." is about the move; "You are sharp" would be about the person and is banned.
2. **A banner never insults the user.** No string in the library may be mocking, belittling, sarcastic-at-the-user, or condescending. A banner is encouragement or a constructive suggested move. The seed bank §G note states it directly: "They must not insult the user — they describe the move and stay encouraging."
3. **No verdict / winner / loser / right / wrong / true / false / proven / defeated / person-label tokens** in any banner string, anywhere — not the headline, not the helper line, not the screen-reader label, not the `bannerCode`. The ban-list is in §10 and is test-enforced.
4. **A banner is advisory and reflects `authoritative: false` metadata.** A banner never asserts a fact about the world, never declares a claim true or false, never declares a debate decided. It reports *how the move played as a game move* — a structural / play-quality observation only. The whole MCP layer is `authoritative: false` (MCP-001 §3.5); a banner inherits that and never escalates a `0/1` binary into a hard claim.
5. **Score never blocks posting; a banner never blocks posting.** A banner appears *after* a move exists (post-submit) or *advisory pre-send in opted-in strict modes only* (MCP-001 §4). It has no `block` field, no gate. The most a banner can do is suggest a next move.
6. **Popularity / heat / virality / crowd reaction is never praised as evidence.** A banner never says "lots of people agree — strong move" or "this is trending". The anti-amplification banners (mapped from C17 `popularity_used_as_evidence`, the `uses_popularity_as_evidence` catalog id) explicitly say popularity is *not* proof and route to a source request — they classify the *text move*, never rank the crowd. Engagement credit and factual-standing credit stay on separate axes (`antiAmplification.ts`); no banner conflates them.
7. **A raw `classifierId` / `reasonCode` / `feedbackCode` / `RefereePointCategory` never reaches a banner surface.** Every banner is selected via an internal mapping; the user only ever sees the curated human string. New `bannerCode`s added to `gameCopy` are mapped to plain language in the same change; an unmapped code is **suppressed, not echoed** (this is already how `toPlainLanguage` behaves and is test-enforced).
8. **A banner never auto-acts.** It does not delete, hide, modify, flag, or score anything. It is a string the UI displays.
9. **Low confidence never produces a hard claim.** A `low`-confidence packet/reading produces a *softer* banner ("This might read as a hot take — a mechanism would land it.") or **no banner at all**, never a confident-sounding line. (MCP-001 §14: "a `low`-confidence binary produces a softer banner or no banner and never moves score on its own".)
10. **The banner model is pure TypeScript.** No imports from Supabase, React, or any network library — same rule as the rules engine, the point-standing economy, and the MCP-003 ledger. It runs identically on client and in an Edge Function. The thin render component is the only React file and it only consumes the model's output.
11. **No meaning is conveyed by color alone** (`accessibility-targets`). Every banner carries a tone band that is expressed by an icon-glyph + a text prefix + the copy itself, not only by a background color. Every banner is a complete screen-reader sentence.
12. **v1 scope guards.** No voting, no winner-producing scoring, no search, no push, no OAuth, no public API is introduced by this card. A banner is a display string; it adds none of those.

## 4. Data model

**No new database table, no migration, no Edge Function** — MCP-008 is design-only, and even the future implementation card delivers a **pure-TS model** (a string library + selection function). The packet and the ledger result are computed upstream; the banner library only *reads* them.

The new *types* the implementation card would add (proposed surface — not yet code):

```ts
// src/features/refereeBanners/  (proposed — not yet code)

/** The 12 issue-named banner categories. A category groups banners by what
 *  the move did; it is NOT a score category and NOT a verdict. */
export type RefereeBannerCategory =
  | 'continuity'          // the move tied to / drifted from the parent
  | 'evidence_debt'       // the move needs evidence / a debt opened
  | 'hot_take'            // the move is spicy / contrarian but playable
  | 'clever_rebuttal'     // the move is a sharp, on-point challenge
  | 'source_chain_gap'    // a source / quote / trail link is missing
  | 'branch_suggestion'   // the move reads better on a branch
  | 'tangent_suggestion'  // the move is a tangent — branch on purpose or tie back
  | 'synthesis_readiness' // the thread is close to a synthesis
  | 'quote_needed'        // the move needs the exact quote / excerpt
  | 'mechanism_needed'    // the move needs the causal mechanism named
  | 'mode_mismatch'       // the move's register does not fit the room mode
  | 'pacing_cooldown';    // pacing / cooldown / turn-limit notice

/** Three tone bands — §6. The band is the ONLY thing the render layer needs
 *  to pick an icon + a text prefix; it is never expressed by color alone. */
export type RefereeBannerTone =
  | 'celebratory'   // recognises a good move — "Nice narrowing move."
  | 'nudge'         // suggests a constructive next step — "This needs a quote anchor."
  | 'routing_hint'; // suggests a structural lane — "This looks like a branch, not a reply."

/** Confidence the banner is allowed to express — derived from the packet /
 *  reading, never invented. Drives §7 softening / suppression. */
export type RefereeBannerConfidence = 'high' | 'medium' | 'low';

/** One banner string in the library. `bannerCode` is the internal key; it is
 *  added to gameCopy.PLAIN_LANGUAGE_COPY so toPlainLanguage covers it. The
 *  user only ever sees `headline` (+ optional `helperLine`). */
export interface RefereeBanner {
  /** Stable internal id — snake_case — added to PLAIN_LANGUAGE_COPY. Never user-visible raw. */
  bannerCode: string;
  category: RefereeBannerCategory;
  tone: RefereeBannerTone;
  /** The visible banner line. ≤ 64 chars. Ban-list clean. A complete clause. */
  headline: string;
  /** Optional second line — a concrete suggested move. ≤ 80 chars. Ban-list clean. */
  helperLine?: string;
  /** A non-color tone marker: a short glyph/word the render layer shows so tone
   *  is not color-only. e.g. 'star' (celebratory), 'arrow' (nudge), 'branch' (routing). */
  toneGlyph: 'star' | 'arrow' | 'branch';
  /** The complete screen-reader sentence — §8. Always a full sentence with the
   *  tone stated in words ("Referee note:" / "Referee suggestion:"). */
  accessibilityLabel: string;
  /** The minimum confidence at which this banner may show. A banner whose
   *  effective confidence falls below this is suppressed (§7). */
  minConfidence: RefereeBannerConfidence;
  /** True for banners that, when their source confidence is `low`, are shown
   *  in their softened sibling form instead of suppressed (§7.2). */
  hasSoftenedSibling: boolean;
}

/** The deterministic selection input — built upstream from a SemanticRefereePacket
 *  (MCP-001) and/or a LedgerResult.categoryReadings (MCP-003). The banner library
 *  NEVER reads a raw conversation; it reads only this already-classified bundle. */
export interface BannerSelectionInput {
  /** The packet's binaries that read `value: 1`, in packet order. Read-only. */
  positiveBinaries: ReadonlyArray<{
    classifierId: string;            // an MCP-002 classifier id (A1..G10)
    confidence: 'low' | 'medium' | 'high';
  }>;
  /** The ledger's category readings, if a LedgerResult is available. Read-only. */
  categoryReadings: ReadonlyArray<{
    category: string;                // a MCP-003 RefereePointCategory
    feedbackCode: string;            // a MCP-003 feedbackCode
    confidence: 'low' | 'medium' | 'high';
    outcome: string;                 // a MCP-003 ReconciliationOutcome
  }>;
  /** Optional packet-level routing — MCP-001 SemanticRouteSuggestion. */
  routeSuggestion?: string;
}

/** The selection result — at most ONE banner per move (§5.4). */
export interface BannerSelectionResult {
  /** The chosen banner, or null when nothing is worth surfacing / all suppressed. */
  banner: RefereeBanner | null;
  /** Why this banner won (or why none did) — for dev/debug + tests, never user-visible. */
  selectionTrace: string;
}
```

The `RefereeBanner.headline` / `helperLine` / `accessibilityLabel` are all JSON-serializable strings; the model is side-effect-free and pure — same constraint as `src/lib/constitution/engine.ts`, `src/features/pointStanding/`, and the proposed MCP-003 `refereeLedger` model.

### 5.3 The classifier-id → banner mapping (scope item 2)

The library ships a deterministic map: **every one of MCP-002's 90 classifier ids (A1–G10) maps to one or more `bannerCode` candidates.** The proposed surface:

```ts
/** Every MCP-002 classifier id → an ordered, non-empty list of bannerCode
 *  candidates. The first entry whose preconditions hold (§5.4) wins. The
 *  map's KEY SET is exactly the 90 seed-bank ids — a test asserts it. */
export const CLASSIFIER_TO_BANNERS: Readonly<Record<string, ReadonlyArray<string>>>;

/** Every MCP-003 feedbackCode → a bannerCode. The ledger path (a move with a
 *  LedgerResult) selects through this; the packet path selects through
 *  CLASSIFIER_TO_BANNERS. The two paths converge in selectBanner (§5.4). */
export const FEEDBACK_CODE_TO_BANNER: Readonly<Record<string, string>>;
```

Mapping rules:

1. **Total coverage.** `Object.keys(CLASSIFIER_TO_BANNERS)` is exactly the 90 seed-bank ids — every id A1…G12, B1…B12, C1…C18, D1…D14, E1…E12, F1…F12, G1…G10. A test (`refereeBannerLibrary.test.ts`) iterates the documented seed-bank id list and asserts each is a key, and that each value is a non-empty array of valid `bannerCode`s.
2. **Many-to-one is allowed.** Several classifier ids can route to the same banner (A2 `responds_to_parent_partially` and the MCP-003 `partial_parent_tie` feedbackCode both reach `continuity_partial_tie`). One classifier id can offer multiple candidates (E12 `hot_take_without_mechanism` offers `hot_take_needs_mechanism` then `mechanism_give_it_a_path` — the second is the fallback when a mechanism-category banner is more useful).
3. **Some classifiers are silent.** A few seeds intentionally map to *no banner* — they are routing-only or suppress-only signals, not user feedback. F2 `post_submit_soft_feedback_only` (the seed bank marks it "no copy — silent route"), F11 `friction_would_feel_punitive` ("suppresses the friction — no user copy"), B5 `jump_branch_consumes_one_chance`, and B6 `branch_jump_not_allowed` (deterministic-dominant notices owned by the GAME-006 jump ledger UI, not the referee banner). These map to the sentinel `[]` (empty array) and a test asserts that an empty mapping is *intentional* by checking those four ids against a documented `INTENTIONALLY_SILENT_CLASSIFIERS` set — so a missing mapping (a bug) and a deliberate silence (correct) are distinguishable.
4. **No raw id ever surfaces.** The map's *values* are `bannerCode`s; the `bannerCode` is mapped through `gameCopy` to the human `headline`. The classifier id `A1` and the reason code `parent_continuity_answers_scope_claim` never appear in a `RefereeBanner` field.

### 5.4 Deterministic selection — one banner per move

> **Given a `BannerSelectionInput`, `selectBanner` returns the same `RefereeBanner | null` every time. No randomness, no time, no model call.**

```ts
/** The single pure entry point. No persistence, no network, no React. */
export function selectBanner(input: BannerSelectionInput): BannerSelectionResult;
```

The algorithm:

1. **Build the candidate pool.** For every positive binary in `input.positiveBinaries`, look up `CLASSIFIER_TO_BANNERS[classifierId]` and append its candidates. For every `categoryReading`, look up `FEEDBACK_CODE_TO_BANNER[feedbackCode]` and append it. Skip `INTENTIONALLY_SILENT_CLASSIFIERS`.
2. **Apply confidence handling (§7).** For each candidate, compute its effective confidence from the source binary/reading. If the effective confidence is below the banner's `minConfidence`, either swap to its softened sibling (if `hasSoftenedSibling`) or drop it.
3. **Rank by a fixed category priority.** When multiple candidates survive, exactly one is chosen by a **documented, deterministic priority order** — never random, never "most recent". The priority order (highest first), chosen so the *most actionable* feedback wins:
   1. `pacing_cooldown` (a pacing/cooldown notice is time-sensitive — show it first)
   2. `source_chain_gap`
   3. `evidence_debt`
   4. `quote_needed`
   5. `mechanism_needed`
   6. `tangent_suggestion`
   7. `branch_suggestion`
   8. `mode_mismatch`
   9. `continuity`
   10. `synthesis_readiness`
   11. `clever_rebuttal`
   12. `hot_take`
   Within a single category, ties break by the candidate's **first appearance** in the seed-bank id order (A before B before C …), which is itself deterministic. (Rationale: a nudge that helps the user repair the move outranks a celebratory line — but celebratory categories are never *suppressed*, they simply lose to an actionable nudge when both apply; if only a celebratory candidate survives, it shows.)
4. **Return at most one banner.** A move surfaces **one** banner — never a stack. (Banner spam reads as nagging and undermines the "fun" goal.) `selectionTrace` records which candidates were in the pool and why the winner won — for dev/debug and the test suite, never user-visible.
5. **`null` is a valid, common result.** When the pool is empty, or every candidate was suppressed by confidence, `selectBanner` returns `{ banner: null, selectionTrace: '<reason>' }`. The render layer shows nothing. A quiet move with no notable structural property gets no banner — that is correct and intended.

This guarantees scope item 2's "banner selection is deterministic given the packet": the same `BannerSelectionInput` always yields the same `BannerSelectionResult`.

## 6. Scope item 3 — Tone bands

Every banner sits in exactly one of **three tone bands**. The tone band is a property of the banner, fixed in the library; it is not recomputed per move.

| Tone band | When | Voice | `toneGlyph` | Examples |
|---|---|---|---|---|
| **`celebratory`** | the move did something well — a clean tie, a sharp rebuttal, a clean branch, a resolved debt, a playable hot take | recognises the move; warm, brief, never gushing; never "you won" | `star` | "Sharp — and it stays on point." · "Clean parent tie." · "Evidence debt resolved — nice." · "Spicy, but playable." |
| **`nudge`** | the move would be stronger with one concrete next step — a quote, a mechanism, a source, a tighter restatement | suggests *one* constructive action; encouraging, never scolding; "needs X" framed as opportunity, not failure | `arrow` | "This needs a quote anchor." · "Hot take — give it a mechanism and it lands." · "No origin yet — where did this come from?" |
| **`routing_hint`** | the move would sit better in a different structural lane — a branch, a tangent, a "you decide" choice | suggests *where* the move goes; neutral, never demoting; the move is welcome, just better placed | `branch` | "This looks like a branch, not a reply." · "Good tangent — worth its own branch." · "Reply here, branch, or tangent? — you decide." |

### 6.1 Tone-band rules

- **A tone band is never a verdict.** `celebratory` is not "you are right"; it recognises *play quality* (continuity, branch hygiene, a coherent hot take). `nudge` is not "you are wrong"; it names a *next move*. `routing_hint` is not "your move is bad"; it is a *map suggestion*.
- **The band is never expressed by color alone** (§8, `accessibility-targets`). The `toneGlyph` (`star` / `arrow` / `branch`) and the screen-reader prefix ("Referee note:" for celebratory, "Referee suggestion:" for nudge/routing) carry the band in non-color form. A grayscale render is still fully legible.
- **`celebratory` never praises popularity or heat.** A celebratory banner is earned by a structural property of the move (it tied to the parent, it named a mechanism, it is a coherent hot take) — never by engagement count, never by crowd reaction (doctrine constraint 6).
- **`nudge` never punishes.** "Needs a quote anchor" is an *opportunity* line, not a penalty line. There is no "you failed to" / "you should have" framing anywhere. The point-standing economy's doctrine — concession is a repair, evidence debt is a prompt — is the model: a debt-opened banner is a `nudge`, never a `celebratory`-for-the-challenger-at-the-author's-expense and never a punishment for the author.
- **`routing_hint` never demotes the person.** "This belongs on a branch" is about the *move's structural fit*, not a quality judgement. The seed bank B4 `outer_realm_recommended` copy ("This reads as low-signal here — a side lane fits it better") is the strongest routing line and even it is about the *move in this lane*, never the person.

## 7. Scope item 4 — Confidence handling

> **A low-confidence packet produces a softer banner or no banner — never a hard claim.** (Issue scope item 4, verbatim; MCP-001 §14.)

### 7.1 Where confidence comes from

The banner library never invents confidence. It reads it from the already-classified input:

- **Packet path** — each `SemanticBinarySample.confidence` (`'low' | 'medium' | 'high'`, MCP-001 §7) is the binary's own uncertainty. The candidate banners from that binary inherit it.
- **Ledger path** — each `CategoryReading.confidence` (`LedgerConfidence`, MCP-003 §5) is the ledger's reconciled confidence. MCP-003 §5.1 already produces a `low_confidence_semantic` outcome (`delta: 0`, `confidence: 'low'`, a softer feedback code) and a `conflict_routed` outcome (`delta: 0`, `confidence: 'low'`, `requiresUserChoice: true`). The banner for a `conflict_routed` reading is **always** the `routing_hint` "Reply here, branch, or tangent? — you decide." (`pacing_you_decide_the_lane`) — never a confident celebratory or nudge line.

### 7.2 The softening / suppression rule

Each `RefereeBanner` declares a `minConfidence` and a `hasSoftenedSibling` flag. The rule in `selectBanner` step 2:

| Effective confidence of the source | Banner's `minConfidence` | Action |
|---|---|---|
| `high` or `medium` | `low` or `medium` | Banner shows as authored. |
| `high` | `high` | Banner shows as authored. |
| `medium` | `high` | Banner is downgraded — swap to its softened sibling if `hasSoftenedSibling`, else drop. |
| `low` | any | If `hasSoftenedSibling`, swap to the softened sibling (which itself has `minConfidence: 'low'`). If not, **drop the banner** (it does not enter the ranked pool). |

**Softened siblings.** A banner that makes a positive structural observation gets a *softened twin* — a hedged form with `minConfidence: 'low'`. The softening is lexical: an assertive phrasing becomes a tentative one.

| Confident form (`minConfidence: 'medium'`+) | Softened sibling (`minConfidence: 'low'`) |
|---|---|
| "Clean parent tie." | "This looks like it ties to the parent." |
| "Sharp — and it stays on point." | "This reads as a sharp, on-point move." |
| "Spicy, but playable." | "This might read as a hot take — still playable." |
| "This belongs on a branch." | "This might read better on a branch." |
| "The evidence doesn't cover the whole claim yet." | "The evidence might not cover the whole claim yet." |

The softened sibling is itself a library entry (it has its own `bannerCode`, e.g. `continuity_clean_tie_soft`) so the ban-list test and the plain-language mapping test cover it too. It is **never** more assertive than the confident form; a test asserts every softened sibling contains a hedge token (`looks like` / `might` / `reads as` / `may`).

### 7.3 The hard rule low confidence enforces

- A `low`-confidence binary **never** produces a celebratory line that sounds like a verdict ("Clean parent tie." implies certainty — at `low` confidence it becomes "This looks like it ties to the parent." or is dropped).
- A `low`-confidence binary **never** produces a `nudge` that asserts a defect as fact. "The evidence doesn't cover the whole claim" at `low` confidence becomes "The evidence might not cover the whole claim yet" — a hedge, not an accusation.
- A `conflict_routed` reading (MCP-003 §5.1 rule 6 — the layers disagreed, neither is authoritative) **never** produces a confident banner of any tone. It produces exactly one banner: the reversible-choice `routing_hint`. This is the same doctrine MCP-003 §5.2 calls "the single most important rule" — a conflict routes, never penalizes, never asserts.
- When in doubt, **no banner.** `selectBanner` returning `{ banner: null }` is always safe. A missing banner costs nothing; a wrong confident banner erodes trust in the referee.

## 8. Scope item 5 — Accessibility

Banners are a visible UI surface, so `accessibility-targets` applies in full.

### 8.1 Every banner works as a screen-reader announcement

Each `RefereeBanner` carries a pre-authored `accessibilityLabel` — a **complete sentence** that states the tone in words and reads cleanly on its own (a screen-reader user gets one shot per element; `accessibility-targets` "the label is intentionally verbose because screen-reader users get one shot per element").

The label format:

```
"<tone prefix> <headline> <helperLine, if present>"
```

- **Tone prefix** — `celebratory` → "Referee note:" · `nudge` / `routing_hint` → "Referee suggestion:". The prefix states the tone band in *words*, so a screen-reader user gets the same celebratory-vs-suggestion distinction a sighted user gets from the glyph and color.
- **Examples:**
  - `continuity_clean_tie` → "Referee note: Clean parent tie. The referee likes how this picks up the thread."
  - `quote_needed_exact_excerpt` → "Referee suggestion: Source is here — the exact quote is still needed."
  - `branch_belongs_on_branch` → "Referee suggestion: This probably belongs on a branch."
- The `accessibilityLabel` is ban-list scanned exactly like `headline` / `helperLine` (§10). No verdict token, no person label, no raw code.
- The word "referee" is a placeholder pending the operator's scoreboard-framing-word decision (§13, MCP-001 §23.4); if the operator picks "scorekeeper" / "coach" / "moderator", the prefix changes in one place (the prefix constant) and every `accessibilityLabel` is regenerated from `headline` + `helperLine` by a pure builder, so the choice is a one-constant change.

### 8.2 No meaning conveyed by color alone

The render component (a thin follow-up, not this card) MUST, per `accessibility-targets` "Color is never the only signal":

- Show the `toneGlyph` (`star` / `arrow` / `branch`) next to the headline. The glyph is shape, not color — it survives a grayscale snapshot. A test renders the banner in grayscale and asserts the glyph + the text alone communicate the tone.
- The tone is **also** in the text: `nudge` banners contain an explicit suggested action ("give it a mechanism", "ask for the exact quote"); `routing_hint` banners contain a lane word ("branch", "tangent"); `celebratory` banners contain recognition language. So even without the glyph the tone is legible.
- Background color may *reinforce* the tone (a warmer tint for celebratory, a neutral tint for nudge/routing) but it is **never the only carrier** — and the §3.11 doctrine + the future render card's test enforce that.
- Color contrast: the banner text meets WCAG AA (4.5:1 body / 3:1 large) against the banner background — a constraint on the render card, noted here so it is not forgotten.

### 8.3 Dynamic-announcement etiquette

A banner appears post-submit (after a move posts). Per `accessibility-targets` "Screen-reader announcements for dynamic changes", the render card should announce the new banner **once** via `AccessibilityInfo.announceForAccessibility(banner.accessibilityLabel)` when it first appears — not on every re-render, not on hover, not on focus. This is a render-card constraint; the model's job is only to provide the ready-to-announce `accessibilityLabel` string. The banner element itself also carries `accessibilityRole="text"` (it is not interactive; if a banner ever gains a tappable "do this move" affordance, that affordance is a separate `Pressable` with its own `accessibilityRole="button"` + `accessibilityLabel` + 44×44 hit target — out of scope for MCP-008's string library).

### 8.4 Reduce-motion

If the render card animates a banner in (slide / fade), it must honor `AccessibilityInfo.isReduceMotionEnabled()` and snap instead. The string model is motion-free; this is a render-card note carried here so the follow-up card does not miss it.

## 10. API / interface contracts & the ban-list

### 10.1 Load-bearing signatures

- `selectBanner(input: BannerSelectionInput): BannerSelectionResult` — the single pure entry point (§5.4). Deterministic: same input → same output, no randomness, no `Date.now()`, no network.
- `CLASSIFIER_TO_BANNERS: Readonly<Record<string, ReadonlyArray<string>>>` — keys are exactly the 90 MCP-002 seed ids; values are non-empty `bannerCode` arrays (or `[]` only for the four `INTENTIONALLY_SILENT_CLASSIFIERS`).
- `FEEDBACK_CODE_TO_BANNER: Readonly<Record<string, string>>` — keys are exactly the MCP-003 §8 `feedbackCode`s; values are `bannerCode`s.
- `REFEREE_BANNER_LIBRARY: ReadonlyArray<RefereeBanner>` — the ≥ 103-entry library; every `bannerCode` is unique; every entry's `headline`/`helperLine`/`accessibilityLabel` is ban-list clean.
- `buildBannerAccessibilityLabel(banner: RefereeBanner, framingWord: string): string` — pure; rebuilds the screen-reader sentence when the operator's framing word changes (§8.1, §13).
- `RefereeBanner` has **no** `block` field, **no** `winner` field, **no** `truthValue` field, **no** `score` scalar, **no** `authoritative` field set to anything but the inherited `false` semantics — enforced by a schema-shape assertion in `refereeBannerBanList.test.ts`.
- The model imports from the MCP-002 seed-bank id list and the MCP-003 `feedbackCode`/type vocabulary **read-only**; it never constructs a `SemanticRefereePacket` and never mutates a `CategoryReading`.

### 10.2 The ban-list (scope item: "Ban-list test defined across all banner strings")

Every produced string — every `headline`, every `helperLine`, every `accessibilityLabel`, and every `bannerCode` — is scanned against the doctrine ban-list. The list is the union of the MCP-001 §19 / MCP-002 / MCP-003 §8.2 ban-lists and the anti-amplification banned-user-labels:

```
winner · loser · won · lost · win · lose · right · wrong · true · false ·
correct · incorrect · proven · disproven · defeated · beat · crush ·
liar · lying · dishonest · bad faith · manipulative · troll · bot ·
astroturfer · propagandist · extremist · stupid · idiot · dumb · smart ·
clueless · pathetic · ignorant
```

Plus the structural assertions:

- **No raw internal code** — no `headline` / `helperLine` / `accessibilityLabel` matches `looksLikeInternalCode` (the existing `gameCopy.ts` helper). A `bannerCode` *is* snake_case, but a `bannerCode` is never user-visible — the test asserts no `bannerCode` string appears inside any user-facing field.
- **No popularity-as-praise** — no banner string contains `popular` / `trending` / `viral` / `everyone agrees` / `most people` *in a praising frame*. (The anti-amplification banners may *name* popularity to say it is **not** proof — e.g. "Popularity isn't proof — what's the source?" — so the test scans for praise-context, not bare token presence; the implementer encodes this as: a banner containing a popularity token MUST also contain a not-proof phrase. This mirrors how MCP-002 keeps `popularity_used_as_evidence`'s copy doctrine-clean.)
- **No second-person insult** — no banner addresses the user with a belittling construction. The test asserts no string matches `you (are|'re) (wrong|losing|behind|bad)` and the like.
- **Every `bannerCode` maps** — `toPlainLanguage(bannerCode)` is non-null for every banner in the library (the plain-language mapping was extended in the same change). An unmapped code is a test failure (`test-discipline`, doctrine constraint 7).

## 11. Edge cases

The implementer must handle each:

- **Empty `BannerSelectionInput`** — no positive binaries, no category readings (a quiet, unremarkable move). `selectBanner` returns `{ banner: null, selectionTrace: 'empty_pool' }`. The render layer shows nothing. This is the **common** case for ordinary moves — most moves get no banner, and that is correct.
- **All candidates suppressed by low confidence** — the pool was non-empty but every candidate fell below `minConfidence` and had no softened sibling. `selectBanner` returns `{ banner: null, selectionTrace: 'all_suppressed_low_confidence' }`. No banner — never a confident fallback.
- **A classifier id not in `CLASSIFIER_TO_BANNERS`** — should be impossible (the test asserts total coverage), but a defensive `selectBanner` skips an unknown id and records it in `selectionTrace` rather than throwing. A new seed-bank id added later without a banner mapping is caught by the coverage test, not by a runtime crash.
- **A `feedbackCode` not in `FEEDBACK_CODE_TO_BANNER`** — same: skipped defensively, recorded in the trace; the coverage test catches a missing mapping.
- **Conflicting candidates from packet path and ledger path** — both paths fed the pool and they point at different categories. The category-priority order (§5.4 step 3) resolves it deterministically; there is no tie that randomness breaks.
- **A `conflict_routed` `CategoryReading`** — produces exactly the `pacing_you_decide_the_lane` routing banner, never a confident line (§7.3). If a `conflict_routed` reading and a high-confidence celebratory binary both appear, `pacing_cooldown` is the top priority category (it owns the `you_decide` banner) so the routing choice wins — correct, because an unresolved routing decision is more actionable than a compliment.
- **A packet whose `authoritative` is somehow not `false`** — impossible per MCP-001's literal type, but the banner model never reads `authoritative` at all (it only reads `binaries` / `scoreHints` / `categoryReadings`), so a malformed packet cannot make a banner assert authority. Noted for completeness.
- **A binary with `value: 0`** — `BannerSelectionInput.positiveBinaries` carries only `value: 1` binaries by construction (the upstream builder filters). A `value: 0` binary contributes no banner — the *absence* of a property is not a banner ("does not respond to parent" is not surfaced as "this fails" — that would be a punishment; the *presence* of `responds_to_parent_partially: 1` surfaces the constructive `continuity_partial_tie` nudge instead).
- **Popularity / heat in the input** — there is no popularity field in `BannerSelectionInput`. Heat / engagement count is structurally *not an input* to banner selection, so no banner can ever be selected *because* a move is popular (doctrine constraint 6 made structural).
- **Concurrent edits** — argument bodies are immutable after posting (soft-delete only — CLAUDE.md Supabase conventions); the upstream packet is computed once per `{roomId, parentId, contentHash, promptVersion}` (MCP-001 §12). A banner is therefore stable for a posted move; no concurrent-edit race exists.
- **Offline / network failure** — the banner model is pure-TS with no network. The *packet* is fetched upstream; if that fetch fails the upstream passes the mock-provider stub (MCP-001 §10) and `selectBanner` runs on whatever (possibly empty) input it gets — typically returning `null`. The banner model itself never makes a network call and never errors on a missing packet.
- **Permission-denied** — not applicable: the model has no auth surface; it consumes already-fetched, already-classified data. The observer/participant/own-bubble matrix is owned by the rail / dock cards (SC-001 / SC-004); a banner is *display only* and the same banner string is doctrine-safe regardless of who reads it.
- **Doctrine edge case — "can heat boost a banner to celebratory?"** — no. There is no heat/engagement input; §11 "Popularity / heat in the input" bullet. A banner is celebratory only for a *structural* property of the move.
- **Doctrine edge case — "can a banner say the other side lost a point?"** — no. A banner is about *the move that was just posted*, never about the opponent and never about a point's resolution. "Evidence debt resolved — nice." celebrates the *repairing move*; it does not say the challenger "lost". Concession banners ("Narrow concession noted — the broad point still stands.") follow MCP-003 §8.3 exactly — a repair, never a defeat.
- **Doctrine edge case — "what if a low-confidence binary is the only signal?"** — §7.2: it shows in softened form if it has a softened sibling, otherwise no banner. It never shows as a confident line.

## 12. Test plan (future implementation card)

All tests are pure-model; **no test makes a live call.** The three files the issue names verbatim, plus three guards consistent with MCP-001 §19 / MCP-003 §15:

- `__tests__/refereeBannerLibrary.test.ts` — coverage:
  - every one of the 12 `RefereeBannerCategory` values has at least its floor count of banners (§5.1);
  - the library has ≥ 100 banners total;
  - every `bannerCode` is unique;
  - **every MCP-002 seed-bank id (all 90, A1…G10)** is a key of `CLASSIFIER_TO_BANNERS`, and every value is a non-empty `bannerCode` array **except** the four `INTENTIONALLY_SILENT_CLASSIFIERS` (which are `[]` *and* are asserted to be exactly those four);
  - every MCP-003 `feedbackCode` (the 20 in MCP-003 §8) is a key of `FEEDBACK_CODE_TO_BANNER`;
  - every `bannerCode` referenced by either map exists in `REFEREE_BANNER_LIBRARY`;
  - every `bannerCode` resolves through `toPlainLanguage` to a non-null string (plain-language mapping coverage).
- `__tests__/refereeBannerBanList.test.ts` — the doctrine ban-list (§10.2) scanned across every `headline`, `helperLine`, `accessibilityLabel`, **and** every `bannerCode`; no user-facing field matches `looksLikeInternalCode`; no banner praises popularity (a popularity token implies a not-proof phrase in the same string); no second-person insult construction; `RefereeBanner` has no `block` / `winner` / `truthValue` / `score`-scalar field (schema-shape assertion). This is the "ban-list test defined across all banner strings" the acceptance criteria require.
- `__tests__/refereeBannerConfidence.test.ts` — confidence handling:
  - a `low`-confidence source for a banner with a softened sibling yields the **softened** banner (and the softened banner contains a hedge token);
  - a `low`-confidence source for a banner with no softened sibling yields **no banner** (suppressed);
  - a `medium`/`high` source yields the banner as authored;
  - a `conflict_routed` `CategoryReading` yields exactly the `you decide the lane` routing banner and never a confident celebratory/nudge line;
  - every softened sibling is no more assertive than its confident form (contains a hedge token);
  - low confidence never produces a banner whose tone is `celebratory` with a verdict-shaped headline.
- `__tests__/refereeBannerSelection.test.ts` (guard) — `selectBanner` is deterministic: the same `BannerSelectionInput` yields the same `BannerSelectionResult` across repeated calls; the category-priority order is respected (a `source_chain_gap` candidate beats a co-occurring `hot_take` candidate); at most one banner is ever returned; an empty input returns `{ banner: null }`.
- `__tests__/refereeBannerForbiddenImports.test.ts` (guard) — the `refereeBanners` model imports nothing from Supabase / React / a network library (source-scan, the MCP-001 §19 / MCP-003 §15 pattern); `gameCopy.ts` is the only cross-feature import and it is the plain-language map.
- `__tests__/refereeBannerAccessibility.test.ts` (guard) — every `RefereeBanner.accessibilityLabel` is a non-empty complete sentence that begins with the tone prefix; `buildBannerAccessibilityLabel` is pure and produces the same string for the same input; changing the framing word changes only the prefix; every banner has a non-`undefined` `toneGlyph` so the render layer always has a non-color tone signal.

Coverage bar: pure-TS model → every public function (`selectBanner`, `buildBannerAccessibilityLabel`) has unit tests including failure / empty / suppressed cases (`test-discipline`). The library data files are covered by the library + ban-list tests.

- **Will block / inform future cards:** the thin **referee-banner render component** card (a `<View>`/`<Text>` that consumes `selectBanner`'s output, applies the `toneGlyph`, honors reduce-motion, announces once — out of scope here per §15) and **MCP-010** (the override UX — a `conflict_routed` banner is the entry point to MCP-010's reversible choice; MCP-008 produces the routing banner string, MCP-010 builds the choice UI).
- **Coordinates with (not blocked by):** the live-provider pilot card — the banner library works identically with the `mock` provider (MCP-001 §10), so it does not wait on a live provider.
- Skills consulted: `cdiscourse-doctrine`, `accessibility-targets`, `point-standing-economy`, `evidence-doctrine`, `test-discipline`.

## 14. Risks

- **Voice drift across 103 hand-authored strings.** A library this size, written by an implementer, can drift into scolding or verdict-shaped copy on the long tail. **Mitigation:** §5.2 locks the voice with a representative sample across all 12 categories; the ban-list test (§12) scans *every* string including the long tail; and the "no second-person insult" + "no popularity praise" assertions catch the two most likely drifts. A reviewer should still read the full library, but the test net is wide.
- **Scope creep toward a banner stack / banner spam.** It is tempting to show every applicable banner. §5.4 fixes "at most one banner per move" and the selection test asserts it. A future card that wants a banner *history* or a banner *stack* is a new card with its own doctrine review — MCP-008's contract is one banner per move.
- **The category-priority order is a judgement call.** §5.4 step 3 ranks `pacing_cooldown` / `source_chain_gap` highest and `hot_take` lowest. If product feedback later wants celebratory banners to win more often, the order is a single documented constant — a low-risk, well-localized change, not a contract change. Flagged so it is a conscious decision.
- **`scoreHints` are not directly consumed.** MCP-008's `BannerSelectionInput` is built from `positiveBinaries` + `categoryReadings`, not the six `scoreHints` integers — because MCP-003 already turns `scoreHints` into `CategoryReading`s (MCP-003 §6) and the banner should read the *reconciled* result, not the raw hint. This is deliberate (it avoids the banner library double-interpreting a hint MCP-003 already interpreted) but means the upstream builder must run MCP-003 *before* MCP-008 when a ledger result is available. Flagged as a sequencing note for the upstream-wiring card.
- **The render component is a separate card.** MCP-008 delivers the *string library + selection model + accessibility strings* but not the `<View>`. The color-independence, reduce-motion, and announce-once requirements (§8) are *noted* here but *enforced* by the render card's tests. If the render card is not filed, the accessibility contract is designed but not shipped — flagged so the render card is not forgotten.
- **No migration / no operator deploy** — pure-TS model, a string library + a pure function. Lowest operational risk class. The only `gameCopy.ts` change is an additive extension (no existing key altered), so no runner-side regression (the same precedent META-001 / LIFE-001 set when they extended `PLAIN_LANGUAGE_COPY`).

## 15. Out of scope

- **The packet contract** — owned by MCP-001. MCP-008 consumes `SemanticRefereePacket` / `SemanticBinarySample` / `scoreHints` read-only and never redefines them.
- **The prompt seed bank** — owned by MCP-002. MCP-008 maps *from* the 90 classifier ids; it does not author or revise a seed.
- **The referee ledger** — owned by MCP-003. MCP-008 consumes the `CategoryReading` / `feedbackCode` output; it does not score, does not reconcile, does not touch `src/features/pointStanding/`.
- **Trigger gating / caching / cost control** — owned by MCP-004. MCP-008 does not decide *when* a packet is produced.
- **The Edge Function / MCP boundary** — owned by MCP-009. The banner model is pure-TS and runs wherever it is called.
- **The override / appeal UX** — owned by MCP-010. MCP-008 produces the `conflict_routed` routing banner string; MCP-010 builds the reversible-choice UI behind it.
- **The banner render component** — the actual `<View>`/`<Text>` banner chrome (background tint, the `toneGlyph` icon asset, the slide/fade animation, the `announceForAccessibility` call, the optional tappable "do this move" affordance) is a thin **separately-filed follow-up card**. MCP-008 delivers the string library, the selection model, the tone-band model, the confidence rule, and the ready-to-render `accessibilityLabel` / `toneGlyph` — i.e. everything the render card consumes. The render card's tests enforce §8.2 / §8.3 / §8.4.
- **A user-facing winner / resolution outcome** — RULE-007 / GAME-007, doctrine-gated and separate. No banner crowns anyone.
- **Banner history / banner stacking / a per-room banner feed** — out of scope; MCP-008 is one banner per move.
- **Localization / i18n of banner strings** — the library is English v1; an i18n pass is a future card. The model is structured so a translated string table could slot in (`bannerCode` is locale-independent).
- **Production code, tests, migrations, Edge Function deploys** — none in this design pass; the issue scope is "design doc only".

## 16. Doctrine self-check

- **cdiscourse-doctrine — score is gameplay analysis, never truth:** every banner describes a *structural play property* of the move (continuity, branch fit, mechanism present, source attached); no banner asserts a claim is true / false / proven; §3.3, §10.2 ban-list. ✓
- **cdiscourse-doctrine — score / banner never blocks posting:** a banner appears *after* a move exists; it has no `block` field; the schema-shape test asserts the absent field; §3.5, §10.1. ✓
- **cdiscourse-doctrine — no winner / loser:** no banner names a winner or loser; concession banners render as a repair ("the broad point still stands"), never a defeat; the ban-list test scans `winner`/`loser`/`won`/`lost`/`defeated`; §3.3, §10.2, §11. ✓
- **cdiscourse-doctrine — AI never decides who is right / never assigns truth / `authoritative: false`:** banners are generated from `authoritative: false` metadata and inherit it; the model never reads `authoritative` and never escalates a `0/1` binary into a hard claim; §3.4. ✓
- **cdiscourse-doctrine — AI advisory only, never auto-acts:** a banner is a display string; it deletes / hides / flags / scores nothing; §3.8. ✓
- **cdiscourse-doctrine — popularity / heat is never evidence and never praised:** there is no heat / engagement input to `BannerSelectionInput`; no banner praises popularity; anti-amplification banners say popularity is *not* proof; the ban-list test enforces the popularity-praise rule; §3.6, §10.2, §11. ✓
- **cdiscourse-doctrine — pure model, no Supabase / React / network:** the `refereeBanners` model is pure-TS, same constraint as the rules engine / the economy / the MCP-003 ledger; enforced by `refereeBannerForbiddenImports.test.ts`; §3.10, §12. ✓
- **cdiscourse-doctrine — plain language, no snake_case / raw code in UI:** every banner is selected via an internal `bannerCode`; the user sees only the curated `headline`/`helperLine`; every `bannerCode` is mapped through `gameCopy.toPlainLanguage`; an unmapped code is suppressed; no user-facing field matches `looksLikeInternalCode`; §3.7, §10.2, §12. ✓
- **cdiscourse-doctrine — describe the move, never the person; no insults:** every banner is about the posted move; the ban-list excludes person labels (`liar`, `troll`, `manipulative`, …) and the test scans for second-person insult constructions; §3.1, §3.2, §10.2. ✓
- **accessibility-targets — no meaning by color alone:** every banner carries a `toneGlyph` (shape, not color) and a tone word in the text and the screen-reader prefix; a grayscale render stays legible; §8.2. ✓
- **accessibility-targets — works as a screen-reader announcement:** every banner carries a pre-authored complete-sentence `accessibilityLabel` with the tone stated in words; announce-once etiquette is specified for the render card; §8.1, §8.3. ✓
- **point-standing-economy — concession is a repair, not a defeat:** concession / debt-resolved banners are framed as repair and recognition ("Evidence debt resolved — nice.", "the broad point still stands"), never "lost" / "gave up"; §6.1, §11. ✓
- **point-standing-economy — a debt is a prompt, never a punishment:** the `evidence_debt` banners are `nudge` tone ("Good pressure. Evidence debt opened." → "a source closes it"), never a penalty line for the author; §6.1. ✓
- **evidence-doctrine — engagement credit and factual-standing credit stay separate:** no banner conflates the two; popularity banners route to a *source request*, never to factual standing; §3.6, §10.2. ✓
- **evidence-doctrine — evidence debt is a prompt, never a falsehood verdict:** the `source_chain_gap` / `evidence_debt` banners ask "where did this come from?" and "a source closes it" — never "this is false"; §5.2, §6.1. ✓
- **test-discipline — tests are part of done, mock-first, ban-list test:** §12 — six test files defined for the implementation card, all pure-model, including the ban-list, the confidence, the determinism, the forbidden-imports, and the accessibility guards. ✓
- **v1 scope guards:** a banner is a display string — it introduces no voting, no winner-producing scoring (§15 — winner is RULE-007/GAME-007), no search, no push, no OAuth, no public API. ✓
