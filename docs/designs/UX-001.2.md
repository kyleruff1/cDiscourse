## Goal

UX-001.2 makes the Timeline the first substantive in-room object beneath the UX-001.1 AppHeader and one compact room/context strip. Today, a room-active 1366×768 web viewport stacks ten pre-Timeline layers (~951px before the first node) — Timeline is buried below AppHeader, MainAppShell tabBar, DebateDetailHeader, MainAppShell roomToolbar, ArgumentGameSurface.header, ArgumentScoreTracker, TimelineSelectedReadoutPanel (with the SC-003 sidecar embedded), ArgumentTimelineMap.controlsRow, container padding, and a 120px internal rail offset. The brief's hard cap is 200px wide / 168px tablet / 128px phone. This card collapses, relocates, or nullifies every pre-Timeline layer (per the brief's chrome-nullification authority that supersedes prior aesthetic placements) so the Timeline starts immediately under the AppHeader plus one compact strip. Functional behavior — activeMessageId, Timeline/Cards switching, IX-003 keyboard nav, branch topology, composer submit, role/actor gating, Account/Admin mounting, no-verdict copy — is preserved.

The design respects doctrine: no truth labels in any new copy, no verdict tokens, no internal codes leak; the rules engine remains pure; no Anthropic/xAI/X API call; no service-role; no schema/migration/Edge Function/dependency; no new route. The Timeline's visual grammar (shape, stroke, color, branch lanes) is unchanged — UX-001.2 is a chrome-consolidation card, not a Timeline-rewrite card.

## Q3 — Global tab bar room-active behavior

**Gate variable.** The existing `hasDebate = Boolean(state.snapshot.selectedDebateId)` at `App.tsx` line 364 is the source of truth. The conditional render at line 597 (`activeTab === 'arguments' && hasDebate && currentDebate && !notificationsOpen`) already isolates the room-active branch. We add ONE more gate to the tabBar render: when `activeTab === 'arguments' && hasDebate && currentDebate && !notificationsOpen`, the tabBar View is NOT rendered.

**Conditional rendering pattern.** Wrap the existing `<View style={styles.tabBar}>` (lines 494-517) in a conditional. The simplest pattern is:

```tsx
const roomActive = activeTab === 'arguments' && hasDebate && Boolean(currentDebate) && !notificationsOpen;
// ...
{!roomActive ? (
  <View style={styles.tabBar}>
    {/* existing tab map */}
  </View>
) : null}
```

We introduce a single local `roomActive` boolean in MainAppShell (computed once near line 433 after `tabs`/`activeTab`) so the test surface is one identifier, not three. The render branch at line 597 reuses the same boolean.

**Restoration on room exit.** The existing `deselectDebate()` (called by `handleLeaveRoom` at line 467 and by tapping the home logo via `dispatch({type: 'DEBATE_DESELECTED'})` in AppRoot) clears `state.snapshot.selectedDebateId`. As `hasDebate` becomes `false`, `roomActive` flips to `false`, and the tabBar re-renders. No new state.

**Room-exit escape path.** The compact room/context strip carries the room-exit affordance (one Pressable, `accessibilityRole="button"`, `accessibilityLabel="Leave argument"` matching the current copy at `DebateDetailHeader.tsx` line 180, `onPress={onLeave}` wired the same way). This is the user's escape path with the tab bar hidden. The Pressable meets 44×44 hit target via `minHeight: 44` (already present at line 253) or explicit `hitSlop` if visual is smaller.

**Account/Admin reachability tests.** Three tests guard the regression:
- When `state.snapshot.selectedDebateId === null`, the tabBar renders with all visible tabs (`getVisibleTabs(role, isDev)`).
- When `state.snapshot.selectedDebateId === 'some-id'` AND `activeTab === 'arguments'`, the tabBar does NOT render.
- When `state.snapshot.selectedDebateId === 'some-id'` AND the user calls `handleLeaveRoom()`, the tabBar re-renders.

**Notifications interaction.** The existing notifications path (line 524) is unchanged. When `notificationsOpen === true`, the notifications screen takes the body and the room is not rendered; the tabBar's gate `!roomActive` is `true` (because `!notificationsOpen` is false in the gate), so the tabBar STAYS visible — Notifications is reached from the tabBar's Arguments tab, then the in-arguments toolbar exposes "Notifications". This preserves the QOL-040 flow exactly.

**What we do NOT do.** No compact 32-36px tab bar (brief explicitly forbids this). No new state. No new navigation. No router. No Account/Admin screen mount change. No role resolution change.

## Q4 — Compact room/context strip specification

The strip replaces `DebateDetailHeader` as the in-room object beneath the AppHeader. It is one row, height ≤ 48 / 56 / 64 per band. It carries the title excerpt, the Timeline/Cards toggle, the room-exit control, and one overflow trigger for invite + dev chips.

**Per-band height arithmetic.**

| Band | Cap | Composition | Total |
|---|---|---|---|
| Phone (0-599) | 48 | `paddingVertical: 6 × 2 = 12` + content row `minHeight: 32` (44 hit-target via `hitSlop: 6`) + `borderBottomWidth: 1` = 12 + 32 + 1 = **45** | 45 ≤ 48 ✓ |
| Tablet (600-1279) | 56 | `paddingVertical: 8 × 2 = 16` + content row `minHeight: 36` (44 hit-target via `hitSlop: 4`) + `borderBottomWidth: 1` = 16 + 36 + 1 = **53** | 53 ≤ 56 ✓ |
| Wide (1280+) | 64 | `paddingVertical: 10 × 2 = 20` + content row `minHeight: 40` (44 hit-target via `hitSlop: 2`) + `borderBottomWidth: 1` = 20 + 40 + 1 = **61** | 61 ≤ 64 ✓ |

The Pressable hit-target of 44×44 is preserved via `hitSlop` — visual height stays inside the cap, accessibility-targets §1 stays satisfied.

**Content per band (left-to-right reading order).**

| Slot | Phone | Tablet | Wide |
|---|---|---|---|
| Room-exit control | Glyph `‹` + `aria-label="Leave argument"`, 32×32 visual / 44×44 hit | Glyph `‹` + label `Leave`, 36 tall | Glyph `‹` + label `Leave`, 40 tall |
| Title (room or debate) | One line, `numberOfLines={1}`, `flex: 1`, font 13/16 | One line, `numberOfLines={1}`, `flex: 1`, font 14/18 | One line, `numberOfLines={1}`, `flex: 1`, font 15/20 |
| Status chip | Hidden if status is `open` (default); shown as 2-char glyph `[D]`/`[L]`/`[A]` for draft/locked/archived | Visible at tablet — short text "OPEN" / "DRAFT" / "LOCKED" | Visible at wide — same text |
| Side chip | Hidden | Visible — "FOR" / "AGAINST" / "OBS" / "MOD" | Visible — same |
| Private-room badge | Hidden if absent; shown as `[P]` glyph with `accessibilityLabel` from `ROOM_VISIBILITY_COPY.badge_private_a11y` | Visible — "PRIVATE" | Visible — "PRIVATE" |
| Timeline/Cards toggle | Two 32×32 segmented buttons | Two 36×36 segmented buttons + label below (on tablet, fits) | Two 40×40 segmented buttons + label |
| Overflow trigger (`⋯`) | One 32×32 button, opens an inline `<View>` (no Modal) carrying Invite / Make-private / Dev-chips | Same | Same |

The Pressable order (left → right) is fixed across bands so screen readers get a predictable order.

**Truncation rule.** Title takes `flex: 1`, `numberOfLines={1}`, `ellipsizeMode="tail"`. If the title is too long for the available space (after the room-exit + status chips on the left and the toggle + overflow on the right), the tail truncates. The brief is explicit: "If the room title is long, truncate. The Timeline gets priority."

**Accessibility surface.** The strip is NOT a single Pressable group — it's a row of independent Pressables, each with `accessibilityRole`, `accessibilityLabel`, and `accessibilityState` populated. Screen-reader users hear:
1. "Leave argument, button" (or "Back to Arguments, button")
2. The title as `<Text>` (read as content, not interactive)
3. The status chip's content (if shown)
4. "Timeline, tab, selected" / "Cards, tab" — segmented toggle
5. "More room options, button" — overflow

**Overflow content.** When the user taps `⋯`, an inline `<View>` (NOT a `Modal`, NOT a route push — TL-003 doctrine) appears below the strip. Inside the overflow:
- Invite trigger → reuses the existing `InvitePanel` mount path; the inline panel renders below the strip exactly as today (lines 691-702 of App.tsx)
- Make-private action (when `eligibility.allowed` is true and visibility is public) — same `handleOpenConfirm` callback; the make-private modal mounts via the existing `MakePrivateConfirmation` (line 198 of DebateDetailHeader)
- Dev chips (only when `__DEV__`) — Tree (dev) / Tracks (dev) — same callbacks the App.tsx roomToolbar currently exposes; these stay dev-only

The overflow itself counts as a transient surface that opens BELOW the strip, so it doesn't violate the cap; it adds vertical chrome only when actively shown (user-initiated).

**DebateDetailHeader.tsx modifications.** The current file's render body (lines 131-206) is replaced by the compact strip. The current styles (lines 209-262) are replaced with new compact-strip styles. The new component takes the same props (`debate`, `participantSide`, `onLeave`, `roomContract`, `currentUserId`, `publicRoomSeatMap`, `onVisibilityChanged`) PLUS three new props for the consolidated toggle / overflow:

```tsx
interface Props {

  viewMode?: ArgumentViewMode;
  /** UX-001.2 — Timeline/Cards toggle callback. */
```

The new props are OPTIONAL so any external test fixture that mounts `DebateDetailHeader` without them gets a strip that omits the toggle and overflow (graceful degradation). The internal layout is band-aware via `useHeaderBreakpoint().band`. The GAME-004 seat strip (`RoomContractSeatStrip`) is removed from the default strip render and folded into the overflow (acceptance criterion 5 of GAME-004 — "Renders only when the contract projection is supplied" — is preserved; the seat strip's `roomContract` prop just moves to the overflow render).

## Risks and trade-offs

### What could go wrong

**1. The Q10 arithmetic is tight (~9px slack on wide).** The brief's hard cap of 200px wide leaves no room for design drift. If the implementer's `paddingVertical` on the strip ends up at 12 instead of 10 (an aesthetic call), the wide sum becomes 199 instead of 191 — still passes but with 1px slack. If the AppHeader's actual measured height under RN-web differs from the design token (e.g. SafeAreaView adds 4-8px on iOS or a status bar inset on Android), the sum may exceed 200. **Mitigation:** the implementer measures the AppHeader's actual rendered height on each platform during the test phase; if a platform overshoot is found, the strip tightens further.

**2. The microMoment exemption.** Treating the microMoment as a transient banner (Q10) is a design decision. If the operator later wants microMoment to count toward the cap, the strip must compress more (to 32/40/48) and the test must check both states. **Mitigation:** the design explicitly states this assumption and lists "microMoment exemption" as a deferred operator decision in the Phase 2 framing.

**3. TimelineMiniMap collapsed state adds 32px.** The mini-map is not in the brief's 11-blocker list, but its collapsed header (28px chip + 4px padding) sits between the overlay and the rail. If it's left in place, the offset budget at the rail's top edge becomes 120+52+32+0+19 = 223 on wide — fails. **Mitigation:** the design moves the mini-map's collapsed header into the overlay container (top-right) as option (a) in Q9. If the implementer measures and finds the mini-map's `<View>` adds layout height even when its content is in the overlay, a small structural change is needed; the design specifies "either move the header chip into overlay OR accept the slack from strip tightening — implementer chooses based on measurement". 

**4. DebateDetailHeader rewrite scope.** The current file is 263 lines. The rewrite replaces ~80% of the body. Any existing tests on `DebateDetailHeader` (e.g. tests asserting "TWO rows of content", "title at fontSize 16") will fail. **Mitigation:** Q12 category 5 lists this explicitly; the implementer updates each affected test as part of the card. The `MakePrivateConfirmation` modal and `transitionRoomToPrivate` API are untouched, so the QOL-039 visibility flow is preserved.

**5. ArgumentTimelineMap's band overlay change.** Today the bands sit above the rail. After UX-001.2 they overlay the rail with `pointerEvents: 'none'` and `opacity: 0.6`. If the band labels conflict visually with the rail's nodes (overlap with node text or stub markers), the implementer may need to push them above by 6-10px. **Mitigation:** the design allows `BAND_RAIL_OFFSET` up to 12/12/16 — enough headroom to push bands above without violating the cap.

**6. ArgumentGameSurface.tsx body restructuring.** Moving 3 components (microMoment out, ScoreTracker down, ReadoutPanel down) without breaking the `mode === 'stack'` branch (which uses `<ArgumentBubbleStack>` instead of the timeline path). **Mitigation:** the design explicitly preserves the stack branch (the `mode === 'stack'` JSX in lines 783-804 is unchanged). The microMoment is moved out of both branches; the score tracker and readout panel relocations apply only to the timeline branch.
