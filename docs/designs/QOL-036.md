## 0. How this design was built

QOL-036 is an **additive extension of an already-shipped pure-TS model**
(`EvidenceArtifact` in `src/features/evidence/evidenceModel.ts`). It is not a new
surface. The design is derived from three grounded sources, in order:

1. The **band-space rent storyboard** "Evidence object model" table
   (`docs/ux-storyboards/band-space-rent-private-evidence-argument.md`, lines
   268–299) — the authoritative field list.
2. The **QOL-036 card body** acceptance criteria (the four `## Acceptance`
   bullets).
3. The **shipped EV-001 model** — the exact `EvidenceArtifact` interface this
   card extends without breaking.

Everything below is the consequence of holding those three against the doctrine
skills (`cdiscourse-doctrine`, `evidence-doctrine`, `supabase-edge-contract`).
Where the storyboard and EV-001 disagree on a name, EV-001's already-shipped
vocabulary wins and the storyboard field is mapped onto it (see §6 "Naming
alignment").

## 1. Goal & scope

Extend the EV-001 `EvidenceArtifact` so a **payment screenshot** can be attached
as a structured object: a monetary amount, a date, a **redacted** payer/payee, a
note/memo text, a **claimed applicability** (which obligation or month the
payment is asserted to cover), and a `confidence` field **pinned to
`user_asserted`**. The new fields are **additive** — every existing field on
`EvidenceArtifact` keeps its name, type, and meaning; every existing consumer
(EV-002, EV-003, EV-005, the receipt-chip, the timeline contract) compiles and
behaves identically. No raw financial account data is required, accepted, or
stored. A payment evidence object **never** produces a truth value or a
point-standing delta on its own.

### In scope

- A new optional `payment?: PaymentEvidenceMetadata` sub-object on
  `EvidenceArtifact` — a pure-TS interface, all fields optional or pinned.
- A new `EvidenceArtifactKind` value `'payment_screenshot'`.
- An `EvidenceConfidence` type whose only value relevant to this card is
  `'user_asserted'`; the field on a payment object is **always** that value.
- A redaction contract: `PaymentParty` carries only a redacted display token and
  an optional role label — never a raw account number, card number, routing
  number, IBAN, phone number, or email.
- A claimed-applicability sub-shape (`ClaimedApplicability`) — the data the
  QOL-037 applicability dispute consumes. QOL-036 only **defines and stores** the
  claimed side; QOL-037 owns the dispute, the `disputedApplicability`, and the
  applicability *status*.
- The adapter extension: `EvidenceAttachmentInput` gains an optional `payment`
  field, and `buildEvidenceArtifacts` copies it through, classifies the kind as
  `payment_screenshot` when present, and runs a redaction guard.
- A read-side display contract: how a payment artifact contributes to the
  EV-001 `ReceiptChipContract` and `TimelineEvidenceContract` (it contributes a
  *kind* and a label, never a *status upgrade* and never a verdict).
- A pure-TS redaction helper (`redactPaymentParty`, `assertNoRawAccountData`)
  that the model and the QOL-030 `add_evidence` box both call.

### Out of scope (own cards — see §13)

- The applicability **dispute** flow, `disputedApplicability`, the applicability
  *status* enum, and the structured evidence responses (Accept / Dispute date /
  Dispute amount / Dispute applicability / …) — **QOL-037**.
- The evidence-debt obligation layer — **EV-003**.
- The `add_evidence` **box UI** (the structured-form renderer, the field
  layout, the toggle) — **QOL-030 / QOL-031**. QOL-036 ships the *data the box
  binds to*, plus a field-contract table the box implementer reads; it does not
  ship a `.tsx` component.
- Image **upload + storage** of the actual screenshot file — explicitly deferred
  (EV-003 "Out of scope" already names "Evidence file upload + storage" as a
  separate card). QOL-036 stores screenshot **metadata** and a redacted
  reference, not the binary.
- Any persisted `argument_evidence_artifacts` table / migration — EV-001 chose
  the adapter-only path (b); QOL-036 keeps it. The persistence follow-up is the
  EV-001 "path (c)" card.

## 3. Problem statement

EV-001's `EvidenceArtifact` models *inspectable-text* evidence well: `url`,
`quote`, `sourceText`, a `kind`, a `sourceChainStatus`. It has **no structure
for a payment**. In the band-space rent storyboard the entire dispute is about
the *applicability* of a payment — which month a $120 transfer covered — and
that requires fields EV-001 does not have:

- There is no `amount`, no `date`, no `noteText` — so today a payment can only be
  pasted as free text into the argument `body`. Free text is un-disputable
  field-by-field: a challenger cannot say "I accept the amount, I dispute the
  month" because there are no separate fields to point at.
- There is no `claimedApplicability` — so the assertion "this covers March rent"
  has nowhere structured to live, and the QOL-037 applicability dispute has
  nothing to attach to.
- There is no redaction contract — so a screenshot or transfer reference risks
  carrying a raw account/card number into a stored JSONB snapshot.
- There is no `confidence` field — so nothing in the type system records that a
  payment screenshot is a **user assertion**, not a system-verified fact.

The cost of not fixing this: the storyboard's central interaction (dispute one
axis of a payment, accept the others) is impossible, and payment text leaks into
`body` where it cannot be redacted or separately disputed.

**Doctrine framing.** A payment screenshot proves *at most that a payment object
exists*. It never proves what the payment was *for*. QOL-036 must encode that in
the type system: `confidence` is pinned to `user_asserted`, and the model
carries **no** field that could be read as "this payment is verified / true /
proven". Existence, amount, date, note, and applicability are **separate axes**
so a dispute can be precise — but none of them is a truth verdict.

## 4. How it integrates with the one-box interface (QOL-030 `add_evidence` box)

QOL-036 is a **data card**. It produces no bespoke surface. Its single UI
integration target is the QOL-030 one-box composer's `add_evidence` box type
(QOL-030 §6.2; QOL-031 lists `Add evidence` as the Evidence-group Act-popout
entry, key `E`, target = node).

The integration contract — what QOL-036 hands the box, and what the box owns:

| Concern | Owner |
|---|---|
| The `PaymentEvidenceMetadata` interface + field semantics | **QOL-036** (this card) |
| The field-contract table (label, input kind, required?, max length) | **QOL-036** §7.4 — the box reads it |
| The redaction helper the box calls on payer/payee before storing | **QOL-036** (`redactPaymentParty`) |
| The structured-form renderer that lays the fields out | **QOL-030 `renderSchema('add_evidence', …)`** |
| The "Attach evidence" toggle + the box chrome | **QOL-030 / QOL-031** |
| Posting the move (the box submits via `submit-argument`) | **QOL-030** — the only write path |
| The applicability *dispute* responses (`respond_to_evidence` box) | **QOL-037** |

The box's `add_evidence` draft buffer (QOL-030 §6.6) carries a
`PaymentEvidenceMetadata` value. On post, the box serialises it into the
existing `attached_evidence[]` request array as **one attachment with a
`payment` sub-object** (see §7.2). No new request field at the top level; no new
Edge Function; the `submit-argument` contract grows by one optional nested
object inside an array element it already accepts.

When QOL-030 is not yet built, QOL-036 still lands cleanly: it is a pure-TS model
extension with its own tests. The model is consumable the moment it merges; the
box wires to it later. QOL-036 has **no hard dependency on QOL-030** — only
EV-001.

A payment artifact, once posted, renders through the **already-shipped EV-001
display path**: `buildEvidenceArtifacts` → `summarizeArtifactsForReceiptChip`
→ the receipt chip on the node (EV-002's `ReceiptChip`), and the EV-002
source-chain popover. QOL-036 adds the *kind* (`payment_screenshot`) and a
*payment label*; it does **not** add a new chip, popover, or pill component.

## 5. Doctrine constraints that shape the design

From `cdiscourse-doctrine`, `evidence-doctrine`, `supabase-edge-contract`:

1. **Evidence is never truth.** `confidence` is pinned to `user_asserted`. The
   model carries no `verified`, `confirmed`, `proven`, `valid`, or boolean
   "is-true" field. A payment object never emits a `PointStandingDelta` — it is
   inert data; only later *moves* (concession, sourcing, applicability dispute)
   can move standing, via the existing point-standing engine.
2. **Existence ≠ applicability.** Separate fields (`amount`, `date`, `noteText`,
   `claimedApplicability`) so a dispute can target one axis. QOL-036 stores only
   the **claimed** side; the disputed side and the status are QOL-037's.
3. **Popularity is not evidence.** No `payment` field references engagement,
   virality, likes, views, follower count, or "verified author". A payment is an
   inspectable artifact; its weight comes from inspectability, never popularity.
4. **No raw financial account data — ever.** `payer`/`payee` are `PaymentParty`
   objects that carry only a *redacted display token* (e.g. `"•••• 4821"` or a
   free-text role like `"the landlord"`) and an optional role label. The adapter
   runs `assertNoRawAccountData` and refuses to emit an artifact whose payment
   fields contain a digit run matching an account/card/routing/IBAN shape. Raw
   account data is **rejected at the model boundary**, not merely masked.
5. **No AI.** QOL-036 is pure TypeScript — no Anthropic / xAI / OpenAI / X API
   call, in the app or anywhere. No AI is consulted to classify or value a
   payment.
6. **No service-role, no direct insert.** QOL-036 adds no Supabase write path.
   The write path stays `submit-argument`. No migration in v1 (adapter-only,
   per EV-001's path (b)).
7. **Plain language.** Every system-generated string (kind label, field label,
   chip contribution) is plain English; no `snake_case` code surfaces. Banned
   tokens (`proof`, `true`, `false`, `winner`, `loser`, `verdict`, person
   labels) never appear in any QOL-036-produced string. Settlement copy uses
   "settled" (the storyboard forbids "case closed").

## 6. Naming alignment — storyboard vs. shipped EV-001

The storyboard's evidence-object table and the shipped EV-001 model use slightly
different names. EV-001 has shipped; its vocabulary wins. The mapping:

| Storyboard field | QOL-036 resolution |
|---|---|
| `kind` = `payment_screenshot` | New `EvidenceArtifactKind` value `'payment_screenshot'`. |
| `platform` | `payment.platform` — a **generic** free-text label (e.g. "a payment app"), never a brand requirement. |
| `date` | `payment.paidAt` — ISO-8601 date string. (`date` is avoided as a field name; `paidAt` disambiguates from `EvidenceArtifact.createdAt`.) |
| `amount` | `payment.amount` — a structured `{ value, currency }`, see §7.1. |
| `payer` / `payee` | `payment.payer` / `payment.payee` — `PaymentParty` objects, redacted. |
| `noteText` | `payment.noteText` — the memo on the record. |
| `claimedApplicability` | `payment.claimedApplicability` — a `ClaimedApplicability` sub-shape. |
| `disputedApplicability` | **Not QOL-036.** Owned by QOL-037 (it is a dispute output, not submitter data). |
| `confidence` (`user_asserted`) | `payment.confidence` — `EvidenceConfidence`, **pinned** to `'user_asserted'`. |
| `submittedBy` / `challengedBy` | Already covered: `EvidenceArtifact.addedByUserId` is the submitter; `challengedBy` is a QOL-037 dispute-record field, not artifact data. |
| `status` (`applicability_disputed` → `stronger`) | **Not QOL-036.** The applicability *status* enum is QOL-037; the source-chain *status* stays EV-001's `sourceChainStatus`. |

QOL-036 therefore adds exactly: one kind, one nested `payment` object, and the
supporting sub-types and helpers. It does **not** add `disputedApplicability`,
an applicability-status enum, or a dispute record — those are QOL-037, and
QOL-036's `payment` object is shaped so QOL-037 can attach them additively
later.

## 7. Data model & file changes

### 7.1 New types (additive to `src/features/evidence/evidenceModel.ts`)

All pure TypeScript. No React. No Supabase. No network. No `Date.now()` in the
model — the adapter copies timestamps from its input, exactly as EV-001 does.

```ts
// ── Extend the existing kind union (ADDITIVE — append one value) ──
export type EvidenceArtifactKind =
  | 'url'
  | 'quote'
  | 'source_text'
  | 'dataset'
  | 'screenshot_redacted'
  | 'manual_citation'
  | 'payment_screenshot';   // NEW (QOL-036)

/**
 * How confidently the SYSTEM treats the evidence's contribution. For a payment
 * object this is ALWAYS 'user_asserted' — a payment screenshot is a user's
 * assertion, never a system-proven fact. The type is a union so a future
 * card *could* add admin-confirmed states, but QOL-036 pins every payment
 * object to 'user_asserted' and a test enforces it.
 */
export type EvidenceConfidence = 'user_asserted';

/** A monetary amount. Structured so a challenger can dispute the amount axis
 *  alone (QOL-037). `currency` is an ISO-4217-style code; free-form is allowed
 *  for informal use ("USD", "GBP"). No field implies the amount is verified. */
export interface EvidenceAmount {
  /** The numeric amount. Non-negative. Stored as a number, not a string. */
  value: number;
  /** Currency code or short label. Max 8 chars. Defaults to caller-supplied. */
  currency: string;
}

/**
 * A redacted party on a payment record. Carries NO raw account data. The
 * displayToken is a user-or-adapter-redacted string ("•••• 4821",
 * "the landlord", "me"); roleLabel is an optional plain-language role.
 * `assertNoRawAccountData` runs over both fields.
 */
export interface PaymentParty {
  /** Redacted, display-safe token. Max 48 chars. Never a raw account number. */
  displayToken: string;
  /** Optional plain-language role, e.g. "payer", "the landlord". Max 48 chars. */
  roleLabel?: string;
}

/**
 * What the submitter ASSERTS the payment applies to. This is the CLAIMED side
 * only. QOL-037 adds the disputed side + the applicability status as separate,
 * additive fields on a dispute record — never by mutating this object.
 */
export interface ClaimedApplicability {
  /** Free-text plain-language statement: "March practice-room rent". Max 160. */
  statement: string;
  /** Optional coarse period label the submitter asserts: "March 2026". Max 32.
   *  Advisory only — never parsed into a Date, never validated as "correct". */
  periodLabel?: string;
  /** Optional opaque reference to the obligation this is claimed to cover
   *  (e.g. a room-scoped obligation id minted by a future card). QOL-036 does
   *  not create obligations; the field is reserved so QOL-037 can link one. */
  obligationRef?: string;
}

/**
 * Structured metadata for a payment / transfer screenshot. Every field is
 * OPTIONAL except `confidence`, which is pinned to 'user_asserted'. The object
 * itself is optional on EvidenceArtifact — present only for payment evidence.
 *
 * DOCTRINE: this object is INERT. It carries no truth value and emits no
 * point-standing delta. It is the data QOL-030's add_evidence box writes and
 * QOL-037's applicability dispute reads.
 */
export interface PaymentEvidenceMetadata {
  /** Always 'user_asserted'. Pinned. A test asserts no other value is stored. */
  confidence: EvidenceConfidence;
  /** Generic platform label: "a payment app". Never a required brand. Max 48. */
  platform?: string;
  /** ISO-8601 date (YYYY-MM-DD) the payment was made, as asserted. Optional. */
  paidAt?: string;
  /** The monetary amount, if any. */
  amount?: EvidenceAmount;
  /** Redacted payer. */
  payer?: PaymentParty;
  /** Redacted payee. */
  payee?: PaymentParty;
  /** The memo / note text on the record. Plain text. Max 280. */
  noteText?: string;
  /** What the submitter asserts the payment covers. */
  claimedApplicability?: ClaimedApplicability;
  /** True when an image artifact is attached. QOL-036 stores only that an image
   *  exists + its redaction state — NOT the binary. Upload is a deferred card. */
  hasScreenshotImage?: boolean;
  /** Confirms the submitter affirmed the screenshot/record is redacted of
   *  payer/payee account data before attaching. Defaults false. The box must
   *  set this true via an explicit user affirmation. */
  redactionConfirmed?: boolean;
}

// ── Extend EvidenceArtifact (ADDITIVE — append one optional field) ──
export interface EvidenceArtifact {
  // ... every existing field UNCHANGED ...
  /** Present only when kind === 'payment_screenshot'. Optional, additive. */
  payment?: PaymentEvidenceMetadata;
}

// ── Extend the adapter input (ADDITIVE — append one optional field) ──
export interface EvidenceAttachmentInput {
  // ... existing url / label / sourceText / quote / kind UNCHANGED ...
  /** Optional structured payment metadata. When present, the adapter
   *  classifies kind as 'payment_screenshot' and runs the redaction guard. */
  payment?: PaymentEvidenceMetadata;
}
```

### 7.2 Helpers (additive to `evidenceModel.ts`)

```ts
/** Frozen, for exhaustive tests. */
export const ALL_EVIDENCE_CONFIDENCES: ReadonlyArray<EvidenceConfidence>;

/**
 * Account-data shapes QOL-036 refuses to store. Conservative digit-run
 * detection — NOT a Luhn check (false negatives are acceptable; false
 * positives over-redact, which is the safe direction). Detects: 12–19 digit
 * runs (card / account), IBAN-shaped tokens, routing-number-shaped 9-digit
 * runs adjacent to bank keywords. Exported for tests + the box.
 */
export function detectRawAccountData(text: string): boolean;

/**
 * Throw-free guard. Returns the offending field name(s) if any payment field
 * contains raw account data; returns [] when clean. The adapter calls this and
 * DROPS the payment sub-object (keeping the artifact, downgrading kind) if it
 * is non-empty — it never silently stores raw account data, and never throws
 * in production (a thrown error in a render path is worse than a degraded
 * artifact). The QOL-030 box calls it BEFORE post to show an inline warning.
 */
export function findRawAccountDataFields(
  payment: PaymentEvidenceMetadata,
): ReadonlyArray<string>;

/**
 * Redact a free-text party string into a PaymentParty. If the input contains a
 * raw account-data run, the run is masked to "••••" + last 4 (when ≥4 trailing
 * digits) or fully masked. Role-only inputs ("the landlord") pass through. The
 * QOL-030 add_evidence box calls this on the payer/payee text fields before it
 * builds the PaymentEvidenceMetadata value.
 */
export function redactPaymentParty(
  rawText: string,
  roleLabel?: string,
): PaymentParty;

/**
 * Plain-language label for the payment kind, used by the receipt chip + a11y.
 * Returns e.g. "Payment record". NEVER returns a snake_case code, never a
 * verdict token.
 */
export function getPaymentEvidenceLabel(): string;

/**
 * Build a one-line, plain-language, redaction-safe summary of a payment object
 * for the receipt chip / side panel header. e.g.
 *   "Payment record — $120 on 2026-03-03, noted 'practice space'"
 * Omits any absent field. NEVER includes payer/payee raw data, NEVER a verdict.
 */
export function summarizePaymentEvidence(
  payment: PaymentEvidenceMetadata,
): string;
```

### 7.3 Adapter behavior change (`buildEvidenceArtifacts`)

`buildEvidenceArtifacts` (already in `evidenceModel.ts`) gains payment handling.
Existing behavior for non-payment attachments is **byte-for-byte unchanged**.

1. **An attachment is non-empty** if `url`, `sourceText`, `quote`, **or
   `payment`** is present. (Today: the first three. Adding `payment` means a
   payment-only attachment — no url/quote/sourceText — is no longer dropped.)
2. **Kind classification** (`classifyEvidenceKind`): if `att.payment` is present
   and `att.kind` is not explicitly set, the kind is `'payment_screenshot'`. An
   explicit `att.kind` still wins (consistent with EV-001's rule).
3. **Redaction guard:** for a payment attachment, the adapter runs
   `findRawAccountDataFields(att.payment)`. If it returns non-empty, the adapter
   **strips the `payment` sub-object**, downgrades the kind to
   `'screenshot_redacted'` (the artifact is still emitted — a missing screenshot
   beats a leaked account number), and the artifact's `label` falls back through
   the existing chain. The adapter **never** stores raw account data. This is a
   deterministic, throw-free degradation.
4. **`confidence` pin:** when the adapter emits a `payment` object, it sets
   `payment.confidence = 'user_asserted'` regardless of the input value. The
   input cannot smuggle a different confidence.
5. **`sourceChainStatus`:** a payment artifact runs the **existing** EV-001
   decision table over `(url, sourceText, quote)`. A payment object alone (no
   url/quote/sourceText) yields `unverified` — *the payment metadata is not a
   source pointer*. This is doctrine: a payment screenshot is an artifact, not a
   primary-source chain. It never auto-promotes to `source_and_quote` or
   `primary_present`. (If the user *also* attaches a url + quote, the existing
   table runs normally and the chain status reflects the url/quote, not the
   payment.)
6. **`risk`:** stays `'unknown'` (the EV-001 default; no payment-specific risk
   classifier in this card).
7. **Determinism:** same input → deeply-equal output, exactly as EV-001.

### 7.4 `add_evidence` box field-contract table (for the QOL-030 implementer)

QOL-030's `renderSchema('add_evidence', …)` lays these out. QOL-036 owns the
contract; QOL-030 owns the layout.

| Field | Input kind | Required? | Max | Notes |
|---|---|---|---|---|
| amount value | numeric | no | — | non-negative |
| amount currency | short text | no | 8 | defaults to a sensible label |
| paid-at date | date picker | no | — | ISO-8601; no "is this correct?" check |
| payer | text → `redactPaymentParty` | no | 48 | redacted on blur |
| payee | text → `redactPaymentParty` | no | 48 | redacted on blur |
| note text | text | no | 280 | the memo on the record |
| claimed applicability statement | text | no | 160 | "March practice-room rent" |
| claimed applicability period label | short text | no | 32 | advisory only |
| screenshot attached | toggle | no | — | sets `hasScreenshotImage`; binary upload deferred |
| redaction confirmed | required affirmation checkbox | yes-if-image | — | sets `redactionConfirmed`; box blocks post if image attached and unconfirmed |
| confidence | (not a field) | — | — | always `user_asserted`; shown as read-only helper text "Marked as: your assertion" |

The box must show, near submit, the inline warning surfaced by
`findRawAccountDataFields` if any field still contains an account-data run — but
even if the user ignores it, the adapter degrades safely (§7.3 step 3).

| State | When | Plain-language rendering |
|---|---|---|
| **No payment** | artifact has no `payment` | nothing payment-specific; the artifact renders as its base kind. |
| **Payment attached** | `payment` present, all fields clean | receipt chip shows the `payment_screenshot` kind; tapping shows `summarizePaymentEvidence` (e.g. "Payment record — $120 on 2026-03-03, noted 'practice space'"). |
| **Payment, partial** | `payment` present, some fields absent | summary omits absent fields gracefully ("Payment record — noted 'practice space'"). |
| **Redaction degraded** | adapter stripped `payment` for account data | artifact still present as `screenshot_redacted`; no payment summary; no error shown to a *reader* (the warning was the *submitter's*). |
| **Redaction unconfirmed (compose-time)** | box, image attached, `redactionConfirmed` false | the box (QOL-030) blocks post with helper "Confirm the screenshot is redacted before attaching." — a compose-time state, not a stored one. |
| **Account-data warning (compose-time)** | box, `findRawAccountDataFields` non-empty | inline warning near the field: "This looks like an account number — please remove it. Account numbers are never stored." |

Forbidden in every payment-related string, per the storyboard and doctrine:
`proof`, `true`, `false`, `winner`, `loser`, `verdict`, `proven`, `disproven`,
`case closed`, and every person label (`liar`, `dishonest`, `bad faith`,
`manipulative`, `extremist`, `propagandist`, `troll`, `bot`, `astroturfer`).
A ban-list test enforces it across `getPaymentEvidenceLabel` and
`summarizePaymentEvidence`.

## 9. Permission / rules implications

- **Who can attach a payment object:** the same actors who can attach any
  evidence — a participant composing an `add_evidence` move (QOL-030). QOL-036
  adds **no new permission**; it rides the existing composer + role gates.
- **Own-bubble rule unchanged:** an author cannot edit their own posted move
  (Stage 6.1.8 / 6.4 — own bubble exposes only `Qualifiers` + `Request
  deletion`). A payment object is part of the move's evidence; once posted it is
  immutable like the rest of the move. Correcting a payment is a **new move**,
  not an edit — consistent with how evidence debt resolves "by later moves, not
  retroactive edits" (`evidence-doctrine`).
- **No point-standing delta from QOL-036.** Attaching a payment object emits
  **no** `PointStandingDelta`. The point-standing engine
  (`src/features/pointStanding/`) is untouched by this card. Standing moves only
  when a *move* does something the engine already grades (concession, narrowing,
  sourcing, synthesis, evasion). A bare payment attachment is none of those.
- **Anti-amplification stays intact.** A payment object is an inspectable
  artifact — it earns *factual-standing eligibility* the same way any artifact
  does, *only when a move uses it to resolve a debt or back a claim*. QOL-036
  does not let a payment object earn standing on its own, and adds nothing that
  could let popularity substitute for it.
- **No validation block.** A payment object never blocks a post. Per EV-001,
  missing evidence hard-blocks only `argument_type='evidence'`; QOL-036 changes
  none of that. A malformed or absent payment field is advisory at most.
- **RLS:** QOL-036 adds no table and no policy. The `payment` sub-object lives
  inside the existing `arguments.client_validation` / `server_validation` JSONB
  snapshot (the same place `attachedEvidence` already lives). Visibility is
  governed by the existing `public.arguments` RLS — a payment object is visible
  exactly when its parent argument row is. In the band-space storyboard the room
  is private; the existing private-room RLS (QOL-039's territory) is what keeps
  the payment object private. QOL-036 relies on it; it does not re-implement it.

## 10. Edge cases

The implementer must handle each. Tests in §11.

1. **`payment` present but every field empty/absent except the pinned
   `confidence`.** The attachment counts as non-empty (rule 1 extended). The
   artifact is emitted with kind `payment_screenshot`; `summarizePaymentEvidence`
   returns just "Payment record". No crash.
2. **`payment` with a raw card number in `noteText`.** `findRawAccountDataFields`
   returns `['noteText']`; the adapter strips `payment`, downgrades kind to
   `screenshot_redacted`. No raw digits stored. Deterministic.
3. **`payer` typed as "•••• 4821".** Already masked — `detectRawAccountData`
   sees only a 4-digit run (below the 12-digit threshold) → clean → passes
   through unchanged.
4. **`payer` typed as a full 16-digit card number.** `redactPaymentParty` masks
   it to "•••• " + last 4; `findRawAccountDataFields` then sees the masked form
   → clean. If the box did NOT call `redactPaymentParty` and the raw number
   reaches the adapter, the §7.3 step-3 guard strips the whole `payment` object.
5. **`amount.value` negative or `NaN`.** The adapter clamps negative to 0 and
   drops a `NaN`/non-finite amount entirely (treats `amount` as absent). No
   throw. `summarizePaymentEvidence` omits a dropped amount.
6. **`paidAt` is a non-ISO string ("last Tuesday").** Stored verbatim — QOL-036
   does **not** parse or validate `paidAt` into a `Date` (parsing would imply
   the date is "verified"). It is displayed as-is. The field-contract table asks
   the box to use a date picker, but the model accepts any string defensively.
7. **`claimedApplicability.periodLabel` says "March" but `paidAt` is in
   February.** QOL-036 stores both **without comparing them** — detecting a
   mismatch is the QOL-037 *applicability dispute*, a user action, never an
   automatic system verdict. The model never flags this.
8. **Two payment artifacts on one move.** Adapter emits both, indices 0 and 1
   per EV-001's id rule. The receipt chip's `kinds` array lists
   `payment_screenshot` once (unique-preserving, per EV-001's
   `summarizeArtifactsForReceiptChip`).
9. **A payment artifact alongside a url+quote artifact.** Two artifacts; the
   chip's worst-status rule runs over both. The payment artifact contributes
   `unverified`; the url+quote artifact contributes `source_and_quote`. EV-001's
   worst-status ordering decides the aggregate — QOL-036 changes none of it.
10. **Concurrent edits.** Not applicable — the model is pure and read-only over
    already-fetched data; the adapter has no I/O.
11. **Offline / network failure.** Not applicable — pure model, no fetch.
12. **Permission-denied.** RLS on `public.arguments` already gates the JSONB
    snapshot the `payment` object lives in. The model performs no auth check.
13. **`submit-argument` boundary.** Today `submit-argument` maps
    `attached_evidence[]` with only `url`/`label`/`source_text`. A `payment`
    sub-object sent in a request element is currently *ignored* by that mapper —
    it is not an error, just not threaded through. QOL-036 does **not** modify
    the Edge Function (the card is model + adapter). Threading `payment` into the
    request is wired by **whichever card builds the `add_evidence` box's submit
    path (QOL-030)** or the EV-001 path-(c) persistence card. QOL-036's design
    explicitly flags this as a downstream wiring step (see §13 Out of scope and
    §15 Open questions) so the implementer does not silently assume the payload
    round-trips.
14. **A payment object inside a public room.** Allowed by the model — the
    storyboard's *private* framing is a room-policy concern (QOL-039), not a
    model constraint. QOL-036 stores redacted payer/payee regardless of room
    visibility; redaction is unconditional, never relaxed for private rooms.
15. **`redactionConfirmed` false but no image (`hasScreenshotImage` false).**
    Fine — `redactionConfirmed` is only *required* by the box when an image is
    attached. A metadata-only payment object needs no image confirmation.
16. **Verdict token in a user-typed `noteText` ("this proves I paid").** The
    model stores user free text verbatim — it does **not** sanitise user
    content (consistent with EV-001 edge case 13). The ban-list only governs
    *system-generated* strings (`getPaymentEvidenceLabel`,
    `summarizePaymentEvidence`'s own scaffolding — the user's `noteText` is
    quoted inside it but the scaffold words around it are clean).

## 12. Doctrine & safety self-check

Each line confirms a hard constraint is respected:

- **Evidence is never truth.** `confidence` is pinned to `user_asserted`; the
  `PaymentEvidenceMetadata` interface has no `verified`/`proven`/`true`/`valid`
  field (structural test enforces). `summarizePaymentEvidence` /
  `getPaymentEvidenceLabel` carry no verdict token (ban-list test). The
  storyboard's "settled, never case closed" rule is honoured — QOL-036 produces
  no settlement string at all.
- **No point-standing delta from evidence existence.** QOL-036 imports nothing
  from `src/features/pointStanding/`; attaching a payment emits no delta. Tested.
- **Existence ≠ applicability.** Separate `amount` / `paidAt` / `noteText` /
  `claimedApplicability` fields; QOL-036 stores only the *claimed* side and
  leaves room for QOL-037's *disputed* side additively.
- **No raw financial account data.** `detectRawAccountData` /
  `findRawAccountDataFields` / `redactPaymentParty` reject account/card/routing/
  IBAN shapes at the model boundary; the adapter strips a `payment` object that
  contains them. A `JSON.stringify` digit-run scan in tests proves no 12+ digit
  run survives.
- **Popularity is not evidence.** No `payment` field references engagement,
  virality, likes, views, follower count, or "verified author"; amplification
  ban-list test enforces.
- **No AI.** Pure TypeScript. No Anthropic / xAI / OpenAI / X API call anywhere
  in the card. No AI is consulted to classify or value a payment.
- **No service-role, no direct insert into `public.arguments`.** QOL-036 adds no
  write path. The write path stays `submit-argument`. No client touches a
  service-role key.
- **No migration, no RLS change.** Adapter-only, per EV-001's path (b). The
  `payment` object lives in the existing JSONB snapshot, gated by existing
  `public.arguments` RLS.
- **No `.env` change.** No env var, no key reference.
- **Plain language.** Every system-generated string is plain English; no
  `snake_case` code surfaces; ban-list + snake_case tests enforce.
- **Rules engine untouched.** `src/lib/constitution/engine.ts` is not edited.
  The QOL-036 model lives in `src/features/evidence/`, pure TS.
- **Additive — no breaking change.** Every existing `EvidenceArtifact` field and
  every existing evidence helper keeps its signature; the new fields are
  optional; EV-002 / EV-003 / EV-005 compile unchanged. An explicit
  "additive-no-break" test asserts an `EvidenceArtifact` without `payment` still
  type-checks and flows through every existing helper.
- **v1 scope guards.** No voting, no search, no push, no OAuth, no public API.
- **Terminology.** "argument" not "debate"; never "game"; never
  "proof/true/winner/loser". `npm run ux:terminology:audit` stays green.

## 16. Open questions

1. **`submit-argument` round-trip.** Today the Edge Function's
   `attached_evidence` mapper drops a `payment` sub-object silently. QOL-036
   deliberately does not modify the function (model-only card). Should the
   implementer add a one-line passthrough in `submit-argument` so a
   QOL-036-produced `payment` object survives a real post **before** QOL-030
   ships — or is it cleaner to leave the function untouched until QOL-030 wires
   the box? *Lean:* leave it untouched; QOL-036 is consumable as a pure model
   today, and the function change belongs with the box that produces the
   payload (avoids a deployed Edge Function that accepts a field nothing yet
   sends). Operator decides at design review.
2. **`amount` as `number` vs. minor-units integer.** QOL-036 specifies
   `amount.value: number` (a plain decimal). A cents/minor-units integer would
   avoid floating-point rounding. *Lean:* keep `number` for v1 — payment
   *evidence* is display-and-dispute data, not an accounting ledger; precision
   beyond two decimals is not a requirement. Revisit if a future card does
   arithmetic on amounts.
3. **`obligationRef` on `ClaimedApplicability`.** The field is reserved so
   QOL-037 can link a payment to a structured obligation. No card currently
   *mints* obligation ids. Should QOL-036 ship the field as reserved-but-unused
   (current design) or drop it until QOL-037 needs it? *Lean:* ship it reserved
   — adding it later forces every `ClaimedApplicability` consumer to migrate;
   one optional string costs nothing now (same reasoning EV-001 used for
   `EvidenceRisk`).
4. **Redaction strictness.** `detectRawAccountData` uses a conservative 12+
   digit-run heuristic. It will over-redact a long *non-account* number (e.g. a
   long order id). The safe direction is over-redaction. Confirm the operator
   accepts occasional false positives (an over-masked benign number) in
   exchange for never storing a real account number. *Lean:* yes — false
   positive degrades a label; false negative leaks PII.
