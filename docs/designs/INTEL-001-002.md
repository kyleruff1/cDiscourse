## Goal

Two pure-TS derivations turn signals the pipeline already stores into *repair-and-friction intelligence*, with zero new provider spend and zero schema change.

- **INTEL-001** detects **dodge-chains** (consecutive `did_not_address` marks along one thread — a friction/escalation reading that replaces tone-guessing) and computes a room's **debt-answer-rate** (a repair metric from the evidence-debt ledger). Both feed **heat inputs (engagement lane only)** and the **mediator summary weighting**, never factual standing.
- **INTEL-002** computes the **specificity KPI** — `marker-linked replies / total replies`, per-room and aggregate — and renders it as an **admin-only advisory readout** on the existing `AdminArgumentsTab` lane. Never user-facing, never per-person, never a leaderboard.

Doctrine that shapes every decision below:

- **cdiscourse-doctrine §1/§2/§3** — heat means *activity / friction*, never correctness, popularity, or consensus. A pile of `did_not_address` marks or unanswered debts is friction, **never** a truth or a standing signal.
- **point-standing-economy / evidence-doctrine (anti-amplification)** — the load-bearing boundary. Engagement credit and factual-standing eligibility are separate. Neither INTEL module may emit a `PointStandingDelta` or import any `pointStanding` path. A `did_not_address` chain feeds heat/mediator; it can **never** lower a claim's standing (the exact FEEDBACK-001 pin, extended).
- **Thread-scoped, never person-scoped** (#900 binding) — chains describe THREADS (argument-id runs), never a user. No per-person dodge counts.
- **Rooms + aggregates only, no per-person rates** (#901 binding) — the specificity KPI is a room and program metric, never an author scorecard.
- **Graceful zero-data** — marks are sparse until `move_marks` flips; markers are sparse until `timestamp_rebuttals` flips. Every derivation is total on empty input and every readout has an honest empty state.

## Envelope assumptions (vs FEEDBACK-002 — the implementer reconciles)

A second designer concurrently owns `docs/designs/FEEDBACK-002.md`, which defines the shared **DerivedSignal envelope** convention. This design does **not** touch that file. There is no `DerivedSignal` type in the codebase yet (confirmed by grep). INTEL modules therefore return **plain data types** as their source of truth, and a thin adapter in `roomIntelSignals.ts` wraps them into the envelope. The following are numbered ASSUMPTIONS the implementer must reconcile against the shipped FEEDBACK-002 envelope; each is written so INTEL can ship even if FEEDBACK-002 slips (no hard dependency).

1. **Envelope wraps, never replaces.** Assume `DerivedSignal<T>` is a wrapper (roughly `{ kind, source: 'derived', advisory: true, lane, data: T, inputHash? }`). INTEL's plain types (`DodgeChainDerivation`, `RoomDebtAnswerRate`, `RoomIntelSignals`, `RoomSpecificityKpi`, `AggregateSpecificityKpi`) are the load-bearing shapes; the adapter maps `data: <plain type>`. If FEEDBACK-002 instead expects INTEL to return the envelope directly, only `roomIntelSignals.ts` changes.
2. **Lane discriminator = engagement.** Assume the envelope carries a lane/axis discriminator distinguishing engagement-lane from standing-lane signals. INTEL declares **`engagement`** and never `standing`. Reconcile the exact literal (`'engagement'` vs `'activity'` vs enum) with FEEDBACK-002.
3. **Advisory / non-user-facing marker.** Assume an `advisory: true` (or `userFacing: false`) field. INTEL sets it. Reconcile the field name.
4. **No standing-bound numeric field.** Assume the envelope has NO score/delta/weight field that could route into standing. If FEEDBACK-002 adds a generic `weight`, INTEL populates only descriptive counts/rates — never a value the standing pipeline reads. The anti-amplification pin (T-AA) asserts this structurally.
5. **Envelope module location.** Assume the type lives in a shared module (likely `src/features/feedback/derivedSignal.ts`). INTEL imports it **type-only**. Until it exists, `roomIntelSignals.ts` defines its plain types independently and the adapter is a no-op passthrough, so INTEL is not blocked by FEEDBACK-002's timing.
6. **Deterministic input hash (if any).** If the envelope carries an `inputHash`, INTEL computes it with a clock-free, order-independent hash (the `deriveMediatorBoardState.computeInputHash` idiom). Reconcile the hashing helper. INTEL never uses `Date.now()` in the hash.

> Implementer action: after FEEDBACK-002 merges, reconcile assumptions 1–6 in `roomIntelSignals.ts` only; the derivation modules and their tests are envelope-agnostic and should not change.

### INTEL-001 — dodge-chain

```ts
// src/features/intel/dodgeChainModel.ts
/** Minimal tree node the derivation needs (id + tree parent). */
export interface DodgeChainNodeInput { id: string; parentId: string | null; }

/** One dodge-chain: a maximal run of >=2 consecutive unaddressed moves on one thread. */
export interface DodgeChain {
  /** The shallowest (head) node of the run — the chain's stable anchor. */
  anchorArgumentId: string;
  /** Ordered head -> leaf. Length >= MIN_DODGE_CHAIN_LENGTH (2). Argument ids only — never a person. */
  memberArgumentIds: readonly string[];
  /** Node count in the run (== memberArgumentIds.length), >= 2. */
  length: number;
}

export interface DodgeChainDerivation {
  /** Maximal single-thread unaddressed paths (len >= 2), sorted by anchorArgumentId. */
  chains: readonly DodgeChain[];
  /** Distinct nodes participating in ANY chain (deduped — branch prefixes counted once). */
  unaddressedChainNodeCount: number;
  /** Longest chain length in nodes (0 when no chain). The primary "consecutive on one thread" measure. */
  longestChainLength: number;
  /** chains.length. */
  chainCount: number;
}

export const MIN_DODGE_CHAIN_LENGTH = 2; // a single dodge is not a chain.

export function deriveDodgeChains(input: {
  unaddressedMoveIds: readonly string[];   // FEEDBACK-001 MoveMarkAggregate.unaddressedMoveIds
  nodes: readonly DodgeChainNodeInput[];
}): DodgeChainDerivation;
```

**The chain rule (precise, deterministic):**
- Let `U` = the set of "unaddressed" nodes = nodes whose id is in `unaddressedMoveIds`. (A `did_not_address` mark on move `M` means *M did not answer the point it replied to* — so a run of these is a stretch of the thread that is not connecting.)
- A node `c` is **linked to its parent in a chain** iff `c ∈ U` and `parent(c) ∈ U` (the parent edge is a tree edge; both endpoints unaddressed).
- A **chain head** = a node `h ∈ U` such that `parent(h) ∉ U` (or `h` has no parent / its parent is not in the room set).
- From each head, walk downward through `U`-children. Each **maximal head→leaf path** of length ≥ `MIN_DODGE_CHAIN_LENGTH` is a `DodgeChain`. When a node has multiple `U`-children the path branches; each maximal branch is emitted as its own chain (they share the head prefix).
- **Branch inflation is neutralised at the feed:** the heat/mediator feed uses `unaddressedChainNodeCount` (distinct nodes, deduped) and `longestChainLength` — never `Σ chain.length` — so a fan-out of unaddressed replies cannot multiply the escalation signal.
- **Cycle/orphan safety:** a `parentId` pointing outside the node set is treated as "no parent" (head). A parent-walk guard caps traversal at node count (the `evidenceDebtModel.buildSubtreeMembership` idiom) so a malformed cycle cannot loop.
- **Determinism:** chains sorted by `anchorArgumentId`; members ordered head→leaf; sibling branches expanded in child-id sorted order. No clock, no randomness.

### INTEL-001 — debt-answer-rate

```ts
// src/features/intel/debtAnswerRateModel.ts
import type { EvidenceDebt } from '../evidence/evidenceDebtModel';

export interface RoomDebtAnswerRate {
  debateId: string;
  totalDebts: number;
  /** Debts a supplying/resolving move landed on (supplied | accepted_by_participant | accepted_by_both). */
  answeredDebts: number;
  /** Still owed (OPEN_EVIDENCE_DEBT_STATUSES: requested | challenged | unresolved | stale). */
  openDebts: number;
  /** Discharged (accepted_by_both). Subset of answeredDebts. */
  settledDebts: number;
  /** Relocated to a side-branch (branched). Excluded from the rate denominator. */
  branchedDebts: number;
  /**
   * answeredDebts / (totalDebts - branchedDebts). NULL when the denominator is 0
   * (no answerable debts) — never a fabricated 0%/100%. A REPAIR metric, room-level;
   * NEVER a per-claim standing and NEVER a per-person rate.
   */
  answerRate: number | null;
}

export function deriveRoomDebtAnswerRate(input: {
  debateId: string;
  debts: readonly EvidenceDebt[];  // already derived by EV-003 deriveEvidenceDebts (clock-free here)
}): RoomDebtAnswerRate;
```

**Status → bucket mapping (reuses EV-003's own unions):**
- `answered` = `supplied` | `accepted_by_participant` | `accepted_by_both` (a source landed and was not re-challenged/declined).
- `open` = `OPEN_EVIDENCE_DEBT_STATUSES` = `requested` | `challenged` | `unresolved` | `stale` (`challenged` = supplied-then-questioned ⇒ still owed).
- `branched` = relocated; **excluded** from both numerator and denominator (penalising a legitimate relocation would be perverse — mirrors EV-003 excluding `branched` from the open set).
- `settled` = `accepted_by_both` (reported separately; subset of answered).

### INTEL-002 — specificity KPI

```ts
// src/features/intel/markerSpecificityModel.ts
/** Minimal reply-move input (root/reply discriminator + type for the secondary breakdown). */
export interface SpecificityReplyInput {
  id: string;
  parentId: string | null;     // null => room root (excluded from the denominator)
  argumentType: string | null; // for the secondary rebuttal-typed breakdown
  status: string | null;       // 'deleted' / inactive rows are excluded
  inactiveAt?: string | null;
}
/** Minimal active-marker input (reply linkage only). */
export interface SpecificityMarkerInput { replyArgumentId: string | null; deletedAt: string | null; }

export interface RoomSpecificityKpi {
  debateId: string;
  /** Reply moves = non-root, non-deleted, non-inactive arguments. The denominator. */
  totalReplies: number;
  /** Distinct reply ids that appear as reply_argument_id on >=1 active marker. The numerator. */
  markerLinkedReplies: number;
  /** markerLinkedReplies / totalReplies. NULL when totalReplies === 0 ("No replies yet"). */
  rate: number | null;
  // Secondary: fidelity to the Design-Pass phrasing "% of rebuttals".
  totalRebuttals: number;         // argumentType === 'rebuttal' subset of replies
  markerLinkedRebuttals: number;
  rebuttalRate: number | null;
}

export interface AggregateSpecificityKpi {
  roomCount: number;
  totalReplies: number;           // pooled sum
  markerLinkedReplies: number;    // pooled sum
  /** POOLED: Σ markerLinkedReplies / Σ totalReplies (NOT the mean of per-room rates). NULL when Σ totalReplies === 0. */
  rate: number | null;
  totalRebuttals: number;
  markerLinkedRebuttals: number;
  rebuttalRate: number | null;
}

export function deriveRoomSpecificityKpi(input: {
  debateId: string;
  replies: readonly SpecificityReplyInput[];
  markers: readonly SpecificityMarkerInput[];
}): RoomSpecificityKpi;

export function deriveAggregateSpecificityKpi(rooms: readonly RoomSpecificityKpi[]): AggregateSpecificityKpi;
```

**KPI rules:**
- **Denominator = reply moves** (`parentId !== null`, not `deleted`, not inactive). The room root is excluded (it targets nothing).
- **Numerator = distinct reply ids** appearing as `replyArgumentId` on an active marker (`deletedAt === null`, `replyArgumentId !== null`). Deduped, so two markers on one reply count the reply once.
- **Rate** = numerator / denominator, `null` when denominator is 0.
- **Aggregate = pooled** (Σ/Σ), never the mean of per-room rates (a 1-reply room must not swing the program KPI as much as a 200-reply room).
- **Rebuttal-typed secondary** — the Design Pass phrases the KPI as "% of *rebuttals*." Primary uses all reply moves (robust, marker-linkage is available on any reply); the `rebuttalRate` restricts the denominator to `argumentType === 'rebuttal'` for fidelity. Interpretive decision D8 (operator-review-flagged).
- **No per-person axis.** `created_by` on markers and `authorId` on replies are never read/grouped. Pinned by T-NP.

## Design decisions

1. **`dodgeChainModel` consumes the FEEDBACK-001 aggregate (`unaddressedMoveIds`), not raw rows.** Reuses the shipped, active-only, no-score derivation; keeps INTEL one layer above marks and structurally unable to see a person.
2. **Chain = maximal single-thread unaddressed path (len ≥ 2); feed uses deduped node count + longest length.** Most literally matches "consecutive `did_not_address` on one thread" while neutralising branch inflation at the heat/mediator seam.
3. **`debtAnswerRateModel` takes already-derived `EvidenceDebt[]` and is clock-free.** Staleness was resolved upstream by `deriveEvidenceDebts(nowMs)`; the rate is a pure function of statuses ⇒ deterministic, no injected clock.
4. **`answerRate` is `null` on a zero denominator; `branched` excluded.** Honest zero-data; no fabricated 0/100%; relocation is not counted as failure or success.
5. **Heat feed = one additive optional `dodgeChainHits` term, gated on `move_marks`.** Additive-optional keeps every existing `computeConversationHeat` caller/test green; the `move_marks` gate makes it byte-identical live (marks default OFF). Engagement-lane only.
6. **No second heat term for debt-answer-rate.** The gallery heat already counts `sourceChainHits` / `evidenceHits`; adding an unanswered-debt term would double-count the same evidence pressure. Debt-answer-rate feeds the **mediator weighting** (`unresolvedDebtPressure`) instead. The optional heat param exists but is intentionally not wired (documented; reversible).
7. **Mediator feed = optional `weightingSignals` on the existing `MediatorBoardOptions`, consumed only in `deriveNextAction`'s tie-break, through the single derivation.** Absent ⇒ byte-identical. This honours the "single derivation" invariant (the weighting enters the one deriver via its existing options seam; it invents no state and changes no label). *Fallback M-B if the reviewer wants `deriveMediatorBoardState.ts` strictly byte-identical: ship an `applyDodgeWeighting(board, signals)` overlay the rail consumes instead — noted as the operator/reviewer choice.*
8. **INTEL-002 denominator = all reply moves; `rebuttalRate` is the secondary fidelity metric.** Marker-linkage is available on any reply and `parentId` is a robust root discriminator; the Design-Pass "rebuttals" phrasing is honoured by the secondary breakdown. **Interpretive — operator-review-flagged.**
9. **Marker-visibility zero is labelled, not hidden.** Where an admin lacks marker visibility for a room the rate reads 0; the model carries `markerVisibilityCaveat` and the readout copy is a literal count ("N of M replies pinned a moment") plus one aggregate caveat line ("counts reflect markers visible to you"). A truly global aggregate needs a service-role admin read (follow-up card). Honest, not fabricated.
10. **INTEL-002 aggregate is pooled (Σ/Σ), not a mean of rates.** Small rooms cannot distort the program KPI.
11. **No new feature flag.** INTEL-001 reuses `move_marks` (already OFF); INTEL-002 is admin-gated + graceful-zero. Adding a flag would be ceremony over an inert-by-default, advisory surface.

## Edge cases

**INTEL-001 dodge-chain:**
- Empty `unaddressedMoveIds` (marks OFF / sparse) ⇒ `{ chains: [], unaddressedChainNodeCount: 0, longestChainLength: 0, chainCount: 0 }`.
- One isolated `did_not_address` (no unaddressed parent/child) ⇒ no chain (len 1 < 2).
- Two consecutive ⇒ one chain len 2.
- Branch: parent unaddressed with two unaddressed children ⇒ two chains sharing the head; `unaddressedChainNodeCount` counts the head once.
- `unaddressedMoveIds` containing an id absent from `nodes` ⇒ ignored (not in the tree).
- `parentId` pointing outside the node set ⇒ treated as a head (no crash).
- Malformed cycle in `parentId` ⇒ parent-walk guard caps traversal; no infinite loop.
- Determinism: same inputs (any order) ⇒ deep-equal output.

**INTEL-001 debt-answer-rate:**
- No debts ⇒ `totalDebts 0, answerRate null`.
- All `branched` ⇒ denominator 0 ⇒ `answerRate null` (not 0%).
- Mixed statuses ⇒ counts partition exactly; `answered + open + branched === total`.
- `challenged` counts as open (still owed), not answered.

**INTEL-002 specificity:**
- Zero replies (root-only room) ⇒ `totalReplies 0, rate null` ⇒ "No replies yet."
- Replies but zero visible markers ⇒ `markerLinkedReplies 0, rate 0` ⇒ "0 of N replies pinned a moment" + caveat (could be sparse OR invisible; not a verdict).
- Marker with `replyArgumentId === null` (standalone marker) ⇒ excluded from the numerator.
- Two markers on one reply ⇒ reply counted once (deduped).
- Deleted/inactive reply ⇒ excluded from the denominator; a stale marker pointing at it ⇒ not counted (its reply isn't in the denominator).
- Aggregate over zero rooms ⇒ `roomCount 0, rate null`.
- Aggregate where every room has `rate null` ⇒ pooled `null`.

**Doctrine edges:**
- "Can a wall of `did_not_address` marks lower a claim's standing?" — **No.** INTEL imports no `pointStanding`; the signal feeds heat/mediator-weighting only. Pinned by T-AA.
- "Does the specificity KPI shame the author who writes vague rebuttals?" — **No.** No per-person axis exists; the KPI is room/aggregate only. Pinned by T-NP.
- "Does a high dodge-chain count mean one side is losing?" — **No.** It is thread friction; the mediator weighting only re-orders *which open point to address next*, never who is right.

## Risks

- **Sparse-data honesty (both cards).** Marks are OFF and markers are behind `timestamp_rebuttals`; most rooms show empty dodge derivations and 0% specificity. Every derivation is total-on-empty and every readout has an honest empty state, but the operator must read "0%" as "0 visible so far," not "the redesign failed." Called out in copy + Operator steps.
- **The marker-visibility gap (INTEL-002 aggregate).** `timestamp_markers` RLS has no admin-broad arm, so the admin aggregate is bounded by the admin's room/circle membership; a room with invisible markers reads 0. Mitigated by the `markerVisibilityCaveat` + literal-count copy (D9); a truly authoritative global aggregate needs a service-role admin read — out of scope, flagged as a follow-up.
- **The per-person trap (both cards).** Both signals sit one join away from an author scorecard (`created_by` on markers, `authorId` on replies, marker `created_by`). The design forbids any per-person grouping; **T-NP** is the pin, with a firing control. A reviewer must treat a per-person leak as a Block, not a nit.

- **Mediator single-derivation drift.** The `weightingSignals` addition touches `deriveMediatorBoardState.ts`. The FEEDBACK-001 pin kept that file zero-diff. **T-MW** proves absent-option byte-identity (incl. `inputHash`); if the reviewer prefers strict zero-diff, fall back to the M-B overlay (D7). Flagged for operator/reviewer.
- **Heat double-count.** Reason for D6 (no debt-answer heat term) — the existing `sourceChainHits`/`evidenceHits` already carry evidence pressure. If a future edit wires the optional debt heat param, it must be reconciled against those terms.
- **Existing snapshots.** The heat, mediator, and admin-tab modified files may have structural snapshot tests; the additive/optional changes must keep the absent/flag-off branch byte-identical (proved by the byte-identity tests).
