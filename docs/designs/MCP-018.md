## 1. Goal (one paragraph)

MCP-018 builds the **operator-hosted MCP-server adapter** for the `semantic-referee` Edge Function: the implementation of MCP-009's reserved `mcp` provider slot. When `SEMANTIC_REFEREE_PROVIDER=mcp`, a new `mcpAdapter.ts` reaches an operator-hosted MCP server over HTTPS (`SEMANTIC_REFEREE_MCP_URL`, authenticated with `SEMANTIC_REFEREE_MCP_TOKEN`, both read via `Deno.env.get()` inside the Edge Function only) and asks it to classify one argument move's structure. It mirrors MCP-017's live-provider discipline step-for-step. The design is shaped by doctrine constraints that are blockers if violated: (1) **`mcp` is opt-in by environment / DB config, never the default** — `providerRoutingCore.ts` keeps its doctrine-critical `?? 'mock'` fallback byte-identical; an unset provider picks `mock`. (2) **The MCP-server call lives only in the Edge Function** — the `fetch` is in `mcpAdapter.ts` inside the `_shared/semanticReferee/` tree, the MCP URL + token are read there via `Deno.env.get()`, and a source-scan test enforces that no client code, no `src/`, no `app/`, no `.env*`, no git file ever names or contains the token. (3) **The adapter can fail without failing the user** — url-missing, token-missing, network failure, non-OK HTTP, parse failure, or an MCP-server output that fails the schema + content-safety validator all return a typed `unavailable` outcome that the routing core translates to an `{ enabled: false, reason }` outcome; the caller still gets a valid `ClassifyMoveOutcome` and the user never sees an error. (4) **Every MCP-server response is re-validated through the boundary contract** — even a successful response runs through the Deno `SemanticRefereePacketSchema` and `scanPacketContent` before it leaves the boundary; any smuggled `block` field, `authoritative: true`, verdict token, or off-contract value forces the deterministic fallback; `providers.ts`'s `validateOrFallback` is the registry-level third wall. (5) **The packet is structural only** — the adapter sends the same redacted move + parent + room context the `anthropic` provider sends, asking only for the bounded structural binaries of MCP-001's catalog-v0; nothing in the request asks the MCP server for a verdict, a winner, a truth value, or a popularity reading, and the MCP server is contractually required to honor the same catalog.

## 3. Doctrine constraints (the lines a reviewer enforces)

Any violation is a blocker. These are the MCP-017 §3 constraints, re-pointed at `mcp`.

1. **Mock is still the default after MCP-018.** `providerRoutingCore.ts` keeps the `?? 'mock'` fallback byte-identical. `SEMANTIC_REFEREE_PROVIDER=mcp` is opt-in; unset means `mock`. The existing `semanticProviderRegistry.test.ts` "default is mock" test must keep passing untouched. (Acceptance criterion 4.)
2. **The adapter never throws to the client.** Every error path — `url_missing`, `token_missing`, `api_error`, `rate_limited`, `network_error`, `parse_failure`, `validation_failed` — returns a typed `ProviderResult` of kind `unavailable`. `providerRoutingCore.ts` translates it into `{ enabled: false, reason }`; `classifyMove` callers always receive a structurally valid `ClassifyMoveOutcome`. The Edge Function returns HTTP 200 on every one of these paths.
3. **The MCP token + URL are read only via `Deno.env.get()` inside `mcpAdapter.ts`.** `SEMANTIC_REFEREE_MCP_URL` and `SEMANTIC_REFEREE_MCP_TOKEN` are never: imported from another file; passed to the client; echoed in a response body; logged (the function may log: function name, caller user id, `roomId`, decision name, provider name, duration, error class — nothing else); mentioned in `app/`, `src/`, `.env`, `.env.local`, `.env.example`, or any git-tracked file other than as a documented name in this design doc and the runbook.
4. **Every MCP-server response is re-validated through the boundary contract.** The adapter's parsed JSON runs through (a) the Deno `SemanticRefereePacketSchema` in `schema.ts` — strict objects, `authoritative: z.literal(false)`, `value` in `{0,1}`, `scoreHints` in `0..3`; then (b) `scanPacketContent` in `contentSafetyScan.ts` — verdict / person / secret / PII / smuggled-`block` / chain-of-thought / raw-prompt detection. A failure on **either** step returns `validation_failed` and the caller falls back to the deterministic packet. `providers.ts`'s `validateOrFallback` is the registry-level third wall.
5. **The request asks for structure, never for truth.** The MCP-tool request carries the redacted move + parent + room context + the requested catalog-v0 classifier ids — the same payload `buildAnthropicRequestBody` assembles, minus the Anthropic-Messages envelope. Nothing in it asks whether anything is true, correct, right, wrong, factual, proven, popular, or who is winning. The MCP server is contractually required to honor MCP-001's catalog (it classifies move structure, never truth — MCP-009 §"MCP-server placement").
6. **The MCP server can't smuggle a verdict.** The Deno `SemanticRefereePacketSchema` is `.strict()` everywhere and has no `block`, no `verdict`, no `truthValue`, no `winner` field to populate. `scanPacketContent` additionally rejects a verdict token hiding inside a `reasonCode`. `authoritative` is hard-pinned to `false` by the adapter's identity stamp; the MCP server cannot override it.
7. **No service-role, no privileged write.** MCP-018 reads no `SUPABASE_SERVICE_ROLE_KEY`, builds no service-role client, writes no table, no migration, no RLS change. The adapter only does an outbound `fetch` to the operator's MCP server.
8. **Cost is bounded by upstream gating.** MCP-012's trigger gates, batching (≤5 classifiers per call), in-memory LRU cache, token budget, and 2-attempt retry policy cap the per-room call rate. MCP-018 changes none of these and adds no new caller.

## 4. Data model

**No new data model. No new type. No migration. No new table. No RLS change.**

- `mcpAdapterCore.ts` **reuses** `ProviderResult` and `ProviderUnavailableReason` — but with **one addition to the reason vocabulary** (see "Defect / mismatch resolution 1" below). It imports `ClassifyMoveRequest` / `SemanticRefereePacket` from `types.ts` and `ProviderResult` from `anthropicClassifierCore.ts` (a **type-only** import — see resolution 2).
- `mcpAdapter.ts` reuses `SemanticRefereePacketSchema` (from `schema.ts`), `scanPacketContent` (from `contentSafetyScan.ts`), `PACKET_VERSION` + `ClassifyMoveRequest` + `SemanticRefereePacket` (from `types.ts`), and `SEED_PROMPT_VERSION` (from `seedPrompt.ts`). All exist.

### Defect / mismatch resolution 1 — the `mcp` failure-reason vocabulary

MCP-017's `ProviderUnavailableReason` (in `anthropicClassifierCore.ts`) is:

```ts
'key_missing' | 'api_error' | 'rate_limited' | 'network_error' | 'parse_failure' | 'validation_failed'
```

The `mcp` adapter has **two configuration inputs** (a URL and a token), not one key. "key_missing" is the wrong word for a missing URL. The design resolves this cleanly and minimally:

- `mcpAdapterCore.ts` declares its own boundary-internal `McpUnavailableReason`:
  ```ts
  export type McpUnavailableReason =
    | 'url_missing'
    | 'token_missing'
    | 'api_error'
    | 'rate_limited'
    | 'network_error'
    | 'parse_failure'
    | 'validation_failed';
  ```
  It is the `mcp` adapter's own failure vocabulary, the exact analog of `ProviderUnavailableReason`. `mcpAdapterCore.ts` also declares its own result union `McpProviderResult = { kind: 'success'; packet } | { kind: 'unavailable'; reason: McpUnavailableReason }`.
- **`ClassifyMoveDisabledReason` does NOT need widening.** The routing core must translate an `unavailable` `mcp` result into a `ClassifyMoveOutcome` `reason`. Inspect the current type:
  - The Deno `ClassifyMoveDisabledReason` (in `types.ts`, widened by MCP-017) and its Node twin (in `src/lib/edgeFunctions.ts`) are **already** the nine-value union: `'disabled' | 'not_configured' | 'not_implemented' | 'key_missing' | 'api_error' | 'rate_limited' | 'network_error' | 'parse_failure' | 'validation_failed'`.
  - Of `McpUnavailableReason`'s seven values, **five** (`api_error`, `rate_limited`, `network_error`, `parse_failure`, `validation_failed`) are already in that union. Only `url_missing` and `token_missing` are not.
  - The design **maps `url_missing` and `token_missing` onto the existing `'not_configured'` reason** at the routing-core translation step. Rationale: a missing MCP URL or token *is* a not-configured provider — `'not_configured'` already exists and already means exactly "the operator selected this provider but did not finish wiring it." The caller (`classifyMove` consumer) treats every `{ enabled: false }` reason identically — fall back to the deterministic layer-1 result — so the precise reason string is a diagnostic, not a behavioural fork. Mapping onto an existing value means **MCP-018 touches no type file at all** — a strictly smaller blast radius than MCP-017's deliberate widening, and it keeps `semanticDenoNodeParity.test.ts` trivially green.
  - The translation lives in the routing core (see §6) as an explicit small map. A reviewer should expect: `url_missing → 'not_configured'`, `token_missing → 'not_configured'`, everything else passed through unchanged (it is already a legal `ClassifyMoveDisabledReason`).

  **Implementer note — the one decision left open, with a recommendation:** an alternative is to widen `ClassifyMoveDisabledReason` with `'url_missing' | 'token_missing'` for byte-faithful diagnostics, mirroring MCP-017's widening. The design **recommends the no-widening map-to-`not_configured` approach** — it is smaller, touches zero type files, needs zero parity-test confirmation, and loses nothing the caller acts on. If a future operator wants the finer-grained diagnostic in logs, `mcpAdapter.ts` may additionally `console.warn` the boundary-internal `McpUnavailableReason` (the *reason class* only — never the URL, never the token); that is a log line, not a contract change. The implementer picks the map-to-`not_configured` path unless the reviewer explicitly asks for the widening.

### Defect / mismatch resolution 2 — `ProviderResult` reuse vs a new `McpProviderResult`

`providerRoutingCore.ts`'s injected `SemanticRefereeProviderDeps.runAnthropic` is typed `(request) => Promise<ProviderResult>`, and `ProviderResult` (in `anthropicClassifierCore.ts`) is `{ kind: 'success'; packet } | { kind: 'unavailable'; reason: ProviderUnavailableReason }`. Because `McpUnavailableReason` differs from `ProviderUnavailableReason` (it has `url_missing`/`token_missing` instead of `key_missing`), the `mcp` adapter's result is **not** assignable to `ProviderResult`.

The design resolves this with a tiny, deliberate type decision:

- `mcpAdapterCore.ts` declares `McpProviderResult` (its own success/unavailable union, parameterized on `McpUnavailableReason`).
- `SemanticRefereeProviderDeps` gains `runMcp: (request: ClassifyMoveRequest) => Promise<McpProviderResult>` — a **new** member alongside `runMock` / `runFixture` / `runAnthropic`. `runMcp` has its **own** result type; it is not forced to share `runAnthropic`'s. This is `+1` member and `+1` type-only import of `McpProviderResult` into `providerRoutingCore.ts` — `providerRoutingCore.ts` already does a type-only import of `ProviderResult` from `anthropicClassifierCore.ts`, so adding a type-only import of `McpProviderResult` from `mcpAdapterCore.ts` is the identical, established pattern and keeps the routing core zod-free.
- The routing core's `mcp` branch awaits `deps.runMcp(request)`, then translates the result to `ClassifyMoveOutcome` via the small reason-map from resolution 1.

This is the exact shape MCP-017 used for `runAnthropic`; MCP-018 adds the parallel `runMcp`. No existing dep member changes.

## 5. File changes

### New files

- **`supabase/functions/_shared/semanticReferee/mcpAdapterCore.ts`** (~130 lines) — the **zod-free, Jest-importable** pure core. The exact analog of `anthropicClassifierCore.ts`. Exports:
  - `MCP_CLASSIFY_TOOL_NAME = 'classify_semantic_move'` — the MCP tool name the server exposes (MCP-009 §"MCP-server placement" names exactly this tool).
  - `MCP_REQUEST_TIMEOUT_MS` — a bounded request timeout (recommend `15_000`; passed to `fetch` via `AbortSignal.timeout`). MCP-017's Anthropic provider has no explicit timeout; an operator-hosted server is a less-known quantity, so a timeout is prudent — a timed-out `fetch` throws and maps to `network_error` (see resolution note below).
  - `buildMcpToolRequestBody(request: ClassifyMoveRequest): Record<string, unknown>` — assembles the MCP tool-invocation body: `{ tool: MCP_CLASSIFY_TOOL_NAME, input: { moveBodyRedacted, parentBodyRedacted?, roomContext, requestedClassifiers, contentHash, roomId, moveId?, parentId?, promptVersionHint? } }`. Pure — exported so tests assert its shape with no network call. Same `request` → byte-identical output (no `Date.now()`, no randomness). It carries the redacted bodies and the requested catalog ids and **nothing** that asks for a verdict.
  - `extractMcpPacket(responseJson: unknown): unknown | null` — pulls the packet object out of the MCP server's response envelope. The MCP convention is a result wrapper (`{ result: {...} }` / `{ output: {...} }` / a `content[]` array of typed blocks); `extractMcpPacket` accepts a small, documented set of envelope shapes — first `result`, then `output`, then a `content[].type === 'json'`/`'text'` block (JSON-parsing a text block via `parseJsonFromContent`) — and returns the inner object, or `null` on any miss. Never throws.
  - `parseJsonFromContent(text: unknown): unknown | null` — extract + `JSON.parse` the first `{...}` object; returns `null` on any failure. **Re-implemented identically to `anthropicClassifierCore.ts`'s version** (a ~12-line pure helper). Per acceptance criterion 5, MCP-018 must not import or modify `anthropicClassifierCore.ts`'s function-level exports beyond the type-only `ProviderResult`-family import; the cleanest path is a local copy of this tiny helper, with a comment pointing at the sibling. (Alternative: extract `parseJsonFromContent` into a shared zod-free `jsonExtract.ts` both cores import — but that *modifies* `anthropicClassifierCore.ts`'s import line, which the strict reading of criterion 5 discourages. The design recommends the local copy.)
  - `sanitizeMcpRawPayload(raw: unknown): Record<string, unknown>` — keeps only a small allow-list of non-sensitive envelope keys (`tool`, `stop_reason`/`status`, `usage` if present) — the analog of `sanitizeRawPayload`. Never returns the raw body, the token, headers, or the inner packet's free-text fields. Used for the (sanitized) log line only.
  - `McpUnavailableReason` and `McpProviderResult` — the boundary-internal types from §4 resolution 1 + 2.
  This file imports **only** `types.ts` (for `ClassifyMoveRequest` / `SemanticRefereePacket`) — zero `npm:` imports — so the `_helpers/semanticRefereeDeno.ts` Jest bridge can `require()` it.
- **`supabase/functions/_shared/semanticReferee/mcpAdapter.ts`** (~150 lines) — the Deno-only MCP-server-call orchestrator. The exact analog of `anthropicProvider.ts`. Exports `runMcpAdapter(request: ClassifyMoveRequest): Promise<McpProviderResult>`. It:
  1. Reads `SEMANTIC_REFEREE_MCP_URL` via `Deno.env.get()` → `{ kind: 'unavailable', reason: 'url_missing' }` if absent/empty.
  2. Reads `SEMANTIC_REFEREE_MCP_TOKEN` via `Deno.env.get()` → `{ kind: 'unavailable', reason: 'token_missing' }` if absent/empty.
  3. Builds the body with `buildMcpToolRequestBody` and `POST`s the MCP URL (headers `Content-Type: application/json`, `Authorization: Bearer <token>` — the standard MCP HTTP auth; the `Bearer ` prefix is assembled from a fragment so the repo secret-literal scan stays green, exactly as `redaction.ts` / `contentSafetyScan.ts` assemble their patterns), with `signal: AbortSignal.timeout(MCP_REQUEST_TIMEOUT_MS)`.
  4. Maps a thrown `fetch` (DNS/TLS/connection-reset/timeout) → `'network_error'`; a non-OK response → `'rate_limited'` for HTTP 429, `'api_error'` otherwise.
  5. `sanitizeMcpRawPayload`s the response; `extractMcpPacket` → `'parse_failure'` on any miss.
  6. Stamps the contract identity fields onto the parsed object — `packetVersion: PACKET_VERSION`, `provider: 'mcp'`, `authoritative: false`, `promptVersion: SEED_PROMPT_VERSION` (or `request.promptVersionHint`), `modelVersion`, `inputHash`, `contentHash`/`roomId`/optional ids — via a `stampPacketIdentity` helper that is the structural twin of `anthropicProvider.ts`'s. `provider` is hard-set to `'mcp'`; `authoritative` is hard-pinned to `false`. (`modelVersion`: the MCP server may report its own model id in the envelope; if absent, stamp a constant like `'operator-mcp-server'` so the schema's `modelVersion: z.string().min(1)` is satisfied. The implementer keeps a `DEFAULT_MCP_MODEL_VERSION` constant in `mcpAdapterCore.ts` for this.)
  7. Validates the stamped object through `SemanticRefereePacketSchema.safeParse()` **and** `scanPacketContent()` → `'validation_failed'` on either failure.
  8. Returns `{ kind: 'success', packet: Object.freeze(...) }` only on a clean pass.
  This file imports `schema.ts` (→ `npm:zod@4`) so it is **not Jest-importable** — covered by `semanticMcpSourceScan.test.ts` source scans + the operator-gated integration suite, exactly as `anthropicProvider.ts` is. It mirrors `anthropicProvider.ts` step-for-step (and **does not import it**).
- **`docs/deployment/mcp-018-mcp-adapter-runbook.md`** (~200 lines) — operator runbook (outline in §9).
- Test files — see §8 for the full list.

> **`AbortSignal.timeout` note.** It is supported in the Supabase Edge runtime (Deno) and in Node 18+, so the Jest-side core tests can exercise `buildMcpToolRequestBody` without it (the timeout is applied in `mcpAdapter.ts`, the non-Jest file). If the implementer prefers to match `anthropicProvider.ts` exactly (no timeout), that is acceptable — a hung `fetch` would then rely on the platform default and MCP-012's retry policy. The design **recommends the explicit timeout** for an operator-hosted, less-predictable endpoint; it is a one-line `signal:` addition and is the only intentional divergence from the `anthropicProvider.ts` mirror.

### MCP transport decision (recorded so the implementer does not re-derive it)

MCP-009 §"MCP-server placement" says the adapter "connects over HTTPS (MCP streamable-HTTP / SSE transport)". The Model Context Protocol's HTTP transport is, at the wire level, JSON request/response — a streaming variant exists but the `classify_semantic_move` tool returns a single bounded structured result, not a token stream. **The design specifies the adapter does exactly one `POST` with a JSON body and reads exactly one JSON response body** — the same call shape `anthropicProvider.ts` uses against the Anthropic Messages API. This:
- keeps `mcpAdapter.ts` a faithful mirror of `anthropicProvider.ts` (one `fetch`, one `.json()`),
- keeps it Jest-source-scannable with the same harness,
- imposes no SSE-parsing dependency,
- and is sufficient for a single-result classification tool.

If the operator's MCP server *only* speaks SSE, that is a server-side configuration concern; the operator can put a thin JSON-result endpoint in front, or a future card can add an SSE code path. The v1 adapter is JSON-over-HTTPS. `extractMcpPacket` is deliberately tolerant of a few common MCP result-envelope shapes (`result` / `output` / a `content[]` block) so the operator's exact server framework is not over-constrained.

5. **Smoke test.** Hit `classifyMove` from a single test room; verify the returned packet carries `provider: 'mcp'` and `authoritative: false`. If the layer returns `{ enabled: false, reason: 'not_configured' }`, a URL or token is missing/empty or the URL is not `https`.
6. **Rollback.** One command, instant, no redeploy: `npx supabase secrets set SEMANTIC_REFEREE_PROVIDER=mock` (or set the DB row back to `'mock'`).
7. **Rotation.** How to rotate `SEMANTIC_REFEREE_MCP_TOKEN` cleanly (one `secrets set`; the function picks it up on next cold start).
8. **Disable the layer entirely.** `npx supabase secrets set SEMANTIC_REFEREE_ENABLED=false` (or DB `enabled = false`).
9. **Monitoring.** The MCP server is a new operational surface — what to watch (its uptime, its latency vs the `MCP_REQUEST_TIMEOUT_MS` budget). The expected per-room call rate given MCP-012's gating.
10. **Security note.** The model key (if the MCP server calls Anthropic/another model) lives on the MCP server, not in Supabase. The MCP server is operator-hosted and must be access-controlled; the `SEMANTIC_REFEREE_MCP_TOKEN` is the only thing the Edge Function presents.

## 11. Risks

- **The MCP server returns plausible but doctrine-violating output.** Mitigation: identical to MCP-017 — the `.strict()` `SemanticRefereePacketSchema` rejects every smuggleable field, `scanPacketContent` catches a verdict / person / secret token inside a contract-valid string, either failure → `validation_failed` → deterministic fallback, and `providers.ts`'s `validateOrFallback` is the registry-level third wall. A misbehaving MCP server cannot reach the user. The adapter additionally hard-stamps `provider: 'mcp'` and `authoritative: false` — the server cannot override those.
- **The token is sent to a plaintext endpoint.** Mitigation: the adapter rejects a non-`https` `SEMANTIC_REFEREE_MCP_URL` as `url_missing` before any `fetch` — a source-shape test asserts the `https` guard. The token only ever travels over TLS.
- **An unknown MCP server framework returns an unrecognised envelope shape.** Mitigation: `extractMcpPacket` accepts a small documented set of envelope shapes (`result` / `output` / `content[]`); an unrecognised shape returns `null` → `parse_failure` → deterministic fallback (graceful, never a throw). The runbook documents the expected `classify_semantic_move` tool-result contract so the operator configures the server to match.
- **An operator-hosted server is slower / flakier than a managed API.** Mitigation: `MCP_REQUEST_TIMEOUT_MS` bounds the wait; a timeout maps to `network_error` → deterministic fallback; MCP-012's retry policy and trigger gating cap the call rate. The runbook tells the operator to watch latency against the budget.
- **`mcpAdapter.ts` drifts from `anthropicProvider.ts`.** Mitigation: the two are deliberately parallel; a reviewer diffs them. The only intentional divergences are documented (the `AbortSignal.timeout`, the two-input URL+token read vs one key, the `mcp` provider stamp, the MCP envelope extraction). A future Anthropic-side fix should prompt the same fix here.
- **The `mcp` provider becomes admin-selectable by accident.** Mitigation: MCP-018 does **not** touch `adminSemanticConfigSchemas.ts` or `AdminSemanticRefereeTab` — the write-path `z.enum` stays three-mode, the UI control stays disabled. The existing `adminSemanticConfigSchemas.test.ts` "rejects `providerMode: 'mcp'` on the write path" assertion keeps passing untouched and is the guard. The design explicitly forbids the implementer from enabling the slot in the Admin UI.
- **`providerRouting.ts` becoming further zod-coupled.** It is *already* zod-coupled (it imports `anthropicProvider.ts`); importing the also-zod-coupled `mcpAdapter.ts` changes nothing. The routing *logic* stays under real test via `providerRoutingCore.ts` + injected spy deps — the MCP-017 arrangement, unchanged.
- **The inverted `mcpAdapter.ts`-absence assertion looks like a removed guard.** Known and resolved — §5 documents the single inverted line in `semanticNoLiveCallInRouter.test.ts` and the rewritten "mcp stub" test in `semanticProviderRegistry.test.ts`. A reviewer should expect both; they are the literal point of the card.
- **The MCP secrets deployed to the wrong Supabase project.** Mitigation: `secrets set` / `functions deploy` operate on the `--linked` project; the runbook calls this out. Operator discipline; a test cannot enforce it.

## 12. Doctrine self-check

- **cdiscourse-doctrine §1 — no truth labels:** the MCP-tool request asks only for the bounded structural binaries of catalog-v0; no verdict / truth field exists on the packet; `scanPacketContent` rejects any verdict token in a `reasonCode` or span; the MCP server is contractually required to honor the structural catalog. ✓
- **cdiscourse-doctrine §1 — score never blocks posting:** MCP-018 changes no ledger field; the packet has no `block` field; the `.strict()` schema rejects a smuggled one; `frictionSuggestion` stays advisory by contract. ✓
- **cdiscourse-doctrine §3 — popularity is not evidence:** the request carries no popularity-asking field; the `uses_popularity_as_evidence` classifier *flags* popularity-as-evidence as a structural property — it never grants standing. ✓
- **cdiscourse-doctrine §4 / §7 — AI never runs on the client:** the live `fetch` to the operator-hosted MCP server lives only in `_shared/semanticReferee/mcpAdapter.ts`, reached by the Edge Function. The MCP server is operator-hosted and never part of the Expo app. `src/lib/edgeFunctions.ts` calls the Edge Function; the function calls the MCP server; no client code ever does. A source-scan test enforces it. ✓
- **cdiscourse-doctrine §6 — secrets are server-only:** `SEMANTIC_REFEREE_MCP_URL` + `SEMANTIC_REFEREE_MCP_TOKEN` are read only via `Deno.env.get()` inside `mcpAdapter.ts`; they appear in no `src/` / `app/` / `.env*` / git file; the runbook and this design name them only as documented variables. The token is sent only over TLS (the `https`-only URL guard). ✓
- **cdiscourse-doctrine §5 — the rules engine is sacred:** `src/lib/constitution/engine.ts` is untouched; MCP-018 adds no import to it. ✓
- **supabase-edge-contract — disabled-by-default + fail-safe + no service-role:** `SEMANTIC_REFEREE_PROVIDER` stays `'mock'` by default; the disabled-by-default check runs before any provider/URL/token; every error path returns `{ enabled: false, reason }` HTTP 200; the function builds only a caller-scoped (RLS) client and performs no privileged write — no `SUPABASE_SERVICE_ROLE_KEY` anywhere; the boundary never throws to the client; logs carry only function name / caller id / `roomId` / decision / provider / duration / error class. ✓
- **supabase-edge-contract — migration discipline:** MCP-018 writes no migration, adds no table, changes no RLS. (`semantic_referee_runtime_config`'s `provider_mode` CHECK already allows `'mcp'` — ADMIN-AI-001 — so no migration is needed even for the future admin-selectable follow-up.) ✓
- **test-discipline:** every public function of the new zod-free `mcpAdapterCore.ts` has happy-path + failure-case tests; the not-Jest-importable `mcpAdapter.ts` is covered by source-scan + source-shape + operator-gated integration suites (the `anthropicProvider.ts` standard); ban-list + reason-map guards are included; test count rises and `docs/core/current-status.md` is updated. ✓
- **roadmap §7 — operator-gated:** the deploy, the secret set, and the provider flip are operator actions; the card produces code only; the first live MCP-server call is the operator's. ✓

## Appendix A — open questions / mismatches resolved into the design

The #224 issue body is prescriptive; there are **few open questions**, and each is resolved in-design rather than left for the implementer:

- **OQ-1 (resolved — §4 resolution 1):** the `mcp` adapter has a URL *and* a token, so MCP-017's `key_missing` reason does not fit. Resolved by a boundary-internal `McpUnavailableReason` with `url_missing` / `token_missing`, **mapped onto the existing `'not_configured'`** `ClassifyMoveDisabledReason` at the routing step — so **MCP-018 widens no type and touches no Node/Deno type file**, a strictly smaller blast radius than MCP-017. The implementer follows the map-to-`not_configured` path unless the reviewer asks for a `ClassifyMoveDisabledReason` widening.
- **OQ-2 (resolved — §4 resolution 2):** the `mcp` result is not assignable to MCP-017's `ProviderResult` (different reason union). Resolved by a parallel `McpProviderResult` type and a **new** `runMcp` member on `SemanticRefereeProviderDeps` with its own result type — alongside, not replacing, `runAnthropic`. No existing dep member changes.
- **OQ-3 (resolved — §5 / §6 "MCP transport decision"):** MCP-009 says "streamable-HTTP / SSE". Resolved: the v1 adapter does one JSON `POST` and reads one JSON response (mirroring `anthropicProvider.ts`); `extractMcpPacket` tolerates a small set of MCP result-envelope shapes. SSE is a server-side concern or a future card.
- **OQ-4 (resolved — §5):** `parseJsonFromContent` is a ~12-line pure helper that already exists in `anthropicClassifierCore.ts`, but AC5 forbids modifying that file. Resolved by a documented **local copy** in `mcpAdapterCore.ts` (with a parity test feeding both copies the same inputs), in preference to a shared-module extraction that would edit `anthropicClassifierCore.ts`'s import line.
- **OQ-5 (recorded — §5):** MCP-018 adds an explicit `AbortSignal.timeout` that `anthropicProvider.ts` does not have — the single intentional divergence from the mirror, justified because the MCP endpoint is operator-hosted and less predictable. A timed-out `fetch` maps to `network_error`. The implementer may omit it to match `anthropicProvider.ts` exactly; the design recommends keeping it.
- **No issue-vs-code mismatch blocks the card.** One issue-body imprecision worth flagging to the implementer: #224's AC4 says "the `?? 'mock'` code fallback in `providers.ts` is unchanged" — post-ADMIN-AI-001 the `?? 'mock'` fallback physically lives in `providerRoutingCore.ts` (`providers.ts` re-exports the routing core and adds the DB-resolution layer). The intent is identical — *the mock default must stay byte-identical* — and §3 / §5 of this design point at `providerRoutingCore.ts` as the actual location. The implementer keeps that `??  'mock'` line in `providerRoutingCore.ts` untouched.
