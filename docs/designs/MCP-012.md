## Goal (one paragraph)

MCP-012 builds the **semantic call router** as pure TypeScript: a deterministic, fixture-only translator from a UI / game event into the decision *"should a semantic-referee classifier call be made, and if so, of what?"* — and **never** *"should this post be allowed?"*. It realizes MCP-004's design (trigger gates, batching ≤5/call, in-memory cache store, token budget, retry policy) as five new files under `src/features/semanticReferee/`, consuming — not re-creating — MCP-011's frozen contract. The card is shaped by doctrine constraints that are blockers if violated: (1) **the router is advisory — it never blocks an ordinary post**: every "no call" path (a forbidden event, a room in `off` mode, an over-budget payload, an exhausted retry) degrades silently to deterministic layer 1; `submit-argument` never consults the router (cdiscourse-doctrine §1, §4; MCP-004 §"doctrine self-check"). (2) **No provider call lives here** — the router is pure decision logic; the only thing that ever *fires* a call is MCP-009's Edge Function, built later by MCP-016. This card adds no network import, no Supabase import, no React import, no `.env` read, no `Deno` reference, no Edge Function (cdiscourse-doctrine §4, §7; roadmap §10). (3) **Mock provider is the default** — the card's tests inject MCP-011's `mockFixtureProvider` through a provider-spy seam; live calls remain operator-gated and out of this card entirely (roadmap §7). (4) **Forbidden events produce zero provider calls** — `keystroke`, `hover`, `timeline_selection`, `observer_browsing`, `scroll`, `focus`, `blur`, `gallery_browsing` are an exhaustive enum that `evaluateTrigger` refuses *unconditionally, before any flag/mode check*; the `semanticNoCallOnDraftEdit` spy test proves it. (5) **Cost control is a doctrine concern** — trigger gating, caching, batching, and the token budget together make an uncapped layer structurally impossible (MCP-004 §"token budget"). This card does **not** redesign MCP-011's contract, MCP-009's boundary, or MCP-002's seed bank — it gates the call path between MCP-004's plan and MCP-009's boundary.

## API / interface contracts

### 1. `evaluateTrigger` — the allowed/forbidden taxonomy

`evaluateTrigger(input)` is the single chokepoint. It returns a `TriggerDecision`, never a post decision.

**Forbidden events — refused unconditionally, before any flag/mode/role check:**

| `SemanticForbiddenEvent` | Decision | `reasonCode` |
|---|---|---|
| `keystroke` | `{ allowed: false }` | `trigger_forbidden_event` |
| `hover` | `{ allowed: false }` | `trigger_forbidden_event` |
| `timeline_selection` | `{ allowed: false }` | `trigger_forbidden_event` |
| `observer_browsing` | `{ allowed: false }` | `trigger_forbidden_event` |
| `scroll` | `{ allowed: false }` | `trigger_forbidden_event` |
| `focus` | `{ allowed: false }` | `trigger_forbidden_event` |
| `blur` | `{ allowed: false }` | `trigger_forbidden_event` |
| `gallery_browsing` | `{ allowed: false }` | `trigger_forbidden_event` |

When `event.kind === 'forbidden'`, `evaluateTrigger` returns `{ allowed: false, reasonCode: 'trigger_forbidden_event' }` as the **very first statement** — it does not read `featureLayerEnabled`, `semanticClassificationMode`, or `actorRole`. This ordering is load-bearing: it guarantees a forbidden event cannot reach a code path that fires a call even if the flag and mode are permissive.

**Allowed moments — `allowed: true` only when (flag on) AND (mode permits) AND (per-moment precondition holds):**

The evaluation order for `event.kind === 'trigger'` is fixed so the most general refusals are checked first:

1. `featureLayerEnabled === false` → `{ allowed: false, reasonCode: 'trigger_layer_disabled' }`.
2. `semanticClassificationMode` absent or `=== 'off'` → `{ allowed: false, reasonCode: 'trigger_room_mode_off' }` (fail-closed on absent mode).
3. `actorRole === 'observer' || actorRole === 'moderator'` → `{ allowed: false, reasonCode: 'trigger_role_not_participant' }`.
4. Per-moment precondition (table below).
5. If all pass → `{ allowed: true, moment, reasonCode: 'trigger_<moment>_allowed' }`.

| `moment` | Mode requirement | Per-moment precondition (after steps 1–3) | Refusal `reasonCode` if precondition fails |
|---|---|---|---|
| `post_submit` | `metadata_only` or `metadata_and_chip` | none beyond steps 1–3 | — |
| `pre_send_review` | `metadata_and_chip` **only** | `preSendReviewOptIn === true` | `trigger_pre_send_not_opted_in`; if mode ≠ `metadata_and_chip` → `trigger_pre_send_mode_not_strict` |
| `evidence_inspection` | `metadata_only` or `metadata_and_chip` | none beyond steps 1–3 | — |
| `branch_routing` | `metadata_only` or `metadata_and_chip` | `branchRouteAmbiguous === true` | `trigger_branch_route_not_ambiguous` |
| `synthesis_readiness` | `metadata_only` or `metadata_and_chip` | `clusterLifecycleEventCount >= SYNTHESIS_TRIGGER_MIN_EVENTS` (`6`) | `trigger_synthesis_below_threshold` |
| `referee_feedback` | `metadata_only` or `metadata_and_chip` | none — but it is a *derived* trigger: `allowed: true` means "read existing metadata", never "fire a call" | — |

`SYNTHESIS_TRIGGER_MIN_EVENTS = 6` is exported from `triggerGates.ts` as a `const` (operator-tunable, MCP-004 §"operator decisions" #3). For `pre_send_review`, the mode check is evaluated **before** the opt-in check, so a non-strict room gets `trigger_pre_send_mode_not_strict` and a strict-but-not-opted-in room gets `trigger_pre_send_not_opted_in` — two distinct, testable codes.

`evaluateTrigger` is **total**: every `(event × mode × flag × role × precondition)` combination yields exactly one `TriggerDecision`. No throw, no `undefined`.

### 2. `planClassifierBatches` — batching algorithm

**Batch groups** (MCP-004 §"batching"; partition the 23-id catalog in `ALL_SEMANTIC_CLASSIFIER_IDS`):

| Group | Classifier ids | Count |
|---|---|---|
| **A — Parent continuity** | `responds_to_parent`, `quote_anchors_parent`, `answers_clarification`, `introduces_new_issue` | 4 |
| **B — Branch routing & mode fit** | `suggests_side_branch`, `suggests_diagonal_tangent`, `fits_selected_debate_mode`, `contains_unplayable_insult_only`, `contains_playable_hot_take` | 5 |
| **C — Evidence pressure** | `asks_for_evidence`, `provides_evidence`, `evidence_supports_claim`, `creates_source_chain_gap`, `uses_popularity_as_evidence` | 5 |
| **D — Evidence provenance** | `cites_retraction`, `uses_satire_as_evidence`, `is_satire_or_parody`, `narrows_claim`, `concedes_narrow_point` | 5 |
| **E — Constructive movement & friction** | `requests_clarification`, `ready_for_synthesis`, `needs_pre_send_pause`, `shifts_to_person_or_intent` | 4 |

`SEMANTIC_BATCH_GROUPS` is a frozen array of the five `SemanticBatchGroup` objects. The union of the five `classifierIds` lists equals `ALL_SEMANTIC_CLASSIFIER_IDS` exactly (23 ids, no overlap, no omission) — a test asserts this partition property against MCP-011's catalog so the two cannot drift.

**Algorithm — `planClassifierBatches(requested)`:**

```
input:  requested: readonly SemanticClassifierId[]   (the trigger's chosen classifier set)
output: ClassifierBatch[]                            (each inner array ≤ BATCH_CAP, order-normalized)

1. Normalize the request: dedupe (Set), then sort lexically. Call this `normalized`.
   - If `normalized` is empty, return [] (no batch, no call).
2. For each batch group A..E in fixed order:
     - intersection = normalized ids that belong to this group (preserve sorted order).
     - if intersection is non-empty, emit it as one ClassifierBatch.
   - Because each group has ≤ 5 ids, each emitted batch is ≤ BATCH_CAP by construction.
3. Ignore any requested id that is not in catalog v0 — it cannot be a member of any
   group, so it is silently dropped (the upstream caller validates ids; the batcher
   is defensive). A test covers the unknown-id drop.
4. Return the list of emitted batches. One group → one batch; a request spanning
   three groups → three batches. A request never produces a batch > 5.
```

Properties the implementer must hold and the test suite asserts:

- **Never larger than 5.** No emitted batch exceeds `BATCH_CAP`. Guaranteed because each group is ≤ 5 and a batch is a *subset* of one group.
- **Order-normalized.** Ids within a batch are lexically sorted; `['responds_to_parent','quote_anchors_parent']` and `['quote_anchors_parent','responds_to_parent']` both plan to `[['quote_anchors_parent','responds_to_parent']]` (sorted). This is the cache-correctness guarantee — the same classifier set always plans identically, so the same `hashClassifierSet` result and the same cache key.
- **One batch per group touched.** A request touching only group C plans exactly one batch. A request is never split into more batches than the number of groups it spans.
- **`planClassifierBatches` does NOT call a provider.** It returns a plan. Whether each batch is then a cache hit or a real call is decided downstream (cache → budget → MCP-016 boundary).
- **Per-trigger cap.** A single trigger's classifier set is expected to span at most two groups (≤ 2 batches, ≤ 10 classifiers — MCP-004 §"batching"). The batcher itself does not enforce a 2-batch ceiling (a 23-id request legitimately plans 5 batches); the *caller* picks a per-moment set within the cap. A test documents that a curated per-moment set produces ≤ 2 batches and a comment in `classifierBatching.ts` records the expectation.

### 3. `SemanticPacketCache` — in-memory LRU store

Store-only. The class API:

```ts
export class SemanticPacketCache {
  constructor(options?: SemanticPacketCacheOptions);   // capacity default 256
  get(key: SemanticCacheKey): SemanticRefereePacket | undefined;  // marks recently-used on hit
  set(key: SemanticCacheKey, packet: SemanticRefereePacket): void; // evicts LRU at capacity
  has(key: SemanticCacheKey): boolean;                  // does NOT mark recently-used
  delete(key: SemanticCacheKey): boolean;
  clear(): void;
  get size(): number;
  get capacity(): number;
}
```

**Eviction policy — LRU, recency-of-access:**

- Internally a `Map<string, SemanticCacheEntry>` keyed by `serializeSemanticCacheKey(key)` (imported from MCP-011 — never re-derived). A `Map` preserves insertion order; the store exploits that for O(1) LRU.
- `get(key)` on a hit: delete the entry and re-`set` it into the `Map` so it moves to the most-recent (last) position; return the packet. `has` does **not** re-order — it is a pure membership test.
- `set(key, packet)`: if the serialized key already exists, delete it first (so the re-insert moves it to most-recent and the new packet overwrites). Then insert. If `map.size > capacity` after insert, evict the least-recently-used entry — the **first** key in `Map` iteration order — until `size <= capacity`.
- `DEFAULT_CACHE_CAPACITY = 256` (MCP-004 §"caching": 256 entries per warm instance — a sane default, not an operator decision). `capacity` is settable via `SemanticPacketCacheOptions` for tests; a capacity `< 1` is clamped to `1`.
- **LRU is a memory bound, NOT a correctness mechanism and NOT invalidation.** An evicted entry simply causes a future re-classification (a cost, not a wrong answer). There is **no time-based TTL**: `lruTick` orders accesses; it is never compared against a clock. Invalidation happens only by `promptVersion` bump changing the key (MCP-004 §"caching") — the store has no `expire` method by design.
- The cache stores **successful packets only**. A deterministic fallback packet (an exhausted-retry or over-budget result) is **not** cached — caching it would pin a degraded result past a provider recovery (MCP-004 §"retry/fallback"). The store does not enforce this (it stores whatever `set` receives); the *caller* (MCP-016, later) must only `set` on a verified-successful classification. A comment in `semanticCache.ts` records this contract.

The cache holds MCP-011's `SemanticRefereePacket` verbatim — it adds, removes, renames no field, and runs no validation (validation already happened at MCP-011's `parseSemanticPacket` before a packet is ever cached).

### 4. `tokenBudget` — budget descriptors

```ts
export function estimatePacketTokens(payload: TokenBudgetPayload): number {
  // Conservative, tokenizer-free. Over-estimates so it fails SAFE.
  //   inputChars  = moveBodyRedacted.length
  //               + (parentBodyRedacted?.length ?? 0)
  //               + (extraContextChars ?? 0)
  //               + requestedClassifiers.length * PER_CLASSIFIER_ID_CHARS  (e.g. 32)
  //   inputTokens = Math.ceil(inputChars / CHARS_PER_TOKEN)   // CHARS_PER_TOKEN = 3.5
  //   return inputTokens + OUTPUT_TOKEN_RESERVE               // OUTPUT_TOKEN_RESERVE = 450
}

export function isWithinBudget(payload: TokenBudgetPayload): TokenBudgetVerdict {
  const estimated = estimatePacketTokens(payload);
  return { ok: estimated <= SEMANTIC_PACKET_TOKEN_BUDGET, estimated, ceiling: SEMANTIC_PACKET_TOKEN_BUDGET };
}
```

- `SEMANTIC_PACKET_TOKEN_BUDGET = 1500` — operator-confirmable (MCP-004 §"operator decisions" #1). The *mechanism* (refuse over-budget, fall back to layer 1, never block the post) does not change with the number.
- `estimatePacketTokens` over-counts (3.5 chars/token is pessimistic vs ~4 real English) so `isWithinBudget` refuses *slightly before* the true ceiling, never after — a doctrine-aligned fail-safe.
- Over-budget → the router does not plan a call for that payload; the consumer falls back to layer 1. **Refusing an over-budget packet never blocks a post** — `post_submit` already happened; `pre_send_review` is advisory. The implementer must not truncate the body to fit (truncation classifies a different text — a false-positive risk; MCP-004 §"token budget").
- `PER_CLASSIFIER_ID_CHARS` (a small `const`, e.g. 32) accounts for each classifier id's prompt footprint. Tunable; documented as a heuristic, not an operator decision.

### 5. `retryPolicy` — retry descriptors

```ts
export const SEMANTIC_RETRY_POLICY: SemanticRetryPolicy = {
  maxAttempts: 2,             // one initial attempt + at most one retry
  backoffMs: [250],          // one short backoff before the single retry
  attemptTimeoutMs: 8000,    // per-attempt wall clock (MCP-004 §"retry" recommends 8 s)
};

export const RETRYABLE_ERROR_CLASSES: readonly SemanticErrorClass[] =
  ['network_timeout', 'provider_5xx', 'provider_rate_limited'];

export const TERMINAL_ERROR_CLASSES: readonly SemanticErrorClass[] =
  ['key_missing', 'validation_failed', 'parse_failure', 'disabled', 'over_budget'];

export function shouldRetry(errorClass: SemanticErrorClass): boolean {
  return RETRYABLE_ERROR_CLASSES.includes(errorClass);
}
```

- `maxAttempts` is the literal `2` — bounded and total; the retry loop (executed later in MCP-016's Edge Function) can never spin.
- `shouldRetry` returns `true` only for transient classes; terminal classes go straight to the deterministic fallback. `RETRYABLE_ERROR_CLASSES` and `TERMINAL_ERROR_CLASSES` together cover every `SemanticErrorClass` member with no overlap — a test asserts the partition.
- This file is a pure *policy descriptor*. It performs no retry, sets no timer, makes no call. MCP-016 reads it.

## Edge cases

The implementer must handle each:

- **`event.kind === 'forbidden'`** — `{ allowed: false, reasonCode: 'trigger_forbidden_event' }` returned before any other check, for all eight events. The acceptance-criterion case.
- **`featureLayerEnabled === false`** — `{ allowed: false, reasonCode: 'trigger_layer_disabled' }` for every trigger moment.
- **`semanticClassificationMode` absent / `undefined`** — treated as `'off'` → `trigger_room_mode_off`. Fail-closed.
- **`semanticClassificationMode === 'off'`** — `trigger_room_mode_off` for every moment.
- **`actorRole === 'observer'`** — `trigger_role_not_participant` for every moment. Belt-and-suspenders with the `observer_browsing` forbidden event.
- **`actorRole === 'moderator'`** — `trigger_role_not_participant`. Moderators are not players in the PvP loop.
- **`pre_send_review` in a non-`metadata_and_chip` room** — `trigger_pre_send_mode_not_strict` (mode check before opt-in check).
- **`pre_send_review` in `metadata_and_chip` but `preSendReviewOptIn === false`** — `trigger_pre_send_not_opted_in`.
- **`branch_routing` with `branchRouteAmbiguous` false / absent** — `trigger_branch_route_not_ambiguous`. Layer 1 is confident; layer 2 adds nothing.
- **`synthesis_readiness` with `clusterLifecycleEventCount` below `6` / absent** — `trigger_synthesis_below_threshold` (absent treated as `0`).
- **`referee_feedback`** — `allowed: true` (if flag + mode + role pass), but it is a *derived* trigger: the consuming code reads cache/ledger and shows no banner if nothing is cached. It must never originate a provider call. A test asserts `evaluateTrigger` returns it allowed and a comment documents the derived-trigger contract.
- **`planClassifierBatches([])`** — empty input → `[]` (no batch, no call).
- **`planClassifierBatches` with a duplicate id** — deduped before batching; `['responds_to_parent','responds_to_parent']` → one batch with one id.
- **`planClassifierBatches` with an unknown / non-catalog id** — silently dropped (it belongs to no group). If *all* requested ids are unknown → `[]`.
- **`planClassifierBatches` with the full 23-id catalog** — plans exactly five batches A–E, each ≤ 5. (Not a normal per-trigger request, but the function is total.)
- **Cache `get` for a key never `set`** — returns `undefined`. `has` returns `false`.
- **Cache `set` overwriting an existing key** — replaces the packet and moves the entry to most-recently-used; `size` does not increase.
- **Cache eviction at capacity** — inserting the `(capacity + 1)`-th distinct key evicts the least-recently-used entry; `size` stays `== capacity`. A `get` on the evicted key returns `undefined`.
- **Cache `get` updates recency** — after `get`, that key is no longer the eviction target; the next-oldest is.
- **Cache `has` does NOT update recency** — a `has` followed by a `set` at capacity may still evict the `has`-probed key. Documented and tested.
- **No time-based expiry** — advancing a fake clock (Jest fake timers) does not invalidate any cache entry. A test asserts this.
- **`promptVersion` bump** — `buildSemanticCacheKey` produces a different `SemanticCacheKey` (different `promptVersion` field) → a different serialized key → a cache miss → a legitimate fresh call. The old entry is unreachable and LRU-evicted eventually. This is the *intended* invalidation path.
- **`classifierSet` ordering** — `planClassifierBatches` sorts within each batch; `hashClassifierSet` (MCP-011) also sorts + dedupes. `['a','b']` and `['b','a']` key identically. A test asserts the same cache key for reordered classifier requests.
- **`estimatePacketTokens` with an empty `moveBodyRedacted`** — returns `OUTPUT_TOKEN_RESERVE` plus the classifier-id overhead (a small number, well within budget). An empty move is not classified upstream anyway (`submit-argument` rejects empty bodies deterministically — unrelated to this card).
- **`isWithinBudget` exactly at the ceiling** — `estimated === SEMANTIC_PACKET_TOKEN_BUDGET` → `ok: true` (inclusive boundary).
- **`isWithinBudget` for an over-length payload** — `ok: false`; the caller skips the call and falls back to layer 1; **the post is unaffected**.
- **`shouldRetry` for every `SemanticErrorClass`** — `true` for the three transient classes, `false` for the five terminal classes; total over the union.
- **Doctrine edge case — "could a gate misfire block a post?"** — no: `evaluateTrigger` returns a decision about *whether to classify*, never *whether to post*. `submit-argument` does not import or call `evaluateTrigger`. A gate that refuses everything means "no semantic metadata this session" — the game runs on layer 1.
- **Doctrine edge case — "could the cache serve a stale verdict?"** — a cached packet is keyed by `contentHash` + `promptVersion` (a posted body is immutable; a prompt change bumps the version). A hit is always the correct packet for that body under that prompt. And the packet is advisory regardless.
- **Doctrine edge case — "could heat / popularity sneak into a trigger condition?"** — `TriggerEvaluationInput` has **no** engagement / like-count / view-count / heat field. Triggers fire only on structural completed actions. A test asserts the input type carries no engagement field (a structural / type-level check).

## Risks

- **`semanticCache.ts` re-deriving the key.** The single biggest mistake the implementer can make is writing a local `serializeSemanticCacheKey` or `fnv1a32` in `semanticCache.ts` instead of importing MCP-011's. The roadmap §6.2 reconciliation forbids it; `semanticRouterForbiddenImports.test.ts` asserts the import is present and the re-derivation is absent.
- **`classifierSet` ordering bug.** If a batch's classifier ids are not order-normalized, two requests for the same set in different orders produce different `hashClassifierSet` results and miss the cache, doubling cost. `planClassifierBatches` sorts within each batch and `hashClassifierSet` (MCP-011) also sorts — the normalization is belt-and-suspenders and load-bearing; `semanticBatching.test.ts` and `semanticCache.test.ts` both assert it.
- **LRU recency semantics.** `get` must refresh recency, `has` must not. A `Map`-reinsert on `get` is the idiomatic O(1) implementation; getting this subtly wrong (e.g. refreshing on `has`, or not refreshing on `get`) breaks the eviction order. The test suite pins both behaviors explicitly.
- **Async creep.** MCP-004's retry section describes a retry *loop* with backoff and timeout — that is the *executor*, owned by MCP-016's Edge Function. If the implementer adds `async` / `setTimeout` / a real retry loop to `retryPolicy.ts`, the card has overstepped its boundary. `retryPolicy.ts` is a *descriptor* only; `semanticRouterForbiddenImports.test.ts` asserts no `async` in any of the five files.
- **Token estimator drift.** The char-based heuristic is deliberately conservative; if a future model tokenizes very differently the estimate could be off. Mitigation: the estimator over-counts (fails safe), the ceiling has headroom, and MCP-009's 8,000-char schema cap is the hard upstream bound. Adding a real tokenizer dependency now is a dependency-policy violation — do not.
- **Verdict-token false positives in the ban-list scan.** `branch_routing` / `trigger_branch_route_not_ambiguous` contain the segment `route`, not a verdict token; `pre_send_review` contains `send`. The ban-list test must match whole `snake_case` segments (split on `_`, exact-equal a banned token), exactly as MCP-011's validator `tokenSegments` does — a substring scan would false-positive. The test file documents this.
- **GAME-003 not yet landed.** `SemanticClassificationMode` is GAME-003's. Because this card reproduces the enum locally and fails closed on an absent mode, it can land before GAME-003; if GAME-003 later renames the field, this card's `TriggerEvaluationInput` type follows it (a one-type edit). The gate logic is stable.
- **Existing tests.** This card adds five pure-TS files and one additive `index.ts` edit; it modifies no existing production file and no existing Edge Function. No existing CDiscourse test should need updating. MCP-011's `semanticRefereeNoLiveAi.test.ts` scans only MCP-011's five files (a hard-coded `SR_FILES` list) — it will not pick up MCP-012's new files; MCP-012's own `semanticRouterForbiddenImports.test.ts` covers them. (Optionally the implementer may extend `SR_FILES` to include the new files; not required — a separate test file is the cleaner, lower-merge-risk choice and is what this design specifies.)
