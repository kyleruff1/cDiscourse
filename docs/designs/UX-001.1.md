## §1 — Goal & scope

UX-001.1 ships Phase 1 of the seven-card UX-001 epic: **brand and app
shell correction**. Phase 1's deliverable is a polished, breakpoint-aware
application shell (header + global surface tokens) that subsequent
cards (UX-001.2 timeline-first viewport, UX-001.3 composer, UX-001.4
Act/Inspect/Go, UX-001.5 metadata, UX-001.6 cross-device QA, UX-001.7
visual design system) consume verbatim.

Phase 1 does NOT touch the timeline (UX-001.2), composer (UX-001.3),
Act/Inspect/Go menus (UX-001.4), metadata layer (UX-001.5), or the
full visual design system consolidation (UX-001.7). Phase 1 owns:

1. Breakpoint strategy (extending the existing 2-state to 3-state).
2. Header structure + density per breakpoint.
3. Logo treatment per breakpoint.
4. Header height per breakpoint (tightened from 152 to 120 wide; new
   intermediate 96 for tablet/desktop).
5. Typography baseline for the shell only (minimum needed for header
   text — wordmark fallback + tagline). UX-001.7 will extend.
6. Phase 1 framing section that UX-001.2 reads as primary input.

Doctrine constraints (see §15):
- `cdiscourse-doctrine` — no verdict copy; no popularity tokens; no
  service-role; no AI calls in production paths; no internal codes.
- `expo-rn-patterns` — RN primitives only; no Bootstrap; no new dep
  (existing serif italic stack remains).
- `accessibility-targets` — 44×44 hit target; WCAG AA contrast; reduce
  motion N/A (no animations added).
- `timeline-grammar` — shell is global chrome, not timeline grammar. No
  timeline tokens leak into header.

## §3 — Problem statement

The shipped header reads as "a logo in a bar" rather than a
brand-forward app chrome. Specific observed problems:

- **Phone-tablet-desktop collapse to two states.** A 768dp portrait
  iPad and a 1440dp desktop monitor get the same 152px header and 110px
  logo, despite having very different vertical-real-estate budgets and
  visual-density needs.
- **Wide header burns viewport.** 152px on a 1024dp-tall iPad portrait
  is 14.8% of vertical real estate. On laptops with browser chrome
  this approaches 20% — the timeline gets shoved below the fold.
- **No documented density contract per breakpoint.** The right slot
  has only a preferences gear today; UX-001.4 will need to add Act /
  Inspect / Go triggers and has no contract to know how much room is
  available.
- **No typography baseline.** Inline `fontSize: 18` / `fontSize: 28` /
  `fontSize: 14` literals are scattered across `AppHeader.tsx` (wordmark
  fallback) and `AppHeaderTagline.tsx`. Subsequent cards (UX-001.2 -
  UX-001.5) will inline more typography decisions ad-hoc until UX-001.7
  consolidates. Each card pays a small token-drift tax.

UX-001.1 resolves all four by:

1. Promoting the breakpoint hook to 3-state.
2. Tightening the wide header to 120px and adding an intermediate 96px
   for the tablet/desktop band.
3. Documenting right-slot capacity per breakpoint.
4. Shipping a minimum typography baseline (3 type tokens) that UX-001.7
   extends.

export const BRAND = {
  // ── EXISTING — preserved byte-identical ───────────────────────
  surface: { app: { bg: '#08060F' }, appElevated: { bg: '#13101D' } },
  text: { primary: '#F5EDE0', muted: '#B6AFA1', taglineFg: '#E6DCC8' },
  accent: { cream: '#F5EDE0', creamHairline: 'rgba(245, 237, 224, 0.18)' },
  headerHeightPx: 64 as const,         // narrow / phone (unchanged)
  logoHeightPx: 44 as const,           // narrow / phone (unchanged)
  logoHeightPxWide: 110 as const,      // legacy wide alias (unchanged)
  headerHeightPxWide: 152 as const,    // legacy wide alias (unchanged)
  headerWideBreakpointPx: 720 as const,// legacy wide breakpoint (unchanged)
  taglineText: 'Just get to the bottom of it' as const,

  // ── NEW — UX-001.1 Phase 1 additions ──────────────────────────

  /**
   * UX-001.1 — 3-state breakpoint contract.
   *
   * `phone`   :   0 - 599  dp  (compact mobile + small Android)
   * `tablet`  : 600 - 1279 dp  (tablet portrait + tablet landscape + small desktop)
   * `wide`    : 1280+      dp  (laptop + desktop + wide desktop)
   *
   * The "phone" upper bound is 599 (not 720 or 768) because:
   *   - Material Design 3 + iOS Human Interface Guidelines both treat
   *     ≥600dp as "tablet class" for typography + density.
   *   - A 5.4-inch iPhone in portrait is 390dp; a Galaxy Fold inner is
   *     584dp. 599dp covers all common phones.
   *   - Tablet portrait (iPad 768dp, iPad mini 744dp) enters the
   *     "tablet" band — the band exists specifically so a portrait
   *     tablet does not get a phone-sized header.
   *
   * The "wide" lower bound is 1280 (not 1024 or 1440) because:
   *   - 1280 is the de-facto laptop minimum (MacBook Air, most 13" laptops).
   *   - 1024 is too aggressive: iPad landscape would jump to "wide" and
   *     get a header tuned for a desktop, which is the wrong density.
   *   - 1440 is too conservative: a 13" MacBook would be stuck on the
   *     "tablet" band despite being a primary desktop class.
   *
   * The legacy `headerWideBreakpointPx: 720` is PRESERVED for back-compat
   * (it still drives the existing `isWide` boolean), but new code should
   * read these three new constants. See §5 (Q1 verdict) for the
   * back-compat strategy.
   */
  breakpoints: {
    phone:  { minPx: 0,    maxPx: 599  },
    tablet: { minPx: 600,  maxPx: 1279 },
    wide:   { minPx: 1280, maxPx: Number.POSITIVE_INFINITY },
  } as const,

  /**
   * UX-001.1 — Logo height per band. Same single PNG asset scales to
   * each height via `resizeMode="contain"`; no asset variants needed.
   *
   * Phone   : 44px (preserved from Stage 1)
   * Tablet  : 80px (NEW; sized to fit the 96px tablet header with 8+8 padding)
   * Wide    : 96px (NEW; sized to fit the 120px wide header with 12+12 padding)
   *
   * The legacy `logoHeightPxWide: 110` is PRESERVED as a back-compat
   * alias (existing tests pin it), but new code consumes
   * `logoHeightByBand` instead. The reduction from "wide=110" to
   * "wide=96" is a joint reconciliation with the wide header tightening
   * from 152 to 120 — see §9 for the math.
   */
  logoHeightByBand: {
    phone:  44 as const,
    tablet: 80 as const,
    wide:   96 as const,
  },

  /**
   * UX-001.1 — Header outer height per band.
   *
   * Phone   :  64px (preserved from Stage 1)
   * Tablet  :  96px (NEW; tighter than the legacy 152 wide; fits 80px logo + 8+8 padding)
   * Wide    : 120px (TIGHTENED from 152px; fits 96px logo + 12+12 padding)
   *
   * Rationale for tightening 152 → 120 on wide:
   *   - Epic non-negotiable: "header height does not bury the active
   *     board on any tested viewport".
   *   - 152px on a 900px-tall laptop browser is 16.9% — too much.
   *   - 120px on the same viewport is 13.3% — comfortable.
   *   - 96px logo at 120px header = 24px combined padding (12+12).
   *
   * Rationale for adding 96 on tablet:
   *   - 768dp iPad portrait at 152px header = 14.8% viewport; at 96px
   *     = 9.4%. The tighter header keeps the timeline above the fold.
   *   - 80px logo at 96px header = 16px combined padding (8+8).
   */
  headerHeightByBand: {
    phone:  64  as const,
    tablet: 96  as const,
    wide:   120 as const,
  },

  /**
   * UX-001.1 — Typography baseline (minimum needed for header).
   *
   * Three tokens only. UX-001.7 extends this into a full type scale
   * (display / heading / body / caption etc.) covering the rest of the
   * app. Phase 1 scope is intentionally tight.
   *
   * Tokens:
   *   - `wordmarkFallback` — sizes for the text-only fallback when the
   *     PNG asset is missing. Per band (phone / tablet / wide).
   *   - `tagline`           — sizes for the AppHeaderTagline component.
   *     Per variant (inline / stacked).
   *   - `header`            — the small-print right-slot label sizes
   *     (e.g. a future "Act" button label). Per band.
   *
   * All values are platform-neutral dp/px. `lineHeight` is included
   * because RN text layout uses it for vertical alignment.
   */
  typography: {
    wordmarkFallback: {
      phone:  { fontSize: 18, lineHeight: 22, fontWeight: '800' as const, letterSpacing: 0.6 },
      tablet: { fontSize: 28, lineHeight: 34, fontWeight: '800' as const, letterSpacing: 0.6 },
      wide:   { fontSize: 34, lineHeight: 40, fontWeight: '800' as const, letterSpacing: 0.6 },
    },
    tagline: {
      inline:  { fontSize: 18, lineHeight: 24, letterSpacing: 0.2, fontWeight: '400' as const },
      stacked: { fontSize: 14, lineHeight: 18, letterSpacing: 0.2, fontWeight: '400' as const },

    header: {
      // Right-slot label sizes (e.g. icon label on tablet+, hidden on phone).
      phone:  { fontSize: 0,  lineHeight: 0,  fontWeight: '500' as const }, // icon-only on phone
      tablet: { fontSize: 12, lineHeight: 16, fontWeight: '500' as const },
      wide:   { fontSize: 13, lineHeight: 18, fontWeight: '500' as const },

## §5 — Q1 Breakpoint strategy (VERDICT)

**Verdict: Option A — extend `useHeaderBreakpoint.ts` to 3-state, keep
`isWide` as a legacy alias.**

### Concrete pixel values

| Band | Min (dp) | Max (dp) | Maps to |
|---|---|---|---|
| `phone` | 0 | 599 | Phones (iPhone, Android phones, Galaxy Fold inner) |
| `tablet` | 600 | 1279 | Tablets (iPad portrait + landscape, Android tablets) + small desktop |
| `wide` | 1280 | ∞ | Laptops + desktops + wide-desktop |

### Why 3 bands, not 4 or 2

The epic recommends 4 bands (phone / tablet / desktop / wide-desktop at
0/600/1024/1440). The 4-band model is over-engineered for an opener:

- **Tablet portrait (iPad 768) and tablet landscape (iPad 1024)** are
  the same physical device with the same visual density needs.
  Splitting them would force the header to redesign its right slot at
  device rotation — a UX bug, not a feature.
- **The actual visual breakpoint that matters** is "phone vs not-phone"
  for typography / spacing, and "laptop+desktop class vs everything
  smaller" for logo + density. That's 3 bands.
- **4 bands × 5 cards (UX-001.2-001.6) = 20 breakpoint × design
  decisions.** 3 bands × 5 cards = 15. The savings compound.
- **Going to 2 bands** (keeping today's `isWide` model) was the
  alternative considered but rejected — see Q5 verdict (the wide-band
  header height must differ between tablet and desktop to satisfy the
  epic's "no buried board" non-negotiable).

### Back-compat (the bridge)

The existing `isWide: boolean` field on `HeaderBreakpoint` is preserved.
Its new semantic is `band !== 'phone'` (true for tablet AND wide). This
preserves every existing call site:

- `AppHeader.tsx` line 55: `const taglineVariant = isWide ? 'inline'
  : 'stacked'`. Still correct: a tablet's tagline should be inline next
  to the larger 110px logo, same as a desktop's.
- All 86 existing BRAND-001 + BRAND-002 tests still pass because
  `resolveHeaderBreakpoint(720).isWide === true` remains true (720dp
  falls in the tablet band; tablet is not phone; therefore `isWide` is
  true).

The legacy `BRAND.headerWideBreakpointPx: 720`, `BRAND.logoHeightPxWide:
110`, `BRAND.headerHeightPxWide: 152` constants are PRESERVED so the
existing test suite continues to read them. New code consumes the new
`BRAND.breakpoints` / `BRAND.logoHeightByBand` / `BRAND.headerHeightByBand`
maps instead. UX-001.7 may deprecate the legacy aliases as a follow-up;
this card does not.

### Why useWindowDimensions, not CSS media queries

Per `expo-rn-patterns` skill — Expo React Native with web support uses
`useWindowDimensions()` as the canonical primitive. CSS media queries
are web-only and would not apply on native. The existing hook already
uses `useWindowDimensions`; UX-001.1 preserves that pattern.

## §6 — Q2 Dark surface hierarchy (VERDICT)

**Verdict: Option B — keep BRAND's 2-level shell scale; document the
mental-model mapping to BRAND-002's `SURFACE_TOKENS` 14-key scale.**

### The split

Two surface families ship in this repo today, by design:

| Family | Hex | Purpose | Owned by |
|---|---|---|---|
| **BRAND shell** (`surface.app`, `surface.appElevated`) | `#08060F`, `#13101D` | The brand-forward dark background that frames the entire app outside the room | BRAND-001 (PR #55) |
| **SURFACE_TOKENS room family** (`base`, `elevated`, `overlay`, `raised`, `border`, `divider`, `inputBg`, `inputBorder`, `placeholder`, `focusRing`, `textPrimary`, `textSecondary`, `textMuted`, `textInverse`) | `#020617`, `#0b1220`, `#0f172a`, `#162033`, ... | The "non-room screens" tokens consumed by Account, Admin, Invite, Auth, legacy composer | BRAND-002 (PR #138) |

### Mental model: primary / secondary / tertiary

The epic frames the dark surface hierarchy as "primary / secondary /
tertiary". UX-001.1 maps that mental model to the shipped tokens:

| Epic mental-model layer | Concrete token | Where it appears |
|---|---|---|
| **Primary** (the app's branded canvas — the deepest layer) | `BRAND.surface.app.bg` (`#08060F`) | App backdrop in `App.tsx:appRoot`, AppHeader background |
| **Secondary** (a card / panel on the canvas) | `BRAND.surface.appElevated.bg` (`#13101D`) | Currently used by future card surfaces; reserved for the Phase 1 header divider position |
| **Tertiary** (a popout / overlay above a card) | `SURFACE_TOKENS.overlay` (`#0f172a`) | PreferencesPopout, ProfileTagPopout (already consumes this via BRAND-002) |

The mapping is documented in `src/lib/designTokens.ts` as a JSDoc
comment block on the existing BRAND object (no new token shipped). The
test pins the mapping by asserting that the four named hex values are
present and unchanged.

### Why not Option A (extend BRAND to 3 levels)

- BRAND-002 already shipped 14 dark surface tokens. Adding a third
  level to BRAND would create a "which surface library is canonical?"
  ambiguity for UX-001.2-001.7 implementers.
- The two families already serve different purposes by design: BRAND is
  the brand-forward shell (cream-on-black, almost black `#08060F`);
  SURFACE_TOKENS is the "screens that need to be functional and
  legible" family (slightly bluer `#020617` to pair with light text
  weights at AA).
- Subsequent cards (especially UX-001.5 metadata pass) will need
  `SURFACE_TOKENS.border` / `divider` / `inputBorder` for the
  annotation chips and badges — those already exist and are
  AA-verified.

### Phase 1 framing for UX-001.2-001.7

UX-001.2 (timeline-first viewport) consumes `BRAND.surface.app.bg` as
the room's primary background. The room's own internal surfaces
(`ArgumentGameSurface` uses `#020617` / `#0b1220` directly today —
these match `SURFACE_TOKENS.base` / `.elevated`) are inside-the-room
chrome and are out of UX-001.1 scope.

## §7 — Q3 Responsive logo treatment (VERDICT)

**Verdict: 3-state logo height; same single PNG asset; left-aligned with
band-specific minWidth on the home pressable.**

### Logo height per band

| Band | Logo height | Source token | Rationale |
|---|---|---|---|
| phone | 44 px | `BRAND.logoHeightByBand.phone` (preserved from Stage 1) | Fits 64px header with 10+10 padding (stacked tagline below) |
| tablet | 80 px | `BRAND.logoHeightByBand.tablet` (NEW) | Fits 96px tablet header with 8+8 padding (inline tagline) |
| wide | 96 px | `BRAND.logoHeightByBand.wide` (NEW) | Fits 120px wide header with 12+12 padding (inline tagline) |

The legacy `BRAND.logoHeightPxWide: 110` constant is **preserved** for
back-compat (existing tests pin it). New code consumes
`logoHeightByBand` instead.

**Note:** the "tablet 80 < legacy wide 110" reduction is intentional.
The legacy 110px logo was sized for the legacy 152px header; the new
96px wide header cannot fit a 110px logo with vertical padding. The
joint reconciliation of Q3 + Q5 is captured in §9.

### Why the same PNG asset

`assets/branding/civic-discourse-logo.png` is 2.31 MB and high-resolution
enough to scale from 44 to 96 without aliasing under
`resizeMode="contain"`. No size variants are shipped:

- A 96px display height on a 2× retina device → 192px of physical
  pixel rendering. The source asset is comfortably above that
  resolution (asset width × height not measured in this design but
  asserted >10 KiB minimum and confirmed PNG-format by the existing
  asset tests at `__tests__/appHeader.test.ts:160-182`).
- Shipping size variants would add 3 assets, ~6 MB total. Single asset

  `expo-rn-patterns`.

### Positioning per band

| Band | Position | Inner layout | Right slot |
|---|---|---|---|
| phone | Left-aligned, inside `homePressable` with `minWidth: 120`. Tagline stacked beneath logo. | Column | Single icon (preferences gear today) |
| tablet | Left-aligned, `minWidth: 200`. Tagline inline next to logo. | Row | Icon + small label allowed |
| wide | Left-aligned, `minWidth: 240`. Tagline inline. | Row | Icon + small label + optional second control |

### Why left-aligned (not centered)

- The home pressable + logo + tagline group reads as the brand+nav

- Centering would compete with the right slot for visual hierarchy —
  the user's eye wouldn't know whether the logo or the right-slot
  controls are primary.
- The epic's non-negotiable "header looks intentional, not incidental"
  reads as "convention-conformant, brand-forward" — left-aligned with
  generous `minWidth` is the canonical pattern.

The PNG asset path is `assets/branding/civic-discourse-logo.png`

## §8 — Q4 Header information density (VERDICT)

**Verdict: per-band density contract with explicit right-slot capacity.**

### Per-band density specification

| Band | Logo + tagline | Right slot capacity | Examples (existing + future) |
|---|---|---|---|
| **phone** (0-599 dp) | Logo 44px + tagline stacked (14px text) | **1 icon-only control** (~44×44 hit target) | Preferences gear (current). UX-001.4 may add an Act/Go menu trigger as a second control IF the visible-icon design fits without crowding the home pressable. |
| **tablet** (600-1279 dp) | Logo 80px + tagline inline (18px text) | **Up to 3 icon-only controls OR 2 icon+label controls** | Preferences gear; UX-001.4 will likely add Act/Inspect/Go pop-up triggers here. |

### What "capacity" means

Capacity is the **maximum** number of controls UX-001.4 (or later
cards) may place in the right slot without making the header visually
crowded. The right slot is the existing `<View style={styles.rightSlot}
... />` block at lines 104-106 of `AppHeader.tsx` — UX-001.1 does NOT
remove it or rename it.

The actual content of the right slot is OWNED BY THE PARENT
(`App.tsx`'s `preferencesTrigger`). UX-001.1 specifies only the
capacity contract; UX-001.4 will introduce the actual triggers.

### Why an explicit capacity contract

UX-001.4 (Act / Inspect / Go simplification) is a multi-trigger card.
Without a documented capacity contract, the implementer of UX-001.4 has
no way to know whether they may add 1, 2, or 4 triggers per breakpoint.
The contract here is the single source of truth.

### What NOT to put in the right slot

The right slot is for **global** controls (preferences, sign-out gate,
future search if voting/search ever ship in v2). Per-debate or
per-screen controls (e.g. "compose" buttons, side filters, view-mode
toggles) belong in the screen body, not the header. UX-001.2 and
UX-001.3 will respect this boundary.

## §9 — Q5 Header height per breakpoint (VERDICT)

**Verdict: phone 64 / tablet 96 / wide 120. Tighten legacy 152 → 120.**

### Final values (joint Q3 + Q5 reconciliation)

The header height per band is reconciled with the logo height per band
(Q3) so that each logo fits its header with comfortable vertical
padding. Inline layout (tablet + wide): logo + tagline are in a row;
tagline `lineHeight: 24` is shorter than the logo so the row's height
equals the logo height. Stacked layout (phone): tagline sits below the
logo so the combined column height is `logo + 4px gap + tagline lineHeight`.

| Band | Logo height | Header height | Layout | Combined visual | Padding budget |
|---|---|---|---|---|---|
| phone | 44 | 64 | Stacked (col) | 44 + 4 + 18 = 66 (clips 2 via lineHeight collapse — matches Stage 1) | 0 (tight; Stage 1 baseline) |
| tablet | 80 | 96 | Inline (row) | max(80, 24) = 80 | 16 (8 top + 8 bottom) |
| wide | 96 | 120 | Inline (row) | max(96, 24) = 96 | 24 (12 top + 12 bottom) |

```ts
logoHeightByBand: {
  phone:  44 as const,
  tablet: 80 as const,
  wide:   96 as const,
},
headerHeightByBand: {
  phone:  64  as const,
  tablet: 96  as const,
  wide:   120 as const,
},
```

The legacy `BRAND.logoHeightPxWide: 110` and `BRAND.headerHeightPxWide:
152` constants are PRESERVED unchanged for back-compat. New code
consumes `logoHeightByBand` / `headerHeightByBand` instead. Existing
tests pinning `logoHeightPxWide === 110` and `headerHeightPxWide ===
152` continue to pass.

### Why these specific numbers (not 110 / 132)

The first-draft proposal was "tablet logo 110, wide logo 132". When
joint-reconciled with the epic's "no buried board" non-negotiable
(forcing wide header ≤ 120 to free viewport for the timeline), neither
proposed logo height fits:

- A 110px tablet logo cannot fit a 96px tablet header.
- A 132px wide logo cannot fit a 120px wide header.

The two ways out are:
- **Grow the headers** (wide back to 152, tablet to 124+) → reintroduces
  the buried-board problem the epic explicitly forbids.
- **Shrink the logos** (tablet to 80, wide to 96) → preserves the
  tightened headers AND keeps the wide logo visibly bigger than the
  tablet logo.

The second path is the only one compatible with all of: (a) epic's "no
buried board" non-negotiable, (b) epic's "logo visibly larger on
tablet/wide" intent, (c) BRAND-001 Stage 1 baseline (phone unchanged).

### Compared with the legacy Stage 2 sizes

| | Phone | Tablet | Wide | Legacy "wide" |
|---|---|---|---|---|
| Header height | 64 (unchanged) | 96 (NEW) | 120 (NEW; tighter) | 152 (preserved alias only) |
| Logo height | 44 (unchanged) | 80 (NEW) | 96 (NEW) | 110 (preserved alias only) |
| Viewport % at 800-tall | 8.0% | 12.0% | 15.0% | 19.0% (was) |
| Viewport % at 1024-tall iPad portrait | 6.3% | 9.4% | n/a (≥1280) | 14.8% (was) |

### Rationale recap

- **96px wide logo > 80px tablet logo:** wide is visibly 1.2× bigger
  than tablet — meets the epic's "logo visibly larger on wide" intent.
- **80px tablet logo > 44px phone logo:** tablet is visibly 1.82×
  bigger than phone — meets the epic's "logo larger on tablet" intent.
- **120px wide header < 152px legacy wide:** new wide header is 21%
  shorter than legacy, frees real estate for the timeline (UX-001.2).
- **96px tablet header < 152px legacy wide:** tablet portrait iPad now
  gets a 9.4% viewport header instead of 14.8%.

## §12 — Q8 Typography and spacing baseline (VERDICT)

**Verdict: Option A — ship the minimum typography needed for the header
in this card. UX-001.7 extends to the full app typography scale.**

### What ships in Phase 1

The new `BRAND.typography` object specified in §4. Three sub-objects:

- `wordmarkFallback` — per-band sizes (phone 18px, tablet 28px, wide
  34px). Read by `AppHeader.tsx` for the text-only fallback (when the
  PNG fails to load).
- `tagline` — per-variant sizes (inline 18px, stacked 14px). Read by
  `AppHeaderTagline.tsx`. Replaces the inline `fontSize: 18` /
  `fontSize: 14` literals.
- `header` — per-band sizes for the right-slot label
  (`fontSize: 12` tablet, `13` wide, `0` phone = icon-only).
  Reserved for UX-001.4's Act/Inspect/Go triggers. Not consumed by
  UX-001.1 itself — declared in advance so UX-001.4 doesn't have to
  introduce typography ad-hoc.

### What does NOT ship in Phase 1

- A full type scale (display / heading / body / caption / code /
  micro). That's UX-001.7's scope.
- Spacing scale extension. `SPACING` already exports `xs/s/m/l/xl`;
  UX-001.7 will extend if needed.
- Font weight scale (only specific weights used by header are typed in
  Phase 1).
- Line-height base unit ratio system (Phase 1 specifies absolute
  lineHeight values per token only).
- Dynamic type / accessibility scaling (covered by
  `allowFontScaling` on RN Text components; Phase 1 keeps this
  unchanged).

### Why ship the baseline now (not defer entirely)

UX-001.2 (timeline-first viewport) will inline typography decisions for
the timeline grammar (already partially typed in
`argumentGameSurfaceModel.ts` etc., but the SHELL-level "this is what
header text looks like" is not). UX-001.3 (composer) will need
input/label sizes. UX-001.4 (Act/Inspect/Go) will need menu trigger
labels.

Without a Phase 1 baseline, each card pays a typography decision tax:
"what fontSize for a menu trigger label?" "what lineHeight on a sidecar
title?" The Phase 1 `header` sub-object gives UX-001.4 its answer
already; the other cards will inline their own typography until UX-001.7
codifies them.

The alternative (defer to UX-001.7) leaves UX-001.2-001.6 to either:
- Inline literal fontSize values (paying the drift tax 5 times); or
- Wait for UX-001.7 (gates the entire epic on the last card).

Shipping a minimum baseline now is the tighter call.

### Why not a full type scale

A full type scale is the SCOPE of UX-001.7. Doing it in UX-001.1 would
violate the per-card discipline. The Phase 1 baseline is intentionally
the smallest viable surface — 3 sub-objects, 8 total tokens — that
unblocks UX-001.4 specifically.

## §13 — Edge cases

- **SSR initial render width=0.** Preserved from Stage 2: `width <= 0`

  asserts `resolveBand(0) === 'wide'`.

- **Negative width.** Preserved from Stage 2: defensive,
  `resolveBand(-1) === 'wide'`. Never crashes.

- **Very narrow phone (≤ 320 dp).** Falls in the phone band. Header
  stays at 64px with tagline stacked beneath the 44px logo. No layout
  shift. Test: `resolveBand(320) === 'phone'`.

- **Very wide ultrawide monitor (3440+ dp).** Falls in the wide band.
  Header stays at 120px with 96px logo. No upper cap — wider screens
  just have more empty space in the header. Test:
  `resolveBand(3440) === 'wide'`.

- **Boundary exactly at 599 / 600 / 1279 / 1280.** Specified per Q1
  verdict: `<= 599` is phone, `>= 600 && <= 1279` is tablet, `>= 1280`
  is wide. Test:
  - `resolveBand(599) === 'phone'`
  - `resolveBand(600) === 'tablet'`
  - `resolveBand(1279) === 'tablet'`
  - `resolveBand(1280) === 'wide'`

- **Dynamic type accessibility scaling (allowFontScaling).** The
  existing `AppHeaderTagline` sets `allowFontScaling` to true (default).
  The new `BRAND.typography.tagline.inline.fontSize: 18` is a base
  value — RN scales it up if the OS-level text size preference is
  larger. The header height per band has 16-24px of padding budget to
  absorb a 1.2× text scale; beyond that the tagline will clip via
  `numberOfLines={1}`. No layout failure. The wordmark fallback
  similarly has padding budget. Documented; no specific test (would
  require RN render-test infra not currently in repo).

- **RTL languages.** RN flexbox handles RTL automatically (logical
  flexDirection swaps). The logo + tagline group reverses direction;
  the right slot becomes the left slot. AppHeader's `justifyContent:
  space-between` keeps the visual hierarchy intact. No specific
  handling needed.

- **PNG asset missing (resolves to undefined).** Wordmark fallback
  renders. Per-band fontSize via `BRAND.typography.wordmarkFallback[band]`.
  Existing test `BRAND-001 AppHeader component › text-only wordmark
  fallback exists for the missing-asset case` continues to pass.

- **Reduce motion.** No animations are added in this card. Preserved
  from Stage 2: header is static, no `AccessibilityInfo` check needed.

- **Doctrine edge case — does header copy ever ship verdict tokens?**
  No copy added. The only header text is (1) the wordmark fallback
  ("CivilDiscourse" — proper noun, frozen), (2) the tagline ("Just
  get to the bottom of it" — frozen, doctrine-cleared in Stage 2). No
  new strings introduced.

- **Doctrine edge case — does the typography token leak verdict
  vocabulary?** Token KEYS are structural (`wordmarkFallback`,
  `tagline`, `header`, `inline`, `stacked`, `phone`, `tablet`, `wide`).
  Token VALUES are numbers + small enum strings (`'500'`, `'800'`,
  `'400'`). Test: ban-list scan asserts no token key matches
  `FORBIDDEN_TOKEN_TOKENS`.

- **Doctrine edge case — does the breakpoint number imply "tablet =
  truth" or similar?** Numbers carry no semantic meaning. The
  `'tablet'` enum string is a device-class label, not a verdict on the
  content shown.

- **Concurrent breakpoint resize on web.** `useWindowDimensions`
  triggers re-render; `useHeaderBreakpoint` returns the new band; the
  header re-renders with new heights / logo size. React's render
  semantics handle the swap (preserved from Stage 2).

- **Existing 86 BRAND tests must still pass.** Pre-flight diff against

  a preserved-value (Stage 1 + Stage 2 keys all preserved verbatim) or
