## Goal

Ship a pure-TS visual-symmetry MAP that turns the nine issue-listed validation / semantic codes into a typed visual descriptor — `{ chipKind, edgeStyle, bandKind, iconHint, label, helperLine }` — so that node renderers, dock chips, sidecar panels, and gallery-card affordances can read one map and produce visually-consistent affordances for each axis WITHOUT inventing new labels and WITHOUT leaking raw snake_case codes to normal-user surfaces. The card is the *visual* sibling of RULE-001 (which is the *tool* sibling) and RULE-003 (which is the *lifecycle UX* sibling). All three describe the same axis vocabulary from three angles; EV-004 commits to the **chip + edge + band + icon** angle. The doctrine constraints that shape the design are `cdiscourse-doctrine` §1 (no truth labels), §2 (heat is activity), §3 (popularity is not evidence; doctrine-pinned phrase appears verbatim), §9 (no internal code leak to user surfaces), and `evidence-doctrine` (engagement vs factual-standing separation, anti-amplification semantic).

## What EV-004 IS and IS NOT

### EV-004 IS

- A pure-TS module: `src/features/evidence/evidenceSymmetryMap.ts`. Exports a frozen `Record<EvidenceSymmetryCode, EvidenceSymmetryEntry>` plus a total `getEvidenceSymmetry(code)` reader and the small string-union types (`ChipKind`, `EdgeStyle`, `BandKind`) consumers commit to.
- A label / helper-line parity contract — `entry.label === toPlainLanguage(code)`, asserted at test time. Where RULE-003 ships a helperLine for the same code (e.g. `synthesis_ready` lives in `LIFECYCLE_UX_MAP['synthesis_ready']`), EV-004's `helperLine` + `iconHint` are equal to RULE-003's by-reference reads — no fresh labels.
- A doctrine-pinned phrasing for `anti_amplification`: `helperLine === 'Popularity is not proof — show the source.'` (the exact phrase from the brief, also matching the doctrine §3 phrase already used in `RULE_TO_UI_AFFORDANCE['anti_amplification'].toolLabel`).
- An *additive* extension of RULE-003's `IconHint` union (4 new values; chosen approach (a) — see §"IconHint reuse decision").

### EV-004 IS NOT

- A render component. No React. No `<View>`, no `<Text>`, no `Pressable`, no animation, no color tokens, no SVG. The `IconHint` / `ChipKind` / `EdgeStyle` / `BandKind` values are *semantic vocabulary*; consumers pick the actual icon implementation, line dash array, and band background.
- A new validation engine. EV-004 reads codes that the constitution engine + the rule-axis pipeline already emit; it does NOT decide which code applies. The check-side belongs to `src/lib/constitution/engine.ts` + `src/features/pointStanding/antiAmplification.ts`, neither of which EV-004 touches.
- A new entry in `PLAIN_LANGUAGE_COPY`. Every `label` field reads through `toPlainLanguage(code)`; if the source-of-truth label changes, EV-004 follows automatically. EV-004 NEVER authors a new label.
- A renderer of any of the nine codes. EV-004 ships the *descriptor*; consumers (EV-001/EV-002 sidecar panels in later cards, ST-002 suggestion chips, GAL-002 gallery cards, IX-002 keyboard hint surface, the upcoming RULE reference card) consume `getEvidenceSymmetry(code)` verbatim.
- A side-stratified surface. Like SW-002, EV-004 is room-level vocabulary — no per-side asymmetry.
- An animation hint emitter. No motion, no easing, no reduce-motion branches. Visuals that consume the map decide their own motion policy.

### EV-004 ↔ RULE-001 ↔ RULE-003 ↔ gameCopy boundary

| Concern | Owner | EV-004 stance |
|---|---|---|
| Axis label (`source_chain` → "Source trail") | `gameCopy.PLAIN_LANGUAGE_COPY` | Read-through only via `toPlainLanguage(code)`. NEVER re-authored. |
| Axis → TOOL (toolLabel / composerLead / suggestedMove) | RULE-001 (`ruleToUiMap.ts`) | Out of scope — EV-004 does NOT shadow toolLabel. |
| Lifecycle / manual-tag / auto-metadata → UX triple | RULE-003 (`lifecycleUxMap.ts`) | Reused for `synthesis_ready` (which lives in `LIFECYCLE_UX_MAP`). EV-004 pulls helperLine + iconHint by-reference from there. |
| `IconHint` union | RULE-003 | Extended additively in option (a). EV-004 contributes 4 new values; never redefines existing ones. |
| `ChipKind`, `EdgeStyle`, `BandKind` | EV-004 | New small unions, never verdict-flavored. |
| Rendering (React, color tokens, line dash) | Consumers (later cards) | EV-004 ships zero rendering. |

The boundary is enforced two ways:
- A source-scan test asserts `evidenceSymmetryMap.ts` contains zero references to `React`, `View`, `Text`, `Pressable`, `supabase`, `fetch`, `anthropic`, `xai`, `Date.now`, `Math.random`.
- A type-import-only scan asserts cross-module imports are `import type { ... }` — no runtime value imports from RULE-003 except the `IconHint` union (which is type-only) and the `LIFECYCLE_UX_MAP` constant (which is a value but is consumed read-only for the `synthesis_ready` helperLine / iconHint parity assertion at module-load time, identical to how `LIFECYCLE_UX_MAP` itself imports `getPointLifecyclePlainLabel`).

## Data model

### Module location

**New file:** `src/features/evidence/evidenceSymmetryMap.ts`.

**New folder:** `src/features/evidence/` — the v1 home for the Epic 6 evidence vocabulary. EV-004 is the first file. EV-001 / EV-002 / EV-003 will land sibling modules (`evidenceArtifactModel.ts`, `sourceChainPopoverModel.ts`, `evidenceDebtChipModel.ts`) in the same folder later; this card does NOT create those.

### ChipKind union

```ts
/**
 * EV-004 — Chip kinds. Each value names a SEMANTIC chip class; consumers
 * pick the actual icon, color, and shape. NEVER a verdict glyph; NEVER
 * an engagement / amplification name; NEVER a person-attribution name.
 */
export type ChipKind =
  | 'source_trail'    // source_chain — issue-named "source trail chip"
  | 'receipt'         // evidence_debt — issue-named "receipt chip"
  | 'scope'           // scope axis chip
  | 'definition'      // definition axis chip
  | 'logic'           // logic axis chip
  | 'mechanism'       // causal axis chip
  | 'amplification'   // anti_amplification axis chip
  | 'synthesis'       // synthesis_ready axis chip
  | 'resolved';       // max_depth_reached → routed to lifecycle band (R4); see "Render-time qualification".
```

`resolved` is included here because the `ChipKind` field is still defined for `max_depth_reached`; consumers MAY render it as a chip on an admin / debug surface even though normal-user surfaces render the band instead (per R4). The `bandKind` is the primary visual carrier for `max_depth_reached`; the chip is the fallback.

### EdgeStyle union

```ts
/**
 * EV-004 — Edge styles. Describes the TIMELINE EDGE running into a node
 * carrying the axis chip. NEVER a verdict-flavored line treatment.
 * `null` is a legal value: most axes do not modify the edge. Only the
 * source_chain edge is modified — its "source trail dotted edge" is the
 * issue-named visual.
 */
export type EdgeStyle =
  | 'dotted'    // source_chain — the issue's "dotted edge"
  | 'solid'     // explicit default for axes that intentionally render the
                // standard rail. Most axes use `null` (no edge override).
  | 'kinked';   // tangent / branch kink — reserved for future cards; EV-004
                // does NOT emit it (no code in the v1 vocabulary maps to it).
```

The `kinked` value is reserved in the union so that future BR-002 / tangent-marker cards land additively without changing the union shape. EV-004 emits only `'dotted'` (for `source_chain`) and `null` (for the other eight codes).

### BandKind union

```ts
/**
 * EV-004 — Band kinds. A "band" is a horizontal strip rendered behind a
 * node OR a cluster header to signal a lifecycle / structural state. NOT
 * a standing band (those live in `argumentScoreModel.StandingBand`).
 * NEVER verdict-flavored.
 */
export type BandKind =
  | 'resolved';   // archived_or_resolved lifecycle — used by R4 for
                  // max_depth_reached → lifecycle band routing.
```

EV-004 commits to ONLY `'resolved'` in v1. Future expansion (e.g. an `'open'` band, an `'attention'` band) is additive. The union is small on purpose — most axes don't need a band; chips + edges + icons carry the symmetry.

### EvidenceSymmetryEntry shape

```ts
import type { IconHint } from '../rulesUx/lifecycleUxMap';

  /** Identical to the EvidenceSymmetryCode enum value — present for round-trip ergonomics. */
  code: EvidenceSymmetryCode;
  /** Read from `toPlainLanguage(code)` at module-load time. NEVER newly authored here. */
  label: string;
  /** ≤ 80 chars. Single-line tooltip / a11y helper. Plain English. */

  /** Reused from RULE-003's `IconHint` union. EV-004 extends the union additively (4 new values). */

```

### Decision and justification

**Option (a) — additive extension of `IconHint` in `lifecycleUxMap.ts`** is chosen because:

1. The issue's acceptance criteria require visual symmetry between the rule code and the icon. Mapping `definition` to `question_mark` is a *semantic* match (both ask "what does this mean?") but visually conflates definition disputes with `clarified` lifecycle states. Real visual symmetry needs the new icon.
2. The four new icons (`key_term`, `logic_chain`, `causal_arrow`, `crowd_slash`) are structural / activity descriptors, NOT verdict glyphs. They pass RULE-003's `_forbiddenLifecycleTokens()`-style scan (no `checkmark`, `x`, `crown`, `flame`, `thumbs_up`, `thumbs_down`, `trophy`, `shield`, `warning`, `medal`, `gavel`, `star`).
3. Additive-only — every existing `IconHint` value stays; every consumer of `IconHint` continues to compile. The RULE-003 ban-glyph test passes unchanged because none of the four new values is verdict-shaped.

## API / interface contracts

### Exported API

```ts
// src/features/evidence/evidenceSymmetryMap.ts

export type EvidenceSymmetryCode = /* 9-member string union — see Data model */;

export type ChipKind = /* 9-member string union — see Data model */;
export type EdgeStyle = /* 3-member string union (one is reserved) — see Data model */;
export type BandKind = /* 1-member string union (v1) — see Data model */;

export interface EvidenceSymmetryEntry {
  code: EvidenceSymmetryCode;
  label: string;
  helperLine: string;
  iconHint: IconHint;        // re-exported as a type-only import from rulesUx/lifecycleUxMap
  chipKind: ChipKind;
  edgeStyle: EdgeStyle | null;
  bandKind: BandKind | null;
}

/** The 9-entry frozen map. */
export const EVIDENCE_SYMMETRY_MAP: Readonly<Record<EvidenceSymmetryCode, EvidenceSymmetryEntry>>;

/** All 9 codes, in declaration order. Useful for tests + docs. */
export const ALL_EVIDENCE_SYMMETRY_CODES: ReadonlyArray<EvidenceSymmetryCode>;

/** Total, pure reader. Same input → same output. */
export function getEvidenceSymmetry(code: EvidenceSymmetryCode): EvidenceSymmetryEntry;
```

### Consumer contract

Consumers call `getEvidenceSymmetry(code)` with a code from `EvidenceSymmetryCode` and render the returned fields verbatim:

- `entry.label` → chip / panel label string. Never re-format.
- `entry.helperLine` → tooltip / a11y helper. Never re-format.
- `entry.iconHint` → semantic icon name; consumer maps to the actual icon implementation.
- `entry.chipKind` → semantic chip class; consumer maps to chip style.
- `entry.edgeStyle` (or null) → rail line dash array; consumer maps to the actual stroke style. `null` means "use rail default".
- `entry.bandKind` (or null) → behind-node band; consumer maps to band background. `null` means "no band".

**Fallback policy:** there is NO fallback string. If a future code arrives that is NOT a member of `EvidenceSymmetryCode`, callers MUST use a defensive variant — proposed: `getEvidenceSymmetryOrSuppress(code: string | null | undefined): EvidenceSymmetryEntry | null`, returning `null` for unknown codes. The defensive variant is OPTIONAL to ship in v1 — the type-level total `getEvidenceSymmetry` is the canonical API and is the only one EV-004 promises. Consumers that read from an untyped source (e.g. an Edge Function response) should narrow before calling.

For symmetry with `mapRuleToUiAffordance` / `mapRuleToUiAffordanceOrSuppress` in `ruleToUiMap.ts`, EV-004 ships both:

```ts
export function getEvidenceSymmetry(code: EvidenceSymmetryCode): EvidenceSymmetryEntry;
export function getEvidenceSymmetryOrSuppress(code: string | null | undefined): EvidenceSymmetryEntry | null;
```

The `OrSuppress` variant normalises whitespace / hyphens / case (mirrors `ruleToUiMap.normalise`) and returns `null` for unknown codes. No raw snake_case echo, no error string.

## Render-time qualification (COPY-001 R1–R5)

EV-004 is itself a consumer-target — the descriptors it ships will be rendered by future cards. The five COPY-001 rules apply at the consumer layer, not inside EV-004; this section documents how each rule shapes EV-004's choices.

### R1 — scope-strict positioning

> A surface that renders cluster-scope codes MUST NOT render move-scope codes in the same visual band, and vice versa.

**EV-004 stance:** All nine codes are *axis-scope* (or runner-status, in the case of `max_depth_reached`). None is cluster-scope or move-scope in the LIFE-001 / META-001 sense. Consumers that render an EV-004 chip on a per-move surface MUST place it in the move-chip strip, not the cluster header band. Consumers that render an EV-004 chip on a cluster surface (e.g. a synthesis chip on a cluster ready for summary) MUST place it in the cluster header band, not the move-chip strip. The chip CONTENT (label + iconHint) is the same; the POSITION is the consumer's responsibility.

### R2 — same-label dedup

> When a single dock / chip strip / sidecar panel would render the cluster-scope label AND a move-scope chip with the same label on the same render, the move-scope chip is suppressed. Cluster header wins.

**EV-004 stance:** The `synthesis_ready` label "Ready for synthesis" matches the manual-tag label `ready_for_synthesis` (both render the same plain-language string). If a consumer renders both the EV-004 `synthesis_ready` chip AND the META-001 `ready_for_synthesis` chip on the same surface, R2 requires the consumer to suppress the duplicate per the rule's wording. EV-004 does NOT enforce dedup itself — the descriptor is correct; the consumer's R2-aware filter is responsible.

### R3 — provenance suffix on cross-level reuse

> When the same label is rendered from two semantic levels, the consumer attaches a provenance hint adjacent to the chip.

**EV-004 stance:** `evidence_debt` is reused across three semantic levels — RULE-001 axis ("Evidence debt"), META-001 manual tag ("Evidence debt"), and now EV-004 visual chip ("Evidence debt"). Per R3, when a surface renders the same "Evidence debt" label from two sources, the consumer attaches a provenance hint (axis dot, tagger avatar, etc.). EV-004 surfaces the label correctly; the consumer does the provenance.

### R4 — runner-status suppression in normal-user surfaces

> The runner-status codes (`synthesis`, `concession`, `submit_failed`, `validation_failed_after_retries`, `three_in_a_row_failures`, `max_depth_reached`) MUST NOT appear in normal-user UI as primary labels. They are operator / corpus / AN-003 diagnostics. When a runner output bubbles up to a normal user, the consumer renders the LIFECYCLE state of the affected cluster (`archived_or_resolved` → "Resolved") instead.

**EV-004 stance:** `max_depth_reached` IS a runner-status code (it appears in the COPY-001 audit's runner-status list). Per R4, normal-user surfaces MUST NOT render the chip with label "Deep unresolved chain"; instead they MUST render the band `bandKind: 'resolved'` with the helperLine pulled from `LIFECYCLE_UX_MAP.archived_or_resolved`. The EV-004 entry encodes this routing by:

1. Setting `bandKind: 'resolved'` (the primary visual carrier — a band, not a chip).
2. Reading `helperLine` and `iconHint` by-reference from `LIFECYCLE_UX_MAP.archived_or_resolved`, so the user-facing copy is the lifecycle copy, NOT the runner-status copy.
3. Leaving `chipKind: 'resolved'` available for admin / debug surfaces that intentionally show the runner code (e.g. AN-003 diagnostics) — but normal-user surfaces SHOULD NOT use the chip.

A consumer-side guard SHOULD wrap a normal-user render in:
```ts
const isRunnerStatusCode = code === 'max_depth_reached';
if (isRunnerStatusCode) {
  // render the band; suppress the chip.
} else {
  // render the chip per chipKind + iconHint + label.
}
```

This guard is the consumer's responsibility; EV-004 documents it but does NOT enforce it (the descriptor is the same shape for all codes).

### R5 — axis-vs-manual-tag dedup on shared semantic root

> When a chip strip would render an axis chip AND a manual-tag chip whose labels share the dominant semantic root word, the consumer renders the axis chip first and suppresses the manual-tag chip. R5 is hygiene, not doctrine. v1 fallback: render both.

**EV-004 stance:** Several EV-004 codes share a semantic root with META-001 manual tags / lifecycle codes:

| EV-004 axis | Shared root | META-001 / LIFE-001 sibling |
|---|---|---|
| `scope` | scope | `scope_issue` (manual tag) |
| `definition` | definition | `definition_issue` (manual tag) |
| `causal` | mechanism | `causal_mechanism` (manual tag) |
| `evidence_debt` | evidence | `evidence_debt` (manual tag) — same code |
| `synthesis_ready` | synthesis | `synthesis_ready` (lifecycle) + `ready_for_synthesis` (manual tag) |

If a consumer chip strip would render BOTH the EV-004 axis chip and the META-001 manual-tag chip with the same root, R5 says (optional v1.1) to render the axis chip first and suppress the manual-tag chip. v1 may render both. EV-004 does not encode R5; consumers do.

## Edge cases

The implementer must handle the following — none requires production code inside EV-004 (the descriptor is the same shape for all codes), but several drive test assertions:

- **Empty inputs.** `getEvidenceSymmetry` is typed total over `EvidenceSymmetryCode`, so `undefined` / empty string are not callable at the type level. The optional `getEvidenceSymmetryOrSuppress` variant returns `null` for empty / null / undefined / unknown codes. Test asserted.
- **Future code without entry.** When a future card adds a value to `EvidenceSymmetryCode` without adding a map entry, TypeScript fails to compile (`Record<EvidenceSymmetryCode, EvidenceSymmetryEntry>` is exhaustive). A coverage test (`for (const code of ALL_EVIDENCE_SYMMETRY_CODES) expect(EVIDENCE_SYMMETRY_MAP[code]).toBeDefined()`) catches the inverse direction. Test asserted.
- **Multi-validator surface.** A node may carry multiple axis chips (e.g. `evidence_debt` + `scope`). Per R3 + R5, the consumer chooses which chips to render and in what order. EV-004 does not order chips; each call returns one descriptor. Test asserted at the descriptor-shape level.
- **`max_depth_reached` on a normal-user surface.** Per R4, the band wins. The chip is available only for admin / debug. Test asserts both branches of the R4 invariant — the band is present, the chip is also present (R4 is a consumer policy, not a descriptor change).
- **Unknown / empty input to `OrSuppress`.** Returns `null`. Test asserted with: `''`, `null`, `undefined`, `'unknown_code'`, `'completely-made-up'`.
- **Case / whitespace / hyphen normalisation.** `getEvidenceSymmetryOrSuppress('SOURCE_CHAIN')`, `'source-chain'`, `'  source_chain  '`, `'Source Chain'` all resolve to the `source_chain` entry. Test asserted.
- **Concurrent edits.** None — the map is frozen at module load. The reader is pure. No state to reconcile.
- **Permission-denied paths.** None — EV-004 has no auth surface. No RLS, no service-role, no Edge Function.
- **Offline / network failure.** None — EV-004 is pure-TS, no network.
- **Doctrine edge case 1 — what if a future axis is added with a verdict-flavored noun?** The ban-list test iterates `ALL_EVIDENCE_SYMMETRY_CODES` and re-scans every `label` + `helperLine` against the COPY-001 verdict ban list (with the `right` / `wrong` / `validated` hardening from COPY-001 §8). Adding a new code without a doctrine-clean label is a test failure.
- **Doctrine edge case 2 — what if a consumer renders the EV-004 chip without R4?** EV-004 cannot prevent this; the rule is consumer policy. The risk is mitigated by the design doc's explicit R4 §, the helper-line by-reference from `LIFECYCLE_UX_MAP.archived_or_resolved` (so even a non-R4-aware consumer renders the right COPY), and a documented test invariant (the `max_depth_reached` entry's `helperLine` matches `LIFECYCLE_UX_MAP.archived_or_resolved.helperLine`).
- **Doctrine edge case 3 — what if `gameCopy.PLAIN_LANGUAGE_COPY[code]` changes?** EV-004 reads through `toPlainLanguage`, so the label tracks automatically. The parity test asserts `entry.label === toPlainLanguage(code)` for all 9 codes; a drift on either side is caught.
- **Doctrine edge case 4 — what if RULE-003 changes `LIFECYCLE_UX_MAP.synthesis_ready.helperLine` or `LIFECYCLE_UX_MAP.archived_or_resolved.helperLine`?** EV-004 reads by-reference at module-load time, so the two entries auto-track. The parity test asserts the equality at run time; a drift on either side is caught.
- **Doctrine edge case 5 — `anti_amplification` helperLine drift.** The doctrine §3 pin requires the EXACT phrase. A test asserts `entry.helperLine === 'Popularity is not proof — show the source.'`. Changing the phrase requires both code and test edits (intentional friction).
