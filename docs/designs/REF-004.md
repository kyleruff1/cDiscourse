REF-003 (`RefereeCardView.tsx` / `refereeCardAssembly.ts`) · the shipped one-box chassis (QOL-030/031/032/033)

This card writes **no production code** (GATE-A is design only). It freezes the wiring that turns three
already-shipped popouts plus the shipped Referee Card into one loop: **Act changes the issue · Inspect explains the
issue · Go moves around the issue map · the box drafts the next move.**

---

## Goal (one paragraph)

REF-003 shipped the Referee Card on the active node with zone-3 next-move buttons that deep-link into the composer
through a v1 stop-gap handler (`handleRefereeMove`, `ArgumentGameSurface.tsx:1389-1397`). REF-004 closes the loop: it
routes those buttons through the **same** Act entry→box dispatch the board-level Act popout already uses (so "Act
changes the issue" is one code path, not two); it adds an Inspect handoff so the Open Issue's full detail — including
the only raw classifier codes anywhere in the product — lives in Inspect; and it adds a Go handoff so the board can be
focused by open-issue state ("show me the open source requests"), dimming non-matching nodes and never hiding them.
The doctrine that shapes every line: the deterministic Constitution engine
(`src/domain/constitution/engine.ts`, mirrored at `supabase/functions/_shared/constitution/`) is the **sole**
submission gate; the Referee Card / Open Issue layer is **consultative**, runs only after an argument is stored, never
blocks/rejects/routes/delays a post, never emits a verdict or person label, and always offers an engine-valid move (or
collapses cleanly). `nextBestMoves` is — and stays — a **subset** of `buildActPopout` survivors, so the card can never
present a button the engine would reject; a structurally-invalid-here move is converted to a recovery route, never
rendered as a rejectable button. This card edits **no file in the engine/role gating path** — the diff is provably
gate-neutral.

## Data model

**No new persisted data model. No migration. No Edge path. No new DB column.** REF-002's `DisagreementContract`
(`OpenIssue`) is derived-only and unchanged; REF-004 consumes it. The card adds three **pure, in-memory, view-only**
shapes:

```ts
// 1. Go addition — src/features/arguments/oneBox/goPopoutModel.ts (NEW exports)
//    Maps an active OpenIssue's IssueState to the EXISTING Go lens that best
//    focuses comparable nodes. NO new FocusLensId, NO timelineDensityLensModel
//    change. `IssueState` is a TYPE-ONLY import (erased at build → no runtime

//    runtime graph).

  source_requested:'evidence',    // → evidence_requested lens
  quote_requested: 'evidence',    // → evidence_requested lens
  answered:        'active_path',  // resolved-ish → focus the path to the active node
```

Notes that make the table load-bearing:

- **Every Act handoff is an engine+role survivor by construction.** `nextBestMoves` (REF-002,
  `deriveNextBestMoves`, `openIssueModel.ts:821-871`) is the intersection of `deriveSuggestedMoves` with
  `flattenActPopout(buildActPopout(...))`. The card never holds an `ActEntryId` the gates removed. The "if present
  after hard gates" condition is therefore enforced **upstream in REF-002**, not re-checked at the surface.
- **Narrow / Concede / Synthesize carry a real `argumentType`** (`concession` / `concession` / `synthesis`) and so are
  subject to the engine gate; when the parent makes them invalid, REF-002 already substitutes the **`branch_tangent`
  recovery route** (`argumentType: null`, always valid for a participant on a node) and the invalid entry is never
  rendered. See §"Recovery-route conversion."
- **Confirm and Branch are engine-exempt** (`argumentType: null`) and survive whenever the role permits.
- **The Inspect "section" is which of the fixed 7 sections the issue's detail leans on** — Inspect's section *set*

- **The Go column is one issue-scoped action.** The active issue has exactly one `state`; `handleRefereeFocusIssue`
  jumps to `issue.targetNodeId` and applies `issueStateToGoLens(issue.state)`. The per-row entries above are the lens
  that results once the corresponding move lands and the issue re-derives.

## Act routing — the exact function/prop chain (the handler swap)

**Live chassis finding:** `ActPopout` (`ActPopout.tsx:269-425`) has **no pre-focus / highlight prop** — it renders all
gated groups and dispatches on entry tap (`ActPopoutProps`, `ActPopout.tsx:180-259`; no `focusedEntryId`). The
Inspect→Act handoff already shipped acknowledges this: `handleInspectHandoffToAct` "does not scroll/focus the specific
entry inside Act (deferred)" (`ArgumentGameSurface.tsx:1498-1509`). Therefore **"open Act pre-focused on the mapped
entry" is not supported by the live chassis.**

**Decision (resolving the card's "open Act focused / or direct `actEntryToQuickAction` into the box" choice):** a
Referee Card zone-3 button **already encodes the specific surviving `ActEntryId`**. Re-opening the Act menu just to make
the user find that same entry would be a redundant step. So REF-004 routes the button **directly into the one-box
through the Act entry→box dispatch the board Act mount already uses** — `handleBoardActSelectBoxType`
(`ArgumentGameSurface.tsx:1429-1455`), which is `actEntryToQuickAction(entryId)` (`actPopoutModel.ts:785`) →
`quickActionToPreset(quickAction, activeParentType)` → `handleAction('reply', activeMessageId ?? '', preset)`. This is
**Act's own bridge** (no parallel path), so the loop closes semantically ("Act changes the issue") without a pre-focus
prop and without a second composer path.

**The swap (behavior-preserving extraction):**

1. Extract the body of `handleBoardActSelectBoxType` into a shared `useCallback`
   `enterBoxForActEntry(entryId: ActEntryId)`:
   ```ts
   const enterBoxForActEntry = useCallback((entryId: ActEntryId) => {
     const quickAction = actEntryToQuickAction(entryId);
     const preset = quickAction !== null
       ? quickActionToPreset(quickAction as QuickActionLabel, activeParentType)
       : null;
     handleAction('reply', activeMessageId ?? '', preset);
   }, [activeMessageId, activeParentType, handleAction]);
   ```
2. `handleBoardActSelectBoxType(entryId)` becomes `enterBoxForActEntry(entryId); setBoardActVisible(false);` —
   **byte-equivalent behavior** (the board-Act anchor tests stay green).
3. `handleRefereeMove` (the leaf `onMove` payload is `MoveSuggestion`; the surface receives `(move, ctx)` via the
   existing Stack→Card binding) is **swapped** to:
   ```ts
   const handleRefereeMove = useCallback(
     (move: MoveSuggestion, _ctx: { activeMessageId: string | null }) => {
       enterBoxForActEntry(move.actEntryId);  // recovery routes (branch_tangent) route here too
     },
     [enterBoxForActEntry],
   );
   ```
   The v1 `null`-quickAction early-return is dropped: every constructive zone-3 entry
   (`ask_source`/`ask_quote`/`add_evidence`/`narrow`/`concede`/`confirm`/`challenge`/`branch_tangent`/`synthesize` —
   the `PADDING_POOL`, `openIssueModel.ts:534-544`) has a non-null `actEntryToQuickAction`; `enterBoxForActEntry`
   correctly opens the composer with no forced preset for the (non-padded) `reply`-class entry, identical to the board
   Act path. The leaf `RefereeCardView.onMove` signature is **unchanged** (REF-003 "handler-swap contract,"
   `docs/reviews/REF-003.md:171-179`).

The `ActPopout` / `actPopoutModel` files are **not edited** — only consumed. No pre-focus prop is added.

**The issue's full detail + raw keys.** REF-001's Inspect contract (REF-001 §"Inspect contract", lines 371-378) makes

- **Raw provenance** (a clearly-diagnostic sub-section, Inspect-only): `rawIssueId`, each `rawObservationCodes` entry,

The fixed-7-section InspectPopout core (`inspectPopoutModel.ts`, `InspectPopout.tsx`) is **unchanged** — its anchor
suite stays green. The `inspectContentBuilder.ts` change is **additive only** (`buildInspectOpenIssueDetail` is a new

considered and **declined** (it would modify `buildInspectContent` and risk the inspect anchor tests for no doctrinal
gain — the issue's plain summary now lives in the overlay's plain-language top block).

## Recovery-route conversion (surface behavior)

REF-002 already guarantees the model half: `nextBestMoves ⊆ buildActPopout` survivors, and a gate-removed suggestion
surfaces the `branch_tangent` recovery route (`isRecoveryRoute: true`, `recoveredFromCode` set) or is **dropped** —
never the invalid action (`deriveNextBestMoves`, `openIssueModel.ts:847-868`). REF-004's job is the **surface
behavior**:

- **An engine-invalid intended move is never rendered as an active button.** The card renders only
  `issue.nextBestMoves`; since that list already excludes gate-removed entries, there is no "click → rejected" path.
  RefereeCardView's zone-3 already renders every entry **enabled** (`accessibilityState={{ disabled: false }}`,
  `RefereeCardView.tsx:170`) — correct, because every entry is a survivor.
- **The recovery offer reads as a constructive move, never a rejection.** A recovery `MoveSuggestion` carries
  `actEntryId: 'branch_tangent'` and its normal Act label **"Open a side issue"** (`label` read from the Act entry
  definition, `actPopoutModel.ts:307`). The card shows that label verbatim — it never says "invalid," "rejected," or
  names the move that was removed. The matrix's "Branch this side issue / Narrow / Ask source" recovery copy is exactly
  the normal Act-entry label of the engine-valid alternative.
- **The recovery routes through the same Act flow.** Tapping it calls `handleRefereeMove(move)` →
  `enterBoxForActEntry('branch_tangent')` → opens the `branch_tangent` box (preset `'branch'`). The engine is never
  bypassed: `branch_tangent` is itself a `buildActPopout` survivor; the recovery enters an **engine-valid** box.
- **The engine stays the sole gate.** Nothing in this path re-validates, resurrects, or overrides a move the engine
  rejected; the rejected move stays rejected and the user is handed an engine-valid alternative. This is the §1 "score
  never blocks; advisory only" rule expressed as UI.

## Go handoff

**Affordance.** A second optional affordance, **"Focus on board,"** renders on the Referee Card when
`onRefereeNavigate` is provided **and** `issue.targetNodeId != null` (no node → nothing to focus). It calls
`onRefereeNavigate('focus_on_board')`. The surface handler `handleRefereeFocusIssue`:

1. `setActiveMessageId(issue.targetNodeId)` + `setSelectionStatus('explicit')` + `setMicroMomentDismissed(true)` —
   the **jump** half (the existing activation path `handleGoJump` uses, `ArgumentGameSurface.tsx:1515-1532`).
2. `if (mode !== 'timeline') setMode('timeline')` — a lens dims **timeline** nodes, so the board must be in timeline
   view for the lens to read.
3. `setGoLens(issueStateToGoLens(issue.state))` — the **filter** half (the existing `setGoLens` path the GoPopout's
   `onSelectLens` uses, `ArgumentGameSurface.tsx:2147`).

**The Go model addition.** `issueStateToGoLens` (and the frozen `ISSUE_STATE_TO_GO_LENS` map) is the **single new
pure export** in `goPopoutModel.ts`. It maps each of REF-002's 8 `IssueState`s onto one of the **existing**
timeline-meaningful Go lenses — `evidence` (→ `evidence_requested`), `unresolved` (→ `needs_response`), or `active_path`
(topology). **No new `FocusLensId`, no `timelineDensityLensModel.ts` change, no new Go popout entry.** The actual
dimming is the shipped `applyTimelineLens` (`timelineDensityLensModel.ts:571-593`), whose `LensEmphasis` is only
`'bright' | 'dimmed'` — **a lens structurally cannot remove a node.** "Go always dims, never hides" is preserved by
construction.

This is the minimal addition the card asks for: it reuses the shipped lens vocabulary and the shipped dim projection;
it only adds the issue-state→lens map so the active issue picks the right existing lens. The `buildGoPopout` gate/entry
logic is untouched.

## Edge cases

- **No active issue (`refereeCardIssue == null`)** → no Referee Card mounts (REF-003); the `InspectOpenIssueDetail`
  overlay is gated on `refereeCardIssue != null` → renders nothing; the new affordances never appear.
- **Observer / own-bubble role** → `nextBestMoves` is empty (REF-002 §"Edge cases"); zone 3 collapses; **no Act
  routing fires** (no buttons). "View details" still works (Inspect is read-only for everyone). "Focus on board" works
  iff `targetNodeId != null`. Observer asking source via the rail set is a REF-006-RAIL concern, not this card.
- **`onRefereeNavigate` omitted** (every REF-003 direct-render test, and any caller that does not opt in) → the
  secondary row does not render; the card is byte-identical to REF-003. Pinned by re-running the REF-003 suites.
- **`issue.targetNodeId == null`** → "Focus on board" affordance is not rendered (nothing to jump to); "View details"
  still renders. `handleRefereeFocusIssue` is never reachable with a null target.
- **Recovery-route move tapped** → routes through `enterBoxForActEntry('branch_tangent')` to the engine-valid branch
  box; never a rejected post (§"Recovery-route conversion").
- **Gate-removed suggestion with `branch_tangent` also gated out** (e.g. observer) → REF-002 **drops** the suggestion;
  zone 3 simply shows fewer (or zero) buttons. No fabricated action.
- **Cards (stack) mode when "Focus on board" is pressed** → `handleRefereeFocusIssue` switches to timeline first, so
  the lens has nodes to dim; the jump still selects the target node in either mode.
- **Lens matches zero nodes** → `applyTimelineLens` returns `isEmpty: true` with **every node bright** (the board is
  never silently emptied; `timelineDensityLensModel.ts:584-592`). The Go popout's existing empty-note copy applies.
- **Raw provenance empty** (`refereeObservations` and `userAllegations` both empty, e.g. a teaching-state issue) →
  `hasRawProvenance: false`; the overlay's diagnostic block renders the one-line "No raw provenance on this issue."
  note (never a blank section, never a raw echo).
- **Family-J / sensitive observation in the input** → already gated out of `refereeObservations` by REF-002's
  `HUB_NON_PRODUCTION_FAMILIES` re-assertion (`openIssueModel.ts:556-580`); it can never reach
  `rawObservationCodes`. Person-directed `userAllegations` carry `composer_only` / `moderator_only` visibility
  (REF-002); the overlay surfaces allegation `sourceCode`s as **diagnostic raw provenance in Inspect only** — never on
  the public card.
- **Heat / popularity tries to influence the lens or the routing** → it does not: `issueStateToGoLens` reads only the
  procedural `IssueState`; the routing reads only `move.actEntryId`. No field reads engagement, views, virality, or a
  strength band.
- **Concurrent edit / late classifier arrival** → the issue re-derives each render (REF-003); the overlay + affordances
  re-read the current `refereeCardIssue`. Nothing blocks or delays a post.

## Risks

- **`enterBoxForActEntry` extraction silently changes board-Act behavior.** Mitigation: the extraction is

  `handleBoardActSelectBoxType` keeps its trailing `setBoardActVisible(false)`. The UX-001.4 board-Act handler tests are
  named anchors that must stay green; the implementer runs them before/after the extraction.
- **Import-cycle / layering smell** from `goPopoutModel → refereeLoop (type)` and `inspectContentBuilder → refereeLoop
  (value: label maps)`. Mitigation: the `goPopoutModel` edge is **type-only** (erased at build → no runtime require).
  The `inspectContentBuilder` edge is value (frozen label-map constants) but the runtime chain
  `inspectContentBuilder → openIssueModel → {actPopoutModel, suggestedMovesModel}` never re-imports
  `inspectContentBuilder`, so it is **acyclic**. The implementer verifies with `npx tsc --noEmit` (a real cycle would
  surface) and, if jest reports a cycle warning, imports directly from `openIssueModel.ts` rather than the barrel.
- **Plumbing churn across 4 files** for `onRefereeNavigate`. Mitigation: it is a single optional pass-through prop per
  level, mirroring the shipped `onRefereeMove` plumbing exactly; when omitted the render is byte-equivalent (REF-003
  direct-render tests prove it).
- **Referee Card crowding.** Two new affordances + 2-3 move buttons could overflow on phone. Mitigation: the affordance
  row is visually subordinate (secondary color, smaller) and wraps; "Focus on board" hides when `targetNodeId == null`.

- **Existing tests that may need updating:** none expected — all changes are additive and the REF-003 / Act / Inspect /
  Go anchor suites are consume-only. If `RefereeCardView.test.tsx` asserts an exact child count, the implementer adds
