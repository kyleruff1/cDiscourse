**Release:** 6.7

## §0 — Card-vs-reality discrepancies (read first)

Every prior card surfaced several; IX-003 is no different. The card body is short
("Arrow L/R moves active node. Home/End → root/latest on web. Accessibility roles +
selected state. Nodes expose type, ordinal, strength, branch, active.") and the
real `ArgumentTimelineMap.tsx` is already partway there. Concrete deltas:

1. **"Buttons have accessibility roles and selected states" — already half-done.**
   `NodeDot`'s `<Pressable>` already has `accessibilityRole="button"`,
   `accessibilityState={{ selected: node.isActive }}`, and an `accessibilityLabel`
   (`ArgumentTimelineMap.tsx` lines 239–243). The control chips (`timeline-prev`,
   `timeline-next`, `timeline-jump-latest`, `timeline-jump-root`,
   `timeline-toggle-mode`) and the info button already have
   `accessibilityRole="button"` but **no `accessibilityState`** — they are
   stateless actions, so `selected` does not apply. IX-003's "selected state"
   acceptance line is satisfied by the *node* Pressables; the control chips need
   no `selected`. The card over-states the remaining work here. IX-003's real
   job for "roles + selected state" is: (a) keep the node contract, (b) add
   `accessibilityState={{ disabled }}` to Prev/Next when at an end (currently
   they silently no-op — see discrepancy 6), and (c) prove the contract with
   tests, which do not exist today.

2. **"Nodes expose type, ordinal, strength, branch, active" — strength and
   branch are MISSING from the label.** The accessibilityLabel is built in
   `buildArgumentTimelineMap` Pass 4 (`argumentGameSurfaceModel.ts` lines
   1369–1381). It currently emits: `Message <ord> of <total>`, `kindLabel`
   (type), `actorLabel`, `side <X>`, relative time, `active`, `latest`,
   `junction with N replies`, `detached`. It does **not** mention
   `standingBand` (strength) or `branchId`/`lane` (branch). VG-004 then appends
   a navigation fragment in `NodeDot` ("active move" / "on the active path" /
   "has an attached source") and an "opening claim" suffix for root. **So
   IX-003 must add strength + branch to the screen-reader contract** — this is
   the substantive a11y work, not the keyboard work.

3. **Active-node-change path is single, clean, and reusable.** Tap →
   `NodeDot.onPress` → `handleNodeTap` (`ArgumentTimelineMap.tsx` line 568) →
   `onActivate(messageId)` → `ArgumentGameSurface.handleActivate` → `setActiveMessageId`.
   Arrow keys MUST drive the **same** `onActivate` callback. No new state path.
   Note `handleNodeTap` has an SC-004 branch (when `onSelectTarget` is wired it
   also sets the dock target). IX-003's arrow keys should call `onActivate`
   ONLY (move the active node); they must NOT open the SC-004 dock or the
   SC-002 popover. Enter/Space is the key that opens detail (see §"API").

4. **No existing keyboard handling anywhere near the timeline.** A repo-wide
   grep for `onKeyDown` / `nativeEvent…key` / `tabIndex` in `src/` returns
   zero hits inside the argument feature. IX-003 introduces the first keyboard
   layer. There is no prior pattern to copy; the design below establishes one.

5. **Pure prev/next helpers already exist.** `timelineMapPrevId(map, id)` and
   `timelineMapNextId(map, id)` (`argumentGameSurfaceModel.ts` lines 1538–1552)
   already return the chronological neighbour by `map.nodes` index. IX-003's
   Arrow Left/Right is exactly chronological prev/next, so the new
   `keyboardNavigationModel.ts` traversal function will **delegate to these**
   rather than re-deriving. Home/End map to `map.rootMessageId` /
   `map.latestMessageId`, both already on the model.

6. **Prev/Next/Latest controls silently no-op at the ends.**
   `ArgumentGameSurface.handlePrev`/`handleNext` call `getPreviousMessageId` /
   `getNextMessageId` and only `setActiveMessageId` if a neighbour exists.
   There is no disabled visual or a11y state. IX-003 will compute
   `isAtFirst` / `isAtLatest` in the keyboard model and thread it so Prev/Next
   can render `accessibilityState={{ disabled }}` and a dimmed style. This is a
   small scope addition justified by the a11y acceptance line; it is NOT a
   behaviour change (the buttons already no-op).

7. **The card scopes Home/End "on web" but arrows are unqualified.** Web
   keyboard events require RN-web-specific capture (see §"API"). On native
   there is no hardware keyboard in the normal phone flow, so ALL key handling
   (arrows included) is effectively web-only. The native path is unchanged and
   complete via tap + VoiceOver/TalkBack rotor. The design treats the entire
   key layer as a `Platform.OS === 'web'` enhancement; this matches the
   accessibility-targets skill ("On native, none of the above applies").

8. **`accessibility-targets` skill names richer key bindings than the card.**
   The skill lists Up/Down (parent / first child), Enter/Space (open popover),
   Esc (close). The card only names L/R + Home/End. **This design implements
   exactly the card's acceptance criteria (L/R/Home/End) plus Enter/Space/Esc
   as a low-cost, in-grammar addition** (Enter/Space simply triggers the
   existing tap-to-open-detail effect; Esc closes whatever overlay is open).
   Up/Down (parent / first child) is explicitly deferred — see §"Out of scope"
   — because it needs a tree-walk the card did not budget effort for and
   risks colliding with vertical page scroll on web.

No conflicts with doctrine or v1 scope were found. The card is buildable as
written; §0 items 2 and 6 are the only places the design adds beyond the
literal card text, and both are required by the accessibility acceptance line.

## Goal (one paragraph)

IX-003 makes the horizontal argument timeline (`ArgumentTimelineMap.tsx`)
fully operable by keyboard and legible to screen readers, without changing how
it works for touch. On web, Arrow Left/Right walk the active node backward and
forward in chronological order, and Home/End jump to the root and the latest
move — all routed through the existing single `onActivate` active-node-change
path so keyboard and tap stay identical. Every timeline node Pressable already
carries `accessibilityRole="button"` and `accessibilityState={{ selected }}`;
IX-003 completes the screen-reader contract so each node also announces its
**strength band** (the existing `standingBand`, exposed in plain language as a
gameplay standing — never a verdict, per cdiscourse-doctrine §1) and its
**branch** (mainline vs. a side branch). The keyboard traversal logic and the
accessibility-label construction are extracted into a pure, React-free model
(`keyboardNavigationModel.ts` + a label helper) so the whole feature is
unit-testable without a renderer, matching this repo's source-scan +
pure-helper test discipline. No new dependency: web key capture uses RN-web's
existing `onKeyDown` / `tabIndex` props on `<View>` / `<Pressable>`.

## Data model

**No new persisted data model. No SQL. No migration.** IX-003 is pure UI +
pure-TS model.

Three new pure-TypeScript types live in the new
`src/features/arguments/keyboardNavigationModel.ts`:

```ts
/** The four keys IX-003 binds plus the two open/close keys. */
export type TimelineNavKey =
  | 'ArrowLeft'
  | 'ArrowRight'
  | 'Home'
  | 'End'
  | 'Enter'
  | ' '          // Space — KeyboardEvent.key for the spacebar is a single space
  | 'Escape';

/** The set of strings KeyboardEvent.key may produce that we treat as one of
 *  the above. Space arrives as ' ' (or, rarely, 'Spacebar' on old engines). */
export const TIMELINE_NAV_KEYS: ReadonlyArray<string> = Object.freeze([
  'ArrowLeft', 'ArrowRight', 'Home', 'End', 'Enter', ' ', 'Spacebar', 'Escape',
]);

/** Outcome of pressing a nav key. Pure — the component applies the effect. */
export type TimelineNavEffect =
  | { type: 'none' }                              // key not handled / nothing to do
  | { type: 'activate'; messageId: string }       // move the active node
  | { type: 'open_detail'; messageId: string }    // Enter/Space on the active node
  | { type: 'close_overlay' };                    // Escape

export interface TimelineNavInput {
  /** The raw KeyboardEvent.key value. */
  key: string;
  /** Current active node id (map.activeNode?.messageId ?? null). */
  activeMessageId: string | null;
  /** The built timeline map — the source of node order, root, latest. */
  map: ArgumentTimelineMapModel;
  /** True when an overlay (SC-002 popover or SC-004 dock) is open; only
   *  then does Escape produce 'close_overlay'. */
  hasOpenOverlay: boolean;
}
```

The accessibility label work reuses the existing `TimelineStandingBand` union
and the existing `STANDING_BAND_SOFT_LABEL` map (`standingBandCopy.ts`); no new
band type is introduced. One small new interface for the label builder:

```ts
/** Minimal node shape the label builder needs — a structural subset of
 *  ArgumentTimelineMapNode so the helper can be tested with tiny fixtures. */
export interface NodeAccessibilityInput {
  ordinal: number;
  totalNodes: number;
  kindLabel: string;
  sideLabel: string;          // 'Aff' | 'Neg' | 'Obs' | 'Mod' | '—'
  standingBand: TimelineStandingBand;
  branchLabel: string;        // 'mainline' or 'side branch' — see helper below
  isActive: boolean;
  isLatest: boolean;
  isRoot: boolean;
  isDetached: boolean;
  isJunction: boolean;
  junctionChildCount: number;
  relativeOrAbsoluteTime: string;
}
```

## API / interface contracts

### `keyboardNavigationModel.ts`

```ts
/** True when `key` is one IX-003 binds. Cheap guard for the onKeyDown handler. */
export function isTimelineNavKey(key: string): boolean;

/**
 * Pure traversal. Given the current map + active node + a key, return the
 * effect the component should apply. Never throws. Never mutates input.
 *
 *  - ArrowLeft  → activate the chronological PREVIOUS node
 *                 (map.nodes[idx-1]); 'none' if already at index 0.
 *  - ArrowRight → activate the chronological NEXT node
 *                 (map.nodes[idx+1]); 'none' if already at the last index.
 *  - Home       → activate map.rootMessageId; 'none' if root is already
 *                 active or rootMessageId is null.
 *  - End        → activate map.latestMessageId; 'none' if latest is already
 *                 active or latestMessageId is null.
 *  - Enter / Space → 'open_detail' for the active node; 'none' if no active
 *                 node. (Mirrors the second-tap / info-icon effect.)
 *  - Escape     → 'close_overlay' iff hasOpenOverlay; else 'none'.
 *  - anything else → 'none'.
 *
 * When activeMessageId is null, ArrowLeft/Home behave as "go to root",
 * ArrowRight/End behave as "go to latest" — so a keyboard user with no
 * selection yet still gets a sensible first move.
 */
export function resolveTimelineNavEffect(input: TimelineNavInput): TimelineNavEffect;

/**
 * Build the full plain-language screen-reader label for one node.
 * Order: "<kindLabel> on side <sideLabel>, position <ordinal> of <total>,
 *         <strength soft label>, <branchLabel>[, opening claim]
 *         [, currently active][, latest move][, junction with N replies]
 *         [, detached — parent unavailable], posted <time>".
 *
 * The strength fragment is STANDING_BAND_SOFT_LABEL[band] — e.g.
 * "Strongly supported", "Needs work", "Neutral", "No reading yet". It is
 * gameplay standing, never a verdict (see §"Doctrine self-check").
 * `side <X>` is omitted when sideLabel is '—'.
 */
export function buildNodeAccessibilityLabel(input: NodeAccessibilityInput): string;

/**
 * Derive the plain-language branch descriptor for a node.
 *  - isDetached            → 'detached from the main thread'
 *  - lane === 0            → 'on the main line'
 *  - lane !== 0            → 'on a side branch'
 * (Lane numbers themselves are never read aloud — "+2"/"-1" mean nothing
 * to a listener. The mainline-vs-side distinction is what matters.)
 */
export function deriveBranchLabel(node: {
  lane: number;
  isDetached: boolean;
}): string;
```

### `ArgumentTimelineMap.tsx` — keyboard handler shape (web-only)

```ts
// Inside ArgumentTimelineMap, web-only. No new prop on the public Props
// interface — the handler reads existing props (`map`, `onActivate`,
// `selectedTarget`, `onSelectTarget`) and local state (`popoverMessageId`).
const handleKeyDown = useCallback((e: { key: string; preventDefault: () => void }) => {
  if (Platform.OS !== 'web') return;
  if (!isTimelineNavKey(e.key)) return;
  const effect = resolveTimelineNavEffect({
    key: e.key,
    activeMessageId: map.activeNode?.messageId ?? null,
    map,
    hasOpenOverlay: popoverMessageId !== null || selectedTarget != null,
  });
  if (effect.type === 'none') return;
  e.preventDefault();           // arrows must not scroll the page
  if (effect.type === 'activate') onActivate(effect.messageId);
  else if (effect.type === 'open_detail') handleInfoTap(effect.messageId);
  else if (effect.type === 'close_overlay') {
    setPopoverMessageId(null);
    onSelectTarget?.(null);
  }
}, [map, onActivate, popoverMessageId, selectedTarget, onSelectTarget, handleInfoTap]);
```

On the outer root `<View>` (web only — guarded via `Platform.select`):
`onKeyDown={handleKeyDown}` and `{...(Platform.OS === 'web' ? { tabIndex: 0 } : {})}`.
RN-web forwards `onKeyDown` and `tabIndex` to the underlying DOM node; on
native these props are simply not passed. No new dependency — `onKeyDown`
and `tabIndex` are part of RN-web's `View` prop surface today.

### Accessibility contract per node (after IX-003)

```
accessibilityRole = "button"                       (unchanged)
accessibilityState = { selected: node.isActive }   (unchanged)
accessibilityHint  = (unchanged — tap-to-activate / tap-again-to-open)
accessibilityLabel = buildNodeAccessibilityLabel({...}) + VG-004 fragment
```

Example label for an active, well-supported rebuttal on the Aff side, 4th of
12, on a side branch:

> "rebuttal on side Aff, position 4 of 12, Well supported, on a side branch,
> currently active, posted 8m ago, active move"

(The trailing "active move" is the VG-004 fragment; it is redundant with
"currently active" but harmless and already shipped — IX-003 does not remove
it to avoid touching VG-004's contract.)

## Edge cases

The implementer MUST handle every one of these:

- **Empty timeline (0 nodes).** `ArgumentTimelineMap` already early-returns the
  empty placeholder before any node renders. `resolveTimelineNavEffect` must
  still return `{ type: 'none' }` for every key when `map.nodes.length === 0`
  (no crash). Tested.
- **Single node.** ArrowLeft/ArrowRight → `'none'` (no neighbour). Home/End →
  `'none'` (the single node is both root and latest and already active).
  Enter/Space → `open_detail`. Tested.
- **Active node is the root.** ArrowLeft → `'none'`; Home → `'none'` (already
  there); ArrowRight → next; End → latest.
- **Active node is the latest.** ArrowRight → `'none'`; End → `'none'`;
  ArrowLeft → previous; Home → root.
- **No active node (`activeMessageId === null`).** ArrowLeft/Home → activate
  root; ArrowRight/End → activate latest. A keyboard user must always be able
  to get a selection from a cold start.
- **Detached nodes.** Detached nodes ARE in `map.nodes` (the map never drops
  them) so ArrowLeft/Right will land on them in chronological order — correct,
  they are real messages. `deriveBranchLabel` returns
  "detached from the main thread" for them. `Home`/`End` are unaffected
  (root/latest are not detached by construction). Tested.
- **Active node disappeared (admin removal mid-session).**
  `ArgumentGameSurface` already snaps `activeMessageId` to latest in a
  `useEffect`. The keyboard model reads `map.activeNode` which is rebuilt each
  render, so by the time a key is pressed the active node is valid again.
  No extra handling needed; documented so the implementer does not add any.
- **ArrowLeft/Right at the ends.** `'none'` — the active node does NOT wrap.
  No wrap-around (a listener pressing Right repeatedly should land on the
  latest and stop, not jump back to root). Tested explicitly.
- **Web vs native.** All key handling is `Platform.OS === 'web'`-guarded. On
  native, `handleKeyDown` returns immediately, `tabIndex` is not passed, and
  the experience is unchanged (tap + VoiceOver/TalkBack rotor). The native
  path remains complete per `expo-rn-patterns` ("native path must still be a
  complete experience").
- **Space key bubbling.** On web, Space inside a focused scroll container
  triggers page-down. The handler `preventDefault()`s Space when it resolves
  to a non-`none` effect, so Space opens detail instead of paging. If the
  active node is null Space → `none` and is NOT preventDefaulted (let the
  browser do its default — there is nothing to open).
- **Escape with nothing open.** `hasOpenOverlay === false` → `'none'`, not
  preventDefaulted, so a parent modal's Esc handling is not swallowed.
- **Reduce-motion interaction.** Keyboard navigation triggers the existing
  auto-scroll-to-active `useEffect` (`scrollRef.current.scrollTo({ animated:
  true })`). Under reduce-motion this animated scroll is a motion. **Scope
  decision:** IX-003 does NOT change the auto-scroll — it is system scroll
  inertia, which the accessibility-targets skill explicitly says to KEEP
  ("Scroll inertia — system-controlled"). The implementer must NOT add a
  reduce-motion branch here; documented to prevent over-engineering.
- **Focus management after activate.** When a key moves the active node, DOM
  focus stays on the timeline group `<View>` (it owns the `onKeyDown`). The
  newly-active node is announced because its `accessibilityState.selected`
  flips and the auto-scroll brings it into view. IX-003 does NOT move DOM
  focus onto individual node Pressables on arrow press (that would fight the
  group handler). Tab still moves focus through individual nodes; arrows move
  the *selection* within the group. This is the standard "roving selection in
  a composite widget" pattern and is the deliberate design — documented so the
  implementer does not add `.focus()` calls.
- **Concurrent edits / offline.** Not applicable — IX-003 makes no network
  call and no write. New messages arriving simply rebuild `map`; the active
  node id is preserved across the rebuild because it is keyed by id.
- **Permission-denied paths.** Not applicable — keyboard nav only changes
  local selection; it never posts. Observer (read-mode) users get the exact
  same keyboard nav as participants.
- **Doctrine edge case — "what if the screen reader makes strength sound like
  a verdict?"** It must not. The label uses `STANDING_BAND_SOFT_LABEL`
  ("Strongly supported", "Needs work", "Neutral", "No reading yet"), never
  "correct"/"true"/"winning". A ban-list test asserts this (see §"Test plan").
- **Doctrine edge case — "does Home/End or arrow nav touch the score?"** No.
  Keyboard nav only calls `onActivate` (selection). It cannot post, cannot
  flag, cannot score. Strength is read-only exposure of the already-computed
  `standingBand`.

## Dependencies (cards / docs / files)

- **Reads** `ArgumentTimelineMapModel` from `argumentGameSurfaceModel.ts`
  (`nodes`, `activeNode`, `rootMessageId`, `latestMessageId`) at the
  `buildArgumentTimelineMap` output — IX-003 consumes the model, never
  rebuilds it.

## Risks

- **RN-web `onKeyDown` / `tabIndex` prop surface.** RN-web forwards these to
  the DOM today; they are not in the typed `ViewProps` of `react-native`'s
  TypeScript definitions. The implementer will need to either spread them via
  a `Platform.OS === 'web'` object (`{...(Platform.OS === 'web' ? { onKeyDown,
  tabIndex } : {})}`) or cast the props object. Casting with a narrow,
  commented `as` is acceptable per the TS conventions ("no `any` unless
  unavoidable and explicitly commented" — here it is a documented RN-web
  prop). **Mitigation:** prefer the spread-object form, which needs no cast;
  the spread object can be typed as `Record<string, unknown>`.
- **`preventDefault` on Arrow keys.** Preventing default on arrows stops page
  scroll, which is what we want INSIDE the timeline — but the handler is on
  the timeline group `<View>`, so it only fires when that group has focus.
  Risk: if a future card nests a focusable scrollable inside the timeline,
  arrow keys could be swallowed. **Mitigation:** the handler only
  `preventDefault`s when the effect is non-`none`; benign keys pass through.
- **Tab order with many nodes.** Adding `tabIndex={0}` to every node Pressable
  on a 250-message timeline creates 250 tab stops. **Mitigation:** the card's
  primary keyboard model is Arrow-key roving selection on the *group* (one tab
  stop). Decision: give the **group** `tabIndex={0}` and the individual nodes
  `tabIndex={-1}` (focusable programmatically / reachable by screen-reader,
  but NOT in the Tab sequence). This keeps the Tab sequence short and matches
  the ARIA composite-widget pattern. The implementer must use `-1` on nodes,
  `0` on the group — call this out in the design as the chosen value.
- **Existing tests that might need updating.** Any current test that asserts
  the *exact* string of a node `accessibilityLabel` will break when strength +
  branch are added. A grep for `accessibilityLabel` assertions in
  `__tests__/` covering `argumentGameSurfaceModel` / timeline should be run
  first; the implementer updates those expected strings to include the new
  fragments. This is a known, contained churn — the design flags it so it is
  not a surprise.
- **No live keyboard event in Jest.** Jest/JSDOM does not simulate real
  keypresses without RTL. The test plan deliberately tests the *pure model*
  (`resolveTimelineNavEffect`) directly and *source-scans* the wiring — it
  does NOT try to fire synthetic key events. This matches repo discipline but
  means the actual `onKeyDown` → effect → `onActivate` plumbing is verified by
  the **browser verification checklist**, not by an automated test. Accept
  this; it is the repo norm for UI wiring.
- **No migration, no Edge Function, no operator deploy** — zero deploy risk.

## Out of scope

Explicit — to stop scope creep when the implementer reads this:

- **Up/Down arrow keys (parent / first child).** The `accessibility-targets`
  skill lists them; the card does not, and the card's effort is M. Up/Down
  needs a parent/child tree-walk and risks colliding with vertical page
  scroll. Deferred to a follow-up (could be folded into IX-002 or a new card).
- **Mini-map keyboard nav** — IX-002's job.
- **Zoom / density keyboard shortcuts** — IX-001's job.
- **Moving DOM focus onto individual nodes on arrow press** — IX-003 uses
  group-level roving selection, not per-node focus. (See §"Edge cases".)
- **Reduce-motion change to the auto-scroll** — auto-scroll is system scroll
  inertia, explicitly kept per the accessibility skill.
- **Keyboard nav for the Stack mode** (`ArgumentBubbleStack`). IX-003 is the
  *timeline* card. Stack already has Prev/Next buttons; a keyboard layer for
  Stack would be a separate card.
- **`announceForAccessibility` on every active-node change.** The
  accessibility skill says announcements should be sparing; the
  `accessibilityState.selected` flip + the node label already convey the
  change. No live-region spam. (The existing branch-expand announcement in
  `ArgumentTimelineMap` is untouched.)
- **New control buttons or visible keyboard hints UI.** IX-003 adds no new
  on-screen affordance; it makes the existing ones operable. (A visible
  "keyboard shortcuts" legend could be a nice future polish card.)
- **Touch gesture changes.** The card title says "mobile touch" but touch
  already works (tap-to-activate ships). IX-003 does not change touch.

## Doctrine self-check

- **cdiscourse-doctrine §1 — no truth labels; score never blocks posting; no
  service-role.** The screen-reader strength fragment is
  `STANDING_BAND_SOFT_LABEL[band]` — "Strongly supported" / "Needs work" /
  "Neutral" / "No reading yet" — a gameplay standing band, never "correct" /
  "true" / "winner". A ban-list test enforces it. Keyboard nav only changes
  *selection*; it cannot post, score, or block anything. No service-role, no
  Supabase call, no migration.

  exposes `standingBand` (strength) and branch (mainline/side). It does NOT
  read or announce `heat`, `temperatureBand`, reply count, or any engagement
  signal as "strength". The branch label is topology, not popularity.
- **cdiscourse-doctrine §4/§7 — no AI calls from the app.** None. Pure model +
  RN-web key handling.
- **cdiscourse-doctrine §5 — rules engine is sacred.** Untouched. IX-003 does
  not import `engine.ts`.
- **cdiscourse-doctrine §9 — plain language.** The accessibility label emits
  only plain English ("rebuttal", "on side Aff", "Strongly supported",
  "on a side branch", "currently active", "posted 8m ago"). No internal codes
  (`standingBand` enum keys, `branchId`, lane numbers) are ever spoken.

- **accessibility-targets — keyboard nav (IX-003 is the named card).** Arrow
  L/R = chronological prev/next; Home = root; End = latest; Enter/Space = open

  Hit target stays 44×44 (`TIMELINE_NODE_SIZE = 44`, unchanged). Tab order is

## Browser verification checklist (Expo web)

Run `npx expo start --web`, open a debate room, switch to Timeline mode, and
verify by keyboard + screen reader. This card gets a live browser check before
merge — these are the concrete things the human verifies:

- [ ] Click once anywhere on the timeline frame so it has focus (or press Tab
      until the timeline group is focused — there should be ONE tab stop for
      the whole node group, not one per node).
- [ ] **Arrow Right** moves the active-node ring/glow one node later
      (chronologically); the timeline auto-scrolls to keep it in view.
- [ ] **Arrow Left** moves the active node one node earlier.
- [ ] Arrow Right repeatedly stops at the latest node (no wrap to root).
      Arrow Left repeatedly stops at the root node (no wrap to latest).
- [ ] **Home** jumps the active node to the root (opening claim).
- [ ] **End** jumps the active node to the latest move.
- [ ] Arrow keys do NOT scroll the whole web page while the timeline is
      focused (page stays put; only the timeline scrolls to the active node).
- [ ] **Enter** or **Space** on a focused timeline opens the per-node
      popover/detail for the active node; Space does not page-down.
- [ ] **Escape** closes an open popover / action dock and does nothing when
      nothing is open.
- [ ] Turn on a screen reader (VoiceOver on Safari, NVDA on Windows). Focus a
      node and confirm it announces, in plain English: the **type**
      (e.g. "rebuttal"), the **side**, the **ordinal** ("position 4 of 12"),
      the **strength** ("Strongly supported" / "Needs work" / etc. — never
      "correct" or "true"), the **branch** ("on the main line" / "on a side
      branch"), and the **active state** ("currently active") when it is the
      active node.
- [ ] The focused timeline group shows a visible focus outline (browser
      default ring is acceptable) so a keyboard user can see where focus is.
- [ ] The active node's `selected` state is announced by the screen reader
      (it should say "selected" / "pressed" for the active node).
- [ ] Prev / Next control chips appear dimmed and are announced as "disabled"
      when the active node is at the first / last position respectively.
- [ ] On a native build (Expo Go / simulator) the room still works by tap and
      VoiceOver/TalkBack rotor — no regression, no crash from the web-only
      props.
