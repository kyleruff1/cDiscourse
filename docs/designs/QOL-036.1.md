## §0 — Card-vs-reality discrepancies

The card body (issue #270) reads:

> The pill consumes the following composition-layer mutations (introduced by MCP-CAT-001):
> - `evidence_applicability_disputed` → pill state becomes "applicability disputed"
> - `corroborating_document_attached` → pill state becomes "corroborated"
> - `evidence_debt_opened` → pill state becomes "source requested"
> - `evidence_debt_resolved` → pill state becomes "source supplied"

The card refers to **the payment-evidence pill** (singular). The repo, however, ships **three** orthogonal chips on every evidence artifact — and a payment artifact may render up to all three at once:

| Axis | Chip | State enum | Pure-model derive function |
|---|---|---|---|
| Existence / source-chain (EV-002) | `ReceiptChip` | `SourceChainStatus` | `summarizeArtifactsForReceiptChip` |
| Obligation / debt (EV-003) | `EvidenceDebtChip` | `EvidenceDebtStatus` | `getNodeEvidenceDebtChip` |
| Applicability (QOL-037) | `ApplicabilityChip` | `ApplicabilityStatus` | `summarizeApplicabilityChip` |

The four card-body labels do **not** invent a new fourth chip; they map cleanly onto **two existing axes**:

| Card-body label | Existing axis | Existing enum value |
|---|---|---|
| "applicability disputed" | Applicability (QOL-037) | `applicability_disputed` |
| "corroborated" | Applicability (QOL-037) | `applicability_supported` |
| "source requested" | Obligation (EV-003) | `requested` |
| "source supplied" | Obligation (EV-003) | `supplied` |

This is the most faithful reading of the card body against the shipped code, and it preserves QOL-036's §15 *Out of scope* boundary: "The applicability status enum ... [is] QOL-037." This card is a **consumer** of QOL-037's and EV-003's already-shipped status enums; it does not invent a fourth enum or a new chip.

**This card therefore "pill" = "the set of axis chips a payment artifact already renders."** The integration wires composition-layer mutations into the layer-1 derivation flow so that classifier-detected applicability disputes / corroborations / source requests / source supplies surface on those existing chips without requiring a user-action `EvidenceResponseRecord` to land first.

No genuine card-vs-reality discrepancy; the card body is a faithful description of a consumer-only integration. The implementer reads §5 below for the exact mapping table.

## §3 — Problem statement

Today the three axis chips on a payment evidence artifact each derive from **layer-1 user-action signals**:

| Chip | Today's signal |
|---|---|
| `ApplicabilityChip` | `deriveApplicabilityStatus(responses, options)` — walks `EvidenceResponseRecord[]` (`dispute_applicability` choice opens a dispute; `accept` by the disputer closes it; `corroboratedByArgumentIds` from caller opens `applicability_supported`). |
| `EvidenceDebtChip` | `deriveEvidenceDebts(...)` — walks argument rows for `source_request` / `quote_request` / etc. tag codes. |
| `ReceiptChip` | `summarizeArtifactsForReceiptChip(artifacts)` — walks the artifact's url / quote / sourceText fields. |

The composition layer ships four mutation types whose **structural meaning is identical** to those three layer-1 signals — but the composition layer detects them from **classifier output** rather than requiring the user to also click an explicit Act-popout entry that emits a tag. Today nothing in the payment-evidence chip derivation reads `useSemanticReferee.getMutationsForMove(payment.argumentId)`, so a classifier-detected applicability dispute / corroboration / source request / supply leaves the chips static until a user also takes the layer-1 action.

The cost of not fixing this:

- The band-space-rent storyboard's promise "the system catches when A challenges B's payment date and flips the applicability chip on B's payment without B having to file a structured `dispute_applicability` response" is unmet (M-A challenges B's payment via classifier signal alone; chip stays `applicability_undisputed`).
- QOL-036's chain prompt explicitly hinted at this integration ("mutations driving evidence-pill state"); the chain prompt was never satisfied.
- The pattern this card establishes is the **prototype** for the broader sweep in §6.1 of the prior reconciliation — EV-001..EV-005, GAME-004..006, BR-001..004 all have the same structural shape (existing chip + new classifier-detected mutations). Doing it once correctly here gives the others a copy-paste template.

**Doctrine framing.** The composition layer is deterministic (no AI call in the production app — runs inside `useSemanticReferee.finalizeReady` after `classifyMove` returns). The mutations are structural enum values, never verdicts. This card simply extends the *signal sources* for two existing chip-status enums; it adds no new chip, no new enum value, no new verdict, no truth label, and no AI dependency from the consumer side.

## §4 — Data model & file changes

### §4.1 — New types (`src/features/evidence/paymentEvidencePillState.ts`)

```ts
/**
 * QOL-036.1 — Payment-evidence pill state derivation, consumer of COMP-001
 * mutations + QOL-037 applicability + EV-003 obligation derivations.
 *
 * Pure TypeScript. No React. No Supabase. No network. No async.
 *
 * Doctrine:
 *   - Adds NO new chip, NO new enum value, NO new component, NO new verdict.
 *   - Reads composition mutations as ADDITIVE signals on top of layer-1
 *     derivations; never replaces layer-1.
 *   - Mutations are structural enum values; this module never authors copy.
 *     All user-facing strings come from QOL-037 / EV-003's already-locked copy.
 */

import type { NodeVisualMutation } from '../semanticReferee/compositionTypes';
import type { EvidenceArtifact } from './evidenceModel';
import type {
  ApplicabilityChipContract,
  ApplicabilityStatus,
  EvidenceResponseRecord,
} from './evidenceApplicabilityModel';
import type {
  EvidenceDebt,
  EvidenceDebtStatus,
} from './evidenceDebtModel';

/**
 * Where each axis status was last produced from. Tracked so the test suite +
 * a future Inspect-§E read-view can show provenance, and so a debugger can
 * tell whether the chip flipped via layer-1 (user action) or layer-2
 * (classifier signal).
 *
 * `layer1` — derived from EvidenceResponseRecord[] or EV-003 tag walk only.
 * `layer2` — derived from a composition-layer mutation only.
 * `layer1_with_layer2_corroboration` — both signals agree.
 * `layer2_overrides_layer1` — chronological conflict; see PILL_STATE_CONFLICT_RULE.
 */
export type PaymentEvidencePillProvenance =
  | 'layer1'
  | 'layer2'
  | 'layer1_with_layer2_corroboration'
  | 'layer2_overrides_layer1';

/**
 * The composed pill state for one payment evidence artifact. Carries the two
 * axis statuses + provenance + the existing chip contracts. The renderer
 * reads `applicabilityChip` and `debtChipStatus` directly — the contracts are
 * QOL-037's and EV-003's already-locked output shapes.
 *
 * Existence axis (ReceiptChip) is NOT touched by this card — none of the four
 * mutations describe the source-chain. The receipt chip continues to derive
 * from `summarizeArtifactsForReceiptChip` over the artifact's url/quote/
 * sourceText fields.
 */
export interface PaymentEvidencePillState {
  /** The payment artifact this pill state is for. */
  readonly artifactId: string;
  /** The parent argument id (the move the payment is attached to). */
  readonly parentMoveId: string;
  /** The composed applicability status (after merging mutations + responses). */
  readonly applicabilityStatus: ApplicabilityStatus;
  /** The QOL-037 chip contract for the composed applicability status.
   *  When status is `applicability_undisputed`, isVisible: false. */
  readonly applicabilityChip: ApplicabilityChipContract;
  /** The composed obligation status, or null when no debt is open or
   *  resolved on this node. */
  readonly debtChipStatus: EvidenceDebtStatus | null;
  /** Provenance for the applicability axis. */
  readonly applicabilityProvenance: PaymentEvidencePillProvenance;
  /** Provenance for the obligation axis. null when debtChipStatus is null. */
  readonly debtProvenance: PaymentEvidencePillProvenance | null;
}

/**
 * The frozen four-row mapping table the card body specifies. Implementer
 * MUST reference this constant; tests assert its shape. Each entry names the
 * source mutation type, the target axis, the target enum value, and the
 * `targetMoveId` the mutation must address for the mapping to fire.
 *
 * Note on `targetMoveId`: the composition layer emits mutations targeting
 * specific moves (COMP-001 §6.3). For pill-state consumption the mutation
 * must target the move the payment artifact is attached TO (i.e. the
 * artifact's `argumentId`). A mutation targeting some OTHER move does not
 * flip THIS payment's chip — that mutation is for a different node's chips.
 */
export const MUTATION_TO_PILL_STATE: readonly {
  readonly mutation:
    | 'evidence_applicability_disputed'
    | 'corroborating_document_attached'
    | 'evidence_debt_opened'
    | 'evidence_debt_resolved';
  readonly axis: 'applicability' | 'obligation';
  readonly enumValue: ApplicabilityStatus | EvidenceDebtStatus;
  readonly targetSelector: 'parent_of_corroborating_move' | 'artifact_parent';
}[] = Object.freeze([
  Object.freeze({
    mutation: 'evidence_applicability_disputed',
    axis: 'applicability',
    enumValue: 'applicability_disputed',
    targetSelector: 'artifact_parent',
  }),
  Object.freeze({
    mutation: 'corroborating_document_attached',
    axis: 'applicability',
    enumValue: 'applicability_supported',
    // corroborating_document_attached fires on the CORROBORATING move, not on
    // the move that holds the payment artifact. The deriver walks the
    // corroborator's parent chain to find which payment artifact it
    // corroborates (the ancestor whose artifact the corroborator references)
    // OR consumes a caller-supplied corroboratedByArgumentIds[] (the same
    // mechanism QOL-037's deriveApplicabilityStatus already accepts).
    targetSelector: 'parent_of_corroborating_move',
  }),
  Object.freeze({
    mutation: 'evidence_debt_opened',
    axis: 'obligation',
    enumValue: 'requested',
    targetSelector: 'artifact_parent',
  }),
  Object.freeze({
    mutation: 'evidence_debt_resolved',
    axis: 'obligation',
    enumValue: 'supplied',
    targetSelector: 'artifact_parent',
  }),
]);

/**
 * The frozen conflict-resolution rule. See §5.3.
 */
export const PILL_STATE_CONFLICT_RULE = Object.freeze({
  /** Within an axis, the later signal wins. */
  withinAxis: 'last_wins_by_source_move_created_at' as const,
  /** Cross-axis mutations do not conflict (applicability + obligation are
   *  orthogonal axes; both stack on the same artifact). */
  crossAxis: 'orthogonal_stack' as const,
  /** When a mutation and a layer-1 record disagree, the more recent signal
   *  wins by created_at. Ties prefer layer-1 (user-action signal). */
  layerConflict: 'last_wins_layer1_wins_ties' as const,
});

/**
 * Input bundle for `derivePaymentEvidencePillState`.
 */
export interface DerivePaymentEvidencePillStateInput {
  /** The payment evidence artifact whose pill state we are composing. */
  readonly artifact: EvidenceArtifact;
  /** Composition-layer mutations whose targetMoveId === artifact.argumentId.
   *  Caller obtains via `useSemanticReferee.getMutationsForMove(artifact.argumentId)`.
   *  May be empty (observer mode, no classifier run, etc.). */
  readonly mutationsTargetingArtifactParent: readonly NodeVisualMutation[];
  /** Composition-layer mutations whose sourceMoveId corroborates this
   *  artifact (mutations of type `corroborating_document_attached` whose
   *  ancestor chain includes the artifact's parent). Caller pre-filters
   *  these — the deriver does not walk ancestor chains itself (pure
   *  data-in, data-out). */
  readonly corroboratingMutations: readonly NodeVisualMutation[];
  /** QOL-037 evidence-response records targeting this artifact. May be empty. */
  readonly responses: readonly EvidenceResponseRecord[];
  /** EV-003 evidence debts attached to the artifact's parent node. May be empty. */
  readonly debts: readonly EvidenceDebt[];
  /** Optional argument-id → ISO-8601 createdAt map for chronological merge.
   *  Required only when the caller wants accurate layer-conflict resolution;
   *  when absent, the deriver falls back to layer-1-wins (safe default). */
  readonly moveCreatedAtById?: ReadonlyMap<string, string>;
}

/**
 * Derive the composed pill state. Pure. Deterministic.
 */
export function derivePaymentEvidencePillState(
  input: DerivePaymentEvidencePillStateInput,
): PaymentEvidencePillState;
```

## §5 — Mutation-to-state mapping (the four mappings + conflict resolution rule)

### §5.1 — The four mappings (canonical contract)

The mapping is the frozen `MUTATION_TO_PILL_STATE` constant. The implementer MUST reference this constant; the test suite asserts its shape.

| # | Mutation | COMP-001 rule | Axis | Target enum value | Existing chip | Visibility |
|---|---|---|---|---|---|---|
| 1 | `evidence_applicability_disputed` | R-EV-APP-01 (`compositionLayer.ts:~959`) | Applicability (QOL-037) | `applicability_disputed` | `ApplicabilityChip` (becomes visible, tone `attention`) | Wherever the artifact is visible (existing RLS) |
| 2 | `corroborating_document_attached` | R-CAT-Corroborating (`compositionLayer.ts:~1036`) | Applicability (QOL-037) | `applicability_supported` | `ApplicabilityChip` (becomes visible, tone `info`) | Wherever the artifact is visible (existing RLS) |
| 3 | `evidence_debt_opened` | R-EV-01 (`compositionLayer.ts:~502`) | Obligation (EV-003) | `requested` | `EvidenceDebtChip` (becomes visible) | Wherever the artifact + node are visible (existing RLS) |
| 4 | `evidence_debt_resolved` | R-EV-02 (`compositionLayer.ts:~519`) | Obligation (EV-003) | `supplied` | `EvidenceDebtChip` (status flips to `supplied`) | Wherever the artifact + node are visible (existing RLS) |

#### Mapping unambiguity (Q1)

**Verdict: PASS.** Each mutation maps to exactly one axis + one enum value:

- Each of the four mutations is named by a COMP-001 rule that fires on a SINGLE classifier signal (R-EV-APP-01, R-CAT-Corroborating, R-EV-01, R-EV-02). The rules are pairwise non-overlapping in the catalog: `disputes_evidence_applicability` ≠ `supplies_corroborating_document` ≠ `asks_for_evidence` ≠ (`provides_evidence` && `evidence_supports_claim`).
- The four target enum values are pairwise distinct (`applicability_disputed`, `applicability_supported`, `requested`, `supplied`).
- The two axes (applicability + obligation) are **orthogonal** — see §5.2.

#### Target-move selection (clarifying who the mutation acts on)

- **Mutations 1, 3, 4** target the parent move (the move the artifact is attached to) via the composition layer's existing `targetMoveId` field. The deriver matches on `mutation.targetMoveId === artifact.argumentId`. Caller filters when looking up.
- **Mutation 2** (`corroborating_document_attached`) fires on the **corroborating** move (`targetMoveId === currentMoveId`), not on the artifact's parent. The deriver therefore reads a **separate** input bucket — `corroboratingMutations` — which the caller pre-filters by walking the corroborator's parent chain to find which artifact it corroborates. This matches QOL-037's existing `corroboratedByArgumentIds` parameter (see `evidenceApplicabilityModel.ts:344`).

### §5.2 — Orthogonality (applicability vs obligation)

The applicability axis and the obligation axis are **independent**. Both chips can render on the same artifact at the same time:

- `applicability_disputed` AND `requested` — "Applicability disputed" + "Source requested" — both chips visible (the artifact's applicability is challenged AND a source is being asked for; both are open obligations on different axes).
- `applicability_supported` AND `supplied` — both chips visible — "applicability addressed" + "source supplied".

A cross-axis pair is **never a conflict** — the mutations stack additively. `PILL_STATE_CONFLICT_RULE.crossAxis === 'orthogonal_stack'`.

### §5.3 — Conflict resolution (Q3)

#### Within an axis (`PILL_STATE_CONFLICT_RULE.withinAxis === 'last_wins_by_source_move_created_at'`)

When the same move's classifier output fires BOTH within-axis mutations (e.g. both `evidence_debt_opened` AND `evidence_debt_resolved`), the deriver picks the one whose `sourceMoveId` has the latest `createdAt`. Equal timestamps prefer the later array index (stable, deterministic).

**Realistic case where this matters:**
- A single move both opens a debt against parent (it `asks_for_evidence`) AND resolves an earlier debt on the same node (it `provides_evidence` AND `evidence_supports_claim`). Per the COMP-001 rule order (R-EV-01 fires before R-EV-02 in `compositionLayer.ts`), both mutations land in the same composition result. The within-axis last-wins rule says: the later mutation in chronological order wins. When both are the same `sourceMoveId`, the deriver picks `evidence_debt_resolved` (R-EV-02 runs second and produces the resolution; the "open" was for a DIFFERENT downstream debt and is a separate composition record).

**Edge case** — both mutations have identical `createdAt`: the deriver picks the one whose `sourceClassifier` sorts higher in the canonical ordering `['provides_evidence', 'evidence_supports_claim', 'asks_for_evidence', ...]` (stable / deterministic / documented).

#### Across layers (`PILL_STATE_CONFLICT_RULE.layerConflict === 'last_wins_layer1_wins_ties'`)

When layer-1 (user-action `EvidenceResponseRecord` / EV-003 debt walk) says `applicability_supported` and layer-2 (composition mutation `evidence_applicability_disputed`) says `applicability_disputed`:

- If `moveCreatedAtById` is supplied: pick the signal whose source move has the later `createdAt`.
- If `moveCreatedAtById` is absent (caller didn't pass a map): **layer-1 wins** (safe default — user action is the ground truth when chronology is unknown).
- On exact-timestamp tie: layer-1 wins (user-action signal is the more deliberate input).

#### Across both layers AND axes (cross-axis stack)

Orthogonal — both axes render independently. Layer-1 + layer-2 conflicts are resolved per-axis using the rule above; one axis's resolution does not influence the other.

### §5.4 — Mapping table for the implementer

The implementer reads this table and copies its semantics into `MUTATION_TO_PILL_STATE`. Tests assert the exact constant shape — adding a fifth row, dropping a row, or changing an `enumValue` is a test failure.

## §7 — Permission / rules implications (Q4 — visibility scope)

### §7.1 — Visibility scope

**Pill state visibility = artifact visibility.** Both chips this card composes are rendered inline on a payment artifact's node; the existing `public.arguments` RLS gates the artifact row and therefore gates both chips. The deriver runs **on the client** as a pure function over data the client already fetched via the existing RLS path; it adds no new visibility surface.

#### Intersection with QOL-036 visibility rules

QOL-036 §9 ("Permission / rules implications"): a payment object lives inside the existing `arguments.client_validation` / `server_validation` JSONB snapshot, gated by `public.arguments` RLS. **Preserved.**

#### Intersection with QOL-039 visibility rules (private rooms)

QOL-039's RLS replacements make private rooms invisible to non-participants. A payment artifact in a private room is therefore invisible to observers; the deriver never runs for an artifact the client cannot read; pill state is never leaked. **Preserved by construction** — this card adds no new data path.

#### Intersection with COMP-001 observer-mode behavior

COMP-001 §6.3: the `crossNodeMutations` map is optional on `ArgumentSurfaceState` — "absent when no composition has run for the room (e.g., observer / read-only mode)." For an observer:

- `mutationsTargetingArtifactParent` is empty (the hook hasn't run `classifyMove`).
- `corroboratingMutations` is empty for the same reason.
- The deriver falls through to layer-1 derivation only.
- Pill state for observers = pre-QOL-036.1 behavior. **Backward-compatible.**

**Verdict (Q4): pill state is visible to everyone who can see the artifact; the composition signal source only fires for viewers whose `useSemanticReferee` hook has run.** This is a one-way enrichment: composition mutations CAN flip the chip (for participants whose hook ran); their absence CAN'T un-flip a layer-1-derived chip. There is no observer-vs-participant divergence in the rendered chip's STATE — only in whether a classifier-detected signal contributes.

### §7.2 — Own-bubble / actor-aware rules

Preserved from QOL-036 §9 + Stage 6.1.8 / 6.4: the deriver is read-only and acts on already-fetched data; it does not gate edits, does not surface own-bubble actions, and does not interact with `request-argument-deletion`.

### §7.3 — Anti-amplification

Preserved from `evidence-doctrine` and `antiAmplification.ts`:

- A `corroborating_document_attached` mutation flips the applicability chip to `applicability_supported` (status flip), but emits NO `PointStandingDelta` — the deriver imports nothing from `src/features/pointStanding/`. A status flip is a STRUCTURAL indicator, not a standing reward.
- Repetition or popularity NEVER contributes to applicability_supported; only an actual classifier-detected corroborating document (`supplies_corroborating_document=1`) does.

### §7.4 — Validation

No new validation block. The deriver never blocks a post; it is read-only over already-fetched data.

### §7.5 — RLS / Edge Function / migration

## §8 — Edge cases

The implementer MUST handle each. Tests in §9.

1. **No mutations + no responses + no debts** → pill returns `applicability_undisputed`, `debtChipStatus: null`, `applicabilityProvenance: 'layer1'`, `debtProvenance: null`. Renders nothing (both chips suppressed).
2. **Mutation present but artifact is not a payment_screenshot** → the deriver still works (it's a pure data-in / data-out function; payment-specificity is enforced by the *caller* deciding which artifacts to pass through). A test fixture asserts the deriver behaves identically for `kind: 'url'` artifacts, with the caller's responsibility documented.
3. **Mutation `evidence_applicability_disputed` AND a layer-1 `dispute_applicability` EvidenceResponseRecord for the same artifact** → both signals agree; provenance is `layer1_with_layer2_corroboration`; status is `applicability_disputed`.
4. **Mutation `evidence_applicability_disputed` AND a layer-1 `accept` by the disputing party** → conflict. With `moveCreatedAtById` supplied: last-wins by `createdAt`. Without: layer-1 wins → `applicability_supported` (the `accept` resolved the dispute).
5. **Mutation `corroborating_document_attached` whose corroborator move's ancestor chain does NOT include the artifact's parent** → caller filters this out before calling the deriver (corroborator's ancestor doesn't match this artifact). The deriver only sees corroborating mutations the caller pre-filtered; an empty `corroboratingMutations` array gives `applicability_undisputed` (no corroboration effect on this artifact).
6. **Two `evidence_debt_opened` mutations against the same parent move from different classify-runs** → both land in `mutationsTargetingArtifactParent`. Deriver picks one `requested` state (idempotent — opening the same debt twice gives one debt). The EV-003 debt walker handles the actual debt-count separately; pill state is "at least one debt is open" → `requested`.
7. **`evidence_debt_resolved` mutation with no corresponding `evidence_debt_opened` in the same room** → conceptually impossible per COMP-001 (R-EV-02 only fires when `findOpenEvidenceDebtForResolution` returns a match), but defensively: the deriver treats the resolution as `supplied` for the obligation axis (last-wins). The EV-003 debt walker has its own correctness guard for the underlying debt row.
8. **Mutation targets a moveId that is NOT the artifact's parent** → caller pre-filters by `mutation.targetMoveId === artifact.argumentId`. The deriver assumes its input is correctly filtered; passing wrong-target mutations is a caller bug. A defensive test asserts the deriver doesn't crash if a wrong-target mutation slips in (it simply doesn't apply it — the mutation has no effect because the deriver only consumes mutations whose `targetMoveId` matches the artifact's parent OR whose `sourceMoveId` resolves to a corroborating ancestor).
9. **`moveCreatedAtById` is supplied but is missing some entries** → for any missing move id the deriver treats `createdAt` as `''` (empty string, sorts first). Last-wins resolution still works for entries that ARE in the map; missing-vs-present pair defaults to layer-1.
10. **Empty composition state (observer mode)** → both mutation arrays are empty; pill state = layer-1-only derivation. Backward-compatible.
11. **Concurrent edits** — N/A. Pure model, read-only over already-fetched data.
12. **Offline / network failure** — N/A. Pure model, no fetch.
13. **Permission-denied** — N/A. Pill is derived from data the client has already fetched; if RLS denied the row, the artifact isn't passed to the deriver.
14. **Settled / accepted_by_both debts** → EV-003's debt walker produces this status from layer-1 tags. The deriver does not flip `accepted_by_both` back to `supplied` via composition mutation — `accepted_by_both` is "more advanced" in the EV-003 lifecycle (`STATUS_SEVERITY_ORDER` places `supplied` before `accepted_by_participant` and `accepted_by_both`). The deriver respects the existing EV-003 worst-status-wins rule by preferring `accepted_by_both` over `supplied` when both signals fire on the same node.
15. **`evidence_applicability_disputed` AND `corroborating_document_attached` on the same artifact from two different moves** → within-axis last-wins by source move createdAt. The later signal wins; provenance is `layer2`. Documented in fixture test.
16. **Verdict token in EvidenceResponseRecord.clarificationBody** → the deriver does NOT sanitize user free text (consistent with QOL-036 §10 edge case 16). The deriver's own output strings (none, since copy comes from QOL-037 / EV-003) are ban-list-clean.
