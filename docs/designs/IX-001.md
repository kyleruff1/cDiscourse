# IX-001 — Timeline Density + Focus Lenses — Design Stub

**Status:** Design stub — created 2026-05-21. Defines the density core that
QOL-033 (Go popout) consumes but does not itself specify.

## Source storyboards / source analysis

- `docs/ux-ui-project-board.md` — IX-001 "Timeline zoom and density modes".
- `docs/designs/QOL-033.md` — the Go popout designs the *lenses* but consumes an
  undefined density/zoom core.
- `docs/project-audits/2026-05-21-roadmap-collision-supersession-analysis.md` §5
  — IX-001 flagged blank/under-defined.
- `docs/ux-storyboards/one-box-interface-model.md` — the Go popout.

## Problem statement

The argument entry screen (Conversation Gallery) and the in-room timeline become
visually noisy when activity is dense — imported corpus rooms, flooded entry
lists, long arguments. There is no shared model for **how dense** a surface
renders, and no shared model for **which subset** of items is in focus. QOL-033
provides the *Go popout* that exposes these controls, but the density tiers and
the lens definitions themselves are undefined. IX-001 defines them.

## Why this card matters now

- QOL-033 cannot be implemented without a density/lens model to render.
- The gallery flooding problem (repeated corpus posts crowding the entry list)
  needs density + a "one card per conversation" rule to be tractable.
- The gallery and the in-room timeline should **share one visual grammar** —
  IX-001 is where that shared grammar is defined.

## User-facing behavior

- A density control changes how much vertical/horizontal space each item takes,
  without hiding data.
- A focus lens dims (never deletes) non-matching items so the user can scan one
  concern at a time. A lens is reversible and stacks with search/sort.
- Density + lens choices persist per session and are shared between the gallery
  and the room timeline.

## Data / semantic fields needed

**Density modes** (4):

`spacious · balanced · compact · scan`

**Focus lenses** (each a deterministic predicate over already-derived fields —
no AI, no truth, no popularity):

`needs_response · no_rebuttal · heating_up · hot · quiet_plain ·
evidence_requested · source_chain_pressure · private_invites · my_active_rooms ·
recently_updated · settled_locked`

Each lens reads existing model output (LIFE-001 stage, EV-003 evidence debt,
heat/temperament from the gallery model, room visibility, participant identity,
timestamps). **A lens never implies truth and never implies popularity is
evidence** — "hot" is an *activity* signal, "heating up" is *momentum*, neither
is correctness.

## UI states

- Density: one of the 4 modes; `balanced` default; preserves the active node on
  change.
- Lens: zero or one active lens; "no lens" = everything at full opacity.
- A lens that matches zero items shows a plain-language empty note ("nothing
  needs a response right now") and leaves the surface unfiltered.

## Permission / rules implications

- Density and lenses are **view state only** — no write, no content mutation,
  no effect on standing or validation.
- `my_active_rooms` and `private_invites` are viewer-scoped (read the viewer's
  participation / invite records via existing RLS-bound queries).

## Dependencies

- The conversation gallery model (Epic 11 / `conversationGalleryModel`) — heat,
  temperament, dedupe.
- LIFE-001 lifecycle stages — `needs_response` / `no_rebuttal` / `settled`.
- EV-003 — `evidence_requested` / `source_chain_pressure`.
- IX-003 — keyboard navigation must reach density + lens controls.
- QOL-033 (Go popout) — the surface that exposes these controls in-room.

## Supersedes / superseded-by

**Partially superseded by QOL-033** — the Go popout *re-houses* the lens
controls. IX-001 is **not** dead: it owns the density-tier and lens *definitions*
that QOL-033 renders. Build IX-001's pure model; build the QOL-033 surface over
it. Do not build a standalone IX-001 React shell.

## Out of scope

- The Go popout surface itself (QOL-033).
- The mini-map (IX-002, re-housed by QOL-033).
- Voting / promotion-driven sorting.

## Acceptance criteria

- 4 density modes + 11 focus lenses defined as a pure deterministic model.
- **One gallery item per conversation** — repeated corpus posts collapse into a
  single conversation card where the data supports it (dedupe rule documented).
- Search, pagination, sort (by created · by activity · by engagement state)
  hooks are part of the model.
- Lenses dim, never delete; data stays intact behind a lens.
- Gallery and room timeline share the visual grammar.
- Keyboard-compatible with IX-003.
- No lens label implies truth or popularity-as-evidence.

## Test targets

- Density model — 4 tiers, active-node preserved across change.
- Each lens predicate — membership table; lens dims, never removes.
- Dedupe — N repeated corpus posts → 1 conversation card.
- Sort comparators — created / activity / engagement-state.
- Ban-list scan over lens labels (no verdict / popularity-as-proof copy).

## Implementation notes

- Pure-TS model in `src/features/` (beside the gallery model). No migration.
- Reuse `conversationGalleryModel` and `timelineMiniMapModel` outputs — IX-001
  is a projection/config layer, not a new data source.

## Open questions

- Should density persist device-local (AsyncStorage, like PR-001 preferences)
  or per-session only? (Lean: per-session for v1, device-local later.)
- Can two lenses stack, or strictly one at a time? (Lean: one at a time for v1.)
- Exact dedupe key for "one card per conversation" — root `debate` id, or a
  fuzzier conversation grouping when corpus duplicates exist?
