## 0. How this design was built

IX-001 is **not a screen**. It is the pure **density + lens core** that two
surfaces consume: the Conversation Gallery (Epic 11, shipped) and the in-room
timeline's Go popout (QOL-033, in flight). The stub correctly says "build the
pure model; build the QOL-033 surface over it; do not build a standalone IX-001
React shell." This design honours that exactly.

It is derived in three grounded passes:

1. **Reconcile what already ships (§2).** A density mode (`TimelineDensityMode`)
   and a density *preference* (`DensityPreference`) already exist in the code.
   IX-001 must extend, not fork, them. QOL-033 already names three density tiers.
   The lens vocabulary must read *existing* model output (gallery card fields,
   `PointLifecycleState`, `EntryOpportunity`) — no new data source.
2. **Define the two pure modules (§3–§7)** as the consequence: a density mode
   set that is a strict superset of the shipped 3, and 11 lens predicates that
   are each a deterministic boolean function over an already-built card / node.
3. **State the integration contract (§8–§9)** so QOL-033 and the gallery screen
   each consume one symbol set and nothing drifts.

Where this design and the stub's `4 density modes` literal disagree with the
shipped 3-mode `TimelineDensityMode`, **the shipped code wins for the timeline**
and IX-001's 4th mode is an *additive gallery-only* tier — see §3.1. The stub's
intent (four density tiers exist) is preserved; the collision is resolved
explicitly rather than papered over.

## 1. Goal & scope

The Conversation Gallery and the in-room timeline both become visually noisy
when activity is dense — corpus-flooded entry lists, 250-node argument boards,
long arguments. There is no shared model for **how dense** a surface renders and
no shared model for **which subset** of items is "in focus" so a user can scan
one concern at a time.

IX-001 defines, as **one pure deterministic projection/config layer**:

- **4 density modes** — how much space each item takes, *without hiding data*.
- **11 focus lenses** — each a deterministic predicate that *dims* (never
  deletes) non-matching items so the user scans one concern at a time.
- **The "one card per conversation" dedupe rule** — a concrete, documented
  dedupe key so repeated corpus posts collapse into a single conversation card.
- **The shared gallery ⇄ room visual grammar** for density + lens rendering.
- **Search / sort / pagination hooks** wired into the same config object.

In scope: the pure `timelineDensityLensModel` module + a tiny `useDensityLens`
session-state hook. **Out of scope:** any React shell of its own — the gallery
screen and QOL-033's Go popout *render* this model; IX-001 ships no surface.

```
                 ┌──────────────────────────────────────┐
                 │  timelineDensityLensModel.ts (pure)   │
                 │  4 density modes · 11 lens predicates │
                 │  dedupe key · sort comparators        │
                 └──────────────────────────────────────┘
                       ▲                         ▲
        ┌──────────────┘                         └──────────────┐
 ConversationGalleryScreen                  QOL-033 Go popout
 (Epic 11, shipped surface)                 (in-room timeline)
 lenses dim gallery cards                   lenses dim timeline nodes
 density sizes cards                        density sizes the board
```

## 3. Density model

### 3.1 The four density modes — reconciling the collision

The stub names four density modes: `spacious · balanced · compact · scan`. The
codebase already ships a **three-mode** `TimelineDensityMode`
(`compact · normal · expanded`) used by VG-004's `resolveNodeGapPx`, and PR-001
mirrors it as `DensityPreference`. QOL-033 §3.2 also lists three
(`Compact · Normal · Expanded`). A fourth mode that the timeline cannot honour
would be a silent contract break.

**Resolution.** IX-001 defines `GalleryDensityMode` as a **strict superset** of
the shipped 3-mode `TimelineDensityMode`, with the stub's names reconciled to
the shipped names and a 4th *gallery-only* "scan" tier added:

```ts
/**
 * IX-001 — Density modes. A STRICT SUPERSET of VG-004's TimelineDensityMode.
 *
 *   - 'expanded' | 'normal' | 'compact'  → the three shipped TimelineDensityMode
 *                                           values, 1:1, by the same name.
 *   - 'scan'                             → an additional gallery-only ultra-
 *                                           dense tier (one-line rows). It maps
 *                                           DOWN to 'compact' for the in-room
 *                                           timeline (the board never goes
 *                                           tighter than 'compact' — node hit
 *                                           targets must stay ≥ 44px).
 *
 * The stub's literal names (spacious/balanced/compact/scan) are reconciled to
 * the SHIPPED names so VG-004 / PR-001 / QOL-033 are not forked:
 *   stub 'spacious'  ≡  'expanded'
 *   stub 'balanced'  ≡  'normal'     (the default)
 *   stub 'compact'   ≡  'compact'
 *   stub 'scan'      ≡  'scan'       (new, gallery-only effect)
 */
export type GalleryDensityMode = 'expanded' | 'normal' | 'compact' | 'scan';

export const ALL_GALLERY_DENSITY_MODES: ReadonlyArray<GalleryDensityMode> =
  Object.freeze(['expanded', 'normal', 'compact', 'scan']);

/** The default. Mirrors the stub's `balanced default`. */
export const DEFAULT_GALLERY_DENSITY: GalleryDensityMode = 'normal';
```

The fourth mode is genuinely the stub's `scan` tier — it is not dropped. It is
honoured fully on the gallery (where rows can be one line) and *clamped* on the
timeline (where a node must keep a 44px hit target). This is documented in the
type and enforced by `toTimelineDensityMode` below.

### 3.2 Density → timeline spacing (no new spacing math)

The in-room timeline already reflows from a density mode via `resolveNodeGapPx`.
IX-001 adds **one** pure converter — it never re-implements spacing:

```ts
/**
 * Project a GalleryDensityMode onto the shipped TimelineDensityMode.
 * 'scan' clamps to 'compact' — the board never renders tighter than
 * 'compact' so every node keeps its ≥44px hit target (accessibility-targets).
 */
export function toTimelineDensityMode(mode: GalleryDensityMode): TimelineDensityMode {
  switch (mode) {
    case 'expanded': return 'expanded';
    case 'normal':   return 'normal';
    case 'compact':  return 'compact';
    case 'scan':     return 'compact'; // clamp — see §3.1
  }
}
```

QOL-033's Density control therefore calls
`resolveNodeGapPx(toTimelineDensityMode(mode))` — `64 / 44 / 28 / 28` px — and
nothing else changes on the board. **`scan` on the timeline is identical to
`compact`** by design; QOL-033 may either hide `scan` in-room or show it as a
synonym of `compact` (its choice — see §9).

### 3.3 Density → gallery card sizing

The gallery card has no equivalent of `resolveNodeGapPx`. IX-001 supplies a
**descriptive spec object** the gallery card renderer reads — it carries *which
optional regions render* and *line clamps*, never raw pixels (RN styling owns
pixels). Doctrine: density **never hides data** — it changes excerpt length and
which *optional* regions show; the full record is always one tap away.

```ts
/** What a gallery card renders at a given density. Descriptive, not pixels. */
export interface GalleryDensitySpec {
  mode: GalleryDensityMode;
  /** Lines of the first-post excerpt to show (0 = excerpt region hidden). */
  firstPostExcerptLines: number;
  /** Lines of the latest-post excerpt to show. */
  latestPostExcerptLines: number;
  /** Render the ConversationMiniTimeline rail on the card? */
  showMiniTimeline: boolean;
  /** Render the full signal chip strip, or only the single highest-tone chip? */
  signalChips: 'all' | 'primary_only';
  /** Render the stats row (moves · replies · participants)? */
  showStatsRow: boolean;
  /** Card vertical rhythm hint for the renderer's style map. */
  rhythm: 'roomy' | 'standard' | 'tight' | 'single_line';
}

export const GALLERY_DENSITY_SPECS: Readonly<Record<GalleryDensityMode, GalleryDensitySpec>> =
  Object.freeze({
    expanded: { mode: 'expanded', firstPostExcerptLines: 3, latestPostExcerptLines: 3,
                showMiniTimeline: true,  signalChips: 'all',          showStatsRow: true,  rhythm: 'roomy' },
    normal:   { mode: 'normal',   firstPostExcerptLines: 2, latestPostExcerptLines: 2,
                showMiniTimeline: true,  signalChips: 'all',          showStatsRow: true,  rhythm: 'standard' },
    compact:  { mode: 'compact',  firstPostExcerptLines: 1, latestPostExcerptLines: 1,
                showMiniTimeline: true,  signalChips: 'primary_only', showStatsRow: true,  rhythm: 'tight' },
    scan:     { mode: 'scan',     firstPostExcerptLines: 0, latestPostExcerptLines: 1,
                showMiniTimeline: false, signalChips: 'primary_only', showStatsRow: false, rhythm: 'single_line' },
  });

export function resolveGalleryDensitySpec(mode: GalleryDensityMode): GalleryDensitySpec {
  return GALLERY_DENSITY_SPECS[mode];
}
```

**Doctrine — `scan` hides no data.** Even in `scan`, the card still shows the
bucket headline, title, heat pill, the single highest-tone signal chip, and the
latest move line. The *first-post excerpt*, *mini-timeline*, and *stats row* are
**optional surface regions**, not data — every one is reachable by opening the
room or stepping the density back up. `scan` is "one line per conversation",
not "less truth per conversation".

### 3.4 Active-item preservation across a density change

The stub's hard rule: a density change "preserves the active node". This is a
**caller contract**, stated here so QOL-033 and the gallery both honour it, and
it is *free* because density changes nothing about identity:

- **Timeline:** `resolveNodeGapPx` reflows every node's `x` but never touches a
  node `id` or `ordinal`. The host re-`scrollTo`s the still-existent active node
  by `id`. VG-004 already documents this exact property
  (`argumentGameSurfaceModel.ts` §1242 comment).
- **Gallery:** a density change is a pure `GalleryDensitySpec` swap — it does not
  re-sort, re-paginate, or re-filter. The card the user was looking at keeps its
  `debateId`, list index, and page. The screen keeps scroll position.

IX-001 exposes a tiny assertion helper so the contract is testable:

```ts
/**
 * True when a density change is identity-preserving for `activeId`.
 * Density NEVER changes membership/order, so this is always true when
 * `activeId` is in both lists — the test exists to lock the invariant.
 */
export function densityChangePreservesActive(
  before: ReadonlyArray<{ id: string }>,
  after: ReadonlyArray<{ id: string }>,
  activeId: string | null,
): boolean {
  if (activeId == null) return true;
  return before.some((x) => x.id === activeId) === after.some((x) => x.id === activeId);
}
```

## 4. Focus-lens model

### 4.1 What a lens is

A **focus lens** is a deterministic predicate `(item) => boolean`. The surface
renders every item; items where the predicate is **false** are *dimmed* to a
faint, still-legible, still-interactive state. A lens **never removes an item,
never blocks navigation to it, and never deletes data**. At most **one** lens is
active at a time (stub lean, §15-Q2); "no lens" = every item at full opacity.

A lens stacks *on top of* search + sort + bucket filter — it changes opacity,
not membership. Search/sort/bucket decide *which cards are in the list*; the
lens decides *which cards in that list are emphasized*.

```ts
/** The 11 focus lenses. `none` is the unfiltered baseline (everything bright). */
export type FocusLensId =
  | 'none'
  | 'needs_response'
  | 'no_rebuttal'
  | 'heating_up'
  | 'hot'
  | 'quiet_plain'
  | 'evidence_requested'
  | 'source_chain_pressure'
  | 'private_invites'
  | 'my_active_rooms'
  | 'recently_updated'
  | 'settled_locked';

export const ALL_FOCUS_LENSES: ReadonlyArray<FocusLensId> = Object.freeze([
  'none', 'needs_response', 'no_rebuttal', 'heating_up', 'hot', 'quiet_plain',
  'evidence_requested', 'source_chain_pressure', 'private_invites',
  'my_active_rooms', 'recently_updated', 'settled_locked',
]);

export const DEFAULT_FOCUS_LENS: FocusLensId = 'none';
```

That is 11 named lenses plus the `none` baseline — the stub's list of 11,
verbatim, with `none` added as the required "no lens" state.

### 4.3 The 11 lens predicates — deterministic functions over existing fields

Each predicate is a pure boolean over an **already-built** `ConversationGalleryCard`
(gallery side) — no AI, no `Date.now()` in the model (recency is computed
against a caller-supplied `nowMs`, mirroring `conversationGalleryModel`'s own
`nowMs` injection so the function stays deterministic and testable).

```ts
/** Recency window for `recently_updated`. */
export const RECENTLY_UPDATED_WINDOW_MS = 24 * 60 * 60 * 1000; // 1 day

/** LIFE-001 states that mean "a source/quote is owed and unanswered". */
const EVIDENCE_DEBT_STATES: ReadonlySet<PointLifecycleState> =
  new Set<PointLifecycleState>(['source_requested', 'quote_requested']);

/** LIFE-001 states that mean "this point still needs a move from someone". */
const NEEDS_RESPONSE_STATES: ReadonlySet<PointLifecycleState> = new Set<PointLifecycleState>([
  'open', 'rebutted', 'clarified', 'source_requested', 'quote_requested', 'narrowed',
]);

/** LIFE-001 states that mean "closed". */
const SETTLED_STATES: ReadonlySet<PointLifecycleState> =
  new Set<PointLifecycleState>(['archived_or_resolved']);

/** Context a few lenses need beyond the card itself. */
export interface LensContext {
  /** Caller's clock, injected for determinism (never `Date.now()` here). */
  nowMs: number;
  /**
   * Debate ids the viewer has a pending invite to. RLS-bound; loader-
   * supplied (see §6 — the invite query). Empty when not loaded.
   */
  invitedDebateIds?: ReadonlySet<string>;
}

/** One lens predicate. `card` is already built; `ctx` carries clock + invites. */
export type GalleryLensPredicate = (card: ConversationGalleryCard, ctx: LensContext) => boolean;

export const GALLERY_LENS_PREDICATES: Readonly<Record<FocusLensId, GalleryLensPredicate>> =
  Object.freeze({
    // The baseline — every card matches, so nothing is ever dimmed.
    none: () => true,

    // 1. Latest move is awaiting a reply. Lifecycle-primary; bucket fallback.
    needs_response: (c) =>
      c.rootClusterLifecycleState != null
        ? NEEDS_RESPONSE_STATES.has(c.rootClusterLifecycleState)
        : (c.hasNoRebuttal || c.unresolvedReason != null),

    // 2. Opening claim, zero rebuttals. Pure structural card field.
    no_rebuttal: (c) => c.hasNoRebuttal,

    // 3. Momentum — gaining pace. Reads the heat LEVEL, never popularity.
    heating_up: (c) => c.heatLevel === 'warming' || c.bucket === 'gaining_heat',

    // 4. High recent activity. ACTIVITY, never correctness (doctrine §2).
    hot: (c) => c.heatLevel === 'hot' || c.heatLevel === 'overheated',

    // 5. Calm room — easy first move. Plain/pedantic temperament + cold heat.
    quiet_plain: (c) =>
      (c.temperament === 'plain' || c.temperament === 'pedantic')
      && c.heatLevel === 'cold',

    // 6. A source was asked for and is still owed. EV-003 evidence debt.
    evidence_requested: (c) =>
      (c.rootClusterLifecycleState != null && EVIDENCE_DEBT_STATES.has(c.rootClusterLifecycleState))
      || c.evidentiaryRisk === 'high',

    // 7. The room is tracing a claim's source chain. EV-003 source-chain risk.
    source_chain_pressure: (c) =>
      c.sourceChainRisk === 'high' || c.sourceChainRisk === 'medium'
      || c.platformSupportWarning,

    // 8. A private room the viewer is invited to. Viewer-scoped (ctx).
    private_invites: (c, ctx) =>
      c.openStatus !== 'archived'
      && (ctx.invitedDebateIds?.has(c.debateId) ?? false)
      && !c.hasUserJoined,

    // 9. A room the viewer joined that is still going. Viewer-scoped.
    my_active_rooms: (c) =>
      c.hasUserJoined && (c.openStatus === 'open' || c.openStatus === 'draft'),

    // 10. A move landed within the recency window. Clock injected via ctx.
    recently_updated: (c, ctx) =>
      ctx.nowMs - c.sortKeys.latestActivityMs <= RECENTLY_UPDATED_WINDOW_MS
      && c.sortKeys.latestActivityMs > 0,

    // 11. The room has closed. Lifecycle-primary; openStatus fallback.
    settled_locked: (c) =>
      (c.rootClusterLifecycleState != null && SETTLED_STATES.has(c.rootClusterLifecycleState))
      || c.openStatus === 'locked' || c.openStatus === 'archived'
      || c.bucket === 'resolved_or_synthesized',
  });
```

**Doctrine notes baked into the predicates:**

- `heating_up` / `hot` read `heatLevel` — `conversationGalleryModel`'s heat is
  *already* defined as "active friction, not popular" (its own header comment;
  it explicitly excludes view/retweet/follower counts). IX-001 does not add a
  popularity input. Heat is an **activity** signal here, full stop.
- Every predicate has a **fallback path** for cards where the LIFE-001
  `rootClusterLifecycleState` is not yet loader-populated (it is optional on the
  card today). The fallback uses only fields that always exist (`bucket`,
  `hasNoRebuttal`, `openStatus`, risk axes). No lens is ever inert pre-LIFE-001.
- `private_invites` and `my_active_rooms` are **viewer-scoped**: `hasUserJoined`
  is already viewer-derived on the card; invites come from `ctx.invitedDebateIds`
  which the loader fills from an RLS-bound query (§6). No cross-user leakage.

### 4.4 The room-side (timeline) lens predicates

In the Go popout, a lens dims **timeline nodes**, not gallery cards. QOL-033 §3.3
already specs three node lenses by LIFE-001 stage (`Active path` / `Unresolved` /
`Evidence`). IX-001 makes these the **node projection** of the same `FocusLensId`
union so the two surfaces share one vocabulary — a node lens predicate operates
over an `ArgumentTimelineMapNode`:

```ts
/** A node-side lens predicate (Go popout). Pure over a built map node. */
export type TimelineLensPredicate = (
  node: ArgumentTimelineMapNode,
  ctx: TimelineLensContext,
) => boolean;

/** Context the active-path lens needs. */
export interface TimelineLensContext {
  /** Message ids on the path from root to the active node (topology). */
  activePathIds: ReadonlySet<string>;
}

/**
 * The subset of FocusLensId that is meaningful on the in-room timeline.
 * Gallery-only lenses (private_invites, my_active_rooms, recently_updated,
 * settled is room-level not node-level) are NOT node lenses — see §4.5.
 */
export const TIMELINE_LENS_IDS: ReadonlyArray<FocusLensId> = Object.freeze([
  'none', 'needs_response', 'no_rebuttal', 'heating_up', 'hot',
  'evidence_requested', 'source_chain_pressure',
]);
```

| `FocusLensId` | Dims to faint every node EXCEPT… | Reads (already on the node) |
|---|---|---|
| `none` | (nothing dimmed) | — |
| `needs_response` | nodes whose `lifecycleState ∈ NEEDS_RESPONSE_STATES` | `node.lifecycleState` (LIFE-001) |
| `no_rebuttal` | nodes whose `lifecycleState === 'open'` and have no child edge | `node.lifecycleState`, child-edge count |
| `heating_up` | nodes whose `temperatureBand === 'warm'` | `node.temperatureBand` |
| `hot` | nodes whose `temperatureBand === 'hot'` | `node.temperatureBand` |
| `evidence_requested` | nodes whose `lifecycleState ∈ EVIDENCE_DEBT_STATES` | `node.lifecycleState` |
| `source_chain_pressure` | nodes of kind `source_chain` or with a dotted evidence-debt ring | `node.kindColorFamily`, debt ring flag |

This is **QOL-033's `Unresolved` lens generalised**: QOL-033's `Unresolved`
is exactly `needs_response`, and its `Evidence` is exactly the union of
`evidence_requested` ∪ `source_chain_pressure`. QOL-033 may continue to present
its three named lenses to the user — they are now *compositions of* IX-001's
shared union, not a parallel set. QOL-033's `Active path` lens is a **topology**
filter (root→active path), not a stage filter; IX-001 keeps it as a separate
`activePathLens(node, ctx)` helper (it reads `ctx.activePathIds`), exactly as
QOL-033 §3.3 already says ("topology, not stage").

### 4.5 Why some lenses are gallery-only

`private_invites`, `my_active_rooms`, `recently_updated`, and `settled_locked`
are **room-level** facts — they describe a whole conversation, not a node. They
have no node-level meaning, so they are absent from `TIMELINE_LENS_IDS`. This is
deliberate and documented so QOL-033 does not try to render an inapplicable
lens. The Go popout's Lens control offers only `TIMELINE_LENS_IDS`; the gallery's
Lens control offers all of `ALL_FOCUS_LENSES`. One union, two scoped views.

### 4.6 Applying a lens — the dim projection

A lens never filters a list. It annotates each item with a render hint:

```ts
/** Per-item render emphasis a lens produces. */
export type LensEmphasis = 'bright' | 'dimmed';

/** An item paired with its lens verdict. The list length never changes. */
export interface LensedItem<T> {
  item: T;
  emphasis: LensEmphasis;
}

/** Apply a gallery lens. Returns EVERY card, each tagged bright|dimmed. */
export function applyGalleryLens(
  cards: ReadonlyArray<ConversationGalleryCard>,
  lens: FocusLensId,
  ctx: LensContext,
): { items: LensedItem<ConversationGalleryCard>[]; matchCount: number; isEmpty: boolean } {
  const predicate = GALLERY_LENS_PREDICATES[lens];
  let matchCount = 0;
  const items = cards.map((card) => {
    const match = predicate(card, ctx);
    if (match) matchCount += 1;
    // `none` keeps everything bright; any other lens dims non-matches.
    const emphasis: LensEmphasis = lens === 'none' || match ? 'bright' : 'dimmed';
    return { item: card, emphasis };
  });
  // A lens that matches zero cards "leaves the surface unfiltered" (stub):
  // every card reverts to bright and the caller shows `emptyNote`.
  const isEmpty = lens !== 'none' && matchCount === 0;
  if (isEmpty) {
    return { items: items.map((i) => ({ ...i, emphasis: 'bright' as const })), matchCount: 0, isEmpty: true };
  }
  return { items, matchCount, isEmpty: false };
}
```

`applyTimelineLens(nodes, lens, ctx)` is the structural twin over
`ArgumentTimelineMapNode`. **Critical invariants, both functions:**

1. `result.items.length === input.length` — always. A lens never drops an item.
2. When `matchCount === 0` and `lens !== 'none'`, every item is forced back to
   `bright` and `isEmpty` is `true` — the surface renders unfiltered and the
   caller shows `FOCUS_LENS_COPY[lens].emptyNote` ("nothing needs a response
   right now"). This is the stub's explicit "matches zero → plain note, leaves
   the surface unfiltered" rule.
3. `emphasis` is `'dimmed'`, never `'hidden'` — there is no `hidden` value in
   the `LensEmphasis` union, structurally guaranteeing a lens cannot delete.

The render layer maps `dimmed` to an opacity (≈ 0.4) — but a dimmed card/node
stays a full `Pressable` with its full `accessibilityLabel`; it is still
tappable, still keyboard-reachable, still navigable (see §11).

## 5. Dedupe — "one card per conversation"

### 5.1 The rule and the concrete key

The stub asks for "one gallery item per conversation — repeated corpus posts
collapse into a single conversation card, dedupe rule documented." The gallery
model **already ships** the mechanism (`dedupeConversationCards`, Stage 6.3).
IX-001's job is to **fix it as the documented rule** and pin the open question.

**The dedupe key is `ConversationGalleryCard.canonicalConversationKey`**,
produced by `deriveCanonicalConversationKey({ debate, rootArgumentBody, hintHash })`.
It is a string chosen by the first available of four tiers:

| Tier | Key shape | When it fires |
|---|---|---|
| 1 | `hash:<hintHash>` | Caller has an explicit `sourceHash` / `scenarioId` (≥ 6 chars). The strongest, most precise key. |
| 2 | `root:<normalised root body>::title:<cleaned title>` | Root argument body is ≥ 24 chars. Cleaned title strips the corpus suffix tags (`[xai-adv …]`, `[ai-corpus …]`, `[stress …]`, `[scenario-N]`, `[run-N]`, `[seed-N]`). |
| 3 | `title:<cleaned title>::resolution:<resolution>` | No root body, but a non-empty cleaned title. |
| 4 | `resolution:<resolution>::id:<debate id>` | Nothing else available — a *stable but non-collapsing* key (each debate stays its own card). |

`dedupeConversationCards(cards, 'collapse_generated')` groups by this key, keeps
the **latest-activity** card as the visible "primary", and folds older variants'
`debateId`s into `duplicateDebateIds` with a `duplicateCount`. The DB rows are
**never deleted** — dedupe is purely visual; every collapsed debate id stays
individually addressable for an expandable "run history".

### 5.2 The open-question resolution

The stub's open question: "exact dedupe key — root `debate` id, or a fuzzier
conversation grouping?" **Decision:** the four-tier key above, with tier 2 as
the workhorse for corpus floods. Rationale:

- A raw `debate.id` key would **never collapse** corpus duplicates — every
  `[xai-adv …]` run is a distinct `debate` row, so id-keying defeats the card's
  entire purpose.
- A purely "fuzzy" semantic grouping would risk **false merges** of genuinely
  distinct rooms that happen to share a topic — and any fuzziness implies a
  similarity model, which edges toward AI. Doctrine forbids that.
- The four-tier key is **deterministic, AI-free, and corpus-aware**: tier 1
  (explicit hash) is exact; tier 2 (`cleanTitleForDedupe` + normalised root
  body) collapses the corpus families precisely because the runners append a
  *known, regex-strippable* suffix and reuse the *same* root body across a run;
  tier 4 guarantees no unrelated rooms ever merge (each keeps its `id`).

`cleanTitleForDedupe` is the load-bearing piece — it strips exactly the suffix
families the corpus runners emit (`SUFFIX_TAG_PATTERNS` in the gallery model).
IX-001 **adopts it unchanged**; this design does not re-implement dedupe.

### 5.3 Dedupe and lenses compose

Order of operations the gallery screen runs (and QOL-033 is unaffected — the
in-room timeline has no dedupe):

```
build cards
  → dedupeConversationCards(cards, dedupeMode)          // §5 — N rooms → 1 card
  → bucket / search filter (existing gallery behavior)
  → sortConversationGalleryCards(cards, sortMode)       // §7
  → applyGalleryLens(cards, lens, ctx)                  // §4.6 — tags bright|dimmed
  → paginateConversationGalleryCards(cards, size, page) // §7
  → render each LensedItem at resolveGalleryDensitySpec(density)  // §3.3
```

Dedupe runs **before** the lens so a lens predicate sees the *primary* card of a
group (the latest-activity one) — the user lenses conversations, not corpus
runs. The lens runs **before** pagination so `matchCount` / `isEmpty` are
computed over the whole filtered set, not one page.

## 6. Viewer-scoped data — the invite query

Two lenses are viewer-scoped. `my_active_rooms` needs nothing new
(`card.hasUserJoined` already exists). `private_invites` needs the set of debate
ids the viewer has a *pending invite* to — `ctx.invitedDebateIds`.

IX-001 **does not** write this query — it specifies the contract so the gallery
loader (`useGalleryArguments` neighbourhood) fills `LensContext`:

- A `SELECT debate_id FROM debate_invites WHERE invitee_user_id = auth.uid() AND status = 'pending'`-shaped read, **RLS-bound** (the invitee may read only their own invite rows). If a `debate_invites` table does not yet exist (invites are a P2 item — QOL-038/040), `invitedDebateIds` is simply `undefined` and the `private_invites` lens predicate returns `false` for every card — the lens is *present but empty*, showing its `emptyNote`. No crash, no half-feature.
- **No service-role.** The read uses the standard anon-key client + RLS, exactly like every other gallery query (`listArgumentsForDebateIds` pattern).
- **No new write, no new Edge Function, no migration** in IX-001. If QOL-038
  later adds the invites table, `private_invites` lights up for free — the
  predicate already handles both states.

This is the stub's "`my_active_rooms` and `private_invites` are viewer-scoped
(read the viewer's participation / invite records via existing RLS-bound
queries)" made concrete.

### 7.3 Reset rules — when a control change resets the page

A pure helper encodes which config changes must snap `pageIndex` back to 0 (so
the user is not stranded on a now-empty page):

```ts
/**
 * Given a config change, return the next config with pageIndex reset
 * to 0 when the change alters the visible set/order. Density does NOT
 * reset the page (it never changes membership — §3.4). Lens does NOT
 * reset the page (it dims, never removes — §4.6). Search/sort DO reset.
 */
export function applyViewConfigChange(
  prev: DensityLensViewConfig,
  patch: Partial<DensityLensViewConfig>,
): DensityLensViewConfig {
  const next = { ...prev, ...patch };
  const membershipChanged =
    ('searchQuery' in patch && patch.searchQuery !== prev.searchQuery) ||
    ('sortAxis'    in patch && patch.sortAxis    !== prev.sortAxis);
  if (membershipChanged) next.pageIndex = 0;
  return next;
}
```

This matches QOL-033's contract that **density preserves the active item** and
the stub's contract that **a lens stacks with search/sort** without yanking the
viewport. Search and sort genuinely reorder the list, so they reset the page;
density and lens do not, so they do not.

## 8. Shared visual grammar (gallery ⇄ room)

The stub's hard requirement: "Gallery and room timeline share the visual
grammar." IX-001 owns the *density + lens* slice of that grammar; the
*node-shape* slice is `timeline-grammar` (VG-001/VG-004), which IX-001 reuses
and never forks. The shared rules:

| Concern | Shared rule | Where enforced |
|---|---|---|
| **Density tiers** | Both surfaces use `GalleryDensityMode`; the timeline projects via `toTimelineDensityMode` → `resolveNodeGapPx`. One mode set, one default (`normal`). | §3 |
| **Lens vocabulary** | Both surfaces use `FocusLensId`; the timeline uses the `TIMELINE_LENS_IDS` subset. One union. | §4 |
| **Dim treatment** | A dimmed item (card or node) drops to ≈ 0.4 opacity, keeps its border/shape, stays interactive. Dimming is opacity-only — never a color the user could read as a verdict. | §11, this table |
| **Heat = activity** | "hot" / "heating up" are an activity tint in both surfaces — the gallery's heat pill and the timeline's `temperatureBand` tone tint both cap their visual weight and both say "activity, not a result" in copy / a11y label. | timeline-grammar; §4.2 copy |
| **Empty-lens note** | Both surfaces, when a lens matches zero items, show `FOCUS_LENS_COPY[lens].emptyNote` and render the surface unfiltered. | §4.6 |
| **Shape carries type** | The lens/density layer never changes a node's shape or a card's structure — VG-001 shape grammar is untouched. Density changes *spacing and excerpt length*; a lens changes *opacity*. Neither touches the type/strength encoding. | timeline-grammar |

The single grammatical sentence: **density changes how much room an item takes;
a lens changes how bright an item is; neither ever changes what an item *is*.**

## 11. Edge cases

- **Empty gallery (0 cards)** — `applyGalleryLens([], lens, ctx)` returns
  `{ items: [], matchCount: 0, isEmpty: lens !== 'none' }`. The screen shows its
  existing "no rooms found" empty state; a lens does not manufacture a different
  message over a genuinely empty list.
- **Lens matches zero cards** — `isEmpty: true`; every card forced back to
  `bright`; the screen shows `FOCUS_LENS_COPY[lens].emptyNote` ("nothing needs a
  response right now") and renders the list unfiltered. The stub's exact rule.
- **Lens matches every card** — `matchCount === cards.length`; every card
  `bright`; no dimming; no empty note. Indistinguishable from `none` visually,
  which is correct.
- **`rootClusterLifecycleState` not loaded** (the field is optional today) —
  every lifecycle-driven predicate falls back to always-present fields
  (`bucket`, `hasNoRebuttal`, `openStatus`, risk axes). No lens is inert; no
  predicate throws on `undefined`.
- **`invitedDebateIds` not loaded / no invites table** — `private_invites`
  predicate returns `false` for all cards → the lens shows its `emptyNote`. The
  lens is present and honest, not broken.
- **Density `scan` on the timeline** — `toTimelineDensityMode('scan')` clamps to
  `'compact'`; the board never renders tighter than 28px gaps; node hit targets
  stay ≥ 44px. No accessibility regression.
- **Density change mid-scroll** — identity-preserving (§3.4);
  `densityChangePreservesActive` is `true`; the host keeps the active node/card
  in view. Verified by a named test.
- **Concurrent edit / new move arrives while a lens is active** — the gallery /
  timeline rebuilds its cards/nodes; `applyGalleryLens` re-runs on the new list
  (the model holds no mutable state). A card that *was* dimmed may become bright
  (e.g. a reply arrives → it leaves `no_rebuttal`) — that is correct, live
  behavior, not a bug.
- **A dimmed card is tapped** — a dimmed item is a full `Pressable`; the tap
  opens the room exactly as a bright card would. Dimming is emphasis, never a
  permission gate. (Verified: `LensEmphasis` has no `disabled` value.)
- **Search query + lens + sort all active** — they compose (§5.3 order): search
  & sort decide the list, the lens dims within it, density sizes it. Switching
  the lens does **not** reset the page (§7.3); switching search/sort does.
- **`nowMs` not supplied / `0`** — `recently_updated` would treat every room as
  "not recent" (`0 - latestActivityMs` is negative, but the
  `latestActivityMs > 0` guard and the window check mean a `0` clock yields no
  matches). The hook always supplies `Date.now()` at the call site, *outside*
  the pure model — the model itself never calls `Date.now()`.
- **Single-mode lens picker on a tiny viewport** — the Lens control is a
  horizontally scrollable chip row (matches the existing bucket-chip row); 12
  chips never crowd because they scroll. Density is 4 chips — fits inline.
- **Doctrine edge case — "what if heat tries to influence the lens set?"** — it
  does not. `heating_up` / `hot` are *themselves* the activity lenses; heat is
  never an input to a *non-heat* lens (e.g. `needs_response` reads lifecycle,
  not heat). And no lens, heat-based or not, changes a card's standing, score,
  or any validation — a lens is opacity only.
- **Doctrine edge case — "what if dedupe merges two genuinely different
  rooms?"** — tier 4 of the dedupe key (`resolution::id`) guarantees that two
  rooms with no shared hash, no ≥24-char shared root body, and no shared cleaned
  title each keep their own `debate.id` in the key and therefore stay separate
  cards. Over-merge is structurally impossible for unrelated rooms.

## 14. Risks

- **Density-mode collision (the biggest risk, resolved).** The codebase ships a
  3-mode `TimelineDensityMode`; the stub said 4 modes with different names. A
  naive 4th mode would silently break VG-004 / PR-001 / QOL-033. **Mitigation:**
  §3.1 makes `GalleryDensityMode` a *strict superset* with the shipped names
  reused 1:1 and `scan` clamping to `compact` in-room. The implementer must
  **not** rename `TimelineDensityMode` or add a 4th value to it — IX-001's 4th
  mode lives only in `GalleryDensityMode`. A forbidden-imports/shape test should
  assert `TimelineDensityMode` still has exactly 3 members.
- **LIFE-001 not yet loader-wired onto the card.** `rootClusterLifecycleState`
  is optional today. If the implementer forgets the fallback branch, several
  lenses would be inert. **Mitigation:** every lifecycle predicate's fallback is
  specified in §4.3 and has a dedicated "lifecycle-absent" test case in §12.
- **Invites table may not exist.** `private_invites` depends on a
  `debate_invites`-shaped table that is a P2 item (QOL-038). **Mitigation:** the
  predicate degrades to "no matches → empty note" when `invitedDebateIds` is
  `undefined`; the lens ships present-but-empty, not broken. No IX-001 migration.
- **Dim opacity vs contrast.** A dimmed card at 0.4 opacity must still pass
  legibility — a dimmed item is *de-emphasized*, not *hidden*, and must remain
  readable (accessibility-targets). **Mitigation:** the render layer (gallery
  card / QOL-033) owns the exact opacity; the design fixes ≈ 0.4 as a floor and
  a component test should assert dimmed text still meets a sane contrast against
  the surface. The dim is opacity-only — never a color swap a user could read as
  a verdict.
- **Existing gallery tests.** `ConversationGalleryScreen` gains controls;
  `conversationGalleryModel.ts` is **not** touched, so its 50-test suite stands.
  The screen's own render tests may need a couple of fixture bumps for the new
  controls — expected; the test count goes up.
- **No migration, no Edge Function, no operator deploy** — pure client code.
  Low operational risk.

## 17. Doctrine & safety self-check

- **cdiscourse-doctrine §1 — score is gameplay analysis, never truth; score
  never blocks posting.** A density mode and a focus lens are *view state only*
  — they change spacing and opacity. Neither is ever an input to the
  Constitution engine, `argumentScoreModel`, `antiAmplification`, or any
  validation gate. No lens label says winner / loser / correct / true / false /
  right / wrong — a ban-list test (§12) enforces it over every authored string.
- **cdiscourse-doctrine §2 — heat means activity / friction, never
  correctness.** The `hot` and `heating_up` lenses read `ConversationGalleryCard.
  heatLevel`, whose source model (`computeConversationHeat`) is *already*
  defined as "active friction, not popular" and explicitly excludes
  view/retweet/follower counts. The lens `helper` copy says so out loud
  ("activity, not correctness" / "momentum, not a result"). Heat is never an
  input to a *non-heat* lens. A `hot` room can be wrong; the lens makes no
  claim about that.
- **cdiscourse-doctrine §3 — popularity is not evidence.** No lens predicate
  reads a popularity-shaped field; the model file contains no view / like /
  retweet / follower / vote / share-count identifier (a source-scan test
  enforces it). A lens grants no factual standing to anything — it changes
  opacity, nothing else.
- **cdiscourse-doctrine §4 — AI moderator limits.** IX-001 contains no AI. It
  decides nothing about right/wrong; it deletes/hides/modifies no content (a
  lens *dims* — `LensEmphasis` has no `hidden` value, structurally); it assigns
  no truth value; it returns no flags.
- **cdiscourse-doctrine §5 — the rules engine is sacred.** IX-001 does not
  import, touch, or reference `engine.ts`.
- **cdiscourse-doctrine §6 — secrets.** No key, no token, no `.env` access; a
  source-scan test asserts no `ANTHROPIC_API_KEY` / `SERVICE_ROLE` reference.
- **cdiscourse-doctrine §7 — no AI calls from the production app.** Pure-TS
  model + one React hook; no Anthropic / xAI / X / OpenAI; a forbidden-imports
  test asserts no `fetch` / network import.
- **cdiscourse-doctrine §8 — Supabase conventions.** **No migration**, no RLS
  change, no service-role. The one viewer-scoped read (`private_invites`'
  invites query) is RLS-bound and lives in the gallery loader, not IX-001's
  files — IX-001 only consumes the resulting `invitedDebateIds` set. No direct
  insert into `public.arguments`; IX-001 writes nothing at all.
- **cdiscourse-doctrine §9 — plain language.** Every lens label/helper/empty
  note is plain English; no internal code (`source_chain`,
  `point_lifecycle_state`, `temperature_band`) reaches a user-facing string — a
  ban-list test scans for underscore-cased tokens.
- **cdiscourse-doctrine §10 — v1 scope.** No voting, no real-time co-editing, no
  OAuth, no public API, no push. **Search note:** IX-001's "search hook" is the
  *already-shipped* gallery room-finder filter over titles/excerpts
  (`getConversationSearchText`, Stage 6.3); doctrine §10's "no argument search"
  bans *full-text search across argument bodies*, which IX-001 neither builds nor
  proposes. IX-001 adopts an existing filter; it adds no new search capability.
- **timeline-grammar — shape carries type/strength; color is supplementary; no
  truth-laden visuals.** IX-001 changes only *spacing* (density) and *opacity*
  (lens). It never changes a node's shape, stroke, strength encoding, or a
  card's structure. The VG-001 token system is untouched and reused. Dimming is
  opacity-only — never a color a user could read as a verdict.
- **accessibility-targets — 44px targets, color independence, SR contract,
  reduce-motion.** Density and Lens controls are plain `Pressable` chips with
  `accessibilityRole="button"` + `accessibilityState={{ selected }}` + ≥44px hit
  targets (`hitSlop` where the visual is smaller). A **dimmed** card/node keeps
  its full `accessibilityLabel` and stays a focusable, tappable, keyboard-
  reachable control — dimming is visual emphasis, never a removal from the
  accessibility tree. `scan` density clamps to `compact` in-room so node hit
  targets never drop below 44px. A density / lens change is not an animation;
  no reduce-motion concern beyond what VG-004 already handles.
- **expo-rn-patterns — RN primitives, no new dependency.** The model is pure TS
  (no React import — enforced by test). The hook uses only `useState`/
  `useCallback`. The gallery controls reuse the existing chip-row pattern. No
  new dependency, no AsyncStorage in v1 (Q1).
