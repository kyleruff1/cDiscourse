## Goal

VG-002 promotes the existing timeline rail — six 4px-tall `<View>` segments per edge, currently rendered in `ArgumentTimelineMap.tsx` via `EdgeStrip` — into a richer **gradient wave rail** that visibly encodes the three orthogonal signals the doctrine permits us to show on the connector between nodes:

1. **Node color blend** (`prev → next` argument-kind family — already encoded via `gradientStops` from `argumentGameSurfaceModel.ts`).
2. **Tone overlay** — heat / friction / activity. Reads existing `toneBand` + `temperatureBand` per edge. Warm hue means *active unresolved discussion*. It never asserts truth.
3. **Evidence-risk overlay** — the worst `SourceChainStatus` between the parent and child nodes. Reads EV-001's `summarizeArtifactsForReceiptChip` over the artifact list for each endpoint. A *clean saturated track* describes the trail's anchoring (`source_and_quote` / `primary_present`); a *dotted / kinked track* describes weak or missing trail.
4. **Active-path glow** — already partially encoded by `edge.isActivePath`; VG-002 promotes it to a real glow overlay via `View` `shadow*` / `elevation` props, plus an additive translucent overlay strip on top of the segment.

The rail surface answers one question: *what shape is this conversation taking?* It never tells the user a claim is correct, true, winning, supported, validated, or anchored-to-truth. "Clean saturated track" describes the **trail quality**, not the **claim quality**. Heat is **friction**, not consensus. These distinctions are doctrine anchors; the design's self-check enumerates them line by line.

The card ships **no new dependency**. The repo carries none of `expo-linear-gradient`, `react-native-svg`, `react-native-skia`, `@shopify/react-native-skia`, `react-native-reanimated`, or `react-native-gesture-handler` (verified in `package.json`). The visual is built with stacked `<View>` strips, `mixHex` interpolation (already exported from `argumentGameSurfaceModel.ts`), and the `shadow*` / `elevation` props that RN's core `View` already supports.

A second-order goal: define the **BR-001 seam**. BR-001 (Issue #7, Branch model) ships tangent-kink detection later. VG-002 must render "Tangent path kinks/breaks visibly" today, with no BR-001 dependency. The design introduces a `RailSegmentInput.branchKind` field (`'main' | 'tangent' | 'kink_start' | 'kink_end' | 'detached'`) populated by a deterministic placeholder derivation in this card. When BR-001 ships, the placeholder is swapped for BR-001's real kink-detection; the rail's visual output stays untouched.

## Visual model — segment anatomy

Each rail segment is a single thin `<View>` strip (4px tall, `RAIL_THICKNESS`; 6px when the edge is the first-clash edge per existing `FIRST_CLASH_RAIL_THICKNESS`), positioned + rotated between two nodes exactly as `EdgeStrip` does today. VG-002 layers up to **five Z-stacked sub-layers** inside that strip wrapper, in this order (lowest → highest):

| Z | Layer | What it encodes | Rendering |
|--:|-------|-----------------|-----------|
| 0 | **Base color band** — 6 stacked `<View>`s | `prev → next` argument-kind blend | Existing `EDGE_SEGMENTS = 6` array, each segment a `mixHex` step over `edge.gradientStops`. Unchanged from today. |
| 1 | **Tone wash** — single `<View>` overlay | Tone (`calm/measured/heated/hostile`) + temperature (`cool/mild/warm/hot`) | Semi-transparent (`rgba(toneColor, alpha)`), `alpha` derived from temperature: `cool=0`, `mild=0.10`, `warm=0.25`, `hot=0.45`. Hostile/heated tones push hue toward warm; calm/measured tones stay close to base. **Color hue means activity, NEVER correctness.** |
| 2 | **Evidence-track texture** — 1 `<View>` or 6 sub-strips | `SourceChainStatus` between endpoints | When `source_and_quote` / `primary_present` → a single thin `<View>` of `RECEIPT_CHIP_RING_COLOR` (teal-700, `#0f766e`, reused from EV-002) at full opacity beneath the tone wash → reads as a "clean saturated track". When `source_no_quote` / `unverified` / `no_source` → no overlay (base color reads through). When `broken` → 6 sub-strips with `opacity: [1, 0.3, 1, 0.3, 1, 0.3]` along the length, producing a visible "dotted track" without `borderStyle` (which RN does not honor on small `<View>` strips). |
| 3 | **Branch / kink marker** — single mini-`<View>` at strip end | `RailSegmentInput.branchKind` | `main` → no marker. `tangent` → render the strip but at a 6° offset (extra `rotateZ` baked into the wrapper transform) and at 60% opacity (the rail "kinks" off the mainline visually). `kink_start` → a 6×6 angled stub `<View>` appears at the parent end of the strip. `kink_end` → a 6×6 angled stub at the child end. `detached` → no strip rendered (matches today's behavior). |
| 4 | **Active-path glow** — `<View>` overlay | `edge.isActivePath` | Rendered ONLY when active. A second `<View>` 2px wider on each side, with `shadowColor: '#a5b4fc'`, `shadowOpacity: 0.65`, `shadowRadius: 6`, `shadowOffset: { width: 0, height: 0 }`, `elevation: 6` (Android). Non-active edges retain today's `opacity: 0.55` dim. |

Each `<View>` is `pointerEvents: 'none'` — none of the rail is tappable. Per accessibility plan, the rail as a whole exposes one summary `accessibilityLabel`; per-segment a11y entry remains via the existing node Pressables and EV-002's source-chain popover.

The center rail bar (the gray track behind the message row, currently `styles.rail`) stays as a single full-width `<View>`. VG-002 does NOT touch that strip — it is the "no edge here" baseline.

## Rail-input contract (the BR-001 seam)

A new pure-TS module `src/features/arguments/railSegmentModel.ts` exports:

```ts
import type {
  ArgumentTimelineMapEdge,
  ArgumentTimelineMapNode,
  TimelineToneBand,
  TimelineTemperatureBand,
} from './argumentGameSurfaceModel';
import type { SourceChainStatus, EvidenceArtifact } from '../evidence/evidenceModel';

/** The kind of segment to render. BR-001 will own deterministic
 *  population once branch detection lands. Today VG-002 derives it
 *  from a placeholder (see `derivePlaceholderBranchKind`). */
export type RailBranchKind =
  | 'main'
  | 'tangent'
  | 'kink_start'
  | 'kink_end'
  | 'detached';

export interface RailSegmentInput {
  /** Stable id for memoization. Mirror `edge.edgeId`. */
  segmentId: string;
  /** Endpoint message ids (for memoization / debug). */
  fromMessageId: string;
  toMessageId: string;
  /** From-node x/y in scroll-coord pixels. */
  x1: number;
  y1: number;
  x2: number;
  y2: number;
  /** 6-stop color array passed to the base layer. */
  gradientStops: ReadonlyArray<string>;
  /** Tone band (drives the tone-wash hue). */
  toneBand: TimelineToneBand;
  /** Temperature band (drives the tone-wash alpha). */
  temperatureBand: TimelineTemperatureBand;
  /** Worst-status SourceChainStatus across both endpoints. */
  sourceChainStatus: SourceChainStatus;
  /** Branch kind — placeholder today, BR-001 owns later. */
  branchKind: RailBranchKind;
  /** True when this edge lies on the active path. Drives the glow layer. */
  isActivePath: boolean;
  /** True for the root→first-rebuttal edge. Drives `FIRST_CLASH_RAIL_THICKNESS`. */
  isFirstClash: boolean;
}

export interface RailSegmentStyle {
  /** Wrapper transform: x, y, width (length), rotateZ deg, optional extra
   *  kink rotation when branchKind === 'tangent'. */
  wrapper: {
    left: number;
    top: number;
    width: number;
    height: number;
    transformRotateZDeg: number;
    opacity: number;
    /** When true, render with the angled-stub start/end markers per branchKind. */
    showKinkStartStub: boolean;
    showKinkEndStub: boolean;
  };
  /** Color hex per sub-strip in the base band (always 6 entries). */
  baseSubStripColors: ReadonlyArray<string>;
  /** Tone-wash overlay (alpha 0 means "no overlay"). */
  toneWash: { color: string; alpha: number };
  /** Evidence-track overlay — null means "no overlay" (base reads through). */
  evidenceTrack:
    | null
    | { mode: 'solid'; color: string }
    | { mode: 'dotted_pattern'; color: string; alphaPattern: ReadonlyArray<number> };
  /** Glow layer (null means "not glowing"). Always null when !isActivePath. */
  glow:
    | null
    | { color: string; shadowOpacity: number; shadowRadius: number; elevation: number };
  /** Plain-language a11y description fragment for this segment. Used to
   *  build the rail's whole-rail accessibilityLabel. */
  accessibilityFragment: string;
}

export function deriveRailSegmentStyle(input: RailSegmentInput): RailSegmentStyle;

/** Placeholder derivation BR-001 will replace. Pure. Today's rule:
 *    - if either endpoint's argumentType maps to a tangent qualifier
 *      (`branch_this_off`, `tangent_or_joke`) OR the edge is detached
 *      from the active path AND its child node has `kindColorFamily ===
 *      'flag'` → `'tangent'`.
 *    - if the edge is detached (parent not in the loaded node set) →
 *      `'detached'`.
 *    - otherwise → `'main'`.
 *  No `'kink_start'` / `'kink_end'` emitted by the placeholder; BR-001
 *  will introduce them when the explicit kink object is modeled. */
export function derivePlaceholderBranchKind(args: {
  fromNode: ArgumentTimelineMapNode;
  toNode: ArgumentTimelineMapNode;
  isDetached: boolean;
}): RailBranchKind;

/** Build a `RailSegmentInput` from an existing edge + the node lookup +
 *  the EV-001 artifact map. Pure. The mapper that the room shell calls
 *  once per render. */
export function buildRailSegmentInput(args: {
  edge: ArgumentTimelineMapEdge;
  fromNode: ArgumentTimelineMapNode;
  toNode: ArgumentTimelineMapNode;
  artifactsByMessageId: Record<string, ReadonlyArray<EvidenceArtifact>>;
}): RailSegmentInput;
```

When BR-001 ships, the only change is to swap `derivePlaceholderBranchKind` for BR-001's `deriveBranchKindFromConstitutionModel(...)`. `RailSegmentInput.branchKind` field, the dispatch table, and the renderer stay frozen. EV-002's surface lock pattern (locked dispatch table + frozen string constants) is mirrored intentionally.

## State-to-style matrix

The rail's visual is a deterministic function of four dimensions: `branchKind × sourceChainStatus × toneBand × isActivePath`. Naively that is 5 × 6 × 5 × 2 = 300 rows. We collapse it by treating tone + temperature as a continuous overlay (one alpha lookup) and by keeping the glow as binary (on iff `isActivePath`). That leaves 5 × 6 = **30 structural states**, of which only the most semantically meaningful 20 actually differ. The full table is below; tests assert each row produces the expected `RailSegmentStyle` shape.

Legend:
- *base*: 6-stop `mixHex` blend over `edge.gradientStops` (parent kindColor → midpoint blend → child kindColor → standingColor → toneColor). Unchanged from existing `EdgeStrip`.
- *tone wash α*: `cool=0`, `mild=0.10`, `warm=0.25`, `hot=0.45`, `unknown=0.05`.
- *teal*: `#0f766e` (the `RECEIPT_CHIP_RING_COLOR` already exported from EV-002).
- *dotted pattern*: alpha array `[1, 0.3, 1, 0.3, 1, 0.3]` over the 6 sub-strips.

| Row | branchKind | sourceChainStatus | Evidence track layer | Wrapper opacity | Kink stubs | Doctrine note |
|---:|---|---|---|---:|---|---|
| 1 | `main` | `no_source` | none | 1.0 | none | "no overlay" reads as "trail unspecified" — neutral, not negative |
| 2 | `main` | `unverified` | none | 1.0 | none | weak trail; UI prompts in EV-002, rail stays neutral |
| 3 | `main` | `source_no_quote` | solid teal at 0.5 alpha | 1.0 | none | trail partially anchored — half-saturated |
| 4 | `main` | `source_and_quote` | **solid teal at 1.0 alpha** | 1.0 | none | "clean saturated track" per AC; describes *trail*, not *claim* |
| 5 | `main` | `primary_present` | **solid teal at 1.0 alpha** + extra `glow` stays binary | 1.0 | none | strongest trail; doctrine: still not "true" |
| 6 | `main` | `broken` | **dotted pattern teal at 0.6 alpha** | 1.0 | none | weak trail; non-accusatory; pairs with EV-002 broken popover |
| 7 | `tangent` | any | inherit from row 1–6 | **0.6** | none | dimmed strip + 6° rotateZ kink baked into wrapper |
| 8 | `kink_start` | any | inherit from row 1–6 | 1.0 | start stub only | BR-001 emits this; placeholder never does |
| 9 | `kink_end` | any | inherit from row 1–6 | 1.0 | end stub only | BR-001 emits this; placeholder never does |
| 10 | `detached` | n/a | not rendered | 0 | n/a | parent missing; rail surface skips the strip entirely |

The `isActivePath` dimension flips the glow layer on/off and sets `opacity = 1` on non-active rows (the existing `opacity: 0.55` dim is preserved for non-active non-glow edges). The `toneBand` + `temperatureBand` dimensions feed the tone wash via a single lookup, never alter the structural row.

The matrix is intentionally compressed to ten structural rows precisely because the doctrine forbids inflating it: every additional "warmth shade" risks readers projecting truth onto a tone. Hot edges look warm; quiet edges look cool; that is the whole tonal vocabulary.

## Performance plan

Acceptance criterion: **Legible at 250+ messages.** Today's `ArgumentTimelineMap` renders every edge unconditionally inside one horizontal `ScrollView`. At 250 messages with ~1 edge per message, we'd render ~250 wrapper `<View>`s + ~6 base sub-strips each + (now) up to 3 overlay `<View>`s — naive total ≈ 250 × (1 + 6 + 3) = **2,500 `<View>` instances on the rail layer**, plus 250 nodes + decorations. That's slow on lower-end Android in single-frame layout passes and creates measurable React reconciliation cost.

### Chosen strategy: virtualized rail segment rendering (option a)

Render only the rail strips whose endpoint x-positions overlap the visible horizontal viewport (plus a small buffer). Specifically:

1. `ArgumentTimelineMap` tracks `scrollX` via the existing `<ScrollView>` `onScroll` callback (throttled to 16ms via a simple `lastFrame` timestamp check — no `requestAnimationFrame` needed at the test surface).
2. A pure helper `visibleSegmentSlice(segmentsInputs, scrollX, viewportWidth, bufferPx)` returns the slice of `RailSegmentInput[]` whose `[min(x1,x2), max(x1,x2)]` intersects `[scrollX - bufferPx, scrollX + viewportWidth + bufferPx]`. `bufferPx` defaults to one viewport width on each side.
3. The component renders the slice; off-screen edges are simply not in the tree. Style objects for the slice are memoized via a `Map<segmentId, RailSegmentStyle>` cached on the model — once a segment is derived, its style is stable until its `RailSegmentInput` deep-equality changes.
4. The 6 sub-strip color arrays for the base layer are memoized inside `deriveRailSegmentStyle` — for 250 messages the cache holds 250 entries at most, then is keyed by `segmentId`.

#### Back-of-envelope cost estimate

At 250 messages:
- Viewport (default phone landscape ≈ 800px) × buffer (2× viewport) = 2,400px window across a `scrollWidth` of ~250 × 72 = 18,000px.
- Visible+buffered slice ≈ `2400 / 18000 × 250 ≈ 33 segments` rendered at any time.
- Each segment renders **at most 10 `<View>` instances** (1 wrapper + 6 base sub-strips + 1 tone wash + 1 evidence track + 1 glow). Many states render fewer (e.g. `main` + `no_source` + non-active drops 2 overlays).
- **Peak rail-layer `<View>` count ≤ 330** at any moment, regardless of `nodes.length`. That is constant-bounded by viewport size, not by message count.
- Memoized style cache: `O(n)` entries (one per segment), each ~150 bytes → ~38 KB at 250 messages. Acceptable.

Tests assert: `visibleSegmentSlice` returns `≤ 50` segments for a synthetic 250-segment fixture at default viewport+buffer settings.

#### Why not the alternatives

- **(b) Decimation (collapse adjacent same-color segments).** Rejected: collapsing edges visually merges two messages into one rail strip, which the visual grammar uses to communicate *count* (each edge corresponds to one parent→child link). Collapsing introduces a covert truth claim ("these two are the same"), which violates the no-truth-label doctrine in spirit. Also: edges at depth boundaries collapse poorly into the lane model.
- **(c) Pre-compute styles at room load.** Rejected as primary because the rail must re-derive when the active path changes (every node tap), which would re-run the whole pre-compute. Memoization keyed per segment (already in the plan) gets the same win at lower cost.
- **(d) Memoization alone, no virtualization.** Rejected: memoization avoids re-renders but does not avoid the *initial* layout pass over 2,500 `<View>` instances on the first render of a 250-message room. Virtualization is the only strategy that bounds peak `<View>` count by viewport, not by total messages.

Memoization (option d) is layered *on top* of virtualization, not in place of it. Both ship in this card.

## Accessibility plan

Per `accessibility-targets` skill, color independence is the most load-bearing accessibility requirement for this card because every signal the rail encodes is initially a color signal.

### Color-independent channels (parallel to every color signal)

| Signal | Color encoding | Parallel non-color channel |
|---|---|---|
| Argument-kind blend | base 6-stop gradient | Node shape (already encoded by VG-001 node Pressable) + node ordinal in `accessibilityLabel` |
| Tone / heat | tone-wash hue + alpha | Heat is summarized in the rail's whole-rail a11y label ("3 active branches") and per-node a11y label ("currently active") — the per-edge color is **not** the screen-reader entry point |
| Evidence track | solid teal vs dotted vs none | EV-002 receipt chip on the node carries identical information as TEXT. Tapping the node opens the chip; the popover headline ("No source yet", "Source and quote", "Source trail is weak") states the trail status in plain language. The rail itself is a *visual reinforcement*, never the only signal. |
| Active path | glow + opacity 1.0 vs 0.55 | The active node already carries `accessibilityState={{ selected: true }}` + an `accessibilityLabel` suffix "currently active". The rail glow is decorative. |
| Tangent kink | 6° rotateZ + 60% opacity | When `branchKind === 'tangent'`, the child node's existing droppedTag chip ("Tangent") fires (already implemented in the existing `argumentGameSurfaceModel.mapDroppedTags`). The kink is a visual reinforcement. BR-001 will introduce a more explicit chip when it lands; until then we lean on the existing tag. |
| Source-chain "broken" dotted pattern | 6-strip alpha pattern | EV-002 chip says "Source trail is weak" in plain language. The dotted rail mirrors it visually. |

### Reduce-motion fallbacks

The active-path glow is rendered as a static `<View>` overlay with `shadowOpacity`. It is **not animated** in v1 — no pulse, no fade-in. So `prefersReducedMotion` does not change the glow. Documented as a known limitation: a future card (VG-004 hot-tail momentum, or QOL-016) may add a pulse animation; that card MUST respect `AccessibilityInfo.isReduceMotionEnabled()` per the existing pattern in EV-002.

Tone wash color transitions when the active path changes: today's `EdgeStrip` already mounts/unmounts edges without animation. VG-002 preserves that. No transition / interpolation animation is added.

If a future card animates the rail (gradient sweep, pulse, color interpolation), the design assumes that card will read `useReduceMotion()` (a tiny pure-TS hook over `AccessibilityInfo`) and skip the animation when the user has opted out. The VG-002 design includes a TODO comment in the rail component pointing at this.

### Screen-reader summary

The rail as a whole exposes one summary `accessibilityLabel` on the timeline's outer `<View>` (already present as `accessibilityLabel="timeline-map-scroll"` on the `ScrollView`). VG-002 promotes this to a richer plain-language summary derived from the model:

```
"Trail: {N} messages, {M} active branches, source attached on {K},
 {L} branches need a source."
```

Numbers come from the existing `ArgumentTimelineMapModel.nodes` + `artifactsByMessageId`; the summary is pure-TS and tested. The label is **rebuilt on every model change**; per-edge label fragments (one per `RailSegmentStyle.accessibilityFragment`) are concatenated only into this whole-rail label — they are not exposed as individual focusable elements, which would explode the focus order at 250 messages.

Per-segment tap: the rail is `pointerEvents: 'none'`. Node tap (already in place) is the per-segment a11y entry point.

### Tap targets

Rail strips are not Pressable. No tap target requirement. Node Pressables already meet `MIN_TAP_TARGET_PX = 44` per VG-001.

### Doctrine-anchored plain language

- `accessibilityFragment` for each segment uses plain English: "active branch with source attached", "weak source trail", "tangent off mainline". Never: "true", "false", "correct", "validated", "winning", "supported", "anchored to truth".
- The whole-rail summary uses "attached" / "needs a source" (matching EV-002's "Ask for source"), never "missing evidence" (sounds accusatory).

## Edge cases

The implementer MUST handle each. Tests are in §"Test plan".

1. **Empty rail (`map.edges.length === 0`).** `GradientWaveRail` renders nothing. The whole-rail `accessibilityLabel` reads "Trail: 0 messages." Existing `<View>` for the center rail bar continues to render the gray track.
2. **Single-message rail (`nodes.length === 1`).** Root node only, zero edges. Same as edge case 1 — `GradientWaveRail` renders nothing. Active-path glow logic short-circuits.
3. **All-tangent rail.** Every edge's `branchKind === 'tangent'`. Each strip renders with 6° rotateZ + 60% opacity. The whole-rail a11y label includes "all branches are tangents off the mainline".
4. **All-broken-chain rail.** Every edge's `sourceChainStatus === 'broken'`. Each strip renders the 6-strip dotted alpha pattern. Whole-rail a11y label: "{N} of {N} branches need a stronger source." (No "broken" verb-token, no "wrong" — matches EV-002's broken-chain copy.)
5. **All-detached rail (`isDetached: true` for every parent reference).** No strips render. Whole-rail a11y label: "Trail: parents not loaded for this view." The existing `nodeDetached` styling on individual nodes communicates per-node detachedness; rail surface stays empty.
6. **Dark theme** (the only theme today per BRAND-001). Base colors render against `#020617` / `#08060F`. Tone wash + teal evidence track have been verified for ≥3:1 contrast against this background in the design's color audit. Light theme is **not** a v1 surface; if PR-001 introduces it later, the chip's contrast pass is the dependency, not VG-002's.
7. **250-message stress case.** Per the performance plan: `visibleSegmentSlice` returns ≤ 50 segments at any time. Memoized style cache survives across scroll frames. Test asserts the slice size and peak `<View>` count.
8. **Observer / read-mode viewer (`isReadModeViewer === true`).** Rail rendering is unchanged. Observers see the rail just like participants. No CTA wiring on the rail (rail is `pointerEvents: 'none'`); the observer-disable contract lives on EV-002's popover, not here.
9. **Active path changes mid-scroll.** When `map.activeNode` changes, `isActivePath` changes for the affected edges. Memoization key includes `isActivePath`, so the cache entries for the affected edges invalidate. Test asserts ≤ 2 cache invalidations per active-node change (the old active edge + the new active edge, in the typical case).
10. **`mixHex` receives a malformed color string.** EV-001's adapter never emits malformed hex; the existing `EdgeStrip` doesn't guard. VG-002 inherits the same trust. If a future change introduces a non-`#rrggbb` color, the `mixHex` helper produces a meaningless but non-crashing string — defensive, never throws.
11. **Edge crosses the visible viewport boundary partially.** `visibleSegmentSlice` includes any segment whose x-range intersects the viewport+buffer, so partial visibility is treated as visibility. No partial-rendering optimization is attempted.
12. **`branchKind === 'kink_start'` or `'kink_end'` from BR-001.** Today's placeholder never emits these. The dispatch table handles them deterministically (start/end stub `<View>`). When BR-001 ships, the visual just starts appearing — no renderer change needed.
13. **Concurrent edits.** Not applicable — the rail is a derived view over the already-fetched edge list. The existing `useArgumentRoomMessages` realtime subscription path delivers new messages; the rail re-derives on the next render.
14. **Reduce-motion preference.** No animation in v1; preference does not change the visual. Documented above.
15. **Color-blind mode (PR-001 future).** Tone wash + teal evidence track use color tokens that PR-001 will pass through a remapper. The chip text + a11y labels remain unchanged across modes — color-independence holds.
16. **First-clash edge thickness.** Existing `FIRST_CLASH_RAIL_THICKNESS = 6` is preserved. The 5-layer stack scales: all layers use the wrapper's `height` so they auto-thicken on first-clash edges.
17. **Edge with zero length** (e.g. parent and child at the same `(x, y)` because the lane collision fallback put them on top). Strip width clamps to a minimum of 2px (existing `length = Math.max(2, …)`). All layers render at that minimum.

## Risks

1. **`derivePlaceholderBranchKind` produces "tangent" too aggressively.** If the placeholder over-marks tangents (e.g. because `flag` is the catch-all family), the rail visually fragments more than the conversation actually is. Mitigation: the placeholder's flag-family rule is gated on `isDetached === true` AND non-active-path — both must be true. Test fixture exercises the false-positive case and asserts `'main'`. BR-001 will tighten this anyway.
2. **`mixHex` performance at 250 messages.** Each segment's base layer calls `mixHex` 6 times. 250 segments × 6 calls = 1500 `mixHex` calls per render. The function is pure arithmetic and runs in microseconds; not a concern at expected scales. Memoization protects scroll re-renders.
3. **Dotted pattern via `[1, 0.3, ...]` alpha may look wrong on some Android backgrounds.** Alpha at 0.3 against the dark `#020617` backdrop should yield ~3:1 contrast with the teal track. If a future surface tone is much lighter, the pattern could blend out. Mitigation: the dotted pattern is the *secondary* signal — the EV-002 chip text is the *primary* signal. Visual is reinforcement, not the only channel.
4. **Active-path glow uses `shadowColor` + `shadowOpacity` which behaves differently on iOS vs Android.** iOS renders shadow as a soft halo; Android renders via `elevation` which produces a rectangular drop shadow. Mitigation: the rail strip is thin (4-6px) so the shadow shapes look similar across platforms; the `elevation: 6` value is chosen empirically to roughly match the iOS halo. If a designer rejects the Android look at Review, a future card may layer a duplicate `<View>` offset by 1px as a synthetic glow — not in scope for VG-002.
5. **`isActivePath` cache invalidation chatter.** Every active-node tap invalidates 2 cache entries (old + new active edges). With memoization keyed per `segmentId`, this is cheap, but a fixture test asserts ≤ 2 invalidations.
6. **`pointerEvents: 'none'` on every rail layer.** If a future card wants segment-tap to open a per-segment popover, the rail surface will need to introduce a Pressable layer. Documented as out-of-scope here; surface stays declarative.
7. **`RAIL_SOURCE_CHAIN_TEAL` drift risk.** The hex is duplicated between `railSegmentModel.ts` and EV-002's `ReceiptChip.tsx`. Test asserts they are equal — if EV-002 changes its hex, the rail test fires immediately.
8. **The whole-rail a11y label could become noisy on long conversations.** "Trail: 250 messages, 12 active branches, source attached on 87, 41 branches need a source" is a lot. Considered acceptable because screen-reader users get one shot per element; verbose-but-accurate beats "Trail: 250 messages." Future card could add a more compact form.
9. **EV-002 + VG-002 ordering on the popover/chip render path.** EV-002 already requires the rail to honor its teal hex. If VG-002 ships first by accident (it shouldn't — EV-002 is already merged per design status), the teal would still render; only the chip integration would be missing. No regression risk.
10. **Performance on web (`Platform.OS === 'web'`).** Web RN renders `<View>` as `<div>`, and `shadow*` props map to CSS `box-shadow`. The CSS shadow is faster than the iOS shadow path. No concern; the virtualized slice keeps DOM size bounded.
