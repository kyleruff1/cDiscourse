## Goal (one paragraph)

CDiscourse forces a debater to hold five vocabularies at once — Constitution argument types, RULE-005 move channels,
LIFE-001 lifecycle states, MCP boolean Observation families, and UX-001.5A user Allegations. REF-002 builds the single
**pure derivation** that converges them into one turn-level reading: the **Open Issue** (`DisagreementContract`). Given
the already-derived seams for one selected node — its lifecycle state, evidence debt, manual/auto metadata, machine
observations, the seeded referee banner, the suggested-moves input, and the Act-popout survivor set — the model derives
`relation · axis · burden · state · referee observation · next-best moves` deterministically and emits no block. The
doctrine that shapes every line: the deterministic engine (`src/domain/constitution/engine.ts`, mirrored at
`supabase/functions/_shared/constitution/`) is the **sole** submission gate; this model is **consultative**, runs only
**after** an argument is stored, **never** re-validates or contradicts the engine, **never** emits a verdict or a person
label, and `nextBestMoves` is always a **subset** of the engine+role-gate survivors so the card can never present a
button the engine would reject. v1 is **derived-only**: no persistence, no migration, no Edge path
(cdiscourse-doctrine §1, §4, §5, §9, §10a; operator ratification constraints 1–5).

```ts
export type DisagreementAxis =
  | 'evidence' | 'definition' | 'scope' | 'causal'
  | 'logic' | 'value' | 'framing' | 'process';                       // 8 — OpenIssue axes (NOT the
                                                                     //     7-value constitution DisagreementAxis;
                                                                     //     distinct type, same name)

export type RelationToParent =
  | 'supports' | 'challenges' | 'asks_source' | 'asks_quote'
  | 'narrows' | 'branches' | 'concedes' | 'synthesizes' | 'replies'; // 9 — `replies` is the internal FALLBACK

export type IssueBurden =
  | 'source_owed' | 'quote_owed' | 'reply_owed' | 'clarification_owed' | 'none';   // 5

export type IssueState =
  | 'open' | 'answered' | 'source_requested' | 'quote_requested'
  | 'narrowed' | 'conceded' | 'synthesis_ready' | 'moved_on';        // 8

export interface IssueObservation {
  sourceCode: string;          // banner `bannerCode` OR mark `rawKey` — NEVER user-visible raw
  line: string;                // plain-language line (banner.headline OR mark.label) — already ban-list clean
  toneGlyph: 'star' | 'arrow' | 'branch' | null;   // carried from the banner; null when mark-seeded
  kind: 'machine_observation'; // always — Allegations live in userAllegations
}

export interface IssueAllegation {
  sourceCode: string;          // USER_ALLEGATION rawKey — NEVER user-visible raw
  line: string;                // plain-language label
  kind: 'user_allegation';
  visibility: 'public' | 'composer_only' | 'moderator_only';
}

export interface MoveSuggestion {
  actEntryId: string;          // a surviving ActEntryId, or the branch_tangent recovery entry
  label: string;               // plain-language button label, read from the Act entry definition
  accessibilityLabel: string;  // verbose a11y label, read from the Act entry definition
  isRecoveryRoute: boolean;    // true when this replaced a gate-removed suggestion
  recoveredFromCode: string | null;   // the recovered SuggestedMoveCode — diagnostic only
}

export interface DisagreementContract {
  id: string;                  // `issue:<targetNodeId>:<relation>:<axis-or-debt>` — deterministic, never a DB row, never shown raw
  roomId: string | null;
  targetNodeId: string | null;
  targetQuote: string | null;          // exact quote/excerpt; NEVER AI-summarized
  contestedProposition: string;        // deterministic truncation/excerpt; NO AI synthesis
  axis: DisagreementAxis;
  relationToParent: RelationToParent;
  burden: IssueBurden;
  state: IssueState;
  refereeObservations: ReadonlyArray<IssueObservation>;   // ≤ 1 on the public card; Family J never present
  userAllegations: ReadonlyArray<IssueAllegation>;        // never merged into Observations
  nextBestMoves: ReadonlyArray<MoveSuggestion>;           // 2–3; engine+role-gate survivors only
}

/** Alias REF-001 used throughout. Both names export the same shape. */
export type OpenIssue = DisagreementContract;
```

## Derivation rules per field (REF-001 §4B, frozen)

### `roomId`, `targetNodeId`
Passed through. `buildOpenIssue` returns `null` when **either** is null (no issue object exists; no card mounts).

### `targetQuote`
Prefer `quoteAnchor` (exact). Else `targetExcerpt` **only if** it is an exact excerpt (used verbatim, no synthesis).
Else `null`. **Never AI-summarized.** Cap is verbatim-with-ellipsis truncation only.

### `contestedProposition`
Deterministic excerpt: `quoteAnchor ?? targetExcerpt ?? ''`, truncated to ≤ 160 chars with a trailing `…` when cut on a
word boundary. **No AI synthesis.** A user-editable title is a later card (out of scope).

### `axis` — `deriveOpenIssueAxis` (6-tier precedence, high→low)

| Tier | Source (live field) | Rule |
|---|---|---|
| 1 — evidence/quote debt | `openEvidenceDebts[].debtKind`; `sourceChainStatus` | Any open debt → `evidence` (every `EvidenceDebtKind` — source/quote/receipt/context/primary_record — is evidence-resolvable). `sourceChainStatus ∈ {no_source, broken, source_no_quote, unverified}` corroborates `evidence`. |
| 2 — explicit `disagreementAxis` | a **specific** Family B (`disagreement_axis`) machine observation in `machineObservations` whose `rawKey` is a `disputes_*` axis | Per §4B normalization: `disputes_definition`→definition, `disputes_scope`→scope, `disputes_causal_link`→causal, `disputes_analogy`/scheme→logic, `disputes_value_weighting`/`disputes_decision_criterion`/`disputes_priority_order`→value, `disputes_relevance`→framing, `disputes_fact`/`disputes_evidence_applicability`/`disputes_interpretation`/`disputes_generalization`→evidence. The generic `disagreement_present` umbrella is **not** explicit (falls to tier 5). Family resolved via `lookupMachineObservationDefinitionByCompoundKey`. |
| 3 — manual tag | `manualTags` | `definition_issue`→definition; `scope_issue`/`narrowed_claim`→scope; `causal_mechanism`→causal; `needs_source`/`needs_quote`/`evidence_debt`→evidence; `tangent`→process. |
| 4 — lifecycle debt (axis) | `lifecycleAxis` (`PointLifecycleAxis`) | `fact`→**evidence**, `evidence`/`source`/`quote`→evidence, `definition`→definition, `scope`→scope, `causal`→causal, `value`→value, `logic`→logic, `unaxed`→(no signal). |
| 5 — semantic observation | any remaining Family B umbrella mark; `activeDisagreementKind` | `disagreement_present` umbrella → framing; `activeDisagreementKind` `framing`/`context`→framing, **`fact`→framing** (the receiver-stance `fact` is issue-focus language, NOT evidence — operator constraint 3), `none`→(no signal). |
| 6 — process fallback | — | When nothing above resolves → **`process`** (never empty). |

**`fact` split (operator constraint 3, REF-001 note lines 213–216).** Three distinct `fact` sources are kept distinct
in code, and **no OpenIssue axis is ever literally `fact`**:
- `PointLifecycleAxis === 'fact'` → `evidence` (tier 4).
- Family B `disputes_fact` mark → `evidence` (tier 2).
- `ActiveDisagreementKind === 'fact'` → `framing` (tier 5).

### `relationToParent` — `deriveOpenIssueRelation` (5-tier precedence + reply-neutrality)

Map the highest-available source to one of the 9 relation values. `replies` is the **internal fallback** chosen only
when no more-specific relation derives (operator constraint 2 — never preferred over a more specific procedural relation).

| Tier | Source | Mapping (omitted ⇒ `replies`) |
|---|---|---|
| 1 | `selectedActEntryId` | `challenge`→challenges, `ask_source`→asks_source, `ask_quote`→asks_quote, `narrow`→narrows, `concede`/`offer_concession`→concedes, `synthesize`→synthesizes, `branch_tangent`→branches, `confirm`/`add_evidence`→supports, `reply`/`clarify`/`respond_to_*`→replies. |
| 2 | `selectedChannel` (`MoveChannel`) | `challenge`→challenges, `ask_source`→asks_source, `ask_quote`→asks_quote, `narrow`→narrows, `concede`→concedes, `synthesize`→synthesizes, `branch_tangent`→branches, `confirm`/`add_evidence`→supports, `reply`/`clarify`/`meta_process`→replies. |
| 3 | `storedArgumentType` | `rebuttal`/`counter_rebuttal`→challenges, `evidence`→supports, `concession`→concedes, `synthesis`→synthesizes, `thesis`/`claim`/`clarification_request`→replies. |
| 4 | `manualTags` | `concession_offered`→concedes, `narrowed_claim`→narrows, `ready_for_synthesis`→synthesizes, `tangent`→branches, `needs_source`→asks_source, `needs_quote`→asks_quote, `definition_issue`/`scope_issue`/`causal_mechanism`/`evidence_debt`→challenges. |
| 5 | `lifecycleState` transition | `source_requested`→asks_source, `quote_requested`→asks_quote, `narrowed`→narrows, `conceded`→concedes, `synthesis_ready`→synthesizes, `branch_recommended`→branches, `rebutted`→challenges, else→replies. |
| 6 | fallback | **`replies`**. |

**Reply-neutrality upgrade (doctrine — side ≠ agreement).** Applied **only** when the derived relation is `replies`:
if `sameSideAsParent && carriesSupportEvidence` → `supports`. **Never** inferred from side alone — `carriesSupportEvidence`
is mandatory. `clarify` / `clarification_request` map to `replies` (no `clarifies` relation exists); their specificity is
carried by `burden = clarification_owed` + `axis = definition`, not by the relation.

### `burden` — `deriveOpenIssueBurden` (precedence: evidence/quote owed > clarification > reply > none)

1. **Highest-precedence open evidence debt** (source-class before quote, mirroring the §4B/state precedence): a
   `source`/`receipt`/`context`/`primary_record` debt → `source_owed`; a `quote` debt → `quote_owed`. When no explicit
   debt: `sourceChainStatus ∈ {no_source, broken}` → `source_owed`; `source_no_quote` → `quote_owed`.
2. else a definition/clarify signal (`manualTags` has `definition_issue`; `lifecycleAxis === 'definition'`;
   `selectedActEntryId`/`selectedChannel === 'clarify'`; `storedArgumentType === 'clarification_request'`) →
   `clarification_owed`.
3. else the point is open/contested and awaits a response (`lifecycleState ∈ {open, rebutted}`, or relation ∈
   {challenges, asks_source, asks_quote} with no terminal state) → `reply_owed`. A bare `replies` relation by itself
   does **not** create `reply_owed` and **never clears a debt** (operator constraint 2).
4. else (terminal: `narrowed`/`conceded`/`confirmed`/`synthesis_ready`/`moved_on`, no open debt) → `none`.

### `state` — `deriveOpenIssueState` (19-row lifecycle base map, then debt override)

**Step A — base map** from `lifecycleState` (null → derive from debt/concession/synthesis flags, default `open`):

| # | `PointLifecycleState` | → `IssueState` | Rationale | Interpretive |
|---|---|---|---|---|
| 1 | `open` | `open` | direct name match | no |
| 2 | `answered` | `answered` | direct name match | no |
| 3 | `rebutted` | `answered` | a rebuttal is a substantive reply that engaged the point | **yes** (alt: `open`) |
| 4 | `clarified` | `answered` | a clarification answered the definitional ask | mild |
| 5 | `sourced` | `answered` | a source was attached, discharging the ask; the point is engaged | mild |
| 6 | `quote_requested` | `quote_requested` | direct name match | no |
| 7 | `source_requested` | `source_requested` | direct name match | no |
| 8 | `narrowed` | `narrowed` | direct (REF-001 trace 3) | no |
| 9 | `conceded` | `conceded` | direct name match | no |
| 10 | `confirmed` | `answered` | confirmation settles/affirms the point (no `confirmed` IssueState) | **yes** |
| 11 | `synthesis_ready` | `synthesis_ready` | direct name match | no |
| 12 | `moved_on_by_affirmative` | `moved_on` | direct family | no |
| 13 | `moved_on_by_negative` | `moved_on` | direct family | no |
| 14 | `ignored_by_affirmative` | `moved_on` | unanswered → procedurally moved past (no `ignored` IssueState; "moved on" never names a person — LIFE-001 doctrine §3) | mild |
| 15 | `ignored_by_negative` | `moved_on` | same | mild |
| 16 | `ignored_by_both` | `moved_on` | same | mild |
| 17 | `exhausted` | `moved_on` | axis exhausted → past productive engagement | mild |
| 18 | `branch_recommended` | `moved_on` | direct (REF-001 trace 4, ratified) | no |
| 19 | `archived_or_resolved` | `answered` | reached resolution / settled | **yes** (alt: `moved_on`) |

> The live `PointLifecycleState` union (`pointLifecycleModel.ts:80`) has **19** values; REF-001's prose says "18" — the
> known descriptive nit (operator constraint 4). All 19 are enumerated above. The doctrine-positive consequence of
> rows 14–16: the side-specific `ignored_by_*` framing collapses to the person-neutral `moved_on` on the issue card, so
> the public surface never reads as an accusation that a side "ignored" anything.

**Step B — open-debt override (debt outranks lifecycle resolution; REF-001 line 195/231).** If the highest-precedence
open evidence debt is source-class with a still-owed status (`requested`/`challenged`/`unresolved`/`stale`) →
`source_requested`; if quote with a still-owed status → `quote_requested`. The override fires **even when** the base map
yielded `answered` (e.g. lifecycle `sourced` but a fresh quote debt is open → `quote_requested`).

### `id`
`` `issue:${targetNodeId}:${relationToParent}:${firstOpenDebtKind ?? axis}` `` where `firstOpenDebtKind` is the
highest-precedence open debt's `debtKind` (source-class before quote) when present, else the derived `axis`.
Deterministic + stable for identical inputs; never a DB row; never rendered raw.

### `refereeObservations` (≤ 1 on the public card; Family J never present)
1. If `bannerSelection?.banner` is non-null → one `IssueObservation`:
   `{ sourceCode: banner.bannerCode, line: banner.headline, toneGlyph: banner.toneGlyph, kind: 'machine_observation' }`.
   The banner is **consumed, never re-selected** (operator constraint 5 — no `selectBanner` call).
2. Else, optionally one summarized **production-family** (`A–I`) machine observation from `machineObservations`:
   `{ sourceCode: mark.rawKey, line: mark.label, toneGlyph: null, kind: 'machine_observation' }`, picked by
   lowest `priority` value. The **J gate** (below) drops any `sensitive_composer` mark first.
3. Else → `[]` (zone 1 falls back to the relation label).

Array length is always **≤ 1** (the card-level summary; Inspect's grouped provenance is REF-003/argumentDetailModel's
job, out of scope here).

### `userAllegations` (never merged into Observations)
Each input `userAllegations` mark (kind `user_allegation`) → one `IssueAllegation`
`{ sourceCode: mark.rawKey, line: mark.label, kind: 'user_allegation', visibility }`. **Visibility** from the mark's
disposition/surface: `disposition === 'composer_only'` or `defaultSurface === 'composer'` → `composer_only`;
`disposition === 'hidden_sensitive'` → `moderator_only`; else `public`. Person-directed concerns therefore never render
on the public card. Allegations are returned in their own field, **never** concatenated into `refereeObservations`.

### `nextBestMoves` (= suggestions ∩ surviving Act entries)
`buildOpenIssue` computes
`actGroups = buildActPopout({ targetKind: 'node', role: viewerRole, stage: lifecycleState, parentType, rules })`, then
calls `deriveNextBestMoves({ suggestionInput, actGroups, maxMoves: 3 })`. The join (REF-001 §"nextBestMoves semantics"):

**Step 1 — map each `SuggestedMoveCode` → `ActEntryId`** (exhaustive; test-pinned):

| `SuggestedMoveCode` | → `ActEntryId` |
|---|---|
| `ask_source` | `ask_source` |
| `ask_quote` | `ask_quote` |
| `narrow` | `narrow` |
| `concede` | `concede` |
| `confirm` | `confirm` |
| `challenge_mechanism` | `challenge` |
| `challenge_scope` | `narrow` |
| `branch_tangent` | `branch_tangent` |
| `synthesize` | `synthesize` |

**Step 2 — intersect with Act survivors.** Flatten `actGroups` to the set of surviving `ActEntryId`s. For each suggested
move in `deriveSuggestedMoves` priority order, up to `maxMoves`:
- mapped id **is** a survivor → emit `MoveSuggestion` (label + accessibilityLabel read from the surviving
  `ActPopoutEntry`), `isRecoveryRoute: false`.
- mapped id was **gate-removed** → do **not** emit the invalid action. Emit the **recovery route**: the `branch_tangent`
  survivor ("Open a side issue"), `isRecoveryRoute: true`, `recoveredFromCode: <the SuggestedMoveCode>`. If
  `branch_tangent` is itself not a survivor (e.g. observer role) → the suggestion is **dropped**, never forced.

**Doctrine consequence:** the card can never present a button the engine/role gate would reject — there is no
"click → rejected" path. `nextBestMoves ⊆ buildActPopout survivors` always.

### `openIssueToRefereeCard` (pure view-model mapping — consumes the contract, never re-derives)
- `zone1RelationLine` = `issue.refereeObservations[0]?.line ?? RELATION_LABEL[issue.relationToParent]`.
- `zone1ToneGlyph` = `issue.refereeObservations[0]?.toneGlyph ?? null`.
- `zone2OpenTaskLine` = when `burden !== 'none'`: `BURDEN_AXIS_LINE[burden][axis]`; when `burden === 'none'`:
  `ISSUE_STATE_TERMINAL_LINE[state]` (e.g. "Ready to synthesize." / "Conceded.").
- `zone3Moves` = `issue.nextBestMoves`.
- `refereeNoteSentence` = frozen template: `Referee note: {relation}. The open task is {burden}. Best next moves:
  {m1} · {m2} · {m3}.` — when `burden === 'none'` the middle clause reads the terminal line; when `nextBestMoves` is
  empty the "Best next moves" clause is omitted.
- `accessibilityLabel` = one complete sentence assembled from the same parts.

This function reads only already-derived fields; it **never** calls `selectBanner` and **never** re-implements banner
selection (operator constraint 5; no parallel referee surface).

## Family-J / sensitive gating (operator constraint; REF-001 §"sensitiveSurface")

The model **re-asserts** the production-roster gate even though upstream already filters, and **never hardcodes a family
roster** — it derives the allow-list from the mirror:

```
PRODUCTION_FAMILIES = ALL_MACHINE_OBSERVATION_FAMILIES.filter(f => !HUB_NON_PRODUCTION_FAMILIES.includes(f))   // A–I
markAllowed(mark) = resolveFamily(mark) === 'other' || PRODUCTION_FAMILIES.includes(resolveFamily(mark))
resolveFamily(mark) = lookupMachineObservationDefinitionByCompoundKey(mark.source, mark.rawKey)?.family ?? 'other'
```

Any `machineObservations` mark whose resolved family is `sensitive_composer` (Family J) is **dropped before**
`refereeObservations` is built — even when a J-derived mark is present in the input (test-pinned). This mirrors
`argumentDetailModel.isHubAllowedFamily` (:760) exactly, so a registry rename / new family flows through the single
mirror constant. Unknown rawKeys (`'other'`) are kept (grouped as a generic observation), identical to the hub.

## Module discipline

Total (never throws on well-typed input — every helper returns a default rather than throwing), deterministic (same
input → same output), JSON-serializable output (no functions, no `Map`/`Set`, no class instances in
`DisagreementContract`), non-mutating (inputs treated read-only; helpers build new arrays). **No** React, Supabase,
fetch/XMLHttpRequest, anthropic, xai, router, or `Date.now`. **Never** in the submit path — the model runs at render
time, after an argument is stored, and emits no block/reject/route/delay.

## Edge cases

- **Null `roomId` or `targetNodeId`** → `buildOpenIssue` returns `null`; no card mounts.
- **Root claim (`parentType === null`)** → engine gate keeps only `null`-`argumentType` Act entries;
  `relationToParent` defaults to `replies`; `nextBestMoves` may be `add_evidence`/`branch_tangent` only.
- **Observer role** → Act survivors are watch/join/chime/view-qualifiers only; none map to a `SuggestedMoveCode` →
  `nextBestMoves` is `[]`; zone 3 collapses; zones 1–2 still read the issue.
- **Own bubble** → Act survivors are `view_qualifiers` + `request_deletion` only → `nextBestMoves` `[]`; zone 3
  collapses (own-bubble safety: no edit/disagree/flag/score on your own move).
- **Empty `rules`** → all typed Act entries removed; only `null`-type entries can survive; if none, zone 3 collapses.
- **Gate-removed suggestion** → recovery route (`branch_tangent`) substitutes; if it too is removed, the suggestion is
  dropped — never a rejected button.
- **Multiple open evidence debts** → highest-precedence open debt (source-class before quote) drives
  axis/burden/state; the rest surface only in Inspect (out of scope here).
- **No banner** → zone 1 falls back to the relation label; `refereeObservations` may be a single mark-seeded
  observation, or `[]`.
- **Unknown / unmapped `IssueState` for the rail** → `toPlainLanguageOrSuppress` returns `null` → the rail entry is
  suppressed, never echoed raw.
- **Family J observation present in `machineObservations`** → dropped by the re-asserted J gate; never reaches
  `refereeObservations`; stays composer-only (test-pinned).
- **Person-directed user Allegation** → `visibility = composer_only`/`moderator_only`; never on the public card.
- **Heat / popularity / strength band tries to influence the issue → it does not.** No field reads engagement, views,
  virality, or a standing band (mirrors `suggestedMovesModel` doctrine). The derivation is a pure function of
  structural move facts.
- **Concurrent edits / async classifier arrival** → the issue is re-derived on each render from current inputs; a late
  classifier result simply changes the next derivation. Nothing blocks or delays the post.

## Doctrine self-check

**Acceptance-gate invariant (verbatim):**
*AI/MCP classifiers MUST NEVER be the submission acceptance gate. The deterministic rules engine
(`src/domain/constitution/engine.ts`, mirrored for the server at `supabase/functions/_shared/constitution/`) is the
sole gate. Classifiers run after an argument is stored. No path may block, reject, route, or delay an ordinary user
post.*
→ Respected: the Open Issue model is **consultative**, derives only, emits no block. `nextBestMoves` is a **subset** of
engine+role-gate survivors — it never resurrects a move the engine rejects and never gates one. The model runs at render
time, after storage; it never sits in the post path.

- **cdiscourse-doctrine §1 (no truth labels; score never blocks).** Every emitted label is from the frozen §4B copy
  maps; the 16 prohibited verdict/person tokens appear in this doc only as prohibitions and in the `banlist` test spec.
  The card never says a user is wrong, never accuses motive, always gives a move (or collapses cleanly). Relation/state
  describe the MOVE's gameplay state, never truth.
- **§2 (heat = activity/friction).** No field reads heat; `moved_on`/`open` describe procedural state, not importance.
- **§3 (popularity is not evidence).** No field reads engagement, views, virality, or a standing band. Evidence debt is
  an obligation marker, never a truth verdict.
- **§4 (AI moderator limits).** No AI call from the production app; the model consumes already-persisted Observations;
  it never triggers classification, never assigns a truth value, never returns an authoritative flag.
- **§5 (engine is sacred).** Types-only imports of domain unions; pure / side-effect-free / JSON-serializable; never
  re-validates or contradicts the engine.
- **§9 (plain language).** Rail states route through `toPlainLanguageOrSuppress`; unknown codes suppressed, never
  echoed; the `moved_on` addition keeps coverage green; zero raw `snake_case` in any user-facing string.
- **§10a (Observations vs Allegations).** `refereeObservations` (machine) and `userAllegations` (user) are separate
  fields, never merged. Sensitive Observations stay composer-only; Family J (`sensitive_composer`,
  `productionEnabled: false`) is **never** a derivation input — the model re-asserts the `HUB_NON_PRODUCTION_FAMILIES`
  gate, even when a J mark is in the input.
- **§10 (v1 scope guards).** No voting/winner scoring, OAuth, push, public API, or search introduced.
- **Doctrine edge case — "what if heat tries to influence the axis/burden? — it doesn't":** no input is a
  heat/popularity/standing signal; the derivation is a pure function of structural move facts.
