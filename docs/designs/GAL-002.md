## Goal

The Conversation Gallery already shows a per-card "micro-moment" entry hint (`microMomentLabel` on `ConversationEntryHint`), shipped in Stage 6.4. Today that hint is built from inline string literals authored locally inside `deriveConversationEntryHint` — `'Be the first rebuttal'`, `'Ask for the source'`, `'Watch first — quiet room'`, etc. The copy is correct, but the doctrine path that produced it pre-dates Wave 2: those strings are not routed through RULE-003, the hint cannot reach the SC-004 dock or the `quickActionPresets.ts` composer surface deterministically, and there is no formal coupling to the SW-002 `RoomActivityProfile.entryOpportunity` signal that LIFE-001 + META-001 + SW-002 now produce.

GAL-002 is a **refactor of that existing function** that:

1. Replaces the inline string literals with reads from RULE-003 (`getLifecycleUx`, `getManualTagUx`, `getAutoMetadataUx`).
2. Switches the derivation away from `bucket` (a gallery-internal classification) onto **root-cluster lifecycle state** when present, per the issue's "Timeline Tree Game Board scope additions" section (`Once LIFE-001 lands, the first-suggested-move hint source moves to the root-cluster lifecycle state of the room`).
3. Adds a deterministic mapping from the resulting hint to a `QuickActionLabel` preset and a `TimelineNodeActionDockActionCode`, so the SC-004 dock and the COMPOSER-001 preset map can both consume it.
4. Reconciles the 7-value gallery hint vocabulary (the issue's literal list: `Be the first rebuttal · Ask for the source · Challenge the mechanism · Narrow the claim · Offer synthesis · Watch first · Join when ready`) with the 9-value ST-002 `SuggestedMoveCode` union, reusing names where they overlap and introducing 3 new gallery-only meta-action codes for the room-entry meta-actions ST-002 does not cover.
5. Adds a freshness guarantee: every gallery load recomputes the hint from current inputs, no caching, no persisted state.

The doctrine constraints that shape the design (all from `cdiscourse-doctrine`, `timeline-grammar`, `expo-rn-patterns`, and the issue body):

1. **No AI call. No popularity / engagement input.** The deriver reads lifecycle / manual tag / auto metadata / SW-002 `entryOpportunity` only. It never reads move count, engagement velocity, view counts, or anything from the heat / temperament bands as a primary signal. SW-002's `entryOpportunity` is itself activity-derived (not popularity-derived), so consuming it does not violate doctrine §3.
2. **No truth / verdict / amplification / person-attribution tokens.** Every produced string passes the COPY-001 post-hardening ban-list, the LIFE-001 / META-001 forbidden-token lists, and the SW-002 activity ban-list. Plain-language copy comes from RULE-003 — never freshly authored inline.
3. **No forced gate.** The hint is advisory. An observer can ignore it; the explicit Join action on the in-room rail (Stage 6.4) remains the only join surface. The gallery `Pressable` still enters the room as observer when the user has not joined.
4. **No render of the hint chip beyond the existing micro-moment line.** The visual surface is unchanged from Stage 6.4: a single secondary line above the room body. GAL-002 does not add a new chip, a new badge, or a new action button to the gallery card.
5. **Score never blocks anything.** The hint is a navigation suggestion. It cannot prevent the user from observing, joining, or posting.

## What GAL-002 IS and IS NOT

### GAL-002 IS

- A pure-TS **refactor** of the existing `deriveConversationEntryHint(card)` function. Same call site (`ConversationGalleryScreen.tsx:289`), same per-load recomputation, broadened typed return value.
- A **strengthening** of the hint's plain-language contract. Verb-phrase + helper-line copy now reads from RULE-003 (`getLifecycleUx(state).label`, `getLifecycleUx(state).helperLine`) and from the gallery-only meta-action label table for the 3 room-entry meta-actions. No inline string literal escapes into the produced hint.
- A **deterministic mapping** to (a) a `QuickActionLabel` preset (when a composer action lines up) and (b) a `TimelineNodeActionDockActionCode` (when a dock action lines up). Both are `| null` for the room-entry meta-actions that don't open the composer.
- A **derivation** that recomputes on every call. The deriver is a stateless pure function — no module-level cache, no memoisation, no closure-captured `WeakMap`. The Stage 6.3 / 6.4 gallery already rebuilds cards on every load via `useGalleryArguments`; GAL-002 keeps the model layer matching that contract.
- An **enrichment of `ConversationGalleryCard`** with a single optional field: `rootClusterLifecycleState?: PointLifecycleState | null`. This is the LIFE-001 input the issue calls out. The gallery model already consumes lifecycle-shaped tag/flag inputs; the new field is a typed conduit for the cluster-level lifecycle state plumbed from the upcoming gallery-load query. Until the loader populates it, the deriver gracefully falls back to bucket-driven derivation (parity with Stage 6.4 behaviour).

### GAL-002 IS NOT

- **A render component.** `ConversationGalleryScreen.tsx` already renders the micro-moment line inside the room shell (via `ArgumentGameSurface.entryHint`). GAL-002 does not add a render surface. The only ConversationGalleryScreen change is the import + type widening of the returned hint shape (additive — existing fields preserved).
- **A new gallery section restructure.** That work is GAL-001. GAL-002 keeps `groupGalleryCardsBySection` exactly as-is.
- **An AI call.** No `fetch`, no `anthropic`, no `xai`. The deriver is sync and pure.
- **A forced action gate.** The hint is advisory and the user can ignore it. Consistent with ST-002's `Score never blocks posting` rule.
- **A popularity / engagement signal.** No move count, no participant count, no virality, no follower count read as a primary signal. SW-002's `entryOpportunity` is consumed because SW-002 itself is doctrine-compliant (activity-only); the deriver still ignores raw counts.
- **A persisted hint store.** No Supabase write, no AsyncStorage, no module-level memo. Per-load recomputation only.
- **A new quick-action preset.** Every `presetKey` mapping is to an existing `QuickActionLabel` member (`reply`, `source`, `challenge`, `narrow`, `concede`, `synthesize`, etc.). New presets are a follow-up card (PRESET-001 / GAL-003).
- **A render of the SW-002 entry-opportunity signal beyond the hint.** GAL-002 reads `entryOpportunity` as a tie-breaker input; it does not surface the SW-002 chip on the gallery card. That is SW-001 / future work.

## Reconciliation with the existing `deriveConversationEntryHint`

This is the most important design decision in GAL-002. There are two reasonable options:

### Option A — New sibling function (`deriveGalleryEntryHint`), old function deprecated

Add a new function `deriveGalleryEntryHint(card): GalleryEntryHint` to `conversationGalleryModel.ts`. Mark the existing `deriveConversationEntryHint` `@deprecated`. Migrate `ConversationGalleryScreen.tsx` to call the new function and pass a forward-compatible hint shape. Leave the old function in place for one release so any external test fixture that asserts on `microMomentLabel` continues to work; remove in a follow-up.

### Option B — Replace `deriveConversationEntryHint` with `deriveGalleryEntryHint`

Rename the function, replace the `ConversationEntryHint` type with a broader `GalleryEntryHint`, and update every caller in the same PR. This includes `ConversationGalleryScreen.tsx`, the `ArgumentGameSurface.entryHint` prop type (`src/features/arguments/ArgumentGameSurface.tsx:120`), the `ArgumentTreeScreen.tsx:47` + `:242` entry-hint prop pass-through, the `App.tsx:92` `entryHint` state, and the two existing test files (`__tests__/seamlessConversationEntry.test.ts` and `__tests__/roomEntryDefaultMode.test.ts`).

### Chosen design: **Option B — Replace**.

Justification:

1. The existing `ConversationEntryHint` type is a 2-field shape (`activate` + `microMomentLabel`). The GAL-002 hint shape extends it with `code` + `verbPhrase` + `helperLine` + `presetKey` + `dockAction`. A clean replacement avoids a permanent two-shape co-existence and a deprecation tail.
2. The number of callers is small and entirely in-repo (5 production files + 2 tests). All callers are already aware of the hint type; no external API surface depends on it.
3. The `activate` field (`'root' | 'latest' | 'first_open_challenge'`) is **preserved verbatim** on the new type. The room-shell behaviour of pre-activating the right message is unchanged. Replacement is purely an extension of the existing shape, not a behaviour change to the activate semantics.
4. The function name change clarifies intent — "gallery entry hint" is what this is; "conversation entry hint" was a Stage 6.4 placeholder name that pre-dates the gallery-specific framing.

## Data model

### New types in `src/features/debates/conversationGalleryModel.ts`

```ts

 * GAL-002 — Gallery entry hint vocabulary. The 7 codes from the issue.
 *
 * 4 codes overlap with ST-002's `SuggestedMoveCode` (`ask_source`,
 * `challenge_mechanism`, `narrow`, `synthesize`) and reuse the exact
 * string spelling for symmetry. 3 codes are gallery-only meta-actions
 * (room-entry choices; not move-type choices) and have no ST-002
 * equivalent.
 */
export type GalleryEntryHintCode =
  | 'be_first_rebuttal'
  | 'ask_source'
  | 'challenge_mechanism'
  | 'narrow'
  | 'synthesize'
  | 'watch_first'
  | 'join_when_ready';

export const ALL_GALLERY_ENTRY_HINT_CODES: ReadonlyArray<GalleryEntryHintCode> = Object.freeze([
  'be_first_rebuttal',
  'ask_source',
  'challenge_mechanism',
  'narrow',
  'synthesize',
  'watch_first',
  'join_when_ready',
]);

/**
 * GAL-002 — Hint shape. Replaces Stage 6.4 `ConversationEntryHint`.
 *
 * - `activate` preserves the Stage 6.4 semantic (which message the room
 *   shell pre-activates on entry).
 * - `code` is the typed union member.
 * - `verbPhrase` ≤ 8 words. The call-to-action the gallery card renders.
 * - `helperLine` is the RULE-003 helper-line (when lifecycle-derived)
 *   or a constant gallery-meta helper (when meta-action-derived).
 *   `verbPhrase + helperLine` ≤ 200 chars total.
 * - `presetKey` is an existing `QuickActionLabel` value, or `null` for
 *   the 3 gallery-only meta-actions.
 * - `dockAction` is the matching SC-004 dock code, or `null` when no
 *   dock action lines up.
 */
export interface GalleryEntryHint {
  activate: 'root' | 'latest' | 'first_open_challenge';
  code: GalleryEntryHintCode;
  verbPhrase: string;
  helperLine: string;
  presetKey: QuickActionLabel | null;
  dockAction: TimelineNodeActionDockActionCode | null;
}

/**
 * GAL-002 — Card-level extension. The gallery card already carries
 * everything the deriver needs except the LIFE-001 root-cluster
 * lifecycle state. Optional + nullable so loaders that pre-date the
 * field still produce valid cards; the deriver falls back to the
 * Stage 6.4 bucket-driven branch when this is null.
 */
export interface ConversationGalleryCard {
  // … existing fields preserved …
  /**
   * GAL-002 — Root-cluster lifecycle state (LIFE-001). When present,
   * primary signal for `deriveGalleryEntryHint`. When null, the
   * deriver falls back to the existing bucket-driven derivation.
   * Loader-populated; the model does not derive it.
   */

  /**
   * GAL-002 — Optional SW-002 `entryOpportunity` snapshot. When
   * present, used as a tie-breaker between `watch_first` and
   * `join_when_ready` (and to escalate to `narrow`/`synthesize` on
   * deep-existing-clash). Loader-populated.
   */
  entryOpportunity?: EntryOpportunity | null;
}
```

The `ConversationGalleryCard` extension is **additive** — every existing field is preserved unchanged, so existing tests / fixtures / call sites continue to compile. The two new fields are optional + nullable; the deriver tolerates their absence.

## API / interface contracts

### Public signature

```ts
export function deriveGalleryEntryHint(card: ConversationGalleryCard): GalleryEntryHint;
```

- **Pure.** No React, no Supabase, no fetch, no `Math.random`, no `Date.now`.
- **Total.** Returns a valid `GalleryEntryHint` for every input. The fallback hint (`watch_first`) covers the case where no signal applies.
- **Deterministic.** Same input → deep-equal output. No closure-captured state.
- **No caching.** Each call recomputes from `card.rootClusterLifecycleState`, `card.bucket`, `card.hasNoRebuttal`, `card.entryOpportunity`, `card.openStatus`. No `WeakMap`, no module-level memo.

### Verb-phrase + helper-line tables

```ts
/**
 * Verb phrases — each ≤ 8 words. Sourced from:
 *  - RULE-003 `getLifecycleUx(state).label` for lifecycle-derived hints
 *    (lifecycle labels are already ≤ 32 chars by RULE-003 contract).
 *  - The 3 gallery-meta phrases below for the 3 room-entry meta-action
 *    codes (`be_first_rebuttal` / `watch_first` / `join_when_ready`).
 *
 * The 3 meta phrases are short, plain-language, ban-list-clean. They
 * are the ONLY new authored strings introduced by GAL-002 and they pass
 * the COPY-001 + SW-002 + LIFE-001 + META-001 forbidden-token lists.
 */
const META_VERB_PHRASES = {
  be_first_rebuttal: 'Be the first rebuttal',     // 4 words
  watch_first: 'Watch first',                      // 2 words
  join_when_ready: 'Join when ready',              // 3 words
} as const;
```

For the 4 lifecycle-overlap hints (`ask_source`, `challenge_mechanism`, `narrow`, `synthesize`), the verb phrase comes from the relevant SC-004 dock label OR the matching RULE-003 lifecycle label, capped at 8 words. The implementer should use the following sourcing rule (verified during implementation):

- `ask_source` → `getLifecycleUx('source_requested').label` (RULE-003 plain label, currently "Source requested" — 2 words).
- `challenge_mechanism` → Falls back to a verb phrase composed from the META-001 `causal_mechanism` manual-tag label via `getManualTagUx('causal_mechanism').label`. If that label is too long (verify ≤ 8 words at implementation time), use the SC-004 dock label `'Challenge mechanism'` from `ACTION_LABELS[challenge]` — but only if it's already in the ban-list-cleared SC-004 vocabulary. Implementer note: the ST-002 model uses the string literal `'Challenge mechanism'` at `SUGGESTED_MOVE_LABELS.challenge_mechanism` (line 246 of `suggestedMovesModel.ts`); reusing that exact string keeps both models in lockstep without inventing a new phrase.
- `narrow` → `getLifecycleUx('narrowed').label` (RULE-003 plain label, "Narrowed").
- `synthesize` → `getLifecycleUx('synthesis_ready').label` (RULE-003 plain label, "Synthesis ready").

If any of these RULE-003 labels exceeds 8 words (they don't today, but the implementer must verify at write time), the deriver caps at 8 words by truncating-on-whitespace.

## Derivation logic

The deriver is a single function with the following priority order. Each rule returns a `GalleryEntryHint` and the function returns the first match. The fallback at the bottom (`watch_first`) covers the case where no signal applies (cool, archived, or signal-less rooms).

### Priority order

1. **Empty-thread / no-rebuttal short-circuit.**
   - If `card.hasNoRebuttal === true` OR `card.moveCount === 0` → `be_first_rebuttal` (activate: `'root'`).
2. **Root-cluster lifecycle state — primary signal (LIFE-001).**
   - If `card.rootClusterLifecycleState` is non-null, dispatch on it per the Mapping table below.
3. **Bucket fallback (Stage 6.4 parity).**
   - If `rootClusterLifecycleState` is null/absent, dispatch on `card.bucket` per the same Mapping table's "Stage 6.4 bucket parity" column. This preserves the existing behaviour for any loader that does not yet plumb the lifecycle state.
4. **SW-002 `entryOpportunity` tie-breaker.**
   - If `entryOpportunity === 'easy_first_move'` and the fallback would otherwise pick `watch_first`, promote to `watch_first` (no change) — the meta-action stays. If `entryOpportunity === 'deep_existing_clash'` and the fallback would pick `watch_first`, promote to `narrow` or `synthesize` per the Mapping table.
   - If `entryOpportunity === 'mid_thread_join'` and the fallback would pick `watch_first`, promote to `join_when_ready`.
5. **Archived / locked / draft override.**
   - If `card.openStatus !== 'open'` → `watch_first` (activate: `'latest'`). The hint reads "Watch first — this room is closed" via the helper line. (The implementer reads `getLifecycleUx('archived_or_resolved').helperLine` when the state is also `archived_or_resolved`.)
6. **Neutral fallback.**
   - `watch_first` (activate: `'latest'`).

### Doctrine §3 carve-out

Per `cdiscourse-doctrine` §3 ("Popularity is not evidence"), the hint MUST NOT be derived from move count alone. The deriver reads:

- `rootClusterLifecycleState` (LIFE-001 — structural, no engagement input)
- `bucket` (an already-doctrine-clean classification from Stage 6.3; the bucket itself does not encode popularity — it encodes friction / unresolved-axis pressure / source-chain risk)
- `hasNoRebuttal` (structural: 0 rebuttals exist)
- `entryOpportunity` (SW-002 — activity, not popularity)
- `openStatus` (debate row field)

The deriver explicitly does NOT read `moveCount`, `rebuttalCount`, `participantCount`, `heatScore`, `heatLevel` (as a *primary* hint signal), `temperament`, or any sortKey. A dedicated test (`doctrine: hint is not derived from move count alone`) constructs two cards with identical lifecycle / bucket / no-rebuttal / openStatus but vastly different `moveCount` (3 vs 30) and asserts the hint is the same.

## Edge cases

The deriver is total; every branch is documented.

| Case | Behaviour |
| --- | --- |
| Card with `moveCount === 0` (no messages yet) | `be_first_rebuttal`. The card may have `hasNoRebuttal === false` if there's no root either; the `moveCount === 0` short-circuit catches both. |
| Card with `moveCount === 1` and `rebuttalCount === 0` (root posted, no replies) | `be_first_rebuttal` (because `hasNoRebuttal === true`). |
| Card with `rootClusterLifecycleState === null` | Falls back to bucket-driven derivation. The hint is fully populated; the `code` is determined by the bucket-parity column above. |
| Card with `rootClusterLifecycleState === undefined` (loader did not plumb it) | Same as `null` — falls back to bucket. The deriver does not throw on `undefined`. |
| Card with `archived_or_resolved` lifecycle | `watch_first`, activate `latest`. The helper line uses `getLifecycleUx('archived_or_resolved').helperLine` ("An admin or synthesis closed this cluster."). |
| Card with `openStatus === 'archived'` but `rootClusterLifecycleState !== 'archived_or_resolved'` | The `openStatus` override fires first → `watch_first`. |
| Card with `openStatus === 'draft'` | `watch_first`. The user can still observe a draft room. |
| Card with `openStatus === 'locked'` | `watch_first`. |
| Card with `entryOpportunity === 'easy_first_move'` and quiet lifecycle | Hint stays `watch_first` (per the rule "if the deriver would already pick `watch_first`, leave it"). |
| Card with `entryOpportunity === 'deep_existing_clash'` and the fallback would otherwise pick `watch_first` | Promoted to `narrow`. Rationale: a deep clash means there's a structural foothold for a first move; narrowing scope is the gallery-appropriate suggestion. |
| Card with `entryOpportunity === 'mid_thread_join'` and the fallback would otherwise pick `watch_first` | Promoted to `join_when_ready`. |
| Card with multiple unresolved metadata (e.g. both `source_requested` AND `definition_issue` manual tags on the root cluster) | The deriver dispatches on `rootClusterLifecycleState` first; the lifecycle's worst-priority state per LIFE-001's priority table (`LIFECYCLE_PRIORITY`) wins. No special-case logic in GAL-002 — LIFE-001 already resolves priority. |
| Card from a generated / test duplicate room (Stage 6.3 collapse) | Hint derives normally from the primary card's lifecycle + bucket. Duplicate collapse is visual; the deriver runs against the merged primary card. |
| Card with no `bucket` (impossible — `bucket` is always populated by `classifyConversationBucket`) | The deriver still works; the bucket fallback table has a row for every `ConversationBucket` value including `'all_open'`. |
| Card with `hasUserJoined === true` (the user is already in the room) | The deriver does not branch on `hasUserJoined` for the hint code — the hint describes the *next move shape*, not the relationship to the user. The render layer (gallery card action text) is responsible for showing "Continue →" vs "Observe →"; that is unchanged from Stage 6.4. |
| All inputs missing / minimal card (`hasNoRebuttal: false`, `moveCount: 0`, `bucket: 'all_open'`, no lifecycle, no entryOpportunity, `openStatus: 'open'`) | `be_first_rebuttal` (because of the `moveCount === 0` short-circuit). |

## Risks

1. **The `activate` field is preserved verbatim.** Existing seamless-entry tests assert on `activate` values per bucket. If the implementer accidentally shifts an `activate` choice (e.g. from `'latest'` to `'root'`) the room shell will pre-activate a different message and the existing tests will fail. The Mapping table above pins every `activate` choice to the Stage 6.4 behaviour; the implementer must double-check by running the existing tests with the new function before adjusting any row.
2. **The `ConversationGalleryCard` shape extension is additive.** New optional fields (`rootClusterLifecycleState`, `entryOpportunity`). Some test fixtures build minimal cards inline; those fixtures keep working because the fields are optional. The deriver tolerates `undefined`. Verify the existing `conversationGalleryModel.test.ts` fixtures still pass without modification (they should).
3. **RULE-003 label length check at implementation time.** The verb-phrase table reads RULE-003 labels. If a RULE-003 label ever exceeds 8 words (none does today), the deriver must cap on word boundary. The implementer adds a defensive cap in the helper, but a test pinning the cap behaviour (`a 12-word label is truncated to 8 words plus ellipsis`) protects against future RULE-003 changes.
4. **Cross-call mutation risk.** The deriver is pure, but `ConversationGalleryCard` is a regular interface — callers can mutate the card between two calls. The freshness test asserts the deriver re-reads on each call (no caching), so mutation is correctly observed.
5. **The 3 new authored strings (`META_HELPER_LINES`).** Three short plain-language phrases are the only new copy GAL-002 introduces. They pass the ban-list, but they are still a small new copy surface. The implementer ensures each string is reviewed against the COPY-001 ban-list (the test catches obvious tokens; the implementer should also read them for tone).
6. **`ArgumentGameSurface.tsx` render-layer change.** The micro-moment banner today renders a single line; GAL-002 turns it into a two-line block. This is a tiny visual change but visible. The implementer must verify on web + native that the banner does not break the surrounding layout (current banner is a 1-line View with `paddingHorizontal: 16` etc.; adding a second line increases height by ~16-20px). If layout is too tight, the implementer falls back to rendering only `verbPhrase` and exposing `helperLine` via `accessibilityLabel` only. The fallback is doctrine-clean (still plain-language) and reduces the visible-surface delta to zero.
7. **Test-count discipline.** Adding ~25-30 tests must not regress any existing test. The existing 1805 tests / 70 suites are the baseline; GAL-002 lands at +25 to +35 new tests, no removals. The implementer must run the full suite before commit and confirm.
